# 主从复制原理

## 这个问题为什么存在？

> 单台数据库承受所有读写压力，性能瓶颈明显。
> 单台数据库如果崩溃，数据可能丢失（即使有Redo Log，恢复也需要时间）。

**核心需求**：
1. **读写分离**：主库写，从库读，提升整体性能
2. **数据冗余**：从库是主库的副本，主库崩溃可以切换
3. **备份**：从库可以停止复制，用于备份

---

## 它是怎么解决问题的？

### 主从复制的核心原理

```
主从复制流程（异步复制）
┌──────────┐  1.写 binlog  ┌──────────┐
│  主库       │  ──────────>  │  binlog    │
│  (Master)  │               │  (二进制日志)│
└──────────┘               └──────────┘
                                       │
                                       │ 2.推送 binlog
                                       ▼
                              ┌──────────────────┐
                              │  IO线程（从库）    │
                              │  拉取 binlog       │
                              └──────────────────┘
                                       │
                                       │ 3.写入 relay log
                                       ▼
                              ┌──────────────────┐
                              │  relay log          │
                              │  (中继日志)        │
                              └──────────────────┘
                                       │
                                       │ 4.读取 relay log
                                       ▼
                              ┌──────────────────┐
                              │  SQL线程（从库）   │
                              │  重放 binlog       │
                              └──────────────────┘
                                       │
                                       ▼
                              ┌──────────────────┐
                              │  从库数据           │
                              └──────────────────┘
```

### 核心组件

| 组件 | 作用 | 位置 |
|------|------|------|
| binlog（二进制日志） | 记录主库的数据变更（DDL+DML） | 主库 |
| relay log（中继日志） | 从库暂存从主库拉取的binlog | 从库 |
| IO线程 | 从库的IO线程，拉取主库的binlog | 从库 |
| SQL线程 | 从库的SQL线程，重放relay log | 从库 |
| dump线程 | 主库为每个从库创建dump线程，推送binlog | 主库 |

### binlog的三种格式

| 格式 | 内容 | 优点 | 缺点 |
|------|------|------|------|
| STATEMENT | SQL语句本身 | 日志量小 | 可能有主从不一致（如UUID()、NOW()） |
| ROW | 行的变更（前镜像+后镜像） | 主从完全一致 | 日志量大 |
| MIXED | 混合模式（默认STATEMENT，必要时用ROW） | 平衡 | - |

```sql
-- 查看binlog格式
SELECT @@binlog_format;  -- 默认ROW

-- 修改binlog格式（需要重启）
SET GLOBAL binlog_format = 'ROW';

-- 永久配置（my.cnf）
-- [mysqld]
-- binlog_format = ROW
-- binlog_row_image = FULL  -- 记录前镜像和后镜像
```

**为什么推荐ROW格式？**

```
STATEMENT格式的问题：
主库：INSERT INTO t VALUES (UUID());  -- UUID()在从库执行，结果不同
→ 主从数据不一致

ROW格式：
记录的是行的变更（哪个字段从什么值改成什么值）
→ 主从完全一致，但从库压力大时，binlog传输可能延迟
```

## 它和相似方案的本质区别是什么？

### 异步复制 vs 半同步复制 vs 组复制

| 维度 | 异步复制（默认） | 半同步复制 | 组复制（MGR） |
|------|----------------|-------------|-------------|
| 一致性 | 最终一致（可能丢数据） | 强一致（至少同步到一个从库） | 强一致（多数派确认） |
| 性能 | 最好 | 中等（多一个ACK） | 较差（需要协调） |
| 配置复杂度 | 简单 | 中等 | 复杂 |
| 适用场景 | 可以接受秒级延迟 | 不能丢数据（金融） | 需要自动故障转移 |

**异步复制的问题**：

```
主库提交事务 → 返回成功给客户端
                ↓
        异步推送binlog到从库
        
如果主库崩溃，且binlog还没推送到从库
→ 从库提升为主库后，这部分事务丢了
```

**半同步复制的解决**：

```
主库提交事务 → 等待至少一个从库ACK（收到binlog）
                ↓
        返回成功给客户端
        
保证：只要客户端收到"成功"，这个事务一定已经同步到至少一个从库
```

**配置半同步复制**：

```sql
-- 主库安装插件
INSTALL PLUGIN rpl_semi_sync_master SONAME 'semisync_master.so';

-- 从库安装插件
INSTALL PLUGIN rpl_semi_sync_slave SONAME 'semisync_slave.so';

-- 主库开启半同步
SET GLOBAL rpl_semi_sync_master_enabled = 1;

-- 从库开启半同步
SET GLOBAL rpl_semi_sync_slave_enabled = 1;
STOP SLAVE IO_THREAD;
START SLAVE IO_THREAD;
```

## 正确使用方式

### 1. 配置主从复制（GTID模式，推荐）

**GTID（Global Transaction ID）**：每个事务都有一个全局唯一的ID，方便故障切换。

**主库配置（my.cnf）**：

```ini
[mysqld]
server-id = 1
log-bin = mysql-bin
binlog_format = ROW
binlog_row_image = FULL
gtid_mode = ON
enforce_gtid_consistency = ON
```

**从库配置（my.cnf）**：

```ini
[mysqld]
server-id = 2
relay-log = relay-bin
read-only = 1  # 从库只读（super用户除外）
gtid_mode = ON
enforce_gtid_consistency = ON
```

**建立复制关系**：

```sql
-- 主库：创建复制用户
CREATE USER 'repl'@'%' IDENTIFIED BY 'password';
GRANT REPLICATION SLAVE ON *.* TO 'repl'@'%';

-- 从库：配置主库信息
CHANGE MASTER TO
  MASTER_HOST = 'master_host',
  MASTER_USER = 'repl',
  MASTER_PASSWORD = 'password',
  MASTER_AUTO_POSITION = 1;  # GTID模式，自动找位置

-- 启动复制
START SLAVE;

-- 查看复制状态
SHOW SLAVE STATUS\G
-- 关键字段：
-- Slave_IO_Running: Yes  → IO线程正常
-- Slave_SQL_Running: Yes → SQL线程正常
-- Seconds_Behind_Master: 0 → 主从延迟0秒
```

### 2. 监控主从延迟

```sql
-- 从库执行
SHOW SLAVE STATUS\G

-- 关键字段：
Seconds_Behind_Master  -- 主从延迟（秒）

-- 更准确的方式：对比主从的binlog位置
-- 主库：
SHOW MASTER STATUS\G
-- 从库：
SELECT 
    MASTER_POS_WAIT('mysql-bin.000001', 1234, 0);  
    -- 等待从库同步到指定位置，返回等待秒数
```

**通过shell调用长查询监控（不在SQL中运行）**：

```bash
# 持续监控主从延迟
watch -n 1 "mysql -e 'SHOW SLAVE STATUS\G' | grep 'Seconds_Behind'"
```

### 3. 处理主从延迟

**常见原因**：

```
1. 从库硬件差（CPU、磁盘比主库差）
2. 从库单线程重放（SQL线程只有一个）
3. 从库有大查询，阻塞了复制
4. 网络延迟（主库和从库跨机房）
5. 主库写入压力大，从库追不上
```

**优化方案**：

```sql
-- 方案1：从库开启并行复制（MySQL 5.7+）
SET GLOBAL slave_parallel_workers = 8;  -- 8个并行线程
SET GLOBAL slave_parallel_type = 'LOGICAL_CLOCK';

-- 方案2：从库使用更快的磁盘（SSD）
-- 方案3：读写分离时，关键查询走主库（避免延迟导致读不到最新数据）
```

## 边界情况和坑

### 坑1：主从数据不一致

```
现象：
- 主从延迟为0
- 但查询结果不一致

原因：
1. 主库用了STATEMENT格式的binlog，且有非确定性函数（UUID()、NOW()、RAND()）
2. 从库被手动写入了数据（违反了"从库只读"原则）
3. 主从版本不一致，某些SQL在主从表现不同
```

**解决**：

```sql
-- 1. 主从都用ROW格式的binlog
SET GLOBAL binlog_format = 'ROW';

-- 2. 从库设置只读（超级用户除外）
SET GLOBAL read_only = 1;

-- 3. 定期用pt-table-checksum检查主从一致性
-- pt-table-checksum --databases=db_name h=master_host
```

### 坑2：从库SQL线程追不上主库

```
现象：
Seconds_Behind_Master 越来越大
从库CPU使用率很低（单线程重放）

原因：
MySQL 5.6及之前，SQL线程是单线程的
主库并发写入，从库单线程重放 → 追不上
```

**解决**：

```sql
-- MySQL 5.7+：开启并行复制
SET GLOBAL slave_parallel_workers = 8;
SET GLOBAL slave_parallel_type = 'LOGICAL_CLOCK';
-- LOGICAL_CLOCK：基于组提交（Group Commit）的并行重放
```

### 坑3：读写分离，读到脏数据

```
场景：
1. 主库写入订单（事务提交）
2. 立即读订单列表（读从库）
3. 从库延迟2秒 → 读不到刚才写入的订单

后果：
- 用户刚下完单，跳到订单列表，看不到订单
- 用户以为下单失败，重复下单
```

**解决**：

```sql
-- 方案1：刚写入后，强制读主库（粘性读）
-- 比如：下单后，跳转到订单详情页（按订单ID查，走主库）

-- 方案2：用全局事务ID（GTID）判断从库是否已经同步
SELECT WAIT_UNTIL_SQL_THREAD_AFTER_GTIDS('uuid:tx_id', 1);
-- 等待从库同步到指定GTID，超时1秒

-- 方案3：关键业务强制走主库，非关键业务走从库
```

### 坑4：从库被写入，复制断裂

```
现象：
SHOW SLAVE STATUS\G
Slave_SQL_Running: No
Last_SQL_Error: ...Duplicate entry '123' for key 'PRIMARY'...

原因：
- 从库被手动写入了数据（比如直接INSERT）
- 主库后续写入相同数据 → 主键冲突 → SQL线程停止
```

**解决**：

```sql
-- 1. 从库设置只读
SET GLOBAL read_only = 1;
SET GLOBAL super_read_only = 1;  -- 连super用户也只读

-- 2. 如果已经冲突，手动跳过这个事务（慎重！）
STOP SLAVE;
SET GLOBAL sql_slave_skip_counter = 1;
START SLAVE;

-- 或者，通过shell调用pt-slave-restart自动跳过错误
-- pt-slave-restart --error-numbers=1062,1032 h=slave_host
```
