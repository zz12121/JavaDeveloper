## 参考答案

### 一、最简实现：SET NX EX + Lua 脚本释放

```
加锁（一条命令，原子）：
  SET lock:order:123  <UUID>  NX  EX  30
                      ↑唯一标识  ↑互斥  ↑超时防死锁

释放锁（Lua 脚本，原子）：
  先比较 value 是不是自己的，是才 DEL
```

**为什么释放锁必须用 Lua 脚本？**

```
❌ 错误：GET + DEL 两步操作，不是原子的
  T1: GET 比较，value 匹配 = 自己的锁
  T2: 锁刚好超时，被其他客户端占了
  T3: DEL → 删掉了别人的锁！

✅ 正确：Lua 脚本，GET + DEL 在 Redis 单线程中原子执行
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
```

### 二、3 大经典坑

#### 坑 1：锁超时，但业务没执行完（最常见 ⚠️）

```
T0:  加锁成功，超时 30 秒
T15: Full GC 停顿 20 秒（或下游服务超时）
T35: GC 结束，锁已超时被删，其他客户端已获锁
T36: 业务执行完，调用 unlock() → 删掉了别人的锁！
```

**解决方案**：
1. **watchdog 自动续期**（Redisson 默认开启，推荐）
2. 锁超时时间设长（治标不治本）
3. **业务幂等设计**（最终兜底）

---

#### 坑 2：Redis 主从切换，锁丢失

```
客户端 A 在 master 加锁成功
master 挂了，还未同步给 slave
slave 晋升为新 master（锁数据丢失）
客户端 B 在新 master 也加锁成功
→ 两个客户端同时持有锁！
```

**解决方案对比**：

| 方案 | 原理 | 缺点 |
|------|------|------|
| 接受风险 | 业务幂等兜底 | 偶尔重复执行 |
| ZooKeeper 锁 | CP 系统，强一致，不会丢锁 | 性能差（约 1/3 的 Redis） |
| Redlock | 向 N 个独立 Redis 节点加锁，过半成功 | 依赖系统时钟，争议大，性能差 |

> 生产建议：**大部分场景用单节点 + watchdog**，对一致性极高要求才考虑 ZooKeeper。

---

#### 坑 3：value 不是唯一标识

```java
// ❌ 错误：value 固定，任何客户端都能释放锁
jedis.set("lock:key", "1", "NX", "EX", 30);
jedis.del("lock:key");  // 谁都能删！

// ✅ 正确：value 用 UUID，加锁和释放用同一个 UUID
String value = UUID.randomUUID().toString();  // 每次加锁生成唯一 ID
jedis.set("lock:key", value, "NX", "EX", 30);
// 释放时用 Lua 脚本比较 UUID 再删除
```

### 三、watchdog 机制（Redisson 核心）

```
加锁时不设置超时（或设置较短时间）
→ 后台 watchdog 线程每 10 秒检查一次
→ 如果锁还被当前客户端持有，自动续期到 30 秒
→ 业务执行完 unlock()，watchdog 停止

效果：锁的有效期 = 业务真实执行时间，不会超时
```

```java
// Redisson 使用示例（推荐直接用，不要自己实现）
RLock lock = redissonClient.getLock("lock:order:123");
try {
    lock.lock();         // 加锁，watchdog 自动续期
    processOrder();      // 业务处理
} finally {
    if (lock.isHeldByCurrentThread()) {
        lock.unlock();   // 释放锁，watchdog 停止
    }
}
```

### 四、Redlock 算法（了解即可）

向 **N 个独立 Redis 节点**（无主从关系）依次加锁：
- 超过半数（N/2 + 1）成功，且总耗时 < 锁超时时间 → 加锁成功
- 否则向所有节点发送释放锁命令

**争议**：Martin Kleppmann（DDIA 作者）认为系统时钟漂移会导致 Redlock 失效；Redis 作者 Antirez 反驳。**实际生产中大多数场景不需要 Redlock**。
