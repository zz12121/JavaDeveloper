## 知识链提示

```
事务与 ACID
  → [[06_数据库/03_事务与锁/ACID实现原理]]
    → Atomicity（原子性）：全部成功或全部失败 → undo log
    → Consistency（一致性）：事务前后数据完整性不变 → 由 A/I/D 共同保证
    → Isolation（隔离性）：事务之间互不干扰 → 锁 + MVCC
    → Durability（持久性）：提交后数据永久保存 → redo log
  → 事务隔离级别
    → [[06_数据库/03_事务与锁/事务隔离级别]]
      → READ UNCOMMITTED：脏读 / 不可重复读 / 幻读 都有
      → READ COMMITTED：解决脏读（MVCC 读已提交）
      → REPEATABLE READ（MySQL 默认）：解决不可重复读（MVCC 可重复读）
      → SERIALIZABLE：解决幻读（全表加锁，性能最差）
    → MySQL InnoDB 的 RR 级别**实际上也解决了幻读**
      → 靠 Next-Key Lock（行锁 + 间隙锁）
      → 但仅限于当前读（SELECT ... FOR UPDATE）
  → 三个并发问题
    → 脏读：读到了其他事务未提交的数据
    → 不可重复读：同一事务内，两次读同一行，结果不一样（被其他事务改了）
    → 幻读：同一事务内，两次范围查询，行数不一样（被其他事务插/删了）
```

## 核心追问

| 追问 | 考点 |
|------|------|
| RR 级别下，普通 SELECT 能看到幻读吗？ | 快照读看不到，当前读才能看到 |
| 什么是快照读？什么是当前读？ | MVCC 快照 vs 加锁读 |
| 为什么 RC 级别也会有不可重复读？ | 每次读都生成新快照 |
| 幻读和不可重复读的区别是什么？ | 不可重复读是「改」，幻读是「增/删」 |

---

## 参考答案要点

**ACID**：
- A：原子性 → undo log
- C：一致性 → 业务规则
- I：隔离性 → 锁 + MVCC
- D：持久性 → redo log

**隔离级别 vs 并发问题**：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---------|------|-----------|------|
| READ UNCOMMITTED | ✗ | ✗ | ✗ |
| READ COMMITTED | ✓ | ✗ | ✗ |
| REPEATABLE READ | ✓ | ✓ | ✗（InnoDB 实际解决了） |
| SERIALIZABLE | ✓ | ✓ | ✓ |
