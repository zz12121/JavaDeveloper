# JIT编译

## 这个问题为什么存在？

> 从「没有这个东西会怎样」出发，解释问题的根源。

如果没有JIT（Just-In-Time）编译器，Java程序只能**解释执行**：

```java
// 解释执行的代价
public class Test {
    public static void main(String[] args) {
        long sum = 0;
        for (int i = 0; i < 1000000; i++) {
            sum += i;  // 每次循环都要解释执行
        }
    }
}
```

**痛点**：
1. **启动快但运行慢**：解释器逐条解释字节码，执行速度慢
2. **无法优化**：解释执行无法做深度优化（如内联、逃逸分析）
3. **重复解释**：同一个方法被调用多次，每次都要解释

**核心问题**：如何让Java程序在**运行时**将热点代码编译为本地机器码，以提高执行速度？

JIT编译器的本质是**运行时优化**：在程序运行时，将频繁执行的热点代码编译为本地机器码，并以缓存起来，后续直接执行机器码。

## 它是怎么解决问题的？

> 不止讲「是什么」，要讲清楚「为什么这样设计能解决问题」。

### 核心机制

#### 1. 解释器 vs JIT编译器

```
┌─────────────────────────────────────────────────────────────┐
│                  解释器 vs JIT编译器                         │
│                                                             │
│  ┌─────────────────────────────┐  ┌─────────────────────┐│
│  │  解释器（Interpreter）     │  │  JIT编译器               ││
│  │  - 逐条解释字节码        │  │  - 热点代码编译为机器码  ││
│  │  - 启动快                │  │  - 运行快                ││
│  │  - 运行慢                │  │  - 启动慢（需要编译时间）││
│  └─────────────────────────────┘  └─────────────────────┘│
│                                                             │
│  HotSpot VM的解决方案：混合模式                              │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  1. 启动时使用解释器（启动快）                          │  │
│  │  2. 热点代码触发JIT编译（运行快）                      │  │
│  │  3. 后续直接执行编译后的机器码                            │  │
│  └──────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

**为什么需要混合模式？**
- **启动快**：解释器启动快，适合短期运行或冷启动场景
- **运行快**：JIT编译器将热点代码编译为本地机器码，执行速度快

#### 2. 分层编译（Tiered Compilation）

JDK 7引入分层编译，将JIT编译分为**5个层次**：

```
┌─────────────────────────────────────────────────────────────┐
│                  分层编译（Tiered Compilation）            │
│                                                             │
│  层次0：解释执行（Interpreter）                           │
│       │                                                    │
│       ▼                                                    │
│  层次1：C1编译（Client Compiler，简单优化）              │
│       │                                                    │
│       ▼                                                    │
│  层次2：C1编译（带Profile信息）                         │
│       │                                                    │
│       ▼                                                    │
│  层次3：C2编译（Server Compiler，深度优化）              │
│       │                                                    │
│       ▼                                                    │
│  层次4：C2编译（带Profile信息，最优）                  │
│                                                             │
│  为什么需要5个层次？                                    │
│  - 层次0：启动快                                       │
│  - 层次1~2：快速编译，简单优化（C1编译器）                │
│  - 层次3~4：深度优化，生成最优代码（C2编译器）            │
└─────────────────────────────────────────────────────────────┘
```

**为什么需要分层编译？**
- **平衡启动速度和运行速度**：先用C1快速编译，再用C2深度优化
- **渐进式优化**：先收集Profile信息，再基于Profile做深度优化

#### 3. C1编译器（Client Compiler）

**特点**：
- 编译速度快（启动快）
- 优化简单（局部优化）
- 适合**客户端应用**（如IDEA、Android Studio）

**优化技术**：
1. **方法内联**（Method Inlining）：将小方法调用替换为方法体
2. **去虚拟化**（Devirtualization）：如果类没有子类，直接调用
3. **冗余消除**（Redundancy Elimination）：消除重复计算

#### 4. C2编译器（Server Compiler）

**特点**：
- 编译速度慢（启动慢）
- 优化深度深（全局优化）
- 适合**服务端应用**（吞吐量优先）

**优化技术**：
1. **逃逸分析**（Escape Analysis）：分析对象是否逃逸出方法/线程
2. **锁消除**（Lock Elision）：如果锁对象没有逃逸，消除锁
3. **锁粗化**（Lock Coarsening）：将相邻的锁合并为一个
4. **标量替换**（Scalar Replacement）：将对象拆解为基本类型，分配在栈上
5. **栈上分配**（Stack Allocation）：对象未逃逸时，直接在栈上分配

#### 5. 热点代码探测

JIT编译器如何判断哪些代码是"热点"？

**基于计数器的热点探测**：
```
┌─────────────────────────────────────────────────────────────┐
│                  热点代码探测                             │
│                                                             │
│  方法调用计数器（Invocation Counter）                       │
│      - 每次方法调用 +1                                   │
│      - 超过阈值（如10000次）→ 触发JIT编译                   │
│                                                             │
│  回边计数器（Back Edge Counter）                           │
│      - 每次循环回边 +1                                   │
│      - 超过阈值 → 触发OSR（On-Stack Replacement）           │
│                                                             │
│  OSR（栈上替换）：                                            │
│  - 循环执行过程中，JIT编译完方法，直接替换栈帧              │
│  - 避免等到下一次方法调用才用编译后的代码                    │
└─────────────────────────────────────────────────────────────┘
```

**为什么需要两个计数器？**
- **方法调用计数器**：探测热点方法
- **回边计数器**：探测热点循环（OSR）

#### 6. 内联优化（Method Inlining）

**为什么要内联？**
- 方法调用有开销（压栈、跳转、返回）
- 内联后，编译器可以做更深入的优化（如逃逸分析）

**内联的条件**：
1. 方法体小（< 35字节，可调整）
2. 频繁调用（热点方法）
3. 可以被内联（非native、非抽象方法）

**示例**：
```java
// 内联前
public int add(int a, int b) {
    return a + b;
}

public void test() {
    int result = add(1, 2);  // 方法调用
}
```

```java
// 内联后（JIT编译后的伪代码）
public void test() {
    int result = 1 + 2;  // 直接计算结果
}
```

#### 7. 逃逸分析（Escape Analysis）

**什么是逃逸？**
- **方法逃逸**：对象被作为返回值，或作为参数传递到其他方法
- **线程逃逸**：对象被其他线程访问（如赋值给static变量）

**逃逸分析后的优化**：
1. **栈上分配**：对象未逃逸，直接在栈上分配（避免GC）
2. **标量替换**：对象未逃逸，拆解为基本类型（避免对象头开销）
3. **锁消除**：锁对象未逃逸，消除锁（减少同步开销）

**示例**：
```java
// 逃逸分析：对象未逃逸
public void test() {
    Object obj = new Object();  // obj未逃逸
    synchronized (obj) {  // 锁消除
        // ...
    }
}
```

### 源码关键路径

#### C1/C2编译器的触发

```cpp
// hotspot/src/share/vm/compiler/compilationPolicy.cpp (简化)
void CompilationPolicy::event(methodHandle method) {
    // 1. 方法调用计数器 +1
    int count = method->invocation_count() + 1;
    method->set_invocation_count(count);
    
    // 2. 判断是否超过阈值
    if (count >= CompileThreshold) {
        // 3. 触发JIT编译
        compile(method);
    }
}

void compile(methodHandle method) {
    // 4. 根据编译层次选择C1或C2
    if (TieredCompilation) {
        if (method->compilation_level() < 3) {
            compile_C1(method);  // C1编译
        } else {
            compile_C2(method);  // C2编译
        }
    } else {
        compile_C2(method);  // 只用C2
    }
}
```

#### 内联优化的实现

```cpp
// hotspot/src/share/vm/compiler/bytecodeInfo.cpp (简化)
bool BytecodeInfo::should_inline(methodHandle method) {
    // 1. 方法体大小
    if (method->code_size() > MaxInlineSize) {
        return false;  // 方法体太大，不内联
    }
    
    // 2. 是否频繁调用
    if (!method->is_hot()) {
        return false;  // 不是热点方法，不内联
    }
    
    // 3. 是否可以被内联
    if (method->is_native() || method->is_abstract()) {
        return false;  // native或抽象方法，不内联
    }
    
    return true;
}
```

## 它和相似方案的本质区别是什么？

> 对比不是列表，要解释「为什么选 A 不选 B」。

### 解释执行 vs JIT编译

| 维度 | 解释执行 | JIT编译 |
|------|----------|----------|
| **启动速度** | 快 | 慢（需要编译时间） |
| **运行速度** | 慢 | 快（本地机器码） |
| **内存占用** | 少 | 多（CodeCache） |
| **优化程度** | 无 | 深度优化 |

**为什么选混合模式？**
- **启动快**：解释器
- **运行快**：JIT编译器

### C1编译器 vs C2编译器

| 维度 | C1（Client） | C2（Server） |
|------|---------------|---------------|
| **编译速度** | 快 | 慢 |
| **优化程度** | 简单 | 深度 |
| **适用场景** | 客户端应用 | 服务端应用 |
| **吞吐量** | 低 | 高 |

**为什么需要两个编译器？**
- **分层编译**：先用C1快速编译，再用C2深度优化
- **平衡**：启动速度和运行速度

## 正确使用方式

> 不是 API 手册，要解释「为什么这样用是对的」。

### 1. 开启分层编译（默认已开启）

```bash
# JDK 8+ 默认开启分层编译
-XX:+TieredCompilation

# 如果关闭，只用C2编译器
-XX:-TieredCompilation
```

**为什么这样用？**
- **分层编译**平衡了启动速度和运行速度
- JDK 8+ 默认开启，无需手动设置

### 2. 调整JIT编译阈值

```bash
# 方法调用计数器阈值（默认10000）
-XX:CompileThreshold=10000

# 调整回边计数器阈值（默认10700）
-XX:OnStackReplacementThreshold=10700
```

**为什么调整？**
- **调小**：更早触发JIT编译（运行更快，但启动更慢）
- **调大**：更晚触发JIT编译（启动更快，但前期运行慢）

### 3. 查看JIT编译日志

```bash
# 打印JIT编译日志
-XX:+PrintCompilation

# 输出示例：
# 123  1       java.lang.String::charAt (25 bytes)
# - 123：编译ID
# - 1：编译层次（1=C1，4=C2）
# - java.lang.String::charAt：被编译的方法
# - 25 bytes：方法体大小
```

**为什么需要日志？**
- 确认热点方法是否被编译
- 排查性能问题（如某方法一直未被编译）

### 4. 调整CodeCache大小

```bash
# CodeCache大小（存储JIT编译后的机器码）
-XX:ReservedCodeCacheSize=256m  # 预留大小
-XX:InitialCodeCacheSize=32m    # 初始大小
```

**为什么调整？**
- CodeCache满了后，JIT编译会停止（不再优化）
- 如果应用很大（如IDEA），需要增大CodeCache

## 边界情况和坑

> 不是列举，要解释「坑的成因」。

### 坑1：CodeCache满了，JIT编译停止

```bash
# 现象：应用运行一段时间后，性能下降
# 原因：CodeCache满了，JIT编译停止

# 解决方案：
# 1. 增大CodeCache
-XX:ReservedCodeCacheSize=256m

# 2. 查看CodeCache使用情况
jcmd <pid> Compiler.codecache
```

**成因**：CodeCache满了后，JIT编译会停止，后续热点方法不再被编译。

### 坑2：分层编译导致启动变慢

```bash
# 现象：应用启动时间变长
# 原因：分层编译需要编译更多层次

# 解决方案：
# 1. 关闭分层编译（不推荐）
-XX:-TieredCompilation

# 2. 调整编译阈值（更早编译）
-XX:CompileThreshold=5000
```

**成因**：分层编译需要编译多个层次，启动时间变长。

### 坑3：内联失效

```java
// 内联失效的场景
public class Test {
    // 1. 方法体太大（> MaxInlineSize）
    public void bigMethod() {
        // 方法体超过35字节
    }
    
    // 2. 递归方法（无法内联）
    public int factorial(int n) {
        if (n <= 1) return 1;
        return n * factorial(n - 1);
    }
    
    // 3. 虚方法（有多态性，不一定能内联）
    public void virtualMethod(A a) {
        a.method();  // 虚方法调用
    }
}
```

**成因**：内联有条件（方法体小、非递归、非虚方法等）。

### 坑4：逃逸分析失效

```java
// 逃逸分析失效的场景
public class Test {
    // 1. 对象逃逸出方法
    public Object escapeMethod() {
        Object obj = new Object();
        return obj;  // 对象逃逸
    }
    
    // 2. 对象被其他线程访问
    public void threadEscape() {
        Object obj = new Object();
        staticObj = obj;  // 对象逃逸到其他线程
    }
}
```

**成因**：逃逸分析需要对象未逃逸，才能做栈上分配、标量替换、锁消除。

## 我的理解（可选）

> 用自己的话重新表述一遍，检验是否真正理解。

JIT编译的本质是**运行时优化**：在程序运行时，将热点代码编译为本地机器码。

**核心思想**：
- **混合模式**：解释器（启动快）+ JIT编译器（运行快）
- **分层编译**：C1（快速编译）+ C2（深度优化）
- **热点探测**：方法调用计数器 + 回边计数器

**为什么需要JIT编译？**
- **运行速度**：JIT编译后的机器码执行速度快
- **深度优化**：内联、逃逸分析、锁消除等

**最容易被忽略的点**：
- CodeCache满了后，JIT编译会停止
- 分层编译会影响启动时间
- 内联和逃逸分析都有条件限制

## 面试话术总结

1. **什么是JIT编译？**
   "JIT（Just-In-Time）编译器是JVM在运行时将热点代码编译为本地机器码的组件。HotSpot VM使用混合模式：启动时用解释器（启动快），热点代码用JIT编译器（运行快）。"

2. **分层编译是什么？**
   "分层编译（Tiered Compilation）将JIT编译分为5个层次：层次0（解释执行）、层次1~2（C1编译，快速编译）、层次3~4（C2编译，深度优化）。目的是平衡启动速度和运行速度。"

3. **热点代码如何探测？**
   "基于计数器：1）方法调用计数器（Invocation Counter）：统计方法调用次数；2）回边计数器（Back Edge Counter）：统计循环回边次数。超过阈值后触发JIT编译。"

4. **JIT有哪些优化技术？**
   "主要有：1）方法内联（Method Inlining）；2）逃逸分析（Escape Analysis）：栈上分配、标量替换、锁消除；3）锁消除（Lock Elision）；4）锁粗化（Lock Coarsening）。"

5. **CodeCache是什么？满了会怎样？**
   "CodeCache是存储JIT编译后机器码的区域。满了后，JIT编译会停止，后续热点方法不再被编译，性能下降。解决方案：增大ReservedCodeCacheSize。"
