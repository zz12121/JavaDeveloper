# 镜像安全

> 镜像是「打包好的运行环境」，如果镜像里有漏洞、有后门、用 root 跑，容器被突破就是宿主机沦陷的前奏。镜像安全不是「可选优化」，是**生产部署的底线**。

---

## 这个问题为什么存在？

不安全的镜像会导致什么？

```
场景 A：镜像里有漏洞
  基于 ubuntu:14.04 构建（已经 EOL，无安全更新）
  → 镜像扫描发现 200+ 个高危漏洞
  → 攻击者用已知 CVE 突破容器

场景 B：用 root 跑容器
  容器内是 root 用户
  → 容器被突破，攻击者拿到 root 权限
  → 配合「容器逃逸」漏洞（如 CVE-2019-5736），直接拿到宿主机 root

场景 C：敏感信息写死在镜像里
  Dockerfile 中：ENV DB_PASSWORD=my-secret-password
  → 任何能拉取镜像的人，都能看到密码
  → `docker history` 也能看到构建时的环境变量

场景 D：镜像来源不可信
  从 Docker Hub 拉了 johndoe/my-mysql:latest
  → 里面可能被植入了挖矿程序
  → 你公司服务器变成了挖矿节点
```

**镜像安全要解决的核心问题**：

1. **漏洞管理**：镜像里有没有已知 CVE？
2. **最小权限**：容器能不能不用 root 跑？
3. **供应链安全**：镜像来源可信吗？有没有被篡改？
4. **敏感信息**：密码、密钥不能进镜像

## 它是怎么解决问题的？

### 手段 1：用非 root 用户运行容器

```dockerfile
FROM eclipse-temurin:11-jre

# 创建非 root 用户
RUN addgroup --system app && adduser --system --group app

# 切换到 app 用户
USER app

# 或者指定 UID/GID（避免和用户 ID 冲突）
RUN groupadd -r -g 1000 app && useradd -r -u 1000 -g app app
USER 1000:1000
```

**效果**：容器被突破后，攻击者只有「app」用户权限，无法直接 `apt-get install` 工具。

### 手段 2：用多阶段构建减小攻击面

见「Dockerfile 最佳实践」章节。

**核心思想**：编译阶段用完整的 JDK 镜像，运行阶段只拷贝编译产物到 JRE 镜像 → 攻击面最小。

### 手段 3：镜像漏洞扫描

```
常用工具：
  - Trivy（开源，最简单）
  - Snyk（商业 + 开源，和 Git Hub 集成好）
  - Anchor（商业，功能最强）
  - Docker Scan（Docker 官方，底层用 Snyk）
```

**使用 Trivy 扫描镜像**：

```bash
# 安装 Trivy
brew install trivy   # macOS
# 或下载二进制：https://github.com/aquasecurity/trivy/releases

# 扫描镜像
trivy image my-app:1.0.0

# 输出：
# Total: 45 (HIGH: 12, CRITICAL: 3)
# 列出所有 CVE，附带来源包和修复版本
```

### 手段 4：签名和验证镜像（Docker Content Trust）

```bash
# 启用 Docker Content Trust（DCT）
export DOCKER_CONTENT_TRUST=1

# 推送镜像时会要求签名
docker push my-app:1.0.0
# 需要输入密钥密码（第一次会生成密钥对）

# 拉取镜像时会验证签名
docker pull my-app:1.0.0
# 如果镜像没有签名或被篡改，拉取失败
```

**效果**：防止「镜像被中间人篡改」或「来源不可信的镜像」。

### 手段 5：用 `.dockerignore` 防止敏感文件进镜像

```
# .dockerignore
.env
*.pem
*.key
secrets/
config/production/
```

**为什么重要**：`COPY.` 会把当前目录所有文件复制进镜像，包括 `.env`（可能包含密码）。

## 它和相似方案的本质区别是什么？

### 镜像漏洞扫描 vs 运行时安全监控

| 维度 | 镜像漏洞扫描 | 运行时安全（Falco、Aqua） |
|------|---------------|----------------------|
| 检查时机 | 构建时 / 部署前 | 运行时（容器启动后） |
| 检查内容 | 已知 CVE、不安全的基线配置 | 异常行为（如容器内向宿主机写文件） |
| 拦截能力 | 可以阻止漏洞镜像部署 | 可以实时告警或阻止异常行为 |
| 部署复杂度 | 低（CI 加一条命令） | 高（需要部署 Agent 或 eBPF 探针） |

**结论**：两者互补。镜像扫描是「入场检查」，运行时监控是「在岗监督」。

### 私服镜像仓库 vs 公有 Docker Hub

| 维度 | 私有仓库（Harbor + 漏洞扫描） | Docker Hub 公有镜像 |
|------|--------------------------|----------------|
| 镜像来源 | 可控（只放行扫描通过的镜像） | 不可控（任何人都可以推送镜像） |
| 漏洞扫描 | 可以集成（Harbor 内置 Trivy） | 部分官方镜像有扫描，用户镜像没有 |
| 访问速度 | 内网高速 | 外网，速度慢 |
| 合规 | 满足企业合规要求 | 不满足（数据不能出域） |

## 正确使用方式

### 1. CI 中集成镜像扫描

```yaml
# GitLab CI 示例
scan_image:
  stage: test
  script:
    - trivy image --exit-code 1 --severity CRITICAL,HIGH my-app:$CI_COMMIT_SHA
  # 如果有 CRITICAL 或 HIGH 漏洞，CI 失败，阻止部署
```

**关键参数**：
- `--exit-code 1`：发现漏洞时返回非零退出码（CI 失败）
- `--severity`：只检查指定严重程度的漏洞
- `--ignore-unfixed`：忽略没有修复方案的漏洞（减少误报）

### 2. 用非 root 用户运行 + 只读文件系统

```dockerfile
FROM eclipse-temurin:11-jre

RUN addgroup --system app && adduser --system --group app
USER app

# 只让需要的目录可写（其他都是只读）
# 在 docker run 时加 --read-only 参数
```

```bash
# 运行时加 --read-only，进一步加强安全
docker run -d --read-only --user 1000:1000 my-app:1.0.0
```

**效果**：即使容器被突破，攻击者也无法写入文件（无法植入后门程序）。

### 3. 不用 `latest` 标签

```dockerfile
# ❌ 危险
FROM ubuntu:latest

# ✅ 安全
FROM ubuntu:22.04
```

**原因**：`latest` 会变，今天构建没问题，明天可能引入新漏洞。

### 4. 定期重建镜像（即使代码没变）

```
问题：
  基础镜像（如 ubuntu:22.04）会更新（安全补丁）
  但你的镜像还是 3 个月前构建的，包含已知漏洞
```

**解决**：每周重建一次镜像（即使代码没变），用最新的基础镜像。

```bash
# 强制重新构建（不用缓存）
docker build --no-cache -t my-app:1.0.0 .
```

## 边界情况和坑

### 1. 非 root 用户无法绑定低位端口

```
报错：
  java.net.SocketException: Permission denied
  → 因为尝试绑定 80 端口（需要 root 权限）
```

**解决**：

```bash
# 方法 1：用高位端口（如 8080），前置 Nginx 反向代理
docker run -p 8080:8080 my-app

# 方法 2：用 root 启动，然后降权（不推荐，有短暂 root 窗口）
```

### 2. 多阶段构建时，文件属主是 root

```dockerfile
FROM maven:3.8 AS builder
RUN mvn package   # 生成的 jar 属主是 root

FROM eclipse-temurin:11-jre
COPY --from=builder /app/target/*.jar /app/app.jar
USER app
# ❌ 启动失败：app 用户无法读取 root 创建的 jar
```

**解决**：

```dockerfile
FROM maven:3.8 AS builder
RUN mvn package

FROM eclipse-temurin:11-jre
COPY --from=builder /app/target/*.jar /app/app.jar
USER app
RUN chown app:app /app/app.jar   # ✅ 改权限
# 或者在 builder 阶段就用非 root 用户编译
```

### 3. Trivy 扫描误报（False Positive）

```
场景：
  Trivy 报 `glibc` 有 CVE-2023-xxxxx
  但你的应用根本不用 `glibc`（是基础镜像自带的）
```

**解决**：用 `.trivyignore` 忽略指定的 CVE：

```
# .trivyignore
CVE-2023-xxxxx
```

**但**：要先确认「这个 CVE 真的不会影响我的应用吗？」，不要盲目忽略。

### 4. Docker Content Trust 密钥丢失

```
场景：
  启用 DCT 后，推送镜像需要密钥
  密钥丢了 → 无法推送新版本镜像
```

**解决**：备份密钥（`~/.docker/trust/`），或者用 KMS（密钥管理服务）集中管理。
