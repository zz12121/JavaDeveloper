# 常见坑与最佳实践

> MyBatis 看起来简单，但大量"能用但有问题"的隐性错误藏在日常使用中——SQL注入、映射失效、参数丢失、连接泄漏。这些问题不会在功能测试中暴露，却会在生产环境制造严重故障。

---

## 这个问题为什么存在？

MyBatis 的设计哲学是「SQL 可控」——开发者手写 SQL，框架负责映射和执行。这个哲学的代价是**正确性高度依赖开发者的细节意识**。

和其他 ORM 框架（比如 JPA 完全自动生成 SQL）不同，MyBatis 把 SQL 的控制权交给了你，但没有帮你守住所有安全边界：

- `#{}` 和 `${}` 只差一个字符，一个安全一个危险，但 MyBatis **不会主动报错**，`${}` 在 SQL 注入被触发前看起来和 `#{}` 一样正常
- ResultMap 中 `<id/>` 和 `<result/>` 功能不同，但写错只在**嵌套集合**场景才暴露，单表查询完全没症状
- `@Param` 在多参数时必须写，但**单个参数时不写也能用**，这种不一致的行为让开发者产生"不用 @Param 也行"的误解
- Spring 管理了 SqlSession 的生命周期，但**非 Spring 环境下必须手动关闭**，这个上下文切换容易被遗忘

这些问题的根源不是"开发者粗心"，而是 MyBatis 的 API 设计在灵活性和安全性之间选择了灵活——它假设你清楚每一步在做什么。

---

## 它是怎么解决问题的？

### `#{}` 和 `${}`：预编译 vs 字符串替换

```java
// #{}
// → 调用 PreparedStatement.setParameter()
// → 生成预编译 SQL: SELECT * FROM user WHERE name = ?
// → 参数值被 JDBC 驱动安全转义，SQL 注入不可能发生

// ${}
// → 直接字符串拼接
// → 生成 SQL: SELECT * FROM user WHERE name = 'xxx' OR '1'='1'
// → 用户输入原样进入 SQL，注入攻击直接生效
```

MyBatis 通过 `#{}` 调用 JDBC 的预编译参数绑定，参数值不会参与 SQL 语法解析，从根源上杜绝了注入。`${}` 则是纯字符串替换，和手动拼接 SQL 没有区别。

`${}` 的合法场景仅限于**结构性 SQL 片段**——表名、列名、ORDER BY 方向——这些位置不能用预编译参数占位：

```xml
<!-- 合法：表名是结构性元素，不能用 #{} -->
<select id="selectByTable" resultType="User">
    SELECT * FROM ${tableName} WHERE id = #{id}
</select>

<!-- 合法：排序方向是关键字，不能用 #{} -->
<select id="selectByCondition" resultType="User">
    SELECT * FROM user ORDER BY ${orderByColumn} ${orderDir}
</select>
```

用 `${}` 时必须在 Java 侧做白名单校验：

```java
private static final Set<String> ALLOWED_COLUMNS = Set.of("name", "age", "create_time");

public List<User> selectByCondition(String column, String direction) {
    if (!ALLOWED_COLUMNS.contains(column)) {
        throw new IllegalArgumentException("非法排序字段: " + column);
    }
    if (!"ASC".equalsIgnoreCase(direction) && !"DESC".equalsIgnoreCase(direction)) {
        throw new IllegalArgumentException("排序方向只能是 ASC 或 DESC");
    }
    return userMapper.selectByCondition(column, direction);
}
```

### LIKE 查询的正确姿势

`#{}` 在 LIKE 中有一个微妙问题：

```xml
<!-- ❌ 不行：#{} 在字符串拼接内部，JDBC 会把整个 '%#{keyword}%' 当作普通字符串 -->
WHERE name LIKE '%#{keyword}%'
<!-- 生成的 SQL: WHERE name LIKE '%?%' — ? 被当成字面量，不是参数占位符 -->

<!-- ❌ 危险：${} 能用但有注入风险 -->
WHERE name LIKE '%${keyword}%'

<!-- ✅ 方案一：CONCAT 函数（推荐） -->
WHERE name LIKE CONCAT('%', #{keyword}, '%')

<!-- ✅ 方案二：Java 侧拼接 -->
<!-- Java: keyword = "%" + input + "%" -->
WHERE name LIKE #{keyword}
```

`CONCAT('%', #{keyword}, '%')` 为什么安全？因为 `#{keyword}` 仍然走预编译参数绑定，`${}` 那个版本才是直接字符串拼接。`CONCAT` 函数只是改变了 SQL 语法结构，参数绑定机制没有变。

### ResultMap 的 `<id/>`：嵌套去重的关键

```xml
<resultMap id="userMap" type="User">
    <id column="id" property="id"/>       <!-- 标记主键 -->
    <result column="name" property="name"/>
    <collection property="orders" ofType="Order">
        <id column="order_id" property="id"/>  <!-- 必须也标记主键 -->
        <result column="order_no" property="orderNo"/>
    </collection>
</resultMap>
```

MyBatis 用 `<id/>` 标记的列来判断**对象是否相同**。在 JOIN 查询返回多行结果时，如果一行结果的 `id` 值和上一行相同，MyBatis 就认为这是同一个 User 对象，只会合并 `collection` 中的 Order。如果漏写 `<id/>`，MyBatis 无法判断哪些行属于同一个 User，结果就是**重复的 User 对象**，每个 User 只带一个 Order。

### `@Param` 的必要性

```java
// 单个参数：不加 @Param 也能用（MyBatis 用参数名 #{id} 直接匹配）
User selectById(Long id);

// 多个参数：不加 @Param → 编译后参数名变成 arg0, arg1（或 param1, param2）
List<User> selectByAgeAndName(Integer age, String name);
// XML 中 #{age} → 找不到参数名 → org.apache.ibatis.binding.BindingException

// 修复
List<User> selectByAgeAndName(@Param("age") Integer age, @Param("name") String name);
```

Java 编译时默认不保留方法参数名（`javac` 的 `-parameters` 选项默认关闭），所以 MyBatis 在运行时拿到的是 `arg0, arg1` 这样的名字。`@Param` 显式告诉 MyBatis 参数应该叫什么。

### foreach 的括号

```xml
<!-- ❌ 生成：1, 2, 3 — 缺少括号，IN 语法错误 -->
<foreach collection="ids" item="id" separator=",">
    #{id}
</foreach>

<!-- ✅ 生成：(1, 2, 3) -->
<foreach collection="ids" item="id" open="(" separator="," close=")">
    #{id}
</foreach>
```

`open` 和 `close` 分别在遍历结果前后追加内容，`separator` 是元素之间的分隔符。`IN` 查询必须用括号包裹，这是 SQL 语法要求。

## 深入原理

### `#{}` 的底层：PreparedStatement 参数绑定

```
#{name} 在 MyBatis 内部的处理流程：
1. 解析 XML/注解中的 SQL，识别 #{} 占位符
2. 生成 MappedStatement，将 #{} 替换为 JDBC 的 ? 占位符
3. 执行时调用 PreparedStatement.setString(index, value)
4. JDBC 驱动负责转义参数值

关键：参数值在第 4 步才绑定，此时 SQL 语法已经解析完毕
→ 参数值不可能改变 SQL 结构 → SQL 注入不可能发生
```

### `${}` 的底层：OGNL 表达式直接求值

```
${name} 在 MyBatis 内部的处理流程：
1. 解析 XML/注解中的 SQL，识别 ${} 占位符
2. 执行时用 OGNL 表达式引擎对 ${name} 求值
3. 将求值结果直接拼接进 SQL 字符串
4. 将完整的 SQL 字符串传给 JDBC 执行

关键：SQL 在第 4 步才确定结构
→ 用户输入在第 2 步就参与了 SQL 拼接 → 注入可能发生
```

这就是为什么 `${}` 只适合表名、列名等结构性元素——它们的值必须是 SQL 关键字或标识符，不可能通过预编译参数传递。

### `selectOne` 的 TooManyResultsException

```java
// MyBatis 源码（DefaultSqlSession）：
@Override
public <T> T selectOne(String statement, Object parameter) {
    List<T> list = this.selectList(statement, parameter);
    if (list.size() == 1) {
        return list.get(0);
    } else if (list.size() > 1) {
        throw new TooManyResultsException(
            "Expected one result (or null) but found: " + list.size());
    }
    return null;
}
```

`selectOne` 内部就是调用 `selectList`，然后检查结果数量。如果 SQL 返回了多行，直接抛异常——MyBatis 宁可报错也不帮你"取第一条"，因为"取第一条"的行为往往是 bug 的遮盖（应该只返回一条数据却返回了多条，说明查询条件有误）。

## 正确使用方式

### SQL 编写规则

```xml
<!-- 始终用 #{} 传值 -->
SELECT * FROM user WHERE name = #{name}

<!-- ${} 仅用于结构性元素，且必须白名单校验 -->
SELECT * FROM ${tableName} WHERE status = #{status}
ORDER BY ${orderColumn} ${orderDir}

<!-- LIKE 查询用 CONCAT -->
WHERE name LIKE CONCAT('%', #{keyword}, '%')

<!-- foreach 注意 open/close -->
WHERE id IN
<foreach collection="ids" item="id" open="(" separator="," close=")">
    #{id}
</foreach>

<!-- ResultMap 中主键字段必须用 <id/> -->
<resultMap id="userWithOrders" type="User">
    <id column="id" property="id"/>
    <collection property="orders" ofType="Order">
        <id column="order_id" property="id"/>
    </collection>
</resultMap>
```

### 多参数方法必须加 @Param

```java
// 两个及以上参数 → 每个都加 @Param
List<User> selectByCondition(@Param("name") String name,
                             @Param("age") Integer age,
                             @Param("status") Integer status);
```

### 批量插入的性能优化

```xml
<!-- JDBC URL 加 rewriteBatchedStatements=true，MySQL 批量性能提升 5~10 倍 -->
<!-- jdbc:mysql://localhost:3306/mydb?rewriteBatchedStatements=true -->

<!-- 批量插入用 foreach 拼接多行 VALUES -->
<insert id="batchInsert">
    INSERT INTO user (name, age, email) VALUES
    <foreach collection="list" item="user" separator=",">
        (#{user.name}, #{user.age}, #{user.email})
    </foreach>
</insert>
```

没有 `rewriteBatchedStatements=true` 时，MySQL 的 `PreparedStatement.addBatch()` 实际上是一条一条执行的（每条都是一次网络往返）。开启后 MySQL 驱动会将多条 INSERT 重写为一条 `INSERT INTO ... VALUES (...), (...), (...)` 的多行插入，网络往返从 N 次降为 1 次。

### 事务交给 Spring 管理

```java
// ✅ 正确：用 @Transactional，不要手动操作 SqlSession
@Transactional
public void createUserWithRole(User user, Long roleId) {
    userMapper.insert(user);
    userRoleMapper.insert(new UserRole(user.getId(), roleId));
    // 异常时 Spring 自动回滚，不需要 sqlSession.rollback()
}

// ❌ 错误：手动 commit/rollback（Spring 环境下）
sqlSession.insert("user.insert", user);
sqlSession.commit();  // 不要这样做
```

Spring 整合 MyBatis 后，SqlSession 的生命周期由 `SqlSessionTemplate` 管理（它实现了 `SqlSession` 接口，但实际每次调用都从 ThreadLocal 获取当前事务的 SqlSession）。手动调用 `commit()`/`rollback()` 会干扰 Spring 的事务管理。

## 边界情况和坑

### Oracle 的 null 值必须指定 jdbcType

```xml
<!-- MySQL：可以不写 jdbcType，驱动能推断 -->
<insert id="insert">
    INSERT INTO user (name, age) VALUES (#{name}, #{age})
</insert>

<!-- Oracle：参数为 null 时必须指定 jdbcType，否则报 "Invalid column type" -->
<insert id="insert">
    INSERT INTO user (name, age) VALUES (#{name, jdbcType=VARCHAR}, #{age, jdbcType=NUMERIC})
</insert>
```

原因：Oracle 的 JDBC 驱动在参数为 null 时无法推断 SQL 类型，必须显式告诉驱动这是一个 VARCHAR 还是 NUMERIC。MySQL 的驱动更智能，能自动推断。团队规范建议统一——要么全写（兼容 Oracle），要么全不写（仅 MySQL）。

### N+1 查询

```java
// 典型的 N+1 场景
List<User> users = userMapper.selectList(null);  // 1 次查询
for (User user : users) {
    List<Order> orders = orderMapper.selectByUserId(user.getId());  // N 次查询
}
// 总计 1 + N 次查询，100 个用户就是 101 次

// 修复方案一：JOIN 一次查出（推荐）
// 用 resultMap 的 <collection> 做嵌套映射，一条 SQL 搞定

// 修复方案二：IN 查询 + 内存组装
List<User> users = userMapper.selectList(null);  // 1 次
List<Long> ids = users.stream().map(User::getId).toList();
List<Order> orders = orderMapper.selectByUserIds(ids);  // 1 次，WHERE id IN (...)
// 在内存中按 userId 分组组装  // 总计 2 次查询
```

N+1 问题不会在开发环境（少量数据）暴露，只在生产环境（大量数据）才变成性能瓶颈——这种"开发时正常、上线后变慢"的问题最危险。

### `@MapperScan` 冲突

```java
// ❌ 两个 @MapperScan 有重叠 → 同一个 Mapper 被注册两次 → 启动报错
@MapperScan("com.example.mapper")
@MapperScan("com.example.mapper.user")  // 子包重叠

// ✅ 用一个 @MapperScan 指定多个包
@MapperScan(basePackages = {"com.example.mapper", "com.other.mapper"})
```

Mapper 是 MyBatis 通过动态代理创建的对象（`MapperProxy`），注册到 Spring 容器时用接口全限定名作为 Bean Name。同一个接口被注册两次，Spring 就会抛 `BeanDefinitionOverrideException`。

### SqlSession 连接泄漏（非 Spring 环境）

```java
// ❌ 忘了 close() → 连接永远不归还连接池 → 连接池耗尽
SqlSession session = sqlSessionFactory.openSession();
User user = session.selectOne("user.selectById", 1);
// 没有 close()

// ✅ try-with-resources 自动关闭
try (SqlSession session = sqlSessionFactory.openSession()) {
    User user = session.selectOne("user.selectById", 1);
}  // 自动调用 close()，归还连接到连接池
```

在 Spring 环境下这个问题不会发生——`SqlSessionTemplate` 始终在线程结束时自动关闭。但在 MyBatis 的纯原生使用方式（比如集成测试、工具脚本）中，忘记关闭 SqlSession 是最常见的连接泄漏原因。

### `autoMapping` 和驼峰转换的关系

```xml
<!-- resultMap 的 autoMapping="true" 依赖驼峰转换是否开启 -->
<resultMap id="userMap" type="User" autoMapping="true">
    <result column="dept_name" property="deptName"/>
    <!-- 期望：未显式映射的字段（如 create_time → createTime）也能自动映射 -->
</resultMap>

<!-- 前提：必须开启驼峰转换 -->
<setting name="mapUnderscoreToCamelCase" value="true"/>
<!-- 否则 create_time 无法映射到 createTime → 字段为 null -->
```

`autoMapping="true"` 只是告诉 MyBatis "没显式映射的字段也尝试映射"，但映射规则依赖列名和属性名能匹配上。如果数据库是下划线命名（`create_time`）而 Java 是驼峰命名（`createTime`），不开驼峰转换就匹配不上。
