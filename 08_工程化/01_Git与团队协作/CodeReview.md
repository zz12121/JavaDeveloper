# CodeReview

> Code Review 不是「找茬」，是**知识传播 + 质量门禁 + 团队对齐」的三合一机制。没有 CR 的团队，代码质量完全依赖个人能力；有 CR 的团队，代码质量是集体智慧的产物。

---

## 这个问题为什么存在？

没有 Code Review 的团队是什么样的？

```
场景 A：核心开发者离职
→ 他写的代码没人看得懂
→ 修一个 bug 引入两个新 bug
→ 因为没人知道那段代码的意图是什么

场景 B：初级开发者写了烂代码
→ 没人看就合进去了
→ 生产出了事故，才发现那段代码有问题
→ 「我以为没问题啊，又没人说」

场景 C：团队代码风格百花齐放
→ 有人用驼峰，有人用下划线
→ 有人 2 空格缩进，有人 4 空格
→ 同一个功能，5 种实现方式

场景 D：没人知道「全貌」
→ 每个人只看自己的模块
→ 架构腐烂了没人发现
→ 直到重构成本已经高到无法承受
```

**Code Review 解决的四个根本问题**：

1. **知识孤岛**：把「只有作者懂」变成「团队共懂」
2. **质量门禁**：在代码合并前拦截明显问题
3. **规范对齐**：让团队代码风格和架构决策保持一致
4. **集体拥有**：任何代码都有至少两个人知道来龙去脉

## 它是怎么解决问题的？

### CR 的基本流程

```
开发者  ──→  创建 PR/MR（Pull Request / Merge Request）
          │
          ↓
     CI 自动检查（编译、单测、Lint、覆盖率）
          │
          ↓（CI 通过）
    Reviewer 阅读代码，提出 Comment
          │
          ↓（修改后）
    Reviewer 再次审查，Approve
          │
          ↓
    合并到目标分支
```

### CR 的核心检查点

```
Code Review Checklist（按优先级排序）：

【必须拦截】
□ 功能正确性：有没有明显的逻辑错误？边界条件处理了吗？
□ 安全漏洞：SQL 注入、XSS、敏感信息硬编码？
□ 性能问题：N+1 查询、大对象持有、死锁风险？
□ 测试覆盖：新功能有单测吗？边界场景覆盖了吗？

【强烈建议】
□ 代码可读性：变量名有意义吗？函数职责单一吗？
□ 架构一致性：新代码符合项目架构规范吗？
□ 错误处理：异常有被正确处理吗？还是悄悄吞掉了？
□ 日志质量：出问题时能从日志定位到根因吗？

【可选（团队规范）】
□ 代码风格：缩进、命名、括号位置（用自动化工具检查，不要人工看）
□ 注释质量：复杂逻辑有解释吗？
```

### 工具支持

```
GitHub / GitLab / Gitea 的 PR/MR 功能：
  - 行级评论（针对具体代码行提意见）
  - Review 状态（Approve / Request Changes / Comment）
  - CI 集成（通过 YAML 配置自动跑测试）
  - COPILOT / CodeRabbit AI Review（AI 辅助审查）

IDE 插件：
  - IntelliJ IDEA：GitLens（查看代码历史、blame）
  - VS Code：GitLens 插件
```

## 它和相似方案的本质区别是什么？

### Code Review vs 结对编程（Pair Programming）

| 维度 | Code Review | 结对编程 |
|------|-------------|-----------|
| 时机 | 代码完成后 | 代码编写中 |
| 成本 | 低（不占用两人同时时间） | 高（两人同时投入） |
| 实时性 | 有延迟（可能需要多次来回） | 实时（问题立即发现） |
| 知识传播 | 异步，Reviewer 可以深入学习 | 实时，但可能流于「一个人写一个人看」 |
| 适合场景 | 大多数团队 | 复杂逻辑、新人 onboarding |

**结论**：不是二选一。理想情况是「结对编程做复杂模块 + CR 做日常审查」。

### Code Review vs 静态分析工具（SonarQube、Checkstyle）

| 维度 | Code Review | 静态分析工具 |
|------|-------------|--------------|
| 检查内容 | 架构、业务逻辑、可读性 | 代码规范、常见 bug 模式、安全漏洞 |
| 误报率 | 低（人工判断） | 高（规则固定，不理解业务上下文） |
| 覆盖率 | 依赖 Reviewer 认真程度 | 100% 覆盖（每个文件都会扫描） |
| 成本 | 人工时间 | 自动化，几乎零边际成本 |

**结论**：工具做「机械检查」，人做「智能判断」。**两者必须结合**，不能互相替代。

```
正确分工：
  静态分析工具 → 代码规范、常见 bug 模式、安全漏洞
  Code Review   → 架构一致性、业务逻辑正确性、可读性
```

## 正确使用方式

### 1. 作为提交者：怎么让你的 PR 更容易被 Review

```
✅ 好的 PR：
  - 标题清晰：「fix: 修复用户登录时 null 检查缺失导致的 NPE」
  - 变更范围小（< 300 行，聚焦一个功能）
  - 描述包含：解决了什么问题、改了哪些文件、测试怎么跑
  - 自测通过，CI 全绿
  - 复杂逻辑有注释解释「为什么这样写」

❌ 烂的 PR：
  - 标题：「改了点东西」
  - 一个 PR 改了 3000 行，混着格式化、重构、新功能
  - 没有描述，Reviewer 要自己猜意图
  - CI 挂了一堆测试，提交者说「不管，你先帮我看看逻辑」
  - 重构和功能混在一起，Reviewer 不知道该关注什么
```

**PR 描述模板**：
```markdown
## 背景
解决了什么问题？为什么需要这个改动？

## 改动说明
- 改了哪些文件
- 关键设计决策（为什么选 A 不选 B）

## 测试
- 怎么测试这个改动
- 覆盖率变化（如果有）

## 风险提示
有没有 breaking change？需不需要更新文档？
```

### 2. 作为 Reviewer：怎么提有建设性的意见

```
✅ 好的 Review 评论：
  「这个循环可以提取成独立方法，提高可读性。另外建议加一个边界检查，防止 index 越界。」

  「这里的异常处理是不是太宽泛了？catch (Exception e) 会吃掉 RunTimeException，建议改成具体的异常类型。」

❌ 烂的 Review 评论：
  「这代码写的什么鬼」（没有建设性，只是指责）
  「我觉得这样不好」（没说为什么不好，也没说怎么改）
  「。」（无意义的评论）
```

**评论的分类**：
```
【必须修改】（用 TODO 或 fix 标记）
  「这里有个 NPE 风险，user 可能为 null，需要加检查。」

【建议修改】（用 suggestion 标记）
  「建议把这个方法提取成独立函数，可读性会更好。不强制，看你意见。」

【提问】（用 question 标记）
  「这里的算法复杂度是 O(n²)，有没有可能优化到 O(n log n)？」

【称赞】（用 praise 标记）
  「这个错误处理写得很完善，学习了！」
```

### 3. 用「Comment 类型」提高沟通效率

```
GitHub/GitLab 支持的单据类型：
  - Comment：一般性评论
  - Approve：通过，可以合并
  - Request Changes：有必须修改的问题，修改后再 Review
  - Pending Review：还在看，暂时不表态
```

**Reviewer 的责任边界**：
- Reviewer 不是「质量担保人」——最终质量责任在**提交者**
- Reviewer 的建议可以被拒绝，但要**说明理由**
- 如果 Reviewer 和提交者无法达成一致，找 Tech Lead 仲裁

### 4. 设置团队 CR 规范

```markdown
## 团队 Code Review 规范

1. **响应时间**：PR 创建后 24 小时内完成 Review（紧急修复除外）
2. **Reviewer 分配**：至少 1 人 Approve 才能合并；关键模块需要 2 人
3. **CI 门禁**：CI 不通过，不能合并（GitHub/GitLab 可以设置 Branch Protection）
4. **PR 大小限制**：单个 PR 不超过 500 行（超过就拆分）
5. **禁止自己 Approve**：自己写的代码不能自己 Approve（需要至少一人审查）
```

## 边界情况和坑

### 1. 「LGTM」文化（Look Good To Me，走形式）

```
问题：
Reviewer 不看代码，直接点 Approve
→ CR 变成了「橡皮图章」，没有实际价值
→ 出事故后互相推诿：「我看过了啊，是 XX 写的」

原因：
- Reviewer 太忙，没时间认真看
- PR 太大，不想仔细看
- 团队文化：「大家都是同事，别太严格」
```

**解决方案**：
- PR 拆小（< 300 行），降低 Review 成本
- 把 CR 计入绩效考核（质量贡献，不只是写代码）
- 用 AI 工具（GitHub COPILOT、CodeRabbit）做初轮审查，人专注于架构和逻辑

### 2. Reviewer 和提交者「吵起来」

```
典型场景：
Reviewer：「这个地方你应该用策略模式」
提交者：「不需要，用 if-else 更简单」
Reviewer：「你这代码质量太差了」
提交者：「你行你上啊」
```

**正确做法**：
- 技术决策**看文档、看数据**，不要看「我觉得」
- 如果争议无法解决，找 Tech Lead 仲裁
- 记住：**CR 是关于代码的讨论，不是对人的评价**

### 3. 紧急修复时省略 CR

```
场景：
生产 P0 事故，需要立即修复
→ 提交者：「来不及 CR 了，我先合并，事后补 CR」

后果：
→ 「事后补 CR」永远不会发生
→ 下次又变成「来不及」
```

**正确做法**：
- 真正的紧急修复：设置「Emergency Merge」流程（合并后 24 小时内必须补 CR）
- 用 `CODEOWNERS` 文件指定关键模块的负责人，至少一个人快速 Approve
- 如果真的「每分钟损失 10 万」，可以先合并，但**必须事后复盘**

### 4. 自动化工具替代了人的判断

```
问题：
团队配置了 SonarQube，所有代码规范都交给工具检查
→ 人认为「工具都检查过了，我不用看代码规范了」
→ 但工具只能检查「规则内的东西」，架构问题、业务逻辑错误工具发现不了
```

**正确分工**：
```
工具（SonarQube、Checkstyle、SpotBugs）：
  - 代码规范（命名、缩进、复杂度）
  - 常见 bug 模式（空指针、资源未关闭）
  - 安全漏洞（SQL 注入、硬编码密码）

人（Code Review）：
  - 架构一致性（新代码符合架构规范吗？）
  - 业务逻辑（这个改动真的解决了问题吗？有没有副作用？）
  - 可读性（变量名有意义吗？函数职责单一吗？）
```
