# 阻塞队列

> 阻塞队列（BlockingQueue）是生产者-消费者模式的最佳实践。线程池的 workQueue 就是阻塞队列——任务满了生产者阻塞，任务空了消费者阻塞。理解阻塞队列是理解线程池参数含义和使用场景的前提。

---

## 这个问题为什么存在？

### 为什么需要阻塞队列？

在多线程协作中，生产者和消费者之间的速度通常不一致：

```
问题1：生产者快于消费者
→ 缓冲区无限增长 → OOM

问题2：消费者快于生产者
→ 消费者空转浪费 CPU
→ 或频繁轮询检查有没有新数据

问题3：手动用 wait/notify 实现
→ 代码复杂，容易出错（notify vs notifyAll、虚假唤醒、锁的释放）
```

**BlockingQueue 用一个数据结构同时解决了这三个问题**：
- 队列满了 → put() 阻塞生产者
- 队列空了 → take() 阻塞消费者
- 所有同步逻辑内置，调用方不需要关心 wait/notify

---

## 它是怎么解决问题的？

### BlockingQueue 接口

```java
public interface BlockingQueue<E> extends Queue<E> {
    // 插入操作（3 种策略）
    boolean add(E e);         // 队列满 → IllegalStateException
    boolean offer(E e);       // 队列满 → 返回 false
    void put(E e)             // 队列满 → 阻塞等待
        throws InterruptedException;
    boolean offer(E e, long timeout, TimeUnit unit)  // 队列满 → 超时等待
        throws InterruptedException;

    // 移除操作（3 种策略）
    E remove();               // 队列空 → NoSuchElementException
    E poll();                 // 队列空 → 返回 null
    E take()                  // 队列空 → 阻塞等待
        throws InterruptedException;
    E poll(long timeout, TimeUnit unit)  // 队列空 → 超时等待
        throws InterruptedException;

    // 检查操作
    E element();              // 队列空 → NoSuchElementException
    E peek();                 // 队列空 → 返回 null

    // 批量操作
    int drainTo(Collection<? super E> c);       // 一次性取出所有元素
    int drainTo(Collection<? super E> c, int maxElements);
    int remainingCapacity();                    // 剩余容量
}
```

```
四组操作的核心区别：满/空时的行为

| 操作 | 抛异常 | 返回特殊值 | 阻塞 | 超时 |
|------|--------|-----------|------|------|
| 插入 | add() | offer() | put() | offer(e, time) |
| 移除 | remove() | poll() | take() | poll(time) |
| 检查 | element() | peek() | — | — |

面试中经常考 add vs offer vs put 的区别——记住这个表格。
```

### 7 种实现类

```
BlockingQueue
├── ArrayBlockingQueue      — 数组结构，有界，公平可选
├── LinkedBlockingQueue     — 链表结构，有界/无界
├── SynchronousQueue        — 不存储元素，直接传递
├── PriorityBlockingQueue   — 优先级排序，无界
├── DelayQueue              — 延迟取出元素
├── LinkedTransferQueue     — 链表结构，直接传递 + 队列
└── LinkedBlockingDeque     — 双端队列
```

### ArrayBlockingQueue

```java
// 有界数组队列，创建时必须指定容量
BlockingQueue<String> queue = new ArrayBlockingQueue<>(100);

// 可选公平模式
BlockingQueue<String> fairQueue = new ArrayBlockingQueue<>(100, true);
```

```
底层原理：
- 内部用 Object[] 数组存储元素
- 用 ReentrantLock + 两个 Condition（notEmpty / notFull）实现同步
- 公平模式：通过 ReentrantLock(true) 实现

特点：
1. 必须指定容量，创建后不能扩容
2. 内存连续，CPU 缓存友好
3. 入队和出队共用一把锁 → 吞吐量不如 LinkedBlockingQueue
4. 公平模式下严格 FIFO（适合需要严格顺序的场景）
```

### LinkedBlockingQueue

```java
// 有界（推荐）
BlockingQueue<String> queue = new LinkedBlockingQueue<>(100);

// 无界（不推荐——Integer.MAX_VALUE 容量，可能 OOM）
BlockingQueue<String> unbounded = new LinkedBlockingQueue<>();
// 注意：Executors.newFixedThreadPool() 用的就是无界 LinkedBlockingQueue
```

```
底层原理：
- 用链表存储（Node 内部类）
- 入队和出队用两把独立的锁（putLock / takeLock）→ 并发性能好
- 但 size 变量用 AtomicInteger，每次操作需要 CAS 更新

为什么 Executors.newFixedThreadPool 的队列是无界的？
这是一个设计缺陷——线程数固定 + 队列无界 →
任务堆积在队列中不会触发拒绝策略 → 内存持续增长 → OOM
这也是为什么推荐手动创建 ThreadPoolExecutor 的原因。
```

### SynchronousQueue

```java
// 不存储元素的队列——每个 put 必须等待一个 take
BlockingQueue<String> queue = new SynchronousQueue<>();

// 线程A
queue.put("hello");  // 阻塞，直到有线程调用 take()

// 线程B
String msg = queue.take();  // 拿到 "hello"，put() 同时解除阻塞
```

```
特点：
1. 容量为 0——不存储任何元素
2. 每个插入操作必须等待对应的移除操作
3. 公平模式用队列实现 FIFO，非公平模式用栈实现 LIFO
4. 吞吐量通常高于 ArrayBlockingQueue（因为没有元素存储的开销）

适用场景：生产者和消费者需要"手递手"交接
典型使用：Executors.newCachedThreadPool() 的任务队列
→ 提交的任务直接交给线程处理，不需要在队列中缓冲
```

### PriorityBlockingQueue

```java
// 优先级队列——元素按自然顺序或 Comparator 排序
PriorityBlockingQueue<Task> queue = new PriorityBlockingQueue<>(11,
    Comparator.comparingInt(Task::getPriority).reversed()
);
```

```
特点：
1. 无界队列（不推荐用于有界场景）
2. 元素必须实现 Comparable 或提供 Comparator
3. 内部用堆（数组实现的二叉小顶堆/大顶堆）
4. take() 返回优先级最高/最低的元素
5. 只用一把锁（和 ArrayBlockingQueue 类似）

注意：PriorityBlockingQueue 是无界的，不断 put 会 OOM。
如果需要有界优先级，需要自己包装或在 put 前检查 size。
```

### DelayQueue

```java
// 延迟队列——元素到期才能取出
class DelayedTask implements Delayed {
    private final long startTime;
    private final String name;

    DelayedTask(long delay, String name) {
        this.startTime = System.currentTimeMillis() + delay;
        this.name = name;
    }

    @Override
    public long getDelay(TimeUnit unit) {
        return unit.convert(startTime - System.currentTimeMillis(), TimeUnit.MILLISECONDS);
    }

    @Override
    public int compareTo(Delayed o) {
        return Long.compare(this.getDelay(TimeUnit.MILLISECONDS), o.getDelay(TimeUnit.MILLISECONDS));
    }
}

DelayQueue<DelayedTask> queue = new DelayQueue<>();
queue.put(new DelayedTask(5000, "5秒后执行"));   // 5 秒后可取出
queue.put(new DelayedTask(1000, "1秒后执行"));    // 1 秒后可取出

// take() 会阻塞，直到最短的延迟到期
DelayedTask task = queue.take();  // 1 秒后返回"1秒后执行"
```

```
底层：PriorityBlockingQueue + Delayed 接口
- 内部用 PriorityQueue 排序（按剩余延迟时间）
- getDelay() <= 0 时元素才可被取出
- take() 通过 Condition.awaitNanos() 实现定时阻塞

应用场景：
1. 定时任务调度
2. 缓存过期清理
3. 延迟消息发送
4. 订单超时自动取消
```

### 阻塞队列与线程池的关系

```
线程池的 workQueue 就是 BlockingQueue。

四种线程池的队列选择：

newFixedThreadPool    → LinkedBlockingQueue（无界）→ 可能 OOM
newCachedThreadPool   → SynchronousQueue（容量0）→ 不缓冲
newSingleThreadExecutor → LinkedBlockingQueue（无界）→ 可能 OOM
newScheduledThreadPool → DelayedWorkQueue（延迟队列）

手动创建线程池时的队列选择原则：
- 有界 ArrayBlockingQueue → 防止 OOM，但任务可能被拒绝
- 有界 LinkedBlockingQueue → 同上，但吞吐量更高
- SynchronousQueue → 需要立即处理，不需要缓冲
- PriorityBlockingQueue → 任务有优先级
```

## 和相似方案的区别

### 7 种阻塞队列对比

| 实现 | 底层结构 | 有界 | 排序 | 锁数量 | 适用场景 |
|------|---------|------|------|--------|---------|
| ArrayBlockingQueue | 数组 | ✓（固定） | FIFO | 1 把 | 固定容量，需要严格 FIFO |
| LinkedBlockingQueue | 链表 | 可选 | FIFO | 2 把 | 吞吐量优先 |
| SynchronousQueue | 无存储 | ✓（0） | — | — | 直接传递，不需要缓冲 |
| PriorityBlockingQueue | 堆 | ✗（无界） | 优先级 | 1 把 | 任务有优先级 |
| DelayQueue | 堆 | ✗（无界） | 延迟时间 | — | 延迟任务 |
| LinkedTransferQueue | 链表 | ✗（无界） | FIFO | — | 高并发直接传递 |
| LinkedBlockingDeque | 双向链表 | 可选 | FIFO | 2 把 | 工作窃取模式 |

### BlockingQueue vs ConcurrentLinkedQueue

| 维度 | BlockingQueue | ConcurrentLinkedQueue |
|------|--------------|----------------------|
| 阻塞操作 | put/take（满/空时阻塞） | 无（只返回 null） |
| 实现 | 用锁（ReentrantLock） | 无锁（CAS） |
| 容量 | 有界/无界 | 无界 |
| 适用场景 | 生产者-消费者 | 高并发非阻塞场景 |
| 线程池 | workQueue | 不用于线程池 |

## 正确使用方式

### 1. 生产者-消费者标准模板

```java
BlockingQueue<Task> queue = new ArrayBlockingQueue<>(100);

// 生产者
ExecutorService producers = Executors.newFixedThreadPool(4);
producers.submit(() -> {
    try {
        while (running) {
            Task task = produceTask();
            queue.put(task);  // 队列满时自动阻塞
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    }
});

// 消费者
ExecutorService consumers = Executors.newFixedThreadPool(4);
consumers.submit(() -> {
    try {
        while (running) {
            Task task = queue.take();  // 队列空时自动阻塞
            processTask(task);
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    }
});
```

### 2. 线程池队列容量选择

```java
// ✓ 有界队列——防止 OOM
new ThreadPoolExecutor(
    4, 8, 60, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(1000),  // 有界！
    new ThreadPoolExecutor.CallerRunsPolicy()
);

// ✗ 无界队列——可能 OOM
new ThreadPoolExecutor(
    4, 8, 60, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>()  // Integer.MAX_VALUE！
);
```

### 3. drainTo 批量取元素

```java
// 一次性取出多个元素，减少锁的获取次数
List<Task> batch = new ArrayList<>(100);
queue.drainTo(batch, 100);  // 最多取 100 个

// 批量处理比逐个 take 效率高
```

### 4. 优雅关闭消费者

```java
// 方式1：毒丸对象（Poison Pill）
Task POISON = new Task();  // 特殊的终止标记
queue.put(POISON);  // 生产者结束时放入

// 消费者
Task task = queue.take();
if (task == POISON) return;  // 收到毒丸，退出

// 方式2：interrupt
consumerThread.interrupt();  // 让 take() 抛 InterruptedException
```

## 边界情况和坑

### 1. LinkedBlockingQueue 无界的 OOM 陷阱

```java
// ✗ Executors.newFixedThreadPool 的队列是无界的
ExecutorService pool = Executors.newFixedThreadPool(4);
// 所有任务堆积在 LinkedBlockingQueue 中
// 线程数只有 4，队列无限增长 → OOM

// ✓ 手动创建，指定有界队列
new ThreadPoolExecutor(4, 4, 0, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(1000)  // 有界
);
```

### 2. size() 在高并发下不准确

```java
// LinkedBlockingQueue.size() 用 AtomicInteger 维护
// 但在并发环境下，size() 返回的值在调用后可能已经变化
// 不要用 size() == 0 来判断队列是否为空
// 要用 isEmpty() 或直接 take()（会阻塞等待）
```

### 3. poll() 返回 null 的歧义

```java
Object item = queue.poll();
if (item == null) {
    // 是队列为空？还是队列中确实存了一个 null？
}

// 避免歧义：BlockingQueue 不允许存 null 元素
// add(null) / put(null) → NullPointerException
// 所以 poll() 返回 null 只有一种含义：队列空了
```

### 4. DelayQueue 的 getDelay 精度问题

```java
// DelayQueue 的 take() 通过 awaitNanos 实现
// 但 awaitNanos 不是绝对精确的——受 OS 调度影响
// 在 Windows 上可能有 15ms 的时钟精度误差
// 对精度要求极高的场景，需要结合 ScheduledExecutorService
```

### 5. PriorityBlockingQueue 的内存增长

```java
// PriorityBlockingQueue 是无界的
// 如果元素不断 put 且没有足够快的 take
// 内部数组会不断扩容 → OOM

// 如果需要限制大小：
int maxSize = 10000;
if (queue.size() >= maxSize) {
    queue.poll();  // 移除优先级最低的元素
}
queue.put(newElement);
```

### 6. SynchronousQueue 的 offer 和 poll 总是失败

```java
SynchronousQueue<String> q = new SynchronousQueue<>();

q.offer("hello");  // 返回 false（没有等待中的消费者）
String s = q.poll();  // 返回 null（没有等待中的生产者）

// SynchronousQueue 中：
// offer() 成功 = 有消费者正好在 take() 等待
// poll() 成功 = 有生产者正好在 put() 等待
// 没有时就立即返回 false/null，不阻塞
```
