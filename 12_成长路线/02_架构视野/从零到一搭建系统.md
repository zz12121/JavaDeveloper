# 从零到一搭建系统

> 从 0 到 1 搭建系统，考验的不是"会用多少框架"，而是"能不能让系统稳定跑到 1 万 QPS"。

---

## 一、搭建流程总览

```
阶段 1：需求分析（1 周）
阶段 2：技术选型（3 天）
阶段 3：架构设计（1 周）
阶段 4：MVP 实现（2-4 周）
阶段 5：压测 & 优化（1 周）
阶段 6：灰度上线（1 周）
阶段 7：复盘 & 规划（3 天）
```

---

## 二、阶段详解

### 2.1 需求分析

```
输出物：
1. 功能需求列表（PRD 评审）
2. 非功能需求（QPS、延迟、数据量）
3. 用户量预估（DAU、峰值 QPS）
4. 数据量预估（日增多少条？1 年后多少？）

关键公式：
峰值 QPS = 日 PV × 集中系数（通常 0.2） / 秒数（86400）
例：日 PV 100 万，峰值 QPS ≈ 100万 × 0.2 / 86400 ≈ 2 QPS（开心！）
例：秒杀场景，峰值 QPS 可能达 10 万+
```

### 2.2 技术选型

```
选型清单：

□ 语言（Java / Go / Python？）
  → Java：生态成熟，人手好找
  → Go：高并发原生支持，部署简单

□ 框架（Spring Boot / Gin / Django？）
  → Spring Boot：生态最完整
  → Gin：轻量高性能

□ 数据库（MySQL / PostgreSQL / MongoDB？）
  → 关系型：MySQL（最成熟）
  → 文档型：MongoDB（Schema 灵活）

□ 缓存（Redis / Memcached？）
  → Redis：功能丰富（推荐）
  → Memcached：极简高性能

□ 消息队列（Kafka / RabbitMQ / RocketMQ？）
  → 日志流处理：Kafka
  → 业务消息：RocketMQ（事务消息）

□ 搜索（Elasticsearch / Solr？）
  → ES：生态好，实时搜索

□ 监控（Prometheus / SkyWalking？）
  → Prometheus + Grafana（最主流）
```

### 2.3 架构设计

```
输出物：
1. 系统架构图（哪几个服务？怎么交互？）
2. 数据库设计（ER 图 + 表结构）
3. API 设计（RESTful / RPC 接口定义）
4. 缓存设计（Key 设计规范）
5. 异常处理方案（降级 / 熔断 / 限流）
6. 安全方案（认证 / 鉴权 / 防刷）
```

### 2.4 MVP 实现

```
MVP（最小可行产品）原则：
- 核心功能先上，锦上添花后做
- 能手动的不写代码（第一版运营数据导入可以 Excel）
- 能简单的不搞复杂（第一版用单体，别上微服务）

第一版技术栈建议（Java 后端）：
- Spring Boot 2.x + Maven
- MySQL 8.x + Druid 连接池
- Redis（Lettuce 客户端）
- Spring Data JPA / MyBatis Plus
- Spring Security + JWT（认证）
- Swagger / SpringDoc（API 文档）
- Docker + Docker Compose（本地开发环境）
```

### 2.5 压测 & 优化

```
压测工具选择：
- wrk：HTTP 接口压测（简单高效）
- JMeter：复杂场景压测（GUI 操作）
- k6：现代压测工具（JS 脚本，CI 集成好）

压测指标：
- QPS（每秒请求数）
- P50 / P99 延迟
- 错误率
- CPU / 内存 / 磁盘 IO

优化优先级：
1. 数据库慢查询（加索引 / 改 SQL）
2. 缓存命中率（热点数据缓存）
3. 连接池配置（数据库连接池 / Redis 连接池）
4. JVM 参数（堆大小 / GC 算法）
5. 代码层面（减少对象创建 / 使用池化）
```

### 2.6 灰度上线

```
灰度策略：
1. 内部员工先用（发现问题）
2. 灰度 5% 用户（观察错误率）
3. 灰度 20% → 50% → 100%
4. 每个阶段观察 1-2 天

降级开关：
- 功能降级：非核心功能可以关闭
- 限流：超出容量直接拒绝
- 熔断：依赖服务挂了自动熔断
```

---

## 三、技术债管理

```
从 0 到 1 必然有技术债，要管理而不是消除：

记录：
- 有哪些技术债？
- 不处理的后果是什么？
- 处理的优先级？

分类：
- 必须还：影响稳定性 / 安全性
- 建议还：影响扩展性
- 可以拖：不影响业务
```

---

## 四、常见陷阱

| 陷阱 | 后果 | 正确做法 |
|------|------|---------|
| 过度设计（V1 就上微服务） | 开发慢、运维复杂 | V1 用单体，后续拆分 |
| 没有压测就上线 | 流量一来就挂 | 上线前必须压测 |
| 没有降级方案 | 依赖挂了全挂 | 每个依赖都有降级 |
| 没有监控 | 出了问题不知道 | 上线前监控必须就绪 |
| 没有灰度 | 全量上线出问题影响所有人 | 必须灰度发布 |

---

## 五、上线检查清单

```
□ 数据库备份策略（全量 + 增量）
□ 监控告警配置（CPU / 内存 / 延迟 / 错误率）
□ 降级开关就位（配置中心可控）
□ 限流规则配置（网关层 + 服务层）
□ 回滚方案（代码回滚 + 数据回滚）
□ 值班安排（第一周必须有人盯）
□ 压测报告（QPS / 延迟 / 瓶颈点）
```
