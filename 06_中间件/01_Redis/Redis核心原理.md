# Redis核心原理

## 这个问题为什么存在？

> Redis是"单线程"的——为什么单线程还能达到10万+ QPS？  
> 为什么Redis是"原子性"的？Pipeline和事务有什么区别？

理解Redis的**核心原理**，才能：
1. **正确选型**：什么时候用Redis，什么时候不用
2. **性能调优**：知道瓶颈在哪（网络、内存、CPU）
3. **回答面试深挖**：从"怎么用"上升到"为什么快"

---

## 它是怎么解决问题的？

### Redis为什么快？（高频面试题）

**回答思路**：不只是"内存操作"，要分层解释。

```
Redis高性能的原因（分层）：
┌──────────────────────────────────────┐
│ 1. 内存操作（主要）                      │
│    → 内存随机访问 ~纳秒级                │
│    → 磁盘随机IO ~毫秒级（差10万倍）     │
│                                       │
│ 2. 单线程架构（避免锁竞争）             │
│    → 无多线程上下文切换开销               │
│    → 无锁竞争（不需要synchronized）       │
│                                       │
│ 3. IO多路复用（epoll/kqueue）          │
│    → 一个线程监听多个客户端连接            │
│    → 谁有数据来了，处理谁                │
│                                       │
│ 4. 高效数据结构（SDS/跳表/ZipList）  │
│    → SDS：O(1)获取长度，减少内存分配    │
│    → 跳表：范围查询O(log n)             │
│    → ZipList：连续内存，缓存友好          │
└──────────────────────────────────────┘
```

**单线程的误解**：

```
误解：Redis是单线程的 → 性能差
事实：Redis的"单线程"指的是"处理命令的核心逻辑是单线程"

Redis实际上是多线程的：
┌──────────────────────────────────────┐
│ Redis线程模型                          │
│  ├── 主线程（单线程）                 │
│  │   → 处理命令（网络IO + 命令执行）   │
│  │   → Redis 6.0之前，网络IO也是主线程│
│  │                                    │
│  ├── 后台线程（多线程，Redis 4.0+）  │
│  │   → BIOSYNC（异步关闭文件描述符）    │
│  │   → BioSyncAOF（异步刷AOF缓冲区）   │
│  │                                    │
│  └── IO线程（多线程，Redis 6.0+）   │
│      → 只负责网络IO（读/写）           │
│      → 命令执行还是单线程（保证原子性）  │
└──────────────────────────────────────┘
```

### Redis的单线程为什么不需要锁？

```
单线程模型的优势：
┌──────────────────────────────────────┐
│ 场景：两个客户端同时执行INCR counter   │
│                                       │
│ 多线程模型（需要锁）：                   │
│  1. 线程A读counter=0                  │
│  2. 线程B读counter=0                  │
│  3. 线程A写counter=1                  │
│  4. 线程B写counter=1（覆盖！）        │
│  → 需要加锁（synchronized）            │
│                                       │
│ 单线程模型（无锁）：                    │
│  1. 主线程读counter=0（处理客户端A） │
│  2. 主线程写counter=1                 │
│  3. 主线程读counter=1（处理客户端B） │
│  4. 主线程写counter=2                 │
│  → 不需要锁！天然原子性                │
└──────────────────────────────────────┘
```

**但是**：Redis 的 "单线程" 是指 **命令执行** 是单线程的。如果是多命令操作，还是需要用事务（MULTI/EXEC）或Lua脚本保证原子性。

### Redis事件循环（Event Loop）

Redis 用 **epoll**（Linux）或 **kqueue**（macOS/BSD）实现 IO 多路复用。

```
Redis事件循环（简化）
┌──────────────────────────────────────┐
│ while (1) {                           │
│     // 1. 等待事件（阻塞，直到有事件发生）│
│     aeProcessEvents() {                │
│         // 用epoll_wait等待多个客户端      │
│         epoll_wait(epfd, events, ...); │
│                                       │
│         // 2. 处理文件事件（客户端读写）  │
│         for (each event) {              │
│             if (读事件)  readQueryFromClient(); │
│             if (写事件)  writeToClient(); │
│         }                              │
│                                       │
│         // 3. 处理时间事件（定时任务）   │
│         processTimeEvents();             │
│     }                                  │
│ }                                      │
└──────────────────────────────────────┘
```

**文件事件 + 时间事件**：

```
文件事件（客户端读写）：
┌──────────────────────────────────────┐
│ 客户端1 → Redis（读事件）            │
│ 客户端2 ← Redis（写事件）            │
│ 多个客户端，一个线程处理               │
└──────────────────────────────────────┘

时间事件（定时任务）：
┌──────────────────────────────────────┐
│ 1. 过期Key的主动删除（hz配置）       │
│ 2. 异步任务（后台线程的通知）         │
│ 3. 触发RDB快照（BGSAVE）            │
│ 4. 触发AOF重写（BGREWRITEAOF）      │
└──────────────────────────────────────┘
```

### Pipeline vs 事务（MULTI/EXEC）

| 维度 | Pipeline | 事务（MULTI/EXEC） |
|------|-----------|-------------------|
| 目的 | 减少网络RTT（批量发送命令） | 保证原子性（一组命令要么都执行，要么都不执行） |
| 原子性 | ❌ 不保证（命令是一条条执行的） | ✅ 保证（全部执行完才提交） |
| 报错处理 | 某条命令失败，其他命令继续执行 | 某条命令失败，后续命令继续执行（Redis事务不支持回滚！） |
| 适用场景 | 批量写入/读取（性能优化） | 需要原子性的操作（比如转账） |

**Redis事务不支持回滚（重点）**：

```
MySQL事务：某条SQL出错 → 回滚所有操作
Redis事务：某条命令出错 → 继续执行后续命令（不回滚）

为什么？
→ Redis认为：命令出错是程序员的问题（语法错误），应该在开发阶段发现
→ 不支持回滚，设计更简单（不需要undo log）
```

### Lua脚本的原子性

**问题**：MULTI/EXEC 事务不保证"中间结果不对外可见"

```
场景：检查余额是否足够，足够才扣款
MULTI
GET balance      -- 读到100
-- 此时另一个客户端把balance改成了50
DECRBY balance 80
EXEC
→ 扣款成功（但此时余额是50，扣80就变负数了！）
```

**Lua脚本解决**：整个脚本是原子执行的（不会被其他命令打断）

```lua
-- 用Lua脚本保证原子性
local balance = redis.call('get', KEYS[1])
if (tonumber(balance) >= tonumber(ARGV[1])) then
    return redis.call('decrby', KEYS[1], ARGV[1])
else
    return -1  -- 余额不足
end
```

**为什么Lua脚本是原子的？**
→ Redis执行Lua脚本时，不会处理其他客户端的命令（整个脚本执行完才处理下一个）

**注意**：Lua脚本不能执行太长时间（默认5秒超时），否则会阻塞其他请求。

### Redis 6.0 的多线程IO

**问题**：单线程处理网络IO，CPU核心数增加也无法提升吞吐量。

**Redis 6.0的解决**：多线程IO（只负责网络读写，命令执行还是单线程）

```
Redis 6.0+ 的线程模型
┌──────────────────────────────────────┐
│ 主线程（单线程）                     │
│  ├── 接收客户端命令（IO线程辅助）    │
│  ├── 执行命令（核心，单线程保证原子）│
│  └── 发送结果（IO线程辅助）          │
│                                       │
│ IO线程（多线程，可配置）              │
│  ├── 读取客户端请求（分摊主线程）     │
│  └── 写回客户端响应（分摊主线程）     │
└──────────────────────────────────────┘
```

**配置**：

```bash
# Redis 6.0+
io-threads 4          # IO线程数（建议：CPU核心数 / 2）
io-threads-do-reads yes  # 是否用IO线程处理读（默认只处理写）
```

## 它和相似方案的本质区别是什么？

### Redis vs Memcached

| 维度 | Redis | Memcached |
|------|-------|-------------|
| 数据结构 | 6种（String/List/Hash/Set/ZSet/Bitmap） | 只有String |
| 持久化 | 支持（RDB/AOF） | 不支持（纯内存） |
| 高可用 | 支持（主从/哨兵/Cluster） | 需要客户端自己实现 |
| 单线程/多线程 | 命令执行单线程（6.0+ IO多线程） | 多线程（线程池） |
| 内存效率 | SDS/ZipList等压缩编码 | slab分配器（也有内存优化） |
| 适用场景 | 复杂数据结构、需要持久化 | 纯缓存（简单Key-Value） |

**本质区别**：Redis不只是缓存，还是**数据结构服务器**（支持复杂操作）。

---

### Redis单线程 vs 多线程数据库（MySQL）

| 维度 | Redis（单线程） | MySQL（多线程） |
|------|-------------------|---------------------|
| 并发模型 | 事件驱动（epoll） | 每连接一线程（或线程池） |
| 锁竞争 | 无（单线程） | 有（行锁、表锁、间隙锁） |
| 上下文切换 | 无 | 有（线程切换开销） |
| 适用场景 | 简单命令、内存操作 | 复杂查询、磁盘IO |

**为什么Redis能用单线程？**
→ 命令执行是内存操作（纳秒级），单线程足够快
→ 如果是磁盘IO（像MySQL），单线程会阻塞（需要多线程）

## 正确使用方式

### 1. 利用Pipeline提升吞吐量

```java
// ❌ 不用Pipeline：每次操作都等网络RTT
for (int i = 0; i < 10000; i++) {
    jedis.set("key:" + i, "value:" + i);  // 1万次网络往返
}

// ✅ 用Pipeline：批量发送，一次网络往返
Pipeline p = jedis.pipelined();
for (int i = 0; i < 10000; i++) {
    p.set("key:" + i, "value:" + i);
}
p.sync();  // 一次网络往返，执行1万次操作
```

**注意**：Pipeline不保证原子性（只是减少网络RTT）。如果需要原子性，用Lua脚本。

---

### 2. 用Lua脚本保证原子性

```java
// ✅ 用Lua脚本：检查余额 + 扣款，原子执行
String lua = `
    local balance = redis.call('get', KEYS[1])
    if (tonumber(balance) >= tonumber(ARGV[1])) then
        return redis.call('decrby', KEYS[1], ARGV[1])
    else
        return -1
    end
`;
Object result = jedis.eval(lua, 
    Collections.singletonList("balance:1000"), 
    Collections.singletonList("80"));
```

---

### 3. Redis 6.0+ 开启多线程IO

```bash
# redis.conf（Redis 6.0+）
io-threads 4              # IO线程数（建议：CPU核心数 / 2）
io-threads-do-reads yes    # 是否用IO线程处理读请求
```

**注意**：
- 命令执行还是单线程（保证原子性）
- 只在高吞吐量场景才有效果（低于10万 QPS不需要开）

## 边界情况和坑

### 坑1：Lua脚本执行时间太长

```
现象：
- Lua脚本中有大循环（比如遍历1000万个Key）
- Redis阻塞（其他客户端的命令都排队）
- 客户端超时

解决：
1. Lua脚本要尽量简单（只做Redis命令的组合）
2. 避免在Lua中做复杂计算
3. 如果必须复杂计算，考虑放到客户端做
```

---

### 坑2：Pipeline中的命令过多

```
现象：
- Pipeline一次发送10万条命令
- 客户端内存溢出（缓存所有命令的响应）
- 或：超过`client-output-buffer-limit`（服务端断开连接）

解决：
Pipeline分批执行（比如每1000条一批）
```

---

### 坑3：Redis事务不支持回滚

```
现象：
- MULTI/EXEC事务中，某条命令失败了
- 但后续命令继续执行（不回滚）
- 数据处于"部分完成"状态

解决：
1. 用Lua脚本（支持复杂的原子操作）
2. 或者：业务层自己处理"部分完成"的情况
```

---

### 坑4：单线程模型的CPU瓶颈

```
现象：
- Redis CPU占用100%（单核）
- 但机器有多个CPU核心（其他核心空闲）

解决：
1. Redis 6.0+ 开启多线程IO（io-threads配置）
2. 或者用Redis Cluster（多个节点，每个节点用单线程）
```
