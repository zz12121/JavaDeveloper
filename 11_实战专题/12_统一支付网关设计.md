# 统一支付网关设计

> 公司对接微信支付、支付宝、银联、Apple Pay 等多个支付渠道，各业务线各自对接，代码重复、对账混乱、渠道切换困难。设计一个统一的支付网关，实现多渠道收口、异步回调幂等、渠道降级、T+1 对账。

---

## 业务背景

```
现状痛点：
  订单服务直接调微信支付 SDK
  会员服务直接调支付宝 SDK
  积分商城直接调 Apple Pay SDK
  → 每个业务线都要处理：签名验签、回调幂等、退款、对账
  → 微信升级 SDK → 改 10 个服务
  → 对账：10 个服务各自对账，总账对不上
  → 渠道故障：某个渠道挂了，业务线不知道，用户支付失败

目标架构：
  业务线 ──→ 支付网关 ──→ 微信支付
                       ──→ 支付宝
                       ──→ 银联
                       ──→ Apple Pay
  业务线不直接接触支付 SDK，只和支付网关交互
```

```java
// 策略模式：统一接口 + 多渠道适配
public interface PaymentChannelAdapter {

    /** 渠道标识 */
    String getChannel();

    /** 创建预支付单，返回支付参数（给前端调起支付） */
    PayResult createPayment(PaymentRequest request);

    /** 查询支付状态 */
    PaymentStatus queryPayment(String channelOrderNo);

    /** 发起退款 */
    RefundResult refund(RefundRequest request);

    /** 验证回调签名 */
    boolean verifyCallback(Map<String, String> params, String signature);

    /** 解析回调参数 */
    CallbackResult parseCallback(Map<String, String> params);
}

// 微信支付适配器
@Component
public class WechatPayAdapter implements PaymentChannelAdapter {

    @Override
    public String getChannel() { return "WECHAT"; }

    @Override
    public PayResult createPayment(PaymentRequest request) {
        // 调用微信支付 SDK（统一下单接口）
        WxPayUnifiedOrderV3Request wxReq = new WxPayUnifiedOrderV3Request();
        wxReq.setOutTradeNo(request.getPaymentNo());
        wxReq.setDescription(request.getSubject());
        wxReq.setAmount(new WxPayUnifiedOrderV3Request.Amount()
            .setTotal(request.getAmount())
            .setCurrency(request.getCurrency()));
        wxReq.setNotifyUrl(callbackUrl);

        WxPayUnifiedOrderV3Result result = wxPayService.createOrderV3(wxReq);

        return PayResult.builder()
            .paymentNo(request.getPaymentNo())
            .channelOrderNo(result.getPaymentNo())   // 微信预支付单号
            .payParams(result.getPayInfo())           // JSAPI/APP 调起参数
            .build();
    }

    @Override
    public boolean verifyCallback(Map<String, String> params, String signature) {
        // 微信 V3：RSA 验签 + AEAD 解密
        return wxPayService.verifyV3Signature(params, signature);
    }

    @Override
    public CallbackResult parseCallback(Map<String, String> params) {
        WxPayOrderNotifyV3Result result = wxPayService.parseOrderNotifyV3Result(
            params, signature);
        return CallbackResult.builder()
            .channelOrderNo(result.getOutTradeNo())
            .tradeStatus(result.getTradeState())       // SUCCESS / NOTPAY / CLOSED
            .paidAmount(result.getAmount().getTotal())
            .paidTime(result.getSuccessTime())
            .build();
    }
}
```

```java
// 渠道路由：根据业务场景选择最优渠道
@Service
public class PaymentChannelRouter {

    @Autowired
    private List<PaymentChannelAdapter> adapters;

    @Autowired
    private ChannelHealthService healthService;

    /**
     * 路由规则：
     * 1. 业务线指定渠道 → 直接用
     * 2. 用户偏好渠道 → 优先用
     * 3. 渠道健康检查 → 自动剔除故障渠道
     * 4. 按金额路由 → 小额走微信/支付宝，大额走银联
     */
    public PaymentChannelAdapter route(RouteContext context) {
        // 1. 业务指定
        if (context.getChannel() != null) {
            PaymentChannelAdapter adapter = getAdapter(context.getChannel());
            if (healthService.isHealthy(context.getChannel())) {
                return adapter;
            }
            log.warn("指定渠道不可用: {}, 降级路由", context.getChannel());
        }

        // 2. 按金额和场景自动选择
        if (context.getAmount() < 500_00) {
            // 小额：优先微信 → 支付宝 → 银联
            return selectFirstHealthy("WECHAT", "ALIPAY", "UNION");
        } else {
            // 大额：优先银联 → 微信 → 支付宝
            return selectFirstHealthy("UNION", "WECHAT", "ALIPAY");
        }
    }

    private PaymentChannelAdapter selectFirstHealthy(String... channels) {
        for (String channel : channels) {
            if (healthService.isHealthy(channel)) {
                return getAdapter(channel);
            }
        }
        throw new PaymentException("所有支付渠道均不可用");
    }
}
```

## 核心设计三：异步回调幂等处理

```
回调处理是支付系统最关键的一环：
  - 渠道可能重复通知（网络超时重试）
  - 业务处理必须幂等
  - 先记账再回调业务线
  - 回调业务线失败要重试

回调处理流程：
┌─────────────┐
│ 渠道回调     │
│（微信/支付宝）│
└──────┬──────┘
       │
       ▼
┌─────────────┐     失败     ┌──────────┐
│ 1. 验签     │ ──────────→ │ 返回 FAIL │
│（防伪造）   │              │ 渠道重试  │
└──────┬──────┘              └──────────┘
       │ 通过
       ▼
┌─────────────┐     重复     ┌──────────┐
│ 2. 幂等检查 │ ──────────→ │ 返回 OK  │
│（Redis锁）  │  已处理过     │（不重复） │
└──────┬──────┘              └──────────┘
       │ 首次
       ▼
┌─────────────┐
│ 3. 更新支付单│
│ status→SUCCESS│
│ 回填渠道单号 │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 4. 发 MQ 通知│
│ → 订单服务   │
│ → 会员服务   │
│ → 营销服务   │
└──────┬──────┘
       │
       ▼
┌──────────┐
│ 返回 SUCCESS│
│（渠道停止重试）│
└──────────┘
```

```java
@RestController
@RequestMapping("/callback")
public class PaymentCallbackController {

    @Autowired
    private PaymentCallbackService callbackService;

    /**
     * 微信支付回调
     * 注意：必须同步返回 SUCCESS/FAIL，不能做耗时操作
     */
    @PostMapping("/wechat")
    public Map<String, String> wechatCallback(
            @RequestBody String body,
            @RequestHeader("Wechatpay-Signature") String signature,
            @RequestHeader("Wechatpay-Serial") String serial) {

        try {
            // 1. 验签 + 解密
            callbackService.verifyAndProcess("WECHAT", body, signature, serial);

            // 2. 成功，返回 SUCCESS（XML 格式或 JSON 格式取决于微信版本）
            return Map.of("code", "SUCCESS", "message", "成功");
        } catch (Exception e) {
            log.error("微信回调处理失败", e);
            // 返回 FAIL，微信会重试
            return Map.of("code", "FAIL", "message", e.getMessage());
        }
    }

    /**
     * 支付宝回调
     */
    @PostMapping("/alipay")
    public String alipayCallback(HttpServletRequest request) {
        try {
            Map<String, String> params = getRequestParams(request);
            callbackService.verifyAndProcess("ALIPAY", params);
            return "success";  // 支付宝要求返回纯文本 "success"
        } catch (Exception e) {
            log.error("支付宝回调处理失败", e);
            return "failure";
        }
    }
}
```

```java
@Service
public class PaymentCallbackService {

    @Autowired
    private PaymentChannelRouter router;

    @Autowired
    private PaymentRequestMapper paymentMapper;

    @Autowired
    private RedisTemplate<String, String> redis;

    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    /**
     * 核心：验证 + 幂等 + 更新 + 通知
     * 整个过程必须在回调响应之前完成（渠道一般要求 5 秒内响应）
     */
    public void verifyAndProcess(String channel, Object rawData, String... signData) {
        // 1. 验签
        PaymentChannelAdapter adapter = router.getAdapter(channel);
        CallbackResult result = adapter.verifyAndParse(rawData, signData);

        String paymentNo = result.getPaymentNo();

        // 2. 幂等检查（Redis SETNX，防止重复处理）
        String lockKey = "pay:callback:lock:" + paymentNo;
        Boolean locked = redis.opsForValue().setIfAbsent(lockKey, "1", 24, TimeUnit.HOURS);
        if (Boolean.FALSE.equals(locked)) {
            log.info("回调已处理，幂等跳过: paymentNo={}", paymentNo);
            return;
        }

        // 3. 更新支付单（乐观锁，只有 INIT/PROCESSING 才更新）
        int rows = paymentMapper.updateStatusByCallback(
            paymentNo,
            "INIT,PROCESSING",   // 可更新的状态
            "SUCCESS",            // 目标状态
            result.getChannelOrderNo(),
            result.getPaidAmount(),
            result.getPaidTime()
        );

        if (rows == 0) {
            // 状态不是 INIT/PROCESSING，说明已处理过或已关闭
            log.info("支付单状态已变更，跳过: paymentNo={}", paymentNo);
            return;
        }

        // 4. 异步通知业务线（发 MQ，不阻塞回调响应）
        PaymentSuccessEvent event = new PaymentSuccessEvent();
        event.setPaymentNo(paymentNo);
        event.setBizOrderNo(result.getBizOrderNo());
        event.setBizType(result.getBizType());
        event.setChannel(channel);
        event.setPaidAmount(result.getPaidAmount());

        rocketMQTemplate.syncSend("payment-success-topic", JSON.toJSONString(event));

        log.info("支付回调处理成功: paymentNo={}, channel={}", paymentNo, channel);
    }
}
```

## 核心设计四：T+1 对账

```
对账流程：
  T日：产生交易 → 支付网关记录本地流水
  T+1日 01:00：拉取渠道对账文件
              ↓
         本地流水 vs 渠道流水 逐笔比对
              ↓
         差异 → 短款（渠道有网关没有）→ 补单
         差异 → 长款（网关有渠道没有）→ 主动查单确认
              ↓
         生成对账报告 → 邮件/钉钉通知财务
```

```java
@Component
public class ReconciliationTask {

    @Autowired
    private PaymentRequestMapper paymentMapper;

    @Autowired
    private ReconciliationService reconService;

    /**
     * 每日凌晨 1 点执行，处理前一天的账
     */
    @Scheduled(cron = "0 0 1 * * ?")
    public void dailyReconciliation() {
        String date = LocalDate.now().minusDays(1).format(DateTimeFormatter.ISO_DATE);

        // 1. 拉取各渠道对账文件
        Map<String, List<ChannelBill>> channelBills = new HashMap<>();
        for (String channel : Arrays.asList("WECHAT", "ALIPAY", "UNION")) {
            List<ChannelBill> bills = reconService.downloadBill(channel, date);
            channelBills.put(channel, bills);
        }

        // 2. 查询本地流水
        List<PaymentRequest> localPayments = paymentMapper.selectByPaidDate(date);

        // 3. 逐笔比对
        ReconciliationResult result = reconService.compare(localPayments, channelBills);

        // 4. 处理差异
        for (Discrepancy diff : result.getDiscrepancies()) {
            switch (diff.getType()) {
                case SHORT:  // 渠道有，网关没有 → 可能是回调丢失
                    reconService.supplementPayment(diff);
                    break;
                case LONG:   // 网关有，渠道没有 → 主动查渠道确认
                    reconService.queryAndConfirm(diff);
                    break;
                case MISMATCH:  // 金额不一致 → 告警人工处理
                    reconService.alertAmountMismatch(diff);
                    break;
            }
        }

        // 5. 生成对账报告
        reconService.generateReport(date, result);
    }
}
```

```java
// 对账比对核心逻辑
@Service
public class ReconciliationService {

    public ReconciliationResult compare(
            List<PaymentRequest> localPayments,
            Map<String, List<ChannelBill>> channelBills) {

        // 构建本地流水 Map（key = biz_order_no）
        Map<String, PaymentRequest> localMap = localPayments.stream()
            .collect(Collectors.toMap(
                p -> p.getBizOrderNo() + ":" + p.getChannel(),
                Function.identity()
            ));

        // 构建渠道流水 Map
        Map<String, ChannelBill> channelMap = channelBills.values().stream()
            .flatMap(List::stream)
            .collect(Collectors.toMap(
                b -> b.getOutTradeNo() + ":" + b.getChannel(),
                Function.identity()
            ));

        ReconciliationResult result = new ReconciliationResult();

        // 遍历本地流水，找长款（本地有渠道没有）
        for (Map.Entry<String, PaymentRequest> entry : localMap.entrySet()) {
            if (!channelMap.containsKey(entry.getKey())) {
                result.addDiscrepancy(Discrepancy.longEntry(entry.getValue()));
            } else {
                // 金额比对
                ChannelBill bill = channelMap.get(entry.getKey());
                if (!entry.getValue().getAmount().equals(bill.getAmount())) {
                    result.addDiscrepancy(Discrepancy.mismatch(entry.getValue(), bill));
                }
                channelMap.remove(entry.getKey());  // 已匹配，从 map 移除
            }
        }

        // 剩余的渠道流水 = 短款（渠道有本地没有）
        for (ChannelBill bill : channelMap.values()) {
            result.addDiscrepancy(Discrepancy.shortEntry(bill));
        }

        return result;
    }
}
```

## 渠道健康监控与降级

```java
@Component
public class ChannelHealthService {

    @Autowired
    private RedisTemplate<String, String> redis;

    // 滑动窗口统计渠道成功率
    private static final String HEALTH_KEY = "pay:channel:health:";

    /**
     * 记录一次调用结果
     */
    public void recordResult(String channel, boolean success, long costMs) {
        String key = HEALTH_KEY + channel;

        // 记录到 Redis sorted set（score = 时间戳，value = 成功/失败）
        long now = System.currentTimeMillis();
        String value = (success ? "1" : "0") + ":" + costMs;
        redis.opsForZSet().add(key, value, now);

        // 只保留最近 5 分钟
        redis.opsForZSet().removeRangeByScore(key, 0, now - 5 * 60 * 1000);
    }

    /**
     * 判断渠道是否健康（最近 5 分钟成功率 > 90%）
     */
    public boolean isHealthy(String channel) {
        String key = HEALTH_KEY + channel;
        long now = System.currentTimeMillis();

        // 最近 5 分钟的总调用数
        Long total = redis.opsForZSet().count(key, now - 5 * 60 * 1000, now);
        if (total == null || total == 0) {
            return true;  // 没有数据，默认健康
        }

        // 成功数（value 以 "1:" 开头的）
        // Redis 不支持复杂的 ZSet 查询，改用计数器
        String successKey = HEALTH_KEY + channel + ":success";
        String failKey = HEALTH_KEY + channel + ":fail";
        Long successCount = redis.opsForValue().get(successKey) != null
            ? Long.parseLong(redis.opsForValue().get(successKey)) : 0;
        Long failCount = redis.opsForValue().get(failKey) != null
            ? Long.parseLong(redis.opsForValue().get(failKey)) : 0;

        long allCount = successCount + failCount;
        if (allCount < 10) {
            return true;  // 样本太少，不做判断
        }

        double successRate = (double) successCount / allCount;
        return successRate >= 0.9;
    }
}
```

## 生产踩坑记录

### 坑 1：微信回调验签失败

```
原因：
  1. 微信 V3 用 RSA 签名，需要用微信平台证书验签，不是商户证书
  2. 证书过期未更新
  3. 回调 body 被 Servlet 提前读取，Spring 再读就空了

解决：
  1. 定时拉取微信平台证书（每小时），缓存到本地
  2. 用 ContentCachingRequestWrapper 包装请求，允许多次读取 body
  3. 验签失败不要返回 SUCCESS，返回 FAIL 让微信重试
```

### 坑 2：回调处理耗时过长，渠道超时重试

```
原因：回调里做了太多事（更新支付单 + 更新订单 + 发优惠券 + 记录积分）
     → 超过渠道要求的 5 秒响应时间
     → 渠道认为你没收到，又发了一次
     → 两次同时处理 → 幂等没做好 → 优惠券发了两次

解决：
  回调接口只做三件事：验签 → 更新支付单 → 发 MQ
  其他业务逻辑由 MQ 消费者异步处理
  确保回调在 200ms 内返回
```

### 坑 3：退款后渠道回调延迟

```
场景：用户退款 → 渠道处理成功 → 但回调延迟了 2 小时才到
     → 用户投诉「退款怎么还没到账」

解决：
  1. 退款后主动轮询查询退款状态（每分钟查一次，最多查 10 次）
  2. 轮询到成功就直接更新本地状态，不等回调
  3. 同时支持回调更新（双重保障）
```

## 追问链

### 追问 1：如果用户支付成功了但回调丢失了怎么办？

> "主动查询补偿。用户在前端点击「我已付款」→ 前端调网关的查询接口 → 网关调渠道的订单查询接口 → 如果渠道返回已支付 → 网关更新本地状态。另外，对账任务每天凌晨也会发现这类差异（短款），自动补单。双保险。"

### 追问 2：支付网关怎么保证和业务系统的一致性？

> "支付网关只管支付流水，不管业务状态。一致性靠事件驱动：支付成功 → 网关发 MQ → 业务线消费。业务线消费要幂等（先查状态再更新）。如果 MQ 消费失败，有重试机制。最坏情况下靠 T+1 对账兜底。不搞分布式事务，用最终一致性就够了。"

### 追问 3：退款和原支付必须走同一个渠道吗？

> "是的，微信支付的钱必须退回微信。退款时从原支付单取出渠道信息，自动路由到对应渠道。如果原渠道不可用（极罕见），就走渠道的手动退款流程，不自动处理。"

### 追问 4：怎么防止重复下单（用户连续点了两次支付）？

> "支付单号 = 业务单号 + 渠道，数据库唯一索引 `UNIQUE KEY uk_biz (biz_order_no, biz_type)`。同一笔业务单同一个渠道只能创建一个支付单，第二次插入会失败，返回已有的支付单号和支付参数。前端拿到支付参数后调起支付，即使调两次，支付渠道侧的 out_trade_no 是同一个，不会重复扣款。"

### 追问 5：微信支付和支付宝的回调格式完全不同，怎么统一处理？

> "策略模式。定义统一的 `PaymentChannelAdapter` 接口，每个渠道实现自己的 `verifyCallback` 和 `parseCallback`。回调入口按渠道分发到不同的 Controller 方法（`/callback/wechat`、`/callback/alipay`），各自验签后转换为统一的 `CallbackResult` 对象，后续处理逻辑完全一样。新增渠道只需要加一个 Adapter 实现类，不影响已有代码。"

## 排查 Checklist

```
□ 回调接口是否幂等？ → Redis SETNX + 数据库状态乐观锁
□ 回调响应时间 < 200ms？ → 只做验签+更新+发MQ，不放业务逻辑
□ 渠道证书有没有过期？ → 定时拉取平台证书，告警
□ 支付单有唯一索引吗？ → biz_order_no + biz_type
□ 对账任务跑了吗？ → 每日 T+1，差异告警
□ 渠道降级生效了吗？ → 健康检查 + 自动切换
□ 退款有轮询补偿吗？ → 主动查询，不只依赖回调
□ MQ 消费有重试吗？ → 死信队列兜底
□ 支付金额用分存储了吗？ → 绝对不能用浮点数存金额
□ 日志记录完整吗？ → 关键操作全链路日志
```
