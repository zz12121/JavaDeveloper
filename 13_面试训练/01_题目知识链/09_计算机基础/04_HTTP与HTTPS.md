## 知识链提示

这道题应该让你联想到：

- `TLS握手` → ClientHello / ServerHello / 密钥交换 / Finished
- `对称加密vs非对称` → AES（对称）快，RSA/ECDHE（非对称）慢
- `数字证书` → CA 签名，证书链验证
- `HTTPS性能` → TLS 握手开销，Session 复用，OCSP Stapling
- `HTTP_2与TLS` → HTTP/2 要求必须跑在 TLS 上（实际部署）

## 参考要点（盲答后再看）


**HTTP vs HTTPS**：

```
HTTP  = TCP → HTTP（明文，端口80）
HTTPS = TCP → TLS → HTTP（加密，端口443）

安全风险（HTTP）：
- 窃听（同一 WiFi 下抓包看所有请求）
- 篡改（ISP 插入广告）
- 中间人（伪造服务器）
```

**TLS 1.2 握手过程**（RSA 密钥交换，2-RTT）：

```
客户端                                服务器
    |                                    |
    |------- ClientHello --------------->|
    |  (支持的加密套件、随机数、TLS版本)     |
    |<------ ServerHello ---------------|
    |  (选定的加密套件、随机数)               |
    |<------ Certificate --------------|
    |  (服务器证书，含公钥)                  |
    |<------ ServerHelloDone ----------|
    |                                    |
    |------- ClientKeyExchange -------->|
    |  (用服务器公钥加密「预主密钥」)          |
    |------- ChangeCipherSpec -------->|
    |------- Finished ----------------->|
    |                                    |
    |<------ ChangeCipherSpec ---------|
    |<------ Finished ------------------|
    |                                    |
  (后续用协商的对称密钥加密通信)
```

**为什么不全用非对称加密？**

```
非对称加密（RSA）：慢（比对称加密慢 100~1000 倍）
对称加密（AES）：快

TLS 设计：
- 用非对称加密「协商」对称密钥（只做一次，握手阶段）
- 用对称密钥加密后续所有数据（快）
```

**TLS 1.3 的改进**（1-RTT，甚至 0-RTT）：

```
TLS 1.2：2-RTT（ClientHello → ServerHello... → 客户端验证证书 → 才开始传数据）
TLS 1.3：1-RTT（ClientHello 里就带上支持的密钥交换方式，服务器回复里直接发证书+密钥）

0-RTT：客户端缓存了服务器的公钥
       → 第一个包就能带加密的应用数据（不安全，可能被重放攻击）
```

**证书链验证**：

```
浏览器内置根证书（Root CA）
验证过程：
  服务器证书 → 中间CA证书 → 根CA证书
  用根CA的公钥验证中间CA的签名
  用中间CA的公钥验证服务器证书的签名
  → 信任链完整 → 证书可信
```
