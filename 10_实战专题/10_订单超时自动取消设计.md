# 订单超时自动取消设计

> 电商下单后 30 分钟未支付自动关闭，释放库存和优惠券。看似简单，但涉及延迟精度、消息可靠性、支付并发窗口、兜底补偿等生产问题，是系统设计高频考题。

---

## 题目一：订单超时自动取消

### 业务背景

```
电商下单流程：
  用户提交订单 → 扣减库存 → 等待支付 → 支付成功 → 订单完成
                                   ↓
                              超时未支付
                                   ↓
                            自动取消 → 释放库存 → 释放优惠券

核心约束：
  - 超时时间：30 分钟（部分业务 15 分钟）
  - 延迟上限：超过超时时间 ≤ 1 秒触发取消（用户体感）
  - 吞吐量：日均订单 50 万，峰值 QPS 2000 单/s
  - 不能漏取消：漏一个 = 库存永远不释放 = 资损
  - 不能误取消：用户正在支付时取消 = 投诉
```

---

### 方案对比

| 方案 | 实现方式 | 延迟精度 | 吞吐能力 | 可靠性 | 复杂度 | 推荐场景 |
|------|---------|---------|---------|--------|--------|---------|
| 定时任务扫表 | `@Scheduled` 轮询 DB | 差（依赖扫描频率） | 中（全表扫描压力） | 高 | 低 | 订单量小、精度要求不高 |
| DelayQueue | JVM 内存队列 | 高（毫秒级） | 低（受内存限制） | 低（宕机丢失） | 低 | 单机、不重要的场景 |
| RocketMQ 延迟消息 | 下单时发延迟消息 | 较高（秒级） | 高 | 高（消息不丢） | 中 | **推荐，RocketMQ 生态内首选** |
| Redis Key 过期通知 | SETEX + Keyspace 通知 | 较高 | 高 | 中（通知可能丢） | 中 | 已有 Redis 基础设施 |
| 时间轮（Netty HashedWheelTimer） | 内存时间轮轮询 | 高（毫秒级） | 中 | 低（不持久化） | 中 | 单机低延迟任务 |

**生产推荐**：RocketMQ 延迟消息 + 定时任务兜底扫表，双保险。

### 方案详解 A：RocketMQ 延迟消息（主方案）

#### 整体流程

```
用户下单
   │
   ▼
┌─────────────────┐     延迟消息      ┌─────────────────┐
│  订单服务        │ ──────────────→   │  RocketMQ Broker │
│  创建订单(待支付) │  delayLevel=16   │  延迟 30 分钟投递  │
│  发送延迟消息     │                   └────────┬────────┘
└─────────────────┘                            │
                                         30 分钟后投递
                                                │
                                                ▼
                                       ┌─────────────────┐
                                       │  取消订单消费者   │
                                       │  1. 查订单状态    │
                                       │  2. 未支付 → 取消  │
                                       │  3. 释放库存/优惠券│
                                       └─────────────────┘
```

#### 核心代码

```java
// ===== 1. 下单时发送延迟消息 =====
@Service
public class OrderService {

    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    @Autowired
    private OrderMapper orderMapper;

    public OrderDTO createOrder(CreateOrderRequest request) {
        // 1. 创建订单
        Order order = new Order();
        order.setOrderNo(generateOrderNo());
        order.setUserId(request.getUserId());
        order.setStatus(OrderStatus.UNPAID);
        order.setTimeoutAt(LocalDateTime.now().plusMinutes(30));
        orderMapper.insert(order);

        // 2. 发送延迟消息（delayLevel=16 对应 RocketMQ 30 分钟）
        OrderTimeoutMessage msg = new OrderTimeoutMessage();
        msg.setOrderNo(order.getOrderNo());
        msg.setUserId(request.getUserId());
        msg.setTimeoutAt(order.getTimeoutAt());

        rocketMQTemplate.syncSend(
            "order-timeout-topic",
            MessageBuilder.withPayload(JSON.toJSONString(msg))
                .setDelayLevel(16)  // RocketMQ 延迟级别 16 = 30min
                .build()
        );

        return OrderDTO.from(order);
    }
}
```

```java
// ===== 2. 消费者：取消超时订单 =====
@Component
@RocketMQMessageListener(
    topic = "order-timeout-topic",
    consumerGroup = "order-timeout-consumer-group",
    consumeMode = ConsumeMode.CONCURRENTLY  // 并发消费，提高吞吐
)
public class OrderTimeoutConsumer implements RocketMQListener<String> {

    @Autowired
    private OrderMapper orderMapper;
    @Autowired
    private StockService stockService;
    @Autowired
    private CouponService couponService;

    @Override
    public void onMessage(String message) {
        OrderTimeoutMessage msg = JSON.parseObject(message, OrderTimeoutMessage.class);

        // 1. 幂等检查：订单已处理则直接跳过
        Order order = orderMapper.selectByOrderNo(msg.getOrderNo());
        if (order == null || order.getStatus() != OrderStatus.UNPAID) {
            log.info("订单已处理，跳过: orderNo={}, status={}",
                     msg.getOrderNo(),
                     order == null ? "不存在" : order.getStatus());
            return;  // 已支付/已取消/不存在 → 不处理
        }

        // 2. 二次确认：再查一次，防止并发窗口问题
        // （用户可能在消息到达前 1 秒刚完成支付）
        order = orderMapper.selectByOrderNoForUpdate(msg.getOrderNo());
        if (order.getStatus() != OrderStatus.UNPAID) {
            return;
        }

        // 3. 执行取消
        try {
            cancelOrder(order);
            log.info("订单超时取消成功: orderNo={}", msg.getOrderNo());
        } catch (Exception e) {
            log.error("订单取消失败: orderNo={}", msg.getOrderNo(), e);
            throw new RuntimeException("取消失败，触发重试", e);
        }
    }

    private void cancelOrder(Order order) {
        // 3.1 更新订单状态（加乐观锁，防止并发修改）
        int rows = orderMapper.updateStatus(
            order.getId(),
            OrderStatus.UNPAID,
            OrderStatus.CANCELLED_TIMEOUT,
            "SYSTEM",
            "下单超时自动取消"
        );
        if (rows == 0) {
            log.warn("订单状态已变更，跳过取消: orderNo={}", order.getOrderNo());
            return;
        }

        // 3.2 释放库存
        stockService.releaseStock(order.getOrderItems());

        // 3.3 释放优惠券
        couponService.releaseCoupon(order.getUserId(), order.getCouponId());

        // 3.4 记录操作日志
        orderLogMapper.insert(OrderLog.cancel(order.getId(), "SYSTEM", "超时自动取消"));
    }
}
```

```sql
-- 乐观锁更新：只更新 UNPAID 状态
UPDATE `order`
SET status = 'CANCELLED_TIMEOUT',
    cancel_reason = '下单超时自动取消',
    cancel_time = NOW(),
    version = version + 1
WHERE id = ? AND status = 'UNPAID' AND version = ?;
-- 返回 0 表示状态已被其他线程修改
```

### 方案详解 B：定时任务兜底（补偿方案）

MQ 延迟消息本身很可靠，但生产环境总有极端情况：Broker 宕机、消息积压、消费端 Bug。需要一个**兜底扫表机制**。

```java
@Component
public class OrderTimeoutCompensateTask {

    @Autowired
    private OrderMapper orderMapper;

    /**
     * 每 1 分钟扫描一次超时未支付订单
     * 注意：只扫过去 30~35 分钟的，避免和 MQ 延迟消息冲突
     */
    @Scheduled(cron = "0 * * * * ?")
    public void compensateTimeoutOrders() {
        // 扫描 35 分钟前创建且仍未支付的订单（比 MQ 延迟多留 5 分钟 buffer）
        LocalDateTime threshold = LocalDateTime.now().minusMinutes(35);
        List<Order> timeoutOrders = orderMapper.selectUnpaidOrdersBefore(threshold);

        log.info("兜底扫描发现超时订单: count={}", timeoutOrders.size());

        for (Order order : timeoutOrders) {
            try {
                // 复用取消逻辑，幂等处理
                orderTimeoutService.cancelIfUnpaid(order);
            } catch (Exception e) {
                log.error("兜底取消失败: orderNo={}", order.getOrderNo(), e);
                // 失败不抛异常，不影响其他订单处理
            }
        }
    }
}
```

```sql
-- 高效扫描（利用 status + created_at 复合索引）
SELECT id, order_no, user_id, status, created_at
FROM `order`
WHERE status = 'UNPAID'
  AND created_at < #{threshold}
LIMIT 500;

-- 索引
CREATE INDEX idx_status_created ON `order` (status, created_at);
```

**兜底任务的关键设计**：
```
1. 扫描窗口 = 超时时间 + 5 分钟 buffer
   → MQ 30 分钟投递，兜底 35 分钟才扫
   → 正常情况下 MQ 已经处理了，兜底扫到 0 条
   → MQ 故障时兜底才真正工作

2. 限制扫描数量 LIMIT 500
   → 防止一次扫太多拖垮 DB
   → 500 × 1 分钟 = 每分钟处理 500 单，够用

3. 幂等处理
   → 取消逻辑天然幂等（status='UNPAID' 条件更新）
   → 重复取消不影响

4. 分布式锁（多实例部署时）
   → 用 Redis SETNX 保证同一时刻只有一个实例执行扫表
```

### 方案详解 C：Redis Key 过期通知（备选方案）

适用于没有 RocketMQ 但有 Redis 的场景。

```java
// ===== 1. 下单时设置过期 Key =====
public OrderDTO createOrder(CreateOrderRequest request) {
    Order order = orderMapper.insert(buildOrder(request));

    // 设置带过期时间的标记 Key
    String key = "order:timeout:" + order.getOrderNo();
    redisTemplate.opsForValue().set(
        key,
        order.getId().toString(),
        30,
        TimeUnit.MINUTES
    );

    return OrderDTO.from(order);
}

// ===== 2. Redis Key 过期监听器 =====
@Component
public class RedisKeyExpirationListener extends KeyExpirationEventMessageListener {

    public RedisKeyExpirationListener(RedisMessageListenerContainer container) {
        super(container);
    }

    @Override
    public void onMessage(Message message, byte[] pattern) {
        String expiredKey = message.toString();

        // 只处理订单超时 Key
        if (expiredKey.startsWith("order:timeout:")) {
            String orderNo = expiredKey.replace("order:timeout:", "");
            log.info("收到订单超时事件: orderNo={}", orderNo);

            try {
                orderTimeoutService.cancelIfUnpaid(orderNo);
            } catch (Exception e) {
                // 过期通知不保证送达，需要兜底扫表
                log.error("处理订单超时失败: orderNo={}", orderNo, e);
            }
        }
    }
}
```

```
Redis 过期通知的坑：
1. 不保证准时：过期时间到了，通知可能有几秒延迟
2. 不保证送达：Redis 重启、网络抖动，通知就丢了
3. 不保证不重复：极端情况下可能收到两次
4. 需要配置 notify-keyspace-events Ex

结论：只能作为辅助方案，不能作为主方案，必须配合兜底扫表
```

### 生产踩坑记录

#### 坑 1：用户正在支付，订单被取消

```
场景：用户点了支付 → 跳转支付宝 → 支付宝回调还在路上 → 30 分钟到了 → 订单取消
     → 支付宝回调到达 → 更新为已支付 → 但库存已经释放了

解决：
1. 支付回调加幂等校验：
   - 回调到达时检查订单状态
   - 如果已取消 → 自动退款（金额原路退回）
   - 如果是 UNPAID → 更新为已支付

2. 适当延长超时时间：
   - 普通商品 30 分钟
   - 需要跳转第三方支付的 45 分钟（留足支付链路时间）

3. 短信提醒：
   - 超时前 5 分钟发短信提醒「您的订单即将关闭」
   - 实际超时率降低 40%+
```

```java
// 支付回调幂等处理
public void handlePayCallback(PayCallback callback) {
    Order order = orderMapper.selectByOrderNo(callback.getOrderNo());

    switch (order.getStatus()) {
        case UNPAID:
            // 正常流程：更新为已支付
            orderMapper.updateStatus(order.getId(), OrderStatus.PAID);
            break;

        case CANCELLED_TIMEOUT:
            // 异常流程：已超时取消但支付成功 → 自动退款
            log.warn("订单已取消但收到支付成功回调: orderNo={}", order.getOrderNo());
            refundService.autoRefund(order, "订单超时取消，自动退款");
            break;

        case PAID:
            // 重复回调 → 直接忽略
            break;

        default:
            log.warn("未知状态收到支付回调: orderNo={}, status={}",
                     order.getOrderNo(), order.getStatus());
    }
}
```

#### 坑 2：MQ 消息积压导致延迟取消

```
场景：大促期间，MQ 消息积压 100 万+
     → 原本 30 分钟投递的消息，实际 2 小时后才被消费
     → 库存被占用太久，影响正常销售

解决：
1. 消费者扩容：临时增加消费线程数
2. 死信队列告警：积压超过阈值触发告警
3. 兜底扫表兜底：积压时兜底任务先顶上
4. 消息 TTL：过期太久（如 2 小时）的消息直接丢弃
```

#### 坑 3：定时任务扫表把 DB 扫挂了

```
场景：订单表 2000 万行，扫 status='UNPAID' 全表
     → 慢查询告警 → DB CPU 飙高 → 影响在线业务

解决：
1. 必须走索引：status + created_at 复合索引
2. 限制扫描范围：只扫过去 35~40 分钟的
3. LIMIT 分批：每次最多 500 条
4. 低峰期加频率：凌晨 2-6 点每 30 秒扫一次，其他时段每 5 分钟
```

### 追问链

#### 追问 1：RocketMQ 延迟消息的实现原理是什么？

> "RocketMQ 的延迟消息不是真的延迟投递，而是投递到一个特殊的 Topic（`SCHEDULE_TOPIC_XXXX`），内部维护多个时间层级（1s/5s/10s/30s/1m/.../30m/1h/2h），每个层级对应一个队列。Broker 有一个定时线程不停扫描，到时间就把消息投递到目标 Topic。所以 RocketMQ 延迟消息只支持固定的 18 个级别，不支持任意延迟时间。如果需要任意延迟（比如 7 分钟），可以用开源的 RocketMQ 延迟消息插件或自研时间轮。"

#### 追问 2：如果延迟时间不固定（不同商品超时时间不同），怎么处理？

> "RocketMQ 原生只支持 18 个固定级别。几种处理方式：1. 选最接近的级别，比如 25 分钟就用 delayLevel=16（30 分钟），提前取消不影响业务。2. 开启 Broker 的 `timerWheelEnable`（5.x+），支持任意毫秒级延迟。3. 自研方案：下单时写入 Redis sorted set（score=过期时间戳），独立定时任务每秒扫描到期的 order 投递到普通 MQ。方案 3 灵活性最高但复杂度也最高。"

#### 追问 3：取消订单时释放库存失败怎么办？

> "用本地消息表 + 重试机制：1. 取消订单操作写入本地消息表（和订单更新在同一个事务里）。2. 后台任务扫描消息表，对失败的释放库存操作进行重试（指数退避，最多重试 5 次）。3. 超过 5 次仍未成功的，发告警到运维群，人工介入。核心原则：**释放库存不能因为异常就跳过，宁可重试也不能漏**。"

#### 追问 4：定时任务在多实例部署时怎么保证不重复执行？

> "三种方式：1. Redis 分布式锁（SETNX + 过期时间），最常用。2. 数据库锁（`SELECT ... FOR UPDATE`），简单但性能差。3. 调度框架（XXL-JOB / ElasticJob），自带分片和互斥。推荐用 XXL-JOB，它天然支持路由策略（第一个、最后一个、轮询、随机、一致性哈希），不用自己管分布式锁。"

#### 追问 5：超时前提醒用户（倒计时 5 分钟发短信）怎么实现？

> "下单时同时发两条延迟消息：一条 30 分钟（取消订单），一条 25 分钟（提醒用户）。提醒消息的消费逻辑就是发一条短信。注意：提醒消息发送前也要检查订单状态，如果已经支付了就不发短信。"

### 排查 Checklist

```
□ 延迟消息的 delayLevel 对应的时间对吗？ → 16 = 30min，不要配错
□ 消费者是幂等的吗？ → 先查状态再取消，状态非 UNPAID 直接跳过
□ 有兜底扫表机制吗？ → 必须有，MQ 不可靠时兜底
□ 扫表有走索引吗？ → status + created_at 复合索引
□ 取消和释放库存在同一个事务吗？ → 如果不是，用本地消息表保证最终一致
□ 支付回调有处理「已取消」的情况吗？ → 自动退款
□ 多实例部署有分布式锁吗？ → 兜底任务需要互斥
□ MQ 积压有告警吗？ → 消费延迟超过阈值触发
□ 取消失败有重试吗？ → 指数退避 + 告警
□ 有操作日志吗？ → 谁取消的、什么时间、什么原因
```
