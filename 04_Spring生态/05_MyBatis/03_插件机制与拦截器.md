# 插件机制与拦截器

## 这个问题为什么存在？

> 你想在执行 SQL 之前打印一条日志，或者在查询结果返回后做脱敏处理。直接改 MyBatis 源码？不现实。

MyBatis 提供了 **插件（Interceptor）机制**，允许你在四个关键节点「插一脚」。

**四个可拦截的接口**（称为 MyBatis 的「四大神兽」）：
1. `Executor`（执行器）— 影响所有 SQL 的执行
2. `ParameterHandler`（参数处理器）— 影响参数的设置
3. `ResultSetHandler`（结果集处理器）— 影响结果集的处理
4. `StatementHandler`（语句处理器）— 影响 SQL 预编译和执行

---

## 它是怎么解决问题的？

### 插件原理：责任链 + 动态代理

```
MyBatis 在创建这 4 个接口的实现类时，
不是直接 new，而是先经过插件链包装（代理）

以 Executor 为例：
  → Configuration.newExecutor()
    → executor = (Executor) interceptorChain.pluginAll(executor)
      → 遍历所有 Interceptor
      → 每个 Interceptor 都用 Plugin.wrap() 包装一次
      → 返回层层代理对象（责任链）
```

**Plugin.wrap() 核心逻辑**：
```java
// Plugin.java
public static Object wrap(Object target, Interceptor interceptor) {
    // 1. 获取该 Interceptor 要拦截的方法签名
    Map<Class<?>, Set<Method>> signatureMap = getSignatureMap(interceptor);
    Class<?> type = target.getClass();
    // 2. 找出 target 实现了哪些接口
    Class<?>[] interfaces = getAllInterfaces(type, signatureMap);
    if (interfaces.length > 0) {
        // 3. 创建 JDK 动态代理
        return Proxy.newProxyInstance(
            type.getClassLoader(),
            interfaces,
            new Plugin(target, interceptor, signatureMap)  // ← Plugin 实现 InvocationHandler
        );
    }
    return target;  // 不需要代理，返回原对象
}

// Plugin.invoke() — 代理方法
@Override
public Object invoke(Object proxy, Method method, Object[] args) {
    // 判断该方法是否需要被拦截
    Set<Method> methods = signatureMap.get(method.getDeclaringClass());
    if (methods != null && methods.contains(method)) {
        return interceptor.intercept(new Invocation(target, method, args));
    }
    return method.invoke(target, args);  // 不拦截，直接调用原方法
}
```

### Interceptor 接口

```java
public interface Interceptor {
    // 核心方法：拦截逻辑写在这里
    Object intercept(Invocation invocation) throws Throwable;

    // 包装目标对象（创建代理），默认实现调用 Plugin.wrap()
    default Object plugin(Object target) {
        return Plugin.wrap(target, this);
    }

    // 读取插件配置（<property> 配置）
    default void setProperties(Properties properties) {}
}
```

### 签名注解 @Intercepts

```java
@Intercepts({
    @Signature(
        type = Executor.class,           // 拦截的接口
        method = "query",                // 拦截的方法名
        args = {MappedStatement.class, Object.class, RowBounds.class, ResultHandler.class}
        // ↑ 方法参数类型（用来区分重载方法）
    ),
    @Signature(
        type = Executor.class,
        method = "update",
        args = {MappedStatement.class, Object.class}
    )
})
public class MyPlugin implements Interceptor {
    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        // 前置逻辑
        System.out.println("SQL 即将执行...");
        // 执行原方法
        Object result = invocation.proceed();
        // 后置逻辑
        System.out.println("SQL 执行完成");
        return result;
    }
}
```

## 源码关键路径

### 四大组件的代理时机

```
Configuration 类中的创建方法
  → newExecutor()         → 拦截 Executor
  → newParameterHandler() → 拦截 ParameterHandler
  → newResultSetHandler() → 拦截 ResultSetHandler
  → newStatementHandler() → 拦截 StatementHandler

每个方法内都有：
  target = (Xxx) interceptorChain.pluginAll(target);
```

### 分页插件原理（最经典的插件应用）

```
PageHelper 的核心原理：
  1. 在执行查询前，把分页参数（pageNum, pageSize）放入 ThreadLocal
  2. 拦截 Executor.query() 方法
  3. 在 SQL 执行前，根据数据库方言改写 SQL（加 LIMIT/OFFSET）
     → MySQL：SELECT * FROM t LIMIT ?, ?
     → Oracle：SELECT * FROM (SELECT t.*, ROWNUM rn FROM t WHERE ROWNUM <= ?) WHERE rn > ?
  4. 查询完成后，把 ThreadLocal 中的参数清除
```

## 它和相似方案的本质区别是什么？

| 方案 | 拦截层面 | 灵活性 | 典型应用 |
|------|---------|--------|---------|
| **MyBatis 插件** | JDBC 层（SQL 执行前后） | 高（可改 SQL/参数/结果） | 分页、乐观锁、数据脱敏 |
| **Spring AOP** | Service 层（方法调用前后） | 中（只能改参数/返回值） | 日志、事务、鉴权 |
| **数据库代理（如 ShardingSphere）** | 网络层（SQL 解析+路由） | 高（支持分库分表） | 分库分表、读写分离 |
| **JDBC 驱动层拦截** | 更底层（Statement 执行） | 最高（但难度最大） | 连接池监控、慢 SQL 记录 |

**本质区别**：MyBatis 插件在 **ORM 框架内部**拦截，能拿到 `MappedStatement`（SQL 元数据），可以改写 SQL；Spring AOP 在 ORM 外面，只能拿到方法参数。

## 正确使用方式

### 自定义插件：SQL 执行时间监控

```java
@Intercepts({
    @Signature(type = StatementHandler.class, method = "prepare", args = {Connection.class, Integer.class})
})
public class SqlCostInterceptor implements Interceptor {

    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        long start = System.currentTimeMillis();
        Object result = invocation.proceed();  // 执行原方法
        long cost = System.currentTimeMillis() - start;
        System.out.println("SQL 执行耗时：" + cost + "ms");
        if (cost > 1000) {
            System.out.println("⚠️ 慢 SQL 警告！");
        }
        return result;
    }

    @Override
    public Object plugin(Object target) {
        return Plugin.wrap(target, this);
    }
}
```

### mybatis-config.xml 中注册插件

```xml
<configuration>
    <plugins>
        <plugin interceptor="com.example.MyPlugin">
            <property name="someProperty" value="100"/>
        </plugin>
        <plugin interceptor="com.example.SqlCostInterceptor"/>
    </plugins>
</configuration>
```

### Spring Boot 中注册插件

```java
@Configuration
public class MyBatisConfig {

    @Bean
    public Interceptor sqlCostInterceptor() {
        return new SqlCostInterceptor();
    }

    @Bean
    public SqlSessionFactory sqlSessionFactory(DataSource dataSource) throws Exception {
        SqlSessionFactoryBean factoryBean = new SqlSessionFactoryBean();
        factoryBean.setDataSource(dataSource);
        // 设置插件
        factoryBean.setPlugins(sqlCostInterceptor(), pageInterceptor());
        return factoryBean.getObject();
    }
}
```

## 边界情况和坑

### 坑1：插件执行顺序问题

```
多个插件拦截同一个方法时，形成代理链：
  最外层的代理 → 第二层 → ... → 原对象

执行顺序：
  外层的 intercept() 先执行「前半段」
    → invocation.proceed() 进入下一层
      → ...（层层嵌套）
        → 原方法执行
      → 返回后执行「后半段」
    → 外层 intercept() 返回

配置顺序 = 代理包装顺序（先配置的先被包装，在最外层）
```

### 坑2：ThreadLocal 忘记清理

```
分页插件（PageHelper）经典坑：
  ① ThreadLocal 中设置了 page 参数
  ② 异常导致方法提前返回
  ③ ThreadLocal 未清理
  ④ 线程池复用 → 下一个请求命中了错误的分页参数！

正确做法：在 finally 块中清理 ThreadLocal
```

### 坑3：拦截 too many methods

```
@Signature 的 args 必须精确匹配方法签名
如果方法重载且参数类型不同，需要写多个 @Signature

错误示例：只写一个 Signature，但 Executor.query() 有多个重载版本
→ 只拦截了匹配的那个，其他版本没拦截
```

### 坑4：插件影响性能

```
每个被拦截的对象都会被 JDK 动态代理包装
代理层数 = 插件数（每个插件包一层）
方法调用时，每层的 invoke() 都会被执行 → 性能有损耗

建议：只保留必要的插件，定期 review 插件列表
```

### 坑5：拦截 ResultSetHandler 时 resultMap 已经解析完成

```
如果想在插件中修改查询结果：
  → 拦截 ResultSetHandler.handleResultSets() 太晚（结果已映射）
  → 应该拦截 StatementHandler.prepare() + 改 SQL（加字段）
    或者拦截 ResultSetHandler 对结果做后处理（脱敏等）
```

## 四大拦截点对比

| 拦截接口 | 拦截时机 | 能做什么 | 经典应用 |
|---------|---------|---------|---------|
| `Executor` | SQL 执行入口（query/update） | 改写 SQL 参数、控制是否执行 | 分页、乐观锁版本号自动填充 |
| `StatementHandler` | SQL 预编译（prepare） | 改写 SQL 语句本身 | 分表（表名替换）、加 WHERE 条件 |
| `ParameterHandler` | 参数设置（setParameters） | 改入参值 | 数据加密（入库前加密） |
| `ResultSetHandler` | 结果集处理（handleResultSets） | 改出参值 | 数据脱敏（手机号脱敏显示） |

## 我的理解

MyBatis 插件机制的本质是 **JDK 动态代理 + 责任链模式**。

- `InterceptorChain.pluginAll()` 把目标对象层层包装
- 每个插件都是一个 `InvocationHandler`（通过 `Plugin.wrap()` 创建）
- 方法调用时从外到内层层经过插件的 `intercept()` 方法

**最强大的地方**：能拿到 `MappedStatement`，知道当前执行的是哪条 SQL，甚至能拿到 BoundSql 改写 SQL 语句。

**最危险的地方**：插件在所有 Mapper 方法执行时都会触发，一旦有 bug 影响面极大；ThreadLocal 必须清理。

**实际项目建议**：
- 分页用 PageHelper（成熟方案）✅
- 数据脱敏可以放在 `ResultSetHandler` 插件 ⚠️（注意性能）
- 慢 SQL 监控推荐用 p6spy（比自己写插件靠谱）✅
- 尽量不要自己写改写 SQL 的插件 ❌（容易出 bug，用 MyBatis 动态 SQL 代替）
