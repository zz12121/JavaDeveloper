# Redis高可用架构

## 这个问题为什么存在？

> 单机 Redis 的两个致命问题：1. **容量瓶颈**——内存有上限，单机几十 GB，上百 GB 就很贵了。2. **可用性**——单机挂了就全挂。问题是：**怎么在单机限制下扩展容量、提高可用性？**

Redis 通过**集群方案**解决：数据分片（扩展容量）+ 副本机制（提高可用性）。

## 架构演进路径

```
单机 → 主从复制 → 哨兵（Sentinel）→ Redis Cluster
│         │              │               │
│      读写分离        自动故障转移      数据分片+高可用
│      副本备份         监控+选主         无中心化
```

### 主从复制：基础副本机制

主节点接收所有写请求，从节点异步复制主节点的数据。

```
客户端 → 主节点（写）→ 主节点向从节点同步数据
       ↓
   从节点（读）
```

#### 复制原理（完整同步 + 命令传播）

**第一次同步（完整同步）**：
```
从节点发送 PSYNC ? -1
主节点执行 BGSAVE，生成 RDB 快照
主节点把 RDB 文件发送给从节点
主节点在后台把 BGSAVE 期间的写命令存入 repl_backlog 缓冲区
从节点清空本地数据，加载 RDB 文件
主节点把 repl_backlog 中的命令发送给从节点
→ 同步完成
```

**增量同步（命令传播）**：
```
主节点每执行一条写命令 → 把命令追加到 repl_backlog 缓冲区
从节点每 1s 发送 PSYNC <runId> <offset>
主节点返回从 offset 开始的增量命令
```

#### `repl_backlog_buffer`：复制积压缓冲区

主节点维护一个固定大小的环形缓冲区（默认 1MB），存放最近执行的写命令。从节点断线重连后：

- 断线 < 缓冲区大小 → 增量同步（PSYNC）
- 断线 > 缓冲区大小 → 完整重同步

这个参数影响断线重连的成功率，生产环境建议设大一些。

#### 主从复制的坑

1. **从节点读到的数据可能过期**：主节点删除过期 key 后，发命令给从节点删除。但 Redis 的惰性删除 + 定时删除，可能导致从节点读到已过期的数据。
2. **主从延迟**：异步复制，从节点可能落后主节点几毫秒到几秒。**不能把主从当强一致性用**。

### 哨兵（Sentinel）：自动故障转移

主从复制解决的是备份问题，**哨兵解决的是"谁是我的主"的问题**。

```
┌─────────────┐
│   哨兵节点1   │  ← 监控主节点健康状态
├─────────────┤
│   哨兵节点2   │  ← 监控主节点健康状态
├─────────────┤
│   哨兵节点3   │  ← 监控主节点健康状态
└─────────────┘
        ↓
  哨兵集群自动选主
        ↓
  从节点晋升为主节点
        ↓
  客户端重定向到新主节点
```

#### 故障检测流程

```
1. 主观下线（SDown）：单个哨兵 ping 主节点，超过 down-after-milliseconds 无响应
2. 客观下线（ODown）：多个哨兵都认为主节点主观下线 → 客观下线
3. 哨兵集群选举 Leader：Raft 协议，谁的优先级最高/runid 最小
4. Leader 哨兵执行故障转移：
   a. 从从节点中选出一个新的主节点（优先级 > 复制偏移量 > runid）
   b. 向其他从节点发送 SLAVEOF 命令，指向新主节点
   c. 通知客户端新的主节点地址
```

#### 哨兵的坑

1. **故障转移期间有短暂的不可写窗口**（几秒）
2. **客户端需要支持 Sentinel**：客户端连接哨兵，由哨兵返回当前主节点地址
3. 哨兵本身也要高可用，所以**至少需要 3 个哨兵节点**（防止脑裂）

### Redis Cluster：数据分片 + 高可用

#### 数据分片机制

Redis Cluster 把整个数据空间划分为 **16384 个槽（slot）**，每个 key 通过 CRC16 计算槽编号：

```
slot = CRC16(key) % 16384
```

```
[0-5460]   → 节点A
[5461-10922] → 节点B
[10923-16383] → 节点C
```

集群中每个节点负责一部分槽，并且每个槽有 N 个副本（主从）。

#### 客户端重定向

客户端访问某个节点，如果 key 不在该节点的槽范围内：

```
客户端 → 节点A（请求 key 在节点B的槽）
    ↓
节点A 返回 MOVED <slot> <node_B>
    ↓
客户端直接连接节点B
```

`MOVED` 是**永久重定向**：客户端缓存 slot→节点映射，后续请求直接打目标节点。

`ASK` 是**临时重定向**：槽正在迁移中，源节点告诉客户端去目标节点找，但不更新缓存。

#### 槽迁移过程

```
1. 目标节点准备接收新槽：CLUSTER SETSLOT <slot> IMPORTING <source_node>
2. 源节点开始迁移：CLUSTER SETSLOT <slot> MIGRATING <target_node>
3. 源节点对 key 逐个 MIGRATE 到目标节点
4. 迁移完成后：所有节点 CLUSTER SETSLOT <slot> NODE <target_node>
5. ASK 重定向在此期间生效，MOVED 在所有节点更新后生效
```

迁移过程中，源节点对**已迁移走的 key** 返回 ASK（临时重定向），对**还没迁移的 key** 正常处理。这是**双写**过程，需要业务侧配合处理。

### 槽迁移中的 ASK vs MOVED

| | ASK | MOVED |
|---|---|---|
| 类型 | 临时重定向 | 永久重定向 |
| 原因 | 槽正在迁移 | 槽已完成迁移 |
| 客户端行为 | 去目标节点找，但继续用旧节点查其他 key | 更新本地缓存，后续直接打目标节点 |
| 是否更新客户端路由缓存 | 否 | 是 |

## 它和相似方案的本质区别是什么？

| 方案 | 数据分片 | 自动故障转移 | 客户端路由 | 复杂度 | 适用场景 |
|---|---|---|---|---|---|
| 主从复制 | ❌ | ❌ | 手动 | 低 | 数据备份，读写分离 |
| 哨兵 | ❌ | ✅ | 哨兵返回地址 | 中 | 小规模，高可用主节点 |
| Redis Cluster | ✅（16384槽） | ✅ | 客户端缓存+MOVED | 高 | 大规模，水平扩展 |

### Codis / Twemproxy vs Redis Cluster

Codis 是第三方分片方案（豌豆荚开源），通过代理层做分片，支持平滑迁移。

| | Redis Cluster | Codis |
|---|---|---|
| 架构 | 无中心化，节点直连 | 代理层，客户端连代理 |
| 迁移 | ASK/MOVED，需客户端配合 | 代理透明，业务无感知 |
| 多键操作 | 同一槽内多键可支持 | 所有操作都支持（代理转发） |
| 运维复杂度 | 高（需要智能客户端） | 低（代理屏蔽细节） |

## 正确使用方式

### 哨兵配置（最小高可用）

```bash
# sentinel.conf
sentinel monitor mymaster 127.0.0.1 6379 2  # 2 个哨兵同意则客观下线
sentinel down-after-milliseconds mymaster 5000  # 5 秒无响应则主观下线
sentinel parallel-syncs mymaster 1  # 故障转移时同时同步的从节点数
sentinel failover-timeout mymaster 180000  # 故障转移超时
```

### Redis Cluster 配置

```bash
# 至少 6 个节点（3 主 3 从），才能容忍 1 个节点挂掉
# 每个主节点至少 1 个从节点
# 槽分配：16384 / 3 = 5461 个槽/主节点
```

### 客户端使用注意事项

```java
// Jedis Cluster 客户端
JedisCluster jedis = new JedisCluster(nodes, timeout, maxRedirections);
String value = jedis.get("name");  // 内部处理 MOVED 重定向

// ⚠️ 注意：多键操作必须在同一槽
// 这些 key 如果不在同一槽，操作会失败
jedis.mset(key1, val1, key2, val2);  // ❌ 不同槽会报错
```

跨槽操作：使用 `hash tags`（`{user:1}:name` 和 `{user:1}:age` 在同一槽）。

## 边界情况和坑

### 脑裂问题（Split-Brain）

```
网络分区：A 和 B 分开了
    ↓
哨兵认为 A 挂了，选举 B 为新主
    ↓
旧主 A 还活着，继续接收写请求
    ↓
网络恢复，哨兵让 A 当从节点
    ↓
A 的数据被清空同步到 B → 丢数据！
```

解决方案：`min-slaves-to-write` + `min-slaves-max-lag`：
```bash
min-slaves-to-write 1  # 至少 1 个从节点可写
min-slaves-max-lag 10   # 从节点延迟不超过 10 秒
```
主节点发现从节点都不满足条件，主动拒绝写入，避免脑裂期间的数据丢失。

### Cluster 节点扩缩容

- **扩容**：新节点加入集群 → 分配槽 → 从原节点迁移槽
- **缩容**：槽迁出 → 下线节点（必须先迁走槽，否则数据丢失）

槽迁移期间用 `redis-cli --cluster reshard` 可以在线迁移，ASK 重定向保证业务不中断。

### 从节点选举优先级

故障转移时，Redis Cluster 从节点选举新主节点的优先级：
1. `replica-priority` 配置值（越大优先级越高）
2. 复制偏移量（`info replication` 中的 `master_repl_offset`，越接近主节点越好）
3. `runid` 字典序（越小越好）

## 我的理解

Redis 高可用有三条路：主从复制是基础，哨兵在它的基础上加了自动选主，Cluster 在哨兵的基础上加了数据分片。每一步都在解决上一步没解决的问题。

面试中最高频的追问是**槽迁移的 ASK vs MOVED**，要能解释清楚：ASK 是迁移过程中的临时重定向，MOVED 是迁移完成后的永久重定向，两者对客户端的影响不同。

## 面试话术

**Q：Redis的高可用方案有哪些？分别说说原理**

"Redis的高可用方案有三个演进阶段：

**第一阶段：主从复制**
- 原理：主节点写，从节点异步复制主节点的数据
- 作用：读写分离（从节点承担读请求）+ 数据备份
- 问题：主节点挂了，需要手动切换（改客户端配置）

**第二阶段：哨兵（Sentinel）**
- 原理：哨兵集群监控主节点，主节点挂了自动选举从节点晋升为主节点
- 故障检测：主观下线（单个哨兵认为主节点挂了）→ 客观下线（多数哨兵都认为挂了）→ 选举Leader哨兵 → Leader执行故障转移
- 作用：自动故障转移，客户端通过哨兵获取当前主节点地址
- 问题：不支持数据分片（单机内存上限）

**第三阶段：Redis Cluster**
- 原理：数据分片（16384个槽），每个槽有主从副本
- 客户端路由：MOVED重定向（永久）+ ASK重定向（迁移中临时）
- 作用：数据分片（突破单机内存上限）+ 高可用（每个分片有主从）
- 适合大规模场景（数据量超过单机内存）

我们生产环境用的是Redis Cluster（数据量大，需要分片）。"

**Q：Redis Cluster的槽迁移过程是怎样的？ASK和MOVED有什么区别？**

"槽迁移过程：
1. 目标节点设置`CLUSTER SETSLOT <slot> IMPORTING <source_node>`（准备接收）
2. 源节点设置`CLUSTER SETSLOT <slot> MIGRATING <target_node>`（准备迁出）
3. 源节点逐个把槽内的key迁移到目标节点（`MIGRATE`命令）
4. 迁移完成后，所有节点更新路由表：`CLUSTER SETSLOT <slot> NODE <target_node>`

**ASK vs MOVED**：
- ASK是临时重定向：槽正在迁移中，告诉客户端'去目标节点找这个key，但别更新本地缓存'（因为迁移还没完成）
- MOVED是永久重定向：槽已经迁移完成，告诉客户端'以后这个槽的key都去目标节点找'（客户端更新本地路由缓存）

核心区别：ASK不更新客户端缓存，MOVED更新缓存。"

**Q：Redis哨兵的选举过程是怎样的？**

"哨兵的选举分两轮：

**第一轮：哨兵Leader选举**（用Raft协议）
- 主观下线：单个哨兵ping主节点，超过`down-after-milliseconds`无响应 → 主观下线
- 客观下线：多个哨兵都认为主节点主观下线 → 客观下线
- Leader选举：哨兵互相投票，票数超过半数的成为Leader，负责执行故障转移

**第二轮：从节点选举**（Leader哨兵选主）
- 优先级：`replica-priority`配置（越小优先级越高）
- 复制偏移量：哪个从节点的数据最接近主节点（偏移量最大）
- runid：如果以上都相同，选runid最小的从节点

选出来后，Leader哨兵向被选中的从节点发送`SLAVEOF NO ONE`（晋升为主节点），然后让其他从节点指向新主节点。"

## 本文总结

| 方案 | 数据分片 | 自动故障转移 | 客户端路由 | 适用场景 |
|------|---------|------------|-----------|---------|
| 主从复制 | ❌ | ❌（手动切换） | 手动（写主读从） | 读写分离、数据备份 |
| 哨兵（Sentinel） | ❌ | ✅ | 哨兵返回主节点地址 | 小规模，高可用主节点 |
| Redis Cluster | ✅（16384槽） | ✅ | 客户端缓存+MOVED/ASK重定向 | 大规模，水平扩展 |

**核心要点**：
1. 主从复制是基础，哨兵解决自动故障转移，Cluster解决数据分片
2. ASK是临时重定向（迁移中），MOVED是永久重定向（迁移完成）
3. 哨兵选举分两轮：哨兵Leader选举（Raft）+ 从节点选举（优先级>偏移量>runid）
4. Cluster至少6个节点（3主3从）才能容忍1个节点挂掉
5. 脑裂问题：`min-slaves-to-write` + `min-slaves-max-lag`配置可以防止

**面试关键**：
- 不只是背方案，要理解**演进路径**（为什么需要哨兵？为什么需要Cluster？）
- 槽迁移的ASK vs MOVED是高频追问
- 哨兵的选举过程（Raft协议）也可能被问到
