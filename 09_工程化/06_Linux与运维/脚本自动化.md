# 脚本自动化

> 重复的操作必须自动化。每次手动执行，都是在累积「人为失误」的风险。

---

## 这个问题为什么存在？

Java 开发的典型重复操作：

```
场景 A：每天手动重启应用
→ 写成脚本 + crontab，每天凌晨 3 点自动重启

场景 B：多个服务需要按顺序启动
→ 写成启动脚本，检查前一个服务的端口是否可用，再启动下一个

场景 C：需要定期清理日志
→ 写成脚本 + crontab，每周清理一次 7 天前的日志

场景 D：需要在多台机器上执行相同操作
→ 写成脚本，用 for 循环遍历所有机器，SSH 执行
```

**为什么不用 GUI 工具？**

服务器通常没有 GUI，即使有，GUI 操作也无法「版本控制」「复用」「审计」。

---

## 它是怎么解决问题的？

### Shell 脚本基础结构

```bash
#!/bin/bash
# 脚本名：restart_app.sh
# 用途：重启 Java 应用

set -e          # 任何命令失败立即退出（防止错误累积）
set -u          # 使用未定义变量时报错（防止变量名拼写错误）
set -o pipefail # 管道中任意命令失败，整个管道失败

APP_NAME="myapp"
PID_FILE="/var/run/${APP_NAME}.pid"

# 停止应用
if [ -f "$PID_FILE" ]; then
    PID=$(cat "$PID_FILE")
    kill "$PID"
    sleep 5
    if ps -p "$PID" > /dev/null 2>&1; then
        kill -9 "$PID"   # 强制杀死
    fi
    rm -f "$PID_FILE"
fi

# 启动应用
nohup java -jar "/opt/${APP_NAME}.jar" > "/var/log/${APP_NAME}.log" 2>&1 &
echo $! > "$PID_FILE"

echo "$(date): ${APP_NAME} restarted" >> "/var/log/${APP_NAME}_restart.log"
```

### 定时任务：`crontab`

```bash
# 查看当前用户的 crontab
crontab -l

# 编辑 crontab
crontab -e

# 格式：分 时 日 月 周 命令
#         │  │  │  │  │
#         │  │  │  │  └─ 0-6（0=周日）
#         │  │  │  └──── 1-12
#         │  │  └─────── 1-31
#         │  └────────── 0-23
#         └───────────── 0-59
```

**常用时间表达式**：

```bash
0 3 * * *         # 每天凌晨 3 点
*/10 * * * *       # 每 10 分钟
0 9-18 * * 1-5    # 工作日 9 点到 18 点，每小时一次
0 0 1 * *          # 每月 1 号凌晨
```

**实战：定期清理日志**

```bash
# 每天凌晨 2 点清理 7 天前的日志
0 2 * * * find /data/logs -name "*.log" -mtime +7 -delete
```

### 批量操作：`for` 循环 + SSH

```bash
#!/bin/bash
# 在多台机器上执行相同操作

SERVERS="192.168.1.101 192.168.1.102 192.168.1.103"

for SERVER in $SERVERS; do
    echo "=== $SERVER ==="
    ssh "$SERVER" "systemctl restart myapp && echo 'restart ok' || echo 'restart failed'"
done
```

**免密登录配置**（一次性操作）：

```bash
# 生成密钥对（如果还没有）
ssh-keygen -t rsa -b 4096

# 把公钥复制到目标服务器
ssh-copy-id user@192.168.1.101
```

### 日志轮转：`logrotate`

Java 应用自己不轮转日志（除非用了 logback 的 `SizeAndTimeBasedRollingPolicy`），但系统级的日志轮转用 `logrotate`。

```bash
# /etc/logrotate.d/myapp
/var/log/myapp/*.log {
    daily                 # 每天轮转
    rotate 7              # 保留 7 份
    compress              # 轮转后压缩
    delaycompress         # 延迟压缩（本次不压缩，下次压缩）
    missingok             # 日志文件不存在时不报错
    notifempty           # 空文件不轮转
    create 0644 myapp myapp   # 轮转后创建新文件，设置权限
    postrotate
        systemctl reload myapp   # 轮转后通知应用重新打开日志文件
    endscript
}
```

## 深入原理

### Shell 脚本 vs Python 脚本

| 维度 | Shell 脚本 | Python 脚本 |
|------|-----------|-------------|
| 适合场景 | 系统管理、文件操作、调用系统命令 | 复杂逻辑、数据处理、API 调用 |
| 性能 | 高（直接调用系统命令） | 中等（解释执行） |
| 可读性 | 差（语法诡异） | 好（语法清晰） |
| 跨平台 | 差（依赖 GNU coreutils） | 好（Windows/Mac/Linux 都能跑） |
| 错误处理 | 弱（`set -e` 也不够完善） | 强（异常机制完善） |

**经验法则**：
- 操作文件/进程/系统 → Shell
- 处理 JSON/API/复杂逻辑 → Python

### `cron` vs `systemd timer`

现代 Linux 发行版（CentOS 7+, Ubuntu 16.04+）推荐用 `systemd timer` 替代 `cron`。

| 维度 | cron | systemd timer |
|------|------|--------------|
| 精确度 | 分钟级 | 秒级 |
| 日志集成 | 需要自己配置（mail/to file） | 自动集成 journalctl |
| 依赖管理 | 不支持 | 支持（等网络就绪后再执行） |
| 分布管理 | 每个用户有自己的 crontab | 集中管理（系统级 + 用户级） |

**但目前 `cron` 仍是主流**，因为更简单、更广为人知。

## 正确使用方式

### 1. 脚本开头总是加 `set -euo pipefail`

```bash
#!/bin/bash
set -e       # 命令失败立即退出
set -u       # 使用未定义变量报错
set -o pipefail  # 管道失败也算失败

# 没有这三行，下面这种错误会被静默忽略：
cat not_exist.txt | grep "keyword"   # not_exist.txt 不存在，但脚本继续执行
```

### 2. 用 `logger` 记录脚本执行日志

```bash
#!/bin/bash
# 把脚本的输出记录到系统日志（/var/log/syslog 或 journalctl）
logger -t my_backup "Backup started"

# 执行备份
rsync -avz /data /backup

logger -t my_backup "Backup finished, exit code: $?"
```

**好处**：所有脚本的执行记录集中在系统日志里，方便排查（用 `journalctl -t my_backup` 查看）。

### 3. 用 `flock` 防止脚本重复执行

```
问题场景：定时任务执行时间超过了间隔时间（比如备份脚本需要 2 小时，但 crontab 设置每 1 小时执行一次）
后果：多个备份进程同时运行，打满磁盘 IO
```

```bash
#!/bin/bash
# 用 flock 保证同一时间只有一个实例运行

(
flock -n 9 || exit 1   # 如果锁已经被占用，直接退出

# 以下是脚本主体
rsync -avz /data /backup
) 9>/var/lock/my_backup.lock
```

### 4. 脚本调试技巧

```bash
# 方法 A：bash -x 打印每条执行的命令（最常用）
bash -x restart_app.sh

# 方法 B：在脚本里加 set -x
#!/bin/bash
set -x   # 开启调试模式
...
set +x   # 关闭调试模式

# 方法 C：echo 关键变量
echo "APP_NAME=$APP_NAME"
echo "PID=$PID"
```

## 边界情况和坑

### 1. `crontab` 的环境变量问题

```
问题：手动执行脚本正常，但 crontab 执行失败
原因：crontab 的环境变量很少（没有 PATH 等），命令找不到
```

**解决**：

```bash
# 方法 A：在 crontab 开头设置 PATH
PATH=/usr/local/bin:/usr/bin:/bin
0 3 * * * /opt/scripts/restart_app.sh

# 方法 B：脚本里用绝对路径
# 不要用 nginx，要用 /usr/sbin/nginx
```

### 2. `crontab` 的 `%` 需要转义

```bash
# 错误：% 在 crontab 里有特殊含义（换行符）
0 3 * * * echo "Backup at $(date +%Y-%m-%d)" >> /tmp/backup.log

# 正确：转义 %
0 3 * * * echo "Backup at $(date +\%Y-\%m-\%d)" >> /tmp/backup.log
```

### 3. Shell 脚本里的整数比较

```bash
# 错误：用 > < 比较整数（这是字符串比较）
if [ $COUNT > 10 ]; then ...

# 正确：用 -gt -lt -eq
if [ $COUNT -gt 10 ]; then ...
```

| 运算符 | 含义 |
|--------|------|
| `-gt` | 大于（greater than） |
| `-lt` | 小于（less than） |
| `-eq` | 等于（equal） |
| `-ne` | 不等于（not equal） |
| `-ge` | 大于等于 |
| `-le` | 小于等于 |

### 4. Windows 编辑的脚本在 Linux 上执行报错

```
问题：在 Windows 上用 Notepad++ 编辑的脚本，在 Linux 上执行报错
原因：Windows 换行符是 CRLF（\r\n），Linux 只认 LF（\n）
```

**解决**：

```bash
# 查看文件换行符
file script.sh
# 输出：script.sh: Bourne-Again shell script, ASCII text, with CRLF line terminators

# 转换换行符
dos2unix script.sh
```
