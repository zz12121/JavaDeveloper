# 持久层框架 — 概览

> 持久层框架解决的是「Java 对象」与「数据库」之间的映射问题。本维度覆盖 MyBatis 和 MyBatis-Plus 两大主流框架。

---

## 为什么需要持久层框架？

JDBC 直接使用有以下痛点：
- 大量样板代码（加载驱动、创建连接、处理结果集、关闭资源）
- SQL 散落在代码里，改 SQL 要重新编译
- 结果集映射手动处理，字段多时非常繁琐
- 没有缓存机制、没有事务管理

ORM 框架（如 Hibernate/JPA）过度封装，复杂 SQL 难以优化。

**MyBatis 的定位**：半自动化 ORM —— SQL 由开发者编写（可控），结果映射和参数绑定自动化（省事）。

**MyBatis-Plus 的定位**：MyBatis 的增强工具，在 MyBatis 基础上只做增强不做改变，解决单表 CRUD 重复劳动问题。

## MyBatis vs MyBatis-Plus vs JPA

| 维度 | MyBatis | MyBatis-Plus | JPA / Hibernate |
|------|----------|----------------|---------------|
| SQL 控制 | 完全手写（可控） | 单表自动，复杂 SQL 手写 | 全自动（HQL） |
| 学习曲线 | 低（懂 SQL 即可） | 低（MP 用法简单） | 高（脏检查、持久化上下文） |
| 复杂查询 | XML 手写，灵活 | XML 手写 + Wrapper 条件 | Criteria API |
| 性能 | 高（SQL 可控） | 高（同 MyBatis） | 中（N+1、脏检查开销） |
| 数据库移植 | 差（SQL 可能有方言） | 差（同 MyBatis） | 好（HQL 数据库无关） |
| 适合场景 | 复杂业务 SQL、需要针对性优化 | 快速开发 + 复杂 SQL 并存 | 简单 CRUD、快速开发 |

**结论**：国内项目推荐 MyBatis + MyBatis-Plus 组合。单表 CRUD 用 MP 的 BaseMapper（省 80% 样板代码），复杂联查用原生 MyBatis 的 XML（灵活可控）。

---

*持久层框架的核心是「映射自动化 + SQL 可控」。理解这个平衡，才能用好 MyBatis 系列框架。*
