# 常见场景

> Git 的命令有很多，但 90% 的工作场景只需要 10 个命令。剩下 10% 的场景（误删代码、合并冲突、历史篡改）才是区分「会用 Git」和「懂 Git」的分水岭。

---

## 这个问题为什么存在？

Git 的灵活性是把双刃剑——它不限制你怎么用，但你用错了就得自己收拾烂摊子。

```
典型灾难场景：
A. 提交了敏感信息（密码、密钥）→ 需要彻底从历史中清除
B. 合并冲突不知道怎么解决 → 乱改一通，把别人的代码弄丢了
C. 想找回 3 天前删掉的代码 → 不知道怎么查历史
D. 同事 forceful 了你的提交 → 历史被覆盖
E. rebase 搞砸了 → 不知道怎么恢复
```

理解这些场景的正确处理方式，比背命令重要 100 倍。

---

## 它是怎么解决问题的？

### 场景 1：合并冲突（Merge Conflict）

```
冲突产生的原因：
同一文件的同一区域，两个分支都改了 → Git 无法自动合并
```

**冲突文件的内容格式**：
```
<<<<<<< HEAD
const timeout = 5000;   // 你的修改
=======
const timeout = 3000;   // 对方的修改
>>>>>>> feature/payment
```

**正确处理流程**：
```bash
# 1. 看到冲突提示
CONFLICT (content): Merge conflict in config.js
Automatic merge failed; fix conflicts and then commit the result.

# 2. 打开冲突文件，手动编辑（保留需要的代码，删除标记）
# 编辑后：
const timeout = 5000;   # 或 3000，或两者结合

# 3. 标记为已解决
git add config.js

# 4. 完成合并
git commit   # 不要加 -m，让 Git 用默认的合并提交消息
```

**用工具处理冲突（更高效）**：
```bash
# VS Code：内置三方合并工具，可视化选择「当前更改」或「传入的更改」
# IntelliJ IDEA：右键文件 → Git → Resolve Conflicts
# 命令行工具：
git mergetool   # 配置后自动打开可视化合并工具
```

**避免冲突的策略**：
```
1. 小步提交（每次提交范围小 → 冲突概率低）
2. 频繁拉取远程最新代码（git pull --rebase）
3. 拆分大文件（把常量、配置、逻辑分文件 → 减少同一文件冲突）
4. 沟通！改公共代码前先问问有没有人在改同一文件
```

### 场景 2：撤销提交（Undo Commit）

```
「撤销」有 4 种含义，用不同的命令：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
含义 1：提交消息写错了
命令：git commit --amend
效果：替换最新提交（会改哈希）

含义 2：提交的内容想修改（补充文件 / 去掉文件）
命令：git add <补充的文件>
     git commit --amend
效果：把暂存区的内容合并到最新提交

含义 3：想撤销提交，但保留变更在暂存区
命令：git reset --soft HEAD~1
效果：撤销提交，变更回到暂存区（像 git add 之后）

含义 4：想撤销提交，保留变更在工作区
命令：git reset --mixed HEAD~1   （等同于 git reset HEAD~1）
效果：撤销提交，变更回到工作区（像 git add 之前）

含义 5：想彻底丢弃最新提交的所有变更
命令：git reset --hard HEAD~1
效果：工作区、暂存区、提交全部回退（⚠️ 危险！）
```

### 场景 3：找回删除的代码

```bash
# 方法 1：用 git log 找历史提交
git log --oneline -- feature/login.js
# 输出：
# a1b2c3d 完成登录功能
# f6e5d4c 初始提交

# 查看该提交改了什么
git show a1b2c3d

# 恢复单个文件到某个提交的状态
git checkout a1b2c3d -- feature/login.js

# 方法 2：用 git reflog 找「丢失」的提交
git reflog
# a1b2c3d HEAD@{0}: reset --hard HEAD~1：回退
# f6e5d4c HEAD@{1}: commit：完成了登录功能   ← 这个就是你要的！
git cherry-pick f6e5d4c   # 把该提交应用到当前分支
```

### 场景 4：修改历史提交（Interactive Rebase）

```bash
# 场景：想合并多个「工作中间提交」为一个干净提交
# 例如：提交了 3 次 "WIP"（work in progress），想合并为一个

git rebase -i HEAD~3
# 打开编辑器，显示：
# pick a1b2c3d 添加登录接口
# pick f6e5d4c 修复 NPE
# pick d2c3b1a 补充单元测试

# 修改：
# pick a1b2c3d 添加登录接口
# squash f6e5d4c 修复 NPE          ← 合并到上一个 pick
# squash d2c3b1a 补充单元测试      ← 合并到上一个 pick

# 保存后，Git 会让你重新编辑合并后的提交消息
```

**`rebase -i` 的指令**：
| 指令 | 作用 |
|------|------|
| `pick` | 保留该提交 |
| `reword` | 保留，但修改提交消息 |
| `edit` | 保留，但暂停让你修改内容 |
| `squash` | 合并到上一个提交 |
| `fixup` | 合并到上一个提交，丢弃提交消息 |
| `drop` | 删除该提交 |

### 场景 5：暂存未提交的变更（git stash）

```bash
# 场景：正在开发功能 A，突然需要切分支修紧急 bug
# 但功能 A 还没写完，不想提交半成品

git stash          # 把当前工作区和暂存区的变更「暂存」起来
git checkout main
# 修 bug、提交...
git checkout feature/A
git stash pop      # 恢复暂存的变更，并从 stash 列表删除

# 查看 stash 列表
git stash list
# stash@{0}: WIP on feature/A: a1b2c3d 添加登录接口

# 恢复但不删除
git stash apply stash@{0}
```

## 深入原理

### `git reset` vs `git revert`

| 维度 | `git reset` | `git revert` |
|------|--------------|---------------|
| 历史修改 | 直接删除提交（改写历史） | 创建一个新提交，抵消目标提交的修改 |
| 适用场景 | 本地分支（未推送） | 公共分支（已推送） |
| 安全性 | 危险（会丢失提交） | 安全（不删除历史） |
| 远程同步 | 需要 `push --force` | 正常 `push` |

```bash
# revert 示例：撤销某个提交，但不删除它
git revert a1b2c3d
# 会创建一个新提交，内容是该提交的反向修改
# 历史变成：... → a1b2c3d → 新提交（撤销 a1b2c3d）
```

**黄金规则**：**已经推送到远程的提交，永远用 `revert`，不要用 `reset`。**

### `git merge` vs `git rebase`

| 维度 | `git merge` | `git rebase` |
|------|--------------|---------------|
| 历史形态 | 保留分支，产生合并提交 | 线性历史，不产合并提交 |
| 安全性 | 安全（不改写已有提交） | 危险（改写提交哈希） |
| 适用场景 | 公共分支合并 | 本地分支同步远程最新代码 |
| 冲突处理 | 一次解决 | 每个提交都可能冲突（需要逐步解决） |

## 正确使用方式

### 1. 提交前先拉取最新代码

```bash
# ❌ 错误做法：直接提交，导致远程有更新时 push 被拒绝
git commit -m "fix"
git push   # 失败：远程有更新

# ✅ 正确做法：先拉取，再提交
git pull --rebase origin main   # 用 rebase 方式拉取（保持线性历史）
git push
```

### 2. 写有意义的提交消息

```
❌ 坏例子：
fix
update
WIP
改了点东西

✅ 好例子：
fix: 修复用户登录时 NPE（当用户名为 null 时）
feat: 添加订单导出 Excel 功能
refactor: 将支付回调处理逻辑从 Controller 移到 Service 层
test: 补充 UserService 的单元测试覆盖率到 85%
```

**推荐格式**（Conventional Commits）：
```
<type>(<scope>): <subject>
空行
<body>
空行
<footer>
```

### 3. 用 `.git/hooks/` 做提交前检查

```bash
# pre-commit 钩子示例：提交前跑代码格式化
#!/bin/sh
# .git/hooks/pre-commit
npm run lint   # 如果 lint 失败，提交被中止
```

**更推荐**：用 [pre-commit](https://pre-commit.com/) 框架管理钩子（支持 Python、Node、Docker 等多种语言）。

## 边界情况和坑

### 1. `git reset --hard` 后无法恢复未提交的变更

```
惨痛教训：
修改了 2 小时的代码，还没 git add
手贱执行了 git reset --hard
→ 所有未提交的变更全部丢失，Git 也找不回来
```

**防御习惯**：
- 重要修改至少 `git add` 一次（进入暂存区就被 Git 追踪了）
- 或者用 `git stash` 暂存
- 配置 Git 自动备份：`git config --global alias.backup '!git stash push -m "auto-backup-$(date)"'`

### 2. `git clean -fd` 删除的未追踪文件无法恢复

```bash
git clean -fd   # 删除所有未追踪的文件和目录
# 这些文件从未被 git add 过，Git 没有它们的任何记录 → 永久丢失
```

**防御**：先 `git clean -fdn`（dry run，只显示会删什么）。

### 3. rebase 公共分支会导致他人的历史分叉

```bash
# 你在 feature 分支 rebase 了 main
git checkout feature/A
git rebase main
git push --force-with-lease

# 你的同事基于旧 feature/A 做了新提交
# 现在他的历史和你 push 的不一致 → 合并时出现大量「幽灵冲突」
```

**黄金规则**：**只 rebase 你自己的本地分支，永远不 rebase 已经推送给别人的分支。**

### 4. Windows 下文件名大小写修改不被 Git 识别

```bash
# Windows 默认文件系统不区分大小写
git mv Readme.md README.md   # 报错：大小写冲突
```

**解决**：
```bash
git config core.ignorecase false   # 让 Git 区分大小写
# 或者先改成临时名字
git mv Readme.md temp.md
git mv temp.md README.md
```

### 5. 大文件提交后，仓库永久膨胀

```
问题：
不小心提交了一个 500MB 的文件
然后下一提交删除了它
→ 仓库大小仍然是 500MB+，因为 Git 历史中还有它
```

**修复**（需要改写历史，谨慎！）：
```bash
# 用 BFG Repo-Cleaner（比 git filter-branch 快得多）
bfg --delete-files *.mp4 repo.git
git reflog expire --expire=now --all
git gc --prune=now --aggressive
```
