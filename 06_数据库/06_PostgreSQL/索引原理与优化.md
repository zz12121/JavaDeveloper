# 索引原理与优化

## 这个问题为什么存在？

MySQL 只有 B+ 树索引和 Hash 索引（Memory 引擎），但 PostgreSQL 支持 **6 种索引类型**，针对不同场景选择不同索引。

```
PostgreSQL 索引类型：
  B-tree    （默认，适合 =、>、<、BETWEEN、IN）
  Hash      （适合 =，PG 10+ 支持 WAL 复制）
  GIN       （倒排索引，适合 JSONB、数组、全文搜索）
  GiST      （通用搜索树，适合几何、范围、相似度）
  BRIN      （块范围索引，适合时序数据、大表）
  SP-GIST   （空间分区 GiST，适合非平衡数据结构）
```

**核心问题**：
1. 为什么 PostgreSQL 需要这么多索引类型？各适合什么场景？
2. GIN 索引是怎么支持 JSONB 查询的？
3. BRIN 索引为什么比 B-tree 小 100 倍？
4. 部分索引（Partial Index）是什么？为什么 MySQL 没有？

理解这些，才能真正发挥 PostgreSQL 的索引优势——特别是 **JSONB + GIN** 和 **部分索引**。

---

## 它是怎么解决问题的？

### 一、B-tree 索引（默认）

B-tree 是 PostgreSQL 的**默认索引**（CREATE INDEX 不指定类型就是 B-tree）。

```
B-tree 适合的操作：
  =、>、>=、<、<=、BETWEEN、IN
  不支持：% 模糊匹配（要用 GIN 或三元组索引）

B-tree 索引结构（和 MySQL 类似）：
  [root]
    ├── [internal node]
    │     ├── [leaf] → ctid1, ctid2, ...
    │     └── [leaf] → ctid3, ctid4, ...
    └── [internal node]
          └── [leaf] → ctid5, ...
```

**B-tree 索引的独特性（vs MySQL）**：

| 特性 | PostgreSQL | MySQL（InnoDB） |
|------|------------|----------------|
| NULL 值索引 | ✅ 支持（可以建索引，可以查询） | ✅ 支持（但只能用 `IS NULL`/`IS NOT NULL`） |
| 多列索引 | 最左前缀，但支持 **任意组合**（只要包含最左列） | 最左前缀，严格 |
| 覆盖索引 | Index-Only Scan（需要 Visibility Map 干净） | 覆盖索引（索引包含查询所需所有列） |
| 部分索引 | ✅ 支持（Partial Index） | ❌ 不支持 |

#### 部分索引（Partial Index）——PostgreSQL 独有！

```sql
-- 只给 "状态为活跃的订单" 建索引（其他行不建索引）
CREATE INDEX idx_active_orders ON orders (user_id)
WHERE status = 'active';

-- 查询时，只有 status='active' 的查询能用这个索引
SELECT * FROM orders WHERE user_id = 123 AND status = 'active';  -- ✅ 用索引
SELECT * FROM orders WHERE user_id = 123 AND status = 'closed';  -- ❌ 不用索引
```

**为什么好？**
1. **索引更小**（只存部分行）
2. **维护成本更低**（UPDATE status 不会影响索引，除非从 active → 非 active 或反过来）
3. **统计信息更准**（优化器知道 active 的基数）

**典型场景**：
```sql
-- 只给未读消息建索引（已读消息不需要快速查询）
CREATE INDEX idx_unread_messages ON messages (user_id, created_at)
WHERE is_read = false;
```

### 二、Hash 索引

Hash 索引只支持 `=` 操作（不支持范围查询）。

```sql
CREATE INDEX idx_hash ON users USING hash (email);
```

**PG 10 之前的问题**：Hash 索引不写 WAL → Crash 后需要 REINDEX → 不能用于复制。

**PG 10 之后**：Hash 索引支持 WAL → 可以用于复制，但 **B-tree 仍然更通用**（B-tree 也支持 `=`，且支持范围查询）。

**结论**：除非表非常大且 **只有 `=` 查询**，否则用 B-tree。

### 三、GIN 索引（倒排索引，最强大！）

GIN（Generalized Inverted Index）是 PostgreSQL 的**杀手锏**，适合 **多值类型**（数组、JSONB、全文搜索）。

#### GIN 原理

```
B-tree：key → ctid 列表（一个 key 对应多个行）
GIN：  key → ctid 列表（一个 key 对应多个行，但 "key" 是 "元素"）

例子：articles 表
  id | title | tags (数组)
  1  | ...   | ['postgresql', 'database']
  2  | ...   | ['database', 'performance']

GIN 索引（tags 列）：
  'postgresql' → {ctid(1)}
  'database'   → {ctid(1), ctid(2)}   ← 多个行包含同一个 key
  'performance'→ {ctid(2)}
```

#### JSONB + GIN ——PostgreSQL 的核心优势

```sql
CREATE TABLE products (
    id    SERIAL PRIMARY KEY,
    name  TEXT,
    attrs JSONB                     -- 商品属性（不同商品有不同属性）
);

-- 创建 GIN 索引（可以对 JSONB 的任何 key 查询）
CREATE INDEX idx_products_attrs ON products USING gin (attrs);

-- 查询：包含指定 key 的商品
SELECT * FROM products WHERE attrs ? 'color';  -- 有 "color" 属性

-- 查询：JSONB 包含指定值
SELECT * FROM products WHERE attrs @> '{"color": "red"}';  -- color = red

-- 查询：JSONB 路径匹配
SELECT * FROM products WHERE attrs #>> '{color}' = 'red';
```

**为什么比 MySQL 的 JSON 好？**

| 维度 | PostgreSQL JSONB + GIN | MySQL JSON + 表达式索引 |
|------|------------------------|------------------------|
| 索引 | GIN 索引，支持任意 key 查询 | 只能对 **固定路径** 建索引（`GEN GENERATED ALWAYS AS (json_col->>'$.key')`） |
| 查询灵活性 | 高（可以查询任意 key） | 低（只能查建了索引的路径） |
| 性能 | GIN 索引很快（倒排） | 表达式索引 = B-tree，也很快，但不够灵活 |

#### 全文搜索 + GIN

```sql
-- tsvector 类型（全文搜索向量）
CREATE INDEX idx_articles_content ON articles USING gin (to_tsvector('english', content));

-- 全文搜索查询
SELECT * FROM articles
WHERE to_tsvector('english', content) @@ to_tsquery('english', 'postgresql & performance');
```

### 四、GiST 索引（通用搜索树）

GiST（Generalized Search Tree）是一种 **框架**（不是具体索引结构），可以支持各种数据类型。

**最常用场景：几何类型、范围类型、相似度搜索**

```sql
-- 几何类型（需要 PostGIS 扩展）
CREATE EXTENSION postgis;
CREATE INDEX idx_locations ON locations USING gist (geom);
SELECT * FROM locations WHERE ST_DWithin(geom, 'SRID=4326;POINT(116 39)', 1000);

-- 范围类型（时间范围、数值范围）
CREATE TYPE salary_range AS RANGE (SUBTYPE = int);
CREATE TABLE jobs (id int, salary salary_range);
CREATE INDEX idx_salary ON jobs USING gist (salary);
SELECT * FROM jobs WHERE salary @> 5000;  -- 薪水范围包含 5000
```

### 五、BRIN 索引（块范围索引，超省空间）

BRIN（Block Range Index）适合 **顺序插入的大表**（如时序数据、日志）。

```
原理：
  把表按物理顺序分成一个个"块范围"（默认 128 个页面 = 1MB）
  对每个块范围，记录：最小值、最大值
  查询时，用最小值/最大值过滤掉不可能包含目标数据的块范围

优势：
  - 索引极小（只存每个块范围的最小值/最大值）
  - 适合时序数据（时间顺序插入，BRIN 效果最好）
```

```sql
-- 时序数据表（按时间顺序插入）
CREATE TABLE sensor_data (
    id    BIGSERIAL PRIMARY KEY,
    ts    TIMESTAMPTZ DEFAULT now(),
    value FLOAT
);

-- BRIN 索引（比 B-tree 小 100 倍）
CREATE INDEX idx_sensor_ts ON sensor_data USING brin (ts);

-- B-tree 索引大小：~ 500MB（1 亿行）
-- BRIN 索引大小：~ 5MB（1 亿行）
```

**限制**：只适合 **物理顺序和索引列顺序高度相关** 的表（如时序数据按时间顺序插入）。

### 六、表达式索引（Expression Index）

可以对**函数/表达式的结果**建索引（MySQL 也叫"函数索引"，但 PG 更早支持）。

```sql
-- 对 lower(email) 建索引（实现大小写不敏感查询）
CREATE INDEX idx_users_email_lower ON users (lower(email));

-- 查询（必须用相同的表达式，才能用索引）
SELECT * FROM users WHERE lower(email) = 'alice@example.com';  -- ✅ 用索引
SELECT * FROM users WHERE email = 'alice@example.com';        -- ❌ 不用索引（除非有普通索引）
```

### 七、覆盖索引与 Index-Only Scan

```sql
-- 建一个覆盖索引（包含所有查询需要的列）
CREATE INDEX idx_cover ON orders (user_id, created_at, status, total_amount);

-- Index-Only Scan（不需要回堆，直接从索引返回数据）
SELECT user_id, created_at, status, total_amount
FROM orders
WHERE user_id = 123
ORDER BY created_at DESC
LIMIT 10;
```

**为什么需要 Visibility Map（VM）？**

```
Index-Only Scan 需要判断：
  "索引指向的堆表行，是不是对所有事务都可见？"
  → 如果可见，不需要回堆（快）
  → 如果不可见，需要回堆检查 MVCC 可见性（慢）

Visibility Map：
  标记哪些页面"所有元组都对所有人可见"
  VACUUM 会更新 VM
  → VACUUM 越及时，Index-Only Scan 越快
```

## 它和相似方案的本质区别是什么？

### PostgreSQL 索引 vs MySQL 索引

| 维度 | PostgreSQL | MySQL（InnoDB） |
|------|------------|----------------|
| 索引类型 | 6 种（B-tree、Hash、GIN、GiST、BRIN、SP-GIST） | 1 种（B-tree，Memory 引擎有 Hash） |
| JSON 索引 | GIN（支持任意 key） | 表达式索引（只能固定路径） |
| 部分索引 | ✅ 支持 | ❌ 不支持 |
| 表达式索引 | ✅ 支持 | ✅ 支持（MySQL 8.0+） |
| 覆盖索引 | Index-Only Scan（需要 VM 干净） | 覆盖索引（索引包含查询列） |
| 索引大小 | GIN/BRIN 可以很小 | B-tree 都比较大 |

**本质区别**：PostgreSQL 的索引更灵活（部分索引、GIN），适合复杂查询（JSONB、全文搜索、几何）。MySQL 的索引更简单，但覆盖索引不需要 VM 维护。

## 正确使用方式

### 正确用法

**1. JSONB 字段用 GIN 索引**

```sql
CREATE INDEX idx_data_gin ON my_table USING gin (data jsonb);
-- 可以对 data 列的任何 key 做条件查询，都能用上 GIN 索引
```

**为什么正确**：JSONB 的查询条件不固定（不同行有不同 key），B-tree 索引无法支持。GIN 是倒排索引，适合多值类型。

**2. 大表的时间字段用 BRIN 索引（时序数据）**

```sql
CREATE INDEX idx_ts_brin ON sensor_data USING brin (ts);
```

**为什么正确**：时序数据按时间顺序插入，BRIN 索引极小（是 B-tree 的 1/100），维护成本低。

**3. 只对部分行建索引（Partial Index）**

```sql
-- 只给未处理的订单建索引
CREATE INDEX idx_pending_orders ON orders (user_id, created_at)
WHERE status = 'pending';
```

**为什么正确**：索引更小，维护成本更低，优化器统计信息更准。

**4. 定期 VACUUM，保持 Visibility Map 干净（让 Index-Only Scan 更快）**

```sql
VACUUM (ANALYZE, VERBOSE) my_table;
```

### 错误用法及后果

**错误1：对 JSONB 用 B-tree 索引**

```sql
-- ❌ 错误：B-tree 不支持 JSONB 的任意 key 查询
CREATE INDEX idx_data_btree ON my_table (data);
```

**后果**：索引建了，但 `WHERE data @> '{"color":"red"}'` 用不上索引，全表扫描。

**修复**：用 GIN 索引。

**错误2：对低选择性列建 B-tree 索引**

```sql
-- ❌ 错误：性别只有 2 个值，B-tree 索引效果差
CREATE INDEX idx_gender ON users (gender);
```

**后果**：优化器可能选择 **全表扫描**（因为索引选择性太低，回堆成本比全表扫描还高）。

**修复**：删除索引，或者建 **部分索引**（如果查询通常只关心一种性别）。

**错误3：BRIN 索引用在不按顺序插入的表**

```sql
-- ❌ 错误：UUID 主键不是顺序插入的
CREATE INDEX idx_id_brin ON users USING brin (id);
```

**后果**：BRIN 的最小值/最大值过滤效果很差（每个块范围都包含各种 UUID），索引几乎没用，还浪费空间。

**修复**：用 B-tree 索引。

## 边界情况和坑

### 坑1：GIN 索引更新很慢（写入放大）

```
GIN 索引的写入成本比 B-tree 高 10 倍以上！
原因：GIN 是倒排索引，插入一行可能导致多个 key 的 ctid 列表更新

解决：
  1. 用 "fastupdate" 参数（PG 9.1+，默认开启）
     → 先把更新存到 pending list，定期合并到主 GIN 索引
  2. 降低 "gin_pending_list_limit"（控制 pending list 大小）
  3. 如果写入非常频繁，考虑用 "表达式索引 + B-tree" 替代 GIN
```

### 坑2：部分索引的 WHERE 条件必须和查询完全匹配

```sql
-- 建索引
CREATE INDEX idx_active ON users (id) WHERE status = 'active';

-- ✅ 能用索引
SELECT * FROM users WHERE id = 123 AND status = 'active';

-- ❌ 不能用索引（条件不一样）
SELECT * FROM users WHERE id = 123 AND status = 'inactive';
```

**解决**：如果查询条件不固定，不要用部分索引（用普通索引）。

### 坑3：表达式索引必须用相同的表达式才能命中

```sql
-- 建索引
CREATE INDEX idx_lower ON users (lower(email));

-- ✅ 能用索引
SELECT * FROM users WHERE lower(email) = 'alice@example.com';

-- ❌ 不能用索引（表达式不一样）
SELECT * FROM users WHERE email = 'alice@example.com';
```
