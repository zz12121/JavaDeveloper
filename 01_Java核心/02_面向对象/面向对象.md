# 面向对象

> 面向对象不是「三大特性背诵题」，是 Java 解决「如何管理复杂度」这个工程问题的系统性答案。理解设计动机，比背下封装/继承/多态的定义重要一百倍。

---

## 这个问题为什么存在？

面向过程编程（C 语言时代）的问题：程序规模一大，全局变量和函数之间的依赖关系就乱成一锅粥——任何人修改任何函数，都可能影响到完全不相关的地方。

```
面向过程的问题：
- 数据和操作数据的函数分离：谁都可以改 struct 里的数据
- 模块边界模糊：函数和全局变量交织，依赖难以追踪
- 复用靠复制代码，代码一处改，处处都要改
```

面向对象（OOP）解决的是同一个根本问题：**如何给代码划定边界，让大规模软件可以被独立开发、独立修改、独立测试。**

三大特性（封装、继承、多态）是这个解决方案的三个维度，不是独立的概念——它们相互配合，服务于同一个目标。

---

## 封装：边界的核心

### 为什么需要封装？

```java
// 没有封装：银行账户余额可以被任何人随意修改
class Account {
    public double balance;
}

account.balance = -999999;  // 任何代码都能执行，没有任何约束
```

封装的本质：**把「可以改变的内部实现」藏起来，只暴露「稳定的接口」**。

```java
// 有封装：通过方法控制修改行为
class Account {
    private double balance;  // 外部不能直接访问

    public boolean withdraw(double amount) {
        if (amount <= 0 || amount > balance) {
            return false;  // 前置条件检查
        }
        balance -= amount;
        return true;
    }

    public double getBalance() {
        return balance;  // 只读访问
    }
}
```

**封装带来的工程价值**：
- 修改内部实现时，只要接口不变，调用方不需要改
- 可以在 setter/方法里做参数校验，防止非法状态
- 单元测试的边界更清晰

### 访问控制修饰符

```
private    → 仅本类可见（最严格）
（包私有） → 同包可见（默认，无修饰符）
protected  → 同包 + 子类可见
public     → 所有人可见（最宽松）
```

**设计原则**：**最小暴露原则（Principle of Least Privilege）**——能 private 就 private，必要时才放宽。

### getter/setter 不等于封装

一个常见误区：给所有字段加 getter/setter，以为就是封装了。

```java
// 这不是封装，是伪封装
class Person {
    private String name;
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }  // 跟 public 字段没区别
}
```

真正的封装是：**字段的修改必须通过有业务语义的方法**。如果一个 setter 里什么都不做，直接赋值，和 public 字段几乎等价。

## 继承：代码复用与分类

### 为什么需要继承？

```java
// 没有继承：狗和猫都有「吃」「睡」的行为，需要重复写
class Dog {
    String name;
    void eat() { System.out.println(name + " 在吃东西"); }
    void sleep() { System.out.println(name + " 在睡觉"); }
    void bark() { System.out.println("汪汪！"); }
}

class Cat {
    String name;
    void eat() { System.out.println(name + " 在吃东西"); }   // 完全重复！
    void sleep() { System.out.println(name + " 在睡觉"); }  // 完全重复！
    void meow() { System.out.println("喵喵~"); }
}
```

继承的目标：**提取公共行为到父类，子类只写自己特有的部分**。

```java
class Animal {
    String name;
    void eat() { System.out.println(name + " 在吃东西"); }
    void sleep() { System.out.println(name + " 在睡觉"); }
}

class Dog extends Animal {
    void bark() { System.out.println("汪汪！"); }   // 只写特有的
}

class Cat extends Animal {
    void meow() { System.out.println("喵喵~"); }   // 只写特有的
}
```

### 继承的代价：强耦合

**继承是 Java 中耦合度最高的关系**——子类和父类之间牵一发动全身。

```java
// 父类修改了方法，子类可能悄悄就出 bug 了
class Animal {
    // 假设原来是：void eat() { ... }
    // 现在改成：return 值了
    boolean eat(int calories) { ... }  // 子类的 void eat() 变成重载，不再是重写！
}
```

**继承的正确场景**：is-a 关系（狗 is-a 动物）。如果用继承只是为了复用代码，通常用**组合**（has-a）更好。

```
继承 vs 组合：
继承（is-a）：Dog extends Animal → 狗是动物，合理
组合（has-a）：Car has Engine  → 汽车有引擎，合理
             Car extends Engine → 汽车是引擎？错误！
```

### 方法覆写（Override）

```java
class Animal {
    void makeSound() {
        System.out.println("...");
    }
}

class Dog extends Animal {
    @Override  // 强烈建议加：让编译器帮你确认是覆写而非重载
    void makeSound() {
        System.out.println("汪汪！");
    }
}
```

**覆写规则**：
```
返回值类型：可以是父类返回类型的子类（协变返回类型）
访问修饰符：只能放宽，不能收紧（子类 public 覆写父类 protected ✅，反之 ❌）
异常声明：只能抛出父类方法声明的检查型异常的子类
```

### super 关键字

```java
class Animal {
    void eat() {
        System.out.println("动物吃东西");
    }
}

class Dog extends Animal {
    @Override
    void eat() {
        super.eat();                        // 调用父类实现（扩展而非替换）
        System.out.println("狗更喜欢骨头");
    }
}
```

## 多态：灵活性的来源

### 为什么需要多态？

多态让你写出「对接口编程」的代码，而不是「对实现编程」。

```java
// 没有多态：每加一种动物，就要改这个方法
void makeAllSounds(Dog dog, Cat cat, Bird bird) {
    dog.bark();
    cat.meow();
    bird.tweet();
}

// 有多态：对接口编程，新增动物类型不需要改这个方法
void makeAllSounds(List<Animal> animals) {
    for (Animal a : animals) {
        a.makeSound();  // 运行时自动分派到具体实现
    }
}
```

### 多态的实现原理：虚方法表（vtable）

```
对象在内存中的布局（HotSpot）：

┌────────────────┐
│  对象头 (Mark Word, 8 bytes)   │  → 锁状态、GC 信息
├────────────────┤
│  类型指针 (Klass*, 4/8 bytes)  │  → 指向方法区的 Class 对象
├────────────────┤
│  实例字段数据              │  → name, age 等字段
└────────────────┘

Class 对象（方法区）中有虚方法表（vtable）：
Dog.vtable:
  [0] → Dog.makeSound()
  [1] → Animal.eat()       （继承的，没覆写）
  [2] → Object.toString()  （继承的，没覆写）
  ...
```

调用 `animal.makeSound()` 时：
1. 从对象头的类型指针找到 Class 对象
2. 在 vtable 中查找 `makeSound` 的槽位（每个类相同槽位固定）
3. 执行对应的方法

这就是为什么多态调用比直接调用稍慢——多了一次 vtable 查找（但 JIT 会内联热点调用，消除这个开销）。

### 编译时多态 vs 运行时多态

```java
class Animal {
    void eat() { System.out.println("动物吃东西"); }
    static void staticMethod() { System.out.println("Animal static"); }
}

class Dog extends Animal {
    @Override
    void eat() { System.out.println("狗吃骨头"); }
    static void staticMethod() { System.out.println("Dog static"); }
}

Animal a = new Dog();
a.eat();           // 运行时多态 → 调用 Dog.eat()（看实际类型）
a.staticMethod();  // 编译时确定 → 调用 Animal.staticMethod()（看声明类型）
```

**静态方法、final 方法、private 方法不参与多态**——编译器在编译期就能确定调用目标（invokestatic / invokespecial），不需要 vtable 查找。

### 向上转型与向下转型

```java
// 向上转型（upcasting）：安全，隐式
Animal a = new Dog();   // Dog → Animal，隐式转换，安全

// 向下转型（downcasting）：有风险，显式
Dog d = (Dog) a;        // Animal → Dog，需要显式转换，可能 ClassCastException

// 先用 instanceof 检查再转型（Java 14+ 可以直接用模式匹配）
if (a instanceof Dog dog) {    // Java 16+ 模式匹配
    dog.bark();                 // 直接用，无需再转
}
```

## 接口 vs 抽象类

这是高频面试题，但更重要的是理解**为什么有两种机制**。

### 设计目标不同

```
抽象类：is-a 关系，模板方法模式
  → 有一部分实现是公共的，子类共享；另一部分是抽象的，子类必须覆写
  → 单继承，一个类只能有一个父类

接口：can-do 关系（能力契约）
  → 完全描述「能做什么」，不关心「怎么实现」
  → 多实现，一个类可以实现多个接口
```

```java
// 抽象类：模板方法
abstract class DataProcessor {
    // 模板方法：定义算法骨架
    final void process() {
        readData();       // 固定步骤
        transformData();  // 抽象，子类实现
        writeData();      // 固定步骤
    }

    abstract void transformData();  // 子类必须实现

    void readData() { /* 通用实现 */ }
    void writeData() { /* 通用实现 */ }
}

// 接口：能力契约
interface Serializable { }         // 可序列化
interface Comparable<T> { ... }    // 可比较
interface Runnable { void run(); } // 可运行
```

### Java 8+ 接口的演进

Java 8 引入 `default` 方法和 `static` 方法，缩小了接口和抽象类的差距：

```java
interface Collection<E> {
    // 抽象方法：必须实现
    boolean add(E element);
    Iterator<E> iterator();

    // default 方法：可选覆写（主要用于接口演进，不破坏现有实现类）
    default boolean isEmpty() {
        return size() == 0;
    }

    // static 方法：接口的工厂方法或工具方法
    static <E> List<E> of(E... elements) { ... }
}
```

**为什么引入 default 方法？**

Java 8 给 `Collection` 加了 `stream()` 方法，但 `Collection` 有成千上万个实现类。如果直接加抽象方法，所有实现类都要改——这是 API 演进的兼容性难题。`default` 方法提供了一个有默认实现的扩展点，现有实现类不用动。

### 选择指南

```
选抽象类的场景：
- 多个子类有公共的状态（字段）和实现
- 需要通过模板方法控制算法骨架
- 子类之间存在 is-a 关系（概念上是同一类事物）

选接口的场景：
- 描述「能力」而非「是什么」（Runnable、Serializable、Comparable）
- 需要多实现（同时实现多个能力）
- 没有状态，只有行为约定
- 解耦：不想限定实现者必须继承某个类
```

## Object 类：所有类的根

Java 中所有类都隐式继承 `Object`，其中最重要的三个方法：

### equals() 和 hashCode() 的契约

```java
// 契约：如果 a.equals(b) 为 true，则 a.hashCode() 必须等于 b.hashCode()
// 反之不成立：hashCode 相同的两个对象不一定 equals

class Point {
    int x, y;

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Point)) return false;
        Point p = (Point) o;
        return x == p.x && y == p.y;
    }

    @Override
    public int hashCode() {
        return Objects.hash(x, y);  // 必须与 equals 一致
    }
}
```

**只覆写 equals 不覆写 hashCode 的后果**：

```java
Point p1 = new Point(1, 2);
Point p2 = new Point(1, 2);
p1.equals(p2);  // true（我们覆写了 equals）

Set<Point> set = new HashSet<>();
set.add(p1);
set.contains(p2);  // false！因为 hashCode 不同，p2 落在不同桶
// HashMap、HashSet 先比 hashCode，再比 equals
```

### toString()

```java
@Override
public String toString() {
    return "Point{x=" + x + ", y=" + y + "}";
}
// 不覆写则默认：ClassName@hashCode（如 Point@1b6d3586），毫无意义
```

### clone()

`clone()` 是浅拷贝，容易踩坑：

```java
class MyList implements Cloneable {
    int[] data;

    @Override
    public MyList clone() {
        try {
            MyList copy = (MyList) super.clone();
            copy.data = data.clone();  // 必须手动深拷贝引用类型字段！
            return copy;
        } catch (CloneNotSupportedException e) {
            throw new AssertionError();
        }
    }
}
```

**实际开发很少用 clone()**，推荐用拷贝构造器或工厂方法代替（Effective Java 建议）。

## 正确使用方式

### 组合优于继承

```java
// 不好：通过继承复用 ArrayList 的行为
class CountingList<E> extends ArrayList<E> {
    int addCount = 0;

    @Override
    public boolean add(E e) {
        addCount++;
        return super.add(e);
    }

    @Override
    public boolean addAll(Collection<? extends E> c) {
        addCount += c.size();
        return super.addAll(c);   // ❌ addAll 内部会调用 add()，addCount 被双计了
    }
}

// 好：通过组合持有 ArrayList，只暴露需要的行为
class CountingList<E> {
    private final List<E> list = new ArrayList<>();
    int addCount = 0;

    public boolean add(E e) {
        addCount++;
        return list.add(e);  // ✅ 精确控制，不受父类内部实现影响
    }
}
```

**为什么 extends ArrayList 出问题**？子类依赖父类的内部实现细节（addAll 会调用 add），父类一改就 bug 了。这叫**脆弱的基类问题（Fragile Base Class）**。

### 多用接口，少用具体类型

```java
// 不好：声明为具体类型
ArrayList<String> list = new ArrayList<>();

// 好：声明为接口类型（未来可以换成 LinkedList、CopyOnWriteArrayList 而不影响调用方）
List<String> list = new ArrayList<>();
```

### final class 的意义

```java
// String、Integer 等都是 final 的
public final class String { ... }

// 为什么 String 必须 final？
// 如果允许继承，可以写一个子类重写 equals()，破坏字符串的语义，
// 导致 HashMap 中的 String key 出现难以追踪的 bug
```

## 边界情况和坑

### 1. 构造函数中调用可覆写方法

```java
class Parent {
    Parent() {
        doSomething();  // ❌ 危险！如果被子类覆写，子类字段还没初始化
    }
    void doSomething() { }
}

class Child extends Parent {
    int value = 10;

    @Override
    void doSomething() {
        System.out.println(value);  // 打印 0，不是 10！
        // 原因：super() 在子类字段初始化之前执行
    }
}
```

### 2. equals() 对称性陷阱

```java
// 如果父类和子类都覆写了 equals，可能破坏对称性
class Point {
    int x, y;
    @Override
    public boolean equals(Object o) {
        if (!(o instanceof Point)) return false;
        Point p = (Point) o;
        return x == p.x && y == p.y;
    }
}

class ColorPoint extends Point {
    Color color;
    @Override
    public boolean equals(Object o) {
        if (!(o instanceof ColorPoint)) return false;  // 比 Point 更严格
        return super.equals(o) && color.equals(((ColorPoint)o).color);
    }
}

Point p = new Point(1, 2);
ColorPoint cp = new ColorPoint(1, 2, RED);

p.equals(cp);   // true（Point.equals 只比 x, y）
cp.equals(p);   // false（ColorPoint.equals 要求 instanceof ColorPoint）
// 对称性破坏了！
```

解决方案：**继承层次中有自定义状态时，放弃 equals 继承，用组合代替**（Effective Java 第 10 条）。

### 3. 接口默认方法冲突

```java
interface A {
    default void hello() { System.out.println("A"); }
}
interface B {
    default void hello() { System.out.println("B"); }
}

class C implements A, B {
    // ❌ 编译错误：hello() 的默认实现冲突，必须覆写
    @Override
    public void hello() {
        A.super.hello();  // 显式指定用 A 的实现
    }
}
```

### 4. 静态绑定的误解

```java
Animal a = new Dog();
a.eat();          // 运行时多态，调用 Dog.eat()
((Dog) a).eat();  // 等价，也是 Dog.eat()
```

但字段访问是静态绑定的：

```java
class Animal { String name = "Animal"; }
class Dog extends Animal { String name = "Dog"; }  // 字段隐藏，不是覆写

Animal a = new Dog();
a.name;  // "Animal"（编译时类型决定）
((Dog) a).name;  // "Dog"（显式转型后）
```

## 我的理解

OOP 的三个特性本质上是一个整体：

- **封装**：划定边界，把「变化的部分」锁在对象内部，对外暴露稳定接口
- **继承**：建立 is-a 层级，提取公共行为。但有强耦合代价，能用组合就用组合
- **多态**：对接口编程，运行时动态分派，让代码可以在不修改的情况下扩展（开闭原则）

面试时聊 OOP，最高段位不是背定义——是**说清楚什么时候该用继承、什么时候用组合，为什么组合优于继承，以及多态背后的 vtable 机制**。
