## synchronized

**关联知识文档**：[[02_并发编程/03_锁机制/锁机制]]

**第一问**：「synchronized 的底层原理是什么？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 它是怎么加锁的？monitorenter / monitorexit 是什么？ | 字节码层面 |
| 第二层 | JDK 6 之后做了哪些优化？ | 锁升级（偏向锁 → 轻量级锁 → 重量级锁）|
| 第三层 | 偏向锁撤销的代价大吗？什么场景下会触发批量重偏向？ | 伪共享 + Epoch 机制 |

**对比型考法**：「synchronized 和 ReentrantLock 有什么区别？什么时候用哪个？」

**实战型考法**：「你在项目里用过 synchronized 吗？为什么用它而不是 Lock？」

**面试话术总结**：

> 回答思路：先说「底层实现」，再说「锁升级过程」，最后说「与 ReentrantLock 的选型」。

- **底层实现**：`synchronized` 编译后产生 `monitorenter`（加锁）和 `monitorexit`（解锁）指令，基于对象头的 Mark Word 和 Monitor 对象实现
- **对象头 Mark Word**：存储锁状态（无锁/偏向/轻量/重量）、GC 年龄、hashCode 等信息，64 位 JVM 占 64 bit
- **锁升级过程**（必须能口述）：
  1. **偏向锁**（默认启用）：第一个线程获取锁，CAS 将线程 ID 写入 Mark Word，之后同一个线程进入无需任何同步操作
  2. **轻量级锁**（偏向锁撤销后）：CAS 将 Mark Word 替换为指向 Lock Record（栈帧中的锁记录）的指针，自旋等待
  3. **重量级锁**（自旋失败）：膨胀为 Monitor 对象，未获取锁的线程阻塞，涉及操作系统级别的互斥量
- **批量重偏向**（JDK 6+）：当同一个类的对象被撤销偏向锁超过 20 次，JVM 将该类剩余的对象批量重偏向到当前线程（Epoch 机制）
- **与 ReentrantLock 选型**：
  - 用 `synchronized`：简单场景、不需要超时/公平/中断、JVM 自动释放锁不怕死锁
  - 用 `ReentrantLock`：需要可中断锁 `lockInterruptibly()`、超时获取 `tryLock()`、公平锁、Condition 精确唤醒

## volatile

**关联知识文档**：[[02_并发编程/02_JMM内存模型/JMM内存模型]]、[[02_并发编程/02_JMM内存模型/volatile]]

**第一问**：「volatile 保证了什么？没保证什么？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 它是怎么实现可见性的？内存屏障是什么？ | StoreLoad / StoreStore / LoadLoad 屏障 |
| 第二层 | happens-before 是什么？它和「时间先后」的区别？ | JMM 语义 / 因果关系 vs 时间顺序 |
| 第三层 | double-check 单例为什么需要 volatile？ | 对象创建的重排序问题 |

**实战型考法**：「什么情况下你会用 volatile？什么情况下必须用 synchronized？」

**面试话术总结**：

> 回答思路：先说「保证什么/不保证什么」，再讲「内存屏障原理」，最后说「典型应用场景」。

- **保证**：可见性（一个线程修改，其他线程立即可见）、有序性（禁止指令重排序）
- **不保证**：原子性（`i++` 仍然不安全）
- **内存屏障**（volatile 写后 / 读前插入）：
  - 写操作后插入 `StoreStore` + `StoreLoad` 屏障
  - 读操作前插入 `LoadLoad` + `LoadStore` 屏障
  - `StoreLoad` 是最重的屏障，同时禁止读写重排序
- **double-check 单例**（经典考点）：
  ```java
  // 缺少 volatile 的问题：指令重排序
  // instance = new Singleton() 实际分三步：
  // 1. 分配内存  2. 调用构造函数  3. 将引用指向内存
  // 没有 volatile，2 和 3 可能重排序，线程 B 拿到未初始化的对象
  private static volatile Singleton instance;
  ```
- **happens-before**：volatile 写 happens-before 后续对这个变量的读；但「时间上先发生」≠「happens-before」，后者是 JMM 的因果关系语义

## AQS

**关联知识文档**：[[02_并发编程/04_AQS/AQS]]

**第一问**：「AQS 是什么？它的工作原理是什么？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | AQS 的核心数据结构是什么？ | CLH 双向队列 + state 变量 |
| 第二层 | 独占模式和共享模式有什么区别？ | ReentrantLock（独占）/ Semaphore（共享）|
| 第三层 | `tryAcquire()` 和 `tryRelease()` 是怎么配合队列工作的？ | CAS 修改 state + 队列出入 |

**面试话术总结**：

- **AQS 核心思想**：用 `volatile int state` 表示同步状态 + FIFO CLH 双向队列管理等待线程
- **两种模式**：
  - **独占模式**（`tryAcquire`/`tryRelease`）：同一时刻只有一个线程获取锁（ReentrantLock、ReentrantReadWriteLock 写锁）
  - **共享模式**（`tryAcquireShared`/`tryReleaseShared`）：允许多个线程同时获取（Semaphore、CountDownLatch、ReentrantReadWriteLock 读锁）
- **加锁流程**：`tryAcquire()` CAS 修改 state 成功 → 获取锁；失败 → 包装为 Node 加入 CLH 队列尾 → `LockSupport.park()` 阻塞；前驱节点释放锁时 `unparkSuccessor()` 唤醒后继
- **基于 AQS 的工具**：ReentrantLock、ReentrantReadWriteLock、Semaphore、CountDownLatch、CyclicBarrier（内部用 ReentrantLock）

## CAS 与 ABA 问题

**关联知识文档**：[[02_并发编程/04_CAS与原子类/CAS与原子类]]

**第一问**：「CAS 是什么？有什么问题？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | CAS 的全称是什么？底层怎么实现的？ | Compare-And-Swap / CPU 指令（x86 的 `cmpxchg`）|
| 第二层 | 什么是 ABA 问题？怎么解决？ | AtomicStampedReference |
| 第三层 | CAS 有什么性能问题？ | 自旋开销 / 长时间自旋浪费 CPU |

**面试话术总结**：

- **CAS 原理**：三个参数（内存值 V、预期值 A、新值 B），如果 V == A 则将 V 更新为 B（原子操作），否则返回失败，通常配合自旋重试
- **底层实现**：利用 CPU 的 `cmpxchg` 指令，配合内存屏障保证原子性（Java 用 `Unsafe.compareAndSwapInt()` 调用）
- **ABA 问题**：线程 1 读到 A，线程 2 将 A→B→A，线程 1 CAS 成功但值已被修改过
- **ABA 解决方案**：`AtomicStampedReference`，每次修改携带一个版本号（stamp），CAS 同时比较值和版本号
- **CAS 性能问题**：高竞争时大量线程自旋浪费 CPU；解决：使用锁（`synchronized` 在竞争激烈时会自动升级为重量级锁让线程阻塞等待，而不是自旋）

## 线程池

**关联知识文档**：[[02_并发编程/07_线程池/线程池]]

**第一问**：「线程池的核心参数有哪些？执行流程是什么？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 为什么不能用 `Executors` 创建线程池？ | FixedThreadPool/SingleThreadExecutor 的无界队列导致 OOM |
| 第二层 | 核心线程数怎么设置？CPU 密集型和 IO 密集型？ | CPU 核数+1 / CPU 核数 * 2（或更精确的公式）|
| 第三层 | 线程池的拒绝策略有哪些？自定义过吗？ | 4 种默认 + 自定义策略与服务降级联动 |

**实战型考法**：「你的项目里线程池参数是怎么设置的？依据是什么？」

**面试话术总结**：

> 回答思路：先说「7 大参数 + 执行流程」，再说「为什么不用 Executors」，最后说「参数设置原则」。

- **7 大核心参数**：
  - `corePoolSize`：核心线程数，常驻不销毁
  - `maximumPoolSize`：最大线程数
  - `keepAliveTime` + `unit`：非核心线程空闲存活时间
  - `workQueue`：任务队列（ArrayBlockingQueue / LinkedBlockingQueue / SynchronousQueue）
  - `threadFactory`：线程工厂（可自定义线程名，便于排查）
  - `handler`：拒绝策略
- **执行流程**：核心线程未满 → 创建核心线程；核心线程满 → 入队列；队列满 → 创建非核心线程（直到 maxPoolSize）；maxPoolSize 满 → 执行拒绝策略
- **为什么不用 Executors**：`newFixedThreadPool` 和 `newSingleThreadExecutor` 用 `LinkedBlockingQueue`（容量 `Integer.MAX_VALUE`），任务堆积导致 OOM；`newCachedThreadPool` 的 `maxPoolSize` 是 `Integer.MAX_VALUE`，高并发创建大量线程也 OOM
- **线程数设置**：
  - CPU 密集型：`N + 1`（N = CPU 核数），+1 是为了防止页缺失导致的空闲
  - IO 密集型：`N * (1 + W/C)`（W = IO 等待时间，C = CPU 计算时间），通常 2N 或 10N
  - 最佳方式：通过压测 + 监控来确定

## ThreadLocal

**关联知识文档**：[[02_并发编程/10_ThreadLocal/ThreadLocal]]

**第一问**：「ThreadLocal 有什么坑？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 为什么说 ThreadLocal 会导致内存泄漏？WeakReference 不是会自动回收吗？ | Entry 持有 value 的强引用 + 线程池复用 |
| 第二层 | 为什么 JDK 要用 WeakReference 作为 key？ | ThreadLocal 对象被回收后，Entry 的 key 自动变 null |
| 第三层 | InheritableThreadLocal 和普通 ThreadLocal 有什么区别？ | 子线程继承父线程的值 |

**实战型考法**：「你在项目里用过 ThreadLocal 吗？用来存什么？怎么防止内存泄漏？」

**面试话术总结**：

- **原理**：每个 `Thread` 对象内部有一个 `ThreadLocalMap`，key 是 `ThreadLocal` 的弱引用，value 是实际存储的对象
- **内存泄漏原因**：
  1. ThreadLocal 对象（key）是弱引用，GC 后 key 变 null，但 value 是强引用
  2. 如果线程被线程池复用（不会销毁），value 一直不被回收
  3. 没有手动 `remove()` → 内存泄漏
- **为什么 key 用弱引用**：如果 key 是强引用，ThreadLocal 外部引用断了之后 key 无法回收，Map 中的 Entry 永远存在。弱引用让 GC 能回收 key
- **正确使用**：每次用完后必须 `threadLocal.remove()`（尤其在线程池场景）
- **InheritableThreadLocal**：子线程创建时自动复制父线程的 ThreadLocal 值；但线程池复用线程时不会重新复制（池化线程不是"新建"的）

## ConcurrentHashMap

**关联知识文档**：[[02_并发编程/05_并发容器/并发容器]]

**第一问**：「ConcurrentHashMap 在 JDK 7 和 JDK 8 有什么区别？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | JDK 7 用的是 Segment 分段锁，为什么 JDK 8 换了？ | 锁粒度更细（桶级别）/ CAS + synchronized |
| 第二层 | JDK 8 的 `size()` 方法为什么不是精确的？ | 多线程并发计数，LongAdder 近似值 |
| 第三层 | JDK 8 在什么情况下会触发红黑树转换？ | 链表长度 ≥ 8 且数组长度 ≥ 64 |

**对比型考法**：「ConcurrentHashMap 和 Hashtable 有什么区别？」

**面试话术总结**：

- **JDK 7**：Segment 数组 + HashEntry 链表，每个 Segment 继承 ReentrantLock，锁粒度是 Segment 级别（默认 16 个 Segment）
- **JDK 8**：Node 数组 + 链表/红黑树，锁粒度是**桶级别**（链表头节点 `synchronized`），put 操作：CAS 尝试插入头节点 → 失败则 `synchronized` 锁住头节点 → 链表尾部追加
- **为什么 JDK 8 换掉 Segment**：
  - 锁粒度更细（16 段 → 任意多个桶）
  - 大部分场景不需要加锁（CAS 乐观）
  - 内存占用更小（不用 Segment 数组）
- **`size()` 实现**：`baseCount` + `CounterCell[]`（LongAdder 思想），多线程并发更新各自的 CounterCell，`size()` 时求和，高并发下是近似值
- **红黑树转换**：链表长度 ≥ 8 **且** 数组长度 ≥ 64 → 红黑树；红黑树节点数 ≤ 6 → 退化为链表（防止频繁转换）

## JMM 与 happens-before

**关联知识文档**：[[02_并发编程/02_JMM内存模型/JMM内存模型]]

**第一问**：「Java 内存模型（JMM）是什么？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | JMM 规定了什么？和 JVM 运行时数据区是一回事吗？ | 线程间可见性/有序性规则，不是内存布局 |
| 第二层 | happens-before 的八条规则能说几条？ | volatile / monitor / 线程启动 / 线程终止 等 |
| 第三层 | as-if-serial 语义是什么？ | 单线程内执行结果必须和顺序执行一致 |

**面试话术总结**：

- **JMM 是什么**：JLS（Java Language Specification）定义的抽象规范，规定多线程下变量的读写顺序和可见性规则。**不是** JVM 运行时数据区（堆/栈/方法区）
- **happens-before 八条规则**（必须能列举 5 条以上）：
  1. 程序顺序规则：同一线程内的操作，前面的操作 happens-before 后面的
  2. volatile 规则：volatile 写 happens-before 后续对该变量的读
  3. 监视器锁规则：unlock happens-before 后续对同一锁的 lock
  4. 线程启动规则：`Thread.start()` happens-before 该线程的所有操作
  5. 线程终止规则：线程所有操作 happens-before `Thread.join()` 返回
  6. 线程中断规则：`interrupt()` happens-before 被中断线程检测到中断
  7. 对象终结规则：构造函数执行完毕 happens-before `finalize()`
  8. 传递性：A happens-before B，B happens-before C → A happens-before C
- **as-if-serial**：不管编译器/CPU 怎么重排序，单线程执行结果必须和顺序执行一样；这是 JMM 给程序员的承诺

## 死锁

**关联知识文档**：[[02_并发编程/12_死锁与活锁/死锁与活锁]]

**第一问**：「什么是死锁？怎么产生？怎么排查？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 死锁的四个必要条件是什么？ | 互斥 / 不可抢占 / 占有并等待 / 循环等待 |
| 第二层 | 怎么预防死锁？怎么避免？ | 固定加锁顺序 / tryLock 超时 / 死锁检测 |
| 第三层 | `jstack` 看到的死锁信息是什么样的？ | `Found one Java-level deadlock` |

**实战型考法**：「你在线上遇到过死锁吗？怎么排查和解决的？」

**面试话术总结**：

- **四个必要条件**（必须能列举）：
  1. **互斥**：资源同时只能被一个线程持有
  2. **不可抢占**：已获取的资源不能被强制收回
  3. **占有并等待**：持有资源同时等待其他资源
  4. **循环等待**：线程间形成环形等待链
- **预防策略**：
  - **破坏循环等待**：所有线程按固定顺序获取锁（如按对象的 hashCode 排序）
  - **破坏占有并等待**：一次性获取所有资源（一次性 `synchronized` 多个对象）
  - **破坏不可抢占**：用 `ReentrantLock.tryLock(timeout)` 超时释放
- **排查方法**：
  - `jstack <pid>`：输出线程堆栈，搜索 `Found one Java-level deadlock`
  - `jconsole` / `VisualVM`：可视化查看线程状态
  - `Arthas thread -b`：线上快速定位死锁线程
  - 日志告警：通过线程监控检测 BLOCKED 状态线程数

## 原子类

**关联知识文档**：[[02_并发编程/04_CAS与原子类/原子类]]

**第一问**：「Java 有哪些原子类？原理是什么？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | `AtomicInteger` 的 `incrementAndGet()` 是怎么保证原子性的？ | Unsafe CAS 自旋 |
| 第二层 | `LongAdder` 比 `AtomicLong` 快在哪里？ | 分散竞争到多个 Cell |
| 第三层 | `AtomicReference` 能实现什么？CAS 更新引用类型 | 无锁并发数据结构 |

**面试话术总结**：

- **原子类分类**：
  - 基本类型：`AtomicInteger`、`AtomicLong`、`AtomicBoolean`
  - 数组类型：`AtomicIntegerArray`、`AtomicReferenceArray`
  - 引用类型：`AtomicReference`、`AtomicStampedReference`（解决 ABA）
  - 累加器：`LongAdder`、`LongAccumulator`
- **AtomicInteger 原理**：基于 `Unsafe.compareAndSwapInt()`，CAS 自旋修改内存值
- **LongAdder 原理**（高频追问）：
  - 核心思想：**分散竞争**。初始只有一个 `base`，竞争激烈时创建多个 `Cell[]`，每个线程更新自己的 Cell
  - `sum()` 返回所有 Cell + base 的和（高并发下是近似值）
  - `AtomicLong` 所有线程 CAS 竞争同一个变量 → 大量自旋失败 → 性能差
  - `LongAdder` 分散到多个 Cell → CAS 冲突大幅减少 → 吞吐量高
  - 代价：`sum()` 不是强一致性的

## 锁机制对比

**关联知识文档**：[[02_并发编程/03_锁机制/锁机制]]

**第一问**：「Java 有哪些锁？怎么选？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 悲观锁和乐观锁的区别？ | synchronized（悲观）/ Atomic（乐观）|
| 第二层 | 读写锁和普通锁的区别？适用于什么场景？ | 读多写少场景，读读不互斥 |
| 第三层 | StampedLock 是什么？和读写锁有什么区别？ | 乐观读模式，避免写饥饿 |

**面试话术总结**：

- **锁分类维度**：
  | 维度 | 悲观锁 | 乐观锁 |
  |------|--------|--------|
  | 代表 | synchronized、ReentrantLock | Atomic 类、StampedLock |
  | 思路 | 先加锁再操作 | 不加锁，CAS 失败再重试 |
  | 适用 | 高竞争 / 临界区大 | 低竞争 / 临界区小 |
- **ReentrantReadWriteLock**：
  - 读锁（共享）：多个线程可同时持有读锁
  - 写锁（独占）：只有一个线程持有，且读锁全释放后才能获取
  - 适用：读多写少（缓存场景）
  - 缺点：写锁饥饿（读操作多时写锁一直拿不到）
- **StampedLock**（JDK 8）：
  - 三种模式：写锁、读锁、**乐观读**（不加锁，只获取 stamp，读完 CAS 校验）
  - 乐观读：`tryOptimisticRead()` → 读数据 → `validate(stamp)` → 成功则返回，失败则升级为悲观读锁
  - 不可重入、不支持 Condition，适合对性能要求极高的读多写少场景

## CompletableFuture 异步编程

**关联知识文档**：[[02_并发编程/09_CompletableFuture/CompletableFuture异步编排]]

**第一问**：「CompletableFuture 用过吗？它解决了什么问题？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | `thenApply`、`thenAccept`、`thenRun` 有什么区别？ | 有返回值 / 无返回值消费 / 无返回值不消费 |
| 第二层 | `supplyAsync` 不传线程池的话用的是什么？ | ForkJoinPool.commonPool() |
| 第三层 | 多个异步任务怎么组合？`allOf` 和 `anyOf` 的区别？ | 全部完成 vs 任意一个完成 |

**实战型考法**：「你在项目里怎么用 CompletableFuture 做并发调用的？」

**面试话术总结**：

- **核心 API**：
  - 创建：`supplyAsync()`（有返回值）/ `runAsync()`（无返回值）
  - 转换：`thenApply()`（有入参有返回）/ `thenAccept()`（有入参无返回）/ `thenRun()`（无入参无返回）
  - 组合：`thenCompose()`（flatMap）/ `thenCombine()`（合并两个结果）
  - 等待：`allOf()`（等待全部完成）/ `anyOf()`（任意一个完成）
- **线程池选择**：`supplyAsync()` 不传线程池 → 默认用 `ForkJoinPool.commonPool()`，线程数 = CPU 核数 - 1（注意：IO 密集任务会阻塞公共池，建议自定义线程池）
- **异常处理**：`exceptionally()`（恢复）/ `handle()`（无论成功失败都处理）/ `whenComplete()`（回调但不处理异常）

## P8 架构型考法汇总

> P8 级别开放设计题，考察技术整合与架构决策能力。

1. **如何设计一个任务调度系统（支持优先级、延迟执行、失败重试）？**
   - 考点：线程池定制、DelayedQueue、重试策略（指数退避）、分片调度
2. **如何设计一个高并发场景下的库存扣减系统？**
   - 考点：无锁设计（CAS）、分段锁、Redis Lua 原子操作、兜底一致性
3. **如何设计一个线程安全的缓存框架（类似 Guava Cache）？**
   - 考点：ConcurrentHashMap、读写锁、淘汰策略（LRU/LFU）、容量控制
4. **如何设计一个支持百万级定时任务的调度平台？**
   - 考点：时间轮（Timing Wheel）、分桶、持久化、集群选举
