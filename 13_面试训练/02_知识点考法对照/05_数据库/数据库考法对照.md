## B+ 树索引

**关联知识文档**：[[06_数据库/02_索引/B+树索引]]

**第一问**：「为什么 MySQL 用 B+ 树而不是 B 树？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | B 树和 B+ 树的核心区别？ | 非叶子节点是否存数据 / 叶子节点是否用链表连接 |
| 第二层 | B+ 树的高度一般是多少？怎么算的？ | 3~4 层（IO 次数 = 树高）|
| 第三层 | 联合索引的最左前缀原则是什么？ | 索引项按定义顺序排列 |

**对比型考法**：「MySQL 索引用 B+ 树，MongoDB 用的是什么？为什么？」

**面试话术总结**：

- **B+ 树 vs B 树**：B+ 树非叶子节点只存索引键（不存数据）→ 每个节点能存更多键 → 树更矮 → IO 次数更少；叶子节点用双向链表连接 → 范围查询高效
- **为什么不用其他数据结构**：
  - 哈希：等值查询 O(1) 但不支持范围查询
  - 红黑树：内存数据结构，不考虑磁盘 IO
  - B 树：范围查询需要中序遍历多个节点
- **最左前缀原则**：联合索引 `(a, b, c)`，查询条件必须从最左列开始才走索引；`(a)`、`(a, b)`、`(a, b, c)` 能走索引，`(b, c)`、`(c)` 不能
- **树高计算**：假设主键 bigint（8B）+ 指针（6B InnoDB），每个非叶子节点约 1170 个索引项（16KB/14B），3 层树能存约 1170^2 ≈ 1300 万行，所以通常 3~4 层就够

## MVCC

**关联知识文档**：[[06_数据库/03_MVCC/MVCC实现原理]]

**第一问**：「MVCC 是什么？它解决了什么问题？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | Read View 是什么？包含哪些字段？ | creator_trx_id / m_ids / min_trx_id / max_trx_id |
| 第二层 | Read Committed 和 Repeatable Read 的 Read View 生成时机？ | 每次语句生成 vs 每次事务生成 |
| 第三层 | undo log 在 MVCC 里扮演什么角色？ | 版本链，通过回滚段追溯历史版本 |

**实战型考法**：「RR 级别下，为什么还是会出现幻读？」

**面试话术总结**：

- **MVCC 本质**：多版本并发控制，每个事务看到的是数据的一个**快照**（Snapshot），读写不互相阻塞
- **InnoDB 实现**：每行数据有两个隐藏列 `trx_id`（最后修改的事务 ID）和 `roll_pointer`（指向 undo log 中的上一版本），形成**版本链**
- **Read View 判断可见性**：
  - `trx_id < min_trx_id`：该版本在 Read View 创建前已提交 → 可见
  - `trx_id >= max_trx_id`：该版本在 Read View 创建后才启动 → 不可见
  - `min_trx_id <= trx_id < max_trx_id`：如果在 `m_ids` 中 → 不可见；不在 → 可见
- **RC vs RR 的区别**：RC 每次 SELECT 都生成新的 Read View → 能看到其他事务已提交的最新数据；RR 只在第一次 SELECT 时生成 → 整个事务看到一致的快照
- **RR 幻读场景**：当前读（`SELECT ... FOR UPDATE` / `INSERT` / `UPDATE`）会加锁，间隙锁防止部分幻读；但第一次快照读后其他事务插入的数据，第二次快照读仍然看不到（不幻读），但当前读会看到（幻读）。所以 RR 只能防止「快照读的幻读」，不能完全防止「当前读的幻读」

## 事务隔离级别

**关联知识文档**：[[06_数据库/04_事务/事务]]

**第一问**：「MySQL 默认的隔离级别是什么？为什么？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 四种隔离级别分别解决什么问题？ | 脏读 / 不可重复读 / 幻读 |
| 第二层 | RR 级别能解决幻读吗？靠什么解决的？ | 快照读（MVCC）+ 当前读（Next-Key Lock）|
| 第三层 | 你在项目里调整过隔离级别吗？为什么调？ | 性能 vs 一致性权衡 |

**面试话术总结**：

- **四种隔离级别**（必须能背出）：
  | 级别 | 脏读 | 不可重复读 | 幻读 |
  |------|------|-----------|------|
  | Read Uncommitted | ❌ | ❌ | ❌ |
  | Read Committed | ✅ | ❌ | ❌ |
  | Repeatable Read | ✅ | ✅ | ⚠️ 部分 |
  | Serializable | ✅ | ✅ | ✅ |
- **MySQL 默认 RR 的原因**：性能和一致性的平衡点。InnoDB 在 RR 下通过 MVCC + Next-Key Lock 实现了很高的并发安全性
- **快照读 vs 当前读**：快照读（普通 `SELECT`）走 MVCC，不加锁；当前读（`SELECT ... FOR UPDATE` / `INSERT` / `UPDATE`）加锁（Record Lock / Gap Lock / Next-Key Lock）

## InnoDB 锁

**关联知识文档**：[[06_数据库/04_事务/事务]]

**第一问**：「InnoDB 有哪些锁？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | Record Lock / Gap Lock / Next-Key Lock 区别？ | 行锁 / 间隙锁 / 行锁+间隙锁 |
| 第二层 | 间隙锁锁的是什么？防什么？ | 索引间隙 / 防止幻读（阻止插入）|
| 第三层 | 唯一索引 + 等值查询，会加间隙锁吗？ | 不会，只加 Record Lock |

**实战型考法**：「你遇到过死锁吗？怎么排查的？」

**面试话术总结**：

- **锁类型**：
  - **Record Lock**（行锁）：锁定索引上的某一条记录
  - **Gap Lock**（间隙锁）：锁定索引记录之间的间隙（开区间），防止其他事务在此间隙插入新记录
  - **Next-Key Lock**（临键锁）：Record Lock + Gap Lock，左开右闭区间（InnoDB 默认加锁方式）
  - **Insert Intention Lock**（插入意向锁）：特殊间隙锁，INSERT 前检查间隙，不互相阻塞
- **加锁规则**（重要）：
  - 唯一索引等值查询：命中 → Record Lock；未命中 → Gap Lock
  - 非唯一索引等值查询：无论是否命中，都加 Next-Key Lock（向左扫描到第一个不满足条件的值）
  - 范围查询：始终加 Next-Key Lock
- **死锁排查**：`SHOW ENGINE INNODB STATUS` → `LATEST DETECTED DEADLOCK` → 看两个事务各自持有的锁和等待的锁

## 索引失效场景

**关联知识文档**：[[06_数据库/02_索引/索引失效场景]]

**第一问**：「什么情况下索引会失效？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 对索引列做了函数操作会失效吗？ | 失效，优化器无法使用索引 |
| 第二层 | `OR` 条件什么时候导致索引失效？ | OR 两侧有一个没索引就全失效 |
| 第三层 | 优化器什么情况下会选择全表扫描而不是走索引？ | 回表成本高 / 统计信息不准确 |

**实战型考法**：「你的慢查询优化过吗？怎么优化的？」

**面试话术总结**：

- **常见索引失效场景**（必须能列举）：
  1. 对索引列使用函数（`WHERE DATE(create_time) = '2024-01-01'` → 改为范围查询）
  2. 隐式类型转换（`WHERE id = '1'`，id 是 int → 字符串转数字不失效，数字转字符串失效）
  3. `LIKE '%abc'`（左模糊 → 索引无法匹配前缀；`'abc%'` 走索引）
  4. `OR` 条件一侧无索引
  5. 违反最左前缀原则
  6. `WHERE <>` 或 `NOT IN`（优化器通常不走索引）
  7. `IS NOT NULL`（在某些版本下可能不走索引）
  8. 索引区分度太低（如性别字段只有 0/1，回表成本高于全表扫描）
- **优化思路**：`EXPLAIN` 分析执行计划 → 看 type（至少 ref 级别）、rows、Extra → 针对性优化

## EXPLAIN 执行计划

**关联知识文档**：[[06_数据库/05_SQL优化/EXPLAIN执行计划]]

**第一问**：「EXPLAIN 的各个字段分别看什么？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | `type` 字段从好到差的排列？ | system > const > eq_ref > ref > range > index > ALL |
| 第二层 | `Extra` 字段里 `Using filesort` 和 `Using temporary` 是什么？ | 文件排序 / 临时表 |
| 第三层 | `Using index` 和 `Using index condition` 的区别？ | 覆盖索引 / 索引下推 |

**面试话术总结**：

- **核心字段**：
  - **type**：访问类型，至少达到 `ref`；`ALL` 是全表扫描（必须优化）
  - **key**：实际使用的索引
  - **rows**：预估扫描行数
  - **Extra**：额外信息
- **Extra 关键值**：
  - `Using index`：覆盖索引（从索引树就能获取全部数据，不需要回表）→ 性能最好
  - `Using index condition`（ICP）：索引下推，WHERE 条件的过滤在存储引擎层完成，减少回表次数（MySQL 5.6+）
  - `Using filesort`：MySQL 需要额外排序（不在索引有序范围内）→ 优化：调整索引或 ORDER BY 列
  - `Using temporary`：使用临时表（通常在 GROUP BY / DISTINCT 时出现）→ 优化：检查索引或调整 SQL
  - `Using where`：在 Server 层过滤行（存储引擎返回数据后进一步过滤）

## 主从复制

**关联知识文档**：[[06_数据库/07_主从复制/主从复制]]

**第一问**：「MySQL 主从复制的原理是什么？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | Binlog 有三种格式？区别？ | STATEMENT / ROW / MIXED |
| 第二层 | 主从延迟怎么解决？ | 半同步复制 / 并行复制 / 写读分离策略 |
| 第三层 | GTID 是什么？解决了什么问题？ | 全局事务 ID，方便故障切换和数据恢复 |

**面试话术总结**：

- **主从复制流程**：Master 写入数据 → Binlog → Dump Thread 发送 → Slave IO Thread 接收写入 Relay Log → Slave SQL Thread 重放
- **Binlog 三种格式**：
  - `STATEMENT`：记录 SQL 语句，日志小但可能主从不一致（`now()` 等函数）
  - `ROW`：记录每行数据变化，日志大但最安全准确（MySQL 5.7+ 默认）
  - `MIXED`：自动选择，一般用 STATEMENT，不安全的用 ROW
- **主从延迟原因**：单线程 SQL 重放（MySQL 5.6 前）；大事务；从库硬件差
- **解决方案**：并行复制（`slave_parallel_workers`）、半同步复制（`rpl_semi_sync_master_wait_for_replica_count`）、读写分离策略（写主读从、关键读走主）

## SQL 优化实战

**关联知识文档**：[[06_数据库/05_SQL优化/SQL优化实战]]

**第一问**：「你做过哪些 SQL 优化？效果怎么样？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 优化大表分页查询 `LIMIT 1000000, 10`？ | 延迟关联 / 游标分页 |
| 第二层 | 如何优化 `COUNT(*)`？ | InnoDB 是索引扫描，不是全表扫描 |
| 第三层 | 批量插入怎么做最快？ | 单条 INSERT vs `INSERT ... VALUES (...),(...),(...)` vs `LOAD DATA` |

**面试话术总结**：

- **分页优化**：`LIMIT offset, size` 在 offset 很大时需要扫描 offset + size 行 → 用延迟关联：
  ```sql
  SELECT t1.* FROM table t1
  INNER JOIN (SELECT id FROM table ORDER BY xxx LIMIT 1000000, 10) t2 ON t1.id = t2.id
  ```
  或用游标分页（记住上一页最后一条的 ID，`WHERE id > last_id LIMIT 10`）
- **`COUNT(*)`**：InnoDB 的 `COUNT(*)` 遍历最小非聚簇索引（二级索引），不是全表扫描；MyISAM 存了行数直接返回。如果频繁 `COUNT`，考虑用 Redis 缓存计数
- **批量插入**：单条 INSERT 多次网络往返 → 多值 INSERT 减少网络开销 → `LOAD DATA INFILE` 最快（跳过 SQL 解析）
