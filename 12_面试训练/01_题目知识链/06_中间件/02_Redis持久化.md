# Redis 持久化方式有哪两种？各自特点是什么？

> ⚠️ **先盲答**：Redis 是内存数据库，宕机数据会丢吗？怎么解决的？

---

## 盲答引导

1. RDB 和 AOF 分别是什么？—— 快照 vs 日志
2. RDB 的 `SAVE` 和 `BGSAVE` 有什么区别？—— 阻塞 vs 后台 fork
3. AOF 的「重写」（Rewrite）是什么？—— 压缩 AOF 文件
4. 生产环境推荐用哪种？—— RDB + AOF 混合（Redis 4.0+）

---

## 知识链提示

```
Redis 持久化
  → [[Redis持久化]]
    → RDB（Redis DataBase）
      → 原理：fork 子进程 → 生成内存快照 → 写入 dump.rdb
      → 触发方式
        → SAVE（阻塞，不用）
        → BGSAVE（后台，fork 子进程）
        → 自动触发：save 900 1（900 秒内至少 1 次写 → 触发 BGSAVE）
      → 优点：文件紧凑，恢复快
      → 缺点：可能丢失最近一次快照后的数据
    → AOF（Append Only File）
      → 原理：每条写命令追加到 appendonly.aof
      → 刷盘策略
        → always：每条都 fsync（慢，安全）
        → everysec（默认）：每秒 fsync（推荐）
        → no：依赖 OS 刷盘（快，不安全）
      → AOF 重写（Rewrite）
        → AOF 文件会越来越大 → fork 子进程 → 重写 AOF（去重冗余命令）
      → 优点：最多丢失 1 秒数据
      → 缺点：文件大，恢复慢
    → 混合持久化（Redis 4.0+）
      → AOF 重写时，把 RDB 格式 + 重写后的 AOF 增量合并
      → 文件头：RDB 格式（恢复快）
      → 文件尾：AOF 增量（数据全）
```

---

## 核心追问

| 追问 | 考点 |
|------|------|
| BGSAVE 时，新的写请求会怎样？ | Copy-On-Write（写时复制），父子进程内存隔离 |
| AOF 文件损坏了怎么办？ | redis-check-aof 工具修复 |
| 主从同步时，RDB 是怎么用的？ | 主库 BGSAVE → 传 RDB 给从库 → 从库加载 |
| 生产环境如何配置？ | 开启 AOF（everysec）+ RDB 定时备份 |

---

## 参考答案要点

**RDB vs AOF**：

| 维度 | RDB | AOF |
|------|-----|-----|
| 持久化方式 | 内存快照 | 命令日志 |
| 恢复速度 | 快 | 慢 |
| 数据丢失 | 可能丢较多 | 最多丢 1 秒 |
| 文件大小 | 小（压缩） | 大（未压缩） |

**混合持久化（推荐）**：AOF 重写时，前半段用 RDB 格式（恢复快），后半段用 AOF 增量（数据全）。
