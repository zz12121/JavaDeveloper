## Git 核心原理

**关联知识文档**：[[09_工程化/01_Git/Git]]

**第一问**：「Git 的底层模型是什么？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | Git 存储的不是差异，而是什么？ | 快照（完整文件内容的 blob 对象）|
| 第二层 | blob、tree、commit 三个对象的关系？ | commit → tree → blob |
| 第三层 | `reset --soft`、`--mixed`、`--hard` 的区别？ | 只移 HEAD / HEAD + 暂存区 / 全部回退 |

**实战型考法**：「你遇到过分支合并冲突吗？怎么解决的？」

**面试话术总结**：

- **Git 对象模型**：四种对象 —— blob（文件内容）、tree（目录结构）、commit（提交信息+tree引用）、tag（标签）
- **三个区域**：工作区（Working Directory）→ 暂存区（Staging Area/Index）→ 本地仓库（Repository）
- **`reset` 三种模式**（必须能说清区别）：
  - `--soft`：只移动 HEAD 指针，暂存区和工作区不变（退回 commit 阶段）
  - `--mixed`（默认）：移动 HEAD + 重置暂存区，工作区不变（退回 add 阶段）
  - `--hard`：移动 HEAD + 重置暂存区 + 重置工作区（全部回退，**不可恢复**）
- **分支策略**：Git Flow（master/develop/feature/release/hotfix，适合发布周期长的项目）；GitHub Flow（只有 main + feature 分支，适合持续部署）；GitLab Flow（带环境分支）
- **`rebase` vs `merge`**：`rebase` 产生线性历史（更整洁），`merge` 保留分支历史（更完整）。公共分支禁用 `rebase`（会重写历史）

## Maven 核心机制

**关联知识文档**：[[09_工程化/02_Maven/Maven]]

**第一问**：「Maven 的生命周期是什么？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 三套生命周期分别是什么？ | clean / default / site |
| 第二层 | 依赖传递和依赖排除怎么做？ | `<exclusions>` / `<optional>true</optional>` |
| 第三层 | 依赖冲突怎么解决？Maven 的调解原则？ | 最短路径优先 / 先声明优先 |

**实战型考法**：「你遇到过 `NoSuchMethodError` 吗？怎么排查的？」

**面试话术总结**：

- **三套生命周期**（互不依赖）：
  - `clean`：`pre-clean` → `clean` → `post-clean`
  - `default`（核心）：`validate` → `compile` → `test` → `package` → `integration-test` → `verify` → `install` → `deploy`
  - `site`：生成项目文档
- **依赖调解**：当多个版本同时存在时，Maven 选「最近」的（路径最短优先）；路径相同时「先声明优先」（pom.xml 中靠前的优先）
- **`NoSuchMethodError`**：编译时用的是 A 版本（有这个方法），运行时加载的是 B 版本（没有这个方法）。排查：`mvn dependency:tree` 看依赖树，找到冲突版本 → `exclusions` 排除

## Docker 核心概念

**关联知识文档**：[[09_工程化/03_Docker/Docker]]

**第一问**：「Docker 和虚拟机的区别是什么？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 容器的本质是什么？ | 共享宿主机内核的进程隔离（Namespace + Cgroup）|
| 第二层 | 镜像的分层存储是什么原理？ | Union File System（联合文件系统）|
| 第三层 | 多阶段构建怎么减小镜像体积？ | 第一阶段编译 → 第二阶段只拷贝产物 |

**面试话术总结**：

- **Docker vs 虚拟机**：
  | 维度 | Docker | 虚拟机 |
  |------|--------|--------|
  | 内核 | 共享宿主机内核 | 每个 VM 独立 Guest OS |
  | 隔离 | Namespace（进程/网络/文件系统）| 硬件级虚拟化 |
  | 启动速度 | 秒级 | 分钟级 |
  | 资源占用 | MB 级 | GB 级 |
- **镜像分层**：每条 Dockerfile 指令生成一层（只读层）；容器运行时在最上层加一层可写层。`COPY`/`ADD`/`RUN` 都会产生新层
- **多阶段构建**：
  ```dockerfile
  FROM maven:3.8 AS build
  COPY . /app && cd /app && mvn package
  FROM openjdk:17-jre-slim
  COPY --from=build /app/target/app.jar /app.jar
  ENTRYPOINT ["java", "-jar", "/app.jar"]
  ```
  第一阶段 600MB+，最终镜像只有 100MB 左右

## CI/CD 流程

**关联知识文档**：[[09_工程化/04_CI_CD/CI_CD]]

**第一问**：「你们项目的 CI/CD 流程是怎样的？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | CI 和 CD 的区别？ | 持续集成 vs 持续部署/交付 |
| 第二层 | 流水线里有哪些关键阶段？ | 构建 → 单元测试 → 代码扫描 → 打包 → 部署 |
| 第三层 | 蓝绿部署和金丝雀部署的区别？ | 流量切换方式 / 风险范围 |

**实战型考法**：「你配置过 Jenkins Pipeline 吗？遇到过什么坑？」

**面试话术总结**：

- **CI（持续集成）**：开发提交代码 → 自动触发构建+测试+静态扫描 → 确保代码能集成
- **CD（持续部署/交付）**：CI 通过后 → 自动部署到测试环境 → 验证通过 → 自动部署到生产环境
- **部署策略**：
  - **蓝绿部署**：准备两套完全相同的环境（蓝/绿），切换流量（DNS/Nginx）。优点：秒级回滚；缺点：需要双倍资源
  - **金丝雀发布**：先将新版本部署给少量用户（5%），观察无异常后逐步扩大（25% → 50% → 100%）。优点：风险可控；缺点：发布慢
  - **滚动更新**：逐步替换旧版本实例（K8s 默认策略）。优点：资源利用高；缺点：回滚慢
