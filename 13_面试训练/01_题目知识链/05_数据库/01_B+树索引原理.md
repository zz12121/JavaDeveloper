## 知识链提示

```
题目：为什么用 B+ 树
  → [[B+树索引原理]]
    → 磁盘 IO 是瓶颈 → 树的高度决定 IO 次数
      → 红黑树太高（百万数据 ≈ 20 层）→ 20 次磁盘 IO，太慢
      → B+ 树矮胖（百万数据 ≈ 3 层）→ 3 次磁盘 IO
    → B+ 树 vs B 树
      → B 树：非叶子节点也存 data → 单次 IO 能加载的索引项更少 → 树更高
      → B+ 树：非叶子节点只存 key → 单次 IO 能加载更多索引项 → 树更矮
    → B+ 树的优势
      → 范围查询快（叶子节点是链表）
      → 全表扫描快（扫链表即可，不用中序遍历整棵树）
      → 查询性能稳定（所有查询都到叶子节点，IO 次数稳定）
  → [[聚簇索引vs非聚簇]]
    → InnoDB：B+ 树的叶子节点直接存整行数据
    → MyISAM：B+ 树的叶子节点存数据文件的地址
```

## 参考答案要点

**红黑树不行**：树太高，磁盘 IO 次数 = 树高，性能差。

**B 树 vs B+ 树**：
- B 树：每个节点都存 data，单次 IO 能加载的索引项少，树高更高
- B+ 树：非叶子节点只存 key，单次 IO 能加载更多索引项，树高更低
- B+ 树叶子节点是链表，范围查询不需要中序遍历

**本质原因**：数据库索引的核心瓶颈是**磁盘 IO 次数**，B+ 树通过「矮胖 + 链表」把 IO 次数压到最低。

> 💡 **一句话记忆**：红黑树太高，B 树太胖，B+ 树矮胖 + 链表，最适合磁盘。
