# 技术选型方法论

## 这个问题为什么存在？

> 技术选型错误，事后重构成本极高。Kafka换RabbitMQ，代码要重写一遍。问题是：**怎么科学地选型，不拍脑袋？**

没有方法论，你会遇到：
- 跟风选型（别人用Kafka，我也用），结果根本不需要高吞吐
- 忽略团队能力（选了Go，但团队全是Java），维护成本极高
- 不Proof of Concept（PoC），上线后性能达不到

---

## 它是怎么解决问题的？

### 四个评估维度

```

维度1：业务需求（核心）
  - 吞吐量要求？（Kafka百万QPS vs RabbitMQ万QPS）
  - 延迟要求？（RabbitMQ毫秒级 vs Kafka秒级）
  - 消息可靠性？（Kafka可能丢消息 vs RabbitMQ确认机制）
  - 事务消息？（RocketMQ支持 vs Kafka不支持）

维度2：团队能力（现实）
  - 团队熟悉Java？（选Java系：Kafka、RocketMQ）
  - 团队规模？（小团队选托管服务，别自建K8s）
  - 学习成本？（Redis简单，ZooKeeper复杂）

维度3：社区活跃度（未来）
  - GitHub Star数？（Kafka 26k vs Pulsar 12k）
  - 发版频率？（活跃项目bug修复快）
  - 文档完善度？（Spring Cloud文档完善 vs 小众框架）

维度4：运维成本（落地）
  - 部署复杂度？（MySQL主从 vs Vitess（分库分表中间件））
  - 监控成熟度？（Prometheus + Grafana生态）
  - 托管服务？（AWS Kafka vs 自建Kafka）
```

**实际案例**：

```


场景：业务需要"事务消息"（支付完成后发通知）
选项：Kafka（不支持）、RocketMQ（支持）
结论：选RocketMQ（业务需求优先）
```

```

场景：创业公司，3个Java开发
选项：自建Kubernetes vs 用云托管（如阿里云EDAS）
结论：选云托管（团队能力有限，运维成本必须低）
```

```markdown
# 技术选型：消息中间件

## 候选方案
- Kafka

- RocketMQ

## 评估维度
| 维度 | Kafka | RabbitMQ | RocketMQ |
|---|---|---|---|
| 吞吐量 | 极高（百万QPS） | 中（万QPS） | 高（十万QPS） |
| 延迟 | 高（批量发送） | 低（实时推送） | 中 |
| 事务消息 | ❌ 不支持 | ❌ 不支持 | ✅ 支持 |
| 社区活跃度 | 极高（26k star） | 高（12k star） | 中（国内阿里系） |
| 运维成本 | 高（ZooKeeper依赖） | 低（单机部署简单） | 中（NameServer简单） |

## 决策
选RocketMQ：
  1. 业务需要事务消息（Kafka、RabbitMQ不支持）
  2. 吞吐量十万QPS，满足需求（不必追Kafka的百万QPS）
  3. 阿里系，文档全中文（团队学习成本低）

## PoC验证
- 部署RocketMQ 2台，压测
- 验证事务消息功能
- 验证监控（Prometheus + Grafana）
```

## 它和相似方案的本质区别是什么？

| | 拍脑袋选型 | 跟风选型 | 方法论选型 |
|---|---|---|---|
| 依据 | 无（感觉） | 别人用啥我用啥 | 四个维度评估 |
| 风险 | 极高（事后重构） | 高（需求不匹配） | 低（评估过） |
| 适用场景 | ❌ 不推荐 | ❌ 不推荐 | ✅ 所有技术选型 |

**本质区别**：拍脑袋和跟风都是**赌博**，方法论是**评估**。

## 正确使用方式

### 使用步骤

```

步骤1：列出候选方案（最多3个）
  → 太多方案，分析不过来

步骤2：按四个维度打分（1-5分）
  → 业务需求（权重50%）
  → 团队能力（权重20%）
  → 社区活跃度（权重15%）
  → 运维成本（权重15%）

步骤3：PoC验证（Top2的方案）
  → 小流量验证，别直接全量

步骤4：选型汇报（架构评审）
  → 讲清楚"为什么选A不选B"
```

### 实际案例：选Kafka还是RabbitMQ？

```

业务需求：
  - 日志流（峰值百万QPS） → Kafka胜出
  - 延迟要求低（秒级可接受） → Kafka满足
  - 消息可能丢（日志不要求强可靠） → Kafka可以

团队能力：
  - 团队熟悉Java（Kafka是Java写的） → Kafka可以
  - 运维能力中（Kafka依赖ZooKeeper） → 勉强可以

决策：选Kafka（日志流场景）
  - 如果业务是"订单消息"（延迟低、事务消息），才选RocketMQ
```

## 边界情况和坑

### 坑1：跟风选型（别人用啥我用啥）

```

场景：别人用Kubernetes，我也用
结果：3人团队，运维K8s忙不过来，业务没时间开发
```

**解决方案**：
1. **按团队能力选型**（3人团队 → 用云托管）
2. **PoC验证**（运维成本到底高不高？）
3. **架构评审**（评委问"为什么选K8s？"，你要答得上来）

### 坑2：忽略业务需求（过度设计）

```

场景：业务峰值QPS才100，选了Kafka（百万QPS）
结果：运维成本高，而且大部分能力用不上
```

**解决方案**：
1. **业务需求优先**（100 QPS → RabbitMQ足够）
2. **不过度设计**（满足需求即可，不追新技术）
3. **未来扩展**（选RabbitMQ，不行再换Kafka，但要PoC验证迁移成本）

### 坑3：不PoC，直接全量

```

场景：选了RocketMQ，没PoC
      上线后，发现事务消息有bug（特定场景失败）
结果：重构，浪费2周
```

**解决方案**：
1. **PoC必须做**（小流量验证，别直接全量）
2. **PoC覆盖核心场景**（事务消息、高并发、故障转移）
3. **PoC结果写入文档**（供后人参考）
