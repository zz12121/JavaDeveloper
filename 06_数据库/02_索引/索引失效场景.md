# 索引失效场景

## 这个问题为什么存在？

> 建了索引，查询却没走索引——性能反而比全表扫描更差。

原因：**索引有使用条件**。违反这些条件，MySQL 优化器会**放弃索引，改走全表扫描**。

理解索引失效的底层原因，才能写出高效 SQL。

---

## 它是怎么解决问题的？

### 场景 1：最左前缀原则

```sql
-- 联合索引 (name, age, email)
CREATE INDEX idx_name_age_email ON users(name, age, email);

-- ✅ 走索引：使用了最左前缀
SELECT * FROM users WHERE name = '张三';
SELECT * FROM users WHERE name = '张三' AND age = 30;
SELECT * FROM users WHERE name = '张三' AND age = 30 AND email = 'a@b.com';

-- ❌ 不走索引：跳过了最左列 name
SELECT * FROM users WHERE age = 30;
SELECT * FROM users WHERE email = 'a@b.com';

-- ⚠️ 索引中断：使用了 name，但跳过了 age
SELECT * FROM users WHERE name = '张三' AND email = 'a@b.com';
-- 只能用到 (name) 部分，email 条件无法用索引（索引在 name 上中断了）
```

**为什么中断就不走后续列？**

> 联合索引按 (name, age, email) 的顺序排列。  
> 知道 `name = '张三'`，可以定位到一批连续的数据块（age 和 email 都在这些块里有序）。  
> 如果不知道 name，只知道 age，MySQL 无法定位，只能全表扫描。

---

### 场景 2：索引列使用函数或运算

```sql
-- ❌ 不走索引：索引列参与了运算
SELECT * FROM orders WHERE YEAR(create_time) = 2024;
SELECT * FROM users WHERE age + 1 > 30;
SELECT * FROM users WHERE id * 2 = 100;  -- id 是主键索引

-- ✅ 走索引：把运算移到右边
SELECT * FROM orders WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01';
SELECT * FROM users WHERE age > 29;
SELECT * FROM users WHERE id = 50;
```

**为什么函数导致索引失效？**

> 索引按「原始值」排序。  
> `YEAR(create_time)` 是计算后的值，MySQL 需要**扫描每一行**，对每行调用 `YEAR()` 函数，再比较 → 无法利用 B+ 树的有序性。

---

### 场景 3：隐式类型转换

```sql
-- phone 是 varchar(20) 字段，建了索引
-- ❌ 不走索引：字符串列用了数字比较（MySQL 会把字符串转成数字）
SELECT * FROM users WHERE phone = 13800138000;
-- MySQL 实际执行：CAST(phone AS signed) = 13800138000
-- 每行都要 CAST → 索引失效

-- ✅ 走索引：使用字符串字面量
SELECT * FROM users WHERE phone = '13800138000';
```

**MySQL 类型转换规则**：

| 比较类型 | 转换方向 | 示例 |
|----------|---------|------|
| 数字 vs 字符串 | 字符串 → 数字 | `phone = 138...` → CAST(phone AS DOUBLE) = 138... |
| 数字 vs 日期 | 日期 → 数字 | `date = '2024-01-01'` → date = 20240101 |

---

### 场景 4：LIKE 左边通配

```sql
-- name 字段建了索引
-- ❌ 不走索引：LIKE 以通配符开头
SELECT * FROM users WHERE name LIKE '%三';       -- 左边 %
SELECT * FROM users WHERE name LIKE '%张%';      -- 左边 %

-- ✅ 走索引：LIKE 以非通配符开头
SELECT * FROM users WHERE name LIKE '张三%';      -- 右 %
SELECT * FROM users WHERE name = '张三丰';        -- 完全匹配
```

**为什么左边通配符不走索引？**

> B+ 树按值排序，`'%三'` 不知道字符串以什么开头，无法定位。  
> `'张三%'` 知道以「张」开头，可以定位到一批连续数据块。

**如果必须匹配任意位置**：用 Elasticsearch 的全文索引，或业务层做「前缀索引」。

---

### 场景 5：OR 条件中包含非索引列

```sql
-- name 有索引，age 无索引
-- ❌ 不走索引（or 中有无索引列）
SELECT * FROM users WHERE name = '张三' OR age = 30;
-- 优化器认为：既然 age 要全表扫，不如整体全表扫

-- ✅ 走索引（全用索引列）
SELECT * FROM users WHERE name = '张三' OR email = 'a@b.com';  -- email 也有索引

-- ✅ 拆成 UNION（强制走索引）
SELECT * FROM users WHERE name = '张三'
UNION
SELECT * FROM users WHERE age = 30;
```

**例外**：InnoDB 的统计信息准确时，优化器可能用**索引合并**（Index Merge）。

---

### 场景 6：WHERE 子句中对索引列使用了 NOT

```sql
-- status 有索引
-- ❌ 通常不走索引（取决于优化器判断）
SELECT * FROM orders WHERE status != 'completed';
SELECT * FROM orders WHERE status NOT IN ('completed', 'cancelled');

-- ✅ 可以走索引
SELECT * FROM orders WHERE status IS NOT NULL;  -- 特殊：可以走索引
```

**为什么 NOT 通常不走索引？**

> B+ 树从根到叶是**有序的**，`!=` 条件不知道从哪开始找。  
> 必须扫描到叶子节点，把所有不等于目标值的记录都找出来。  
> 当 `!=` 排除了大部分数据时（结果集小），优化器可能仍走索引。

---

### 场景 7：优化器认为全表扫描更快

```sql
-- orders 表有 100 万行，status 索引有 100 个不同值
-- 查询 status = 'pending'（pending 状态有 80 万行）

SELECT * FROM orders WHERE status = 'pending';  -- 结果集占 80%
-- 80% 的数据都要返回 → 走索引 → 回表 80 万次 → 比全表扫描还慢
-- 优化器选择：全表扫描（顺序读 100 万行）更快

-- 优化器选择走索引的条件：结果集占全表比例足够小
```

**MySQL 优化器的决策依据**：

```sql
-- 查看索引基数（每种值大概有多少行）
SHOW INDEX FROM orders;
-- Cardinality: 反映索引列不同值的数量（估算值，不精确）

-- 强制使用索引（不推荐，仅用于测试）
SELECT * FROM orders USE INDEX (idx_status) WHERE status = 'pending';
```

---

### 场景 8：复合索引列顺序错误

```sql
-- 复合索引 (a, b, c)
-- 用户查询只用到了 a 和 c，没有 b

-- ❌ 索引顺序不当：查 a 和 c，但联合索引是 (a,b,c)
SELECT * FROM t WHERE a = 1 AND c = 2;
-- 只用到 a，c 条件无法用索引

-- ✅ 调整：建 (a, c) 或 (a, c, b)（根据查询频率调整）
CREATE INDEX idx_a_c ON t(a, c);
```

**调整索引顺序的原则**：把**高频条件**放在前面，**范围查询**放在后面（范围查询会导致索引中断）。

---

### 场景 9：JOIN 时字符集/排序规则不一致

```sql
-- users.name (utf8mb4)
-- logs.username (utf8_bin)

-- ❌ 不走索引：字符集或排序规则不一致
SELECT * FROM users u
JOIN logs l ON u.name = l.username;

-- ✅ 统一字符集和排序规则
ALTER TABLE logs MODIFY username VARCHAR(50) CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

**为什么字符集不一致不走索引？**

> MySQL 比较两个不同字符集的列时，需要逐行转换，索引无法加速。

---

### 快速排查：索引失效的检查步骤

### 1. 用 EXPLAIN 查看执行计划

```sql
EXPLAIN SELECT * FROM users WHERE name = '张三' AND age > 25;

-- 关键字段：
-- type:      ALL=全表扫描, ref=索引查找, range=范围扫描, const=常量查找
-- key:       实际使用的索引名（NULL=没用到索引）
-- rows:      预计扫描的行数（越大越可疑）
-- Extra:     Using index=覆盖索引, Using filesort=文件排序, Using temporary=临时表
```

### 2. 检查 Extra 字段的常见警告

| Extra 值 | 含义 | 性能影响 |
|----------|------|---------|
| `Using filesort` | 文件排序（内存或磁盘） | ⚠️ 大数据量时慢 |
| `Using temporary` | 临时表 | ⚠️ 消耗内存/磁盘 |
| `Using index condition` | 索引下推（ICP） | ✅ 优化 |
| `Using index` | 覆盖索引，无需回表 | ✅ 最优 |
| `Using where` | 需要在 Server 层过滤 | ⚠️ 正常，量大时慢 |

### 3. 检查是否有隐式类型转换

```sql
EXPLAIN SELECT * FROM users WHERE phone = 13800138000;
-- 看 Warning，可能有 "type conversion"
```

## 深入原理

### 优化器 Cost Model：为什么放弃索引？

MySQL 优化器在决定是否走索引时，会计算两条路径的**代价**：

```
路径1：走索引
  代价 = 索引树遍历代价 + 回表代价（每次回表≈一次随机 I/O）
  如果结果集很大 → 回表次数多 → 代价高

路径2：全表扫描
  代价 = 数据页顺序读代价（顺序 I/O 比随机 I/O 快得多）
  InnoDB 的数据页有预读机制，顺序读非常高效
```

**关键结论**：全表扫描用顺序 I/O，走索引 + 回表用随机 I/O。当结果集超过表数据的约 20-30% 时，优化器会认为全表扫描更快。

```sql
-- 查看优化器的代价计算（MySQL 8.0+）
EXPLAIN FORMAT=JSON SELECT * FROM orders WHERE status = 'pending';
-- 关注 "cost_info" 字段
```

### 索引树遍历的底层约束

B+ 树索引的有序性决定了它能高效处理以下查询模式：

```
能走索引：等值查询（=、IN）、范围查询（>、<、BETWEEN）、前缀匹配（LIKE 'abc%'）
不能走索引：函数变换（YEAR(col)）、隐式转换（varchar→int）、左边通配（LIKE '%abc'）
```

原因：B+ 树按「原始值的字节序」排列。对索引列做任何变换，都无法利用树的有序性进行二分查找。

### 为什么联合索引要遵守最左前缀原则？

联合索引 (a, b, c) 在 B+ 树中按「先比较 a，a 相同再比较 b，b 相同再比较 c」排序。跳过 a 直接查 b，就像在字典中跳过拼音首字母直接查第二字母——无法定位。

## 正确使用方式

### 编码规范：避免索引失效

1. **索引列不参与运算**。把运算移到等号右边
2. **类型匹配**。Java 中 String 参数要用  绑定，不要拼接数字
3. **LIKE 只用前缀匹配**。全文搜索用 Elasticsearch 或 MySQL 全文索引
4. **OR 条件两边都有索引**。否则拆成 UNION
5. **联合索引按查询频率排列**。高频等值条件在前，范围查询在后
6. **覆盖索引优先**。查询只涉及索引列时，MySQL 直接从索引返回，不需要回表

### 使用 EXPLAIN 验证

```sql
-- 每条上线的 SQL 都应该 EXPLAIN 一下
EXPLAIN SELECT * FROM users WHERE name = '张三' AND age > 25;
-- 关注：type（不能是 ALL）、key（不能是 NULL）、rows（不能太大）
```

## 边界情况和坑

### 1. IN 在某些情况下走索引

```sql
-- age 有索引
-- ✅ 走索引（IN 的值少时）
SELECT * FROM users WHERE age IN (20, 25, 30);

-- ⚠️ IN 里值很多时，可能退化为全表
SELECT * FROM users WHERE age IN (20, 21, 22, ..., 60);
-- 如果这些值覆盖了大部分数据 → 走全表扫描

-- ✅ BETWEEN 范围查询可以走索引
SELECT * FROM users WHERE age BETWEEN 20 AND 30;
```

### 2. IS NULL 可能走索引

```sql
-- name 有索引
-- ✅ 可以走索引（NULL 值少时）
SELECT * FROM users WHERE name IS NULL;
-- InnoDB 索引可以存储 NULL 值

-- ❌ 如果 NULL 值太多，优化器可能放弃索引
```

### 3. 主键索引和非主键索引的区别

```sql
-- id 是主键（有聚集索引），name 是普通索引

SELECT * FROM users WHERE id = 1;       -- 直接走聚集索引，O(1)
SELECT * FROM users WHERE name = '张三'; -- 先走 name 的辅助索引（回表查聚集索引）
```

**主键查询永远走索引**（因为聚集索引和数据行一起存储，查找主键等于查找数据本身）。
