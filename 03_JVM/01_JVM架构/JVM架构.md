# JVM架构

> JVM是"一次编写到处运行"的核心——一个标准化的软件计算机，把字节码翻译成具体平台的机器码。重点：四大模块职责、执行引擎混合模式、JVM vs JRE vs JDK 的区别。

---

## 这个问题为什么存在？

在没有JVM之前，C/C++程序直接编译为本地机器码，一份代码需要在每个操作系统、每个CPU架构上单独编译、单独维护。

**核心问题**：如何让同一份代码，在任何支持Java的平台上都能正确执行？

JVM的答案是：引入一层「虚拟机器」——所有平台都实现一个能读懂同一份字节码的JVM，Java程序只需要编译一次（生成字节码），到处运行。

---

## 它是怎么解决问题的？

JVM本质上是一台**软件模拟的计算机**，由四大模块组成：

```
JVM
├── 类加载子系统        — 把 .class 文件加载进内存
├── 运行时数据区       — 管理内存：堆、栈、方法区
├── 执行引擎           — 解释器 + JIT 编译器
└── 本地接口           — 调用C/C++等本地代码
```

### 1. 类加载子系统

负责将 `.class` 文件加载到内存，形成可以被JVM使用的Java类。

核心机制是**双亲委派模型**：

```
自定义ClassLoader
        ↑ 都先委派父加载器
Extension ClassLoader
        ↑
Bootstrap ClassLoader（加载 rt.jar 核心类库）
```

收到类加载请求时，先不自己尝试加载，而是委派给父加载器。只有当父加载器反馈无法完成时，才由当前加载器自己尝试加载。

**为什么这样设计？**
- **避免重复加载**：确保 `java.lang.Object` 等核心类在JVM中只存在一份
- **安全性**：防止核心API被篡改（即使自定义了 `java.lang.String`，也不会被加载）

### 2. 运行时数据区

定义了JVM运行程序时所需的内存布局，分为**线程共享**和**线程私有**两大类：

```
┌─────────────────────────────────────────┐
│              JVM 运行时数据区             │
├──────────────┬──────────────────────────┤
│  线程私有      │  线程共享                │
├──────────────┼──────────────────────────┤
│ PC寄存器      │  堆（Heap）              │
│ JVM栈        │  （存放对象实例和数组）     │
│ 本地方法栈    │  方法区（Method Area）      │
│              │  （类信息、常量、静态变量） │
└──────────────┴──────────────────────────┘
```

| 区域 | 线程 | 作用 | 异常 |
|------|------|------|------|
| PC寄存器 | 私有 | 记录当前线程执行的字节码行号 | 无 |
| JVM栈 | 私有 | 存储栈帧（局部变量表、操作数栈） | `StackOverflowError` |
| 本地方法栈 | 私有 | 为 Native 方法服务 | `StackOverflowError` |
| 堆 | 共享 | 存放对象实例 | `OutOfMemoryError` |
| 方法区（元空间） | 共享 | 类信息、常量、静态变量 | `OutOfMemoryError` |

### 3. 执行引擎

负责执行字节码指令，采用**解释器 + JIT编译器**的混合模式：

| 方式 | 特点 | 适用 |
|------|------|------|
| 解释器 | 逐条解释执行，启动快 | 冷启动、短期运行 |
| JIT编译器 | 将热点代码编译为本地机器码，运行快 | 长期运行的热点代码 |

**为什么混合？** 只解释→运行慢；只JIT→启动慢。混合模式兼顾两者：启动初期用解释器，运行一段时间后JIT把热点代码编译为本地码。

HotSpot的热点探测：某个方法或循环执行次数超过阈值（默认10000次），触发JIT编译。

### 4. JVM vs JRE vs JDK

```
JDK（Java Development Kit）
  └── JRE（Java Runtime Environment）
        └── JVM（Java Virtual Machine）
```

- **JVM**：只负责运行字节码，是JRE的子集
- **JRE**：JVM + 核心类库（rt.jar等），可以运行Java程序，但不能编译
- **JDK**：JRE + 编译器（javac）+ 调试工具（jstack、jmap等），可以开发Java程序

## 深入原理

### HotSpot VM的实现细节

HotSpot是目前最常用的JVM实现（OpenJDK和Oracle JDK都用它）。

**分层编译（Tiered Compilation）**：
JIT编译器本身也分层次，不是只有一种：

| 层次 | 名称 | 说明 |
|------|------|------|
| 0 | 解释器 | 无编译，直接解释执行 |
| 1 | C1编译器（客户端） | 简单优化，编译速度快 |
| 2 | C2编译器（服务端） | 深度优化，编译速度慢但生成代码质量高 |
| 3 | 解释器 + 性能监控 | 解释执行的同时收集性能数据 |

默认开启分层编译（`-XX:+TieredCompilation`），让程序在启动阶段和稳定阶段都能获得较好的性能。

### 类加载的底层流程

`.class` 文件加载进JVM，并不是一步完成的，而是分为**加载 → 链接 → 初始化**三个阶段：

```
加载（Loading）
  └── 通过类的全限定名获取 .class 文件的二进制字节流
      │
链接（Linking）
  ├── 验证（Verification）：确保字节码不会危害JVM安全
  ├── 准备（Preparation）：为类静态变量分配内存，赋零值
  └── 解析（Resolution）：将常量池中的符号引用替换为直接引用
      │
初始化（Initialization）
  └── 执行 <clinit>() 方法，为静态变量赋程序员设定的值
```

**为什么初始化要最后做？** 因为静态变量的赋值可能依赖于其他类已经加载完成，必须等链接阶段把依赖都解析清楚之后，才能安全地执行初始化代码。

## 正确使用方式

### 1. 理解JVM参数层面的概念

```bash
# JVM 本身不需要手动启动，由 java 命令拉起
java -Xms2g -Xmx2g -XX:+UseG1GC MyApp

# 区分 JDK 工具和 JVM 概念
javac      # JDK 工具，编译 .java → .class
java       # JDK 工具，启动 JVM
jstack     # JDK 工具，但作用是诊断 JVM 内线程状态
```

**为什么这样用？** `java` 命令启动的进程就是JVM实例，JDK工具是通过进程通信（attach机制）去干预正在运行的JVM。

### 2. 自定义类加载器的正确场景

自定义类加载器不是为了「炫技」，而是有以下合理场景：

- **热部署**：Web容器（Tomcat）为每个Web应用创建独立的类加载器，可以单独重载某个应用而不影响其他应用
- **加密保护**：对 `.class` 文件加密，自定义类加载器在 `findClass()` 时解密
- **模块隔离**：OSGi、Spring Boot DevTools 都用自定义类加载器实现隔离

**注意**：自定义类加载器不一定需要「打破双亲委派」。只有像SPI这类「核心接口由Bootstrap加载，但实现类在classpath」的场景，才需要部分打破双亲委派（即 `Thread.getContextClassLoader()` 的设计背景）。

## 边界情况和坑

### 1. 一个类在JVM中「唯一」的条件

不是「全限定名相同就唯一」——类在JVM中的唯一标识是：`类加载器 + 全限定名`。

```java
// 不同的ClassLoader加载同一个.class文件，会产生两个独立的Class对象
ClassLoader cl1 = new MyClassLoader();
ClassLoader cl2 = new MyClassLoader();

Class<?> c1 = cl1.loadClass("com.example.User");
Class<?> c2 = cl2.loadClass("com.example.User");

System.out.println(c1 == c2);  // false！
System.out.println(c1.equals(c2));  // false！
```

**成因**：JVM通过 `ClassLoader + 类全名` 来标识一个类。这是Tomcat能同时部署多个同名Web应用、以及SPI机制能正常工作的底层基础。

### 2. 静态变量存在哪里？

很多人误以为「静态变量在堆里」——实际上：

- **静态变量本身**存储在**方法区（元空间）**，不在堆里
- **静态变量指向的对象**存储在**堆里**（因为对象永远在堆里分配）

```java
class Example {
    static Object obj = new Object();
    //          ↑           ↑
    //     在元空间     对象在堆里
}
```

### 3. JVM栈溢出不一定是代码写错了

除了无限递归会导致 `StackOverflowError`，还有一种情况是**栈深度本身设置得太小**：

```bash
# 默认栈深度通常够用，但递归层级很深时需要调整
-Xss256k   # 每个线程的栈大小设为256KB（默认1M）
```

栈越小 → 能创建的线程数越多（因为总内存固定），但越容易发生栈溢出。这是「线程数 vs 栈深度」的经典权衡。

### 4. JDK 8之后没有「永久代」了

JDK 7：方法区实现是「永久代（PermGen）」，受 `-XX:MaxPermSize` 控制，容易OOM。

JDK 8+：方法区实现改为「元空间（Metaspace）」，使用本地内存，默认不限制大小（受物理内存上限约束）。

**坑**：如果应用大量使用动态代理（如Spring AOP），元空间会持续增长。需要设置 `-XX:MaxMetaspaceSize` 防止耗尽系统内存。
