# 线上问题排查手册

> 排查问题的能力 = 工具熟练度 + 排查思路 + 业务理解。本文是 SOP，不是银弹。遇到新问题，按照框架走，遇到卡点再深入。

---

## 通用排查框架（5 步法）

```
① 现象观察    → 什么时间、什么服务、什么现象？
② 影响评估    → 影响多少用户？数据是否受损？
③ 快速止血    → 先恢复服务，再排查根因
④ 根因分析    → 按系统层次逐步定位
⑤ 复盘预防    → 补充监控/告警，避免同类问题
```

## 第一类：接口响应慢 / 超时

### 排查路径

```
接口慢
   │
   ├── CPU 高吗？
   │      ├── 是 → 看线程堆栈，找热点方法
   │      │         jstack / Arthas profiler
   │      │         → 死循环？频繁 GC？加密/压缩计算？
   │      │
   │      └── 否 → 看线程状态
   │                BLOCKED → 锁等待
   │                WAITING → 等 IO / 等 MQ
   │                RUNNABLE 但耗时 → 外部依赖慢
   │
   ├── 外部依赖慢？
   │      ├── DB → explain / 慢查询日志 / 连接池
   │      ├── Redis → bigkey / 连接池 / 热 key
   │      ├── 第三方 API → 超时配置 / 限流
   │      └── MQ → 消费堆积 / 消费者挂了
   │
   └── 资源耗尽？
          ├── DB 连接池满 → HikariCP pool exhausted
          ├── Redis 连接池满 → Jedis 连接数
          ├── Tomcat 线程池满 → 请求排队
          └── 文件描述符耗尽 → too many open files
```

### 工具速查

```bash
# 1. Arthas 在线诊断（不需要重启，线上直接用）
# 启动
java -jar arthas-boot.jar
# 常用命令
dashboard              # 系统概览（CPU、内存、线程、GC）
thread -n 5            # CPU 最高的 5 个线程
thread -b              # 找出阻塞其他线程的罪魁祸首
thread --state BLOCKED # 所有 BLOCKED 状态的线程
watch com.xxx.Service method '{params, returnObj, throwExp}' 
    -x 2               # 观察方法调用入参、返回值、异常
trace com.xxx.Service method '#cost > 500'  
    # 追踪方法调用耗时，输出 > 500ms 的调用
stack com.xxx.Service method 
    # 打印方法调用的堆栈

# 2. JVM 相关
jstat -gcutil <pid> 1000    # GC 概况，每秒刷新
jstack <pid> > thread.log   # 导出线程堆栈（用于离线分析）
jmap -histo <pid> | head -30 # 堆中对象分布

# 3. 数据库
show processlist;           # 正在执行的所有 SQL
explain <sql>;              # 执行计划
show engine innodb status;  # InnoDB 锁等待情况
```

## 第二类：OOM / 内存问题

### 排查路径

```
OOM
   │
   ├── Java heap space → 堆内存溢出
   │      ├── 对象泄漏？ → jmap -histo / MAT 分析 dump
   │      │               → ThreadLocal / 静态集合 / 流未关闭
   │      │
   │      └── 正常但内存不够？ → -Xmx 太小？流量突增？
   │
   ├── Metaspace → 类元数据溢出
   │      → 动态代理/CGLib 生成了大量类 / 热加载类加载器泄漏
   │
   ├── Direct buffer → NIO 直接内存
   │      → Netty/HTTP 客户端直接内存未释放
   │
   └── unable to create native thread → 线程数耗尽
           → 线程池未关闭 / 频繁创建线程
```

### 工具速查

```bash
# 1. 配置 OOM 自动 dump
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/var/log/app/heap.hprof

# 2. 手动触发 dump（低峰期）
jmap -dump:format=b,file=heap.hprof <pid>

# 3. 分析 dump
# MAT（Memory Analyzer Tool）打开 .hprof 文件
# → Leak Suspects Report（自动分析泄漏）
# → Dominator Tree（按 retained size 排序）

# 4. Arthas 在线分析（不用 dump）
vmtool --action getInstances --className java.lang.String --limit 20
ognl '@com.xxx.Cache@MAP'    # 查看某个静态集合的内容
```

## 第三类：服务宕机 / 进程退出

### 排查路径

```
进程退出
   │
   ├── 被 K8s/OOM Killer 杀死
   │      → kubectl describe pod / dmesg | grep -i kill
   │      → 容器内存限制？宿主机内存不足？
   │
   ├── 被外部健康检查踢出
   │      → 健康检查接口响应超时
   │      → 原因：GC 停顿期间无响应 / 接口 hang 住
   │
   ├── 抛出未捕获异常
   │      → logback 日志搜 ERROR
   │      → 线程池拒绝策略抛异常？
   │
   └── JVM 致命错误（hs_err_pid.log）
           → StackOverflow / SIGSEGV
           → 看 hs_err_pid.log 文件内容
```

## 第四类：CPU 使用率高

### 排查路径

```
CPU 高
   │
   ├── GC 频繁？
   │      → jstat -gcutil 看 FGC 次数
   │      → G1/Parallel GC 停顿时间
   │      → 对象分配率高？内存泄漏？
   │
   ├── 死循环？
   │      → top -Hp <pid> 找到高 CPU 线程
   │      → jstack <pid> 导出，对比线程 ID
   │      → Arthas profiler 抓 CPU 火焰图
   │
   ├── 正则表达式灾难（ReDoS）？
   │      → 搜索日志中的正则处理
   │      → 编译时用 Chalkboard/OWASP 扫描
   │
   └── 加密/压缩/HASH 计算密集？
           → 正常，看业务类型
           → 加缓存减少重复计算
```

```bash
# CPU 火焰图生成
# 1. 用 Arthas profiler
profiler start --event cpu
# 等一段时间（30秒-1分钟）
profiler stop --format html > flame.html

# 2. 用 perf + 方糖工具
perf record -F 99 -p <pid> -g -- sleep 30
perf script | ./stackcollapse-perf.pl | ./flamegraph.pl > flame.svg
```

## 第五类：消息队列消费堆积

### 排查路径

```
MQ 堆积
   │
   ├── 消费者挂了？
   │      → 容器/进程是否存活
   │      → 健康检查失败被摘除
   │
   ├── 消费者慢？
   │      → 消费逻辑里有 DB 查询/远程调用慢
   │      → 加线程池并发消费
   │
   ├── 消费者报错？
   │      → 日志搜 ERROR / Exception
   │      → 消息重试死循环？
   │
   └── 生产速率 > 消费速率？
           → 正常，需要扩容消费者
           → 或加缓冲队列削峰
```

```bash
# Kafka 消费者滞后检查
kafka-consumer-groups.sh --bootstrap-server <host>:9092 
  --group <group> --describe

# RocketMQ 堆积检查
mqadmin consumerProgress -n <namesrv>:9876 -g <group>
```

## 第六类：网络超时 / 抖动

### 排查路径

```
网络超时
   │
   ├── DNS 解析慢？
   │      → DNS 缓存/本地 DNS 劫持
   │      → 使用 IP 直连或配置可靠的 DNS 服务器
   │
   ├── TCP 连接耗时？
   │      → 长连接复用 / 连接池配置
   │      → 防火墙/NLB 超时断开
   │
   ├── NLB 限流？
   │      → 云厂商 NLB 有 QPS 限制
   │      → 告警配置 + 申请扩容
   │
   └── 内网质量差？
           → 跨 AZ 延迟 / 丢包
           → traceRoute / MTR 排查
```

## 快速止血清单

```
□ 接口超时 → 超时配置先调大 + 降级非核心功能
□ OOM → 重启服务 + dump 分析
□ CPU 高 → jstack 导出 + 找热点代码
□ MQ 堆积 → 扩容消费者 + 告警上游
□ 连接池满 → 重启 + 限流 + 连接池参数调优
□ 磁盘满 → 清理日志/dump文件 + 扩容
□ 服务不可用 → K8s 自动重启 + 切流量
```

## 常用命令速查表

```bash
# 进程相关
ps -ef | grep java                    # 找 Java 进程
top -Hp <pid>                         # 看线程 CPU
lsof -p <pid>                         # 进程打开的文件描述符

# JVM 诊断
jstat -gcutil <pid> 1000              # GC 每秒刷新
jstack <pid> > thread.log             # 线程堆栈
jmap -histo <pid> | head -30           # 对象分布
jmap -dump:format=b,file=heap.hprof <pid>  # 堆 dump

# 网络
netstat -an | grep <port> | wc -l     # 连接数
ss -s                                  # Socket 统计
dmesg | tail -50                       # 系统日志

# Docker/K8s
kubectl get pods                       # Pod 列表
kubectl describe pod <pod>             # Pod 详情
kubectl logs <pod> --tail=100          # 日志
docker stats                            # 容器资源使用

# 数据库
show processlist;                      # 活跃连接
explain <sql>;                         # 执行计划
show engine innodb status;              # InnoDB 状态

# Redis
redis-cli info clients                 # 客户端连接数
redis-cli --bigkeys                    # 大 key
redis-cli slowlog get 10               # 慢查询
```
