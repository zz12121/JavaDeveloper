# 内存模型（JMM + happens-before）

## 这个问题为什么存在？

> CPU 有缓存，编译器会重排序 → 同一行代码，在不同线程里「看到的顺序」可能不一样。

**问题场景**：

```java
// 线程 A
flag = true;        // 操作 1
value = 42;         // 操作 2

// 线程 B
if (flag) {         // 操作 3
    print(value);   // 操作 4
}
```

**期望**：如果 B 看到 `flag=true`，那 `value` 一定是 42。  
**现实**：由于编译器和 CPU 重排序，操作 1 和 2 可能被重排 → B 看到 `flag=true` 时，`value` 可能还是 0（未初始化）。

JMM（Java Memory Model）就是**定义「什么情况下，线程 A 的操作对线程 B 可见」的规则**。

---

## JMM 的核心概念

### 主内存 vs 工作内存

```
线程 A                      线程 B
┌────────────┐            ┌────────────┐
│ 工作内存A  │ ←── load ─→│ 工作内存B  │
│  (CPU缓存) │            │  (CPU缓存) │
└─────┬──────┘            └─────┬──────┘
      │        ┌────────┐      │
      └───────→│ 主内存  │←──────
               │(RAM)   │
               └────────┘
```

- **主内存**：所有线程共享（堆内存）
- **工作内存**：每个线程私有（CPU 缓存 + 寄存器）

**线程间通信**（传递数据）必须经过：**工作内存 → 主内存 → 工作内存**。

---

## happens-before 规则（核心）

`happens-before` 是 JMM 的**核心语义**：如果 A happens-before B，那么 **A 的操作结果对 B 可见**。

### 8 条 happens-before 规则

#### 规则 1：程序顺序规则

```java
// 在同一线程内：
int a = 1;    // A1
int b = 2;    // A2
// A1 happens-before A2（同线程内，代码顺序即 happens-before）
```

> **同线程内**，程序的每个操作 happens-before **后续操作**（代码顺序）。

---

#### 规则 2：volatile 变量规则

```java
// 线程 A
volatile boolean ready = true;    // 写 volatile

// 线程 B
while (ready) {                    // 读 volatile
    // ...
}
// ready 写 happens-before ready 读
// → A 的所有写操作（不只是 ready）对 B 可见
```

> **volatile 写的值 happens-before 后续对同一个 volatile 的读**。

**这就是 `volatile` 保证可见性的 JMM 层面原理**：不是「缓存刷新」，而是 **JMM 定义了 happens-before 关系**。

---

#### 规则 3：监视器锁规则

```java
// 线程 A
synchronized (lock) {
    x = 42;      // 线程 A 的操作
}                // 解锁

// 线程 B
synchronized (lock) {  // 加锁
    print(x);    // 一定能读到 42
}
```

> **同一个锁的解锁 happens-before 后续对这个锁的加锁**。  
> 相当于：解锁时强制把工作内存刷回主内存，加锁时强制清空工作内存（从主内存重新加载）。

---

#### 规则 4：线程启动规则

```java
Thread B = new Thread(() -> {
    // 线程 B 的所有操作
    print(x);
});

x = 42;          // 主线程赋值
B.start();       // 启动线程 B

// x=42 happens-before B.start()
// → B 一定能读到 x=42
```

> **`Thread.start()` 调用 happens-before 被启动线程的任意操作**。

---

#### 规则 5：线程终止规则

```java
Thread B = new Thread(() -> {
    x = 42;
});
B.start();
B.join();

// B 的所有操作 happens-before join() 返回
// → 主线程一定能读到 x=42
```

> **`Thread.join()` 返回 happens-before 检测到线程终止的代码**。

---

#### 规则 6：传递性

```java
// 如果 A happens-before B，B happens-before C
// 则 A happens-before C（传递性）
```

---

#### 规则 7 & 8：中断和 final

```java
// 规则 7：中断
// 对 Thread.interrupt() 的调用 happens-before
// 被中断线程检测到中断状态（InterruptedException）

// 规则 8：final 字段
// 构造函数的 final 写入 happens-before
// 构造函数返回前，this 引用不会逸出
// → 其他线程在构造函数完成后，能安全读到 final 字段
```

## 实战：利用 happens-before 写正确的并发代码

### 1. 单例模式（双重检查 + volatile）

```java
class Singleton {
    private static volatile Singleton instance;

    static Singleton getInstance() {
        if (instance == null) {           // ①
            synchronized (Singleton.class) {
                if (instance == null)
                    instance = new Singleton();  // ②
            }
        }
        return instance;
    }
}
```

**为什么 `volatile` 是必须的？**

```
instance = new Singleton() 不是原子操作：
  1. 分配内存
  2. 调用构造函数
  3. instance = 内存地址（引用赋值）

步骤 2 和 3 可能重排序（编译器优化）：
  → 分配内存 → instance = 地址 → 调用构造函数（还没完成！）

线程 B 在 ① 看到 instance != null
  → 拿到一个「构造函数还没跑完」的对象
  → 调用方法时 NPE

volatile 的 happens-before 规则：
  写入 volatile（步骤 3） happens-before 读取 volatile（步骤 ①）
  → 构造函数（步骤 2） happens-before volatile 写（步骤 3）
  → 线程 B 读到 volatile 时，构造函数一定已经完成
```

---

### 2. 生产者-消费者（wait/notify + synchronized）

```java
// 生产者
synchronized (queue) {
    while (queue.isFull())
        queue.wait();

    queue.add(item);        // happens-before notify
    queue.notify();         // 唤醒消费者
}

// 消费者
synchronized (queue) {
    while (queue.isEmpty())
        queue.wait();

    Object item = queue.remove();  // 一定能拿到生产者添加的值
    queue.notify();
}
```

> `synchronized` 的 happens-before 规则：  
> 解锁（notify 之前） happens-before 加锁（wait 之后） → 保证了可见性。

---

### 3. 线程间传递数据的正确姿势

```java
// ❌ 错误：普通变量没有 happens-before 保证
// 线程 A
data = new Data();
ready = true;

// 线程 B
if (ready) {
    use(data);  // data 可能还是 null！
}

// ✅ 正确：用 volatile
volatile boolean ready = false;
volatile Data data;

// 线程 A
data = new Data();
ready = true;  // happens-before

// 线程 B
if (ready) {
    use(data);  // data 一定已初始化
}
```

## 它和相似方案的本质区别是什么？

### happens-before vs as-if-serial

| | happens-before | as-if-serial |
|--|----------------|--------------|
| **作用对象** | 线程间（跨线程） | 单线程内 |
| **含义** | 如果 A hb B，A 的结果对 B 可见 | 单线程看起来像串行执行 |
| **关系** | JMM 的可见性保证 | 编译器的优化约束 |
| **联系** | 单线程内 as-if-serial 本身就保证 hb | 多线程下额外需要 hb |

**as-if-serial 的例子**：

```java
// 单线程内，编译器和 CPU 可以重排，但必须看起来像串行：
a = 1;      // A1
b = 2;      // A2
print(a+b); // A3

// 重排后：
b = 2;      // 先执行
a = 1;      // 后执行
print(a+b); // 3（和原来一样，因为 a+b 只关心 a 和 b 的值）
```

---

### happens-before vs 内存屏障（底层实现）

> **happens-before 是语义层**（告诉程序员「保证什么」）  
> **内存屏障是实现层**（告诉 CPU「怎么保证」）  

```
happens-before 规则
        ↓
JMM 实现（HotSpot）
        ↓
内存屏障（StoreLoad / LoadLoad / ...）
        ↓
CPU MESI 协议 / 缓存刷写
```

**常见的 happens-before 实现**：

| happens-before 场景 | 内存屏障 |
|---------------------|----------|
| volatile 写之后 | StoreLoad（最重，禁止写之后的 Load 重排到写之前） |
| volatile 读之前 | LoadLoad（禁止读之后的 Load 重排到读之前） |
| synchronized 入 | Load + Acquire（禁止之后的操作重排到之前） |
| synchronized 出 | Store + Release（释放锁时，把所有写刷回主内存） |

## 边界情况和坑

### 1. 误以为「synchronized 保证了可见性就足够了」

```java
// ❌ 错误：synchronized 块外的普通变量，对其他线程不可见
class Demo {
    boolean flag = false;  // 普通变量

    void writer() {
        flag = true;  // 没有 synchronized
    }

    void reader() {
        if (flag)    // 读取也可能看不到最新的值
            print("...");
    }
}
```

> `flag = true` 在 synchronized 块外，**不保证**对其他线程可见（可能被 CPU 缓存）。  
> 必须：`flag` 用 `volatile`，或者 `writer()` 和 `reader()` 都加 `synchronized`。

---

### 2. happens-before 不保证「先后顺序」

> **happens-before 是偏序关系，不是全序关系**。

```java
// 线程 A：a = 1, b = 2
// 线程 B：x = a, y = b

// 可能的结果：
// x=1, y=2（正常）
// x=0, y=2（a 的写对 B 不可见）
// x=1, y=0（b 的写对 B 不可见）
// x=0, y=0（都不可见）

// 但不可能：x=1, y=0 且 a=1, b=2?（如果 a 的写对 B 可见，b 的写也对 B 可见）
```

**这说明**：两个线程之间的 happens-before 只约束「同一个变量的操作」，不约束「不同变量的操作」。  
→ **不同变量的写，不一定有 happens-before 关系**（这就是竞态条件的根源）。

---

### 3. 构造函数里的 this 逸出

```java
// ❌ 错误：构造函数里把 this 传给其他线程
class Unsafe extends Thread {
    Unsafe() {
        new Thread(this).start();  // this 逸出！在构造函数里启动了另一个线程
    }
}

// 另一个线程可能在构造函数完成前就访问了 this
```

> 正确做法：在构造函数完成后，才把对象引用传给其他线程。  
> `volatile` + 构造函数结束后再启动线程，可以解决这个问题。

---

## 我的理解

**happens-before 是 JMM 给程序员的「承诺」**：满足这些规则的操作，我保证可见性。

**8 条规则里，最常用的是**：
1. 程序顺序规则（同线程内，代码顺序即 hb）
2. volatile 变量规则（写 hb 读）
3. 监视器锁规则（解锁 hb 加锁）
4. 线程启动规则（start() hb 线程的操作）

**理解 happens-before 的关键**：它不是「谁先执行」，而是「谁的结果对谁可见」。  
时间上的先后 ≠ happens-before 关系。
