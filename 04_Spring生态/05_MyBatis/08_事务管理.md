# 事务管理

## 这个问题为什么存在？

> MyBatis 需要执行 SQL，SQL 需要事务。但 MyBatis 自己管理事务吗？还是交给 Spring？

MyBatis 有一套**自己的事务管理**（基于 JDBC `Connection`），但**实际项目中几乎全用 Spring 事务管理**。

理解两套事务机制，才能排查事务相关的问题。

---

## 它是怎么解决问题的？

### MyBatis 原生事务（了解即可）

```
MyBatis 事务接口：Transaction
  ├── JdbcTransaction（默认）    基于 JDBC Connection 的 commit()/rollback()
  └── ManagedTransaction（容器管理） 交给外部容器（如 Java EE 容器）

配置方式（mybatis-config.xml）：
  <transactionManager type="JDBC"/>   ← 默认，MyBatis 自己管事务
  <transactionManager type="MANAGED"/> ← 交给容器
```

**JdbcTransaction 核心逻辑**：

```java
// JdbcTransaction.java
@Override
public void commit() {
    if (connection != null && !connection.getAutoCommit()) {
        connection.commit();  // ← 直接调用 JDBC Connection
    }
}

@Override
public void rollback() {
    if (connection != null && !connection.getAutoCommit()) {
        connection.rollback();
    }
}

// 关键点：setAutoCommit(false) 才开启事务
@Override
public void openConnection() {
    connection = dataSource.getConnection();
    if (autoCommit != null) {
        connection.setAutoCommit(autoCommit);  // ← 关键！
    }
}
```

### Spring 整合后的事务（实际项目用这个！）

```
Spring 事务管理 MyBatis 的方式：
  → 用 SpringManagedTransactionFactory 替换 MyBatis 的 TransactionFactory
  → SpringManagedTransaction 把 Connection 交给 Spring 的 DataSourceTransactionManager 管理
  → @Transactional 注解由 Spring AOP 拦截，和 MyBatis 无关
  → MyBatis 执行 SQL 时，从 TransactionSynchronizationManager 获取当前事务的 Connection
```

**整合核心类**：

```java
// SpringManagedTransaction.java
@Override
public Connection getConnection() {
    // 从 Spring 事务上下文获取 Connection
    this.connection = DataSourceUtils.getConnection(this.dataSource);
    return this.connection;
}
// DataSourceUtils.getConnection() 核心逻辑：
//   1. 从 TransactionSynchronizationManager 获取当前事务的 Connection
//   2. 如果没有事务，从 DataSource 获取新 Connection（autoCommit=true）
//   3. 如果有事务，返回事务绑定的 Connection（autoCommit=false）

// MyBatis 的 SqlSessionTemplate（Spring 整合核心）
// 每次操作都从 TransactionSynchronizationManager 获取当前 SqlSession
// 保证同一事务内使用同一个 SqlSession（同一 Connection）
```

## 源码关键路径

```
@Transactional 方法调用
  → Spring AOP 拦截（TransactionInterceptor）
    → DataSourceTransactionManager.getTransaction()
      → TransactionSynchronizationManager.bindResource(dataSource, Connection)
        → Connection 绑定到当前线程（ThreadLocal）
    → 调用业务方法
      → mapper.insert() / mapper.select()
        → SqlSessionTemplate.SqlSessionInterceptor.invoke()
          → SqlSessionUtils.getSqlSession()
            → SessionHolder holder = TransactionSynchronizationManager.getResource(sessionFactory)
            → 如果当前有事务 → 返回已有的 SqlSession
            → 如果没有事务 → 创建新 SqlSession（autoCommit=true）
          → DefaultSqlSession.insert()/select()
            → Executor.update()/query()
              → StatementHandler.prepare()
                → connection = transaction.getConnection()
                  → DataSourceUtils.getConnection()  ← 从 Spring 获取 Connection
              → StatementHandler.update()/query()
    → 业务方法结束
    → DataSourceTransactionManager.commit()
      → connection.commit()
    → TransactionSynchronizationManager.unbindResource(dataSource)
```

## 它和相似方案的本质区别是什么？

| 对比维度 | MyBatis 原生事务 | Spring 事务管理 | Java EE 容器管理事务（CMT） |
|---------|----------------|----------------|-------------------|
| 配置方式 | mybatis-config.xml | @Transactional 注解 | EJB 注解（@TransactionAttribute） |
| 功能 | 基础（commit/rollback） | 丰富（传播行为/隔离级别/只读/超时） | 丰富 |
| 和 Spring 整合 | 不需要（被 Spring 替换） | ✅ 深度整合 | ❌ |
| 实际使用情况 | 很少（只有独立用 MyBatis 时） | 最常用 | 很少 |

**本质区别**：MyBatis 原生事务是**薄封装**（直接调 JDBC Connection），Spring 事务是**完整的事务管理框架**（支持传播行为、隔离级别、只读事务、超时、回滚规则等）。

## 正确使用方式

### Spring 整合后的正确配置

```java
// Spring Boot 自动配置（无需手动配置）
// 只需要：
// 1. 配置数据源（application.yml）
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/test
    username: root
    password: 123456
    driver-class-name: com.mysql.cj.jdbc.Driver

// 2. 开启事务注解（Spring Boot 自动开启了，不需要手动加）
// @EnableTransactionManagement  ← Spring Boot 已经自动配置了

// 3. 配置事务管理器（Spring Boot 自动配置了）
// PlatformTransactionManager  ← 自动根据数据源创建
```

### @Transactional 正确使用

```java
@Service
public class UserService {
    @Autowired
    private UserMapper userMapper;

    // ✅ 正确：public 方法（Spring AOP 才能拦截）
    @Transactional
    public void insertUser(User user) {
        userMapper.insert(user);
        // 如果这里抛 RuntimeException → 自动回滚
    }

    // ❌ 错误：private 方法（AOP 无法拦截，事务不生效）
    @Transactional
    private void insertUserPrivate(User user) {
        userMapper.insert(user);
    }

    // ✅ 指定回滚规则
    @Transactional(rollbackFor = Exception.class)  // Checked 异常也回滚
    public void insertWithRollback(User user) throws Exception {
        userMapper.insert(user);
        if (user.getAge() < 0) {
            throw new Exception("年龄不能为负");  // 会触发回滚（因为 rollbackFor=Exception.class）
        }
    }

    // ✅ 只读事务（优化性能）
    @Transactional(readOnly = true)
    public User getUserById(Integer id) {
        return userMapper.selectById(id);
        // 数据库可能对只读事务做优化（如 MySQL 不加写锁）
    }
}
```

## 边界情况和坑

### 坑1：自调用导致事务不生效

```java
@Service
public class UserService {
    public void insertTwoUsers(User u1, User u2) {
        insertUser(u1);  // ← 自调用，@Transactional 不生效！
        insertUser(u2);
    }

    @Transactional
    public void insertUser(User user) {
        userMapper.insert(user);
    }
}
// 解决：拆到另一个 Service，或用 AopContext.currentProxy()
```

### 坑2：MyBatis 原生事务和 Spring 事务混用

```
❌ 错误：手动调用 sqlSession.commit()
  try (SqlSession session = sqlSessionFactory.openSession()) {
      session.getMapper(UserMapper.class).insert(user);
      session.commit();  // ← MyBatis 原生事务
  }
// 如果 Spring 的 @Transactional 也作用在这个方法上 → 事务冲突！

✅ 正确：Spring 整合后，不要手动调用 sqlSession.commit()
  让 Spring 管理事务
```

### 坑3：事务隔离级别的实现

```
MyBatis 不支持设置事务隔离级别！
  → 隔离级别是 Connection 的属性
  → MyBatis 的 JdbcTransaction 没有设置隔离级别的代码

✅ 正确：在 Spring 的 @Transactional 中设置
  @Transactional(isolation = Isolation.READ_COMMITTED)
  public void method() { ... }

底层实现：Spring 在获取 Connection 后，调用 connection.setTransactionIsolation()
```

### 坑4：多线程下事务不传播

```
@Transactional
public void mainMethod() {
    new Thread(() -> {
        userMapper.insert(...);  // ← 新线程，没有事务！
    }).start();
}
// 原因：Spring 事务上下文在 ThreadLocal 中，新线程获取不到

解决：
  ① 不在事务方法内开新线程
  ② 或者让新线程也加入事务（分布式事务，如 Seata）
```

## MyBatis 事务相关配置

| 配置项 | 默认值 | 说明 |
|--------|-------|------|
| `defaultExecutorType` | `SIMPLE` | Executor 类型（BATCH 用于批量） |
| `defaultTransactionIsolationLevel` | `null`（使用数据库默认） | 事务隔离级别（MyBatis 原生，不推荐用） |
| `useManagedTransaction` | `false` | 是否用容器管理事务 |

**实际使用建议**：这些配置都不用管，交给 Spring 的 `@Transactional` 管理。

## 我的理解

MyBatis 有自己的事务管理机制（`JdbcTransaction`），但实际项目中**完全不需要关心**——Spring 通过 `SpringManagedTransaction` 把事务管理接管了。

**关键理解**：
- MyBatis 的 `SqlSession` 底层用 `Connection`
- Spring 通过 `TransactionSynchronizationManager` 把 `Connection` 绑定到当前线程
- MyBatis 执行 SQL 时，从 Spring 获取当前事务的 `Connection`
- 所以 `@Transactional` 能控制 MyBatis 的 SQL 执行

**实际建议**：
- 事务配置全用 Spring 的 `@Transactional` ✅
- MyBatis 配置中不需要配 `transactionManager` ⚠️（Spring Boot 自动配置好了）
- 记住自调用问题（最常见事务失效原因）⚠️
