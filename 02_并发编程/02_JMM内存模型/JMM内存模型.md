# JMM 内存模型

> JMM 是 Java 并发编程的理论基石——它不是 JVM 的内存布局，而是一套规范，定义线程之间如何通过内存交互。volatile 和 synchronized 的语义都由 JMM 定义，理解 JMM 才能正确推导任意并发场景下的可见性。

---

## 这个问题为什么存在？

单线程程序中，变量的读写是直觉性的——写了就能读到。但在多线程环境下，这个直觉被打破：

- **硬件层面**：CPU 多级缓存导致同一变量在不同核心上可能有不同副本
- **编译器层面**：编译器优化可能重排指令顺序
- **结果**：一个线程的写入，另一个线程可能永远看不到，或者看到"半成品"状态

如果没有一套规范来定义"什么情况下写入对其他线程可见"，并发编程就成了靠运气的黑暗森林。JMM 就是这套规范。

---

## 核心脉络

JMM 围绕两个核心概念展开：

### 主内存 vs 工作内存

```
┌─────────────────────────────────────────┐
│                主内存                    │
│         （所有共享变量存储在这里）         │
│                                         │
│   x = 0                                 │
└────────┬──────────────────┬──────────────┘
         │ store / write    │ read / load
         ↓                  ↑
  ┌──────────────┐   ┌──────────────┐
  │ 线程A 工作内存│   │ 线程B 工作内存│
  │ x 的副本     │   │ x 的副本     │
  │ assign: x=1 │   │ use: 读取x   │
  └──────────────┘   └──────────────┘

线程不能直接读写主内存，必须通过工作内存中转。
线程之间的工作内存互不可见——只能通过主内存间接通信。
```

### happens-before 规则

happens-before 是 JMM 给程序员的可见性推导工具。如果 A happens-before B，则 A 的结果对 B 可见。

8 条规则中，最常用的 5 条：

1. **程序顺序**：同一线程内，前面的操作 hb 后面的操作
2. **监视器锁**：unlock hb 后续同一锁的 lock
3. **volatile**：volatile 写 hb 后续同一变量的 volatile 读
4. **传递性**：A hb B，B hb C → A hb C
5. **start/join**：start() hb 线程内所有操作；线程内所有操作 hb join() 返回

完整的规则推导和实战案例见 [[02_并发编程/02_JMM内存模型/JMM内存模型详解|JMM 内存模型详解]]。

### 可见性机制全景

- **volatile**（详见 [[02_并发编程/02_JMM内存模型/volatile|volatile]]）：可见性 ✓ 强制刷新；有序性 ✓ 禁止重排；原子性 ✗ 仅读写原子；不阻塞
- **synchronized**（详见 [[02_并发编程/03_锁机制/锁机制|锁机制]]）：可见性 ✓ 释放时刷新；有序性 ✓ 临界区有序；原子性 ✓ 互斥；阻塞 ✓
- **Atomic\\***（详见 [[02_并发编程/05_CAS与原子类/CAS与原子类|CAS与原子类]]）：可见性 ✓ volatile 语义；有序性 ✓ volatile 语义；原子性 ✓ CAS；不阻塞
- **final**：可见性 ✓ 初始化安全；有序性 —；原子性 —；不阻塞

## 与其他维度的关系

```
JMM内存模型 ←→ 锁机制        （监视器锁规则定义了 synchronized 的可见性语义）
JMM内存模型 ←→ CAS与原子类    （Atomic* 内部用 volatile 保证可见性）
JMM内存模型 ←→ 并发容器        （ConcurrentHashMap 的 table 引用用 volatile 保证扩容可见性）
JMM内存模型 ←→ 线程池          （线程间任务结果传递依赖 happens-before 保证）
JMM内存模型 ←→ 虚拟线程        （结构化并发中的任务可见性依赖 JMM 保证）
```
