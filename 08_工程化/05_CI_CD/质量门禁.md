# 质量门禁

> 质量门禁（Quality Gate）是 CI/CD 流水线中的"守门人"，确保不合格的代码不能进入下一阶段。它是工程效率和质量的核心保障机制。

---

## 一、质量门禁概述

### 1.1 什么是质量门禁

```
代码提交 → 构建 → 单元测试 → 代码检查 → 安全扫描 → 质量门禁判定 → 部署
                                                    ↓
                                              通过 ✓  →  继续部署
                                              失败 ✗  →  阻断流水线，通知开发者
```

**核心思想**：在软件交付流程中设置自动化的质量检查点，只有满足预设标准的代码才能继续推进。

### 1.2 质量门禁的价值

| 价值 | 说明 |
|------|------|
| 左移质量 | 在开发阶段发现问题，修复成本远低于生产环境 |
| 标准化 | 团队统一的质量标准，不依赖个人自觉性 |
| 自动化 | 无人值守，每次提交都自动检查 |
| 可追溯 | 每次构建的质量数据都有记录，可回溯 |
| 信心保障 | 合规的代码才能上线，降低线上故障率 |

## 二、质量门禁的层级

### 2.1 三层门禁模型

```
┌─────────────────────────────────────────────┐
│           第三层：发布门禁（Release Gate）       │
│   集成测试 + 性能测试 + 安全审计 + 灰度验证      │
├─────────────────────────────────────────────┤
│           第二层：集成门禁（Integration Gate）    │
│   代码审查 + 集成测试 + 代码覆盖率 + 安全扫描     │
├─────────────────────────────────────────────┤
│           第一层：提交门禁（Commit Gate）         │
│   编译检查 + 单元测试 + 代码格式 + 静态分析       │
└─────────────────────────────────────────────┘
```

### 2.2 各层级详细指标

#### 第一层：提交门禁（Commit Gate）

| 指标 | 阈值示例 | 工具 |
|------|---------|------|
| 编译成功 | 必须通过 | Maven/Gradle |
| 单元测试通过率 | 100% | JUnit/TestNG |
| 代码格式检查 | 0 Error | Checkstyle |
| 静态代码分析 | 0 Critical Bug | SonarQube/SpotBugs |
| 提交信息规范 | 符合 Conventional Commits | commitlint |

#### 第二层：集成门禁（Integration Gate）

| 指标 | 阈值示例 | 工具 |
|------|---------|------|
| 代码审查（Code Review） | 至少 1 人 Approve | GitLab/GitHub PR |
| 单元测试覆盖率 | ≥ 80%（新增代码 90%） | JaCoCo |
| 集成测试通过率 | 100% | Testcontainers/SpringBootTest |
| 安全漏洞扫描 | 0 High/Critical | OWASP Dependency-Check |
| API 兼容性检查 | 无 Breaking Change | Swagger/Diff |

#### 第三层：发布门禁（Release Gate）

| 指标 | 阈值示例 | 工具 |
|------|---------|------|
| 性能基线 | 响应时间 P99 < 500ms | JMeter/Gatling |
| 压力测试 | 吞吐量不低于基线 90% | wrk/k6 |
| 安全审计 | 通过 | SAST/DAST |
| 灰度验证 | 错误率 < 0.1% | APM（SkyWalking/Prometheus） |
| 回归测试 | 通过 | Selenium/Appium |

## 三、代码覆盖率

### 3.1 覆盖率类型

| 类型 | 含义 | 优先级 |
|------|------|--------|
| 行覆盖率（Line Coverage） | 执行到的代码行百分比 | 基础 |
| 分支覆盖率（Branch Coverage） | if/else 分支都执行到的百分比 | 重要 |
| 方法覆盖率（Method Coverage） | 调用到的方法百分比 | 一般 |
| 类覆盖率（Class Coverage） | 实例化到的类百分比 | 一般 |
| 条件覆盖率（Condition Coverage） | 每个条件的 true/false 都执行 | 严格 |

### 3.2 JaCoCo 配置

```xml
<!-- Maven JaCoCo 插件配置 -->
<plugin>
    <groupId>org.jacoco</groupId>
    <artifactId>jacoco-maven-plugin</artifactId>
    <version>0.8.11</version>
    <executions>
        <execution>
            <goals>
                <goal>prepare-agent</goal>
            </goals>
        </execution>
        <execution>
            <id>report</id>
            <phase>test</phase>
            <goals>
                <goal>report</goal>
            </goals>
        </execution>
        <execution>
            <id>check</id>
            <phase>verify</phase>
            <goals>
                <goal>check</goal>
            </goals>
            <configuration>
                <rules>
                    <rule>
                        <!-- 增量覆盖率：只检查本次变更的代码 -->
                        <element>PACKAGE</element>
                        <limits>
                            <limit>
                                <counter>LINE</counter>
                                <value>COVEREDRATIO</value>
                                <!-- 新增代码行覆盖率不低于 90% -->
                                <minimum>0.90</minimum>
                            </limit>
                            <limit>
                                <counter>BRANCH</counter>
                                <value>COVEREDRATIO</value>
                                <minimum>0.80</minimum>
                            </limit>
                        </limits>
                    </rule>
                </rules>
            </configuration>
        </execution>
    </executions>
</plugin>
```

### 3.3 覆盖率阈值设置建议

```
整体覆盖率：
- 核心业务模块 ≥ 80%
- 工具类/通用组件 ≥ 90%
- Controller 层 ≥ 60%（重点测 Service 层）

增量覆盖率（推荐）：
- 新增代码行覆盖率 ≥ 90%
- 新增代码分支覆盖率 ≥ 80%

不要盲目追求 100% 覆盖率：
- getter/setter 不需要测试
- 简单的 CRUD 可以适当放宽
- 异常分支的覆盖性价比低时可以豁免
```

## 四、静态代码分析

### 4.1 工具选型

| 工具 | 语言 | 特点 |
|------|------|------|
| SonarQube | 多语言 | 最主流，有质量门禁功能 |
| SpotBugs | Java | FindBugs 的继任者 |
| Checkstyle | Java | 代码风格检查 |
| PMD | Java | 代码质量检查 |
| Error Prone | Java | Google 出品，编译期检查 |
| Alibaba Java Coding Guidelines | Java | 阿里巴巴 Java 开发手册的自动检查 |

### 4.2 SonarQube 质量门禁规则

```yaml
# sonar-project.properties
sonar.qualitygate.wait=true  # 流水线等待质量门禁结果

# 质量门禁规则示例：
# - 新增代码覆盖率 ≥ 80%
# - 重复率 ≤ 3%
# - 可靠性评级 ≥ A（0 Bug）
# - 安全性评级 ≥ A（0 Vulnerability）
# - 代码异味密度 ≤ 5%
# - 技术债比率 ≤ 5%
```

### 4.3 代码异味等级

| 级别 | 说明 | 示例 |
|------|------|------|
| Blocker | 必须修复 | 空指针风险、SQL 注入、硬编码密码 |
| Critical | 强烈建议修复 | 未关闭的资源、空 catch 块 |
| Major | 建议修复 | 过长方法（>80行）、圈复杂度过高 |
| Minor | 可选修复 | 魔法数字、命名不规范 |
| Info | 提示 | 注释缺失、import 未使用 |

## 五、CI/CD 平台中的质量门禁

### 5.1 GitLab CI/CD

```yaml
# .gitlab-ci.yml
stages:
  - build
  - test
  - quality-gate
  - deploy

build:
  stage: build
  script:
    - mvn clean compile

unit-test:
  stage: test
  script:
    - mvn test
  coverage: '/Total.*?([0-9]{1,3})%/'

sonarqube-check:
  stage: quality-gate
  script:
    - mvn verify sonar:sonar -Dsonar.qualitygate.wait=true
  allow_failure: false  # 质量门禁失败则阻断

deploy:
  stage: deploy
  script:
    - kubectl apply -f k8s/
  when: manual  # 门禁通过后手动触发
```

### 5.2 GitHub Actions

```yaml
# .github/workflows/ci.yml
name: CI with Quality Gate

on: [push, pull_request]

jobs:
  quality-gate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build & Test
        run: mvn clean verify

      - name: Check Coverage
        run: |
          # 解析 JaCoCo 报告，覆盖率低于阈值则失败
          COVERAGE=$(awk -F',' '/Total/{print $4}' target/site/jacoco/jacoco.csv | tr -d '%')
          if (( $(echo "$COVERAGE < 80" | bc -l) )); then
            echo "Coverage $COVERAGE% is below 80%"
            exit 1
          fi

      - name: SonarQube Scan
        uses: sonarsource/sonarqube-scan-action@v2
        with:
          args: >
            -Dsonar.qualitygate.wait=true
```

### 5.3 Jenkins Pipeline

```groovy
pipeline {
    agent any

    stages {
        stage('Build') {
            steps {
                sh 'mvn clean compile'
            }
        }

        stage('Test') {
            steps {
                sh 'mvn test'
            }
            post {
                always {
                    junit '**/target/surefire-reports/*.xml'
                }
            }
        }

        stage('Quality Gate') {
            steps {
                sh 'mvn verify sonar:sonar'
            }
        }

        stage('Deploy') {
            when {
                branch 'main'
            }
            steps {
                sh 'kubectl apply -f k8s/'
            }
        }
    }
}
```

## 六、安全扫描门禁

### 6.1 SAST（静态应用安全测试）

```
在编译期扫描源代码中的安全漏洞

工具：
- SonarQube Security
- SpotBugs + FindSecBugs
- Semgrep
- Checkmarx

检查内容：
- SQL 注入、XSS、CSRF
- 硬编码密码/密钥
- 不安全的反序列化
- 路径遍历
- 不安全的加密算法
```

### 6.2 SCA（软件成分分析）

```
扫描项目依赖中的已知漏洞

工具：
- OWASP Dependency-Check
- Snyk
- Trivy

场景：log4j2 漏洞（CVE-2021-44228）爆发时，
通过 SCA 可以快速定位哪些项目使用了受影响版本
```

### 6.3 DAST（动态应用安全测试）

```
在运行时向应用发送攻击请求，检测安全漏洞

工具：
- OWASP ZAP
- Burp Suite

通常在集成测试或预发布环境执行
```

## 七、质量门禁的实践建议

### 7.1 渐进式引入

```
阶段 1（基础）：编译 + 单元测试通过
阶段 2（增强）：+ 代码覆盖率 ≥ 80% + Checkstyle
阶段 3（完善）：+ SonarQube 扫描 + 安全检查
阶段 4（高级）：+ 性能基线 + 灰度验证
```

### 7.2 常见陷阱

| 陷阱 | 解决方案 |
|------|---------|
| 门禁太严格，流水线频繁失败 | 分级门禁，Blocker 必过，Major 可豁免 |
| 门禁太松，形同虚设 | 设置合理的阈值，定期调整 |
| 覆盖率造假 | 看分支覆盖率，不要只看行覆盖率 |
| 扫描工具误报 | 配置排除规则，但要定期审查 |
| 质量门禁太慢影响交付 | 并行执行，增量分析 |

### 7.3 质量门禁与团队文化

```
好的质量门禁：
✓ 快速反馈（< 10 分钟）
✓ 规则透明（团队共同制定）
✓ 可追踪（每次构建的质量报告）
✓ 可调整（定期回顾和优化阈值）
✓ 自动化（无需人工干预）

坏的质量门禁：
✗ 太慢（> 30 分钟）
✗ 规则黑箱（没人知道为什么不过）
✗ 大量误报（开发者的抱怨）
✗ 不可调整（阈值僵化）
✗ 绕过机制（"先合并再说"）
```

## 八、面试高频问题

| 问题 | 关键回答 |
|------|---------|
| 你们的质量门禁有哪些指标？ | 编译通过、单测覆盖率≥80%、SonarQube 0 Critical、安全扫描 0 High |
| 代码覆盖率多少合适？ | 核心模块 80-90%，整体 70-80%，增量代码 90% |
| SonarQube 和 Checkstyle 的区别？ | Checkstyle 查代码风格，SonarQube 查代码质量+安全+重复率 |
| 如何处理质量门禁失败？ | 1. 看报告定位问题 2. 修复并提交 3. 紧急情况可由 Tech Lead 审批豁免 |
| SAST 和 DAST 的区别？ | SAST 扫源码（白盒），DAST 测运行中的应用（黑盒） |
