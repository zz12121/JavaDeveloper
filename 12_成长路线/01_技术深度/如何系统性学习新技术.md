# 如何系统性学习新技术

> 技术学习的核心不是"看完了"，而是"能用了、能讲了、能选型了"。

---

## 一、学习的四个阶段

```
阶段 1：会用（API Level）
→ 能跟着文档跑通 Demo，能完成 CRUD 任务
→ 标志：能独立完成功能开发

阶段 2：理解（Why Level）
→ 知道为什么这样设计，能读源码，能排查问题
→ 标志：能定位线上 Bug，能回答"为什么"

阶段 3：能选（Trade-off Level）
→ 面对多个方案能说出优劣，能根据场景选型
→ 标志：能做技术选型文档，能在评审中辩论

阶段 4：能创造（Create Level）
→ 能设计类似的系统，能给社区贡献代码
→ 标志：能讲清楚背后的设计哲学
```

## 二、系统性学习路径（以学习一门新技术为例）

### 2.1 第一步：建立全局认知（1-2 天）

```
目标：知道这门技术是"干什么的、解决什么问题、在什么场景下用"

动作：
1. 官方文档的 Overview / Getting Started（别跳过！）
2. 找一篇高质量的中文入门文章（建立直觉）
3. 画一张「它解决什么问题」的图
4. 问自己：没有它之前，人们是怎么解决的？

示例（学 Redis）：
- 问题：MySQL 查询慢，高并发下扛不住
- 解决：内存 KV 缓存，单机 QPS 10万+
- 没有它之前：加 MySQL 从库、加本地缓存（Ehcache）
- 优劣：Redis 是集中式缓存，Ehcache 是本地缓存
```

### 2.2 第二步：跑通核心 Demo（3-5 天）

```
目标：动手把最核心的功能跑起来，建立"手感"

原则：
- 不要一上来就看源码
- 不要试图一次性学完所有功能
- 每跑通一个 Demo，就问"为什么这样用"

学习顺序（以 Spring Boot 为例）：
1. 最简单的 RESTful API（理解 IoC/DI）
2. 加上数据库（理解 JPA/MyBatis 整合）
3. 加上 Redis 缓存（理解缓存抽象）
4. 加上 Spring Security（理解过滤链）
→ 每步都问：这一步解决了什么问题？
```

### 2.3 第三步：读核心源码（1-2 周）

```
目标：理解"为什么"，而不只是"怎么用"

源码阅读顺序（以 HashMap 为例）：
1. 构造函数（理解初始化逻辑）
2. put 方法（理解核心流程）
3. hash 函数（理解为什么要这样设计）
4. resize 方法（理解扩容机制）
5. treeifyBin 方法（理解为什么用红黑树）

技巧：
- 先读注释！JDK 源码的注释是最权威的文档
- 用 Debug 模式跟着走一遍
- 画流程图，比看代码有效 10 倍
- 遇到不懂的先标记，不要卡住
```

### 2.4 第四步：做小项目练手（2-4 周）

```
目标：在真实场景中用起来，遇到问题、解决问题

练手项目要求：
- 覆盖该技术 70% 的核心功能
- 有真实的业务场景（不是 Todo List）
- 能部署运行，能压测
- 写技术博客记录踩坑过程

示例（学 Kafka）：
练手项目：实现一个日志收集系统
- Producer 发送应用日志
- Consumer 消费并写入 Elasticsearch
- 处理 consumer rebalance、消息丢失问题
```

### 2.5 第五步：输出倒逼输入（持续）

```
目标：能讲清楚，才是真懂了

输出形式：
1. 技术博客（写给别人看，逼自己弄懂）
2. 团队技术分享（PPT + Demo）
3. 参与开源（提 PR、修 Bug、写文档）
4. 面试别人（能讲清楚说明真懂了）

费曼学习法：
- 选择一个概念
- 用最简单的语言讲给"小白"听
- 卡住的地方就是没真正懂的地方
- 回头重新学习，直到能讲清楚
```

## 三、不同技术的学习策略

| 技术类型 | 学习重点 | 推荐顺序 |
|---------|---------|---------|
| 框架类（Spring） | 设计思想 > API 使用 | 官方文档 → 核心注解 → 源码 → 自动配置原理 |
| 中间件类（Redis/Kafka） | 使用场景 > 原理 | 基本命令 → Java 客户端 → 核心原理 → 生产最佳实践 |
| 语言类（Java/Go） | 语言特性 > 语法 | 基础语法 → 集合框架 → 并发包 → JVM → 新特性 |
| 工具类（Maven/Docker） | 能用就行 | 跟着文档做，遇到报错再深入 |
| 理论类（DDD/微服务） | 思想 > 具体实现 | 看书 → 看案例 → 自己设计一个系统 |

## 四、学习中的常见陷阱

### 陷阱 1：只看不练

```
症状：看了很多文章，觉得"懂了"，一写代码就懵
解药：学完一个知识点，立刻写 Demo 验证
规则：看 1 小时，练 2 小时
```

### 陷阱 2：上来就啃源码

```
症状：还没搞懂基本用法，就去读源码，越读越懵
解药：先跑通 Demo，再读源码；先读注释，再读代码
规则：至少先写 500 行使用该技术的代码，再碰源码
```

### 陷阱 3：收藏夹吃灰

```
症状：收藏了几百篇文章，一篇都没仔细看过
解药：收藏的文章，当天或第二天必须看完；看不完不要收藏
规则：同时进行的"学习中技术"不超过 3 个
```

### 陷阱 4：追求完美学习

```
症状：想"系统学完再动手"，结果永远学不完
解药：MVP 思维——先掌握 20% 的功能解决 80% 的问题
规则：80/20 原则，先搞定最常用的部分
```

### 陷阱 5：不输出

```
症状：学了很多，但说不清楚，面试一问就虚
解药：学完就写博客 / 做分享，输出倒逼输入
规则：每学完一个模块，输出一篇笔记
```

## 五、学习资源分级

### 5.1 官方文档（⭐⭐⭐⭐⭐ 必读）

```
优点：最权威、最全面、更新及时
缺点：英文、枯燥

阅读技巧：
- 先看 Overview，建立全局认知
- Quick Start 跟着做一遍
- 重点看 Configuration / API 章节
- 遇到不懂的先跳过，不要卡住
```

### 5.2 经典书籍（⭐⭐⭐⭐ 系统学习）

```
Java 后端必读书：
- 《Effective Java》（Java 编程最佳实践）
- 《Java 并发编程实战》（并发圣经）
- 《深入理解 Java 虚拟机》（JVM 必读）
- 《Spring 实战》（Spring 全家桶）
- 《数据密集型应用系统设计》（分布式必读）
```

### 5.3 优质博客 / 公众号（⭐⭐⭐ 补盲点）

```
优点：有中文、有图、有实战经验
缺点：碎片化、质量参差不齐

筛选标准：
- 有没有源码分析？
- 有没有踩坑记录？
- 有没有自己的思考（而不是搬运）？
```

### 5.4 视频课程（⭐⭐ 入门用）

```
优点：直观、适合入门
缺点：节奏慢、信息密度低

使用建议：
- 只看入门部分，建立直觉
- 进阶部分看文档和源码，不要依赖视频
```

## 六、学习节奏建议

```
工作日（每天 1-2 小时）：
- 早上通勤：听技术播客 / 看技术文章
- 午休：刷 1-2 道算法题（保持手感）
- 晚上：深度学习 1 小时（跟着文档做 Demo）

周末（每天 4-6 小时）：
- 周六上午：系统学习（看书 / 读源码）
- 周六下午：练手项目
- 周日：输出（写博客 / 整理笔记）

每季度：
- 回顾本季度学了什么
- 输出一篇总结文章
- 调整下一季度学习方向
```

## 七、检验学习成果的标准

```
初级检验（会用）：
□ 能不看文档写出 Hello World
□ 能独立完成该功能模块的开发
□ 能回答面试官的基础问题

中级检验（理解）：
□ 能讲清楚核心设计思想
□ 能读懂核心源码
□ 能排查该技术的常见线上问题

高级检验（能选）：
□ 能在 2-3 个方案中说出优劣
□ 能根据业务场景做技术选型
□ 能在技术评审中辩论

顶级检验（能创造）：
□ 能设计一个类似的中间件
□ 能给该技术的开源社区贡献代码
□ 能带团队落地该技术
```
