# 短链接系统

## 这个问题为什么存在？

> 长链接不方便分享——短信里的链接太长、微博有字数限制、二维码太密扫描困难。问题是：**怎么把长链接映射成短链接，且能跳转回去？**

没有短链接系统，你会遇到：
- 短信内容超长，用户看不全
- 微博发不出去（140字限制）
- 二维码太密，扫码识别率低

---

## 它是怎么解决问题的？

### 核心架构：ID生成 + 存储 + 跳转

```
用户输入长链接 → 系统生成短链接 → 返回短链接
    │
    ▼
┌─────────────────────────────┐
│ 1. ID生成（自增/雪花算法） │
│ 2. 存储映射关系（MySQL/Redis）  │
│ 3. 短链接跳转（301/302）    │
└──────────┬──────────────────┘
           │
           ▼
用户访问短链接 → 查映射关系 → 302跳转 → 长链接
```

### 关键技术点

#### 1. ID生成（核心）

```
方案1：自增ID + Base62编码
  ID: 1 → 短链接: http://t.cn/1
  ID: 1000 → 短链接: http://t.cn（不同Base62编码）

方案2：雪花算法（分布式）
  ID: 7256246525782548 → Base62 → 6字符
```

```java
// Base62编码（将自增ID转成短字符串）
public class Base62 {
    private static final String CHARS =
            "abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789";

    public static String encode(long num) {
        if (num == 0) return "0";
        StringBuilder sb = new StringBuilder();
        while (num > 0) {
            sb.append(CHARS.charAt((int)(num % 62)));
            num /= 62;
        }
        return sb.reverse().toString();  // 注意反转
    }
}

// 生成短链接
@PostMapping("/shorten")
public String shortenUrl(@RequestParam String longUrl) {
    // 1. 检查是否已存在（避免重复）
    String existShortKey = redis.get("long2short:" + longUrl);
    if (existShortKey != null) {
        return BASE_URL + existShortKey;  // 已存在，直接返回
    }

    // 2. 生成自增ID（从MySQL获取）
    long id = idGenerator.nextId("short_url");
    String shortKey = Base62.encode(id);

    // 3. 存储映射关系
    redis.set("short:" + shortKey, longUrl);
    redis.set("long2short:" + longUrl, shortKey);
    mysql.insert("INSERT INTO short_url (short_key, long_url) VALUES (?, ?)",
            shortKey, longUrl);

    return BASE_URL + shortKey;
}
```

**注意**：自增ID要用**号段模式**或**雪花算法**，避免每次都访问DB。

#### 2. 跳转（301 vs 302）

```
301（永久重定向）：浏览器缓存长链接，不再请求短链接服务
         → 优点：省流量
         → 缺点：无法统计点击量（浏览器不请求服务器）

302（临时重定向）：浏览器每次都请求短链接服务
         → 优点：能统计点击量
         → 缺点：每次都要请求服务器
```

```java
@GetMapping("/{shortKey}")
public ResponseEntity<Void> redirect(@PathVariable String shortKey) {
    String longUrl = redis.get("short:" + shortKey);
    if (longUrl == null) {
        return ResponseEntity.notFound().build();
    }

    // 302重定向（临时，浏览器每次都请求）
    HttpHeaders headers = new HttpHeaders();
    headers.setLocation(URI.create(longUrl));
    return ResponseEntity.status(HttpStatus.FOUND).headers(headers).build();
}
```

**生产推荐**：**302**（能统计点击量）。如果流量太大，可以用301 + 异步统计。

#### 3. 高并发优化

```
优化1：Redis缓存（短链接 → 长链接）
  → 命中率99%，几乎没有DB访问

优化2：布隆过滤器（防止缓存穿透）
  → 不存在的短链接，直接返回404

优化3：MySQL读写分离
  → 写主库，读从库（读多写少）
```

```java
// 布隆过滤器：防止缓存穿透
private final BloomFilter<String> bloomFilter =
        BloomFilter.create(Funnels.stringFunnel(), 100000000, 0.01);  // 1亿数据，1%误判率

@GetMapping("/{shortKey}")
public ResponseEntity<Void> redirect(@PathVariable String shortKey) {
    // 1. 布隆过滤器快速判断（不存在直接返回404）
    if (!bloomFilter.mightContain(shortKey)) {
        return ResponseEntity.notFound().build();
    }

    // 2. Redis查询
    String longUrl = redis.get("short:" + shortKey);
    if (longUrl == null) {
        // 3. MySQL查询（罕见，缓存穿透才到这里）
        longUrl = mysql.queryForObject(
                "SELECT long_url FROM short_url WHERE short_key = ?", shortKey);
        if (longUrl == null) {
            return ResponseEntity.notFound().build();
        }
        redis.set("short:" + shortKey, longUrl, 3600);  // 缓存1小时
    }

    // 4. 302重定向
    HttpHeaders headers = new HttpHeaders();
    headers.setLocation(URI.create(longUrl));
    return ResponseEntity.status(HttpStatus.FOUND).headers(headers).build();
}
```

## 深入原理

| | 自增ID + Base62 | 雪花算法 + Base62 | UUID + Base62 |
|---|---|---|---|
| 有序性 | ✅ 有序 | ✅ 趋势有序 | ❌ 无序 |
| 碰撞率 | 0%（自增不重复） | 极低（时钟回拨才有问题） | 极低（但太长） |
| 性能 | 高（号段模式） | 极高（无网络IO） | 中（需要查重） |
| 适用场景 | 中小型（单机自增） | 大型（分布式） | 不推荐（太长） |

**本质区别**：
- **自增ID**：简单，但分布式下需要号段模式或雪花算法
- **雪花算法**：分布式首选，但时钟回拨是致命问题
- **UUID**：太长，Base62编码后还是长，不适合短链接

## 正确使用方式

### 防滥用（签名 + 限流）

```java
// 1. 签名（防止被恶意调用，生成大量短链接）
@PostMapping("/shorten")
public String shortenUrl(@RequestParam String longUrl,
                       @RequestParam String sign) {
    // 验证签名（简单示例，生产要用HMAC）
    if (!validateSign(longUrl, sign)) {
        throw new RuntimeException("签名错误");
    }

    // 2. 限流（每个用户每分钟最多生成10个）
    String userKey = "shorten:limit:" + getUserId();
    Long count = redis.incr(userKey);
    if (count == 1) {
        redis.expire(userKey, 60);  // 60秒过期
    }
    if (count > 10) {
        throw new RuntimeException("操作过于频繁");
    }

    // 生成短链接...
}
```

### 点击统计（异步）

```java
@GetMapping("/{shortKey}")
public ResponseEntity<Void> redirect(@PathVariable String shortKey) {
    String longUrl = getLongUrl(shortKey);

    // 异步统计点击量（不阻塞跳转）
    CompletableFuture.runAsync(() -> {
        redis.incr("click:" + shortKey);  // 先写Redis
        // 每1000次，刷回MySQL（避免频繁写DB）
        if (redis.get("click:" + shortKey) % 1000 == 0) {
            mysql.update("UPDATE short_url SET click_count = ? WHERE short_key = ?",
                    redis.get("click:" + shortKey), shortKey);
        }
    });

    // 302重定向...
}
```

## 边界情况和坑

### 坑1：哈希冲突（不同长链接生成相同短链接）

```
场景：Base62编码后，不同长链接可能有相同短链接（哈希冲突）
后果：用户A的短链接，跳转到用户B的长链接
```

**解决方案**：
1. **用自增ID，不哈希**（这是最安全的，自增ID不会重复）
2. **检查冲突**（生成短链接后，查DB是否已存在，存在就重新生成）

```java
// 防冲突：生成短链接后，检查是否已存在
public String generateShortKey(String longUrl) {
    long id = idGenerator.nextId("short_url");
    String shortKey = Base62.encode(id);

    // 检查是否已存在（罕见，但保险）
    String existLongUrl = mysql.queryForObject(
            "SELECT long_url FROM short_url WHERE short_key = ?", shortKey);
    if (existLongUrl != null && !existLongUrl.equals(longUrl)) {
        // 冲突了，重新生成（递归调用，但概率极低）
        return generateShortKey(longUrl);
    }
    return shortKey;
}
```

### 坑2：缓存穿透（大量请求不存在的短链接）

```
场景：恶意请求不存在的短链接（如 http://t.cn/aaaaaa）
后果：Redis没有，MySQL也没有，但每次都打到MySQL
```

**解决方案**：
1. **布隆过滤器**（快速判断不存在，直接返回404）
2. **缓存空值**（不存在的短链接也缓存，TTL短一点，如5分钟）

### 坑3：数据倾斜（某些短链接被大量访问）

```
场景：某个短链接被病毒传播，访问量暴增
后果：Redis热点，单节点打挂
```

**解决方案**：
1. **本地缓存**（JVM缓存热点短链接，减少Redis访问）
2. **Redis Cluster**（热点数据自动迁移到不同节点）
