# 分布式一致性协议

## 这个问题为什么存在？

> 单体事务靠数据库保证ACID。但分布式环境下，**一笔业务横跨多个服务、多个库**，每个服务只管自己的事务，问题是：**怎么保证跨服务的数据一致性？**

没有一致性协议，你会遇到：
- 订单创建成功，扣库存失败 → 超卖
- 支付成功，通知失败 → 用户没收到确认
- 服务A提交，服务B回滚 → 数据不一致

---

## 它是怎么解决问题的？

### 2PC（两阶段提交）：经典但有问题

```
协调者（事务管理器）
       │
  ┌────┴────┐
  │         │
服务A    服务B
（订单）  （库存）
```

**阶段1：Prepare**
```
协调者 → 所有参与者：Prepare？
参与者 → 写undo/redo日志，返回Yes/No
```

**阶段2：Commit/Rollback**
```
如果全部Yes → 协调者发Commit → 所有参与者提交
如果任一No  → 协调者发Rollback → 所有参与者回滚
```

```java
// 2PC 伪代码（协调者视角）
public class TwoPhaseCommit {
    public void commit(Transaction tx) {
        // Phase 1: Prepare
        Map<Participant, Boolean> votes = new HashMap<>();
        for (Participant p : tx.participants) {
            votes.put(p, p.prepare(tx));
        }

        // Phase 2: Commit or Rollback
        if (votes.values().stream().allMatch(v -> v == true)) {
            for (Participant p : tx.participants) {
                p.commit(tx);
            }
        } else {
            for (Participant p : tx.participants) {
                p.rollback(tx);
            }
        }
    }
}
```

**致命缺陷**：
1. **同步阻塞**：Prepare后，资源被锁住，等待协调者指令
2. **单点故障**：协调者挂了，参与者永远等待
3. **脑裂**：协调者发Commit后挂了，部分参与者收到，部分没收到

### 3PC（三阶段提交）：解决单点故障，但网络分区仍无解

```
CanCommit → PreCommit → DoCommit
   │           │           │
 询问        准备        提交
（无锁）    （持锁）   （执行）
```

**比2PC多了一个CanCommit阶段**：先询问"能不能提交"，大家都行再进入Prepare。

**优点**：协调者挂了，参与者超时后会自动中止（不会永远持锁）
**缺点**：网络分区时，参与者可能收到不同的指令，仍然不一致

> 结论：**3PC没有完全解决2PC的问题**，实际生产中很少直接使用。

### TCC（Try-Confirm-Cancel）：业务层面的2PC

```
Try：预留资源（不是真正执行）
Confirm：确认执行（真正扣减）
Cancel：取消预留（释放资源）
```

```java
// 电商下单的TCC实现
public class OrderTccService {

    @Transactional
    public void tryCreateOrder(Order order) {
        // Try：插入"预留状态"的订单，不真正生效
        order.setStatus("PENDING");
        orderMapper.insert(order);
        // 冻结库存（不是真正扣减）
        inventoryMapper.freeze(order.getProductId(), order.getAmount());
    }

    @Transactional
    public void confirmCreateOrder(Order order) {
        // Confirm：订单生效，扣减真实库存
        order.setStatus("CONFIRMED");
        orderMapper.updateStatus(order);
        inventoryMapper.deduct(order.getProductId(), order.getAmount());
    }

    @Transactional
    public void cancelCreateOrder(Order order) {
        // Cancel：订单取消，释放冻结库存
        order.setStatus("CANCELLED");
        orderMapper.updateStatus(order);
        inventoryMapper.unfreeze(order.getProductId(), order.getAmount());
    }
}
```

**为什么TCC比2PC好**：2PC是资源层（数据库锁），TCC是业务层（逻辑预留），持锁时间短。

### Saga：长事务的最终一致性方案

```
订单创建 → 扣库存 → 扣余额 → 通知用户
   │         │
   └─── 失败 → 反向补偿 ───────────────┘
             Cancel库存  Cancel余额
```

```java
// Saga：每个操作都有对应的补偿操作
public class OrderSaga {
    public void execute(Order order) {
        try {
            createOrder(order);       // 正向操作1
            deductInventory(order);   // 正向操作2
            deductBalance(order);     // 正向操作3
            notifyUser(order);        // 正向操作4
        } catch (Exception e) {
            // 失败，按相反顺序补偿
            compensateNotifyUser(order);
            compensateBalance(order);
            compensateInventory(order);
            compensateOrder(order);
        }
    }
}
```

**适用场景**：流程长、涉及多个服务的**最终一致性**场景（如电商下单）。

## 它和相似方案的本质区别是什么？

| 方案 | 一致性 | 性能 | 复杂度 | 适用场景 |
|---|---|---|---|---|
| 2PC | 强一致 | 低（同步阻塞） | 低（依赖DB） | 传统企业应用（很少用） |
| 3PC | 强一致（理论上） | 低 | 中 | 几乎不用 |
| TCC | 最终一致 | 高（业务层） | 高（需改代码） | 金融、支付（强一致要求） |
| Saga | 最终一致 | 高（异步补偿） | 中（需写补偿） | 长流程业务（电商下单） |
| 本地消息表 | 最终一致 | 中（异步轮询） | 低 | 对一致性要求不高的场景 |
| 事务消息（MQ） | 最终一致 | 高（MQ保证） | 低 | 主流方案（RocketMQ事务消息） |

**本质区别**：
- **2PC/3PC**：资源层一致，但性能差，实际很少用
- **TCC**：业务层一致，性能好，但侵入业务代码
- **Saga**：补偿机制，适合长事务
- **事务消息**：主流方案，MQ保证半消息+回查

## 正确使用方式

### 场景1：支付系统用TCC

```
Try：冻结账户金额（不是真正扣减）
Confirm：真正扣减（只有Try全成功才执行）
Cancel：解冻金额（Try失败后执行）
```

### 场景2：电商下单用Saga

```
正向：创建订单 → 扣库存 → 扣余额 → 发通知
失败：发通知失败的补偿（标记订单为"通知失败"，重试）
```

### 场景3：RocketMQ事务消息（推荐）

```java
// RocketMQ事务消息：半消息 + 本地事务 + 回查
public class OrderTransactionProducer {

    public void createOrder(Order order) {
        // 1. 发送半消息（对消费者不可见）
        TransactionSendResult result = producer.sendMessageInTransaction(
                new Message("order_topic", order.toString().getBytes()),
                order
        );

        // 2. 执行本地事务（创建订单）
        // 3. 根据本地事务结果，提交或回滚半消息
    }

    // 本地事务执行状态（MQ会回查这个方法）
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        String orderId = msg.getKeys();
        Order order = orderMapper.selectById(orderId);
        if (order == null) return ROLLBACK;
        if (order.getStatus().equals("CONFIRMED")) return COMMIT;
        return UNKNOW;  // 继续回查
    }
}
```

## 边界情况和坑

### 坑1：TCC的空回滚和悬挂

```
空回滚：Try还没执行，Cancel先来了（网络超时，协调者认为Try失败）
悬挂：Try迟到，在Cancel之后才执行（Cancel已经释放资源，Try又预留了）
```

**解决方案**：
1. 记录事务状态表（Try、Confirm、Cancel都写日志）
2. 空回滚：Cancel时检查Try是否执行，没执行就记录"空回滚"标记
3. 悬挂：Try时检查是否已经有Cancel记录，有就释放资源直接返回

### 坑2：Saga的补偿失败

```
场景：正向操作执行到第三步失败了，开始补偿，但补偿也失败
结果：数据处于"部分补偿"状态，不一致
```

**解决方案**：
1. **补偿操作必须幂等**（可以重试）
2. **记录补偿状态**（哪些补偿成功了，哪些失败了）
3. **人工介入**（补偿失败超过阈值，告警人工处理）

### 坑3：2PC的协调者单点

```
场景：协调者在Phase 2挂了
结果：参与者永远持锁等待
```

**解决方案**：协调者高可用（ZooKeeper选主），新协调者从日志恢复状态。

## 我的理解

分布式一致性协议的核心矛盾：**强一致性和高性能不可兼得**。

- **2PC**：理论上强一致，但实际性能差（同步阻塞），几乎不用
- **TCC**：业务层强一致，性能好，但侵入代码，需要实现三个接口
- **Saga**：补偿机制，适合长事务，但一致性弱（中间状态对外可见）
- **事务消息**：主流方案（RocketMQ），MQ保证半消息+回查，对业务侵入小

实际生产中：
- **强一致要求（支付）** → TCC
- **最终一致即可（电商）** → Saga 或 事务消息
- **传统企业应用** → 2PC（但性能和可用性都很差）

**面试最高频追问**：
1. "2PC的问题是什么？" → 同步阻塞、单点故障、脑裂
2. "TCC的三个阶段？" → Try（预留）、Confirm（确认）、Cancel（取消）
3. "Saga和TCC的区别？" → Saga是补偿，TCC是预留；Saga一致性弱，TCC强一致

---

*分布式一致性的本质：用复杂度换一致性，但永远没有"完美方案"，只有trade-off。*
