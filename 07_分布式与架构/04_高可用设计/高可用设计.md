# 高可用设计

## 这个问题为什么存在？

> 系统不可用，用户就流失。但**100%可用是神话**，问题是：**怎么在可控成本下，最大化系统可用性？**

没有高可用设计，你会遇到：
- 服务挂了，所有用户无法使用（单点故障）
- 流量突增，系统雪崩（没有限流）
- 新版本有bug，全量用户受影响（没有灰度）
- 网络抖动，用户投诉"卡"（没有超时+重试）

---

## 它是怎么解决问题的？

### 高可用的核心公式

```
可用性 = 正常运行时间 / (正常运行时间 + 故障时间)

99.9%      → 一年宕机8.76小时
99.99%     → 一年宕机52.6分钟
99.999%    → 一年宕机5.26分钟（挑战物理极限）
```

### 五大核心手段

```
手段1：限流（Rate Limit）
  → 控制QPS，防止系统过载

手段2：降级（Degrade）
  → 关闭非核心功能，保核心功能

手段3：幂等性（Idempotency）
  → 重试不导致重复操作

手段4：超时与重试（Timeout & Retry）
  → 快速失败，释放资源

手段5：灰度发布（Canary Release）
  → 新版本先给小部分用户用，没问题再全量
```

### 高可用架构模式

#### 1. 主从模式（Master-Slave）

```
主库（写） → 同步/异步 → 从库（读）
  │
  └─ 主库挂了 → 从库晋升（故障转移）
```

**适用**：数据库（MySQL主从）、Redis主从

#### 2. 集群模式（Cluster）

```
负载均衡（Nginx/LVS）
       │
  ┌────┴────┐
  实例1   实例2   实例3
  （都跑相同代码，无状态）
```

**适用**：应用层（Tomcat、Spring Boot）

#### 3. 异地多活（Multi-Site Active-Active）

```
北京机房（主） ← 数据同步 → 上海机房（备）
  │                              │
  └── 北京挂了 ── 流量切上海 ──┘
```

**适用**：金融、电商（可用性要求极高）

## 它和相似方案的本质区别是什么？

| 手段 | 目的 | 触发条件 | 处理方式 |
|---|---|---|---|
| **限流** | 控制QPS | QPS > 阈值 | 拒绝/排队 |
| **降级** | 保核心功能 | 系统资源不足 | 关闭非核心功能 |
| **熔断** | 保护调用方 | 下游错误率 > 阈值 | 快速失败，不发起调用 |
| **超时** | 防止资源耗尽 | 调用超时 | 释放资源，快速失败 |
| **重试** | 容忍临时故障 | 网络抖动 | 等待后重试（指数退避） |

**本质区别**：
- **限流**："我处理能力有限，多来的我不接了"
- **降级**："资源不够，我关掉非核心功能"
- **熔断**："下游挂了，我不调了"
- **超时+重试**："调用慢，但我不能一直等"

## 正确使用方式

### 高可用设计的"三板斧"

```
第1板：限流（入口挡流量）
  → 令牌桶/漏桶，控制QPS

第2板：降级（资源不够时保核心）
  → 关闭推荐、评论等非核心功能

第3板：容错（下游挂了不影响我）
  → 熔断、超时、重试
```

### 监控 + 告警（高可用的眼睛）

```
指标1：QPS（每秒请求数）
  → 限流阈值参考

指标2：响应时间（P95、P99）
  → 超时时间设置参考

指标3：错误率（%）
  → 熔断触发条件

指标4：可用性（SLA）
  → 99.9%？99.99%？
```

## 边界情况和坑

### 坑1：限流阈值设太高，系统还是挂了

```
场景：限流阈值设10000 QPS，但系统只能处理5000 QPS
后果：限流没生效，系统还是被压垮
```

**解决方案**：
1. **压测**（知道系统的最大处理能力）
2. **限流阈值 < 系统最大处理能力**（留有余地）
3. **多级限流**（入口限流 + 接口限流）

### 坑2：降级了，但没恢复

```
场景：系统压力大，降级了非核心功能
      压力下来后，没自动恢复
后果：用户一直看不到推荐、评论
```

**解决方案**：
1. **降级手动 + 恢复手动**（避免自动恢复导致再次雪崩）
2. **监控压力，人工判断恢复时机**

### 坑3：灰度发布，但数据库不兼容

```
场景：v1.1改了表结构（加了字段）
      v1.0还在跑，读老字段 → 报错
```

**解决方案**：
1. **灰度前，保证向前兼容**（v1.1先加字段，v1.0还能跑）
2. **灰度顺序**：先升级DB Schema，再灰度应用
