## 参考要点（盲答后再看）


**进程 vs 线程**：

```
进程（Process）：
  - 资源分配单位（内存、文件句柄、信号...）
  - 独立地址空间（一个进程崩溃不影响其他进程）
  - 切换开销大（要切换页表、刷新 TLB）

线程（Thread）：
  - 调度执行单位（CPU 调度的最小单位）
  - 同一进程的线程共享地址空间（堆、方法区、文件句柄...）
  - 各自独立：程序计数器、虚拟机栈、本地方法栈
  - 切换开销小（不需要切换地址空间）
```

**Java 线程模型（1:1）**：

```
Java Thread  → 映射到 → OS Thread（内核态线程）
  - 优点：利用多核，操作系统自动调度
  - 缺点：创建/切换成本高（每次都要进内核）

对比其他语言：
  - Go：M:N 模型（M 个协程映射到 N 个线程，N ≈ CPU 核数）
  - Java 19 之前：只有 1:1
  - Java 19+：Virtual Thread（M:N）
```

**Virtual Thread（Java 19+）**：

```
传统线程：
  - 每个 Thread = 1 个 OS Thread（约 1MB 栈内存）
  - 创建 10 万个 → 100GB 内存（不可能）

Virtual Thread：
  - 由 JVM 调度（用户态），不绑定固定 OS Thread
  - 阻塞时（如 I/O），JVM 自动把 Virtual Thread 从 OS Thread 摘下来
  - 10 万个 Virtual Thread → 只需几百个 OS Thread（内存几 MB）
  - 适合：I/O 密集型应用（Web 服务器）
```

**协程 = 用户态线程**：

```
线程阻塞：
  1. 应用调用 read() → 进入内核
  2. 内核把线程标记为「阻塞」
  3. 上下文切换到其他线程
  4. I/O 完成 → 内核唤醒线程

协程挂起：
  1. 协程调用 async_read() → 应用层自己处理（不进内核）
  2. 应用层调度器把 CPU 让给其他协程
  3. I/O 完成 → 应用层调度器恢复协程
  → 全程在内核态之外（快！）
```
