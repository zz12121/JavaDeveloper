# Redis持久化机制

## 这个问题为什么存在？

> Redis 是内存数据库，数据全在内存里。断电、重启、崩溃——内存中的数据全部丢失。问题是：**怎么让内存数据在 Redis 退出后不丢失，同时又不让持久化本身成为性能瓶颈？**

这本质上是在两个目标之间找平衡：**数据安全** vs **读写性能**。

## 它是怎么解决问题的？

### 两种持久化策略对比

```
┌─────────────────────────────────────────────────────────────┐
│                         Redis 进程                          │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────┐  │
│  │  写操作      │→ │  内存数据      │→ │  RDB 文件（快照）  │  │
│  │  所有修改命令 │  │  (dict + 过期)│  │  AOF 文件（日志） │  │
│  └──────────────┘  └──────────────┘  └──────────────────┘  │
│                                    ↕                       │
│                              恢复时加载                      │
└─────────────────────────────────────────────────────────────┘
```

**RDB**：定时生成全量快照，存到磁盘。恢复快，但两次快照之间数据可能丢失。
**AOF**：记录每条写命令，恢复时重放命令。数据几乎不丢，但文件大、重放慢。

### RDB：快照持久化

#### 工作原理

RDB 通过 `BGSAVE` 或 `SAVE` 命令触发，核心是** fork 子进程**：

```
主进程 fork() → 子进程
    ↓
子进程遍历内存，写入临时 .rdb 文件
    ↓
写完后用原子 rename 替换旧 .rdb 文件
    ↓
子进程退出
```

`fork()` 是**写时复制（COW）**：子进程共享父进程内存页，只有父进程修改某个内存页时才复制一份。所以 fork 开销主要是遍历内存结构的指针，大小固定，不会翻倍占用内存。

#### 触发方式

```bash
# 手动触发
SAVE  # 阻塞主进程（不推荐）
BGSAVE  # fork 子进程后台执行

# 自动触发（配置文件）
save 900 1      # 900秒内 ≥1 次写操作 → 触发 BGSAVE
save 300 10     # 300秒内 ≥10 次写操作
save 60 10000   # 60秒内 ≥10000 次写操作
```

#### RDB 的优缺点

| 优点 | 缺点 |
|---|---|
| 全量快照，恢复速度快（毫秒级） | fork 需要时间，大数据集上可能卡顿 |
| 文件紧凑（COPY-on-Write 期间只有变化的页） | 最后一次快照到崩溃之间的数据丢失 |
| 适合做灾备、冷备份 | fork() 在大数据集上内存压力较大 |
| 更紧凑，适合日志、统计等定期归档 | 单进程，无法做到实时持久化 |

### AOF：命令日志持久化

#### 工作原理

AOF 把每条写命令追加到 `.aof` 文件末尾：

```
SET name "zhang"  →  append 到 aof 文件
INCR age         →  append 到 aof 文件
HSET user name   →  append 到 aof 文件
```

恢复时重放所有命令，恢复完整数据集。

#### 三种 fsync 策略

**`appendfsync` 的取值决定了数据安全级别**：

```bash
# always：每条命令都 fsync（刷盘）
# → 最多丢 1 条命令，数据最安全，但性能最差
appendfsync always

# everysec：每秒 fsync 一次（默认）
# → 最多丢 1 秒数据，性能和数据安全的折中
appendfsync everysec

# no：让操作系统决定什么时候 fsync
# → 不丢命令（交给系统），但系统崩溃可能丢大量数据
appendfsync no
```

**`fsync` 是 AOF 性能的决定因素**：每条命令都要写文件，但刷盘是独立的。`everysec` 之所以能做到"每秒万级 QPS"，是因为**写文件是异步的**，主线程只负责把命令追加到缓冲区，刷盘由后台线程每秒执行一次。

#### AOF 重写（rewrite）

随着命令积累，AOF 文件越来越大。重写机制：**把内存中的数据直接写成新的 SET 命令**，跳过历史命令。

```
# 重写前（1万条 INCR）
INCR counter  →  10000条记录，文件 200KB
INCR counter
...

# 重写后（1条）
SET counter 10000  →  1条记录，文件 1KB
```

重写也是 **fork 子进程** 做的，不阻塞主进程。

```bash
# 手动触发
BGREWRITEAOF

# 自动触发（当 AOF 文件大于指定大小时）
auto-aof-rewrite-percentage 100  # AOF 文件比上次重写后大一倍
auto-aof-rewrite-min-size 64mb  # 最小 64MB 才触发
```

### RDB + AOF 混合持久化

Redis 4.0 引入混合模式，解决两者的痛点：

```bash
aof-use-rdb-preamble yes
```

工作方式：
```
RDB 部分（开头）：fork 子进程生成 RDB 格式的全量快照
  ↓
AOF 部分（后面）：后续命令以 AOF 格式追加
```

恢复时：Redis 识别到开头是 RDB 格式，先加载 RDB 快速恢复，再重放后面的 AOF 命令补齐。兼顾了 **快速恢复（RDB）** 和 **低数据丢失（AOF）**。

## 它和相似方案的本质区别是什么？

| 策略 | 数据完整性 | 性能 | 恢复速度 | 适用场景 |
|---|---|---|---|---|
| RDB | 中（最多丢一次快照间隔的数据） | 高 | 快 | 灾备、冷备份、关机器前快照 |
| AOF everysec | 高（最多丢1秒） | 中 | 慢（重放所有命令） | 数据重要、可以接受稍慢恢复 |
| AOF always | 极高（最多丢1条） | 低 | 慢 | 数据极度重要 |
| 混合持久化 | 高（结合两者） | 中 | 快 | Redis 4.0+ 推荐 |

### 和 Memcached 持久化方案对比

Memcached **没有内置持久化**，纯内存。Redis 通过 AOF/RDB 提供了可选持久化，这是两者定位不同的根本原因——Redis 不只是缓存，还是数据结构服务器。

## 正确使用方式

### 生产环境推荐配置

```bash
# 混合持久化（Redis 4.0+）
appendonly yes
aof-use-rdb-preamble yes
appendfsync everysec

# RDB 定时快照
save 900 1
save 300 10
save 60 10000

# AOF 重写
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb

# RDB 压缩
rdbcompression yes
```

### 不要用 `SAVE` 在生产环境

`SAVE` 阻塞主进程，所有客户端请求都要等。这在生产环境会导致服务不可用。一定要用 `BGSAVE`。

### AOF 和 RDB 的恢复顺序

```
Redis 启动
    ↓
有 AOF 文件？→ YES → 只加载 AOF（包含混合格式的话自动处理）
    ↓ NO
只有 RDB 文件？→ 加载 RDB
```

Redis 优先用 AOF 恢复，因为 AOF 数据更完整。如果同时存在，先加载 AOF。

## 边界情况和坑

### `fork()` 在大内存机器上的问题

Redis 在 32GB 内存的机器上 fork 一个 28GB 的进程，fork 本身会阻塞数十秒（取决于内存大小）。这是 `BGSAVE` 在大数据集上可能卡顿的原因。

解决方案：
- 控制 Redis 使用内存不超过机器物理内存的一半
- 使用 Linux 的 `THP`（Transparent Huge Pages）可以减少 fork 开销，但有副作用（建议关闭）
- 大数据量场景考虑 Redis Cluster 分片

### AOF 文件损坏

AOF 文件可能被截断（Redis 崩溃时正在写盘）。Redis 有修复机制：

```bash
redis-check-aof --fix appendonly.aof
```

修复方式：删除文件末尾不完整的命令。**这可能导致少量数据丢失**。

### RDB 和 AOF 同时开，内存占用翻倍？

不会。RDB fork 时子进程共享父进程内存（COW），实际内存增长是修改的页面大小，不是整个数据集。AOF 的写命令在缓冲区里，不会再复制一份内存数据。

### 子进程和父进程的内存关系

```
fork() 之后：
  父进程内存 ← 共享（COW，只有修改的页才复制）
  子进程内存 ← 共享（COW）

随着数据修改：
  父子进程的共同页面被复制，各自在独立页面修改
  峰值内存 = 父进程修改的页面 + 子进程修改的页面
  通常总内存峰值 ≈ 2x 原始数据（极端情况）
```
