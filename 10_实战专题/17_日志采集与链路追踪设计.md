# 日志采集与链路追踪设计

> 微服务架构下，一次请求可能经过 5~10 个服务。出了问题怎么快速定位？日志分散在几十台机器上，怎么关联？线上报了一个 500 错误，如何从日志中还原完整的调用链路？

---

## 题目一：日志采集与链路追踪设计

### 业务背景

```
微服务调用链示例：
  用户请求 → 网关 → 用户服务 → 订单服务 → 库存服务 → 支付服务 → 通知服务
  任何一个环节出问题，用户看到的都是"系统异常"

排查痛点：
  1. 日志散落在各服务各自的日志文件中，没有统一入口
  2. 一个请求跨 5 个服务，日志时间戳对不上（各机器时钟偏差）
  3. 无法区分哪些日志属于同一次请求
  4. 慢请求排查困难 —— 不知道瓶颈在哪个服务、哪个方法
  5. 错误日志被海量 INFO 日志淹没
  6. 日志量太大，存储成本高（日均 TB 级别）
  7. 日报/周报需要手动统计错误率，效率低

目标：
  - 一次请求的完整调用链路可视化
  - 从入口到出口，每一步耗时一目了然
  - 错误日志自动关联到调用链路
  - 慢请求自动发现和告警
  - 日志统一查询入口（按 TraceId/服务名/时间范围）
```

### 方案对比

| 方案 | 技术栈 | 优势 | 劣势 | 适用场景 |
|------|--------|------|------|---------|
| ELK Stack | Elasticsearch + Logstash + Kibana | 生态成熟、查询能力强 | 资源消耗大、ES 集群运维成本高 | 中大型企业，已有 ES 基础设施 |

| PLG Stack | Loki + Promtail + Grafana | 轻量、和 Prometheus 生态整合 | 日志全文搜索能力弱 | 日志量中等、K8s 环境 |

### 方案详解 A：TraceId 全链路透传（基础设施）

#### TraceId 生成与透传

```
请求链路：
  Client → Gateway → Service A → Service B → Service C

TraceId 在网关生成，通过 HTTP Header 在服务间透传：
  ┌─────────┐  X-Trace-Id: abc123  ┌───────────┐  X-Trace-Id: abc123  ┌───────────┐
  │ Gateway  │ ───────────────────→ │ Service A │ ───────────────────→ │ Service B │
  │  生成     │  X-Span-Id: span-1  │           │  X-Span-Id: span-2  │           │
  │ abc123   │  X-Parent-Span: -   │           │  X-Parent-Span: 1   │           │
  └─────────┘                      └───────────┘                      └───────────┘

每个服务内部的日志自动带上 TraceId + SpanId：
  [abc123][span-1] INFO  Gateway   - 收到请求 /api/order
  [abc123][span-2] INFO  ServiceA  - 开始处理订单, userId=12345
  [abc123][span-3] INFO  ServiceB  - 扣减库存, skuId=67890
  [abc123][span-3] ERROR ServiceB  - 库存不足异常
  [abc123][span-2] ERROR ServiceA  - 调用库存服务失败
```

#### TraceId 过滤器实现

```java
/**
 * Servlet Filter —— 从请求头中获取或生成 TraceId，放入 MDC
 * MDC（Mapped Diagnostic Context）是 Logback 的线程上下文，
 * 在当前线程的所有日志中自动追加 MDC 中的变量
 */
@Component
@Order(Ordered.HIGHEST_PRECEDENCE)
public class TraceFilter implements Filter {

    private static final String TRACE_ID_HEADER = "X-Trace-Id";
    private static final String SPAN_ID_HEADER = "X-Span-Id";
    private static final String PARENT_SPAN_HEADER = "X-Parent-Span-Id";

    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
                         FilterChain chain) throws IOException, ServletException {
        HttpServletRequest httpRequest = (HttpServletRequest) request;
        HttpServletResponse httpResponse = (HttpServletResponse) response;

        // 获取或生成 TraceId
        String traceId = httpRequest.getHeader(TRACE_ID_HEADER);
        if (traceId == null || traceId.isEmpty()) {
            traceId = generateTraceId();
        }

        // 生成当前 SpanId
        String spanId = generateSpanId();
        String parentSpanId = httpRequest.getHeader(PARENT_SPAN_HEADER);

        // 放入 MDC，后续所有日志自动携带
        MDC.put("traceId", traceId);
        MDC.put("spanId", spanId);
        if (parentSpanId != null) {
            MDC.put("parentSpanId", parentSpanId);
        }

        // 设置响应头，方便前端排查
        httpResponse.setHeader(TRACE_ID_HEADER, traceId);

        try {
            chain.doFilter(request, response);
        } finally {
            // 请求结束清空 MDC（防止线程池复用导致 MDC 泄漏）
            MDC.clear();
        }
    }

    /**
     * 生成 TraceId：时间戳 + 机器标识 + 序列号
     * 长度固定 32 位，方便日志检索
     */
    private String generateTraceId() {
        String timestamp = Long.toHexString(System.currentTimeMillis());
        String machineId = Long.toHexString(getMachineId() & 0xFFFF);
        String sequence = Long.toHexString(sequence.incrementAndGet() & 0xFFFF);
        return String.format("%s%s%04s%s%04s", timestamp, machineId,
                machineId, sequence, sequence);
    }
}
```

#### Feign/RPC 调用自动透传 TraceId

```java
/**
 * Feign 拦截器 —— 在调用下游服务时自动透传 TraceId 和 SpanId
 */
@Component
public class FeignTraceInterceptor implements RequestInterceptor {

    @Override
    public void apply(RequestTemplate template) {
        String traceId = MDC.get("traceId");
        String spanId = MDC.get("spanId");

        if (traceId != null) {
            template.header("X-Trace-Id", traceId);
            template.header("X-Parent-Span-Id", spanId);
            template.header("X-Span-Id", generateSpanId()); // 新 Span
        }
    }
}

/**
 * RestTemplate 拦截器
 */
@Component
public class RestTemplateTraceInterceptor implements ClientHttpRequestInterceptor {

    @Override
    public ClientHttpResponse intercept(HttpRequest request, byte[] body,
                                         ClientHttpRequestExecution execution) throws IOException {
        String traceId = MDC.get("traceId");
        String spanId = MDC.get("spanId");
        if (traceId != null) {
            request.getHeaders().set("X-Trace-Id", traceId);
            request.getHeaders().set("X-Parent-Span-Id", spanId);
            request.getHeaders().set("X-Span-Id", generateSpanId());
        }
        return execution.execute(request, body);
    }
}
```

#### 异步线程 TraceId 传递

```java
/**
 * 问题：@Async 或线程池中，MDC 会丢失（子线程拿不到父线程的 TraceId）
 *
 * 解决：自定义 TaskDecorator，在线程池任务提交时复制 MDC
 */
@Component
public class MdcTaskDecorator implements TaskDecorator {

    @Override
    public Runnable decorate(Runnable runnable) {
        // 在父线程中捕获当前 MDC
        Map<String, String> contextMap = MDC.getCopyOfContextMap();
        return () -> {
            try {
                // 在子线程中恢复 MDC
                if (contextMap != null) {
                    MDC.setContextMap(contextMap);
                }
                runnable.run();
            } finally {
                MDC.clear();
            }
        };
    }
}

// 配置线程池
@Configuration
public class AsyncConfig {

    @Bean
    public Executor asyncExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(8);
        executor.setMaxPoolSize(16);
        executor.setQueueCapacity(1000);
        executor.setTaskDecorator(new MdcTaskDecorator()); // 关键！
        executor.initialize();
        return executor;
    }
}
```

#### 自定义 Span（手动埋点）

```java
/**
 * SkyWalking 自动拦截 Spring MVC、Dubbo、Feign、MyBatis、Redis 等调用
 * 但某些业务逻辑需要手动埋点，比如批量处理、定时任务
 */
@Trace
@TraceCrossThread  // 跨线程传播
@Service
public class OrderProcessService {

    /**
     * @Trace 注解标记的方法会被自动追踪
     * @Tags 注解添加业务标签，方便后续查询
     */
    @Trace(operationName = "processOrderBatch")
    @Tags({
        @Tag(key = "batch.size", value = "arg[0]"),
        @Tag(key = "batch.type", value = "arg[1]")
    })
    public void processBatch(int size, String type) {
        // 自动被追踪
    }

    /**
     * 编程式手动创建 Span（更灵活）
     */
    public void processSingleOrder(Long orderId) {
        // 创建一个 Span
        AbstractSpan span = ContextManager.createSpan("processSingleOrder");
        span.tag("orderId", String.valueOf(orderId));

        try {
            // 业务逻辑
            orderMapper.updateStatus(orderId, OrderStatus.PROCESSING);

            // 创建子 Span
            AbstractSpan childSpan = ContextManager.createSpan("validateOrder");
            childSpan.tag("orderId", String.valueOf(orderId));
            try {
                validateOrder(orderId);
            } finally {
                childSpan.end();
            }

        } catch (Exception e) {
            span.errorOccurred(); // 标记异常
            span.log(e);
            throw e;
        } finally {
            span.end();
        }
    }
}
```

### 日志采集架构（ELK）

#### 日志采集流程

```
应用服务                    采集层                    存储层                    查询层
┌──────────┐           ┌──────────┐             ┌────────────┐           ┌──────────┐
│ Service A │──.log──→│ Filebeat │──→ JSON ──→ │            │           │          │
├──────────┤           ├──────────┤             │            │  查询     │ Kibana   │
│ Service B │──.log──→│ Filebeat │──→ JSON ──→ │ Elasticsearch│←────────│ 或       │
├──────────┤           ├──────────┤             │  (3节点集群) │           │ Grafana  │
│ Service C │──.log──→│ Filebeat │──→ JSON ──→ │            │           │          │
├──────────┤           └──────────┘             └────────────┘           └──────────┘
│ Gateway   │──.log──→│ Logstash │──→ 过滤/格式化 → ES
└──────────┘           └──────────┘
  每台机器部署             轻量采集器               分布式存储                统一查询

关键设计点：
  1. Filebeat 采集日志 → 发给 Logstash（可选）或直接写 ES
  2. Logstash 做日志解析：去敏感字段、提取结构化字段、GeoIP 解析
  3. ES 索引按天滚动：app-logs-2026.05.02，方便按天清理
  4. 索引生命周期管理（ILM）：热数据 7 天 → 温数据 30 天 → 冷数据 90 天 → 删除
```

### 日志分级存储与成本优化

```
日志分级：
  INFO 级别    → 保留 3 天（量最大，排查价值低）
  WARN 级别    → 保留 7 天
  ERROR 级别   → 保留 30 天（排查价值最高）
  慢请求日志    → 保留 14 天
  链路追踪数据  → 保留 7 天

成本优化策略：
  1. 采样：非核心服务采样率 10%，核心服务 100%
  2. 日志压缩：ES 索引使用 best_compression
  3. 冷热分离：3天内热节点 SSD，3~30天温节点 HDD，30天+归档或删除
  4. 日志量控制：限制单条日志最大 10KB，避免大对象 JSON 打印
  5. 关键路径：打印出入参摘要，不要打印完整的请求体
```

### 异常链路自动发现与告警

```java
/**
 * 全局异常处理器 —— 统一拦截异常并记录到追踪系统
 */
@RestControllerAdvice
@Slf4j
public class GlobalExceptionHandler {

    @Autowired
    private AlertService alertService;

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleException(Exception e) {
        String traceId = MDC.get("traceId");

        // 记录异常日志（自动带 TraceId）
        log.error("[Unhandled] traceId={}, exception={}", traceId, e.getMessage(), e);

        // 根据异常类型决定告警级别
        AlertLevel level = classifyAlertLevel(e);

        // 发送告警
        alertService.sendAlert(AlertEvent.builder()
                .traceId(traceId)
                .level(level)
                .message(e.getMessage())
                .exceptionClass(e.getClass().getSimpleName())
                .stackTrace(getStackTrace(e))
                .build());

        return ResponseEntity.status(500)
                .body(new ErrorResponse(traceId, "系统异常，请联系管理员"));
    }

    private AlertLevel classifyAlertLevel(Exception e) {
        if (e instanceof HttpRequestMethodNotSupportedException) {
            return AlertLevel.LOW; // 客户端错误，低优先级
        }
        if (e instanceof FeignException.ServiceUnavailable) {
            return AlertLevel.HIGH; // 下游服务不可用，高优先级
        }
        if (e instanceof DeadlockLoserDataAccessException) {
            return AlertLevel.CRITICAL; // 数据库死锁，紧急
        }
        return AlertLevel.MEDIUM;
    }
}
```

### 高频追问

#### Q1：TraceId 在消息队列（MQ）中如何传递？

```
场景：Service A → RocketMQ → Service B
MQ 消息没有 HTTP Header，TraceId 怎么传？

方案 1（RocketMQ）：将 TraceId 放入 Message 的 Properties
  Message msg = new Message("topic", "tag", body);
  msg.putUserProperty("traceId", MDC.get("traceId"));
  msg.putUserProperty("spanId", MDC.get("spanId"));

  消费者从 Properties 中取出：
  String traceId = msg.getProperty("traceId");
  MDC.put("traceId", traceId);

方案 2（Spring Cloud Stream）：在消息拦截器中自动注入
  @Bean
  public ProducerInterceptor producerInterceptor() {
      return message -> {
          message.getHeaders().put("traceId", MDC.get("traceId"));
          return message;
      };
  }
```

#### Q2：日志量太大，ES 存不下来怎么办？

```
真实数据量参考：
  50 个微服务 × 每个服务日均 5GB 日志 = 250GB/天
  一个月 = 7.5TB → 3 个副本 = 22.5TB 存储

优化策略：
1. 减少日志量
   - 关闭不必要的框架 DEBUG 日志
   - 避免循环中打日志
   - 大对象只打摘要（ID + 关键字段），不全量 toString

2. 日志分级存储（最有效）
   - INFO 日志 → 保留 3 天（ES 热节点）
   - WARN/ERROR → 保留 30 天
   - 3 天后的 INFO → 归档到对象存储（OSS/S3）

3. 采样
   - 健康检查、心跳等高频无价值日志不采集
   - 链路追踪按比例采样（如 10%）
   - ERROR 级别全量采集

4. ES 索引优化
   - 关闭不需要的字段（_source）
   - 使用 keyword 类型而非 text
   - 索引模板设置 number_of_replicas=1（非关键日志）
```

#### Q3：多机房部署时日志如何集中管理？

```
方案：每个机房独立部署 Filebeat + Kafka → 集中到总部的 ES 集群

  机房 A                     机房 B                     总部
  ┌────────┐                ┌────────┐                ┌────────┐
  │ App    │→ Filebeat →    │ App    │→ Filebeat →    │ Kafka  │→ Logstash → ES
  │ Servers│   Kafka A      │ Servers│   Kafka B      │ (聚合) │
  └────────┘                └────────┘                └────────┘

关键点：
  - 机房间的数据传输用 Kafka（高吞吐、断点续传）
  - 机房网络中断时本地 Kafka 积压，恢复后自动补传
  - ES 集群只在总部部署（跨机房 ES 集群延迟太大）
  - 机房ID作为日志字段，方便按机房筛选
```

### 生产 Checklist

```
□ TraceId 是否在所有入口（HTTP/MQ/RPC/定时任务）中正确生成
□ TraceId 是否在 Feign/Dubbo/MQ/线程池中正确透传
□ MDC 是否在请求结束后 clear（防止线程池复用泄漏）
□ 日志格式是否统一（JSON 格式 + MDC 字段）
□ 日志是否有分级存储策略（ERROR 保留时间 > INFO）
□ 敏感信息（手机号/身份证/密码）是否脱敏
□ 链路追踪 Agent 是否有性能影响监控（CPU/内存/RT）
□ ES 索引是否有生命周期管理（ILM）避免磁盘满
□ 错误日志是否自动关联到链路追踪
□ 告警规则是否覆盖核心 SLI（成功率/P99/错误率）
□ 日志采集是否有降级策略（ES 宕机时不能影响业务）
□ 是否有 TraceId 的前端展示（用户反馈问题时能提供 TraceId）
□ 日志查询是否有权限控制（开发只能看自己负责的服务）
```
