## 参考要点（盲答后再看）


**堆内 vs 堆外**：

| 维度 | 堆内（HeapByteBuffer） | 堆外（DirectByteBuffer） |
|------|------------------------|--------------------------|
| 分配开销 | 小 | 大（需系统调用） |
| GC 管理 | ✅ 自动回收 | ❌ GC 管不到（需 Cleaner） |
| IO 性能 | 差（需拷贝到堆外再 IO） | 好（直接和 Channel 交互） |
| 容量限制 | -Xmx | -XX:MaxDirectMemorySize |

**DirectByteBuffer 分配流程**：
```java
// 简化
ByteBuffer.allocateDirect(capacity);
  → Bits.reserveMemory()       // 检查 MaxDirectMemorySize
  → Unsafe.allocateMemory()  // 系统调用 mmap / malloc
  → 创建 Cleaner（PhantomReference，GC 时触发释放）
```

**Cleaner 机制（关键）**：
- DirectByteBuffer 里有一个 `Cleaner` 对象（PhantomReference 的子类）
- GC 发现 DirectByteBuffer 只有 Cleaner 引用 → 把 Cleaner 放入 ReferenceQueue
- ReferenceHandler 线程执行 `Cleaner.clean()` → 调用 `Unsafe.freeMemory()`

**Netty 堆外内存的原因**：
```
堆内 ByteBuf：
  socket.write(堆内Buf) 
    → 先拷贝到堆外临时缓冲区（JVM 保证）
    → 再发给内核
    → 两次拷贝 ❌

堆外 ByteBuf：
  socket.write(堆外Buf)
    → 直接发给内核
    → 一次拷贝 ✅
```

**OOM 排查（Direct buffer memory）**：
1. 检查 `-XX:MaxDirectMemorySize`（默认等于 -Xmx）
2. 检查是否忘了 `ByteBuffer.clear()` 或 Netty 的 `ByteBuf.release()`
3. 用 `jcmd <pid> VM.native_memory` 查看 Native Memory 占用

**MaxDirectMemorySize 默认值**：等于 -Xmx（JDK8 之后）
