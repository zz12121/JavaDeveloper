# 消息通知中心设计

> 公司有 App 推送、短信、邮件、站内信、钉钉/企微机器人等多个通知渠道，各业务线各自对接，发送逻辑散落各处。需要一个统一的通知中心：统一接入、渠道适配、发送优先级管理、限流防刷、送达状态追踪。

---

## 业务背景

```
现状痛点：
  订单服务自己调短信 SDK → 会员服务也自己调 → 积分商城也自己调
  → 重复造轮子，每条通知逻辑各写一遍
  → 短信渠道变更（换供应商）→ 改 10 个服务
  → 发送失败没人知道，用户说「没收到验证码」时才排查
  → 营销短信发了 100 万条，运营商限速，正常短信被挤占
  → 没有防刷机制，恶意用户疯狂触发短信接口

需求：
  1. 统一接入：业务线只调一个 API，不管底层用什么渠道发
  2. 模板管理：通知内容模板化，支持变量替换
  3. 优先级：交易通知 > 安全通知 > 营销通知
  4. 限流：防刷（单用户 N 条/分钟）+ 渠道限速（运营商限制）
  5. 送达状态：追踪每条通知的投递状态（已发送/送达/失败/退订）
  6. 发送记录：审计、对账、用户投诉时追溯
```

## 核心设计二：统一发送接口

```java
// ===== 业务线调用的统一接口 =====
@Service
public class NotificationService {

    @Autowired
    private NotificationDispatcher dispatcher;

    @Autowired
    private RateLimitService rateLimitService;

    @Autowired
    private TemplateService templateService;

    /**
     * 发送通知（同步，适合验证码等需要即时发送的场景）
     */
    public SendResult send(String templateCode, String receiver, Map<String, String> variables) {
        return send(templateCode, receiver, variables, null);
    }

    /**
     * 发送通知（支持指定优先级和延迟发送）
     */
    public SendResult send(String templateCode, String receiver,
                           Map<String, String> variables, SendOption option) {
        // 1. 限流检查
        if (!rateLimitService.tryAcquire(templateCode, receiver)) {
            return SendResult.rejected("发送频率超限，请稍后再试");
        }

        // 2. 获取模板
        Template template = templateService.getByCode(templateCode);
        if (template == null || template.getStatus() == 0) {
            return SendResult.failed("模板不存在或已禁用");
        }

        // 3. 渲染内容（变量替换）
        String content = renderTemplate(template.getContent(), variables);

        // 4. 构建消息
        NotificationMessage msg = NotificationMessage.builder()
            .msgId(generateMsgId())
            .templateCode(templateCode)
            .channel(template.getChannel())
            .receiver(receiver)
            .content(content)
            .priority(option != null ? option.getPriority() : template.getPriority())
            .bizType(template.getBizType())
            .build();

        // 5. 分发发送
        return dispatcher.dispatch(msg);
    }

    /**
     * 异步批量发送（适合营销推送、运营通知等）
     */
    public void sendAsync(String templateCode, List<String> receivers,
                          Map<String, String> variables) {
        for (String receiver : receivers) {
            // 发到 MQ，异步处理
            mqTemplate.send("notification-async-topic",
                new AsyncSendMsg(templateCode, receiver, variables));
        }
    }

    private String renderTemplate(String template, Map<String, String> variables) {
        String result = template;
        for (Map.Entry<String, String> entry : variables.entrySet()) {
            result = result.replace("${" + entry.getKey() + "}", entry.getValue());
        }
        return result;
    }
}
```

## 核心设计三：限流防刷

```
限流层级：
┌─────────────────────────────────────────────────┐
│  全局限流：所有通知渠道总 QPS 上限                   │
│  例：短信 500 条/s，邮件 100 条/s                  │
├─────────────────────────────────────────────────┤
│  用户级限流：单用户单渠道频率限制                     │
│  例：同手机号 1 条/60s，同邮箱 5 条/分钟             │
├─────────────────────────────────────────────────┤
│  业务级限流：单业务线每日发送量限制                   │
│  例：营销短信 10 万条/天，验证码无限制               │
├─────────────────────────────────────────────────┤
│  渠道级限流：各运营商/供应商的速率限制                 │
│  例：阿里云短信签名级 200 条/s                      │
└─────────────────────────────────────────────────┘
```

```java
@Component
public class RateLimitService {

    @Autowired
    private StringRedisTemplate redis;

    // 各模板的限流规则（实际应放配置中心）
    private static final Map<String, RateRule> RULES = Map.of(
        "LOGIN_SMS",       new RateRule(1, 60, 10, 3600),   // 1条/分钟, 10条/小时
        "REGISTER_SMS",    new RateRule(1, 60, 5, 86400),    // 1条/分钟, 5条/天
        "ORDER_SMS",       new RateRule(5, 60, 50, 3600),    // 5条/分钟, 50条/小时
        "MARKETING_SMS",   new RateRule(3, 60, 20, 86400),   // 3条/分钟, 20条/天
        "SECURITY_SMS",    new RateRule(3, 60, 30, 3600)     // 3条/分钟, 30条/小时
    );

    /**
     * 滑动窗口限流
     */
    public boolean tryAcquire(String templateCode, String receiver) {
        RateRule rule = RULES.get(templateCode);
        if (rule == null) return true;  // 没有配置规则，不限流

        String key = "notify:limit:" + templateCode + ":" + receiver;
        long now = System.currentTimeMillis();

        // 用 Redis sorted set 实现滑动窗口
        // score = 时间戳, value = 请求ID
        redis.opsForZSet().add(key, UUID.randomUUID().toString(), now);

        // 移除窗口外的记录
        redis.opsForZSet().removeRangeByScore(key, 0, now - rule.getWindowMs());

        // 统计窗口内的数量
        Long count = redis.opsForZSet().count(key, 0, now);

        // 设置过期时间，防止 key 永远存在
        redis.expire(key, rule.getWindowMs() / 1000 + 10, TimeUnit.SECONDS);

        return count != null && count <= rule.getMaxCount();
    }

    @Data
    @AllArgsConstructor
    private static class RateRule {
        private int maxCount;       // 窗口内最大数量
        private int windowSeconds;  // 窗口时长（秒）
        // 以下是日级限流的备用字段
        @SuppressWarnings("unused")
        private int dailyMax;       // 日最大数量
        @SuppressWarnings("unused")
        private int dailyWindow;    // 日窗口时长（秒）
        private long getWindowMs() { return windowSeconds * 1000L; }
    }
}
```

## 核心设计四：优先级队列分发

```
为什么要分优先级？
  短信通道有限（运营商限制 500 条/s）
  如果营销短信把通道占满了
  → 用户的验证码发不出去 → 用户无法登录 → 事故

队列设计：
  ┌───────────────────┐
  │  MQ Topic         │
  │                   │
  │  3 个消费者组      │
  │  ├─ P1 交易通知组  │  ← 并发 50，最高优先
  │  ├─ P2 安全通知组  │  ← 并发 30
  │  └─ P3 营销通知组  │  ← 并发 10，最低优先
  └───────────────────┘
```

```java
@Component
@RocketMQMessageListener(
    topic = "notification-async-topic",
    consumerGroup = "notify-priority-1",  // 交易通知消费者组
    consumeMode = ConsumeMode.CONCURRENTLY,
    consumeThreadMax = 50                 // 高并发
)
public class TransactionNotifyConsumer implements RocketMQListener<String> {

    @Autowired
    private ChannelAdapterManager channelManager;

    @Override
    public void onMessage(String message) {
        AsyncSendMsg msg = JSON.parseObject(message, AsyncSendMsg.class);

        // 只处理交易和安全类通知（优先级 1~3）
        if (msg.getPriority() > 3) {
            // 重新投递到低优先级队列（或直接丢弃，由低优先级消费者处理）
            return;
        }

        ChannelAdapter adapter = channelManager.getAdapter(msg.getChannel());
        try {
            SendResult result = adapter.send(msg.getReceiver(), msg.getContent());

            // 更新发送记录
            notificationLogMapper.updateStatus(msg.getMsgId(),
                result.isSuccess() ? "SENT" : "FAILED",
                result.getChannelMsgId(), result.getErrorCode());
        } catch (Exception e) {
            log.error("通知发送失败: msgId={}", msg.getMsgId(), e);
            throw new RuntimeException(e);  // 触发 MQ 重试
        }
    }
}
```

## 发送失败重试与退订管理

```java
// 重试策略：指数退避 + 区分可重试错误
@Component
public class NotificationRetryTask {

    @Autowired
    private NotificationLogMapper logMapper;

    @Autowired
    private ChannelAdapterManager channelManager;

    @Scheduled(cron = "0 */5 * * * ?")  // 每 5 分钟扫描一次
    public void retryFailedNotifications() {
        // 查找需要重试的记录
        List<NotificationLog> failedList = logMapper.selectRetryable(
            new Date(),  // next_retry_at <= now
            3            // retry_count < max_retry
        );

        for (NotificationLog log : failedList) {
            try {
                ChannelAdapter adapter = channelManager.getAdapter(log.getChannel());
                ChannelSendResult result = adapter.send(log.getReceiver(), log.getContent());

                if (result.isSuccess()) {
                    logMapper.updateStatus(log.getMsgId(), "SENT",
                        result.getChannelMsgId(), null);
                } else {
                    // 判断是否可重试（运营商限速可重试，号码错误不可重试）
                    if (isRetryable(result.getErrorCode())) {
                        int nextRetry = log.getRetryCount() + 1;
                        long delay = calculateBackoff(nextRetry);
                        logMapper.updateRetry(log.getMsgId(), nextRetry,
                            new Date(System.currentTimeMillis() + delay));
                    } else {
                        // 不可重试，直接标记失败
                        logMapper.updateStatus(log.getMsgId(), "FAILED",
                            null, result.getErrorCode());
                    }
                }
            } catch (Exception e) {
                log.error("重试失败: msgId={}", log.getMsgId(), e);
            }
        }
    }

    private boolean isRetryable(String errorCode) {
        // 号码格式错误、退订 → 不可重试
        // 运营商限速、网络超时 → 可重试
        return !Set.of("INVALID_NUMBER", "UNSUBSCRIBED", "BLACKLIST")
                   .contains(errorCode);
    }

    private long calculateBackoff(int retryCount) {
        // 5min, 15min, 45min
        return (long) (5 * Math.pow(3, retryCount - 1) * 60 * 1000);
    }
}
```

```java
// 用户退订管理
@Service
public class UnsubscribeService {

    /**
     * 用户回复 TD 退订营销短信（运营商标准）
     */
    public void handleUnsubscribe(String phone, String channel) {
        // 1. 标记用户退订状态
        unsubscribeMapper.insert(phone, channel, "USER_REPLY");

        // 2. 写入 Redis 快速查询
        String key = "notify:unsub:" + channel + ":" + phone;
        redis.setex(key, 365 * 24 * 3600, "1");  // 1 年有效

        // 3. 如果是全部退订
        redis.setex("notify:unsub:ALL:" + phone, 365 * 24 * 3600, "1");
    }

    /**
     * 发送前检查退订状态
     */
    public boolean canSend(String phone, String channel, String bizType) {
        // 营销类通知检查退订
        if ("MARKETING".equals(bizType)) {
            if (redis.hasKey("notify:unsub:SMS:" + phone)) return false;
            if (redis.hasKey("notify:unsub:ALL:" + phone)) return false;
        }
        // 交易类通知不受退订限制
        return true;
    }
}
```

## 生产踩坑记录

### 坑 1：短信供应商故障，全网通知瘫痪

```
场景：主短信供应商宕机 → 所有短信发不出去
     → 验证码发不了 → 用户无法注册登录
     → 影响所有业务

解决：
1. 多供应商接入：阿里云短信 + 腾讯云短信，互为备份
2. 自动切换：主供应商连续失败 N 次 → 自动切到备用
3. 供应商级健康检查：定时发测试短信，成功率 < 95% 告警
4. 关键通知多渠道路由：验证码短信 + 站内信双发
```

### 坑 2：营销活动瞬间 10 万条推送把通道打满

```
场景：运营一键群发 → 10 万条消息同时涌入 MQ
     → 消费者全在处理营销消息
     → 验证码消息排在后面 → 送达延迟 5 分钟

解决：
1. 优先级隔离：交易通知和营销通知用不同的 MQ 队列和消费者组
2. 营销消息削峰：运营群发后不是立刻发，而是按速率匀速发出（如 100 条/s）
3. 流量整形：用 RocketMQ 的延迟消息分批投递
```

### 坑 3：手机号前缀被运营商拦截

```
场景：170/171 虚拟运营商号段被大量标记为骚扰
     → 短信拒收率高达 40%
     → 但这些用户确实是正常用户

解决：
1. 识别高风险号段，改用语音验证码
2. 号段黑名单可配置化（不用硬编码）
3. 多渠道降级：短信失败 → 自动发语音验证码或站内信
```

## 追问链

### 追问 1：短信验证码为什么不能纯数字连续的？

> "纯数字连续（如 123456）容易被撞库。生成策略：1. 排除 000000、123456、111111 等弱码。2. 至少包含 4 个不同数字。3. 相邻两位差值不能都为 1 或 -1。4. 验证码 6 位就够了，4 位暴力破解成本太低（10^4 = 1 万种组合）。另外验证码输入错误超过 5 次要锁定该手机号 30 分钟。"

### 追问 2：大量用户同时请求验证码怎么扛？

> "关键瓶颈在短信通道（运营商限制 500 条/s），不在系统本身。应对策略：1. 图形验证码前置：请求短信前先过图形验证码，过滤 80% 机器流量。2. IP 限流：单 IP 每分钟最多 5 次。3. 设备指纹：同一设备每分钟最多 1 次。4. 如果短信通道满载，提示用户「稍后重试」，不要让请求堆积到 MQ。"

### 追问 3：站内信怎么实现未读消息提醒？

> "双写：发站内信时同时写数据库（inbox 表）和 Redis（未读计数）。未读数 = `INCR user:unread:{userId}`。用户读消息时 `DECR`。如果 Redis 挂了，从数据库 COUNT 恢复。App 端通过长连接/轮询获取未读数变化，有新消息时推送刷新。"

### 追问 4：消息通知中心的模板怎么支持多语言？

> "模板表增加 locale 字段，同编码不同语言各一条记录：`(LOGIN_SMS, zh_CN)`、`(LOGIN_SMS, en_US)`。发送时根据用户的语言偏好（用户表存 locale）选择对应模板。变量值本身也做多语言映射：`${product_name}` 根据语言查不同的翻译。"

### 追问 5：通知量太大，notification_log 表怎么处理？

> "三个策略：1. 热冷分离：最近 7 天放 MySQL（支持查询），历史数据归档到 Elasticsearch（支持检索）或对象存储（只存不查）。2. 分表：按月分表 `notification_log_202605`、`notification_log_202606`。3. 营销通知日志保留 30 天，交易通知日志保留 1 年（合规要求不同）。"

## 排查 Checklist

```
□ 限流规则配置合理吗？ → 验证码 1条/分钟，营销 3条/分钟
□ 短信供应商有备份吗？ → 至少两个供应商，自动切换
□ 回调状态能追踪吗？ → channel_msg_id + 送达回调
□ 退订状态有检查吗？ → 营销消息发前检查退订
□ 优先级隔离了吗？ → 交易通知和营销通知独立队列
□ 发送失败有重试吗？ → 指数退避 + 区分可重试错误
□ 模板变量有 XSS 过滤吗？ → 站内信内容要转义
□ 发送记录能查询吗？ → 用户投诉时能追溯
□ 渠道切换对业务透明吗？ → 业务线不需要改代码
□ 全局限流有告警吗？ → 接近运营商限额时提前告警
```
