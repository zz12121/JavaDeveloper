# 运行时数据区

> JVM 执行程序时需要科学划分内存——哪些区域是线程私有的、哪些是共享的，直接决定了线程安全和 GC 的设计。重点：PC 寄存器、JVM 栈帧结构、堆 vs 栈的职责分工、方法区（元空间）的演进。

---

## 这个问题为什么存在？

如果 JVM 不规定内存区域划分，所有数据（对象、局部变量、方法信息）都混在一起：

- **无法做垃圾回收**——不知道哪些内存可以被回收
- **线程不安全**——没有线程私有的内存区域，多线程互相干扰
- **无法实现规范**——不同的 JVM 实现会有完全不同的内存布局，破坏"一次编写，到处运行"

**核心问题**：JVM 如何科学地划分内存，使得程序正确运行、内存可被管理、且线程之间互不干扰？

---

## 它是怎么解决问题的？

JVM 把运行时数据区划分为两大类：**线程私有**（随线程生灭，无需 GC）和**线程共享**（需要 GC 管理）。

```
┌─────────────────────────────────────────────┐
│             JVM 运行时数据区                │
├──────────────┬──────────────────────────┤
│  线程私有     │  线程共享                  │
├──────────────┼──────────────────────────┤
│ PC 寄存器    │  Java 堆（Heap）          │
│ JVM 栈       │   （存放对象实例和数组）     │
│ 本地方法栈   │  方法区（Method Area）      │
│              │  （类信息、常量、静态变量） │
└──────────────┴──────────────────────────┘
```

### 1. 程序计数器（PC Register）

**作用**：记录当前线程正在执行的字节码指令地址。

- 如果线程正在执行 Java 方法：PC 寄存器存的是**字节码指令地址**
- 如果线程正在执行 Native 方法：PC 寄存器值为 **undefined**

**为什么是线程私有？** 线程切换后需要恢复到正确的执行位置，每个线程必须有独立的 PC。

**为什么不会 OOM？** PC 寄存器只占很少空间（一个 int 大小），且生命周期跟线程一致，线程结束即释放。

### 2. Java 虚拟机栈（JVM Stack）

**作用**：每个方法执行时都会创建一个**栈帧（Stack Frame）**，存储局部变量表、操作数栈、动态链接、方法返回地址。

```
栈帧（Stack Frame）
├── 局部变量表（Local Variables）
│   └── 存放方法参数和局部变量，以 slot 为单位
│       （32 位类型占 1 个 slot，long/double 占 2 个 slot）
├── 操作数栈（Operand Stack）
│   └── 字节码指令从这里弹出操作数，计算结果再压入
├── 动态链接（Dynamic Linking）
│   └── 将常量池中的符号引用转换为直接引用
└── 方法返回地址（Return Address）
    └── 记录方法退出后应该返回的位置
```

**为什么是栈结构？** 方法调用天然是「后进先出」，栈是最佳数据结构。方法 A 调用方法 B，B 的栈帧压在 A 上面，B 执行完弹出，A 继续执行。

**StackOverflowError vs OutOfMemoryError**：
- 栈深度超过 `-Xss` 设置的值 → `StackOverflowError`（递归太深）
- 栈需要扩展但内存不足 → `OutOfMemoryError`（少见）

### 3. 本地方法栈（Native Method Stack）

为 JVM 调用的 Native 方法（C/C++ 实现）服务。

HotSpot VM 把**本地方法栈和虚拟机栈合二为一**——不需要单独处理，统一用同一个栈结构。

### 4. Java 堆（Heap）

存放**对象实例和数组**，是 JVM 管理的最大一块内存，也是垃圾回收的主要区域。

堆的详细结构（新生代/老年代、Eden/Survivor、TLAB）在 `[[03_JVM/05_堆内存结构/堆内存结构]]` 中详述。

**为什么对象必须放在堆上？** 对象可能被多个方法、多个线程访问，放在线程私有的栈上无法满足共享需求。

### 5. 方法区（Method Area）

存储已被 JVM 加载的**类信息、常量、静态变量、JIT 编译后的代码**。

**方法区的实现演进**：

| JDK 版本 | 实现 | 位置 | 限制 |
|-----------|------|------|------|
| JDK 7 及之前 | 永久代（PermGen） | JVM 堆内 | `-XX:MaxPermSize` |
| JDK 8 及之后 | 元空间（Metaspace） | 本地内存 | 默认无上限（受物理内存限制） |

**为什么要用元空间替换永久代？**
- 永久代大小难以调优，设置不当容易 OOM
- 元空间使用本地内存，不易 OOM，且垃圾回收更高效

### 6. 运行时常量池（Runtime Constant Pool）

方法区的一部分，存放编译期生成的**字面量和符号引用**。

```java
// 源码
String s = "hello";
int x = 10;
```

编译后的 .class 文件中，这些信息都存在**常量池**里。类加载后，常量池被读入内存，成为运行时常量池。

**与字符串常量池的区别**：字符串常量池（String Table）在 JDK 7+ 移到**堆中**，存放的是 `String` 对象实例；运行时常量池在**方法区（元空间）**，存放的是符号引用和字面量。

## 深入原理

### 栈帧里的「动态链接」在做什么？

Java 是动态链接语言——`.class` 文件里不会把方法调用的目标直接写成内存地址，而是写成**符号引用**（类名 + 方法名 + 描述符）。

```java
// 源码
obj.foo();
```

编译后的字节码是 `invokevirtual #2`（#2 是常量池索引）。运行时，JVM 需要通过「动态链接」把 #2 这个符号引用解析为**实际的内存地址**。

**为什么不在类加载时就全部解析好？** 因为 Java 支持运行时动态绑定（多态）——`obj` 的实际类型在编译期不知道，只能在**第一次执行到这行**时才解析，这叫「**延迟解析（Lazy Resolution）**」。

### 栈溢出 vs 堆溢出

| | 栈溢出（StackOverflowError） | 堆溢出（OutOfMemoryError: Java heap space） |
|--|---------------------------|----------------------------------------|
| **原因** | 栈深度超过 `-Xss` | 对象过多且无法被 GC 回收 |
| **触发** | 递归太深 | 不断创建对象且被引用持有 |
| **解决** | 改为迭代，或增大 `-Xss` | 修复内存泄漏，或增大 `-Xmx` |

### 为什么基本类型存在栈上，对象存在堆上？

**基本类型**：生命周期短，且不需要被共享，放在栈上访问速度快，且无需垃圾回收。

**对象**：可能被多个方法、多个线程访问，必须放在共享区域（堆）上。

**例外**：JIT 编译器的**逃逸分析**发现对象未逃逸出方法，可以把对象**标量替换**到栈上（拆成基本类型存在栈帧里），避免堆分配。

## 正确使用方式

### 1. 理解栈深度设置 vs 线程数

```bash
# 每个线程的栈大小
-Xss256k   # 每个线程栈 256KB

# 计算：如果堆设为 2G，系统内存 4G
# 最多能创建约 (4G - 2G) / 256k ≈ 8192 个线程
# 但实际会受操作系统限制（ulimit -u）
```

**为什么不用默认 1M 栈？** 高并发应用（如 Netty）需要几万甚至几十万线程，栈越小能创建的线程越多。但栈太小容易 `StackOverflowError`，需要根据实际递归深度调整。

### 2. 方法区（元空间）大小需要设置上限

```bash
# JDK 8+：元空间默认无上限，但必须设置上限防止失控
-XX:MetaspaceSize=128m     # 初始大小（触发首次 GC 的阈值）
-XX:MaxMetaspaceSize=256m  # 最大上限
```

**为什么会失控？** Spring Boot、MyBatis 大量使用 CGLIB 动态代理，每次代理都会生成新的类，元空间会持续增长。不设置上限可能在运行很久后耗尽系统内存。

## 边界情况和坑

### 1. 递归没有终止条件 → StackOverflowError

```java
public void recursive() {
    recursive();  // 无限递归，栈深度不断增加
}
// 错误信息：java.lang.StackOverflowError
```

**排查**：`StackOverflowError` 的堆栈信息非常有用——看栈顶的方法，就是无限递归的那个方法。

### 2. 线程数达到操作系统上限

```bash
# 每个线程需要占用栈空间（-Xss）
# 线程数过多时，即使堆还有很多空间，也无法创建新线程
# 错误信息：java.lang.OutOfMemoryError: unable to create new native thread
```

**解决**：减小 `-Xss`（栈小一点，能创建更多线程），或增大操作系统的线程数限制（`ulimit -u`）。

### 3. 静态变量存在哪里？

很多人误以为「静态变量在堆里」——实际上：

- **静态变量本身**存放在**方法区（元空间）**
- **静态变量指向的对象**存放在**堆里**

```java
class Example {
    static Object obj = new Object();
    //   ↑ 在元空间      ↑ 这个对象在堆里
}
```

### 4. 栈帧里的局部变量表，存的可能是引用

```java
public void test() {
    Object o = new Object();
    // 局部变量 o 存在栈帧的局部变量表里
    // 但 o 指向的 Object 实例存在堆里
}
```

**本质**：栈里存的是「地址」（引用），对象本体永远在堆里。这也是为什么 Java 的栈帧可以很小——它只存基本类型和引用，大对象都在堆上。
