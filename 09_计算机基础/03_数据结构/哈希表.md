# 哈希表

> 哈希表是面试中考察频率最高的数据结构，HashMap 的底层原理几乎是 Java 后端面试的必问题。

---

## 一、哈希表基本原理

### 1.1 核心思想

**通过哈希函数将 Key 映射到数组索引，实现 O(1) 的查找、插入和删除。**

```
Key: "apple"   → hash("apple") = 5    → table[5] → Value
Key: "banana"  → hash("banana") = 2   → table[2] → Value
Key: "cherry"  → hash("cherry") = 5   → table[5] → 冲突！
```

### 1.2 哈希函数设计原则

| 原则 | 说明 |
|------|------|
| 确定性 | 相同的 key 总是得到相同的 hash |
| 均匀性 | hash 值尽可能均匀分布 |
| 高效性 | 计算速度快 |
| 雪崩效应 | key 的微小变化导致 hash 值的大变化 |

---

## 二、哈希冲突解决

### 2.1 四种主要方法

#### 方法一：链地址法（Separate Chaining）⭐ 最常用

```
table[0] → null
table[1] → [key1|val1] → [key5|val5] → null
table[2] → [key2|val2] → null
table[3] → null
table[4] → [key3|val3] → [key4|val4] → [key6|val6] → null

Java: HashMap, LinkedHashMap, ConcurrentHashMap
```

#### 方法二：开放地址法（Open Addressing）

```
探测序列：h(k), h(k)+1, h(k)+4, h(k)+9, ... (二次探测)

table[0]: key="apple"
table[1]: key="banana"
table[2]: key="cherry"（原本 hash=1，冲突后探测到位置 2）
table[3]: null

Java: ThreadLocal.ThreadLocalMap
```

| 探测方式 | 序列 | 优缺点 |
|---------|------|--------|
| 线性探测 | h+1, h+2, h+3... | 简单，但容易聚集 |
| 二次探测 | h+1, h+4, h+9... | 减少聚集 |
| 双重哈希 | h₁(k), h₁(k)+h₂(k)... | 分布最均匀 |

#### 方法三：再哈希（Rehashing）

使用多个哈希函数，第一个冲突就用第二个，以此类推。

#### 方法四：建立公共溢出区

将冲突元素统一存放到溢出表中。

### 2.2 方法对比

| 方法 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 链地址法 | 实现简单，删除方便 | 指针开销，缓存不友好 | 通用场景（HashMap） |
| 开放地址法 | 无指针开销，缓存友好 | 删除复杂，聚集问题 | 元素数量固定（ThreadLocal） |

## 三、HashMap 深度剖析（JDK 8+）

### 3.1 数据结构

```
JDK 8: 数组 + 链表 + 红黑树

table:  [  ] [  ] [  ] [  ] [  ] [  ] [  ] [  ]
          |              |              |
          ↓              ↓              ↓
        [k1|v1]       [k2|v2]       [TreeNode(红黑树)]
          ↓
        [k3|v3]
          ↓
        [k4|v4]        (链表长度 < 8)

当链表长度 ≥ 8 且数组长度 ≥ 64 时，链表转为红黑树
当红黑树节点 ≤ 6 时，红黑树退化回链表
```

### 3.2 关键参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| DEFAULT_INITIAL_CAPACITY | 16 | 初始容量 |
| MAXIMUM_CAPACITY | 2^30 | 最大容量 |
| DEFAULT_LOAD_FACTOR | 0.75 | 负载因子 |
| TREEIFY_THRESHOLD | 8 | 链表转红黑树阈值 |
| UNTREEIFY_THRESHOLD | 6 | 红黑树退化链表阈值 |
| MIN_TREEIFY_CAPACITY | 64 | 最小树化数组容量 |

### 3.3 Hash 计算与索引定位

```java
// 1. 计算 hash（扰动函数）
static final int hash(Object key) {
    int h;
    // 高 16 位与低 16 位异或，让高位也参与运算
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}

// 2. 定位桶索引
// index = hash & (capacity - 1)
// 等价于 hash % capacity，但位运算更快
// 前提：capacity 必须是 2 的幂次方
```

**为什么 capacity 必须是 2 的幂次方？**
- `hash & (n-1)` 等价于 `hash % n`，但位运算比取模快
- 当 n 是 2^k 时，n-1 的二进制全是 1（如 16-1=15=0b1111），能确保 hash 的低位全部参与运算

**为什么用 h ^ (h >>> 16)？**
- 数组通常不大（< 2^16），hash 的高 16 位在 `hash & (n-1)` 时完全被丢弃
- 异或使得高位信息也能影响索引，减少碰撞

### 3.4 Put 流程

```java
public V put(K key, V value) {
    return putVal(hash(key), key, value, false, true);
}

final V putVal(int hash, K key, V value, boolean onlyIfAbsent, boolean evict) {
    Node<K,V>[] tab; Node<K,V> p; int n, i;

    // 1. 初始化数组（延迟初始化，第一次 put 时才创建）
    if ((tab = table) == null || (n = tab.length) == 0)
        n = (tab = resize()).length;

    // 2. 计算索引，桶为空直接放入
    if ((p = tab[i = (n - 1) & hash]) == null)
        tab[i] = newNode(hash, key, value, null);

    else {
        Node<K,V> e; K k;

        // 3. key 已存在，直接覆盖
        if (p.hash == hash && ((k = p.key) == key || (key != null && key.equals(k))))
            e = p;

        // 4. 红黑树节点
        else if (p instanceof TreeNode)
            e = ((TreeNode<K,V>)p).putTreeVal(this, tab, hash, key, value);

        // 5. 链表节点，遍历插入
        else {
            for (int binCount = 0; ; binCount++) {
                if ((e = p.next) == null) {
                    p.next = newNode(hash, key, value, null);
                    // 链表长度 ≥ 7（binCount 从 0 开始），尝试树化
                    if (binCount >= TREEIFY_THRESHOLD - 1)
                        treeifyBin(tab, hash);
                    break;
                }
                if (e.hash == hash && ((k = e.key) == key || (key != null && key.equals(k))))
                    break;
                p = e;
            }
        }
        if (e != null) { // 已存在的 key
            V oldValue = e.value;
            if (!onlyIfAbsent || oldValue == null)
                e.value = value;
            afterNodeAccess(e);
            return oldValue;
        }
    }

    ++modCount;
    // 6. 超过阈值，扩容
    if (++size > threshold)
        resize();
    afterNodeInsertion(evict);
    return null;
}
```

### 3.5 扩容机制

```
触发条件：size > capacity × loadFactor

扩容过程：
oldCapacity = 16 → newCapacity = 32
阈值：16 × 0.75 = 12 → 32 × 0.75 = 24

元素重新分配（JDK 8 优化）：
- 不需要重新计算 hash
- 使用 hash & oldCapacity 判断元素在新数组的位置
  - hash & oldCapacity == 0 → 留在原位置
  - hash & oldCapacity != 0 → 移动到 原位置 + oldCapacity

示例（oldCap = 16, newCap = 32）：
hash = 5  → 5 & 16 = 0  → 留在 index 5
hash = 21 → 21 & 16 = 16 → 移到 index 21 (5 + 16)
hash = 3  → 3 & 16 = 0  → 留在 index 3
hash = 19 → 19 & 16 = 16 → 移到 index 19 (3 + 16)

迁移前：table[3] → [hash=3] → [hash=19] → [hash=35]
迁移后：table[3] → [hash=3] → [hash=35]
        table[19] → [hash=19]

→ 拆分为两条链表，保持原有顺序！
```

### 3.6 Get 流程

```java
public V get(Object key) {
    Node<K,V> e;
    return (e = getNode(hash(key), key)) == null ? null : e.value;

// 1. 计算 hash，定位桶
// 2. 桶为空 → 返回 null
// 3. 桶头节点匹配 → 返回
```

## 四、HashMap 的线程安全问题

### 4.1 JDK 7 并发死循环

```
场景：多线程同时扩容，头插法导致链表成环

线程A 和 线程B 同时在链表 [3] → [7] → [5] 上扩容
线程A 暂停在 e.next = newTable[i] 处
线程B 完成扩容：newTable[3] = [7] → [3]（反转了）
线程A 恢复：e.next = [7]（newTable[3]），形成环！

JDK 8 改为尾插法解决了这个问题，但 HashMap 仍然不是线程安全的！
```

### 4.2 ConcurrentHashMap（JDK 8）

```
JDK 7: Segment 数组 + ReentrantLock（分段锁）
- 默认 16 个 Segment，每个 Segment 一把锁
- 并发度 = Segment 数量

JDK 8: Node 数组 + CAS + synchronized（锁粒度从段级降到桶级）

put 流程：
1. 计算 hash，定位桶
2. 桶为空 → CAS 自旋写入 Node（无锁）
3. 桶非空 → synchronized 锁住头节点（桶级锁）
4. 判断是链表还是红黑树，执行对应操作
5. 添加完成后检查是否需要树化

size 计算：
- 使用 CounterCell 数组 + baseCount（LongAdder 思想）
- 多线程各自更新自己的 CounterCell
- 汇总时累加所有 CounterCell + baseCount
```

| 维度 | JDK 7 | JDK 8 |
|------|-------|-------|
| 锁粒度 | Segment（段级） | 桶级（Node 级） |
| 锁策略 | ReentrantLock | CAS + synchronized |
| 数据结构 | 数组 + 链表 | 数组 + 链表 + 红黑树 |
| 并发度 | 固定（Segment 数量） | 随容量动态 |
| size 计算 | 先尝试不加锁，失败后加锁 | LongAdder 方案 |

## 五、HashSet 与 LinkedHashSet

### 5.1 HashSet

```java
// HashSet 底层就是 HashMap！
public class HashSet<E> {
    private transient HashMap<E, Object> map;
    private static final Object PRESENT = new Object(); // 哑值

    public boolean add(E e) {
        return map.put(e, PRESENT) == null;
    }

    public boolean contains(Object o) {
        return map.containsKey(o);
    }
}
// 所有操作都是 O(1)，存储的是 Key，Value 固定为 PRESENT
```

## 六、哈希表性能分析

### 6.1 负载因子选择

```
负载因子 α = 元素数量 / 桶数量

α 太大 → 冲突增多 → 查找退化为 O(n)
α 太小 → 空间浪费 → 但查找接近 O(1)

Java 默认 α = 0.75：
- 泊松分布下，当 α = 0.75 时，链表长度为 8 的概率仅为 0.00000006
- 是时间和空间的良好平衡点

α = 1.0：空间利用率最高，但冲突概率高
α = 0.5：冲突概率低，但空间浪费 50%
```

### 6.2 装填因子对性能的影响

| 负载因子 | 空间利用率 | 平均查找长度 | 适用场景 |
|---------|-----------|-------------|---------|
| 0.5 | 50% | ~1.5 | 查找敏感 |
| 0.75 | 75% | ~2.0 | 通用（Java 默认） |
| 1.0 | 100% | ~3.5 | 空间敏感 |
| 2.0 | 200%（开放地址） | ~5.0 | 极致省内存 |

## 七、哈希攻击与防护

### 7.1 哈希碰撞攻击（Hash Collision DoS）

```
攻击原理：
1. 攻击者找到大量 hash 相同的 key（如 "Aa" 和 "BB" 的 hash 相同）
2. 向 HashMap 插入大量碰撞的 key
3. 所有元素退化为一条链表
4. 插入/查找从 O(1) 退化为 O(n)
5. CPU 被大量 hash 计算拖垮

防护：
- JDK 8：链表长度 ≥ 8 转红黑树，查找保持 O(log n)
- 使用随机种子（但 Java String.hashCode 是固定的）
- 限制单个 IP 的请求频率
```

## 八、面试高频问题

| 问题 | 关键回答 |
|------|---------|
| HashMap 底层数据结构 | 数组 + 链表 + 红黑树（JDK 8+） |
| 为什么用红黑树不用 AVL | 插入时旋转次数更少，HashMap 写多读少 |
| 扩容机制 | 2 倍扩容，使用 hash & oldCapacity 拆分链表 |
| 为什么容量是 2 的幂 | 位运算代替取模，hash 分布更均匀 |
| 为什么 0.75 负载因子 | 泊松分布下的时空平衡点 |
| 线程安全吗 | 不安全，用 ConcurrentHashMap |
| JDK 7 死循环原因 | 头插法 + 多线程并发扩容导致链表成环 |
| key 可以为 null 吗 | 可以（HashMap），不能（ConcurrentHashMap、HashTable） |
