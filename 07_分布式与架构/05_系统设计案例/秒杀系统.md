# 秒杀系统

## 这个问题为什么存在？

> 电商大促（双11、618），**瞬间流量是平时的100倍**，数据库直接被打挂。问题是：**怎么在极致并发下，保证不超卖、系统不崩？**

没有秒杀设计，你会遇到：
- 100万人抢1000件商品 → 数据库直接挂
- 超卖（1000件库存，卖了5000件）
- 用户支付成功，但库存已扣光（库存不一致）

---

## 它是怎么解决问题的？

### 核心思路：层层过滤，只有少量请求到数据库

```

                 100万QPS
                     │
            ┌───────┴───────┐
            │  CDN静态化       │  ← 静态资源不走后端
            │  （商品详情页）    │
            └───────┬───────┘
                      │ 过滤掉90%
                      ▼
                   10万QPS
                     │
            ┌───────┴───────┐
            │  Nginx限流       │  ← 令牌桶，限制到1000 QPS
            └───────┬───────┘
                      │ 过滤掉99%
                      ▼
                   1000 QPS
                     │
            ┌───────┴───────┐
            │  Redis预扣库存   │  ← 原子操作，库存<=0就拒绝
            └───────┬───────┘
                      │ 过滤掉80%
                      ▼
                   200 QPS
                     │
            ┌───────┴───────┐
            │  异步下单          │  ← MQ削峰，异步创建订单
            └───────┬───────┘
                      │
                      ▼
                  数据库
```

### 关键技术点

#### 1. CDN静态化（前端）

```
商品详情页 → 静态化 → 放到CDN
  → 用户请求 → CDN直接返回（不走后端）
  → 减少后端流量90%
```

#### 2. Nginx限流（入口）

```nginx
# 限制每个IP每秒10次请求（防止script攻击）
limit_req_zone $binary_remote_addr zone=seckill:10m rate=10r/s;

server {
    location /seckill/ {
        limit_req zone=seckill burst=20 nodelay;
        proxy_pass http://seckill-service;
    }
}
```

#### 3. Redis预扣库存（核心）

```java
// ❌ 错误：查库存 → 减库存（两步操作，不是原子的）
public boolean deductStock(Long itemId) {
    int stock = redis.get("stock:" + itemId);
    if (stock > 0) {
        redis.decr("stock:" + itemId);  // 不是原子的！并发下超卖
        return true;
    }
    return false;
}

// ✅ 正确：用Lua脚本，原子操作
public boolean deductStock(Long itemId, int amount) {
    String lua = """
        local stock = tonumber(redis.call('get', KEYS[1]))
        if stock == nil or stock < tonumber(ARGV[1]) then
            return 0  -- 库存不足
        end
        return redis.call('decrby', KEYS[1], ARGV[1])
        """;
    Long result = jedis.eval(lua,
            Collections.singletonList("stock:" + itemId),
            Collections.singletonList(String.valueOf(amount)));
    return result != null && result >= 0;
}
```

#### 4. MQ异步下单（削峰）

```java
@PostMapping("/seckill")
public String seckill(Long itemId, int amount) {
    // 1. Redis预扣库存（原子操作）
    if (!redisService.deductStock(itemId, amount)) {
        return "库存不足";
    }

    // 2. 发送消息到MQ（异步创建订单）
    OrderMessage msg = new OrderMessage(userId, itemId, amount);
    rabbitTemplate.convertAndSend("seckill-order", msg);

    return "排队中，请稍后查看订单";  // 不直接返回订单号
}

// 3. 消费者：异步创建订单（削峰）
@RabbitListener(queues = "seckill-order")
public void createOrder(OrderMessage msg) {
    orderService.create(msg);  // 写入数据库（流量已削峰）
}
```

## 它和相似方案的本质区别是什么？

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| **同步下单**（不推荐） | 简单 | 数据库直接挂 | 流量极低（<100 QPS） |
| **Redis预扣 + 异步下单**（推荐） | 高并发，削峰 | 复杂度高，用户需轮询结果 | 秒杀、大促 |
| **全链路限流**（Nginx+网关+Redis） | 多层防护 | 合法请求也被限流 | 秒杀 |

**本质区别**：
- **同步下单**：流量直接打到DB，架构简单但撑不住高并发
- **异步下单**：MQ削峰，流量从100万QPS → 200 QPS（DB承受得住）

## 正确使用方式

### 秒杀系统的"四个要点"

```
要点1：静态化   → CDN扛流量（90%流量过滤）
要点2：限流      → Nginx + 网关（99%流量过滤）
要点3：预扣库存 → Redis原子操作（防止超卖）
要点4：异步下单 → MQ削峰（保护数据库）
```

### 防止脚本攻击（黄牛）

```java
// 1. 验证码（前端）
// 秒杀前，先填验证码（防止script自动提交）

// 2. 用户ID限流（后端）
// 同一个用户ID，5秒内只能提交1次
public boolean trySeckill(Long userId, Long itemId) {
    String key = "seckill:user:" + userId;
    Long result = jedis.setnx(key, "1");
    if (result == 1) {
        jedis.expire(key, 5);  // 5秒内只能抢一次
        // 继续秒杀逻辑...
    }
    return false;  // 5秒内重复提交，拒绝
}
```

## 边界情况和坑

### 坑1：Redis预扣成功，但DB扣减失败

```
场景：Redis预扣了库存，但DB扣减时宕机
后果：Redis库存少了，DB库存没少 → 数据不一致
```

**解决方案**：
1. **异步重试**（MQ消费失败，重试）
2. **定期对账**（Redis库存 vs DB库存，差异告警）
3. **库存回补**（DB扣减失败，Redis库存加回去）

### 坑2：用户支付成功，但库存已扣光

```
场景：Redis预扣库存成功，用户去支付
      支付成功时，库存已经被其他人抢光了
后果：用户付了钱，但没货发
```

**解决方案**：
1. **支付前再次检查库存**（支付时，再查一次DB库存）
2. **超时释放**（15分钟没支付，库存加回去）
3. **告诉用户"抢购成功，但库存不足，自动退款"**

### 坑3：脚本攻击（黄牛用脚本抢）

```
场景：黄牛用1000个账号，每秒提交10000次请求
后果：正常用户抢不到
```

**解决方案**：
1. **验证码**（秒杀前先填验证码）
2. **用户ID限流**（同一个用户ID，5秒内只能提交1次）
3. **风控系统**（检测异常账号，拉黑）
