## 知识链提示

这道题应该让你联想到：

- `传统线程模型` → OS 线程 1:1 映射，内存开销大（1MB/线程）
- `载体线程 Carrier Thread` → 虚拟线程挂载的载体
- `阻塞时让出` → 虚拟线程的核心优势：阻塞时不占载体
- `线程局部变量陷阱` → ThreadLocal 在虚拟线程里可能内存泄漏
- `Project Loom` → Java19 预览，Java21 正式

## 参考要点（盲答后再看）


**虚拟线程本质**：
- JVM 管理的轻量级线程，不 1:1 绑定 OS 线程
- 阻塞时：自动「卸载」虚拟线程，载体线程可以去运行其他虚拟线程
- 适合：大量 IO 等待的场景（Web 请求、DB 查询）

**与传统线程对比**：

| 维度 | 传统线程 | 虚拟线程 |
|------|---------|---------|
| 调度者 | OS 内核 | JVM |
| 内存开销 | ~1MB/线程 | ~几百字节/虚拟线程 |
| 阻塞代价 | 占着 OS 线程 | 让出载体，几乎无代价 |
| 适用场景 | CPU 密集 | IO 密集 |
| 数量上限 | 几千个 | 几百万个 |

**synchronized 的「引脚」问题**：
```java
// ❌ 虚拟线程在 synchronized 块内阻塞时，无法卸载（被「引脚」在载体上）
synchronized (lock) {
    Thread.sleep(1000);  // 占着载体！
}
// ✅ 改用 ReentrantLock
lock.lock();
try { Thread.sleep(1000); }
finally { lock.unlock(); }
```

**如何启用虚拟线程（Spring Boot）**：
```java
// Java21+ 直接
Thread.startVirtualThread(() -> { ... });

// 线程池方式
ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor();
```

**与 Go goroutine 对比**：
| | 虚拟线程 | goroutine |
|--|---------|------------|
| 调度器 | JVM 内置 | Go runtime |
| 栈管理 | 惰性分配，可增长 | 动态增长 |
| 抢占 | 协作式（代码里插入 yield 点） | 协作式 + 异步抢占（Go1.14+） |
