# 代码质量平台设计

> 代码质量核心：静态分析 + 覆盖率 + 门禁规则 + 质量度量。

---

## 代码质量平台架构

```
开发提交代码
   │
   ▼
┌─────────┐  PR触发 ┌─────────┐  分析  ┌─────────┐
│  Git    │ ────→  │  CI     │ ────→   │ SonarQube│
│  仓库    │        │  流水线  │         │  静态分析 │
└─────────┘        └─────────┘         └────┬────┘
                                            │
                     ┌──────────────────────┼──────────────────────┐
                     │                      │                      │
                     ▼                      ▼                      ▼
              ┌────────────┐         ┌────────────┐         ┌────────────┐
              │  JaCoCo    │         │  门禁规则   │         │  质量报表   │
              │  覆盖率     │         │  PR Block  │         │  Grafana   │
              └────────────┘         └────────────┘         └────────────┘
```

## 场景 A：SonarQube静态分析集成

### 现象

```
代码存在空指针、未关闭资源等问题才上线
重复代码多，维护成本高
安全漏洞（SQL注入、XSS）未被检测
```

### 根因

```
未集成静态分析工具，靠人工Review发现
CI流程无质量检测，问题代码合入主干
无统一质量规则，各项目标准不一致
```

### 解决方案

```yaml
# SonarQube集成到GitLab CI
sonar-check:
  stage: test
  script:
    - mvn sonar:sonar \
      -Dsonar.projectKey=myapp \
      -Dsonar.host.url=http://sonar.example.com:9000 \
      -Dsonar.login=$SONAR_TOKEN \
      -Dsonar.java.binaries=target/classes \
      -Dsonar.coverage.jacoco.reportPaths=target/jacoco.xml
  dependencies:
    - unit-test
```

**静态分析关键**：
- 集成SonarQube到CI流水线，PR提交时自动分析
- 自定义质量规则（如禁止System.out.println、强制资源关闭）
- 新增代码覆盖率必须≥80%，否则PR被Block

## 场景 B：JaCoCo代码覆盖率

### 现象

```
功能上线后发现有分支未测试，出现Bug
测试用例只覆盖主流程，边界条件遗漏
覆盖率数据造假（只跑测试不校验覆盖率）
```

### 根因

```
未集成覆盖率工具，无覆盖率门禁
测试用例设计不合理，只测正常流程
覆盖率数据未上报，无法追踪改进
```

### 解决方案

```xml
<!-- JaCoCo Maven插件配置 -->
<plugin>
  <groupId>org.jacoco</groupId>
  <artifactId>jacoco-maven-plugin</artifactId>
  <version>0.8.10</version>
  <executions>
    <execution>
      <goals><goal>prepare-agent</goal></goals>
    </execution>
    <execution>
      <id>report</id>
      <phase>test</phase>
      <goals><goal>report</goal></goals>
    </execution>
    <execution>
      <id>check</id>
      <phase>verify</phase>
      <goals><goal>check</goal></goals>
      <configuration>
        <rules>
          <rule>
            <element>PACKAGE</element>
            <limits>
              <limit>
                <counter>LINE</counter>
                <value>COVEREDRATIO</value>
                <minimum>0.80</minimum>  <!-- 行覆盖率≥80% -->
              </limit>
              <limit>
                <counter>BRANCH</counter>
                <value>COVEREDRATIO</value>
                <minimum>0.70</minimum>  <!-- 分支覆盖率≥70% -->
              </limit>
            </limits>
          </rule>
        </rules>
      </configuration>
    </execution>
  </executions>
</plugin>
```

**覆盖率关键**：
- 增量覆盖率（新增代码）必须≥80%，否则PR不通过
- 结合SonarQube展示覆盖率趋势，持续追踪改进
- 排除生成代码、配置类，避免覆盖率虚高

## 场景 C：门禁规则PR Review

### 现象

```
问题代码合入主干，导致线上故障
PR Review流于形式，未检查核心逻辑
无权限控制，任何人都能合入代码
```

### 根因

```
未设置PR门禁，通过所有检查就能合入
Review人未指定，责任不明确
无自动化检查，依赖人工Review发现所有问题
```

### 解决方案

```yaml
# GitHub PR门禁规则（.github/CODEOWNERS）
# 核心模块必须指定Review人
/src/main/java/com/example/core/**/*.java @core-team
/pom.xml @dev-leads
```

```groovy
// Jenkins流水线门禁检查
pipeline {
    stages {
        stage('Quality Gate') {
            steps {
                // 等待SonarQube质量门禁通过
                timeout(time: 10, unit: 'MINUTES') {
                    waitForQualityGate abortPipeline: true
                }
            }
        }
    }
}
```

**门禁关键**：
- 核心模块修改必须2人以上Approval
- SonarQube质量门禁不通过，PR自动Block
- 增量覆盖率、静态分析问题数为0才允许合入

## 场景 D：质量度量体系

### 现象

```
无法衡量代码质量改进效果
各项目质量指标不透明，没有对比
技术债越积越多，无人跟进
```

### 根因

```
无统一质量度量指标，各项目自定标准
质量数据未可视化，无法追踪趋势
无技术债管理流程，问题代码无人修改
```

### 解决方案

```
质量度量核心指标：
1. 代码重复率：≤5%（SonarQube检测）
2. 圈复杂度：单方法≤15，单类≤200
3. 测试覆盖率：增量代码≥80%，全量≥60%
4. 严重问题数：Blocker/Critical问题数为0
5. 技术债：每周必须偿还10%以上
```

**度量关键**：
- 每周生成质量报表，发送给大家
- 建立质量排名，落后项目指定改进计划
- 技术债纳入迭代任务，定期偿还

## 排查 Checklist

```
□ SonarQube集成到CI了吗？ → PR提交自动触发分析
□ 有覆盖率门禁吗？ → 增量覆盖率≥80%
□ PR需要Approval吗？ → 核心模块至少2人Approval
□ 质量指标可视化了吗？ → Grafana展示覆盖率/问题数趋势
□ 技术债有跟进吗？ → 每周偿还10%以上技术债
□ 静态分析规则自定义了吗？ → 禁止空指针/未关闭资源等
□ 覆盖率数据真实吗？ → 排除生成代码/配置类
□ 门禁不通过会Block PR吗？ → Sonar质量门禁+CI检查
```
