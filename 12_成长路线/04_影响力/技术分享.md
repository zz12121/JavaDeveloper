# 技术分享

> 能讲清楚，才是真懂了。技术分享是建立个人影响力最直接的路径。

---

## 一、技术分享的准备流程

### 1.1 选题

```
选题原则：
✅ 解决团队实际痛点（最近线上问题多？）
✅ 你有深度实践（不能是"刚看了篇文章"）
✅ 有争议性（能引发讨论更好）
✅ 能形成系列（一次讲不完，分两次）

好选题示例：
- "我们是怎么把订单查询从 2s 优化到 200ms 的"（有故事）
- "Java 8 Stream 用错了会慢 3 倍"（有反直觉点）
- "微服务拆分踩过的 5 个坑"（有经验教训）

❌ 差选题：
- "Redis 入门教程"（太基础，网上有的是）
- "DDD 理论详解"（纯理论，没有落地）
- "我读的一次论文"（和团队工作无关）
```

### 1.2 结构设计（45 分钟分享模板）

```
开场（3 分钟）：
- 今天要解决什么问题？
- 这个问题不解决会怎样？

背景（5 分钟）：
- 现有方案的痛点（用数据！）
- 为什么现在要解决？

正题（25 分钟）：
- 方案设计（架构图 + 时序图）
- 核心技术点（讲深，不要泛泛而谈）
- 踩过的坑（最有价值的部分！）

效果（5 分钟）：
- 优化前后对比（性能 / 稳定性数据）
- 团队收益

互动（7 分钟）：
- Q&A
- 讨论：你们有没有更好的方案？
```

## 二、PPT 制作技巧

### 2.1 每一页 PPT 的原则

```
✅ 一页一个核心观点
✅ 有图不堆字（架构图 > 10 行文字）
✅ 代码最多 10 行（多了大家不看了）
✅ 关键数据用大字号突出
✅ 有对比（优化前 vs 优化后）

❌ 满屏文字（照着念，谁听得下去？）
❌ 没有图示（纯文字描述系统架构）
❌ 代码 50 行（大家看不清也看不懂）
```

### 2.2 图示规范

```
架构图：
- 用不同颜色区分系统边界
- 箭头标注协议（HTTP/gRPC/Kafka）
- 标注关键指标（QPS、延迟）

时序图：
- 标注每一步的耗时
- 异常路径用红色虚线

数据对比图：
- 永远用同一量纲对比
- 优化前后的柱状图并排
- 关键提升用红色标注（如"提升 10 倍！"）
```

## 三、分享现场技巧

### 3.1 开场抓人

```
❌ 差开场："大家好，今天我讲一下 Redis 缓存..."
✅ 好开场："上个月我们线上出了 3 次缓存雪崩，导致订单丢失。今天分享我们是怎么彻底解决的。"

✅ 更好的开场（抛问题）："大家觉得 Redis 缓存穿透和缓存击穿是一回事吗？很多人答错，今天 5 分钟讲清楚。"
```

### 3.2 控制节奏

```
- 每 10 分钟一个小互动（提问 / 投票）
- 核心技术点讲 2 遍（换种说法）
- 看到有人眼神游离，讲个踩坑故事拉回来
- 预留 Q&A 时间（至少 10 分钟）
```

### 3.3 处理提问

```
会的问题：
→ 先重复一遍问题（让所有人听到）
→ 举例说明（不要只讲概念）
→ 询问"我讲清楚了吗？"

不会的问题：
→ 诚实说"这个我没深入研究，会后确认后回复"
→ 不要强行回答（会被问住）

有争议的问题：
→ "这个确实是 trade-off，我们的场景是... 你们怎么看的？"
→ 引导讨论，不要硬输
```

## 四、分享后的动作

```
□ 24 小时内发出 PPT + 录屏视频
□ 整理 Q&A 记录（有人问了什么，你答了什么）
□ 把核心内容改写成技术博客（一份时间，两份产出）
□ 收集反馈（哪部分讲快了？哪部分最有收获？）
□ 下次改进
```

## 五、不同类型分享的侧重点

| 类型 | 时长 | 重点 |
|------|------|------|
| 技术方案评审 | 30-45 分钟 | 方案对比 + 风险评估 |
| 技术专题分享 | 45-60 分钟 | 深度 + 踩坑经验 |
| 新员工培训 | 60-90 分钟 | 基础 + 动手 Demo |
| 外部会议演讲 | 20-30 分钟 | 故事性 + 核心亮点 |

## 六、建立个人技术品牌

```
内部影响力：
→ 定期（每季度 1-2 次）技术分享
→ 主动在 Code Review 中传播最佳实践
→ 把踩坑经验写成内部 Wiki

外部影响力：
→ 技术博客（掘金 / CSDN / 个人公众号）
→ 开源贡献（提 PR、写文档）
→ 技术会议演讲（先内部分享，再投外部 CFP）
→ 技术社区活跃（回答 Stack Overflow / 知乎技术问题）

影响力飞轮：
  写博客 → 有读者 → 被邀请演讲 → 认识更多牛人 → 更多学习机会 → 更强的技术深度 → 更好的博客...
```
