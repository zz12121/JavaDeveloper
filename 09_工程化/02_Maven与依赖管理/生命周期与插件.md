# 生命周期与插件

> Maven 不是「构建工具」，是**构建生命周期管理框架**。它的核心设计是「约定优于配置」——目录结构、构建阶段、插件绑定，都是预先定义好的。不理解生命周期，配置插件时就是在碰运气。

---

## 这个问题为什么存在？

没有统一构建生命周期时，每个项目都有自己的构建方式：

```
石器时代的构建：
项目 A：ant build.xml（自定义任务）
项目 B：手工 javac + jar 打包
项目 C：idea 导出 ar
项目 D：...（每个项目都不一样）

问题：
1. 新人接手项目，不知道怎么构建
2. CI/CD 没法通用（每个项目构建命令不同）
3. 构建结果不可预测（每个人本地构建结果可能不一样）
```

**Maven 的解决方案**：定义**标准生命周期**，所有项目共用同一套构建命令（`mvn clean package`）。

---

## 它是怎么解决问题的？

### 三套生命周期

Maven 有 **3 套独立的生命周期**，每套由多个 **Phase（阶段）** 组成：

```
【1. Clean Lifecycle】—— 清理项目
  pre-clean → clean → post-clean

【2. Default Lifecycle】—— 核心构建（编译、测试、打包）
  validate → initialize → generate-sources → process-sources
  → generate-resources → process-resources → compile
  → process-classes → generate-test-sources → process-test-sources
  → test-compile → process-test-classes → test
  → prepare-package → package → pre-integration-test
  → integration-test → post-integration-test → verify
  → install → deploy

【3. Site Lifecycle】—— 生成项目文档站点
  pre-site → site → post-site → site-deploy
```

**关键认知**：执行某个 Phase，它**之前的所有 Phase 都会按顺序执行**：

```bash
mvn install
# 实际执行：validate → ... → compile → ... → test → ... → package → install

mvn package
# 实际执行：validate → ... → compile → ... → test → package
# install 不会执行
```

### Phase vs Goal（阶段 vs 目标）

```
Phase（阶段）：生命周期中的一个步骤（如 compile、test、package）
Goal（目标）：插件的具体功能（如 compiler:compile、surefire:test）

关系：
  Phase 本身不做任何事，它「绑定」插件 Goal 来完成实际工作
```

**默认绑定关系**（部分）：

| 生命周期 Phase | 默认绑定的插件 Goal | 作用 |
|---------------|-------------------|------|
| `process-resources` | `resources:resources` | 复制 `src/main/resources` 到输出目录 |
| `compile` | `compiler:compile` | 编译 `src/main/java` |
| `test` | `surefire:test` | 运行单元测试 |
| `package` | `jar:jar`（或 `war:war`） | 打包 |
| `install` | `install:install` | 安装到本地仓库 |
| `deploy` | `deploy:deploy` | 推送到远程仓库 |

### 插件类型

```xml
<!-- 1. Build Plugins（构建插件）—— 在 pom.xml 的 <build><plugins> 中配置 -->
<build>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-compiler-plugin</artifactId>
            <version>3.10.1</version>
            <configuration>
                <source>11</source>    <!-- 源码 Java 版本 -->
                <target>11</target>    <!-- 目标字节码版本 -->
            </configuration>
        </plugin>
    </plugins>
</build>

<!-- 2. Reporting Plugins（报告插件）—— 在 <reporting><plugins> 中配置 -->
<!-- 用于 mvn site 生成项目报告 -->
<reporting>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-javadoc-plugin</artifactId>
            <version>3.4.0</version>
        </plugin>
    </plugins>
</reporting>
```

## 深入原理

### Maven Lifecycle vs Gradle Tasks

| 维度 | Maven | Gradle |
|------|--------|--------|
| 构建模型 | 固定生命周期（三套） | 完全自定义 Task 依赖图 |
| 灵活性 | 低（只能改插件配置） | 高（可以定义任意 Task 和依赖关系） |
| 学习曲线 | 低（命令固定：`mvn clean package`） | 高（需要理解 Task 依赖模型） |
| 增量构建 | 弱（需要插件支持） | 强（Task 输入/输出变化检测） |
| 自定义构建逻辑 | 需要写 Maven 插件（Java/Mojo） | 直接在 build.gradle 中写 Groovy/Kotlin |

**为什么 Maven 的生命周期是优势？**

统一！任何人拿到 Maven 项目，都知道：
- `mvn test` 跑测试
- `mvn package` 打包
- `mvn install` 安装到本地仓库

Gradle 项目可以自定义 Task，新人可能不知道该跑哪个 Task。

**为什么 Gradle 的 Task 模型更灵活？**

Maven 的生命周期一旦定义就无法修改。如果你想在 `compile` 之前加一个自定义步骤，只能写插件并绑定到某个 Phase。Gradle 可以直接：
```groovy
task myTask {
    dependsOn compileJava   // 定义在哪个 Task 之后运行
}
```

### Maven Plugin vs Ant Task

| 维度 | Maven Plugin | Ant Task |
|------|--------------|----------|
| 约定 | 强（有默认绑定） | 无（全靠自己配置） |
| 复用性 | 高（插件可以发布到 Maven Central） | 低（通常项目内自定义） |
| 配置方式 | 声明式（XML） | 命令式（XML 脚本） |

Ant 是命令式的：`svn compile` → 你需要写 `javac` 任务的详细配置。Maven 是声明式的：配置插件，生命周期自动绑定。

## 正确使用方式

### 1. 常用构建命令组合

```bash
# 完整构建（编译 + 测试 + 打包 + 安装到本地仓库）
mvn clean install

# 跳过测试（慎用，只用于快速验证编译）
mvn clean package -DskipTests

# 只跑测试，不打包
mvn test

# 离线构建（不检查远程仓库更新，速度快）
mvn clean install -o

# 只构建指定模块（多模块项目）
mvn clean install -pl user-service -am
# -pl：指定模块
# -am：同时构建该模块依赖的其他模块
```

### 2. 配置编译器插件（指定 Java 版本）

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <version>3.10.1</version>
    <configuration>
        <source>11</source>
        <target>11</target>
        <encoding>UTF-8</encoding>
        <!-- Java 9+ 推荐用 release 代替 source/target -->
        <!-- <release>11</release> -->
    </configuration>
</plugin>
```

### 3. 用 `maven-surefire-plugin` 控制测试

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-surefire-plugin</artifactId>
    <version>3.0.0-M7</version>
    <configuration>
        <!-- 跳过测试 -->
        <skipTests>false</skipTests>
        <!-- 只运行指定的测试类 -->
        <includes>
            <include>**/*Test.java</include>
        </includes>
        <!-- 排除指定的测试类 -->
        <excludes>
            <exclude>**/*IntegrationTest.java</exclude>
        </excludes>
        <!-- JVM 参数 -->
        <argLine>-Xmx512m</argLine>
    </configuration>
</plugin>
```

### 4. 用 `maven-shade-plugin` 打 fat jar（包含所有依赖）

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-shade-plugin</artifactId>
    <version>3.4.0</version>
    <executions>
        <execution>
            <phase>package</phase>   <!-- 绑定到 package 阶段 -->
            <goals>
                <goal>shade</goal>
            </goals>
            <configuration>
                <!-- 指定 Main-Class -->
                <transformers>
                    <transformer implementation="org.apache.maven.plugins.shade.resource.ManifestResourceTransformer">
                        <mainClass>com.example.Main</mainClass>
                    </transformer>
                </transformers>
            </configuration>
        </execution>
    </executions>
</plugin>
```

## 边界情况和坑

### 1. `mvn clean` 删除的是 `target/` 目录

```bash
mvn clean   # 删除 target/ 目录

# 问题：如果 target/ 里有你手动放的文件（不推荐），会被删除
# 更严重：如果 target/ 是软链接，clean 可能不按预期工作
```

**解决**：不要往 `target/` 里手动放东西，它是构建输出目录。

### 2. 插件版本不指定会导致不可重现构建

```xml
<!-- ❌ 不指定版本 -->
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <!-- 没有写 version -->
</plugin>

# Maven 会自动使用「最新版本」，今天构建成功，明天可能失败（插件升级了）
```

**正确做法**：所有插件都指定版本，或者用 `<pluginManagement>` 统一锁定。

### 3. `package` 和 `install` 的区别容易被忽略

```
mvn package：
  → 只打包到 target/ 目录
  → 不安装到本地仓库（~/.m2/repository）
  → 其他项目依赖这个模块时会找不到

mvn install：
  → 打包 + 安装到本地仓库
  → 其他本地项目可以依赖这个模块
```

**多模块项目**：改了公共模块（如 `common`），必须 `mvn install`，否则依赖它的模块用的还是旧版本。

### 4. 插件目标（Goal）可以直接执行，不触发生命周期

```bash
# 直接执行插件目标（不触发之前的 Phase）
mvn compiler:compile   # 只编译，不跑 test
mvn surefire:test      # 只跑测试，不编译（假设已经编译过了）

# 用途：调试插件问题时，可以单独执行某个 Goal
```

### 5. `mvn site` 生成的文档很慢，且很少用

```
现实：
  mvn site 生成的文档看起来很专业，但维护成本高
  大多数团队直接用 Markdown + GitLab/GitHub Pages
  site 生命周期基本被废弃了
```

**建议**：不需要配置 `site` 生命周期，用 ReadTheDocs 或 GitLab Pages 代替。
