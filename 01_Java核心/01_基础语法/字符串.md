# 字符串

> String 是 Java 中最特殊的类——不可变、final、有专属的常量池、享有编译器特殊待遇。理解 String 的设计，是理解 Java 内存模型和 JVM 优化的重要入口。

---

## 这个问题为什么存在？

字符串是程序中最常用的数据类型。Java 的 String 需要同时解决三个问题：

1. **内存效率**——程序中大量重复字符串（状态码、配置 key、URL），每个都创建新对象会浪费大量内存
2. **安全性**——文件路径、数据库连接、网络参数都是字符串，如果内容可以被篡改，整个系统的安全防线就会崩溃
3. **线程安全**——字符串几乎会在所有线程间共享，每次读写都要加锁的性能代价不可接受

**没有不可变性会怎样？** 假设 String 是可变的：

```java
String url = "jdbc:mysql://localhost:3306/mydb";
// 另一个线程或模块持有 url 的引用，并修改了内容
// → 数据库连接指向了错误的地址
// → 所有基于 url.hashCode() 的缓存全部失效
```

---

## 它是怎么解决问题的？

### 不可变性设计

String 被声明为 `final class`（不可继承），内部所有字段都是 `private final`（不可修改）。**final class** 详见 [[01_Java核心/01_基础语法/final关键字|final关键字]]。

```java
public final class String
    implements java.io.Serializable, Comparable<String>, CharSequence {

    // JDK 8：char 数组存储
    private final char value[];

    // JDK 9+：byte 数组 + coder 标识
    private final byte[] value;
    private final byte coder;  // LATIN1 或 UTF16
}
```

JDK 9 的改动是因为：大多数字符串只包含 Latin-1 字符（每个只需 1 字节），用 `char[]` 存储会浪费一半内存。改为 `byte[] + coder` 后，Latin-1 字符串的内存占用直接减半。

不可变带来的三个好处：

```
1. 字符串常量池可以安全共享引用——值不会被篡改
2. hashCode 只需计算一次，缓存复用——因为内容永不变
3. 天然线程安全——不需要同步，任何线程读到的都是同一个值
```

### 字符串常量池

```
字面量创建："hello"
┌─────────────────────────────────────────┐
│ 1. 检查常量池是否有"hello"              │
│         │                               │
│    ┌────┴────┐                          │
│    │ 找到？  │                          │
│    └────┬────┘                          │
│       是 │     否                       │
│         ▼       ▼                       │
│   返回引用   在常量池创建"hello"         │
│              返回新引用                  │
└─────────────────────────────────────────┘

new String("hello")
┌─────────────────────────────────────────┐
│ 1. 先确保常量池中有"hello"（字面量）    │
│ 2. 在堆中创建新的 String 对象           │
│ 3. 返回堆中的引用                        │
└─────────────────────────────────────────┘
```

```java
String a = "hello";            // 常量池
String b = "hello";            // 常量池，和 a 同一引用
String c = new String("hello"); // 堆上新建对象

a == b;       // true（同一常量池引用）
a == c;       // false（c 在堆上，a 在常量池）
a.equals(c);  // true（值相同）
```

```
常量池的实际影响：
- 大量动态拼接的唯一字符串 → 内存浪费
- JDK 7 之前常量池在 PermGen，可能导致 PermGen OOM
- JDK 7+ 移到堆中，不会被单独撑爆，但大量 intern() 仍会增加 GC 压力
```

### String、StringBuilder、StringBuffer

```java
// String：不可变，每次操作生成新对象
String s = "hello";
s = s + " world";  // 原来的 "hello" 对象还在内存里，新建了一个 "hello world"

// StringBuilder：可变，非线程安全
StringBuilder sb = new StringBuilder();
sb.append("hello").append(" world");  // 同一个对象上操作

// StringBuffer：可变，线程安全（synchronized）
StringBuffer sf = new StringBuffer();
sf.append("hello").append(" world");  // 每次 append 都有锁开销
```

```
性能差异：单线程下 StringBuilder 比 StringBuffer 快约 30%，
因为避免了 synchronized 的 lock/unlock 开销。

实际选择：
- 不需要频繁修改 → String
- 单线程频繁拼接 → StringBuilder
- 多线程共享同一个字符串缓冲区 → StringBuffer（但这种场景极少见）
```

### 编码问题：UTF-16 与 Unicode

Java 内部使用 UTF-16 编码。一个 `char` 是一个 UTF-16 代码单元（2 字节），但一个 Unicode 字符（码点）可能需要 1 个或 2 个代码单元：

```java
// 基本多文种平面（BMP）内的字符：1 个 char
"a".length();           // 1
"中".length();          // 1

// 补充字符（emoji、生僻字）：2 个 char
"🎯".length();          // 2（一个字符，但占两个代码单元）

// 获取真实字符数
"🎯hello".codePointCount(0, "🎯hello".length());  // 6

// 正确遍历（处理补充字符）
for (int i = 0; i < str.length(); ) {
    int cp = str.codePointAt(i);
    System.out.println(new String(Character.toChars(cp)));
    i += Character.charCount(cp);  // 补充字符跳两个 char
}
```

## 深入原理

### intern() 方法的实现与演变

```java
String s1 = new String("hello");        // 堆中对象
String s2 = s1.intern();                 // 将字符串放入常量池，返回常量池引用
```

```
JDK 6：intern() 把字符串复制到 PermGen 的字符串常量池
  → 大量调用会导致 PermGen OOM
  → 解决方案：尽量不用 intern()

JDK 7+：intern() 把字符串引用放入堆的 StringTable（哈希表）
  → 不再复制字符数组，只是多了一个引用
  → 但大量不同字符串仍会导致 StringTable 膨胀，增加 Full GC 压力
  → 可通过 -XX:StringTableSize 调整桶数量

实际建议：
- 正常开发不需要手动调用 intern()
- 如果有大量重复字符串需要去重，考虑 Guava Interner 或自己维护 Set
```

### substring() 的内存泄漏（JDK 6）

```java
// JDK 6：substring() 返回的字符串共享原字符数组
String huge = new String(new char[10_000_000]);  // 10M
String tiny = huge.substring(0, 1);
// tiny 仍持有 10M 字符数组的引用！原数组无法被 GC

// JDK 7+ 修复：创建新数组
String substring(int begin, int endIndex) {
    return new String(value, begin, subLen, coder);  // 复制字符，不共享
}
```

JDK 6 选择共享数组是因为 substring 是高频操作，复制开销大。但实际场景中，大字符串取小子串的情况远多于大字符串取大子串，导致内存浪费的代价超过了复制数组的开销。JDK 7 选择了正确性优先。

### String 拼接的编译器优化

```java
// 常量拼接：编译期直接合并
String s = "hello" + "world";  // 编译后：String s = "helloworld";

// 变量拼接：JDK 8 生成 StringBuilder
String a = "hello";
String s = a + "world";  // 编译后：new StringBuilder().append(a).append("world").toString();

// 变量拼接：JDK 9+ 使用 invokedynamic
String a = "hello";
String s = a + "world";  // 编译后：invokedynamic makeConcatWithConstants
```

JDK 9 的改进避免了每次拼接都创建 StringBuilder 对象，由 JVM 运行时决定最优策略。但在循环内多次拼接，手动 StringBuilder 仍然更可控。

## 正确使用方式

### 字符串判等

```java
// 永远用 equals()，不用 ==
String a = new String("hello");
String b = "hello";
a == b;        // false：引用比较
a.equals(b);   // true：内容比较

// 常量放前面，避免 NPE
if ("hello".equals(userInput)) { }  // ✓ userInput 为 null 也不会 NPE

// 更简洁：Objects.equals() 自动处理 null
if (Objects.equals(userInput, "hello")) { }
```

### 字符串拼接的选择

```java
// 少量固定拼接：直接 +（可读性最好）
String sql = "SELECT * FROM user WHERE id = " + userId;

// 循环内拼接：手动 StringBuilder
StringBuilder sb = new StringBuilder();
for (OrderItem item : items) {
    sb.append(item.getName()).append(": ").append(item.getPrice());
}

// 格式化：MessageFormat 或 String.format
String msg = String.format("%s共%d件，总价%.2f元", name, count, total);

// 日志：用占位符，不要提前拼接
log.info("用户{}下单成功，金额{}", userId, amount);  // ✓
log.info("用户" + userId + "下单成功，金额" + amount);  // ✗ 即使日志级别不打印，拼接也会执行
```

### 不要用 new String() 创建字符串

```java
String s = new String("hello");  // 多创建了一个对象，毫无意义
String s = "hello";               // 直接使用常量池
```

## 边界情况和坑

### 1. split() 的正则陷阱

```java
"a,b,c,".split(",");     // ["a", "b", "c"] — 末尾空字符串被丢弃
"a.b.c".split(".");       // [] — 点号是正则元字符，匹配任意字符
"a|b|c".split("\\|");    // ["a", "b", "c"] — 竖线需要转义

// 保留末尾空字符串
"a,b,c,".split(",", -1);  // ["a", "b", "c", ""]
```

split() 的第一个参数是正则表达式，不是普通字符。`"."` 匹配的是"任意字符"，所以 `"a.b.c".split(".")` 会把每个字符之间的"空隙"都匹配到，返回空数组。对于简单的固定分隔符，用 `indexOf()` + `substring()` 性能更好。

### 2. String.valueOf(null) vs toString()

```java
Object obj = null;
String.valueOf(obj);   // "null"（字符串）
obj.toString();        // NPE！

int[] arr = {1, 2, 3};
Arrays.toString(arr);  // "[1, 2, 3]"
arr.toString();        // "[I@15db9742" — 数组的默认 toString() 打印的是内存地址
```

`String.valueOf()` 对 null 做了防御处理，返回字符串 `"null"`。而 `toString()` 是空指针炸弹。数组的 `toString()` 不会打印内容，必须用 `Arrays.toString()` 或 `Arrays.deepToString()`。

### 3. new String(byte[]) 的编码陷阱

```java
byte[] bytes = getFromNetwork();
String s = new String(bytes);  // 使用平台默认编码，可能乱码

// 正确：明确指定编码
String s = new String(bytes, StandardCharsets.UTF_8);

// 反向：字符串转字节也要指定编码
byte[] out = s.getBytes(StandardCharsets.UTF_8);
```

`getBytes()` 无参版本使用平台默认编码（Windows 是 GBK，Linux 通常是 UTF-8），代码在不同环境运行会出问题。始终显式指定编码。

### 4. 循环内拼接的性能陷阱

```java
// 每次循环都创建新的 StringBuilder + 新的 String 对象
String result = "";
for (String s : list) {
    result += s;  // O(n²) 的时间复杂度和对象创建
}

// 复用一个 StringBuilder：O(n)
StringBuilder sb = new StringBuilder();
for (String s : list) {
    sb.append(s);
}
```

单次 `+` 拼接编译器会优化为 StringBuilder，但在循环内每次 `+=` 都是一个独立的拼接操作，编译器无法跨循环优化。
