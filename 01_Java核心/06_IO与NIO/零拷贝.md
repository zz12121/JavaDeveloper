# 零拷贝

> 零拷贝不是"零次拷贝"，而是"减少甚至消除 CPU 参与的数据拷贝"——让数据直接在内核态和设备之间流动，绕过用户空间，大幅提升 IO 吞吐量。Kafka 和 RocketMQ 的高性能都依赖零拷贝。

---

## 这个问题为什么存在？

传统 IO 中，数据从磁盘到网络需要经过**4 次拷贝和 4 次上下文切换**，其中 CPU 拷贝是纯开销——数据只是从一个缓冲区搬到另一个缓冲区，没有任何加工。

```
传统 IO（read + write）：

磁盘
  ↓ DMA 拷贝（1次拷贝）
内核缓冲区（Page Cache）
  ↓ CPU 拷贝（2次拷贝）  ← 浪费！CPU 只是搬运工
用户空间缓冲区（JVM 堆）
  ↓ CPU 拷贝（3次拷贝）  ← 浪费！CPU 又搬了一次
Socket 缓冲区
  ↓ DMA 拷贝（4次拷贝）
网卡
```

**核心问题**：CPU 拷贝（第 2、3 次）完全没有必要——数据不需要在用户空间做任何修改，只是路过。

---

## 它是怎么解决问题的？

### mmap：内存映射（3 次拷贝）

```
磁盘
  ↓ DMA 拷贝
内核缓冲区（Page Cache） ← mmap 让用户进程直接映射这块内存
  ↓ （没有 CPU 拷贝！用户进程直接读写 Page Cache）
Socket 缓冲区
  ↓ DMA 拷贝
网卡
```

**mmap 的原理**：让用户空间和内核空间共享同一块物理内存。应用程序读写这块内存就等于读写内核缓冲区，省去了一次 CPU 拷贝。

```java
// Java NIO 的 mmap
FileChannel channel = FileChannel.open(Paths.get("file.txt"),
    StandardOpenOption.READ);
MappedByteBuffer mappedBuf = channel.map(
    FileChannel.MapMode.READ_ONLY, 0, channel.size());

// mappedBuf 直接映射到内核 Page Cache
// 读写 mappedBuf 就是读写 Page Cache，没有额外拷贝
```

**mmap 的优势**：
- 减少一次 CPU 拷贝（从 4 次减到 3 次）
- 上下文切换次数不变（还是 4 次）

**mmap 的代价**：
- 映射大文件时占用大量虚拟地址空间
- 如果文件比物理内存大，会导致 page fault 频繁
- 并发修改同一 mmap 区域有信号量开销

### sendfile：真正的零拷贝（2 次拷贝）

```
磁盘
  ↓ DMA 拷贝
内核缓冲区（Page Cache）
  ↓ DMA 拷贝（带 Scatter/Gather）  ← CPU 完全不参与！
网卡
```

```java
// Java NIO 的 sendfile
FileChannel fileChannel = FileChannel.open(Paths.get("file.txt"),
    StandardOpenOption.READ);
long transferred = fileChannel.transferTo(0, fileChannel.size(), socketChannel);
// 底层调用 Linux sendfile(2)
```

**sendfile 的原理**（Linux 2.1+）：
1. DMA 把数据从磁盘读到 Page Cache
2. **不拷贝到 Socket 缓冲区**，而是把 Page Cache 的地址和长度信息附加到 Socket 缓冲区
3. DMA 根据 Socket 缓冲区中的描述信息，直接把数据从 Page Cache 传到网卡

**CPU 完全不参与数据搬运**——这就是"零拷贝"名称的来源。

**sendfile + Scatter/Gather**（Linux 2.4+）：
- 如果网卡支持 DMA Scatter/Gather，连 Socket 缓冲区描述信息都不需要了
- DMA 控制器直接从 Page Cache 中读取数据和偏移量，传到网卡
- 这是目前最高效的文件传输方式

### 各方案对比

| 方案 | 拷贝次数 | CPU 拷贝 | 上下文切换 | Java API |
|------|---------|---------|----------|---------|
| 传统 read+write | 4 | 2 | 4 | `InputStream`/`OutputStream` |
| mmap | 3 | 1 | 4 | `FileChannel.map()` |
| sendfile | 2 | 0 | 2 | `FileChannel.transferTo()` |
| sendfile+SG-DMA | 2 | 0 | 2 | `FileChannel.transferTo()` |

## 深入原理

### Page Cache 的角色

```
为什么读文件时数据要先经过 Page Cache？
  → 操作系统为了减少磁盘 IO，把空闲内存用作磁盘缓存
  → 读文件时先检查 Page Cache，命中则直接返回（内存速度）
  → 未命中则从磁盘读入 Page Cache

零拷贝的优化都是围绕 Page Cache 展开的：
  → mmap：让用户进程直接访问 Page Cache
  → sendfile：直接从 Page Cache 传到网卡
```

**Page Cache 的影响**：
- 顺序读性能极高（预读机制）
- 随机读性能一般（预读失效）
- 写入 Page Cache 后进程即可返回（异步刷盘）

### Kafka 的零拷贝

```
Kafka 消费者拉取消息：

传统方式（无零拷贝）：
  磁盘 → Page Cache → 用户空间 → Socket 缓冲区 → 网卡
  CPU 参与 2 次拷贝

Kafka 的方式（sendfile 零拷贝）：
  磁盘 → Page Cache → 网卡
  CPU 零次拷贝

Kafka 为什么快？三个关键：
  1. 顺序写磁盘（Page Cache + 异步刷盘）
  2. 零拷贝（sendfile 消费者拉取）
  3. 批量压缩（减少网络传输量）
```

### RocketMQ 的零拷贝

```
RocketMQ 存储层用 mmap（不是 sendfile）：
  → 因为 RocketMQ 需要在用户空间处理消息
  → mmap 让用户空间直接读写 Page Cache
  → 消费时直接从 Page Cache 读取

为什么不用 sendfile？
  → sendfile 数据不经过用户空间，无法在传输前修改消息
  → RocketMQ 需要对消息做过滤、标记等处理
```

## 正确使用方式

### transferTo 的使用

```java
// ✅ 文件传输（零拷贝）
try (FileChannel in = FileChannel.open(Paths.get("source.txt"));
     FileChannel out = FileChannel.open(Paths.get("dest.txt"),
         StandardOpenOption.WRITE, StandardOpenOption.CREATE)) {
    long transferred = in.transferTo(0, in.size(), out);
    System.out.println("传输了 " + transferred + " 字节");
}
```

**注意**：`transferTo()` 不保证一次传输完所有数据，需要循环调用：
```java
long position = 0;
long count = in.size();
while (position < count) {
    long transferred = in.transferTo(position, count - position, out);
    position += transferred;
}
```

### mmap 的使用

```java
// ✅ 大文件随机读（mmap 比 read 高效）
try (FileChannel channel = FileChannel.open(Paths.get("large.dat"))) {
    MappedByteBuffer buf = channel.map(
        FileChannel.MapMode.READ_ONLY, 0, channel.size());

    // 直接通过内存地址访问文件内容
    int value = buf.getInt(100);  // 读第 100 字节的 int
    value = buf.getInt(10000);    // 读第 10000 字节的 int
    // 操作系统自动处理缺页中断
}
```

### 选型决策

```
文件传输（磁盘 → 网络）：
  → sendfile（transferTo）——最快，零 CPU 拷贝

文件随机读：
  → mmap ——直接访问任意偏移，操作系统处理缺页中断

文件顺序读：
  → BufferedInputStream（小文件）或 mmap（大文件）

需要修改数据再发送：
  → mmap（数据经过用户空间，可以修改）
  → 不能用 sendfile（不经过用户空间）
```

## 边界情况和坑

### 坑 1：mmap 大文件导致 OOM

```java
// ❌ 映射 10GB 文件
FileChannel channel = FileChannel.open(Paths.get("huge.dat"));
MappedByteBuffer buf = channel.map(MapMode.READ_ONLY, 0, 10_000_000_000L);
// 虽然不会立即消耗 10GB 物理内存，但会占用 10GB 虚拟地址空间
// 32 位 JVM 上直接 OutOfMemoryError
```

**成因**：mmap 映射的是虚拟地址空间。32 位 JVM 虚拟地址空间只有 4GB，映射大文件会导致地址空间耗尽。64 位 JVM 影响较小，但仍然需要注意。

**解决方案**：分段映射（每次映射一小段）：
```java
long fileSize = channel.size();
long chunkSize = Integer.MAX_VALUE;  // 每次最多映射 ~2GB
for (long offset = 0; offset < fileSize; offset += chunkSize) {
    long size = Math.min(chunkSize, fileSize - offset);
    MappedByteBuffer buf = channel.map(MapMode.READ_ONLY, offset, size);
    // 处理这一段
}
```

### 坑 2：mmap 文件在写入时进程崩溃导致文件损坏

```java
// READ_WRITE 模式的 mmap
MappedByteBuffer buf = channel.map(MapMode.READ_WRITE, 0, 1024);
buf.putInt(0, 42);
// 数据写入 Page Cache，但还没刷到磁盘
// 如果此时进程崩溃或断电，数据可能丢失
```

**成因**：mmap 写入的是 Page Cache，不是直接写磁盘。操作系统会在某个时刻异步刷盘。

**解决方案**：需要保证持久化时，调用 `MappedByteBuffer.force()`：
```java
buf.putInt(0, 42);
buf.force();  // 强制刷盘（类似 fsync）
```

### 坑 3：transferTo 在某些场景下退化为传统拷贝

```java
// 文件 → 文件：零拷贝
in.transferTo(0, in.size(), fileOut);  // ✅ 用 sendfile

// 文件 → 自定义处理 → 网络：不能零拷贝
in.transferTo(0, in.size(), socketChannel);
// 中间不需要处理 → ✅ 零拷贝
// 如果中间需要处理数据（加密、压缩）→ 必须经过用户空间
```

### 坑 4：mmap 的内存可见性问题

```java
// 进程 A 通过 mmap 写入
MappedByteBuffer bufA = channel.map(MapMode.READ_WRITE, 0, 1024);
bufA.putInt(0, 42);

// 进程 B 通过 mmap 读取（同一个文件）
MappedByteBuffer bufB = channel.map(MapMode.READ_ONLY, 0, 1024);
int value = bufB.getInt(0);  // 不一定能读到 42！
```

**成因**：mmap 的可见性依赖操作系统的内存一致性模型。进程 A 写入 Page Cache 后，进程 B 不一定能立即看到修改（尤其是不同机器上挂载的共享文件系统）。

### 坑 5：MappedByteBuffer 的 GC 问题

```java
// MappedByteBuffer 映射的内存不受 JVM GC 直接管理
// 在 Java 8 中，即使 MappedByteBuffer 对象被 GC，映射的内存可能不会立即释放
// 需要依赖 Cleaner 机制（不确定性）

// Java 9+ 改善了这个问题，但仍建议及时解除映射
```
