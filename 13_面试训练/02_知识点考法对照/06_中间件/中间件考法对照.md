## Redis 数据结构与底层编码

**关联知识文档**：[[07_中间件/01_Redis/Redis数据结构]]

**第一问**：「Redis 的 String 类型底层是什么？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | embstr 和 raw 什么时候切换？ | 44 字节阈值（JDK 9+ 是 43）|
| 第二层 | SDS 相比 C 字符串有什么优势？ | 空间预分配 / 惰性释放 / 二进制安全 / O(1) 获取长度 |
| 第三层 | Redis 有几种数据编码？RedisObject 结构是什么？ | type + encoding + ptr |

**实战型考法**：「一个 String 类型的 Key 最大能存多大值？」

**面试话术总结**：

- **RedisObject**：每个 Redis 值都是一个 RedisObject，包含 `type`（5 种大类型）、`encoding`（底层编码）、`ptr`（指向实际数据）
- **String 编码**：`int`（纯整数，long 范围）→ `embstr`（≤44 字节，RedisObject 和 SDS 一次分配）→ `raw`（>44 字节，两次分配）
- **Hash 编码**：`ziplist`（元素少且小）→ `hashtable`（元素多）
- **List 编码**：`quicklist`（双向链表 + ziplist，控制每个节点大小）
- **Set 编码**：`intset`（全是整数且数量少）→ `hashtable`
- **ZSet 编码**：`ziplist`（元素少且小）→ `skiplist + hashtable`
- **SDS 优势**：O(1) 获取长度（C 字符串 O(n)）；空间预分配（修改时预留空间减少分配次数）；惰性释放（缩短时不立即释放，等下次分配复用）；二进制安全（`\0` 不是结束标志）

## Redis 持久化

**关联知识文档**：[[07_中间件/01_Redis/Redis持久化]]

**第一问**：「Redis 持久化方式有哪些？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | RDB 和 AOF 各自的优缺点？ | 性能 vs 数据安全性 |
| 第二层 | 为什么说 RDB 可能丢数据？ | 快照间隔内宕机 |
| 第三层 | 混合持久化是什么？ | Redis 4.0+，RDB 头部 + AOF 增量尾部 |

**实战型考法**：「你用的什么持久化策略？为什么？」

**面试话术总结**：

- **RDB**：fork 子进程 → 子进程写内存快照到 dump.rdb。优点：文件小、恢复快、对性能影响小。缺点：间隔内数据可能丢失
- **AOF**：记录每条写命令（`appendonly yes`）。三种刷盘策略：`always`（每条 fsync，最安全但最慢）/ `everysec`（每秒 fsync，最多丢 1 秒，推荐）/ `no`（OS 决定，可能丢失较多）
- **AOF 重写**：AOF 文件过大时，Redis 会重写（bgrewriteaof）→ 合并多条命令为一条（如多次 `INCR` → `SET`），减小文件体积
- **混合持久化**（Redis 4.0+ `aof-use-rdb-preamble yes`）：AOF 重写时先写 RDB 快照（二进制，体积小），之后的增量命令仍用 AOF 格式 → 兼顾恢复速度和数据安全

## Redis 高可用

**关联知识文档**：[[07_中间件/01_Redis/Redis高可用]]

**第一问**：「Redis 有哪些高可用方案？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 主从复制是怎么做的？ | 全量同步（RDB）+ 增量同步（repl_backlog）|
| 第二层 | Sentinel 是怎么实现故障转移的？ | 主观下线 + 客观下线 + Leader 选举 |
| 第三层 | Redis Cluster 的槽位机制？ | 16384 个槽，每个主节点负责一部分 |

**面试话术总结**：

- **主从复制**：全量同步（Slave 发送 PSYNC → Master fork 子进程生成 RDB → 发送 RDB + 缓冲区增量）；增量同步（复制偏移量 + repl_backlog 环形缓冲区）
- **Sentinel 故障转移**：
  1. Sentinel 每秒 ping 主节点和其他 Sentinel
  2. 超过 `down-after-milliseconds` 未响应 → 主观下线
  3. 超过半数 Sentinel 确认 → 客观下线
  4. Sentinel 选举 Leader（Raft 算法）→ Leader 执行故障转移
  5. 选一个从节点做新主节点（优先级、复制偏移量、runid）
- **Redis Cluster**：16384 个 Hash Slot，`CRC16(key) % 16384` 决定数据存到哪个节点。节点间 Gossip 协议通信。客户端收到 MOVED/ASK 重定向

## Redis 分布式锁

**关联知识文档**：[[07_中间件/01_Redis/Redis分布式锁]]

**第一问**：「Redis 分布式锁怎么实现？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | `SETNX + EXPIRE` 有什么问题？ | 非原子操作 / 锁过期但业务未完成 |
| 第二层 | Redisson 的看门狗机制是什么？ | 后台线程自动续期 |
| 第三层 | RedLock 为什么被批评？ | 时钟漂移 / 实际意义不大 |

**实战型考法**：「你在项目里用过分布式锁吗？用的什么方案？」

**面试话术总结**：

- **基础实现**：`SET key value NX EX 30`（原子操作，加锁+过期时间一体）
- **锁续期**：Redisson 看门狗（Watchdog）：加锁时不指定过期时间 → 后台线程每 10s（过期时间的 1/3）检查锁是否存在 → 存在则续期 30s → 业务执行完主动释放；如果宕机看门狗停止运行 → 锁自动过期
- **RedLock**（Redis 作者提出）：向多个（≥3）独立 Redis 节点获取锁，多数成功才算获取成功；被批评原因：依赖时钟同步、实际部署中多个 Redis 实例可能物理机相同
- **实际选型**：大多数场景单节点 Redisson 够用；跨数据中心场景考虑 ZooKeeper（CP）或 etcd

## Kafka 高可靠与高吞吐

**关联知识文档**：[[07_中间件/02_Kafka/Kafka]]

**第一问**：「Kafka 为什么能做到高吞吐？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 顺序写磁盘为什么快？ | 操作系统页缓存 + 预读 |
| 第二层 | 零拷贝是什么？减少了哪几步拷贝？ | sendfile() / mmap |
| 第三层 | ISR 是什么？为什么需要它？ | 副本同步列表 / 数据可靠性 |

**实战型考法**：「Kafka 消息丢失怎么排查和防止？」

**面试话术总结**：

- **高吞吐原因**：
  - **顺序写**：追加写入（Append-Only Log），利用磁盘顺序写性能（远超随机写）
  - **页缓存**：Kafka 不直接 `fsync`，依赖 OS 页缓存，由 OS 异步刷盘
  - **零拷贝**：`sendfile()` 零拷贝（数据从页缓存直接到网卡，跳过用户态），普通 IO 需要 4 次拷贝（磁盘→内核→用户态→Socket 缓冲区→网卡），sendfile 只有 2 次
  - **批处理**：生产者批量发送、消费者批量拉取
- **ISR（In-Sync Replicas）**：与 Leader 保持同步的副本集合。`acks=all`（等所有 ISR 副本确认）→ 最安全但延迟高；`acks=1`（等 Leader 确认）→ 折中；`acks=0`（不等确认）→ 最快但可能丢
- **消息防丢**：生产者端 `acks=all` + `retries>0` + 幂等（`enable.idempotence=true`）；Broker 端 `min.insync.replicas ≥ 2`；消费者端手动提交 offset

## RabbitMQ 消息模型

**关联知识文档**：[[07_中间件/02_消息队列/RabbitMQ架构与交换机]]

**第一问**：「RabbitMQ 和 Kafka 有什么区别？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | RabbitMQ 的消息投递模型？ | Exchange → Queue → Consumer |
| 第二层 | RabbitMQ 怎么保证消息不丢失？ | 确认机制 + 持久化 + 消费者手动 ACK |
| 第三层 | 消息堆积怎么处理？ | 死信队列 + 消费者扩容 + 惰性队列 |

**面试话术总结**：

- **核心区别**：
  | 维度 | RabbitMQ | Kafka |
  |------|----------|-------|
  | 模型 | 消息队列（消费后删除）| 发布-订阅日志（可重复消费）|
  | 吞吐量 | 万级 | 百万级 |
  | 延迟 | 微秒级 | 毫秒级 |
  | 适用 | 业务消息、RPC | 日志、大数据流 |
- **RabbitMQ 消息防丢**：
  - 生产者端：`confirm` 模式（Broker 收到消息后回调确认）
  - Broker 端：Queue 持久化 + Message 持久化
  - 消费者端：手动 ACK（`basic.ack`），不要自动确认
- **死信队列（DLX）**：消息被拒绝（`basic.reject` requeue=false）/ TTL 过期 / 队列满 → 进入死信交换机 → 路由到死信队列 → 用于延迟消费或异常消息处理

## Elasticsearch 基础

**关联知识文档**：[[07_中间件/03_Elasticsearch/Elasticsearch]]

**第一问**：「Elasticsearch 的倒排索引是什么？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 正排索引和倒排索引的区别？ | 文档→词 vs 词→文档 |
| 第二层 | 分词器是什么？IK 分词器有哪些模式？ | ik_smart（粗粒度）/ ik_max_word（细粒度）|
| 第三层 | Elasticsearch 和 MySQL 什么时候选哪个？ | 全文检索 vs 精确查询 |

**面试话术总结**：

- **倒排索引**：将文档内容分词 → 建立词语到文档 ID 的映射表（词 → [doc1, doc3, doc7]）；搜索时先查倒排索引找到文档 ID，再从正排索引获取完整文档
- **分词器流程**：Character Filters（字符处理）→ Tokenizer（分词）→ Token Filters（大小写、同义词、停用词）
- **选型原则**：需要全文检索、模糊搜索、相关性排序 → ES；需要事务、JOIN、精确查询 → MySQL；可以组合使用（MySQL 做主存储，ES 做搜索引擎）

## ZooKeeper 基础

**关联知识文档**：[[07_中间件/04_ZooKeeper/ZooKeeper]]

**第一问**：「ZooKeeper 是什么？有什么用？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | ZAB 协议是什么？ | 写操作通过 Leader 广播保证顺序一致性 |
| 第二层 | ZooKeeper 的临时节点有什么用？ | 服务注册发现 / 分布式锁 / Leader 选举 |
| 第三层 | Watch 机制有什么注意事项？ | 一次性触发 / 客户端重连后需要重新注册 |

**面试话术总结**：

- **ZooKeeper 本质**：分布式协调服务，提供：配置管理、命名服务、分布式锁、集群管理、Leader 选举
- **数据模型**：树形结构（类似文件系统），节点类型：持久节点、持久顺序节点、临时节点（Session 断开自动删除）、临时顺序节点
- **ZAB 协议**：写请求发给 Leader → Leader 生成 ZXID（事务ID）→ 广播给所有 Follower → 多数确认后 Leader 提交 → Follower 同步
- **选型**：强一致性场景（如分布式锁、配置中心）→ ZK；服务注册发现 → Nacos/Eureka 更轻量
