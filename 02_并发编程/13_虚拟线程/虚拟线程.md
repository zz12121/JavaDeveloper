# 虚拟线程

> 虚拟线程（Virtual Threads）是 Java 21 正式发布的特性，也是 Project Loom 的核心成果。它让 Java 程序员终于能像写同步代码一样写出高并发程序——不需要回调、不需要 Reactive、不需要 CompletableFuture 链式编排。一句话：**用最简单的代码风格，获得最高的并发能力。**

---

## 这个问题为什么存在？

### 平台线程的困境

Java 传统的 `Thread`（现在叫"平台线程"，Platform Thread）本质是操作系统的线程：

```
一个平台线程 = 一个 OS 线程 = 一个内核调度单元

代价：
- 内存：每个线程栈 512KB~1MB，1 万个线程 ≈ 5~10GB 栈内存
- 创建成本：系统调用，涉及内核态切换
- 调度成本：线程上下文切换需要保存/恢复 CPU 寄存器，涉及内核态/用户态切换
- 数量上限：通常一台机器几千到几万个是极限

现代应用的并发需求：
- 高并发 Web 服务：10 万+ 并发请求很常见
- 每个请求可能要等 I/O（数据库、RPC、Redis）
- I/O 等待时线程被阻塞，但线程资源被白白占用

矛盾：需要大量线程来扛并发，但 OS 线程太贵了
```

### 已有的解决方案及其代价

```
方案 1：线程池（有限线程 + 队列）
  → 线程复用，但并发上限 = 线程池大小
  → 线程被 I/O 阻塞时，其他任务在队列里等
  → 需要精心调优线程池参数
  → 本质是"用吞吐换线程数"

方案 2：异步非阻塞（Netty EventLoop、CompletableFuture）
  → 少量线程处理大量连接（事件驱动）
  → 但代码风格变成回调地狱或链式编排
  → 调试困难，堆栈碎片化
  → 异常处理复杂，学习曲线陡
  → 本质是"用代码复杂度换并发能力"

方案 3：协程（Go goroutine、Kotlin coroutine）
  → 用户态轻量线程，由运行时调度
  → 几乎可以无限创建
  → 但这不是 Java 的方案……

虚拟线程 = Java 的协程实现
  → 像线程池一样是平台线程的"多路复用"
  → 像协程一样轻量，可以创建百万级
  → 代码风格 = 普通同步代码，零学习成本
```

---

## 它是怎么解决问题的？

### 核心原理：Continuation（续体）

```
虚拟线程的关键：把"线程"和"OS 线程"解耦

平台线程：一个虚拟线程被挂载到一个载体线程（Carrier Thread，即平台线程）上运行
  ┌──────────────────────────────────────┐
  │  载体线程（Carrier Thread = 平台线程）  │
  │  ┌─────────────────────────────────┐  │
  │  │  虚拟线程 A（执行中）              │  │
  │  └─────────────────────────────────┘  │
  └──────────────────────────────────────┘

遇到 I/O 阻塞时：
  ┌──────────────────────────────────────┐
  │  载体线程（Carrier Thread）            │
  │  ┌─────────┐    ┌─────────┐          │
  │  │ VT-A     │    │ VT-B    │          │
  │  │ (挂起)   │    │ (执行中) │          │
  │  └─────────┘    └─────────┘          │
  └──────────────────────────────────────┘
  VT-A 遇到阻塞 I/O → JVM 自动把 VT-A 从载体线程卸载
  载体线程空闲 → JVM 自动把 VT-B 挂载上去继续执行
  I/O 完成后 → VT-A 被重新挂载到某个载体线程上恢复执行
```

```
这个"卸载-恢复"的机制叫 Continuation（续体）：

Continuation = 一个可暂停和恢复的执行单元

当虚拟线程遇到阻塞操作（I/O、sleep、Lock 等）时：
1. JVM 保存当前执行状态（栈帧）到堆内存中
2. 把虚拟线程从载体线程卸载
3. 载体线程去执行其他虚拟线程
4. 阻塞操作完成后，从堆内存恢复状态
5. 虚拟线程被挂载到某个空闲的载体线程上继续执行

注意：这一切对开发者完全透明！你写的还是普通的同步代码。
```

### 创建虚拟线程

```java
// 方式 1：直接创建
Thread vt = Thread.startVirtualThread(() -> {
    System.out.println("Hello from virtual thread: " + Thread.currentThread());
});

// 方式 2：使用 Thread.ofVirtual() Builder（推荐，可配置）
Thread vt = Thread.ofVirtual()
    .name("my-vt", 0)       // 线程名前缀，0 表示自动编号
    .start(() -> {
        System.out.println("Hello");
    });

// 方式 3：使用虚拟线程执行器（最常用）
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 100_000; i++) {
        executor.submit(() -> {
            // 每个任务分配一个独立的虚拟线程
            var result = fetchData();  // 阻塞 I/O？没问题！
            processData(result);
        });
    }
}
// 注意：这个 Executor 不需要指定线程池大小！
// 因为每个任务一个虚拟线程，虚拟线程几乎不占资源
```

### 关键特性

```
1. 极轻量
   → 栈大小动态调整，初始只有几百字节
   → 100 万个虚拟线程 ≈ 几百 MB（vs 平台线程 1TB+）

2. 数量几乎无限
   → 不受 OS 线程数限制
   → 受限于可用内存（堆上的栈帧）

3. 调度由 JVM 管理
   → 不依赖 OS 调度器
   → M:N 调度：M 个虚拟线程映射到 N 个载体线程
   → N 默认 = CPU 核心数

4. 和平台线程的 API 完全兼容
   → Thread.sleep() 照常用
   → synchronized 照常用
   → ReentrantLock 照常用
   → 唯一例外：不支持 ThreadLocal（后面会讲）
```

### 不阻塞载体线程的操作

```
以下操作会让虚拟线程自动从载体线程卸载（不阻塞载体线程）：

✅ 大多数阻塞 I/O：
   - Socket 读写（InputStream.read, OutputStream.write）
   - 网络请求（HttpClient, HttpURLConnection）
   - 文件 I/O（大部分场景）
   - SocketChannel, ServerSocketChannel

✅ 其他阻塞操作：
   - Thread.sleep()
   - LockSupport.park()
   - CountDownLatch.await()
   - CompletableFuture.join() / .get()

⚠️ 注意：synchronized 块内的阻塞不会卸载！
   → 这是虚拟线程最大的坑之一，后面详细说
```

## 和相似方案的区别

### 虚拟线程 vs 平台线程

```
                      平台线程              虚拟线程
──────────────────────────────────────────────────────────
底层实现              OS 线程               JVM 用户态调度
栈大小                固定 512KB~1MB        动态，初始几百字节
创建成本              高（系统调用）          极低
切换成本              高（内核态切换）        低（用户态切换）
数量上限              几千~几万              百万级（受内存限制）
阻塞行为              阻塞载体线程           自动卸载
代码风格              同步（简单）           同步（同样简单！）
ThreadLocal           ✅ 支持               ❌ 不建议用
```

### 虚拟线程 vs 协程（Go/Kotlin）

```
                      Java 虚拟线程           Go goroutine        Kotlin 协程
─────────────────────────────────────────────────────────────────────
实现层                JVM 层                 语言运行时           编译器（CPS变换）
调度模型              M:N                    M:N                 M:N（Dispatcher）
阻塞语义              和平台线程一样           和 Go 线程一样       suspend/resume
代码风格              同步代码                同步代码             同步代码（suspend函数）
生态兼容              和现有 Java 代码 100%兼容  Go 生态         Kotlin 生态
学习成本              零                     零                  中等（suspend 语义）
堆栈跟踪              完整                   完整                需要特殊工具
```

### 虚拟线程 vs CompletableFuture

```
场景：调用 3 个 RPC 服务，合并结果

// CompletableFuture（链式编排）
CompletableFuture<A> futureA = CompletableFuture.supplyAsync(() -> rpcA());
CompletableFuture<B> futureB = CompletableFuture.supplyAsync(() -> rpcB());
CompletableFuture<C> futureC = CompletableFuture.supplyAsync(() -> rpcC());

CompletableFuture.allOf(futureA, futureB, futureC).join();
Result result = combine(futureA.join(), futureB.join(), futureC.join());

// 虚拟线程（最简单的方式）
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    Future<A> futureA = executor.submit(() -> rpcA());
    Future<B> futureB = executor.submit(() -> rpcB());
    Future<C> futureC = executor.submit(() -> rpcC());
    Result result = combine(futureA.get(), futureB.get(), futureC.get());
}

// 两者效果相同，但虚拟线程版本代码更直观
// 不需要理解 CompletableFuture 的链式 API
// 堆栈信息更完整，调试更方便
```

## 正确使用方式

### 什么时候用虚拟线程

```
✅ 适合的场景：
- I/O 密集型任务（HTTP 请求、数据库查询、文件读写、RPC 调用）
- 高并发 Web 服务（每个请求一个虚拟线程）
- 需要大量并发等待的场景（爬虫、消息消费、定时任务）

❌ 不适合的场景：
- CPU 密集型任务（大量计算、图像处理）
  → 虚拟线程不能让 CPU 变快，100 万个虚拟线程争抢 8 个 CPU 核心 = 大量上下文切换
  → CPU 密集任务用 ForkJoinPool 或普通线程池
```

### Pinning 问题（载体线程被钉住）

```
虚拟线程最大的坑：Pinning（钉住）

当虚拟线程在以下两种情况下阻塞时，无法从载体线程卸载：

1. 在 synchronized 块/方法内阻塞
   synchronized (lock) {
       socket.read();  // ❌ 载体线程被钉住！
   }

2. 在 native 方法内阻塞（JDK 内部的 native 调用）
   某些 JDK 内部操作通过 native 方法实现，无法卸载

为什么？
→ synchronized 的锁机制和 JVM 的载体线程绑定逻辑有耦合
→ JDK 团队正在修复（JDK 24+ 开始支持 synchronized 卸载）

如何避免？
→ 把 synchronized 换成 ReentrantLock
   ReentrantLock lock = new ReentrantLock();
   lock.lock();
   try {
       socket.read();  // ✅ 虚拟线程正常卸载
   } finally {
       lock.unlock();
   }
```

### ThreadLocal 的问题

```
ThreadLocal 和虚拟线程不兼容：

问题：
- 平台线程数量少（几十~几百），ThreadLocal 内存可控
- 虚拟线程数量大（百万级），每个虚拟线程都有一份 ThreadLocal 副本
  → 100 万虚拟线程 × 每个 ThreadLocal 几 KB = 内存爆炸

解决方案：
- 尽量不用 ThreadLocal
- 如果必须传递上下文，用以下方式：
  1. 方法参数显式传递（最推荐）
  2. Scoped Values（JDK 21+ 预览特性，Java 的"上下文变量"）
     ScopedValue.where(CURRENT_USER, user)
               .run(() -> doWork());  // 在 run 内可以访问 CURRENT_USER
     // run 结束后自动清除，不会泄漏
```

### 最佳实践

```
1. 用 Executors.newVirtualThreadPerTaskExecutor() 替代固定大小线程池
   → 不要为虚拟线程设置线程池大小
   → 每个任务一个虚拟线程

2. 把 synchronized 换成 ReentrantLock
   → 避免 Pinning 问题

3. 避免使用 ThreadLocal
   → 用 ScopedValue 或方法参数传递

4. 不要用虚拟线程做 CPU 密集计算
   → CPU 密集任务用 ForkJoinPool

5. 拥抱"每个请求一个线程"的简单编程模型
   → 不再需要复杂的事件驱动或异步编排
   → 回到最简单的代码风格
```

## 边界情况和坑

### 坑 1：synchronized 内的阻塞操作

```java
// ❌ 危险：载体线程被钉住
public synchronized void handleRequest(Socket socket) {
    byte[] data = socket.getInputStream().readAllBytes();  // 钉住！
    process(data);
}

// ✅ 安全：换成 ReentrantLock
private final ReentrantLock lock = new ReentrantLock();
public void handleRequest(Socket socket) {
    lock.lock();
    try {
        byte[] data = socket.getInputStream().readAllBytes();  // 正常卸载
        process(data);
    } finally {
        lock.unlock();
    }
}
```

### 坑 2：虚拟线程的生命周期管理

```java
// ❌ 危险：创建虚拟线程但不等待
for (int i = 0; i < 1000; i++) {
    Thread.startVirtualThread(() -> {
        heavyIO();
    });
}
// main 方法结束 → JVM 退出 → 虚拟线程可能还没执行完

// ✅ 安全：用 ExecutorService + shutdown
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    List<Future<?>> futures = new ArrayList<>();
    for (int i = 0; i < 1000; i++) {
        futures.add(executor.submit(() -> heavyIO()));
    }
    for (Future<?> f : futures) {
        f.get();  // 等待所有任务完成
    }
}
```

### 坑 3：Semaphore 和虚拟线程

```java
// ✅ 虚拟线程 + Semaphore = 限流
// 虽然可以创建百万虚拟线程，但不代表可以百万并发请求
// 数据库连接、API 限流等仍然需要 Semaphore 控制并发数

Semaphore dbPool = new Semaphore(20);  // 数据库最多 20 并发

try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 10_000; i++) {
        executor.submit(() -> {
            dbPool.acquire();
            try {
                queryDatabase();
            } finally {
                dbPool.release();
            }
        });
    }
}
```

### 坑 4：锁竞争下的虚拟线程

```
虚拟线程解决的是 I/O 阻塞问题，不是锁竞争问题

如果多个虚拟线程争抢同一把锁：
→ 没拿到锁的虚拟线程会被挂起（park），不阻塞载体线程（✅）
→ 但大量虚拟线程争一把锁 = 大量 park/unpark = 上下文切换开销
→ 本质问题还是"锁粒度太大"或"共享资源竞争太激烈"

解决思路：
→ 优化锁粒度（和平台线程时代的建议一样）
→ 用无锁数据结构（ConcurrentHashMap、原子类）
→ 用乐观锁代替悲观锁
```
