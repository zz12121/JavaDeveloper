# Java 日志框架核心原理

> 日志是线上问题排查的第一手资料。Java 日志生态以 SLF4J 门面 + Logback/Log4j2 实现为核心。理解门面模式、MDC 链路追踪、异步日志机制，是正确配置日志和排查日志问题的关键。

---

## 这个问题为什么存在？

Java 应用运行中产生的日志需要满足多个需求：**分级输出、按环境调整、输出到不同目的地、不拖慢业务性能**。但 Java 历史上出现了多套日志框架，带来了三个核心问题：

1. **日志框架不统一**——项目依赖 A 库用 Log4j，B 库用 JUL，C 库用 Commons Logging，日志输出混乱，无法统一配置
2. **切换成本高**——代码直接依赖具体实现，换日志框架要改大量业务代码
3. **性能问题**——日志输出涉及磁盘 I/O 和字符串格式化，高并发下可能成为瓶颈

SLF4J + Logback/Log4j2 的组合就是为了同时解决这三个问题：**门面解耦 + 可插拔实现 + 高性能异步输出**。

---

## 它是怎么解决问题的？

### SLF4J 门面模式

SLF4J 是日志的**接口层**，不负责实际输出。所有代码面向 SLF4J 编程，底层实现可替换：

```java
// 获取 Logger（推荐用类名）
private static final Logger log = LoggerFactory.getLogger(MyClass.class);

// 或 Lombok 简写
@Slf4j
public class MyClass { }
```

切换实现（Logback → Log4j2）只需换 Maven 依赖，不改一行业务代码。第三方库的日志通过 SLF4J Bridge 统一适配到同一个实现。

### 日志级别体系

```
ERROR（错误） > WARN（警告） > INFO（信息） > DEBUG（调试） > TRACE（追踪）
```

只输出 >= 当前设置级别的日志。设置为 INFO → 只输出 ERROR、WARN、INFO。参数化日志（`{}` 占位符）只在日志确实输出时才执行格式化，避免了 `log.debug("user: " + user)` 这种无条件字符串拼接的性能开销。

### Logback 核心架构

```
Logger（记录器）→ Level（级别）→ Appender（输出目的地）→ Filter（过滤器）→ Layout（格式化）
```

| 组件 | 职责 | 配置示例 |
|------|------|----------|
| **Logger** | 按包名层级组织，子 Logger 继承父级 | `<logger name="com.example" level="DEBUG">` |
| **Appender** | 输出目的地 | Console / RollingFile / Async |
| **Layout** | 格式化输出 | PatternLayout |
| **Filter** | 过滤日志 | LevelFilter / ThresholdFilter |

Logger 继承关系：root（根，默认 DEBUG）→ com → com.example → com.example.service。`additivity="false"` 让日志不向父 Logger 传播，避免重复输出。

### 滚动文件策略

```xml
<appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
    <file>logs/app.log</file>
    <rollingPolicy class="ch.qos.logback.core.rolling.SizeAndTimeBasedRollingPolicy">
        <fileNamePattern>logs/app.%d{yyyy-MM-dd}.%i.log.gz</fileNamePattern>
        <maxFileSize>100MB</maxFileSize>
        <maxHistory>30</maxHistory>
        <totalSizeCap>10GB</totalSizeCap>
    </rollingPolicy>
</appender>
```

### MDC 链路追踪

MDC（Mapped Diagnostic Context）是 SLF4J 提供的**线程级上下文 Map**。请求进入时 `MDC.put("traceId", uuid)`，Pattern 中 `%X{traceId}` 引用，同一请求的所有日志都带相同 traceId，可以 grep 快速串联完整日志链路。

异步线程中 MDC 丢失问题通过 `TaskDecorator` 解决——在任务提交前拷贝 MDC 上下文，执行前恢复。

### SpringBoot 日志配置

```yaml
spring:
  logging:
    level:
      root: INFO
      com.example: DEBUG
      org.springframework: WARN
    file:
      name: logs/app.log
```

切换 Log4j2：排除 `spring-boot-starter-logging`，引入 `spring-boot-starter-log4j2`。`logback-spring.xml` 支持 `<springProfile>` 多环境配置。

## 深入原理

### 异步日志：Logback AsyncAppender vs Log4j2 Disruptor

```
性能对比（参考值）：
  同步日志：~1万条/秒
  Logback AsyncAppender：~10万条/秒
  Log4j2 AsyncLogger：~1800万条/秒
```

**Logback AsyncAppender**：内部维护一个 BlockingQueue，业务线程把日志事件放入队列，由异步线程取出写入。队列满时根据 `discardingThreshold` 决定是否丢弃。

**Log4j2 AsyncLogger**：基于 LMAX Disruptor（无锁环形队列 + CAS），消除线程竞争，性能远超 Logback。配置 `<AsyncRoot>` 即可全局异步。

**异步日志的代价**：
- 队列满时可能丢日志（配置 `neverBlock=true` 可改为不阻塞但丢日志）
- 异步线程的异常与原始请求上下文脱节
- JVM 崩溃时队列中未输出日志丢失（可接受）
- `includeLocation=true` 获取调用行号有性能开销，生产环境建议关闭

### Logback vs Log4j2 对比

| 维度 | Logback | Log4j2 |
|------|---------|--------|
| **性能** | 好（AsyncAppender） | 更好（Disruptor AsyncLogger） |
| **SpringBoot 默认** | 是 | 需排除 Logback |
| **JSON 格式** | 需额外依赖 | 原生支持 |
| **Lambda 延迟求值** | 不支持 | 支持 |
| **GC 压力** | 格式化产生临时 String | 可配置 StringBuilder 复用 |

SpringBoot 默认 Logback，满足 90% 场景。极高并发、需要 JSON 原生格式、Lambda 延迟求值时选 Log4j2。

## 正确使用方式

### 生产环境日志配置要点

```
① 日志级别设 INFO，业务包可 DEBUG 但必须有开关
② 文件滚动策略（按天+按大小，保留 30 天，总大小限制 10GB）
③ 使用异步日志减少对业务线程的阻塞
④ 日志格式结构化（JSON），方便 [[07_中间件/03_Elasticsearch|ELK]] 采集解析
⑤ 敏感信息脱敏（手机号/身份证/密码）
⑥ 开启 MDC traceId 实现全链路追踪
⑦ 监控 ERROR 日志告警（接入钉钉/邮件）
⑧ 避免循环日志（Object.toString 导致无限递归）
```

### 日志脱敏

```java
// 自定义 Logback Converter
public class SensitiveDataConverter extends ClassicConverter {
    @Override
    public String convert(ILoggingEvent event) {
        String msg = event.getFormattedMessage();
        msg = msg.replaceAll("(1\\d{2})\\d{4}(\\d{4})", "$1****$2");      // 手机号
        msg = msg.replaceAll("(\\d{6})\\d{8}(\\d{4})", "$1********$2");   // 身份证
        msg = msg.replaceAll("(\\w{3})\\w+@(\\w+)", "$1***@$2");          // 邮箱
        return msg;
    }
}
```

### 参数化日志 vs 字符串拼接

```java
// ✅ 参数化：只在日志输出时格式化
log.debug("User {} login from IP {}", username, ip);

// ❌ 拼接：无论日志是否输出都执行 toString()
log.debug("User " + username + " login from IP " + ip);
```

## 边界情况和坑

### SLF4J 绑定冲突

```
现象：启动时报警告
  SLF4J: Class path contains multiple SLF4J bindings.
  SLF4J: Found binding in [...logback-classic-1.2.11.jar]
  SLF4J: Found binding in [...log4j-slf4j-impl-2.17.2.jar]

成因：classpath 中同时存在 Logback 和 Log4j2 的 SLF4J 实现
解决：mvn dependency:tree | grep log → 排除多余的日志依赖
```

### 异步线程 MDC 丢失

```
现象：@Async 方法或线程池中的日志没有 traceId

成因：MDC 基于 ThreadLocal，异步线程拿不到主线程的 MDC
解决：使用 TaskDecorator 在任务提交前拷贝、执行前恢复 MDC 上下文
```

### 日志打不出来排查步骤

```
1. 确认日志级别：Logger 的级别是否 >= 配置的级别？
2. 确认 additivity：是否被父 Logger 的配置覆盖？
3. 确认 Appender：有没有配置 appender-ref？
4. 确认文件权限：日志目录是否有写入权限？
5. 确认依赖冲突：是否只有一个 SLF4J 实现？
```

### 循环日志导致 StackOverflow

```
现象：应用启动后 CPU 100%，日志文件暴增

成因：实体类的 toString() 方法互相引用
  Order.toString() → User.toString() → Order.toString() → ...
  日志框架调用 toString() 格式化时触发无限递归

解决：排除循环引用字段，或使用 @ToString.Exclude（Lombok）
```
