# 灰度发布策略

## 这个问题为什么存在？

> 新版本直接全量上线，一旦有问题，**全部用户受影响**。问题是：**怎么让新版本先给小部分用户用，没问题再全量？**

没有灰度发布，你会遇到：
- 新版本有bug，所有用户都无法下单
- 回滚要重新部署，耗时长
- 无法做A/B测试（新UI用户喜不喜欢？）

---

## 它是怎么解决问题的？

### 四大发布策略

#### 1. 蓝绿部署（Blue-Green Deployment）

```
蓝环境（v1.0，线上）
绿环境（v1.1，待上线）

步骤：
  1. 绿环境部署v1.1，但不接流量
  2. 测试绿环境（内部测试）
  3. 切换流量（Nginx/网关改upstream），所有流量到绿环境
  4. 如果v1.1有问题，切回蓝环境（秒级回滚）
```

```yaml
# Nginx：蓝绿切换（改upstream，reload）
upstream backend {
    server green-server:8080;  # 切到绿环境
    # server blue-server:8080;  # 蓝环境（回滚时用）
}
```

**优点**：秒级切换、秒级回滚
**缺点**：需要双倍资源（蓝、绿各一套）

#### 2. 金丝雀发布（Canary Release）

```
95% 流量 → v1.0（稳定版）
5% 流量  → v1.1（金丝雀，试探）

观察5%的报错率、响应时间
  → 没问题，逐步扩大v1.1的流量（10% → 50% → 100%）
  → 有问题，一键回滚到v1.0
```

```yaml
# Spring Cloud Gateway：权重路由（金丝雀）
spring:
  cloud:
    gateway:
      routes:
        - id: order-v1    # 95%流量
          uri: lb://order-service-v1
          predicates:
            - Weight=group1, 95
        - id: order-v2    # 5%流量（金丝雀）
          uri: lb://order-service-v2
          predicates:
            - Weight=group1, 5
```

**优点**：逐步放量，风险可控
**缺点**：需要流量分割能力（网关支持）

#### 3. A/B 测试

```
用户A（新用户） → v1.1（新UI）
用户B（老用户） → v1.0（旧UI）

目标：对比转化率，决定用哪个版本
```

```java
// 网关层：根据用户信息分流（A/B测试）
@Component
public class ABTestFilter implements GlobalFilter {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String userId = exchange.getRequest().getHeaders().getFirst("X-User-Id");
        int hash = Math.abs(userId.hashCode()) % 100;

        if (hash < 50) {
            // 50%用户 → v1.0
            exchange.getAttributes().put("version", "v1.0");
        } else {
            // 50%用户 → v1.1
            exchange.getAttributes().put("version", "v1.1");
        }

        return chain.filter(exchange);
    }
}
```

#### 4. 滚动发布（Rolling Update）

```
3个实例（v1.0）
  → 先替换1个（v1.1）
  → 再替换1个（v1.1）
  → 最后替换1个（v1.1）
```

```yaml
# K8s Deployment：滚动发布
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # 最多多1个实例
      maxUnavailable: 1  # 最多少1个实例
  template:
    spec:
      containers:
        - name: order
          image: order-service:v1.1
```

**优点**：不需要双倍资源
**缺点**：新旧版本同时在线，可能不兼容

## 它和相似方案的本质区别是什么？

| | 蓝绿部署 | 金丝雀发布 | A/B测试 | 滚动发布 |
|---|---|---|---|---|
| 流量切换 | 全量切换 | 逐步放量 | 按用户分组 | 逐步替换实例 |
| 回滚速度 | 秒级（切流量） | 秒级（切流量） | 秒级（切流量） | 慢（重新部署） |
| 资源占用 | 双倍 | 单倍（加少量实例） | 单倍 | 单倍 |
| 适用场景 | 关键服务，要求秒级回滚 | 常规发布，风险可控 | 产品验证（UI、流程） | K8s环境，默认策略 |

**本质区别**：
- **蓝绿**：全量切换，资源占用高，但回滚最快
- **金丝雀**：逐步放量，资源占用低，风险可控
- **A/B测试**：不是为了稳定，而是为了**产品验证**
- **滚动发布**：K8s默认，资源占用最低，但回滚慢

## 正确使用方式

### 场景1：Spring Cloud Gateway + Nacos（动态配置）

```java
// 动态权重：从Nacos拉取权重配置
@RefreshScope
@Component
public class CanaryWeightConfig {

    @Value("${canary.weight:5}")  // 默认5%（Nacos动态刷新）
    private int canaryWeight;

    public boolean isCanary(String userId) {
        int hash = Math.abs(userId.hashCode()) % 100;
        return hash < canaryWeight;  // hash < 5 → 走金丝雀
    }
}
```

### 场景2：K8s Ingress（基于Header的金丝雀）

```yaml
# Istio VirtualService：按Header灰度
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: order-service
spec:
  hosts:
    - order.example.com
  http:
    - match:
        - headers:
            canary:
              exact: "true"   # Header含canary=true，走v1.1
      route:
        - destination:
            host: order-service-v1.1
    - route:
        - destination:
            host: order-service-v1.0  # 默认走v1.0
```

## 边界情况和坑

### 坑1：灰度流量失控（权重配置错误）

```
场景：金丝雀权重设成100%（本来想设5%）
后果：全部流量到新版本，新版本有问题 → 全部用户受影响
```

**解决方案**：
1. **灰度权重 gradual increase**（5% → 10% → 50% → 100%）
2. **灰度前，小流量压测**（验证新版本性能）
3. **告警**：新版本报错率超过阈值，自动回滚

### 坑2：数据库不兼容（新旧版本Schema不同）

```
场景：v1.1改了表结构（加了字段）
      v1.0还在跑，读老字段 → 报错
```

**解决方案**：
1. **灰度前，保证向前兼容**（v1.1先加字段，v1.0还能跑）
2. **灰度顺序**：先升级DB Schema，再灰度应用
3. **灰度期间，禁止DB回滚**（新Schema可能已写入数据）

### 坑3：回滚困难（状态已写入新版本）

```
场景：v1.1已写入新格式的数据
      回滚到v1.0，v1.0读不懂新格式 → 报错
```

**解决方案**：
1. **灰度期间，双写**（新旧格式都写，保证回滚后还能读旧格式）
2. **灰度前，做好数据迁移方案**
3. **禁止回滚**（如果数据已写入新格式，只能往前修，不能回滚）
