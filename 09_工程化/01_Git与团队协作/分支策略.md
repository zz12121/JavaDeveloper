# 分支策略

> 分支策略不是「规定怎么建分支」，而是**规定代码如何在分支间流动**。好的分支策略让发布可预测，坏的分支策略让合并地狱成为日常。

---

## 这个问题为什么存在？

没有分支策略时，团队会遇到这些典型问题：

```
场景 A：所有人往 main 提交
→ main 随时可能挂
→ 一个同事的半成品代码导致全团队无法工作
→ 发布时不知道「哪些功能完成了」「哪些还没稳定」

场景 B：长期分支（开发了 2 个月的分支）
→ 合并时冲突几百个文件
→ 解决冲突后发现功能坏了，不知道哪里改错了

场景 C：没有发布分支
→ 发布到生产环境的代码和开发代码混在一起
→ 生产紧急修复不知道往哪个分支改
→ 修复完忘记合并回开发分支 → 下次发布又把修复覆盖掉
```

**分支策略的核心目标**：
1. **隔离**：开发中的代码不影哨发布
2. **可追踪**：每个发布版本都能对应到代码
3. **可回退**：出问题时能快速回滚到上一个稳定版本
4. **并行**：多个人/多个功能可以同时开发

## 它是怎么解决问题的？

### Git Flow（经典但重）

```
Git Flow 分支模型：
          release/v1.2
               │
    develop─────┬───────────────────────→
                 │
    feature/A ──┘
    feature/B ──┐
                 │
    hotfix ──────┘
                 │
    main ────────┴──────────────────→（永远代表生产状态）
```

**五种分支类型**：

| 分支类型 | 从哪来 | 合到哪去 | 生命周期 | 说明 |
|---------|--------|----------|---------|------|
| `main` | - | - | 永久 | 生产环境对应 |
| `develop` | `main` | - | 永久 | 集成分支 |
| `feature/*` | `develop` | `develop` | 临时 | 功能开发 |
| `release/*` | `develop` | `main` + `develop` | 临时 | 发布准备 |
| `hotfix/*` | `main` | `main` + `develop` | 临时 | 生产紧急修复 |

**发布流程**：
```bash
# 1. 从 develop 拉 release 分支
git checkout -b release/1.2.0 develop

# 2. 在 release 上做最后修复（不改大功能）
# 3. 合并到 main 和 develop
git checkout main
git merge --no-ff release/1.2.0
git tag v1.2.0

git checkout develop
git merge --no-ff release/1.2.0

# 4. 删除 release 分支
git branch -d release/1.2.0
```

**`--no-ff` 的意义**：强制生成一个合并提交，保留分支历史。没有它，Git 会「快进合并」，丢失分支信息。

### GitHub Flow（轻量，适合持续交付）

```
GitHub Flow（只有两种分支）：
    main ────────────→（永远可发布）
       │
    feature/A ──────┘（通过 PR 合并回 main）
```

**规则极简**：
1. `main` 分支永远是可发布状态
2. 新功能从 `main` 拉分支，命名随意
3. 功能分支通过 **Pull Request** 合并回 `main`
4. 合并后立即部署到生产环境

**适合场景**：SaaS 产品、Web 服务、持续交付团队。

### GitLab Flow（上游优先）

```
GitLab Flow 的「上游优先」原则：
    main → pre-production → production
       （提交只允许向下游流动）

修复 bug：
    先在 production 修，再 cherry-pick 到 pre-production，再到 main
    或者：先在 main 修，再向下游 cherry-pick（上游优先）
```

**两种变种**：
- **环境分支**：`main` → `staging` → `production`（每个环境一个分支）
- **发布分支**：`main` 持续开发，定期从 `main` 拉 `release/x.x`（适合有版本号的产品）

### Trunk-Based Development（主干开发）

```
所有开发者每天都往 main（trunk）提交：
    main ──●──●──●──●──●──→（每个人每天至少提交一次）

短期分支（可选）：
    main ──●──────●─────→
              │      │
              └─●   └─●  （分支存活 < 1 天，通过 CI 后立即合并）
```

**核心原则**：
- 所有人每天至少向 `main` 提交一次
- 大功能用 **Feature Flag**（功能开关）而非长期分支
- 依赖强大的 CI/CD —— 每次提交都跑测试，失败立即修复

**适合场景**：Google、Meta 这种工程成熟度极高的团队。

## 深入原理

### Git Flow vs GitHub Flow

| 维度 | Git Flow | GitHub Flow |
|------|----------|--------------|
| 分支数量 | 多（5 种） | 少（2 种） |
| 发布节奏 | 版本化发布（v1.0、v1.1） | 持续发布（每天多次） |
| 合并频率 | 低（功能完成后才合并） | 高（小步提交，快速合并） |
| 适用产品 | 有版本号的客户端软件（APP、桌面软件） | SaaS、Web 服务 |
| 复杂度 | 高，新人学习曲线陡 | 低，5 分钟讲完规则 |

**为什么 Git Flow 不适合 Web 开发？**

Web 服务没有「版本号」概念，随时可以发布。Git Flow 的 release 分支在 Web 场景下没有意义——你不需要「准备发布」这个过程，因为你有 CI/CD 自动部署。

### Git Flow vs Trunk-Based

| 维度 | Git Flow | Trunk-Based |
|------|----------|-------------|
| 分支隔离 | 强（功能分支隔离） | 弱（所有人提交到同一分支） |
| 合并冲突 | 多（长期分支） | 少（每天合并，冲突小） |
| Feature Flag | 不用 | 必须用（隐藏未完成功能） |
| CI/CD 要求 | 低 | 极高（每次提交都跑全量测试） |
| 适合团队 | 传统软件团队 | 高成熟度工程团队 |

**为什么 Trunk-Based 合并冲突更少？**

冲突的本质是「同一段代码被两个人改了」。分支存活时间越长，冲突概率越高。Trunk-Based 要求每天合并，冲突在最小范围内解决。

**但为什么大多数团队不用 Trunk-Based？**

因为需要完善的 Feature Flag 机制 + 极高的 CI/CD 覆盖率 + 全团队纪律。Git Flow / GitHub Flow 用「分支隔离」换来了更低的工程门槛。

## 正确使用方式

### 1. 选对策略（按产品形态选）

```
产品类型 → 推荐策略：
- 有版本号的桌面/移动端软件  → Git Flow
- SaaS / Web 服务           → GitHub Flow 或 GitLab Flow
- 高频发布的互联网产品       → Trunk-Based（需要工程成熟度）
- 开源项目                  → GitHub Flow（简单，贡献者友好）
```

### 2. 功能分支的命名规范

```bash
# 推荐格式
feature/user-auth          # 功能
fix/login-null-pointer    # 修复
hotfix/payment-timeout    # 生产紧急修复
refactor/api-layer        # 重构
docs/api-spec             # 文档
test/order-service        # 测试

# 团队统一即可，不需要过度复杂
```

### 3. 用 PR/MR 做代码审查（不要直接合并）

```bash
# 正确流程（GitHub Flow / GitLab Flow）
git checkout -b feature/user-auth main
# 开发...
git push origin feature/user-auth
# 在 GitHub/GitLab 上创建 Pull Request / Merge Request
# Code Review → CI 通过 → Merge
```

**PR 的 checklist**：
- [ ] 代码能编译
- [ ] 测试通过（或新增了测试）
- [ ] PR 描述清晰（解决了什么问题、改了哪些文件）
- [ ] 至少一个其他人 Review 并 Approve

### 4. 用 `.gitignore` + 钩子保证分支干净

```bash
# pre-commit 钩子示例（用 pre-commit 框架）
repos:
  - repo: https://github.com/pre-commit/pre-commit-hooks
    hooks:
      - id: trailing-whitespace    # 去除行尾空格
      - id: end-of-file-fixer     # 确保文件末尾有换行
      - id: check-yaml            # 检查 YAML 语法
```

## 边界情况和坑

### 1. 长期分支合并冲突地狱

```
惨痛教训：
 feature/login 分支开发了 3 个月
 main 也在持续迭代
 合并时：187 个文件冲突
 解决完冲突，功能挂了，不知道哪里改错了
```

**正确做法**：
- 功能分支每天 rebase 最新的 `main`（保持同步）
- 大功能拆小（一个功能不超过 1-2 周）
- 用 Feature Flag 代替长期分支（Trunk-Based 思路）

### 2. Git Flow 的 release 分支变成「半成品垃圾场」

```bash
# 错误做法
git checkout -b release/1.2.0 develop
# 在 release 上开发了 2 周新功能
# → release 失去了「只做修复」的意义
```

**release 分支只允许**：
- Bug 修复
- 文档更新
- 版本号变更
- **不允许新功能**

### 3. 生产修复忘记合并回开发分支

```
经典惨案：
1. 生产发现 bug，从 main 拉 hotfix 分支修复
2. 修复完，合并回 main，打 tag 发布
3. 忘记合并回 develop
4. 下次发布 develop 时，bug 又回来了
```

**防御措施**：
- 用 Git Flow 工具（`git flow init`）自动化合并流程
- CI 检查：hotfix 分支必须同时合并到 main 和 develop
- GitLab Flow 的「上游优先」原则从流程上避免这个问题

### 4. 删除远程分支后，本地还能看到「已删除分支」

```bash
git branch -a   # 仍然能看到 remotes/origin/old-branch

# 清理本地缓存的远程分支信息
git fetch --prune   # 或 git remote prune origin
```

### 5. `--no-ff` vs `--ff-only` 的选择困惑

```bash
# --no-ff：强制生成合并提交（保留分支历史）
git merge --no-ff feature/A

# --ff-only：只允许快进合并（分支历史线性，不生成合并提交）
git merge --ff-only feature/A
# 如果无法快进，直接拒绝合并

# 使用建议：
# - main/develop 合并功能分支 → --no-ff（保留功能开发历史）
# - feature 分支同步 main 最新代码（rebase）→ 不用 merge，用 rebase
```
