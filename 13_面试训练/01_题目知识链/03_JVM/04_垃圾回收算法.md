# JVM 有哪些垃圾回收算法？各有什么特点？

## 知识链提示

```
垃圾回收算法
  → [[03_JVM/06_垃圾回收算法/垃圾回收算法]]
    → 判断对象存活
      → 引用计数（Python 用，JVM 不用）：循环引用导致内存泄漏
      → 可达性分析（JVM 用）：从 GC Roots 出发，顺着引用链搜索
        → GC Roots 包括：虚拟机栈引用的对象 / 方法区静态属性引用的对象 / 方法区常量引用的对象 / 本地方法栈 JNI 引用的对象 / JVM 内部引用（Class对象等）/ 同步锁持有的对象 / ...
    → 三大基础算法
      → 标记-清除（Mark-Sweep）：先标记所有存活对象，再清除未标记的
        → 缺点：产生内存碎片，分配速度慢
      → 复制（Copying）：把存活对象复制到另一半空间，再整体清空
        → 优点：无碎片，分配简单（指针碰撞）
        → 缺点：浪费一半空间
        → 应用：新生代（Eden → Survivor → 老年代，比例 8:1:1）
      → 标记-整理（Mark-Compact）：标记后移动存活对象，向一端对齐
        → 优点：无碎片
        → 缺点：移动成本高（Stop The World）
        → 应用：老年代（对象存活率高，复制开销大）
```

---

## 核心追问

| 追问 | 考点 |
|------|------|
| 为什么新生代用复制算法而不是标记-整理？ | 新生代对象朝生夕死，复制成本低 |
| 新生代 Eden:Survivor 为什么是 8:1:1？ | 经验值，大部分对象活不过第一次 Minor GC |
| 对象从新生代到老年代有哪些路径？ | 晋升年龄 / 大对象直接进 / 动态年龄判断 |
| 什么是 Stop The World？它是怎么发生的？ | GC 时必须停止所有应用线程（JVM 安全点） |

---

## 参考答案要点

**GC Roots 可达性分析**：
```
对象A（GC Root） → 引用 → 对象B → 引用 → 对象C（可达，活）
对象X → 引用 → 对象Y（没有 GC Root 路径，不可达，回收）
```

**三算法对比**：

| 算法 | 优点 | 缺点 | 适用区域 |
|------|------|------|---------|
| 标记-清除 | 不移动对象 | 内存碎片 | 老年代（CMS） |
| 复制 | 无碎片，分配快 | 浪费50%空间 | 新生代（Eden） |
| 标记-整理 | 无碎片 | 移动成本高 | 老年代 |
