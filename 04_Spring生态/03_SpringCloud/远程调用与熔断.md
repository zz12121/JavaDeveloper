# 远程调用与熔断

## 这个问题为什么存在？

> 服务间调用如果手写 `RestTemplate`，每个调用方都要处理：服务发现、负载均衡、超时、重试、熔断... 重复劳动，且容易不一致。  
> 如果下游服务挂了，上游一直重试 → ** cascading failure（雪崩）**。

**核心问题**：怎么**优雅地**调用下游服务，并在下游出问题时**快速失败**（不拖垮自己）？

---

## 它是怎么解决问题的？

### OpenFeign（声明式 HTTP 客户端）

**问题**：手写 `RestTemplate` 调用，代码冗长，不直观。

```java
// ❌ 手写调用（冗长）
String url = "http://user-service/api/users/" + userId;
User user = restTemplate.getForObject(url, User.class);
```

**OpenFeign 的解法**：**接口 + 注解 → 自动生成实现**。

```java
// ✅ 声明式（优雅）
@FeignClient(name = "user-service", fallback = UserServiceFallback.class)
public interface UserServiceClient {

    @GetMapping("/api/users/{id}")
    User getUserById(@PathVariable("id") Long id);

    @PostMapping("/api/users")
    User createUser(@RequestBody UserDTO dto);
}
```

> 只需要写接口，OpenFeign 在启动时**自动生成实现类**（JDK 动态代理）。  
> 调用方直接注入 `UserServiceClient`，像调用本地方法一样。

---

### OpenFeign 的工作原理

```java
// FeignClientsRegistrar（注册 Feign Client）
public void registerFeignClients(AnnotationMetadata metadata,
                                BeanDefinitionRegistry registry) {
    // 1. 扫描所有 @FeignClient 注解的接口
    Set<BeanDefinition> candidateComponents =
        getScanner().findCandidateComponents(basePackages);

    // 2. 为每个接口生成代理对象（JDK 动态代理）
    for (BeanDefinition candidate : candidateComponents) {
        // 生成 FeignClientFactoryBean（负责创建代理）
        BeanDefinitionBuilder definition = BeanDefinitionBuilder
            .genericBeanDefinition(FeignClientFactoryBean.class);
        // 注册到 Spring 容器
        registry.registerBeanDefinition(beanName, definition.getBeanDefinition());
    }
}
```

**调用流程**：

```
UserServiceClient.getUserById(1L)
    ↓
Feign 动态代理拦截
    ↓
1. 从 Nacos 拉取 user-service 的实例列表
    ↓
2. 负载均衡（选一个实例）
    ↓
3. 构造 HTTP 请求（GET /api/users/1）
    ↓
4. 发送请求（带超时、重试配置）
    ↓
5. 解析响应（JSON → User 对象）
```

### 熔断（Resilience4J）

**问题**：下游服务挂了，上游一直超时 → 线程池打满 → **雪崩**。

**Resilience4J 的解法**：**快速失败**（下游不健康时，直接走降级逻辑）。

**熔断器的三种状态**：

```
CLOSED（关闭，正常转发）
    ↓ 错误率 > 50%（可配置）
OPEN（打开，直接走降级，不转发）
    ↓ 等待 10s（可配置）
HALF_OPEN（半开，放一个请求试试）
    ↓ 成功
CLOSED（恢复）
    ↓ 失败
OPEN（再次打开）
```

**配置示例**：

```yaml
resilience4j:
  circuitbreaker:
    instances:
      userService:
        sliding-window-size: 10       # 滑动窗口大小（统计最近 10 次调用）
        failure-rate-threshold: 50      # 失败率超过 50% → 熔断
        wait-duration-in-open-state: 10s  # OPEN 状态持续 10s
        permitted-number-of-calls-in-half-open-state: 3  # 半开时放 3 个请求
```

---

### 降级（Fallback）

**问题**：下游熔断了（OPEN 状态），调用方收到什么？—— 如果不配置降级，会抛异常。

**降级的解法**：**提供一个「兜底」实现**，熔断时自动调用。

```java
// 1. 编写降级类
@Component
public class UserServiceFallback implements UserServiceClient {

    @Override
    public User getUserById(Long id) {
        // 兜底逻辑：返回一个默认用户（或抛业务异常）
        return new User(id, "未知用户");
    }

    @Override
    public User createUser(UserDTO dto) {
        throw new BusinessException("用户服务不可用，请稍后重试");
    }
}

// 2. 在 @FeignClient 中指定 fallback
@FeignClient(name = "user-service", fallback = UserServiceFallback.class)
public interface UserServiceClient { ... }
```

### 限流（Rate Limiter）

**问题**：下游服务只能承受 100 QPS，上游疯狂调用 → 压垮下游。

**Resilience4J 限流**：

```yaml
resilience4j:
  ratelimiter:
    instances:
      userService:
        limit-for-period: 100    # 每个周期允许 100 个请求
        limit-refresh-period: 1s  # 刷新周期 1s
        timeout-duration: 500ms   # 等待令牌的超时时间
```

**使用**：

```java
@GetMapping("/orders")
@RateLimiter(name = "orderService", fallbackMethod = "getOrdersFallback")
public List<Order> getOrders() {
    return orderService.getOrders();
}

// 降级方法（签名和原方法一致，加一个 Throwable 参数）
public List<Order> getOrdersFallback(Throwable t) {
    return Collections.emptyList();  // 限流时返回空列表
}
```

## 它和相似方案的本质区别是什么？

### OpenFeign vs RestTemplate

| | OpenFeign | RestTemplate |
|---|------------|--------------|
| 编程模型 | 声明式（接口 + 注解） | 命令式（手写调用） |
| 可读性 | 高（像本地方法调用） | 低（URL 拼接冗长） |
| 集成熔断 | ✅（@FeignClient 指定 fallback） | ✅（需要手动加 @CircuitBreaker） |
| 适用场景 | 服务间调用（推荐） | 简单场景、第三方 HTTP 调用 |

---

### Resilience4J vs Hystrix

| | Resilience4J | Hystrix |
|---|---------------|----------|
| 维护状态 | ✅ 活跃 | ❌ Netflix 已停止维护 |
| 编程模型 | 注解驱动（@CircuitBreaker） | 注解驱动（@HystrixCommand） |
| 隔离策略 | 信号量（默认）、线程池 | 线程池（默认） |
| 性能 | 高（轻量级） | 中（线程池开销） |

> **Hystrix 已停止维护**，新项目一律用 **Resilience4J**。

## 正确使用方式

### 1. OpenFeign 的日志级别

```yaml
# 配置 OpenFeign 的日志级别（DEBUG 级别才输出）
logging:
  level:
    com.example.client: DEBUG

# 配置 Feign 的日志级别（NONE / BASIC / HEADERS / FULL）
feign:
  client:
    config:
      default:
        logger-level: FULL  # 打印完整请求和响应
```

**日志级别说明**：

| 级别 | 输出内容 |
|------|----------|
| `NONE` | 不输出（默认） |
| `BASIC` | 请求方法 + URL + 状态码 + 耗时 |
| `HEADERS` | BASIC + 请求头 + 响应头 |
| `FULL` | HEADERS + 请求体 + 响应体 |

---

### 2. 请求/响应拦截器（统一处理 Token）

```java
// 1. 实现 RequestInterceptor
@Component
public class TokenInterceptor implements RequestInterceptor {

    @Override
    public void apply(RequestTemplate template) {
        // 从请求上下文取 Token，添加到 Feign 请求头
        String token = RequestContext.getCurrentContext().getToken();
        template.header("Authorization", "Bearer " + token);
    }
}
```

> **所有 Feign Client 的请求都会经过这个拦截器**，适合统一加 Token、TraceId 等。

---

### 3. 超时配置

```yaml
feign:
  client:
    config:
      default:
        connect-timeout: 2000   # 连接超时 2s
        read-timeout: 5000       # 读取超时 5s
      user-service:              # 针对某个服务单独配置
        connect-timeout: 1000
        read-timeout: 3000
```

> 超时配置**非常重要**，否则下游服务慢时，调用方线程会一直阻塞。

## 边界情况和坑

### 1. Feign 第一次调用失败（熔断误判）

```
服务刚启动 → 第一次调用（慢，可能超时）→ 熔断误判（认为服务不健康）
```

**解决**：配置 `warmup` 期，或者调大超时时间。

```yaml
resilience4j:
  circuitbreaker:
    instances:
      userService:
        minimum-number-of-calls: 10  # 最少 10 次调用后才开始计算失败率
```

---

### 2. Feign 的 GET 请求传复杂对象

```java
// ❌ 错误：GET 请求用 @RequestBody
@FeignClient("user-service")
public interface UserServiceClient {
    @GetMapping("/users")
    List<User> getUsers(@RequestBody UserQuery query);  // GET 不能有 body
}

// ✅ 正确：用 @SpringQueryMap（自动把对象属性转成 query string）
@GetMapping("/users")
List<User> getUsers(@SpringQueryMap UserQuery query);
```

---

### 3. 熔断器的 fallback 方法签名

```java
// ❌ 错误：fallback 方法签名不一致
@GetMapping("/users/{id}")
@CircuitBreaker(name = "userService", fallbackMethod = "getUserFallback")
public User getUser(@PathVariable Long id) { ... }

// fallback 方法签名要和原方法一致，最后加一个 Throwable 参数
public User getUserFallback(Long id, Throwable t) { ... }  // ✅ 正确
```

---

### 4. OpenFeign 的继承特性（不推荐）

```java
// ❌ 不推荐：Feign 支持继承（服务端和客户端用同一个接口）
public interface UserService {
    @GetMapping("/users/{id}")
    User getUser(@PathVariable Long id);
}

// 服务端实现
@RestController
public class UserController implements UserService { ... }

// 客户端 Feign
@FeignClient("user-service")
public interface UserServiceClient extends UserService { ... }
```

> **官方不推荐**这种做法：服务端和客户端耦合太强（接口变更要同时改）。
> 推荐：**服务端和客户端各自维护接口**。

## 我的理解

**远程调用的本质**：

1. **OpenFeign**：声明式 HTTP 客户端，接口 + 注解 → 自动生成代理
2. **Resilience4J**：熔断（快速失败）、限流（保护下游）
3. **降级**：提供一个兜底实现，熔断时自动调用

**面试高频追问**：

1. OpenFeign 的工作原理（扫描 @FeignClient → 生成代理 → 负载均衡 → 发送请求）
2. 熔断器的三种状态（CLOSED / OPEN / HALF_OPEN）
3. Resilience4J vs Hystrix 的区别（维护状态、性能、隔离策略）
4. Feign 的日志级别配置（NONE / BASIC / HEADERS / FULL）
5. 为什么 GET 请求不能用 @RequestBody？
