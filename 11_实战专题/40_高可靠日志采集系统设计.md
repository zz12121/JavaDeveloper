# 高可靠日志采集系统设计

> 日志采集核心：Agent采集 + 批量发送 + ACK机制 + 死信队列。核心矛盾是**高吞吐采集 vs 可靠传输，要保证不丢日志、不重复、且顺序正确**。

---

## 日志采集全链路架构

```
应用服务器 × N
   │
   ├─→ Filebeat Agent ──→  Kafka  ──→ Logstash ──→ Elasticsearch
   │     (轻量级)       (缓冲)     (解析/过滤)     (存储/检索)
   │
   ├─→ Logback → Kafka Appender → Kafka ──→ ...
   │
   └─→ 应用日志文件 → Filebeat → ...
   
   
额外组件：
   Kafka → S3/OSS (归档)
   Elasticsearch → Kibana (可视化)
   Elasticsearch → 告警系统 (监控)
```

## 场景 A：Filebeat/Logstash采集架构

### 现象

```
应用分散在数百台服务器，日志分散
需要统一采集到日志中心
Filebeat 太轻量，功能有限
Logstash 功能强，但占用资源多
```

### 解决方案

```yaml
# 1. Filebeat 配置（推荐作为日志采集 Agent）
# filebeat.yml
filebeat.inputs:
  - type: log
    enabled: true
    paths:
      - /var/log/app/*.log
      - /var/log/app/error/*.log
    fields:
      app: "order-service"
      env: "prod"
    fields_under_root: false
    # 多行日志合并（如 Java 异常栈）
    multiline.pattern: '^[0-9]{4}-[0-9]{2}-[0-9]{2}'
    multiline.negate: true
    multiline.match: after

# 输出到 Kafka（高吞吐缓冲）
output.kafka:
  hosts: ["kafka1:9092", "kafka2:9092", "kafka3:9092"]
  topic: "logs-%{[fields.app]}"  # 按应用分 topic
  partition.round_robin:
    reachable_only: false
  required_acks: 1  # 1=leader确认, -1=all副本确认
  compression: snappy  # 压缩
  max_message_bytes: 1000000  # 单条消息最大 1MB

# 状态持久化（记住读到哪了，重启不丢）
filebeat.registry.path: /var/lib/filebeat/registry
filebeat.registry.cleanup: true

# 性能调优
queue.mem:
  events: 4096        # 内存队列大小
  flush.min_events: 512  # 批量发送阈值
  flush.timeout: 1s      # 批量发送超时
```

```java
// 2. 应用直接写 Kafka（不走文件，更低延迟）
// Logback Kafka Appender 配置
<configuration>
    <appender name="KAFKA" class="com.github.danielwegener.logback.kafka.KafkaAppender">
        <encoder>
            <pattern>%d{yyyy-MM-dd HH:mm:ss} [%thread] %-5level %logger{36} - %msg%n</pattern>
        </encoder>
        <topic>logs-app-name</topic>
        <keyingStrategy>
            <type>thread</type>  <!-- 同一线程日志发到同一分区，保证顺序 -->
        </keyingStrategy>
        <deliveryStrategy>
            <type>async</type>  <!-- 异步发送，不阻塞业务 -->
        </deliveryStrategy>
        <producerConfig>bootstrap.servers=kafka1:9092,kafka2:9092</producerConfig>
        <producerConfig>acks=1</producerConfig>
        <producerConfig>compression.type=snappy</producerConfig>
        <producerConfig>retries=3</producerConfig>
    </appender>
    
    <root level="INFO">
        <appender-ref ref="KAFKA" />
        <appender-ref ref="STDOUT" />
    </root>
</configuration>
```

**采集架构选择**：
- **Filebeat**：轻量级，适合日志文件采集，资源占用少（10MB 内存）
- **Logstash**：功能强（过滤、解析、丰富），但重（几百MB 内存），适合服务端处理
- **Kafka Appender**：直接从应用发到 Kafka，最低延迟，但耦合应用

## 场景 B：批量压缩发送

### 现象

```
每条日志单独发送，网络开销大
未压缩的日志占用大量带宽
Kafka 写入频率过高，影响性能
需要批量 + 压缩，提高吞吐
```

### 解决方案

```java
// 1. 批量发送实现（Kafka Producer）
Properties props = new Properties();
props.put("bootstrap.servers", "kafka1:9092,kafka2:9092");
props.put("batch.size", 16384);       // 批量大小：16KB（凑满一批再发）
props.put("linger.ms", 10);           // 最多等待 10ms（即使没凑满也发）
props.put("compression.type", "snappy");  // 压缩算法：snappy（速度快）
props.put("acks", "1");               // 1=leader确认（平衡可靠性与性能）
props.put("retries", 3);              // 发送失败重试 3 次
props.put("max.in.flight.requests.per.connection", 1);  // 保证顺序

KafkaProducer<String, String> producer = new KafkaProducer<>(props);

// 发送日志（异步，批量累积）
public void sendLog(String topic, String logLine) {
    ProducerRecord<String, String> record = new ProducerRecord<>(topic, logLine);
    producer.send(record, (metadata, exception) -> {
        if (exception != null) {
            log.error("发送日志失败", exception);
            // 写入本地文件兜底
            writeToLocalFile(logLine);
        }
    });
}
```

```yaml
# 2. Filebeat 批量配置
queue.mem:
  events: 4096        # 内存队列：最多缓存 4096 条
  flush.min_events: 512  # 凑满 512 条就发（不用等超时）
  flush.timeout: 1s      # 或等待 1 秒就发

output.kafka:
  bulk_max_size: 2048    # 单次请求最多 2048 条
  compression: snappy    # snappy 压缩（速度快，压缩比适中）
  # 可选压缩：gzip（压缩比高但慢）、lz4（更快）、zstd（新算法）
```

```java
// 3. 压缩算法对比与选择
public class CompressionComparison {
    // snappy：速度快（~500MB/s），压缩比 2:1，适合大多数场景
    // gzip：速度慢（~50MB/s），压缩比 3:1，适合归档
    // lz4：速度快（~400MB/s），压缩比 2:1，适合实时
    // zstd：速度快（~300MB/s），压缩比 3:1，新选择
    
    // 日志场景推荐：snappy（实时）或 zstd（归档）
}
```

**批量发送优化**：
- **batch.size**：太小（频繁发送）、太大（延迟高），建议 16-64KB
- **linger.ms**：设置 5-10ms，平衡延迟和吞吐
- **压缩**：节省 50-70% 带宽，但增加 CPU 开销

## 场景 C：ACK与重试机制

### 现象

```
日志发送后，Kafka 没收到（网络抖动）
发送失败没有重试，日志丢失
ACK 级别设置不当：acks=0 最快但可能丢，acks=-1 最慢但可靠
需要合适的 ACK + 重试策略
```

### 解决方案

```java
// 1. Kafka Producer 可靠性配置
Properties props = new Properties();
props.put("bootstrap.servers", "kafka1:9092,kafka2:9092");
props.put("acks", "1");  // leader 确认（推荐：速度 + 可靠性平衡）
// acks=0: 不等待确认（最快，可能丢）
// acks=1: leader 确认（推荐）
// acks=-1 或 all: 所有副本确认（最可靠，但慢）

props.put("retries", 3);              // 重试次数
props.put("retry.backoff.ms", 100);   // 重试间隔

// 幂等性：防止因重试导致重复消息
props.put("enable.idempotence", true);  // 开启幂等（Kafka 0.11+）
// 幂等原理：每个 Producer 分配 PID，消息带序列号，Broker 去重

KafkaProducer<String, String> producer = new KafkaProducer<>(props);
```

```java
// 2. 发送回调：确认是否成功
public void sendWithCallback(String topic, String key, String value) {
    ProducerRecord<String, String> record = new ProducerRecord<>(topic, key, value);
    
    producer.send(record, new Callback() {
        @Override
        public void onCompletion(RecordMetadata metadata, Exception exception) {
            if (exception != null) {
                // 发送失败：记录失败日志，写入本地文件兜底
                log.error("发送失败: topic={}, key={}", topic, key, exception);
                fallbackToLocalFile(record);
                
                // 告警：连续失败超过阈值
                if (failureCount.incrementAndGet() > 100) {
                    alarmService.send("日志发送连续失败超过100条");
                }
            } else {
                // 发送成功
                failureCount.set(0);  // 重置失败计数
                log.debug("发送成功: offset={}", metadata.offset());
            }
        }
    });
}

// 3. 兜底：本地文件写入（发送失败时使用）
private void fallbackToLocalFile(ProducerRecord<String, String> record) {
    try {
        Files.write(
                Paths.get("/tmp/failed_logs.log"),
                (record.key() + ":" + record.value() + "\n").getBytes(),
                StandardOpenOption.CREATE,
                StandardOpenOption.APPEND
        );
    } catch (IOException e) {
        log.error("兜底写入也失败", e);
    }
}
```

```yaml
# 4. Filebeat ACK 配置
output.kafka:
  required_acks: 1        # leader 确认
  retry.max: 3             # 重试 3 次
  timeout: 30s             # 超时时间

# Filebeat 内部队列持久化（重启不丢）
filebeat.registry.path: /var/lib/filebeat/registry
# registry 记录每个文件读到哪个位置，重启后继续读
```

**ACK 级别选择**：
- **acks=0**：测速、日志允许丢失的场景
- **acks=1**：生产推荐（leader 确认，速度较快）
- **acks=-1**：金融、审计日志（所有副本确认，最可靠）

## 场景 D：消息顺序保证

### 现象

```
同一请求的日志被打乱顺序（如：开始→处理→结束）
Kafka 多分区导致顺序乱
排查问题时，日志链式追踪困难
需要保证关键日志的顺序
```

### 解决方案

```java
// 1. Kafka 顺序保证：同一 Key 发送到同一分区
// 原理：Kafka 用 key 的 hash 决定分区，相同 key 一定在同一分区
// 同一分区内，消息是有序的

public void sendOrderedLog(String traceId, String logLine) {
    // traceId 作为 key，保证同一请求的日志在同一分区
    ProducerRecord<String, String> record = 
            new ProducerRecord<>("logs-topic", traceId, logLine);
    
    producer.send(record);
}

// 2. Partitioner 自定义（可选）
// 默认分区器：key 存在时用 hash；key 不存在时轮询
// 自定义：保证某个字段（如 traceId）在同一分区
public class TraceIdPartitioner implements Partitioner {
    @Override
    public int partition(String topic, Object key, byte[] keyBytes, 
                         Object value, byte[] valueBytes, Cluster cluster) {
        // 用 traceId 计算分区
        List<PartitionInfo> partitions = cluster.partitionsForTopic(topic);
        int numPartitions = partitions.size();
        
        // 简单的 hash 取模（保证相同 traceId 到同一分区）
        return Math.abs(key.hashCode()) % numPartitions;
    }
}
```

```java
// 3. 日志链路追踪：在日志中打印 traceId
// Logback 配置：用 MDC 传递 traceId
public class TraceFilter implements Filter {
    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain) {
        try {
            // 从请求头获取 traceId，或生成新的
            String traceId = getTraceIdFromHeader(request);
            if (traceId == null) {
                traceId = UUID.randomUUID().toString();
            }
            
            // 放入 MDC（日志框架会自动打印）
            MDC.put("traceId", traceId);
            
            chain.doFilter(request, response);
        } finally {
            MDC.remove("traceId");
        }
    }
}

// Logback pattern：%X{traceId} 打印 MDC 中的 traceId
// <pattern>%d{yyyy-MM-dd HH:mm:ss} [%thread] %-5level %logger - traceId=%X{traceId} - %msg%n</pattern>
```

**顺序保证限制**：
- Kafka 只保证**分区内有序**，不保证全局有序
- 如果要求全局有序，只能用**单分区**（但性能差，不推荐）
- 实际方案：**按 traceId 分区**，保证同一请求日志有序（够用）

## 核心参数估算

```
日志量估算：
  假设：100 台服务器，每台 100MB/天，10 个应用
  总日志量 = 100 × 100MB × 10 = 100GB/天
  峰值：晚上 20:00-22:00，占全天 30% → 30GB/2h = 4MB/s
  
Kafka 吞吐需求：
  峰值 4MB/s × 3（副本） = 12MB/s 写入
  单 Kafka broker 可处理 50MB/s → 3 台够用
  
Filebeat 资源占用：
  内存：10-20MB
  CPU：1-5%
  单 Filebeat 可处理 100MB/s 日志（文件读取）
```

## 追问链

### 追问 1：Filebeat 和 Logstash 怎么选？

> "分工不同：**Filebeat**：部署在应用服务器，轻量级采集日志文件，只做简单过滤，发送到 Kafka。**Logstash**：部署在日志服务端，做复杂解析（如 JSON 解析、字段提取、数据丰富），性能要求高（需要 2-4GB 内存）。典型架构：Filebeat → Kafka → Logstash → Elasticsearch。"

### 追问 2：Kafka 消息丢失怎么防？

> "三层防护：1. **Producer 端**：acks=1 或 -1，开启重试（retries=3+）。2. **Broker 端**：replication.factor=3（3副本），min.insync.replicas=2（至少2副本确认）。3. **Consumer 端**：先处理日志，再提交 offset（手动 commit）。同时：Filebeat registry 持久化，重启不丢读取位置。"

### 追问 3：日志顺序乱了怎么查问题？

> "用 traceId 关联：1. 在请求入口生成 traceId，放入 MDC。2. 日志打印 traceId（Logback Pattern 配置）。3. 发送到 Kafka 时用 traceId 作为 key，保证同一请求日志在同一分区。4. 查询时用 traceId 搜索（Elasticsearch），所有相关日志按顺序展示。"

### 追问 4：日志量爆发，Kafka 扛不住怎么办？

> "多级防护：1. **限流**：Filebeat 配置 `harvester_limit`（限制读取速度）。2. **本地缓存**：Kafka 不可用时，Filebeat 会缓存到本地 registry，等恢复后继续发送。3. **扩容 Kafka**：增加 broker 节点，增加分区数。4. **降级**：非核心日志（如 DEBUG）临时关闭。"

## 排查 Checklist

```
□ Filebeat 正在运行吗？ → systemctl status filebeat
□ Filebeat registry 正常吗？ → 检查 /var/lib/filebeat/registry
□ Kafka 可连接吗？ → telnet kafka1 9092
□ Kafka topic 存在吗？ → kafka-topics.sh --list
□ 日志压缩开启了吗？ → compression.type=snappy
□ ACK 级别合适吗？ → acks=1（生产推荐）
□ 重试配置了了吗？ → retries > 0
□ 顺序有保障吗？ → 用 traceId 作为 Kafka key
□ 本地兜底开启了吗？ → 发送失败写入本地文件
□ 磁盘空间够吗？ → df -h（日志文件、registry、Kafka）
```
