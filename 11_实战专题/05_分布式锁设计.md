# 分布式锁设计

> 分布式锁的本质是：**在分布式系统中，让多个节点的同一时刻只有一个能执行某段逻辑**。选型取决于一致性要求、性能要求、依赖引入成本。

---

## 核心需求

```
分布式锁需要满足：
1. 互斥性（基本）
2. 防死锁（不能一直占着）
3. 可重入（同一线程可重复获取）
4. 锁续期（业务超时锁不过期）
5. 高可用（Redis/ZK 挂了不影响核心业务）
```

---

## 方案 A：Redis 分布式锁（最常用）

### 基础实现

```java
// 方案1：原生 Redis（setnx）
public boolean tryLock(String key, String value, long expireSeconds) {
    // setnx + expire 不是原子的！需要用 set nx ex 替代
    String result = jedis.set(key, value, "NX", "EX", expireSeconds);
    return "OK".equals(result);
}

public void unlock(String key, String value) {
    // Lua 脚本保证原子性：只能删自己加的锁
    String lua = 
        "if redis.call('get', KEYS[1]) == ARGV[1] then " +
        "    return redis.call('del', KEYS[1]) " +
        "else " +
        "    return 0 " +
        "end";
    jedis.eval(lua, Collections.singletonList(key), 
               Collections.singletonList(value));
}
```

**问题**：锁过期了但业务没执行完 → 锁失效。

### Redisson（推荐）

```java
@Configuration
public class RedissonConfig {
    @Bean
    public RedissonClient redisson() {
        Config config = new Config();
        config.useSingleServer()
              .setAddress("redis://127.0.0.1:6379")
              .setConnectionPoolSize(64);
        return Redisson.create(config);
    }
}

@Service
public class OrderService {
    @Autowired
    private RedissonClient redisson;
    
    public void createOrder(Long userId) {
        String lockKey = "order:create:" + userId;
        RLock lock = redisson.getLock(lockKey);
        
        try {
            // 尝试加锁，最多等 3 秒，锁持有 10 秒
            if (lock.tryLock(3, 10, TimeUnit.SECONDS)) {
                // 业务逻辑
                createOrderInternal(userId);
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
```

**Redisson 的优点**：
- ✅ 支持可重入
- ✅ 支持锁续期（Watchdog 机制）
- ✅ 支持公平锁/读写锁/信号量

## 方案 B：Zookeeper 分布式锁（强一致）

### Curator 实现

```java
@Configuration
public class ZkConfig {
    @Bean
    public CuratorFramework curatorFramework() {
        RetryPolicy retryPolicy = new ExponentialBackoffRetry(1000, 3);
        CuratorFramework client = CuratorFrameworkFactory.builder()
                .connectString("127.0.0.1:2181")
                .sessionTimeoutMs(5000)
                .retryPolicy(retryPolicy)
                .build();
        client.start();
        return client;
    }
}

@Service
public class OrderService {
    @Autowired
    private CuratorFramework client;
    
    public void createOrder(Long userId) throws Exception {
        String lockPath = "/locks/order/create/" + userId;
        InterProcessMutex lock = new InterProcessMutex(client, lockPath);
        
        try {
            if (lock.acquire(3, TimeUnit.SECONDS)) {
                createOrderInternal(userId);
            }
        } finally {
            if (lock.isAcquiredInThisProcess()) {
                lock.release();
            }
        }
    }
}
```

### ZK 锁的原理

```
/lock/
├── lock-0000000001  ← 第一个请求
├── lock-0000000002  ← 第二个请求，监听前一个
└── lock-0000000003  ← 第三个请求，监听前一个

第一个释放锁 → 第二个收到通知，获取锁
```

**优点**：强一致性，不会锁失效
**缺点**：性能低于 Redis，依赖 ZK 集群

## 方案 C：数据库分布式锁（最简单）

### 基于唯一索引

```sql
-- 尝试获取锁
INSERT INTO distributed_lock (lock_key, lock_value, expire_time)
VALUES ('order:create:123', 'uuid-xxx', NOW() + INTERVAL 30 SECOND);
-- 成功 → 获取到锁
-- 失败（唯一索引冲突）→ 获取锁失败

-- 释放锁
DELETE FROM distributed_lock 
WHERE lock_key = 'order:create:123' AND lock_value = 'uuid-xxx';
```

### 基于 for update（不推荐）

```java
@Transactional
public void createOrder(Long userId) {
    // for update 会锁住这行，直到事务提交
    Lock lock = lockMapper.selectForUpdateByKey("order:create:" + userId);
    if (lock == null) {
        lockMapper.insert(new Lock("order:create:" + userId));
    }
    try {
        createOrderInternal(userId);
    } finally {
        // 事务提交时自动释放锁
    }
}
```

**缺点**：
- 连接池耗尽风险
- 性能差（串行化）
- 锁超时依赖 DB 的 `innodb_lock_wait_timeout`

## 方案对比

| 维度 | Redis（Redisson） | Zookeeper（Curator） | 数据库 |
|------|-------------------|----------------------|--------|
| 一致性 | 最终一致 | 强一致（CP） | 强一致 |
| 性能 | 最高（~1ms） | 中等（~10ms） | 最低（~50ms） |
| 锁续期 | ✅ Watchdog | ✅ 临时节点 | ❌ 需自己实现 |
| 可重入 | ✅ | ✅ | ❌ |
| 依赖 | Redis 集群 | ZK 集群 | MySQL |
| 适用场景 | 高并发、允许偶尔重复 | 强一致要求 | 简单场景、低并发 |

## 失败场景分析

```
场景 1：锁过期但业务没执行完
解决：Redisson 的 Watchdog 自动续期

场景 2：主节点加锁成功，但主从切换
解决：RedLock 算法（不推荐，复杂且不安全）
      实际方案：业务幂等 + 最终一致

场景 3：网络分区，客户端以为锁丢了
解决：锁的唯一 value + Lua 脚本原子删除

场景 4：锁的持有者挂了
解决：锁过期时间 + 看门狗续期
```

---

## 设计原则

```
1. 锁粒度尽量小
   ✅ "order:create:" + userId
   ❌ "order:create:all"

2. 锁超时时间要合理
   太短 → 业务没完锁就丢了
   太长 → 服务挂了锁长期占用

3. 释放锁必须校验 value
   ✅ Lua 脚本原子删除
   ❌ 先 get 再 del

4. 业务必须幂等
   分布式锁不保证 100% 互斥，业务要兜底
```

## 追问链

### 追问 1：Redis 主从切换会不会导致锁丢失？

> "会。主节点加锁成功，但还没同步到从节点，主节点挂了，从节点升级为主节点，另一个请求可以在新主节点上加锁成功——两个客户端同时持有锁。这是 Redis 分布式锁的固有问题。解决方式：1. 业务幂等兜底（最实际）。2. RedLock 算法（复杂，且 Martin Kleppmann 证明不安全）。3. 用 ZK 或 etcd（强一致）。"

### 追问 2：分布式锁和本地锁怎么配合用？

> "先分布式锁（跨 JVM 互斥），再 synchronized（单 JVM 内互斥）。两层锁保护：

```java
public void createOrder(Long userId) {
    RLock distLock = redisson.getLock("order:" + userId);
    distLock.lock(10, TimeUnit.SECONDS);  // 分布式锁
    try {
        synchronized(this) {               // 本地锁
            createOrderInternal(userId);
        }
    } finally {
        distLock.unlock();
    }
}
```

实际中，如果业务是幂等的，只用分布式锁就够了。"

### 追问 3：锁的粒度怎么设计？

> "锁粒度 = 业务粒度。比如创建订单，锁 `order:create:{userId}` 而不是 `order:create:all`。粒度越小，并发度越高。但太细会导致锁数量爆炸，需要在并发度和复杂度之间找平衡。"

### 追问 4：秒杀场景用 Redis 锁还是数据库锁？

> "都不用最好。秒杀应该用：1. Redis 预减库存（`DECR` 原子操作）。2. 库存 > 0 才让进 DB。3. DB 层再校验库存（兜底）。分布式锁在这里会成为性能瓶颈。"

## 排查 Checklist

```
□ 锁能正常获取和释放吗？ → 单元测试 + 压测验证
□ 锁会过期吗？ → 业务执行时间 vs 锁超时时间
□ 释放锁是原子的吗？ → Lua 脚本 or Redisson
□ 业务是幂等的吗？ → 分布式锁不保证 100% 互斥
□ 锁粒度合适吗？ → 太粗影响并发，太细管理复杂
□ Redis 主从切换有预案吗？ → 业务幂等 or 用 ZK
□ 有锁的监控吗？ → 获取锁失败次数、锁持有时间
```
