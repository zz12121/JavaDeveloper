# 分布式 ID 生成方案有哪些？各自有什么优缺点？

> ⚠️ **先盲答**：单机数据库自增 ID 为什么不适合分布式系统？

---

## 盲答引导

1. 分布式 ID 的核心要求是什么？—— 唯一性 / 趋势递增 / 高可用 / 高性能
2. UUID 的问题是什么？—— 无序 / 存储空间大 / 不可读
3. 雪花算法（Snowflake）的核心结构是什么？—— 时间戳 + 机器ID + 序列号
4. 雪花算法的问题是什么？—— 时钟回拨

---

## 知识链提示

```
分布式 ID
  → [[分布式ID]]
    → UUID
      → 优点：本地生成，无网络开销，无单点
      → 缺点：无序（插入 B+ 树效率差）/ 36 字符存储大 / 不可读
    → 数据库自增 ID
      → 单机没问题，分布式：多机需要设置不同起始值 + 步长
      → 缺点：强依赖数据库，性能差，扩展难
    → Redis INCR
      → 利用 Redis 的原子操作 INCR
      → 缺点：强依赖 Redis，网络抖动会影响 ID 生成
    → 雪花算法（Snowflake）
      → 64 位结构
        → 1 位：固定 0（正数）
        → 41 位：时间戳（毫秒级，可用 69 年）
        → 10 位：机器 ID（数据中心 + 机器）
        → 12 位：序列号（每 ms 内自增，最多 4096）
      → 优点：趋势递增，高性能（本地生成）
      → 缺点：强依赖时钟，时钟回拨会重复 ID
        → 解决：发生回拨 → 等待回拨时间过去 / 用上一毫秒序列号延续
    → 百度 UidGenerator
      →改良雪花算法，支持每秒百万级生成
      →时间段 + 序列号，自旋 CAS 抢序列号
    → 美团 Leaf
      →号段模式：一次从数据库拿一批 ID 段，本地自增用完再拿
```

## 核心追问

| 追问 | 考点 |
|------|------|
| 雪花算法的机器 ID 是怎么分配的？ | 手动配置 / Zookeeper 协调分配 |
| 时钟回拨怎么解决？ | 等待法 / 延续上一毫秒序列号 |
| 号段模式和雪花算法的核心区别是什么？ | 雪花本地生成高性能，号段减少数据库依赖 |
| MyCat / ShardingSphere-JDBC 里，分布式 ID 怎么处理？ | 内置雪花算法 |

---

## 参考答案要点

**四种方案对比**：

| 方案 | 唯一性 | 趋势递增 | 性能 | 依赖 |
|------|--------|---------|------|------|
| UUID | ✅ | ❌ | 高 | 无 |
| 数据库自增 | ✅ | ✅ | 低 | 数据库 |
| Redis INCR | ✅ | ✅ | 高 | Redis |
| 雪花算法 | ✅ | ✅ | 最高 | 时钟 |

**实战建议**：大多数场景用雪花算法，特殊场景（强依赖时钟不可用）用号段模式。
