# 类加载机制

> .class 文件只是一串字节码，CPU 不能直接执行。类加载机制就是把它读进 JVM、变成可用的 Class 对象的过程。重点：双亲委派模型、热部署原理、Class.forName vs loadClass。

---

## 这个问题为什么存在？

如果没有类加载机制，JVM 启动时就要把所有 .class 文件全部读进内存——程序可能只用到了 5% 的类，却要为 100% 的类付出加载和内存开销。

**核心问题**：如何「按需加载」.class 文件，把它变成 JVM 里可以用的 Class 对象，同时保证核心类不被篡改？

---

## 它是怎么解决问题的？

### 1. 类加载的完整五阶段

```
加载（Loading）
  └── 通过类的全限定名，找到 .class 文件的二进制字节流
      ├── ① 从 classpath 读取
      ├── ② 从 JAR 包读取
      └── ③ 运行时动态生成（代理类）
      │
链接（Linking）
  ├── 验证（Verification）：魔数 CAFEBABE + 语义校验
  ├── 准备（Preparation）：为 static 变量分配内存，赋零值
  └── 解析（Resolution）：常量池符号引用 → 直接引用
      │
初始化（Initialization）
  └── 执行 <clinit>() 方法（编译器收集所有 static 赋值 + static{} 块）
```

**准备阶段赋零值，初始化阶段才赋真值**：

```java
public static int value = 123;
// 准备阶段：value = 0
// 初始化阶段：value = 123
```

### 2. 初始化时机（懒加载原则）

只有以下 6 种情况才会立即触发类的初始化（**主动引用**）：

```java
1. new、读取/设置 static 字段（非 final）、调用 static 方法
2. 反射调用（Class.forName("com.xxx")）
3. 初始化子类 → 父类必须先初始化
4. 主类（包含 main() 的类）
5. JDK 7+：MethodHandle 解析结果为 REF_getStatic 等
6. JDK 8：接口中 default 方法的实现类被初始化时，接口要先初始化
```

**为什么这样设计？** 大多数类在程序运行期间可能一直用不到，懒加载可以显著减少启动时间和内存占用。

### 3. 双亲委派模型

```
Custom ClassLoader（自定义）
        ↑ 先委派父加载器
AppClassLoader（应用程序类加载器）
        ↑
ExtClassLoader（扩展类加载器）
        ↑
Bootstrap ClassLoader（启动类加载器，加载 rt.jar）
```

收到类加载请求时，**先不自己尝试加载**，而是委派给父加载器。只有当父加载器反馈无法完成时，才由当前加载器自己尝试加载。

**为什么需要双亲委派？**

| 原因 | 说明 |
|------|------|
| 防止核心类被篡改 | `java.lang.Object` 永远由 Bootstrap 加载，用户自定义的 `java.lang.Object` 不会被加载 |
| 避免重复加载 | 父加载器加载过的类，子加载器不会再次加载 |

### 4. 源码关键路径：`ClassLoader.loadClass()`

```java
// ClassLoader.java（简化）
protected Class<?> loadClass(String name, boolean resolve) {
    synchronized (getClassLoadingLock(name)) {
        // ① 先检查是否已加载
        Class<?> c = findLoadedClass(name);
        if (c == null) {
            try {
                if (parent != null) {
                    c = parent.loadClass(name, false); // ② 委派父加载器
                } else {
                    c = findBootstrapClassOrNull(name);    // ③ Bootstrap
                }
            } catch (ClassNotFoundException e) {
                // 父加载器抛出异常，继续
            }
            if (c == null) {
                c = findClass(name);  // ④ 父加载器失败，自己加载
            }
        }
        if (resolve) resolveClass(c);
        return c;
    }
}
```

**自定义类加载器时，应该重写 `findClass()`，而不是 `loadClass()`**：

- 重写 `loadClass()` 会破坏双亲委派模型（除非你明确不想用双亲委派）
- 重写 `findClass()` 可以保留双亲委派逻辑，只在父加载器失败时调用自定义逻辑

## 深入原理

### 破坏双亲委派模型（面试高频）

双亲委派不是强制约束，有以下场景需要「破坏」它：

#### 1. Tomcat 的 WebAppClassLoader

Tomcat 需要隔离 Web 应用——每个 Web 应用用自己的类加载器，应用 A 的类不能被应用 B 看到。

Tomcat 的类加载顺序（**倒置**）：

```
WebAppClassLoader（优先自己加载）
        ↓ 加载失败
    AppClassLoader
        ↓
    ExtClassLoader
```

即：先尝试自己加载，不行再委派父加载器。这样每个 Web 应用加载的类彼此隔离。

#### 2. JDBC / SPI 的破坏（线程上下文类加载器）

JDBC 的接口在 `rt.jar`（由 Bootstrap 加载），但实现类在 `mysql-connector.jar`（由 AppClassLoader 加载）。

**问题**：Bootstrap 加载的类，无法访问 AppClassLoader 加载的类（类加载器的可见性规则）。

**解决**：用**线程上下文类加载器**：

```java
// 在 rt.jar 的 DriverManager 里
ClassLoader cl = Thread.currentThread().getContextClassLoader();
// 用这个 cl 去加载 mysql 的驱动（绕开双亲委派）
```

#### 3. OSGi 的网状类加载

OSGi 允许多个模块（Bundle）共享和隔离类，类加载器之间呈网状关系，完全打破了双亲委派的树状结构。

### `Class.forName()` vs `ClassLoader.loadClass()`

| | `Class.forName()` | `ClassLoader.loadClass()` |
|---|-------------------|------------------------|
| **初始化** | ✅ 执行 `<clinit>()`（初始化） | ❌ 只加载 + 链接，不初始化 |
| **使用场景** | 需要初始化（执行 static 块） | 只要加载（懒初始化） |

```java
// forName 会初始化（执行 static 块）
Class<?> c1 = Class.forName("com.xxx.MyClass");

// loadClass 只加载，不初始化
Class<?> c2 = getClass().getClassLoader().loadClass("com.xxx.MyClass");
```

### 热部署的实现原理

```
1. 自定义类加载器 HotSwapClassLoader
2. 每次加载类时，检查 .class 文件时间戳是否有修改
3. 如有修改，创建一个新的 HotSwapClassLoader 实例
4. 用新实例重新加载类（旧实例 + 旧类被 GC 回收）
```

**局限性**：只能替换方法体，**不能修改方法签名**（新增/删除方法需要重启），因为旧的类对象可能还在被引用，无法完全卸载。

## 正确使用方式

### 1. 自定义类加载器（加密 .class 文件场景）

```java
class EncryptedClassLoader extends ClassLoader {
    private String classDir;

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        try {
            byte[] classData = loadClassData(name);   // 读取加密的 .class
            byte[] decrypted = decrypt(classData);      // 解密
            return defineClass(name, decrypted, 0, decrypted.length);
        } catch (Exception e) {
            throw new ClassNotFoundException(name);
        }
    }
}
```

**用途**：防止反编译。Android 的 `dex` 文件加密也类似原理。

### 2. 线程上下文类加载器（SPI 场景的正确写法）

```java
// ❌ 错误：硬编码用系统类加载器
InputStream is = ClassLoader.getSystemClassLoader()
    .getResourceAsStream("config.properties");

// ✅ 正确：用线程上下文类加载器（兼容 Web 容器）
ClassLoader cl = Thread.currentThread().getContextClassLoader();
InputStream is = cl.getResourceAsStream("config.properties");
```

**为什么？** 在 Web 容器里，正确的类加载器是线程上下文类加载器，不是系统类加载器。

## 边界情况和坑

### 1. 自定义类加载器 parent 设为自己 → 无限递归

```java
// ❌ 错误：parent 是自己 → 无限递归
class MyLoader extends ClassLoader {
    public MyLoader() {
        super(new MyLoader());  // 父加载器是自己 → 死循环
    }
}

// ✅ 正确：parent = null（Bootstrap）或 AppClassLoader
class MyLoader extends ClassLoader {
    public MyLoader() {
        super(ClassLoader.getSystemClassLoader());  // 父 = AppClassLoader
    }
}
```

### 2. 静态代码块里触发类初始化 → 死锁

```java
// 类 A
class A { static { B.class.getClass(); } }  // 初始化 A 时，触发 B 的初始化

// 类 B
class B { static { A.class.getClass(); } }  // 初始化 B 时，触发 A 的初始化
```

两个线程同时初始化 A 和 B → 互相等待 → 死锁。JVM 规范：类初始化是同步的（`<clinit>` 加锁），但不会检测「递归初始化」。

### 3. `ClassNotFoundException` vs `NoClassDefFoundError`

| 异常 | 抛出时机 | 原因 |
|------|---------|------|
| `ClassNotFoundException` | **编译后**，运行时动态加载类失败 | `Class.forName()`、`ClassLoader.loadClass()` 找不到类 |
| `NoClassDefFoundError` | **编译通过**，运行时 JVM 找不到类的二进制表示 | 类在编译时存在，但运行时 .class 文件被删除了 |

```java
// 编译时存在 User.class，运行时 User.class 被删除
User u = new User();  // 运行时会抛 NoClassDefFoundError
```
