# 反射

## 这个问题为什么存在？

> 正常情况下，代码编译时就知道了要调哪个类、哪个方法。但有些场景（如 Spring 依赖注入、序列化框架、ORM）编译期根本不知道要操作谁——必须在**运行期**动态获取类信息、调用任意方法。问题是：**怎么在运行期让代码像"读配置文件"一样去操作任意类？**

反射的本质是**把「类信息」变成普通对象**，让代码可以自省（Introspection）和自控（Reflection）。

## 它是怎么解决问题的？

### 核心 API

```java
// 获取 Class 对象的三种方式
Class<?> clazz1 = User.class;           // 编译期已知类型
Class<?> clazz2 = user.getClass();      // 运行时对象获取
Class<?> clazz3 = Class.forName("com.example.User");  // 字符串动态加载

// 构造对象（运行时实例化）
Object obj = clazz1.getDeclaredConstructor().newInstance();

// 访问字段
Field nameField = clazz1.getDeclaredField("name");
nameField.setAccessible(true);  // ⚠️ 跳过访问检查（private 也可见）
nameField.set(obj, "张三");

// 调用方法
Method setAge = clazz1.getDeclaredMethod("setAge", int.class);
setAge.invoke(obj, 28);  // obj.setAge(28)
```

### `getDeclaredMethod` vs `getMethod`

```java
clazz.getMethod("toString");        // 包括继承的 public 方法
clazz.getDeclaredMethod("doLogic");  // 只看本类声明的方法（包括 private）
```

`getDeclaredMethod` 能找到 `private`、`protected`、`默认访问` 的方法，但不包括继承的。`getMethod` 只能找 `public` 方法，但会逐层找父类。

### `setAccessible(true)` 的意义

```java
Field field = clazz.getDeclaredField("password");
field.setAccessible(true);  // 跳过 Java 的访问控制检查（private 也可以访问）

// 为什么需要？
// 1. Spring IoC 注入 private 字段
// 2. 序列化框架访问 private 字段
// 3. 测试框架访问 private 实现
```

### 反射性能开销的来源

```
反射调用路径（比普通方法调用慢 10~100 倍）：
  method.invoke(obj, args)
      ↓
  校验参数
      ↓
  AccessibleObject.checkAccess()   ← 访问检查
      ↓
  MethodAccessor.invoke()          ← 反射调用

ReflectionFactory 决定用哪种 MethodAccessor：
  - DelegatingMethodAccessorImpl
      → NativeMethodAccessorImpl（Native）
          → 本地方法（JNI 调用）  ← 第一次调用，生成字节码
      → GeneratedMethodAccessor  ← 生成的字节码（约 20 次后切换）
```

**为什么慢**：
1. **访问检查**：`checkAccess()` 每次都检查（关闭检查则绕过）
2. **方法内联失效**：JIT 看不到反射调用的目标，inline 优化失效
3. **虚表调度失效**：反射调用的方法不经过虚表，每次都要查 `Method`
4. **JIT 优化困难**：JIT 依赖逃逸分析，反射的参数逃逸范围不明确

### JDK 8 LambdaForm 优化

JVM 在同一个方法被反射调用 **约 15~20 次** 后，用 `GeneratedMethodAccessor` 生成字节码代替 JNI 调用。但这不是最优解——JDK 8 引入了 `LambdaForm`：

```java
// 编译时常量（字面量）方法 → JIT 优化后内联，反射开销接近零
Method m = String.class.getMethod("length");
m.invoke("hello");  // JIT 优化后直接内联，不走反射
```

**本质**：当反射调用的目标是**不可变类型**（如 `String.length()`），JIT 可以把反射调用内联掉。但对一般业务对象的反射，收益有限。

### `getDeclaredMethod` vs `getMethod` 的性能差异

```
getDeclaredMethod()：线性扫描本类的所有方法（包括 private）
getMethod()：递归向上找父类，public 方法
```

`getDeclaredMethod` 比 `getMethod` 更快（因为只查本类），但如果需要找父类 public 方法，只能用 `getMethod`。

## 它和相似方案的本质区别是什么？

| 方式 | 获取时机 | 灵活性 | 性能 |
|---|---|---|---|
| 直接调用 | 编译期 | 低（硬编码） | 最优 |
| 反射 | 运行期 | 极高（动态操作任意类） | 慢（10~100x） |
| MethodHandle | 运行期 | 高（方法句柄，像指针） | 居中（比反射快，但仍需 lookup） |
| 动态代理 | 运行期 | 高（AOP、RPC 框架） | 取决于实现（JDK vs CGLIB） |

**MethodHandle**：JDK 7 引入，像一个"方法指针"，可以动态指向任意方法，比反射快（少了访问检查），适合在重复调用的场景。

```java
MethodHandles.Lookup lookup = MethodHandles.lookup();
MethodHandle mh = lookup.findVirtual(User.class, "setName", MethodType.methodType(void.class, String.class));
mh.invokeExact(user, "李四");  // 比 method.invoke() 快
```

## 正确使用方式

### 反射 + 缓存（Spring 的做法）

```java
// 不缓存：每次都解析方法元数据，慢
Method m = clazz.getDeclaredMethod("setName", String.class);
m.invoke(obj, "name");

// 缓存后：只解析一次，后续直接用（Spring BeanWrapper 的做法）
private final Map<Method, MethodHandle> methodCache = new ConcurrentHashMap<>();

public Object invokeCached(Object target, String methodName, Object... args) {
    Method m = findMethod(target.getClass(), methodName);
    // 缓存处理后每次调用开销大幅下降
}
```

### 反射创建对象（比 new 慢约 3 倍）

```java
// 反射构造（可控制是否调用构造器）
Object obj = clazz.getDeclaredConstructor().newInstance();

// new 实例（编译期优化、JIT 内联）
User user = new User();
```

反射创建比 `new` 慢约 3 倍，不算严重。但如果**每次请求都反射创建新实例**（而不是复用），则每次都重新分配内存 + GC 压力，实际开销会大得多。

## 边界情况和坑

### 反射绕过泛型限制

```java
List<String> list = new ArrayList<>();
list.add("hello");

// 反射可以突破泛型类型擦除
Method addMethod = list.getClass().getMethod("add", Object.class);
addMethod.invoke(list, 123);  // 绕过泛型限制，list 里实际有 Integer

String s = list.get(1);  // ❌ 运行时 ClassCastException
```

**原因**：编译期泛型检查被绕过了。`add(123)` 实际被擦除为 `add(Object)`。

### 数组反射的特殊处理

```java
Object array = Array.newInstance(String.class, 3);
Array.set(array, 0, "a");
Array.get(array, 0);

// 反射构建数组类型
Class<?> arrayType = Class.forName("[Ljava.lang.String;");  // "[L" 前缀表示对象数组
```

### 私有类加载器的问题

```java
ClassLoader cl = new URLClassLoader(urls);
Class<?> clazz = cl.loadClass("com.example.InternalClass");
clazz.getMethod("doWork");  // ❌ 如果是 sun.* 包，会报 IllegalAccessError
```

## 我的理解

反射的核心价值是**动态性**：Spring IoC、Jackson、MyBatis 都靠反射实现「代码不需要硬编码类名，但能正确操作任意类」。

面试追问最多的是**性能**：为什么反射慢（访问检查 + inline 失效 + 虚表失效），以及 JDK 8 LambdaForm 如何优化（通过 JIT 内联不可变目标）。
