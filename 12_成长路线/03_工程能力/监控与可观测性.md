# 监控与可观测性

> 没有监控的系统是盲目的。可观测性三支柱：日志（Logs）、指标（Metrics）、链路追踪（Traces）。

---

## 一、可观测性三支柱

```
Logs（日志）：
→ 离散的事件记录
→ 适合排查具体问题
→ 工具：ELK、Loki

Metrics（指标）：
→ 随时间变化的数值
→ 适合监控趋势和告警
→ 工具：Prometheus + Grafana

Traces（链路追踪）：
→ 请求在分布式系统中的完整路径
→ 适合排查跨服务慢请求
→ 工具：SkyWalking、Zipkin、Jaeger
```

## 二、日志（Logs）

### 2.1 日志级别规范

| 级别 | 使用场景 | 示例 |
|------|---------|------|
| ERROR | 影响功能，必须处理 | 数据库连不上、支付失败 |
| WARN | 不影响功能，但需要注意 | 配置缺失使用默认值、重试次数达上限 |
| INFO | 关键业务事件 | 用户登录、订单创建、支付成功 |
| DEBUG | 调试信息，生产不开启 | 方法入参、中间变量 |
| TRACE | 更详细的调试信息 | 每行代码执行记录 |

### 2.2 日志最佳实践

```java
// ✅ 好日志
log.info("用户登录成功, userId={}, ip={}", userId, ip);

// ✅ 带上下文的日志
try {
    orderService.create(order);
    log.info("订单创建成功, orderId={}, userId={}", orderId, userId);
} catch (Exception e) {
    log.error("订单创建失败, orderId={}, userId={}", orderId, userId, e);
}

// ❌ 坏日志
log.info("登录成功");  // 没有上下文
log.debug("调试信息");     // 生产没开，等于没记
log.error("出错");        // 没有异常堆栈
```

### 2.3 敏感信息脱敏

```java
// ❌ 密码不能明文记日志
log.info("用户登录, password={}", password); // 严重安全违规！

// ✅ 脱敏处理
log.info("用户登录, userId={}, password={}", userId, "***");
```

### 2.4 日志框架选型

| 框架 | 特点 | 推荐场景 |
|-------|------|---------|
| Logback | Log4j 继任者，性能最好 | Spring Boot 默认 ⭐ 推荐 |
| Log4j2 | 异步日志性能极佳 | 高并发场景 |
| SLF4J | 门面（Facade），不负责实现 | 必须作为统一 API 使用 |

```xml
<!-- logback-spring.xml 核心配置 -->
<configuration>
    <!-- 控制台输出 -->
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>%d{HH:mm:ss} [%thread] %-5level %logger{36} - %msg%n</pattern>
        </encoder>
    </appender>

    <!-- 文件输出（滚动） -->
    <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>logs/app.log</file>
        <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
            <fileNamePattern>logs/app.%d{yyyy-MM-dd}.log</fileNamePattern>
            <maxHistory>30</maxHistory> <!-- 保留 30 天 -->
        </rollingPolicy>
    </appender>

    <root level="INFO">
        <appender-ref ref="CONSOLE"/>
        <appender-ref ref="FILE"/>
    </root>
</configuration>
```

## 三、指标（Metrics）

### 3.1 四大指标类型

| 类型 | 说明 | 示例 |
|------|------|------|
| Counter（计数器） | 只增不减 | HTTP 请求总数、订单创建总数 |
| Gauge（仪表盘） | 可增可减 | 当前在线用户数、JVM 内存使用量 |
| Histogram（直方图） | 统计分布 | 请求延迟分布（P50/P95/P99） |
| Summary（摘要） | 类似 Histogram，客户端计算 | 请求延迟分位数 |

### 3.2 Prometheus 指标定义

```java
// Counter
Counter orderCounter = Counter.build()
    .name("orders_created_total")
    .help("Total orders created")
    .labelNames("status") // 按状态分类
    .register();

orderCounter.labels("success").inc(); // 订单创建成功 +1

// Gauge
Gauge onlineUsers = Gauge.build()
    .name("online_users")
    .help("Current online users")
    .register();

onlineUsers.set(120); // 设置当前值

// Histogram（延迟分布）
Histogram requestLatency = Histogram.build()
    .name("http_request_duration_seconds")
    .help("Request latency in seconds")
    .buckets(0.01, 0.05, 0.1, 0.5, 1.0) // 分桶
    .register();

Histogram.Timer timer = requestLatency.startTimer();
// ... 处理请求 ...
timer.observeDuration();
```

### 3.3 关键业务指标

```
必须监控的业务指标：
□ 订单创建成功率（goals_created_total / goals_total）
□ 支付成功率（payment_success_total / payment_total）
□ 用户注册转化率
□ 核心接口 P99 延迟
□ 错误率（error_total / request_total）
```

### 3.4 Grafana 仪表盘

```
核心仪表盘：
1. 业务总览（订单量、支付量、用户量）
2. 应用监控（JVM、线程池、GC）
3. 接口监控（QPS、延迟、错误率）
4. 中间件监控（Redis、MySQL、Kafka）
5. 告警面板（当前告警列表）
```

## 四、链路追踪（Traces）

### 4.1 核心概念

```
一个 Trace（追踪）由多个 Span（跨度）组成：

Trace ID: abc-123
├── Span-1: API-Gateway (50ms)
│   ├── Span-2: Order-Service (30ms)
│   │   ├── Span-3: MySQL (10ms)
│   │   └── Span-4: Redis (5ms)
│   └── Span-5: Payment-Service (20ms)
└── Span-6: User-Service (15ms)

每个 Span 记录：操作名、开始时间、持续时间、Tags、Logs
```

### 4.2 SkyWalking 集成

```yaml
# agent/skywalking-agent.config
agent.service_name=${SW_AGENT_NAME:your-service-name}
collector.backend_service=${SW_AGENT_COLLECTOR:skywalking-oap:11800}

# Java 启动参数
java -javaagent:/path/to/skywalking-agent.jar \
     -DSW_AGENT_NAME=order-service \
     -DSW_AGENT_COLLECTOR=skywalking-oap:11800 \
     -jar app.jar
```

### 4.3 追踪数据解读

```
慢请求排查流程：
1. 在 SkyWalking UI 中按延迟排序
2. 找到慢的 Trace，点进去看 Span 列表
3. 找到耗时最长的 Span（比如 MySQL Span 耗时 2s）
4. 点开这个 Span 的 Tags，看 SQL 语句
5. 拿 SQL 去数据库执行 EXPLAIN，看执行计划
6. 发现全表扫描 → 加索引 → 解决
```

## 五、告警策略

### 5.1 告警级别

| 级别 | 响应时间 | 通知方式 | 示例 |
|------|---------|---------|------|
| P0（致命） | 立即响应 | 电话 + 短信 + IM | 核心接口错误率 > 10% |
| P1（严重） | 30 分钟内 | 短信 + IM | 数据库主从延迟 > 60s |
| P2（警告） | 2 小时内 | IM 群 | JVM 堆内存使用 > 80% |
| P3（提示） | 工作时间 | IM 群 | 接口 P99 延迟 > 500ms |

### 5.2 告警规则示例（Prometheus Alertmanager）

```yaml
groups:
  - name: app-alerts
    rules:
      - alert: HighErrorRate
        expr: rate(http_requests_total{status=~"5.."}[5m]) > 0.05
        for: 5m   # 持续 5 分钟才告警，避免抖动
        labels:
          severity: critical
        annotations:
          summary: "错误率超过 5%"
          description: "服务 {{ $labels.instance }} 错误率 {{ $value }}"

      - alert: HighLatency
        expr: histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m])) > 0.5
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "P99 延迟超过 500ms"
```

### 5.3 告警最佳实践

```
✅ 告警要有明确的 On-Call 责任人
✅ 告警要能直接给出「排查路径」（附 Runbook 链接）
✅ 避免告警风暴（一个故障触发 100 条告警）
✅ 定期清理无效告警（Alertmanager 静默规则）
❌ 不要对「正常波动」告警（如 CPU 偶尔到 80%）
❌ 不要设置太多 P0 告警（狼来了效应）
```

## 六、SLA/SLO/SLI

```
SLI（Service Level Indicator，指标）：
→ 实际测量的数值
→ 例：可用性 99.9%、P99 延迟 200ms

SLO（Service Level Objective，目标）：
→ 你承诺要达到的 SLI 值
→ 例：可用性 ≥ 99.9%、P99 延迟 ≤ 200ms

SLA（Service Level Agreement，协议）：
→ 写进合同的 SLO，不达标要赔偿
→ 例：可用性 < 99.9%，退还 10% 服务费
```

### 错误预算（Error Budget）

```
概念：SLO 之外的不可用时间「预算」
例：SLO = 99.9% → 一个月错误预算 = 43.2 分钟

规则：
- 错误预算充足 → 可以大胆发布
- 错误预算快用完 → 冻结发布，专注稳定性

这是 SRE 的核心实践！
```

## 七、监控体系搭建路线

```
阶段 1（基础）：
✅ 应用日志（Logback + ELK）
✅ JVM 监控（Micrometer + Prometheus）
✅ 基础告警（接口错误率、JVM OOM）

阶段 2（完善）：
✅ 链路追踪（SkyWalking）
✅ 业务指标（订单量、支付成功率）
✅ 中间件监控（Redis、MySQL、Kafka）

阶段 3（高级）：
✅ SLO/错误预算管理
✅ 容量规划（根据趋势预测何时需要扩容）
✅ 混沌工程（主动注入故障，验证监控有效性）
```
