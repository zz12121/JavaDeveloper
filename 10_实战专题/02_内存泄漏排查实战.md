# 内存泄漏排查实战

> 内存泄漏不是"内存完全没了"，而是**该回收的对象没被回收**，像水管里慢慢积累的水垢——看着没事，直到管道堵死。

---

## 核心概念：Java 内存泄漏 ≠ 内存溢出

```
内存溢出（OOM）：真的没内存了 → java.lang.OutOfMemoryError
内存泄漏：有内存但收不回 → 逐渐积累，最终导致 OOM

关系：内存泄漏是因，OOM 是果（之一）
```

Java 中内存泄漏的本质：**对象被 GC Root 强引用了，但业务上已经不再需要它。**

常见 GC Root：
- 栈帧中的局部变量
- 静态字段
- JNI 引用
- 活动线程

---

## 排查框架

```
现象观察 → 是否 OOM？ → 是否 Metaspace？ → 堆dump → 分析大对象 → 定位引用链
```

## 场景 A：ThreadLocal 泄漏（最经典）

### 现象

```
线上服务运行一段时间后，Full GC 越来越频繁
堆使用率持续上升，不会回落
最终 OOM：Java heap space
```

### 根因分析

```java
// 问题代码
public class UserContext {
    private static final ThreadLocal<User> CURRENT_USER = new ThreadLocal<>();

    public static void set(User user) {
        CURRENT_USER.set(user);
    }
    // ❌ 没有 remove()！
}
```

ThreadLocal 的内存结构：

```
Thread 对象
  └── ThreadLocalMap（每个线程一个）
        └── Entry[]（WeakReference<ThreadLocal> 作为 key，value 是强引用）
              ├── key: WeakReference → ThreadLocal（GC 时可回收）
              └── value: User（强引用！不会被回收！）
```

当 ThreadLocal 实例被回收后，key 变成 null，但 **value 仍然被 Entry 强引用**。如果线程是线程池中的线程（长期存活），这些 value 永远不会被回收。

### 解决方案

```java
// 在请求结束时必须清理
public class UserContextFilter implements Filter {
    @Override
    public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain) {
        try {
            chain.doFilter(req, res);
        } finally {
            UserContext.remove();  // ✅ 必须清理
        }
    }
}
```

### 验证方法

```bash
# Arthas 在线诊断
vmtool --action getInstances \
  --className java.lang.ThreadLocal \
  --limit 10

# 或者直接看 ThreadLocalMap 的 size
ognl '@com.xxx.UserContext@CURRENT_USER'
```

## 场景 B：静态集合持有对象引用

### 现象

```
服务运行 2-3 天后必须重启
堆使用率曲线：阶梯式上升，Full GC 后只能回落一小部分
```

### 根因

```java
// 问题代码：缓存没有淘汰策略
public class CacheManager {
    private static final Map<String, Object> CACHE = new HashMap<>();
    
    public void put(String key, Object value) {
        CACHE.put(key, value);  // ❌ 永远不清理
    }
}
```

静态 Map 的生命周期 = 类的生命周期 = JVM 的生命周期。对象放进去就不会被回收。

### 解决方案

```java
// 方案1：用 WeakHashMap（key 为弱引用，GC 时可回收）
private static final Map<String, Object> CACHE = new WeakHashMap<>();

// 方案2：用 Caffeine（推荐，带淘汰策略）
private static final Cache<String, Object> CACHE = Caffeine.newBuilder()
    .maximumSize(10_000)           // 最大条目数
    .expireAfterWrite(30, TimeUnit.MINUTES)  // 写入30分钟后过期
    .expireAfterAccess(10, TimeUnit.MINUTES)  // 10分钟不访问就过期
    .recordStats()                 // 记录命中率等统计
    .build();

// 方案3：如果确实需要长期缓存，用堆外内存或专门的缓存中间件
```

## 场景 C：资源未关闭（连接/流）

### 现象

```
频繁报错：Cannot get a connection, pool exhausted
数据库连接数持续增长到上限
```

### 根因

```java
// 问题代码
public List<User> queryUsers() {
    Connection conn = dataSource.getConnection();
    Statement stmt = conn.createStatement();
    ResultSet rs = stmt.executeQuery("SELECT * FROM user");
    List<User> users = new ArrayList<>();
    while (rs.next()) {
        users.add(new User(rs.getString("name")));
    }
    // ❌ 没有关闭 conn/stmt/rs
    return users;
}

// 即使加了 close()，异常时也不会执行
public List<User> queryUsers() {
    Connection conn = dataSource.getConnection();
    Statement stmt = conn.createStatement();
    ResultSet rs = stmt.executeQuery("SELECT * FROM user");
    // 如果这里抛异常，下面的 close 不会执行
    List<User> users = new ArrayList<>();
    while (rs.next()) { ... }
    conn.close();  // ❌ 异常时不会执行
    return users;
}
```

### 解决方案

```java
// 方案1：try-with-resources（推荐）
public List<User> queryUsers() throws SQLException {
    String sql = "SELECT * FROM user";
    try (Connection conn = dataSource.getConnection();
         Statement stmt = conn.createStatement();
         ResultSet rs = stmt.executeQuery(sql)) {
        List<User> users = new ArrayList<>();
        while (rs.next()) {
            users.add(new User(rs.getString("name")));
        }
        return users;
    }
    // conn/stmt/rs 自动关闭，即使抛异常
}

// 方案2：使用连接池 + Spring 的 DataSourceUtils
// 方案3：MyBatis/JPA 框架自动管理
```

## 场景 D：内部类持有外部类引用

### 现象

```
Activity/Controller 对象无法被回收（Android 和 Web 都有这个问题）
每次请求都创建新的 Controller，但旧的不会被回收
```

### 根因

```java
public class OrderController {
    private List<Order> orders = new ArrayList<>();

    public void processOrder() {
        // 匿名内部类隐式持有 OrderController 的引用（this$0）
        new Thread(() -> {
            Order order = fetchOrder();
            orders.add(order);  // ← 这里引用了外部类的 orders 字段
        }).start();
        // 如果线程是长期运行的，OrderController 就不会被 GC
    }
}
```

### 解决方案

```java
// 方案1：使用静态内部类（不持有外部类引用）
private static class OrderTask implements Runnable {
    private final List<Order> orders;  // 显式传入，不持有 Controller
    private final OrderService orderService;

    OrderTask(List<Order> orders, OrderService orderService) {
        this.orders = orders;
        this.orderService = orderService;
    }

    @Override
    public void run() {
        Order order = orderService.fetchOrder();
        orders.add(order);
    }
}

// 方案2：用方法引用或 lambda 替代（lambda 如果不捕获 this，不会持有引用）
```

## 排查工具箱

### 1. 在线诊断（不重启）

```bash
# 查看 JVM 内存概况
jstat -gcutil <pid> 1000     # 每秒刷新一次

# 查看堆中各类型的对象数量和大小
jmap -histo <pid> | head -30

# Arthas：实时查看
dashboard                    # 概览面板
memory                       # 内存使用明细
vmtool --action getInstances \
  --className java.lang.String \
  --limit 10                 # 查看某个类的所有实例
```

### 2. 离线分析（需要 dump）

```bash
# 触发堆 dump（会 STW，注意选低峰期）
jmap -dump:format=b,file=heap.hprof <pid>

# 或者自动 OOM 时 dump（启动参数）
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/var/log/app/heap.hprof
```

### 3. 分析 dump 文件

**MAT（Memory Analyzer Tool）**：
```
1. 打开 heap.hprof
2. 看 Dominator Tree → 按 retained size 排序
3. 找最大的对象 → 看 GC Root 引用链
4. 自动执行 Leak Suspects Report
```

**VisualVM**：
```
1. 打开 heap.hprof
2. Classes 标签页 → 按大小排序
3. 实例视图 → 查看引用关系
```

## 追问链

### 追问 1：WeakHashMap 能完全解决缓存泄漏吗？

> "不能。WeakHashMap 的 key 是弱引用，但 value 是强引用。如果 value 持有了 key 的引用，会形成 `key → value → key` 的引用环，key 反而不会被 GC。正确的做法是用 Caffeine 或 Guava Cache，它们有成熟的淘汰策略。"

### 追问 2：怎么区分内存泄漏和内存不足？

> "内存不足是正常的——业务量大，内存确实不够。解决方式是加内存或优化。内存泄漏是不正常的——同样的业务量，使用时间越长内存越高。关键是看 **Full GC 后内存是否回落**，不回落就是泄漏。"

### 追问 3：线上 OOM 了怎么办？

> "1. 看启动参数有没有配置 `HeapDumpOnOutOfMemoryError`，有就直接拿 dump 分析。2. 没有的话，先恢复服务（重启），然后加上 dump 参数复现。3. 如果 OOM 反复出现但 dump 分析不出问题，可能是 native 内存泄漏（直接内存、JNI），用 `pmap` 或 `NativeMemoryTracking` 排查。"

### 追问 4：Metaspace OOM 怎么排查？

> "Metaspace 存的是类元数据。常见原因：1. 动态代理/CGlib 生成了大量代理类。2. 反复加载类（热部署、类加载器泄漏）。3. 大量 String 常量（JDK8+ 不影响 Metaspace，但影响堆）。排查方式：`-XX:MaxMetaspaceSize=256m` 限制上限，配合 `-XX:+TraceClassLoading` 看类加载日志。"

## 排查 Checklist

```
□ OOM 类型？ → Java heap space / Metaspace / Direct buffer / unable to create native thread
□ Full GC 频率？ → jstat -gcutil，关注 FGC 列和 OU（老年代使用率）列
□ 堆中什么对象最多？ → jmap -histo <pid> | head -20
□ 有 HeapDump 吗？ → 确认 -XX:+HeapDumpOnOutOfMemoryError 配置
□ 静态集合是否无限增长？ → 检查 Cache/Map/List 的清理逻辑
□ ThreadLocal 有 remove 吗？ → 搜索 ThreadLocal.set 的地方，确认有对应的 remove
□ 资源是否在 finally/close 中关闭？ → 检查 Connection/Stream/Channel
```
