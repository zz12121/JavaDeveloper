# 事务隔离级别

## 这个问题为什么存在？

> 两个事务同时操作同一行数据，会互相干扰。
> 如果不加控制，会读到"错误"的数据。

**三个经典的并发问题**：

### 脏读（Dirty Read）

```
时间线：
T1: START TRANSACTION;
T1: UPDATE accounts SET balance = 500 WHERE id = 1;  -- 原值是1000，改成500
T2: START TRANSACTION;
T2: SELECT balance FROM accounts WHERE id = 1;  -- 读到500（脏读！）
T1: ROLLBACK;  -- T1回滚了，balance应该是1000
T2: 基于500做了决策 → 错误！
```

**脏读**：读到了别的事务未提交的数据，结果那个事务回滚了。

### 不可重复读（Non-Repeatable Read）

```
时间线：
T1: START TRANSACTION;
T1: SELECT balance FROM accounts WHERE id = 1;  -- 读到1000
T2: START TRANSACTION;
T2: UPDATE accounts SET balance = 500 WHERE id = 1;
T2: COMMIT;
T1: SELECT balance FROM accounts WHERE id = 1;  -- 读到500（不可重复读！）
T1: COMMIT;
```

**不可重复读**：同一个事务内，两次读取同一行，结果不一样（因为别的事务修改了这行）。

### 幻读（Phantom Read）

```
时间线：
T1: START TRANSACTION;
T1: SELECT * FROM users WHERE age > 20;  -- 返回5行
T2: START TRANSACTION;
T2: INSERT INTO users VALUES (NULL, 25, 'Tom');  -- 插入一行age=25的数据
T2: COMMIT;
T1: SELECT * FROM users WHERE age > 20;  -- 返回6行（幻读！）
T1: COMMIT;
```

**幻读**：同一个事务内，两次执行同样的查询，返回的行数不一样（因为别的事务插入或删除了行）。

---

## 它是怎么解决问题的？

### SQL标准定义的四种隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 说明 |
|---------|------|-----------|------|------|
| READ UNCOMMITTED | ❌ 可能 | ❌ 可能 | ❌ 可能 | 能读未提交的数据 |
| READ COMMITTED | ✅ 解决 | ❌ 可能 | ❌ 可能 | 只能读已提交的数据 |
| REPEATABLE READ | ✅ 解决 | ✅ 解决 | ❌ 可能* | InnoDB默认级别 |
| SERIALIZABLE | ✅ 解决 | ✅ 解决 | ✅ 解决 | 完全串行化 |

*注：InnoDB在REPEATABLE READ级别下，通过Gap锁和Next-Key Lock，**实际已经避免了幻读**。这是InnoDB的特殊实现，超过了SQL标准的要求。

### READ UNCOMMITTED（读未提交）

```sql
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;

-- T1
START TRANSACTION;
UPDATE accounts SET balance = 500 WHERE id = 1;

-- T2（READ UNCOMMITTED）
START TRANSACTION;
SELECT balance FROM accounts WHERE id = 1;  -- 读到500（脏读）
```

**实现方式**：什么都不做，读数据时不加锁、不检查版本。

**问题**：脏读、不可重复读、幻读都可能发生。**生产环境几乎不用**。

### READ COMMITTED（读已提交）

```sql
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- T1
START TRANSACTION;
UPDATE accounts SET balance = 500 WHERE id = 1;
-- 不提交

-- T2（READ COMMITTED）
START TRANSACTION;
SELECT balance FROM accounts WHERE id = 1;  -- 读到1000（原值，避免了脏读）
-- T1提交后
SELECT balance FROM accounts WHERE id = 1;  -- 读到500（不可重复读）
```

**实现方式（InnoDB）**：
- 每次读，都读最新的已提交版本（通过MVCC的Read View实现）
- 每次读都生成新的Read View

**能解决的问题**：脏读
**不能解决的问题**：不可重复读、幻读

**使用场景**：Oracle默认级别，很多互联网公司也用这个级别（减少Gap锁，提升并发）。

### REPEATABLE READ（可重复读，InnoDB默认）

```sql
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- T1
START TRANSACTION;
SELECT balance FROM accounts WHERE id = 1;  -- 读到1000

-- T2
START TRANSACTION;
UPDATE accounts SET balance = 500 WHERE id = 1;
COMMIT;

-- T1
SELECT balance FROM accounts WHERE id = 1;  -- 还是读到1000（可重复读！）
COMMIT;
```

**实现方式（InnoDB）**：
- 事务开始时创建Read View，整个事务期间都用同一个Read View
- 读数据只看事务开始前的已提交版本

**能解决的问题**：脏读、不可重复读
**InnoDB还额外解决了幻读**（通过Gap锁+Next-Key Lock）

**InnoDB的特殊之处**：

```
SQL标准要求REPEATABLE READ解决脏读和不可重复读，不要求解决幻读。
但InnoDB通过锁机制，在RR级别也解决了幻读。
这是InnoDB的重要特性，面试经常问。
```

### SERIALIZABLE（串行化）

```sql
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- 所有读操作都加共享锁
-- 所有写操作都加排他锁
-- 并发性能极差，相当于单线程执行
```

**实现方式**：读加S锁，写加X锁，范围查询加Gap锁，完全串行化。

**能解决的问题**：脏读、不可重复读、幻读
**问题**：性能极差，一般不用。

---

## 它和相似方案的本质区别是什么？

### MVCC vs 锁

| 维度 | MVCC（多版本并发控制） | 锁（Locking） |
|------|---------------------|--------------|
| 读操作 | 不加锁，读快照 | 加共享锁 |
| 写操作 | 加排他锁 | 加排他锁 |
| 并发性能 | 高（读写不阻塞） | 低（读写互相阻塞） |
| 实现复杂度 | 高（需要维护版本链） | 低 |
| InnoDB使用场景 | READ COMMITTED和REPEATABLE READ级别 | 写操作、SERIALIZABLE级别 |

**本质区别**：MVCC通过"多版本"实现非阻塞读，锁通过"互斥"实现隔离。

**InnoDB的混用策略**：
- **读不加锁**：通过MVCC读历史版本（快照读）
- **写加锁**：修改数据时必须加排他锁（当前读）
- **范围查询防止幻读**：加Gap锁或Next-Key锁

### READ COMMITTED vs REPEATABLE READ（InnoDB实现对比）

| 维度 | READ COMMITTED | REPEATABLE READ |
|------|----------------|-----------------|
| Read View创建时机 | 每次读都创建新的 | 事务开始时创建，全程共用 |
| 不可重复读 | 可能发生 | 不会发生 |
| 幻读 | 可能发生 | **InnoDB已通过锁解决** |
| Gap锁 | 不加（除了外键和唯一性检查） | 加Gap锁，防止幻读 |
| 并发性能 | 更高（锁范围小） | 较低（锁范围大） |
| 主从复制 | 用ROW格式binlog | 默认格式 |

**选择建议**：
- 如果能接受不可重复读（比如Web应用，每次请求都是新事务），用READ COMMITTED
- 如果需要事务内多次读取结果一致（比如对账系统），用REPEATABLE READ

## 正确使用方式

### 1. 查看和设置隔离级别

```sql
-- 查看全局和会话级隔离级别
SELECT @@global.transaction_isolation;
SELECT @@session.transaction_isolation;

-- 设置会话级隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 设置全局隔离级别（影响新连接）
SET GLOBAL TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

### 2. 根据业务选择隔离级别

```sql
-- 场景1：电商下单（默认RR即可）
-- 需要可重复读，防止在事务执行期间读到别的订单修改

-- 场景2：报表查询（可以用RC）
-- 不需要可重复读，只要不读脏数据即可
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
SELECT SUM(amount) FROM orders WHERE user_id = 123;

-- 场景3：数据一致性要求极高（用Serializable）
-- 比如财务对账，不能有任何并发问题
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;
```

### 3. 避免在事务中执行不必要的查询

```sql
-- ❌ 错误：在事务里做无关查询
START TRANSACTION;
SELECT * FROM users WHERE id = 1;  -- 持有了id=1的行锁
SELECT * FROM products;              -- 无关查询，延长了锁持有时间
UPDATE users SET last_login = NOW() WHERE id = 1;
COMMIT;

-- ✅ 正确：事务尽量短
START TRANSACTION;
SELECT * FROM users WHERE id = 1 FOR UPDATE;  -- 显式加锁
UPDATE users SET last_login = NOW() WHERE id = 1;
COMMIT;
```

## 边界情况和坑

### 坑1：InnoDB的RR级别"已解决幻读"是有限制的

```sql
-- 快照读（Snapshot Read）不会幻读
START TRANSACTION;
SELECT * FROM users WHERE age > 20;  -- 5行（快照读，不加锁）
-- 别的事务插入了一行age=25
SELECT * FROM users WHERE age > 20;  -- 还是5行（MVCC，读快照）
COMMIT;

-- 当前读（Current Read）会幻读，但InnoDB用Gap锁防止了
START TRANSACTION;
SELECT * FROM users WHERE age > 20 FOR UPDATE;  -- 6行（当前读，加Gap锁）
-- 别的事务想插入age=25 → 被Gap锁阻塞，无法插入
COMMIT;
```

**关键理解**：
- **快照读**（普通SELECT）：通过MVCC避免幻读（读历史版本）
- **当前读**（SELECT...FOR UPDATE、SELECT...LOCK IN SHARE MODE、INSERT/UPDATE/DELETE）：通过Gap锁防止别的事务插入，从而避免幻读

### 坑2：READ COMMITTED下的"半一致性读"优化

```sql
-- InnoDB在RC级别下有个特殊优化：semi-consistent read
-- 当UPDATE语句遇到行锁时，会读最新的已提交版本来判断是否满足条件

-- T1
START TRANSACTION;
UPDATE users SET age = 30 WHERE id = 1;  -- 持有id=1的行锁，未提交

-- T2（RC级别）
START TRANSACTION;
UPDATE users SET name = 'Tom' WHERE age = 25;  
-- 对于id=1这行，T2会读已提交版本（age=原值）
-- 如果原值不满足age=25，就跳过这行（不需要等待T1释放锁）
-- 这减少了锁等待
```

### 坑3：隔离级别与Binlog格式的配合

```sql
-- READ COMMITTED级别 + STATEMENT格式binlog = 主从数据不一致！

-- 主库执行
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
START TRANSACTION;
UPDATE t SET b = b + 1 WHERE a = 1;
COMMIT;

-- 如果binlog格式是STATEMENT，从库重放时可能因为执行计划不同导致结果不同
-- （因为RC级别下，别的事务可能提交了新数据，影响UPDATE的行数）

-- 解决：RC级别必须用ROW格式binlog
SET GLOBAL binlog_format = 'ROW';
```

### 坑4：long_transaction中的"过期读"

```sql
-- REPEATABLE READ保证可重复读，但可能导致读到的数据已经过期
START TRANSACTION;
SELECT balance FROM accounts WHERE id = 1;  -- 读到1000
-- 别的事务已经把balance改成了500，并提交
SELECT balance FROM accounts WHERE id = 1;  -- 还是读到1000（可重复读）
-- 这个1000是"过期"数据
```

**这是REPEATABLE READ的特性，不是Bug**。如果需要读最新的已提交数据，用`READ COMMITTED`或者`SELECT...FOR SHARE`（加共享锁读最新版本）。
