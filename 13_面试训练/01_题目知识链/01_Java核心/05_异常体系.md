## 知识链提示

这道题应该让你联想到：

- `[[异常体系]]` → `Error`（不该捕获）/ `RuntimeException`（可以不捕获）/ 检查异常
- `[[try-with-resources]]` → 编译后变成 `try-catch-finally` + `addSuppressed()`
- `[[自动关闭]]` → `Closeable` vs `AutoCloseable`
- `[[异常性能开销]]` → 填充堆栈快照非常耗时，高并发路径不能抛异常
- `[[异常链]]` → `initCause()` / 构造方法传入 `Throwable cause`

## 参考要点（盲答后再看）

<details>
<summary>点击展开</summary>

**异常体系**：
```
Throwable
├── Error（严重错误，不该捕获，如 OutOfMemoryError）
└── Exception
    ├── RuntimeException（unchecked，可以不捕获）
    │   ├── NullPointerException
    │   ├── IndexOutOfBoundsException
    │   └── ...
    └── 其他（checked，必须捕获或声明 throws）
        ├── IOException
        ├── SQLException
        └── ...
```

**finally 与 return**：
```java
// 字节码层面：
// 1. try块执行，返回值存入局部变量 slot
// 2. finally块执行
// 3. 从slot取返回值返回
// 如果finally里也有return，会覆盖try块的返回值！

try { return 1; }
finally { return 2; }  // 最终返回 2，try的return 1 被丢弃
```

**try-with-resources 编译展开**：
```java
// 源码
try (InputStream is = new FileInputStream("a.txt")) {
    // ...
}

// 编译后大致等价于：
InputStream is = new FileInputStream("a.txt");
Throwable primaryExc = null;
try {
    // ...
} catch (Throwable t) {
    primaryExc = t;
    throw t;
} finally {
    if (is != null) {
        if (primaryExc != null) {
            try { is.close(); } catch (Throwable suppressed) {
                primaryExc.addSuppressed(suppressed);  // 关键！
            }
        } else {
            is.close();
        }
    }
}
```

**异常的性能开销**：
- 创建异常对象时调用 `fillInStackTrace()` → 遍历当前线程堆栈 → **非常慢**（比普通方法调用慢 100~1000 倍）
- 高并发路径（如每请求都走的代码）不能依赖异常做控制流

**自定义异常建议**：
- 业务异常 → 继承 `RuntimeException`（不需要强迫调用方写 try-catch）
- 必须让调用方处理的 → 继承 `Exception`
