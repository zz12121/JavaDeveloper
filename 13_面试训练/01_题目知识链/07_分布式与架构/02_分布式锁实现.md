# 分布式锁的实现方案有哪些？各自的优缺点是什么？

> ⚠️ **先盲答**：单机锁（synchronized / ReentrantLock）在分布式场景下为什么不够用了？

---

## 盲答引导

1. 分布式锁的核心要求是什么？—— 互斥 / 高可用 / 防死锁 / 可重入
2. Redis 实现分布式锁，SETNX + TTL 够吗？不够的话还缺什么？—— 原子性 + 续期 + 多机保障
3. Redisson 是怎么解决 Redis 单机故障的？—— RedLock 算法
4. ZooKeeper 实现分布式锁的原理是什么？—— 临时顺序节点 + Watch

---

## 知识链提示

```
分布式锁
  → [[08_分布式与架构/03_分布式锁/Redis分布式锁实现]]
    → 简单实现：SET key value NX PX timeout
      → 问题：单点故障 / 锁续期 / 可重入 / 非原子性
    → 进阶：Redisson（看门狗机制 + 可重入）
      → 自动续期（watchdog），防止业务执行过长导致锁自动释放
    → RedLock（多机 Redisson）
      → 5 个独立 Redis 实例，分别加锁
      → 多数实例加锁成功才算成功（N/2+1）
      → 问题：实现复杂，仍不能 100% 保证正确性（争议）
  → ZooKeeper锁
    → 原理：临时顺序节点 + Watch 机制
    → 优点：天然分布式，无单点问题
    → 缺点：性能比 Redis 低（每个请求都要走 ZK 集群）
  → 数据库锁（最不推荐）
    → 利用唯一索引 / SELECT FOR UPDATE
    → 缺点：性能差，数据库本身是瓶颈
```

## 核心追问

| 追问 | 考点 |
|------|------|
| Redis 分布式锁如果不设置 TTL，会有什么问题？ | 锁永远不释放（进程崩溃 / 机器重启） |
| 为什么 SETNX + TTL 组合不是原子操作？ | SETNX 成功 + 设置 TTL 失败 = 死锁 |
| Redisson 的 watchdog（看门狗）机制是什么？ | 每 1/3 TTL 续期一次，保证业务执行完之前锁不过期 |
| RedLock 为什么被很多人批评？ | 假设 5 个 Redis 实例独立，但时钟漂移可能导致锁失效 |

---

## 参考答案要点

**三种方案对比**：

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| Redis（SETNX） | 性能高 | 单点故障（需 RedLock） | 并发量大、锁时间短 |
| ZooKeeper | 强一致性，无单点 | 性能低（写请求走 ZK） | 可靠性要求极高 |
| 数据库 | 实现简单 | 性能差，数据库是瓶颈 | 低并发场景 |

**Redisson 是最实用的 Redis 方案**：看门狗续期 + 可重入 + Lua 脚本保证原子性。
