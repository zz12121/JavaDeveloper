**类加载过程（5步）**：
```
加载 → 验证 → 准备 → 解析 → 初始化
       （连接阶段：验证+准备+解析）
```

**双亲委派模型（重点）**：
```java
// ClassLoader.loadClass() 的核心逻辑
protected Class<?> loadClass(String name) {
    Class<?> c = findLoadedClass(name);  // 1. 已加载过？
    if (c == null) {
        if (parent != null) {
            c = parent.loadClass(name);  // 2. 委托父加载器
        } else {
            c = findBootstrapClassOrNull(name); // 3. 父为空→Bootstrap
        }
        if (c == null) {
            c = findClass(name);  // 4. 父找不到，自己找
        }
    }
    return c;
}
```

**为什么需要双亲委派**：
1. **防止核心类被篡改**：`java.lang.Object` 永远由 Bootstrap 加载
2. **避免重复加载**：父加载器加载过的，子加载器不会重复加载

**打破双亲委派的场景**：
| 场景 | 为什么打破 | 怎么打破 |
|------|-----------|----------|
| Tomcat | 多个 Web 应用可能有同名类，需要隔离 | WebAppClassLoader 先自己找，找不到再委托父 |
| SPI（JDBC） | 核心接口在 Bootstrap，实现类在 AppClassLoader | 用线程上下文类加载器（TCCL）反向委托 |
| OSGi | 模块化，精细控制类的可见性 | 完全自定义类加载逻辑 |

**Class.forName vs ClassLoader.loadClass**：
| | Class.forName | ClassLoader.loadClass |
|--|----------------|----------------------|
| 初始化 | ✅ 会触发（执行 static 块） | ❌ 只加载，不初始化 |
| 类加载器 | 调用者所在类加载器 | 指定类加载器 |

**自定义 ClassLoader 正确姿势**：
```java
class MyClassLoader extends ClassLoader {
    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        byte[] bytes = loadClassData(name);  // 自定义：从文件/网络/加密文件读
        return defineClass(name, bytes, 0, bytes.length);
    }
}
// ⚠️ 不要随便覆写 loadClass，除非你真的要打破双亲委派
```
