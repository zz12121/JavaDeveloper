# 限流算法

## 这个问题为什么存在？

> 系统处理能力有上限。如果请求量超过上限，所有请求都会变慢甚至雪崩。问题是：**怎么在系统过载前把多余的请求挡在外面，同时又不浪费正常请求的处理能力？**

限流的本质是**在系统容量和请求量之间做流量整形（Traffic Shaping）**。

## 它是怎么解决问题的？

### 四种核心算法对比

```
请求 ──→ [ 算法 ] ──→ 通过 / 拒绝 / 排队
```

#### 1. 固定窗口计数器

```
窗口大小：1秒，阈值：100 次
[0ms ~ 1000ms] 最多 100 次
```

```java
public class FixedWindowRateLimiter {
    private final int limit;       // 阈值
    private final long windowSize;  // 窗口大小（ms）
    private int count;
    private long windowStart;

    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        if (now - windowStart > windowSize) {
            windowStart = now;     // 新窗口
            count = 0;
        }
        if (count < limit) {
            count++;
            return true;
        }
        return false;  // 超限，拒绝
    }
}
```

**缺点：窗口边界突发**。假如阈值 100，第 1 秒最后 100ms 来了 100 个请求，第 2 秒开始 100ms 又来了 100 个请求——**200ms 内处理了 200 个请求，远超阈值**。

#### 2. 滑动窗口

```
窗口大小：1秒（滑动）
────────────────────────────▶ 时间轴
       ↑                      
    窗口起点（当前时间 - 1s）
```

每次请求判断：统计 `[now - windowSize, now]` 内的请求数。精确度比固定窗口高，但实现复杂。

#### 3. 漏桶（Leaky Bucket）

```
   入请求 → [ 漏桶（固定容量）] → 匀速出水 → 服务
                │
           满了就拒绝
```

```java
public class LeakyBucket {
    private final int capacity;    // 桶容量
    private double water;          // 当前水量
    private final double rate;     // 出水速率（个/秒）
    private long lastTime;

    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        double leaked = (now - lastTime) * rate / 1000.0;
        water = Math.max(0, water - leaked);  // 先漏水
        lastTime = now;

        if (water < capacity) {
            water++;
            return true;
        }
        return false;  // 桶满，拒绝
    }
}
```

**特点**：**输出速率固定**，不管入流量多大，都是匀速处理。适合**保护下游**（如调用第三方 API）。

#### 4. 令牌桶（Token Bucket）

```
   令牌生成器 → [ 令牌桶（固定容量）]
                          │
   入请求 → 取走 1 个令牌 → 通过
             没有令牌 → 拒绝
```

```java
public class TokenBucket {
    private final int capacity;    // 桶容量
    private double tokens;         // 当前令牌数
    private final double rate;     // 令牌生成速率（个/秒）
    private long lastTime;

    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        double generated = (now - lastTime) * rate / 1000.0;
        tokens = Math.min(capacity, tokens + generated);  // 补充令牌
        lastTime = now;

        if (tokens >= 1) {
            tokens--;
            return true;
        }
        return false;  // 没有令牌，拒绝
    }
}
```

**特点**：**允许突发**。桶里有令牌就能一次性通过多个请求。这是和漏桶最大的区别。

### 令牌桶 vs 漏桶

| | 令牌桶 | 漏桶 |
|---|---|---|
| 流量特征 | 允许突发 | 匀速输出 |
| 适用场景 | 保护自身（限流入口） | 保护下游（匀速调用） |
| 实现复杂度 | 中 | 低 |
| 突发处理 | ✅ 桶里有令牌就能突发 | ❌ 始终匀速 |

## 分布式限流：Redis + Lua

单机限流用本地计数器即可，但**分布式系统**需要把所有实例的计数汇总：

```lua
-- key: 限流 key，ARGV[1]: 阈值，ARGV[2]: 窗口大小（秒）
local key = KEYS[1]
local limit = tonumber(ARGV[1])
local window = tonumber(ARGV[2])

local current = redis.call('INCR', key)
if current == 1 then
    redis.call('EXPIRE', key, window)  -- 第一个请求，设置过期时间
end
if current > limit then
    return 0  -- 超限
end
return 1  -- 通过
```

**为什么要用 Lua**：`INCR` + `EXPIRE` 不是原子操作，需要用 Lua 脚本保证原子性。

## 它和相似方案的本质区别是什么？

| 方案 | 目的 | 触发条件 | 处理方式 |
|---|---|---|---|
| 限流（Rate Limit） | 控制 QPS | QPS > 阈值 | 拒绝 / 排队 |
| 熔断（Circuit Breaker） | 保护调用方 | 错误率 > 阈值 | 快速失败，不发起调用 |
| 降级（Degrade） | 保证核心功能 | 系统资源不足 | 关闭非核心功能 |
| 超时（Timeout） | 防止雪崩 | 调用超时 | 释放资源，快速失败 |

**熔断是「下游挂了，我不调了」；限流是「我处理能力有限，多来的我不接了」**。

## 正确使用方式

### Guava RateLimiter（单机令牌桶）

```java
RateLimiter limiter = RateLimiter.create(10.0);  // 每秒 10 个令牌

if (limiter.tryAcquire()) {
    processRequest();
} else {
    return Response.error("限流中，请稍后再试");
}
```

### Spring Cloud Gateway + Redis 限流

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://user-service
          predicates:
            - Path=/api/users/**
          filters:
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 10   # 每秒补充 10 个令牌
                redis-rate-limiter.burstCapacity: 20    # 桶容量 20
```

## 边界情况和坑

### 令牌桶的「预热」问题

系统刚启动时空桶，一下子放进来 `capacity` 个请求，把系统打挂。`RateLimiter` 有 `warmUp` 模式解决这个问题：

```java
RateLimiter limiter = RateLimiter.create(10.0, 5, TimeUnit.SECONDS);
// 5 秒内逐渐提升到 10 QPS，预热期通过数逐渐增多
```

### 滑动窗口的精度与内存权衡

把窗口分成 N 个格子（如 10 个，每个 100ms），滑动时移动一个格子。格数越多精度越高，但内存占用越大。

```
10 个格子 × 每个格子一个计数器 = 10 个计数器
精度：100ms；内存：10 个 long
```

### 分布式限流的 Redis 单点问题

Redis 挂了 → 限流失效 → 所有请求都通过 → 系统可能雪崩。

解决方案：
1. **降级策略**：Redis 不可用时，降级到本地限流（每个实例单独限流，阈值 ÷ 实例数）
2. **Redis Cluster**：提高可用性
3. **令牌桶本地化**：每个实例自己维护令牌桶，用 `Redis` 做协调（适合突发不大的场景）

## 我的理解

限流四种算法各有适用场景：

- **固定窗口**：实现简单，但边界突发严重，适合对精度要求不高的场景
- **滑动窗口**：精度高，但内存开销大
- **漏桶**：输出匀速，保护下游，适合调用第三方 API
- **令牌桶**：允许突发，最常用，适合入口限流

面试追问最高频：**令牌桶和漏桶的区别**——令牌桶允许突发（桶里有令牌就能过），漏桶始终匀速（像漏桶一样一滴一滴漏水）。
