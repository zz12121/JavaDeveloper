# 常见坑与最佳实践

## 这个问题为什么存在？

> MyBatis 用起来简单，但**坑特别多**——大都是「看起来能用，实际有问题」的隐性错误。

本文档整理实际项目中最常见的坑，以及对应的 Best Practice。

---

## 坑1：`#{}` 和 `${}` 混用导致 SQL 注入

```java
// ❌ 危险：用户输入直接拼入 SQL
@Select("SELECT * FROM user WHERE name = '${name}'")
User findByName(String name);
// 如果 name = "xxx' OR '1'='1" → 全表泄露！

// ✅ 安全：用 #{}（预编译）
@Select("SELECT * FROM user WHERE name = #{name}")
User findByName(String name);
```

**最佳实践**：永远用 `#{}`，只在**表名/列名/排序字段**才用 `${}`，且必须做白名单校验。

---

## 坑2：ResultMap 中 `id` 和 `result` 的区别

```xml
<resultMap id="userMap" type="User">
    <id column="id" property="id"/>      ← 主键（有特殊优化）
    <result column="name" property="name"/>
</resultMap>
```

**区别**：
- `<id/>`：MyBatis 用它来判断**对象是否相同**（嵌套集合去重的关键）
- `<result/>`：普通字段

**坑**：如果忘了写 `<id/>`，MyBatis 无法去重 → 嵌套 `collection` 出现重复对象。

---

## 坑3：`\#{}` 中的 `jdbcType` 什么时候需要写？

```xml
<!-- 场景：参数值为 null 时，Oracle 必须指定 jdbcType -->
<insert id="insert">
    INSERT INTO user (name) VALUES (#{name, jdbcType=VARCHAR})
</insert>
```

**规则**：
- MySQL：通常不需要写（驱动能推断）
- Oracle：null 值必须指定 `jdbcType`
- **最佳实践**：团队统一规范，要么全写，要么都不写（MySQL 场景）

## 坑4：`selectOne` 和 `selectList` 的返回类型匹配

```java
// ❌ 错误：方法返回单个对象，但 SQL 返回多行 → 抛 TooManyResultsException
User selectByAge(Integer age);  // SQL：SELECT * FROM user WHERE age = ?

// ✅ 正确一：返回 List
List<User> selectByAge(Integer age);

// ✅ 正确二：SQL 加 LIMIT 1
User selectByAge(@Param("age") Integer age);
<!-- XML: SELECT * FROM user WHERE age = #{age} LIMIT 1 -->
```

---

## 坑5：`@Param` 注解遗漏（多个参数时）

```java
// ❌ 错误：MyBatis 无法推断参数名（编译时参数名丢失）
List<User> selectByAgeAndName(Integer age, String name);
<!-- XML 中：#{age}, #{name} → 报错：Parameter 'age' not found -->

// ✅ 正确：加 @Param
List<User> selectByAgeAndName(@Param("age") Integer age, @Param("name") String name);
```

**注意**：如果用 `javac -parameters` 编译（保留参数名），可以不加 `@Param`。

---

## 坑6：`foreach` 的 `open`/`close`/`separator` 写错

```xml
<!-- ❌ 错误：IN 查询少了括号 -->
<foreach collection="ids" item="id" separator=",">
    #{id}
</foreach>
<!-- 生成：1, 2, 3 → SQL 语法错误 -->

<!-- ✅ 正确 -->
<foreach collection="ids" item="id" open="(" separator="," close=")">
    #{id}
</foreach>
<!-- 生成：(1, 2, 3) -->
```

## 坑7：`@MapperScan` 扫包冲突

```java
// ❌ 错误：两个 @MapperScan 重叠扫描
@MapperScan("com.foo.mapper")
@MapperScan("com.bar.mapper")
// 如果 com.foo.mapper 和 com.bar.mapper 有重名 Mapper → 冲突

// ✅ 正确：用一个 @MapperScan，指定多个包（不支持！需要分开写）
@MapperScan(basePackages = {"com.foo.mapper", "com.bar.mapper"})  // ← 支持数组
```

## 坑8：`SqlSession` 未关闭（非 Spring 环境）

```java
// ❌ 错误：SqlSession 未关闭 → 数据库连接泄漏！
SqlSession session = sqlSessionFactory.openSession();
User user = session.getMapper(UserMapper.class).selectById(1);
// 没有 session.close() → 连接池耗尽

// ✅ 正确：try-with-resources（SqlSession extends Closeable）
try (SqlSession session = sqlSessionFactory.openSession()) {
    User user = session.getMapper(UserMapper.class).selectById(1);
}  // ← 自动调用 close()
```

## 坑9：`#{}` 和 `${}` 在 `LIKE` 查询中的正确用法

```xml
<!-- ❌ 错误一：${} 有注入风险 -->
WHERE name LIKE '%${keyword}%'

<!-- ❌ 错误二：#{} 在 LIKE 中失效（生成 'SELECT * FROM t WHERE name LIKE '%?%' → 语法错误） -->
WHERE name LIKE '%#{keyword}%'

<!-- ✅ 正确一：用 CONCAT -->
WHERE name LIKE CONCAT('%', #{keyword}, '%')

<!-- ✅ 正确二：Java 侧拼接好再传 -->
WHERE name LIKE #{keyword}   ← Java 侧：keyword = "%张%"
```

---

## 坑10：`resultMap` 的 `autoMapping="true"` 失效

```xml
<resultMap id="userMap" type="User" autoMapping="true">
    <result column="dept_name" property="deptName"/>
    <!-- 期望：其他字段自动映射 -->
</resultMap>
<!-- 坑：如果列名和属性名不匹配（且没有开启驼峰转换）→ 那些字段为 null -->

<!-- ✅ 解决：开启驼峰转换 -->
<setting name="mapUnderscoreToCamelCase" value="true"/>
```

## 最佳实践总结

### SQL 编写
| 实践 | 说明 |
|------|------|
| 永远用 `#{}` | 防止 SQL 注入 |
| `${}` 只用于表名/列名 | 且必须白名单校验 |
| `LIKE` 用 `CONCAT` | 不要用 `${}` |
| 写 `ResultMap` 时，**必须写 `<id/>` | 影响嵌套集合去重 |

### 性能优化
| 实践 | 说明 |
|------|------|
| 开启 `mapUnderscoreToCamelCase` | 减少 `resultMap` 配置 |
| 批量插入用 `foreach` 拼接 + 分批 | 每批 500~1000 条 |
| JDBC URL 加 `rewriteBatchedStatements=true` | MySQL 批量性能提升 5~10 倍 |
| 避免 N+1：用 `JOIN` + 嵌套结果映射 | 而不是 `select` 嵌套查询 |

### 事务管理
| 实践 | 说明 |
|------|------|
| Spring 整合后，**不要手动调 `sqlSession.commit()`** | 让 `@Transactional` 管理 |
| `@Transactional` 方法必须是 `public` | 否则 AOP 不生效 |
| 批量操作在 `@Transactional` 内执行 | 保证事务一致性 |

### 配置建议
```xml
<!-- mybatis-config.xml 推荐配置 -->
<settings>
    <!-- 驼峰自动映射 -->
    <setting name="mapUnderscoreToCamelCase" value="true"/>
    <!-- 延迟加载（需要时开启） -->
    <setting name="lazyLoadingEnabled" value="false"/>
    <!-- 秒级超时（防止慢 SQL 一直占连接） -->
    <setting name="defaultStatementTimeout" value="30"/>
    <!-- 二级缓存：关闭（用 Redis 替代） -->
    <!-- 不需要配置 <cache/> -->
</settings>
```

## 我的理解

MyBatis 的坑大多集中在**映射规则**和**SQL 拼接**两个方面。

- 映射坑：`#{}` vs `${}`、参数名推断、`ResultMap` 的 `id` 标签
- 性能坑：N+1 问题、批量操作方式选错
- 事务坑：Spring 整合后的自调用问题、SqlSession 未关闭

**核心原则**：
> 「看起来能用」不等于「正确」——MyBatis 不报错不代表逻辑对。
> 如：`${}` 能跑，但有 SQL 注入；不写 `<id/>` 能跑，但嵌套集合去重失败。

**实际项目 checklist**：
- [ ] 所有 `${}` 都有白名单校验
- [ ] 没有 N+1 问题（用 JOIN 代替嵌套 `select`）
- [ ] 批量操作用了 `foreach` 拼接 + 分批
- [ ] `@Transactional` 方法都是 `public`
- [ ] 配置文件中开启了 `mapUnderscoreToCamelCase`
