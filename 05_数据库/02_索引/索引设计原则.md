# 索引设计原则

## 这个问题为什么存在？

> 建了10个索引，查询还是慢。或者建了索引，优化器却不用。
> 
> 索引不是"建得越多越好"——每个索引都要占用磁盘空间，都要在写入时维护。

索引设计本质上是在权衡：
- **查询性能** vs **写入性能**
- **索引覆盖率** vs **索引维护成本**

### 真实场景：索引建了但没用

```sql
-- 表结构
CREATE TABLE orders (
    id INT PRIMARY KEY,
    user_id INT,
    status TINYINT,
    created_at DATETIME,
    amount DECIMAL(10,2)
);

-- DBA建了这些索引
CREATE INDEX idx_user_id ON orders(user_id);
CREATE INDEX idx_status ON orders(status);
CREATE INDEX idx_created_at ON orders(created_at);

-- 查询还是慢
SELECT * FROM orders 
WHERE user_id = 123 
  AND status = 1 
  AND created_at > '2024-01-01'
ORDER BY created_at DESC
LIMIT 20;
```

**问题在哪？**
- 三个单列索引，优化器只能选一个最好的
- 即使选了`idx_user_id`，还要回表过滤`status`和`created_at`
- `ORDER BY created_at`可能无法利用索引排序，需要filesort

**正确的设计应该是联合索引**：

```sql
-- 根据查询模式设计联合索引
CREATE INDEX idx_user_status_created ON orders(user_id, status, created_at);
```

---

## 它是怎么解决问题的？

### 核心原则1：根据WHERE、JOIN、ORDER BY设计索引

索引的价值在于**加速数据定位**。设计索引时，先分析系统的查询模式：

```sql
-- 查询模式1：单表等值查询
SELECT * FROM users WHERE email = 'user@example.com';
-- → 给 email 建索引（高选择性）

-- 查询模式2：范围查询 + 排序
SELECT * FROM orders 
WHERE user_id = 123 
  AND created_at BETWEEN '2024-01-01' AND '2024-01-31'
ORDER BY created_at DESC;
-- → 给 (user_id, created_at) 建联合索引

-- 查询模式3：JOIN查询
SELECT o.* FROM orders o
JOIN users u ON o.user_id = u.id
WHERE u.vip_level = 1;
-- → 给 users.vip_level 建索引（用于JOIN过滤）
-- → 给 orders.user_id 建索引（用于JOIN查找）
```

**设计流程**：

```
1. 收集慢查询日志（slow query log）
2. 对每条慢查询做EXPLAIN分析
3. 识别查询的过滤条件（WHERE）、连接条件（JOIN）、排序字段（ORDER BY）
4. 设计索引覆盖这些条件
5. 上线后持续监控，迭代优化
```

### 核心原则2：联合索引的列顺序——等值条件放前，范围条件放后

这是索引设计最重要的规则，没有之一。

#### 为什么顺序这么重要？

联合索引的B+树是按照**索引列顺序**构建的。以`(A, B, C)`为例：

```
根节点
├── A=1
│   ├── B=2, C=3 → 行数据
│   ├── B=2, C=5 → 行数据
│   └── B=4, C=1 → 行数据
├── A=2
│   ├── B=1, C=2 → 行数据
│   └── B=3, C=4 → 行数据
└── A=3
    └── ...
```

**查询能利用索引的前提是：从左往右匹配，遇到范围查询就停止匹配。**

```sql
-- 索引：(user_id, status, created_at)

-- ✅ 能完全利用索引
WHERE user_id = 123 AND status = 1 AND created_at > '2024-01-01'
-- 等值条件(user_id, status)能匹配，范围条件(created_at)能用于过滤但不能用于后续列

-- ✅ 能部分利用索引
WHERE user_id = 123 AND created_at > '2024-01-01'
-- 只用到 user_id，因为 created_at 是范围查询，跳过了 status

-- ❌ 不能利用索引
WHERE status = 1 AND created_at > '2024-01-01'
-- 没有包含最左前缀 user_id
```

#### 等值条件 vs 范围条件

| 条件类型 | 示例 | 能否作为索引前缀 | 说明 |
|---------|------|----------------|------|
| 等值条件 | `user_id = 123` | ✅ 能 | 精确匹配，继续往后匹配 |
| IN条件 | `user_id IN (1,2,3)` | ✅ 能 | 多个等值，优化器会处理 |
| 范围条件 | `created_at > '2024-01-01'` | ⚠️ 部分 | 能过滤，但后续列无法利用索引 |
| 排序 | `ORDER BY created_at` | ⚠️ 部分 | 能避免filesort，但后续列无法利用 |

**设计规则**：
```
等值条件放最前 → 范围条件放中间 → 排序字段放最后
```

```sql
-- 查询
SELECT * FROM orders 
WHERE user_id = 123       -- 等值，放第1位
  AND status IN (1,2,3)   -- 多等值，放第2位
  AND created_at >= '2024-01-01'  -- 范围，放第3位
ORDER BY created_at DESC   -- 与范围条件同字段，已包含在索引中
LIMIT 20;

-- 正确索引设计
CREATE INDEX idx_uid_status_created ON orders(user_id, status, created_at);
```

### 核心原则3：选择性高的列放前面

**选择性（Selectivity）** = 不重复值数量 / 总行数

```sql
-- 计算各列的选择性
SELECT 
    COUNT(DISTINCT gender) / COUNT(*) AS gender_selectivity,     -- 低（男/女，约0.5）
    COUNT(DISTINCT city) / COUNT(*) AS city_selectivity,          -- 中（几百个城市）
    COUNT(DISTINCT email) / COUNT(*) AS email_selectivity        -- 高（几乎每一行都不同，接近1）
FROM users;

-- 结果示例
-- gender_selectivity: 0.0002 (几乎无过滤能力)
-- city_selectivity: 0.05
-- email_selectivity: 0.99 (几乎唯一)
```

**选择性越高，过滤能力越强，应该放索引越前面。**

```sql
-- ❌ 错误设计：低选择性列放前面
CREATE INDEX idx_gender_city ON users(gender, city);
-- 先用gender过滤，几乎没过滤掉任何行（50%行都匹配）

-- ✅ 正确设计：高选择性列放前面
CREATE INDEX idx_city_gender ON users(city, gender);
-- 先用city过滤，过滤掉大部分行，再用gender过滤少量行
```

**但注意**：这个规则要**结合规则2（等值优先）**。如果低选择性列是等值条件，高选择性列是范围条件，还是要把等值条件放前面：

```sql
-- 即使 gender 选择性低，但它是等值条件，还是要放前面
CREATE INDEX idx_gender_created ON orders(gender, created_at);
-- WHERE gender = 1 AND created_at > '2024-01-01'
```

### 核心原则4：优先设计覆盖索引

**覆盖索引（Covering Index）**：索引包含了查询需要的所有字段，不需要回表。

```sql
-- 查询只需要少数几个字段
SELECT id, user_id, status, created_at 
FROM orders 
WHERE user_id = 123;

-- 普通索引：需要回表
CREATE INDEX idx_user_id ON orders(user_id);
-- 流程：索引找到 user_id=123 的主键ID → 回表查完整行 → 返回所需字段

-- 覆盖索引：不需要回表
CREATE INDEX idx_user_covering ON orders(user_id, status, created_at);
-- 流程：索引直接包含所需字段 → 直接返回
```

**覆盖索引的优势**：
1. **减少随机IO**：回表是随机IO，覆盖索引是顺序IO
2. **减少页读取**：索引页通常比数据页小
3. **减少锁竞争**：InnoDB在索引页加锁，不回表就不需要在数据页加锁

```sql
-- 使用覆盖索引后，EXPLAIN的Extra列会显示"Using index"
EXPLAIN SELECT user_id, status FROM orders WHERE user_id = 123;
-- Extra: Using index  ← 覆盖索引生效
```

**设计技巧**：
```sql
-- 即使SELECT *，也可以通过主键实现"伪覆盖"
-- InnoDB的二级索引叶子节点包含主键，所以：
CREATE INDEX idx_user_status ON orders(user_id, status);
-- 对于 SELECT id, user_id, status FROM orders WHERE user_id = 123
-- 这个索引已经覆盖了所有需要的字段（id在叶子节点中）
```

### 核心原则5：避免索引失效的设计

索引失效的根本原因是**索引字段被"污染"**——MySQL无法直接从索引树定位。

#### 坑1：对索引字段做函数运算

```sql
-- ❌ 索引失效
SELECT * FROM orders WHERE DATE(created_at) = '2024-01-01';
-- DATE()函数作用在索引字段上，优化器放弃使用索引

-- ✅ 改写后索引生效
SELECT * FROM orders 
WHERE created_at >= '2024-01-01' 
  AND created_at < '2024-01-02';
```

#### 坑2：隐式类型转换

```sql
-- phone是VARCHAR类型
CREATE INDEX idx_phone ON users(phone);

-- ❌ 索引失效：传入数字是INT，发生隐式转换
SELECT * FROM users WHERE phone = 13800138000;
-- MySQL会把VARCHAR转成INT比较，相当于对索引字段用了函数

-- ✅ 索引生效：传入字符串
SELECT * FROM users WHERE phone = '13800138000';
```

#### 坑3：最左前缀被破坏

```sql
-- 索引：(a, b, c)
CREATE INDEX idx_abc ON t(a, b, c);

-- ❌ 跳过最左列，索引失效
WHERE b = 2 AND c = 3   -- 没用上a

-- ✅ 使用最左前缀
WHERE a = 1 AND b = 2   -- 用上了(a, b)
WHERE a = 1              -- 用上了(a)
```

## 它和相似方案的本质区别是什么？

### 单列索引 vs 联合索引

| 维度 | 单列索引 | 联合索引 |
|------|---------|---------|
| 适用场景 | 单条件过滤 | 多条件组合过滤 |
| 索引利用率 | 优化器只选一个 | 能利用多个条件 |
| 回表次数 | 可能多次回表 | 联合索引可能覆盖更多字段 |
| 索引维护成本 | 低 | 高（索引更大） |
| 设计难度 | 低 | 高（需要斟酌列顺序） |

**本质区别**：联合索引通过"索引覆盖"减少了回表次数，但设计不当会造成空间浪费。

### B+树索引 vs 哈希索引

| 维度 | B+树索引 | 哈希索引 |
|------|---------|---------|
| 等值查询 | O(logN) | O(1) |
| 范围查询 | ✅ 支持 | ❌ 不支持 |
| 排序 | ✅ 支持（按索引顺序扫描） | ❌ 不支持 |
| 索引大小 | 较大 | 较小 |
| 适用场景 | 综合查询场景 | 纯等值查询（如KV） |

**本质区别**：B+树是有序结构，支持范围；哈希是无序的，只支持等值。

InnoDB的**自适应哈希索引（AHI）**会自动为频繁访问的等值查询建立哈希索引，但这个是InnoDB自动管理的，不需要用户干预。

### 聚簇索引 vs 非聚簇索引

（详见`聚簇索引与非聚簇索引.md`，这里只做对比）

| 维度 | 聚簇索引（主键） | 非聚簇索引（二级索引） |
|------|-----------------|---------------------|
| 叶子节点 | 完整行数据 | 主键值 |
| 回表 | 不需要 | 需要（除非覆盖索引） |
| 数量 | 每张表只有一个 | 可以多个 |
| 查询成本 | 低（一次IO） | 高（可能多次IO） |

**设计启示**：
- 主键要短小（影响所有二级索引的大小）
- 避免使用UUID作为主键（插入乱序，导致页分裂）

## 正确使用方式

### 步骤1：分析查询模式

```sql
-- 开启慢查询日志
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;  -- 超过1秒的查询记录到慢查询日志

-- 分析慢查询
SELECT * FROM mysql.slow_log ORDER BY start_time DESC LIMIT 10;
```

### 步骤2：设计索引

```sql
-- 对每条慢查询，分析其WHERE、JOIN、ORDER BY
EXPLAIN SELECT * FROM orders 
WHERE user_id = 123 
  AND status = 1 
  AND created_at > '2024-01-01'
ORDER BY created_at DESC
LIMIT 20;

-- 根据EXPLAIN结果设计索引
CREATE INDEX idx_user_status_created ON orders(user_id, status, created_at);
```

### 步骤3：在线加索引（避免锁表）

```sql
-- MySQL 5.6+ 支持Online DDL，但仍有影响
-- 建议使用pt-online-schema-change或gh-ost

-- 方式1：Online DDL（MySQL 5.6+）
ALTER TABLE orders ADD INDEX idx_user_status_created(user_id, status, created_at), ALGORITHM=INPLACE, LOCK=NONE;

-- 方式2：使用pt-online-schema-change（推荐生产环境）
-- pt-online-schema-change --alter="ADD INDEX idx_user_status_created(user_id, status, created_at)" D=db,t=orders --execute
```

### 步骤4：验证索引效果

```sql
-- 加索引前后分别执行，对比性能
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 123 AND status = 1;
```

## 边界情况和坑

### 坑1：索引越多，写入越慢

每个索引都要在INSERT/UPDATE/DELETE时维护。

```sql
-- 表有10个索引
CREATE TABLE t (
    id INT PRIMARY KEY,
    a INT, b INT, c INT, ...,
    INDEX idx_a(a), INDEX idx_b(b), ..., INDEX idx_j(j)
);

-- 插入一行，需要更新1个聚簇索引 + 10个二级索引 = 11次索引维护
INSERT INTO t VALUES (1, 1, 1, ...);
```

**经验规则**：
- OLTP系统（频繁写入）：单表索引不超过5-7个
- OLAP系统（频繁查询）：可以适当增加索引

### 坑2：前缀索引的选择性损失

```sql
-- VARCHAR(255)的字段，为了节省空间，建前缀索引
CREATE INDEX idx_title_prefix ON articles(title(20));

-- 问题：前缀可能不够唯一
-- "深入理解Java虚拟机..." 和 "深入理解Android..." 前20个字符相同
-- 前缀索引无法区分这两行，仍需回表验证完整值
```

**如何确定前缀长度**：

```sql
-- 计算不同前缀长度的选择性
SELECT 
    COUNT(DISTINCT LEFT(title, 10)) / COUNT(*) AS sel_10,
    COUNT(DISTINCT LEFT(title, 20)) / COUNT(*) AS sel_20,
    COUNT(DISTINCT LEFT(title, 50)) / COUNT(*) AS sel_50,
    COUNT(DISTINCT title) / COUNT(*) AS sel_full
FROM articles;

-- 当前缀选择性接近完整列选择性时，就是合适的前缀长度
-- 例如：sel_20 = 0.95, sel_full = 0.99，选择前缀20即可
```

### 坑3：索引合并（Index Merge）往往是设计不足的信号

```sql
-- 两个单列索引
CREATE INDEX idx_user_id ON orders(user_id);
CREATE INDEX idx_status ON orders(status);

-- 查询
SELECT * FROM orders WHERE user_id = 123 OR status = 1;

-- EXPLAIN可能显示"Using union(idx_user_id, idx_status)"
-- 这意味着优化器分别扫描两个索引，然后合并结果
```

**索引合并的问题**：
1. 需要扫描两个索引 → IO增加
2. 合并操作消耗CPU
3. 通常不如一个设计良好的联合索引

```sql
-- 更好的设计：根据查询模式建联合索引
-- 如果查询是 OR 条件，考虑是否能改成 UNION
SELECT * FROM orders WHERE user_id = 123
UNION ALL
SELECT * FROM orders WHERE status = 1 AND user_id != 123;
```

### 坑4：NULL值对索引选择性的影响

```sql
-- 如果列有大量NULL，索引的选择性会计算错误
CREATE INDEX idx_middle_name ON users(middle_name);
-- 大部分行middle_name为NULL，这个索引的选择性很低

-- 查询优化器可能认为"扫描全表比走索引便宜"
SELECT * FROM users WHERE middle_name = 'James';
-- 可能不走索引，因为优化器估计匹配行数太多
```

**处理方式**：
```sql
-- 方式1：使用复合索引，把NULL值排在一起
CREATE INDEX idx_middle_name_id ON users(middle_name, id);

-- 方式2：如果业务允许，用空字符串代替NULL
UPDATE users SET middle_name = '' WHERE middle_name IS NULL;
```

## 我的理解

索引设计本质上是一个**空间换时间**的权衡游戏：

1. **先搞清楚查询模式**——盲目建索引是灾难
2. **联合索引的顺序是艺术**——等值放前，范围放后，选择性高的放前
3. **覆盖索引是王道**——能覆盖就覆盖，减少回表
4. **索引不是越多越好**——每个索引都是写入时的负担
5. **用EXPLAIN验证**——直觉不可靠，数据说话

**一个实用的索引设计检查清单**：

```
□ 这个索引能覆盖哪些查询？
□ 联合索引的列顺序是否合理（等值→范围→排序）？
□ 选择性低的列是否不必要地占了索引前缀？
□ 是否能用覆盖索引避免回表？
□ 这个索引的维护成本（写入性能影响）是否可接受？
□ 是否有冗余索引？（例如(a,b)和(a)是冗余的）
```

## 面试话术

**Q：如何设计索引？**

"我会按照这个流程来：

第一，分析查询模式。通过慢查询日志找到需要优化的SQL，用EXPLAIN分析它的执行计划。

第二，设计联合索引。遵循几个原则：
- 等值条件放最前面，范围条件放后面
- 选择性高的列优先放在前面
- 尽量设计成覆盖索引，避免回表
- 避免对索引字段做函数运算

第三，权衡写入性能。索引越多，写入越慢。一般建议单表索引不超过5-7个。

第四，上线后持续监控。用EXPLAIN验证索引是否被使用，根据业务变化调整索引。"

**Q：联合索引(a,b,c)，哪些查询能利用索引？**

"这考察最左前缀原则。
- `WHERE a = 1` —— 能利用，用到a列
- `WHERE a = 1 AND b = 2` —— 能利用，用到a,b列
- `WHERE a = 1 AND b = 2 AND c = 3` —— 能利用，用到全部三列
- `WHERE a = 1 AND c = 3` —— 能利用，但只用到a列，c列无法跳过b直接利用索引
- `WHERE b = 2 AND c = 3` —— 不能利用，没有最左前缀a

还有个特殊情况：如果查询有排序，且排序字段是索引的后续列，也可能利用索引避免filesort。比如`WHERE a = 1 ORDER BY b, c`，能利用索引排序。"

**Q：索引越多越好吗？**

"不是。每个索引都要占用磁盘空间，都要在写入时维护。

索引多的坏处：
1. 写入变慢：每次INSERT/UPDATE/DELETE都要更新所有索引
2. 优化器选择困难：索引太多，优化器可能选错
3. 占用内存：Buffer Pool的空间被索引页占用，数据页可能放不下了

经验上，OLTP系统的单表索引建议不超过5-7个。如果查询模式复杂，可以考虑读写分离，读库多建些索引。"

## 本文总结

| 原则 | 核心要点 |
|------|---------|
| 根据查询模式设计 | 分析WHERE、JOIN、ORDER BY，不要盲目建索引 |
| 等值放前，范围放后 | 联合索引的列顺序决定了能利用多少 |
| 选择性高的放前面 | 过滤能力强的列优先 |
| 优先覆盖索引 | 减少回表，提升性能 |
| 避免索引失效 | 不对索引字段用函数、注意隐式转换 |
| 控制索引数量 | 单表不超过5-7个（OLTP） |
| 用EXPLAIN验证 | 直觉不可靠，看执行计划 |

索引设计不是"背规则"，而是在理解B+树原理的基础上，针对具体查询模式做权衡。最好的索引设计是**让80%的查询都能高效利用索引，同时写入性能可接受**。
