# 消息队列的核心概念是什么？

> ⚠️ **先盲答**：为什么需要消息队列？它解决了什么问题？

---

## 盲答引导

1. 消息队列的核心价值是什么？—— 解耦 / 异步 / 削峰
2. 点对点（P2P）和发布订阅（Pub/Sub）模式有什么区别？
3. 消息可靠性怎么保证？—— 生产者确认 / 消息持久化 / 消费者 ACK
4. 消息重复消费怎么处理？—— 幂等性

---

## 知识链提示

```
消息队列
  → [[09_工程化/04_Kubernetes/核心概念]]
    → 核心价值
      → 解耦：生产者和消费者不需要同时在线，不需要知道彼此地址
      → 异步：生产者发完消息即可返回，不需要等消费者处理完
      → 削峰：突发流量写入 MQ，消费者按自己能力消费
    → 两种消息模型
      → 点对点（Queue）：一个消息只能被一个消费者消费
      → 发布订阅（Topic）：一个消息可以被多个订阅者消费
    → 消息可靠性（三段）
      → 生产者 → MQ：事务消息 / 确认机制（RabbitMQ Confirm / Kafka ACK）
      → MQ → 存储：持久化（写磁盘 / 副本）
      → 存储 → 消费者：消费者 ACK（消费成功才删除消息）
    → 消息有序性
      → 全局有序：一个 Topic 只有一个分区/队列（性能差）
      → 局部有序：按业务 key 哈希到同一分区（Kafka）
    → 消息重复
      → 网络超时 → 生产者重试 → 重复消息
      → 消费者消费完但 ACK 失败 → 重新投递 → 重复消费
      → 解决：消费者幂等（[[08_分布式与架构/05_高可用设计/幂等性设计]]）
```

## 核心追问

| 追问 | 考点 |
|------|------|
| 消息队列能不能保证消息不丢失？ | 不能 100% 保证，只能极大降低概率 |
| Kafka 怎么保证消息不丢？ | acks=all + replication + 关闭自动提交 offset |
| 什么是「死信队列」？ | 无法被正常消费的消息，转入死信队列人工处理 |
| 消息积压了怎么办？ | 增加消费者实例 / 优化消费者处理逻辑 |

---

## 参考答案要点

**三大核心价值**：解耦 + 异步 + 削峰。

**消息可靠性三段保护**：
1. 生产者 → MQ：Confirm 机制
2. MQ 存储：持久化 + 多副本
3. MQ → 消费者：消费者 ACK（手动 ACK，消费成功再提交）

**重复消费**：消费者必须实现幂等（`[[08_分布式与架构/05_高可用设计/幂等性设计]]`）。
