# 高性能连接池设计

> 连接池核心：对象复用 + 公平性 + 动态调整 + 泄漏检测。

## 连接池核心架构

```
应用请求
   │
   ▼
┌─────────┐  获取  ┌─────────┐  复用/创建  ┌─────────┐
│  业务线程 │ ────→  │  连接池  │ ────→    │  空闲队列  │
│  请求连接 │        │  管理器  │         │  已用队列  │
└─────────┘        └────┬────┘         └─────────┘
                        │
     ┌──────────────────┼──────────────────┐
     │                  │                  │
     ▼                  ▼                  ▼
┌─────────┐      ┌─────────┐      ┌─────────┐
│ 泄漏检测 │      │ 动态调整 │      │ JMX监控  │
│ 定时扫描 │      │ min/max │      │ 运行时查看 │
└─────────┘      └─────────┘      └─────────┘
```

## 场景 A：通用对象池设计GenericObjectPool

### 现象

```
每次请求都创建新连接，数据库响应慢
连接频繁建立/销毁，CPU占用高
连接数不受控，数据库连接耗尽
```

### 根因

```
未使用连接池，每次操作都新建连接
无复用机制，资源浪费严重
无连接数限制，导致后端服务压力过大
```

### 解决方案

```java
// 基于Apache Commons Pool2的通用对象池实现
public class ConnectionPool {
    
    // 对象工厂：定义对象的创建、销毁和校验逻辑
    private static class PooledConnectionFactory extends BasePooledObjectFactory<Connection> {
        
        @Override
        public Connection create() throws Exception {
            // 创建新连接（如JDBC/Redis/HTTP连接）
            return DriverManager.getConnection("jdbc:mysql://localhost:3306/mydb");
        }
        
        @Override
        public PooledObject<Connection> wrap(Connection conn) {
            return new DefaultPooledObject<>(conn);
        }
        
        @Override
        public boolean validateObject(PooledObject<Connection> p) {
            // 校验连接是否可用（发送心跳包）
            try {
                return p.getObject().isValid(1);  // 1秒超时
            } catch (SQLException e) {
                return false;
            }
        }
        
        @Override
        public void destroyObject(PooledObject<Connection> p) throws Exception {
            // 销毁连接（关闭物理连接）
            p.getObject().close();
        }
    }

    private final GenericObjectPool<Connection> pool;

    public ConnectionPool() {
        GenericObjectPoolConfig config = new GenericObjectPoolConfig<>();
        config.setMaxTotal(100);       // 最大连接数
        config.setMaxIdle(50);         // 最大空闲连接数
        config.setMinIdle(10);         // 最小空闲连接数
        config.setMaxWaitMillis(3000); // 获取超时时间3秒
        
        this.pool = new GenericObjectPool<>(new PooledConnectionFactory(), config);
    }
    
    /**
     * 从池中获取连接（自动处理等待和超时）
     */
    public Connection borrowConnection() throws Exception {
        return pool.borrowObject();  // 内部会阻塞等待或抛出异常
    }
    
    /**
     * 归还连接到池中（必须finally调用）
     */
    public void returnConnection(Connection conn) {
        pool.returnObject(conn);
    }
}
```

**对象池关键**：
- 用`BasePooledObjectFactory`定义对象的完整生命周期
- `borrowObject()`获取连接，`returnObject()`归还连接
- 池内部维护空闲队列和已用队列，保证线程安全

## 场景 B：公平性与超时策略

### 现象

```
某些线程长期拿不到连接，饥饿现象明显
连接池耗尽时所有线程无限等待
高峰期大量线程堆积，最终OOM
```

### 根因

```
默认采用LIFO（后进先出），后到的线程先拿到连接
无公平性机制，早到的线程可能一直等不到
无超时控制，线程无限阻塞
```

### 解决方案

```java
// 配置公平性和超时参数
public class FairConnectionPool {
    
    public static void main(String[] args) {
        GenericObjectPoolConfig config = new GenericObjectPoolConfig<>();
        
        // 关键配置1：启用FIFO（先进先出）公平模式
        config.setLifo(false);  // false=先进先出，避免饥饿
        
        // 关键配置2：设置获取超时时间
        config.setMaxWaitMillis(5000);  // 5秒超时，超时抛出NoSuchElementException
        
        // 关键配置3：当连接耗尽时的策略
        config.setBlockWhenExhausted(true);  // true=阻塞等待，false=直接失败
        config.setTestOnBorrow(true);  // 借出前校验连接有效性
        
        GenericObjectPool<Connection> pool = 
            new GenericObjectPool<>(new PooledConnectionFactory(), config);
        
        // 使用示例（必须try-with-resources或finally归还）
        try (Connection conn = pool.borrowObject()) {
            // 执行业务逻辑
            doSomething(conn);
        } catch (Exception e) {
            // 处理异常
            log.error("获取连接失败", e);
        }
    }
}
```

**公平性关键**：
- 设置`lifo=false`启用FIFO，保证先到先得
- 必须设置`maxWaitMillis`，防止线程无限阻塞
- 使用try-with-resources确保连接一定被归还

## 场景 C：动态调整min/max

### 现象

```
凌晨低峰期连接数过多，浪费资源
白天高峰期连接数不够，请求排队
手动调参滞后，无法自适应流量变化
```

### 根因

```
min/max固定值，不随负载变化动态调整
缺少运行时监控数据支撑调参决策
无自动化扩缩容机制
```

### 解决方案

```java
/**
 * 动态调整连接池大小的监控器
 * 根据活跃连接数比例自动调整min/max
 */
public class PoolDynamicAdjuster {
    
    private final GenericObjectPool<?> pool;
    private final ScheduledExecutorService scheduler;
    
    public PoolDynamicAdjuster(GenericObjectPool<?> pool) {
        this.pool = pool;
        this.scheduler = Executors.newScheduledThreadPool(1);
        startDynamicAdjust();
    }
    
    private void startDynamicAdjust() {
        // 每60秒检查一次连接池状态并动态调整
        scheduler.scheduleAtFixedRate(() -> {
            
            int active = pool.getNumActive();   // 当前活跃连接数
            int idle = pool.getNumIdle();       // 当前空闲连接数
            int max = pool.getMaxTotal();        // 当前最大连接数
            
            double usageRatio = active * 1.0 / max;  // 使用率
            
            log.info("连接池状态：active={}, idle={}, max={}, ratio={}", 
                     active, idle, max, String.format("%.2f%%", usageRatio * 100));
            
            // 策略1：使用率>80%，扩大max
            if (usageRatio > 0.8 && max < 200) {
                int newMax = Math.min(max + 20, 200);
                pool.setMaxTotal(newMax);
                log.info("连接池扩容：max={} -> {}", max, newMax);
            }
            
            // 策略2：使用率<30%，缩小max
            else if (usageRatio < 0.3 && max > 20) {
                int newMax = Math.max(max - 10, 20);
                pool.setMaxTotal(newMax);
                log.info("连接池缩容：max={} -> {}", max, newMax);
            }
            
            // 策略3：动态调整minIdle（保持合理预热）
            pool.setMinIdle(Math.max(active / 4, 5));
            
        }, 60, 60, TimeUnit.SECONDS);
    }
}
```

**动态调整关键**：
- 监控`active/max`比值，高于80%扩容，低于30%缩容
- 通过JMX在运行时修改`setMaxTotal`/`setMinIdle`
- 避免频繁调整（设冷却期），防止抖动

## 场景 D：泄漏检测与JMX监控

### 现象

```
连接池中的连接越来越少，最终全部耗尽
代码忘记调用returnObject/close()
生产环境无法查看连接池状态，排查困难
```

### 根因

```
无泄漏检测机制，忘记归还的连接无法回收
无运行时状态暴露，问题发现晚
无告警规则，连接池耗尽后才报警
```

### 解决方案

```java
// 1. 泄漏检测配置
GenericObjectPoolConfig config = new GenericObjectPoolConfig<>();

// 启用泄漏追踪（记录连接借出位置）
config.setAbandonedUsageTracking(true);

// 连接借出超过10分钟未归还，视为泄漏并强制回收
config.setRemoveAbandonedTimeout(600);  // 单位：秒
config.setRemoveAbandonedOnBorrow(true);  // 借出时触发检查

// 2. JMX注册（让连接池状态可通过jconsole/jvisualvm查看）
pool = new GenericObjectPool<>(factory, config);
ManagementFactory.getPlatformMBeanServer().registerMBean(
    new StandardMBean(pool, GenericObjectPoolMBean.class),
    new ObjectName("com.example:type=ConnectionPool,name=myPool")
);

// 3. 告警规则（当空闲连接为0且活跃接近上限时告警）
if (pool.getNumIdle() == 0 && pool.getNumActive() >= pool.getMaxTotal() * 0.9) {
    alertManager.sendAlert("连接池即将耗尽！active=" + pool.getNumActive());
}
```

**泄漏检测关键**：
- 启用`removeAbandonedOnBorrow`+`abandonedUsageTracking`，自动回收泄漏连接
- 通过JMX暴露连接池指标（active/idle/waitCount）
- 设置告警阈值，空闲连接为0且活跃率>90%时立即告警

## 排查 Checklist

```
□ 有连接复用吗？ → 用GenericObjectPool替代每次新建
□ 公平性配置了吗？ → lifo=false避免饥饿
□ 有超时限制吗？ → maxWaitMillis≤5s，防止无限等待
□ 支持动态调整吗？ → 根据usageRatio自动扩缩容
□ 开启泄漏检测了吗？ → removeAbandonedTimeout=600s
□ JMX监控接入了吗？ → jconsole可查看连接池实时状态
□ 归还连接有保障吗？ → try-with-resources/finally确保归还
□ 连接校验开启了吗？ → testOnBorrow=true防止借到坏连接
```
