# 分布式基础理论

## 这个问题为什么存在？

> 单体应用所有代码跑在一个进程里，调用是方法调用，数据访问是本机内存。但业务增长后，**性能瓶颈、单点故障、维护困难**逼着你把系统拆成多个服务部署在不同机器上。问题是：**分布式环境和单机环境本质上有何不同？这些差异带来了什么新挑战？**

不理解这些差异，你写的"分布式代码"只是把单机的坑换了个分布式的放大版。

---

## 它是怎么解决问题的？

### 分布式系统的四大挑战

#### 1. 网络不可靠

```
单机调用：方法调用 → 结果立即返回（基本不会失败）
分布式调用：A → 网络 → B → 结果返回（网络可能丢包、超时、乱序）
```

**后果**：你永远不知道对方是"慢"还是"挂了"。

#### 2. 节点故障是常态

```
单机：进程挂了 → 整个系统挂了（单点故障）
分布式：节点A挂了 → 其他节点应该接管（高可用设计）
```

**后果**：你必须假设任何节点随时会挂，设计上要容忍故障。

#### 3. 时钟不同步

```
机器A时间：2024-01-01 10:00:00.000
机器B时间：2024-01-01 10:00:00.100（慢了100ms）
```

**后果**：不能用本地时间戳做全局事件排序，需要**逻辑时钟**（如Lamport时钟、向量时钟）。

#### 4. 网络分区（P）

```
机房A ──✂️── 机房B（光纤被挖断）
      网络分区
```

**后果**：机房A和机房B无法通信，但各自都认为对方挂了。这就是CAP中的P。

### 分布式系统的设计原则

```
原则1：系统设计要容忍故障（Design for failure）
         → 任何服务都可能挂，要有熔断、降级、重试

原则2：避免单点故障（Eliminate single point of failure）
         → 关键组件必须冗余（如ZooKeeper集群、Redis主从）

原则3：用幂等性应对重试（Idempotency）
         → 网络超时后重试，不能导致重复扣款

原则4：最终一致性可接受（Eventual consistency）
         → 不强求实时一致，但保证经过一段时间后数据一致
```

## 它和相似方案的本质区别是什么？

### 单体架构 vs 分布式架构

| | 单体架构 | 分布式架构 |
|---|---|---|
| 调用方式 | 方法调用（快，基本不失败） | RPC/HTTP（慢，可能失败） |
| 数据一致性 | ACID事务（强一致） | 分布式事务（最终一致） |
| 部署 | 一次部署所有功能 | 每个服务独立部署 |
| 扩展性 | 垂直扩展（加内存、CPU） | 水平扩展（加机器） |
| 维护成本 | 低（一个代码库） | 高（服务拆分、服务治理） |

**本质区别**：单体架构是**集中式思维**，分布式架构是**分散式思维**。分布式用**复杂度**换**可扩展性**。

### 分布式系统 vs 集群

| | 分布式系统 | 集群 |
|---|---|---|
| 定义 | 多个不同服务协同工作 | 多个相同服务冗余部署 |
| 目标 | 业务拆分，各司其职 | 高可用，负载均衡 |
| 示例 | 订单服务 + 库存服务 | 3个Tomcat实例负载均衡 |

**本质区别**：分布式系统强调**功能拆分**，集群强调**冗余高可用**。实际系统中两者并存。

## 正确使用方式

### 1. 调用远程服务必须设超时

```java
// ❌ 错误：没有超时，线程一直阻塞
String result = restTemplate.getForObject(url, String.class);

// ✅ 正确：设超时，避免雪崩
RestTemplate restTemplate = new RestTemplate();
HttpComponentsClientHttpRequestFactory factory =
        new HttpComponentsClientHttpRequestFactory();
factory.setConnectTimeout(3000);  // 连接超时3秒
factory.setReadTimeout(5000);     // 读取超时5秒
restTemplate.setRequestFactory(factory);
```

### 2. 远程调用必须重试，但要有策略

```java
// ✅ 正确：指数退避重试
public class RetryTemplate {
    public <T> T execute(Callable<T> task) throws Exception {
        int maxRetries = 3;
        long backoff = 1000;  // 初始退避1秒

        for (int i = 0; i < maxRetries; i++) {
            try {
                return task.call();
            } catch (Exception e) {
                if (i == maxRetries - 1) throw e;
                Thread.sleep(backoff);
                backoff *= 2;  // 指数退避：1s → 2s → 4s
            }
        }
        return null;
    }
}
```

### 3. 分布式锁用Redis或ZooKeeper

```java
// ✅ 正确：用Redis的SETNX实现分布式锁
public boolean tryLock(String key, String value, int expireSeconds) {
    // SET key value NX EX expireSeconds
    String result = jedis.set(key, value, "NX", "EX", expireSeconds);
    return "OK".equals(result);
}
```

## 边界情况和坑

### 坑1：忽略网络分区

```
场景：机房A和机房B网络中断，但各自都运行正常
错误：两个机房都认为自己是最新数据，继续写入
后果：脑裂，数据永久不一致
```

**解决方案**：
1. **奇数节点 + 多数派确认**（如ZooKeeper，3个节点允许1个挂）
2. **租约机制**（Lease），过期后自动释放锁

### 坑2：分布式系统用本地锁

```
场景：单机锁（synchronized）在分布式环境下无效
后果：两个机器同时修改同一资源，数据冲突
```

**解决方案**：用分布式锁（Redis、ZooKeeper）替代本地锁。

### 坑3：时间戳做全局排序

```
场景：用System.currentTimeMillis()做全局事件排序
后果：机器时钟不同步，事件顺序错乱
```

**解决方案**：用**逻辑时钟**（Lamport时钟、向量时钟）或**全局递增ID**（雪花算法）。
