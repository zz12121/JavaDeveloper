# 集合框架

> 集合框架不是「API 速查表」，是 Java 设计者解决「如何组织和高效访问数据」这个问题的系统性答案。理解设计动机，比背下 API 重要一百倍。

---

## 这个问题为什么存在？

没有集合框架之前，你需要什么？

- 需要「可变长序列」？自己写一个动态数组，手动扩容。
- 需要「键值对映射」？自己写一个哈希表，处理冲突。
- 需要「不重复集合」？自己写一个数组，每次插入前遍历检查。

**问题不是「能不能写」，而是「每个人都在重复造同一个轮子」**——而且大多数人造出来的轮子有 bug。

Java 集合框架（JCF）解决的是两个根本问题：

1. **数据结构的抽象**：统一接口，让代码不依赖具体实现。换 LinkedList 改 ArrayList，调用方不用动。
2. **通用算法的复用**：`Collections.sort()`、`Collections.shuffle()` 对任何 `List` 都有效，不需要为每种数据结构重写排序。

---

## 它是怎么解决问题的？

### 核心接口层级

```
Iterable          — 可遍历（iterator()）
  └── Collection  — 单值集合
        ├── List  — 有序、可重复
        ├── Set   — 无序、去重
        └── Queue — 队列操作
  └── Map          — 键值对（注意：Map 不继承 Collection）
```

这是一个**设计与职责分离**的经典案例：
- `List/Set/Queue` 管「值」
- `Map` 管「键-值关联」
- `Iterator` 管「遍历行为`

**为什么 Map 不继承 Collection？** 因为 Map 的核心操作（put/get）跟 Collection（add/remove）语义完全不同，硬塞在一起会让两个接口都变得别扭。这是设计上的诚实——不愿意为了「看起来统一」而牺牲语义清晰。

### List：顺序容器

**ArrayList**：用数组实现，get O(1)，插入/删除 O(n)。

```
容量 vs 长度：
- length：当前元素个数
- capacity：底层数组大小（自动扩容：1.5倍）

为什么是 1.5 倍？不是 2 倍？
2 倍扩容：浪费内存（长期运行后平均 50% 空闲空间）
1.5 倍：内存利用率更高，但扩容更频繁（时间换空间）
```

**LinkedList**：双向链表，插入/删除 O(1)，但 get O(n)。**现实场景**：几乎用不到 LinkedList。

```
典型误区：把 LinkedList 当作「频繁插入删除场景」的首选。
实际情况：LinkedList 的节点是离散分布的，CPU 缓存命中极差。
大多数「频繁插入删除」实际上是在操作末尾追加，ArrayList 反而更快。
真实需要 LinkedList 的场景：LRU 缓存（双向链表天然支持 O(1) 移动节点）。
```

### Set：去重容器

| 实现 | 底层 | 特点 |
|------|------|------|
| HashSet | HashMap（值作为 key，固定 PRESENT 对象作为 value） | O(1) 查找/插入，无序 |
| LinkedHashSet | LinkedHashMap | 保持插入顺序 |
| TreeSet | 红黑树 | 自动排序，O(log n) |

**HashSet 底层就是 HashMap**，这个实现细节揭示了一个重要原则：组合优于继承。HashSet 不是「继承 HashMap」，而是「持有 HashMap 实例」，自己只暴露 Set 的接口。

```
// HashSet 源码本质
private transient HashMap<E, Object> map;
private static final Object PRESENT = new Object();

public boolean add(E e) {
    return map.put(e, PRESENT) == null;  // key 是元素，value 是固定对象
}
```

### Map：键值对

Map 是集合框架里最深的部分，也是面试最常问的。我们重点说 HashMap。

## HashMap 深度解析

### 为什么需要哈希表？

数组的痛点：查找要 O(n)（二分要 O(log n) 但要求有序）。
哈希表的目标：**把「关键字→存储位置」的映射直接算出来**，查找变 O(1)。

### 哈希函数

HashMap 的哈希计算有两步：

```java
// JDK 8 源码
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}

// put 时计算槽位
int index = (n - 1) & hash
```

**为什么要 `h ^ (h >>> 16)`？**

```
key.hashCode() 只低 16 位有值（大多数对象的内存地址相关性集中在低位）
直接用 hashCode() & (n-1) 的话，高位信息全丢了
>>> 16 后异或：把高位信息混合到低位，让 hash 分布更均匀
```

这个技巧叫做 **「扰乱函数」（perturbation）**，本质是用空间换均匀性。

### 底层数据结构演变

```
JDK 7：数组 + 链表（Entry[])
JDK 8+：数组 + 链表 + 红黑树（Node → TreeNode）
```

**链表 → 红黑树的阈值是 8**，不是随便选的：

```
泊松分布计算：链表长度超过 8 的概率 ≈ 0.0000006
即：99.9999% 的链表长度 ≤ 8
选 8 是为了让红黑树在真正需要的极端场景才触发
```

```
JDK 8 红黑树的条件：
- 链表长度 > 8 → 树化
- 链表长度 < 6  → 退化回链表（避免在临界点反复树化/退化）
```

### put 流程

```
put(key, value)
  ├── 计算 hash(key)
  ├── 计算 index = (n-1) & hash
  ├── 遍历该桶的链表/红黑树
  │     ├── 找到相同 key → 覆盖 value
  │     └── 没找到 → 插入尾部（链表）/ 插入红黑树
  └── size++ → 检查是否超过 threshold → 触发 resize()
```

### resize 机制

触发条件：`size > threshold`（threshold = capacity × loadFactor）

```
loadFactor = 0.75（平衡点：时间与空间的 trade-off）
- 0.5：空间利用率低，但扩容不频繁（碰撞少）
- 0.75：HashMap 默认值，综合最优
- 1.0：空间利用率高，但碰撞多，性能下降
```

**resize 做了两件事**：
1. 容量翻倍（16 → 32 → 64...）
2. **重新计算每个元素的 index**（因为 n-1 变了）

```
JDK 7 扩容：rehash，所有元素重新算位置（重新 hash 一遍）
JDK 8 扩容：不需要重新 hash，直接看 hash 的第 (log2(n)) 位是 0 还是 1
          为 0 → 原位置
          为 1 → 原位置 + 旧容量

这个优化叫「二倍扩容位判断」，避免了重复计算 hash
```

### 并发问题（ JDK 7 的死循环）

JDK 7 的 HashMap 在并发 resize 时会产生**链表环形引用**，导致 get() 死循环。

```
原因：JDK 7 扩容时是**头插法**，并发环境下会反转链表顺序。
多个线程同时 resize：线程 A 和 B 都在操作同一个桶的两条链表，
头插法可能导致 A 的 next 指向 B，B 的 next 指向 A → 环形链表。
下次 get 这个桶时，遍历永不停止 → CPU 100%。
```

JDK 8 用**尾插法**解决了这个问题，但 HashMap 本身仍然**不是线程安全的**。并发场景必须用 `ConcurrentHashMap`。

## 并发容器

### ConcurrentHashMap 的演进

```
JDK 7：Segment[] + HashEntry，分段锁（默认 16 段）
        锁的粒度：一个 Segment（一段 bucket）
        问题：段数固定，并发度上限是 16

JDK 8：去掉 Segment，用 CAS + synchronized
        锁的粒度：一个 bucket
        并发度提升到数组长度（可认为无上限）
```

**JDK 8+ 的 put 流程**：

```java
// 简化的核心逻辑
1. 计算 hash，定位到 bucket
2. if (该 bucket 为空) {
       CAS 插入，失败则重试
   } else {
       synchronized (该 bucket) {
           // 链表或红黑树插入（单线程，安全）
       }
   }
```

**为什么 bucket 级别的 synchronized 不是性能瓶颈？**

因为大多数情况下，hash 足够均匀，不同线程操作的 bucket 很少冲突。真正冲突时，synchronized 的时间极短（只是链表操作），不会长时间阻塞。

### 常见并发容器对比

| 容器 | 底层 | 适用场景 |
|------|------|---------|
| ConcurrentHashMap | CAS + synchronized | 高并发键值存储 |
| CopyOnWriteArrayList | 写时复制 | 读多写少（迭代不阻塞） |
| ConcurrentLinkedQueue | 无界链表 CAS | 高并发队列 |
| BlockingQueue | ReentrantLock + Condition | 生产者-消费者 |
| ConcurrentSkipListMap | 跳表 | 需要有序的高并发 Map |

## 正确使用方式

### 选型决策树

```
需要键值映射？
├── 线程安全？
│   ├── 高并发 → ConcurrentHashMap
│   └── 只需要单一写锁 → Collections.synchronizedMap()
├── 需要保持插入顺序？ → LinkedHashMap
├── 需要按自然顺序遍历？ → TreeMap
└── 普通场景 → HashMap（默认选这个）

需要 List？
├── 高随机访问 → ArrayList（默认选这个）
├── 需要队列/栈 → ArrayDeque
└── 极少数真正的链表场景 → LinkedList

需要 Set？
├── 高并发 → ConcurrentHashMap.newKeySet()
├── 需要有序 → TreeSet
├── 需要保持插入顺序 → LinkedHashSet
└── 普通场景 → HashSet（默认选这个）
```

### ArrayList vs LinkedList 的真实选择

**99% 的场景用 ArrayList**。LinkedList 只在以下场景合理：

- 实现 LRU 缓存（双向链表 O(1) 移动节点）
- 实现数据结构本身（作为其他高级数据结构如链表、栈、队列的构件）
- 在已知是**在任意位置频繁插入删除**且数据量极大的场景（但实际测试往往 ArrayList 更快）

## 边界情况和坑

### 1. hashCode() 和 equals() 的坑

**自定义对象做 key 必须同时重写 hashCode() 和 equals()**：

```java
class Person {
    String name;
    int age;

    // 只重写 equals，没重写 hashCode → HashMap 中的「幽灵」
    @Override
    public boolean equals(Object o) {
        return o instanceof Person && ((Person) o).name.equals(name) && ((Person) o).age == age;
    }
    // 必须加上：
    @Override
    public int hashCode() {
        return Objects.hash(name, age);  // 保证相同对象一定生成相同 hash
    }
}
```

如果只重写 equals，不重写 hashCode：相同内容的两个对象 equals 相等，但 hashCode 不同——HashMap 会认为它们是不同的 key，**放在不同的桶里**，查找时找不到。

### 2. fail-fast 机制

当迭代过程中检测到集合被结构性修改（非迭代器自身的 remove/add），立即抛出 `ConcurrentModificationException`。

```java
// 常见错误写法
List<String> list = new ArrayList<>();
list.add("A"); list.add("B");
for (String s : list) {       // 底层是 Iterator
    if ("A".equals(s)) {
        list.remove(s);        // ConcurrentModificationException!
    }
}

// 正确写法
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if ("A".equals(it.next())) {
        it.remove();           // 用迭代器自身的 remove，不抛异常
    }
}
```

**为什么 fail-fast 是合理的？**

这本质上是一个**检测而非预防**机制。它不是为了「在并发下安全使用」，而是检测到程序员的 bug（用集合自己的引用修改了集合，同时在迭代它）。这种 bug 如果静默忽略，会导致极其难追踪的数据不一致问题。

### 3. 空值问题

```
HashMap：允许一个 null key
HashSet：允许一个 null 元素
ConcurrentHashMap：不允许 null key（因为 CAS 无法区分「key 不存在」和「value 为 null」）
```

这是一个著名的设计决策。ConcurrentHashMap 作者 Doug Lea 的解释：允许 null 会让 API 在并发场景下产生歧义（返回 null 到底是没有这个 key，还是 value 本身是 null？），为了消除歧义，索性禁止。

### 4. 扩容因子不是越高越好

很多人觉得 0.75 太保守，调到 0.9 能省内存。但：

```
loadFactor = 0.9 的问题：
- 碰撞概率急剧上升
- 链表变长，查找从 O(1) 退化为 O(n)
- 在极端 hashCode 实现下（所有 key 都落在同一桶），性能退化到 O(n)
- 扩容频率降低节省的时间，被碰撞查找浪费的时间抵消，甚至更差
```

### 5. HashMap 内存开销

```
HashMap.Entry = 16 bytes × 2 (key+value) + 8 (next) + 4 (hash) + 12 (对象头+填充) ≈ 40 bytes/条
vs Redis 的简单字符串 KV ≈ 100+ bytes/条

结论：HashMap 适合「内存受限但数据量可控」的场景。
数据量 > 1000 万条时，考虑分库分表或换成专门的大数据存储。
```

## 我的理解

集合框架的核心设计哲学就两句话：

1. **接口抽象行为，实现交付性能**。代码依赖接口（List/Map），运行时分派到具体实现（ArrayList/HashMap）。
2. **没有银弹，只有 trade-off**。ArrayList 快但占连续内存，LinkedList 灵活但缓存不友好，ConcurrentHashMap 高并发但比 HashMap 重。每一种选择都有代价。

面试时聊集合框架，最高段位不是说「HashMap 用数组加链表加红黑树」——这是初中级水平。**最高段位是说清楚「什么时候该选哪个」「为什么这样选」「不这样选会付出什么代价」**。
