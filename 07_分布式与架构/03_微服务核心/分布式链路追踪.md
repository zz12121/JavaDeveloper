# 分布式链路追踪

## 这个问题为什么存在？

> 单体应用，一个请求在一个进程里，打日志就能排查问题。但微服务架构，**一个请求横跨10个服务**，问题是：**怎么把整个调用链串起来，快速定位故障？**

没有链路追踪，你会遇到：
- 用户投诉"订单创建慢"，你不知道是订单服务慢，还是支付服务慢
- 一个请求经过10个服务，每个服务都有日志，但串不起来
- 排查问题要靠"猜"，一个个服务查日志

---

## 它是怎么解决问题的？

### 核心概念：TraceId + SpanId

```
一个请求（Trace）：
  TraceId = abc123

调用链：
  ┌── 订单服务（SpanId=1, parentSpanId=0）────────────┐
  │  ┌── 库存服务（SpanId=1.1, parentSpanId=1）────┐ │
  │  │  ┌── 数据库（SpanId=1.1.1, parent=1.1）──┐│ │
  │  │  └──────────────────────────────────────────┘│ │
  │  └───────────────────────────────────────────────┘ │
  │  ┌── 支付服务（SpanId=1.2, parentSpanId=1）────┐ │
  │  │  ┌── 第三方支付（SpanId=1.2.1, parent=1.2）┐│ │
  │  │  └──────────────────────────────────────────┘│ │
  │  └───────────────────────────────────────────────┘ │
  └──────────────────────────────────────────────────────┘
```

**关键ID**：
- **TraceId**：整个调用链的唯一标识（一个请求一个TraceId）
- **SpanId**：每个服务调用的唯一标识（订单是1，库存是1.1）
- **parentSpanId**：父SpanId（库存的parent是订单）

### 传播方式：通过请求头传递

```
订单服务 → 调用库存服务：
  Http Header:
    X-Trace-Id: abc123
    X-Span-Id: 1.1
    X-Parent-Span-Id: 1
```

```java
// Spring Cloud Sleuth 自动传播（无需手写）
// 1. 订单服务接收请求（TraceId不存在，新建一个）
// 2. 订单服务调用库存服务，Sleuth自动把TraceId、SpanId放入Http Header
// 3. 库存服务接收请求，从Http Header提取TraceId、SpanId，继续传递
```

## 它和相似方案的本质区别是什么？

| | SkyWalking | Zipkin | Jaeger |
|---|---|---|---|
| 性能 | 高（异步上报，gRPC） | 中（HTTP上报） | 高（gRPC/UDP） |
| UI | 丰富（拓扑图、性能指标） | 简单（调用链列表） | 中（调用链+依赖图） |
| 存储 | ES、MySQL、H2 | 内存、ES、MySQL | Cassandra、ES |
| 侵入性 | 低（Java Agent，无代码侵入） | 中（需引入Sleuth） | 中（需引入SDK） |
| 适用场景 | 生产推荐（性能好，UI强） | 简单场景、快速集成 | K8s环境、CNCF生态 |

**本质区别**：
- **SkyWalking**：**无侵入**（Java Agent），性能好，UI强，生产首选
- **Zipkin**：**早期标准**，但性能差（HTTP上报），UI简单
- **Jaeger**：**CNCF生态**，适合K8s环境

## 正确使用方式

### 方式1：SkyWalking（推荐，无侵入）

```
步骤1：下载SkyWalking APM，启动（默认端口11800 gRPC，12800 HTTP）
步骤2：应用启动时加Java Agent
```

```bash
# 应用启动命令（加SkyWalking Agent）
java -javaagent:/path/to/skywalking-agent.jar \
     -DSW.AGENT.SERVICE_NAME=order-service \
     -DSW.AGENT.COLLECTOR.BACKEND_SERVICE=localhost:11800 \
     -jar order-service.jar
```

**SkyWalking的优点**：
1. **无代码侵入**（Java Agent，不修改一行代码）
2. **性能高**（异步gRPC上报，不影响业务）
3. **UI强大**（调用链、拓扑图、性能指标、告警）

### 方式2：Spring Cloud Sleuth + Zipkin（有代码侵入）

```xml
<!-- pom.xml -->
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-sleuth</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-sleuth-zipkin</artifactId>
</dependency>
```

```yaml
# application.yml
spring:
  zipkin:
    base-url: http://localhost:9411
  sleuth:
    sampler:
      probability: 1.0  # 采样率100%（生产建议0.1，10%采样）
```

## 边界情况和坑

### 坑1：采样率100%，性能炸了

```
场景：生产环境，采样率100%
后果：每个请求都上报，网络IO + 存储压力巨大
```

**解决方案**：
1. **采样率调低**（生产环境10%采样足够）
   ```yaml
   spring:
     sleuth:
       sampler:
         probability: 0.1  # 10%采样
   ```
2. **只对慢请求采样**（SkyWalking支持）

### 坑2：线程池、异步调用，TraceId丢失

```
场景：主线程有TraceId，但提交到线程池后，子线程没有TraceId
后果：异步调用的链路断了
```

**解决方案**：
1. **用MDC传递**（Spring Cloud Sleuth自动支持，但自定义线程池要手动传递）
   ```java
   // ✅ 正确：用Sleuth的LazyTraceExecutor，自动传递TraceId
   @Bean
   public Executor taskExecutor() {
       return new LazyTraceExecutor(beanFactory);
   }
   ```
2. **SkyWalking Agent自动支持**（无需额外配置）

### 坑3：日志里没有TraceId，排查困难

```
场景：链路追踪系统能看到调用链，但业务日志里没有TraceId
后果：看到慢请求，但无法关联到业务日志
```

**解决方案**：
1. **日志模板里加TraceId**（Logback/Log4j2）
   ```xml
   <!-- logback.xml -->
   <pattern>%d{yyyy-MM-dd HH:mm:ss} [%thread] %-5level %logger{36} - TraceId=%X{traceId} - %msg%n</pattern>
   ```
2. **MDC自动填充**（Sleuth/SkyWalking都会自动填充MDC）
