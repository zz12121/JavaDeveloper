# 字符串

## 这个问题为什么存在？

> 从「没有这个东西会怎样」出发，解释问题的根源。

Java中的字符串处理看似简单，但`String`类却是Java中使用最频繁的类，其设计背后蕴含着对**性能**、**安全**和**不可变性**的深度考量。

**为什么字符串需要不可变？**
1. **字符串常量池的实现依赖不可变性**：JDK7之前字符串常量池在PermGen（永久代），JDK7移到了Heap区。相同内容的字符串可以共享同一块内存，减少内存占用。
2. **安全性**：数据库连接URL、文件路径、网络连接参数等都是字符串，如果可变可能导致安全问题。
3. **线程安全**：不可变对象天然线程安全，无需同步开销。
4. **hashCode缓存**：字符串的hashCode只计算一次，可被缓存和复用。

**为什么需要String、StringBuilder、StringBuffer三种字符串类型？**
- `String`：不可变，每次操作都会生成新对象，适合不需要频繁修改的场景
- `StringBuilder`：可变，非线程安全，性能最优，适合单线程下的频繁字符串操作
- `StringBuffer`：可变，线程安全，但有synchronized开销，适合多线程共享场景

## 它是怎么解决问题的？

> 不止讲「是什么」，要讲清楚「为什么这样设计能解决问题」。

### 核心机制

#### String的底层实现

```java
public final class String
    implements java.io.Serializable, Comparable<String>, CharSequence {
    
    // JDK9之前用char数组存储字符
    private final char value[];
    
    // JDK9+用byte数组存储，更节省内存
    private final byte[] value;
    private final coder;  // 编码器标识：LATIN1或UTF16
}
```

**设计演进**：JDK9将`char[]`改为`byte[]`+`coder`，因为大多数字符串只包含Latin-1字符（只需1字节），使用`char[]`会浪费一半内存。

#### 字符串常量池

```java
// 字符串常量池是HotSpot VM中的一个特殊区域
// 字面量创建时，会先在常量池中查找是否有相同内容的字符串
String s1 = "hello";  // 直接从常量池获取
String s2 = "hello";  // s1 == s2 为true，共享同一引用

// new String() 每次都会在堆中创建新对象
String s3 = new String("hello");  // s3 != s1
```

**为什么这样设计？**
- 字符串是程序中最常见的对象，减少重复字符串可以显著节省内存
- 常量池中的字符串永不GC，即使没有任何引用也会保留（设计缺陷，JDK7才移出PermGen解决）

#### ASCII图：字符串创建流程

```
字面量创建："hello"

│ 1. 检查常量池是否有"hello"              │

│    └────┬────┘                          │

└─────────────────────────────────────────┘
```

#### String.intern()方法

```java
String s1 = new String("hello");        // 堆中对象
String s2 = s1.intern();                 // 将字符串放入常量池，返回常量池引用

System.out.println(s1 == s2);  // false：s1是堆对象，s2是常量池对象
System.out.println(s2 == "hello");  // true：都是常量池对象
```

**intern()的坑点**：
- JDK6：常量池在PermGen，会导致OOM（永久代空间有限）
- JDK7+：常量池移到Heap，但大量调用intern()仍可能导致Full GC压力
- JDK11+：StringTable大小可配置（`-XX:StringTableSize`）

### StringBuilder vs StringBuffer

```java
// StringBuilder - 非线程安全，性能最优
StringBuilder sb = new StringBuilder();
sb.append("hello");
sb.append("world");
sb.toString();

// StringBuffer - 线程安全，但有synchronized开销
StringBuffer sf = new StringBuffer();
sf.append("hello");
sf.append("world");
sf.toString();
```

**源码对比**：
```java
// StringBuilder.append()
public StringBuilder append(String str) {
    super.append(str);  // 直接调用AbstractStringBuilder
    return this;
}

// StringBuffer.append()
public synchronized StringBuffer append(String str) {
    super.append(str);
    return this;
}
```

**性能差异**：单线程下StringBuilder比StringBuffer快约30%，因为避免了synchronized的lock/unlock开销。

### 字符串拼接的编译器优化

```java
// 编译前
String s = "hello" + "world" + "java";

// 编译后（javac优化）
String s = "helloworldjava";  // 编译器合并常量
```

```java
// 编译前（变量拼接）
String a = "hello";
String s = a + "world";

// 编译后（使用StringBuilder）
String s = new StringBuilder().append(a).append("world").toString();
```

**为什么编译器不合并变量拼接？**
- 变量值在编译期不确定，无法合并
- 生成StringBuilder代码是JVM的标准优化策略

**性能优化建议**：循环内拼接使用StringBuilder，避免重复创建
```java
// 慢：循环内toString()会产生大量中间String对象
for (String item : list) {
    result = result + item;  // 每次+都创建新StringBuilder
}

// 快：使用一个StringBuilder复用
StringBuilder sb = new StringBuilder();
for (String item : list) {
    sb.append(item);
}
String result = sb.toString();
```

### 常见字符串方法的实现细节

#### substring()的内存泄漏问题

```java
// JDK6及之前：返回的substring共享原字符串的char数组
String s = new String(new char[100000]);  // 占用10万字符
String sub = s.substring(0, 1);  // 只用1个字符

// sub引用了原s的char数组，原10万字符无法被GC
// 导致内存泄漏
```

```java
// JDK7+：创建新char数组，避免内存泄漏
String substring(int beginIndex, int endIndex) {
    // 计算子串长度
    int subLen = endIndex - beginIndex;
    // JDK9+使用Byte[]实现，这里简化
    return new String(value, beginIndex, subLen, coder);  // 创建新数组
}
```

**为什么JDK6选择共享char数组？**
- substring是高频操作，共享避免了大量内存复制
- 但当原字符串很大但只取很小子串时，会导致大对象无法释放

#### split()的正则表达式陷阱

```java
// 陷阱1：末尾空字符串被丢弃
"a,b,c,".split(",");  // ["a", "b", "c"] - 末尾空字符串丢失

// 陷阱2：特殊字符需要转义
"a.b.c".split(".");  // [] - 点号是正则元字符，需要"\\."
"a|b|c".split("|");  // ["a", "|", "b", "|", "c"] - 竖线是正则元字符

// 正确做法
"a.b.c".split("\\.");   // ["a", "b", "c"]
"a|b|c".split("\\|");  // ["a", "b", "c"]

// 保留末尾空字符串
"a,b,c,".split(",", -1);  // ["a", "b", "c", ""]
```

**split()实现原理**：使用正则表达式引擎解析分隔符，性能较差。对于简单分隔符，可用indexOf()替代。

### 编码问题：UTF-8与Unicode

```java
// Java内部使用UTF-16编码（码点范围0x0000-0xFFFF，一个char对应一个代码单元）
// 但UTF-16是变长编码，补充字符（码点>0xFFFF）需要两个char

// 获取真实字符数（考虑补充字符）
int length(String s) {
    return s.codePointCount(0, s.length());
}

// 遍历字符串的正确方式
for (int i = 0; i < s.length(); ) {
    int cp = s.codePointAt(i);
    System.out.println(Character.toChars(cp));
    i += Character.charCount(cp);  // 补充字符跳两个char
}
```

**为什么"你好".length()可能是4？**
```java
"你好".length();  // 在UTF-16视角下是4（两个补充字符各占两个char）
"你好".codePointCount(0, 2);  // 真正的字符数是2
```

## 它和相似方案的本质区别是什么？

> 对比不是列表，要解释「为什么选 A 不选 B」。

### String vs StringBuilder vs StringBuffer

| 维度 | String | StringBuilder | StringBuffer |
|------|--------|--------------|--------------|
| **可变性** | 不可变 | 可变 | 可变 |
| **线程安全** | 安全（不可变） | 不安全 | 安全（synchronized） |
| **性能** | 每次创建新对象 | 复用数组 | 有同步开销 |
| **适用场景** | 字符串常量、配置key | 单线程频繁拼接 | 多线程共享字符串 |

**选择原则**：
- 优先使用String，除非有性能问题
- 单线程拼接用StringBuilder
- 多线程共享状态用StringBuffer（或不用字符串）

### 字符串池化策略

```java
// 字符串池 - 自动优化相同字面量
String a = "hello";
String b = "hello";

// new String() - 绕过池
String c = new String("hello");

// intern() - 手动入池
String d = c.intern();
```

**为什么不用new String()创建字符串池？**
- 毫无意义：创建了一个永远不会被池化的String对象
- 唯一的"作用"是浪费内存
- 唯一合理的场景是：当字符串很长时，避免立即入池（节省PermGen空间，JDK6时代）

### intern()的使用场景

```java
// 场景1：大量重复字符串，可节省内存
// 但JDK7+才可靠，JDK6会OOM
Set<String> s = new HashSet<>();
for (String word : largeWordList) {
    s.add(word.intern());  // 节省内存，但JDK6会OOM
}

// 场景2：比较字符串引用而非内容（性能优化）
// 在JVM中，相同的intern字符串在堆中只有一份
String s1 = getFromDb();  // 返回"status"
String s2 = getFromDb();  // 返回"status"
s1 == s2;  // false
(s1 = s1.intern()) == (s2 = s2.intern());  // true
```

**为什么不推荐手动intern()？**
- 需要谨慎管理StringTable大小
- 大量不同字符串调用intern()会导致StringTable膨胀
- 现代JVM的GC已经优化了小字符串的处理

## 正确使用方式

> 不是 API 手册，要解释「为什么这样用是对的」。

### 字符串判等的正确方式

```java
// 永远用equals()，不用 ==
String a = new String("hello");
String b = "hello";

a == b;        // false：引用比较
a.equals(b);   // true：内容比较

// 性能优化：常量放前面
if ("hello".equals(userInput)) { }  // 避免NPE
if (userInput != null && userInput.equals("hello")) { }  // 兜底写法
```

**为什么常量放前面？**
- `"hello".equals(userInput)` 即使userInput为null也不会NPE
- 但语义上是在问"用户输入是否等于hello"，应该将用户输入作为主体

**实用建议**：用`Objects.equals(a, b)`更简洁，自动处理null

### 字符串拼接的选择

```java
// 1. 少量固定拼接：直接+
String sql = "SELECT * FROM user WHERE id = " + userId;  // 可读性好

// 2. 动态拼接：StringBuilder
StringBuilder sb = new StringBuilder();
for (OrderItem item : items) {
    sb.append(item.getName())
      .append(": ")
      .append(item.getPrice())
      .append("; ");
}

// 3. 格式化拼接：MessageFormat
String msg = MessageFormat.format("{0}共{1}件，总价{2}元", name, count, total);

// 4. 模板替换：Apache Commons Text
String template = "Hello, ${name}!";
String result = StrSubstitutor.replace(template, map);

// 5. 日志拼接：使用占位符
log.info("用户{}下单成功，金额{}", userId, amount);
```

**为什么不用+拼接所有字符串？**
- 循环内+会创建大量中间对象
- 但少量拼接+的可读性最好

### 处理中文字符串的正确方式

```java
// 1. 确保源文件编码为UTF-8
// javac -encoding UTF-8 Main.java

// 2. 字符长度计算（考虑补充字符）
int charCount = str.codePointCount(0, str.length());

// 3. 字节长度计算（用于数据库、IO）
byte[] gbk = str.getBytes("GBK");
byte[] utf8 = str.getBytes("UTF-8");

// 4. 正确遍历字符
IntStream chars = str.codePoints();  // Java9+，返回IntStream

// 5. 截取字符串（考虑中文）
public static String substring(String s, int start, int end) {
    int[] cps = s.codePoints().skip(start).limit(end - start)
                        .toArray();
    return new String(cps, 0, cps.length);
}
```

### 避免字符串常见错误

```java
// 错误1：使用new String()创建字符串
String s = new String("hello");  // 多创建一个对象，无意义

// 错误2：字符串比较用 ==
if (str == "yes") { }  // 可能false

// 错误3：循环内字符串+
String result = "";
for (String s : list) {
    result += s;  // 大量中间对象
}

// 错误4：忽略编码问题
byte[] bytes = str.getBytes();  // 使用平台默认编码，可能乱码
byte[] bytes = str.getBytes("UTF-8");  // 指定编码

// 错误5：split()处理特殊字符
"a.b".split(".");  // 返回空数组！点号是正则表达式
"a.b".split("\\.");  // 正确
```

## 边界情况和坑

> 不是列举，要解释「坑的成因」。

### 坑1：字符串常量的gc不掉

```java
// JDK7之前：常量池在PermGen，即使没有引用也不会GC
String s = new String(new char[100000]).intern();
s = null;  // s引用断开
// 常量池中的100000字符数组永不GC，可能OOM

// JDK7+：常量池移到堆，但大量intern字符串仍会导致Full GC压力
```

**成因**：常量池的设计是"只进不出"，目的是支持类加载器卸载时才清理，但实际效果是很多无用的常量占着内存。

### 坑2：substring内存泄漏（JDK6）

```java
// JDK6中，substring()返回的字符串共享原字符数组
String huge = new String(new char[10_000_000]);  // 10M
String tiny = huge.substring(0, 1);
// tiny仍持有10M字符数组的引用！

// 解决：JDK7+已修复，或手动创建新字符串
String safe = new String(huge.substring(0, 1));
```

### 坑3：new String(byte[])编码问题

```java
byte[] bytes = getFromNetwork();
String s = new String(bytes);  // 使用平台默认编码，可能乱码

// 正确：知道原始编码
String s = new String(bytes, StandardCharsets.UTF_8);
```

### 坑4：String.valueOf() vs toString()

```java
Object obj = null;
String.valueOf(obj);   // "null"字符串
obj.toString();        // NPE!

// 数组的valueOf()和toString()行为不同
int[] arr = {1, 2, 3};
Arrays.toString(arr);  // "[1, 2, 3]"
arr.toString();       // "[I@15db9742" - 内存地址！
```

### 坑5：字符串拼接的性能陷阱

```java
// 看似优化，实际没优化
String s = "";
for (int i = 0; i < 1000; i++) {
    s += "a";  // 每次都创建新的StringBuilder + String
}

// 真正优化：复用StringBuilder
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 1000; i++) {
    sb.append("a");
}
String s = sb.toString();
```

### 坑6：StringTableSize配置不当

```java
// -XX:StringTableSize 设置过小会导致链表过长，查询性能下降
// JDK11默认1009太小，高并发场景建议设置65536
-XX:StringTableSize=65536
```

## 我的理解（可选）

> 用自己的话重新表述一遍，检验是否真正理解。

String是Java中最特殊的类：不可变、final类、字符串常量池、特殊编译器待遇。

**为什么String要设计成不可变？**
- 服务于常量池：相同字符串共享内存
- 服务于安全：文件路径、网络连接不会在不知情的情况下被修改
- 服务于性能：hashCode可缓存

**三种字符串类型的取舍**：
- String：不可变，适合常量；每次修改都创建新对象
- StringBuilder：可变+非线程安全，最优性能
- StringBuffer：可变+线程安全，有synchronized开销

**最容易被忽略的坑**：
- JDK6的substring内存泄漏
- split()的第一个参数是正则表达式
- "中文".length()可能不等于2
- String.valueOf(null)返回"null"字符串

## 面试话术总结

1. **String为什么是不可变的？**
   "String被设计成不可变主要基于三个原因：1）支持字符串常量池，相同内容的字符串可以共享内存，减少内存占用；2）安全性，字符串常用于文件路径、数据库连接等敏感场景，被恶意修改会导致安全问题；3）线程安全，不可变对象天然线程安全，无需同步。另外String的hashCode可以被缓存，提升哈希容器的性能。"

2. **String、StringBuilder、StringBuffer的区别？**
   "String是不可变类，每次拼接或修改都会创建新对象；后两者是可变类。StringBuilder非线程安全但性能最好，适用于单线程场景；StringBuffer线程安全但有synchronized开销，适用于多线程共享字符串状态。实际开发中，优先使用String，频繁拼接用StringBuilder，多线程共享用StringBuffer。"

3. **字符串常量池是什么？有什么作用？**
   "字符串常量池是JVM堆中的一块特殊区域，存储字符串字面量。当创建字符串字面量时，JVM会先在常量池中查找是否存在相同内容的字符串，存在则返回该引用，不存在则创建。通过常量池，相同内容的字符串字面量可以共享同一块内存，减少内存占用。但需要注意JDK6常量池在永久代，大量使用可能导致OOM，JDK7移到了堆区。"

4. **new String("abc")创建了几个对象？**
   "最多创建两个对象：1）字符串字面量'abc'，在类加载时创建在常量池中；2）new关键字在堆中创建的String对象。如果常量池中已有'abc'，则只创建1个堆对象。intern()方法可以将new出来的字符串放入常量池，返回常量池引用。"

5. **JDK6和JDK9对String做了哪些优化？**
   "JDK6时String用char[]存储，一个字符占2字节；JDK9改为byte[]+coder标识，Latin-1字符只占1字节，UTF-16字符才占2字节，内存节省约50%。另外JDK7的substring()不再共享原字符数组，避免了内存泄漏问题。"
