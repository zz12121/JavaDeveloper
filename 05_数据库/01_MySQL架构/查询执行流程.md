# 查询执行流程

## 这个问题为什么存在？

> 你写了一条SQL：`SELECT * FROM users WHERE age > 20 ORDER BY name LIMIT 10;`
> 
> MySQL是怎么把它变成结果的？中间经历了什么？哪里可能慢？

不理解查询执行流程，你就无法回答：
- **为什么这个查询慢？** —— 不知道慢在哪个阶段
- **怎么优化？** —— 不知道该从哪个方向入手
- **Explain的每个字段是什么意思？** —— 不知道它们对应执行流程的哪个阶段

```
一条SQL的一生：
客户端 ──▶ 1.通信 ▶ 2.缓存 ▶ 3.解析 ▶ 4.优化 ▶ 5.执行 ▶ 6.返回结果
               │
               └── 每个阶段都可能成为性能瓶颈
```

---

## 它是怎么解决问题的？

### 完整执行流程（6大阶段）

```
┌─────────────────────────────────────────────────────┐
│               一条SELECT的一生                        │
│                                                     │
│  1. 通信协议 (Protocol)                             │
│     │                                               │
│     ▼                                               │
│  2. 查询缓存 (Query Cache)  [MySQL 8.0已移除]       │
│     │                                               │
│     ▼                                               │
│  3. 解析器 (Parser)                                │
│     ├── 词法分析 (Lexical Analysis)                  │
│     └── 语法分析 (Syntax Analysis)                  │
│     │                                               │
│     ▼                                               │
│  4. 预处理器 (Preprocessor)                         │
│     └── 检查表/列是否存在，展开*                    │
│     │                                               │
│     ▼                                               │
│  5. 优化器 (Optimizer)                              │
│     ├── 选择索引                                     │
│     ├── 决定JOIN顺序                                │
│     └── 重写查询                                     │
│     │                                               │
│     ▼                                               │
│  6. 执行器 (Executor)                               │
│     ├── 调用存储引擎接口                             │
│     ├── 过滤WHERE条件                               │
│     ├── 排序 (ORDER BY)                             │
│     ├── 分组 (GROUP BY)                             │
│     └── 返回结果                                     │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 阶段1：通信协议 (Protocol)

**职责**：负责客户端和MySQL服务器之间的数据传输。

```
MySQL协议（基于TCP）：
1. 握手认证
   - 服务器发送握手包 (协议版本、服务器版本、线程ID、盐值)
   - 客户端响应 (用户名、密码加密、数据库名)
   - 服务器验证，返回认证结果

2. 命令请求
   - 客户端发送命令包 (COM_QUERY、COM_PING等)
   - 例如: COM_QUERY + "SELECT * FROM users"

3. 结果响应
   - 服务器返回结果集 (列定义 + 数据行)
   - 或者返回OK/ERROR包
```

**MySQL支持多种协议**：
```
协议类型         说明                                适用场景
──────────────────────────────────────────────────────────
TCP/IP          标准网络协议，默认3306端口           远程连接
Unix Socket     本地进程间通信，不走网络               Linux本地连接
Named Pipe       Windows进程间通信                     Windows本地连接
Shared Memory    Windows共享内存                       Windows本地连接
```

**半双工通信** (Half-Duplex)：

```
客户端 ──────▶ 服务器   (客户端发送请求，不能同时接收)
                
服务器 ──────▶ 客户端   (服务器返回结果，不能同时接收)

问题：
- 客户端发送请求后，只能等待，不能发其他请求 (同一个连接)
- 服务器开始返回结果后，客户端必须接收完所有结果
```

**为什么MySQL用半双工？**

- 简化设计（不需要处理同时读写）
- 大多数场景是"一问一答"，半双工够用
- 如果需要并发，用连接池（多个连接）

### 阶段2：查询缓存 (Query Cache) —— MySQL 8.0已移除

**为什么移除？**

```
查询缓存的问题：
1. 命中率极低
   - 只要表有任何写入，所有相关缓存全部失效
   - 对于写入频繁的应用，缓存命中率接近0

2. 锁竞争
   - 每次查询都要检查缓存 (需要加锁)
   - 高并发下，锁竞争严重

3. 内存管理复杂
   - 缓存过大 → 淘汰策略开销
   - 缓存过小 → 命中率低
```

**MySQL 5.7及之前**：

```sql
-- 查看查询缓存状态
SHOW VARIABLES LIKE 'query_cache%';

-- query_cache_type: ON/OFF/DEMAND
-- query_cache_size: 缓存大小 (默认0，表示不开启)

-- 强制不走缓存
SELECT SQL_NO_CACHE * FROM users WHERE id = 1;
```

**替代方案**：
- 应用层缓存（Redis、Memcached）
- 只缓存"读多写少"的热点数据

### 阶段3+4：解析器 + 预处理器

#### 3.1 词法分析 (Lexical Analysis)

**职责**：把SQL字符串拆成token。

```
SQL: "SELECT id, name FROM users WHERE age > 20"

Token列表：
1. SELECT    (关键字)
2. id        (列名)
3. ,         (标点符号)
4. name      (列名)
5. FROM      (关键字)
6. users     (表名)
7. WHERE     (关键字)
8. age       (列名)
9. >         (操作符)
10. 20       (常量)
```

**如果词法分析失败** → 报语法错误（一般是拼写错误）。

#### 3.2 语法分析 (Syntax Analysis)

**职责**：根据MySQL的语法规则，生成解析树（Parse Tree）。

```
解析树：
SELECT
├── 列: id
├── 列: name
├── FROM: users
└── WHERE
    └── 条件: age > 20
```

**如果语法分析失败** → 报语法错误（一般是语法不对）。

```sql
-- 错误示例
SELECT * FORM users;  -- FORM 应为 FROM
-- ERROR 1064: You have an error in your SQL syntax

SELECT * FROM;  -- 缺少表名
-- ERROR 1064: You have an error in your SQL syntax
```

#### 3.3 预处理器 (Preprocessor)

**职责**：语义检查（解析器只做语法检查，不做语义检查）。

```
预处理器的检查：
1. 表是否存在？
   - SELECT * FROM uusers;  -- 表名拼错 → 报错

2. 列是否存在？
   - SELECT id, namee FROM users;  -- 列名拼错 → 报错

3. 权限检查
   - 当前用户是否有SELECT权限？

4. 展开通配符 *
   - SELECT * FROM users;
   - 预处理器把 * 展开成所有列名 (id, name, age, ...)
```

**如果预处理器失败** → 报语义错误。

```sql
-- 表不存在
SELECT * FROM nonexistent_table;
-- ERROR 1146: Table 'test.nonexistent_table' doesn't exist

-- 列不存在
SELECT id, nonexistent_col FROM users;
-- ERROR 1054: Unknown column 'nonexistent_col' in 'field list'
```

### 阶段5：优化器 (Optimizer) —— 最复杂的部分

**职责**：基于成本（Cost-Based）选择最优执行计划。

```
优化器的输入：解析树
优化器的输出：执行计划 (EXPLAIN可以看到)

优化器做的事情：
1. 选择索引
   - 用哪个索引？（可能有多个索引可选）
   - 是否用索引？（全表扫描可能更快）

2. 决定JOIN顺序
   - 三表JOIN：A JOIN B JOIN C
   - 先JOIN哪两张表？（影响中间结果集大小）

3. 重写查询
   - 简化条件 (WHERE 1=1 AND age > 20 → WHERE age > 20)
   - 子查询优化 (IN vs EXISTS)

4. 估算成本
   - IO成本：预计要读多少页？
   - CPU成本：预计要比较多少行？
```

#### 5.1 索引选择

```
可能有多个索引可选：
CREATE INDEX idx_age ON users(age);
CREATE INDEX idx_name ON users(name);
CREATE INDEX idx_age_name ON users(age, name);

查询：SELECT * FROM users WHERE age > 20 AND name = 'Tom';

优化器评估：
- idx_age: 范围扫描，可能返回大量行
- idx_name: 等值查询，返回少量行
- idx_age_name: 覆盖索引，且age>20可以走索引

→ 选择 idx_age_name (成本最低)
```

**优化器怎么估算成本？**

```
成本 = IO成本 + CPU成本

IO成本：
- 全表扫描：需要读 N 个页 (Page)
  - 每个页16KB，100万行数据 ≈ 几千个页
  - 如果Buffer Pool不命中 → 几千次磁盘IO

- 索引扫描：需要遍历 M 个索引节点
  - B+树高度一般2-4层 → 2-4次IO就能定位到数据

CPU成本：
- 需要比较多少行？
- 需要做多少次函数调用？
```

**查看优化器的选择**：

```sql
-- 开启优化器跟踪
SET optimizer_trace="enabled=on";

-- 执行你的SQL
SELECT * FROM users WHERE age > 20;

-- 查看优化器的决策过程
SELECT * FROM information_schema.OPTIMIZER_TRACE\G

输出JSON，包含：
- considered_execution_plans: 考虑过的执行计划
- best_access_path: 最终选择的访问路径
- rows_estimated: 估算的行数
- cost_info: 成本信息
```

#### 5.2 JOIN顺序选择

```
三表JOIN示例：
SELECT *
FROM orders o
JOIN customers c ON o.customer_id = c.id
JOIN products p ON o.product_id = p.id
WHERE o.order_date > '2024-01-01';

可能的JOIN顺序：
1. 先JOIN orders和customers → 中间结果集大 → 成本高
2. 先JOIN orders和products → 中间结果集小 → 成本低
3. 利用索引，先过滤order_date → 再JOIN

优化器会评估每种顺序的成本，选择最优的。
```

**STRAIGHT_JOIN** （强制JOIN顺序）：

```sql
-- 如果你知道最优的JOIN顺序，可以强制优化器按照你的顺序执行
SELECT *
FROM orders o
STRAIGHT_JOIN customers c ON o.customer_id = c.id
STRAIGHT_JOIN products p ON o.product_id = p.id
WHERE o.order_date > '2024-01-01';
```

### 阶段6：执行器 (Executor)

**职责**：按照执行计划，调用存储引擎接口，获取/过滤数据。

```
执行器的工作流程：
1. 调用存储引擎接口，获取第一行/第一批数据
2. 判断是否符合WHERE条件
3. 如果符合，加入结果集
4. 继续获取下一行
5. 直到所有数据都处理完
6. 如果有ORDER BY/GROUP BY，进行排序/分组
7. 返回结果给客户端
```

#### 6.1 全表扫描 (Full Table Scan)

```
执行流程：
1. 执行器调用 handler->rnd_init() 初始化全表扫描
2. 循环：
   a. 执行器调用 handler->rnd_next() 获取下一行
   b. 存储引擎返回一行 (从Buffer Pool或磁盘)
   c. 执行器检查WHERE条件
   d. 如果满足，加入结果集
   e. 如果不满足，继续下一行
3. 直到handler->rnd_next() 返回"没有更多行"
4. 执行器调用 handler->rnd_end() 结束扫描
```

#### 6.2 索引扫描 (Index Scan)

```
执行流程：
1. 执行器调用 handler->index_read() 定位到索引的第一条记录
2. 循环：
   a. 存储引擎返回当前索引记录 (包含主键值)
   b. 如果是覆盖索引 → 直接返回
   c. 如果不是覆盖索引 → 回表 (根据主键查聚簇索引)
   d. 执行器检查WHERE条件
   e. 如果满足，加入结果集
   f. 执行器调用 handler->index_next() 获取下一条索引记录
3. 直到不满足条件或没有更多记录
```

#### 6.3 排序 (ORDER BY)

```
MySQL的排序方式：
1. 索引排序 (Using index)
   - 如果ORDER BY的列是索引的一部分，且顺序一致 → 直接按索引顺序读
   - 例如: CREATE INDEX idx_age_name ON users(age, name);
           SELECT * FROM users ORDER BY age, name;  -- 可以用索引排序

2. 文件排序 (Using filesort)
   - 如果不能利用索引排序 → MySQL需要自己排序
   - 排序算法：
     a. 如果排序数据 < sort_buffer_size → 内存排序 (快速排序)
     b. 如果排序数据 > sort_buffer_size → 磁盘临时文件排序 (归并排序)
```

**优化ORDER BY**：

```sql
-- ✗ 慢：文件排序
SELECT * FROM users ORDER BY name;

-- ✓ 快：利用索引排序
CREATE INDEX idx_name ON users(name);
SELECT * FROM users ORDER BY name;  -- Using index
```

#### 6.4 分组 (GROUP BY)

```
分组方式：
1. 松散索引扫描 (Loose Index Scan)
   - 利用索引的有序性，直接跳到每个组的第一个记录
   - 例如: CREATE INDEX idx_age_name ON users(age, name);
           SELECT age, COUNT(*) FROM users GROUP BY age;
           → 每个age值只取第一条记录，不用扫描所有行

2. 紧凑索引扫描 (Tight Index Scan)
   - 按顺序扫描索引，遇到不同的组就输出
   - 比全表扫描快，但不如松散索引扫描

3. 临时表分组
   - 如果无法利用索引 → 创建临时表，把分组字段作为临时表的主键
   - 性能最差
```

## 它和相似方案的本质区别是什么？

### MySQL vs PostgreSQL 的查询优化器

```
                     MySQL优化器                 PostgreSQL优化器
─────────────────────────────────────────────────────────────
优化器类型           基于成本 (CBO)                基于成本 (CBO)
统计信息             索引统计 (cardinality)        详细统计 (直方图、MCV)
JOIN顺序优化         有限 (最多61个表)             强大 (动态规划算法)
子查询优化           较弱                         强大 (子查询提升)
并行查询             不支持 (8.0+)                支持 (9.6+)
适用场景             简单查询，Web应用             复杂查询，分析型应用
```

**为什么PostgreSQL的优化器更强？**

- PostgreSQL的统计信息更详细（直方图、高频值MCV）
- 优化器用动态规划算法，能找到最优JOIN顺序
- 支持并行查询（多个CPU核心同时工作）

**但是**，MySQL的优化器对简单查询足够了，而且更稳定（优化器越复杂，越容易选错索引）。

### MySQL 5.7 vs MySQL 8.0 的执行流程差异

```
                     MySQL 5.7                  MySQL 8.0
─────────────────────────────────────────────────────────────
查询缓存             支持                         已移除
窗口函数            不支持                       支持
CTE (WITH子句)      不支持                       支持
优化器              旧优化器                     重写优化器 (更智能)
直方图              不支持                       支持 (ANALYZE TABLE)
```

**直方图 (Histogram)**：

```sql
-- MySQL 8.0 可以收集列值的分布统计
ANALYZE TABLE users UPDATE HISTOGRAM ON age WITH 10 BUCKETS;

-- 优化器利用直方图，更准确地估算基数
-- 例如: WHERE age > 20
-- 如果有直方图 → 知道age>20的记录占比 → 更准确选择索引
```

## 正确使用方式

### 1. 用EXPLAIN分析查询

```sql
-- 查看执行计划
EXPLAIN SELECT * FROM users WHERE age > 20;

-- 关键字段：
-- id: 查询的序列号
-- select_type: 查询类型 (SIMPLE、PRIMARY、SUBQUERY、DERIVED)
-- table: 访问的表
-- type: 访问类型 (ALL、index、range、ref、eq_ref、const)
-- possible_keys: 可能使用的索引
-- key: 实际使用的索引
-- rows: 估算的扫描行数
-- Extra: 附加信息 (Using index、Using filesort、Using temporary)
```

**访问类型 (type) 从好到坏**：

```
system > const > eq_ref > ref > range > index > ALL

const:     主键或唯一索引等值查询 (最多返回1行)
ref:        非唯一索引等值查询 (可能返回多行)
range:      索引范围扫描 (BETWEEN、IN、>、<)
index:      全索引扫描 (比ALL快，因为索引文件小)
ALL:        全表扫描 (最慢，需要优化)
```

### 2. 避免SELECT *

```sql
-- ✗ 错误：SELECT *
SELECT * FROM users WHERE age > 20;

-- 问题：
-- 1. 返回不必要的列 → 网络传输开销大
-- 2. 无法利用覆盖索引 → 必须回表
-- 3. 如果列被删除或重命名 → 应用层报错

-- ✓ 正确：只查需要的列
SELECT id, name FROM users WHERE age > 20;

-- 如果id和name有联合索引 → 覆盖索引，不需要回表
CREATE INDEX idx_age_name ON users(age, name);
-- 执行：SELECT id, name FROM users WHERE age > 20;
-- Extra: Using index (覆盖索引，性能极佳)
```

### 3. 利用索引排序，避免filesort

```sql
-- ✗ 慢：filesort
CREATE INDEX idx_age ON users(age);
SELECT * FROM users ORDER BY name;  -- 无法利用索引排序

-- ✓ 快：利用索引排序
CREATE INDEX idx_name ON users(name);
SELECT * FROM users ORDER BY name;  -- Using index

-- 复合索引排序
CREATE INDEX idx_age_name ON users(age, name);

-- ✓ 可以利用索引排序 (ORDER BY是索引前缀)
SELECT * FROM users ORDER BY age, name;

-- ✗ 无法利用索引排序 (ORDER BY不是索引前缀)
SELECT * FROM users ORDER BY name;
```

### 4. 分页优化 (LIMIT优化)

```sql
-- ✗ 慢：LIMIT 100000, 10
SELECT * FROM users ORDER BY id LIMIT 100000, 10;
-- 问题：需要先读100010行，然后丢弃前100000行 → 浪费IO

-- ✓ 快：用主键游标
SELECT * FROM users WHERE id > 100000 ORDER BY id LIMIT 10;
-- 直接从id=100001开始读 → 只需要读10行

-- 前提：id是连续的，且查询有ORDER BY id
```

## 边界情况和坑

### 坑1：优化器选错索引

```
现象：
- 明知有更好的索引，但EXPLAIN显示用的不是它
- 或者昨天还用对索引，今天突然用错了

原因：
- 统计信息不准确 (cardinality不准)
- 数据分布变化 (某个值突然变得很常见)
```

**解决**：

```sql
-- 1. 重新收集统计信息
ANALYZE TABLE users;

-- 2. 强制用索引
SELECT * FROM users FORCE INDEX(idx_age) WHERE age > 20;

-- 3. 用优化器提示
SELECT /*+ INDEX(users idx_age) */ * FROM users WHERE age > 20;
```

### 坑2：隐式类型转换导致索引失效

```sql
-- 场景：phone是VARCHAR类型，但查询用数字
CREATE INDEX idx_phone ON users(phone);

-- ✗ 慢：隐式类型转换 (索引失效)
SELECT * FROM users WHERE phone = 13800138000;
-- MySQL会把VARCHAR转成数字 → 对每个记录做转换 → 索引失效

-- ✓ 快：类型匹配
SELECT * FROM users WHERE phone = '13800138000';
-- 或者用显式转换
SELECT * FROM users WHERE CAST(phone AS UNSIGNED) = 13800138000;
-- 但是，CAST会导致索引失效 → 还是慢
```

### 坑3：ORDER BY + LIMIT 的"错误"优化

```sql
-- 场景：ORDER BY + LIMIT，优化器可能选择错误的索引
SELECT * FROM users WHERE age > 20 ORDER BY name LIMIT 10;

-- 优化器可能认为：
-- "用idx_name索引，可以避免filesort，虽然扫描行数多，但LIMIT 10很快"

-- 但实际上：
-- idx_name无法过滤age>20 → 需要扫描大量索引记录 → 反而慢

-- 解决：强制用过滤性好的索引
SELECT * FROM users FORCE INDEX(idx_age) WHERE age > 20 ORDER BY name LIMIT 10;
```

### 坑4：临时表导致性能问题

```sql
-- 场景：GROUP BY或ORDER BY的列不是索引
SELECT age, COUNT(*) FROM users GROUP BY age ORDER BY COUNT(*) DESC;

-- 执行过程：
-- 1. 创建临时表 (内存或磁盘)
-- 2. 把分组结果写入临时表
-- 3. 对临时表排序
-- 4. 返回结果

-- 如果临时表太大 → 从内存临时表转成磁盘临时表 → 性能急剧下降

-- 查看是否用了临时表
EXPLAIN SELECT age, COUNT(*) FROM users GROUP BY age;
-- Extra: Using temporary; Using filesort
```

**优化**：

```sql
-- 1. 用索引避免临时表
CREATE INDEX idx_age ON users(age);
```
