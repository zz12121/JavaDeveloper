# 内存溢出与泄漏

## 这个问题为什么存在？

> 从「没有这个东西会怎样」出发，解释问题的根源。

Java虽然有垃圾回收机制，但**内存溢出（OutOfMemoryError）**和**内存泄漏（Memory Leak）**仍然会发生。

**如果没有对这些问题的预防和排查机制**：
1. **内存溢出**：程序突然崩溃，没有任何提示
2. **内存泄漏**：可用内存越来越少，性能逐渐下降，最终OOM
3. **无法排查**：不知道哪里出了问题

**核心问题**：如何**预防**内存问题，以及发生问题后如何**排查**和**修复**？

内存溢出与泄漏的本质是**内存管理不当**：要么申请的内存超过了JVM限制，要么不再使用的对象无法被GC回收。

### 核心机制

#### 1. 内存溢出的几种类型

```
┌─────────────────────────────────────────────────────────────┐
│                  OutOfMemoryError 类型                        │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 1. Java heap space（堆溢出）                        │    │
│  │    - 堆内存不足以分配新对象                          │    │
│  │    - 可能原因：内存泄漏、堆设置过小                    │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 2. Metaspace（元空间溢出）                        │    │
│  │    - 元空间（方法区）内存不足                        │    │
│  │    - 可能原因：加载了太多类（动态代理、CGLIB）         │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 3. Unable to create new native thread               │    │
│  │    - 无法创建新的线程                                │    │
│  │    - 可能原因：线程数超出系统限制                       │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 4. Direct buffer memory（直接内存溢出）              │    │
│  │    - 使用NIO的DirectByteBuffer分配直接内存           │    │
│  │    - 可能原因：分配了太多直接内存                      │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 5. Requested array size exceeds VM limit             │    │
│  │    - 请求的数组大小超过JVM限制（如new int[Integer.MAX_VALUE]）│
│  └─────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

**为什么会有这么多类型？**
- JVM内存分为多个区域，每个区域都可能溢出
- 不同类型需要不同的排查方法

#### 2. 内存泄漏的常见原因

**1. 静态集合持有对象引用**：
```java
public class MemoryLeak {
    private static List<Object> list = new ArrayList<>();
    
    public void add(Object obj) {
        list.add(obj);  // 对象永远无法被GC
    }
}
```

**2. ThreadLocal使用不当**：
```java
public class ThreadLocalLeak {
    private static ThreadLocal<Object> threadLocal = new ThreadLocal<>();
    
    public void process() {
        threadLocal.set(new Object());
        // 忘记调用threadLocal.remove()！
        // 线程池中的线程会复用，导致Object无法被GC
    }
}
```

**3. 未关闭资源**：
```java
public void readFile() throws IOException {
    FileInputStream fis = new FileInputStream("file.txt");
    // 使用fis...
    // 忘记fis.close()！
    // 文件句柄泄漏，最终无法打开新文件
}
```

**4. 监听器未注销**：
```java
public class ListenerLeak {
    public void register() {
        EventListener listener = new EventListener() {
            @Override
            public void onEvent(Event e) {
                // 处理事件
            }
        };
        EventBus.register(listener);
        // 忘记EventBus.unregister(listener)！
    }
}
```

**5. 外部连接未关闭**：
```java
public void query() throws SQLException {
    Connection conn = DriverManager.getConnection(url, user, password);
    // 使用conn...
    // 忘记conn.close()！
    // 数据库连接泄漏，最终无法获取新连接
}
```

**命令行工具**：

**可视化工具**：

2. **jvisualvm**：更强大的监控和分析工具
3. **MAT（Memory Analyzer Tool）**：分析堆转储，找出内存泄漏
4. **JProfiler**：商业工具，功能强大

### 源码关键路径

#### OutOfMemoryError的抛出过程

```java
// hotspot/src/share/vm/memory/collectedHeap.cpp (简化)
oop CollectedHeap::mem_allocate(size_t size) {
    // 1. 尝试在TLAB中分配
    if (UseTLAB) {
        oop result = allocate_from_tlab(size);
        if (result != null) return result;
    }
    
    // 2. 尝试在Eden区分配
    oop result = allocate_from_eden(size);
    if (result != null) return result;
    
    // 3. Eden区满，触发Minor GC
    do_minor_collection();
    result = allocate_from_eden(size);
    if (result != null) return result;
    
    // 4. 如果还失败，尝试在老年代分配
    result = allocate_from_old_gen(size);
    if (result != null) return result;
    
    // 5. 老年代也满，触发Full GC
    do_full_collection();
    result = allocate_from_old_gen(size);
    if (result != null) return result;
    
    // 6. 仍然失败，抛出OutOfMemoryError
    THROW_MSG_0(vmSymbols::java_lang_OutOfMemoryError(),
           "Java heap space");
}
```

#### ThreadLocal内存泄漏的原因

```
┌─────────────────────────────────────────────────────────────┐
│                  ThreadLocal 内存泄漏原理                       │
│                                                             │
│  Thread类中有一个ThreadLocalMap：                        │
│  ThreadLocalMap {                                       │
│      Entry[] table;  // Entry是弱引用                        │
│  }                                                      │
│                                                             │
│  Entry结构：                                             │
│  Entry extends WeakReference<ThreadLocal> {                │
│      Object value;  // 强引用！                            │
│  }                                                      │
│                                                             │
│  泄漏场景：                                               │
│  1. ThreadLocal被回收（弱引用） → key变为null              │
│  2. 但value仍然是强引用，无法被GC                      │
│  3. 如果线程不结束（如线程池），value永远无法被GC      │
│                                                             │
│  解决方案：                                               │
│  使用完ThreadLocal后，调用remove()清理value             │
└─────────────────────────────────────────────────────────────┘
```

## 它和相似方案的本质区别是什么？

> 对比不是列表，要解释「为什么选 A 不选 B」。

### 内存溢出 vs 内存泄漏

| 维度 | 内存溢出（OOM） | 内存泄漏（Memory Leak） |
|------|-------------------|----------------------|
| **定义** | 申请的内存超过JVM限制 | 不再使用的对象无法被GC |
| **表现** | 程序崩溃，抛出OOM | 可用内存越来越少，性能下降 |
| **原因** | 堆设置过小、内存泄漏 | 静态集合、ThreadLocal、未关闭资源 |
| **解决** | 增加-Xmx、修复内存泄漏 | 找到泄漏根源，释放引用 |

**为什么需要区分？**
- 内存溢出不一定是内存泄漏，可能是堆设置过小
- 内存泄漏不一定会立即OOM，可能是长时间运行后才出现问题

### 堆转储分析工具对比

| 工具 | 优点 | 缺点 |
|------|------|------|
| **MAT** | 功能强大，免费 | 界面不够友好 |
| **JProfiler** | 功能最强大，界面友好 | 商业工具，收费 |
| **jvisualvm** | JDK自带，免费 | 功能相对简单 |

**为什么选MAT？**
- **免费**：开源工具
- **功能强大**：可以找出内存泄漏、分析对象占用
- **Eclipse插件**：可以集成到Eclipse中使用

## 正确使用方式

> 不是 API 手册，要解释「为什么这样用是对的」。

### 1. 开启GC日志，便于排查问题

```bash
# JDK 9+ 开启GC日志
-Xlog:gc*:file=gc.log:time,uptime,level,tags

# JDK 8 开启GC日志
-XX:+PrintGC
-XX:+PrintGCDetails
-XX:+PrintGCTimeStamps
-Xloggc:gc.log
```

**为什么这样用？**
- GC日志可以帮助判断是内存溢出还是内存泄漏
- 可以分析GC频率、GC时间，优化JVM参数

### 2. OOM时自动生成堆转储

```bash
# OOM时生成堆转储
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/path/to/dumps/heap.hprof
```

**为什么这样用？**
- OOM时自动生成堆转储，便于事后排查
- 不需要手动执行jmap，避免错过现场

### 3. 使用try-with-resources关闭资源

```java
// ❌ 错误：不关闭资源
public void readFile() throws IOException {
    FileInputStream fis = new FileInputStream("file.txt");
    // 使用fis...
    // 忘记fis.close() → 文件句柄泄漏
}

// ✅ 正确：使用try-with-resources
public void readFile() throws IOException {
    try (FileInputStream fis = new FileInputStream("file.txt")) {
        // 使用fis...
    }  // 自动关闭
}
```

**为什么这样用？**
- 编译器会自动生成finally块，调用close()
- 即使抛出异常，资源也会被正确关闭

### 4. 使用线程池时，注意ThreadLocal清理

```java
// ❌ 错误：使用完ThreadLocal不清理
public class ThreadLocalLeak {
    private static ThreadLocal<Object> threadLocal = new ThreadLocal<>();
    
    public void process() {
        threadLocal.set(new Object());
        // 忘记threadLocal.remove()！
        // 线程池中的线程会复用，导致Object无法被GC
    }
}

// ✅ 正确：在finally中清理ThreadLocal
public void process() {
    try {
        threadLocal.set(new Object());
        // 使用threadLocal
    } finally {
        threadLocal.remove();  // 确保在finally中清理
    }
}
```

**为什么这样用？**
- 线程池中的线程会复用，如果不清理ThreadLocal，value会一直存在
- 在finally中清理，即使抛出异常也能执行

### 5. 定期检查静电集合大小

```java
// ❌ 错误：静电集合无限增长
public class MemoryLeak {
    private static List<Object> list = new ArrayList<>();
    
    public void add(Object obj) {
        list.add(obj);  // 对象永远无法被GC
    }
}

// ✅ 正确：定期清理或使用WeakReference
public class SafeCache {
    private static Map<String, WeakReference<Object>> cache = new HashMap<>();
    
    public void add(String key, Object obj) {
        cache.put(key, new WeakReference<>(obj));
    }
    
    public Object get(String key) {
        WeakReference<Object> ref = cache.get(key);
        return ref == null ? null : ref.get();
    }
}
```

**为什么这样用？**
- WeakReference指向的对象，在GC时会被回收
- 避免静电集合导致内存泄漏

## 边界情况和坑

> 不是列举，要解释「坑的成因」。

### 坑1：使用MAT分析堆转储时，无法找到泄漏根源

```java
// 生成堆转储
jmap -dump:format=b,file=heap.hprof <pid>

// 使用MAT打开heap.hprof，但找不到明显的内存泄漏
```

**成因**：有些内存泄漏是**缓慢增长**的，短时间内看不出来。

**解决方案**：
1. 多次生成堆转储，对比对象增长情况
2. 使用MAT的**Histogram**功能，按类名统计对象数量
3. 使用**Dominator Tree**功能，找到占用内存最大的对象

### 坑2：ThreadLocal内存泄漏在线程池中尤其严重

```java
// 线程池 + ThreadLocal = 内存泄漏高风险
ExecutorService pool = Executors.newFixedThreadPool(10);

pool.submit(() -> {
    ThreadLocal.set(new Object());
    // 忘记ThreadLocal.remove()！
    // 线程池中的线程会复用，导致Object无法被GC
});
```

**成因**：线程池中的线程会复用，如果不清理ThreadLocal，value会一直存在。

**解决方案**：
1. 在finally中调用ThreadLocal.remove()
2. 使用**TransmittableThreadLocal**（阿里开源），支持线程池上下文传递

### 坑3：使用反射时，可能会产生大量代理类，导致元空间溢出

```java
// 使用CGLIB动态生成代理类
Enhancer enhancer = new Enhancer();
enhancer.setSuperclass(Target.class);
enhancer.setUseCache(false);  // 不使用缓存
enhancer.setCallback((MethodInterceptor) (obj, method, args, proxy) -> 
    proxy.invokeSuper(obj, args));
enhancer.create();  // 每次都生成新的代理类
```

**成因**：每次调用create()都会生成新的代理类，元空间会被占满。

**解决方案**：
1. 使用缓存，避免重复生成代理类
2. 增加元空间大小：`-XX:MaxMetaspaceSize=512m`
3. 使用`-XX:+TraceClassLoading`和`-XX:+TraceClassUnloading`监控类加载和卸载

### 坑4：使用NIO时，DirectByteBuffer可能导致直接内存溢出

```java
// 分配直接内存
ByteBuffer buffer = ByteBuffer.allocateDirect(1024 * 1024);  // 1MB

// 如果分配太多，会导致OOM：Direct buffer memory
```

**成因**：直接内存不受Java堆限制，但受操作系统可用内存限制。

**解决方案**：
1. 增加直接内存大小：`-XX:MaxDirectMemorySize=1g`
2. 及时释放直接内存：`((DirectBuffer) buffer).cleaner().clean()`
3. 使用`-XX:+DisableExplicitGC`谨慎，可能会导致DirectByteBuffer无法被回收

### 坑5：使用第三方库时，可能会发生内存泄漏

```java
// 使用第三方库，但不知道内部实现
ThirdPartyLib.process(data);

// 如果ThirdPartyLib内部有内存泄漏，很难排查
```

**成因**：第三方库的内部实现不透明，可能无法及时发现内存泄漏。

**解决方案**：
1. 使用`-XX:+HeapDumpOnOutOfMemoryError`自动生成堆转储
2. 使用MAT分析堆转储，找到占用内存最大的对象
3. 查看第三方库的文档，是否有内存泄漏的已知问题

## 我的理解（可选）

> 用自己的话重新表述一遍，检验是否真正理解。

内存溢出与泄漏的本质是**内存管理不当**：要么申请的内存超过了JVM限制，要么不再使用的对象无法被GC回收。

**核心思想**：
- **预防**：合理使用内存，避免内存泄漏
- **排查**：使用工具（jmap、MAT等）分析内存问题
- **修复**：找到泄漏根源，释放引用

**为什么需要关注内存问题？**
- **稳定性**：内存溢出会导致程序崩溃
- **性能**：内存泄漏会导致性能下降
- **可维护性**：及时发现和修复内存问题，提高代码质量

**最容易被忽略的点**：
- ThreadLocal使用不当会导致内存泄漏（尤其在线程池中）
- 静态集合、未关闭资源、监听器未注销等都可能导致内存泄漏
- OOM时自动生成堆转储（`-XX:+HeapDumpOnOutOfMemoryError`）非常重要

## 面试话术总结

1. **OutOfMemoryError有哪些类型？**
   "常见的OOM类型有：1）Java heap space（堆溢出）；2）Metaspace（元空间溢出）；3）Unable to create new native thread（无法创建新线程）；4）Direct buffer memory（直接内存溢出）；5）Requested array size exceeds VM limit（数组大小超过JVM限制）。"

2. **内存泄漏的常见原因？**
   "常见原因有：1）静态集合持有对象引用；2）ThreadLocal使用不当；3）未关闭资源（文件句柄、数据库连接等）；4）监听器未注销；5）外部连接未关闭。"

3. **如何排查内存泄漏？**
   "排查流程：1）开启GC日志；2）分析GC日志，判断是内存溢出还是内存泄漏；3）内存泄漏时，生成堆转储（jmap/jcmd）；4）使用MAT分析堆转储，找到占用内存最大的对象；5）查看对象到GC Roots的引用链，找到泄漏根源；6）修复代码。"

4. **ThreadLocal为什么会导致内存泄漏？**
   "ThreadLocalMap中的Entry是弱引用（key是ThreadLocal，value是强引用）。如果ThreadLocal被回收，key变为null，但value仍然是强引用，无法被GC。如果线程不结束（如线程池），value永远无法被GC。解决方案：使用完ThreadLocal后，调用remove()清理value。"

5. **有哪些工具可以排查内存问题？**
   "命令行工具有：jps（查看Java进程）、jstat（查看JVM统计信息）、jmap（生成堆转储）、jstack（生成线程转储）、jcmd（多功能工具）。可视化工具有：jconsole、jvisualvm、MAT（Memory Analyzer Tool）、JProfiler。"
