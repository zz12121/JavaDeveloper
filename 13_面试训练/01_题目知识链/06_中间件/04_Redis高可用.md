# Redis 高可用方案有哪些？

> ⚠️ **先盲答**：Redis 单机有单点故障问题，怎么解决？

---

## 盲答引导

1. 主从复制（Replication）解决了什么问题？—— 读写分离 + 数据冗余
2. 哨兵（Sentinel）解决了什么问题？—— 主库故障自动切换
3. Redis Cluster（集群）解决了什么问题？—— 数据分片 + 高可用
4. 主从复制 + 哨兵 vs Redis Cluster，什么时候用哪个？

---

## 知识链提示

```
Redis 高可用
  → [[Redis高可用]]
    → 主从复制（Replication）
      → 一主多从：主库写，从库读（读写分离）
      → 复制流程
        → 从库连接主库 → 发送 SYNC 命令
        → 主库 BGSAVE → 传 RDB → 从库加载
        → 后续：主库把写命令发给从库（异步）
      → 问题：主库故障，无法自动切换（需人工或 Sentinel）
    → 哨兵（Sentinel）
      → 监控主库：定期 PING，判断主库是否下线
      → 自动故障转移：选一个从库升主，其他从库切到新主
      → 配置提供者：客户端从 Sentinel 获取主库地址
      → 选举：哨兵之间 Raft 协议选 leader
    → Redis Cluster（官方集群方案）
      → 数据分片：16384 个槽位，每个主节点负责部分槽位
      → 高可用：每个主节点有从节点，主故障自动切换
      → 客户端路由：客户端计算 key 的槽位 → 直接连对应节点
        → 重定向：MOVED（槽位迁移）/ ASK（迁移中）
      → 限制：不支持多 key 跨槽位操作（可用 hash tag 解决）
```

---

## 核心追问

| 追问 | 考点 |
|------|------|
| 主从复制是同步还是异步？ | 异步（主库写完后立即返回，不等待从库） |
| Sentinel 怎么判断主库「客观下线」？ | 多个 Sentinel 都认为主库下线（quorum 机制） |
| Redis Cluster 的槽位为什么是 16384？ | 2 的 14 次方，兼顾心跳包大小和迁移粒度 |
| 什么是 Redis Cluster 的「槽位迁移」？ | 扩容时，把部分槽位从旧节点迁到新节点（在线迁移） |

---

## 参考答案要点

**三种方案对比**：

| 方案 | 数据分片 | 自动故障转移 | 适用场景 |
|------|---------|--------------|---------|
| 主从复制 | ❌ | ❌ | 读写分离，数据量小 |
| 主从 + 哨兵 | ❌ | ✅ | 高可用，数据量中等 |
| Redis Cluster | ✅ | ✅ | 大数据量，分布式 |
