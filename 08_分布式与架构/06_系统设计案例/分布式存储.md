# 分布式存储

## 这个问题为什么存在？

> 单机存储有容量上限（磁盘满了）、性能上限（IOPS不够）。问题是：**怎么把数据分散到多台机器，且保证可靠性和性能？**

没有分布式存储，你会遇到：
- 数据量超过单机磁盘（如10TB数据，单机只有4TB磁盘）
- 读写QPS超过单机上限（如10万QPS，单机只能扛1万）
- 单机挂了，数据丢了（没有副本）

---

## 它是怎么解决问题的？

### 核心架构：分片 + 副本

```

数据分片（Sharding）
  → 把数据分散到多台机器（解决容量和性能上限）

副本（Replication）
  → 每个分片有多个副本（解决可靠性和可用性）

                Client
                  │
            ┌─────┴─────┐
            │ NameNode │  ← 中心化：元数据管理（HDFS）
            └─────┬─────┘
                  │
        ┌─────────┼─────────┐
        ▼         ▼         ▼
   DataNode1  DataNode2  DataNode3
   (Shard 0)  (Shard 1)  (Shard 2)
   × 3副本    × 3副本    × 3副本
```

### 关键技术点

#### 1. 一致性Hash（数据分片）

```
传统Hash：Hash(key) % N
问题：N变化（扩容/缩容），几乎全部数据重新分片

一致性Hash：Hash(key) → 环上位置
   → 扩容时，只影响相邻分片（DHT算法）
   → 减少数据迁移量
```

```java
// 一致性Hash（简化版）
public class ConsistentHash {
    private final SortedMap<Long, String> ring = new TreeMap<>();

    public void addNode(String node) {
        // 每个节点生成150个虚拟节点（均匀分片）
        for (int i = 0; i < 150; i++) {
            ring.put(hash(node + "#" + i), node);
        }
    }

    public String getNode(String key) {
        long hash = hash(key);
        // 找到第一个 >= hash 的节点（环上顺时针）
        SortedMap<Long, String> tail = ring.tailMap(hash);
        Long nodeHash = tail.isEmpty() ? ring.firstKey() : tail.firstKey();
        return ring.get(nodeHash);
    }
}
```

#### 2. CAP权衡（一致性 vs 可用性）

```
CP系统（HDFS、ZooKeeper）：
  → 网络分区时，保一致性，拒绝服务
  → 适用：金融、元数据

AP系统（Cassandra、DynamoDB）：
  → 网络分区时，保可用，允许数据不一致
  → 适用：社交、内容分发
```

#### 3. 副本策略（NWR）

```
N = 副本总数（如3）
W = 写入副本数（如2）
R = 读取副本数（如2）

W + R > N  → 强一致性（读写都超过半数）
W = 1, R = 1  → 最终一致性（性能最高）
```

```java
// Cassandra：可调一致性
// 强一致性：W=QUORUM(2), R=QUORUM(2)
// 最终一致性：W=ONE(1), R=ONE(1)

Session session = cluster.connect("ks");
// 写入：一致性级别 QUORUM（超过半数副本写入成功）
PreparedStatement ps = session.prepare(
        "INSERT INTO orders (id, status) VALUES (?, ?)")
        .setConsistencyLevel(ConsistencyLevel.QUORUM);
```

## 深入原理

| | 中心化（HDFS） | 去中心化（Cassandra） | 对象存储（S3） |
|---|---|---|---|
| 元数据管理 | NameNode（中心） | 分布式（每节点都知全貌） | 中心化（S3Gateway） |
| 扩展性 | 差（NameNode瓶颈） | 好（无中心） | 极好（分片自动） |
| 一致性 | 强一致（CP） | 可调（NWR） | 最终一致（AP） |
| 适用场景 | 大数据分析（Hadoop） | 高并发读写（Cassandra） | 海量小文件（图片、视频） |

**本质区别**：
- **中心化**：元数据集中管理，简单但单点瓶颈
- **去中心化**：元数据分布式，扩展性好但复杂
- **对象存储**：基于HTTP，最适合海量小文件

## 正确使用方式

### 场景1：日志存储（HDFS）

```java
// HDFS：适合存大文件（如日志、视频）
Configuration conf = new Configuration();
conf.set("fs.defaultFS", "hdfs://namenode:8020");

FileSystem fs = FileSystem.get(conf);
// 写入大文件（分块，每块128MB，3副本）
FSDataOutputStream out = fs.create(new Path("/logs/app.log"));
out.write(logData.getBytes());
out.close();
```

### 场景2：用户行为存储（Cassandra）

```java
// Cassandra：适合写多读少、高并发
// 数据模型：按RowKey分片，Columns动态扩展

CREATE TABLE user_events (
    user_id UUID,
    event_time TIMESTAMP,
    event_type TEXT,
    event_data MAP<TEXT, TEXT>,
    PRIMARY KEY (user_id, event_time)
) WITH CLUSTERING ORDER BY (event_time DESC);
// 查询：按user_id分片，事件按时间倒排
```

## 边界情况和坑

### 坑1：脑裂（中心化系统）

```
场景：NameNode HA（主从）
      主NameNode网络通信故障（被认为挂了）
      从NameNode晋升（新主）
      原主恢复（还认为自己是大哥）
结果：两个主NameNode，元数据冲突
```

**解决方案**：
1. **Fencing机制**（SSH杀掉原主进程）
2. **共享存储**（JN，JournalNode，保证元数据一致）

### 坑2：数据倾斜（一致性Hash）

```
场景：一致性Hash后，某个分片数据量是其他的10倍
后果：该分片成为热点，性能瓶颈
```

**解决方案**：
1. **虚拟节点**（每个物理节点生成100~200个虚拟节点，均匀分片）
2. **手动调整**（热点分片拆成两个）

### 坑3：副本丢失（磁盘故障）

```
场景：3副本，同时坏了2个
后果：数据丢失（只剩1个副本，无法满足W=2）
```

**解决方案**：
1. **EC（纠删码）**（如Reed-Solomon，3+2，允许2个副本丢失）
2. **快速恢复**（磁盘故障，立即从其他副本恢复）
