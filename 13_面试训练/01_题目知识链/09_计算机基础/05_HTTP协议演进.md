# 题目：HTTP/1.1、HTTP/2、HTTP/3 的核心区别是什么？

## 盲答引导

1. HTTP/1.1 的「队头阻塞」是什么意思？一个请求慢了会怎样？
2. HTTP/2 的「多路复用」是怎么解决的？为什么还会有队头阻塞？
3. HTTP/2 的「头部压缩」（HPACK）是怎么做的？和 HTTP/1.1 比省了多少？
4. HTTP/3 为什么抛弃 TCP，改用 QUIC（基于 UDP）？
5. 从 HTTP/1.1 升级到 HTTP/2，需要改应用代码吗？需要换证书吗？

## 知识链提示

这道题应该让你联想到：

- `[[HTTP_1.1局限]]` → 队头阻塞、头部冗余、连接无法复用
- `[[HTTP_2多路复用]]` → 二进制分帧、Stream 独立
- `[[HTTP_3_QUIC]]` → 解决 TCP 队头阻塞、0-RTT、连接迁移
- `[[TLS在HTTP演进中]]` → H2 必须 TLS，H3 内置加密
- `[[服务器推送]]` → HTTP/2 Server Push（H3 里改成了「早期数据」）

## 核心追问

1. HTTP/2 的多路复用，底层还是同一个 TCP 连接，TCP 层的队头阻塞还存在吗？为什么？
2. HTTP/2 的「流优先级」是什么？有什么实际应用场景？
3. QUIC 的「连接迁移」是什么意思？为什么 TCP 做不到？
4. HTTP/3 里，TLS 加密是在 QUIC 层做的还是单独做的？
5. 为什么国内很多网站还没上 HTTP/3？（部署复杂度、CDN 支持、浏览器兼容）

**三代协议对比**：

| | HTTP/1.1 | HTTP/2 | HTTP/3 |
|--|----------|--------|--------|
| 传输层 | TCP | TCP | QUIC（UDP） |
| 编码 | 文本 | 二进制 | 二进制 |
| 多路复用 | ❌（队头阻塞） | ✅（Stream 级别） | ✅（Stream 独立，无队头阻塞） |
| 头部压缩 | ❌（明文，重复发送） | ✅（HPACK） | ✅（QPACK） |
| 服务器推送 | ❌ | ✅（Server Push） | ✅（早期数据） |
| 加密 | 可选 | 强制 TLS | 内置（不能关） |
| 队头阻塞 | TCP 层 + 应用层 | TCP 层还有（一个包丢，所有流等） | 无（每个流独立） |
| 连接迁移 | ❌ | ❌ | ✅（Connection ID） |
| 握手 RTT | 无 TLS：0-RTT；有 TLS：2-RTT | 2-RTT | 1-RTT（可 0-RTT） |

**HTTP/1.1 队头阻塞**：

```
HTTP/1.1 默认开启 keep-alive（一个 TCP 连接处理多个请求）
但请求是「串行」的：
  请求1 → 等响应1 → 请求2 → 等响应2

如果响应1 很慢 → 请求2 一直排队（队头阻塞）

解决（不完美）：浏览器开多个 TCP 连接（通常6个）并行请求
```

**HTTP/2 多路复用**：

```
把请求/响应拆成「帧」（Frame）
同一个 TCP 连接里，多个 Stream（流）的帧交错发送
每个帧带 Stream ID → 接收方重新组装

结果：请求1 的响应慢，不影响请求2 的帧传送（应用层无队头阻塞）

但！TCP 层还有队头阻塞：
  如果 TCP 的一个包丢了 → TCP 重传 → 所有 Stream 都等这个包
  （因为 TCP 保证有序交付，后面的包即使到了也要等）
```

**HTTP/3 彻底解决队头阻塞**：

```
QUIC（UDP 上实现）：
- 每个 Stream 独立（一个 Stream 的包丢了，不影响其他 Stream）
- 内置 TLS 1.3（加密在 QUIC 层做）
- 连接标识用 Connection ID（不是 IP+端口）
  → 客户端换 WiFi → IP 变了 → 但 Connection ID 没变 → 连接不断！
```

**HPACK 头部压缩原理**：

```
HTTP/1.1 每次请求都带完整的头部（User-Agent、Cookie 通常不变）
HTTP/2 用 HPACK：
  - 静态表（常见头部，预定义索引）
  - 动态表（连接内缓存用过的头部，后续用索引代替）
  - 哈夫曼编码（进一步压缩）
→ 通常能省 50%~90% 的头部大小
```
