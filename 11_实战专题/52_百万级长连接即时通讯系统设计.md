## 场景 A：Netty Reactor多线程模型

### 现象

```
10万+并发连接，CPU利用率只有20%
消息处理延迟高，P99达到500ms+
单机只能支撑1万连接，无法水平扩展
```

### 根因

```
用阻塞IO模型，每个连接占用一个线程
未用Netty的Reactor模式，IO和业务处理混杂
线程数过多，上下文切换开销大
```

### 解决方案

```java
/**
 * Netty Reactor主从多线程模型（支持百万级连接）
 * 
 * 架构：1个Boss线程(NIO accept) + N个Worker线程(NIO IO)
 *       + M个Business线程池(处理业务逻辑)
 */
public class IMServer {

    public void start(int port) {
        // 1. Boss线程组：只负责接受新连接
        EventLoopGroup bossGroup = new NioEventLoopGroup(1);
        
        // 2. Worker线程组：负责IO读写（建议=CPU核数*2）
        EventLoopGroup workerGroup = new NioEventLoopGroup(
            Runtime.getRuntime().availableProcessors() * 2);
        
        // 3. 业务线程池：处理非IO操作（避免阻塞Worker线程）
        EventExecutorGroup businessGroup = new DefaultEventExecutorGroup(64);

        ServerBootstrap bootstrap = new ServerBootstrap();
        bootstrap.group(bossGroup, workerGroup)
            .channel(NioServerSocketChannel.class)
            
            // 关键配置：TCP参数调优
            .option(ChannelOption.SO_BACKLOG, 1024)         // 全连接队列长度
            .option(ChannelOption.SO_REUSEADDR, true)       // 允许端口复用
            
            // 子通道参数：每个连接的配置
            .childOption(ChannelOption.TCP_NODELAY, true)   // 禁用Nagle算法，减少小包延迟
            .childOption(ChannelOption.SO_KEEPALIVE, true)  // 操作系统级心跳
            .childOption(ChannelOption.SO_RCVBUF, 64 * 1024)  // 接收缓冲区64KB
            .childOption(ChannelOption.SO_SNDBUF, 64 * 1024)  // 发送缓冲区64KB
            
            .childHandler(new ChannelInitializer<Channel>() {
                @Override
                protected void initChannel(Channel ch) {
                    ChannelPipeline pipeline = ch.pipeline();
                    
                    // 协议解码/编码
                    pipeline.addLast("decoder", new IMProtocolDecoder());
                    pipeline.addLast("encoder", new IMProtocolEncoder());
                    
                    // 业务处理器：交给独立线程池执行（防止阻塞IO线程！）
                    pipeline.addLast(businessGroup, "imHandler", new IMHandler());
                }
            });

        try {
            ChannelFuture future = bootstrap.bind(port).sync();
            log.info("IM服务启动成功, port={}", port);
            future.channel().closeFuture().sync();
        } finally {
            bossGroup.shutdownGracefully();
            workerGroup.shutdownGracefully();
        }
    }
}
```

**Reactor关键**：
- Boss线程只做accept（1个足够），Worker线程做IO读写
- **必须**把业务逻辑交给独立线程池（businessGroup），不能在IO线程中做DB/MQ等耗时操作
- `TCP_NODELAY=true`关闭Nagle算法，对IM场景（小消息、低延迟）至关重要

## 场景 B：心跳保活与断线重连

### 现象

```
客户端显示在线，但实际已断网
大量僵尸连接占用服务端资源
网络抖动导致频繁断线重连
```

### 根因

```
无心跳机制，无法检测假死连接
无重连策略，断开后无法恢复会话
心跳间隔不合理，要么太费流量要么检测慢
```

### 解决方案

```java
/**
 * 心跳保活机制实现
 * 1. 服务端定时扫描空闲连接（超过30秒没收到数据视为断开）
 * 2. 客户端每15秒发送一次心跳包
 * 3. 断线后客户端指数退避重连
 */
public class HeartbeatHandler extends ChannelInboundHandlerAdapter {
    
    // 心跳间隔：15秒
    private static final int HEARTBEAT_INTERVAL_SEC = 15;
    
    // 超时阈值：30秒（连续3次心跳未响应则判定断开）
    private static final int TIMEOUT_THRESHOLD_SEC = HEARTBEAT_INTERVAL_SEC * 3;

    @Override
    public void channelActive(ChannelHandlerContext ctx) throws Exception {
        // 连接建立后启动心跳定时任务
        scheduleHeartbeat(ctx);
        super.channelActive(ctx);
    }

    /**
     * 定时发送心跳（客户端侧）
     * 服务端收到心跳后回复pong
     */
    private void scheduleHeartbeat(ChannelHandlerContext ctx) {
        ctx.executor().schedule(() -> {
            if (ctx.channel().isActive()) {
                // 发送轻量心跳包（仅6字节：type=0x0A, timestamp）
                ctx.writeAndFlush(IMHeartbeatPacket.create());
                // 继续下一次心跳调度
                scheduleHeartbeat(ctx);
            }
        }, HEARTBEAT_INTERVAL_SEC, TimeUnit.SECONDS);
    }

    @Override
    public void userEventTriggered(ChannelHandlerContext ctx, Object evt) throws Exception {
        if (evt instanceof IdleStateEvent) {
            IdleStateEvent idleEvt = (IdleStateEvent) evt;
            
            // 读超时：30秒内没有收到任何数据 → 判定断开
            if (idleEvt.state() == IdleState.READER_IDLE) {
                String userId = ctx.channel().attr(AttributeKey.valueOf("userId")).get();
                log.warn("用户{}心跳超时，关闭连接", userId);
                
                // 通知业务层用户离线
                userService.offline(userId);
                ctx.close();
            }
        }
    }
}
```

```java
/**
 * 客户端指数退避重连策略
 */
public class ReconnectStrategy {
    
    private static final int MAX_RETRY_COUNT = 5;      // 最大重试次数
    private static final long BASE_DELAY_MS = 1000;    // 基础延迟1秒
    private static final long MAX_DELAY_MS = 30_000;   // 最大延迟30秒
    
    public void reconnect(String serverHost, int port) {
        for (int i = 0; i < MAX_RETRY_COUNT; i++) {
            try {
                // 指数退避：1s -> 2s -> 4s -> 8s -> 16s（最大30s）
                long delay = Math.min(BASE_DELAY_MS * (1L << i), MAX_DELAY_MS);
                Thread.sleep(delay);
                
                log.info("尝试第{}次重连...", i + 1);
                connect(serverHost, port);  // 尝试重新连接
                return;  // 重连成功，退出
                
            } catch (Exception e) {
                log.error("第{}次重连失败", i + 1);
            }
        }
        log.error("重连{}次均失败，放弃连接", MAX_RETRY_COUNT);
    }
}
```

**心跳关键**：
- 客户端每15秒发心跳，服务端30秒未收数据则踢掉连接
- 用Netty的`IdleStateHandler`自动检测读超时
- 重连采用指数退避，避免风暴效应

## 场景 C：消息推送QoS（ACK/重试）

### 现象

```
消息发出后对方未收到，且无法感知
弱网环境下丢包严重，聊天记录缺失
群发消息部分人收到，部分人未收到
```

### 根因

```
无ACK确认机制，不知道消息是否送达
无持久化存储，离线消息丢失
无重试机制，失败即丢弃
```

### 解决方案

```
QoS等级设计：
- QoS 0（最多一次）：普通文本消息，允许少量丢失（如"正在输入"状态）
- QoS 1（至少一次）：重要消息，需要ACK确认，失败自动重试（如聊天消息）
- QoS 2（恰好一次）：支付类消息，去重保证不重复（如转账通知）

消息流转：
Client → Gateway → MQ(Kafka) → Consumer → DB存储 → 推送目标用户
                                    ↓
                              收到目标用户ACK后标记为已送达
```

```java
/**
 * QoS 1消息推送实现（至少一次投递）
 */
public class MessagePushService {

    private final RedisTemplate<String, String> redisTemplate;
    private final KafkaTemplate<String, String> kafkaTemplate;

    /**
     * 推送消息到目标用户（带重试机制）
     */
    public void pushMessage(Long targetUserId, IMMessage message) {
        String msgId = message.getId();  // 全局唯一消息ID
        
        // 1. 先存入Redis待确认队列（用于ACK确认）
        redisTemplate.opsForList()
            .leftPush("im:pending:" + targetUserId, JSON.toJSONString(message));
        
        // 2. 通过WebSocket推送
        boolean sent = imConnectionManager.sendToUser(targetUserId, message);
        
        if (!sent) {
            // 用户离线，消息进入离线存储
            offlineStorage.save(targetUserId, message);
        }
        // 如果在线但推送失败（网络问题），依赖ACK超时重试
    }

    /**
     * 处理目标用户的ACK确认
     * 收到ACK后将消息从pending队列移除
     */
    public void onAck(Long userId, String msgId) {
        // 从待确认队列移除该消息
        ListOperations<String, String> ops = redisTemplate.opsForList();
        ops.remove("im:pending:" + userId, 0, msgId);
        
        // 标记消息为已送达
        messageDao.updateStatus(msgId, "DELIVERED");
    }

    /**
     * 定时任务：重试未确认的消息（每隔5分钟扫一次）
     */
    @Scheduled(fixedRate = 5 * 60 * 1000)
    public void retryPendingMessages() {
        Set<String> keys = redisTemplate.keys("im:pending:*");
        keys.forEach(key -> {
            while (true) {
                String msgJson = (String) redisTemplate.opsForList().rightPop(key);
                if (msgJson == null) break;
                
                IMMessage msg = JSON.parseObject(msgJson, IMMessage.class);
                Long targetUserId = extractUserIdFromKey(key);
                
                if (msg.getRetryCount() >= 3) {
                    // 超过3次重试，转为离线存储或告警
                    offlineStorage.save(targetUserId, msg);
                    alertManager.sendAlert("消息投递失败: " + msgId);
                } else {
                    // 重试推送
                    pushMessage(targetUserId, msg.incrementRetryCount());
                }
            }
        });
    }
}
```

## 场景 D：协议选型（WebSocket/MQTT）

### 现象

```
HTTP轮询方式消耗服务器资源，延迟高
自定义协议解析复杂，跨平台兼容差
移动端弱网环境下连接不稳定
```

### 根因

```
协议选型不当，不适合IM长连接场景
未考虑移动端省电/省流量需求
协议头部冗余大，带宽浪费严重
```

### 解决方案

| 对比维度 | WebSocket | MQTT | 自定义TCP |
|---------|-----------|------|----------|
| 适用场景 | Web/App通用IM | IoT/移动设备 | 高性能定制IM |
| 头部开销 | 较大（HTTP握手） | 极小（2字节固定头） | 最小（可自定义） |
| 移动端友好度 | 一般 | 优秀（QoS内置） | 需自己实现 |
| 浏览器支持 | 原生支持 | 需JS库 | 不支持 |
| 实现复杂度 | 中等 | 低（有成熟SDK） | 高 |

```
推荐方案：
- App端：使用MQTT协议（内置QoS、心跳、离线消息，适合移动端）
- Web端：使用WebSocket（浏览器原生支持，开发简单）
- 服务端统一转换为内部二进制协议（Protobuf序列化）
```

```java
/**
 * 自定义IM协议定义（基于Protobuf，兼顾性能和扩展性）
 * 
 * 报文结构：
 * [4字节魔数] [1字节版本] [1字节类型] [2字节长度] [N字节Body]
 * 类型值：0x01=登录  0x02=心跳  0x03=单聊  0x04=群聊  0x05=ACK
 */
@Proto
message IMMessage {
    string id = 1;           // 消息唯一ID（UUID）
    int64 fromUserId = 2;    // 发送者ID
    int64 toUserId = 3;      // 接收者ID（私聊）或groupId（群聊）
    MessageType type = 4;    // 消息类型：TEXT/IMAGE/FILE/AUDIO
    bytes content = 5;       // 消息体（根据类型不同解析）
    int64 timestamp = 6;     // 时间戳
    int32 qosLevel = 7;      // QoS等级：0/1/2
}
```

**协议选型关键**：
- App端优先MQTT（省电、省流、内置QoS），Web端用WebSocket
- 内部通信统一Protobuf二进制协议，压缩率高、解析快
- 协议版本号预留，方便后续兼容性升级

## 排查 Checklist

```
□ 用Reactor多线程了吗？ → Boss(1)+Worker(CPU*2)+Business(64)分离
□ 业务逻辑交给线程池了吗？ → 不能阻塞IO线程！
□ 有心跳保活吗？ → 15s心跳/30s超时/指数退避重连
□ TCP_NODELAY开启了吗？ → IM场景必须关闭Nagle算法
□ 消息有ACK确认吗？ → QoS 1重要消息必须有ACK+重试
□ 离线消息存储了吗？ → 用户离线时消息入库，上线后拉取
□ 协议选择合理吗？ → App用MQTT/Web用WS/内部用Protobuf
□ 单机能撑多少？ → 调整文件描述符限制ulimit -n 1000000
```
