# 数据库分布式锁

## 这个问题为什么存在？

> Redis锁性能高，但引入Redis增加运维成本；ZooKeeper锁强一致，但部署复杂。问题是：**能不能用已有的数据库实现分布式锁？**

能用，但性能和可靠性是硬伤。适合**并发量低、对性能要求不高**的场景。

---

## 它是怎么解决问题的？

### 方案一：基于唯一约束（悲观锁）

```
原理：利用DB的唯一约束，insert成功则获得锁
      insert失败则没获得锁
```

```java
// 获取锁：insert，成功则获得锁
public boolean tryLock(String lockKey, long expireSeconds) {
    try {
        String sql = """
            INSERT INTO distributed_lock (lock_key, owner, expire_time)
            VALUES (?, ?, DATE_ADD(NOW(), INTERVAL ? SECOND))
            """;
        jdbcTemplate.update(sql, lockKey, getOwnerId(), expireSeconds);
        return true;  // 插入成功，获得锁
    } catch (DuplicateKeyException e) {
        return false; // 插入失败，锁已被占用
    }
}

// 释放锁：删除记录
public void unlock(String lockKey) {
    String sql = "DELETE FROM distributed_lock WHERE lock_key = ? AND owner = ?";
    jdbcTemplate.update(sql, lockKey, getOwnerId());
}
```

**问题**：
1. **死锁风险**：客户端挂了，锁记录永远不删除（需要用定时任务清理过期锁）
2. **性能差**：每次加锁都是磁盘IO

### 方案二：基于乐观锁（版本号）

```
原理：用版本号（version）实现CAS
      更新时检查版本号，一致才更新成功
```

```java
// 乐观锁：适合"冲突不频繁"的场景
public boolean updateWithOptimisticLock(Long id, int oldVersion, String newName) {
    String sql = """
        UPDATE users
        SET name = ?, version = version + 1
        WHERE id = ? AND version = ?
        """;
    int rows = jdbcTemplate.update(sql, newName, id, oldVersion);
    return rows > 0;  // 返回true表示更新成功
}

// 使用：
int version = getUserVersion(id);
boolean success = updateWithOptimisticLock(id, version, "newName");
if (!success) {
    // 版本号冲突，重试或抛异常
    retryOrThrow();
}
```

**适用场景**：**读多写少**，冲突不频繁（如商品详情页更新）。

### 方案三：基于悲观锁（SELECT ... FOR UPDATE）

```
原理：用DB的行锁，锁定记录
      事务提交后才释放锁
```

```java
// 悲观锁：适合"冲突频繁"的场景
@Transactional
public void processWithPessimisticLock(Long id) {
    // SELECT ... FOR UPDATE 锁定这行
    String sql = "SELECT * FROM orders WHERE id = ? FOR UPDATE";
    Order order = jdbcTemplate.queryForObject(sql, new OrderRowMapper(), id);

    // 处理业务逻辑...
    processOrder(order);

    // 事务提交时，自动释放行锁
}
```

**适用：** 冲突频繁，需要强一致（如**金融转账**）。

## 它和相似方案的本质区别是什么？

| | 数据库锁（唯一约束） | 数据库锁（乐观锁） | 数据库锁（悲观锁） | Redis锁 | ZooKeeper锁 |
|---|---|---|---|---|---|
| 性能 | 低（磁盘IO） | 中（读不锁） | 低（行锁阻塞） | 极高 | 中 |
| 一致性 | 中（依赖DB） | 最终一致（重试） | 强一致（行锁） | 最终一致 | 强一致 |
| 死锁风险 | 有（客户端挂了） | 无（无锁） | 有（需要超时） | 无（TTL） | 无（临时节点） |
| 适用场景 | 并发低 | 读多写少 | 冲突频繁 | 高并发 | 一致性要求高 |

**本质区别**：数据库锁**性能差**，但**不需要额外组件**（Redis、ZK），适合**并发量低**的内部系统。

## 正确使用方式

### 场景1：定时任务防重复执行（唯一约束）

```java
// 定时任务：每天凌晨2点执行，但多台机器会同时触发
@Scheduled(cron = "0 0 2 * * ?")
public void dailyTask() {
    if (tryLock("dailyTask", 3600)) {  // 锁超时1小时
        try {
            // 业务逻辑...
            processDailyData();
        } finally {
            unlock("dailyTask");
        }
    } else {
        log.info("其他机器正在执行，跳过");
    }
}
```

### 场景2：商品库存扣减（乐观锁）

```java
// 乐观锁：扣减库存
public boolean deductInventory(Long productId, int deductAmount) {
    int maxRetries = 3;
    for (int i = 0; i < maxRetries; i++) {
        // 1. 读取当前库存和版本号
        Inventory inv = getInventory(productId);
        if (inv.getStock() < deductAmount) {
            throw new RuntimeException("库存不足");
        }

        // 2. 更新库存，带版本号检查
        String sql = """
            UPDATE inventory
            SET stock = stock - ?, version = version + 1
            WHERE id = ? AND version = ?
            """;
        int rows = jdbcTemplate.update(sql, deductAmount, productId, inv.getVersion());
        if (rows > 0) {
            return true;  // 扣减成功
        }
        // 版本号冲突，重试
    }
    throw new RuntimeException("扣减失败，请重试");
}
```

## 边界情况和坑

### 坑1：锁超时，但业务没执行完

```
场景：
  T0: 获取锁成功，超时3600秒
  T1: 业务执行慢，超过3600秒
  T3601: 锁超时，DB自动删除记录
  T3602: 另一台机器获取锁成功
  → 两个机器同时执行业务！
```

**解决方案**：
1. **定时任务续期**（类似Redis的watchdog）
2. **锁超时时间设长一点**（如24小时），但治标不治本
3. **业务幂等**（最终兜底）

### 坑2：死锁（SELECT ... FOR UPDATE）

```
场景：
  事务A：锁定订单1 → 锁定订单2
  事务B：锁定订单2 → 锁定订单1
  → 死锁！
```

**解决方案**：
1. **按固定顺序加锁**（如按订单ID排序）
2. **设锁等待超时**（`innodb_lock_wait_timeout`）
3. **检测死锁**（DB会自动检测并回滚一个事务）

### 坑3：连接池耗尽

```
场景：所有连接都在等待锁（SELECT ... FOR UPDATE）
后果：连接池耗尽，整个系统不可用
```

**解决方案**：
1. **设锁等待超时**（`innodb_lock_wait_timeout = 5`，默认50秒太长）
2. **用乐观锁替代悲观锁**（读多写少场景）
3. **限流**（控制并发量，不超过连接池大小）
