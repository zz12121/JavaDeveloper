# 备份与恢复策略

## 这个问题为什么存在？

> 数据库被误删了、被黑客勒索了、磁盘坏了 → 数据全没了。
> 没有备份 = 等死。

**备份的核心目标**：
1. **数据恢复**：误删、故障后恢复数据
2. **数据审计**：历史数据回溯
3. **测试环境**：用生产备份恢复测试环境

---

## 它是怎么解决问题的？

### 备份类型

| 备份类型 | 说明 | 优点 | 缺点 |
|---------|------|------|------|
| **逻辑备份** | 导出SQL语句（`mysqldump`） | 跨版本恢复、可读、可编辑 | 恢复慢（需要执行SQL） |
| **物理备份** | 直接拷贝数据文件（`xtrabackup`） | 恢复快（直接拷贝回去） | 需要相同版本、不可读 |
| **增量备份** | 只备份变化的数据 | 备份快、占用空间小 | 恢复时需要全量+所有增量 |
| **全量备份** | 备份全部数据 | 恢复简单 | 备份慢、占用空间大 |

### 逻辑备份：`mysqldump`

**全量备份**：

```bash
# 备份单个数据库
mysqldump -u root -p --single-transaction --routines --triggers --events db_name > backup.sql

# 备份所有数据库
mysqldump -u root -p --single-transaction --all-databases > full_backup.sql

# 压缩备份
mysqldump -u root -p --single-transaction db_name | gzip > backup.sql.gz
```

**关键参数**：

| 参数 | 作用 |
|------|------|
| `--single-transaction` | 对InnoDB，在单个事务内备份，保证一致性 |
| `--lock-all-tables` | 对MyISAM，锁全表保证一致性（会阻塞写） |
| `--routines` | 备份存储过程和函数 |
| `--triggers` | 备份触发器 |
| `--events` | 备份事件 |
| `--where` | 只备份部分数据（比如只备份最近1年的订单） |

**恢复**：

```bash
# 恢复全量备份
mysql -u root -p db_name < backup.sql

# 恢复压缩备份
gunzip < backup.sql.gz | mysql -u root -p db_name
```

### 物理备份：`xtrabackup`（推荐生产环境）

**全量备份**：

```bash
# 备份到 /data/backup/full
xtrabackup --backup --target-dir=/data/backup/full --user=root --password=xxx

# 准备（使备份一致）
xtrabackup --prepare --target-dir=/data/backup/full
```

**增量备份**：

```bash
# 1. 先全量备份
xtrabackup --backup --target-dir=/data/backup/full --user=root --password=xxx

# 2. 第一次增量备份（基于全量）
xtrabackup --backup --target-dir=/data/backup/inc1 --incremental-basedir=/data/backup/full --user=root --password=xxx

# 3. 第二次增量备份（基于第一次增量）
xtrabackup --backup --target-dir=/data/backup/inc2 --incremental-basedir=/data/backup/inc1 --user=root --password=xxx
```

**恢复**：

```bash
# 1. 准备全量备份
xtrabackup --prepare --apply-log-only --target-dir=/data/backup/full

# 2. 应用第一次增量
xtrabackup --prepare --apply-log-only --target-dir=/data/backup/full --incremental-dir=/data/backup/inc1

# 3. 应用第二次增量（最后一次不要 --apply-log-only）
xtrabackup --prepare --target-dir=/data/backup/full --incremental-dir=/data/backup/inc2

# 4. 拷贝回数据目录
systemctl stop mysqld
rm -rf /var/lib/mysql/*
xtrabackup --move-back --target-dir=/data/backup/full
chown -R mysql:mysql /var/lib/mysql
systemctl start mysqld
```

### 基于二进制日志（binlog）的PITR（时间点恢复）

**场景**：今天下午3点，误删了一个表。全量备份是昨天晚上12点。

```
恢复流程：
1. 恢复全量备份（昨天12点）
2. 重放binlog（从昨天12点 → 今天下午3点前）
3. 数据恢复到今天的上午状态（误删前的状态）
```

**操作步骤**：

```bash
# 1. 恢复全量备份
mysql -u root -p db_name < full_backup.sql

# 2. 重放binlog（从全量备份时间 → 误删前的时间）
mysqlbinlog --start-datetime="2024-01-01 00:00:00" \
           --stop-datetime="2024-01-02 14:59:59" \
           /var/lib/mysql/mysql-bin.000001 \
           | mysql -u root -p

# 3. 如果知道具体的binlog位置（更精确）
mysqlbinlog --start-position=1234 --stop-position=5678 \
           /var/lib/mysql/mysql-bin.000001 \
           | mysql -u root -p
```

## 它和相似方案的本质区别是什么？

### 逻辑备份 vs 物理备份

| 维度 | 逻辑备份（`mysqldump`） | 物理备份（`xtrabackup`） |
|------|------------------------|--------------------------|
| 备份速度 | 慢（需要执行SELECT） | 快（直接拷贝文件） |
| 恢复速度 | 慢（需要执行SQL） | 快（直接拷贝回去） |
| 跨版本恢复 | 支持 | 不支持（需要相同版本） |
| 可读性 | 可读（SQL文件） | 不可读（二进制文件） |
| 适用场景 | 小数据量（< 50GB）、跨版本 | 大数据量（> 50GB）、同版本 |

**本质区别**：逻辑备份是"逻辑层"（SQL语句），物理备份是"物理层"（数据文件）。

### 全量备份 vs 增量备份

| 维度 | 全量备份 | 增量备份 |
|------|----------|----------|
| 备份速度 | 慢（每次都备份全部） | 快（只备份变化） |
| 恢复速度 | 快（只需要恢复一份） | 慢（全量 + 所有增量） |
| 存储空间 | 大 | 小 |
| 恢复复杂度 | 简单 | 复杂（需要按顺序恢复） |

**本质区别**：权衡"备份时间/空间" vs "恢复时间/复杂度"。

## 正确使用方式

### 1. 生产环境备份策略（推荐）

```
策略：全量备份（每周） + 增量备份（每天） + binlog（实时）

1. 每周日凌晨2点：全量备份
   0 2 * * 0 /usr/bin/xtrabackup --backup --target-dir=/data/backup/full_$(date +\%Y\%m\%d)

2. 每天凌晨2点：增量备份（基于昨天的备份）
   0 2 * * * /usr/bin/xtrabackup --backup --target-dir=/data/backup/inc_$(date +\%Y\%m\%d) --incremental-basedir=/data/backup/full_or_prev_inc

3. binlog实时同步到备份服务器
   # 主库配置：log-bin=mysql-bin
   # 备份服务器：实时拷贝binlog到异地
```

### 2. 验证备份可用性（很重要！）

```
坑：备份成功，但恢复时发现备份文件损坏 → 等于没备份！

解决：定期恢复演练
1. 每月做一次恢复演练（把备份恢复到测试环境）
2. 验证数据完整性（比对记录数、关键数据）
```

```bash
# 恢复演练脚本（cron任务）
#!/bin/bash
# 恢复昨天的备份到测试环境
xtrabackup --prepare --target-dir=/data/backup/full
# ... 恢复到测试环境
# 验证数据
mysql -u root -p -e "SELECT COUNT(*) FROM db_name.orders" | mail -s "Backup Verify" dba@company.com
```

### 3. 备份文件的管理

```bash
# 备份文件保留策略
# 全量备份：保留最近4周（4个）
# 增量备份：保留最近7天（7个）
# binlog：保留最近7天

# 自动清理脚本
find /data/backup/full -name "full_*" -mtime +28 -exec rm -rf {} \;
find /data/backup/inc -name "inc_*" -mtime +7 -exec rm -rf {} \;
```

## 边界情况和坑

### 坑1：逻辑备份时锁表（MyISAM）

```
现象：
mysqldump备份MyISAM表时，默认锁全表
→ 备份期间，所有写操作被阻塞
```

**解决**：
1. 改用InnoDB（支持`--single-transaction`，不锁表）
2. 如果必须用MyISAM，在从库上备份（影响小）

### 坑2：物理备份恢复时版本不一致

```
现象：
备份是MySQL 5.7的
恢复到MySQL 8.0 → 报错（数据文件格式不兼容）
```

**解决**：
- 物理备份只能恢复到相同版本（或严格兼容的版本）
- 如果要跨版本恢复，用逻辑备份（`mysqldump`）

### 坑3：备份时磁盘空间不足

```
现象：
备份进行中，磁盘满了
→ 备份失败，可能产生不完整的备份文件
```

**解决**：
1. 备份前检查磁盘空间（`df -h`）
2. 备份到独立磁盘（不要备份到数据库同一块磁盘）
3. 压缩备份（`mysqldump | gzip` 或 `xtrabackup --compress`）

### 坑4：binlog没有实时同步到异地

```
场景：
主库磁盘坏了，binlog也在同一块磁盘上
→ binlog丢了，无法做PITR
```

**解决**：
1. 主库配置：`sync_binlog=1`（每次事务提交都刷盘）
2. 实时拷贝binlog到异地（比如用`mysqlbinlog`远程实时拷贝）
3. 或者：使用存储层同步（如云数据库的自动备份）
