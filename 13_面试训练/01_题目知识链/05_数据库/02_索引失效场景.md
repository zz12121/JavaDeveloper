# 索引失效的常见场景有哪些？

> ⚠️ **先盲答**：索引失效是什么意思？为什么索引会失效？

---

## 盲答引导

1. 什么是「索引失效」？—— 明明建了索引，但 explain 显示没走索引
2. 你在实际项目中遇到过哪些导致索引失效的 SQL 写法？
3. 联合索引的「最左前缀原则」具体是什么意思？
4. 索引失效和「索引不生效」是一回事吗？有区别吗？

---

## 知识链提示

```
索引失效场景
  → [[06_数据库/02_索引/索引失效场景]]
    → 最左前缀原则
      → 联合索引 (a, b, c) → 只走了 a 才有效
        → WHERE a = 1 AND c = 2 → 只用了 a 的索引，c 不生效
        → WHERE b = 1 → 完全不走索引
        → 秘密：联合索引是先按 a 排序，再按 b 排序，c 同理
    → 函数/运算导致索引失效
      → WHERE YEAR(create_time) = 2024 → 索引失效
        → 原因：对索引列做了函数运算，B+树的有序性被破坏
        → 改写：WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01'
      → WHERE amount + 1 = 100 → 索引失效
    → 隐式类型转换
      → 字段是 VARCHAR，传参是数字 → MySQL 会把字段都转成数字再比较 → 失效
      → phone = 13800138000（字段是 VARCHAR）→ 失效
    → OR 条件
      → WHERE a = 1 OR b = 2 → 如果 a 有索引、b 没索引 → 全表扫描
    → LIKE 左模糊
      → WHERE name LIKE '%明' → 失效（B+树只能从左到右匹配）
      → WHERE name LIKE '张%' → 不失效（前缀匹配，B+树有序）
    → NULL 值
      → 索引列允许 NULL → IS NULL / IS NOT NULL 可能失效（取决于 NULL 比例）
```

## 核心追问

| 追问 | 考点 |
|------|------|
| 联合索引 (a,b,c)，WHERE a=1 AND c=2 走索引吗？为什么？ | 只走 a，c 失效 |
| 索引条件下推（ICP）是什么？它和索引失效是一回事吗？ | 优化手段，不是失效 |
| 覆盖索引能解决索引失效问题吗？ | 能，避免回表 |
| 如何快速判断一条 SQL 是否走了索引？ | EXPLAIN 看 type/key 字段 |

---

## 参考答案要点

**八大失效场景**：
1. 违反最左前缀原则 → 联合索引必须从左边开始匹配
2. 对索引列使用函数 → `YEAR(date)` 等
3. 对索引列做运算 → `amount + 1 = 100`
4. 隐式类型转换 → VARCHAR 字段传数字
5. OR 条件中有字段无索引 → 全表扫描
6. LIKE 左模糊 → `%keyword`
7. 不等于 / IS NULL / IS NOT NULL → 可能因数据分布失效
8. 优化器判断扫描范围超过一定比例 → 放弃索引
