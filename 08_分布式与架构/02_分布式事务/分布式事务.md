# 分布式事务

> 单体应用里，多个操作在同一个数据库事务里，要么全成要么全败。拆成微服务后，订单服务调库存服务、库存服务调支付服务——任何一个环节失败，已经提交的事务无法回滚。问题是：**跨服务、跨数据库，怎么保证数据一致性？**

这本质上是在**强一致性（ACID）** 和 **可用性（CAP）** 之间找平衡。没有一致性协议，你会遇到：
- 订单创建成功，扣库存失败 → 超卖
- 支付成功，通知失败 → 用户没收到确认
- 服务A提交，服务B回滚 → 数据不一致

## 它是怎么解决问题的？

### 2PC：两阶段提交（理论模型）

2PC 是分布式事务的起点——引入**协调者（Coordinator）**，让多个参与者共同决定提交还是回滚。

```
    协调者（Coordinator）
         │
    ┌────┼────────┬────────┐
    │    │        │        │
    ▼    ▼        ▼        ▼
    TM1  TM2      TM3      TM4  （参与者 / 事务管理器）
```

**阶段一：Prepare（投票阶段）**

```
协调者 → 所有参与者：Can you commit?
参与者1 → 执行事务（不提交），写 Undo/Redo 日志，返回 Yes
参与者2 → 执行事务（不提交），返回 Yes
参与者3 → 执行失败，返回 No
```

**阶段二：Commit / Rollback（提交阶段）**

```
所有参与者都返回 Yes → 协调者发送 Commit → 所有参与者提交
有任何参与者返回 No → 协调者发送 Rollback → 所有参与者回滚
```

```java
// 2PC 伪代码（协调者视角）
public class TwoPhaseCommit {
    public void commit(Transaction tx) {
        // Phase 1: Prepare
        Map<Participant, Boolean> votes = new HashMap<>();
        for (Participant p : tx.participants) {
            votes.put(p, p.prepare(tx));
        }

        // Phase 2: Commit or Rollback
        if (votes.values().stream().allMatch(v -> v == true)) {
            for (Participant p : tx.participants) {
                p.commit(tx);
            }
        } else {
            for (Participant p : tx.participants) {
                p.rollback(tx);
            }
        }
    }
}
```

**致命缺陷**：
1. **同步阻塞**：Prepare后，资源被锁住，等待协调者指令
2. **单点故障**：协调者挂了，参与者永远等待
3. **数据不一致**：阶段二部分 Commit 成功、部分失败
4. **无法彻底解决**：协调者宕机 + 参与者也宕机 → 无法判断应该提交还是回滚

### 3PC：引入超时机制

3PC 在 2PC 的基础上加了 `CanCommit` 阶段（询问是否能提交，不执行业务逻辑），并引入超时机制：

```
CanCommit → PreCommit → DoCommit
   │           │           │
 询问        准备        提交
（无锁）    （持锁）   （执行）
```

如果参与者在 `PreCommit` 后超时没收到 `DoCommit`，**默认提交**。减少了协调者故障后的阻塞，但**网络分区下仍可能不一致**。

> 结论：**3PC 没有完全解决 2PC 的问题**，实际生产中很少直接使用。了解即可。

### TCC：业务层面的两阶段

TCC 不依赖数据库的事务机制，而是由**业务逻辑**实现两阶段：

```
Try 阶段：预留资源（如冻结库存 10 件，而不是扣减 10 件）
Confirm 阶段：确认预留（冻结 → 实际扣减）
Cancel 阶段：取消预留（冻结 → 释放回原库存）
```

```java
// 电商下单的 TCC 实现
public class OrderTccService {

    @Transactional
    public void tryCreateOrder(Order order) {
        // Try：插入"预留状态"的订单，不真正生效
        order.setStatus("PENDING");
        orderMapper.insert(order);
        // 冻结库存（不是真正扣减）
        inventoryMapper.freeze(order.getProductId(), order.getAmount());
    }

    @Transactional
    public void confirmCreateOrder(Order order) {
        // Confirm：订单生效，扣减真实库存
        order.setStatus("CONFIRMED");
        orderMapper.updateStatus(order);
        inventoryMapper.deduct(order.getProductId(), order.getAmount());
    }

    @Transactional
    public void cancelCreateOrder(Order order) {
        // Cancel：订单取消，释放冻结库存
        order.setStatus("CANCELLED");
        orderMapper.updateStatus(order);
        inventoryMapper.unfreeze(order.getProductId(), order.getAmount());
    }
}
```

**为什么 TCC 比 2PC 好**：2PC 是资源层（数据库锁），TCC 是业务层（逻辑预留），持锁时间短。

### 本地消息表 + MQ：最终一致性

```
本地事务：
  1. 业务操作（如创建订单）
  2. 写消息表（订单ID + 消息内容）【同一个本地事务】

后台任务：
  扫描消息表 → 发送到 MQ → 收到 ACK 后删除消息表记录

消费者：
  消费消息 → 执行对应业务（如扣减库存）→ 确认消费
```

**关键**：消息表和业务操作在同一个数据库事务里，保证"业务做了，消息一定发出去"。消费者需要**幂等**（同一条消息消费多次，结果一样）。

### Saga 模式：长流程事务

Saga 适合**流程长、无法冻结资源**的场景——每个操作都有对应的补偿操作：

```
订单创建 → 扣库存 → 扣余额 → 通知用户
   │         │
   └─── 失败 → 反向补偿 ───────────────┘
             Cancel库存  Cancel余额
```

```java
// Saga：每个操作都有对应的补偿操作
public class OrderSaga {
    public void execute(Order order) {
        try {
            createOrder(order);       // 正向操作1
            deductInventory(order);   // 正向操作2
            deductBalance(order);     // 正向操作3
            notifyUser(order);        // 正向操作4
        } catch (Exception e) {
            // 失败，按相反顺序补偿
            compensateNotifyUser(order);
            compensateBalance(order);
            compensateInventory(order);
            compensateOrder(order);
        }
    }
}
```

## 深入原理

### 方案对比总览

| 方案 | 一致性 | 性能 | 复杂度 | 适用场景 |
|---|---|---|---|---|
| XA（2PC） | 强一致 | 低（同步阻塞） | 低（数据库内置） | 单体应用多数据源 |
| 3PC | 强一致（理论上） | 低 | 中 | 几乎不用 |
| TCC | 最终一致 | 中 | 高（需写三个接口） | 金融、扣款等强一致需求 |
| Saga | 最终一致 | 高 | 中（需写补偿） | 长流程业务（旅游、电商下单） |
| 本地消息表 + MQ | 最终一致 | 高 | 中 | 对一致性要求不高的业务 |
| AT（自动补偿，Seata） | 最终一致 | 中 | 低 | 希望无侵入的分布式事务 |

**本质区别**：
- **2PC/3PC**：资源层一致，但性能差，实际很少用
- **TCC**：业务层一致，性能好，但侵入业务代码
- **Saga**：补偿机制，适合长事务
- **事务消息**：主流方案，MQ 保证半消息 + 回查

### Seata 的 AT 模式

Seata 通过**拦截 JDBC 操作**，自动生成反向补偿 SQL：

```
业务 SQL：UPDATE inventory SET count = count - 10 WHERE id = 1
Seata 自动记录前镜像：count = 100
Seata 自动记录后镜像：count = 90
回滚时：UPDATE inventory SET count = 100 WHERE id = 1
```

对业务代码零侵入，但依赖 Seata Server（TC——事务协调器）。

## 正确使用方式

### TCC 接口设计

```java
public interface InventoryTccService {
    @TwoPhaseBusinessAction(name = "deduct", commitMethod = "confirm", rollbackMethod = "cancel")
    boolean tryDeduct(BusinessActionContext context, Long orderId, Integer count);

    boolean confirm(BusinessActionContext context);  // 确认：冻结 → 扣减
    boolean cancel(BusinessActionContext context);    // 取消：冻结 → 释放
}
```

### RocketMQ 事务消息

```java
// RocketMQ 事务消息：半消息 + 本地事务 + 回查
public class OrderTransactionProducer {

    public void createOrder(Order order) {
        // 1. 发送半消息（对消费者不可见）
        TransactionSendResult result = producer.sendMessageInTransaction(
                new Message("order_topic", order.toString().getBytes()),
                order
        );
        // 2. 执行本地事务（创建订单）
        // 3. 根据本地事务结果，提交或回滚半消息
    }

    // 本地事务执行状态（MQ 会回查这个方法）
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        String orderId = msg.getKeys();
        Order order = orderMapper.selectById(orderId);
        if (order == null) return ROLLBACK;
        if (order.getStatus().equals("CONFIRMED")) return COMMIT;
        return UNKNOW;  // 继续回查
    }
}
```

### 幂等消费（消息队列方案必须）

```java
@KafkaListener(topics = "inventory-deduct")
public void consume(ConsumerRecord<String, String> record) {
    String msgId = record.key();  // 用消息 ID 做幂等
    if (redisTemplate.opsForValue().setIfAbsent("msg:" + msgId, "1")) {
        process(record.value());  // 第一次消费，执行业务
    }
    // 重复消息，直接 ACK
}
```

## 边界情况和坑

### 坑1：TCC 的空回滚和悬挂

```
空回滚：Try 还没执行，Cancel 先到了（网络超时重试）。Cancel 需要判断 Try 是否执行过，没执行过也要能处理。
悬挂：Cancel 比 Try 先执行（网络严重超时）。Try 执行时，发现已经被 Cancel 了，不能再执行。
```

**解决方案**：
1. 记录事务状态表（Try、Confirm、Cancel 都写日志）
2. 空回滚：Cancel 时检查 Try 是否执行，没执行就记录"空回滚"标记
3. 悬挂：Try 时检查是否已经有 Cancel 记录，有就释放资源直接返回

### 坑2：Saga 的补偿失败

```
场景：正向操作执行到第三步失败了，开始补偿，但补偿也失败
结果：数据处于"部分补偿"状态，不一致
```

**解决方案**：
1. **补偿操作必须幂等**（可以重试）
2. **记录补偿状态**（哪些补偿成功了，哪些失败了）
3. **人工介入**（补偿失败超过阈值，告警人工处理）

### 坑3：2PC 协调者宕机

```
协调者在阶段一之后宕机：
  → 所有参与者都停留在 "等待协调者指令" 状态
  → 资源（数据库连接）一直不释放
  → 需要人工介入，或者新的协调者从日志恢复状态
```

Seata 的 TC 通过**事务日志持久化**解决这个问题：新 TC 启动后，从日志恢复所有未完成事务的状态。

### 坑4：MQ 消息重复发送

```
生产者发送消息 → Broker 写入成功 → 网络超时 → 生产者重试 → 同一条消息发送两次
```

消费者必须幂等。常见方案：
1. 数据库唯一约束（推荐）
2. Redis `SET NX`（设置过期时间）
3. 状态机判断（已处理过的状态不再处理）
