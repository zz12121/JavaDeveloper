# 架构评审Checklist

> 架构评审不是"批斗大会"，而是提前发现问题、对齐认知的机制。好的 Checklist 能让评审效率提升 3 倍。

---

## 一、评审前的准备

```
提交者准备：
□ 技术方案文档（必须！口头讲不清）
□ 架构图（现状 + 目标）
□ 核心流程时序图
□ 性能基线数据
□ 风险评估表
□ 实施计划（分阶段 + 回滚方案）

评审者准备：
□ 提前 1 天读完方案文档
□ 列出疑问点
□ 准备类似场景的经验
```

## 二、评审 Checklist（按维度）

### 2.1 功能完整性

```
□ 是否覆盖了所有功能需求？
□ 边界场景是否考虑？（空值、最大值、并发）
□ 异常处理是否周全？（超时、失败、降级）
□ 接口定义是否清晰？（入参/出参/错误码）
□ 幂等性是否保证？（重试场景）
```

### 2.2 性能

```
□ 是否有性能目标和基线？
□ 数据库查询是否会有全表扫描？（看执行计划）
□ 是否有缓存方案？（缓存穿透/击穿/雪崩如何应对）
□ 是否有消息队列削峰？（突发流量）
□ 是否有压测方案？（上线前的压测计划）
```

### 2.3 可用性

```
□ 是否有单点故障？（所有组件都要有高可用方案）
□ 降级方案是什么？（依赖服务挂了怎么办？）
□ 限流方案是什么？（超出容量怎么保护？）
□ 熔断方案是什么？（下游慢了怎么保护上游？）
□ 监控告警是否完善？（出了问题能及时发现吗？）
```

### 2.4 一致性

```
□ 分布式事务如何处理？（AT/TCC / SAGA / XA？）
□ 数据库和缓存的一致性如何保证？（删除缓存 vs 更新缓存？）
□ 消息队列的消息丢失/重复如何处理？
□ 多副本一致性如何保证？（Raft/Paxos？）
```

### 2.5 安全性

```
□ 是否有身份认证和鉴权？（JWT/OAuth2？）
□ 敏感数据是否加密？（密码/手机号/身份证）
□ 是否有 SQL 注入 / XSS 防护？
□ 是否有请求签名/防重放？（支付场景）
□ 日志是否可能泄露敏感信息？
```

### 2.6 可维护性

```
□ 代码是否有单元测试？（覆盖率目标？）
□ 是否有完整的 API 文档？
□ 是否有运维手册？（部署/扩容/故障处理）
□ 关键配置是否外置？（配置中心）
□ 是否考虑了版本兼容？（向前/向后兼容）
```

## 三、各类型方案的评审重点

### 3.1 性能优化方案

```
必问：
1. 现状性能基线是多少？（QPS？延迟？）
2. 目标性能是多少？
3. 瓶颈定位依据是什么？（火焰图？慢查询日志？）
4. 优化后如何验证？（压测对比）
5. 降级方案是什么？（优化失败怎么办？）
```

### 3.2 数据迁移方案

```
必问：
1. 数据量多大？迁移窗口多久？
2. 迁移期间双写一致性如何保证？
3. 回滚方案是什么？（迁移失败怎么恢复？）
4. 迁移失败的告警机制？
5. 验证数据一致性的工具？
```

### 3.3 新引入中间件方案

```
必问：
1. 单机 QPS 能到多少？（官方数据 vs 实际压测）
2. 集群方案是什么？（主从？分片？）
3. 监控方案是什么？（关键指标有哪些？）
4. 故障恢复 RTO 是多少？
5. 团队有没有人会运维？
```

## 四、评审会议的高效组织

### 4.1 会前

```
□ 方案文档提前 2 天发出
□ 邀请相关人（不要太多，5 人以内最好）
□ 标注「需要重点讨论的部分」
□ 准备 Demo 或原型（如果有）
```

### 4.2 会中

```
时间盒：最长 1 小时
主持人：控制节奏，不要跑题
记录员：记录所有 Action Item
决策：每个讨论点都要有结论

讨论顺序：
1. 背景和目标（5 分钟）
2. 方案设计（20 分钟）
3. 风险评估（15 分钟）
4. 实施计划（10 分钟）
5. 行动项汇总（10 分钟）
```

### 4.3 会后

```
□ 24 小时内发出会议纪要
□ 每个 Action Item 都有负责人和截止日期
□ 下次会议前检查 Action Item 完成情况
```

## 五、评审中的常见陷阱

| 陷阱 | 正确做法 |
|------|---------|
| 陷入语法细节 | 关注架构决策，语法问题代码 Review 时看 |
| 过度设计（YAGNI） | 问：这个功能 6 个月内需要吗？ |
| 忽视运维成本 | 每个方案都要问：谁来运维？出问题怎么办？ |
| 没有 baseline | 优化方案必须有基线数据 |
| 没有回滚方案 | 任何有风险的变更都要有回滚方案 |

## 六、评审意见的分类

```
Blocker（必须修改后才能推进）：
- 安全风险
- 数据一致性问题
- 性能目标明显达不到

Major（强烈建议修改）：
- 可用性隐患
- 扩展性差
- 严重的技术债

Minor（可以后续优化）：
- 代码风格
- 非核心路径的性能优化
- 文档完善度

Nit（可选）：
- 命名建议
- 注释补充
```
