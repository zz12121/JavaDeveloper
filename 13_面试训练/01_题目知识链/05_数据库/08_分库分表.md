## 知识链提示

```
分库分表
  → [[06_数据库/04_SQL优化/分库分表方案]]
    → 什么时候需要分
      → 单表数据量 > 500万~2000万（经验值，取决于硬件和索引设计）
      → 数据库成为性能瓶颈（CPU/IO/连接数）
      → 单机存储容量不够
    → 两种拆分方式
      → 垂直拆分（按列/按业务拆分）
        → 垂直分表：把大字段（text/blob）拆到扩展表
        → 垂直分库：把不同业务模块拆到不同库（订单库/用户库）
        → 解决：单机容量瓶颈、业务耦合
      → 水平拆分（按行拆分）
        → 水平分表：同一业务数据按规则拆到多个表
        → 水平分库：同一业务数据按规则拆到多个库
        → 解决：单表数据量过大
    → 分片规则
      → 范围分片：按时间/ID 范围（好扩容，但可能数据倾斜）
      → 哈希分片：hash(shard_key) % N（数据均匀，但扩容麻烦）
      → 一致性哈希：解决扩容时大量数据迁移
    → 分库分表的问题
      → 分布式事务：[[04_Spring生态/03_SpringCloud/分布式事务]] → Seata / TCC / 最终一致性
      → 跨分片查询：要查多个分片，再聚合（性能差）
      → 跨分片 JOIN：基本不可能，要在应用层做
      → 扩容：数据迁移成本高
```

## 参考答案要点

**垂直 vs 水平**：
- 垂直：按「列/业务」拆 → 解决容量/耦合问题
- 水平：按「行」拆 → 解决单表数据量问题

**分片键选择原则**：
- 尽量选高频查询条件（避免全分片扫描）
- 尽量数据均匀分布（避免倾斜）
- 尽量不变（避免数据迁移）

**分库分表是大招，不要轻易用**——先考虑：加索引 / 读写分离 / 升级硬件。
