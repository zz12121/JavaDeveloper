# 泛型

> Java 5 引入泛型，用编译期类型检查替代运行期的强制转换，让集合框架等通用组件既能保证类型安全，又能保持与旧字节码的兼容。核心手段是**类型擦除**——编译后泛型信息消失，变成 Object 或边界类型。

---

## 这个问题为什么存在？

Java 1.4 及之前，集合框架用 `Object` 接收所有类型，取出来要强制转换，编译期发现不了类型错误，运行时才抛 `ClassCastException`。

```java
// Java 1.4 风格：没有任何类型检查
List list = new ArrayList();
list.add("hello");
list.add(123);  // 编译通过，运行时才发现不对

Integer i = (Integer) list.get(0);  // ClassCastException 运行时才爆
```

**核心问题**：怎么让集合在**编译期**就约束类型，同时又能**兼容旧代码**（旧字节码里没有泛型信息）？

泛型的解决思路是**编译期类型检查 + 运行期类型擦除**——编译时保证类型安全，运行时擦掉类型参数保持和旧字节码兼容。

---

## 它是怎么解决问题的？

### 泛型擦除（Type Erasure）

```
// 编译期：编译器检查类型
List<String> list = new ArrayList<>();
list.add("hello");
list.add(123);   // ❌ 编译错误

// 编译后（字节码）：类型擦除，变成 Object
List list = new ArrayList();
list.add("hello");   // 实际调用 add(Object)
String s = (String) list.get(0);  // 编译器自动插入强转
```

**擦除规则**：
- `<T>` 无边界 → 擦除为 `Object`
- `<T extends Number>` 有边界 → 擦除为 `Number`
- `<T extends Comparable & Serializable>` 多个边界 → 擦除为**第一个边界**（`Comparable`）

**为什么用擦除而不是具体化泛型？**
- Java 5 发布时，已经有大量使用 `List`、`ArrayList` 的旧字节码
- 如果改成具体化泛型（像 C#），旧字节码全部不兼容
- 擦除方案让 `List<String>` 和 `List` 在运行期是**同一个类**，完美兼容

### 桥方法（Bridge Method）

泛型擦除后，多态会遇到问题：

```
// 编译前
interface Comparable<T> {
    int compareTo(T o);
}

class MyString implements Comparable<String> {
    int compareTo(String o) { ... }  // 参数是 String
}

// 擦除后
interface Comparable {
    int compareTo(Object o);  // 擦除为 Object
}

// MyString 类必须有方法：compareTo(Object)
// 但程序员写的是 compareTo(String)
// → 编译器自动生成桥方法！
```

**桥方法（`javap -c MyString.class` 可以看到）**：

```java
// 编译器自动生成
public int compareTo(Object o) {
    return this.compareTo((String) o);  // 转发给实际的方法
}
```

桥方法让**泛型子类的重写**在擦除后仍然正确工作。没有桥方法，`MyString` 就没有符合擦除后签名的方法，多态会失效。

### 为什么禁止 `new T()` 和 `new T[]`？

```java
public class MyContainer<T> {
    private T value = new T();      // ❌ 编译错误
    private T[] elements = new T[10];  // ❌ 编译错误
}
```

**原因**：运行期 `T` 已经被擦除了，JVM 不知道 `T` 是什么类型，无法实例化或分配数组。

```
擦除后：T → Object
new T[10] → new Object[10] ?

但如果 T 实际上是 String，Object[] 不是 String[]，类型不安全！
```

**解决方案**：
```java
// 方案 1：传 Class<T>
public MyContainer(Class<T> clazz) {
    this.arr = (T[]) Array.newInstance(clazz, 10);
}

// 方案 2：用 Object 数组，取时强转
private Object[] arr = new Object[10];
@SuppressWarnings("unchecked")
public T get(int i) { return (T) arr[i]; }
```

### 泛型数组的本质困难

```java
List<String>[] arr = new List<String>[10];  // ❌ 禁止
```

**原因**：数组是**协变且具体化的**（run-time 检查类型），但泛型是**不变且擦除的**（compile-time 检查）。

```java
List<String>[] arr = new List[10];  // ✅ 可以，但会有警告
arr[0] = new ArrayList<Integer>();  // ❌ 运行时才发现类型不匹配？
String s = arr[0].get(0);  // ClassCastException
```

因为数组存储检查只看 `List`（擦除后的类型），不知道里面的泛型参数，所以 JVM 无法在运行时正确检查类型。Java 干脆禁止创建泛型数组，把错误提前到编译期。

## 深入原理

### 类型擦除的代价

擦除让 Java 泛型有了「表达力折扣」——有些在 C# 里能写的东西，Java 写不了：

```java
// ❌ 不能 T.class（运行期 T 被擦除）
public class MyClass<T> {
    Class<T> clazz = T.class;  // 编译错误
}

// ❌ 不能 instanceof T（运行期 T 被擦除）
if (obj instanceof T) { }  // 编译错误

// ❌ 不能重载（擦除后方法签名相同）
public void print(List<String> list) { }
public void print(List<Integer> list) { }  // 编译错误
```

**为什么 C# 没有这些问题？**
- C# 的泛型是**具体化的**：每个泛型类型参数在运行期保留，`List<string>` 和 `List<int>` 是两个不同的类型
- 代价是 C# 1.0 没有像 Java 那样的历史包袱，可以从头设计

### 通配符与 PECS 原则

```java
// 生产者（读取）：用 ? extends T
public void copy(List<? extends Number> src, List<Number> dst) {
    for (Number n : src) dst.add(n);  // ✅ 可以读取为 Number
    src.add(1);  // ❌ 不能写入（不知道具体是 Integer 还是 Double）
}

// 消费者（写入）：用 ? super T
public void addNumbers(List<? super Integer> list) {
    list.add(1);  // ✅ 可以写入 Integer
    Integer i = list.get(0);  // ❌ 读取只能当 Object
}
```

**PECS**（Producer Extends, Consumer Super）：
- 如果集合**只生产**（读取），用 `? extends T`
- 如果集合**只消费**（写入），用 `? super T`

这是 `Collections.copy()`、`Collections.sort()` 等标准库方法的签名设计依据。

### 类型推断（Java 7+）

```java
// Java 7 引入钻石操作符
List<String> list = new ArrayList<>();  // ✅ 编译器推断类型

// 泛型方法类型推断
public <T> T getMiddle(List<T> list) {
    return list.get(list.size() / 2);
}

String s = getMiddle(Arrays.asList("a", "b", "c"));
// ✅ 编译器根据实参推断出 T = String，无需显式指定
```

Java 8 进一步增强了目标类型推断（[[01_Java核心/08_Lambda与Stream/Lambda与函数式|Lambda 类型推断]]），让泛型方法调用更简洁。

## 正确使用方式

### 泛型方法

```java
// 普通泛型类
class Box<T> {
    private T value;
    public T get() { return value; }
}

// 泛型方法（不依赖类的泛型参数）
public <T> T getMiddle(List<T> list) {
    return list.get(list.size() / 2);
}

// 多个泛型参数
public <K, V> Map<K, V> createMap(K key, V value) {
    Map<K, V> map = new HashMap<>();
    map.put(key, value);
    return map;
}
```

泛型方法的类型参数写在返回值之前，调用时通常可以省略（编译器推断）。

### 上界和下界的正确使用

```java
// 上界：只能接收 Number 或其子类（读取安全）
public double sum(List<? extends Number> list) {
    return list.stream().mapToDouble(Number::doubleValue).sum();
}

// 下界：只能接收 Integer 或其父类（写入安全）
public void addIntegers(List<? super Integer> list) {
    list.add(1);
    list.add(2);
}
```

**常见错误**：
```java
// ❌ 同时用 extends 和 super，没有任何灵活性收益
public void process(List<? extends ? super Integer> list) { }  // 语法错误

// ✅ 如果既要读又要写，不要用通配符
public void process(List<Integer> list) {
    list.add(1);
    Integer i = list.get(0);
}
```

### 泛型边界（Bounded Type Parameters）

```java
// 单边界
public <T extends Comparable<T>> T max(T a, T b) {
    return a.compareTo(b) > 0 ? a : b;
}

// 多边界：第一个是类（只能有一个类），后面是接口
public <T extends Number & Comparable<T>> void process(T value) {
    double d = value.doubleValue();
    int cmp = value.compareTo(null);  // 编译期保证有 compareTo 方法
}
```

多边界时，擦除为**第一个边界**（`Number`），所以类应该放在第一位（只能有一个类，放前面效率更高）。

---

## 边界情况和坑

### 坑 1：不能 `T.class`

```java
public class MyClass<T> {
    Class<T> clazz = T.class;  // ❌ 擦除后 T 不存在
}
```

**成因**：运行期 `T` 已经被擦除为 `Object`（或有界类型），JVM 不知道 `T` 的具体类型。

**解决方案**：传 `Class<T>` 进来：
```java
public MyClass(Class<T> clazz) {
    this.clazz = clazz;  // ✅ 保留类型信息
}
```

### 坑 2：不能 `instanceof T`

```java
if (obj instanceof T) { }  // ❌ 运行期 T 被擦除了
```

**解决方案**：用边界类型：
```java
// 如果 T extends Number
if (obj instanceof Number) {  // ✅ 用边界类型检查
    Number n = (Number) obj;
}
```

### 坑 3：桥方法导致反射时的「重复方法」

```java
for (Method m : MyString.class.getDeclaredMethods()) {
    System.out.println(m);
    if (m.isBridge()) {
        System.out.println("  ← 这是桥方法");
    }
}
```

**成因**：编译器为泛型子类生成了桥方法，反射时能看到两个 `compareTo` 方法：
- `compareTo(String)` — 程序员写的
- `compareTo(Object)` — 编译器生成的桥方法

**影响**：手写反射逻辑时，需要过滤掉桥方法（`m.isBridge()`），否则可能重复处理。

### 坑 4：泛型与数组的混用

```java
List<String>[] arr = new List[10];  // ⚠️ 编译警告，但能运行
arr[0] = new ArrayList<String>();   // ✅
arr[0] = new ArrayList<Integer>();  // ⚠️ 编译警告，运行期才发现
```

**成因**：数组是协变的，`ArrayList<String>` 和 `ArrayList<Integer>` 都能赋给 `List` 类型（擦除后的类型），JVM 无法在存储时检查泛型参数。

**解决方案**：用 `List<List<String>>` 替代 `List<String>[]`，彻底避免数组与泛型的混用。

### 坑 5：原始类型（Raw Type）绕过类型检查

```java
List<String> strings = new ArrayList<>();
List raw = strings;  // ⚠️ 原始类型，编译警告
raw.add(123);         // ⚠️ 编译警告，但能运行
String s = strings.get(0);  // ClassCastException！
```

**成因**：原始类型是为了兼容 Java 1.4 之前的代码而保留的，使用原始类型会绕过所有泛型类型检查。

**解决方案**：永远不要使用原始类型，如果类型参数不确定，用 `List<?>`（无界通配符）替代。
