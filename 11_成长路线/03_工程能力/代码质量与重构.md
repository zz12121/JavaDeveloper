# 代码质量与重构

> 代码是写给人看的，只是顺便让机器执行。代码质量决定技术债的积累速度。

---

## 一、代码质量的衡量标准

### 1.1 可维护性指标

| 指标 | 工具 | 好的标准 |
|------|------|---------|
| 圈复杂度（Cyclomatic Complexity） | SonarQube | 方法 < 10，类 < 200 |
| 代码行数（LOC/Method） | Checkstyle | 方法 < 50 行 |
| 嵌套深度 | PMD | < 4 层 |
| 注释覆盖率 | Javadoc | 公共 API 100% |
| 重复率 | SonarQube | < 5% |

### 1.2 可读性标准

```
好的代码：
✅ 变量名自解释（calculateOrderTotalDays，而不是 calOD)
✅ 方法短小（一个方法只做一件事）
✅ 少嵌套（提前 return，减少 else）
✅ 无魔法数字（用常量 MAX_RETRY_COUNT）
✅ 异常处理清晰（不吞异常，不捕获 Exception）

坏的代码：
❌ 一个方法 200 行
❌ 嵌套 6 层 if-else
❌ 变量名 a, b, c, data, list1
❌ 吞掉异常（catch 空块）
❌ 一个类 3000 行
```

## 二、SOLID 原则（设计原则）

### 2.1 单一职责原则（SRP）

```
❌ 坏例子：
class UserService {
    void register() { /* 注册逻辑 */ }
    void sendEmail() { /* 发邮件 */ }
    void writeLog() { /* 写日志 */ }
}

✅ 好例子：
class UserService { void register() { ... } }
class EmailService { void sendEmail() { ... } }
class LogService { void writeLog() { ... } }
```

### 2.2 开闭原则（OCP）

```
❌ 坏例子：
class PaymentService {
    void pay(String type) {
        if ("alipay".equals(type)) { ... }
        else if ("wechat".equals(type)) { ... }
    }
}

✅ 好例子（策略模式）：
interface PaymentStrategy { void pay(); }
class AlipayStrategy implements PaymentStrategy { ... }
class WechatStrategy implements PaymentStrategy { ... }

class PaymentService {
    void pay(PaymentStrategy strategy) {
        strategy.pay();
    }
}
```

### 2.3 里氏替换原则（LSP）

```
❌ 坏例子：
class Bird { void fly() {} }
class Ostrich extends Bird { 
    void fly() { throw new UnsupportedOperationException(); } // 鸵鸟不会飞！
}

✅ 好例子：
interface Bird {}
interface Flyable { void fly(); }
class Sparrow implements Bird, Flyable {}
class Ostrich implements Bird {}
```

### 2.4 接口隔离原则（ISP）

```
❌ 坏例子：
interface Worker {
    void work();
    void eat();  // 机器人不需要 eat！
}

✅ 好例子：
interface Workable { void work(); }
interface Eatable { void eat(); }
class Human implements Workable, Eatable {}
class Robot implements Workable {}
```

### 2.5 依赖倒置原则（DIP）

```
❌ 坏例子：
class OrderService {
    private MySQLDao dao = new MySQLDao(); // 依赖具体
}

✅ 好例子：
class OrderService {
    private OrderDao dao;  // 依赖抽象
    OrderService(OrderDao dao) { this.dao = dao; }
}
```

## 三、重构手法（Martin Fowler）

### 3.1 重新组织函数

| 手法 | 说明 | 示例 |
|------|------|---------|
| 提炼函数（Extract Method） | 把一段代码抽成独立函数 | 长方法拆短 |
| 内联函数（Inline Method） | 函数体太简单，直接内联 | 去掉无意义的一行包装函数 |
| 内联临时变量 | 临时变量只赋值一次，直接用表达式 | `int t = order.getTotal();` → 直接用 `order.getTotal()` |
| 以查询取代临时变量 | 临时变量 → 无参方法 | 让多个方法共享计算结果 |
| 引入解释性变量 | 复杂表达式 → 有意义的变量名 | `if ((a > b) && (c < d))` → `boolean isInRange = ...` |

### 3.2 在对象之间搬移特性

| 手法 | 说明 |
|------|------|
| 搬移函数（Move Method） | 函数在哪个类用得最多，就搬到哪里 |
| 搬移字段（Move Field） | 字段在哪个类用得最多，就搬到哪里 |
| 提炼类（Extract Class） | 一个类职责太多 → 拆成多个类 |
| 内联类（Inline Class） | 类太小 → 合并到使用的类 |

### 3.3 重新组织数据

| 手法 | 说明 |
|------|------|
| 自封装字段（Self Encapsulate Field） | 通过 getter/setter 访问字段 |
| 以对象取代数据值 | 纯数据 → 对象（带行为） |
| 以常量取代魔法数 | `3.14` → `static final PI = 3.1415926` |

## 四、代码 Review 要点

### 4.1 提交者 Checklist

```
□ 代码能编译（没有 // TODO 遗留）
□ 有单元测试（覆盖率达标）
□ 方法不超过 50 行
□ 圈复杂度不超过 10
□ 没有魔法数字
□ 没有硬编码（配置放配置文件）
□ 异常处理得当（不吞异常）
□ 日志关键路径有记录
□ 没有 SQL 拼接（防注入）
```

### 4.2 Reviewer Checklist

```
□ 变量/方法命名是否清晰？
□ 是否有重复代码可以抽取？
□ 是否有过度设计（YAGNI）？
□ 异常处理是否合适？
□ 线程安全是否保证？
□ 边界条件是否考虑？
□ 性能是否有明显问题？
□ 是否有 SQL N+1 问题？
```

## 五、技术债管理

```
技术债分类：
- 鲁莽而不经意的债（❌ 最差）
- 鲁莽而刻意的债（❌ 故意走捷径）
- 谨慎而不经意的债（✅ 当时不知道更好的做法）
- 谨慎而刻意的债（✅ 为了赶工期，有计划地还）

技术债记录模板：
- 位置：xxxService.java:58
- 问题：方法圈复杂度 18，难以测试
- 后果：修改容易引入 Bug
- 还债优先级：High
- 预计工作量：2 人天
```

## 六、重构的安全网

```
重构前必须有：
1. 单元测试（覆盖率 ≥ 70%）
2. 集成测试（核心流程覆盖）
3. CI 流水线（自动跑测试）

重构策略：
- 小步重构（每次改动都能编译、能跑测试）
- 绞杀者模式（Strangler Fig Pattern）
  → 新功能用新方式写，旧功能逐步迁移
```
