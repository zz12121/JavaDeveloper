# 注解

> 注解是给代码元素（类、方法、字段）附加**结构化元数据**的标准化机制。它把"这个类是 Controller""这个方法需要缓存"这类意图直接写在代码上，由编译器或框架在后续阶段读取并执行。

---

## 这个问题为什么存在？

在注解出现之前，Java 程序的元数据主要通过三种方式表达：

1. **XML 配置文件**：Spring、Hibernate 大量使用 XML，导致"配置地狱"——配置和代码分离，改了逻辑忘了改配置
2. **标记接口**：如 `Serializable`，只能标记类、不能携带参数，无法表达"这个方法需要事务"这种细粒度意图
3. **命名约定**：如 JUnit 3 要求方法名以 `test` 开头，没有编译期检查，写错方法名也不会报错

核心矛盾：**程序需要一种既能附着到任意代码元素、又能携带参数、还能在编译期或运行时被读取的元数据机制。** 注解就是为解决这个矛盾而生的。

---

## 它是怎么解决问题的？

### 三层架构：定义 → 读取 → 处理

注解不是"主动执行"的，它的工作流分三层：

```
┌──────────────────────────────────────────────────────────┐
│                    1. 定义层                               │
│                                                          │
│  @Target(METHOD)       @Retention(RUNTIME)               │
│  @interface Cacheable {                                   │
│      String key() default "";                             │
│      long ttl() default 60000;                            │
│  }                                                        │
│  → 注解本身只是接口，参数实际上是方法声明                      │
└────────────────────────┬─────────────────────────────────┘
                         │ 编译器写入字节码
                         ▼
┌──────────────────────────────────────────────────────────┐
│                    2. 读取层                               │
│                                                          │
│  Method method = clazz.getMethod("findById");             │
│  Cacheable ann = method.getAnnotation(Cacheable.class);   │
│  String key = ann.key();    // "user:1"                   │
│  → JVM 通过动态代理生成注解接口的实例                        │
└────────────────────────┬─────────────────────────────────┘
                         │ 框架拿到元数据
                         ▼
┌──────────────────────────────────────────────────────────┐
│                    3. 处理层                               │
│                                                          │
│  cache.put(key, method.invoke(target), ann.ttl());        │
│  → 注解只是触发器，真正逻辑在框架/工具中                     │
└──────────────────────────────────────────────────────────┘
```

### 元注解：控制注解行为的注解

```java
@Target(ElementType.METHOD)         // 能用在什么地方
@Retention(RetentionPolicy.RUNTIME) // 在哪个阶段可见
@Documented                         // 出现在 Javadoc 中
@Inherited                          // 子类能否继承
public @interface MyAnnotation {
    String value() default "";      // 特殊参数名，单参数时可省略 key=
    int count() default 0;
}
```

#### @Retention：三种保留策略

| 策略 | 可见范围 | 原理 | 典型场景 |
|------|---------|------|---------|
| `SOURCE` | 仅源码，编译后丢弃 | 编译器处理完就丢弃，不进字节码 | `@Override`、`@SuppressWarnings` |
| `CLASS` | 字节码中，运行时不可读 | 写入 class 文件的属性表，但 JVM 不加载 | 字节码增强工具（AspectJ） |
| `RUNTIME` | 字节码 + 运行时可反射读取 | JVM 加载类时解析到 `AnnotationData` | `@Test`、`@Autowired`、`@Transactional` |

为什么需要三种而不是两种？`CLASS` 策略的存在是因为有些工具（如 AspectJ）需要在字节码层面做增强，但不希望这些元数据在运行时被任意反射读取——这是一种**信息可见性的精细控制**。

#### @Target：11 种作用目标

| 目标 | 示例 |
|------|------|
| `TYPE` | `@Entity public class User` |
| `FIELD` | `@Id private Long id` |
| `METHOD` | `@Test public void shouldReturn()` |
| `PARAMETER` | `public void find(@PathParam String name)` |
| `CONSTRUCTOR` | `@Inject public UserService()` |
| `LOCAL_VARIABLE` | 局部变量（Lombok 的 `@Cleanup`） |
| `ANNOTATION_TYPE` | 元注解（`@Target` 本身） |
| `PACKAGE` | `package-info.java` 中的 `@Deprecated` |
| `TYPE_PARAMETER` | `class Box<@NotNull T>` （Java 8+） |
| `TYPE_USE` | `List<@NonNull String>` （Java 8+） |
| `RECORD_COMPONENT` | `record Point(@Range int x, @Range int y)` （Java 16+） |

### 编译器处理注解的源码路径

```java
// javac 内部（简化）：com.sun.tools.javac.processing.JavacProcessingEnvironment
public boolean processAnnotations(List<JCCompilationUnit> units) {
    // 1. 扫描所有注解，按 @Retention 分类
    // 2. 对 SOURCE 级注解：执行编译期检查，然后丢弃
    //    例：@Override 检查方法是否真的覆盖了父类方法
    // 3. 对 CLASS/RUNTIME 级注解：写入 class 文件的
    //    RuntimeVisibleAnnotations / RuntimeInvisibleAnnotations 属性表
    // 4. 调用已注册的 APT 处理器
}
```

### 运行时注解的反射读取原理

JVM 加载类时，解析字节码中的 `RuntimeVisibleAnnotations` 属性，存入 `Class` 对象的 `AnnotationData`。当调用 `getAnnotation()` 时：

```java
// java.lang.Class
public <A extends Annotation> A getAnnotation(Class<A> annotationClass) {
    // 1. 从 AnnotationData 中查找
    AnnotationData annotationData = annotationData();
    // 2. 如果找到，通过 AnnotationInvocationHandler（动态代理）生成实例
    return (A) annotationData.annotations.get(annotationClass);
}
```

注解接口的实例是一个 **`InvocationHandler` 动态代理对象**，每次调用 `ann.key()` 时，代理从内部 Map 中取出对应的值返回。这意味着注解实例本质上是一个**键值对的包装器**。

## 深入原理

### 注解 vs XML vs 标记接口：三种元数据表达的本质区别

| 维度 | 注解 | XML | 标记接口 |
|------|------|-----|---------|
| 绑定时机 | 编译期（写入字节码） | 运行期（解析文件） | 编译期（类型系统） |
| 类型安全 | 编译期检查参数类型 | 运行时才发现错误 | 编译期 `instanceof` |
| 粒度 | 任意代码元素 | 通常只对类/方法 | 只能标记类 |
| 携带信息 | 键值对参数 | 任意 XML 结构 | 无 |
| 可修改性 | 改代码 → 重编译 | 改文件 → 不编译 | 改代码 → 重编译 |

选择依据不是"哪个更好"，而是**配置的变更频率**：

- **注解**适合与代码强绑定、几乎不变的配置（`@Test`、`@Override`、`@Entity`）
- **XML** 适合需要动态修改、跨模块共享的配置（数据库连接、消息队列地址）
- **标记接口**在注解出现前是主流，现在只有 `Serializable`、`Cloneable` 等极少数因为历史兼容性而保留

### APT（Annotation Processing Tool）：编译期代码生成

注解本身不生成代码，但配合 APT 可以在编译期生成代码：

```java
@SupportedAnnotationTypes("com.example.MyAnnotation")
@SupportedSourceVersion(SourceVersion.RELEASE_17)
public class MyProcessor extends AbstractProcessor {
    @Override
    public boolean process(Set<? extends TypeElement> annotations,
                           RoundEnvironment roundEnv) {
        for (Element element : roundEnv.getElementsAnnotatedWith(MyAnnotation.class)) {
            // 读取注解参数 → 生成新的 .java 文件
            JavaFileObject file = processingEnv.getFiler()
                .createSourceFile("Generated_" + element.getSimpleName());
            try (Writer writer = file.openWriter()) {
                writer.write("public class Generated_" + element.getSimpleName() + " {}");
            }
        }
        return true;
    }
}
```

**Lombok 的原理就是 APT**：`@Data` 注解被 Lombok 的 APT 处理器读取，直接修改抽象语法树（AST），在编译产物中插入了 getter/setter/equals/hashCode/toString 的字节码。这就是为什么你的代码里没写这些方法，但编译后它们存在。

## 正确使用方式

### 定义注解的实践规范

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
@Documented
public @interface Cacheable {
    String key() default "";              // 有默认值，使用时可选
    long ttl() default 60_000;            // 用下划线分隔大数字
    CacheType type() default CacheType.LOCAL;  // 用枚举限制取值，不用 String
}
```

关键原则：
- **始终显式声明 `@Target` 和 `@Retention`**——它们是注解的使用文档
- **参数用枚举而非 String 限制取值**——编译期就能拦截非法值
- **提供合理的默认值**——大多数场景不需要指定所有参数
- **`@Documented` 让注解出现在 Javadoc 中**——API 使用者需要看到注解的含义

### 读取注解的正确姿势

```java
// 先检查再读取，避免 NullPointerException
if (method.isAnnotationPresent(Cacheable.class)) {
    Cacheable ann = method.getAnnotation(Cacheable.class);
    String key = ann.key().isEmpty() ? method.getName() : ann.key();
    cache.put(key, method.invoke(target), ann.ttl());
}
```

## 边界情况和坑

### 坑 1：注解参数不支持 null

```java
public @interface MyAnn {
    String value() default null;  // 编译错误！
}
```

**成因**：注解参数存储在字节码的 `element_value` 结构中，这是一个常量池索引，没有表示 null 的方式。

**解法**：用空字符串 `""`、空数组 `{}`、或哨兵值（`"__NULL__"`）替代。

### 坑 2：@Inherited 只对类继承有效，对接口无效

```java
@Inherited
public @interface Log { }

@Log
public interface BaseService { }

public class UserService implements BaseService { }

UserService.class.isAnnotationPresent(Log.class);  // false
```

**成因**：JLS 规定 `@Inherited` 仅影响类继承链（`extends`），不影响接口实现（`implements`）。如果需要接口上的注解也能"继承"，必须手动向上遍历：

```java
public static <A extends Annotation> A findAnnotation(Class<?> clazz, Class<A> type) {
    while (clazz != null && clazz != Object.class) {
        A ann = clazz.getAnnotation(type);
        if (ann != null) return ann;
        clazz = clazz.getSuperclass();
    }
    return null;
}
```

### 坑 3：注解参数类型受限

注解参数只能是以下 8 种类型：基本类型、`String`、`Class<?>`、枚举、其他注解、以及以上类型的数组。

不支持 `Object`、`List`、`Map`、`Date` 等复杂类型。**成因**同样是字节码常量池的限制——只有基本类型和字面量可以存入常量池。如果需要复杂参数，可以用 `String` 存储 JSON，运行时反序列化。

### 坑 4：重复注解需要 @Repeatable

Java 8 之前不能在同一元素上重复使用同一注解：

```java
// Java 8+ 写法
@Role("admin")
@Role("operator")           // 直接重复
public class User { }

// 底层机制：@Role 被收集到容器注解 @Roles 中
@Roles({@Role("admin"), @Role("operator")})
public class User { }       // 编译器自动转换
```

如果不声明 `@Repeatable(Roles.class)`，编译器不允许重复使用。这在设计需要叠加使用的注解（如校验规则、权限声明）时容易遗漏。

### 坑 5：注解实例的 equals() 可能返回 false

```java
Method m1 = A.class.getMethod("foo");
Method m2 = B.class.getMethod("foo");
MyAnn a1 = m1.getAnnotation(MyAnn.class);
MyAnn a2 = m2.getAnnotation(MyAnn.class);

a1.equals(a2);  // 可能是 false，即使参数值完全相同
```

**成因**：注解实例是动态代理对象，不同类加载器或不同上下文中可能生成不同的代理实例。**解法**：比较注解的参数值，而非注解对象本身。
