# Spring 事务管理是怎么实现的？

> ⚠️ **先盲答**：`@Transactional` 注解做了什么？它是怎么回滚的？

---

## 盲答引导

1. Spring 事务的实现方式有哪些？—— 编程式（TransactionTemplate）/ 声明式（@Transactional）
2. 事务传播行为有哪几种？—— 7 种，最常用的是 REQUIRED 和 REQUIRES_NEW
3. 什么情况下 `@Transactional` 会失效？—— 自调用 / 非 public / 异常类型不对
4. 事务隔离级别和传播行为是一回事吗？—— 完全不是

---

## 知识链提示

```
Spring 事务管理
  → [[05_持久层框架/01_MyBatis/08_事务管理]]
    → 两种实现
      → 编程式：TransactionTemplate.execute()
        → 灵活，但侵入代码
      → 声明式：@Transactional（AOP 实现）
        → 无侵入，推荐
    → 7 种传播行为（重点 3 个）
      → PROPAGATION_REQUIRED（默认）：没有事务就建一个，有就用现有的
      → PROPAGATION_REQUIRES_NEW：挂起现有事务，新建自己的事务
      → PROPAGATION_SUPPORTS：有就用，没有就以非事务方式执行
    → 事务失效的常见原因
      → 方法没标记为 public（Spring AOP 只拦截 public）
      → 自调用（同一个类里 A() 调 B()，B 有 @Transactional，但不生效）
        → 原因：Spring 事务靠 AOP 代理，自调用绕过代理
        → 解决：通过 ApplicationContext 获取自身代理 / 拆分到另一个类
      → 异常类型不对：默认只回滚 RuntimeException 和 Error， checked Exception 不回滚
        → 解决：@Transactional(rollbackFor = Exception.class)
    → 事务隔离级别
      → 默认：ISOLATION_DEFAULT（用数据库的）
      → 其他：同数据库的 4 个级别
```

## 核心追问

| 追问 | 考点 |
|------|------|
| REQUIRED 和 REQUIRES_NEW 的区别？ | REQUIRED 加入现有事务，REQUIRES_NEW 挂起现有事务 |
| 为什么自调用会导致事务失效？ | Spring 事务靠 AOP 代理，自调用绕过代理 |
| checked Exception 默认不回滚，为什么？ | Spring 设计：checked Exception 是「可恢复」的 |
| 事务传播行为在什么场景下用 REQUIRES_NEW？ | 日志/审计记录，不管主事务是否回滚都要记录 |

---

## 参考答案要点

**7 种传播行为速记**：
- `REQUIRED`（默认）：有就用，没有就建
- `REQUIRES_NEW`：挂起现有，自己新建
- `SUPPORTS`：有就用，没有就算了
- `NOT_SUPPORTED`：不用事务，有就挂起
- `MANDATORY`：必须有事务，没有就抛异常
- `NEVER`：必须没事务，有就抛异常
- `NESTED`：嵌套事务（保存点）

**事务失效的三大元凶**：自调用 / 非 public / 异常类型不对。
