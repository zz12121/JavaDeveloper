# 循环依赖解决方案

## 这个问题为什么存在？

> 如果两个 BeanA、B 相互引用（`A → B，B → A`），Spring 在创建 A 时需要注入 B，创建 B 时需要注入 A。如果没有循环依赖机制，Spring 会陷入无限递归，抛出 `BeanCurrentlyInCreationException`。 

这个问题的本质是：**对象在完全初始化之前，是否需要被其他对象引用**

Spring 的解决思路是**提前暴露未初始化完成的对象引用**（"早期引用"），让依赖方先拿到一个"半成品"，等最终初始化完成后再替换成完整对象。

## 它是怎么解决的？

### 三级缓存（核心机制）

Spring 用 **三个缓存** 来解决循环依赖，定义在 `DefaultSingletonBeanRegistry` 中：

```java
// 一级缓存：存放完全初始化好的 Bean（单例池）
private final Map<String, Object> singletonObjects = new ConcurrentHashMap<>(256);

// 二级缓存：存放早期 Bean（已实例化但未初始化，可能是原始对象也可能是代理对象）
private final Map<String, Object> earlySingletonObjects = new ConcurrentHashMap<>(16);

// 三级缓存：存放 Bean 的 ObjectFactory（可以生成早期引用，惰性地决定是否创建代理）
private final Map<String, ObjectFactory<?>> singletonFactories = new HashMap<>(16);

// 当前正在创建的 Bean 名称集合（用于检测循环依赖）
private final Set<String> singletonsCurrentlyInCreation = new HashSet<>(16);
```

### 完整时序：`AService → BService → AService`

```
getBean("aService")
    ↓
1. 检查一级缓存 → 没有
2. 标记 aService 为"正在创建"（加入 singletonsCurrentlyInCreation）
3. 实例化 A（反射 new ASerivce()）→ 此时 A 的属性还没注入
4. 把 A 的 ObjectFactory 放入【三级缓存】
       key = "aService"
       value = () -> getEarlyBeanReference("aService", A.class, aInstance)
5. 开始填充 A 的属性 → 发现依赖 B → getBean("bService")
    ↓
getBean("bService")
    ↓
1. 检查一级缓存 → 没有
2. 标记 bService 为"正在创建"
3. 实例化 B
4. 把 B 的 ObjectFactory 放入【三级缓存】
5. 开始填充 B 的属性 → 发现依赖 A → getBean("aService")


    ↓
getBean("aService")  【第二次进来-B 依赖 A】
    ↓
1. 检查一级缓存 → 没有
2. 检查 "正在创建集合" → 有！说明发生了循环依赖
3. 检查二级缓存 → 没有
4. 检查三级缓存 → 有！拿到 A 的 ObjectFactory
5. 调用 ObjectFactory.getObject() → 执行 getEarlyBeanReference()
   → 如果 A 不需要 AOP 代理，直接返回原始实例
   → 如果 A 有 AOP，这里生成代理对象（提前代理）
6. 把结果从三级缓存移除，放入【二级缓存】
7. 返回早期的 A（原始对象 或 代理对象）

    ↓
B 拿到早期的 A，注入完成
B 初始化完成
B 放入【一级缓存】，从二、三级缓存移除
    ↓
回到 A 的属性的注入 → A 拿到完整的 B，注入完成
A 初始化完成
A 放入【一级缓存】，从二、三级缓存移除
    ↓
循环依赖解决完成 ✅
```

### `getEarlyBeanReference()` 的关键逻辑


```
getEarlyBeanReference(): 三级缓存中的 ObjectFactory 实际执行的方法

public Object getEarlyBeanReference(String beanName, Class<?> mbd, Object bean) {
    Object exposedObject = bean;
    // 遍历所有 SmartInstantiationAwareBeanPostProcessor
    for (BeanPostProcessor bp : getBeanPostProcessors()) {
        if (bp instanceof SmartInstantiationAwareBeanPostProcessor) {
            // 重点是 AbstractAutoProxyCreator
            exposedObject = ((SmartInstantiationAwareBeanPostProcessor) bp)
                .getEarlyBeanReference(exposedObject, beanName);


        }
    }
    return exposedObject;
}
```

如果发现当前 Bean 需要被 AOP 代理，**这里就会提前生成代理对象**，确保循环依赖时注入的是正确的代理对象（而不是原始对象）。

### 为什么需要三级缓存，而不是两级？

**两级缓存方案（去掉三级）的问题**：

```
假设只用一、二级缓存：
- 实例化 A → 直接把原始对象放入【二级缓存】
- B 创建时从二级缓存拿到 A 的原始对象 → 注入
- A 继续执行 → 被 AOP 代理 → 最终 A 是代理对象
- 问题：B 里注入的 A 是原始对象，不是代理对象！
  Spring 容器里的 A 是代理对象，但 B 用的是原始对象 → 不一致！
```

**三级缓存解决**：把「是否创建代理对象」的决策**延迟到真正发生循环依赖时**。没有循环依赖时，不需要提前创建代理，AOP 正常在初始化完成后做。

```
三级缓存存的不是对象，而是 ObjectFactory（工厂）：
→ 没有循环依赖：工厂不会被调用，AOP 正常在初始化后做
→ 有循环依赖：工厂被调用，提前生成代理，确保注入的是代理对象
```

核心原因总结：**三级缓存实现了「延迟代理」—— 保证 AOP 代理在需要时才创建，同时保证循环依赖注入的是正确的代理对象**。

### 源码关键路径

```java
// org.springframework.beans.factory.support.DefaultSingletonBeanRegistry


protected Object getSingleton(String beanName, boolean allowEarlyReference) {
    // 1. 从一级缓存取
    Object singletonObject = this.singletonObjects.get(beanName);
    if (singletonObject == null && isSingletonCurrentlyInCreation(beanName)) {
        synchronized (this.singletonObjects) {
            // 2. 从二级缓存取
            singletonObject = this.earlySingletonObjects.get(beanName);
            if (singletonObject == null && allowEarlyReference) {
                // 3. 从三级缓存取 ObjectFactory，调用它
                ObjectFactory<?> singletonFactory = this.singletonFactories.get(beanName);


                if (singletonFactory != null) {
                    singletonObject = singletonFactory.getObject();
                    // 放入二级缓存，从三级缓存移除
                    this.earlySingletonObjects.put(beanName, singletonObject);
                    this.singletonFactories.remove(beanName);
                }
            }
        }
    }
    return singletonObject;
}
```

## 它和相似方案的本质区别是什么？

### Spring 循环依赖 vs 其他方案

| 方案 | 原理 | 局限 |
|---|---|---|
| Spring 三级缓存 | 提前暴露早期引用 | 只支持**单例 + 仅限 setter/字段注入** |
| 构造器注入 | 编译期报错，强制解耦 | 无法解决运行时循环依赖 |


| 使用 `ApplicationContext.getBean()` 延迟获取 | 打破编译期依赖 | 代码侵入性强 |
| `@Lazy` 注解 | 注入代理对象，使用时才真正获取 | 增加了代理层，调试复杂 |
| 重构设计（提取公共逻辑） | 根治方案 | 改造成本高 |

### 为什么**原型（prototype）** Bean 无法解决循环依赖？

原型 Bean 每次 `getBean()` 都创建新实例，没有"单例池"可以缓存早期引用。Spring 在 `AbstractBeanFactory.doGetBean()` 中有明确判断：

```java
if (isPrototypeCurrentlyInCreation(beanName)) {
    // 原型 Bean 检测到循环依赖，直接抛异常


    throw new BeanCurrentlyInCreationException(beanName);
}
```

### 为什么**构造器注入**无法解决循环依赖？

构造器注入要求在**实例化阶段**就传入依赖对象。但三级缓存是在「实例化完成之后」才暴露早期引用的，构造器阶段还没有任何对象可以暴露——死锁。

```
A(构造器需要 B) → 实例化 B → B(构造器需要 A) → 实例化 A → 死循环
```

## 正确使用方式

### 推荐做法：用 `@Lazy` 打破构造器注入的循环依赖

```java
@Service
public class AService {
    private final BService bService;


    @Autowired  // 构造器注入（推荐做法，不可变、易于测试）
    public AService(@Lazy BService bService) {  // ← 加 @Lazy
        this.bService = bService;
    }
}
```

`@Lazy` 实际注入的是一个 CGLIB 代理，调用方法时才真正从容器取 Bean，避开了构造器阶段的循环依赖。

### 不推荐但常见：setter 注入（Spring 默认能解决）

```java
@Service
public class AService {
    @Autowired  // setter 注入，Spring 能通过三级缓存解决循环依赖
    public void setBService(BService bService) {
        this.bService = bService;
    }
}
```

### 错误用法：构造器注入且不用 `@Lazy`

```java
@Service
public class AService {
    private final BService bService;


    @Autowired  // ❌ 会抛 BeanCurrentlyInCreationException
    public AService(BService bService) {
        this.bService = bService;
    }
}
```

## 边界情况和坑

### AOP + 循环依赖的代理一致性问题

```java
@Serivce
@Transactional  // A 有事务 AOP
public class AService {
    @Autowired private BService bService;
}


@Service
public class BService {
    @Autowired private AService aService;  // 注入的必须是 A 的代理对象
}
```

关键点：如果 A 有 AOP，`getEarlyBeanReference()` 提前生成代理对象，B 注入的是**代理对象**（不是原始对象），保证 `@Transactional` 正常工作。

如果去掉三级缓存，B 注入的是原始 A 对象，`@Transactional` 失效——这是三级缓存最精妙的地方。

### `@Async` + 循环依赖的特殊问题

`@Async` 是通过 `AsyncAnnotationBeanPostProcessor` 在**初始化后**创建代理的（比其他 AOP 晚）。如果 A 有 `@Async` 且发生循环依赖：

```
A 实例化 → 三级缓存放入 ObjectFactory
     ↓
B 创建 → 从三级缓存拿到 A 的早期引用（此时还没有 @Async 代理！）
     ↓
A 初始化完成 → @Async 创建代理 → 放入一级缓存的是代理对象
     ↓
问题：B 里注入的 A 是原始对象，没有异步能力！
```

Spring 对此有检测：`BeanCurrentlyInCreationException: Error creating bean with name 'AService': 
Bean with name 'AService' has been injected into other beans [...] in its raw version as part of a circular reference...`

解决方案：用 `@Lazy` 或把 `@Async` 方法抽到单独的 Service。

### 在多例（prototype）下只能自己解决

```java
@Scope(ConfigurableBeanFactory.SCOPE_PROTOTYPE)
@Serivce
public class AService {
    @Autowired private AService self;  // ❌ 原型模式下 Spring 不帮解决
}
```

原型 Bean 没有缓存机制，Spring **直接抛异常**。只能自己用 `ApplicationContext.getBean()` 延迟获取，或重构设计。
