# 审批工作流引擎设计

> 企业内部 OA、采购审批、合同审核、费用报销等场景都需要审批流程。流程节点多、审批人动态变化、支持会签/或签/条件分支，如何设计一个灵活且可扩展的工作流引擎？

---

## 题目一：审批工作流引擎设计

### 业务背景

```
典型审批场景：

1. 请假审批：直属上级 → HR（>3天） → 部门经理（>5天）
2. 采购审批：申请人 → 主管（<5万）→ 财务（5~20万）→ 副总裁（>20万）
3. 合同审批：业务员 → 法务审核 → 财务审核 → 部门负责人 → 总经理
4. 报销审批：员工提交 → 直属上级 → 财务（>5000）→ 出纳打款

核心约束：
  - 流程定义灵活：不同业务线有完全不同的审批流程
  - 审批人动态：根据金额、部门、职级等条件动态决定审批人
  - 支持多种审批模式：或签（任一人通过）、会签（所有人通过）、依次审批
  - 支持条件分支：金额不同走不同审批链路
  - 支持驳回/退回：驳回到发起人、驳回到指定节点、驳回到上一步
  - 支持转办/加签/委托：审批人不在时委托他人
  - 审批超时提醒：超时未审批自动催办
  - 审批记录完整：审批意见、操作时间、操作人不可篡改（审计）
  - 流程可追溯：任意时刻可查看当前流转到哪、历史审批记录
```

---

### 方案对比

| 方案 | 实现方式 | 灵活度 | 学习成本 | 性能 | 推荐场景 |
|------|---------|--------|---------|------|---------|
| 硬编码 | 每种审批写独立代码 | 极低 | 低 | 高 | 流程固定、审批类型少 |
| 状态机（Spring Statemachine） | 状态转移图定义 | 中 | 中 | 高 | 流程简单、状态数量有限 |
| 规则引擎（Drools） | 规则文件定义审批链路 | 高 | 高 | 中 | 条件复杂、规则频繁变更 |
| BPMN 引擎（Flowable/Camunda） | 标准 BPMN 2.0 流程定义 | 极高 | 高 | 中 | 企业级 OA、复杂审批流 |
| 自研轻量引擎 | 自定义流程模型 + 节点执行器 | 高 | 中 | 高 | 中等复杂度、不想引入重依赖 |

**生产推荐**：流程复杂度一般的企业，自研轻量引擎（可控性强、依赖少）；大型企业直接用 Flowable（生态成熟、BPMN 标准支持）。

---

### 方案详解 A：自研轻量工作流引擎（推荐生产）

#### 核心模型设计

```
流程定义（ProcessDef）— 描述一种审批类型怎么走
  ├── 节点列表（Node）
  │   ├── 开始节点（StartNode）
  │   ├── 审批节点（ApprovalNode）— 一个人或多个人审批
  │   ├── 条件节点（ConditionNode）— 根据条件走不同分支
  │   ├── 排他网关（ExclusiveGateway）— 多选一
  │   ├── 并行网关（ParallelGateway）— 同时走多条线
  │   └── 结束节点（EndNode）
  └── 连线（Edge）— 节点之间的流转关系 + 条件表达式

流程实例（ProcessInst）— 一次具体的审批流转
  ├── 当前活跃节点
  ├── 流程状态（RUNNING / APPROVED / REJECTED / CANCELLED）
  └── 变量上下文（动态审批人、金额、部门等运行时参数）

任务（Task）— 每个审批人的待办
  ├── 审批人
  ├── 任务状态（PENDING / APPROVED / REJECTED / DELEGATED）
  ├── 审批意见
  └── 到期时间
```

#### 审批人策略解析器

```java
/**
 * 审批人动态计算 —— 根据策略在运行时解析实际审批人
 */
public interface AssigneeStrategy {

    /**
     * 解析审批人列表
     * @param strategy  策略类型（MANAGER/ROLE/DEPT_LEADER/SPECIFIED/EXPRESSION）
     * @param config    策略配置值
     * @param context   流程上下文（包含发起人、金额、部门等变量）
     */
    List<Long> resolveAssignees(String strategy, String config, ProcessContext context);
}

@Component
public class AssigneeStrategyFactory {

    @Autowired
    private List<AssigneeStrategy> strategies;

    private final Map<String, AssigneeStrategy> strategyMap = new HashMap<>();

    @PostConstruct
    public void init() {
        // INITIATOR_MANAGER → 发起人的直属上级
        // ROLE → 指定角色的人员
        // DEPT_LEADER → 发起人部门负责人
        // SPECIFIED → 流程变量中指定的具体人
        // EXPRESSION → SpEL 表达式计算
        for (AssigneeStrategy s : strategies) {
            strategyMap.put(s.getClass().getSimpleName(), s);
        }
    }

    public AssigneeStrategy getStrategy(String type) {
        AssigneeStrategy strategy = strategyMap.get(type);
        if (strategy == null) {
            throw new WorkflowException("不支持的审批人策略: " + type);
        }
        return strategy;
    }
}

// 发起人直属上级策略
@Component
public class InitiatorManagerStrategy implements AssigneeStrategy {

    @Autowired
    private OrganizationService orgService;

    @Override
    public List<Long> resolveAssignees(String strategy, String config, ProcessContext context) {
        Long managerId = orgService.getDirectManager(context.getInitiator());
        if (managerId == null) {
            throw new WorkflowException("未找到发起人的直属上级");
        }
        return Collections.singletonList(managerId);
    }
}

// 角色策略 —— 查询指定角色下的人员
@Component
public class RoleAssigneeStrategy implements AssigneeStrategy {

    @Autowired
    private RoleService roleService;

    @Override
    public List<Long> resolveAssignees(String strategy, String config, ProcessContext context) {
        // config = "FINANCE_MANAGER"，查询该角色下所有人员
        return roleService.getUserIdsByRole(config);
    }
}
```

#### 流程引擎核心服务

```java
@Service
@Slf4j
public class WorkflowEngine {

    @Autowired
    private ProcessDefRepository processDefRepo;
    @Autowired
    private ProcessInstRepository processInstRepo;
    @Autowired
    private TaskRepository taskRepo;
    @Autowired
    private ApprovalRecordRepository recordRepo;
    @Autowired
    private AssigneeStrategyFactory strategyFactory;
    @Autowired
    private ApplicationEventPublisher eventPublisher;

    /**
     * 启动流程
     */
    @Transactional
    public ProcessInstance startProcess(String processKey, String businessKey,
                                         String businessType, Long initiator,
                                         Map<String, Object> variables) {
        // 1. 获取最新版本的流程定义
        ProcessDef processDef = processDefRepo.findLatestByKey(processKey);

        // 2. 创建流程实例
        ProcessInstance instance = new ProcessInstance();
        instance.setProcessDefId(processDef.getId());
        instance.setBusinessKey(businessKey);
        instance.setBusinessType(businessType);
        instance.setInitiator(initiator);
        instance.setVariables(variables);
        instance.setStatus(ProcessStatus.RUNNING);
        processInstRepo.save(instance);

        // 3. 推进到第一个审批节点，创建待办任务
        ProcessContext context = new ProcessContext(instance, processDef, variables);
        List<Node> startNodes = processDef.getStartNodes();
        for (Node node : startNodes) {
            advanceToNode(context, node);
        }

        log.info("流程启动: processKey={}, businessKey={}, instanceId={}",
                processKey, businessKey, instance.getId());
        return instance;
    }

    /**
     * 审批操作（通过/驳回）
     */
    @Transactional
    public void approve(Long taskId, Long operator, Action action, String comment) {
        // 1. 查询任务并校验
        Task task = taskRepo.findById(taskId)
                .orElseThrow(() -> new WorkflowException("任务不存在"));
        if (task.getStatus() != TaskStatus.PENDING) {
            throw new WorkflowException("任务已处理: " + task.getStatus());
        }
        if (!task.getAssignee().equals(operator)) {
            throw new WorkflowException("无权操作此任务");
        }

        // 2. 记录审批操作（审计日志，不可修改）
        ApprovalRecord record = new ApprovalRecord();
        record.setProcessInstId(task.getProcessInstId());
        record.setTaskId(taskId);
        record.setNodeId(task.getNodeId());
        record.setNodeName(task.getNodeName());
        record.setOperator(operator);
        record.setAction(action.getCode());
        record.setComment(comment);
        recordRepo.save(record);

        // 3. 更新任务状态
        task.setStatus(action == Action.APPROVE ? TaskStatus.APPROVED : TaskStatus.REJECTED);
        task.setComment(comment);
        task.setFinishTime(LocalDateTime.now());
        taskRepo.save(task);

        // 4. 根据审批动作推进流程
        ProcessInstance instance = processInstRepo.findById(task.getProcessInstId());
        ProcessDef processDef = processDefRepo.findById(instance.getProcessDefId());
        ProcessContext context = new ProcessContext(instance, processDef, instance.getVariables());

        if (action == Action.APPROVE) {
            handleApprove(context, task);
        } else {
            handleReject(context, task);
        }

        // 5. 发送审批结果事件（通知、状态回调等）
        eventPublisher.publishEvent(new ApprovalEvent(instance, task, action));
    }

    /**
     * 处理通过 —— 检查会签是否全部通过，决定是否推进到下一节点
     */
    private void handleApprove(ProcessContext context, Task completedTask) {
        String nodeId = completedTask.getNodeId();
        Node node = context.getProcessDef().getNode(nodeId);

        // 检查审批类型
        if (node.getApproveType() == ApproveType.COUNTERSIGN) {
            // 会签：检查该节点所有任务是否全部通过
            List<Task> pendingTasks = taskRepo.findPendingByNode(
                    context.getInstance().getId(), nodeId);
            if (!pendingTasks.isEmpty()) {
                log.info("会签节点 {} 还有 {} 个待审批任务", nodeId, pendingTasks.size());
                return; // 还没全部通过，等待其他人
            }
        }

        // 当前节点审批完毕，推进到下一节点
        List<Edge> outgoingEdges = context.getProcessDef().getOutgoingEdges(nodeId);
        if (outgoingEdges.isEmpty()) {
            completeProcess(context, ProcessStatus.APPROVED);
            return;
        }

        // 走网关判断
        Node nextNode = resolveNextNode(context, outgoingEdges);
        if (nextNode == null) {
            completeProcess(context, ProcessStatus.APPROVED);
            return;
        }

        advanceToNode(context, nextNode);
    }

    /**
     * 处理驳回
     */
    private void handleReject(ProcessContext context, Task task) {
        Node node = context.getProcessDef().getNode(task.getNodeId());

        // 会签中一人驳回 → 整个节点驳回
        if (node.getApproveType() == ApproveType.COUNTERSIGN) {
            taskRepo.rejectAllByNode(context.getInstance().getId(), task.getNodeId());
        }

        // 驳回到发起人，流程结束
        context.getInstance().setStatus(ProcessStatus.REJECTED);
        context.getInstance().setEndTime(LocalDateTime.now());
        processInstRepo.save(context.getInstance());

        log.info("流程驳回: instanceId={}, nodeId={}, operator={}",
                context.getInstance().getId(), task.getNodeId(), task.getAssignee());
    }

    /**
     * 推进到下一个节点 —— 动态解析审批人并创建任务
     */
    private void advanceToNode(ProcessContext context, Node node) {
        if (node.getType() == NodeType.END) {
            completeProcess(context, ProcessStatus.APPROVED);
            return;
        }

        if (node.getType() == NodeType.APPROVAL) {
            // 动态解析审批人
            AssigneeStrategy strategy = strategyFactory.getStrategy(
                    node.getAssigneeStrategy());
            List<Long> assignees = strategy.resolveAssignees(
                    node.getAssigneeStrategy(), node.getAssigneeValue(), context);

            if (assignees.isEmpty()) {
                throw new WorkflowException("节点 " + node.getName() + " 未找到审批人");
            }

            // 为每个审批人创建任务
            for (Long assignee : assignees) {
                Task task = new Task();
                task.setProcessInstId(context.getInstance().getId());
                task.setNodeId(node.getId());
                task.setNodeName(node.getName());
                task.setAssignee(assignee);
                task.setTaskType(node.getApproveType().getCode());
                task.setStatus(TaskStatus.PENDING);
                task.setDueTime(LocalDateTime.now().plusHours(node.getTimeoutHours()));
                taskRepo.save(task);
            }

            // 更新流程实例当前节点
            context.getInstance().setCurrentNode(node.getId());
            processInstRepo.save(context.getInstance());
        } else if (node.getType() == NodeType.EXCLUSIVE_GATEWAY) {
            // 排他网关：根据条件选择分支
            List<Edge> edges = context.getProcessDef().getOutgoingEdges(node.getId());
            Node nextNode = resolveNextNode(context, edges);
            advanceToNode(context, nextNode);
        }
    }

    /**
     * 条件解析 —— 根据流程变量计算走哪条分支
     */
    private Node resolveNextNode(ProcessContext context, List<Edge> edges) {
        ExpressionParser parser = new SpelExpressionParser();
        StandardEvaluationContext evalContext = new StandardEvaluationContext();
        context.getVariables().forEach(evalContext::setVariable);

        for (Edge edge : edges) {
            if (edge.getCondition() == null || edge.getCondition().isEmpty()) {
                // 无条件的默认分支
                return context.getProcessDef().getNode(edge.getTo());
            }
            // 用 SpEL 解析条件表达式
            Expression expr = parser.parseExpression(edge.getCondition());
            Boolean result = expr.getValue(evalContext, Boolean.class);
            if (Boolean.TRUE.equals(result)) {
                return context.getProcessDef().getNode(edge.getTo());
            }
        }
        return null; // 没有匹配的分支
    }

    private void completeProcess(ProcessContext context, ProcessStatus status) {
        context.getInstance().setStatus(status);
        context.getInstance().setEndTime(LocalDateTime.now());
        context.getInstance().setCurrentNode(null);
        processInstRepo.save(context.getInstance());
        log.info("流程完成: instanceId={}, status={}",
                context.getInstance().getId(), status);
    }
}
```

#### 会签 vs 或签

```
会签（Countersign）：
  节点 N 有 3 个审批人 [A, B, C]
  ┌──→ A 通过 → 等待
  ├──→ B 通过 → 等待
  └──→ C 通过 → 全部通过 → 推进下一节点
  任一人驳回 → 节点整体驳回

  适用场景：合同审批（法务+财务+业务全部通过才算过）

或签（Or Sign）：
  节点 N 有 3 个审批人 [A, B, C]
  ┌──→ A 通过 → 节点通过 → 推进下一节点（B/C 任务自动关闭）
  ├──→ B 通过 → 同上
  └──→ C 通过 → 同上

  适用场景：主管审批（主管+代理主管任一通过即可）

依次审批（Sequential）：
  节点 N 有 3 个审批人 [A, B, C]
  A 通过后 → 创建 B 的任务 → B 通过后 → 创建 C 的任务

  适用场景：分级审批（逐级上报）
```

#### 转办与委托

```java
/**
 * 转办：审批人将任务转给另一个人，自己不再参与
 */
@Transactional
public void transfer(Long taskId, Long operator, Long targetUser, String comment) {
    Task task = taskRepo.findById(taskId)
            .orElseThrow(() -> new WorkflowException("任务不存在"));
    if (!task.getAssignee().equals(operator)) {
        throw new WorkflowException("无权操作");
    }

    // 记录转办操作
    saveRecord(task, operator, Action.TRANSFER, comment);

    // 关闭原任务，创建新任务给目标人
    task.setStatus(TaskStatus.TRANSFERRED);
    task.setFinishTime(LocalDateTime.now());
    taskRepo.save(task);

    Task newTask = new Task();
    newTask.setProcessInstId(task.getProcessInstId());
    newTask.setNodeId(task.getNodeId());
    newTask.setNodeName(task.getNodeName());
    newTask.setAssignee(targetUser);
    newTask.setTaskType(task.getTaskType());
    newTask.setStatus(TaskStatus.PENDING);
    newTask.setDueTime(task.getDueTime());
    taskRepo.save(newTask);
}

/**
 * 委托：审批人委托他人代为审批，审批结果仍记在原审批人名下
 */
@Transactional
public void delegate(Long taskId, Long operator, Long delegateUser, String comment) {
    Task task = taskRepo.findById(taskId)
            .orElseThrow(() -> new WorkflowException("任务不存在"));
    if (!task.getAssignee().equals(operator)) {
        throw new WorkflowException("无权操作");
    }

    saveRecord(task, operator, Action.DELEGATE, comment);

    // 更新审批人为被委托人，但记录原始审批人
    task.setAssignee(delegateUser);
    task.setDelegatedFrom(operator);
    taskRepo.save(task);
}
```

```java
/**
 * 定时扫描超时未审批的任务，发送催办通知
 * 配合 XXL-JOB 每 15 分钟执行一次
 */
@Component
@Slf4j
public class ApprovalTimeoutHandler {

    @Autowired
    private TaskRepository taskRepo;
    @Autowired
    private NotificationService notificationService;
    @Autowired
    private StringRedisTemplate redisTemplate;

    /**
     * 催办逻辑：
     * 1. 查询所有已超时的待审批任务
     * 2. 用 Redis 防止重复催办（每个任务每天最多催 2 次）
     */
    public void handleTimeout() {
        List<Task> timeoutTasks = taskRepo.findOverduePendingTasks(LocalDateTime.now());

        for (Task task : timeoutTasks) {
            String remindKey = "wf:remind:" + task.getId();
            Long count = redisTemplate.opsForValue().increment(remindKey);
            if (count != null && count == 1) {
                redisTemplate.expire(remindKey, 12, TimeUnit.HOURS);
            }
            if (count != null && count <= 2) { // 每天最多催 2 次
                ProcessInstance inst = processInstRepo.findById(task.getProcessInstId());
                notificationService.sendRemind(task.getAssignee(),
                        "您有一个审批任务已超时，请尽快处理：" + inst.getBusinessKey());

                log.info("催办通知已发送: taskId={}, assignee={}, remindCount={}",
                        task.getId(), task.getAssignee(), count);
            }
        }
    }
}
```

#### 驳回到指定节点

```java
/**
 * 驳回到指定节点 —— 不是驳回到发起人，而是驳回到流程中的某个历史节点
 *
 * 场景：财务审批发现采购申请信息有误，驳回到「提交申请」节点修改后重新走流程
 *
 * 实现思路：
 * 1. 关闭当前节点及之后所有节点的任务
 * 2. 将流程状态重置为 RUNNING
 * 3. 在目标节点重新创建任务（审批人不变或重新计算）
 * 4. 清除目标节点之后的审批记录（可选，取决于审计要求）
 */
@Transactional
public void rejectToNode(Long taskId, Long operator, String targetNodeId, String comment) {
    Task currentTask = taskRepo.findById(taskId)
            .orElseThrow(() -> new WorkflowException("任务不存在"));

    ProcessInstance instance = processInstRepo.findById(currentTask.getProcessInstId());
    ProcessDef processDef = processDefRepo.findById(instance.getProcessDefId());

    // 校验 targetNodeId 在当前流程定义中存在
    if (processDef.getNode(targetNodeId) == null) {
        throw new WorkflowException("目标节点不存在: " + targetNodeId);
    }

    // 记录驳回操作
    saveRecord(currentTask, operator, Action.REJECT, comment);

    // 关闭从目标节点到当前节点之间所有待办任务
    List<Task> pendingTasks = taskRepo.findPendingByInstanceId(instance.getId());
    for (Task t : pendingTasks) {
        t.setStatus(TaskStatus.REJECTED);
        t.setFinishTime(LocalDateTime.now());
        taskRepo.save(t);
    }

    // 在目标节点重新创建任务
    Node targetNode = processDef.getNode(targetNodeId);
    ProcessContext context = new ProcessContext(instance, processDef, instance.getVariables());
    AssigneeStrategy strategy = strategyFactory.getStrategy(targetNode.getAssigneeStrategy());
    List<Long> assignees = strategy.resolveAssignees(
            targetNode.getAssigneeStrategy(), targetNode.getAssigneeValue(), context);

    for (Long assignee : assignees) {
        Task newTask = new Task();
        newTask.setProcessInstId(instance.getId());
        newTask.setNodeId(targetNodeId);
        newTask.setNodeName(targetNode.getName());
        newTask.setAssignee(assignee);
        newTask.setTaskType(targetNode.getApproveType().getCode());
        newTask.setStatus(TaskStatus.PENDING);
        newTask.setDueTime(LocalDateTime.now().plusHours(targetNode.getTimeoutHours()));
        taskRepo.save(newTask);
    }

    // 更新流程实例状态
    instance.setCurrentNode(targetNodeId);
    instance.setStatus(ProcessStatus.RUNNING);
    processInstRepo.save(instance);

    log.info("驳回到指定节点: instanceId={}, from={}, to={}",
            instance.getId(), currentTask.getNodeId(), targetNodeId);
}
```

```java
// Flowable 方式定义流程 —— 用 BPMN XML 或 Flowable API
@Service
public class FlowableApprovalService {

    @Autowired
    private RuntimeService runtimeService;
    @Autowired
    private TaskService taskService;
    @Autowired
    private HistoryService historyService;

    /**
     * 启动流程
     */
    public String startProcess(String processKey, String businessKey,
                                Map<String, Object> variables) {
        ProcessInstance instance = runtimeService.startProcessInstanceByKey(
                processKey, businessKey, variables);
        return instance.getId();
    }

    /**
     * 查询待办任务
     */
    public List<TaskDTO> getMyPendingTasks(Long userId) {
        List<Task> tasks = taskService.createTaskQuery()
                .taskAssignee(String.valueOf(userId))
                .orderByTaskCreateTime().desc()
                .list();

        return tasks.stream()
                .map(t -> new TaskDTO(
                        t.getId(), t.getName(), t.getProcessInstanceId(),
                        t.getCreateTime(), t.getDueDate()))
                .collect(Collectors.toList());
    }

    /**
     * 审批通过
     */
    @Transactional
    public void completeTask(String taskId, Long userId, String comment) {
        // 添加审批意见
        taskService.addComment(taskId, null, comment);
        // 完成任务（自动推进到下一节点）
        taskService.complete(taskId);
    }

    /**
     * 驳回 —— Flowable 需要用运行时 API 手动跳转
     */
    @Transactional
    public void rejectToActivity(String taskId, String targetActivityId, String comment) {
        taskService.addComment(taskId, null, "驳回: " + comment);

        // Flowable 提供的运行时跳转 API
        runtimeService.createChangeActivityStateBuilder()
                .processInstanceId(getProcessInstanceIdByTaskId(taskId))
                .moveActivityIdTo(getActivityIdByTaskId(taskId), targetActivityId)
                .changeState();
    }

    /**
     * 会签 —— Flowable 原生支持多实例
     */
    // BPMN 中配置：
    // <userTask id="finance_review" name="财务审核"
    //           activiti:assignee="${assigneeService.getFinanceReviewer(execution)}">
    //   <multiInstanceLoopCharacteristics
    //     isSequential="false"
    //     activiti:collection="financeReviewerList"
    //     activiti:elementVariable="reviewer">
    //     <completionCondition>${nrOfCompletedInstances == nrOfInstances}</completionCondition>
    //   </multiInstanceLoopCharacteristics>
    // </userTask>
    // completionCondition 控制通过条件：
    //   全部通过: nrOfCompletedInstances == nrOfInstances
    //   一票通过: nrOfCompletedInstances >= 1
    //   比例通过: nrOfCompletedInstances / nrOfInstances >= 0.6
}
```

#### Flowable vs 自研选型

| 维度 | 自研轻量引擎 | Flowable |
|------|------------|----------|
| 上手难度 | 中，需要设计模型 | 高，需要学习 BPMN 规范 |
| 灵活度 | 高，完全可控 | 极高，标准规范但定制需深入 |
| 功能完整度 | 按需实现 | 开箱即用（监听器、定时器、子流程、信号等） |
| 运维成本 | 低，纯 Java 代码 | 中，需要维护 Flowable 引擎 + 28+ 张表 |
| 可视化设计器 | 需自研或用第三方 | 官方 Flowable Modeler |
| 数据库依赖 | 自定义几张表 | 引擎自动创建 28+ 张表 |
| 社区生态 | 无 | 活跃，大量文档和案例 |
| 适用团队规模 | 5~50 人业务团队 | 50+ 人中大型企业 |

#### Q1：如何防止审批人被离职/调岗后流程卡死？

```
方案 1（推荐）：审批人策略始终动态计算
  - 不要在流程启动时就写死审批人
  - 每次推进到新节点时实时查询组织架构获取当前审批人
  - 策略「INITIATOR_MANAGER」→ 每次调接口取最新的直属上级

方案 2：流程管理员干预
  - 提供管理后台，管理员可以「转办」卡住的任务到其他人
  - 定时任务扫描卡住超过 N 天的任务，自动通知管理员

方案 3：自动升级
  - 审批人离职/不可用时，自动升级到上一级审批人
  - 需要和组织架构系统联动
```

#### Q2：如何支持动态加签（审批过程中临时加人）？

```java
/**
 * 前加签：在当前节点前面插入新的审批人
 * 当前审批人审批完后，新加入的人先审，审完再继续
 *
 * 后加签：在当前节点后面插入新的审批人
 * 当前审批人审批完后，原流程继续，同时新加入的人也要审
 */
@Transactional
public void addSignAfter(Long taskId, Long operator, List<Long> newAssignees, String comment) {
    Task currentTask = taskRepo.findById(taskId).orElseThrow();

    // 记录加签操作
    saveRecord(currentTask, operator, Action.ADD_SIGN, comment);

    // 为新加的人创建任务（与当前节点同级）
    for (Long assignee : newAssignees) {
        Task task = new Task();
        task.setProcessInstId(currentTask.getProcessInstId());
        task.setNodeId(currentTask.getNodeId());
        task.setNodeName(currentTask.getNodeName() + "(加签)");
        task.setAssignee(assignee);
        task.setTaskType(ApproveType.COUNTERSIGN.getCode()); // 加签的人都要审
        task.setStatus(TaskStatus.PENDING);
        task.setDueTime(LocalDateTime.now().plusHours(24));
        task.setIsAddedSign(true);
        taskRepo.save(task);
    }

    // 当前审批人的任务标记为「等待加签人员完成」
    currentTask.setStatus(TaskStatus.WAITING_ADD_SIGN);
    taskRepo.save(currentTask);
}
```

#### Q3：审批流程如何支持子流程？

```
场景：采购审批流程中，金额 > 10 万时需要走一个「招投标子流程」

实现方式 1（简单）：条件网关 + 独立流程
  主流程到条件节点 → 满足条件 → 启动一个新的流程实例 → 子流程完成后回调主流程
  优点：解耦，子流程独立管理
  缺点：需要回调机制

实现方式 2（Flowable）：内嵌子流程
  BPMN 中使用 <subProcess> 标签，Flowable 原生支持
  优点：引擎自动管理
  缺点：BPMN 复杂度高

实现方式 3（自研推荐）：节点事件监听
  在条件节点上加 EventListener，满足条件时同步启动子流程
  主流程阻塞等待子流程完成事件
```

#### Q4：高并发下大量审批任务如何优化查询性能？

```
问题：审批人查询待办列表，task 表可能有千万级数据

优化方案：
1. 待办数量用 Redis 计数（避免 COUNT 查询）
   wf:pending_count:{userId} → 待办数量
   审批操作时原子更新计数

2. 待办列表分页查询走二级缓存
   @Cacheable(value = "userTasks", key = "#userId + ':' + #page + ':' + #size")

3. 历史任务归档
   已完成的任务超过 30 天迁移到历史表（wf_task_history）
   保持 wf_task 表只存活跃任务

4. 索引优化
   (assignee, status, create_time) 组合索引覆盖待办查询
   避免回表：用覆盖索引

5. 读写分离
   待办列表查询走从库
```

### 生产 Checklist

```
□ 流程定义是否支持版本管理（旧流程实例走旧版本）
□ 审批人离职/调岗后流程是否会卡死
□ 会签场景下一人驳回是否正确处理其他人的任务
□ 驳回到指定节点后重新审批，审批记录是否完整
□ 并行网关（多条分支同时进行）汇聚时是否正确等待
□ 超时催办是否有限频（不能一秒发 10 条提醒）
□ 审批记录是否不可篡改（审计要求）
□ 流程实例查询是否有索引支撑（按业务Key、发起人、状态查询）
□ 大量待办任务是否做了归档和缓存
□ 转办/加签/委托操作是否记录了操作日志
□ SpEL 条件表达式是否有安全校验（防止表达式注入）
□ 流程引擎接口是否做了幂等（重复提交审批不能重复推进）
```
