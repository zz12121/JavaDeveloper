# 常见的负载均衡算法有哪些？

> ⚠️ **先盲答**：负载均衡是什么？为什么需要它？

---

## 盲答引导

1. 负载均衡在 OSI 哪一层？—— L4（传输层）/ L7（应用层）
2. 轮询 / 随机 / 哈希 / 最小连接数 / 加权负载均衡，原理分别是什么？
3. 一致性哈希属于负载均衡吗？它和普通哈希有什么区别？

## 知识链提示

```
负载均衡
  → [[负载均衡]]
    → 四层 vs 七层
      → 四层（TCP/UDP）：基于 IP + 端口转发，不解析应用层
        → LVS / 云 LB（高性能，几十 Gbps）
        → 无连接保持（无 Session）
      → 七层（HTTP/HTTPS）：解析应用层，按 URL/Header 路由
        → Nginx / Envoy / Spring Cloud Gateway
        → 支持：路径重写 / SSL 卸载 / 灰度发布
    → 算法分类
      → 轮询（Round Robin）：简单，无状态，每台轮着来
      → 加权轮询（Weighted Round Robin）：响应慢的机器权重低
      → 随机（Random）：随机选一台
      → 最少连接（Least Connections）：选当前连接数最少的
      → 一致性哈希：[[一致性哈希]] → 同一 key 的请求打到同一机器
      → 加权最小响应时间（WLC）：最快响应的 + 权重
    → Ribbon 算法（Spring Cloud 客户端负载均衡）
      → RoundRobinRule / RandomRule / WeightedResponseTimeRule / BestAvailableRule / RetryRule
      → [[服务注册发现]] → Ribbon 从 Eureka 获取服务列表后路由
```

## 核心追问

| 追问 | 考点 |
|------|------|
| Ribbon 和 Nginx 都是负载均衡，有什么区别？ | Ribbon（客户端，七层感知，服务级）/ Nginx（服务端，四层或七层，机器级） |
| 一致性哈希做负载均衡时，节点挂了怎么办？ | 该节点负责的 key 全部迁移到下一个节点 |
| 「最少连接数」算法怎么统计连接数？ | 每秒收集一次各后端的活跃连接数 |
| 金丝雀发布（灰度发布）用什么负载均衡实现？ | 七层负载均衡（按 Header/权重路由特定比例流量）|

---

## 参考答案要点

**L4 vs L7 负载均衡**：

| 维度 | 四层（LVS） | 七层（Nginx） |
|------|------------|-------------|
| 性能 | 更高（不解析应用层） | 略低（解析 HTTP） |
| 功能 | 端口转发 | URL 路由 / 重写 / SSL |
| 适用 | 通用转发 | 需要应用层感知的场景 |

**实战算法选择**：一般用轮询或加权轮询；有 Session 保持需求用一致性哈希；有性能差异用 WLC。
