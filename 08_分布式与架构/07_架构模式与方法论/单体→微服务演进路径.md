# 单体→微服务演进路径

## 这个问题为什么存在？

> 单体应用初期开发快、部署简单。但随着业务增长，**代码耦合严重、扩展困难、部署风险高**。问题是：**什么时候拆、怎么拆、拆了之后怎么办？**

过早拆分是最大的架构罪。拆早了，边界还在变，拆了就要不断改；拆晚了，代码耦合严重，拆不动。

---

## 它是怎么解决问题的？

### 演进路径：三个阶段

```
阶段1：模块化单体（推荐起点）
  → 按业务能力分包（order、inventory、payment）
  → 包之间不允许循环依赖（用ArchUnit测）
  → 数据库还没拆，但逻辑已隔离

阶段2：边缘服务剥离（第一步拆分）
  → 把"变更频繁"或"资源需求不同"的功能拆出来
  → 例如：推荐系统（算法密集型，用Python）→ 独立服务

阶段3：全面微服务（慎重）
  → 业务稳定、团队壮大后，再全面拆分
  → 配套：注册中心、配置中心、网关、链路追踪
```

### 拆分策略：四个维度

```
维度1：业务能力（DDD限界上下文）
  → order-service、inventory-service、payment-service

维度2：数据边界（共享数据库是陷阱）
  → 每个服务独立数据库，不允许跨服务查表

维度3：团队结构（康威定律）
  → 3个team → 3个服务（每个team维护一个）

维度4：变更频率（频繁变的独立拆）
  → 促销规则（频繁变） → promotion-service
  → 订单核心（稳定） → order-service
```

```java
// 模块化单体：包之间不允许循环依赖（ArchUnit测试）
@AnalyzeClasses(packages = "com.example")
public class ModularMonolithTest {

    @ArchTest
    public static final ArchRule noCycles = slices().matching("com.example.(*)")
            .should().beFreeOfCycles();

    @ArchTest
    public static final ArchRule orderDoesNotDependOnPayment =
            noClasses().that().resideInAPackage("..order..")
                    .should().dependOnClassesThat().resideInAPackage("..payment..");
}
```

## 深入原理

| | 模块化单体 | 微服务（过早拆分） | 微服务（正确拆分） |
|---|---|---|---|
| 部署 | 整体部署 | 独立部署，但频繁改边界 | 独立部署，边界稳定 |
| 扩展性 | 整体扩展（浪费） | 独立扩展，但边界还在变 | 独立扩展，精准 |
| 复杂度 | 低（无分布式事务） | 极高（边界在变，不断改） | 中（边界稳定，治理完善） |
| 适用阶段 | 创业初期、团队<5人 | ❌ 不推荐 | ✅ 业务稳定、团队>10人 |

**本质区别**：
- **模块化单体**：用包隔离，不做分布式，复杂度低
- **过早微服务**：边界还在变，拆了就要不断改，复杂度×10
- **正确微服务**：业务稳定，团队壮大，才全面拆分

## 正确使用方式

### 第一步：模块化单体（包隔离）

```
com.example.ecommerce
  ├── order          # 订单模块（仅暴露OrderService接口）
  ├── inventory     # 库存模块
  ├── payment       # 支付模块
  ├── shared        # 共享内核（User、Auth）
  └── EcommerceApplication.java
```

**关键**：模块间只通过**接口**通信，不直接依赖实现类。

### 第二步：边缘服务剥离（拆分第一个服务）

```java
// 拆分前：单体内部方法调用
orderService.createOrder();
inventoryService.deduct();  // 同一JVM

// 拆分后：RPC调用（Spring Cloud OpenFeign）
@FeignClient(name = "inventory-service")
public interface InventoryClient {
    @PostMapping("/inventory/deduct")
    void deduct(@RequestBody DeductRequest req);
}
// 调用就跟本地方法一样，但实际上是HTTP请求
inventoryClient.deduct(req);
```

### 第三步：数据库拆分（最难的）

```
拆分前：order表、inventory表、payment表 在同一DB
拆分后：
  order-service     → order_db
  inventory-service → inventory_db
  payment-service  → payment_db

问题：跨表查询（如"查询订单+库存"）怎么办？
解决：
  1. API组合（订单服务调库存服务，内存组装）
  2. CQRS（读写分离，读模型冗余）
  3. 数据同步（Canal，MySQL binlog → 其他服务）
```

## 边界情况和坑

### 坑1：过早拆分（业务边界还在变）

```
场景：刚上线2个月，业务模式还在探索
      拆了微服务，业务方"改一下订单流程"
      结果：3个服务都要改 → 拆分了个寂寞
```

**解决方案**：
1. **模块化单体起步**（包隔离，不拆库）
2. **等业务稳定再拆**（6个月~1年后）
3. **康威定律**：团队结构变化了，再拆（团队大了，需要独立部署）

### 坑2：数据库没拆，还是共享DB

```
场景：服务拆了，但DB还是共享的
结果：
  - 库存服务直接改order表（耦合）
  - DB性能瓶颈（还是单点）
  - 白拆了（没达到独立扩展的目的）
```

**解决方案**：
1. **服务拆分 + 数据库拆分 同步进行**
2. **共享DB只能作为过渡方案**（6个月内必须拆）
3. **用ArchUnit强制检测**（订单服务不能依赖库存表）

### 坑3：分布式事务复杂（拆库后的必然代价）

```
场景：订单创建成功，扣库存失败
结果：数据不一致（有订单，没扣库存）
```

**解决方案**：
1. **Saga模式**（补偿事务）
2. **事务消息**（RocketMQ）
3. **接受最终一致性**（不能 accepts 强一致，就不要拆微服务）
