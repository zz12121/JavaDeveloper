# 一致性哈希是什么？解决了什么问题？

> ⚠️ **先盲答**：普通哈希（hash % N）有什么问题？为什么需要一致性哈希？

---

## 盲答引导

1. 普通哈希 `hash(key) % N` 的问题在哪？—— 节点数量变化时，大量数据需要重新映射
2. 一致性哈希的「环」是什么？—— 2^32 个点的哈希环
3. 虚拟节点（Virtual Node）解决了什么问题？—— 数据倾斜
4. 一致性哈希在哪些场景用过？—— Redis Cluster / Memcached / 负载均衡

---

## 知识链提示

```
一致性哈希
  → [[13_面试训练/01_题目知识链/07_分布式与架构/06_一致性哈希]]
    → 普通哈希的问题
      → hash(key) % N（N 是节点数）
      → 节点增加/减少 → N 变化 → 大多数 key 的映射都变了 → 大量数据迁移
      → 举例：3台 → 4台，75% 的数据需要迁移
    → 一致性哈希的原理
      → 把所有节点哈希映射到一个 [0, 2^32-1] 的哈希环上
      → key 哈希 → 顺时针找到第一个大于等于它的节点 → 负责这个 key
      → 节点变化时，只影响环上该节点附近的 key → 迁移量 ≈ 1/N
    → 数据倾斜问题
      → 节点哈希分布不均匀 → 大量 key 落到少数节点
      → 解决：虚拟节点（每台物理机对应多个虚拟节点，哈希到环上不同位置）
        → 虚拟节点越多，分布越均匀
    → 应用场景
      → Redis 集群：使用一致性哈希（16384 个槽位）
      → [[06_数据库/04_SQL优化/分库分表方案]]：分片键选哈希（不是范围）
      → 负载均衡器：加权一致性哈希
```

## 核心追问

| 追问 | 考点 |
|------|------|
| Redis Cluster 用的是一致性哈希吗？ | 不是，用的是槽位映射（16384 个槽），是另一种分区策略 |
| 一致性哈希的缺点是什么？ | 虚拟节点过多管理复杂 / 某些场景仍需全量迁移 |
| 哈希环上，节点哈希冲突怎么办？ | 再哈希（rehash）或用虚拟节点分散 |
| 为什么说一致性哈希特别适合缓存场景？ | 缓存数据多读少写，节点变更时只需迁移少数数据 |

---

## 参考答案要点

**核心思想**：节点和 key 都哈希到同一个环上，key 顺时针找到第一个节点负责。

**普通哈希 vs 一致性哈希**：
- 普通哈希：节点变 → 大规模数据迁移（O(N)）
- 一致性哈希：节点变 → 局部数据迁移（O(1/N)）

**虚拟节点解决倾斜**：每个物理节点在环上映射多个虚拟节点，使数据分布更均匀。
