## 参考要点（盲答后再看）

**CHM 禁止 null 的原因**：

```java
// 语义问题：get(key) 返回 null，无法区分两种情况
// 情况1：key 不存在 → get() 返回 null
// 情况2：value 本身就是 null → get() 也返回 null

// Hashtable 允许 null：
// get(key) == null → 可能是 key 不存在，也可能是 value 为 null
// synchronizedMap 也允许 null → 但二者歧义共存

// CHM 的选择：宁可牺牲一点便利性，也要消除歧义
// 这使得 get() 的返回值语义清晰：null = key 不存在
```

**并发容器选型**：

```
需要 Map？
├── 不需要排序 → ConcurrentHashMap（⭐ 默认）
└── 需要排序/范围 → ConcurrentSkipListMap

需要 List？
├── 读多写极少 → CopyOnWriteArrayList（⭐）
└── 读写都频繁 → 考虑 Queue 替代

需要 Set？
├── 无序 → ConcurrentHashMap.newKeySet()（⭐ 默认）
├── 有序 → ConcurrentSkipListSet
└── 读多写极少（小集合）→ CopyOnWriteArraySet

需要 Queue？
├── 非阻塞高并发 → ConcurrentLinkedQueue（⭐）
├── 生产-消费（需等待）→ ArrayBlockingQueue / LinkedBlockingQueue
└── 双端（工作窃取）→ ConcurrentLinkedDeque
```

**ConcurrentLinkedQueue 三大坑**：

```java
// 坑1：size() 是 O(n)！
while (!queue.isEmpty()) {  // ✅ 用 isEmpty()
    queue.size();            // ❌ O(n)！每次遍历
}

// 坑2：无界 → 内存泄漏风险
queue.offer(task);  // 生产者速度 > 消费者 → 无限堆积 → OOM
// 解决：用有界阻塞队列或配合 Semaphore 限流

// 坑3：不适合生产-消费模式
// poll() 返回 null 表示空，不阻塞
// 消费者空转浪费 CPU
// 生产-消费用 BlockingQueue
```

**CopyOnWriteArrayList 迭代器**：

```java
CopyOnWriteArrayList<String> list = new CopyOnWriteArrayList<>();
list.add("a");
list.add("b");

Iterator<String> it = list.iterator();
// 迭代器是"快照"——copy-on-write 发生时不抛 CME
list.add("c");
// 迭代器仍看到 [a, b]，不抛 ConcurrentModificationException
// 但也看不到新增的 "c"
```

**并发 Set 对比**：

```
无序 → ConcurrentHashMap.newKeySet()    // O(1)，默认选
有序 → ConcurrentSkipListSet            // O(log n)，基于跳表
读少写多 → CopyOnWriteArraySet          // O(n)，小集合
```
