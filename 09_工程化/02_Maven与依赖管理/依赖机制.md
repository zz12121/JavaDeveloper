# 依赖机制

> Maven 的依赖管理不是「下载 jar 包」这么简单。核心要解决三个问题：依赖传递导致包冲突、版本仲裁规则不透明、依赖范围控制（编译期/测试期/运行期）。不理解这些，遇到 `ClassNotFoundException` 和 `NoSuchMethodError` 就只能瞎猜。

---

## 这个问题为什么存在？

没有依赖管理时，Java 项目的痛点是：

```
石器时代的做法：
1. 从网上手动下载 spring-core-5.2.0.jar
2. 扔进项目的 lib/ 目录
3. 发现缺少依赖 → 再下载 spring-jcl.jar
4. 再发现冲突 → 某个库依赖了旧版本的 Jackson
5. 最终 lib/ 里有 87 个 jar，不知道哪个是哪个的传递依赖
6. 上线时出现 NoSuchMethodError → 包冲突，排查 3 小时
```

**Maven 要解决的核心问题**：

1. **依赖传递**：A 依赖 B，B 依赖 C → 你只需要声明 A，C 自动加入
2. **版本仲裁**：多个库依赖同一个库的不同版本 → 用规则决定用哪个版本
3. **依赖范围**：有些包只在测试时需要（junit），有些只在运行时需要（servlet-api）
4. **冲突解决**：同一个库的不同版本同时存在 → 用机制决定保留哪个

---

## 它是怎么解决问题的？

### 核心概念：坐标（GAV）

```xml
<!-- 任何一个依赖由三个坐标唯一定位 -->
<dependency>
    <groupId>or.springframework</groupId>
    <artifactId>spring-core</artifactId>
    <version>5.2.0.RELEASE</version>
</dependency>
```

- `groupId`：组织/公司（倒序域名）
- `artifactId`：项目名
- `version`：版本号

**仓库查找逻辑**：
```
本地仓库（~/.m2/repository）
    ↓ 找不到
中央仓库（Maven Central：repo1.maven.org）
    ↓ 找不到
私服（Nexus，公司内网）
    ↓ 找不到
构建失败，报 MissingArtifactException
```

### 依赖传递与依赖树

```
你的项目
  ├── spring-core:5.2.0
  │     └── spring-jcl:5.2.0      ← 传递依赖（自动引入）
  └── jackson-core:2.10.0
        └── (无传递依赖)
```

**查看依赖树**：
```bash
mvn dependency:tree
# 输出：
# [INFO] com.example:my-app:jar:1.0-SNAPSHOT
# [INFO] +- or.springframework:spring-core:jar:5.2.0.RELEASE:compile
# [INFO] |  \- or.springframework:spring-jcl:jar:5.2.0.RELEASE:compile
# [INFO] \- com.fasterxml.jackson.core:jackson-core:jar:2.10.0:compile
```

### 版本仲裁规则（Arbitration）

当多个依赖引入了同一个库的不同版本，Maven 用**最近定义优先**（nearest definition）规则：

```
依赖路径 A：
  你的项目 → spring-core:5.2.0 → spring-jcl:5.2.0

依赖路径 B：
  你的项目 → jackson-core:2.10.0 → spring-jcl:3.0.0

仲裁结果：spring-jcl:5.2.0 胜出
（因为路径 A 距离更近，深度为 2；路径 B 深度为 2，
但先声明的优先——实际上是「依赖调解第一原则：路径最近者优先」）
```

**仲裁规则总结**：
1. **路径最近者优先**（深度浅的胜出）
2. **深度相同，先声明者优先**（在 `pom.xml` 中先声明的胜出）
3. **可以用 `<exclusions>` 排除传递依赖**
4. **可以用 `<dependencyManagement>` 锁定版本**

### 依赖范围（Scope）

| Scope | 编译期 | 测试编译期 | 运行期 | 典型例子 |
|-------|---------|-----------|--------|---------|
| `compile`（默认） | ✓ | ✓ | ✓ | spring-core |
| `test` | ✗ | ✓ | ✗ | junit、mockito |
| `provided` | ✓ | ✓ | ✗ | servlet-api（容器提供） |
| `runtime` | ✗ | ✓ | ✓ | mysql-connector-java |
| `system` | ✓ | ✓ | ✗ | 本地 jar（不推荐） |

```xml
<!-- 典型配置 -->
<dependency>
    <groupId>junit</groupId>
    <artifactId>junit</artifactId>
    <version>4.13.2</version>
    <scope>test</scope>     <!-- 只在测试时可用 -->
</dependency>

<dependency>
    <groupId>javax.servlet</groupId>
    <artifactId>javax.servlet-api</artifactId>
    <version>4.0.1</version>
    <scope>provided</scope> <!-- 编译时需要，运行时由容器提供 -->
</dependency>
```

### 依赖冲突解决：exclusions

```xml
<!-- 场景：spring-boot-starter-web 传递依赖了旧版 logback -->
<!-- 你想用 log4j2 代替，需要排除 logback -->

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
    <exclusions>
        <exclusion>
            <groupId>ch.qos.logback</groupId>
            <artifactId>logback-classic</artifactId>
        </exclusion>
    </exclusions>
</dependency>

<!-- 然后引入你想要的版本 -->
<dependency>
    <groupId>org.apache.logging.log4j</groupId>
    <artifactId>log4j-api</artifactId>
    <version>2.17.1</version>
</dependency>
```

## 深入原理

### Maven vs Gradle

| 维度 | Maven | Gradle |
|------|--------|--------|
| 配置方式 | XML（声明式） | Groovy/Kotlin DSL（编程式） |
| 学习曲线 | 低（XML 直观） | 高（需要学 DSL） |
| 构建性能 | 慢（每次全量构建） | 快（增量构建 + 构建缓存） |
| 依赖管理 | 成熟稳定 | 更灵活（支持动态版本、依赖约束） |
| 自定义任务 | 需要写插件（Maven 方式复杂） | 直接写任务（灵活） |
| 生态 | 成熟（插件极多） | 快速增长中 |

**为什么很多老项目还在用 Maven？**

1. **XML 声明式配置**：新人接手项目，打开 `pom.xml` 就能看懂依赖关系
2. **约定优于配置**：目录结构、构建生命周期都是固定的，不需要额外配置
3. **稳定性**：Maven 3.x 已经稳定很多年，构建可重现

**为什么新项目倾向用 Gradle？**

1. **构建速度**：大型单体仓库（Monorepo），Maven 构建要 20 分钟，Gradle 增量构建可能只要 2 分钟
2. **灵活性**：需要自定义构建逻辑时，Gradle 的 DSL 比 Maven 插件开发简单 10 倍
3. **Android 官方支持**：Android 项目只能用 Gradle

### Maven 依赖管理 vs NPM（Node.js）

| 维度 | Maven | NPM |
|------|--------|-----|
| 依赖存储 | 本地仓库（~/.m2）全局共享 | node_modules 每个项目独立 |
| 版本锁定 | 不自动锁定（需要 `dependencyManagement` 或 BOM） | package-lock.json 自动锁定 |
| 依赖冲突 | 仲裁规则自动解决（可能引入错误版本） | 允许同一库的不同版本共存（node_modules 嵌套） |
| 传输效率 | 只下载需要的传递依赖 | 下载所有依赖（包括可选依赖） |

**Maven 的典型问题**：依赖冲突导致运行时 `NoSuchMethodError`，需要手动排查依赖树并用 `exclusions` 解决。NPM 允许同一库的不同版本共存，冲突概率低，但 `node_modules` 体积巨大。

## 正确使用方式

### 1. 用 `dependencyManagement` 统一版本

```xml
<!-- 父 POM 中统一版本 -->
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.apache.commons</groupId>
            <artifactId>commons-lang3</artifactId>
            <version>3.12.0</version>   <!-- 所有子模块强制用这个版本 -->
        </dependency>
    </dependencies>
</dependencyManagement>

<!-- 子模块中只需要声明 groupId + artifactId，版本自动继承 -->
<dependencies>
    <dependency>
        <groupId>org.apache.commons</groupId>
        <artifactId>commons-lang3</artifactId>
        <!-- 不需要写 version -->
    </dependency>
</dependencies>
```

### 2. 用 BOM（Bill of Materials）管理依赖族

```xml
<!-- Spring Boot 的典型用法：引入 BOM 统一管理所有 Spring 依赖版本 -->
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-dependencies</artifactId>
            <version>2.7.0</version>
            <type>pom</type>
            <scope>import</scope>   <!-- import 方式引入 BOM -->
        </dependency>
    </dependencies>
</dependencyManagement>

<!-- 之后引入任何 Spring Boot 依赖都不需要写版本 -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
    <!-- version 自动从 BOM 继承 -->
</dependency>
```

### 3. 定期用 `mvn dependency:analyze` 检查无用依赖

```bash
mvn dependency:analyze
# 输出：
# [WARNING] Used undeclared dependencies found:
# [WARNING]    or.springframework:spring-core:jar:5.2.0:compile
#    ↑ 用了但没声明（靠传递依赖引入的，危险！）
#
# [WARNING] Unused declared dependencies found:
# [WARNING]    junit:junit:jar:4.13.2:test
#    ↑ 声明了但没用（可以删除）
```

**「Used undeclared dependencies」是危险信号**：你依赖了传递依赖，一旦直接依赖升级去掉了该传递依赖，你的代码就编译失败了。应该显式声明所有直接使用的依赖。

## 边界情况和坑

### 1. 传递依赖冲突导致 `NoSuchMethodError`

```
报错：
java.lang.NoSuchMethodError: or.springframework.core.xxx.Method.getJavaMember()

原因：
  编译时用的是 spring-core:5.2.0（有这个方法）
  运行时 Maven 仲裁给了 spring-core:4.3.0（没有这个方法）
```

**排查步骤**：
```bash
# 1. 查看依赖树，找到冲突
mvn dependency:tree -Dverbose

# 2. 在 dependency:tree 输出中找到哪个版本被仲裁了
# 3. 用 exclusions 排除旧版本，或用 dependencyManagement 锁定新版本
```

### 2. `provided` 范围的依赖在 IDE 中能跑，打包后跑不了

```
场景：
  servlet-api 设为 provided（因为 Tomcat 会提供）
  在 IDE 中运行正常（IDE 自动加了依赖）
  打成 jar/war 后运行 → ClassNotFoundException
```

**原因**：`provided` 范围的依赖不参与打包。`mvn package` 后的 jar 里没有 servlet-api。

**解决**：确认运行环境确实提供了该依赖。如果不确定，改成 `compile`（不推荐）或确保容器提供了正确的版本。

### 3. SNAPSHOT 版本导致构建不可重现

```xml
<version>1.0-SNAPSHOT</version>   <!-- SNAPSHOT 版本 -->
```

**问题**：SNAPSHOT 版本每次构建都会检查远程仓库是否有更新。今天构建成功，明天可能失败（因为 SNAPSHOT 变了）。

**解决**：
- 开发阶段可以用 SNAPSHOT
- 发布到生产环境前，必须去掉 SNAPSHOT，用固定版本号
- 用 `mvn -U` 强制更新 SNAPSHOT（谨慎使用）

### 4. Windows 下路径过长导致构建失败

```
报错：
Failed to execute goal org.apache.maven.plugins:maven-compiler-plugin:3.8.1:compile (default-compile) on project xxx: Fatal error compiling: java.lang.RuntimeException: com.sun.tools.javac.main.JavaCompiler is not supported in this environment

根本原因：
  Windows 路径长度限制 260 字符
  .m2/repository 下的依赖路径太深，超过限制
```

**解决**：
```xml
<!-- 方案 1：用短路径安装 Maven（避免中文路径） -->
<!-- 方案 2：启用 Windows 长路径支持（Windows 10+） -->
<!-- 方案 3：用 <localRepository> 配置短路径 -->
<settings>
  <localRepository>D:\.m2\repo</localRepository>
</settings>
```
