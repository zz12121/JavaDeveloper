# Java 工程师能力模型

> 知道自己在哪里，才能规划去哪里。

---

## 一、能力模型总览

Java 工程师的能力分为三个维度：**技术能力**、**工程能力**、**软技能**。

```
                    ┌─────────────────────────────────┐
                    │          综合竞争力               │
                    └──────────┬──────────────────────┘
                               │
        ┌──────────────────────┼──────────────────────┐
        ↓                      ↓                      ↓
   技术能力               工程能力               软技能
   (Know-How)           (Build-Right)         (Work-Smart)
        │                      │                      │
   ├─ Java 核心           ├─ 代码质量           ├─ 沟通协作
   ├─ 框架掌握           ├─ 系统设计           ├─ 项目推进
   ├─ 数据库             ├─ CI/CD              ├─ 技术判断力
   ├─ 中间件             ├─ 监控可观测性       ├─ 学习能力
   └─ 分布式/架构        └─ 安全意识           └─ 影响力
```

## 二、分级能力要求

### 2.1 P4/初级工程师（0-2 年）

**核心目标：能独立完成功能开发**

| 能力域 | 具体要求 |
|--------|---------|
| Java 基础 | OOP 四大特性、集合框架（List/Map/Set）、异常处理、泛型 |
| 并发 | Thread/Runnable/线程池基本使用、synchronized 基础 |
| JVM | 知道 GC 存在，能看懂基本 GC 日志 |
| Spring | SpringBoot 基本使用（CRUD、注解驱动）、IoC/DI 理解 |
| 数据库 | 写 SQL（JOIN/聚合/子查询）、MyBatis 熟练使用 |
| 中间件 | Redis 基本 CRUD 操作、MQ 发消息/消费消息 |
| 工程化 | Git 基本操作（commit/push/PR）、Maven/Gradle |
| 代码质量 | 代码可读（命名规范、方法不超过 50 行）、会写单元测试 |

**成长建议：** 不要满足于会用，要追问"为什么"。每用一个 API，去看一眼它的源码。

---

### 2.2 P5/中级工程师（2-4 年）

**核心目标：能独立负责一个模块/服务的设计与实现**

| 能力域 | 具体要求 |
|--------|---------|
| Java 深度 | 并发编程（JUC 核心类、线程安全、锁机制）、JVM 调优入门 |
| 框架原理 | Spring AOP/事务原理、SpringBoot 自动配置、MyBatis 插件机制 |
| 数据库 | 索引优化、查询分析（EXPLAIN）、事务隔离级别、分页优化 |
| 中间件 | Redis 数据结构选型、缓存三大问题、Kafka/RocketMQ 可靠性保证 |
| 架构入门 | 服务拆分基本原则、RESTful API 设计、接口幂等性 |
| 工程化 | Code Review、CI/CD 基本配置、监控告警接入 |
| 分布式 | 分布式锁（Redis/ZK）、分布式事务了解、Seata 基本使用 |

**成长瓶颈：** 中级工程师的常见瓶颈是"技术深度不够"——会用但不懂原理，遇到复杂问题无从下手。

---

### 2.3 P6/高级工程师（4-7 年）

**核心目标：能主导系统设计，解决复杂技术问题**

| 能力域 | 具体要求 |
|--------|---------|
| Java 专精 | JVM 调优实战（GC 分析/内存泄漏排查）、并发编程专家级 |
| 系统设计 | 高可用（99.99%）、高并发（万级 TPS）、微服务拆分与治理 |
| 性能优化 | 全链路性能分析、SQL 优化、JVM 参数调优、系统瓶颈定位 |
| 架构视野 | 常见架构模式对比、CAP 定理实践、分布式系统核心问题 |
| 中间件深度 | 理解中间件内核（Kafka Partition/Replication、Redis Cluster）|
| 工程质量 | 技术方案设计能力、架构评审、测试策略、技术债管理 |
| 技术影响力 | 技术分享/写作、方案决策、Code Review 能带动团队提升 |

**成长瓶颈：** 高级工程师的瓶颈通常是"影响力"——技术够强，但无法让团队看见你的价值。

---

### 2.4 P7/资深工程师（7+ 年）

**核心目标：能推动技术战略，解决组织级技术问题**

| 能力域 | 具体要求 |
|--------|---------|
| 架构设计 | 复杂系统的整体架构设计（10w+ TPS、跨机房容灾）|
| 技术判断 | 技术选型与 trade-off 决策，平衡业务需求与技术债 |
| 技术规划 | 中长期技术路线规划（季度/年度）|
| 组织能力 | 团队技术能力建设、人才培养、标准制定 |
| 跨团队影响 | 推动跨团队技术协作，解决组织边界问题 |
| 业务结合 | 深度理解业务，能从技术视角提出业务创新方向 |

## 三、技术能力详细分解

### 3.1 Java 核心技术栈

```
┌─ 语言基础
│   ├── 集合框架（内部实现、线程安全、选型）
│   ├── 异常体系（Checked vs Unchecked、最佳实践）
│   ├── 泛型（类型擦除、边界通配符）
│   ├── 反射（Class/Method/Field 操作、性能影响）
│   ├── Lambda & Stream（函数式编程）
│   └── Java 新特性（record/sealed/pattern matching）
│
├─ 并发编程
│   ├── 线程基础（Thread 生命周期、Daemon）
│   ├── JMM（happens-before、volatile）
│   ├── synchronized（偏向锁/轻量锁/重量锁升级）
│   ├── AQS（ReentrantLock/Semaphore/CountDownLatch）
│   ├── 线程池（ThreadPoolExecutor 7个参数、拒绝策略）
│   ├── 并发集合（ConcurrentHashMap/CopyOnWriteArrayList）
│   └── CompletableFuture（异步编排）
│
└─ JVM
    ├── 内存结构（堆/栈/方法区/直接内存）
    ├── 类加载（双亲委派、自定义 ClassLoader）
    ├── GC 算法（标记-清除/标记-整理/复制）
    ├── GC 收集器（Serial/Parallel/CMS/G1/ZGC/Shenandoah）
    └── JVM 调优（参数配置、OOM 排查、GC 日志分析）
```

### 3.2 框架能力矩阵

| 框架 | 初级 | 中级 | 高级 |
|------|------|------|------|
| Spring Core | CRUD/注解 | IoC/AOP 原理 | 扩展点/BeanDefinition |
| Spring Boot | 快速开发 | 自动配置/starter | 自定义 Starter |
| Spring MVC | Controller 编写 | 拦截器/Filter | DispatcherServlet 流程 |
| MyBatis | XML/注解 CRUD | 插件/二级缓存 | 源码定制 |
| Spring Cloud | Feign/Gateway 使用 | 熔断/限流 | 微服务治理架构 |
| Spring Security | 认证/授权配置 | JWT/OAuth2 | 自定义扩展点 |

## 四、软技能评估

### 4.1 沟通能力

```
评估维度：
- 技术方案能否用非技术语言向产品/运营解释？
- 在 Code Review 中，能否给出建设性而非批判性意见？
- 发邮件/写文档，能否逻辑清晰、重点突出？
- 跨团队协作时，能否快速建立信任关系？
```

**自测题：** 把一个复杂技术决策（比如"为什么用 Kafka 不用 RabbitMQ"）用 3 句话解释给非技术人员听。

### 4.2 项目推进能力

```
评估维度：
- 能否分解任务、制定合理计划？
- 遇到阻塞，是主动推进还是等待？
- 能否识别项目风险，提前预警？
- 上线前的 Checklist 是否完整？
```

### 4.3 技术判断力（最难培养）

```
技术判断力 = 见过足够多的方案 × 踩过足够多的坑 × 对业务的理解深度

表现：
- 能说清楚某个技术决策的 trade-off
- 能识别"银弹"（过度设计/不必要复杂性）
- 能区分"学术最优"和"工程最优"
- 能在时间/质量/成本之间做出合理取舍
```

## 五、能力自测方法

### 5.1 面向自己的问题

每隔半年，认真回答以下问题：

```
技术深度：
□ 我上一次深入读某个框架/中间件源码是什么时候？
□ 我能解释我每天使用的 3 个核心技术的底层实现吗？
□ 我上一次解决的最复杂技术问题是什么？

技术广度：
□ 在我当前技术栈之外，我了解哪些其他方向？
□ 我有没有关注过技术趋势（CNCF 毕业项目/Java 新版特性）？

工程能力：
□ 我的代码被 Review 后，有没有反复出现同类问题？
□ 我写的设计文档，同事能看懂并提出有效反馈吗？

影响力：
□ 我有没有帮助团队成员解决过他们自己搞不定的问题？
□ 我最近一次分享技术是什么时候？
```

### 5.2 雷达图自评

为以下 8 个维度打分（1-10）：

```
        Java 核心(8)
            /\
影响力(5) /  \ 分布式(7)
        /    \
工程化(7)----架构(6)
        \    /
数据库(8) \  / 中间件(7)
            \/
         框架(9)
```

看雷达图的最低点，那就是你最需要补强的方向。

## 六、不同阶段的成长重点

| 阶段 | 重点方向 | 常见错误 |
|------|---------|---------|
| 0-1 年 | 打牢基础，学会快速交付 | 急于"深入"，基础没有扎实 |
| 1-3 年 | 深入原理，理解"为什么" | 跳槽频繁，没有在一个系统上深耕 |
| 3-5 年 | 系统设计，提升架构能力 | 只顾技术深度，忽视业务理解 |
| 5-8 年 | 建立影响力，培养团队 | 单打独斗，没有放大个人能力 |
| 8年+ | 技术战略，跨界融合 | 管理与技术两边都做不精 |
