## TCP 三次握手

**关联知识文档**：[[10_计算机基础/01_网络协议/TCP]]

**第一问**：「TCP 三次握手的过程是什么？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 为什么是三次而不是两次？ | 防止已失效的连接请求被服务端接受 |
| 第二层 | 第三次 ACK 丢失了会怎样？ | 服务端重传 SYN+ACK，客户端重发 ACK |
| 第三层 | ISN（初始序列号）为什么是随机的？ | 防止序列号预测攻击（伪造 TCP 报文）|

**实战型考法**：「你用 Wireshark 抓过三次握手包吗？看到了什么？」

**面试话术总结**：

- **三次握手过程**：
  1. Client → Server：`SYN=1, seq=x`（客户端进入 SYN_SENT）
  2. Server → Client：`SYN=1, ACK=1, seq=y, ack=x+1`（服务端进入 SYN_RCVD）
  3. Client → Server：`ACK=1, seq=x+1, ack=y+1`（双方进入 ESTABLISHED）
- **为什么不能两次**：如果只有两次，客户端第一次 SYN 因网络延迟到达服务端 → 服务端建立连接并分配资源 → 客户端已超时关闭 → 但服务端一直在等 → 资源浪费（历史连接攻击）
- **为什么不能四次**：第二步的 SYN+ACK 可以合并为一步，减少一次 RTT
- **第三次 ACK 丢失**：服务端未收到 ACK → 超时重传 SYN+ACK → 客户端收到后重发 ACK。如果客户端已发送数据，数据包中的 ACK 也能确认

## TCP 四次挥手

**关联知识文档**：[[10_计算机基础/01_网络协议/TCP]]

**第一问**：「TCP 断开连接为什么要四次挥手？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | FIN 和 CLOSE_WAIT 的关系？ | 被动关闭方收到 FIN 后进入 CLOSE_WAIT |
| 第二层 | TIME_WAIT 为什么等 2MSL？ | 确保最后一个 ACK 到达 / 让旧报文消失 |
| 第三层 | 大量 TIME_WAIT 怎么解决？ | `tcp_tw_reuse` / `SO_REUSEADDR` / 调整 MSL |

**面试话术总结**：

- **四次挥手过程**：
  1. A → B：`FIN=1`（A → FIN_WAIT_1）
  2. B → A：`ACK=1`（B → CLOSE_WAIT，A → FIN_WAIT_2）
  3. B → A：`FIN=1`（B → LAST_ACK）
  4. A → B：`ACK=1`（A → TIME_WAIT，等 2MSL 后关闭）
- **为什么四次**：TCP 是全双工的，每个方向都需要单独关闭。被动关闭方收到 FIN 后可能还有数据要发送（步骤 2 和步骤 3 之间可能有数据传输）
- **TIME_WAIT 等待 2MSL 的原因**：
  1. 确保最后一个 ACK 能到达服务端（如果丢失，服务端会重传 FIN，A 在 TIME_WAIT 可以重发 ACK）
  2. 让本次连接的所有报文在网络中消失（MSL 是报文最大生存时间），防止旧连接的报文被新连接误接收
- **大量 TIME_WAIT 危害**：占用连接数（端口耗尽）、消耗内存。解决：`net.ipv4.tcp_tw_reuse=1`（允许 TIME_WAIT socket 复用）；客户端短连接用 `SO_REUSEADDR`

## TCP 与 UDP

**关联知识文档**：[[10_计算机基础/01_网络协议/TCP]]

**第一问**：「TCP 和 UDP 有什么区别？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | TCP 的可靠性是怎么保证的？ | 序号 / 确认应答 / 重传 / 校验和 |
| 第二层 | UDP 能做到可靠传输吗？ | QUIC 协议 / 应用层重传 |
| 第三层 | 为什么视频通话用 UDP 而不是 TCP？ | 实时性 > 可靠性 |

**面试话术总结**：

- **核心区别**：
  | 维度 | TCP | UDP |
  |------|-----|-----|
  | 连接 | 面向连接（三次握手）| 无连接 |
  | 可靠性 | 可靠（确认/重传/排序）| 不可靠（尽最大努力交付）|
  | 速度 | 慢（首部 20B、确认机制）| 快（首部 8B、无控制机制）|
  | 模式 | 一对一 | 一对一/一对多/多对多 |
  | 应用 | HTTP/HTTPS/FTP/SMTP | DNS/DHCP/SNMP/视频/游戏 |
- **TCP 可靠性机制**：序号（排序）、确认应答（ACK）、超时重传、滑动窗口（流量控制）、拥塞控制（慢启动/拥塞避免）
- **QUIC**（HTTP/3）：基于 UDP 实现可靠传输，0-RTT 握手、无队头阻塞、连接迁移（切换 WiFi 不中断）

## HTTP 与 HTTPS

**关联知识文档**：[[10_计算机基础/01_网络协议/HTTP协议详解]]

**第一问**：「HTTPS 比 HTTP 多了什么？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | TLS 握手的过程？ | ClientHello → ServerHello → 密钥交换 → Finished |
| 第二层 | 对称加密和非对称加密在 TLS 里分别用在哪？ | 握手用非对称，传输用对称 |
| 第三层 | 怎么验证证书合法性？ | CA 签名链 / CRL / OCSP |

**实战型考法**：「你配置过 HTTPS 证书吗？用的是什么证书？」

**面试话术总结**：

- **HTTPS = HTTP + TLS**：在传输层和应用层之间加了 TLS 加密层
- **TLS 1.2 握手流程**：
  1. ClientHello：支持的 TLS 版本、加密套件列表、随机数
  2. ServerHello：选定 TLS 版本、加密套件、随机数 + 服务端证书
  3. 客户端验证证书（CA 签名链）
  4. 密钥交换：客户端用服务端公钥加密「预主密钥」发送（RSA 方式）或 ECDHE 交换
  5. 双方用预主密钥 + 两个随机数生成「会话密钥」（对称密钥）
  6. Finished：双方用会话密钥加密验证消息，握手完成
- **混合加密**：非对称加密（RSA/ECC）用于握手阶段交换密钥（密钥短、次数少）；对称加密（AES）用于数据传输（性能高）
- **证书验证**：浏览器内置根 CA 证书 → 服务端证书由 CA 签名 → 用 CA 公钥验证签名 → 逐级向上验证到根 CA

## HTTP 协议演进

**关联知识文档**：[[10_计算机基础/01_网络协议/HTTP协议详解]]

**第一问**：「HTTP/1.1、HTTP/2、HTTP/3 有什么区别？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | HTTP/1.1 的队头阻塞问题是什么？ | 一个 TCP 连接上同一时刻只能有一个请求-响应 |
| 第二层 | HTTP/2 的多路复用怎么解决的？ | 二进制帧 + 流（Stream）|
| 第三层 | HTTP/3 为什么基于 UDP？ | TCP 层面的队头阻塞无法解决 → QUIC（UDP）|

**面试话术总结**：

- **HTTP/1.1**：长连接（`Connection: keep-alive`）+ 管道化（Pipelining），但管道化有队头阻塞（前一个响应慢，后面的都等着）
- **HTTP/2 核心特性**：
  - **二进制帧**：报文拆分为多个帧，帧有类型（HEADERS/DATA）
  - **多路复用**：一个 TCP 连接上多个 Stream 并行交错传输，解决 HTTP 层队头阻塞
  - **头部压缩**：HPACK 算法压缩重复的 HTTP Header
  - **Server Push**：服务器可主动推送资源
- **HTTP/3 核心变化**：基于 QUIC（UDP），解决 TCP 层面的队头阻塞（HTTP/2 多个 Stream 共用一个 TCP 连接，TCP 层丢包重传会阻塞所有 Stream）；QUIC 0-RTT/1-RTT 握手；内置 TLS 1.3

## TCP 滑动窗口

**关联知识文档**：[[10_计算机基础/01_网络协议/TCP]]

**第一问**：「TCP 的滑动窗口是做什么用的？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 发送窗口大小由谁决定？ | min(rwnd, cwnd)：接收窗口 + 拥塞窗口 |
| 第二层 | 零窗口探测是什么？ | 接收方缓冲区满后，发送方周期性发送探测包 |
| 第三层 | Nagle 算法是什么？有什么问题？ | 小包合并发送 / 和延迟确认的冲突 |

**面试话术总结**：

- **滑动窗口**：发送方维护发送窗口（已发送已确认 / 已发送未确认 / 可发送未发送 / 不可发送），窗口右沿随 ACK 确认不断右滑
- **流量控制**（接收方）：接收方通过 ACK 报文的 `window` 字段告知发送方自己的接收缓冲区剩余大小，发送方据此调整发送速率
- **拥塞控制**（网络）：发送方根据网络拥塞程度调整 `cwnd`（拥塞窗口），和 `rwnd` 取小值作为实际发送窗口
- **零窗口**：接收方 `window=0` → 发送方停止发送 → 定时发送零窗口探测包（1 字节）→ 接收方 ACK 回复新的窗口大小
- **Nagle 算法**：小包攒够一定量或等到 ACK 回来才发送，减少小包数量。和 TCP 延迟确认（Delayed ACK，200ms 内等数据一起 ACK）可能产生冲突：发送方等 ACK → 接收方等数据 → 死锁。解决：`TCP_NODELAY` 禁用 Nagle

## TCP 拥塞控制

**关联知识文档**：[[10_计算机基础/01_网络协议/TCP]]

**第一问**：「TCP 的拥塞控制算法有哪些？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 慢启动的 ssthresh 是怎么变化的？ | 超时 → ssthresh = cwnd/2，cwnd = 1；快重传 → ssthresh = cwnd/2，cwnd = ssthresh |
| 第二层 | 快重传和快恢复是什么？ | 3 个重复 ACK → 不等超时，直接重传 |
| 第三层 | BBR 算法和 CUBIC 的核心区别？ | 基于带宽估计 vs 基于丢包 |

**实战型考法**：「你调过 TCP 的拥塞控制算法吗？在哪里调的？」

**面试话术总结**：

- **四个阶段**（必须能口述）：
  1. **慢启动**（Slow Start）：`cwnd = 1`，每收到一个 ACK，`cwnd += 1`（指数增长），直到 `cwnd >= ssthresh`
  2. **拥塞避免**（Congestion Avoidance）：`cwnd` 线性增长（每个 RTT 加 1）
  3. **快重传**（Fast Retransmit）：收到 3 个重复 ACK → 立即重传丢失包（不等超时）
  4. **快恢复**（Fast Recovery）：`ssthresh = cwnd/2`，`cwnd = ssthresh`，直接进入拥塞避免（不回到慢启动）
- **超时 vs 快重传**：超时（RTO）说明网络严重拥塞 → `ssthresh = cwnd/2`，`cwnd = 1`（回到慢启动）；3 个重复 ACK 说明轻度拥塞 → `ssthresh = cwnd/2`，`cwnd = ssthresh`（快恢复）
- **BBR vs CUBIC**：CUBIC（Linux 默认）基于丢包判断拥塞（丢包 → 减小窗口），在高丢包网络（如无线网络）过于保守；BBR（Google）基于带宽和 RTT 估计，不依赖丢包信号，在高丢包和高带宽网络下表现更好

## Cookie 与 Session

**关联知识文档**：[[10_计算机基础/01_网络协议/HTTP协议详解]]

**第一问**：「Cookie 和 Session 有什么区别？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | Session 是怎么和客户端关联的？ | Session ID 存在 Cookie 里 |
| 第二层 | 如果客户端禁用 Cookie，Session 还能用吗？ | URL 重写（`jsessionid`）|
| 第三层 | 分布式场景下 Session 怎么处理？ | Session 复制 / 集中存储（Redis）/ Spring Session |

**实战型考法**：「你的项目里登录状态是怎么管理的？JWT 还是 Session？」

**面试话术总结**：

- **Cookie**：客户端存储，大小 4KB 限制，可设过期时间，每次请求自动携带。安全性：`HttpOnly`（防 XSS 读取）、`Secure`（仅 HTTPS）、`SameSite`（防 CSRF）
- **Session**：服务端存储（内存/Redis），通过 Cookie 中的 `JSESSIONID` 关联。无大小限制，但占用服务端资源
- **JWT vs Session**：
  | 维度 | Session | JWT |
  |------|---------|-----|
  | 存储 | 服务端 | 客户端（Token 自包含）|
  | 扩展性 | 需要共享 Session（Redis）| 天然无状态 |
  | 注销 | 服务端删除即可 | 无法主动失效（需黑名单）|
  | 安全性 | Session ID 泄露风险 | Token 泄露风险 + 无法撤销 |
- **分布式 Session**：Session 粘滞（Nginx ip_hash）、Session 复制（Tomcat 集群间同步，性能差）、Spring Session + Redis（推荐）

## 进程与线程

**关联知识文档**：[[10_计算机基础/02_操作系统/进程与线程]]

**第一问**：「进程和线程有什么区别？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 线程共享哪些资源？独享哪些资源？ | 共享堆/代码段；独享栈/PC/寄存器 |
| 第二层 | 进程间通信有哪些方式？ | 管道/消息队列/共享内存/信号量/Socket |
| 第三层 | 协程和线程有什么区别？ | 用户态调度 vs 内核态调度 |

**面试话术总结**：

- **进程 vs 线程**：
  | 维度 | 进程 | 线程 |
  |------|------|------|
  | 定义 | 资源分配的基本单位 | CPU 调度的基本单位 |
  | 地址空间 | 独立 | 共享所属进程的地址空间 |
  | 创建/销毁 | 重（分配独立地址空间）| 轻（共享地址空间）|
  | 切换开销 | 大（上下文切换涉及地址空间）| 小（只需切换寄存器和栈）|
  | 通信 | IPC（管道/消息队列/共享内存）| 直接共享内存（需要同步机制）|
- **进程间通信**：管道（半双工，父子进程）、命名管道（FIFO，无亲缘关系）、消息队列（消息链表）、共享内存（最快，需同步）、信号量（同步机制）、Socket（网络通信）
- **协程**：用户态轻量级线程，由程序自行调度（不涉及内核切换），切换成本极低。一个线程可以包含多个协程。Go 的 goroutine、Kotlin 协程、Java 虚拟线程（Project Loom）都是协程的实现
