# 架构设计案例分析

> 架构设计题 = **需求理解 + 维度拆解 + 权衡取舍 + 估算支撑**。不要急着画架构图，先把问题想清楚。

---

## 通用解题思路（STAR 法用于设计题）

```
① 澄清需求（Scope）
   - 用户量 / QPS / 数据量
   - 核心功能 vs 非核心功能
   - 一致性要求 / 可用性要求

② 估算规模（Scale）
   - QPS 峰值 = 日活 × 平均请求 / 86400 × 峰值系数
   - 存储量 = 单条大小 × 日增量 × 保留天数
   - 带宽 = QPS × 平均响应大小

③ 设计架构（Design）
   - 分层：接入层 / 业务层 / 数据层 / 缓存层
   - 拆服务：按领域拆分（DDD）
   - 高可用：主从 / 集群 / 限流 / 熔断
   - 高性能：缓存 / 异步 / CDN

④ 深入细节（Deep Dive）
   - 数据库 schema 设计
   - 缓存策略
   - 消息队列使用
   - 监控告警

⑤ 演进路线（Evolution）
   - V1：MVP，快速上线
   - V2：性能优化
   - V3：容灾 / 多活
```

## 案例 A：设计一个短链接服务

### 需求澄清

```
功能：输入长链接 → 生成短链接 → 重定向
非功能：高可用、低延迟、可统计点击量
规模假设：
  - 日活 100 万，人均生成 0.1 个短链接
  - 日生成 10 万条，日点击 1000 万次
  - QPS（读）≈ 1000万 / 86400 ≈ 116，峰值 ≈ 1000
  - 存储 10 万条 / 天 × 365 天 ≈ 3650 万条
```

### 核心设计

```
┌──────────┐   写       ┌──────────┐
│  Client  │ ────────→ │  API     │
│          │             │  Gateway │
└──────────┘   读       │          │
       ↑                └────┬─────┘
       │                     │
       │              ┌──────▼────────┐
       └────────────  │  ShortLink    │
                      │  Service      │
                      └──────┬────────┘
                             │
               ┌─────────────┼─────────────┐
               ▼             ▼             ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │  MySQL   │ │  Redis   │ │  Kafaka  │
        │  (存储)  │ │  (缓存)  │ │  (异步)  │
        └──────────┘ └──────────┘ └──────────┘
```

### 短码生成

```java
// 方案1：自增 ID + Base62 编码（简单）
public String generateShortCode(Long id) {
    String base62 = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ";
    StringBuilder sb = new StringBuilder();
    while (id > 0) {
        sb.append(base62.charAt((int)(id % 62)));
        id /= 62;
    }
    return sb.reverse().toString();
}
// 问题：ID 可预测，容易被遍历

// 方案2：雪花算法 ID + Base62（推荐）
public String generateShortCode() {
    long id = snowflake.nextId();
    return Base62.encode(id);
}
// 问题：需要 Base62 库

// 方案3：随机字符串 + 冲突重试
public String generateShortCode() {
    String code;
    do {
        code = RandomStringUtils.randomAlphanumeric(6);
    } while (exists(code));  // 检查冲突
    return code;
}
```

### 重定向实现

```java
@RestController
public class RedirectController {
    @Autowired
    private RedisTemplate<String, String> redis;
    
    @Autowired
    private ShortLinkMapper mapper;
    
    @GetMapping("/{shortCode}")
    public ResponseEntity<Void> redirect(@PathVariable String shortCode) {
        // 1. 查缓存
        String longUrl = redis.opsForValue().get("short:" + shortCode);
        if (longUrl == null) {
            // 2. 查 DB
            ShortLink link = mapper.findByShortCode(shortCode);
            if (link == null) {
                return ResponseEntity.notFound().build();
            }
            longUrl = link.getLongUrl();
            redis.opsForValue().set("short:" + shortCode, longUrl, 24, TimeUnit.HOURS);
        }
        
        // 3. 异步记录点击
        kafkaTemplate.send("click-event", shortCode, 1L);
        
        // 4. 302 重定向
        return ResponseEntity.status(302)
                .header("Location", longUrl)
                .build();
    }
}
```

### 数据库设计

```sql
CREATE TABLE short_link (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    short_code VARCHAR(10) NOT NULL UNIQUE,
    long_url TEXT NOT NULL,
    created_at DATETIME NOT NULL,
    expire_at DATETIME NULL,
    click_count BIGINT DEFAULT 0,
    INDEX idx_short_code (short_code),
    INDEX idx_created_at (created_at)
```

### 四种算法对比

| 算法 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| 固定窗口 | 单位时间计数 | 简单 | 突刺问题 | 简单限流 |
| 滑动窗口 | 更细粒度的窗口 | 比固定窗口精确 | 内存占用高 | 中等精度 |
| 漏桶（Leaky Bucket） | 固定速率出水 | 流量平滑 | 无法应对突发 | 保护下游 |
| 令牌桶（Token Bucket） | 固定速率放令牌 | 允许突发 | 实现稍复杂 | API 限流（推荐） |

### 令牌桶实现

```java
// Guava RateLimiter（最简单）
@RestController
public class ApiController {
    private final RateLimiter rateLimiter = RateLimiter.create(100.0);  // 100 QPS
    
    @GetMapping("/api/data")
    public ResponseEntity<Data> getData() {
        if (!rateLimiter.tryAcquire()) {
            return ResponseEntity.status(429).body(null);  // Too Many Requests
        }
        return ResponseEntity.ok(queryData());
    }
}
```

```java
// Redis + Lua 分布式限流（推荐，集群场景）
@Component
public class RedisRateLimiter {
    private static final String LUA_SCRIPT =
        "local key = KEYS[1] " +
        "local limit = tonumber(ARGV[1]) " +
        "local window = tonumber(ARGV[2]) " +
        "local current = redis.call('get', key) " +
        "if current and tonumber(current) > limit then " +
        "    return 0 " +
        "end " +
        "redis.call('incr', key) " +
        "if tonumber(redis.call('get', key)) == 1 then " +
        "    redis.call('expire', key, window) " +
        "end " +
        "return 1";
    
    public boolean tryAcquire(String key, int limit, int windowSeconds) {
        Long result = (Long) redisTemplate.execute(
            new DefaultRedisScript<>(LUA_SCRIPT, Long.class),
            Collections.singletonList(key),
            String.valueOf(limit),
            String.valueOf(windowSeconds)
        );
        return result == 1L;
    }
}
```

```java
// 更新分数
public void updateScore(Long userId, double score) {
    String key = "rank:leaderboard";
    redisTemplate.opsForZSet().add(key, userId.toString(), score);
}

// 获取 Top N
public List<RankItem> getTopN(int n) {
    String key = "rank:leaderboard";
    Set<ZSetOperations.TypedTuple<String>> results = 
        redisTemplate.opsForZSet().reverseRangeWithScores(key, 0, n - 1);
    
    List<RankItem> ranks = new ArrayList<>();
    int rank = 1;
    for (ZSetOperations.TypedTuple<String> item : results) {
        ranks.add(new RankItem(
            Long.parseLong(item.getValue()),
            item.getScore(),
            rank++
        ));
    }
    return ranks;
}

// 获取用户排名
public Long getUserRank(Long userId) {
    String key = "rank:leaderboard";
    Long rank = redisTemplate.opsForZSet().reverseRank(key, userId.toString());
    return rank == null ? -1 : rank + 1;  // 排名从 1 开始
}
```

```
                用户

         ┌────────────────┐
         │   CDN + 静态页  │
         └────────┬───────┘
                  │
                  ▼
         ┌────────────────┐
         │   API 网关      │
         │  （限流+验证码）│

                  │
        ┌─────────┼─────────┐
        ▼         ▼         ▼
   ┌────────┐┌────────┐┌────────┐
   │ Redis  ││ 秒杀   ││  MQ   │
   │ 预减库存││ 服务   ││ 异步  │
   └────────┘└───┬────┘└───┬───┘
                  │         │
                  │   ┌─────▼─────┐
                  │   │  订单服务  │
                  │   └─────┬─────┘
                  │         │
                  │   ┌─────▼─────┐
                  │   │  支付服务  │
                  │   └─────┬─────┘
                  │         │
                  ▼   ┌─────▼─────┐
             ┌────────┐ │  数据库   │
             │ MySQL  │ │  (订单/  │
             │ (商品) │ │   支付)  │
             └────────┘ └──────────┘
```

### 关键设计点

```
1. 静态化：秒杀页面静态化，放 CDN
2. 隔离：秒杀服务独立部署，不影响主业务
3. 削峰：MQ 异步下单，前端轮询结果
4. 限流：多层限流（IP、用户 ID、接口级）
5. 降级：Redis 挂了→降级到 DB 限流（悲观锁）
6. 防刷：验证码 + 风控 + IP 黑名单
```

## 架构设计 Checklist

```
□ 需求澄清了吗？ → 功能 / 非功能 / 规模估算
□ 存储量估算了吗？ → DB 磁盘 / 缓存内存 / 带宽
□ 有缓存层吗？ → Redis 缓存策略 / 缓存一致性
□ 有异步层吗？ → MQ 削峰 / 解耦
□ 有限流吗？ → 令牌桶 / 漏桶 / 固定窗口
□ 有降级吗？ → 熔断 / 降级开关 / 兜底数据
□ 有监控吗？ → 关键指标 / 告警规则
□ 有安全吗？ → 限流防刷 / SQL 注入 / XSS
□ 有演进路线吗？ → V1/V2/V3 分阶段
```

## 追问链

### 追问 1：设计一个系统，怎么开头？

> "先问清楚规模和约束，不要直接画图。问：用户量多少？读写比例？数据量多大？一致性要求？可用性要求？估算完规模，再决定用不用缓存、用不用 MQ、用不用分库分表。架构是 trade-off，不是堆技术。"

### 追问 2：分布式 ID 怎么生成？

> "1. 雪花算法（推荐，趋势递增，性能好）。2. 数据库自增（简单，但有单点）。3. 号段模式（Leaf，批量取号，减少 DB 压力）。4. UUID（无序，索引分裂，不推荐）。5. Redis INCR（简单，依赖 Redis）。"

### 追问 3：怎么保证缓存和 DB 一致性？

> "Cache Aside 模式：写时先更新 DB，再删除缓存；读时先读缓存，miss 则读 DB 并写回缓存。一致性是最终一致，不是强一致。对一致性要求高的场景，用分布式锁或版本号。"
