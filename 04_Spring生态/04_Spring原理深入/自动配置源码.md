# 自动配置源码

## 这个问题为什么存在？

> Spring Boot 能做到「引入依赖即生效」，靠的是**自动配置**。但自动配置不是魔法，它有一套**严格的源码级机制**：  
> 怎么发现自动配置类？怎么按需加载？怎么保证用户定义的 Bean 优先？

**核心问题**：`@EnableAutoConfiguration` 是怎么把「依赖 Jar 里的配置类」自动注册到容器的？

---

## 它是怎么解决问题的？

### @EnableAutoConfiguration 的入口

```java
@SpringBootApplication
// 等同于：
@Configuration
@EnableAutoConfiguration   // ★ 关键
@ComponentScan

// @EnableAutoConfiguration 的定义：
@Import(AutoConfigurationImportSelector.class)
public @interface EnableAutoConfiguration {}
```

> **核心**：`@Import(AutoConfigurationImportSelector.class)` ——  
> 把 `AutoConfigurationImportSelector` 导入到容器，它负责**选择并导入**所有自动配置类。

---

### AutoConfigurationImportSelector 的工作流程

```
@EnableAutoConfiguration 触发
    ↓
AutoConfigurationImportSelector.selectImports()
    ↓
【1】从配置文件读取所有自动配置类
    ↓
【2】去重（去掉重复的配置类）
    ↓
【3】排除（@SpringBootApplication.exclude 指定的类）
    ↓
【4】过滤（根据条件注解，去掉不生效的配置类）
    ↓
【5】排序（@AutoConfigureAfter / @AutoConfigureBefore）
    ↓
【6】返回要导入的配置类列表 → Spring 注册这些配置类
```

---

### 第 1 步详解：读取自动配置类

**Spring Boot 2.7 之前**（spring.factories）：

```java
// AutoConfigurationImportSelector.getCandidateConfigurations()
List<String> configurations = SpringFactoriesLoader.loadFactoryNames(
    EnableAutoConfiguration.class, classLoader);

// 读取所有 Jar 中的 META-INF/spring.factories：
# spring-boot-autoconfigure-2.7.0.jar!/META-INF/spring.factories
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
  org.springframework.boot.autoconfigure.web.servlet.WebMvcAutoConfiguration,\
  org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration,\
  ...
```

**Spring Boot 2.7 之后**（imports 文件，推荐）：

```java
// 读取所有 Jar 中的 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
// 每行一个自动配置类：
org.springframework.boot.autoconfigure.web.servlet.WebMvcAutoConfiguration
org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration
```

---

### 第 4 步详解：条件过滤（ConditionEvaluator）

**问题**：读取了 100+ 个自动配置类，但项目只引入了 Web 依赖，其他的不应该生效。

**解法**：条件注解（`@ConditionalOnClass`、`@ConditionalOnMissingBean`...）。

```java
// WebMvcAutoConfiguration 的条件
@Configuration
@ConditionalOnClass({ Servlet.class, DispatcherServlet.class })
// ↑ 只有 classpath 有 Servlet API，这个配置类才生效

@ConditionalOnMissingBean(WebMvcConfigurationSupport.class)
// ↑ 只有容器里没有 WebMvcConfigurationSupport Bean，才用自动配置的
public class WebMvcAutoConfiguration { ... }
```

**过滤流程**（源码路径）：

```java
// AutoConfigurationImportSelector.filter()
List<String> filter(List<String> configurations) {
    for (AutoConfigurationImportFilter filter : getAutoConfigurationImportFilters()) {
        // 每个 filter 判断配置类是否满足条件
        // 例如：OnClassCondition（检查 @ConditionalOnClass）
        boolean[] match = filter.match(candidates, this.autoConfigurationMetadata);
        // match[i] = false → 过滤掉第 i 个配置类
    }
}
```

---

### 第 5 步详解：排序（@AutoConfigureAfter）

**问题**：`WebMvcAutoConfiguration` 要在 `HttpMessageConvertersAutoConfiguration` 之后才生效（因为要用到 HttpMessageConverter）。

**解法**：`@AutoConfigureAfter` / `@AutoConfigureBefore`。

```java
@Configuration
@AutoConfigureAfter(HttpMessageConvertersAutoConfiguration.class)
public class WebMvcAutoConfiguration { ... }
```

> Spring 在导入自动配置类时，会按照这些注解**排序**，  
> 保证依赖的配置类先被处理。

---

## 它和相似方案的本质区别是什么？

### Spring Boot 自动配置 vs 传统 Spring XML 配置

| | Spring Boot 自动配置 | 传统 Spring XML 配置 |
|---|---|---|
| 配置方式 | 条件注解（`@Conditional`） | 手动编写 `<bean>` |
| 按需加载 | ✅（条件满足才生效） | ❌（全部加载） |
| 覆盖方式 | 定义同名 Bean 即可覆盖 | 需要 `<bean>` 覆盖（复杂） |
| 学习曲线 | 中（需要理解条件注解） | 低（直接写 XML） |

**关键区别**：自动配置的 Bean 都是 **`@ConditionalOnMissingBean`** —— 如果用户定义了同名 Bean，自动配置的**不生效**（用户定义的优先）。

## 正确使用方式

### 1. 排除特定的自动配置类

```java
// 方式 1：@SpringBootApplication.exclude
@SpringBootApplication(exclude = DataSourceAutoConfiguration.class)
public class Application { ... }

// 方式 2：配置文件
spring:
  autoconfigure:
    exclude:
      - org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration
```

---

### 2. 自定义自动配置（写自己的 Starter）

```
my-starter/
├──src/main/java
│   └── com/example/autoconfigure/
│       ├── MyServiceAutoConfiguration.java
│       └── MyServiceProperties.java
└── src/main/resources
    └── META-INF/
        └── spring/
            └── org.springframework.boot.autoconfigure.AutoConfiguration.imports
```

```java
// 1. Properties 类
@ConfigurationProperties(prefix = "my.service")
public class MyServiceProperties {
    private String name;
    // getter / setter
}

// 2. AutoConfiguration 类
@Configuration
@ConditionalOnClass(MyService.class)
@EnableConfigurationProperties(MyServiceProperties.class)
public class MyServiceAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean
    public MyService myService(MyServiceProperties props) {
        return new MyService(props.getName());
    }
}

// 3. 注册到 imports 文件
// META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
com.example.autoconfigure.MyServiceAutoConfiguration
```

---

### 3. 查看自动配置报告（debug = true）

```bash
# 启动时加 --debug
java -jar app.jar --debug

# 控制台输出：
# POSITIVE_MATCHES（生效的自动配置）
# NEGATIVE_MATCHES（未生效的自动配置 + 原因）
```

> **这是排查自动配置问题的最强工具**！  
> 可以看到每一个自动配置类**为什么生效 / 为什么不生效**。

## 边界情况和坑

### 1. 自动配置类被意外覆盖

```java
// ❌ 错误：定义了和自动配置同名的 Bean，自动配置失效
@Bean
public DataSource dataSource() {
    return new HikariDataSource();  // 覆盖了 DataSourceAutoConfiguration
}
```

> 自动配置类的 Bean 都有 `@ConditionalOnMissingBean` ——  
> **用户定义的同名 Bean 会覆盖自动配置的**。

---

### 2. 自动配置类的执行顺序问题

```java
// ❌ 错误：自定义配置类依赖自动配置的 Bean，但顺序不对
@Configuration
public class MyConfig {

    @Bean
    public MyBean myBean(WebMvcConfigurer webMvcConfigurer) {
        // ❌ WebMvcAutoConfiguration 还没执行，WebMvcConfigurer 不存在
    }
}
```

**解决**：用 `@AutoConfigureAfter` 或让自定义配置类**不依赖**自动配置的 Bean。

---

### 3. spring.factories 和 imports 文件同时存在

```
Spring Boot 2.7+：
  - spring.factories 仍然被读取（兼容旧版本）
  - 但推荐用 imports 文件
  - 如果两个都有，imports 文件的优先级更高
```

> **最佳实践**：新项目只用 `imports` 文件，`spring.factories` 用于兼容旧版 Spring Boot。
