# 微服务核心概念

## 这个问题为什么存在？

> 单体应用所有功能打包在一个进程里，初期开发快、部署简单。但随着业务增长，**代码耦合严重、扩展困难、部署风险高**。问题是：**怎么让系统既能快速迭代，又能独立扩展？**

微服务架构把单体应用拆成多个**独立部署、松耦合**的服务，每个服务负责一个业务能力。

---

## 它是怎么解决问题的？

### 单体 vs 微服务

```
单体架构：
  ┌────────────────────────────┐
  │  订单模块   库存模块   支付模块  │
  │  （全部在一个进程里）        │
  └──────────┬─────────────────┘
               │
            整体部署（改一行，全量部署）

微服务架构：
  ┌────────┐  ┌────────┐  ┌────────┐
  │订单服务│  │库存服务│  │支付服务│
  └───┬───┘  └───┬───┘  └───┬───┘
      │            │            │
   独立部署    独立扩展    独立扩容
```

**微服务解决的核心问题**：
1. **独立部署**：改订单模块，不影响库存和支付
2. **独立扩展**：订单服务压力大，只扩容订单服务
3. **技术异构**：订单用Java，推荐用Python（AI库丰富）
4. **故障隔离**：支付服务挂了，不影响订单创建

### 微服务架构的核心组件

```
                    ┌──────────┐
        ┌──────────│ API网关   │──────────┐
        │          └──────────┘          │
        ▼                                ▼
  ┌──────────┐                    ┌──────────┐
  │订单服务  │                    │支付服务  │
  └────┬─────┘                    └────┬─────┘
       │                               │
       └──────────┬──────────────────┘
                  │
          ┌───────┴────────┐
          │  服务注册中心    │  ← 服务注册与发现
          │  (Eureka/Nacos) │
          └───────┬────────┘
                  │
          ┌───────┴────────┐
          │  配置中心        │  ← 集中管理配置
          │  (Nacos/Apollo) │
          └───────┬────────┘
                  │
          ┌───────┴────────┐
          │  链路追踪        │  ← 分布式调用链
          │ (SkyWalking)     │
          └─────────────────┘
```

## 它和相似方案的本质区别是什么？

### 微服务 vs 单体架构

| | 单体架构 | 微服务架构 |
|---|---|---|
| 部署 | 整体部署 | 每个服务独立部署 |
| 扩展 | 整体扩展（浪费资源） | 按需扩展（精准） |
| 技术栈 | 必须统一 | 可以异构 |
| 故障影响 | 全部挂掉 | 故障隔离 |
| 开发效率 | 初期高，后期低 | 初期低（架构复杂），后期高 |
| 测试 | 简单（单机） | 复杂（分布式） |

**本质区别**：单体是**集中式思维**，微服务是**分散式思维**。微服务用**复杂度**换**敏捷性**。

### 微服务 vs SOA（面向服务的架构）

| | SOA | 微服务 |
|---|---|---|
| 通信协议 | 重量级（SOAP/ESB） | 轻量级（HTTP/REST/gRPC） |
| 服务粒度 | 粗粒度（企业级） | 细粒度（单个业务能力） |
| 数据管理 | 共享数据库 | 每个服务独立数据库 |
| 耦合度 | 高（ESB总线） | 低（无中心化总线） |

**本质区别**：SOA是**企业级**的集成方案（复杂、重量级），微服务是**互联网级**的敏捷方案（简单、轻量）。

## 正确使用方式

### 微服务拆分的四个维度

```
维度1：业务边界  （Domain-Driven Design，限界上下文）
维度2：团队边界  （两个披萨团队，康威定律）
维度3：变更频率  （频繁变更的服务独立部署）
维度4：资源需求  （CPU密集型 vs IO密集型，分开扩展）
```

### 微服务通信方式

```java
// 同步调用（REST）
@RestController
public class OrderController {
    @Autowired
    private RestTemplate restTemplate;

    @GetMapping("/order/{id}")
    public Order getOrder(@PathVariable Long id) {
        // 调用库存服务（同步，阻塞等待）
        Inventory inventory = restTemplate.getForObject(
                "http://inventory-service/inventory/" + id,
                Inventory.class
        );
        // ...
    }
}

// 异步消息（MQ）
@Autowired
private RabbitTemplate rabbitTemplate;

public void createOrder(Order order) {
    // 发送消息，不阻塞
    rabbitTemplate.convertAndSend("order-queue", order);
}
```

## 边界情况和坑

### 坑1：微服务拆分过细（纳米服务）

```
错误：一个服务只有200行代码，拆了10个服务
后果：
  - 分布式事务复杂（10个服务调用，一个失败全部回滚）
  - 网络开销大（原来方法调用，现在10次网络调用）
  - 运维成本高（10个服务要监控、部署、扩容）
```

**解决方案**：**拆分粒度 = 团队能力**。小团队（<5人）别拆太细，3-5个服务足够。

### 坑2：分布式事务（跨服务的数据一致性）

```
场景：订单创建成功，调用库存扣减失败
后果：订单有了，库存没扣 → 超卖
```

**解决方案**：
1. **Saga模式**（补偿事务）
2. **事务消息**（RocketMQ）
3. **TCC**（Try-Confirm-Cancel，金融场景）

### 坑3：服务雪崩（一个服务挂，全部挂）

```
场景：支付服务响应慢，订单服务的所有线程都阻塞在调用支付
后果：订单服务线程池耗尽，整个系统不可用
```

**解决方案**：
1. **熔断**（Circuit Breaker，Hystrix/Resilience4j）
2. **降级**（支付失败时，订单仍创建，标记为"待支付"）
3. **超时控制**（调用支付设置超时，避免无限等待）
