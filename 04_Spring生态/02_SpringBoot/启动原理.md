# 启动原理

## 这个问题为什么存在？

> 没有 SpringBoot 之前，搭一个 Spring 项目要手动配置 `web.xml`、配 `DispatcherServlet`、配 `ApplicationContext`、引一堆 XML... 繁琐且容易出错。  
> SpringBoot 的目标是：**一行 `main` 方法启动一个 Web 应用**。

**核心问题**：SpringBoot 是怎么做到「零配置」启动的？

---

## 它是怎么解决问题的？

### SpringApplication.run() 全流程

```java
@SpringBootApplication
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}
```

`SpringApplication.run()` 内部做了什么？

```
SpringApplication.run()
│
├─ 1. 推断应用类型（SERVLET / REACTIVE）
├─ 2. 加载 ApplicationContextInitializer（META-INF/spring.factories）
├─ 3. 加载 ApplicationListener（事件监听）
├─ 4. 推断主配置类（包含 main 方法的类）
│
├─ 5. 运行 run() 方法（核心）
    │
    ├─ 5.1 启动 StopWatch（计时）
    ├─ 5.2 发布 ApplicationStartingEvent
    ├─ 5.3 准备 Environment（读取配置文件）
    ├─ 5.4 打印 Banner
    ├─ 5.5 创建 ApplicationContext
    │       └─ SERVLET → AnnotationConfigServletWebServerApplicationContext
    ├─ 5.6 执行 ApplicationContextInitializer
    ├─ 5.7 刷新上下文（核心！调用 AbstractApplicationContext.refresh()）
    │       └─ 这一步完成了所有 Bean 的加载、实例化
    ├─ 5.8 调用 ApplicationRunner 和 CommandLineRunner
    ├─ 5.9 发布 ApplicationReadyEvent
    └─ 5.10 返回应用上下文
```

---

### 核心源码路径：SpringApplication.run()

```java
// SpringApplication.java
public ConfigurableApplicationContext run(String... args) {
    StopWatch stopWatch = new StopWatch();
    stopWatch.start();

    // 1. 获取或创建 Bootstrap Registry
    DefaultBootstrapContext bootstrapContext = createBootstrapContext();

    // 2. 配置 Headless 模式（无显示器环境）
    configureHeadlessProperty();

    // 3. 获取 SpringApplicationRunListener（事件发布）
    SpringApplicationRunListeners listeners = getRunListeners(args);

    // 4. 发布「开始启动」事件
    listeners.starting();

    try {
        // 5. 准备 Environment（读取 application.yml、环境变量、命令行参数）
        ApplicationArguments applicationArguments = new DefaultApplicationArguments(args);
        ConfigurableEnvironment environment = prepareEnvironment(listeners, applicationArguments);
        configureIgnoreBeanInfo(environment);

        // 6. 打印 Banner
        Banner printedBanner = printBanner(environment);

        // 7. ★ 创建 ApplicationContext（核心）
        context = createApplicationContext();

        // 8. 准备上下文（设置 Environment、执行 Initializer）
        prepareContext(bootstrapContext, context, environment, listeners, applicationArguments, printedBanner);

        // 9. ★ 刷新上下文（核心中的核心，调用 AbstractApplicationContext.refresh()）
        refreshContext(context);

        // 10. 刷新后处理（空实现，留给子类扩展）
        afterRefresh(context, applicationArguments);

        stopWatch.stop();

        // 11. 发布「启动完成」事件
        listeners.started(context);

        // 12. 调用 Runner（ApplicationRunner / CommandLineRunner）
        callRunners(context, applicationArguments);
    } catch (Throwable ex) {
        handleRunFailure(context, ex, listeners);
        throw new IllegalStateException(ex);
    }

    // 13. 发布「应用就绪」事件
    listeners.running(context);
    return context;
}
```

> **第 9 步 `refreshContext()` 是最核心的**，它调用了 `AbstractApplicationContext.refresh()` —— 这是 Spring 容器的启动万能方法，所有 Bean 的加载、实例化、初始化都在这里完成。

### 自动配置原理（@EnableAutoConfiguration）

**问题**：为什么加一个 `spring-boot-starter-web` 依赖，就有了 Tomcat + Spring MVC？

**关键注解**：`@SpringBootApplication` = `@Configuration` + `@ComponentScan` + **`@EnableAutoConfiguration`**

```java
// @EnableAutoConfiguration 的核心
@Import(AutoConfigurationImportSelector.class)
public @interface EnableAutoConfiguration {}
```

`AutoConfigurationImportSelector` 做了什么？

```java
// AutoConfigurationImportSelector.java
protected List<String> getCandidateConfigurations(...) {
    // 从 META-INF/spring.factories 读取所有自动配置类
    List<String> configurations = SpringFactoriesLoader.loadFactoryNames(
        EnableAutoConfiguration.class, classLoader);
    return configurations;
}
```

**spring.factories 文件内容**（以 `spring-boot-autoconfigure` 为例）：

```properties
# META-INF/spring.factories
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
  org.springframework.boot.autoconfigure.web.servlet.WebMvcAutoConfiguration,\
  org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration,\
  org.springframework.boot.autoconfigure.data.redis.RedisAutoConfiguration,\
  ...
```

> Spring Boot 2.7 之后，改为 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 文件（更简洁）。

### 条件注解（@Conditional）

自动配置类为什么「按需加载」？因为有 **条件注解**。

```java
@Configuration
@ConditionalOnClass({ DataSource.class, EmbeddedDatabaseType.class })
// ↑ 只有 classpath 下有 DataSource 类时，这个配置类才生效
public class DataSourceAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean
    // ↑ 只有容器里没有 DataSource Bean 时，才创建这个 Bean
    public DataSource dataSource() {
        return new EmbeddedDatabaseBuilder().build();
    }
}
```

**常用条件注解**：

| 注解 | 生效条件 |
|------|----------|
| `@ConditionalOnClass` | classpath 有指定类 |
| `@ConditionalOnMissingClass` | classpath 没有指定类 |
| `@ConditionalOnBean` | 容器中有指定 Bean |
| `@ConditionalOnMissingBean` | 容器中没有指定 Bean |
| `@ConditionalOnProperty` | 配置文件中存在指定属性 |
| `@ConditionalOnWebApplication` | 是 Web 应用 |
| `@ConditionalOnNotWebApplication` | 不是 Web 应用 |

> 条件注解是 Spring Boot「**约定优于配置**」的核心实现 mechanism。

### 内置 Tomcat 启动原理

**问题**：为什么运行 `main` 方法就能启动一个 Web 服务器？

```java
// ServletWebServerApplicationContext.onRefresh()
protected void onRefresh() {
    super.onRefresh();
    // 创建 Web 服务器（Tomcat / Jetty / Undertow）
    createWebServer();
}

// 创建 Tomcat
private void createWebServer() {
    TomcatServletWebServerFactory factory = getWebServerFactory();
    // ★ 这里启动了 Tomcat！
    this.webServer = factory.getWebServer(getSelfInitializer());
    // Tomcat 在一个新线程中启动，监听端口
}
```

**流程**：

```
main() 启动
  → SpringApplication.run()
    → createApplicationContext() 创建 Web 应用上下文
      → onRefresh()
        → createWebServer()
          → new Tomcat()  // 创建 Tomcat 实例
          → tomcat.start() // 启动 Tomcat（新线程，监听 8080）
    → refresh() 完成 Bean 加载
  → 应用启动完成，Tomcat 已经在跑着了
```

## 它和相似方案的本质区别是什么？

### Spring Boot 1.x vs 2.x vs 3.x

| | 1.x | 2.x | 3.x |
|--|------|------|------|
| 底层 Spring | Spring 4.x | Spring 5.x（引入 Spring WebFlux） | Spring 6.x（Jakarta EE） |
| Java 版本 | Java 7+ | Java 8+ | Java 17+ |
| 配置发现 | `spring.factories` | `spring.factories` + `imports` 文件 | 只支持 `imports` 文件 |
| 响应式支持 | 无 | Spring WebFlux | Spring WebFlux + R2DBC |

### Spring Boot vs 传统 Spring MVC

| | 传统 Spring MVC | Spring Boot |
|--|-----------------|-------------|
| 配置方式 | XML / Java Config | 零配置（自动配置） |
| 服务器 | 外置（部署到 Tomcat） | 内置（可执行 Jar） |
| 依赖管理 | 手动引一堆 Jar | Starter 依赖（传递依赖自动管理） |
| 启动方式 | 部署到 Servlet 容器 | `main()` 方法直接启动 |

## 正确使用方式

### 1. 自定义横幅（Banner）

```java
// 1. 在 classpath 下创建 banner.txt
// 2. 或者用代码配置
SpringApplication app = new SpringApplication(Application.class);
app.setBannerMode(Banner.Mode.CONSOLE);
app.setBanner(new ResourceBanner(new ClassPathResource("banner.txt")));
```

### 2. 自定义 Runner

```java
// ApplicationRunner（参数封装成 ApplicationArguments）
@Component
public class MyRunner implements ApplicationRunner {
    @Override
    public void run(ApplicationArguments args) {
        System.out.println("应用启动完成，参数：" + args.getOptionNames());
    }
}

// CommandLineRunner（原始命令行参数）
@Component
public class MyCommandRunner implements CommandLineRunner {
    @Override
    public void run(String... args) {
        System.out.println("原始参数：" + Arrays.toString(args));
    }
}
```

### 3. 延迟初始化

```java
// application.yml
spring:
  main:
    lazy-initialization: true  # 所有 Bean 延迟初始化
```

> 延迟初始化可以加快启动速度，但会掩盖 Bean 定义错误（启动时不会报错，运行时才报错）。

## 边界情况和坑

### 1. 主配置类位置不当

```java
// ❌ 错误：主配置类放在太深的包里
package com.example.deep.package.app;
@SpringBootApplication
public class Application { ... }

// SpringBoot 默认扫描 @SpringBootApplication 所在包及其子包
// 如果 Controller 在 com.example.controller，不会被扫描到！
```

**解决**：指定扫描路径

```java
@SpringBootApplication(scanBasePackages = "com.example")
public class Application { ... }
```

---

### 2. 自动配置冲突

```java
// ❌ 自己定义了一个 DataSource，但自动配置也生效了
@Bean
public DataSource myDataSource() {
    return new HikariDataSource();
}
// 可能导致多个 DataSource Bean 冲突
```

**解决**：用 `@ConditionalOnMissingBean` 或在配置类上排除自动配置

```java
@SpringBootApplication(exclude = DataSourceAutoConfiguration.class)
public class Application { ... }
```

---

### 3. main() 方法必须放在根包

```
com.example
  ├── Application.java          ✅ 正确（根包）
  └── controller
        └── UserController.java

com.example.app
  ├── Application.java          ❌ 错误（放在子包，扫描不到同级包）
  └── controller
        └── UserController.java
```

## 我的理解

**SpringBoot 启动的本质**：

1. `SpringApplication.run()` 是入口
2. 核心两步：**创建 Context** → **刷新 Context（`refresh()`）**
3. 自动配置 = `@EnableAutoConfiguration` + `spring.factories`（或 `imports` 文件）+ 条件注解
4. 内置 Tomcat = 在 `onRefresh()` 阶段创建并启动 WebServer
