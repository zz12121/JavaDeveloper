# 千亿级数据量订单查询系统设计

> 千亿级数据查询核心：分库分表 + 读写分离 + 多级缓存 + 归档。核心矛盾是**数据量极大导致单表查询慢，需通过拆分、缓存、归档三层手段解决**。

## 场景 A：分库分表策略（ShardingSphere）

### 现象

```
单表订单数据超过 5000万行
查询响应时间从 50ms 飙升到 5000ms+
全表扫描导致 DB CPU 100%
写入也变慢，影响下单流程
```

### 解决方案

```java
// 1. ShardingSphere 分片配置（order_id 取模分16库，每库64表）
shardingRuleConfiguration:
  tables:
    t_order:
      actualDataNodes: ds_${0..15}.t_order_${0..63}
      databaseStrategy:
        standard:
          shardingColumn: order_id
          shardingAlgorithmName: db-inline
      tableStrategy:
        standard:
          shardingColumn: order_id
          shardingAlgorithmName: table-inline
  
  shardingAlgorithms:
    db-inline:
      type: INLINE
      props:
        algorithm-expression: ds_${order_id % 16}
    table-inline:
      type: INLINE
      props:
        algorithm-expression: t_order_${order_id % 64}
```

```sql
-- 2. 查询时必须带分片键（order_id 或 user_id）
-- 好查询：直接路由到单表
SELECT * FROM t_order WHERE order_id = 123456789;

-- 坏查询：全路由（广播到所有表，性能极差）
SELECT * FROM t_order WHERE create_time > '2024-01-01';
-- 解决：改用 user_id 查询，或引入订单索引表
```

**分片策略选择**：
- **订单查询场景**：80% 查询用 `user_id`，20% 用 `order_id`
- **方案**：双写索引表，`t_order` 按 `order_id` 分片，`t_order_user_index` 按 `user_id` 分片存储 `order_id` 映射
- **user_id 查订单流程**：先查索引表获取 order_id 列表，再查订单主表

```java
// 3. 索引表查询实现
public List<Order> queryByUserId(Long userId) {
    // 1. 查索引表（按 user_id 分片，只查1张表）
    List<Long> orderIds = orderIndexMapper.selectOrderIdsByUserId(userId);
    
    if (orderIds.isEmpty()) {
        return Collections.emptyList();
    }
    
    // 2. 批量查订单主表（可能横跨多个分片）
    return orderMapper.selectByOrderIds(orderIds);
}
```

## 场景 B：读写分离与延迟

### 现象

```
写操作变慢（主库压力大）
读操作读到旧数据（主从延迟 1-5 秒）
用户下单后立刻查询，查不到刚下的单
金融场景要求强一致，不能容忍延迟
```

### 解决方案

```
读写分离策略：
1. 写操作 → 强制走主库
2. 读操作 → 默认走从库，特殊场景走主库
3. 主从延迟监控 → 超过阈值告警
4. 延迟敏感查询 → 强制走主库（如订单支付后查询结果）
```

```java
// 1. ShardingSphere 读写分离配置
readwriteSplittingRuleConfiguration:
  dataSources:
    pr_ds:
      type: Static
      props:
        writeDataSourceName: ds_master
        readDataSourceNames:
          - ds_slave0
          - ds_slave1
        loadBalancerName: round_robin
  
  loadBalancers:
    round_robin:
      type: ROUND_ROBIN

// 2. 强制走主库注解（用于延迟敏感场景）
@Target({ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
public @interface MasterOnly {
}

// 3. AOP 实现强制主库路由
@Aspect
@Component
public class MasterRouteAspect {
    
    @Around("@annotation(masterOnly)")
    public Object routeToMaster(ProceedingJoinPoint pjp, MasterOnly masterOnly) throws Throwable {
        // 设置 Hint，强制走主库
        HintManager hintManager = HintManager.getInstance();
        hintManager.setWriteRouteOnly();
        try {
            return pjp.proceed();
        } finally {
            hintManager.close();
        }
    }
}

// 4. 使用：支付后查询订单，强制走主库
@MasterOnly
public Order queryOrderAfterPayment(Long orderId) {
    return orderMapper.selectById(orderId);
}
```

**主从延迟处理**：
- 写入后 5秒内查询 → 强制走主库（通过本地缓存记录刚写入的 order_id）
- 超过 5秒 → 走从库（延迟已消除）

```java
// 5. 本地记录刚写入的订单，短时间内查主库
@Component
public class RecentWriteCache {
    // 记录最近5秒写入的订单ID
    private final Cache<Long, Boolean> recentWrites = Caffeine.newBuilder()
            .expireAfterWrite(5, TimeUnit.SECONDS)
            .build();
    
    public void recordWrite(Long orderId) {
        recentWrites.put(orderId, true);
    }
    
    public boolean isRecentWrite(Long orderId) {
        return recentWrites.getIfPresent(orderId) != null;
    }
}
```

## 场景 C：多级缓存架构

### 现象

```
直接查 DB，QPS 超过 5000 就扛不住
热点用户（大客户）的订单查询频繁
相同查询条件重复查询，浪费 DB 资源
缓存穿透导致 DB 压力骤增
```

### 解决方案

```
多级缓存架构：
L1：本地缓存（Caffeine）- 缓存 1分钟，容量 10000
L2：Redis 缓存 - 缓存 10分钟，TB 级容量
L3：DB 查询 - 分库分表查询
```

```java
// 1. 多级缓存实现
@Service
public class OrderQueryService {
    
    // L1：本地缓存（JVM 级别，最快）
    private final LoadingCache<String, List<Order>> localCache = 
        Caffeine.newBuilder()
                .maximumSize(10_000)
                .expireAfterWrite(1, TimeUnit.MINUTES)
                .build(this::loadFromRedis);
    
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    
    @Autowired
    private OrderMapper orderMapper;
    
    public List<Order> queryOrders(Long userId, Date startTime, Date endTime) {
        String cacheKey = buildCacheKey(userId, startTime, endTime);
        
        // L1：查本地缓存
        List<Order> result = localCache.get(cacheKey);
        if (result != null) {
            return result;
        }
        
        // L2：查 Redis（不会走到这里，因为本地缓存加载器已包含 Redis）
        return localCache.get(cacheKey);
    }
    
    private List<Order> loadFromRedis(String cacheKey) {
        // L2：查 Redis
        List<Order> cached = (List<Order>) redisTemplate.opsForValue().get(cacheKey);
        if (cached != null) {
            return cached;
        }
        
        // L3：查 DB
        List<Order> orders = queryFromDb(cacheKey);
        
        // 回写 Redis（过期时间随机化，防止缓存雪崩）
        int expire = 600 + new Random().nextInt(120); // 10-12分钟
        redisTemplate.opsForValue().set(cacheKey, orders, expire, TimeUnit.SECONDS);
        
        return orders;
    }
    
    private List<Order> queryFromDb(String cacheKey) {
        // 解析 cacheKey，执行 DB 查询
        // 实际生产中这里需要解析 cacheKey 获取查询条件
        return orderMapper.selectByExample(...);
    }
    
    private String buildCacheKey(Long userId, Date start, Date end) {
        return String.format("order:query:user:%d:start:%d:end:%d", 
                            userId, start.getTime(), end.getTime());
    }
}
```

**缓存问题处理**：
- **缓存穿透**：空结果也缓存（TTL 短一点，如 1分钟）
- **缓存击穿**：用 `synchronized` 或 Redis 分布式锁，只让一个线程查 DB
- **缓存雪崩**：过期时间加随机值

```java
// 2. 防止缓存击穿（分布式锁）
private List<Order> loadFromRedisWithLock(String cacheKey) {
    List<Order> cached = (List<Order>) redisTemplate.opsForValue().get(cacheKey);
    if (cached != null) {
        return cached;
    }
    
    // 获取分布式锁，避免缓存击穿
    String lockKey = cacheKey + ":lock";
    String lockValue = UUID.randomUUID().toString();
    try {
        // 尝试加锁（SET NX EX）
        Boolean locked = redisTemplate.opsForValue()
                .setIfAbsent(lockKey, lockValue, 10, TimeUnit.SECONDS);
        
        if (locked != null && locked) {
            // 获取锁成功，查 DB
            List<Order> orders = queryFromDb(cacheKey);
            
            // 回写缓存
            redisTemplate.opsForValue()
                    .set(cacheKey, orders, 600 + new Random().nextInt(120), TimeUnit.SECONDS);
            return orders;
        } else {
            // 获取锁失败，等待重试
            Thread.sleep(50);
            return loadFromRedisWithLock(cacheKey);
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
        return Collections.emptyList();
    } finally {
        // 释放锁（只能释放自己加的锁）
        if (lockValue.equals(redisTemplate.opsForValue().get(lockKey))) {
            redisTemplate.delete(lockKey);
        }
    }
}
```

## 场景 D：归档与冷热分离

### 现象

```
订单表数据量持续增长，查询越来越慢
90% 的查询集中在最近 3 个月
历史订单（1年前）几乎无人查询，但占用大量存储空间
备份和恢复时间越来越长
```

### 解决方案

```
冷热分离策略：
热数据（近3月）：高性能 SSD，分库分表，多级缓存
温数据（3月-1年）：普通磁盘，归档库，查询走离线通道
冷数据（1年+）：对象存储（如 S3/OSS），压缩存储，按需恢复
```

```sql
-- 1. 归档任务：将1年前订单迁移到归档表
-- 归档表结构与主表相同，但存储在低成本存储上
INSERT INTO t_order_archive_2023
SELECT * FROM t_order_0000  -- 遍历所有分表
WHERE create_time < DATE_SUB(NOW(), INTERVAL 1 YEAR)
  AND create_time >= DATE_SUB(NOW(), INTERVAL 2 YEAR);

-- 2. 验证归档数据完整性
SELECT 
    COUNT(*) AS source_count,
    (SELECT COUNT(*) FROM t_order_archive_2023 WHERE ...) AS archive_count
FROM t_order_0000
WHERE create_time < DATE_SUB(NOW(), INTERVAL 1 YEAR);

-- 3. 确认无误后，删除已归档数据（分批删除，避免长事务）
DELETE FROM t_order_0000
WHERE create_time < DATE_SUB(NOW(), INTERVAL 1 YEAR)
LIMIT 1000;  -- 分批删除，每次1000条
```

```java
// 2. 归档定时任务实现
@Component
public class OrderArchiveScheduler {
    
    @Scheduled(cron = "0 0 2 * * ?")  // 每天凌晨2点执行
    public void archiveOldOrders() {
        LocalDate oneYearAgo = LocalDate.now().minusYears(1);
        LocalDate twoYearsAgo = LocalDate.now().minusYears(2);
        
        // 遍历所有分表（16库 × 64表 = 1024张表）
        for (int dbIndex = 0; dbIndex < 16; dbIndex++) {
            for (int tableIndex = 0; tableIndex < 64; tableIndex++) {
                archiveTable(dbIndex, tableIndex, oneYearAgo, twoYearsAgo);
            }
        }
    }
    
    private void archiveTable(int dbIndex, int tableIndex, LocalDate start, LocalDate end) {
        String tableName = String.format("t_order_%02d_%02d", dbIndex, tableIndex);
        
        // 1. 查询需要归档的数据
        List<Order> ordersToArchive = orderMapper.selectOldOrders(
                tableName, start, end, 1000);  // 每次处理1000条
        
        if (ordersToArchive.isEmpty()) {
            return;
        }
        
        // 2. 写入归档表
        orderArchiveMapper.batchInsert(ordersToArchive);
        
        // 3. 验证归档数据（抽样校验）
        for (Order order : ordersToArchive) {
            Order archived = orderArchiveMapper.selectById(order.getOrderId());
            if (archived == null) {
                log.error("归档数据丢失: orderId={}", order.getOrderId());
            }
        }
        
        // 4. 删除已归档数据（分批）
        for (Order order : ordersToArchive) {
            orderMapper.deleteById(tableName, order.getOrderId());
        }
        
        log.info("归档完成: table={}, count={}", tableName, ordersToArchive.size());
    }
}
```

**归档策略优化**：
- 归档任务分布式调度（用 ElasticJob 或 XXL-Job）
- 归档失败重试机制（记录失败订单，定时重试）
- 归档数据统计（记录归档率、归档速度等指标）

## 核心参数估算

```
数据量估算：
  假设：日均订单 1亿，1年 = 365亿，3年 = 1000亿+
  单表行数控制在 500万以内 → 需要 2000+ 张表
  采用 16库 × 64表 = 1024张表（单表约 1亿 / 1024 ≈ 100万行，合理）

查询 QPS 估算：
  日均 1亿订单，每订单平均查询 2次 → 日均 2亿次查询
  平均 QPS = 2亿 / 86400 ≈ 2300
  峰值 QPS = 平均 × 10 ≈ 23000

缓存容量估算：
  L1 本地缓存：10000 条 × 2KB = 20MB（单机）
  L2 Redis 缓存：热点数据 10% × 1000亿条 × 2KB ≈ 20TB（需集群）
  实际：Redis 只缓存近期数据（近3月），约 5TB
```

## 追问链

### 追问 1：分库分表后，跨分片查询怎么优化？

> "跨分片查询是分库分表的痛点。三种方案：1. **避免跨分片**：业务设计上让查询都带分片键（如查询订单必须带 user_id 或 order_id）。2. **冗余索引表**：对高频查询条件建索引表（如 t_order_user_index）。3. **ES 宽表**：将订单数据同步到 Elasticsearch，复杂查询走 ES。4. **归并结果**：ShardingSphere 会自动归并各分片结果，但性能差，只适合后台查询。"

### 追问 2：主从延迟怎么监控和处理？

> "监控：1. 通过 `SHOW SLAVE STATUS` 获取 `Seconds_Behind_Master`。2. 写入时间戳到主库，从库查询对比。处理：1. 延迟 < 1秒：可接受。2. 延迟 1-5秒：重要查询强制走主库（用 HintManager）。3. 延迟 > 5秒：告警，人工介入，可能从库宕机需要重建。"

### 追问 3：千亿级数据，Redis 缓存容量不够怎么办？

> "三层面优化：1. **只缓存热点数据**：用 LFU 淘汰策略，保留高频访问数据。2. **多级缓存**：L1 本地缓存扛最热数据，减少 Redis 压力。3. **缓存时效性**：缩短过期时间，减少无效缓存。4. **数据压缩**：用 Kryo 或 Protobuf 序列化，减小 value 大小。5. **拆分缓存**：订单基础信息（小）和详情（大）分开缓存。"

### 追问 4：归档数据怎么查询？

> "归档数据查询走**离线通道**：1. **历史查询入口**：在订单查询页面区分'近3个月'和'历史订单'两个 tab。2. **历史查询服务**：专门查询归档库，响应时间可以慢一点（如 5秒内）。3. **导出服务**：用户需要历史数据时用导出功能（生成 Excel/CSV 文件，异步下载）。4. **冷数据恢复**：1年前数据存在对象存储，需要时可恢复到归档库。"

## 排查 Checklist

```
□ 分片键选择合理吗？ → 查询必须带分片键，避免全路由
□ 主从延迟监控了吗？ → Seconds_Behind_Master < 1秒
□ 读写分离配置正确吗？ → 写主库，读从库（特殊场景写 Hint）
□ 缓存命中率多少？ → Redis 命中率 > 80%，本地缓存 > 50%
□ 缓存过期时间合理吗？ → 本地1分钟，Redis 10分钟（加随机值）
□ 归档任务执行了吗？ → 检查归档日志，数据完整性校验
□ 归档后原表数据删除了吗？ → 分批删除，避免长事务
□ 热点数据有缓存吗？ → 大客户订单加本地缓存
□ 跨分片查询有优化吗？ → 加索引表或走 ES
□ 冷数据查询有离线通道吗？ → 历史查询服务 / 导出功能
```
