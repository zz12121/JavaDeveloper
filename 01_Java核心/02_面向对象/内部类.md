# 内部类

> 内部类不是"写在一个类里面的类"那么简单。它是 Java 解决"一个类需要另一个类的紧密配合"的方案，也是理解 Lambda、匿名类、内存泄漏的关键。

---

## 这个问题为什么存在？

有些场景下，一个类型**只在另一个类型的内部才有意义**，离开外部类它就没有独立存在的价值：

```
场景举例：
- Map.Entry：Entry 只在 Map 内部使用，外部不需要直接创建 Entry
- Iterator：迭代器必须持有集合的引用才能遍历，它是集合的一部分
- Comparator：排序策略只在排序时使用，通常写成匿名类或 Lambda
```

如果把这些类型独立成顶级类，会导致：
1. **命名空间污染**：大量只在一个地方使用的类散落在包里
2. **封装被破坏**：内部类需要访问外部类的 private 成员，顶级类做不到
3. **代码不内聚**：紧密关联的逻辑被分散到不同文件

内部类的本质价值：**让紧密耦合的两个类型共享封装边界。**

---

## 它是怎么解决问题的？

### 四类内部类全景

| 类型 | 声明位置 | 编译后文件 | 持有外部类引用 | 访问外部类 |
|------|----------|-----------|---------------|-----------|
| **静态内部类** | 类体内 `static class` | `Outer$Inner.class` | ❌ 不持有 | 仅 static 成员 |
| **成员内部类** | 类体内 `class` | `Outer$Inner.class` | ✅ 隐式持有 | 全部成员（含 private） |
| **局部内部类** | 方法体内 `class` | `Outer$1Inner.class` | ✅ 隐式持有 | 外部全部 + final 局部变量 |
| **匿名内部类** | 表达式 `new X() {}` | `Outer$1.class` | ✅ 隐式持有 | 同局部内部类 |

### 静态内部类 vs 成员内部类（核心区别）

```java
class Outer {
    private int instanceVar = 10;     // 实例变量
    private static int staticVar = 20; // 静态变量

    // 静态内部类：不依赖外部类实例
    static class StaticInner {
        void test() {
            // System.out.println(instanceVar);  // ❌ 编译错误！不能访问实例变量
            System.out.println(staticVar);        // ✅ 可以访问静态成员
        }
    }

    // 成员内部类：依赖外部类实例
    class MemberInner {
        void test() {
            System.out.println(instanceVar);  // ✅ 可以访问实例变量
            System.out.println(staticVar);    // ✅ 可以访问静态成员
        }
    }
}
```

**使用方式的区别**：
```java
// 静态内部类：直接 new，不需要外部类实例
Outer.StaticInner si = new Outer.StaticInner();

// 成员内部类：必须先有外部类实例
Outer outer = new Outer();
Outer.MemberInner mi = outer.new MemberInner();
```

**为什么静态内部类不持有外部类引用？**

因为它在语义上和外部类实例无关——就像 ArrayList 里的 Iterator，迭代器需要引用 ArrayList，但 `ArrayList` 里的 `SubList` 视图并不需要。静态内部类的价值在于**逻辑归属**（它属于外部类的命名空间），而不是**实例关联**。

### 编译器做了什么？——this$0 机制

成员内部类在编译后，编译器自动生成一个指向外部类的字段：

```java
// 源码
class Outer {
    private int x = 10;
    class Inner {
        void print() { System.out.println(x); }
    }
}

// 编译后 Inner.class 等价于：
class Inner {
    final Outer this$0;          // ← 编译器自动生成的外部类引用

    Inner(Outer this$0) {        // ← 构造器被改写，接收外部类实例
        this.this$0 = this$0;
    }

    void print() {
        // 编译器把 x 替换为 this$0.x（通过合成访问器绕过 private）
        System.out.println(this$0.access$000());
    }
}
```

**关键细节**：
1. `this$0` 是 `final` 的——内部类一旦创建，其外部类引用就固定了
2. 访问外部类 `private` 成员时，编译器会在外部类生成**合成访问器方法**（`access$000`），内部类调用这个方法
3. 每个成员内部类对象都比外部类对象**多一个引用的开销**（4/8 字节）

### 局部内部类与匿名内部类

```java
void method() {
    final int localVar = 10;  // 必须是 final 或 effectively final

    // 局部内部类
    class LocalInner {
        void test() {
            System.out.println(localVar);  // ✅ 访问外部局部变量
        }
    }

    // 匿名内部类（局部内部类的语法糖）
    Runnable r = new Runnable() {
        @Override
        public void run() {
            System.out.println(localVar);  // ✅ 同样可以访问
        }
    };
}
```

**为什么局部变量必须是 final / effectively final？**

```
本质原因：变量生命周期不匹配

局部变量在栈上 → 方法返回后就被回收
内部类对象在堆上 → 可能活得比方法调用更长

编译器的解决方案：把局部变量「拷贝」一份到内部类对象里
→ 如果允许修改原变量，拷贝的值就对不上了
→ 所以 Java 强制要求：要么 final，要么 effectively final（不修改就等同于 final）
```

JDK 8 放宽了这个限制：只要变量在捕获后**没有被修改**，就不需要显式写 `final`（effective final）。但这只是语法层面的便利，底层机制没变——仍然是值拷贝。

## 深入原理

### Lambda vs 匿名内部类（字节码层面的根本区别）

```java
// 匿名内部类
Runnable r1 = new Runnable() {
    @Override
    public void run() { System.out.println("hi"); }
};
// 编译后：生成 Outer$1.class 文件
// 字节码：invokevirtual Outer$1.run

// Lambda
Runnable r2 = () -> System.out.println("hi");
// 编译后：不生成额外 .class 文件！
// 字节码：invokedynamic LambdaMetafactory.metafactory
```

| 维度 | 匿名内部类 | Lambda |
|------|-----------|--------|
| 编译产物 | 每个匿名类一个 `.class` 文件 | 无额外 `.class`，运行时动态生成 |
| 字节码指令 | `new` + `invokevirtual` | `invokedynamic` |
| 外部类引用 | 总是持有（`this$0`） | 只捕获实际使用的变量 |
| this 指向 | 指向匿名类实例 | 指向 enclosing 类实例 |
| 可序列化 | ✅ 如果实现的接口可序列化 | ❌ Lambda 默认不可序列化 |

Lambda 不持有外部类引用这个特性很重要——它是解决内部类内存泄漏问题的一种方式。详见 [[01_Java核心/08_Lambda与Stream/Lambda与函数式|Lambda与函数式]] 中 invokedynamic 的完整原理。

### 内部类与内存泄漏

```
泄漏场景（经典 Android 案例）：

Activity {
    void onCreate() {
        // 匿名内部类 Handler 持有 Activity 引用（this$0）
        handler.postDelayed(new Runnable() {
            void run() {
                textView.setText("done");  // 隐式持有 Activity
            }
        }, 60000);  // 1分钟后执行
    }
    void onDestroy() {
        // Activity 销毁了，但 Runnable 还在 MessageQueue 里
        // Runnable → this$0 → Activity → 所有 View
        // Activity 无法被 GC！→ 内存泄漏
    }
}
```

**解决方案**：
```java
// 1. 使用静态内部类（不持有外部类引用）
static class SafeHandler extends Handler {
    private WeakReference<Activity> activityRef;
    SafeHandler(Activity activity) {
        activityRef = new WeakReference<>(activity);
    }
}

// 2. 用 Lambda 替代匿名内部类（不生成 this$0）
handler.postDelayed(() -> {
    // 只捕获需要的变量，不持有整个 Activity
    textView.setText("done");
}, 60000);

// 3. 及时取消
@Override void onDestroy() {
    handler.removeCallbacksAndMessages(null);
}
```

### 内部类能访问外部类 private 的原理

```java
class Outer {
    private int secret = 42;
    class Inner {
        void peek() { System.out.println(secret); }
    }
}
```

编译后在 `Outer.class` 中会生成一个**合成方法**：
```java
// 编译器在 Outer 中生成：
static int access$000(Outer outer) {
    return outer.secret;  // 包级私有，同包的 Inner 可以调用
}
```

这是 Java 语言层面的"friend"机制——编译器通过生成包级私有的合成方法，让内部类绕过 private 访问限制。[[01_Java核心/07_注解与反射/反射|反射]] 可以看到这些方法（`getDeclaredMethods()` 返回带 `synthetic` 标志的方法）。

## 正确使用方式

### 1. 优先用静态内部类

```java
// ✓ 好：不需要访问外部类实例时，用 static 内部类
class Outer {
    private int value;

    static class Helper {
        void help() {
            // 不需要访问 Outer 的实例成员
            System.out.println("help");
        }
    }
}
// 好处：不持有外部类引用，不会有内存泄漏风险
```

### 2. 需要访问外部类实例时才用成员内部类

```java
// 迭代器是典型场景：必须访问集合的内部数据
class MyList<E> {
    private E[] data;

    class Iter implements Iterator<E> {
        int index = 0;
        public boolean hasNext() { return index < data.length; }
        public E next() { return data[index++]; }
    }

    public Iterator<E> iterator() { return new Iter(); }
}
```

### 3. 用 Lambda 替代匿名内部类

```java
// 匿名内部类：每次都会生成一个 .class 文件
Collections.sort(list, new Comparator<String>() {
    @Override
    public int compare(String a, String b) {
        return a.length() - b.length();
    }
});

// Lambda：不生成额外 .class，不持有不必要的外部引用
Collections.sort(list, (a, b) -> a.length() - b.length());

// 方法引用：更简洁
Collections.sort(list, Comparator.comparingInt(String::length));
```

详见 [[01_Java核心/08_Lambda与Stream/Lambda与函数式|Lambda与函数式]] 了解更多 Lambda 用法。

## 边界情况和坑

### 1. 匿名内部类的 this 指向

```java
class Outer {
    String name = "outer";

    void test() {
        new Runnable() {
            String name = "inner";
            public void run() {
                System.out.println(name);       // "inner"（局部变量遮蔽）
                System.out.println(this.name);    // ❌ 编译错误，Runnable 没有 name 字段
                System.out.println(Outer.this.name);  // ✅ "outer"
            }
        }.run();
    }
}
```

**匿名内部类的 `this` 指向匿名类自己的实例**，不是外部类。要访问外部类的 `this`，用 `Outer.this`。Lambda 没有这个问题——Lambda 中的 `this` 指向 enclosing 类实例。

### 2. 静态内部类不能访问外部类的实例成员

```java
class Outer {
    private int x = 10;
    static class Inner {
        void test() {
            System.out.println(x);  // ❌ 编译错误
        }
    }
}
```

如果静态内部类需要访问外部类的实例成员，必须显式传入外部类引用：

```java
static class Inner {
    private final Outer outer;
    Inner(Outer outer) { this.outer = outer; }
    void test() { System.out.println(outer.x); }
}
```

### 3. 内部类 File 命名的坑

```java
class Outer {
    class A {}
    class B {}
}
// 编译后：Outer$A.class, Outer$B.class

class Test {
    // 两个匿名内部类
    Runnable r1 = new Runnable() { public void run() {} };
    Runnable r2 = new Runnable() { public void run() {} };
}
// 编译后：Test$1.class, Test$2.class
// 数字编号按出现顺序，添加/删除匿名类可能导致编号变化
// → 不要依赖匿名类的 .class 文件名做逻辑判断
```

### 4. 局部内部类不能声明静态成员（Java 16+ 除外）

```java
void method() {
    class Local {
        static int x = 10;  // ❌ Java 15 及之前：编译错误
        // Java 16+ 允许局部内部类有 static 成员
    }
}
```
