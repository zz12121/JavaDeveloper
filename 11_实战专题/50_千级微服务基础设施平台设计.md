## 场景 A：K8s容器调度

### 现象

```
千级服务部署慢，调度延迟高
资源利用率低，节点空闲率高
Pod频繁驱逐，服务不稳定
```

### 根因

```
未做资源配额管理，大服务占用所有资源
调度器未优化，未及时调度Pending Pod
未用亲和性/反亲和性，Pod分布不均
```

### 解决方案

```yaml
# K8s资源配额（限制命名空间资源使用）
apiVersion: v1
kind: ResourceQuota
metadata:
  name: service-quota
  namespace: prod
spec:
  hard:
    pods: "100"          # 最多100个Pod
    requests.cpu: "200"  # 总CPU请求不超过200核
    requests.memory: "400Gi"
    limits.cpu: "400"
    limits.memory: "800Gi"
```

```yaml
# Pod反亲和性配置（保证高可用）
apiVersion: apps/v1
kind: Deployment
metadata:
  name: user-service
spec:
  replicas: 3
  template:
    spec:
      affinity:
        podAntiAffinity:  # Pod分布在不同节点
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchLabels:
                app: user-service
            topologyKey: kubernetes.io/hostname
```

**调度关键**：
- 每个服务设置合理的requests/limits，避免资源争抢
- 用亲和性/反亲和性保证Pod高可用（跨节点部署）
- 启用K8s调度器扩展，优化千级服务调度速度

## 场景 B：Istio服务网格

### 现象

```
服务间调用无法追踪，故障定位难
无统一流量管理，灰度发布难实现
服务间认证复杂，每个服务都要实现
```

### 根因

```
未用服务网格，流量管理分散在各服务
无统一可观测性，调用链断裂
服务间认证需每个服务单独实现
```

### 解决方案

```yaml
# Istio VirtualService（流量管理）
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: user-service-vs
spec:
  hosts:
    - user-service
  http:
  - route:
    - destination:
        host: user-service
        subset: v1
      weight: 90
    - destination:
        host: user-service
        subset: v2
      weight: 10  # 10%流量到新版本
```

**服务网格关键**：
- 统一流量管理：灰度、熔断、限流在网格层实现
- 自动注入Sidecar，服务无感知接入
- 统一可观测性：调用链、指标、日志自动采集

## 场景 C：可观测性Prometheus+Grafana+SkyWalking

### 现象

```
服务故障后无法快速定位根因
无统一监控指标，各服务上报格式不一致
调用链断裂，无法追踪完整请求路径
```

### 根因

```
未统一可观测性标准，各服务自定埋点
无集中式监控平台，指标分散
未集成调用链追踪，跨服务调用不可见
```

### 解决方案

```yaml
# Prometheus采集配置
scrape_configs:
  - job_name: 'kubernetes-pods'
    kubernetes_sd_configs:
      - role: pod
    relabel_configs:
      - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_scrape]
        action: keep
        regex: true
```

```xml
<!-- SkyWalking Java Agent配置 -->
<plugin>
  <groupId>org.apache.skywalking</groupId>
  <artifactId>apm-toolkit-logback-1.x</artifactId>
  <version>8.16.0</version>
</plugin>
```

**可观测性关键**：
- 指标（Prometheus）：服务QPS、延迟、错误率
- 日志（ELK/Loki）：集中式日志查询
- 调用链（SkyWalking）：端到端请求追踪

## 场景 D：自助化平台

### 现象

```
服务部署需要运维介入，耗时数小时
配置变更需要提交工单，流程繁琐
开发者无法自助查看服务状态和日志
```

### 根因

```
无自助化平台，所有操作依赖运维
未做权限控制，开发者无法操作自己服务
未集成CI/CD，部署流程不透明
```

### 解决方案

```
自助化平台核心功能：
1. 服务管理：创建/部署/扩缩容/下线自助操作
2. 配置管理：Apollo配置中心集成，在线修改配置
3. 监控查看：集成Grafana，查看服务指标/调用链
4. 日志查询：集成Loki，在线查询服务日志
5. 权限控制：基于RBAC，开发者只能操作自己负责的服务
```

**自助化关键**：
- 集成K8s API，开发者自助部署服务
- 流程自动化：部署→健康检查→流量接入全自动化
- 审计日志：所有操作记录，可追溯

## 排查 Checklist

```
□ 有资源配额限制吗？ → 每个命名空间设置ResourceQuota
□ 用服务网格了吗？ → Istio统一流量管理/可观测性
□ 可观测性覆盖全吗？ → 指标/日志/调用链全覆盖
□ 支持自助部署吗？ → 开发者自助创建/部署服务
□ Pod分布均匀吗？ → 反亲和性保证跨节点部署
□ 服务有熔断限流吗？ → Istio流量策略配置
□ 监控告警及时吗？ → Prometheus AlertManager配置
□ 权限控制合理吗？ → RBAC基于角色的权限控制
```
