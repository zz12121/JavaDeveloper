# 任务调度系统设计

> 任务调度系统核心是：**调度策略 + 执行隔离 + 失败重试 + 分片扩展**。需要支持延迟任务、优先级任务、Cron 任务等场景，是分布式系统的基础中间件。

---

## 架构设计

```
┌─────────────┐   提交   ┌─────────────┐
│  业务应用  │ ───────→ │  调度中心   │
└─────────────┘         │ (Scheduler)  │
                             └──────┬──────┘
                                    │ 分片下发
                    ┌───────────────┼───────────────┐
                    ▼               ▼               ▼
             ┌────────┐      ┌────────┐      ┌────────┐
             │ 执行器 │      │ 执行器 │      │ 执行器 │
             │ (Worker)│      │ (Worker)│      │ (Worker)│
             └────────┘      └────────┘      └────────┘
```

## 场景 A：延迟任务实现（5 秒后执行）

### 方案对比
| 方案 | 优点 | 缺点 | 适用 |
|------|------|------|------|
| DelayQueue | 无锁、JDK 内置 | 单机 | 轻量 |
| Redis ZSet | 分布式、简单 | 需要轮询 | 中小规模 |
| 时间轮（Netty HashedWheelTimer）| O(1) 插入、O(1) 触发 | 内存占用 | 百万级任务 |
| MQ 延迟消息 | 可靠、解耦 | 依赖 MQ | 高可靠 |

### Redis ZSet 实现延迟任务
```java
@Service
public class DelayTaskService {
    private static final String KEY = "delay:tasks";
    
    @Autowired
    private RedisTemplate<String, String> redis;
    
    // 提交延迟任务
    public void submitTask(String taskId, long delaySeconds) {
        double executeTime = System.currentTimeMillis() + delaySeconds * 1000;
        redis.opsForZSet().add(KEY, taskId, executeTime);
    }
    
    // 轮询执行（定时任务，每秒扫一次）
    @Scheduled(fixedDelay = 1000)
    public void pollTasks() {
        double now = System.currentTimeMillis();
        Set<String> tasks = redis.opsForZSet()
            .rangeByScore(KEY, 0, now);
        
        for (String taskId : tasks) {
            // 用 ZREM 原子删除（防止多机重复执行）
            Long removed = redis.opsForZSet().remove(KEY, taskId);
            if (removed > 0) {  // 只有删除成功的才执行
                executeTask(taskId);
            }
        }
    }
}
```

## 场景 B：分片调度（支持水平扩展）

### 问题
```
单机调度：任务多 → 调度线程池满 → 任务堆积
解决：分片（Sharding），每个执行器只执行部分任务
```

### 分片算法
```java
// 任务分片：按任务 ID hash 分片
public class ShardingScheduler {
    private final int shardTotal;  // 总分片数（= 执行器数量）
    private final int shardIndex;   // 当前执行器分片索引
    
    public void schedule() {
        List<Task> allTasks = taskMapper.selectPendingTasks();
        for (Task task : allTasks) {
            // 分片：taskId % shardTotal == shardIndex 才执行
            if (task.getId() % shardTotal == shardIndex) {
                executeAsync(task);
            }
        }
    }
    
    // 动态扩缩容：执行器上线/下线时，重新分配分片
    @ZookeeperWatch(path = "/scheduler/workers")
    public void onWorkersChange(List<String> workers) {
        this.shardTotal = workers.size();
        this.shardIndex = workers.indexOf(myId);
        // 重新分片，可能触发任务迁移
    }
}
```

### 生产方案：XXL-Job / Elastic-Job
- **XXL-Job**：轻量，中心化调度，执行器通过心跳注册
- **Elastic-Job**：去中心化，基于 ZooKeeper 选主，分片更均匀

## 场景 C：失败重试（指数退避）

```java
// 重试策略：指数退避 + 最大重试次数
public class RetryPolicy {
    private final int maxRetries = 5;
    private final long baseDelayMs = 1000;  // 1秒
    
    public long getRetryDelay(int retryCount) {
        if (retryCount >= maxRetries) return -1;  // 放弃
        // 指数退避：1000, 2000, 4000, 8000, 16000
        return baseDelayMs * (1L << retryCount);
    }
}

// 任务执行框架
public class TaskExecutor {
    private final RetryPolicy retryPolicy = new RetryPolicy();
    
    public void executeWithRetry(Task task) {
        int attempt = 0;
        while (attempt <= retryPolicy.maxRetries) {
            try {
                doExecute(task);
                task.markSuccess();
                return;
            } catch (Exception e) {
                attempt++;
                long delay = retryPolicy.getRetryDelay(attempt);
                if (delay < 0) {
                    task.markFailed("重试次数耗尽");
                    return;
                }
                // 延迟重试（提交到延迟队列）
                delayQueue.offer(new DelayedTask(task, delay));
            }
        }
    }
}
```
