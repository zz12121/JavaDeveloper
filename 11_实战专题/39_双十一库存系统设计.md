# 双十一库存系统设计

> 双十一核心：Redis预减 + 分桶抗热Key + 限流削峰 + 降级方案。核心矛盾是**瞬时流量极大（百万QPS），DB扛不住，必须用多级防护保证不超卖、少卖**。

## 场景 A：Redis预减与DB兜底

### 现象

```
双十一零点，库存 Redis 瞬间被打爆
直接查 DB 扣库存，DB 连接池耗尽
Redis 扣减成功，DB 挂了 → 库存不一致
必须 Redis 和 DB 双重保障
```

### 解决方案

```java
// 1. Redis 预减库存（第一道防线）
@Service
public class StockService {
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    @Autowired
    private StockMapper stockMapper;
    
    @Autowired
    private RocketMQTemplate mqTemplate;
    
    // 扣减库存（Redis 预减 + 异步落库）
    public Result deductStock(Long skuId, Integer count, Long userId) {
        String stockKey = "stock:sku:" + skuId;
        
        // 1. 本地布隆过滤器：快速过滤已售罄商品（减少 Redis 请求）
        if (BloomFilter.isSoldOut(skuId)) {
            return Result.error("商品已售罄");
        }
        
        // 2. Redis 原子预减
        Long remain = redisTemplate.opsForValue().decrement(stockKey, count);
        
        if (remain < 0) {
            // 库存不足，回补
            redisTemplate.opsForValue().increment(stockKey, count);
            BloomFilter.markSoldOut(skuId);  // 标记售罄
            return Result.error("库存不足");
        }
        
        // 3. 发送 MQ 消息，异步落库（不影响用户响应）
        StockDeductMsg msg = new StockDeductMsg(skuId, count, userId, System.currentTimeMillis());
        mqTemplate.send("topic_stock_deduct", msg);
        
        return Result.success("抢购成功");
    }
}

// 2. MQ 消费者：异步落库（第二道防线）
@Component
public class StockDeductConsumer {
    
    @Autowired
    private StockMapper stockMapper;
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    @RocketMQMessageListener(
            topic = "topic_stock_deduct",
            consumerGroup = "stock_deduct_group",
            consumeMode = ConsumeMode.ORDERLY  // 顺序消费，同一 sku 顺序处理
    )
    public void onMessage(StockDeductMsg msg) {
        try {
            // DB 层扣减（兜底）
            int rows = stockMapper.deductStock(msg.getSkuId(), msg.getCount());
            
            if (rows == 0) {
                // DB 扣减失败（库存不足），需要回滚 Redis
                String stockKey = "stock:sku:" + msg.getSkuId();
                redisTemplate.opsForValue().increment(stockKey, msg.getCount());
                log.warn("DB扣减失败，回滚Redis: skuId={}", msg.getSkuId());
            }
        } catch (Exception e) {
            log.error("处理库存扣减失败: {}", msg, e);
            throw new RuntimeException(e);  // 重试
        }
    }
}
```

```sql
-- 3. DB 层兜底扣减（防超卖）
UPDATE sku_stock
SET stock = stock - #{count},
    sold = sold + #{count},
    update_time = NOW()
WHERE sku_id = #{skuId}
  AND stock >= #{count};  -- 必须带条件，防超卖
```

**Redis 与 DB 一致性**：
- Redis 预减成功 ≠ 最终扣减成功（DB 可能失败）
- DB 扣减失败 → 回滚 Redis（保证不超卖）
- 定时对账：Redis 库存 vs DB 库存，发现差异告警

## 场景 B：分桶抗热Key

### 现象

```
热门商品（如 iPhone 15）库存 key 被百万 QPS 访问
单个 Redis 分片被打爆（CPU 100%）
即使 Redis Cluster，热 Key 也会导致某个分片过载
必须分散热 Key 压力
```

### 解决方案

```
分桶策略：
原始库存：stock:sku:1001 = 100000
拆分为 100 个桶：
  stock:sku:1001:bucket:0 = 1000
  stock:sku:1001:bucket:1 = 1000
  ...
  stock:sku:1001:bucket:99 = 1000

用户请求随机分配到某个桶，分散压力
```

```java
// 1. 分桶扣减实现
@Service
public class BucketStockService {
    
    private static final int BUCKET_COUNT = 100;  // 100个桶
    
    // 扣减库存（分桶）
    public Result deductStock(Long skuId, Integer count, Long userId) {
        // 随机选择一个桶（也可以用 userId 哈希，保证同一用户落同一桶）
        int bucketId = ThreadLocalRandom.current().nextInt(BUCKET_COUNT);
        String bucketKey = "stock:sku:" + skuId + ":bucket:" + bucketId;
        
        // 尝试扣减该桶
        Long remain = redisTemplate.opsForValue().decrement(bucketKey, count);
        
        if (remain >= 0) {
            // 扣减成功，发送 MQ 落库
            sendDeductMsg(skuId, count, userId);
            return Result.success("抢购成功");
        }
        
        // 该桶不够，尝试其他桶（最多尝试 3 个桶）
        for (int i = 1; i < 3; i++) {
            int nextBucket = (bucketId + i) % BUCKET_COUNT;
            bucketKey = "stock:sku:" + skuId + ":bucket:" + nextBucket;
            
            remain = redisTemplate.opsForValue().decrement(bucketKey, count);
            if (remain >= 0) {
                sendDeductMsg(skuId, count, userId);
                return Result.success("抢购成功");
            }
        }
        
        // 所有桶都尝试失败，标记售罄
        BloomFilter.markSoldOut(skuId);
        return Result.error("库存不足");
    }
    
    // 初始化分桶库存
    public void initBuckets(Long skuId, Integer totalStock) {
        int perBucket = totalStock / BUCKET_COUNT;
        int remainder = totalStock % BUCKET_COUNT;
        
        for (int i = 0; i < BUCKET_COUNT; i++) {
            int bucketStock = perBucket + (i < remainder ? 1 : 0);
            String bucketKey = "stock:sku:" + skuId + ":bucket:" + i;
            redisTemplate.opsForValue().set(bucketKey, bucketStock);
        }
    }
}
```

**分桶优化**：
- **桶数选择**：QPS 越高，桶数越多（100-1000）
- **本地缓存**：JVM 缓存热点商品的桶映射，减少 Redis 访问
- **动态调整**：某个桶售罄后，自动从其他桶调剂

## 场景 C：限流削峰方案

### 现象

```
双十一零点瞬间流量 100万 QPS
服务只能处理 10万 QPS
超过的部分必须被限流或削峰
否则服务雪崩，全部不可用
```

### 解决方案

```java
// 1. 多层限流：网关层 + 应用层
// 网关层：基于 IP + 用户 ID 限流
@Component
public class RateLimitFilter implements GlobalFilter {
    
    // 令牌桶限流（每个用户每秒 10 次请求）
    private final Map<String, RateLimiter> userLimiters = new ConcurrentHashMap<>();
    
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String userId = getUserId(exchange);
        String ip = getIpAddress(exchange);
        
        // 用户级限流（每秒 10 次）
        RateLimiter userLimiter = userLimiters.computeIfAbsent(
                userId, k -> RateLimiter.create(10.0));  // QPS=10
        
        if (!userLimiter.tryAcquire()) {
            exchange.getResponse().setStatusCode(HttpStatus.TOO_MANY_REQUESTS);
            return exchange.getResponse().setComplete();
        }
        
        // IP 级限流（每秒 100 次）
        RateLimiter ipLimiter = ipLimiters.computeIfAbsent(
                ip, k -> RateLimiter.create(100.0));
        
        if (!ipLimiter.tryAcquire()) {
            exchange.getResponse().setStatusCode(HttpStatus.TOO_MANY_REQUESTS);
            return exchange.getResponse().setComplete();
        }
        
        return chain.filter(exchange);
    }
}

        }
    }
```

## 场景 D：降级方案

### 现象

```
Redis 挂了 → 无法扣减库存
DB 挂了 → 无法落库
MQ 挂了 → 异步任务堆积
必须有多级降级方案，保证核心功能可用
```

### 解决方案

```java
// 1. 多级降级策略
@Service
public class StockServiceWithFallback {
    
    // 扣减库存（带降级）
    public Result deductStockWithFallback(Long skuId, Integer count) {
        // Level 1：Redis 可用 → 正常流程
        if (redisAvailable(skuId)) {
            return deductFromRedis(skuId, count);
        }
        
        // Level 2：Redis 挂了 → 降级到 DB 直接扣减（限流后）
        if (dbAvailable()) {
            log.warn("Redis不可用，降级到DB扣减: skuId={}", skuId);
            return deductFromDb(skuId, count);
        }
        
        // Level 3：DB 也挂了 → 返回"系统繁忙，请重试"
        return Result.error("系统繁忙，请稍后重试");
    }
    
    // Redis 扣减
    private Result deductFromRedis(Long skuId, Integer count) {
        try {
            String stockKey = "stock:sku:" + skuId;
            Long remain = redisTemplate.opsForValue().decrement(stockKey, count);
            
            if (remain < 0) {
                redisTemplate.opsForValue().increment(stockKey, count);
                return Result.error("库存不足");
            }
            
            // 异步落库（失败不影响用户）
            sendMqAsync(skuId, count);
            return Result.success("抢购成功");
            
        } catch (Exception e) {
            log.error("Redis扣减异常，降级", e);
            return deductFromDb(skuId, count);  // 降级到 DB
        }
    }
    
    // DB 直接扣减（降级方案，需要限流保护 DB）
    private Result deductFromDb(Long skuId, Integer count) {
        // 这里需要限流：只允许 1000 QPS 访问 DB
        if (!dbRateLimiter.tryAcquire()) {
            return Result.error("系统繁忙，请稍后重试");
        }
        
        int rows = stockMapper.deductStockDirectly(skuId, count);
        if (rows > 0) {
            return Result.success("抢购成功");
        } else {
            return Result.error("库存不足");
        }
    }
}
```

- **MQ 挂**：同步落库（用户等待时间变长）
- **全链路降级**：关闭推荐、评论等非核心功能，保核心下单

## 核心参数估算

```
双十一零点流量估算：
  预热期（22:00-23:59）：50万 QPS
  爆发期（00:00-00:05）：300万 QPS
  平稳期（00:05-02:00）：50万 QPS
  
系统容量设计：
  Redis Cluster：100 个分片，每分片 10万 QPS → 总 1000万 QPS
  实际设计：按 500万 QPS 冗余（考虑热 Key 问题）
  
分桶数计算：
  300万 QPS / 桶数 = 每桶 QPS < 10万（单 Redis 分片能力）
  → 桶数 > 30，取 100 桶（留有余量）
  
限流配置：
  网关层：单用户 10 QPS，单 IP 100 QPS
  DB 层：1000 QPS（降级时保护 DB）
```

## 追问链

### 追问 1：Redis 预减后，DB 扣减失败怎么处理？

> "三步处理：1. **回滚 Redis**：DB 扣减失败，立即回补 Redis 库存。2. **记录失败订单**：记录到失败表，人工处理。3. **告警**：发送告警通知运维。核心原则：宁可少卖（Redis 有库存但 DB 失败，回滚后用户看到库存恢复），不能超卖。"

### 追问 2：分桶后，某个桶售罄但其他桶有库存怎么办？

> "两种方案：1. **动态调剂**：监控各桶库存，某个桶售罄后，从其他桶调剂（Redis 原子操作）。2. **全桶查询**：扣减失败时，遍历所有桶找有库存的（最多查 100 个桶，用 Lua 脚本批量查）。实际生产推荐方案 1，更高效。"

### 追问 3：限流和削峰有什么区别？

> "**限流**：直接拒绝超过阈值的请求（快速失败）。适用于：保护系统不被压垮。**削峰**：把瞬时流量缓存起来，慢慢处理（MQ 排队）。适用于：流量可延迟处理。实际场景：网关层限流（防压垮）+ MQ 削峰（缓冲流量）。"

### 追问 4：怎么保证不超卖又少卖最少？

> "风险控制：1. **Redis 预减**：DECR 原子操作，不会超卖。2. **DB 兜底**：`UPDATE ... WHERE stock >= count`，双重保险。3. **对账机制**：Redis 库存 vs DB 库存，差异告警。4. **少卖控制**：DB 扣减失败时及时回滚 Redis，避免 Redis 库存被占用。5. **监控**：实时监控库存消耗速度，提前预警。"

## 排查 Checklist

```
□ Redis 扣减是原子的吗？ → DECR/Lua 脚本
□ DB 层有 WHERE stock >= count 吗？ → 防超卖必须
□ 分桶数合理吗？ → 每桶 QPS < 单分片能力
□ 限流规则配置了吗？ → 用户级 + IP 级双维度
□ MQ 消费失败会重试吗？ → 死信队列兜底
□ 降级开关生效了吗？ → Redis/DB/MQ 挂了能自动降级
□ 对账任务执行了吗？ → Redis vs DB 库存差异
□ 热 Key 监控了吗？ → Redis 分片 CPU 使用率
□ 布隆过滤器误判率高吗？ → 预计误判率 < 1%
```
