## 为什么需要 MyBatis-Plus？

原生 MyBatis 只解决了「SQL 可控 + 映射自动化」，但：
- 每张表的单表 CRUD 都要手写 XML（10 张表 = 50 个 SQL）
- 多条件查询需要大量动态 SQL 标签
- 分页/逻辑删除/自动填充等通用需求每个项目都要重复实现

**MP 的定位**：MyBatis 之上的增强层，只做增强不做改变。
- 单表 CRUD → BaseMapper / IService（不用写 XML）
- 多条件查询 → Wrapper 条件构造器
- 复杂 SQL → 仍然用 XML/注解（完全兼容）

## 核心概念速查

```
MyBatis-Plus 核心组件
├── BaseMapper<T>              ← 单表 CRUD 接口（无需实现）
├── IService<T>               ← Service 层 CRUD 接口
├── ServiceImpl<M, T>        ← IService 默认实现
├── AbstractWrapper           ← 条件构造器基类
│   ├── QueryWrapper         ← 查询构造器
│   ├── UpdateWrapper        ← 更新构造器
│   └── LambdaQueryWrapper   ← Lambda 版本（推荐）
├── MybatisPlusInterceptor   ← 插件管理器
│   ├── PaginationInnerInterceptor    ← 分页插件
│   ├── OptimisticLockerInnerInterceptor ← 乐观锁插件
│   └── BlockAttackInnerInterceptor     ← 防全表更新插件
├── MetaObjectHandler         ← 自动填充处理器
└── ISqlInjector             ← SQL 注入器（生成 CRUD SQL）
```

## 学习顺序建议

1. **BaseMapper 核心原理** → 理解 MP 最神奇的地方（为什么继承就有 CRUD）
2. **Wrapper 条件构造器** → 日常使用最多的功能
3. **分页插件** → 分页是后台系统标配
4. **逻辑删除/自动填充/乐观锁** → 通用业务需求
5. **代码生成器** → 新项目快速启动
6. **实战与面试** → 避坑 + 面试准备

## 和 MyBatis 模块的关系

```
原生 MyBatis（05_MyBatis/）
    ├── Mapper 代理原理
    ├── 缓存机制
    ├── 插件机制（Interceptor）
    ├── 动态 SQL 解析
    └── ...（MP 完全兼容，可以直接用）

MyBatis-Plus（07_MyBatis-Plus/）
    ├── BaseMapper（单表 CRUD 增强）
    ├── Wrapper（条件构造器）
    ├── 分页/逻辑删除/自动填充/乐观锁（通用功能封装）
    └── 代码生成器（快速生成 Entity/Mapper/Service/Controller）
```

**两者可以无缝混用**：
- 复杂 SQL 仍然写在 XML 里（MyBatis 原生方式）
- 单表 CRUD 直接用 BaseMapper（MP 方式）
- Wrapper 可以和自定义 SQL 结合使用
