# 服务注册与发现

## 这个问题为什么存在？

> 单体时代，服务调用靠localhost + 端口号。但微服务环境下，**服务实例动态变化**（扩缩容、故障重启、灰度发布），硬编码IP不可行。问题是：**服务消费者怎么知道服务提供者的地址？**

没有注册中心，你会遇到：
- 服务A扩容了，服务B不知道新实例的IP
- 服务C挂了，服务D还在往这个地址发请求
- 手动改配置文件，重启所有消费者（无法接受）

---

## 它是怎么解决问题的？

### 核心机制：注册、发现、心跳

```
服务提供者（订单服务）               服务消费者（前端网关）
         │                                  │
         ├─ 1. 注册 ──→ 注册中心           │
         │              (Eureka/Nacos)      │
         │                                  │
         ├─ 2. 心跳 ──→ 注册中心           │
         │              (每30秒)            │
         │                                  │
         │                    ← 3. 拉取服务列表 ──┤
         │                                  │
         └─ 4. 下线 ──→ 注册中心           │
           (优雅关闭 或 心跳超时)           │
```

**时序**：
```
T0: 订单服务启动 → 注册到Eureka（状态=UP）
T30: 订单服务发送心跳 → Eureka更新最后心跳时间
T90: 订单服务挂了，没发心跳
T120: Eureka检测到超时 → 标记订单服务为DOWN，从列表移除
T150: 前端网关拉取最新列表 → 不再调用已挂的订单服务
```

### 两种发现模式

**客户端发现**（Eureka、Nacos）
```
服务消费者 → 从注册中心拉取列表（本地缓存）
          → 自己选一个实例（负载均衡）
          → 直接调用
```
```java
// Eureka客户端：本地缓存 + 负载均衡
@LoadBalanced  // Ribbon：从本地缓存选一个实例
@Bean
public RestTemplate restTemplate() {
    return new RestTemplate();
}

// 调用时，用服务名（不是IP）
String result = restTemplate.getForObject(
        "http://order-service/api/orders", String.class);
```

**服务端发现**（Kubernetes Service、Nginx）
```
服务消费者 → 调用API网关（或Service）
          → 网关/Service负责转发（负载均衡由服务端做）
```

## 它和相似方案的本质区别是什么？

### Eureka（AP） vs Nacos（AP/CP 可切换）

| | Eureka | Nacos |
|---|---|---|
| CAP权衡 | AP（高可用优先） | 可配置（AP或CP） |
| 一致性协议 | 自研（Peer-to-Peer） | Raft（CP模式） |
| 配置管理 | ❌ 不支持 | ✅ 支持（替代Apollo） |
| 健康检测 | 客户端心跳 | 客户端心跳 + 服务端探测 |
| 适用场景 | 纯服务发现 | 服务发现 + 配置管理 |

**本质区别**：Eureka是**纯AP**，保证高可用但数据可能不一致（网络分区时，客户端用本地缓存）；Nacos可以**切换AP/CP**，且集成了配置管理。

### 客户端发现 vs 服务端发现

| | 客户端发现 | 服务端发现 |
|---|---|---|
| 负载均衡位置 | 客户端（本地） | 服务端（网关/Service） |
| 耦合度 | 高（客户端要集成SDK） | 低（对客户端透明） |
| 灵活性 | 高（客户端自选算法） | 低（服务端定死算法） |
| 适用场景 | Spring Cloud（Eureka） | K8s（Service） |

## 正确使用方式

### Eureka：配置心跳和剔除时间

```yaml
# 服务提供者（订单服务）
eureka:
  instance:
    lease-renewal-interval-in-seconds: 30   # 心跳间隔30秒
    lease-expiration-duration-in-seconds: 90 # 90秒没心跳就剔除
  client:
    service-url:
      defaultZone: http://eureka1:8761/eureka,http://eureka2:8761/eureka

# 服务消费者（网关）
eureka:
  client:
    registry-fetch-interval-seconds: 30  # 30秒拉取一次服务列表
```

### Nacos：AP模式（推荐）

```yaml
spring:
  cloud:
    nacos:
      discovery:
        server-addr: nacos1:8848,nacos2:8848
        # 选AP模式（高可用，适合大部分场景）
        # CP模式需要强一致，性能差
```

## 边界情况和坑

### 坑1：Eureka的自我保护模式

```
场景：15分钟内，超过85%的客户端心跳失败（网络抖动）
Eureka：进入自我保护模式，不再剔除任何实例
后果：服务列表里可能有已挂的实例
```

**解决方案**：
1. **生产环境保留自我保护**（避免网络抖动导致所有服务下线）
2. **客户端要有容错机制**（熔断、重试）
3. **网络稳定后可以关闭**（开发环境）

```yaml
eureka:
  server:
    enable-self-preservation: false  # 开发环境关闭
```

### 坑2：服务下线不优雅

```
场景：服务A关闭，但没向Eureka发"下线"请求
后果：Eureka要等90秒心跳超时才剔除，这90秒内请求还会路由到A
```

**解决方案**：
1. **优雅关闭**（Spring Boot Actuator的`/shutdown`端点）
2. **缩短心跳超时时间**（但会增加网络负担）
3. **客户端重试**（调用失败，重试下一个实例）

### 坑3：本地缓存脏数据

```
场景：Eureka宕机，客户端用本地缓存继续调用
后果：某个服务实例其实已经挂了，但客户端不知道
```

**解决方案**：
1. **客户端要有容错**（熔断、重试）
2. **Eureka集群部署**（3个节点，可用性高）
3. **接受现实**（CAP中P必选，A就意味容忍暂时的不一致）
