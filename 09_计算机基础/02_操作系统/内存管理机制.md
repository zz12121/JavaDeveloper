# 内存管理机制

> **核心认知**：操作系统的内存管理是理解 JVM 内存模型、GC、内存泄漏排查的基础。虚拟内存让每个进程"以为"自己拥有全部内存，页表是虚拟地址到物理地址的翻译字典，TLB 是加速翻译的缓存。

---

## 1. 为什么需要虚拟内存

### 1.1 没有虚拟内存的问题

```
问题1：地址冲突
  程序 A 和程序 B 都想使用 0x1000 地址 → 冲突

问题2：内存不足
  程序需要 1GB 内存，但物理内存只有 512MB → 无法运行

问题3：安全隔离
  程序 A 可以直接读写程序 B 的内存 → 安全风险

问题4：内存碎片
  反复分配和释放导致内存碎片化 → 大块内存无法分配
```

### 1.2 虚拟内存的解决方案

```
虚拟内存（Virtual Memory）：
  每个进程拥有独立的虚拟地址空间（32 位系统 4GB，64 位系统 256TB）

  虚拟地址 → 页表（Page Table）→ 物理地址

  ┌────────────────────────────────────────┐
  │  进程 A 的虚拟地址空间（0~4GB）         │
  │  0x00000000 ─┐                        │
  │  0x00001000 ─┤ 页表A ──→ 物理页框 100  │
  │  0x00002000 ─┤ 页表A ──→ 物理页框 200  │
  │  ...        ─┘                        │
  └────────────────────────────────────────┘
  ┌────────────────────────────────────────┐
  │  进程 B 的虚拟地址空间（0~4GB）         │
  │  0x00000000 ─┐                        │
  │  0x00001000 ─┤ 页表B ──→ 物理页框 300  │
  │  0x00002000 ─┤ 页表B ──→ 物理页框 400  │
  │  ...        ─┘                        │
  └────────────────────────────────────────┘
  相同的虚拟地址 → 不同的物理地址（通过各自的页表）
```

**虚拟内存的优势**：
1. **隔离**：每个进程独立地址空间，互不干扰
2. **安全**：进程不能直接访问物理内存，必须通过操作系统
3. **高效利用**：不常用的页面可以换出到磁盘（Swap），释放物理内存
4. **内存超分配**：虚拟地址空间可以大于物理内存

## 2. 页表机制

### 2.1 分页（Paging）

内存被划分为固定大小的块：

| 概念 | 大小 | 说明 |
|------|------|------|
| 页（Page） | 4KB（默认） | 虚拟地址空间的块 |
| 页框（Page Frame） | 4KB（默认） | 物理内存的块 |
| 页表（Page Table） | - | 虚拟页号 → 物理页框号的映射表 |

**地址转换过程**：

```
虚拟地址（32 位）：
┌──────────────┬──────────────┐
│  虚拟页号(VPN) │  页内偏移     │
│   20 位       │   12 位       │
└──────┬───────┴──────────────┘
       │
       │ 查页表
       ↓
页表项（PTE）：
┌──────────────┬──────┬──────┐
│  物理页框号(PFN) │ 有效位 │ 其他  │
│   20 位       │      │      │
└──────┬───────┴──────┴──────┘
       │
       ↓
物理地址 = 物理页框号 + 页内偏移

转换公式：物理地址 = PTE.PFN × 页大小 + Offset
```

### 2.2 多级页表

**问题**：32 位系统，4GB 地址空间，4KB 一页，页表项 4 字节 → 页表大小 = 4GB / 4KB × 4B = **4MB**。每个进程都需要 4MB 的页表，进程多时浪费内存。

**解决方案**：多级页表，只创建用到的页表项。

```
一级页表（页目录）：1024 个条目（每个 4KB）
  二级页表：每个条目指向一个页表页（1024 个 PTE）
    共 1024 × 1024 = 1048576 个页，4GB 完整覆盖

64 位系统：四级页表（PML4 → PDPT → PD → PT → Page）
```

### 2.3 TLB（Translation Lookaside Buffer）

**问题**：每次内存访问都要先查页表，页表在内存中 → 多一次内存访问 → 性能减半。

**解决方案**：TLB，页表的高速缓存，在 CPU 芯片内。

```
地址转换过程（有 TLB）：
  1. 查 TLB → 命中 → 直接得到物理地址（纳秒级）
  2. 查 TLB → 未命中 → 查页表 → 更新 TLB → 返回物理地址

TLB 命中率通常 > 99%，所以多级页表对性能影响很小。
```

**TLB 失效的原因**：
1. **TLB 未命中**：该虚拟页不在 TLB 中（正常，查页表即可）
2. **TLB Flush**：进程切换时需要清空 TLB（不同进程的虚拟地址映射不同）
3. **缺页中断（Page Fault）**：虚拟页不在物理内存中，需要从磁盘加载

## 3. 页面置换算法

当物理内存满了，需要腾出空间给新页面，选择哪个页面换出？

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **OPT（最优）** | 淘汰未来最长时间不使用的页面 | 理论最优 | 需要预知未来，无法实现 |
| **FIFO** | 淘汰最早进入内存的页面 | 简单 | 可能淘汰常用页面（Belady 异常） |
| **LRU（最近最少使用）** | 淘汰最久没被访问的页面 | 性能接近最优 | 实现开销大（需要维护访问时间） |
| **Clock（时钟算法）** | LRU 的近似实现 | 开销小 | 精度不如 LRU |
| **LFU（最不经常使用）** | 淘汰访问次数最少的页面 | 考虑频率 | 新页面可能被误杀 |

**LRU 的近似实现——Clock 算法**：

```
所有页面排成一个环形链表，每个页面有一个"使用位"（Reference Bit）

1. 指针指向某个页面
2. 如果使用位 = 1：清除使用位，指针前进（再给一次机会）
3. 如果使用位 = 0：淘汰该页面
4. 重复直到找到可淘汰的页面

改进（Modified Clock）：
  同时考虑"使用位"和"修改位"（Dirty Bit）
  优先淘汰未修改的页面（不需要写回磁盘）
```

## 4. 缺页中断（Page Fault）

### 4.1 什么是缺页中断

当 CPU 访问的虚拟页不在物理内存中时，触发缺页中断，由操作系统将页面从磁盘加载到物理内存。

```
CPU 访问虚拟地址 0x2000
  → 查 TLB：未命中
  → 查页表：有效位 = 0（页面不在物理内存中）
  → 触发缺页中断
  → 操作系统处理：
    1. 找到一个空闲页框（或换出一个页面）
    2. 从磁盘读取页面到该页框
    3. 更新页表（设置有效位、物理页框号）
    4. 重新执行导致缺页的指令
```

### 4.2 缺页中断的分类

| 类型 | 说明 | 性能影响 |
|------|------|---------|
| 硬缺页（Hard Fault） | 页面不在物理内存中，需要从磁盘读取 | **慢**（磁盘 IO：毫秒级） |
| 软缺页（Soft Fault） | 页面在物理内存中，但不在当前进程的页表中（共享页面） | 快（仅更新页表映射） |
| 次缺页（Minor Fault） | 首次访问新分配的页面（COW、Zero-fill） | 快（内存操作） |

### 4.3 预读（Read Ahead）

操作系统可以预测进程即将访问的页面，提前加载到物理内存：

```
顺序读取文件时，操作系统预读后续几个页面
  → 减少 Page Fault 次数
  → 提高顺序读取性能

MySQL InnoDB 的预读（Read Ahead）就是基于这个原理。
```

## 5. 内存分配

### 5.1 用户空间内存布局（Linux）

```
高地址
┌──────────────────┐ 0xFFFFFFFF
│   内核空间        │ （1GB，用户不可访问）
├──────────────────┤ 0xC0000000
│   栈（Stack）     │  ↓ 向下增长
│   - 函数调用栈     │
│   - 局部变量       │
├──────────────────┤
│                  │
│   ↓ 空闲区域     │  （用于栈和堆的增长）
│                  │
├──────────────────┤
│   堆（Heap）      │  ↑ 向上增长
│   - malloc/new   │
│   - 动态分配      │
├──────────────────┤
│   BSS 段         │  未初始化的全局/静态变量（零填充）
├──────────────────┤
│   数据段（Data）  │  已初始化的全局/静态变量
├──────────────────┤
│   代码段（Text）  │  程序代码（只读）
└──────────────────┘ 0x00000000
低地址
```

### 5.2 内存分配方式

| 方式 | API | 说明 |
|------|-----|------|
| 静态分配 | 编译时确定 | 全局变量、静态变量 |
| 栈分配 | 函数调用时自动分配 | 局部变量（自动释放） |
| 堆分配 | malloc/free、new/delete | 手动管理，灵活但易泄漏 |
| 内存映射 | mmap/munmap | 映射文件到内存，或匿名映射分配大块内存 |

### 5.3 malloc 的底层实现

```
malloc 并不直接调用 brk/sbrk，而是维护一个内存池（arena）：

  小块内存（< 512B）：使用 Fast Bin / Small Bin（类似 Java 的 TLAB）
  大块内存（≥ 512B）：使用 mmap 分配

为什么小块不用 mmap？
  mmap 每次分配都是独立的虚拟内存区域，需要修改页表
  小块内存通过 brk 调整堆顶指针更高效
```

## 6. Linux 内存参数

```bash
# 查看内存使用
free -h
cat /proc/meminfo

# 查看进程内存映射
pmap -x <pid>
cat /proc/<pid>/maps

# 查看进程内存使用
cat /proc/<pid>/status | grep -E "VmRSS|VmSize|VmSwap"

# 查看页面置换统计
vmstat 1
# si: 换入（Swap In）
# so: 换出（Swap Out）

# 查看缺页中断统计
ps -o min_flt,maj_flt -p <pid>
# min_flt: 次缺页
# maj_flt: 主缺页（磁盘读取）
```

## 7. 内存管理与 JVM 的关系

| 操作系统概念 | JVM 对应 |
|-------------|---------|
| 虚拟地址空间 | JVM 运行时数据区（堆、栈、方法区） |
| 物理内存 | 实际分配的内存（-Xms/-Xmx） |
| 页面置换 | GC 的对象回收（思想类似：找不用的页面/对象回收） |
| 缺页中断 | GC 时的内存分配失败 |
| TLB | JIT 编译后的代码缓存 |
| brk/mmap | 堆内存分配（Java 中大部分通过 mmap 分配） |
| 内存映射（mmap） | MappedByteBuffer / DirectByteBuffer |

**关键理解**：
- JVM 堆从操作系统申请的是**虚拟内存**，实际使用的物理内存取决于访问了哪些页面
- `-Xmx4G` 表示虚拟内存最大 4GB，不是物理内存立即分配 4GB
- Java 的 `DirectByteBuffer` 通过 `malloc` 分配堆外内存（不受 GC 直接管理）
- Linux 的 `OOM Killer` 会在物理内存不足时杀掉占用内存最多的进程（包括 JVM）

## 8. 面试高频问题

### Q1: 虚拟内存的工作原理？

**答**：每个进程有独立的虚拟地址空间，通过页表映射到物理内存。CPU 访问虚拟地址时，先查 TLB（页表缓存），命中则直接转换；未命中则查多级页表。若页面不在物理内存中，触发缺页中断，从磁盘加载。TLB 通常命中率 > 99%，缺页中断频率很低（除非内存不足导致频繁换页）。

### Q2: 什么是缺页中断？为什么频繁缺页会影响性能？

**答**：CPU 访问的虚拟页不在物理内存中时触发缺页中断，操作系统从磁盘将页面加载到物理内存。磁盘 IO 速度（毫秒级）远慢于内存访问（纳秒级），频繁缺页（Thrashing）会导致系统大部分时间都在等待磁盘 IO，性能急剧下降。常见于物理内存不足、交换分区使用过多。

### Q3: LRU 算法是什么？MySQL InnoDB 和 JVM 中的 LRU 有什么不同？

**答**：LRU（Least Recently Used）淘汰最久没被访问的页面。MySQL InnoDB 的 Buffer Pool 使用改进的 LRU（冷热数据分离，新页面先进入冷区，被访问后才移入热区），避免全表扫描污染缓存。JVM 的 G1 收集器也使用了类似 LRU 的思想管理 Region。

### Q4: malloc 和 mmap 的区别？

**答**：malloc 用于小块内存分配，通过 brk 调整堆顶指针，效率高；mmap 用于大块内存分配（通常 > 128KB），每次创建独立的虚拟内存区域。mmap 分配的内存可以独立释放（munmap），不会产生内存碎片，但修改页表有额外开销。
