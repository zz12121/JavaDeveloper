# 类加载机制

## 这个问题为什么存在？

> `.class` 文件只是字节码，CPU 不能直接执行。  
> **类加载机制**就是把字节码加载到内存、变成 `Class` 对象、最终变成可直接使用的 Java 类型的过程。

如果没有类加载机制：
- 所有的类都一次性加载 → 启动超慢、内存爆满
- 无法实现**热部署**、**插件化**

---

## 它是怎么解决问题的？

### 类加载的完整流程

```
┌─────────────────────┐
│       加载（Loading）                │
│  ① 通过全限定名获取 .class 的二进制流 │
│  ② 将字节流转化为方法区的运行时结构  │
│  ③ 在堆中生成 java.lang.Class 对象    │
├─────────────────────┤
│       验证（Verification）             │
│  ① 文件格式验证（魔数 CAFEBABE）  │
│  ② 元数据验证（语义分析）            │
│  ③ 字节码验证（程序语义合法）        │
│  ④ 符号引用验证（可达性）            │
├─────────────────────┤
│       准备（Preparation）             │
│  为类变量（static）分配内存并初始化   │
│  → 注意：是零值，不是用户赋值的值  │
│    pubilc static int value = 123;   │
│    准备阶段：value = 0               │
│    初始化阶段：value = 123           │
├─────────────────────┤
│       解析（Resolution）              │
│  将常量池中的符号引用 → 直接引用     │
├─────────────────────┤
│       初始化（Initialization）         │
│  执行 <clinit>() 方法              │
│  （编译器自动收集所有 static 赋值+   │
│   static{} 块，合并成 <clinit>）    │
└─────────────────────┘
```

**初始化时机**（懒加载原则）：

```java
// 只有这 6 种情况，才会立即初始化类（主动引用）
1. new、读取/设置 static 字段（非 final）、调用 static 方法
2. 反射调用（Class.forName("com.xxx")）
3. 初始化子类 → 父类必须先初始化
4. 主类（包含 main() 的类）
5. JDK 7+：MethodHandle 解析结果为 REF_getStatic 等
6. JDK 8：接口中 default 方法的实现类被初始化时，接口要先初始化
```

### 双亲委派模型（高频追问）

```
        Bootstrap ClassLoader（启动类加载器）
                 ↑
        Extension ClassLoader（扩展类加载器）
                 ↑
        Application ClassLoader（应用程序类加载器）
                 ↑
        Custom ClassLoader（自定义类加载器）
```

**工作流程**：

```
1. 收到类加载请求
2. 先委派给父加载器（不自己加载）
3. 父加载器加载失败（ClassNotFound）
4. 才由自己加载
```

**为什么需要双亲委派？**

> 1. **防止核心类被篡改**  
>    `java.lang.Object` 永远由 Bootstrap 加载，  
>    用户自定义的 `java.lang.Object` 不会被加载（保证了核心类的唯一性）。  
> 2. **避免重复加载**  
>    父加载器加载过的类，子加载器不会再次加载。

### 源码关键路径：ClassLoader.loadClass()

```java
// ClassLoader.java
protected Class<?> loadClass(String name, boolean resolve) {
    synchronized (getClassLoadingLock(name)) {
        // ① 先检查是否已加载
        Class<?> c = findLoadedClass(name);
        if (c == null) {
            try {
                if (parent != null) {
                    c = parent.loadClass(name, false); // ② 委派父加载器
                } else {
                    c = findBootstrapClassOrNull(name); // ③ Bootstrap
                }
            } catch (ClassNotFoundException e) {
                // 父加载器抛出异常
            }
            if (c == null) {
                c = findClass(name);  // ④ 父加载器失败，自己加载
            }
        }
        if (resolve) resolveClass(c);
        return c;
    }
}
```

**自定义类加载器时，应该重写 `findClass()`，而不是 `loadClass()`**：

> 重写 `loadClass()` 会破坏双亲委派模型（除非你明确不想用双亲委派）。  
> 重写 `findClass()` 可以保留双亲委派逻辑，只在父加载器失败时调用自定义逻辑。

### 破坏双亲委派模型（面试加分）

#### Tomcat 的 WebAppClassLoader

> Tomcat 需要**隔离 Web 应用**（每个 Web 应用用自己的类加载器）。  
> 如果完全双亲委派，Web 应用 A 的类会被 Web 应用 B 看到（违反隔离性）。

Tomcat 的类加载顺序（**倒置**）：

```
WebAppClassLoader（优先自己加载）
        ↓ 加载失败
    AppClassLoader
        ↓
    ExtClassLoader
```

**即：先尝试自己加载，不行再委派父加载器。**

---

#### JDBC / SPI 的破坏（线程上下文类加载器）

> JDBC 的接口在 `rt.jar`（Bootstrap 加载），  
> 但实现类在 `mysql-connector.jar`（AppClassLoader 加载）。  

> Bootstrap 加载的类，**无法访问 AppClassLoader 加载的类**（类加载器的可见性问题）。  

> **解决**：用**线程上下文类加载器**（Thread Context ClassLoader）：

```java
// 在 rt.jar 的 DriverManager 里
ClassLoader cl = Thread.currentThread().getContextClassLoader();
// 用这个 cl 去加载 mysql 的驱动（绕开双亲委派）
```

## 它和相似方案的本质区别是什么？

### ClassLoader 继承关系 vs 双亲委派关系

| | ClassLoader 继承 | 双亲委派 parent |
|--|-------------------|-------------------|
| **含义** | Java 继承（extends） | 组合关系（成员变量） |
| **Parent** | `URLClassLoader extends SecureClassLoader` | `ClassLoader.parent` |
| **可定制性** | 编译期固定 | 运行时可改（构造函数传入） |

**双亲委派用的是「组合」，不是「继承」**：

```java
class ClassLoader {
    private final ClassLoader parent;  // 组合，不是继承
    protected ClassLoader(ClassLoader parent) {
        this.parent = parent;
    }
}
```

### 热部署 / 热加载的原理

> 热部署：不用重启 JVM，替换 class 文件后立即生效。  

**实现原理**：

```
1. 自定义类加载器 HotSwapClassLoader
2. 每次加载类时，检查 .class 文件是否有修改（时间戳）
3. 如有修改，创建一个新的 HotSwapClassLoader 实例
4. 用新实例重新加载类（旧实例 + 旧类被 GC 回收）
```

**局限性**：

> 只能替换方法体，**不能修改方法签名**（新增/删除方法 → 需要重启）。  
> 因为旧的类对象可能还在被引用，无法完全卸载。

## 正确使用方式

### 1. 自定义类加载器（加密 .class 文件）

```java
class EncryptedClassLoader extends ClassLoader {
    private String classDir;

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        try {
            byte[] classData = loadClassData(name);  // 读取加密的 .class
            byte[] decrypted = decrypt(classData);      // 解密
            return defineClass(name, decrypted, 0, decrypted.length);
        } catch (Exception e) {
            throw new ClassNotFoundException(name);
        }
    }
}
```

**用途**：防止反编译（Android 的 `dex` 文件加密也类似原理）。

---

### 2. 线程上下文类加载器（SPI 场景）

```java
// 正确获取资源的方式（兼容 SPI）
ClassLoader cl = Thread.currentThread().getContextClassLoader();
InputStream is = cl.getResourceAsStream("config.properties");
```

> 不要硬编码用 `ClassLoader.getSystemClassLoader()`，  
> 因为在 Web 容器里，正确的类加载器是线程上下文类加载器。

## 边界情况和坑

### 1. 自定义类加载器没有 `parent = null` → 无限递归

```java
// ❌ 错误：parent 不为 null，形成循环
class MyLoader extends ClassLoader {
    public MyLoader() {
        super(new MyLoader());  // 父加载器是自己 → 无限递归
    }
}

// ✅ 正确：parent = null（Bootstrap）或 AppClassLoader
class MyLoader extends ClassLoader {
    public MyLoader() {
        super(ClassLoader.getSystemClassLoader());  // 父 = AppClassLoader
    }
}
```

---

### 2. 在静态代码块里触发类初始化 → 死锁

```java
// 类 A
class A {
    static { B.getClass(); }  // 初始化 A 时，触发 B 的初始化
}

// 类 B
class B {
    static { A.getClass(); }  // 初始化 B 时，触发 A 的初始化
}

// 两个线程同时初始化 A 和 B → 互相等待 → 死锁
```

**JVM 规范**：类初始化是同步的（`<clinit>` 加锁），但不会检测「递归初始化」。

---

### 3. `Class.forName()` vs `ClassLoader.loadClass()`

| | `Class.forName()` | `ClassLoader.loadClass()` |
|--|-------------------|------------------------|
| **初始化** | ✅ 执行 `<clinit>`（初始化） | ❌ 只加载 + 链接，不初始化 |
| **使用场景** | 需要初始化（执行 static 块） | 只要加载（懒初始化） |

```java
// forName 会初始化（执行 static 块）
Class<?> c1 = Class.forName("com.xxx.MyClass");

// loadClass 只加载，不初始化
Class<?> c2 = getClass().getClassLoader().loadClass("com.xxx.MyClass");
```

## 我的理解

类加载的本质是**「按需加载」+ 「隔离命名空间」**。

**核心要点**：
1. 加载 → 验证 → 准备 → 解析 → 初始化（5 个阶段）
2. 双亲委派：先父后子（防止核心类被篡改）
3. 破坏双亲委派：Tomcat（隔离 Web 应用）、JDBC（SPI）
4. 自定义类加载器：**重写 `findClass()`**，不要重写 `loadClass()`
