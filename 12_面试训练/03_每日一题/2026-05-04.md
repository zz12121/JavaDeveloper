## 参考答案

### 一、MVCC 解决了什么问题？

数据库并发时，如果只用锁保证隔离性，读操作要等写操作完成 → **读性能极差（写阻塞读）**。

MVCC 的解法：为每行数据保存多个版本，让读操作读「一致性快照」，而不是最新数据 → **读写互不阻塞**。

---

### 二、MVCC 的三大核心组件

#### 1. 隐藏列（行数据结构）

InnoDB 每行数据都有两个关键隐藏列：

```
每行数据
  ├── db_trx_id   (6 bytes) — 最近修改这行的事务 ID
  ├── db_roll_ptr (7 bytes) — 回滚指针，指向 undo log 中的旧版本
  └── db_row_id   (6 bytes) — 主键（无主键时 InnoDB 自动生成）
```

#### 2. undo log 版本链

每次 UPDATE/DELETE，旧值写入 undo log，`db_roll_ptr` 串联成链：

```
当前行 (trx_id=30, balance=300)
    ↓ db_roll_ptr
undo log v2 (trx_id=20, balance=200)
    ↓ db_roll_ptr
undo log v1 (trx_id=10, balance=100)
    ↓ db_roll_ptr = NULL（最老版本）
```

#### 3. Read View（快照，可见性判断的核心）

执行快照读时，InnoDB 创建 Read View，记录以下信息：

| 字段 | 含义 |
|------|------|
| `creator_trx_id` | 创建这个 Read View 的事务 ID |
| `trx_ids[]` | 创建时**活跃（未提交）的事务 ID 列表** |
| `min_trx_id` | `trx_ids[]` 中最小的事务 ID |
| `max_trx_id` | 当前已分配的最大事务 ID + 1 |

**可见性判断规则（四条，必须记住）**：

```
判断一个数据版本的 trx_id 是否对当前事务可见：

① trx_id == creator_trx_id        → 自己修改的，可见 ✅
② trx_id < min_trx_id             → 创建 Read View 前已提交，可见 ✅
③ trx_id >= max_trx_id            → 创建 Read View 之后才开始的事务，不可见 ❌
④ trx_id 在 trx_ids[] 中          → 创建时未提交，不可见 ❌

所有条件都不满足 → 沿 db_roll_ptr 找上一个版本，重复判断
```

---

### 三、Read View 的生成时机：RC vs RR 的本质区别

| 隔离级别 | Read View 生成时机 | 效果 |
|----------|--------------------|------|
| **RC（Read Committed）** | **每次 SELECT** 都重新生成 | 每次读到最新已提交数据（不可重复读） |
| **RR（Repeatable Read）** | **事务第一次 SELECT 时**生成，整个事务复用 | 同一事务内读结果固定（可重复读） |

---

### 四、快照读 vs 当前读

| 类型 | 触发方式 | 原理 | 加锁？|
|------|----------|------|-------|
| **快照读** | 普通 `SELECT` | MVCC 读历史版本快照 | 不加锁 |
| **当前读** | `SELECT ... FOR UPDATE` | 读最新已提交数据 | 加排他锁 |
| **当前读** | `SELECT ... LOCK IN SHARE MODE` | 读最新已提交数据 | 加共享锁 |
| **当前读** | `INSERT / UPDATE / DELETE` | 自动当前读 | 加 X 锁 |

**关键点**：`LOCK IN SHARE MODE` 加的是共享锁，但它读的是最新数据，属于**当前读**，不是快照读。

---

### 五、MVCC 能否完全解决幻读？

**RR 级别下：**
- 快照读 → **不会幻读**（Read View 固定）
- 当前读 → **可能幻读**（读最新数据，其他事务新插入的行也会被扫到）

**解决当前读的幻读**：InnoDB 使用 **Next-Key Lock（记录锁 + 间隙锁）**：

```sql
SELECT * FROM orders WHERE user_id = 100 FOR UPDATE;
-- InnoDB 会锁住 user_id=100 的所有行 + 前后间隙
-- 防止其他事务在间隙中插入，彻底防止幻读
```

---

### 六、常见坑

**1. 长事务导致 undo log 膨胀**

长事务不提交时，它的 Read View 始终存在，InnoDB 不能清理它「可见范围」之前的 undo log → undo log 越积越大 → 存储暴涨，查询变慢。

```sql
-- 监控长事务
SELECT * FROM information_schema.INNODB_TRX
WHERE trx_started < NOW() - INTERVAL 1 HOUR;
```

**2. 先读后写的竞态（MVCC 不防写冲突）**

```java
// ❌ 快照读 + 条件判断 + 写，高并发下会出问题
SELECT balance = 100;     // 两个事务都读到 100
if (balance >= 100) UPDATE balance = balance - 100;  // 都扣款 → 超扣

// ✅ 解决：SELECT ... FOR UPDATE（当前读 + 锁）或乐观锁版本号
```
