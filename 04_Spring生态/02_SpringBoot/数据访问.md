# 数据访问

## 这个问题为什么存在？

> 没有 Spring 的数据访问支持，每个 DAO 都要自己管理连接、处理 `SQLException`、手动映射结果集 → 大量重复代码，且异常处理混乱。  
> Spring 的目标是：**用模板方法统一流程，用异常体系统一异常处理**。

**核心问题**：Spring 是怎么简化数据访问的？

---

## 它是怎么解决问题的？

### Spring 数据访问的异常体系

**问题**：JDBC 的 `SQLException` 太底层（一个异常类表达所有错误），调用方无法区分「连接失败」和「SQL 语法错误」。

**Spring 的解法**：将 `SQLException` 翻译成**细分的运行时异常**。

```
SQLException（受检异常，太宽泛）
        ↓ Spring 异常翻译
DataAccessException（非受检异常，细分子类）
    ├── DuplicateKeyException      （主键冲突）
    ├── DataIntegrityViolationException（数据完整性违反）
    ├── CannotAcquireLockException（锁获取失败）
    ├── QueryTimeoutException     （查询超时）
    └── TransientDataAccessException（ transient 错误，可重试）
```

**异常翻译的原理**：

```java
// SQLExceptionTranslator.translate()
public DataAccessException translate(String task, String sql, SQLException ex) {
    // 根据 SQL 状态码（如 "2300" = 完整性约束违反）
    // 和厂商错误码（如 MySQL 1062 = 主键冲突）
    // 翻译成对应的 Spring 异常
    if (isDuplicateKey(ex)) {
        return new DuplicateKeyException(sql, ex);
    }
    // ...
}
```

> **核心价值**：调用方只需捕获 `DataAccessException`，或者干脆不捕获（让全局异常处理器处理）。

### JdbcTemplate（模板方法模式的经典应用）

**问题**：JDBC 操作流程固定（获取连接 → 创建 Statement → 执行 → 处理结果集 → 关闭资源），每个 DAO 都写一遍太冗余。

**Spring 的解法**：模板方法模式，`JdbcTemplate` 负责固定流程，可变部分用回调传入。

```java
// 更新操作（INSERT/UPDATE/DELETE）
@Repository
public class UserDao {
    @Autowired
    private JdbcTemplate jdbcTemplate;

    public int insert(User user) {
        return jdbcTemplate.update(
            "INSERT INTO user(name, email) VALUES (?, ?)",
            user.getName(), user.getEmail()
        );
    }

    // 查询并映射结果集
    public User findById(Long id) {
        return jdbcTemplate.queryForObject(
            "SELECT * FROM user WHERE id = ?",
            new BeanPropertyRowMapper<>(User.class),
            id
        );
    }
}
```

**JdbcTemplate 的好处**：

1. **资源自动管理**（连接在 finally 中关闭）
2. **异常自动翻译**（`SQLException` → `DataAccessException`）
3. **结果集自动映射**（`BeanPropertyRowMapper` 按字段名自动映射）

### 事务管理（与数据访问深度集成）

**问题**：多个 DAO 操作要绑定到同一个事务（同一个数据库连接）。

**Spring 的解法**：`DataSourceUtils` 统一管理连接，保证同一个事务内的多个 DAO 拿到同一个连接。

```java
// DataSourceUtils 的核心逻辑
public static Connection getConnection(DataSource dataSource) {
    // 1. 先尝试从 TransactionSynchronizationManager 取
    //    （如果当前有事务，连接已经绑定在 ThreadLocal）
    Connection con = TransactionSynchronizationManager.getConnection(dataSource);
    if (con != null) return con;

    // 2. 没有事务，从 DataSource 获取新连接
    con = dataSource.getConnection();

    // 3. 如果当前有事务，把连接绑定到 ThreadLocal
    if (TransactionSynchronizationManager.isActualTransactionActive()) {
        TransactionSynchronizationManager.bindResource(dataSource, con);
    }
    return con;
}
```

> **关键点**：`JdbcTemplate` 内部用 `DataSourceUtils.getConnection()` 获取连接，而不是 `dataSource.getConnection()` 直拿。  
> 这样才能和 Spring 的事务管理无缝集成。

### Spring Data JPA（Repository 抽象）

**问题**：即使有 `JdbcTemplate`，还是要写大量 CRUD 模板代码（`findById`、`save`、`delete`...）。

**Spring Data 的解法**：**根据方法名自动生成查询**（不需要写 SQL）。

```java
public interface UserRepository extends JpaRepository<User, Long> {

    // 根据方法名自动生成查询：WHERE email = ?
    User findByEmail(String email);

    // 多条件：WHERE name LIKE ? AND age > ?
    List<User> findByNameLikeAndAgeGreaterThan(String name, int age);

    // 分页 + 排序
    Page<User> findAll(Pageable pageable);

    // @Query 注解写 JPQL
    @Query("SELECT u FROM User u WHERE u.age > :age")
    List<User> findUsersOlderThan(@Param("age") int age);
}
```

**方法名规则**：

| 关键字 | 示例 | 生成的 SQL |
|---------|--------|------------|
| `findBy` | `findByName` | `WHERE name = ?` |
| `Like` | `findByNameLike` | `WHERE name LIKE ?` |
| `And` | `findByNameAndAge` | `WHERE name = ? AND age = ?` |
| `Or` | `findByNameOrEmail` | `WHERE name = ? OR email = ?` |
| `GreaterThan` | `findByAgeGreaterThan` | `WHERE age > ?` |
| `OrderBy` | `findByAgeOrderByNam` | `WHERE age = ? ORDER BY name` |

### MyBatis 集成

**问题**：JPA 不够灵活（复杂 SQL 不好写），JDBC 太繁琐。MyBatis 取中间路线：**手写 SQL，但自动映射结果**。

```xml
<!-- UserMapper.xml -->
<mapper namespace="com.example.mapper.UserMapper">
    <select id="findById" resultType="User">
        SELECT * FROM user WHERE id = #{id}
    </select>

    <insert id="insert" useGeneratedKeys="true" keyProperty="id">
        INSERT INTO user(name, email) VALUES (#{name}, #{email})
    </insert>
</mapper>
```

```java
// Mapper 接口（MyBatis 自动生成实现）
@Mapper
public interface UserMapper {
    User findById(Long id);
    int insert(User user);
}
```

**Spring Boot 自动配置 MyBatis**：

```yaml
# application.yml
mybatis:
  mapper-locations: classpath:mapper/**/*.xml
  type-aliases-package: com.example.entity
```

> `mybatis-spring-boot-starter` 自动配置了 `SqlSessionFactory` 和 `MapperScannerConfigurer`，  
> 所以只需要写 Mapper 接口和 XML，不需要手动配置。

## 它和相似方案的本质区别是什么？

### JdbcTemplate vs MyBatis vs JPA

| | JdbcTemplate | MyBatis | JPA / Hibernate |
|--|--------------|----------|-----------------|
| SQL 控制 | 完全手动 | 手动（XML/注解） | 自动生成（JPQL） |
| 学习曲线 | 低 | 中 | 高 |
| 复杂查询 | 不方便 | ✅ 擅长 | 不擅长（Criteria API 繁琐） |
| 性能优化 | 直接控制 SQL | 直接控制 SQL | 可能产生 N+1 问题 |
| 适用场景 | 简单 CRUD + 复杂报表 | 复杂查询 + 灵活 SQL | 领域模型丰富的业务系统 |

**选型建议**：

- 互联网高并发场景 → **MyBatis**（SQL 可控，性能好）
- 企业级业务系统 → **JPA**（开发效率高，领域模型清晰）
- 简单 CRUD + 复杂报表 → **JdbcTemplate**（直接写 SQL，不失灵活）

## 正确使用方式

### 1. 批量操作（性能优化）

```java
// ❌ 错误：循环单条插入（N 次网络往返）
for (User user : users) {
    jdbcTemplate.update("INSERT ...", user.getName());
}

// ✅ 正确：批量插入（1 次网络往返）
jdbcTemplate.batchUpdate(
    "INSERT INTO user(name, email) VALUES (?, ?)",
    users,
    100,  // 每批 100 条
    (ps, user) -> {
        ps.setString(1, user.getName());
        ps.setString(2, user.getEmail());
    }
);
```

---

### 2. 声明式事务的正确使用

```java
@Service
public class UserService {

    // ✅ 只读事务（数据库可以优化，如 MySQL 快照读）
    @Transactional(readOnly = true)
    public User findById(Long id) {
        return userRepository.findById(id).orElseThrow();
    }

    // ✅ 写操作：指定回滚规则
    @Transactional(rollbackFor = Exception.class)
    public User create(User user) {
        return userRepository.save(user);
    }

    // ❌ 错误：catch 了异常，事务不会回滚
    @Transactional
    public void createV2(User user) {
        try {
            userRepository.save(user);
        } catch (Exception e) {
            log.error("error", e);
            // 异常被吞，Spring 感知不到，不会回滚
        }
    }
}
```

---

### 3. 分页查询

```java
// JPA 分页
@GetMapping("/users")
public Page<User> list(Pageable pageable) {
    return userRepository.findAll(pageable);
}

// 请求：GET /users?page=0&size=10&sort=name,asc
```

```java
// MyBatis 分页（用 PageHelper 插件）
PageHelper.startPage(1, 10);
List<User> users = userMapper.selectAll();
PageInfo<User> pageInfo = new PageInfo<>(users);
```

## 边界情况和坑

### 1. JPA 的 N+1 问题

```java
// ❌ 错误：N+1 问题
List<Order> orders = orderRepository.findAll();
for (Order order : orders) {
    System.out.println(order.getUser().getName());  // 每条触发一条 SQL
}
// 结果：1 条查 Order + N 条查 User = N+1 次查询
```

**解决**：用 `JOIN FETCH` 或 `@EntityGraph`

```java
@Query("SELECT o FROM Order o JOIN FETCH o.user WHERE ...")
List<Order> findWithUser();

// 或者
@EntityGraph(attributePaths = "user")
@Query("SELECT o FROM Order o WHERE ...")
List<Order> findWithUser();
```

---

### 2. MyBatis 的 `#{}` vs `${}`

```xml
<!-- ✅ #{}：预编译（安全，防 SQL 注入） -->
<select id="findById">
    SELECT * FROM user WHERE id = #{id}
    <!-- 生成：SELECT * FROM user WHERE id = ? （参数化查询） -->
</select>

<!-- ❌ ${}：字符串拼接（有 SQL 注入风险） -->
<select id="findByOrder">
    SELECT * FROM user ORDER BY ${column}
    <!-- 如果 column = "name; DROP TABLE user;" → 灾难！ -->
</select>
```

> **规则**：**能用 `#{}` 就用 `#{}`**，`${}` 只用于动态表名、排序字段（且要白名单校验）。

---

### 3. 数据库连接池配置不当

```yaml
# ❌ 错误：连接池太小 + 没有超时设置
spring:
  datasource:
    hikari:
      maximum-pool-size: 5  # 高并发下不够用

# ✅ 正确：根据 QPS 和 RT 计算
spring:
  datasource:
    hikari:
      maximum-pool-size: 20     # 最大连接数
      minimum-idle: 5            # 最小空闲连接
      connection-timeout: 30000   # 获取连接超时（毫秒）
      idle-timeout: 600000       # 连接空闲超时
```

> **连接池大小的计算公式**：  
> `最大连接数 = (核心线程数 * 平均 RT) / 平均 QPS`  
> 经验值：10 ~ 20（小型应用），50 ~ 100（高并发）。

## 我的理解

**Spring 数据访问的本质**：

1. **模板方法模式** → `JdbcTemplate` 统一流程，简化 JDBC
2. **异常体系** → 将 `SQLException` 翻译成细分的运行时异常
3. **事务集成** → `DataSourceUtils` 保证同一事务用同一个连接
4. **Repository 抽象** → Spring Data JPA 根据方法名自动生成查询
5. **MyBatis 集成** → SQL 可控，自动映射结果

**面试高频追问**：

1. Spring 的异常体系（`DataAccessException` 的细分）
2. `JdbcTemplate` 的资源管理方式（为什么不需要手动关闭连接）
3. JPA 的 N+1 问题及解决方案
4. MyBatis 的 `#{}` 和 `${}` 的区别
5. 数据库连接池的配置原则
