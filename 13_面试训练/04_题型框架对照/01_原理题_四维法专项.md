## 一、为什么要用四维法

```
面试官问原理题，不是要你背书，而是要看你：
1. 知道不知道 → 你有没有了解过这个技术
2. 理解多深 → 你能不能讲清楚机制
3. 有没有思考 → 你能不能对比、评价、延伸

四维法让你有话可说、说得有条理、说得有深度。
```

## 二、四维法详解

### 第一维：它是什么（定义）

```
一句话说清楚，不要超过30个字。

举例：
- HashMap：基于哈希表的键值对存储结构
- synchronized：Java内置的互斥锁，保证原子性/可见性/有序性
- JVM：Java虚拟机，字节码的运行环境

技巧：用"是/就是/指的是"开头
```

### 第二维：它解决了什么问题（背景）

```
用一句话说清楚痛点。

举例：
- HashMap：解决了数组查询快但插入慢、链表插入快但查询慢的问题
- 线程池：解决了频繁创建销毁线程带来的性能开销
- Redis：解决了数据库扛不住高并发读的问题

关键：说清楚"不用它之前有什么问题"
```

### 第三维：核心原理是什么（机制）

```
这是重点，需要展开讲。

举例（HashMap）：
"核心原理分三块：
第一，hash函数，通过key的hashCode确定数组下标，
   hash = hash(key) & (length-1)，利用&位运算提升性能
第二，hash冲突解决，冲突时用链表串联，
   当链表长度超过8且数组长度>=64时转红黑树，
   红黑树查询O(log n)比链表O(n)快
第三，扩容机制，数组满了2倍扩容，
   rehash时重新计算每个元素的桶位置"

技巧：用数字/流程/分步骤的方式讲
```

### 第四维：优缺点/使用场景（判断）

```
评价一个东西，要说清楚好在哪、差在哪。

举例（ConcurrentHashMap vs HashMap）：
"ConcurrentHashMap的优点是线程安全，
   1.7用分段锁，1.8用CAS+synchronized，锁粒度更细
缺点是需要额外的并发控制开销，
   如果不需要并发安全，用HashMap性能更好"

场景：
- 高并发读多：ConcurrentHashMap
- 单线程：HashMap
- 需要有序：LinkedHashMap
```

## 三、高频原理题四维法示范

### 题目1：synchronized原理

```
【是什么】
synchronized是Java内置的互斥锁，通过monitor对象实现，
保证同一时刻只有一个线程执行同步代码块。

【解决什么问题】
解决多线程并发访问共享资源时的竞态条件问题，
保证原子性（不可中断）、可见性（写后刷新主存）、有序性（禁止重排序）。

【核心原理】
分三个层面：
1. 对象头：包含Mark Word，存储锁状态
   - 无锁：对象hashCode、分代年龄
   - 偏向锁：线程ID
   - 轻量级锁：指向栈中锁记录的指针
   - 重量级锁：指向Monitor对象的指针
2. 锁升级（不可逆）：
   偏向锁（单线程）→ 轻量级锁（少量线程竞争）→ 重量级锁（多线程竞争）
3. Monitor：
   - Owner：持有锁的线程
   - WaitSet：调用wait()的线程
   - EntryList：等待获取锁的线程

【优缺点/场景】
优点：JVM内置，自动释放，不会死锁
缺点：锁粒度粗，并发度高时性能差
场景：需要保证线程安全的代码块，小段代码用synchronized更简单
```

### 题目2：JVM运行时数据区

```
【是什么】
JVM运行时数据区是Java程序运行时的内存划分，
分为线程私有区和线程共享区。

【解决什么问题】
JVM自动管理内存，程序员不需要手动分配释放，
通过垃圾回收机制自动回收无用对象。

【核心原理】
分两类：

线程私有（随线程创建销毁）：
1. 程序计数器：记录当前线程执行的字节码行号
2. 虚拟机栈：方法调用栈，每个方法一个栈帧
   栈帧包含：局部变量表/操作数栈/动态链接/返回地址
3. 本地方法栈：Native方法调用栈

线程共享（随JVM启动创建）：
4. 堆：对象分配的主要区域，GC重点区域
   新生代（Eden/Survivor×2）+ 老年代
5. 方法区：存储类信息/常量/静态变量
   JDK8用元空间实现，堆外内存

【优缺点/场景】
堆是GC重点：对象优先在Eden分配，大对象直接进老年代
方法区容易被忽略：反射生成的类、动态代理生成的代理类都存在这里
```

### 题目3：MySQL索引B+树

```
【是什么】
B+树是MySQL InnoDB存储引擎的默认索引结构，
是一种自平衡的多路搜索树。

【解决什么问题】
解决了全表扫描的O(n)问题，
把查询时间复杂度降到O(log n)，并且支持范围查询。

【核心原理】
分三层：
1. 叶子节点层：双向链表连接所有记录
   - 存储所有数据（聚簇索引）或主键值（非聚簇索引）
   - 链表保证范围查询高效
2. 非叶子节点层：只存储索引键值和指针
   - 每个节点可存多个键值
   - 减少树的高度，降低IO次数
3. 索引层：根节点常驻内存
   - 深度一般3-4层（百万级数据）
   - 每层对应一次磁盘IO

B树 vs B+树：
B+树非叶子节点不存数据，同样大小能存更多索引
B+树叶子节点链表，范围查询更快
B+树所有查询都要到叶子节点，更稳定

【优缺点/场景】
优点：查询稳定，支持范围查询，IO次数少
缺点：写入时需要维护索引，有写入放大
场景：主键索引、复合索引、范围查询
```

### 题目4：Redis持久化

```
【是什么】
Redis持久化是将内存数据保存到磁盘的机制，
防止Redis重启后数据丢失。

【解决什么问题】
Redis是内存数据库，断电即失，
持久化解决了数据可靠性问题。

【核心原理】
两种机制：

1. RDB（快照）：
   - 定时执行BGSAVE，fork子进程遍历内存写入RDB文件
   - fork时复制页表，不复制内存（Copy-On-Write）
   - 优点：恢复快，文件紧凑
   - 缺点：可能丢失最后一次快照后的数据

2. AOF（追加）：
   - 每个写命令追加到AOF缓冲区，定时fsync刷盘
   - 三种刷盘策略：always/everysec/no
   - rewrite机制压缩AOF文件
   - 优点：数据丢失少（everysec丢失最多1秒）
   - 缺点：文件大，恢复慢

3. 混合持久化（4.0+）：
   - RDB+AOF，先加载RDB，再执行AOF重放
   - 兼顾恢复速度和完整性

【优缺点/场景】
数据安全要求高 → AOF always
追求性能 + 可接受少量丢失 → RDB
最佳 → 混合持久化
```

### 题目5：Spring事务传播

```
【是什么】
事务传播定义了当一个事务方法被另一个事务方法调用时，
事务如何在这个调用链中传播。

【解决什么问题】
解决Service层嵌套调用时事务边界的问题，
比如A方法有事务，调用B方法，B方法是否需要独立事务。

【核心原理】
7种传播行为：
1. REQUIRED（默认）：有事务就加入，没有就新建
2. REQUIRES_NEW：总是新建事务，挂起外层事务
3. SUPPORTS：有就加入，没有就不使用事务
4. NOT_SUPPORTED：不使用事务，挂起外层事务
5. MANDATORY：必须在事务中运行，抛异常
6. NEVER：不能在事务中运行，抛异常
7. NESTED：嵌套事务，用Savepoint实现

【优缺点/场景】
REQUIRED：大多数场景，简单
REQUIRES_NEW：子方法事务独立，比如日志记录
NESTED：子方法失败只回滚到Savepoint，父事务不全部回滚
```
