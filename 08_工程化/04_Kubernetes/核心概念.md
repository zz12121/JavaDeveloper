# Kubernetes核心概念

> K8s 不是「Docker 的 Web UI」，是**容器编排引擎**。它解决的核心问题是：容器多了怎么管理？怎么保证高可用？怎么自动扩缩容？

---

## 这个问题为什么存在？

不用 K8s 时，Docker 容器的痛点是：

```
场景 A：10 个容器怎么管理？
  手动 docker run 10 次？
  → 容器挂了没人重启
  → 要扩容到 20 个，手动跑 10 次
  → 要升级版本，手动停旧容器、启新容器

场景 B：容器怎么暴露到外网？
  docker run -p 8080:8080 my-app
  → 10 个实例要手动分配不同端口（8080、8081、8082...）
  → 前端怎么知道该访问哪个端口？

场景 C：容器之间怎么通信？
  docker network create my-net
  docker run --network my-net ...
  → 100 个容器，手动管理网络？
  → 容器 IP 变了，其他容器怎么知道？

场景 D：配置怎么管理？
  把配置文件 COPY 进镜像？
  → 改配置要重新构建镜像！
  → 开发/测试/生产 配置不同，要构建 3 个镜像？
```

**Kubernetes 解决的核心问题**：

1. **声明式 API**：我告诉 K8s「我要 3 个实例」，它保证始终有 3 个
2. **自修复**：容器挂了 → K8s 自动重启；节点挂了 → 迁移到其他节点
3. **服务发现**：给 Service 一个 DNS 名，自动负载均衡到后端 Pod
4. **配置管理**：ConfigMap/Secret 管理配置，不进镜像

---

## 它是怎么解决问题的？

### 核心架构：Control Plane + Node

```
┌───────────────────────────────────────────────────────────┐
│                 Control Plane（控制平面）                  │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐          │
│  │   API    │  │  Scheduler │  │ Controller │          │
│  │  Server  │  │            │  │  Manager   │          │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘          │
│       │              │              │                    │
│       └──────────────┴──────────────┘                    │
│                     etcd（分布式存储）                      │
├───────────────────────────────────────────────────────────┤
│                   Worker Node（工作节点）                   │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐          │
│  │  Kubelet │  │ Kube-proxy│  │ Container │          │
│  │          │  │            │  │  Runtime   │          │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘          │
│       │              │              │                    │
│       └──────────────┴──────────────┘                    │
│              Pod（一个或多个容器）                           │
└───────────────────────────────────────────────────────────┘
```

**各组件职责**：

| 组件 | 跑在哪 | 职责 |
|------|--------|------|
| `API Server` | Control Plane | 所有操作的入口（kubectl 也连它） |
| `etcd` | Control Plane | 集群状态存储（唯一数据源） |
| `Scheduler` | Control Plane | 把 Pod 调度到合适的 Node |
| `Controller Manager` | Control Plane | 保证「实际状态 = 期望状态」 |
| `Kubelet` | 每个 Node | 管理本节点上的 Pod 生命周期 |
| `Kube-proxy` | 每个 Node | 维护网络规则（Service → Pod 转发） |
| `Container Runtime` | 每个 Node | 真正跑容器的（containerd、CRI-O） |

### 核心资源对象（API Object）

```
K8s 的所有功能都通过「操作资源对象」来完成：

【最小调度单元】Pod
  - 一个或多个容器的集合（共享网络 + 存储）
  - K8s 不直接管理容器，管理的是 Pod

【副本控制】ReplicaSet / Deployment
  - ReplicaSet：保证始终有 N 个 Pod 在跑
  - Deployment：管理 ReplicaSet 的版本（滚动更新）

【服务暴露】Service
  - 提供稳定的访问入口（IP + DNS 名）
  - 负载均衡到后端多个 Pod

【配置管理】ConfigMap / Secret
  - ConfigMap：配置键值对（不加密）
  - Secret：敏感信息（密码、令牌，base64 编码）

【持久化存储】PersistentVolume（PV）/ PersistentVolumeClaim（PVC）
  - PV：集群级别的存储资源
  - PVC：Pod 对存储的请求（类似「我要 10GB 存储」）

【命名空间】Namespace
  - 逻辑隔离（default / kube-system / 自定义）
```

### Pod 的本质

```
Pod ≠ 容器
Pod = 一个或多个紧密关联的容器 + 共享资源

共享什么？
  - 网络：Pod 内所有容器共享一个 IP
  - 存储：Pod 内所有容器共享 Volume

典型场景：
  Pod 内有一个主容器（你的应用）
  + 一个 sidecar 容器（日志收集、监控采集）
  → 它们共享同一份日志文件（通过 Volume）
```

## 它和相似方案的本质区别是什么？

### Kubernetes vs Docker Swarm

| 维度 | Kubernetes | Docker Swarm |
|------|-------------|---------------|
| 复杂度 | 高（几十个概念要学） | 低（和 Docker CLI 几乎一样） |
| 功能完整性 | 极高（自带扩缩容、滚动更新、配置管理） | 中（基础功能有，高级功能缺） |
| 生态 | 绝对主导（所有云厂商都支持） | 日渐式微（Docker 公司自己都在推 K8s） |
| 学习曲线 | 陡 | 缓 |
| 适合场景 | 生产环境、大型集群 | 小团队、快速原型 |

**为什么 Swarm 输了？**

生态。K8s 赢了「容器编排标准」的战争，现在所有工具都优先支持 K8s。

### Kubernetes vs Nomad（HashiCorp）

| 维度 | Kubernetes | Nomad |
|------|-------------|-------|
| 复杂度 | 高 | 低（二进制文件就跑起来了） |
| 功能范围 | 全栈（容器 + 存储 + 网络 + 配置） | 专注调度（存储/网络用 Consul + Vault） |
| 性能 | 中等（大规模集群需要调优） | 高（十万容器级别） |
| 适合场景 | 大多数团队 | 超大规模、或已经用 HashiCorp 全家桶 |

## 正确使用方式

### 1. 用 YAML 声明资源

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: user-service
  namespace: default
spec:
  replicas: 3        # 要 3 个副本
  selector:
    matchLabels:
      app: user-service
  template:     # Pod 模板
    metadata:
      labels:
        app: user-service
    spec:
      containers:
        - name: user-service
          image: my-registry/user-service:1.0.0
          ports:
            - containerPort: 8080
```

```bash
# 创建资源
kubectl apply -f deployment.yaml

# 查看 Pod
kubectl get pods

# 查看 Pod 详情
kubectl describe pod <pod-name>

# 查看日志
kubectl logs <pod-name>

# 进入容器
kubectl exec -it <pod-name> -- /bin/bash

# 删除资源
kubectl delete -f deployment.yaml
```

### 2. 用 Service 暴露服务

```yaml
# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: user-service
spec:
  selector:
    app: user-service    # 选择标签为 app=user-service 的 Pod
  ports:
    - port: 80            # Service 端口
      targetPort: 8080    # Pod 端口
  type: ClusterIP        # 内部访问（默认）
  # type: NodePort      # 外部可访问（占用节点端口）
  # type: LoadBalancer # 云厂商负载均衡器
```

**Service 的三种类型**：

| 类型 | 访问方式 | 适用场景 |
|------|---------|---------|
| `ClusterIP`（默认） | 集群内访问（Service IP 或 DNS 名） | 微服务间调用 |
| `NodePort` | `<NodeIP>:<NodePort>` | 测试环境外部访问 |
| `LoadBalancer` | 云厂商 LB 的 IP | 生产环境外部访问 |

### 3. 用 ConfigMap 管理配置

```yaml
# configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: user-service-config
data:
  application.properties: |
    spring.datasource.url=jdbc:mysql://mysql:3306/mydb
    spring.datasource.username=root
```

```yaml
# 在 Deployment 中引用 ConfigMap
spec:
  containers:
    - name: user-service
      image: my-app:1.0.0
      volumeMounts:
        - name: config
          mountPath: /app/config   # 挂载到容器路径
  volumes:
    - name: config
      configMap:
        name: user-service-config
```

## 边界情况和坑

### 1. Pod 状态一直是 `Pending`

```
现象：
  kubectl get pods
  NAME            READY   STATUS    RESTARTS   AGE
  user-service-xxx   0/1    Pending   0          5m
```

**原因**：没有足够资源的 Node 可以调度。

```bash
# 查看事件（为什么 Pending）
kubectl describe pod <pod-name>
# 输出中会有：
# Warning  FailedScheduling   no nodes available to schedule pods
```

**解决**：加 Node，或者降低 Pod 的资源请求（`resources.requests`）。

### 2. 镜像拉取失败（ImagePullBackOff）

```
现象：
  STATUS: ImagePullBackOff
```

**原因**：镜像不存在，或者私有仓库没配置拉取密钥。

```bash
# 查看详细错误
kubectl describe pod <pod-name>
# 会显示：
# Failed to pull image "my-private-repo/my-app:1.0": rpc error: code = Unknown
```

**解决**：创建 `imagePullSecret`：

```bash
# 创建 docker-registry secret
kubectl create secret docker-registry my-registry-key \
  --docker-server=https://my-registry.com \
  --docker-username=admin \
  --docker-password=secret \
  --docker-email=me@example.com

# 在 deployment 中引用
spec:
  template:
    spec:
      imagePullSecrets:
        - name: my-registry-key
```

### 3. 删除 Pod 后马上又重建了

```
现象：
  kubectl delete pod user-service-xxx
  → 马上又有一个新的 Pod 被创建
```

**原因**：Deployment 保证「实际状态 = 期望状态」。你删了 Pod，Deployment Controller 发现副本数不够，又创建了一个。

**正确做法**：修改 Deployment 的 `replicas` 或直接 `kubectl edit deployment user-service`。

### 4. `ClusterIP` 在集群外无法访问

```
场景：
  在笔记本上（不在 K8s 集群内）访问 ClusterIP 服务
  → 超时
```

**解决**：用 `kubectl port-forward` 临时转发：

```bash
kubectl port-forward service/user-service 8080:80
# 本地访问 http://localhost:8080 即可
```

## 我的理解

K8s 最深刻的设计是**声明式 API（Declarative API）**：

- **命令式**：我告诉系统「怎么做」（如 `docker run`、`kubectl run`）
- **声明式**：我告诉系统「要什么」（如 `kubectl apply -f deployment.yaml`）

K8s 会一直保证「实际状态 = 期望状态」——这个过程叫 **Reconciliation Loop**（调谐循环）。

```
Reconciliation Loop 原理：
  1. 用户声明：要 3 个 Pod
  2. K8s 记录到 etcd
  3. Controller 监听 etcd，发现「期望 3 个，实际 2 个」
  4. Controller 创建 1 个新 Pod
  5. 过了一会，1 个 Pod 挂了
  6. Controller 发现「期望 3 个，实际 2 个」
  7. Controller 又创建 1 个新 Pod
  → 永远保证「实际 = 期望」
```

**面试高频考点**：

1. K8s 的核心架构（Control Plane + Node，各组件职责）—— 必问
2. Pod 和容器的关系 —— Pod 是调度单元，包含一个或多个容器
3. Deployment 和 ReplicaSet 的关系 —— Deployment 管理 ReplicaSet 的版本
4. Service 的三种类型 —— ClusterIP / NodePort / LoadBalancer
5. 你们生产环境用 K8s 吗？怎么部署的？—— 开放题，结合实战

*K8s 是「数据中心的操作系统」。不理解它，就像写代码不理解操作系统一样——能做，但做不好。*
