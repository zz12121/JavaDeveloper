## CAP 定理与 BASE 理论

**关联知识文档**：[[08_分布式与架构/01_分布式基础/CAP定理与BASE理论]]

**第一问**：「CAP 是什么？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 为什么 P（分区容错）必须满足？ | 网络分区不可避免 |
| 第二层 | CP 系统和 AP 系统各举两例？ | ZK/RocketMQ（CP）/ Eureka/Cassandra（AP）|
| 第三层 | BASE 理论和 CAP 的关系？ | BASE 是 AP 的延伸，最终一致性 |

**对比型考法**：「ZooKeeper 是 CP 还是 AP？为什么？」

**面试话术总结**：

- **CAP**：Consistency（一致性）、Availability（可用性）、Partition Tolerance（分区容错）。分布式系统 P 必须保证，所以只能在 C 和 A 之间权衡
- **CP 系统**：优先一致性，分区时不可用（ZK、etcd、HBase）
- **AP 系统**：优先可用性，分区时返回旧数据（Eureka、Cassandra、DynamoDB）
- **BASE 理论**：Basically Available（基本可用）、Soft State（软状态）、Eventually Consistent（最终一致性）。AP 系统的设计原则，接受短暂不一致，保证最终一致

## 分布式事务

**关联知识文档**：[[08_分布式与架构/02_分布式事务/分布式事务]]

**第一问**：「分布式事务有哪些解决方案？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 2PC 有什么缺陷？ | 协调者单点 / 同步阻塞 / 数据不一致 |
| 第二层 | TCC 和 Saga 的区别？ | 资源预留 vs 补偿 |
| 第三层 | 本地消息表和 RocketMQ 事务消息怎么选？ | 实现复杂度 / 可靠性 / 业务侵入性 |

**实战型考法**：「你的项目里怎么处理分布式事务的？」

**面试话术总结**：

- **2PC（两阶段提交）**：阶段一 Prepare（协调者问所有参与者能否提交）→ 阶段二 Commit/Rollback。缺点：同步阻塞、协调者单点、脑裂（网络分区导致部分提交部分回滚）
- **TCC（Try-Confirm-Cancel）**：Try（预留资源，如冻结金额）→ Confirm（确认提交）→ Cancel（取消，回滚）。业务侵入性强，需要编写三个方法
- **Saga**：长事务拆分为多个本地事务，每个事务有对应的补偿操作。执行顺序：T1→T2→T3，如果 T2 失败则执行 C2→C1 补偿。两种模式：编排（Choreography，事件驱动）和协调（Orchestration，中心控制器）
- **本地消息表**：业务操作 + 消息表写入同一本地事务 → 定时任务扫描消息表 → 发送消息 → 消费者幂等处理。可靠但需要定时扫描
- **RocketMQ 事务消息**：半消息（不可消费）→ 执行本地事务 → 提交/回滚消息。如果未确认，Broker 主动回查事务状态。业务侵入性比 TCC 小

## 分布式锁

**关联知识文档**：[[08_分布式与架构/03_分布式锁/分布式锁]]

**第一问**：「分布式锁怎么实现？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | Redis 分布式锁和 ZooKeeper 分布式锁的区别？ | AP vs CP / 性能 vs 一致性 |
| 第二层 | Redisson 的看门狗机制？ | 自动续期防止业务未完成锁过期 |
| 第三层 | 如何实现可重入锁？ | Hash 结构存重入次数 / 线程标识 |

**面试话术总结**：

- **Redis vs ZK vs etcd**：
  | 维度 | Redis | ZooKeeper | etcd |
  |------|-------|-----------|------|
  | 一致性 | AP（Redisson）| CP | CP |
  | 性能 | 高（单线程内存操作）| 中（磁盘写）| 中 |
  | 适用 | 高并发、允许偶尔失效 | 强一致场景（如选主）| K8s 场景 |
- **Redis 锁要点**：`SET key value NX EX 30`（原子加锁+过期）；Redisson 看门狗自动续期；Lua 脚本保证 unlock 原子性（判断值 + 删除）
- **ZK 锁要点**：创建临时顺序节点 → 判断自己是否最小序号 → 是则获取锁，否则 watch 前一个节点。优点：CP、支持公平锁；缺点：性能低于 Redis

## 限流算法

**关联知识文档**：[[08_分布式与架构/04_高可用设计/限流算法]]

**第一问**：「常见的限流算法有哪些？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 令牌桶和漏桶的区别？ | 允许突发 vs 完全平滑 |
| 第二层 | 固定窗口的「临界双倍问题」具体是什么？ | 窗口边界附近请求量翻倍 |
| 第三层 | 分布式限流怎么实现？ | Redis + Lua（滑动窗口/令牌桶）|

**实战型考法**：「你的系统做过限流吗？阈值怎么设的？」

**面试话术总结**：

- **四大算法**：
  - **固定窗口**：单位时间窗口内计数，简单但有「临界双倍」问题（窗口交界处可能 2 倍流量通过）
  - **滑动窗口**：将窗口细分为小格子，滑动计算，解决临界问题
  - **漏桶**：请求进入桶中，以恒定速率处理。**完全平滑**，不允许突发
  - **令牌桶**：以恒定速率向桶中放令牌，请求需要获取令牌，桶满则丢弃。**允许突发**（桶中有积累的令牌时）
- **Guava RateLimiter**：基于令牌桶，支持预热（`warmup`）和平滑突发限流
- **分布式限流**：Redis + Lua 脚本（保证原子性）；Sentinel（阿里巴巴，支持流控、熔断、降级一体化）

## 熔断与降级

**关联知识文档**：[[08_分布式与架构/04_高可用设计/熔断与降级]]

**第一问**：「熔断和降级有什么区别？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 熔断的三个状态是什么？ | CLOSED / OPEN / HALF_OPEN |
| 第二层 | Sentinel 和 Hystrix 的区别？ | 线程隔离 vs 信号量隔离 / Hystrix 已停更 |
| 第三层 | 降级返回什么数据？ | 缓存数据 / 默认值 / 友好提示 |

**面试话术总结**：

- **熔断**：当下游服务异常率超过阈值时，**自动切断调用**（防止级联故障）。三个状态：
  - `CLOSED`：正常状态，请求通过，统计失败率
  - `OPEN`：失败率超过阈值，熔断开启，请求直接被拒绝（快速失败）
  - `HALF_OPEN`：熔断一段时间后进入半开状态，放行少量请求探测下游是否恢复
- **降级**：在系统压力大或服务不可用时，**主动牺牲非核心功能**保证核心可用（如关闭推荐系统、返回缓存数据）
- **Sentinel vs Hystrix**：
  - Hystrix：线程池隔离（每个服务独立线程池，资源开销大）；已停更（2018 年）
  - Sentinel：信号量隔离（轻量）；支持流量整形、热点限流、系统自适应保护；活跃维护

## 负载均衡

**关联知识文档**：[[08_分布式与架构/05_系统设计/负载均衡策略]]

**第一问**：「常见的负载均衡算法有哪些？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 一致性哈希相比普通哈希的优势？ | 节点变更影响范围小 |
| 第二层 | 四层负载均衡和七层负载均衡的区别？ | 传输层（IP+端口）/ 应用层（HTTP Header）|
| 第三层 | 灰度发布用什么策略？ | 按 Header / Cookie / 权重 |

**面试话术总结**：

- **常见算法**：轮询（Round Robin）、加权轮询（Weighted RR）、随机（Random）、加权随机、最小连接数（Least Connections）、一致性哈希（Consistent Hash）
- **一致性哈希**：将节点和请求都映射到 0~2^32 的环上，请求顺时针找最近的节点。虚拟节点解决数据倾斜
- **四层 vs 七层**：四层（LVS、Nginx stream）在传输层转发，性能高但无法基于内容路由；七层（Nginx http、Gateway）在应用层解析请求内容，可基于 URL/Header/Cookie 路由

## 分布式 ID

**关联知识文档**：[[08_分布式与架构/05_系统设计/分布式ID]]

**第一问**：「分布式 ID 怎么生成？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | UUID 的优缺点？ | 生成简单但无序、字符串太长、无业务含义 |
| 第二层 | 雪花算法（Snowflake）的原理？ | 时间戳 + 机器 ID + 序列号 |
| 第三层 | 雪花算法的时钟回拨问题怎么解决？ | 等待回拨结束 / 多worker容忍 |

**面试话术总结**：

- **雪花算法**（Twitter）：64 位 long → 1bit 符号位 + 41bit 时间戳（约 69 年）+ 10bit 机器 ID（1024 台）+ 12bit 序列号（每毫秒 4096 个）
- **时钟回拨**：NTP 校时可能导致时间回拨 → 解决方案：回拨时间短则等待、回拨时间长则报错、多 worker 容忍（美团 Leaf）
- **方案对比**：
  | 方案 | 有序性 | 性能 | 可用性 | 缺点 |
  |------|--------|------|--------|------|
  | UUID | 无序 | 高 | 高 | 字符串太长、无序 |
  | 数据库自增 | 有序 | 低 | 低 | 单点瓶颈 |
  | Redis INCR | 有序 | 高 | 中 | 依赖 Redis |
  | 雪花算法 | 趋势有序 | 高 | 高 | 时钟回拨 |
  | Leaf（美团）| 有序 | 高 | 高 | 依赖 ZK |

## 微服务核心概念

**关联知识文档**：[[08_分布式与架构/03_微服务核心/微服务]]

**第一问**：「微服务架构有哪些核心组件？」

| 追问层次 | 追问 | 考点 |
|---------|------|------|
| 第一层 | 服务注册、配置中心、网关分别做什么？ | 服务发现 / 集中配置 / 路由鉴权 |
| 第二层 | 微服务之间怎么通信？同步 vs 异步？ | Feign/gRPC（同步）/ MQ（异步）|
| 第三层 | 微服务的缺点是什么？ | 运维复杂度高 / 分布式事务 / 链路追踪难 |

**面试话术总结**：

- **核心组件**：服务注册发现（Nacos/Eureka）、配置中心（Nacos/Apollo）、API 网关（Gateway/Zuul）、熔断降级（Sentinel）、链路追踪（SkyWalking/Zipkin）、分布式事务（Seata）、消息队列（RocketMQ/Kafka）
- **服务通信**：同步（HTTP/REST/Feign、gRPC/Protobuf）；异步（MQ 解耦，最终一致性）
- **微服务拆分原则**：按业务领域拆分（DDD 领域驱动设计）、高内聚低耦合、每个服务独立数据库、独立部署
- **微服务 vs 单体**：微服务适合大型团队和复杂业务，单体适合小团队和快速迭代。不要为了微服务而微服务
