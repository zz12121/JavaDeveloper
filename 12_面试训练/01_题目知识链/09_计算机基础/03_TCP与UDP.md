**TCP vs UDP 对比表**：

| | TCP | UDP |
|--|-----|-----|
| 连接性 | 面向连接（三次握手） | 无连接 |
| 可靠性 | 可靠（确认、重传、排序） | 不可靠（尽最大努力） |
| 有序性 | 保证有序（序号 + 排序） | 不保证有序 |
| 流量控制 | 有（滑动窗口） | 无 |
| 拥塞控制 | 有（慢启动、拥塞避免） | 无（应用自己控制） |
| 头部开销 | 20~60 字节 | 8 字节 |
| 传输单位 | 字节流（无边界） | 数据报（保留边界） |
| 广播/多播 | 不支持 | 支持 |
| 适用场景 | 文件传输、邮件、HTTP | 视频、DNS、游戏、QUIC |

**TCP 可靠传输的 4 件大事**：

```
1. 序号和确认应答（SEQ/ACK）
   - 每个字节编序号，接收方确认收到的最后一个字节

2. 超时重传
   - 发送方发数据后启动定时器，超时没收到 ACK → 重传
   - RTT 动态估算超时时间（RTO = RTT + 4*RTT_var）

3. 滑动窗口
   - 接收方告诉发送方「我还能收多少」（窗口大小）
   - 发送方在未收到 ACK 的情况下，可以发窗口大小的数据

4. 拥塞控制
   - 慢启动：指数增长窗口，探测网络容量
   - 拥塞避免：线性增长
   - 快重传/快恢复：收到3个重复 ACK → 立即重传（不等超时）
```

**UDP 提高可靠性的方式**（应用层实现）：

```java
// 在应用层加：序号、确认、超时重传、排序
// 例如：QUIC 协议就是在 UDP 上实现了 TCP 的所有可靠性机制，还加了加密
```

**QUIC 的核心改进**：

```
1. 0-RTT 握手（TCP 是 1-RTT，TLS 额外 1~2 RTT）
2. 解决队头阻塞（TCP 一个包丢了，后面所有包都要等；QUIC 是每个流独立）
3. 连接迁移（客户端换 IP/网络，TCP 连接会断；QUIC 用 Connection ID，不断）
4. 内置加密（TLS 1.3）
```
