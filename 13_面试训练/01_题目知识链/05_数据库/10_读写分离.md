# 读写分离是怎么实现的？有什么坑？

> ⚠️ **先盲答**：读写分离是什么？它解决什么问题？常见实现方式有哪些？

## 盲答引导

1. 读写分离的基本架构是什么？—— 一主多从？多主多从？

## 知识链提示

```
读写分离
  → [[06_数据库/05_高可用/读写分离设计]]
    → 基本架构
      → 一主多从：主库写，多个从库读（最常见）
      → 多主多从：双主互备 + 多从（高可用）
    → 实现方式
      → 应用层实现：Spring 的 AbstractRoutingDataSource
        → 根据注解 / 方法名判断走主库还是从库
        → 优点：简单，无中间件依赖
        → 缺点：侵入业务代码，扩展困难
      → 中间件实现：MyCAT / ShardingSphere-Proxy
        → SQL 解析 → 自动路由到主库/从库
        → 优点：对应用透明，支持分库分表
        → 缺点：引入中间件，增加运维成本
    → 主从延迟的坑
      → 写主库后立刻读从库 → 可能读不到（延迟几毫秒到几秒）
      → 解决方案
        → 写后读自己：同一用户写完后，强制读主库（缓存用户→主库映射）
        → 时间戳方案：写时记录时间戳，读时判断从库延迟是否可接受
        → 缓存分担读压力：热点数据走 Redis，减轻从库压力
```

---

## 参考答案要点

**核心思想**：写操作走主库，读操作走从库，分摊读压力。

**最大坑：主从延迟** → 写完后立刻读从库，可能读不到最新数据。

**三种解决方案**：
1. 写后读自己 → 同一会话/用户强制走主库
2. 先睡几毫秒再读 → 不优雅，但简单有效
3. 关键数据走缓存 → Redis 挡在读前面
