## 二、原理/概念题 — 四维法框架

### 框架模板

```
四维法 = 是什么 + 解决什么问题 + 核心原理 + 优缺点

1. 它是什么？（一句话定义）
2. 它解决了什么问题？（为什么要有它）
3. 核心原理是什么？（怎么工作的）
4. 优缺点/使用场景？（我的判断）
```

### 知识库对照 — 按模块

#### 模块1：Java核心

| 知识点 | 对应知识库文件 | 四维法要点 |
|--------|--------------|-----------|
| **HashMap** | 集合框架 | 是什么：KV存储，通过hash定位<br>解决：O(1)查找<br>原理：数组+链表+红黑树，hash冲突用链表，超过8转红黑树<br>缺点：线程不安全 |
| **ConcurrentHashMap** | 并发容器 | 是什么：线程安全的HashMap<br>解决：并发场景下的安全读写<br>原理：JDK7分段锁，JDK8 CAS+ synchronized +Node<br>vs HashMap：并发安全 |
| **ArrayList vs LinkedList** | 集合框架 | 区别：数组vs链表，随机访问vs插入删除<br>ArrayList：连续内存，寻址快，扩容代价大<br>LinkedList：离内存，插入快，但无法高效随机访问 |
| **synchronized原理** | synchronized关键字 | 是什么：内置锁，保证原子性/可见性/有序性<br>解决：多线程并发访问共享资源的安全问题<br>原理：对象头Mark Word + Monitor + CAS<br>升级：偏向锁→轻量级锁→重量级锁 |
| **volatile原理** | volatile关键字 | 是什么：轻量级同步机制<br>解决：可见性问题，禁止指令重排序<br>原理：内存屏障，写后刷新主存，读前失效CPU缓存<br>不能替代synchronized：不保证原子性 |
| **ThreadLocal** | ThreadLocal线程本地变量 | 是什么：线程本地存储，每个线程独立副本<br>解决：避免对象传递，减少线程间竞争<br>原理：Thread对象持有ThreadLocalMap<br>注意：内存泄漏（弱引用+remove） |
| **AQS原理** | AQS抽象队列同步器 | 是什么：抽象队列同步器，JUC的基石<br>解决：自定义锁的通用实现框架<br>原理：state + CLH队列（双向链表）<br>子类实现：tryAcquire/tryRelease |
| **ReentrantLock** | ReentrantLock与显式锁 | 是什么：可重入锁，显式控制<br>vs synchronized：可中断、可超时、多条件变量、公平锁<br>synchronized：JVM内置，自动释放 |
| **线程池原理** | 线程池 | 是什么：复用线程，避免频繁创建销毁<br>参数：corePoolSize/maxPoolSize/queue/拒绝策略<br>原理：提交任务→核心线程→队列→最大线程→拒绝<br>常见类型：Fixed/Cached/Scheduled |
| **Fork/Join** | ForkJoin框架 | 是什么：分治并行框架<br>解决：大任务拆分并行执行<br>原理：Work-Stealing算法（工作窃取）<br>使用：RecursiveTask/RecursiveAction |
| **CompletableFuture** | CompletableFuture异步编排 | 是什么：异步编程利器<br>解决：回调地狱，串行依赖<br>原理：依赖图+线程池执行<br>用法：thenApply/thenCompose/thenCombine/exceptionally |
| **死锁** | 死锁与活锁 | 是什么：多个线程互相等待对方持有的资源<br>条件：互斥/占有并等待/不抢占/循环等待<br>排查：jstack/jvisualvm<br>解决：打破任一条件，顺序加锁 |

#### 模块4：数据库

| 知识点 | 对应知识库文件 | 四维法要点 |
|--------|--------------|-----------|
| **B+树索引** | B+树索引原理 | 是什么：MySQL默认索引，平衡多路查找树<br>解决：快速定位数据，O(log n)<br>结构：非叶子节点存索引，叶子节点存所有数据+指针<br>特点：范围查询友好，IO次数少 |

| **索引失效** | 索引失效场景 | 失效场景：<> / not in / or / 函数 / 类型转换 / like前通配<br>最佳实践：覆盖索引/最左前缀/区分度高的列<br>为什么：破坏了索引有序性 |
| **事务ACID** | ACID实现原理 | 原子性：undo log回滚<br>一致性：数据库本身保证<br>隔离性：锁+MVCC<br>持久性：redo log+刷盘 |

| **InnoDB锁** | InnoDB锁机制 | 锁类型：行锁（Record/Gap/Next-Key）、表锁<br>行锁：锁定索引记录<br>间隙锁：锁定范围，防止幻读<br>临键锁：Record+Gap，解决幻读 |

| **分库分表** | 分库分表方案 | 是什么：水平拆分数据<br>方案：ShardingSphere/MyCat<br>问题：跨表查询、分布式ID、事务一致性<br>中间件：ShardingSphere-JDBC/Proxy |
| **主从复制** | 主从复制原理 | 原理：binlog→IO线程→relay log→SQL线程执行<br>模式：异步/半同步/全同步<br>问题：主从延迟<br>解决方案：并行复制/降低负载/读写分离 |
| **执行计划** | EXPLAIN执行计划 | 字段：id/select_type/type/key/rows/Extra<br>type：system>const>eq_ref>ref>range>index>ALL（差到好）<br>Extra：Using index/Using filesort/Using temporary |

#### 模块7：计算机基础

| 知识点 | 对应知识库文件 | 四维法要点 |
|--------|--------------|-----------|
| **TCP三次握手** | TCP三次握手 | 过程：SYN→SYN+ACK→ACK<br>为什么三次：同步序列号，确认双方收发能力<br>状态：LISTEN/SYN_SENT/SYN_RCVD/ESTABLISHED |
| **TCP四次挥手** | TCP四次挥手 | 过程：FIN→ACK→FIN→ACK<br>为什么四次：全双工，需要各自关闭<br>TIME_WAIT：2MSL，确保最后ACK到达<br>CLOSE_WAIT：等待处理完数据 |
| **TCP滑动窗口** | TCP滑动窗口 | 是什么：流量控制机制<br>解决：发送方发太快，接收方处理不了<br>原理：rwnd窗口大小，动态调整<br>拥塞控制：cwnd，慢开始/拥塞避免/快恢复 |
| **HTTP vs HTTPS** | HTTP与HTTPS | HTTP：无状态，明文传输，80端口<br>HTTPS：加密传输，443端口，SSL/TLS<br>TLS：非对称加密（握手）+对称加密（传输）<br>数字证书：CA认证，解决中间人攻击 |
| **TCP vs UDP** | TCP与UDP | TCP：面向连接，可靠，字节流，拥塞控制<br>UDP：无连接，不可靠，数据报，无连接管理<br>选择：TCP用于可靠传输（HTTP/邮件）；UDP用于实时性（视频/DNS） |
| **Cookie vs Session** | Cookie与Session | Cookie：存储在客户端，4KB，有大小限制<br>Session：存储在服务端，无大小限制<br>分布式Session：黏性Session/ Session复制/ Session集中存储/ Token方案 |
| **进程 vs 线程** | 进程与线程 | 进程：资源分配单位，独立地址空间<br>线程：CPU调度单位，共享进程资源<br>切换开销：进程大（MMU切换），线程小（共享资源）<br>选择：IO密集多线程，CPU密集多进程 |

## 三、系统设计题 — 四步设计法

### 框架模板

```
第一步：确认需求和约束
第二步：顶层设计（高并发/高可用/可扩展）
第三步：核心模块设计
第四步：权衡取舍
```

### 知识库对应的系统设计题

| 设计题目 | 考察模块 | 核心知识点 | 设计要点 |
|---------|---------|----------|---------|

## 六、问题排查题 — 四步排查法

### 框架模板

```
第一步：确认问题现象（从外到内）
第二步：缩小范围（分段排查）
第三步：定位根因（找到具体原因）
第四步：解决+预防
```

### 知识库对应的排查场景

| 排查场景 | 排查工具/命令 | 核心知识点 |
|---------|-------------|----------|
| **CPU 100%** | top → jstack → Arthas →火焰图 | 死循环/频繁GC/线程竞争 |
| **内存溢出** | jmap -heap → jmap -dump → MAT分析 | 内存泄漏/大对象/数组膨胀 |
| **频繁Full GC** | jstat -gcutil → GC日志分析 | 对象过多/Metaspace膨胀/老年代空间不足 |
| **接口超时** | Arthas trace → 日志 → APM链路追踪 | 慢SQL/外部调用超时/GC停顿 |
| **数据库慢查询** | slow_query_log → explain → processlist | 全表扫描/索引失效/锁等待 |
| **Redis变慢** | slowlog → info commandstats → bigkey | 大key/热key/命令复杂度/内存碎片 |
| **Kafka消费积压** | kafka-consumer-groups → lag监控 | 消费者挂了/处理慢/分区不均 |
| **分布式事务失败** | Seata日志 → TCC日志 → MQ消息 | 网络超时/空回滚/悬挂问题 |

## 七、所有知识点的面试考察层次

> 知道这个知识点会怎么问，才能知道要准备什么。

### 第一层：概念题（知道就行）

```
问法：「是什么？」
举例：「什么是IOC？」「什么是AOP？」
准备：用四维法，一句话定义
```

### 第二层：原理题（理解机制）

```
问法：「底层怎么实现的？」
举例：「synchronized底层是什么？」
「HashMap扩容为什么是2倍？」
准备：源码级别，理解核心流程
```

### 第三层：对比题（融会贯通）

```
问法：「和XXX有什么区别？」
举例：「synchronized和ReentrantLock区别？」
「Redis和Memcached区别？」
准备：列表对比，抓住核心差异
```

### 第四层：实战题（能不能用）

```
问法：「你在项目里怎么用的？」
举例：「项目中怎么用线程池的？」
「Redis缓存怎么设计的？」
准备：STAR法则包装项目经历
```

### 第五层：追问题（思维深度）

```
问法：「如果...会怎样？」「还有呢？」
举例：「如果HashMap不扩容会怎样？」
「如果Redis master挂了怎么办？」
准备：理解边界情况，有自己的思考
```
