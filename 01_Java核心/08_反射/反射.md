# 反射

## 这个问题为什么存在？

> 从「没有这个东西会怎样」出发，解释问题的根源。

在反射出现之前，Java程序的功能在**编译期**就完全确定：

```java
// 没有反射时，调用方法必须写死类名
UserService service = new UserService();
service.login();
```

**痛点**：
1. **无法动态加载类**：类名在编译期就必须确定
2. **无法动态调用方法**：方法名在编译期就必须确定
3. **无法访问私有成员**：封装性过强，导致测试、框架开发困难

**核心问题**：如何让Java程序在**运行时**动态获取类信息、调用方法、修改字段？

反射的本质是**运行时自我检查（Runtime Introspection）与动态调用**。

## 它是怎么解决问题的？

> 不止讲「是什么」，要讲清楚「为什么这样设计能解决问题」。

### 核心机制

#### 1. 反射的三大核心类

```
┌─────────────────────────────────────────────────────────────┐
│                  Java 反射三大核心类                         │
│                                                             │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │  Class         │  │  Field        │  │  Method       │  │
│  │  （类信息）    │  │  （字段信息）   │  │  （方法信息）   │  │
│  │  - getName()   │  │  - get()       │  │  - invoke()    │  │
│  │  - getMethods()│  │  - set()       │  │  - getReturnType│  │
│  │  - newInstance │  │  - setAccessible│  │  - getParameter │  │
│  └──────────────┘  └──────────────┘  └──────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

#### 2. 获取Class对象的三种方式

```java
// 1. 类字面量（编译期已知类型）
Class<UserService> clazz1 = UserService.class;

// 2. 对象.getClass()（运行时已知对象）
UserService service = new UserService();
Class<?> clazz2 = service.getClass();

// 3. Class.forName()（运行时动态加载）
Class<?> clazz3 = Class.forName("com.example.UserService");
```

**为什么需要三种方式？**
- **类字面量**：编译期安全，性能最好
- **对象.getClass()**：已有对象，想获取其类型
- **Class.forName()**：动态加载，适合框架开发（如Spring的Bean加载）

**Class.forName() vs ClassLoader.loadClass()**

| 维度 | Class.forName() | ClassLoader.loadClass() |
|------|-------------------|--------------------------|
| **初始化** | ✅ 执行`<clinit>()` | ❌ 只加载，不初始化 |
| **使用场景** | 需要执行static块 | 只要加载类，不初始化 |
| **示例** | `Class.forName("com.mysql.jdbc.Driver")` | 自定义类加载器 |

#### 3. 反射调用方法

```java
// 正常调用
UserService service = new UserService();
service.login("zhang", "123456");

// 反射调用
Class<?> clazz = Class.forName("com.example.UserService");
Object obj = clazz.newInstance();  // 创建对象

Method method = clazz.getMethod("login", String.class, String.class);
Object result = method.invoke(obj, "zhang", "123456");
```

**为什么反射调用慢？**
```
正常调用：静态绑定 → 直接跳转到方法地址 → 快
反射调用：运行时查找方法 → 安全检查 → 动态调用 → 慢
```

**优化**：`setAccessible(true)`跳过安全检查

```java
Method method = clazz.getMethod("login", String.class, String.class);
method.setAccessible(true);  // 跳过访问检查 → 性能提升约3倍
Object result = method.invoke(obj, "zhang", "123456");
```

#### 4. 访问私有字段

```java
public class UserService {
    private String password;  // 私有字段

    public UserService(String password) {
        this.password = password;
    }
}

// 反射访问私有字段
UserService service = new UserService("123456");
Field field = UserService.class.getDeclaredField("password");
field.setAccessible(true);  // 强制访问私有字段
String password = (String) field.get(service);  // 获取私有字段值 → "123456"

// 修改私有字段
field.set(service, "newPassword");  // 修改私有字段值
```

**为什么setAccessible(true)能访问私有成员？**
- JVM的访问控制是在**解释器/编译器**层面实现的
- `setAccessible(true)`是**权限覆盖**，告诉JVM跳过访问检查

### 源码关键路径

#### Class.forName()的底层实现

```java
// java.lang.Class
public static Class<?> forName(String className) throws ClassNotFoundException {
    // 1. 获取调用者的类加载器
    ClassLoader caller = Reflection.getCallerClass();

    // 2. 调用forName0（native方法）
    return forName0(className, true, caller, caller);  // true → 初始化
}

// hotspot/src/share/vm/prims/jvm.cpp
JVM_ENTRY(jobject, JVM_FindClassFromClassLoader(JNIEnv* env, jstring name, jboolean init, jobject loader))
    // 3. 委托给ClassLoader.loadClass()
    // 4. 如果init=true，执行<clinit>()（初始化static字段+static块）
JVM_END
```

**为什么forName()会初始化类？**
- JDBC驱动需要在static块中注册：`Class.forName("com.mysql.jdbc.Driver")`
- 如果只加载不初始化，驱动不会生效

#### Method.invoke()的底层实现

```java
// sun.reflect.NativeMethodAccessorImpl
public class NativeMethodAccessorImpl implements MethodAccessor {
    public Object invoke(Object obj, Object[] args) throws IllegalArgumentException, InvocationTargetException {
        // 1. 调用native方法
        return invoke0(obj, args);
    }

    private static native Object invoke0(Object obj, Object[] args);
}

// 2. 当调用次数超过阈值（默认15次），生成字节码访问器
// sun.reflect.MethodAccessorGenerator
public class MethodAccessorGenerator {
    public MethodAccessor generateMethod() {
        // 生成字节码：直接调用目标方法
        // 后续调用不再经过native方法，性能大幅提升
    }
}
```

**为什么invoke()前几次慢，后面快？**
- **前15次**：使用native方法调用（慢）
- **15次后**：生成字节码访问器（快），直接调用目标方法

这是**自适应优化**：热点方法自动优化。

## 它和相似方案的本质区别是什么？

> 对比不是列表，要解释「为什么选 A 不选 B」。

### 反射 vs 直接调用

| 维度 | 反射调用 | 直接调用 |
|------|----------|----------|
| **性能** | 慢（查找、检查、动态调用） | 快（静态绑定） |
| **灵活性** | 高（运行时动态决定） | 低（编译期确定） |
| **安全限制** | 可以访问私有成员 | 只能访问公有成员 |
| **编译期检查** | 无（方法名写错，编译不报错） | 有（写错编译报错） |

**为什么需要反射？**
- **框架开发**：Spring、Hibernate都大量使用反射
- **动态代理**：运行时生成代理类
- **注解处理**：运行时读取注解信息

**为什么不直接调用？**
- **性能更好**，但灵活性不足
- **适合**：性能敏感的代码

### 反射 vs 内省（Introspector）

| 维度 | 反射（Reflection） | 内省（Introspector） |
|------|---------------------|--------------------------|
| **作用** | 获取任意类信息 | 专门处理JavaBean |
| **方法** | `getDeclaredMethods()` | `getBeanInfo()` → `getPropertyDescriptors()` |
| **使用场景** | 通用反射 | 操作JavaBean（getter/setter） |

**为什么需要内省？**
- **简化JavaBean操作**：直接通过属性名获取getter/setter
- **Spring BeanUtils**：基于内省实现

```java
// 反射操作JavaBean（复杂）
Method getter = clazz.getMethod("get" + fieldName.substring(0, 1).toUpperCase() + fieldName.substring(1));

// 内省操作JavaBean（简单）
BeanInfo beanInfo = Introspector.getBeanInfo(clazz);
PropertyDescriptor[] pds = beanInfo.getPropertyDescriptors();
for (PropertyDescriptor pd : pds) {
    if (pd.getName().equals(fieldName)) {
        Method getter = pd.getReadMethod();  // 直接获取getter
    }
}
```

## 正确使用方式

> 不是 API 手册，要解释「为什么这样用是对的」。

### 1. 反射性能优化

```java
// ❌ 错误：每次调用都反射
for (int i = 0; i < 100000; i++) {
    Method method = obj.getClass().getMethod("toString");
    method.invoke(obj);
}

// ✅ 正确：缓存Method对象
Method method = obj.getClass().getMethod("toString");
method.setAccessible(true);  // 跳过访问检查
for (int i = 0; i < 100000; i++) {
    method.invoke(obj);
}
```

**为什么这样用？**
- `getMethod()`每次都要查找方法，性能差
- 缓存`Method`对象，避免重复查找
- `setAccessible(true)`跳过安全检查，性能提升约3倍

### 2. 反射创建对象

```java
// ❌ 错误：使用clazz.newInstance()（JDK 9+ 已废弃）
Object obj = clazz.newInstance();

// ✅ 正确：使用Constructor
Constructor<?> ctor = clazz.getConstructor(String.class, int.class);
Object obj = ctor.newInstance("zhang", 30);
```

**为什么这样用？**
- `clazz.newInstance()`只能调用无参构造器，且已废弃
- `Constructor.newInstance()`更灵活，可以调用有参构造器

### 3. 泛型类型擦除的坑

```java
// 问题：泛型在运行时被擦除
public class GenericClass<T> {
    private T data;

    public void setData(T data) {
        this.data = data;
    }
}

// 反射无法获取泛型类型
Method method = GenericClass.class.getMethod("setData", Object.class);  // T → Object
Type[] paramTypes = method.getGenericParameterTypes();  // 可以获取泛型类型
// paramTypes[0] 是 TypeVariableImpl（T），不是具体类型
```

**解决方案**：通过`Field.getGenericType()`获取泛型类型

```java
public class GenericClass<T> {
    private List<String> list;  // 字段的泛型不会被擦除

    // 可以获取到：List<String>
    public void printGenericType() throws Exception {
        Field field = GenericClass.class.getDeclaredField("list");
        Type genericType = field.getGenericType();  // 获取泛型类型
        System.out.println(genericType);  // java.util.List<java.lang.String>
    }
}
```

### 4. 反射破坏单例模式

```java
// 问题：反射可以调用私有构造器
public class Singleton {
    private static final Singleton INSTANCE = new Singleton();

    private Singleton() {
        // 防止反射破坏
        if (INSTANCE != null) {
            throw new RuntimeException("单例已被破坏！");
        }
    }

    public static Singleton getInstance() {
        return INSTANCE;
    }
}

// 反射破坏单例
Constructor<Singleton> ctor = Singleton.class.getDeclaredConstructor();
ctor.setAccessible(true);  // 强制访问私有构造器
Singleton instance2 = ctor.newInstance();  // 创建第二个实例
```

**解决方案**：
1. 枚举单例（枚举的构造器天然防止反射）
2. 在构造器中检查实例是否已存在

## 边界情况和坑

> 不是列举，要解释「坑的成因」。

### 坑1：Class.forName()默认初始化类

```java
// 问题代码
Class<?> clazz = Class.forName("com.example.MyClass");
// → 会执行static块，可能触发不必要的副作用

// 解决方案：使用ClassLoader.loadClass()
ClassLoader cl = Thread.currentThread().getContextClassLoader();
Class<?> clazz = cl.loadClass("com.example.MyClass");  // 只加载，不初始化
```

**成因**：`Class.forName()`的第二个参数`initialize`默认为`true`，会执行类的static块。

### 坑2：反射调用基本类型方法时的类型转换

```java
// 问题代码
Method method = Integer.class.getMethod("parseInt", String.class);
Integer result = (Integer) method.invoke(null, "123");  // ✅ 可以

// 但如果是反射调用构造函数：
Constructor<Integer> ctor = Integer.class.getConstructor(int.class);
Integer obj = ctor.newInstance(123);  // ✅ 可以

// 坑：基本类型 vs 包装类型
Method method2 = MyClass.class.getMethod("setValue", int.class);
method2.invoke(obj, new Integer(123));  // ✅ 自动拆箱
method2.invoke(obj, new Object[]{123});  // ✅ 也可以
```

**成因**：反射方法调用时，基本类型会自动拆箱/装箱。

### 坑3：setAccessible(true)的安全风险

```java
// 问题代码
public class SecurityRisk {
    private String password = "secret";

    public static void main(String[] args) throws Exception {
        SecurityRisk obj = new SecurityRisk();
        Field field = SecurityRisk.class.getDeclaredField("password");
        field.setAccessible(true);  // 绕过访问控制
        System.out.println(field.get(obj));  // 输出："secret"
    }
}
```

**成因**：`setAccessible(true)`可以访问私有成员，破坏封装性。

**解决方案**：
1. **SecurityManager**：配置安全策略，禁止`setAccessible(true)`
2. **模块系统（JPMS）**：JDK 9+的模块系统可以限制反射访问

### 坑4：反射性能差，不要用在循环中

```java
// 问题代码
for (int i = 0; i < 100000; i++) {
    Method method = obj.getClass().getMethod("toString");  // 每次都反射查找
    method.invoke(obj);
}

// 性能对比：
// - 直接调用：~1ms
// - 反射调用（不缓存）：~200ms
// - 反射调用（缓存+setAccessible）：~5ms
```

**成因**：反射调用需要经过：查找方法 → 安全检查 → 动态调用，开销大。

### 坑5：泛型擦除导致的类型不安全

```java
// 问题代码
List<String> list = new ArrayList<>();
list.add("hello");

// 反射绕过泛型检查
Method addMethod = list.getClass().getMethod("add", Object.class);
addMethod.invoke(list, 123);  // 添加Integer成功！

System.out.println(list.get(1));  // ClassCastException：Integer无法转换为String
```

**成因**：泛型只在**编译期**有效，运行时被擦除为`Object`。

## 我的理解（可选）

> 用自己的话重新表述一遍，检验是否真正理解。

反射的本质是**运行时自我检查与动态调用**。

**核心思想**：
- **动态性**：运行时才能确定类名、方法名
- **灵活性**：可以访问私有成员，动态创建对象
- **性能代价**：查找、检查、动态调用都比直接调用慢

**为什么需要反射？**
- **框架开发**：Spring、Hibernate都大量使用
- **动态代理**：运行时生成代理类
- **注解处理**：运行时读取注解信息

**最容易被忽略的点**：
- `setAccessible(true)`可以大幅提升性能
- `Class.forName()`会初始化类（执行static块）
- 泛型在运行时被擦除
- 反射性能差，不要用在循环中

## 面试话术总结

1. **什么是反射？**
   "反射是Java在运行时动态获取类信息、调用方法、修改字段的机制。核心类：Class（类信息）、Field（字段信息）、Method（方法信息）。"

2. **Class.forName()和ClassLoader.loadClass()的区别？**
   "Class.forName()会初始化类（执行static块），ClassLoader.loadClass()只加载不初始化。JDBC驱动需要用Class.forName()来触发驱动注册。"

3. **反射为什么慢？如何优化？**
   "慢的原因：1）需要查找类/方法/字段；2）需要安全检查；3）动态调用。优化：1）缓存Class/Method/Field对象；2）setAccessible(true)跳过安全检查。"

4. **反射有哪些应用场景？**
   "1）框架开发：Spring的依赖注入、Hibernate的ORM；2）动态代理：Spring AOP、RPC框架；3）注解处理：运行时读取注解信息；4）配置化：根据配置文件动态加载类。"

5. **如何防止反射破坏单例模式？**
   "1）枚举单例：枚举的构造器天然防止反射；2）在构造器中检查实例是否已存在，如果存在则抛出异常。"
