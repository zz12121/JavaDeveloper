# 堆内存结构

> Java对象都在堆上分配，而堆不是一整块乱放的内存——它被精心划分为不同的区域，以支撑高效的垃圾回收。

---

## 这个问题为什么存在？

如果堆只是一大块连续内存，不分代、不分区，那垃圾回收时必须**扫描整块堆**来判断哪些对象还活着。

这会带来两个致命问题：

1. **效率极低**：堆可能几十GB，每次GC都要全堆扫描，STW（Stop-The-World）时间长到应用直接卡死
2. **分配慢**：从几十GB的大堆里找一个合适的位置放新对象，没有结构化组织，分配效率很低

**核心矛盾**：对象寿命差异极大——大部分对象「朝生夕死」，少数对象能活很久。如果用同一套回收策略对待所有对象，就是用大炮打蚊子 + 用水枪打坦克，两头都不讨好。

**解决思路**：分代（Generational）。根据「大部分对象短命」这个经验事实（Weak Generational Hypothesis），把堆分成不同区域，对不同寿命的对象用不同的回收策略。

---

## 它是怎么解决问题的？

### 1. 堆的整体布局

```
┌─────────────────────────────────────────────────────────────┐
│                        Java 堆（Heap）                       │
│                     -Xms / -Xmx 控制                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────────────────────────────────────┐    │
│  │              新生代（Young Generation）                 │    │
│  │               -Xmn 控制大小                            │    │
│  │                                                      │    │
│  │  ┌──────────┐  ┌──────────┐  ┌──────────┐          │    │
│  │  │  Eden    │  │Survivor 0│  │Survivor 1│          │    │
│  │  │  (80%)   │  │  (10%)   │  │  (10%)   │          │    │
│  │  └──────────┘  └──────────┘  └──────────┘          │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐    │
│  │              老年代（Old Generation）                   │    │
│  │               堆 - 新生代                              │    │
│  └─────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

### 2. 为什么这样分？

| 区域 | 存什么 | 回收频率 | 回收算法 | 为什么这样设计 |
|------|--------|---------|---------|---------------|
| **Eden** | 新创建的对象 | 极高（Minor GC） | 复制算法 | 绝大多数对象在这里就死了，直接清掉，效率极高 |
| **Survivor** | 经历过一次GC还活着的对象 | 高（Minor GC） | 复制算法 | 两块 Survivor 轮换（From/To），保证存活对象连续排列 |
| **Old** | 经历多次GC仍然活着的对象 | 低（Major GC） | 标记-清除/标记-整理 | 存活率高，复制代价大，用整理算法减少碎片 |

### 3. 对象的完整生命周期

```
new Object()
    │
    ▼
┌─────────┐
│  Eden   │  ← 绝大多数对象死在这里，Minor GC 直接回收
└────┬────┘
     │ Minor GC 后还活着？
     │ 是
     ▼
┌───────────┐
│ Survivor  │  ← From 和 To 轮换，每活过一次GC，年龄 +1
└────┬──────┘
     │ 年龄达到阈值（默认15，-XX:MaxTenuringThreshold）？
     │ 是
     ▼
┌───────────┐
│    Old    │  ← 进入老年代，Major GC / Full GC 才会回收
└───────────┘
```

### 4. 特殊情况：对象直接进入老年代

不是所有对象都走 Eden → Survivor → Old 的标准流程。以下情况会**直接进入老年代**：

- **大对象**（-XX:PretenureSizeThreshold）：超过指定大小的对象直接分配到老年代，避免在 Eden 和 Survivor 之间来回复制
- **大数组**：同上，`byte[]`、`long[]` 等大数组
- **动态年龄判断**：如果 Survivor 中相同年龄的对象总大小超过 Survivor 空间一半，大于等于该年龄的对象直接进入老年代
- **空间担保失败**：Minor GC 时 Survivor 放不下所有存活对象，直接进老年代

### 5. TLAB（Thread Local Allocation Buffer）

**问题**：Eden 区是所有线程共享的，多个线程同时分配对象会产生竞争，即使不用锁，CAS 操作也有开销。

**解决**：JVM 在 Eden 区内为每个线程预先分配一块私有区域——TLAB。

```
┌─────────────────────────────────────────┐
│                  Eden                    │
├────────┬────────┬────────┬──────────────┤
│Thread1 │Thread2 │Thread3 │   公共空间    │
│ TLAB   │ TLAB   │ TLAB   │  (TLAB耗尽后) │
└────────┴────────┴────────┴──────────────┘
```

- 线程创建对象时优先在自己的 TLAB 内分配，**无竞争、无锁**
- TLAB 耗尽后，再从公共 Eden 区 CAS 分配新的 TLAB
- `-XX:+UseTLAB`（默认开启）、`-XX:TLABSize`（手动指定大小）

## 深入原理

### Survivor 为什么是两块？

只用一块 Survivor 可以吗？不行。复制算法要求「从一块复制到另一块」——如果只有一块，复制后原始数据被覆盖，无法区分哪些是存活对象、哪些是未清理的垃圾。

两块 Survivor 的轮换机制：

```
GC前:  [Eden(满)] [From(有存活对象)] [To(空)]
GC后:  [Eden(空)] [From(空→To)]     [To(存活对象→From)]
```

每次 GC 后，From 和 To 角色互换。这保证了：
1. 存活对象始终紧凑排列（无内存碎片）
2. 清理只需要把 Eden + From 整块清空

### 堆内存参数速查

| 参数 | 含义 | 示例 |
|------|------|------|
| `-Xms` | 堆最小值 | `-Xms512m` |
| `-Xmx` | 堆最大值 | `-Xmx2g` |
| `-Xmn` | 新生代大小 | `-Xmn256m` |
| `-XX:SurvivorRatio` | Eden : Survivor 的比例 | `-XX:SurvivorRatio=8`（Eden:From:To = 8:1:1） |
| `-XX:NewRatio` | Old : Young 的比例 | `-XX:NewRatio=2`（老年代是新生代的2倍） |
| `-XX:MaxTenuringThreshold` | 晋升老年代的年龄阈值 | `-XX:MaxTenuringThreshold=15` |
| `-XX:PretenureSizeThreshold` | 大对象直接进老年代的阈值 | `-XX:PretenureSizeThreshold=1M` |
| `-XX:+UseTLAB` | 启用 TLAB | 默认开启 |
| `-XX:TLABSize` | TLAB 大小 | `-XX:TLABSize=256k` |

**注意**：`-Xmn` 和 `-XX:NewRatio` 不要同时使用，两者控制同一件事，同时设会冲突。生产环境建议只设 `-Xms` 和 `-Xmx`（保持一致），让 JVM 自适应调整各代比例。

### 堆外内存

堆不是 JVM 管理的唯一内存。直接内存（Direct Memory）由 `ByteBuffer.allocateDirect()` 使用，不受堆大小限制，但受 `-XX:MaxDirectMemorySize` 约束（默认与 `-Xmx` 相等）。NIO 的零拷贝、Netty 的 ByteBuf 都大量使用直接内存。

---

## 正确使用方式

**1. 堆大小设置原则**

```
-Xms = -Xmx    ← 两个值设一样，避免运行时动态扩缩容的开销
```

堆太小 → GC 频繁，吞吐量下降
堆太大 → GC 停顿时间长，内存浪费

**2. 根据对象特点调整比例**

- 如果应用有大量短命对象（Web 请求处理），适当增大新生代
- 如果应用有大量长命对象（缓存），老年代需要更多空间

**3. 大对象要警惕**

如果业务中有大数组、大 List，考虑：
- 检查是否真的需要一次性加载到内存
- 使用流式处理替代全量加载
- 调大 `-XX:PretenureSizeThreshold` 避免它们进入 Eden 挤占空间

---

## 边界情况和坑

### 1. 动态年龄计算带来的意外晋升

Survivor 空间不够时，JVM 不是按单个对象年龄判断，而是按**同年龄对象总大小**判断。比如 Survivor 中年龄 5 的对象占了 Survivor 空间的 60%（阈值是 50%），那**所有年龄 ≥ 5 的对象**都会被提前晋升到老年代。

这可能导致老年代过早填满。解决方式：增大 Survivor 大小（调小 `-XX:SurvivorRatio`）。

### 2. Full GC 的连锁反应

Minor GC 时，如果老年代没有足够空间作为「担保」，会直接触发 Full GC。Full GC 扫描整个堆，停顿时间长。避免方式：
- 不要让老年代增长过快（检查长命对象、内存泄漏）
- 确保 `-XX:SurvivorRatio` 合理，让对象能在 Survivor 里多熬几轮

### 3. 内存碎片只在老年代出现

新生代用复制算法，天然无碎片。老年代用标记-清除/标记-整理，可能产生碎片。CMS 用标记-清除所以有碎片问题，G1 用整理所以碎片控制更好。

### 4. `-Xms ≠ -Xmx` 的坑

如果 `-Xms` 小于 `-Xmx`，JVM 在运行时会动态扩容和缩容。每次扩缩容都需要重新调整堆的布局，并可能触发 Full GC。生产环境务必保持两者一致。

### 5. MetaSpace（元空间）不在堆内

类信息、常量池等存在 MetaSpace（方法区的实现），使用本地内存，不在堆内。但 MetaSpace 无上限会导致系统内存被耗尽，需要用 `-XX:MaxMetaspaceSize` 限制。OOM 时不要只盯着堆。
