# 序列化

> 序列化解决的是「Java 对象怎么跨 JVM 边界传输和持久化」的问题。Java 原生序列化用 `Serializable` 标记接口 + `ObjectOutputStream` 实现，但存在安全漏洞、性能差、跨语言兼容性差等问题。实际项目中更常用 JSON、Protobuf 等替代方案。

---

## 这个问题为什么存在？

Java 对象只存在于 JVM 堆内存中，但很多场景需要把对象「搬出去」：
- **网络传输**：RPC 调用时把对象从一台机器发送到另一台
- **持久化存储**：把对象保存到磁盘，下次启动时恢复
- **深拷贝**：通过序列化/反序列化实现对象的完整复制

**核心问题**：Java 对象在内存中的布局是 JVM 实现相关的（不同 JVM 的对象布局不同），不能直接把内存字节写出去。需要一个**与平台无关的格式**来描述对象。

---

## 它是怎么解决问题的？

### Java 原生序列化

```java
// 1. 实现 Serializable 标记接口
public class User implements Serializable {
    private String name;
    private int age;
    // transient 字段不参与序列化
    private transient String password;
    // serialVersionUID 用于版本兼容
    private static final long serialVersionUID = 1L;
}

// 2. 序列化
User user = new User("Alice", 25, "secret");
try (ObjectOutputStream oos = new ObjectOutputStream(
        new FileOutputStream("user.dat"))) {
    oos.writeObject(user);
}

// 3. 反序列化
try (ObjectInputStream ois = new ObjectInputStream(
        new FileInputStream("user.dat"))) {
    User restored = (User) ois.readObject();
    // name="Alice", age=25, password=null（transient 字段丢失）
}
```

**为什么 `Serializable` 是标记接口（没有方法）？**
- JVM 通过运行时检查 `instanceof Serializable` 决定是否允许序列化
- 不需要额外方法，JVM 自己知道怎么处理

**`serialVersionUID` 的作用**：
```
序列化时：JVM 把 serialVersionUID 写入字节流
反序列化时：JVM 对比字节流中的 UID 和类当前的 UID
  → 不匹配 → InvalidClassException

如果不手动指定：
  → JVM 根据类结构（字段、方法、修饰符）自动计算
  → 修改类结构后 UID 变化 → 反序列化旧数据失败

建议：始终手动指定 serialVersionUID
```

### transient 关键字

```java
public class User implements Serializable {
    private String name;
    private transient String password;     // 不参与序列化
    private transient Connection conn;      // 不参与序列化
}
```

**为什么需要 transient？**
- 安全性：密码等敏感字段不应被序列化到文件或网络
- 资源管理：数据库连接、线程池等不可序列化的资源
- 减小体积：大字段或容易重新计算的字段

### 自定义序列化

```java
public class User implements Serializable {
    private String name;
    private int age;
    private transient String password;
    private static final long serialVersionUID = 1L;

    // 自定义序列化逻辑
    private void writeObject(ObjectOutputStream oos) throws IOException {
        oos.defaultWriteObject();  // 先让 JVM 写非 transient 字段
        oos.writeUTF(encrypt(password));  // 手动写 transient 字段（加密后）
    }

    // 自定义反序列化逻辑
    private void readObject(ObjectInputStream ois)
            throws IOException, ClassNotFoundException {
        ois.defaultReadObject();  // 先让 JVM 读非 transient 字段
        this.password = decrypt(ois.readUTF());  // 手动读 transient 字段（解密）
    }
}
```

**JVM 怎么调用 writeObject/readObject？** 通过反射。`ObjectOutputStream.writeObject()` 检查目标类是否有 `writeObject` 方法，如果有就用反射调用，没有就用默认序列化逻辑。

### Externalizable 接口

```java
public class User implements Externalizable {
    private String name;
    private int age;

    // 必须有无参构造器！反序列化时先调用无参构造器
    public User() {}

    @Override
    public void writeExternal(ObjectOutput out) throws IOException {
        out.writeUTF(name);
        out.writeInt(age);
    }

    @Override
    public void readExternal(ObjectInput in) throws IOException {
        this.name = in.readUTF();
        this.age = in.readInt();
    }
}
```

**Externalizable vs Serializable**：

| 维度 | Serializable | Externalizable |
|------|-------------|----------------|
| 控制粒度 | 默认序列化 + 可选自定义 | 完全手动控制 |
| 性能 | 默认序列化使用反射，较慢 | 手动控制，更快 |
| 灵活性 | transient 字段不序列化 | 完全自定义哪些字段序列化 |
| 兼容性 | 添加字段后旧数据仍可反序列化（UID 不变时）| 结构变化后通常不兼容 |
| 要求 | 无 | 必须有无参构造器 |

## 深入原理

### 序列化字节流的结构

```
序列化后的字节流结构：
  AC ED 00 05  — 魔数（STREAM_MAGIC）+ 版本号（STREAM_VERSION）
  73          — TC_OBJECT（对象标记）
  72          — TC_CLASSDESC（类描述符）
  00 0A       — 类名长度（10）
  63 6F 6D... — 类名（com.example.User）
  ...         — serialVersionUID
  ...         — 字段描述
  78          — TC_ENDBLOCKDATA
  70          — TC_NULL（无父类）
  ...         — 对象数据
```

`ObjectOutputStream` 把对象的信息（类名、UID、字段描述、字段值）按固定格式写入字节流，`ObjectInputStream` 按相同格式解析。

### 序列化与 ClassLoader

```java
// 反序列化时需要 Class 对象
User user = (User) ois.readObject();
// JVM 通过当前线程的 ClassLoader 加载 User 类
// 如果序列化和反序列化在不同 JVM 中，ClassLoader 必须能找到 User 类
```

**跨 JVM 序列化的前提**：两端的类定义一致（相同的 serialVersionUID、兼容的字段结构）。

### Java 原生序列化的安全问题

```java
// 反序列化可以执行任意代码！
// 攻击者构造恶意的序列化字节流，反序列化时触发 RCE

// 漏洞原理：
// 1. 某些类的 readObject() 方法中执行了危险操作
// 2. 攻击者构造包含这些类的序列化数据
// 3. 目标程序反序列化时，自动调用 readObject()
// 4. readObject() 中的代码执行了攻击者的指令

// 历史 RCE 漏洞：
// - Apache Commons Collections 的 InvokerTransformer
// - Spring Framework 的各种序列化漏洞
// - JDK 自身的 AnnotationInvocationHandler
```

**防御措施**：
- 不信任的数据不要用 Java 原生序列化
- 使用 `ObjectInputFilter`（Java 9+）限制反序列化的类
- 使用白名单，只允许特定类被反序列化

## 正确使用方式

### 序列化方案的选择

| 方案 | 人类可读 | 跨语言 | 性能 | 体积 | 适用场景 |
|------|---------|--------|------|------|---------|
| Java 原生 | ❌ | ❌ | 中 | 中 | Java 内部使用 |
| JSON | ✅ | ✅ | 慢 | 大 | REST API、配置文件 |
| Protobuf | ❌ | ✅ | 快 | 小 | gRPC、高性能 RPC |
| Hessian | ❌ | 部分 | 中 | 中 | Dubbo 默认序列化 |
| Kryo | ❌ | ❌ | 很快 | 小 | Java 高性能序列化 |

### serialVersionUID 的最佳实践

```java
// ❌ 不指定 serialVersionUID
public class User implements Serializable {
    // JVM 自动生成 → 修改类后 UID 变化 → 旧数据无法反序列化
}

// ✅ 手动指定
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    // UID 不变 → 添加字段后旧数据仍可反序列化（新字段为默认值）
}
```

### 反序列化的单例陷阱

```java
// 序列化会破坏单例！
public class Singleton implements Serializable {
    private static final Singleton INSTANCE = new Singleton();
    private Singleton() {}
    public static Singleton getInstance() { return INSTANCE; }
}

// 反序列化时，JVM 会创建新对象
Singleton s1 = Singleton.getInstance();
byte[] data = serialize(s1);
Singleton s2 = deserialize(data);
System.out.println(s1 == s2);  // false！破坏了单例
```

**解决方案**：实现 `readResolve()` 方法：
```java
public class Singleton implements Serializable {
    private static final Singleton INSTANCE = new Singleton();
    private Singleton() {}
    public static Singleton getInstance() { return INSTANCE; }

    // 反序列化时用这个方法替代默认的 new
    protected Object readResolve() {
        return getInstance();
    }
}
```

## 边界情况和坑

### 坑 1：修改类导致反序列化失败

```java
// v1：原始类
public class Config implements Serializable {
    private static final long serialVersionUID = 1L;
    private String host;
    private int port;
}

// v2：添加了新字段
public class Config implements Serializable {
    private static final long serialVersionUID = 1L;  // 保持不变！
    private String host;
    private int port;
    private boolean ssl;  // 新字段
}
// 只要 serialVersionUID 不变，v1 的数据可以被 v2 反序列化
// 新字段 ssl = false（默认值）
```

**如果 serialVersionUID 改了？** → `InvalidClassException`。添加字段不会自动改变 UID，但修改字段类型、删除字段、改变修饰符都会。

### 坑 2：父类未实现 Serializable

```java
public class Animal {
    protected String name;  // 不参与序列化！
}

public class Dog extends Animal implements Serializable {
    private int age;  // 参与序列化
}
// 反序列化 Dog 时，name 不会被恢复
// 因为 Animal 没有实现 Serializable
// Dog 反序列化后 name = null
```

**成因**：只有实现了 `Serializable` 的类及其超类中的字段才会被序列化。

**解决方案**：父类也实现 `Serializable`，或提供 `writeObject/readObject` 手动处理父类字段。

### 坑 3：集合中存入不同类型的对象

```java
List<Object> list = new ArrayList<>();
list.add("hello");
list.add(42);
list.add(new User("Alice", 25));

// 序列化后反序列化
List<Object> restored = (List<Object>) deserialize(data);
String s = (String) restored.get(0);  // ✅
Integer i = (Integer) restored.get(1); // ✅
User u = (User) restored.get(2);      // ✅

// 如果 User 类被删除了，get(2) 会抛 ClassNotFoundException
```

**成因**：序列化字节流中保存了每个对象的类名。反序列化时必须能找到对应的类。

### 坑 4：static 字段不参与序列化

```java
public class Config implements Serializable {
    private static final long serialVersionUID = 1L;
    private static int count = 0;  // 不参与序列化！
    private String name;
}
```

**成因**：`static` 字段属于类，不属于对象。序列化的是对象状态，不包括类状态。

### 坑 5：循环引用

```java
public class Node implements Serializable {
    String value;
    Node next;  // 循环引用：A.next = B, B.next = A

    Node a = new Node("A");
    Node b = new Node("B");
    a.next = b;
    b.next = a;
}
```

**Java 原生序列化能正确处理循环引用**：`ObjectOutputStream` 维护了一个引用表（identity hash table），遇到已序列化的对象时写入引用编号，而不是重新序列化。

但 JSON 序列化（如 Jackson 默认配置）**不能处理循环引用**，会抛异常或导致无限递归。需要用 `@JsonIdentityInfo` 注解处理。
