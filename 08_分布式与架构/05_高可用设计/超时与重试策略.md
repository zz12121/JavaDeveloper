# 超时与重试策略

## 这个问题为什么存在？

> 分布式环境下，网络是不可靠的。调用超时、网络抖动、服务临时不可用都是常态。问题是：**怎么容忍这些临时故障，又不导致资源耗尽？**

没有超时与重试策略，你会遇到：
- 服务A调用服务B，服务B慢，所有线程都阻塞 → 服务A线程池耗尽
- 网络抖动导致一次调用失败，直接抛异常 → 用户体验差
- 重试无策略，所有请求同时重试 → 重试风暴，把下游打挂

---

## 它是怎么解决问题的？

### 超时控制：释放资源，快速失败

```
没有超时：
  A → B（慢） → 线程阻塞 → 线程池耗尽 → A也挂了

有超时：
  A → B（慢） → 3秒超时 → 线程释放 → 快速失败 → 不影响其他请求
```

```java
// ❌ 错误：没有超时设置
String result = restTemplate.getForObject(url, String.class);

// ✅ 正确：设连接超时和读取超时
RestTemplate restTemplate = new RestTemplate();
HttpComponentsClientHttpRequestFactory factory = new HttpComponentsClientHttpRequestFactory();
factory.setConnectTimeout(3000);  // 连接超时3秒
factory.setReadTimeout(5000);     // 读取超时5秒
restTemplate.setRequestFactory(factory);
```

**超时设多少？**
```
推荐：超时时间 = P99响应时间 × 2
  - P99 = 95%的请求在200ms内完成
  - 超时设500ms，既能容忍波动，又不至于长期阻塞
```

### 重试策略：容忍临时故障，但要有策略

#### 策略1：指数退避（Exponential Backoff）

```
第1次重试：等待1秒
第2次重试：等待2秒
第3次重试：等待4秒
第4次重试：等待8秒
  → 每次等待时间翻倍，避免把下游打挂
```

```java
// 指数退避重试
public class RetryTemplate {
    public <T> T execute(Callable<T> task, int maxRetries) throws Exception {
        long backoff = 1000;  // 初始退避1秒
        for (int i = 0; i < maxRetries; i++) {
            try {
                return task.call();
            } catch (Exception e) {
                if (i == maxRetries - 1) throw e;
                Thread.sleep(backoff);
                backoff *= 2;  // 指数增长：1s → 2s → 4s
            }
        }
        return null;
    }
}
```

#### 策略2：重试+熔断组合

```
只有重试 → 下游挂了，一直重试 → 把下游彻底打挂
重试+熔断 → 下游挂了，熔断器打开 → 不再重试，快速失败
```

```java
// Resilience4j：重试 + 熔断 组合
CircuitBreaker circuitBreaker = CircuitBreaker.ofDefaults("payment");
Retry retry = Retry.ofDefaults("payment");

// 先重试，再熔断
Supplier<String> decorated = Decorators.ofSupplier(() -> paymentService.call())
        .withRetry(retry)
        .withCircuitBreaker(circuitBreaker)
        .decorate();
```

#### 策略3：只重试幂等接口

```
❌ 错误：对非幂等接口重试（如"扣款"接口，重试会扣两次）
✅ 正确：只对幂等接口重试（如"查询"、"更新状态"）
```

## 深入原理

| | 超时控制 | 重试策略 | 熔断 |
|---|---|---|---|
| 目的 | 释放资源，快速失败 | 容忍临时故障 | 防止下游挂了还一直调用 |
| 触发条件 | 调用超过设定时间 | 调用失败（异常） | 失败率超阈值 |
| 处理方式 | 抛超时异常 | 等待一段时间后重试 | 快速失败，不发起调用 |
| 适用场景 | 所有远程调用 | 幂等接口，临时故障 | 下游故障 |

**本质区别**：
- **超时**：防止资源耗尽（线程池、连接池）
- **重试**：容忍临时故障（网络抖动、服务临时不可用）
- **熔断**：下游挂了，别再调了（防止雪崩）

## 正确使用方式

### Spring Cloud：Feign + 超时 + 重试

```yaml
feign:
  client:
    config:
      default:
        connectTimeout: 3000   # 连接超时3秒
        readTimeout: 5000      # 读取超时5秒
        loggerLevel: basic

# Hystrix（或Resilience4j）配置熔断
hystrix:
  command:
    default:
      execution:
        isolation:
          thread:
            timeoutInMilliseconds: 10000  # 熔断超时10秒（> Feign超时）
```

### Resilience4j：重试 + 熔断 + 超时

```java
// 1. 配置重试
RetryConfig retryConfig = RetryConfig.custom()
        .maxAttempts(3)           // 最多重试3次
        .waitDuration(Duration.ofMillis(1000))  // 初始等待1秒
        .exponentialBackoffMultiplier(2.0)    // 指数退避，翻倍
        .build();
Retry retry = Retry.of("payment", retryConfig);

// 2. 配置熔断
CircuitBreakerConfig cbConfig = CircuitBreakerConfig.custom()
        .failureRateThreshold(50)  // 失败率50%就熔断
        .waitDurationInOpenState(Duration.ofMillis(10000))  // 熔断10秒
        .build();
CircuitBreaker cb = CircuitBreaker.of("payment", cbConfig);

// 3. 组合使用
Supplier<String> decorated = Decorators.ofSupplier(() -> paymentService.call())
        .withRetry(retry)
        .withCircuitBreaker(cb)
        .decorate();
```

## 边界情况和坑

### 坑1：重试风暴（Retry Storm）

```
场景：订单服务有3个实例，每个都配了重试3次
      支付服务挂了
      结果：3个实例 × 3次重试 = 9次调用 → 支付服务更挂了
```

**解决方案**：
1. **重试 + 熔断必须组合**（熔断打开了，就不重试了）
2. **重试次数 × 实例数 < 下游承受极限**
3. **指数退避 + 随机抖动**（不把重试请求同时发出去）

### 坑2：超时设太短，所有请求都超时

```
场景：P99响应时间是500ms，超时设300ms
结果：95%的请求都超时失败
```

**解决方案**：
1. **超时 = P99 × 2**（留有余地）
2. **监控P99、P999响应时间**（动态调整超时）
3. **区分"连接超时"和"读取超时"**（连接超时设短一点，读取超时设长一点）

### 坑3：非幂等接口重试，导致重复操作

```
场景：支付接口，第一次调用成功但响应超时
      重试 → 又扣了一次钱
```

**解决方案**：
1. **只对幂等接口重试**（查询、更新状态）
2. **非幂等接口，用幂等机制**（Token、唯一约束）
3. **重试前检查业务状态**（如果已支付，就不重试）
