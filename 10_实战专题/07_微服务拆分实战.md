# 微服务拆分实战

> 拆分不是目的，**解决业务复杂度、团队协同效率、系统稳定性**才是。拆得好是微服务，拆不好是分布式单体——所有服务加一起比单体还慢。

---

## 拆分的判断标准

```
什么时候拆：
  ✅ 单体应用代码量 > 50 万行
  ✅ 团队人数 > 20 人（2 pizza team）
  ✅ 不同模块的变更频率差异大
  ✅ 需要独立扩缩容不同模块
  ✅ 技术栈差异大（Python + Java + Go）

什么时候不拆：
  ❌ 系统 QPS < 1000
  ❌ 团队 < 10 人
  ❌ 业务还在快速迭代，边界不清楚
  ❌ 没有微服务治理经验
```

---

## 场景 A：如何确定拆分边界

### 常见错误

```
❌ 按技术层次拆分
  → 用户服务、订单服务、库存服务、支付服务
  → 问题：跨服务调用极其频繁，一个订单流程要调 5-6 个服务

❌ 按数据库表拆分
  → 每个表一个服务
  → 问题：关系型数据被强行拆散，JOIN 变成跨服务调用
```

### 正确思路：领域驱动设计（DDD）

```
拆分边界 = 业务边界（高内聚） + 数据边界（独立数据库）

步骤：
1. 梳理业务流程（用户下单完整链路）
2. 找出核心实体和领域事件
3. 按「高内聚、低耦合」划分限界上下文
4. 确定每个上下文对外暴露的能力（API）
```

```
DDD 拆分示例（电商）：
┌──────────────────────────────────────────────────────┐
│  商品域（Product）                                     │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐               │
│  │ 商品服务 │  │ 库存服务 │  │ 分类服务 │               │
│  └────┬────┘  └────┬────┘  └────┬────┘               │
│       │            │            │                     │
│  ┌────▼────────────▼────────────▼────┐              │
│  │     ProductDB（共享库/独立库均可）  │              │
│  └────────────────────────────────────┘              │
└──────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────┐
│  交易域（Trade）                                       │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐               │
│  │ 订单服务 │  │ 促销服务 │  │ 支付服务 │               │
│  └─────────┘  └─────────┘  └─────────┘               │
└──────────────────────────────────────────────────────┘
```

## 场景 B：服务间通信方案

### 同步通信：HTTP / gRPC

```java
// HTTP（Spring RestTemplate / OpenFeign）
@Service
public class OrderService {
    @Autowired
    private RestTemplate restTemplate;
    
    public Order createOrder(Long userId, Long productId) {
        // 调用用户服务
        User user = restTemplate.getForObject(
            "http://user-service/users/{id}", User.class, userId);
        
        // 调用库存服务
        Stock stock = restTemplate.postForObject(
            "http://stock-service/stock/deduct",
            new DeductRequest(productId, 1), Stock.class);
        
        return orderRepository.save(new Order(user, stock));
    }
}
```

```protobuf
// gRPC（适合高性能场景）
service ProductService {
    rpc GetProduct(GetProductRequest) returns (Product);
    rpc ListProducts(ListProductsRequest) returns (stream Product);
}

// proto 文件定义后，生成各语言客户端
```

### 异步通信：消息队列

```java
// 下单成功后，异步通知其他服务
@Service
public class OrderService {
    @Autowired
    private RocketMQTemplate mq;
    
    public void createOrder(Order order) {
        orderRepository.save(order);
        
        // 发送消息，不阻塞
        mq.convertAndSend("order:topic", "create", order);
        // 库存服务、用户服务、物流服务各自订阅处理
    }
}

// 库存服务订阅
@Component
@RocketMQMessageListener(topic = "order:topic", 
                        selectorExpression = "create",
                        consumerGroup = "stock-consumer")
public class StockConsumer {
    public void consume(Order order) {
        stockService.deduct(order.getProductId(), order.getQuantity());
    }
}
```

**通信方案对比**：

| 方案 | 延迟 | 一致性 | 适用场景 |
|------|------|--------|---------|
| HTTP/REST | 低 | 强一致 | 实时查询、简单调用 |
| gRPC | 最低 | 强一致 | 高性能、内部通信 |
| Dubbo RPC | 低 | 强一致 | Java 生态、高性能 |
| Kafka | 毫秒级 | 最终一致 | 异步解耦、事件流 |
| RocketMQ | 毫秒级 | 最终一致 | 可靠消息、事务消息 |

## 场景 C：分布式事务问题

### 问题

```
订单服务扣用户余额，库存服务扣商品库存
A 成功，B 失败 → 数据不一致
```

### 解决方案 1：Saga 模式（补偿事务）

```java
// 订单创建 = 一系列本地事务的序列
// 每个步骤都有对应的补偿（回滚）操作
public void createOrderSaga(Order order) {
    try {
        // Step 1: 锁定库存（Compensable）
        stockService.reserve(order.getItems());
        
        // Step 2: 扣减余额（Compensable）
        accountService.deductBalance(order.getUserId(), order.getAmount());
        
        // Step 3: 创建订单（Compensable）
        Order savedOrder = orderRepository.save(order);
        
        // Step 4: 发送通知（Compensable）
        notificationService.send(savedOrder);
        
    } catch (Exception e) {
        // 补偿：从后往前依次回滚
        notificationService.cancel(order.getId());
        accountService.refundBalance(order.getUserId(), order.getAmount());
        stockService.release(order.getItems());
    }
}
```

### 解决方案 2：可靠消息最终一致

```java
// RocketMQ 事务消息
@Service
public class OrderService {
    @Autowired
    private RocketMQTemplate mq;
    
    public void createOrder(Order order) {
        // 发送半消息（此时消费者看不到）
        mq.sendMessageInTransaction("order:topic", "create", order, 
            // 事务执行器
            (msg, arg) -> {
                orderRepository.save(order);  // 本地事务
                stockService.deduct(order.getProductId());  // 外部调用
                return MessageBuilder.withPayload(order).build();
            }, order);
    }
}
```

## 场景 D：服务治理

### 核心能力

```
服务治理 = 注册发现 + 负载均衡 + 限流熔断 + 链路追踪 + 配置管理
```

```yaml
# Spring Cloud Gateway 统一入口
spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://user-service  # 负载均衡调用
          predicates:
            - Path=/api/user/**
        - id: order-service
          uri: lb://order-service
          predicates:
            - Path=/api/order/**
      
      # 全局限流
      default-filters:
        - name: RequestRateLimiter
          args:
            redis-rate-limiter.replenishRate: 100
            redis-rate-limiter.burstCapacity: 200

# Sentinel 熔断降级
sentinel:
  rules:
    flow:
      - resource: order-service
        grade: 1
        count: 100
        controlBehavior: 2  # Warm Up
        clusterMode: true
    degrade:
      - resource: order-service
        grade: 2  # 异常比例熔断
        count: 0.5
        timeWindow: 10
```

## 拆分后的挑战

```
1. 运维复杂度倍增
   → 需要：容器化（Docker/K8s）+ 自动化部署 + 监控告警

2. 分布式调试困难
   → 需要：全链路追踪（SkyWalking/Jaeger）+ 统一日志

3. 接口版本管理
   → 需要：API 版本号 + 向下兼容 + 灰度发布

4. 测试成本上升
   → 需要：契约测试（Consumer-Driven Contract）+ 集成测试平台

5. 数据一致性问题
   → 需要：Saga/Seata + 补偿机制 + 幂等设计
```

## 追问链

### 追问 1：微服务 vs 模块化，怎么选？

> "如果团队 < 15 人，系统复杂度可控，**先用模块化**（Maven 多模块项目，物理分离代码但共享进程）。微服务需要：独立的数据库、独立的部署、独立的技术栈——代价很大。正确的路径是：单体 → 模块化 → 边界清晰后 → 微服务。不要为了微服务而微服务。"

### 追问 2：服务拆分了，但调用链很深怎么办？

> "调用链 > 5 层就该警惕了。解决方式：1. 聚合服务层（BFF）对外提供粗粒度接口。2. 避免服务间循环依赖。3. 关键路径用异步消息解耦。4. 如果调用链确实深，确保每层都有超时和熔断。"

### 追问 3：数据库怎么拆分？

> "三种方式：1. 按业务拆分（每个服务独立 DB，最干净）。2. 按表拆分（分库分表，大表拆成多个）。3. 共享库（同个 DB 多个服务访问，不推荐但最简单）。起步阶段建议方案 1，简单直接。"

## 排查 Checklist

```
□ 拆分边界清晰吗？ → 领域驱动设计，不要按表拆分
□ 服务间通信选型合理吗？ → 同步 vs 异步，按场景选择
□ 有分布式事务问题吗？ → Saga / 可靠消息
□ 服务治理完整吗？ → 注册发现 + 限流 + 熔断 + 链路追踪
□ 接口版本管理了吗？ → API 向下兼容
□ 有全链路监控吗？ → SkyWalking / Jaeger
□ 调用链深度合理吗？ → >5 层需警惕
□ 数据库拆分了吗？ → 按业务域独立 DB
□ 有服务契约测试吗？ → 避免接口不兼容
□ 团队准备好了吗？ → DevOps + 自动化部署
```
