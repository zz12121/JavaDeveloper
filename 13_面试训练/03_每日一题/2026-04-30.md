### 一、@Transactional 的工作原理

Spring 事务基于 **AOP 动态代理**实现，核心链路：

```
@Transactional 方法
    ↓ 调用时
代理对象（JDK Proxy / CGLIB）
    ↓ 拦截
TransactionInterceptor.invoke()
    ↓
1. 解析注解属性（传播行为、隔离级别、rollbackFor...）
2. 开启事务 → DataSourceTransactionManager.doBegin()
   ├─ 从 DataSource 获取 Connection
   ├─ connection.setAutoCommit(false)
   └─ 把连接绑定到 ThreadLocal（TransactionSynchronizationManager）
3. 执行业务方法
4. 正常返回 → connection.commit()
   抛出匹配异常 → connection.rollback()
5. 清理 ThreadLocal，恢复 autoCommit
```

**ThreadLocal 绑定连接是核心**：同一个线程中的所有 DAO 都通过 `DataSourceUtils.getConnection(dataSource)` 拿连接，由于 ThreadLocal 的绑定，拿到的是**同一个 Connection**，这才能保证原子性。

### 二、事务失效的 5 大场景

#### 1. 自调用（最高频 ⚠️）

```java
@Service
public class UserService {
    public void createUser() {
        this.insert();  // ❌ this 指向原始对象，不走代理，事务失效
    }

    @Transactional
    public void insert() { ... }
}
```

**原因**：Spring 事务通过 AOP 代理拦截，`this.xxx()` 绕过了代理对象。

**解决**：
- 拆到另一个 Service（推荐）
- `AopContext.currentProxy()` 获取代理（需配置 `exposeProxy=true`）

---

#### 2. 非 public 方法

```java
@Transactional  // ❌ protected/private 方法，Spring 不代理
protected void insert() { ... }
```

Spring AOP 只代理 **public 方法**，非 public 的 `@Transactional` 静默失效。

---

#### 3. 异常被 catch 吞掉

```java
@Transactional(rollbackFor = Exception.class)
public void create() {
    try {
        dao.insert();
    } catch (Exception e) {
        log.error("error", e);  // ❌ 异常被吞，Spring 感知不到，不会回滚
    }
}
```

**解决**：catch 后重新抛出，或手动设置回滚标记：

```java
TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
```

---

#### 4. 默认只回滚 RuntimeException（受检异常陷阱）

```java
@Transactional  // ❌ 默认不回滚受检异常
public void create() throws IOException {
    dao.insert();
    throw new IOException("文件读取失败");  // 不会回滚！
}

// ✅ 正确：明确指定
@Transactional(rollbackFor = Exception.class)
```

Spring 默认规则：**RuntimeException + Error 才回滚**，受检异常（Exception）默认不回滚。

---

#### 5. 多线程下事务不传播

```java
@Transactional
public void create() {
    new Thread(() -> {
        userDao.insert();  // ❌ 新线程拿不到父线程的 ThreadLocal 连接
    }).start();
}
```

Spring 事务绑定在 ThreadLocal 上，**新线程 = 新 ThreadLocal = 以非事务方式执行**。

### 三、传播行为核心对比（高频追问）

| 传播行为 | 行为 | 典型场景 |
|----------|------|----------|
| **REQUIRED**（默认） | 有事务加入，无则新建 | 普通业务方法 |
| **REQUIRES_NEW** | 挂起当前事务，新建独立事务 | 审计日志（不受主事务影响） |
| **NESTED** | 嵌套事务（JDBC Savepoint） | 部分回滚（内层失败不影响外层） |
| **SUPPORTS** | 有事务加入，无事务以非事务执行 | 查询方法 |

**REQUIRES_NEW vs NESTED 本质区别**：

```
REQUIRES_NEW：两个独立事务，B 提交后 A 回滚不影响 B
NESTED：B 是 A 的一部分，A 回滚会把 B 一起回滚（Savepoint 只允许回滚 B 单独失败的场景）
```
