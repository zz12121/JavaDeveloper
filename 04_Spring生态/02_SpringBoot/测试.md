# Spring Boot 测试

> Spring Boot 提供了完整的测试支持——从单元测试到集成测试，从 Web 层到数据访问层，理解 `@SpringBootTest` 和各种切片测试的适用场景，是写出高质量测试的关键。

---

## 这个问题为什么存在？

没有测试框架支持时，验证 Spring Boot 应用的正确性需要启动整个应用、手动发送 HTTP 请求、查看数据库结果——这不是自动化测试，这是手工验证。

Spring Boot 测试体系要解决的核心矛盾是：**Spring 应用依赖 IoC 容器（Bean 的创建和依赖注入），脱离了容器，业务代码根本跑不起来**。

```java
// 没有 Spring 测试的困境
UserService userService = new UserServiceImpl();  // 直接 new
// 问题：userService 里的 @Autowired 字段全是 null！
// 因为没经过 Spring 容器初始化

// 解决方式一：启动完整 Spring 容器（慢，但真实）
// 解决方式二：用 Mock 模拟依赖（快，但可能遗漏集成问题）
// Spring Boot 测试框架提供了这两种方式之间的精细控制
```

---

## 它是怎么解决问题的？

### @SpringBootTest：完整容器测试

```java
@SpringBootTest
class UserServiceTest {

    @Autowired
    private UserService userService;

    @Test
    void testCreateUser() {
        User user = userService.createUser("张三", "123456");
        assertNotNull(user.getId());
    }
}
```

`@SpringBootTest` 会启动完整的 Spring 应用上下文（和正常启动应用一样加载 Bean、配置、自动配置），然后在测试中注入需要的 Bean。

**代价**：每次运行都要启动整个 Spring 容器，如果有 Web 环境还要启动内嵌的 Tomcat——单个测试类可能要 10~30 秒。

### 切片测试：只加载相关配置

```java
// Web 层切片：只加载 Web 相关 Bean（Controller、Filter、Interceptor）
@WebMvcTest(UserController.class)   // 只加载 UserController 相关的 Bean
class UserControllerTest {

    @Autowired
    private MockMvc mockMvc;              // 模拟 HTTP 请求，不启动真实 Tomcat

    @MockBean
    private UserService userService;        // UserService 需要 Mock（因为没加载 Service 层 Bean）

    @Test
    void testGetUser() throws Exception {
        when(userService.getUserById(1L))
            .thenReturn(new User(1L, "张三"));

        mockMvc.perform(get("/users/1"))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.name").value("张三"));
    }
}

// 数据层切片：只加载 JPA/MyBatis 相关 Bean
@DataJpaTest                         // 只加载 JPA 相关 Bean，用内存数据库
class UserRepositoryTest { ... }

@MybatisTest                         // 只加载 MyBatis 相关 Bean
class UserMapperTest { ... }

// 其他切片
@JsonTest                            // 只加载 JSON 序列化/反序列化相关 Bean
@RestClientTest                      // 只加载 RestTemplate 相关 Bean
```

切片测试的核心优势：**启动极快**（只加载相关 Bean），且**隔离性好**（不依赖其他层的实现）。代价是需要用 `@MockBean` 模拟被切片排除的层。

### MockMvc vs WebTestClient

```java
// MockMvc（Servlet 栈，Spring MVC）
@WebMvcTest
class UserControllerTest {
    @Autowired MockMvc mockMvc;

    @Test
    void testGetUser() throws Exception {
        mockMvc.perform(get("/users/{id}", 1L))
            .andExpect(status().isOk());
    }
}

// WebTestClient（Reactive 栈，Spring WebFlux）
@WebFluxTest
class UserControllerTest {
    @Autowired WebTestClient webTestClient;

    @Test
    void testGetUser() {
        webTestClient.get().uri("/users/{id}", 1L)
            .exchange()
            .expectStatus().isOk();
    }
}
```

## 深入原理

### @MockBean 的工作原理

```java
@WebMvcTest(UserController.class)
class UserControllerTest {

    @MockBean               // ← 关键注解
    private UserService userService;

    // 没有 @MockBean：Spring 启动时会报错
    // 因为 @WebMvcTest 不会加载 UserService 的 Bean 定义
    // No qualifying bean of type 'UserService' available

    // 有 @MockBean：Mockito 创建一个 UserService 的 Mock 对象
    // 注册到 Spring 容器，替换真正的 UserService Bean
}
```

`@MockBean` 通过 `MockitoPostProcessor` 在 Spring 容器初始化时介入，用 Mockito 创建一个指定类型的 Mock 对象，并将其注册为 Spring Bean。后续测试中用 `when().thenReturn()` 控制这个 Mock 对象的行为。

**为什么不用 @Mock？** `@Mock`（Mockito 原生注解）创建的 Mock 对象不在 Spring 容器中——`@Autowired` 注入时找不到这个对象。`@MockBean` 是 Spring Boot 对 Mockito 的扩展，确保 Mock 对象被注册到 Spring 容器。

### 测试容器的自动配置

```java
// @DataJpaTest 默认行为：
// 1. 自动配置内存数据库（H2）
// 2. 只加载 JPA 相关 Bean（EntityManager、Repository）
// 3. 自动回滚事务（每个测试方法结束后数据不污染）
```

```java
@DataJpaTest
class UserRepositoryTest {

    @Autowired
    private TestEntityManager entityManager;   // 专为测试设计的 EntityManager 包装

    @Autowired
    private UserRepository userRepository;

    @Test
    void testFindByEmail() {
        entityManager.persist(new User("张三", "zhangsan@test.com"));
        User found = userRepository.findByEmail("zhangsan@test.com");
        assertEquals("张三", found.getName());
    }
}
```

`TestEntityManager` 提供了比标准 `EntityManager` 更简洁的 API（比如 `persistAndFlush()`），并且和 `@DataJpaTest` 的事务自动回滚机制配合良好。

## 正确使用方式

### 标准测试分层策略

```
测试金字塔：
  ┌─────────────────┐
  │   端到端测试     │  ← @SpringBootTest（完整容器，最少）
  │  （E2E Test）   │
  ├─────────────────┤
  │   集成测试       │  ← @WebMvcTest / @DataJpaTest（切片，适量）
  │  （Integration）  │
  ├─────────────────┤
  │   单元测试       │  ← Mockito（无 Spring，最多）
  │  （Unit Test）   │
  └─────────────────┘
```

**单元测试（无 Spring）**：用 Mockito 模拟所有依赖，只测试单个类的方法逻辑，不启动 Spring 容器——速度最快（毫秒级）。

```java
// 纯单元测试：不启动 Spring
class UserServiceTest {

    @Mock
    private UserRepository userRepository;

    @InjectMocks
    private UserServiceImpl userService;

    @Test
    void testGetUserById() {
        when(userRepository.findById(1L))
            .thenReturn(Optional.of(new User(1L, "张三")));
        User user = userService.getUserById(1L);
        assertEquals("张三", user.getName());
    }
}
```

**集成测试（Spring 切片）**：测试 Controller 和 Service 的集成，或 Service 和 Repository 的集成，用切片测试精确控制加载范围。

**端到端测试（完整容器）**：验证整个应用的启动和配置是否正确，覆盖配置文件加载、自动配置生效、Bean 依赖注入等——运行最慢，只跑关键路径。

### 测试配置文件

```yaml
# src/test/resources/application-test.yml
spring:
  datasource:
    url: jdbc:h2:mem:testdb;DB_CLOSE_DELAY=-1;DB_CLOSE_ON_EXIT=FALSE
    driver-class-name: org.h2.Driver
    username: sa
    password:
  jpa:
    hibernate:
      ddl-auto: create-drop
    show-sql: true
```

```java
@SpringBootTest(properties = "spring.profiles.active=test")
// 或
@ActiveProfiles("test")   // 激活 test profile
```

测试环境用内存数据库（H2）替代真实数据库，好处是：测试独立（不依赖外部数据库）、速度快、每次测试后数据自动清理。

## 边界情况和坑

### @Transactional 在测试中的特殊行为

```java
@SpringBootTest
class UserServiceTest {

    @Autowired
    private UserService userService;

    @Test
    @Transactional        // ← 测试方法加 @Transactional
    void testCreateUser() {
        User user = userService.createUser("张三", "123456");
        // 断言...

        // 测试结束后，事务自动回滚！
        // 数据库中没有这条记录（不会影响其他测试）
    }
}
```

**坑**：`@Transactional` 在测试中是**自动回滚**的（和正式代码中的行为不同）。如果想保留测试数据（比如调试时查看数据库），加 `@Commit` 注解：

```java
@Test
@Transactional
@Commit                 // 测试结束后提交事务，不回滚
void testCreateUserAndCommit() { ... }
```

### WebTestClient 的"延迟"断言

```java
// ❌ 错误：直接断言（断言在响应到达前执行）
webTestClient.get().uri("/users/{id}", 1L)
    .exchange()
    .expectStatus().isOk();
User user = objectMapper.readValue(responseBody, User.class);  // responseBody 从哪里来？

// ✅ 正确：用 exchange() 后的断言链
webTestClient.get().uri("/users/{id}", 1L)
    .exchange()
    .expectStatus().isOk()
    .expectBody(User.class)
    .consumeWith(result -> {
        User user = result.getResponseBody();
        assertEquals("张三", user.getName());
    });
```

`WebTestClient` 的 `exchange()` 返回的是 `WebTestClient.ResponseSpec`，所有断言都必须在这个链式调用中完成，不能提前取出响应体。

### @MockBean 覆盖了真正的 Bean

```java
@SpringBootTest
class UserServiceTest {

    @MockBean               // ← 这里 Mock 了 UserService
    private UserService userService;
    // 问题：整个 Spring 容器中的 UserService 都被替换成了 Mock 对象
    // 如果测试的正是 UserService 本身，这就矛盾了！

    @Test
    void testUserService() {
        // userService 是 Mock，不是真正的 UserServiceImpl
        // 这个测试实际上没有测试任何真实逻辑
    }
}
```

**规则**：测试哪个类，就不要 `@MockBean` 哪个类。要测试 `UserService`，用 `@SpringBootTest` 加载真实 Bean，或者用纯 Mockito 单元测试（不启动 Spring）。

### 测试方法的执行顺序和状态共享

```java
@SpringBootTest
class UserRepositoryTest {

    @Autowired
    private UserRepository userRepository;

    @Test
    void test1() {
        userRepository.save(new User("张三"));
        // 没有清理...
    }

    @Test
    void test2() {
        // test1 插入的数据还在吗？
        // 取决于：是否有 @Transactional（自动回滚）？
        // 是否有 @DirtiesContext（让 Spring 重新加载上下文）？
        // 测试框架是否随机排序（JUnit 5 默认是）？
    }
}
```

**最佳实践**：每个测试方法都是独立的，不依赖其他测试方法的执行结果。用 `@BeforeEach` 准备测试数据，用 `@AfterEach` 或 `@Transactional`（自动回滚）清理。
