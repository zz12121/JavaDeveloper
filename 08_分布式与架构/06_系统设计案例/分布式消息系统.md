# 分布式消息系统

## 这个问题为什么存在？

> 系统之间直接调用，耦合严重。订单服务调库存、支付、通知，**一个慢全部慢**。问题是：**怎么解耦、削峰、异步？**

没有消息系统，你会遇到：
- 订单服务依赖3个服务，响应时间 = 三者之和
- 大促时，订单量暴增，数据库直接挂
- 通知服务挂了，订单服务也跟着报错

---

## 它是怎么解决问题的？

### Kafka核心架构

```
           ┌── Producer1（订单服务）
           │
Producer ─┼── Producer2（用户服务）
           │
           └── Producer3（日志服务）

                    Topic：order-event
                    ┌── Partition 0（Leader: Broker1）
                    │    Replica: Broker2, Broker3
                    │
                    ├── Partition 1（Leader: Broker2）
                    │    Replica: Broker1, Broker3
                    │
                    └── Partition 2（Leader: Broker3）
                         Replica: Broker1, Broker2

Consumer Group A（通知服务）:
  Consumer A1 → 消费 Partition 0
  Consumer A2 → 消费 Partition 1
  Consumer A3 → 消费 Partition 2

Consumer Group B（报表服务）:
  Consumer B1 → 消费全部分区（广播）
```

**核心概念**：
- **Topic**：消息主题（如 `order-event`）
- **Partition**：并行单位，一个Topic分多个Partition（提高吞吐）
- **Consumer Group**：消费组，组内**均分Partition**，组间**广播**
- **Offset**：消费者在Partition中的消费进度

### 高吞吐的四大秘诀

#### 1. 零拷贝（Zero Copy）

```
传统IO：磁盘 → 内核缓冲区 → 用户缓冲区 → Socket缓冲区 → 网卡
         （4次拷贝，4次上下文切换）

零拷贝：磁盘 → 内核缓冲区 ────────→ 网卡
         （2次拷贝，2次上下文切换，用sendfile系统调用）
```

```java
// Kafka用零拷贝发送文件（Java NIO）
FileChannel channel = new FileInputStream(file).getChannel();
// transferTo：零拷贝，数据不从内核态拷贝到用户态
channel.transferTo(0, file.length(), socketChannel);
```

#### 2. 顺序写磁盘

```
随机写：磁头寻道 → 慢（10ms寻道时间）
顺序写：磁头不动 → 快（600MB/s，比内存慢不了多少）
```

Kafka把消息**顺序追加**到日志文件（Partition），极大提高写入速度。

#### 3. 批量发送 + 压缩

```java
// Producer配置：批量发送 + 压缩
Properties props = new Properties();
props.put("batch.size", 16384);      // 16KB一批
props.put("linger.ms", 100);         // 最多等待100ms
props.put("compression.type", "snappy");  // 压缩（省带宽）
```

#### 4. 拉模式（Pull）

```
Push模式（RabbitMQ）：Broker主动推 → 消费者处理不过来，就崩了
Pull模式（Kafka）：Consumer主动拉 → 自己控制速率，不怕压垮
```

## 深入原理

| | Kafka | RabbitMQ | RocketMQ |
|---|---|---|---|
| 吞吐量 | 极高（百万QPS） | 中（万QPS） | 高（十万QPS） |
| 延迟 | 高（批量发送） | 低（实时推送） | 中 |
| 消息顺序 | Partition内有序 | 队列有序 | 队列有序 |
| 事务消息 | ❌ 不支持 | ❌ 不支持 | ✅ 支持（半消息） |
| 适用场景 | 日志流、大数据 | 业务消息、路由灵活 | 业务消息、事务消息 |

**本质区别**：
- **Kafka**：为高吞吐设计（日志流、大数据），延迟高但吞吐极大
- **RabbitMQ**：为灵活路由设计（Exchange类型多），延迟低但吞吐一般
- **RocketMQ**：为业务消息设计（事务消息、延迟消息），阿里系首选

## 正确使用方式

### Producer：发消息 + ACK配置

```java
@Configuration
public class KafkaConfig {

    @Bean
    public KafkaTemplate<String, String> kafkaTemplate() {
        Properties props = new Properties();
        props.put("bootstrap.servers", "broker1:9092,broker2:9092");
        props.put("acks", "1");  // Leader写入成功就返回（平衡可靠性和性能）
        // acks=0：不等待确认（最快，但可能丢消息）
        // acks=1：Leader写入成功（推荐）
        // acks=all：所有副本写入成功（最慢，但最可靠）
        props.put("retries", 3);              // 重试3次
        props.put("batch.size", 16384);       // 16KB批量
        props.put("linger.ms", 100);          // 最多等100ms
        return new KafkaTemplate<>(props);
    }
}

// 发送消息
kafkaTemplate.send("order-event", orderId, orderJson);
```

### Consumer：消费 + 手动提交Offset

```java
@KafkaListener(topics = "order-event", groupId = "notify-group")
public void consume(Order order) {
    try {
        notifyService.send(order);  // 业务逻辑
        // 手动提交Offset（防止消费失败但Offset已提交）
        ack.acknowledge();
    } catch (Exception e) {
        // 消费失败，不提交Offset，等待重试
        log.error("消费失败", e);
    }
}
```

**为什么手动提交**？自动提交可能在消费前就提交了Offset，消费失败就丢了。

## 边界情况和坑

### 坑1：消息丢失（Producer端）

```
场景：acks=0（不等待确认），Leader写入前就返回成功
后果：Leader挂了，消息丢了
```

**解决方案**：
1. **acks=1**（Leader写入成功再返回）
2. **acks=all + min.insync.replicas=2**（至少2个副本写入成功）

### 坑2：重复消费（Consumer端）

```
场景：消费成功，但提交Offset失败
后果：重启后，这条消息又被消费一次
```

**解决方案**：
1. **业务幂等**（去重表、唯一约束）
2. **手动提交Offset**（消费成功再提交）

### 坑3：消息顺序错乱

```
场景：同一订单的"创建"和"支付"消息，发到了不同Partition
后果：Consumer先收到"支付"，再收到"创建"（顺序错了）
```

**解决方案**：
1. **同一业务键发到同一Partition**（如用订单ID做Hash）
   ```java
   // 同一订单ID，总是发到同一Partition
   kafkaTemplate.send("order-event", orderId, orderJson);
   ```
2. **Partition内有序**（Kafka保证）
