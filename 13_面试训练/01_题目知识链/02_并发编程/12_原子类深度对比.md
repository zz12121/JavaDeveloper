## 参考要点（盲答后再看）

<details>
<summary>点击展开</summary>

**AtomicInteger 的问题**：
- 所有线程 CAS 同一个 `value` 变量 → 高并发下大量自旋重试
- 缓存行伪共享：`value` 单独占一个缓存行，但 CPU 缓存一致性协议会让每次修改都广播

**LongAdder 的分段思想**：
```java
// 伪代码逻辑
if (cells == null) {
    CAS_BASE();  // 没人竞争，直接 CAS 基值
} else {
    取当前线程的 hash 值 → 找到对应的 Cell → CAS 该 Cell 的值
}
```
- 竞争激烈时，每个线程改自己对应的 Cell，不争夺同一个变量
- `sum()` = base + 所有 Cell 的值（**不是实时精确**，但够用）

**AtomicInteger vs LongAdder**：

| 维度 | AtomicInteger | LongAdder |
|------|--------------|----------|
| 竞争点 | 单变量 CAS | 分散到多个 Cell |
| 适用场景 | 低并发 / 需要精确值 | 高并发计数（如 QPS） |
| 读数精确性 | 精确 | sum() 非实时精确 |
| 内存开销 | 小 | 较大（Cell 数组） |

**@Contended（伪共享优化）**：
```java
// Cell 类上有这个注解，防止多个 Cell 在同一个缓存行
@sun.misc.Contended 
static final class Cell { volatile long value; }
```

**选型建议**：
- 精确计数（如订单号生成）→ `AtomicLong`
- 高并发近似计数（如 QPS 统计）→ `LongAdder`
- 需要自定义累加函数 → `LongAccumulator`
