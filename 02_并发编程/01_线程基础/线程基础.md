# 线程基础

> 线程是 Java 并发编程的起点。理解线程的创建、调度和生命周期，是解决可见性、原子性、有序性三大并发问题的根基。

---

## 这个问题为什么存在？

### 从单线程到多线程

单线程程序一次只能做一件事。但真实程序经常需要同时处理多个任务：

- **IO 等待**：等待数据库查询返回时 CPU 空闲——这段时间可以被利用
- **多核利用**：现代 CPU 多核架构，单线程只能用一个核心
- **响应性**：UI 线程被阻塞 = 界面卡死
- **业务需求**：同时服务多个用户请求、并行处理批量数据

### 多线程为什么难？

线程不难在"创建"，难在"协同"。多个线程共享同一块内存，同时读写同一变量时，**结果不可预测**：

```
int count = 0;

// 线程A 和 线程B 同时执行 count++
// count++ 不是原子操作，分三步：读 → 加 → 写

线程A: 读(0) → 加1 → 写(1)
线程B:          读(0) → 加1 → 写(1)  // 丢失了线程A的更新

// 两次 ++ 后 count 应该是 2，实际是 1
// 这就是「竞态条件」（Race Condition）
```

**并发编程的三大核心问题**：

1. **原子性**：一个操作是否不可被中断？
2. **可见性**：一个线程的修改，另一个线程能否立即看到？
3. **有序性**：指令执行顺序是否和代码顺序一致？

这三个问题贯穿整个并发编程——[[02_并发编程/02_JMM内存模型/JMM内存模型|JMM]] 给出了理论框架，[[02_并发编程/03_锁机制/锁机制|synchronized]]、[[02_并发编程/02_JMM内存模型/volatile|volatile]]、[[02_并发编程/05_CAS与原子类/CAS与原子类|原子类]] 等工具都是在不同层面解决这三个问题。

## 它是怎么解决问题的？

### 线程的创建方式

Java 创建线程有 4 种方式，核心都围绕 `Thread` 类和 `Runnable` 接口：

```java
// 1. 继承 Thread（不推荐——Java 单继承，浪费继承位）
class MyThread extends Thread {
    @Override
    public void run() { /* 任务逻辑 */ }
}
new MyThread().start();

// 2. 实现 Runnable（推荐——解耦任务和线程）
Runnable task = () -> { /* 任务逻辑 */ };
new Thread(task).start();

// 3. 实现 Callable + FutureTask（需要返回值时）
Callable<Integer> task = () -> { return 42; };
FutureTask<Integer> future = new FutureTask<>(task);
new Thread(future).start();
Integer result = future.get();  // 阻塞等待结果

// 4. 线程池（生产环境必须用这种方式）
ExecutorService pool = Executors.newFixedThreadPool(4);
pool.submit(() -> { /* 任务逻辑 */ });
pool.shutdown();
```

```
为什么生产环境必须用线程池？
1. 线程创建和销毁有成本（OS 级别资源分配，约 1MB 栈空间）
2. 无限创建线程 → OOM
3. 线程池复用线程、控制并发度、管理任务队列
4. 详见 → 02_并发编程/08_线程池/线程池
```

### 线程的生命周期

Thread 有六种状态（`Thread.State` 枚举），状态转换关系如下：

```
         start()
 NEW ──────────→ RUNNABLE ←────────┐
                   │      │        │
              调度获得CPU  等待锁    │
              时间片用完  /IO/      │  wait()结束
              yield()   sleep()    │  notify/notifyAll
                   │      │        │  /锁释放
                   │      ↓        │
                   │   BLOCKED ────┘
                   │      │
                   │   WAITING
                   │   (Object.wait()
                   │    Thread.join()
                   │    LockSupport.park())
                   │      │
                   │   TIMED_WAITING
                   │   (Thread.sleep(n)
                   │    Object.wait(n)
                   │    LockSupport.parkNanos(n))
                   │      │
                   ↓      ↓
                TERMINATED
                (run()结束 / 异常退出)
```

常见误区：
- **RUNNABLE**：可运行（含就绪和运行中）（`start()`）
- **BLOCKED**：等待获取监视器锁，等待进入 [[02_并发编程/03_锁机制/锁机制|synchronized]] 块
- **WAITING**：无限期等待（`wait()`、`join()`、`park()`）
- **TIMED_WAITING**：有时限等待（`sleep(n)`、`wait(n)`、`parkNanos(n)`）
- **TERMINATED**：执行完毕（`run()` 正常结束或抛异常）

```
常见误区：
- RUNNABLE 不等于"正在运行"——线程可能在等待 CPU 调度（就绪态）
- BLOCKED 只针对 synchronized——ReentrantLock 的 await() 状态是 WAITING
- Thread.sleep() 不会释放锁，Object.wait() 会释放锁
```

### 线程的核心 API

#### start() vs run()

```java
Thread t = new Thread(() -> System.out.println("hello"));

t.run();    // 在当前线程中执行，不会创建新线程
t.start();  // 创建新线程，在新线程中执行 run()
```

`run()` 是普通方法调用，在当前线程执行；`start()` 会创建新线程并调用 `run()`。这是最基础的知识点，也是最容易混淆的。

#### sleep() vs wait()

| 维度 | sleep() | wait() |
|------|---------|--------|
| 所属类 | Thread | Object |
| 释放锁 | **不释放** | **释放** |
| 使用位置 | 任何地方 | 必须在 synchronized 块内 |
| 唤醒方式 | 超时自动醒 | notify/notifyAll 或超时 |
| 用途 | 暂停当前线程 | 线程间通信/协作 |

```java
// sleep：不释放锁
synchronized (lock) {
    Thread.sleep(1000);  // 其他线程仍然无法获取 lock
}

// wait：释放锁
synchronized (lock) {
    lock.wait();  // 释放 lock，其他线程可以获取
    // 被唤醒后重新竞争锁
}
```

wait() 释放锁的本质原因：它是线程协作机制的一部分——线程释放锁让其他线程进入临界区操作共享数据，操作完毕后通过 notify 唤醒等待线程。sleep() 是纯粹的"暂停"，不涉及线程协作。

#### interrupt() 中断机制

```java
Thread t = new Thread(() -> {
    while (!Thread.currentThread().isInterrupted()) {
        // 检查中断标志
    }
});
t.start();
t.interrupt();  // 设置中断标志为 true
```

中断机制的设计哲学是**协作式中断**——`interrupt()` 只是设置标志，不会强制停止线程。线程需要自行检查并响应。

```
中断机制核心：
1. interrupt() 设置中断标志，不强制停止线程
2. 如果线程在 sleep/wait/join 中被中断 → 抛 InterruptedException 并清除标志
3. 如果线程在运行中 → 只设置标志，需要代码主动检查

错误做法：
- Thread.stop() — 已废弃，不安全（释放锁时对象状态可能不一致）

正确做法：
- interrupt() + InterruptedException + isInterrupted() 三件套
```

**三种中断响应模式**：

```java
// 模式1：轮询检查中断标志
while (!Thread.currentThread().isInterrupted()) {
    doWork();
}

// 模式2：捕获 InterruptedException，恢复标志后退出
try {
    while (true) {
        Thread.sleep(100);
        doWork();
    }
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();  // 恢复中断标志！
    cleanup();
}

// 模式3：双重检查（推荐）
while (!Thread.currentThread().isInterrupted()) {
    try {
        Thread.sleep(100);
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();  // 恢复标志
        break;
    }
}
```

为什么要 `Thread.currentThread().interrupt()`？因为 `InterruptedException` 抛出时 JVM 自动清除了中断标志。如果不恢复，调用栈上层代码就无法感知到中断请求——这是一个非常容易犯的错。

#### join()

```java
Thread t = new Thread(() -> { /* 耗时任务 */ });
t.start();
t.join();      // 当前线程阻塞，等待 t 执行完毕
t.join(3000);  // 最多等 3 秒
```

join() 底层就是 wait()——调用方线程在 t 对象上 wait()，t 执行完毕后 JVM 自动调用 t.notifyAll()。

#### yield()

`Thread.yield()` 提示调度器"我可以让出 CPU"，但**只是一个提示，调度器完全可以忽略**。实际开发中几乎不用。

#### 守护线程 vs 用户线程

```java
Thread daemon = new Thread(() -> {
    while (true) { /* 后台任务 */ }
});
daemon.setDaemon(true);  // 设为守护线程
daemon.start();
```

- **用户线程**：JVM 等待所有用户线程结束后才退出
- **守护线程**：当所有用户线程结束时，JVM 立即退出，不管守护线程是否执行完毕
- 典型守护线程：GC 线程、日志清理线程

### 线程优先级

Java 线程优先级 1~10，但**只映射到 OS 优先级，不同 OS 调度策略不同**。Linux 基本忽略 Java 线程优先级。不要依赖优先级来保证执行顺序。

### 线程组（ThreadGroup）

Java 早期的线程管理机制，现在已过时。stop()/suspend() 有安全问题，现代并发编程用 ExecutorService 管理线程，不需要 ThreadGroup。了解即可。

## 深入原理

### 线程 vs 进程

| 维度 | 进程 | 线程 |
|------|------|------|
| 资源 | 独立内存空间 | 共享进程内存 |
| 创建成本 | 高（OS 分配地址空间） | 低（共享进程资源） |
| 通信 | IPC（管道、共享内存、Socket） | 直接读写共享变量 |
| 切换成本 | 高（切换地址空间） | 低（切换寄存器和栈） |
| 安全性 | 高（内存隔离） | 低（需要同步机制） |
| 崩溃影响 | 不影响其他进程 | 可能导致整个进程崩溃 |

### 平台线程 vs 虚拟线程

**平台线程 vs 虚拟线程（Java 21+）**：

- **映射**：平台线程 1:1 → OS 线程；虚拟线程 M:N → 少量载体线程
- **创建成本**：平台线程 ~1MB 栈空间；虚拟线程 极低（动态增长，几百字节起步）
- **数量上限**：平台线程 几千；虚拟线程 百万级
- **阻塞行为**：平台线程 OS 线程阻塞；虚拟线程 自动卸载，释放载体线程
- **使用方式**：平台线程 通常通过 [[02_并发编程/08_线程池/线程池|线程池]]；虚拟线程 用完即弃，通常不用池

### Runnable vs Callable

| 维度 | Runnable | Callable\<V\> |
|------|----------|-------------|
| 返回值 | 无 | 有（V） |
| 异常 | 不能抛 checked exception | 可以抛 |
| 使用方式 | `new Thread(r).start()` | 配合 FutureTask / 线程池 |
| 引入版本 | Java 1.0 | Java 5 |

## 正确使用方式

### 1. 不要用 Executors 快捷方法创建线程池

```java
// ✗ 队列无界 → OOM
Executors.newFixedThreadPool(10);
Executors.newCachedThreadPool();     // 最大 Integer.MAX_VALUE → OOM
Executors.newSingleThreadExecutor(); // 队列无界 → OOM

// ✓ 手动创建 ThreadPoolExecutor，明确所有参数
new ThreadPoolExecutor(
    2, 4, 60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(1000),  // 有界队列
    r -> new Thread(r, "biz-pool-" + counter.getAndIncrement()),
    new ThreadPoolExecutor.CallerRunsPolicy()
);
```

### 2. 正确的中断处理模板

```java
public void run() {
    try {
        while (!Thread.currentThread().isInterrupted()) {
            doWork();
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();  // 恢复中断标志
    } finally {
        cleanup();
    }
}
```

### 3. 线程命名

```java
// ✗ 线程名是 Thread-1、Thread-2……出问题时无法定位
new Thread(task).start();

// ✓ 有意义的名字
new Thread(task, "order-processing-" + orderId).start();

// 用 ThreadFactory 批量命名
ThreadFactory factory = r -> new Thread(r, "biz-worker-" + counter.getAndIncrement());
```

### 4. 未捕获异常处理

```java
// 线程中抛出的未捕获异常，主线程不会感知到
Thread t = new Thread(() -> { throw new RuntimeException("boom"); });
t.setUncaughtExceptionHandler((thread, ex) -> {
    logger.error("线程 {} 异常", thread.getName(), ex);
});
t.start();

// 线程池中设置全局 Handler
ExecutorService pool = Executors.newFixedThreadPool(4);
((ThreadPoolExecutor) pool).setUncaughtExceptionHandler(
    (t, e) -> logger.error("线程池异常", e)
);
```

## 边界情况和坑

### 1. start() 只能调用一次

```java
Thread t = new Thread(task);
t.start();  // ✓
t.start();  // IllegalThreadStateException！
```

Thread 对象一旦进入 TERMINATED 就不能再 start()。**不要复用 Thread 对象**——线程池的核心价值之一就是解决这个问题。

### 2. ThreadLocal 导致内存泄漏

```java
// 线程池中 ThreadLocal 的经典坑
pool.submit(() -> {
    threadLocal.set(largeObject);
    try {
        doWork();
    } finally {
        // ✗ 忘记清理！线程回收到池中，largeObject 仍被持有
        // ✓ 必须清理
        threadLocal.remove();
    }
});
```

线程池中的线程是复用的，ThreadLocal 的值不会自动清除。**每次使用后必须 remove()**，否则内存泄漏。详见 [[02_并发编程/12_ThreadLocal/ThreadLocal线程本地变量|ThreadLocal]]。

### 3. join() 不抛超时异常

```java
t.join(5000);  // 最多等 5 秒
// 5 秒后 t 还没结束 → join 静默返回，不抛异常
// 需要检查 t.isAlive() 判断是否超时
if (t.isAlive()) {
    // 超时了，线程还在运行
}
```

### 4. 守护线程中不要做 IO 操作

守护线程继承创建者的守护状态——守护线程中创建的子线程也是守护线程。当所有用户线程结束时 JVM 立即退出，守护线程可能被强制终止。**不要在守护线程中做 IO 操作或资源清理**，可能导致数据不一致。

### 5. 线程状态是快照

`Thread.getState()` 返回的是调用瞬间的状态，线程可能在调用后立即切换。不要基于 getState() 做关键决策。

### 6. 未捕获异常导致线程静默死亡

```java
new Thread(() -> {
    throw new RuntimeException("boom");  // 线程直接终止
}).start();
// 主线程完全感知不到异常！
```

不设置 UncaughtExceptionHandler 的话，异常只打印到 System.err。**线程池中的任务异常如果不处理，会被静默吞掉**——`Future.get()` 会包装为 ExecutionException，但 `submit` + 不调用 get 的话异常就消失了。
