## 知识链提示

```
内存溢出与泄漏
  → [[内存溢出与泄漏]]
    → OOM（内存溢出）
      → 本质：申请内存 > 剩余可用内存
      → 常见类型
        → Java heap space：堆内存不够（对象太多 / 内存泄漏）
        → Metaspace：类加载太多（动态代理 / CGLib）
        → Direct buffer memory：NIO 直接内存溢出
        → StackOverflowError：栈溢出（递归无出口）
    → 内存泄漏（Memory Leak）
      → 定义：对象已经不再使用，但 GC 无法回收（仍然被引用着）
      → 常见场景
        → 集合类：往 static Map 里 put()，用完没 remove()
        → 监听器/回调：注册了没注销
        → 线程池：核心线程一直存活，ThreadLocal 没 remove()
          → [[ThreadLocal线程本地变量]] → 线程池场景的泄漏风险
        → 资源没关闭：InputStream / Connection 没 close()
    → 排查步骤
      → 1. 配置 -XX:+HeapDumpOnOutOfMemoryError（OOM 时自动 dump）
      → 2. 用 MAT（Memory Analyzer Tool）打开 hprof 文件
        → Histogram：按类名统计对象数和占用内存
        → Dominator Tree：支配树，找占用内存最大的对象
        → Path To GC Roots：查看这个对象为什么没被回收
```

## 核心追问

| 追问 | 考点 |
|------|------|
| `HashMap` 里用了 `String.substring()` 为什么可能导致内存泄漏（JDK 6 及之前）？ | substring 共享原 char[]，大字符串截取小字符串后原大 char[] 无法释放 |
| MAT 里的「Shallow Heap」和「Retained Heap」分别是什么？ | Shallow：对象自身大小；Retained：对象 + 它引用的所有对象的总大小 |
| `-XX:+HeapDumpOnOutOfMemoryError` 会影响性能吗？ | 只有 OOM 时触发，平时不影响 |
| 为什么 ThreadLocal 会内存泄漏？ | 线程池复用线程 → ThreadLocalMap 的 value 泄漏 |

---

## 参考答案要点

**OOM vs 泄漏**：
- 泄漏：对象无法被 GC 回收（有引用链）
- OOM：申请内存失败（可能是泄漏导致的，也可能是真的需要更多内存）

**最常见的泄漏**：`static Map` 忘 remove、线程池 + ThreadLocal 没 remove、资源没 close。

**排查黄金步骤**：
1. 开启 `-XX:+HeapDumpOnOutOfMemoryError`
2. OOM 后拿到 `java_pid<pid>.hprof`
3. MAT 打开 → Histogram → 找可疑类 → Path To GC Roots
