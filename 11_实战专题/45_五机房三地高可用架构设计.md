## 场景 A：同城双活

### 现象

```
同城北京1、2机房双活部署，但主DB仅在北京1机房，跨机房写延迟达20ms
北京1机房故障后，流量切到北京2机房，但缓存命中率从90%跌到30%
双活流量比例1:1，但跨机房调用链路占比80%，延迟高
```

### 根因分析

同城双活未实现流量闭环，核心组件（DB、缓存）未按机房隔离部署。DB主节点单机房部署，跨机房写请求需经过同城专线，延迟高且存在单点风险。缓存集群未按机房部署，跨机房访问缓存增加额外延迟，且机房故障后缓存失效，穿透到DB。流量调度未做同机房优先，导致大量跨机房调用。

### 解决方案

```java
/**
 * 同城双活流量路由组件：同机房优先，闭环调用
 */
@Component
public class SameIdcTrafficRouter {
    // 机房与服务节点映射：机房ID -> 节点列表
    private final Map<String, List<ServiceInstance>> idcInstanceMap = new ConcurrentHashMap<>();

    // 当前机房ID（从环境变量获取）
    @Value("${idc.id}")
    private String currentIdcId;

    @Autowired
    private RegistryService registryService;

    /**
     * 初始化：从注册中心拉取所有节点，按机房分组
     */
    @PostConstruct
    public void init() {
        List<ServiceInstance> allInstances = registryService.getAllInstances();
        for (ServiceInstance instance : allInstances) {
            String idcId = instance.getMetadata().get("idc_id");
            idcInstanceMap.computeIfAbsent(idcId, k -> new ArrayList<>()).add(instance);
        }
    }

    /**
     * 选择同机房服务实例，失败则选其他机房
     */
    public ServiceInstance chooseInstance(String serviceName) {
        // 1. 优先选择同机房实例
        List<ServiceInstance> sameIdcInstances = idcInstanceMap.get(currentIdcId);
        if (sameIdcInstances != null && !sameIdcInstances.isEmpty()) {
            return loadBalance(sameIdcInstances);
        }

        // 2. 同机房无实例，遍历其他机房
        for (Map.Entry<String, List<ServiceInstance>> entry : idcInstanceMap.entrySet()) {
            if (!entry.getKey().equals(currentIdcId) && !entry.getValue().isEmpty()) {
                return loadBalance(entry.getValue());
            }
        }

        throw new RuntimeException("无可用服务实例");
    }

    /**
     * 负载均衡：轮询
     */
    private int currentIndex = 0;
    private ServiceInstance loadBalance(List<ServiceInstance> instances) {
        synchronized (this) {
            currentIndex = (currentIndex + 1) % instances.size();
            return instances.get(currentIndex);
        }
    }

    /**
     * 同机房DB部署配置：主库、从库均在同一机房
     * 配置示例：
     * spring.shardingsphere.datasource.master.url=jdbc:mysql://10.0.1.1:3306/db?serverTimezone=UTC
     * spring.shardingsphere.datasource.slave.url=jdbc:mysql://10.0.1.2:3306/db?serverTimezone=UTC
     */
}
```

## 场景 B：异地三活

### 现象

```
北京到深圳异地机房延迟40ms，写请求RT从20ms升至60ms
异地机房数据同步延迟5秒，用户查询到旧数据，投诉量上升
异地深圳机房仅承载10%流量，北京机房故障时无法承接全量流量
```

### 根因分析

异地三活未做用户分片，部分用户请求跨异地访问。数据全量同步导致异地机房存储压力大，同步延迟高。异地机房资源预留不足，未做弹性扩容准备，无法承接全量流量。此外，未做读写分离，异地机房的读请求也路由到主机房，增加跨城延迟。

### 解决方案

```java
/**
 * 异地三活用户分片路由：按用户ID哈希分片，用户固定到所属大区
 */
@Component
public class CrossRegionShardRouter {
    // 大区与机房映射：大区ID -> 机房列表
    private static final Map<String, List<String>> REGION_IDC_MAP = new HashMap<>();

    static {
        REGION_IDC_MAP.put("north", Arrays.asList("beijing1", "beijing2"));
        REGION_IDC_MAP.put("east", Arrays.asList("shanghai1", "shanghai2"));
        REGION_IDC_MAP.put("south", Arrays.asList("shenzhen1"));
    }

    /**
     * 根据用户ID路由到所属大区机房
     * @param userId 用户ID
     * @return 机房ID
     */
    public String routeByUserId(Long userId) {
        // 用户ID哈希取模，分为3个大区（北方、东方、南方）
        int shard = (int) (userId % 3);
        String region = shard == 0 ? "north" : shard == 1 ? "east" : "south";
        // 大区内选择同机房（同机房优先）
        List<String> idcList = REGION_IDC_MAP.get(region);
        return idcList.get(0); // 简化：选大区第一个机房
    }

    /**
     * 异地数据同步策略：仅同步用户所属大区的数据到其他大区，减少同步量
     * 示例：北方大区用户数据仅同步到东方、南方大区备份，不全量同步
     */
    public boolean needSyncToRegion(String sourceRegion, String targetRegion, Long userId) {
        String userRegion = routeByUserId(userId);
        // 仅同步用户所属大区的数据到其他大区
        return userRegion.equals(sourceRegion) && !userRegion.equals(targetRegion);
    }
}
```

## 场景 C：流量调度

### 现象

```
DNS TTL设置10分钟，流量切换需30分钟才全量生效
北京用户IP被GSLB调度到深圳机房，延迟从10ms升至50ms
流量切换无灰度，直接全量切流，导致后端服务雪崩
```

### 根因分析

DNS TTL过长导致调度变更生效慢，不适合快速流量切换场景。GSLB调度策略未结合用户IP归属地、机房负载情况，导致调度不准确。流量切换无灰度机制，大流量瞬间切换导致后端服务连接池耗尽、CPU 100%，引发雪崩。

### 解决方案

```java
/**
 * GSLB流量调度组件：结合用户IP、机房负载，灰度切流
 */
@Component
public class GslbTrafficScheduler {
    // 机房负载阈值：CPU使用率>80%则减少流量
    private static final double LOAD_THRESHOLD = 0.8;

    @Autowired
    private IpLocationService ipLocationService;

    @Autowired
    private IdcLoadService idcLoadService;

    /**
     * 调度用户请求到目标机房
     * @param userIp 用户IP
     * @return 目标机房ID
     */
    public String schedule(String userIp) {
        // 1. 通过IP归属地获取用户所属大区
        String userRegion = ipLocationService.getRegionByIp(userIp);
        // 2. 获取大区内所有机房
        List<String> idcList = getRegionIdcList(userRegion);
        // 3. 过滤负载低于阈值的机房
        List<String> availableIdc = idcList.stream()
                .filter(idc -> idcLoadService.getCpuUsage(idc) < LOAD_THRESHOLD)
                .collect(Collectors.toList());
        // 4. 选择负载最低的机房
        return availableIdc.stream()
                .min(Comparator.comparingDouble(idcLoadService::getCpuUsage))
                .orElse(idcList.get(0));
    }

    /**
     * 灰度切流：按比例逐步切换流量
     * @param sourceIdc 源机房
     * @param targetIdc 目标机房
     * @param percent 切换比例（0-100）
     */
    public void grayScaleSwitch(String sourceIdc, String targetIdc, int percent) {
        // 更新调度规则：百分之percent的流量切到目标机房
        // 实际实现：更新配置中心灰度规则，网关拉取规则后生效
```

## 场景 D：容灾切换

### 现象

```
北京1机房断电故障，10分钟后才完成流量切换，业务中断
切换后数据不一致，用户余额显示错误，客诉量暴增
无自动化切换预案，手动操作容易出错，切换时间不可控
```

### 根因分析

容灾切换依赖人工判断和操作，未实现自动化。无数据同步校验机制，切换前未确认数据已同步到目标机房，导致切换后数据不一致。无定期容灾演练，预案不可行，故障时手忙脚乱。此外，未做流量预热，切换后流量瞬间涌入目标机房，导致服务不可用。

### 解决方案

```java
/**
 * 自动化容灾切换组件：数据校验、流量预热、快速切换
 */
@Component
public class DisasterRecoverySwitcher {
    // 数据同步校验阈值：数据延迟<1秒才允许切换
    private static final long SYNC_DELAY_THRESHOLD = 1000L;

    @Autowired
    private DataSyncService dataSyncService;

    @Autowired
    private GslbTrafficScheduler gslbScheduler;

    /**
     * 自动容灾切换
     * @param faultIdc 故障机房ID
     */
    public void autoSwitch(String faultIdc) {
        // 1. 校验数据同步状态：确认故障机房数据已同步到目标机房
        if (!checkDataSync(faultIdc)) {
            log.error("故障机房{}数据未同步完成，禁止切换", faultIdc);
            return;
        }

        // 2. 流量预热：提前将目标机房资源扩容到承接全量流量
        preheatTargetIdc(faultIdc);

        // 3. 灰度切流：先切10%流量，观察5分钟无异常后全量切换
        String targetIdc = getTargetIdc(faultIdc);
        gslbScheduler.grayScaleSwitch(faultIdc, targetIdc, 10);
        try {
            Thread.sleep(5 * 60 * 1000);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }

        // 4. 全量切换
        gslbScheduler.grayScaleSwitch(faultIdc, targetIdc, 100);
        log.info("容灾切换完成：{} → {}", faultIdc, targetIdc);
    }

    /**
     * 校验数据同步状态
     */
    private boolean checkDataSync(String faultIdc) {
        long syncDelay = dataSyncService.getSyncDelay(faultIdc, getTargetIdc(faultIdc));
        return syncDelay < SYNC_DELAY_THRESHOLD;
    }

    /**
     * 流量预热：目标机房扩容，承接全量流量
     */
    private void preheatTargetIdc(String faultIdc) {
        String targetIdc = getTargetIdc(faultIdc);
        // 调用扩容接口，将目标机房实例数扩容到故障机房的2倍
        log.info("目标机房{}开始流量预热，扩容实例", targetIdc);
    }

    /**
     * 获取容灾目标机房（同城另一个机房优先）
     */
    private String getTargetIdc(String faultIdc) {
        // 北京1故障 → 北京2；上海1故障 → 上海2；深圳1故障 → 上海2
        return faultIdc.equals("beijing1") ? "beijing2" :
               faultIdc.equals("beijing2") ? "beijing1" :
               faultIdc.equals("shanghai1") ? "shanghai2" :
               faultIdc.equals("shanghai2") ? "shanghai1" : "shanghai2";
    }
}
```

## 排查 Checklist

```
□ 同城双活流量是否闭环？ → 同机房调用率>90%
□ 异地用户分片是否正确？ → 用户请求路由到所属大区，跨区率<5%
□ DNS TTL是否合理？ → TTL=60秒，切换生效时间<2分钟
□ 灰度切流是否生效？ → 切流比例与预期一致，无异常
□ 容灾切换是否自动化？ → 故障触发后5分钟内完成切换
□ 数据同步延迟是否达标？ → 跨大区同步延迟<5秒
□ 目标机房容量是否足够？ → 承接全量流量时CPU<70%
□ 切换后数据是否一致？ → 数据校验通过，无一致性投诉
□ 容灾演练是否定期执行？ → 每季度至少1次全链路演练
□ 监控是否覆盖全链路？ → 机房状态、流量、延迟、数据同步全监控
```

## 追问链

### 追问 1：同城双活和异地三活的核心区别是什么？

> 同城双活的核心是**低延迟**，同城市机房之间通过专线互联，延迟<5ms，可实现数据强一致（如DB主从同城部署）。异地三活的核心是**容灾**，跨城市延迟>30ms，无法实现强一致，需采用最终一致性，通过用户分片减少跨城调用。

### 追问 2：异地机房数据同步延迟高怎么处理？

> 三种方案：1. **用户分片**：用户固定到所属大区，减少跨城数据同步。2. **异步同步+补偿**：数据先写本地，异步同步到异地，同步失败则发补偿消息重试。3. **读本地、写本地**：异地机房仅负责本地用户读写，不跨区域访问。

### 追问 3：流量调度如何避免用户跨城访问？

> 结合**IP归属地+Cookie标记**：首次访问通过IP判断用户所属大区，将大区信息写入Cookie；后续请求优先按Cookie中的大区调度，避免IP判断错误导致跨城访问。同时GSLB结合机房负载动态调整，确保调度准确性。

### 追问 4：容灾切换如何保证数据一致性？

> 切换前**数据同步校验**：确认故障机房数据已全量同步到目标机房，无丢失。切换后**数据对账**：每小时对比故障机房和目标机房数据，发现不一致则自动修复。关键业务（如支付）采用**同步写多地**，确保切换后数据可用。
