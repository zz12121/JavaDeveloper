# 泛型

## 这个问题为什么存在？

> Java 1.4 及之前，集合框架用 `Object` 接收所有类型，取出来要强制转换，编译期发现不了类型错误，运行时才 `ClassCastException`。问题是：**怎么让集合在编译期就约束类型，同时又能兼容旧代码？**

泛型的解决思路是**编译期类型检查 + 运行期类型擦除**——编译时保证类型安全，运行时擦掉类型参数保持和旧字节码兼容。

## 它是怎么解决问题的？

### 泛型擦除（Type Erasure）

```
// 编译期：编译器检查类型
List<String> list = new ArrayList<>();
list.add("hello");
list.add(123);   // ❌ 编译错误

// 编译后（字节码）：类型擦除，变成 Object
List list = new ArrayList();
list.add("hello");   // 实际调用 add(Object)
String s = (String) list.get(0);  // 编译器自动插入强转
```

**擦除规则**：
- `<T>` 无边界 → 擦除为 `Object`
- `<T extends Number>` 有边界 → 擦除为 `Number`
- `<T extends Comparable & Serializable>` 多个边界 → 擦除为**第一个边界**（`Comparable`）

### 为什么禁止 `new T[]`？

```
public class MyContainer<T> {
    private T[] elements = new T[10];  // ❌ 编译错误：不能 new 泛型数组
}
```

**原因**：运行期 `T` 已经被擦除了，JVM 不知道 `T` 是什么类型，无法分配正确的数组。

```
擦除后：T → Object
new T[10] → new Object[10]  ?

但如果 T 实际上是 String，Object[] 不是 String[]，类型不安全！
```

**解决方案**：用反射或 `Object` 数组 + 强转：

```java
// 方案1：传 Class<T>
public MyContainer(Class<T> clazz) {
    this.arr = (T[]) Array.newInstance(clazz, 10);
}

// 方案2：用 Object 数组，取时强转
private Object[] arr = new Object[10];
@SuppressWarnings("unchecked")
public T get(int i) { return (T) arr[i]; }
```

### 桥方法（Bridge Method）

泛型擦除后，多态会遇到问题：

```
// 编译前
interface Comparable<T> {
    int compareTo(T o);
}

class String implements Comparable<String> {
    int compareTo(String o) { ... }  // 参数是 String
}

// 擦除后
interface Comparable {
    int compareTo(Object o);  // 擦除为 Object
}

// String 类必须有方法：compareTo(Object)
// 但程序员写的是 compareTo(String)
// → 编译器自动生成桥方法！
```

**桥方法（`javap -c String.class` 可以看到）**：

```java
// 编译器自动生成
public int compareTo(Object o) {
    return this.compareTo((String) o);  // 转发给实际的方法
}
```

桥方法让**泛型子类的重写**在擦除后仍然正确工作。

### 泛型数组的本质困难

```
List<String>[] arr = new List<String>[10];  // ❌ 禁止
```

**原因**：数组是**协变且具体化的**（run-time 检查类型），但泛型是**不变且擦除的**（compile-time 检查）。

```
List<String>[] arr = new List[10];  // ✅ 可以，但会有警告
arr[0] = new ArrayList<Integer>();  // ❌ 运行时才发现类型不匹配？
String s = arr[0].get(0);  // ClassCastException
```

因为数组存储检查只看 `List`（擦除后的类型），不知道里面的泛型参数，所以 JVM 无法在运行时正确检查类型。Java 干脆禁止创建泛型数组，把错误提前到编译期。

### PECS 原则（Producer Extends, Consumer Super）

```
// 生产者（读取）：用 ? extends T
public void copy(List<? extends Number> src, List<Number> dst) {
    for (Number n : src) dst.add(n);  // ✅ 可以读取为 Number
    src.add(1);  // ❌ 不能写入（不知道具体是 Integer 还是 Double）
}

// 消费者（写入）：用 ? super T
public void addNumbers(List<? super Integer> list) {
    list.add(1);  // ✅ 可以写入 Integer
    Integer i = list.get(0);  // ❌ 读取只能当 Object
}
```

## 它和相似方案的本质区别是什么？

| 语言 | 泛型实现 | 保留类型到运行期？ | 类型安全 |
|---|---|---|---|
| Java | 类型擦除 | ❌ | 编译期安全 |
| C# | 具体化泛型 | ✅ | 运行期也安全 |
| C++ 模板 | 代码生成（每个类型一份代码）| ✅ | 编译期安全，代码膨胀 |
| Go 1.18+ | 类型擦除（类似 Java）| ❌ | 编译期安全 |

Java 选择擦除是为了**兼容旧字节码**（Java 1.4 的 `List` 和 Java 5 的 `List<String>` 是同一个类）。

## 正确使用方式

### 泛型方法

```java
// 普通泛型类
class Box<T> {
    private T value;
    public T get() { return value; }
}

// 泛型方法（不依赖类的泛型参数）
public <T> T getMiddle(List<T> list) {
    return list.get(list.size() / 2);
}
```

### 上界和下界

```java
// 上界：只能接收 Number 或其子类
public void printNumbers(List<? extends Number> list) { }

// 下界：只能接收 Integer 或其父类
public void addIntegers(List<? super Integer> list) { }
```

### 错误用法：不能重载

```java
public void print(List<String> list) { }
public void print(List<Integer> list) { }  // ❌ 编译错误
```

擦除后两个方法签名一样（`List`），不算重载。

## 边界情况和坑

### 不能 `T.class`

```
public class MyClass<T> {
    Class<T> clazz = T.class;  // ❌ 擦除后 T 不存在
}
```

解决：传 `Class<T>` 进来。

### 不能 `instanceof T`

```
if (obj instanceof T) { }  // ❌ 运行期 T 被擦除了
```

解决：`obj instanceof Number`（用边界类型）。

### 桥方法导致 `isBridge()` 判断

反射时，桥方法会被认为是"合成方法"：

```java
for (Method m : String.class.getDeclaredMethods()) {
    if (m.isBridge()) {
        System.out.println("桥方法：" + m);
    }
}
```

## 我的理解

泛型 = **编译期类型检查** + **运行期类型擦除**。

面试中最常追问：**为什么禁止 `new T[]`**——擦除后 JVM 不知道数组元素类型，无法保证类型安全。**桥方法**也是高频追问：它是编译器为解决「擦除后多态失效」而自动生成的中转方法。
