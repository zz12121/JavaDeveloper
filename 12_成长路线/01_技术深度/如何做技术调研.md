# 如何做技术调研

> 技术调研不是"搜几篇文章"，而是"系统性评估一个技术能不能用、怎么用、有什么坑"。

---

## 一、技术调研的完整流程

```
阶段 1：明确目标（1 小时）
阶段 2：快速评估（1-2 天）
阶段 3：深度验证（3-5 天）
阶段 4：输出报告（1 天）
```

---

## 二、阶段详解

### 2.1 明确目标

```
必须回答的问题：
1. 我们要解决什么问题？（性能？开发效率？运维成本？）
2. 现有技术为什么不行？（量化：慢多少？贵多少？）
3. 调研的技术要解决到什么程度？（QPS 多少？延迟多少？）
4. 决策时间点？（下周？下个月？）
5. 决策者是谁？（CTO？架构组？自己？）

输出：一页调研目标文档
```

### 2.2 快速评估（1-2 天）

```
目标：快速判断"值不值得深入"，避免浪费时间

动作清单：
□ 官方文档 Overview（30 分钟）
□ GitHub Stars / 最近更新 / issue 活跃度（15 分钟）
□ 社区讨论（知乎/V2EX/掘金，搜"XX 坑"/"XX 为什么不用"）（1 小时）
□ 找 1-2 篇踩坑文章（最重要的！）（1 小时）
□ 和团队成员快速聊一轮（有人用过吗？）

快速判断标准：
✅ 继续深入：文档完善 + 社区活跃 + 有大厂落地案例
❌ 直接放弃：文档缺失 + 半年没更新 + 有更好的替代品
⚠️ 谨慎评估：有坑但没更好选择，继续深入
```

### 2.3 深度验证（3-5 天）

```
目标：用数据和事实回答"能不能用"

必做动作：
1. 跑官方 Demo（2 小时）
2. 做 POC（Proof of Concept，概念验证）（2-3 天）
   - 用真实业务场景测试
   - 覆盖核心功能 + 边界场景
   - 记录所有问题和坑
3. 性能压测（1 天）
   - 用真实数据量测试
   - 对比现有技术（基线对比）
4. 安全 / 合规检查（半天）
   - 开源协议是否允许商用？
   - 有没有已知安全漏洞？
```

### 2.4 输出报告

```
技术调研报告模板：

# 技术调研报告：XXX

## 一、背景与目标
- 当前问题
- 调研目标
- 决策时间点

## 二、方案对比
| 维度 | 现有技术 | 候选技术 A | 候选技术 B |
|------|---------|------------|------------|
| 性能 | ... | ... | ... |
| 学习成本 | ... | ... | ... |
| 社区活跃度 | ... | ... | ... |
| 运维成本 | ... | ... | ... |

## 三、POC 验证结果
- 功能满足度
- 性能指标
- 发现的问题和坑

## 四、风险评估
- 技术风险
- 团队风险
- 运维风险

## 五、结论与建议
- 推荐方案
- 落地计划
- 回退方案
```

---

## 三、技术调研的红线

| 红线 | 说明 |
|------|------|
| 没有 POC 就做决策 | 纸上谈兵，必踩坑 |
| 只看官方文档，不看踩坑文章 | 官方文档只说好的 |
| 不考虑团队学习成本 | 技术很好，但没人会用 |
| 不对比至少 2 个方案 | 容易陷入"确认偏误" |
| 不考虑运维成本 | 上线后才知道运维很痛 |

---

## 四、常见技术的调研重点

### 4.1 中间件选型（Redis vs Memcached）

```
必看维度：
- 数据结构丰富度（Redis 支持更多类型）
- 持久化能力（Redis AOF/RDB vs Memcached 纯内存）
- 高可用方案（Redis Sentinel/Cluster vs Memcached 客户端分片）
- 社区活跃度（Redis 遥遥领先）
```

### 4.2 框架选型（Spring Cloud vs Dubbo）

```
必看维度：
- 生态完整度
- 学习曲线
- 公司技术栈匹配度
- 社区和文档
```

### 4.3 数据库选型（MySQL vs PostgreSQL）

```
必看维度：
- 数据量预估（单机能否扛？）
- 查询复杂度（是否需要复杂分析？）
- 事务要求（ACID 级别）
- 团队熟悉度
```
