## 知识链提示

```
主从复制
  → [[主从复制]]
    → 三个线程
      → 主库：Binlog Dump 线程 → 把 binlog 推给从库
      → 从库：I/O 线程 → 接收 binlog，写入 relay log（中继日志）
      → 从库：SQL 线程 → 读 relay log，重放 SQL（单线程！）
        → 多线程复制（MTS）：SQL 线程变成协调线程，多个 worker 线程并行重放
    → 复制类型
      → 异步复制（默认）：主库写完 binlog 就返回，不关心从库是否收到
        → 问题：主库宕机，从库可能丢数据（数据不一致）
      → 半同步复制：主库至少等一个从库 ACK 才返回
        → 解决：减少数据丢失风险（但不能完全避免）
      → 全同步复制：等所有从库都 ACK → 性能太差，几乎不用
    → 主从延迟问题
      → 原因：从库 SQL 线程是单线程（MTS 改进了但仍可能延迟）
      → 影响：读写分离后，写主库、读从库，可能读不到最新数据
      → 解决：读自己写的数据强制走主库 / 先 sleep 再读 / 用缓存
```

## 参考答案要点

**三个线程**：主库 Binlog Dump / 从库 I/O 线程 / 从库 SQL 线程。

**异步复制的问题**：主库宕机 → 从库可能丢数据 → 半同步复制缓解（但不能完全解决）。

**主从延迟**：从库重放 binlog 的速度赶不上主库写入速度 → 读写分离后，读从库可能读到旧数据。
