# 异常处理

## 这个问题为什么存在？

> 从「没有这个东西会怎样」出发，解释问题的根源。

在异常机制出现之前，程序错误处理主要依赖**错误码返回**：

```java
// 没有异常时的错误处理
int result = divide(a, b);
if (result == -1) {
    // 处理除零错误
} else if (result == -2) {
    // 处理溢出错误
}
```

**痛点**：
1. **错误码容易被忽略**：调用者可能不检查返回值
2. **错误处理代码与业务逻辑混杂**：每调用一个方法都要检查错误码
3. **无法携带详细信息**：错误码只能表示错误类型，无法携带上下文
4. **无法跨层传递**：深层错误需要逐层返回，污染所有中间方法

**核心问题**：程序执行过程中，当发生意外情况（文件不存在、网络中断、空指针）时，如何**优雅地**将错误信息传递给调用者，同时不破坏正常的代码流程？

异常机制的本质是**将正常业务逻辑与错误处理代码分离**，并提供**栈轨迹**等丰富信息帮助定位问题。

## 它是怎么解决问题的？

> 不止讲「是什么」，要讲清楚「为什么这样设计能解决问题」。

### 核心机制

#### Java异常体系

```
                Throwable
               /          \
          Error           Exception
                         /        \
                  RuntimeException   检查型异常(IOException等)
```

**设计哲学**：
- `Error`：JVM层面严重问题（OutOfMemoryError、StackOverflowError），**不应该被捕获**
- `Exception`：程序层面可处理的异常
  - **检查型异常（Checked Exception）**：必须捕获或声明抛出（`IOException`、`SQLException`）
  - **非检查型异常（Unchecked Exception）**：`RuntimeException`及其子类，无需强制处理

**为什么需要检查型异常？**
- 强制调用者处理可能失败的情况，提高程序健壮性
- 但过度使用会导致代码臃肿（C#后来取消了检查型异常）

#### try-catch-finally执行顺序

```java
try {
    System.out.println("try");
    return 1;
} catch (Exception e) {
    System.out.println("catch");
    return 2;
} finally {
    System.out.println("finally");
    // return 3;  // ❌ 会覆盖try/catch的返回值
}
// 输出：try → finally → 返回1
```

**执行顺序规则**：
1. `try`块正常执行完毕 → 执行`finally` → 返回
2. `try`块抛出异常 → 执行`catch` → 执行`finally` → 返回或继续抛出
3. `finally`中的`return`会**覆盖**`try`或`catch`中的`return`
4. `finally`中的异常会**覆盖**`try`或`catch`中的异常

**为什么finally总会执行？**
- JVM通过**异常表（Exception Table）**实现：为每个`try`块生成一个异常处理器，无论正常还是异常退出，都会执行`finally`代码

#### 异常表（Exception Table）

```java
// 源码
public void test() {
    try {
        int a = 1 / 0;
    } catch (ArithmeticException e) {
        System.out.println("error");
    }
}
```

编译后的字节码（简化）：
```
Exception table:
   from   to   target  type
      0    4      7    Class ArithmeticException

Code:
   0: iconst_1
   1: iconst_0
   2: idiv
   3: goto 15
   7: astore_1    // catch块开始
   8: getstatic
   ...
  15: return
```

**异常表的工作原理**：
- `from-to`：try块的范围
- `target`：catch块的起始位置
- `type`：捕获的异常类型
- 当异常发生时，JVM查找异常表，找到匹配的异常处理器

#### try-with-resources（Java 7+）

```java
// 传统方式：需要手动关闭资源
InputStream is = null;
try {
    is = new FileInputStream("file.txt");
    // 使用资源
} catch (IOException e) {
    // 处理异常
} finally {
    if (is != null) {
        try {
            is.close();
        } catch (IOException e) {
            // 关闭异常被抑制
        }
    }
}

// try-with-resources：自动关闭
try (InputStream is = new FileInputStream("file.txt");
     OutputStream os = new FileOutputStream("out.txt")) {
    // 使用资源
} catch (IOException e) {
    // 处理异常
}
// 编译器自动生成finally块，调用close()
```

**为什么需要try-with-resources？**
- 手动关闭资源容易遗漏，导致资源泄漏
- 关闭时的异常容易被忽略
- 多个资源的关闭代码非常臃肿

**抑制异常（Suppressed Exception）**：
```java
try (MyResource res = new MyResource()) {
    throw new RuntimeException("主要异常");
} catch (Exception e) {
    // e是主要异常
    // e.getSuppressed() 返回被抑制的异常（如close()抛出的异常）
}
```

#### 自定义异常

```java
// 检查型异常
public class BusinessException extends Exception {
    private final String errorCode;

    public BusinessException(String errorCode, String message) {
        super(message);
        this.errorCode = errorCode;
    }

    public BusinessException(String errorCode, String message, Throwable cause) {
        super(message, cause);  // 保留原始异常栈
        this.errorCode = errorCode;
    }
}

// 非检查型异常
public class SystemException extends RuntimeException {
    public SystemException(String message) {
        super(message);
    }
}
```

**为什么需要自定义异常？**
- **区分异常类型**：调用者可以根据异常类型采取不同处理策略
- **携带业务信息**：错误码、用户提示等
- **保留异常链**：通过`cause`保留原始异常信息

### 源码关键路径

#### 异常抛出机制

```java
// 1. 显式抛出异常
throw new IllegalArgumentException("参数错误");

// 2. JVM自动抛出异常（如空指针）
Object obj = null;
obj.toString();  // JVM抛出NullPointerException

// 3. 底层实现：athrow指令
// 当执行throw语句或JVM检测到异常条件时：
// - 创建异常对象
// - 填充栈轨迹（fillInStackTrace()）
// - 查找异常表，找到匹配的异常处理器
// - 如果找不到，线程终止
```

#### 栈轨迹生成

```java
public class Throwable {
    private StackTraceElement[] stackTrace;

    public synchronized Throwable fillInStackTrace() {
        // 调用native方法获取当前线程的栈帧
        // 填充stackTrace数组
        return this;
    }

    public StackTraceElement[] getStackTrace() {
        return stackTrace;
    }
}
```

**为什么创建异常对象开销大？**
- `fillInStackTrace()`需要**快照当前线程的调用栈**，这是一个**native方法**，开销较大
- 在性能敏感的代码中，应避免频繁抛出和捕获异常

#### 异常链（Exception Chaining）

```java
// 捕获底层异常，包装为业务异常
try {
    dao.save(user);
} catch (SQLException e) {
    // 包装底层异常，保留原始信息
    throw new BusinessException("DB_ERROR", "数据库保存失败", e);
}
```

**为什么需要异常链？**
- 底层异常（如`SQLException`）对上层调用者没有意义
- 但保留底层异常有助于**排查问题**
- 异常链通过`cause`字段实现：`e.getCause()`返回原始异常

## 它和相似方案的本质区别是什么？

> 对比不是列表，要解释「为什么选 A 不选 B」。

### 检查型异常 vs 非检查型异常

| 维度 | 检查型异常（Checked） | 非检查型异常（Unchecked） |
|------|---------------------|------------------------|
| **编译期检查** | 必须捕获或声明抛出 | 无需强制处理 |
| **使用场景** | 调用者必须处理的异常 | 程序错误（bug） |
| **示例** | `IOException`、`SQLException` | `NullPointerException`、`IllegalArgumentException` |
| **设计目的** | 强制调用者处理可能失败的情况 | 表示程序错误，不应被捕获 |

**为什么需要检查型异常？**
- 强制调用者处理可能失败的情况，提高程序健壮性
- 但过度使用会导致代码臃肿（C#后来取消了检查型异常）

**为什么需要非检查型异常？**
- 程序错误（如空指针）不应该被捕获，而应该修复代码
- 如果强制捕获`NullPointerException`，代码会变得非常臃肿

**选择原则**：
- 调用者**必须处理**的情况 → 检查型异常
- 程序错误（bug）→ 非检查型异常

### 异常 vs 错误码

```java
// 错误码方式
int result = service.process(data);
if (result == ERROR_CODE_1) {
    // 处理错误1
} else if (result == ERROR_CODE_2) {
    // 处理错误2
}

// 异常方式
try {
    service.process(data);
} catch (BusinessException e) {
    // 处理业务异常
} catch (SystemException e) {
    // 处理系统异常
}
```

**为什么选异常？**
- **错误信息更丰富**：异常可以携带消息、栈轨迹、原因链
- **代码更清晰**：错误处理与业务逻辑分离
- **无法忽略**：异常必须被捕获或声明抛出，而错误码可以被忽略

**为什么选错误码？**
- **性能更好**：错误码没有栈轨迹填充的开销
- **跨语言兼容**：错误码可以在不同语言间传递

### 异常 vs 断言（Assert）

```java
// 断言：用于调试，生产环境默认关闭
assert obj != null : "obj must not be null";

// 异常：用于运行时检查
if (obj == null) {
    throw new IllegalArgumentException("obj must not be null");
}
```

**为什么选断言？**
- 用于**内部 invariant 检查**，不影响生产环境性能
- 断言失败表示**程序bug**，而非运行时错误

**为什么选异常？**
- 用于**运行时错误检查**，生产环境也需要
- 异常可以**被捕获和处理**，断言失败会直接导致程序终止

## 正确使用方式

> 不是 API 手册，要解释「为什么这样用是对的」。

### 异常处理的最佳实践

```java
// 1. 只捕获你能处理的异常
try {
    Thread.sleep(1000);
} catch (InterruptedException e) {
    // 正确：恢复了中断状态
    Thread.currentThread().interrupt();
    throw new RuntimeException(e);
}

// 2. 不要捕获Throwable或Error
try {
    // ...
} catch (Error e) {  // ❌ 不应该捕获Error
    // ...
}

// 3. 使用多捕获（Java 7+）
try {
    // ...
} catch (IOException | SQLException e) {  // ✅ 减少重复代码
    // 处理异常
}

// 4. 早抛出，晚捕获
// 底层方法：尽早抛出异常
public void process(String input) {
    Objects.requireNonNull(input, "input must not be null");  // 早抛出
    // ...
}

// 高层方法：延迟捕获，统一处理
public void controllerMethod() {
    try {
        service.process(input);
    } catch (Exception e) {  // 晚捕获
        // 统一处理或转换异常
    }
}

// 5. 清理资源用try-with-resources
try (Connection conn = dataSource.getConnection();
     PreparedStatement ps = conn.prepareStatement(sql)) {
    // 使用资源
} catch (SQLException e) {
    // 处理异常
}
// 编译器自动生成finally块，调用close()
```

**为什么这样用是对的？**
- 只捕获能处理的异常，避免吞噬异常
- 不捕获`Error`，因为`OutOfMemoryError`等问题无法通过代码修复
- 多捕获减少重复代码，提高可维护性
- 早抛出：让错误尽早暴露；晚捕获：在合适的层次统一处理
- `try-with-resources`避免资源泄漏

### 异常转换与包装

```java
// 1. 将底层异常包装为业务异常
public void saveUser(User user) throws BusinessException {
    try {
        userDao.save(user);
    } catch (SQLException e) {
        // 包装底层异常，保留原始信息
        throw new BusinessException("DB_ERROR", "保存用户失败", e);
    }
}

// 2. 使用异常链传递原始异常
public void process() throws SystemException {
    try {
        dao.query();
    } catch (SQLException e) {
        // 使用initCause()建立异常链
        SystemException se = new SystemException("查询失败");
        se.initCause(e);
        throw se;
    }
}
```

**为什么需要异常转换？**
- 底层异常（如`SQLException`）对上层调用者没有意义
- 转换为业务异常，让API更清晰

### 自定义异常的设计

```java
// 1. 提供多个构造方法
public class BusinessException extends Exception {
    private final String errorCode;

    // 构造方法1：错误码 + 消息
    public BusinessException(String errorCode, String message) {
        super(message);
        this.errorCode = errorCode;
    }

    // 构造方法2：错误码 + 消息 + 原因
    public BusinessException(String errorCode, String message, Throwable cause) {
        super(message, cause);
        this.errorCode = errorCode;
    }

    // 构造方法3：错误码 + 原因
    public BusinessException(String errorCode, Throwable cause) {
        super(cause);
        this.errorCode = errorCode;
    }
}

// 2. 使用枚举定义错误码
public class BusinessException extends Exception {
    private final ErrorCode errorCode;

    public BusinessException(ErrorCode errorCode, String message) {
        super(message);
        this.errorCode = errorCode;
    }

    public enum ErrorCode {
        USER_NOT_FOUND("USER_001"),
        INVALID_PARAMETER("PARAM_001");

        private final String code;
        ErrorCode(String code) { this.code = code; }
        public String getCode() { return code; }
    }
}
```

**为什么这样设计？**
- 多个构造方法提供灵活性
- 使用枚举定义错误码，避免硬编码错误码字符串

## 边界情况和坑

> 不是列举，要解释「坑的成因」。

### 坑1：finally块中的return覆盖返回值

```java
public int test() {
    try {
        return 1;
    } finally {
        return 2;  // ❌ 会覆盖try中的return 1
    }
}
// 调用test()返回2，且try中的return 1被丢弃
```

**成因**：`finally`块在`try`的`return`之前执行，如果`finally`中也有`return`，会覆盖`try`中的返回值。

**解决方案**：避免在`finally`中使用`return`。

### 坑2：finally块中的异常覆盖原始异常

```java
try {
    throw new RuntimeException("原始异常");
} finally {
    throw new RuntimeException("finally异常");  // ❌ 覆盖原始异常
}
// 原始异常被丢弃，只能看到"finally异常"
```

**成因**：如果`finally`块也抛出异常，会覆盖`try`或`catch`中的异常。

**解决方案**：
```java
// 正确做法：在finally中处理异常，而非抛出
Exception original = null;
try {
    throw new RuntimeException("原始异常");
} catch (Exception e) {
    original = e;
    throw e;
} finally {
    try {
        resource.close();
    } catch (Exception e) {
        if (original != null) {
            original.addSuppressed(e);  // 使用抑制异常
        } else {
            throw e;
        }
    }
}
```

### 坑3：在try块中返回，finally修改返回值的误区

```java
public int test() {
    int i = 0;
    try {
        i = 1;
        return i;  // 返回1
    } finally {
        i = 2;  // 修改i，但返回值已经是1
    }
}
// 调用test()返回1，而非2
```

**成因**：`return i`时，返回值已经**复制**到返回栈中，`finally`中修改`i`不会影响已复制的返回值。

### 坑4：异常性能开销

```java
// 慢：在性能敏感的代码中频繁抛出异常
for (int i = 0; i < 100000; i++) {
    try {
        Integer.parseInt("not-a-number");
    } catch (NumberFormatException e) {
        // 处理异常
    }
}

// 快：先检查，再执行
for (int i = 0; i < 100000; i++) {
    String s = "not-a-number";
    if (s.matches("\\d+")) {  // 先检查
        Integer.parseInt(s);
    }
}
```

**成因**：创建异常对象时，需要调用`fillInStackTrace()`填充栈轨迹，这是一个**native方法**，开销较大。

### 坑5：异常被吞掉（Swallowed Exception）

```java
// 错误：捕获异常后不处理
try {
    riskyOperation();
} catch (Exception e) {
    // ❌ 空catch块，异常被吞掉
}

// 错误：只打印异常，不处理
try {
    riskyOperation();
} catch (Exception e) {
    e.printStackTrace();  // ❌ 在生产环境中无效
}

// 正确：至少记录日志
try {
    riskyOperation();
} catch (Exception e) {
    logger.error("操作失败", e);  // ✅ 记录日志
    throw e;  // 或重新抛出
}
```

**成因**：空`catch`块或只调用`printStackTrace()`会导致异常信息丢失，难以排查问题。

### 坑6：在构造函数中抛出异常

```java
public class User {
    private final String name;

    public User(String name) {
        if (name == null) {
            throw new IllegalArgumentException("name must not be null");
        }
        this.name = name;
    }
}
```

**问题**：如果构造函数抛出异常，对象**没有完全初始化**，可能导致资源泄漏（如文件句柄、数据库连接）。

**解决方案**：
- 在构造函数中避免获取需要显式释放的资源
- 如果必须获取，使用`try-finally`确保资源释放

## 我的理解（可选）

> 用自己的话重新表述一遍，检验是否真正理解。

异常处理的核心思想是**将正常业务逻辑与错误处理代码分离**，让代码更清晰、更健壮。

**核心机制**：
- Java异常体系分为`Error`和`Exception`，`Exception`又分为检查型和非检查型
- `try-catch-finally`通过**异常表**实现，确保`finally`块总会执行
- `try-with-resources`自动关闭资源，避免资源泄漏
- 异常链通过`cause`字段保留原始异常信息

**为什么需要异常？**
- 错误信息更丰富（消息、栈轨迹、原因链）
- 代码更清晰（错误处理与业务逻辑分离）
- 无法忽略（异常必须被捕获或声明抛出）

**最容易被忽略的点**：
- `finally`中的`return`会覆盖`try`中的返回值
- 创建异常对象开销大（需要填充栈轨迹）
- 空`catch`块会吞掉异常，难以排查问题

## 面试话术总结

1. **Error和Exception的区别？**
   "Error表示JVM层面的严重问题（如OutOfMemoryError、StackOverflowError），不应该被捕获，也无法恢复。Exception表示程序层面可处理的异常，分为检查型异常（必须捕获或声明抛出）和非检查型异常（RuntimeException及其子类，无需强制处理）。"

2. **try-catch-finally的执行顺序？**
   "执行顺序：1）try块正常执行完毕 → 执行finally → 返回；2）try块抛出异常 → 执行catch → 执行finally → 返回或继续抛出。需要注意的是，finally中的return会覆盖try或catch中的返回值，finally中的异常会覆盖try或catch中的异常。"

3. **try-with-resources的原理？**
   "try-with-resources是Java 7引入的语法糖，用于自动关闭资源。编译器会自动生成finally块，调用所有资源的close()方法。如果close()抛出异常，会被抑制（suppressed），可以通过getSuppressed()获取被抑制的异常。资源类必须实现AutoCloseable接口。"

4. **为什么异常创建开销大？**
   "创建异常对象时，需要调用fillInStackTrace()方法填充栈轨迹，这是一个native方法，需要快照当前线程的调用栈，开销较大。因此在性能敏感的代码中，应避免频繁抛出和捕获异常。"

5. **检查型异常和非检查型异常的选择原则？**
   "检查型异常用于调用者必须处理的情况（如IOException、SQLException），通过编译期检查强制调用者处理。非检查型异常用于表示程序错误（如NullPointerException、IllegalArgumentException），不应该被捕获，而应该修复代码。选择原则：调用者必须处理的 → 检查型异常；程序错误（bug）→ 非检查型异常。"
