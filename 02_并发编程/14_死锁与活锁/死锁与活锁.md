# 死锁与活锁

> 死锁是最经典的并发 bug，活锁是更隐蔽的并发陷阱。它们都是线程无法继续执行的状态，但成因和解决方式完全不同。理解它们的本质，是写出正确并发程序的基本功。

---

## 这个问题为什么存在？

### 多线程协作的代价

加锁保护共享资源是必要的，但引入了新的风险：

```
场景 1：两个线程各持一把锁，都想拿对方的锁
  线程 A 持有 lock1，等待 lock2
  线程 B 持有 lock2，等待 lock1
  → 两个线程永远等下去 = 死锁

场景 2：两个线程互相让步，反复重试
  线程 A 发现资源被 B 占用，主动释放重试
  线程 B 发现资源被 A 占用，主动释放重试
  → 两个线程永远在"让步-重试"循环 = 活锁

场景 3：一个线程拿了两把锁不释放，其他线程全卡住
  → 饥饿（Starvation）
```

```
为什么不能"设计好就不会出问题"？

1. 真实系统复杂：几十个服务互相调用，锁的获取顺序不可能全局规划
2. 代码会演进：今天单锁，明天加锁，锁的顺序可能在迭代中被打乱
3. 并发 bug 难复现：取决于线程调度顺序，测试可能跑一万次不出问题，上线就死锁

所以：必须从设计层面预防 + 运行时检测 + 应急恢复
```

---

## 它是怎么解决的？

### 一、死锁（Deadlock）

#### 四个必要条件（Coffman 条件）

```
死锁的充要条件——四个必须同时满足，缺一不可：

1. 互斥（Mutual Exclusion）
   资源一次只能被一个线程持有
   → 锁的本质就是互斥，这个条件无法消除

2. 占有并等待（Hold and Wait）
   线程持有至少一个资源，同时等待获取其他资源
   → 持有 lock1 的同时等待 lock2

3. 不可抢占（No Preemption）
   资源不能被强制剥夺，只能由持有者主动释放
   → synchronized 的锁不能被别的线程抢走

4. 循环等待（Circular Wait）
   存在线程的循环等待链：T1→T2→T3→...→T1
   → A 等 B 的锁，B 等 C 的锁，C 等 A 的锁
```

**核心策略：打破其中任意一个条件即可预防死锁。**

```
消除可能性分析：
- 互斥：无法打破（锁就是为了互斥）
- 占有并等待：可以打破（一次性获取所有锁）
- 不可抢占：可以打破（tryLock 超时机制）
- 循环等待：可以打破（统一加锁顺序） ← 最实用
```

#### 死锁代码示例

```java
// 经典死锁：两个线程以不同顺序获取两把锁
Object lock1 = new Object();
Object lock2 = new Object();

// 线程 A：先 lock1 再 lock2
new Thread(() -> {
    synchronized (lock1) {
        System.out.println("A: 持有 lock1，等待 lock2...");
        sleep(100);  // 让线程 B 有时间拿 lock2
        synchronized (lock2) {
            System.out.println("A: 拿到 lock2");
        }
    }
}).start();

// 线程 B：先 lock2 再 lock1
new Thread(() -> {
    synchronized (lock2) {
        System.out.println("B: 持有 lock2，等待 lock1...");
        sleep(100);
        synchronized (lock1) {
            System.out.println("B: 拿到 lock1");
        }
    }
}).start();

// 结果：A 等 B 释放 lock2，B 等 A 释放 lock1 → 死锁
```

#### 预防策略

##### 策略 1：固定加锁顺序（最推荐）

```java
// ❌ 危险：不同方法加锁顺序不同
void transfer(Account from, Account to, int amount) {
    synchronized (from) {       // 先锁 from
        synchronized (to) {     // 再锁 to
            from.debit(amount);
            to.credit(amount);
        }
    }
}
// transfer(A, B) → 先 A 后 B
// transfer(B, A) → 先 B 后 A
// 并发调用两个方向 → 死锁！

// ✅ 安全：统一按某个属性排序加锁
void transfer(Account from, Account to, int amount) {
    Account first = from.getId() < to.getId() ? from : to;
    Account second = from.getId() < to.getId() ? to : from;
    synchronized (first) {
        synchronized (second) {
            from.debit(amount);
            to.credit(amount);
        }
    }
}
// 无论 transfer(A, B) 还是 transfer(B, A)，都是先锁 id 小的
// 打破循环等待条件 ✓
```

##### 策略 2：一次性获取所有锁

```java
// ❌ 分步获取
synchronized (lockA) { /* 操作 */ }
synchronized (lockB) { /* 操作 */ }

// ✅ 一次获取（粗粒度）
synchronized (lockA) {
    synchronized (lockB) {
        // 所有操作在这里完成
    }
}

// ✅ 更优雅：使用 Lock 的 tryLock
Lock lockA = new ReentrantLock();
Lock lockB = new ReentrantLock();

while (true) {
    if (lockA.tryLock()) {
        try {
            if (lockB.tryLock()) {
                try {
                    // 拿到两把锁，执行操作
                    break;
                } finally {
                    lockB.unlock();
                }
            }
        } finally {
            lockA.unlock();
        }
    }
    // 两把锁没全拿到，短暂等待后重试
    Thread.sleep(50);
}
```

##### 策略 3：超时机制

```java
// 使用 tryLock 设置超时，避免无限等待
Lock lock = new ReentrantLock();

if (lock.tryLock(3, TimeUnit.SECONDS)) {
    try {
        // 获取成功，执行操作
    } finally {
        lock.unlock();
    }
} else {
    // 超时获取失败，执行降级逻辑
    log.warn("获取锁超时，当前操作跳过");
    fallback();
}
```

#### 死锁检测

```
运行时检测死锁的手段：

1. jstack（JDK 自带）
   $ jstack <pid>
   → 输出线程转储，自动检测并报告死锁
   → "Found one Java-level deadlock:"

2. jconsole / jvisualvm
   → GUI 工具，"线程"标签页显示死锁

3. ThreadMXBean（代码内检测）
   ThreadMXBean mxBean = ManagementFactory.getThreadMXBean();
   long[] deadlockedThreads = mxBean.findDeadlockedThreads();
   if (deadlockedThreads != null) {
       ThreadInfo[] infos = mxBean.getThreadInfo(deadlockedThreads);
       // 打印死锁信息，触发告警
   }

4. JMC（Java Mission Control）
   → 更高级的监控工具，支持连续监控
```

### 二、活锁（Livelock）

#### 什么是活锁？

```
活锁：线程没有阻塞，但一直在做无用功，无法推进任务

比喻：
- 两个人在走廊面对面碰上，同时往左让，又同时往右让，永远错过
- 两个人都很"礼貌"地让对方先走，结果谁也没过去

和死锁的区别：
- 死锁：线程阻塞（WAITING 状态），什么都不做
- 活锁：线程一直运行（RUNNABLE 状态），但没有实际进展
```

#### 活锁代码示例

```java
// 两个线程互相让步
class Spouse {
    String name;
    boolean isHungry = true;

    void eatWith(Spouse other) {
        while (isHungry) {
            if (other.isHungry) {
                // 对方也饿，礼貌让步
                System.out.println(name + ": 你先吃吧");
                Thread.yield();  // 主动让出 CPU
            } else {
                // 对方不饿了，自己吃
                isHungry = false;
                System.out.println(name + ": 我开始吃了");
            }
        }
    }
}

// 两个 spouse 都饿，互相让步 → 永远吃不上饭
```

#### 活锁的解决方式

```
1. 加入随机退避（Random Backoff）
   → 每次重试前等一个随机时间，打破"同步让步"
   → 这是消息队列、网络重试中的经典做法

2. 指定"负责人"
   → 给其中一个线程赋予优先级或特殊角色
   → "id 小的先执行，id 大的等待"

3. 重试次数限制
   → 超过最大重试次数后，走降级逻辑
   → 避免无限重试
```

```java
// 解决方案：随机退避
Random random = new Random();

void eatWithRetry(Spouse other) {
    int maxRetries = 10;
    for (int i = 0; i < maxRetries && isHungry; i++) {
        if (!other.isHungry) {
            isHungry = false;
            System.out.println(name + ": 我开始吃了");
            return;
        }
        System.out.println(name + ": 你先吃吧");
        try {
            Thread.sleep(random.nextInt(100) + 50);  // 随机退避
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return;
        }
    }
    System.out.println(name + ": 重试次数耗尽，先不吃了");
}
```

### 三、饥饿（Starvation）

#### 什么是饥饿？

```
饥饿：某个线程长期得不到所需资源，虽然其他线程在正常运行

典型场景：
1. 优先级反转——低优先级线程持有锁，高优先级线程等锁
   → 中优先级线程抢占 CPU，低优先级线程得不到执行
   → 高优先级线程间接被中优先级线程"饿死"

2. 非公平锁——新来的线程可以插队
   → 等待时间最长的线程可能一直拿不到锁

3. 线程池任务饥饿——核心线程被长任务占满
   → 短任务排不上队
```

#### 解决方式

```
1. 使用公平锁
   → ReentrantLock(true) 或 synchronized（JVM 尽量公平，但不保证）

2. 合理设置线程优先级
   → 避免极端优先级差异

3. 线程池隔离
   → 快任务和慢任务用不同线程池
   → 快任务不会被慢任务阻塞
```

## 死锁、活锁、饥饿对比

```
                死锁              活锁              饥饿
────────────────────────────────────────────────────────────
线程状态        阻塞（WAITING）   运行（RUNNABLE）  阻塞（WAITING）
是否在执行      ❌ 完全停止       ✅ 一直在跑       ❌ 在等待
任务能否推进    ❌ 完全不能       ❌ 看似在动实则在转  ❌ 单个线程不能
是否影响其他    ✅ 相关线程全卡   ✅ 相关线程全转    ❌ 只影响被饿的线程
诊断难度        中（jstack检测）   高（线程在跑）    高（概率性问题）
根本原因        循环等待+互斥     过度谦让+同步重试  调度策略不公平
```

## 和相似方案的区别

### 预防 vs 避免 vs 检测

```
死锁处理的三种策略（操作系统经典理论）：

1. 死锁预防（Deadlock Prevention）
   → 运行前破坏四个必要条件之一
   → 例：统一加锁顺序、一次性获取所有锁
   → 代价：可能降低并发度（粗粒度锁）

2. 死锁避免（Deadlock Avoidance）
   → 运行中动态判断是否安全，不安全就不分配资源
   → 例：银行家算法（理论完美，实际几乎不用）
   → 代价：需要预知每个线程的最大资源需求

3. 死锁检测与恢复（Deadlock Detection）
   → 允许死锁发生，定期检测并恢复
   → 例：jstack 检测、ThreadMXBean、进程重启
   → 代价：恢复代价可能很大（数据回滚、事务补偿）

Java 实践中：
→ 预防（固定加锁顺序）+ 检测（jstack/jconsole）是最常用组合
→ 避免策略（银行家算法）在实际开发中基本不用
```

## 正确使用方式

### 写并发代码的安全清单

```
□ 多把锁的获取顺序是否全局一致？
□ 是否用 try-finally 保证锁释放？
□ 是否使用了 tryLock + 超时，而不是无脑 synchronized？
□ 重试逻辑是否有随机退避和重试次数限制？
□ 是否有监控手段（jstack、ThreadMXBean）？
□ 线程池是否做了快慢任务隔离？
□ 关键锁路径是否有日志记录获取/释放时间？
```

### 实战：转账死锁的完整解决方案

```java
class SafeTransfer {

    // 方案 1：按账号 id 排序加锁（预防循环等待）
    void transfer(Account from, Account to, int amount) {
        if (from.getId() == to.getId()) return; // 自己转自己

        Account first = from.getId() < to.getId() ? from : to;
        Account second = from.getId() < to.getId() ? to : from;

        synchronized (first) {
            synchronized (second) {
                from.debit(amount);
                to.credit(amount);
            }
        }
    }

    // 方案 2：tryLock + 超时 + 重试（避免无限等待）
    void transferWithTimeout(Account from, Account to, int amount)
            throws InterruptedException {
        Lock fromLock = from.getLock();
        Lock toLock = to.getLock();

        while (true) {
            if (fromLock.tryLock(100, TimeUnit.MILLISECONDS)) {
                try {
                    if (toLock.tryLock(100, TimeUnit.MILLISECONDS)) {
                        try {
                            from.debit(amount);
                            to.credit(amount);
                            return;
                        } finally {
                            toLock.unlock();
                        }
                    }
                } finally {
                    fromLock.unlock();
                }
            }
            Thread.sleep(new Random().nextInt(50) + 10); // 随机退避
        }
    }
}
```

## 边界情况和坑

### 容易忽略的死锁场景

```java
// 坑 1：锁的隐藏获取
// synchronized 方法调用另一个 synchronized 方法
class Parent {
    synchronized void doSomething() { ... }
}
class Child extends Parent {
    @Override
    synchronized void doSomething() {
        super.doSomething();  // 尝试获取 this 的锁 → 但已经是 this 的锁
        // 如果是可重入锁（synchronized），没问题
        // 但如果和外部锁组合，可能产生意外死锁
    }
}
```

```java
// 坑 2：String 常量池导致的隐式共享锁
// String 常量池中相同内容的字符串是同一个对象
synchronized ("lock") { ... }  // 两个完全不相关的代码用了相同的字符串字面量
synchronized ("lock") { ... }  // 竞争同一把锁！→ 可能死锁
// 解决：用 new Object() 作为锁对象
```

```java
// 坑 3：数据库事务中的死锁
// 事务 A: UPDATE account SET balance = balance - 100 WHERE id = 1
//          UPDATE account SET balance = balance + 100 WHERE id = 2
// 事务 B: UPDATE account SET balance = balance - 100 WHERE id = 2
//          UPDATE account SET balance = balance + 100 WHERE id = 1
// → 数据库行锁的死锁和 Java 锁原理一样
// → 解决：按 id 排序更新 + 重试机制
```

### 活锁在实际系统中的案例

```
真实案例：微服务中的重试风暴

服务 A 调用服务 B，B 返回 503（过载）
A 指数退避重试，3s 后再试
但所有 A 的实例同时开始退避 → 同时重试 → B 再次过载 → 再次退避
→ 全部实例进入"退避-重试-再退避"的活锁循环

解决方案：
1. 抖动（Jitter）：退避时间加随机偏移，打散重试时间点
2. 熔断器（Circuit Breaker）：连续失败后直接降级，不再重试
3. 限流：限制对 B 的并发调用数
```

## 我的理解（面试考点）

### 高频面试题

```
Q1: 产生死锁的四个必要条件是什么？
→ 互斥、占有并等待、不可抢占、循环等待
→ 展开说：互斥无法打破，实际开发主要打破"循环等待"（固定加锁顺序）

Q2: 如何预防死锁？
→ 1. 固定加锁顺序（最推荐）
→ 2. 一次性获取所有锁（粗粒度，可能降低并发）
→ 3. tryLock + 超时（不阻塞，超时后降级）
→ 4. 尽量减少锁的粒度，缩短持锁时间

Q3: 死锁、活锁、饥饿的区别？
→ 死锁：线程阻塞，互相等待，完全不推进
→ 活锁：线程运行，互相让步，无效推进
→ 饥饿：某个线程长期得不到资源，其他线程正常推进

Q4: 实际项目中遇到过死锁吗？怎么排查？
→ 用 jstack 查看线程转储
→ 找到 "Found one Java-level deadlock"
→ 分析锁的获取顺序，统一顺序后修复
→ 加监控：ThreadMXBean.findDeadlockedThreads()

Q5: 说说你了解的锁优化策略？
→ 减小锁粒度、缩短持锁时间
→ 读写分离（ReadWriteLock）
→ 用乐观锁代替悲观锁（CAS）
→ 用并发容器代替 synchronized 容器
```

2. 知道用 tryLock + 随机退避是分布式锁的通用模式
3. 理解"String 常量池作锁"这种隐蔽坑

### 一句话总结

```
死锁预防靠"顺序"，活锁解决靠"随机"，饥饿避免靠"公平"。
在并发编程中，最重要的不是写出不会死锁的代码，
而是建立"持锁时间尽量短、锁粒度尽量小、超时必须有"的编码习惯。
```
