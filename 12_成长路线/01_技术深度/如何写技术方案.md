# 如何写技术方案

> 技术方案是架构师的核心输出。方案写好，80% 的落地问题已经被提前发现了。

---

## 一、技术方案的结构

```
1. 背景与目标（为什么做？）
2. 现状分析（现在有什么问题？）
3. 方案设计（怎么做？）
4. 风险评估（有什么风险？）
5. 实施计划（怎么落地？）
6. 附录（详细设计、API 定义等）
```

---

## 二、逐节详解

### 2.1 背景与目标

```
背景（Background）：
- 当前的业务场景和规模
- 现有方案的痛点（用数据说话！）
- 不解决问题的后果

目标（Objective）：
- 功能目标（要达成什么功能）
- 非功能目标（性能、可用性、一致性等）
- 限制条件（时间、成本、人力）

反面教材：
❌ "现有系统性能不好，需要优化"
✅ "现有系统 QPS 500，大促期间超时率 30%，
    目标：支撑 QPS 5000，P99 延迟 < 200ms，
    限制：1 个月内上线，不超过 2 台额外服务器"
```

### 2.2 现状分析

```
必须说清楚：
1. 现有架构图（AS-IS）
2. 核心链路的时序图
3. 当前性能基线（压测数据）
4. 已知的技术债

输出：一张清晰的现状架构图 + 关键指标表
```

### 2.3 方案设计（核心！）

```
方案对比（至少 2 个方案）：
| 维度 | 方案 A | 方案 B |
|------|--------|--------|
| 性能 | ... | ... |
| 复杂度 | ... | ... |
| 成本 | ... | ... |
| 风险 | ... | ... |

推荐方案详解：
1. 目标架构图（TO-BE）
2. 核心流程时序图
3. 数据模型设计（表结构/Redis Key 设计）
4. API 设计（RESTful 或 RPC 接口定义）
5. 异常场景处理（降级/熔断/限流）
6. 灰度发布方案

关键：每个设计决策都要写清楚「为什么选这个」
```

### 2.4 风险评估

```
| 风险 | 概率 | 影响 | 应对方案 |
|------|------|------|---------|
| Redis 主节点宕机 | 低 | 高 | 主从切换 + Sentinel |
| 数据迁移失败 | 中 | 高 | 灰度迁移 + 回滚方案 |
| 性能不达预期 | 中 | 中 | 提前压测 + 降级开关 |
```

### 2.5 实施计划

```
分阶段：
阶段 1（Week 1-2）：数据迁移脚本 + 双写
阶段 2（Week 3）：灰度 5% 流量
阶段 3（Week 4）：灰度 50% 流量
阶段 4（Week 5）：全量切换
阶段 5（Week 6）：下线旧逻辑

每个阶段都要有：
- 具体任务
- 责任人
- 验收标准
- 回滚方案
```

## 三、技术方案的评判标准

```
好方案的特征：
✅ 问题定义清晰（背景和目标具体可衡量）
✅ 方案有对比（不只是"我打算怎么做"）
✅ 设计决策有理由（trade-off 分析）
✅ 异常场景考虑周全（降级/熔断/限流）
✅ 实施计划可落地（分阶段 + 回滚）
✅ 图示清晰（架构图/时序图/ER 图）

坏方案的特征：
❌ 只有代码，没有设计思路
❌ 只说"怎么做"，不说"为什么"
❌ 没有异常场景分析
❌ 实施计划只有一句话"下周上线"
```

## 四、常见方案类型模板

### 4.1 性能优化方案

```
1. 现状：压测数据 + 瓶颈定位（CPU/内存/IO？）
2. 分析：火焰图 + 慢查询日志
3. 方案：缓存？异步？分库分表？
4. 验证：优化前后压测对比
```

### 4.2 重构方案

```
1. 现状：代码复杂度 + 缺陷率数据
2. 目标：降低圈复杂度到 X，覆盖率提升到 Y%
3. 方案：重构策略（绞杀者模式？新旧并行？）
4. 验证：重构前后功能对比测试
```

### 4.3 新技术引入方案

```
1. 现状：现有技术为什么不行
2. 方案：新技术的 POC 验证结果
3. 迁移：数据迁移方案 + 灰度方案
4. 保障：回滚方案 + 监控告警
```

## 五、方案汇报技巧

```
汇报对象不同，重点不同：

向 CTO/技术 VP 汇报：
→ 重点：业务价值、成本收益、风险
→ 少用术语，多用数据

向架构评审委员会汇报：
→ 重点：技术选型理由、性能数据、风险评估
→ 准备好评委可能问的细节

向团队汇报：
→ 重点：实施计划、每个人的任务、时间节点
→ 让大家知道"我要做什么"
```
