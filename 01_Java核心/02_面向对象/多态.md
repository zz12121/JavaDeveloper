# 多态

> 多态的核心是"对接口编程，而非对实现编程"。它让你写出的代码在**不修改的前提下可以扩展**，这是开闭原则的直接体现。

---

## 这个问题为什么存在？

没有多态时，每加一种新类型就要改所有条件分支：

```java
// 没有多态：每加一种动物，这个方法就要改
void makeAllSounds(Dog dog, Cat cat, Bird bird) {
    dog.bark();
    cat.meow();
    bird.tweet();
}

// 有多态：对接口编程，新增动物类型不用改这个方法
void makeAllSounds(List<Animal> animals) {
    for (Animal a : animals) {
        a.makeSound();  // 运行时自动分派到具体实现
    }
}
```

[[01_Java核心/02_面向对象/封装|封装]] 划定了单个类的边界，[[01_Java核心/02_面向对象/继承|继承]] 建立了类型层级，而多态让这些类型层级在**运行时**能被动态分派——三者合一，才构成了完整的 OOP 能力。

---

## 它是怎么解决问题的？

### 编译时多态 vs 运行时多态

```java
class Animal {
    void eat() { System.out.println("动物吃东西"); }
    static void staticMethod() { System.out.println("Animal static"); }
}

class Dog extends Animal {
    @Override
    void eat() { System.out.println("狗吃骨头"); }
    static void staticMethod() { System.out.println("Dog static"); }
}

Animal a = new Dog();
a.eat();           // 运行时多态 → 调用 Dog.eat()（看实际类型）
a.staticMethod();  // 编译时绑定 → 调用 Animal.staticMethod()（看声明类型）
```

**不参与多态的三种方法**：
- `static` 方法：编译期绑定（invokestatic）
- `final` 方法：编译期绑定（invokespecial）
- `private` 方法：不可见，编译期绑定

### 向上转型与向下转型

```java
// 向上转型（upcasting）：安全，隐式
Animal a = new Dog();   // Dog → Animal，隐式转换，总是安全

// 向下转型（downcasting）：有风险，显式
Dog d = (Dog) a;        // Animal → Dog，需要显式转换，可能 ClassCastException

// Java 16+ 模式匹配：转型和检查合一
if (a instanceof Dog dog) {    // 转型成功后可直接用 dog 变量
    dog.bark();
}
```

向下转型成功的前提：**对象的实际类型必须是目标类型或其子类**。如果 `a` 实际是 `Cat`，`(Dog) a` 会抛 `ClassCastException`。

### 接口多态

接口多态比继承多态更灵活（一个类可以实现多个接口）：

```java
interface CanFly { void fly(); }
interface CanSwim { void swim(); }

class Duck implements CanFly, CanSwim {
    public void fly() { System.out.println("鸭子飞"); }
    public void swim() { System.out.println("鸭子游"); }
}

// 对接口编程，不依赖具体类型
void letFly(CanFly f) { f.fly(); }  // ✓ 鸭子、鸟、昆虫都能传进来
```

详见 [[01_Java核心/02_面向对象/抽象类与接口|抽象类与接口]] 中关于接口设计意图的完整分析。

## 深入原理

### 虚方法表（vtable）机制

```
对象在内存中的布局（HotSpot）：

┌──────────────────────┐
│  Mark Word (8 bytes)  │  → 锁状态、GC 年龄
├──────────────────────┤
│  Klass* (4/8 bytes) │  → 指向方法区的 Class 对象
├──────────────────────┤
│  实例字段数据          │  → name, age 等字段
└──────────────────────┘

Class 对象（方法区）中有虚方法表（vtable）：
Dog.vtable:
  [0] → Dog.makeSound()    ← 覆写了
  [1] → Animal.eat()       ← 继承的，没覆写
  [2] → Object.toString()  ← 继承的，没覆写
```

调用 `animal.makeSound()` 时：
1. 从对象头的类型指针找到 Class 对象
2. 在 vtable 中查找 `makeSound` 的槽位（每个类相同槽位固定）
3. 执行对应的方法

**这就是为什么多态调用比直接调用稍慢**——多了一次 vtable 查找。JIT 会对热点调用做内联优化，消除这个开销。

### 方法分派字节码指令

```
invokestatic   → 静态方法，编译期绑定
invokespecial  → 构造器、private 方法、super.Method()，编译期绑定
invokevirtual  → 实例方法，运行时多态分派（查 vtable）
invokeinterface → 接口方法，类似 invokevirtual，但查表方式不同
invokedynamic  → [[01_Java核心/08_Lambda与Stream/Lambda与函数式|Lambda]] 用这个（Java 7+）
```

`invokevirtual` 的分派过程：
```
1. 取操作数栈顶的对象引用（实际类型）
2. 在该类型的 vtable 中找方法对应的槽位
3. 执行槽位中的方法指针
```

### 为什么接口方法分派用 invokeinterface 而不是 invokevirtual？

因为接口可以有多个实现类，且一个类可以实现多个接口——**vtable 的布局在不同实现类中可能不统一**。JVM 对 `invokeinterface` 做了特殊优化（如用 itable 缓存接口方法查找结果），速度接近 `invokevirtual`。

## 正确使用方式

### 1. 声明类型用接口，不用具体类

```java
// 不好：声明为具体类型，未来换实现要改调用方
ArrayList<String> list = new ArrayList<>();

// 好：声明为接口类型
List<String> list = new ArrayList<>();
// 未来换成 LinkedList 只需改这一行
List<String> list = new LinkedList<>();
```

### 2. 接受参数用最窄类型

```java
// 不好：接受 List，但实际只需要读取
void printAll(List<?> list) {
    for (Object o : list) System.out.println(o);
}

// 更好：接受 Iterable，接受范围更窄，更灵活
void printAll(Iterable<?> iter) {
    for (Object o : iter) System.out.println(o);
}
```

### 3. 返回类型用最宽类型（接口）

```java
// 好：返回接口，调用方不依赖具体实现
public List<String> getNames() {
    return new ArrayList<>(names);  // 未来可以换成其他 List 实现
}
```

## 边界情况和坑

### 1. 字段不参与多态（字段隐藏）

```java
class FieldHide {
    String name = "Animal";
}
class Dog extends Animal {
    String name = "Dog";  // 字段隐藏，不是覆写
}

Animal a = new Dog();
System.out.println(a.name);   // "Animal"（编译时类型决定）
System.out.println(((Dog) a).name);  // "Dog"
```

**字段永远看编译时类型**。这也是为什么字段应该 `private`，通过方法访问才能利用多态。

### 2. 静态方法不参与多态

```java
class Animal {
    static void whoAmI() { System.out.println("Animal"); }
}
class Dog extends Animal {
    static void whoAmI() { System.out.println("Dog"); }
}

Animal a = new Dog();
a.whoAmI();  // 打印 "Animal"！看的是声明类型
```

静态方法用 `invokestatic` 指令，编译期就绑定了目标方法。调用 `a.whoAmI()` 时，编译器眼里 `a` 的类型是 `Animal`，所以绑定 `Animal.whoAmI()`。

### 3. 构造器链式调用中的多态陷阱

```java
class Parent {
    Parent() {
        doSomething();  // ❌ 危险！如果被子类覆写，子类字段还没初始化
    }
    void doSomething() { }
}

class Child extends Parent {
    int value = 10;

    Child() {
        super();  // 先调用 Parent()，此时 value 还是默认值 0
    }

    @Override
    void doSomething() {
        System.out.println(value);  // 打印 0，不是 10！
    }
}
```

**最佳实践**：构造函数中不要调用可覆写的方法。用 `[[01_Java核心/01_基础语法/final关键字|final]]` 保护构造函数调用的步骤。

### 4. 自动装箱破坏多态的重载选择

```java
void print(Object o) { System.out.println("Object"); }
void print(Integer i) { System.out.println("Integer"); }

print(42);  // 打印 "Integer"——自动装箱后精准匹配
print(null);  // 编译错误！Integer 和 Object 都是 null 的候选，歧义
```

重载（`overload`）是编译期绑定的，选择"最具体的类型"。`null` 可以匹配任何引用类型，编译器无法确定选哪个。
