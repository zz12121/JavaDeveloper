# 数据库中间件设计

> 类似ShardingSphere，核心：SQL解析 + 路由 + 结果归并 + 分布式主键。核心矛盾是**对业务透明地实现分库分表，让应用像操作单库单表一样操作分布式数据库**。

---

## 数据库中间件整体架构

```
应用层
   │
   ▼
┌─────────┐  拦截  ┌──────────────┐  改写  ┌──────────────┐
│  业务   │ ────→  │  SQL 解析器  │ ────→ │  路由引擎     │
│  代码   │        │  (ANTLR/Calcite)│      │  (分片计算)   │
└─────────┘        └──────────────┘        └──────┬───────┘
                                                   │
              ┌────────────────────────────────────┼──────────────────────────────┐
              │                                    │                              │
              ▼                                    ▼                              ▼
       ┌────────────┐                      ┌────────────┐              ┌────────────┐
       │  结果归并   │                      │  分布式主键 │              │  执行引擎   │
       │  (排序/聚合)│                      │  (雪花/号段)│              │  (多数据源) │

              ┌───────────────────────────────────────────────────────────────┘

       │  DS_0       │         │  DS_1       │   ...   │  DS_N       │

       └────────────┘         └────────────┘         └────────────┘
```

## 场景 A：SQL解析与路由

### 现象

```
业务执行 SELECT * FROM t_order WHERE user_id = 100
实际数据在 t_order_00 到 t_order_63 共64张表
中间件需要解析 SQL，计算应该路由到哪些表
错误路由会导致查不到数据或全路由（性能差）
```

### 解决方案

```java
// 1. SQL 解析：使用 ANTLR 或 Apache Calcite 解析 SQL
// 这里以简化版为例，展示解析流程
public class SQLParseEngine {
    
    // 解析 SQL，提取表名、条件、投影等
    public SQLStatement parse(String sql) {
        // 实际使用中会用 ANTLR 生成的解析器
        // 如 MySQL Lexer/Parser 或 Calcite SQL Parser
        SQLParser parser = new SQLParser(sql);
        return parser.parseStatement();
    }
}

// 2. 路由引擎：根据分片条件计算目标数据源和表
@Service
public class RoutingEngine {
    
    @Autowired
    private ShardingRule shardingRule;
    
    // 计算路由结果
    public RoutingResult route(SQLStatement sqlStatement) {
        String logicTable = sqlStatement.getTableName();  // 逻辑表名，如 t_order
        
        // 获取分片条件（如 WHERE user_id = 100）
        List<ShardingCondition> conditions = extractShardingConditions(sqlStatement);
        
        // 如果没有分片条件 → 全路由（广播到所有表）
        if (conditions.isEmpty()) {
            return createFullRouting(logicTable);
        }
        
        // 有分片条件 → 精确路由
        return calculateRouting(logicTable, conditions);
    }
    
    // 计算精确路由
    private RoutingResult calculateRouting(String logicTable, List<ShardingCondition> conditions) {
        RoutingResult result = new RoutingResult();
        
        for (ShardingCondition condition : conditions) {
            // 1. 计算目标库（按 user_id % 16）
            int dbIndex = condition.getValue("user_id").hashCode() % 16;
            String dataSourceName = "ds_" + dbIndex;
            
            // 2. 计算目标表（按 user_id % 64）
            int tableIndex = condition.getValue("user_id").hashCode() % 64;
            String actualTable = logicTable + "_" + tableIndex;
            
            result.addRoute(dataSourceName, actualTable);
        }
        
        return result;
    }
    
    // 全路由：广播到所有库所有表
    private RoutingResult createFullRouting(String logicTable) {
        RoutingResult result = new RoutingResult();
        for (int db = 0; db < 16; db++) {
            for (int table = 0; table < 64; table++) {
                result.addRoute("ds_" + db, logicTable + "_" + table);
            }
        }
        return result;
    }
}
```

**SQL 改写**：
```java
// 3. 改写 SQL：逻辑表名 → 实际表名
```

## 场景 B：分片规则设计

### 现象

```
分片规则不合理，导致数据倾斜（某些库/表数据特别多）
查询必须带分片键，否则全路由
分片键选择错误，导致90%查询都全路由
```

### 解决方案

```java
// 1. 分片规则配置（支持多种分片算法）
public class ShardingRuleConfig {
    
    // 分片规则定义
    public static class TableRule {
        private String logicTable;                    // 逻辑表名
        private String actualDataNodes;               // 实际数据节点：ds_${0..15}.t_order_${0..63}
        private ShardingStrategy databaseStrategy;    // 库分片策略
        private ShardingStrategy tableStrategy;       // 表分片策略
    }
    
    // 分片策略
    public static class ShardingStrategy {
        private String shardingColumn;               // 分片键
        private ShardingAlgorithm algorithm;          // 分片算法
    }
    
    // 2. 分片算法接口
    public interface ShardingAlgorithm {
        // 根据分片值计算目标节点
        Collection<String> doSharding(
                Collection<String> availableTargets,  // 可用目标（库名或表名列表）
                ShardingCondition condition           // 分片条件
        );
    }
    
    // 3. 取模分片算法（最常用）
    public class ModShardingAlgorithm implements ShardingAlgorithm {
        private final int mod;
        
        public ModShardingAlgorithm(int mod) {
            this.mod = mod;
        }
        
        @Override
        public Collection<String> doSharding(
                Collection<String> availableTargets, 
                ShardingCondition condition) {
            
            Object value = condition.getValue();  // 分片值
            int shardIndex = value.hashCode() % mod;
            if (shardIndex < 0) shardIndex += mod;
            
            // 返回目标节点（如 ds_3）
            String target = "ds_" + shardIndex;
            return Collections.singletonList(target);
        }
    }
    
    // 4. 范围分片算法（用于时间范围查询）
    public class RangeShardingAlgorithm implements ShardingAlgorithm {
        @Override
        public Collection<String> doSharding(
                Collection<String> availableTargets,
                ShardingCondition condition) {
            
            // 时间范围查询：返回多个目标节点
            // 如 WHERE create_time BETWEEN '2024-01-01' AND '2024-03-31'
            // 返回 ds_202401, ds_202402, ds_202403
            DateRange range = (DateRange) condition.getValue();
            List<String> targets = new ArrayList<>();
            
            for (String target : availableTargets) {
                if (isInRange(target, range)) {
                    targets.add(target);
                }
            }
            return targets;
        }
    }
}
```

**分片规则最佳实践**：
- **分片键选择**：选择查询频率最高的字段（如 user_id、order_id）
- **分片数**：建议 2^n，方便取模（16库、64表等）
- **避免数据倾斜**：测试分片键的哈希分布是否均匀

## 场景 C：结果归并

### 现象

```
分库分表后，一个查询路由到多个库多个表
需要把多个数据源的结果合并成一个结果集
涉及：排序、分页、聚合函数（COUNT/SUM/AVG）等
归并不当会导致内存溢出或结果错误
```

### 解决方案

```java
// 1. 结果归并引擎
@Service
public class MergeEngine {
    
    // 归并查询结果
    public MergedResult merge(List<QueryResult> queryResults, SQLStatement sqlStatement) {
        // 根据SQL类型选择归并策略
        if (sqlStatement.hasOrderBy()) {
            return new OrderByMergeResult(queryResults, sqlStatement.getOrderByItems());
        }
        
        if (sqlStatement.hasGroupBy()) {
            return new GroupByMergeResult(queryResults, sqlStatement.getGroupByItems());
        }
        
        if (sqlStatement.isAggregation()) {
            return new AggregationMergeResult(queryResults, sqlStatement.getAggregationItems());
        }
        
        // 默认：遍历归并（按顺序遍历所有结果集）
        return new IteratorMergeResult(queryResults);
    }
}

// 2. 排序归并（流式归并，避免全量加载到内存）
public class OrderByMergeResult implements MergedResult {
    private final PriorityQueue<QueryResult> queue;  // 最小堆
    private final List<OrderByItem> orderByItems;
    
    public OrderByMergeResult(List<QueryResult> results, List<OrderByItem> items) {
        this.orderByItems = items;
        this.queue = new PriorityQueue<>(this::compare);
        
        // 初始化：每个结果集取第一条记录放入堆
        for (QueryResult result : results) {
            if (result.next()) {
                queue.offer(result);
            }
        }
    }
    
    @Override
    public boolean next() {
        if (queue.isEmpty()) {
            return false;
        }
        
        // 取出最小的记录（堆顶）
        QueryResult current = queue.poll();
        currentRecord = current.getRow();
        
        // 如果该结果集还有下一条，继续放入堆
        if (current.next()) {
            queue.offer(current);
        }
        
        return true;
    }
    
    // 比较两个结果集的当前记录
    private int compare(QueryResult r1, QueryResult r2) {
        for (OrderByItem item : orderByItems) {
            Comparable v1 = r1.getValue(item.getColumn());
            Comparable v2 = r2.getValue(item.getColumn());
            int cmp = v1.compareTo(v2);
            if (cmp != 0) {
                return item.isAsc() ? cmp : -cmp;
            }
        }
        return 0;
    }
}

// 3. 分页归并（LIMIT/OFFSET）
public class LimitMergeResult implements MergedResult {
    private final MergedResult underlying;
    private final int offset;
    private final int rowCount;
    private int currentIndex = 0;
    
    public LimitMergeResult(MergedResult underlying, int offset, int rowCount) {
        this.underlying = underlying;
        this.offset = offset;
        this.rowCount = rowCount;
    }
    
    @Override
    public boolean next() {
        // 跳过 OFFSET 条记录
        while (currentIndex < offset) {
            if (!underlying.next()) {
                return false;
            }
            currentIndex++;
        }
        
        // 取 ROW_COUNT 条记录
        if (currentIndex >= offset + rowCount) {
            return false;
        }
        
        currentIndex++;
        return underlying.next();
    }
}
```

**归并优化**：
- **流式归并**：排序用最小堆，不需要全量加载（避免 OOM）
- **内存归并**：数据量小时可以用（更快）
- **分页优化**：每个分片执行 `LIMIT offset + rowCount`，减少数据传输

## 场景 D：分布式主键（雪花/号段）

### 现象

```
分库分表后，不能用数据库自增 ID（会重复）
需要全局唯一的分布式 ID
要求：趋势递增（对 B+ 树友好）、高性能、不依赖 DB
```

### 解决方案

```java
// 1. 雪花算法（Snowflake）实现
public class SnowflakeIdGenerator {
    // 起始时间戳（可以设置为项目启动时间）
    private static final long START_TIMESTAMP = 1704067200000L;  // 2024-01-01
    
    // 位数分配：符号位(1) + 时间戳(41) + 机器ID(10) + 序列号(12) = 64位
    private static final long SEQUENCE_BITS = 12L;
    private static final long WORKER_ID_BITS = 10L;
    
    private static final long MAX_WORKER_ID = ~(-1L << WORKER_ID_BITS);  // 1023
    private static final long MAX_SEQUENCE = ~(-1L << SEQUENCE_BITS);   // 4095
    
    private static final long WORKER_ID_SHIFT = SEQUENCE_BITS;          // 12
    private static final long TIMESTAMP_SHIFT = SEQUENCE_BITS + WORKER_ID_BITS;  // 22
    
    private final long workerId;        // 机器 ID（0-1023）
    private long sequence = 0L;         // 序列号
    private long lastTimestamp = -1L;   // 上次生成 ID 的时间戳
    
    public SnowflakeIdGenerator(long workerId) {
        if (workerId > MAX_WORKER_ID || workerId < 0) {
            throw new IllegalArgumentException("workerId 超出范围");
        }
        this.workerId = workerId;
    }
    
    // 生成下一个 ID（同步方法，线程安全）
    public synchronized long nextId() {
        long timestamp = timeGen();
        
        // 时钟回拨检查
        if (timestamp < lastTimestamp) {
            throw new RuntimeException("时钟回拨，拒绝生成 ID");
        }
        
        // 同一毫秒内
        if (timestamp == lastTimestamp) {
            sequence = (sequence + 1) & MAX_SEQUENCE;  // 序列号+1，超过最大值则归零
            if (sequence == 0) {
                // 序列号用尽，等待下一毫秒
                timestamp = waitNextMillis(lastTimestamp);
            }
        } else {
            // 新的毫秒，序列号重置
            sequence = 0;
        }
        
        lastTimestamp = timestamp;
        
        // 组装 ID：时间戳部分 + 机器ID部分 + 序列号部分
        return ((timestamp - START_TIMESTAMP) << TIMESTAMP_SHIFT)
                | (workerId << WORKER_ID_SHIFT)
                | sequence;
    }
    
    private long waitNextMillis(long lastTimestamp) {
        long timestamp = timeGen();
        while (timestamp <= lastTimestamp) {
            timestamp = timeGen();
        }
        return timestamp;
    }
    
    private long timeGen() {
        return System.currentTimeMillis();
    }
}

// 2. 号段模式（Leaf）实现
@Service
public class LeafSegmentIdGenerator {
    
    @Autowired
    private LeafAllocMapper leafAllocMapper;
    
    // 每个号段缓存：key=业务key，value=号段
    private final Map<String, SegmentBuffer> cache = new ConcurrentHashMap<>();
    
    // 获取下一个 ID（号段模式）
    public long nextId(String key) {
        SegmentBuffer buffer = cache.computeIfAbsent(key, k -> new SegmentBuffer(k));
        return buffer.nextId();
    }
    
    // 号段缓冲区
    private static class SegmentBuffer {
        private final String key;
        private volatile Segment currentSegment;
        private volatile boolean isLoadingNext = false;
        
        public SegmentBuffer(String key) {
            this.key = key;
            this.currentSegment = loadSegment(key, 0);  // 第一次加载，从0开始
        }
        
        public synchronized long nextId() {
            // 如果当前号段用完，切换到下一个号段
            if (currentSegment.isExhausted()) {
                // 加载新号段（实际项目会异步预加载）
                currentSegment = loadSegment(key, currentSegment.getMaxId());
            }
            
            return currentSegment.nextId();
        }
    }
    
    // 加载号段：从 DB 获取一段 ID
    private Segment loadSegment(String key, long fromId) {
        // 1. DB 原子更新：UPDATE leaf_alloc SET max_id = max_id + step WHERE key = ?
        leafAllocMapper.updateMaxId(key, 1000);  // step=1000
        
        // 2. 查询更新后的 max_id
        long maxId = leafAllocMapper.selectMaxId(key);
        
        return new Segment(fromId + 1, maxId);
    }
}

// 号段定义
private static class Segment {
    private final long minId;
    private final long maxId;
    private volatile long currentId;
    
    public Segment(long minId, long maxId) {
        this.minId = minId;
        this.maxId = maxId;
        this.currentId = minId - 1;
    }
    
    public synchronized long nextId() {
        if (currentId >= maxId) {
            throw new IllegalStateException("号段已用完");
        }
        return ++currentId;
    }
    
    public boolean isExhausted() {
        return currentId >= maxId;
    }
    
    public long getMaxId() {
        return maxId;
    }
}
```

**雪花 vs 号段**：
- **雪花**：本地生成，无网络开销，但依赖时钟（时钟回拨会失败）
- **号段**：依赖 DB，但更稳定，适合对时钟敏感的场景

## 核心参数估算

```
性能估算：
  雪花算法：本地生成，QPS 可达 400万/秒（受序列号位数限制）
  号段模式：DB 每次获取1000个号段，QPS 受 DB 限制（约 1000次/秒）
  
号段大小选择：
  step=1000：适合中等流量（1000 QPS 以下）
  step=10000：适合高流量，减少 DB 访问频率
  
雪花算法时钟回拨处理：
  回拨 < 5ms：等待
  回拨 5-100ms：告警，拒绝服务
  回拨 > 100ms：人工介入（可能系统时间被错误修改）
```

## 追问链

### 追问 1：为什么不用 UUID 做分布式 ID？

> "UUID 的问题：1. **无序**：B+ 树插入会导致大量页分裂，性能差。2. **字符串**：比 Long 类型大很多（36字节 vs 8字节），浪费存储和索引空间。3. **可读性差**：排查问题不方便。推荐用雪花算法（趋势递增、Long 类型、性能好）。"

### 追问 2：结果归并的流式排序怎么实现？

> "用**最小堆（PriorityQueue）**：1. 每个分片结果集取第一条记录放入堆。2. 弹出堆顶（最小值），就是当前最小的记录。3. 对该结果集取下一记录，继续放入堆。4. 重复直到堆空。时间复杂度 O(N log K)，N 是总记录数，K 是分片数。不需要全量加载到内存。"

### 追问 3：分片路由时，全路由怎么优化？

> "全路由（不带分片键查询）是性能杀手。优化：1. **业务层优化**：强制查询带分片键（如 queryByUserId）。2. **索引表**：对高频查询条件建索引表（如 t_order_user_index）。3. **ES 宽表**：复杂查询走 Elasticsearch。4. **接受全路由**：后台管理查询可以接受慢一点（但设置超时）。"

### 追问 4：号段模式 DB 成为瓶颈怎么办？

> "优化方案：1. **异步预加载**：号段用到 80% 时，后台线程异步加载下一个号段。2. **多 DB 实例**：不同业务 key 用不同 DB，分散压力。3. **双 Buffer**：两个号段交替使用，一个用完自动切换到另一个，无缝衔接。4. **本地缓存**：号段缓存在 JVM，减少 DB 访问。"

## 排查 Checklist

```
□ SQL 解析正确吗？ → 测试各种 SQL（子查询、JOIN、聚合）
□ 路由计算准确吗？ → 验证分片键哈希后的目标库表
□ 全路由有告警吗？ → 监控不带分片键的查询
□ 结果归并正确吗？ → 测试排序、分页、聚合函数
□ 雪花时钟回拨处理了吗？ → 回拨时拒绝服务 + 告警
□ 号段加载失败会重试吗？ → DB 连接失败有重试机制
□ 分片数合理吗？ → 单表控制在 500万行以内
□ 分布式主键唯一吗？ → workerId 不重复（用 IP + 端口 或 ZooKeeper）
```
