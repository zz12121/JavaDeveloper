# 网络模型

> K8s 网络不是「单机 Docker 网络」。它要解决的问题是：Pod 怎么互相访问？Service 怎么负载均衡？外部流量怎么进来？不理解网络模型，服务间调用就永远在碰运气。

---

## 这个问题为什么存在？

不用 K8s 时（纯 Docker），网络是这样的：

```
场景 A：容器怎么暴露服务？
  docker run -p 8080:8080 my-app
  → 端口冲突问题（100 个容器都要不同端口）
  → 前端怎么知道该连哪个端口？

场景 B：容器 A 怎么访问容器 B？
  docker network create my-net
  → 100 个容器，手动维护网络？
  → 容器 IP 变了，其他容器怎么感知？

场景 C：多节点通信
  Node A 上的容器 → Node B 上的容器
  → 需要手动配置路由表
  → 新增节点要更新所有节点路由
```

**K8s 网络要解决的核心问题**（K8s 网络的 4 个约定）：

1. **所有 Pod 能互相通信**（不管在哪个 Node）
2. **所有 Node 能和所有 Pod 通信**（Node → Pod 直通）
3. **Pod 看到的自己的 IP = 别人看到的 Pod IP**（没有 NAT）
4. **外部流量怎么进集群** → Service / Ingress

---

## 它是怎么解决问题的？

### K8s 网络模型（扁平网络）

```
传统网络（有 NAT）：
  Pod A (10.1.1.2) → (NAT) → Pod B (10.1.2.3)
  → Pod B 看到的是 Node IP，不是 Pod A 的 IP

K8s 网络（无 NAT，扁平）：
  Pod A (10.1.1.2) ──→ Pod B (10.1.2.3)
  → 直接通信，Pod IP 在全集群可路由
```

**实现方式**：K8s 自己不实现网络，它定义了**网络模型**，由 CNI 插件实现。

```
常见 CNI 插件：
  - Calico（BGP/IPIP 模式，性能好）
  - Flannel（VXLAN 模式，配置简单）
  - Cilium（eBPF 模式，性能最好，功能最强）
```

### Service：Pod 的负载均衡器

```
问题：
  Pod IP 是临时的（Pod 重建 = 新 IP）
  → 不能把 Pod IP 写死在配置文件里
```

**Service 的解决方案**：给一组 Pod 一个稳定的虚拟 IP（Cluster IP）和 DNS 名。

```yaml
apiVersion: v1
kind: Service
metadata:
  name: user-service
spec:
  selector:
    app: user-service    # 选择标签为 app=user-service 的 Pod
  ports:
    - port: 80           # Service 端口
      targetPort: 8080   # Pod 端口
  type: ClusterIP
```

**访问方式**：

```bash
# 集群内其他 Pod 访问 user-service：
curl http://user-service/      # DNS 解析 → Cluster IP → 负载均衡到某个 Pod

# Service 的 DNS 名格式：
<service-name>.<namespace>.svc.cluster.local
# 同 namespace 下可以简写：<service-name>
```

### Service 的三种类型

| 类型 | 访问方式 | 适用场景 |
|------|---------|---------|
| `ClusterIP`（默认） | 集群内访问（Cluster IP 或 DNS 名） | 微服务间调用 |
| `NodePort` | `<NodeIP>:<NodePort>`（外部可访问） | 测试环境临时暴露 |
| `LoadBalancer` | 云厂商负载均衡器 IP | 生产环境外部暴露 |

### Ingress：HTTP/HTTPS 路由

```
问题：
  有 10 个服务要暴露到外部
  用 LoadBalancer → 要 10 个 LB，贵！
```

**Ingress 的解决方案**：一个 LB + 基于域名/路径的路由。

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
spec:
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /user
            pathType: Prefix
            backend:
              service:
                name: user-service
                port:
                  number: 80
          - path: /order
            pathType: Prefix
            backend:
              service:
                name: order-service
                port:
                  number: 80
```

**效果**：外部请求 `http://api.example.com/user/xxx` → 路由到 `user-service`，`/order/xxx` → 路由到 `order-service`。

## 它和相似方案的本质区别是什么？

### K8s Service vs Docker Compose 网络

| 维度 | K8s Service | Docker Compose 网络 |
|------|---------------|-------------------|
| 负载均衡 | ✓（自动分发到多个 Pod） | ✗（只能 Round Robin DNS） |
| 服务发现 | DNS（CoreDNS） | DNS（`servicename`） |
| 外部暴露 | Ingress / LoadBalancer | 端口映射（`ports:`） |
| 跨 Node 通信 | ✓（CNI 插件实现） | ✗（单机） |

### K8s Ingress vs Nginx 反向代理（独立部署）

| 维度 | Ingress（K8s 管理） | 独立 Nginx |
|------|---------------------|------------|
| 配置方式 | YAML（kubectl apply） | 手动改 nginx.conf |
| 动态更新 | ✓（Ingress Controller 自动 reload） | ✗（需要手动 reload） |
| SSL 证书管理 | ✓（整合 Cert-Manager，自动续期） | ✗（手动更新证书） |
| 适合场景 | K8s 集群内服务暴露 | 非 K8s 环境 / 复杂路由规则 |

## 正确使用方式

### 1. 用 Headless Service 做 Pod 级发现

```
场景：
  需要直连某个 Pod（如 StatefulSet 的有状态应用）
  → ClusterIP Service 会负载均衡，不知道具体连了哪个 Pod
```

**Headless Service**（ClusterIP=None）：

```yaml
apiVersion: v1
kind: Service
metadata:
  name: mysql-cluster
spec:
  clusterIP: None     # Headless！
  selector:
    app: mysql
```

**效果**：DNS 查询 `mysql-cluster` 返回**所有 Pod IP**（不是单个 VIP）。

```bash
# DNS 查询返回多个 A 记录
nslookup mysql-cluster
# Server:    10.96.0.10
# Address:  10.96.0.10#53

# Name:  mysql-cluster.default.svc.cluster.local
# Address: 10.1.1.2
# Name:  mysql-cluster.default.svc.cluster.local
# Address: 10.1.1.3
```

### 2. 用 `externalName` 访问集群外服务

```
场景：
  K8s 内的服务要访问云数据库（RDS）
  → 不想把数据库连接信息写死 IP
```

**ExternalName Service**：

```yaml
apiVersion: v1
kind: Service
metadata:
  name: cloud-mysql
spec:
  type: ExternalName
  externalName: my-rds.xxx.rds.amazonaws.com
```

**效果**：Pod 内访问 `http://cloud-mysql:3306` → 自动解析到 `my-rds.xxx.rds.amazonaws.com`。

### 3. 用 NetworkPolicy 控制流量

```
场景：
  前端 Pod 只需要访问后端 Pod 的 8080 端口
  → 默认所有 Pod 之间都能互相访问（不安全！）
```

**NetworkPolicy**（需要 CNI 插件支持，如 Calico）：

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-backend
spec:
  podSelector:
    matchLabels:
      app: backend     # 这条规则应用到 label 为 app=backend 的 Pod
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: frontend   # 只允许 frontend Pod 访问
      ports:
        - protocol: TCP
          port: 8080
```

## 边界情况和坑

### 1. Service 的 IP 是虚拟的，无法 ping

```
现象：
  kubectl get svc user-service
  → ClusterIP: 10.96.1.23

  ping 10.96.1.23
  → 超时（ping 不通）
```

**原因**：Service 的 IP 是 `iptables` / `IPVS` 规则，不是真实的网络接口。

**验证 Service 是否正常**：

```bash
# 用 telnet / curl 测试（TCP 连接）
curl http://10.96.1.23

# 查看 iptables 规则（kube-proxy 所为）
iptables -t nat -L KUBE-SERVICES
```

### 2. `NodePort` 端口范围限制

```
现象：
  type: NodePort
  ports:
    - port: 80
      targetPort: 8080
      nodePort: 80    # ❌ 报错：端口超出范围
```

**原因**：`nodePort` 默认范围是 `30000-32767`。

**解决**：

```yaml
# 方法 1：不指定 nodePort，让 K8s 自动分配
ports:
  - port: 80
    targetPort: 8080
    # 不写 nodePort，K8s 会从 30000-32767 中分配一个

# 方法 2：修改 API Server 启动参数（不推荐）
# --service-node-port-range=80-32767
```

### 3. Ingress 需要 Ingress Controller 才工作

```
现象：
  创建了 Ingress，但访问不通
```

**原因**：Ingress 只是配置，`Ingress Controller`（如 Nginx Ingress Controller）才是真正处理流量的组件。

**解决**：先部署 Ingress Controller。

```bash
# 安装 Nginx Ingress Controller
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/main/deploy/static/provider/cloud/deploy.yaml
```

### 4. DNS 缓存导致服务发现延迟

```
现象：
  Pod 重启后 IP 变了
  → 其他 Pod 还是访问旧 IP（DNS 缓存）
```

**解决**：

- 应用层：不用 IP，用 Service DNS 名（DNS 解析是实时的）
- JVM：配置 `-Dsun.net.inetaddr.ttl=0`（禁用 DNS 缓存，不推荐）
- 更好的方案：用 Service Mesh（如 Istio），不依赖 DNS

## 我的理解

K8s 网络最容易被误解的是**「Service 是负载均衡器，不是代理」**。

```
负载均衡器 vs 代理：
  - 负载均衡器：转发流量（L4/L7），客户端感知不到
  - 代理：代表客户端发起请求（如 Nginx 反向代理）

K8s Service 是「负载均衡器」（L4，iptables/IPVS 实现）
Ingress 是「L7 代理」（Ingress Controller 实现，通常是 Nginx/Envoy）
```

**面试高频考点**：

1. K8s 网络的 4 个约定 —— 了解即可
2. Service 的作用和三种类型 —— 必问
3. Ingress 和 Service 的区别 —— Ingress 是 L7（HTTP 路由），Service 是 L4（TCP/UDP 负载均衡）
4. Pod 之间怎么通信？—— CNI 插件实现扁平网络
5. Headless Service 是什么？—— 返回多个 Pod IP，用于有状态应用

*K8s 网络是「分布式系统的网络基础」。不理解它，微服务调用就是「能通但不知道为什么通，不通也不知道为什么不通」。*
