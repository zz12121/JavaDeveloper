# 操作系统基础

> **核心认知**：操作系统是连接硬件和软件的桥梁。理解操作系统不只是"会用 Linux 命令"，而是理解进程管理、内存管理、文件系统、IO 模型这些底层机制——它们是 JVM、并发编程、网络编程的根基。

---

## 1. 操作系统的定义与职责

**操作系统（OS）是管理计算机硬件和软件资源的系统软件**，为应用程序提供运行环境。

```
┌─────────────────────────────┐
│     应用程序（浏览器、JVM…）  │
├─────────────────────────────┤  ← 系统调用接口（API）
│        操作系统内核           │
│  ┌─────────┐ ┌─────────┐   │
│  │ 进程管理 │ │ 内存管理 │   │
│  ├─────────┤ ├─────────┤   │
│  │ 文件系统 │ │ IO管理  │   │
│  └─────────┘ └─────────┘   │
├─────────────────────────────┤  ← 硬件抽象层
│  CPU / 内存 / 磁盘 / 网卡    │
└─────────────────────────────┘
```

**操作系统的核心职责**：
1. **进程管理**：创建/销毁进程、调度、进程间通信
2. **内存管理**：虚拟内存、页表、内存分配与回收
3. **文件系统**：文件的存储、组织、访问控制
4. **IO 管理**：管理输入输出设备、提供统一的 IO 接口
5. **安全与权限**：用户认证、文件权限、进程隔离

## 2. 内核态与用户态

### 2.1 为什么需要区分？

操作系统需要保护自己不被应用程序破坏。比如应用程序不能直接读写磁盘、不能访问其他进程的内存。

```
特权指令（只能内核态执行）：
  - IO 操作（读写磁盘、网卡）
  - 内存管理（修改页表）
  - 进程管理（创建/销毁进程）
  - 访问硬件设备

非特权指令（用户态可执行）：
  - 算术运算
  - 访问用户空间内存
  - 函数调用
```

### 2.2 切换方式

| 方式 | 说明 | 举例 |
|------|------|------|
| 系统调用（System Call） | 应用程序主动请求内核服务 | read()、write()、mmap() |
| 异常（Exception） | 程序执行出错（除零、缺页） | Page Fault |
| 中断（Interrupt） | 外部硬件通知 CPU | 网卡收到数据、定时器中断 |

### 2.3 常见系统调用

```c
// 进程管理
fork()        // 创建子进程
exec()        // 执行新程序
wait()        // 等待子进程结束
exit()        // 终止进程

// 文件操作
open()        // 打开文件
read()        // 读文件
write()       // 写文件
close()       // 关闭文件
stat()        // 获取文件信息

// 内存管理
brk()         // 调整堆大小
mmap()        // 内存映射
munmap()      // 取消映射

// 网络通信
socket()      // 创建套接字
bind()        // 绑定地址
listen()      // 监听
accept()      // 接受连接
connect()     // 发起连接
```

## 3. 中断机制

### 3.1 中断的类型

```
          中断
         ┌────┴────┐
    外部中断      内部中断（异常）
    (硬件)       ┌───┴───┐
    网卡、键盘   故障    陷阱
    定时器       缺页    系统调用
```

| 类型 | 来源 | 是否可恢复 | 举例 |
|------|------|-----------|------|
| 外部中断（硬件中断） | 外部设备 | 是 | 网卡收数据、键盘输入、定时器 |
| 故障（Fault） | CPU 执行指令时 | 是（修复后重新执行） | 缺页中断（Page Fault） |
| 陷阱（Trap） | 程序有意触发 | 是 | 系统调用（int 0x80） |
| 终止（Abort） | 不可恢复的错误 | 否 | 硬件故障、除零错误 |

### 3.2 中断处理流程

```
1. 硬件保存当前执行状态（PC、寄存器等）
2. 根据中断号查找中断向量表（IVT）
3. 跳转到对应的中断处理程序
4. 执行中断处理
5. 恢复之前的状态，继续执行
```

## 5. 操作系统与 Java 的关系

```
操作系统                    Java / JVM
─────────                ─────────────
进程                     JVM 进程
线程                     Java 线程（Thread）
虚拟内存                  JVM 内存模型（堆/栈/方法区）
页面置换                   GC（垃圾回收）
文件描述符                 FileDescriptor / Channel
系统调用                  native 方法
零拷贝（sendfile）         FileChannel.transferTo()
epoll                    Selector / Netty EventLoop
信号                     ShutdownHook
```

**理解操作系统的重要性**：
- **JVM 内存模型** = 操作系统虚拟内存的子集（JVM 堆在用户空间中分配）
- **Java 线程** = 操作系统线程（1:1 模型，每个 Java 线程对应一个 OS 线程）
- **NIO** = 对操作系统 epoll/kqueue 的封装
- **GC** = 操作系统页面置换思想的延伸
- **synchronized** = 操作系统互斥锁 + 条件变量

## 6. 面试高频问题

### Q1: 什么是系统调用？和普通函数调用有什么区别？

**答**：系统调用是应用程序请求操作系统内核服务的接口。区别：普通函数调用在用户态完成，不涉及特权操作；系统调用需要从用户态切换到内核态，执行完成后切回用户态。系统调用有性能开销（上下文切换），因此频繁的 IO 操作应该批量处理（如 BufferedInputStream 减少系统调用次数）。

### Q2: 32 位系统和 64 位系统的区别？

**答**：
- **寻址空间**：32 位最多 4GB（2^32），64 位理论 16EB（2^64），实际受限于硬件
- **寄存器宽度**：64 位可以一次处理 64 位数据
- **JVM 影响**：32 位 JVM 堆上限约 2GB；64 位 JVM 可以使用更大的堆（-Xmx），但对象指针从 4 字节变 8 字节（可用 CompressedOops 压缩到 4 字节）

### Q3: 为什么 Java 中频繁读写文件要用 BufferedInputStream/BufferedOutputStream？

**答**：因为每次 read/write 系统调用都有用户态↔内核态切换的开销。BufferedInputStream 内部维护一个缓冲区（默认 8KB），一次系统调用读取一批数据到缓冲区，后续的 read() 直接从缓冲区返回，减少系统调用次数。这是操作系统层面的优化思想在 Java IO 中的体现。
