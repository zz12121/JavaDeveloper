## 知识链提示

```
类加载机制
  → [[03_JVM/03_类加载机制/类加载机制]]
    → 类加载过程（7步）
      → 加载：从 .class 文件读字节码 → 静态存储结构 → 方法区（元空间）
      → 验证：魔数校验 / 字节码校验 / 符号引用校验
      → 准备：static 变量赋初始值（0 / false / null）
        → static int x = 123 → 准备阶段 x = 0，初始化阶段 x = 123
      → 解析：符号引用 → 直接引用
      → 初始化：执行 <clinit>() 方法（static 赋值 + static 块）
      → 使用 + 卸载
    → 双亲委派模型
      → Bootstrap（C++ 实现，加载 rt.jar）
        → 父亲：无（最顶层）
      → Extension（加载 ext 目录）
        → 父亲：Bootstrap
      → Application（加载 classpath）
        → 父亲：Extension
      → 自定义 ClassLoader
        → 父亲：Application
    → 双亲委派的工作流程
      → 收到加载请求 → 先委派父亲加载
        → 父亲加载成功 → 返回
        → 父亲加载失败 → 自己尝试加载
    → 破坏双亲委派
      → SPI 场景（JDBC）：父加载器需要加载子加载器路径的类
        → 线程上下文类加载器（Thread Context ClassLoader）
      → Tomcat：WebAppClassLoader 先自己加载，再委派父亲
      → OSGi：网状类加载
```

## 核心追问

| 追问 | 考点 |
|------|------|
| `loadClass` 和 `findClass` 的区别？ | loadClass 实现双亲委派逻辑，findClass 是自定义加载逻辑 |
| 一个类能被加载多次吗？ | 不能，同一个 ClassLoader 命名空间内唯一 |
| 两个不同 ClassLoader 加载的同一个类，equals 返回 true 吗？ | false（不同 Class 对象） |
| 什么是「热替换」？怎么实现？ | 自定义 ClassLoader，每次加载新版本 |

---

## 参考答案要点

**双亲委派的核心目的**：防止核心类被篡改（Bootstrap 加载的 `java.lang.Object` 不会被用户自定义类覆盖）。

**破坏双亲委派的经典场景**：
- JDBC：Bootstrap 加载的 `DriverManager` 需要加载 `com.mysql.cj.jdbc.Driver`（在 classpath）→ 用线程上下文类加载器
- Tomcat：Web 应用隔离，每个 WebApp 用自己的 ClassLoader

**自定义 ClassLoader**：重写 `findClass()`，不要重写 `loadClass()`（否则双亲委派被破坏）。
