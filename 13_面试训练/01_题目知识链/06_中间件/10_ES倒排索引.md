# Elasticsearch 的倒排索引是什么？

> ⚠️ **先盲答**：Elasticsearch 是做什么的？它和 MySQL 的全文索引有什么区别？

---

## 盲答引导

1. 正排索引（Forward Index）和倒排索引（Inverted Index）的区别是什么？
2. 倒排索引的核心结构是什么？—— 词典（Term Dictionary）+ 倒排列表（Posting List）
3. 分词（Analyzer）是什么？为什么中文分词比英文难？
4. Elasticsearch 和 MySQL 全文索引各自的适用场景是什么？

---

## 知识链提示

```
Elasticsearch 倒排索引
  → [[Elasticsearch倒排索引]]
    → 正排 vs 倒排
      → 正排（Forward Index）：文档 → 包含哪些词（MySQL 索引的本质）
      → 倒排（Inverted Index）：词 → 出现在哪些文档（搜索引擎的本质）
        → 查询「Java 并发」→ 倒排索引：找到含「Java」的文档 + 含「并发」的文档 → 取交集
    → 倒排索引结构
      → Term Dictionary（词典）：所有分词后的 term，按字典序排列（有序 → 可二分）
      → Posting List（倒排列表）：每个 term 对应的「文档ID + 词频 + 位置」
        → Posting List 是有序数组，支持求交/求并
    → 分词器（Analyzer）
      → 分词（Tokenization）：一段文本 → 多个 token
      → 英文：空格分词，简单
      → 中文：需要词典（jieba / IK）：「我爱你」→ 「我」「爱」「你」
      → 分词质量直接影响搜索效果
    → 分布式架构
      → [[Elasticsearch集群架构]] → 多个分片（Shard）+ 副本（Replica）
        → 每个分片是一个 Lucene 索引
        → 主分片负责写，副本分片负责读
```

---

## 核心追问

| 追问 | 考点 |
|------|------|
| 为什么倒排索引要做分词？ | 不分词只能按完整词匹配，搜索效果差 |
| Elasticsearch 和 MySQL 怎么选？ | 全文搜索 / 聚合分析 / 海量数据 → ES；强事务 / 精确查询 → MySQL |
| 分片数量定了能改吗？ | 能改，但需要 reindex（成本高）|
| 为什么要设置副本分片？ | 高可用 + 提升读吞吐 |

---

## 参考答案要点

**倒排索引的核心**：把「文档 → 词」反过来，变成「词 → 文档」，查询时通过词的 Posting List 求交集。

**中文分词挑战**：英文按空格分词，中文需要词典 + 统计模型（CRF / HMM），分词不准则搜索效果差。

**ES vs MySQL 选型**：
- ES：全文搜索 / 海量数据 / 复杂聚合 / 日志分析
- MySQL：强事务 / 精确查询 / 数据量中等
