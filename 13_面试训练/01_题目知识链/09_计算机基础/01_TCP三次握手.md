**三次握手过程**：

```
客户端                                服务器
  CLOSED                              LISTEN
    |                                    |
    |--------- SYN (seq=x) ------------>|
    |      (请求建立连接，带初始序列号)       |
    |<-------- SYN-ACK (seq=y, ack=x+1) --|
    |   (确认收到，也发自己的初始序列号)       |
    |--------- ACK (ack=y+1) --------->|
    |                                    |
ESTABLISHED                        ESTABLISHED
  (可开始传数据)                        (可开始传数据)
```

**为什么不是两次？——「已失效的连接请求」问题**：

```
场景：客户端发了一个连接请求，网络延迟，客户端超时重发
旧请求：在链路上「迷途」，延迟很久才到达服务器
如果只有两次握手：
  服务器收到旧请求 → 发 SYN-ACK → 进入 ESTABLISHED
  但客户端此时并没有真正的连接请求（旧请求已经超时废弃）
  → 服务器白白浪费资源等待一个不存在的连接

三次握手：第三次 ACK 从客户端发出
  如果客户端没有发真正连接请求，它不会发第三次 ACK
  服务器收不到 ACK，不会进入 ESTABLISHED
```

**SYN Flood 攻击与防御**：

```
攻击：疯狂发 SYN，不回 ACK → 服务器半连接队列爆满 → 无法服务正常用户

防御1：SYN Cookie
  - 服务器不分配资源，把状态编码进 SYN-ACK 的序列号
  - 收到第三次 ACK 时，解码序列号恢复状态
  - 不需要半连接队列

防御2：SYN Cache
  - 半连接保存在哈希表里（内存小），超时清理
```

**第三次 ACK 丢了会怎样？**

```
客户端：发完 ACK 就进入 ESTABLISHED，开始发数据
服务器：没收到 ACK，重传 SYN-ACK（最多重传 tcp_synack_retries 次，默认5）
客户端发的数据包里带 ACK → 服务器收到后进入 ESTABLISHED

结论：能建立成功！第三次握手 ACK 丢失不影响
（但服务器会重传 SYN-ACK，浪费带宽）
```
