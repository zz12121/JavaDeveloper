# 题目：微服务版本管理和接口兼容性是怎么保证的？

## 盲答引导

1. 什么是「破坏性变更」（Breaking Change）？删除一个字段算不算？
2. REST API 里，URL 路径变了（如 `/users` → `/accounts`）算破坏性变更吗？
3. 为什么说「新增字段不算破坏性变更，删除/修改字段才算」？
4. JSON Schema / OpenAPI 的向后兼容（Backward Compatible）规则是什么？
5. 在微服务架构里，如果服务 A 依赖服务 B，怎么保证 B 的版本升级不会让 A 挂掉？

## 参考要点（盲答后再看）


**破坏性变更类型**：

```
破坏性（Breaking Change）：
  - 删除字段 / 枚举值
  - 修改字段类型（String → Integer）
  - 修改字段含义（字段名不变，值变了）
  - 删除接口（删除 GET /users）
  - 修改必填参数（可选 → 必填）
  - 修改响应结构（删字段）

非破坏性（Backward Compatible）：
  - 新增字段（客户端忽略未知字段 → 不崩溃）
  - 新增可选参数
  - 新增接口
```

**版本管理策略**：

```bash
# 方式1：URL 路径（最直观）
GET /v1/users
GET /v2/users

# 方式2：Header
GET /users
Accept: application/vnd.api.v1+json

# 方式3：Query 参数（不推荐，污染 URL）
GET /users?api-version=1
```

**契约测试（Pact / Spring Cloud Contract）**：

```
Consumer（消费者，如前端/调用方）：
  - 定义「我需要什么数据」（期望的响应格式）
  - Pact 生成契约文件

Provider（提供者，如后端服务）：
  - 从 Pact 文件拉取契约
  - 验证「我返回的数据是否满足契约」
  - 如果 Provider 改了返回格式 → Pact 测试失败 → 强制先通知 Consumer

对比：
  集成测试：整个系统跑一遍（Consumer + Provider + DB 都启动）
  契约测试：Consumer 和 Provider 独立验证，不需要启动对方
```

**字段重命名兼容迁移方案**：

```java
// 第一步：Provider 同时返回新旧两个字段（共存期）
@Data
public class UserDTO {
    private String username;    // 旧字段，Consumer 还在用
    private String accountName;  // 新字段，v2 Consumer 开始用
}
// 第二步：等所有 Consumer 都切换到 v2
// 第三步：v3 中删除 username 字段

// API 演进黄金法则：一次只加，不删不改
```

**灰度 + 版本策略**：

```
A/B 测试 + 版本共存：
  - 5% 流量切到 v2（新版本）
  - 95% 流量留在 v1（旧版本）
  - 观察 v2 错误率
  - 逐步扩大 v2 比例 → 100% → v1 下线
```
