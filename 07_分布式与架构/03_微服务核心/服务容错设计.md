# 服务容错设计

## 这个问题为什么存在？

> 微服务调用链路长（订单→库存→支付→通知），一个服务故障可能导致**所有上游服务阻塞**，最终**整个系统雪崩**。问题是：**怎么防止局部故障扩散？**

没有容错设计，你会遇到：
- 支付服务响应慢，订单服务所有线程都阻塞在调用支付
- 订单服务线程池耗尽，连创建订单的接口都不可用
- 最终整个系统崩溃

---

## 它是怎么解决问题的？

### 五大核心手段

```
手段1：熔断（Circuit Breaker）
  → 下游故障率超阈值，快速失败，不发起调用

手段2：降级（Degrade）
  → 非核心功能关闭，保核心功能可用

手段3：隔离（Bulkhead）
  → 不同服务用不同线程池，一个挂了不影响其他

手段4：超时（Timeout）
  → 调用超时，释放资源，快速失败

手段5：重试（Retry）
  → 网络抖动导致失败，重试几次（但要防重试风暴）
```

### 熔断器：三种状态

```
          故障率 > 50%
          ┌──────────────┐
          ▼              │
  CLOSED ──────→ OPEN ──→ HALF_OPEN
    │                      │
    │   成功数 > 阈值      │
    └──────────────────────┘
  正常         熔断          半开（放一个请求试试）
```

```java
// Resilience4j熔断器（简化逻辑）
public class CircuitBreaker {
    private State state = State.CLOSED;
    private int failureCount = 0;
    private final int failureThreshold = 50;  // 50%失败率就熔断
    private final int successThreshold = 10;   // 连续10次成功就关闭熔断

    public <T> T execute(Supplier<T> supplier) {
        if (state == State.OPEN) {
            throw new CircuitBreakerOpenException("熔断中，快速失败");
        }

        if (state == State.HALF_OPEN) {
            // 半开状态：放一个请求试试
            try {
                T result = supplier.get();
                if (++successCount >= successThreshold) {
                    state = State.CLOSED;  // 成功够了，关闭熔断
                }
                return result;
            } catch (Exception e) {
                state = State.OPEN;  // 还是失败，继续熔断
                throw e;
            }
        }

        // CLOSED状态：正常调用，但记录失败率
        try {
            T result = supplier.get();
            return result;
        } catch (Exception e) {
            if (++failureCount >= failureThreshold) {
                state = State.OPEN;  // 失败率超阈值，熔断
            }
            throw e;
        }
    }
}
```

### 舱壁模式（Bulkhead）：线程池隔离

```
不隔离：
  OrderService的线程池（200个线程）
    → 调用PaymentService慢
    → 200个线程全部阻塞
    → OrderService整个不可用

隔离后：
  OrderService的线程池（200个线程）
    → 调用PaymentService用独立线程池（20个线程）
    → 就算PaymentService全部慢，也只占用20个线程
    → 剩余180个线程还能处理其他请求
```

```java
// Resilience4j线程池隔离（简化）
public class Bulkhead {
    private final Semaphore semaphore;
    private final int maxConcurrentCalls;

    public <T> T execute(Supplier<T> supplier) {
        if (!semaphore.tryAcquire()) {
            throw new BulkheadFullException("舱壁满了，快速失败");
        }
        try {
            return supplier.get();
        } finally {
            semaphore.release();
        }
    }
}
```

## 它和相似方案的本质区别是什么？

| 手段 | 目的 | 触发条件 | 处理方式 |
|---|---|---|---|
| **熔断** | 保护调用方 | 下游故障率超阈值 | 快速失败，不发起调用 |
| **降级** | 保证核心功能 | 系统资源不足 | 关闭非核心功能 |
| **隔离** | 防止故障扩散 | 下游慢或故障 | 独立线程池，不扩散 |
| **超时** | 防止资源耗尽 | 调用超时 | 释放资源，快速失败 |
| **重试** | 容忍网络抖动 | 网络临时故障 | 重试几次（指数退避） |

**本质区别**：
- **熔断**："下游挂了，我不调了"（保护调用方）
- **降级**："资源不够，我关掉非核心功能"（保核心）
- **隔离**："你挂了，但不影响我其他功能"（故障不扩散）

## 正确使用方式

### Resilience4j：熔断 + 降级

```java
// 1. 配置熔断器
CircuitBreakerConfig cbConfig = CircuitBreakerConfig.custom()
        .failureRateThreshold(50)          // 失败率50%就熔断
        .waitDurationInOpenState(Duration.ofMillis(10000))  // 熔断10秒
        .slidingWindowSize(100)            // 统计最近100次调用
        .build();
CircuitBreaker circuitBreaker = CircuitBreaker.of("payment", cbConfig);

// 2. 配置超时
TimeLimiterConfig tlConfig = TimeLimiterConfig.custom()
        .timeoutDuration(Duration.ofSeconds(3))  // 3秒超时
        .build();
TimeLimiter timeLimiter = TimeLimiter.of("payment", tlConfig);

// 3. 组合使用：熔断 + 超时 + 降级
Supplier<String> supplier = () -> paymentService.call();

// 熔断保护
Supplier<String> decorated = CircuitBreaker
        .decorateSupplier(circuitBreaker, supplier);

// 超时保护
decorated = TimeLimiter.decorateSupplier(timeLimiter, 
        () -> CompletableFuture.supplyAsync(supplier::get).get());

// 降级：熔断或超时后，返回默认值
decorated = decorateSupplier(decorated, throwable -> {
    log.warn("支付服务不可用，降级处理", throwable);
    return "fallback_result";  // 返回默认值，或抛业务异常
});
```

### Spring Cloud：注解方式

```java
@RestController
public class OrderController {

    // 熔断 + 降级
    @CircuitBreaker(name = "payment", fallbackMethod = "fallback")
    @GetMapping("/order/{id}")
    public Order getOrder(@PathVariable Long id) {
        return paymentService.call();  // 可能抛出异常
    }

    // 降级方法（签名要一样，多一个Throwable参数）
    public Order fallback(Long id, Throwable t) {
        log.warn("支付服务不可用，降级返回", t);
        return Order.builder().id(id).status("PAYMENT_UNAVAILABLE").build();
    }
}
```

## 边界情况和坑

### 坑1：熔断误判（慢调用也算失败）

```
场景：支付服务响应慢（5秒），但不报错
结果：熔断器统计"慢调用"为失败，触发熔断
     但实际支付服务是可用的，只是慢
```

**解决方案**：
1. **配慢调用阈值**（`slowCallDurationThreshold`），慢调用才统计
2. **慢调用比例超阈值才熔断**（`slowCallRateThreshold`）

```java
CircuitBreakerConfig.custom()
        .slowCallDurationThreshold(Duration.ofSeconds(2))  // 超过2秒算慢调用
        .slowCallRateThreshold(50)        // 慢调用比例超50%才熔断
```

### 坑2：重试风暴（Retry Storm）

```
场景：订单服务有3个实例，每个都配了重试3次
      支付服务挂了
      结果：3个实例 × 3次重试 = 9次调用 → 支付服务更挂了
```

**解决方案**：
1. **重试 + 熔断组合**（熔断打开了，就不重试了）
2. **重试次数 × 实例数 < 下游承受极限**
3. **指数退避**（重试间隔越来越长）

### 坑3：超时设置不合理

```
场景1：超时设太短（1秒），实际服务需要2秒
结果：每次都超时，服务永远调不通

场景2：超时设太长（30秒），服务响应慢
结果：线程全部阻塞，资源耗尽
```

**解决方案**：
1. **超时时间 = P95响应时间 × 2**（留有余地）
2. **熔断 + 超时组合**（超时 frequent，触发熔断）

## 我的理解

服务容错的核心是**防止局部故障扩散成全局故障**。

五大手段各有侧重：
- **熔断**：保护调用方，下游挂了就不调了
- **降级**：保核心功能，关闭非核心功能
- **隔离**：故障不扩散，用独立线程池
- **超时**：释放资源，不无限等待
- **重试**：容忍网络抖动，但要防重试风暴

实际生产中，**Resilience4j** 是主流选择（Hystrix已停止维护）。

配置原则：
1. **熔断 + 降级必须组合**（熔断后要有降级逻辑）
2. **超时 + 重试必须组合**（超时后重试，但要指数退避）
3. **隔离是最后手段**（线程池隔离成本高，优先考虑熔断）

**面试最高频追问**：
1. "熔断的三种状态？" → CLOSED（正常）、OPEN（熔断）、HALF_OPEN（试试能不能恢复）
2. "熔断和降级的区别？" → 熔断是"下游挂了，我不调了"；降级是"资源不够，我关掉非核心功能"
3. "怎么防止重试风暴？" → 熔断 + 指数退避 + 限制重试次数

---

*服务容错不是"防止故障"，而是"故障了，系统还能撑住"。*
