# 场景题 — 五步法专项训练

> 框架：澄清需求 → 技术选型 → 数据设计 → 接口设计 → 异常处理

---

## 一、场景题的本质

```
场景题 = 给你一个业务需求，问你怎么设计

区别于系统设计题：
- 系统设计：设计一个完整系统（秒杀/短链）
- 场景题：解决一个具体问题（幂等/缓存一致/分库分表）

面试官在看你：
1. 会不会拆解问题
2. 懂不懂技术选型
3. 能不能权衡取舍
4. 有没有实战经验
```

---

## 二、五步法详解

### 第一步：澄清需求（边界/限制）

```
这是最容易忽略但最重要的一步。

拿到场景题，先问清楚：

必问清单：
1. QPS/并发量多大？（100/1000/10万）
2. 数据规模多大？（1万/100万/10亿）
3. 一致性要求？（强一致/最终一致）
4. 可用性要求？（99.9%/99.99%）
5. 技术栈限制？（只能用XX技术）
6. 时间/成本限制？

技巧：
「我确认一下，这个场景的QPS大概是多少？对一致性有什么要求？」
→ 显得你专业，也帮你自己理清思路。
```

### 第二步：技术选型（用什么技术）

```
根据需求，确定用哪些技术。

选型思路：
1. 需求 → 关键词
2. 关键词 → 技术方案
3. 方案对比 → 最终选择

常用技术选型对照：

| 需求 | 技术方案 |
|------|---------|
| 高并发读 | Redis缓存、CDN、读写分离 |
| 高并发写 | 分库分表、消息队列削峰、异步处理 |
| 数据一致性 | 分布式事务、消息队列、本地消息表 |
| 数据可靠性 | 主从复制、多副本、持久化 |
| 接口幂等 | 唯一键、Token、分布式锁 |
| 限流防刷 | 令牌桶、滑动窗口、Redis计数 |
| 服务解耦 | 消息队列（Kafka/RabbitMQ） |
| 服务高可用 | 熔断降级、多副本、故障转移 |
```

### 第三步：数据设计（怎么存）

```
数据怎么存，往往是场景题的核心。

设计维度：
1. 选什么数据库？（关系型/NoSQL/搜索引擎）
2. 表结构怎么设计？（字段/索引/分库分表）
3. 缓存怎么设计？（Cache Aside/Read Through/Write Through）
4. 消息怎么存？（消息持久化/死信队列）

关键问题：
- 数据量预估 → 决定是否需要分库分表
- 查询模式 → 决定索引设计
- 一致性要求 → 决定缓存策略
```

### 第四步：接口设计（怎么调）

```
接口怎么设计，决定了系统的交互方式。

设计维度：
1. 同步 vs 异步？
2. REST vs RPC？
3. 接口幂等怎么保证？
4. 超时/重试/熔断怎么处理？

模板：
```
接口名：POST /api/xxx
入参：xxx
出参：xxx
幂等：唯一Token
超时：X秒
重试：X次
降级：返回xxx
```
```

### 第五步：异常处理（出问题了怎么办）

```
任何方案都要考虑异常情况。

异常场景：
1. 依赖服务挂了怎么办？
2. 网络超时怎么办？
3. 数据不一致怎么办？
4. 重复请求怎么办？

常用策略：
- 熔断：服务挂了，快速失败，不拖累下游
- 降级：核心功能降级，返回兜底数据
- 重试：临时失败，等一等再试（注意幂等）
- 补偿：最终不一致，用定时任务/消息队列补偿
```

## 三、高频场景题详解

### 场景1：如何保证接口幂等性

```
【第一步：澄清需求】
"确认一下：
- 什么场景？（支付/下单/退款/回调）
- 是前端重试还是后端重试？
- 需要支持多长时间的幂等？"

【第二步：技术选型】
方案对比：
1. 唯一Token：前端生成，每次请求带Token
2. 数据库唯一键：利用唯一索引防重
3. 分布式锁：Redis SETNX + 唯一标识
4. 状态机：订单状态流转防重复操作

选型依据：
- HTTP接口：Token方案
- 写数据库：唯一键方案
- 分布式环境：分布式锁方案

【第三步：数据设计】
Token方案：
- 存储：Redis SET token -> value（请求参数hash），过期时间
- key设计： idempotent:{token}
- 过期时间：业务允许的重试时间窗口

唯一键方案：
- 表设计：唯一索引（business_id + idempotent_key）
- 业务ID：订单号/用户ID+操作类型+时间戳

【第四步：接口设计】
```
POST /api/order/create
Header: X-Idempotent-Token: {token}
Body: { userId, productId, amount }

幂等流程：
1. 检查Token是否存在Redis
2. 不存在 → 执行业务逻辑 → 写入Token → 返回成功
3. 存在 → 返回已处理
```
```

【第五步：异常处理】
- Token过期/Redis挂了 → 降级为数据库唯一键兜底
- 并发请求同一Token → Redis SETNX保证只有一个成功
- Token生成冲突 → UUID/v4，保证全局唯一
```

### 场景2：如何保证缓存与数据库一致性

```
【第一步：澄清需求】
"确认一下：
- 一致性要求多高？（强一致/最终一致）
- 缓存失效容忍多久？（秒级/分钟级）
- 是读多还是写多场景？"

【第二步：技术选型】
三种经典模式：
1. Cache Aside（最常用）：读时缓存，写时删缓存
2. Read Through：缓存自动加载
3. Write Through：写缓存时同步写数据库

Cache Aside详解：
- 读：先读缓存，命中返回；未命中读DB，写缓存，返回
- 写：先写DB，成功后删缓存

【第三步：数据设计】
缓存Key设计：
```
cache:{业务}:{ID}:{版本}
如：cache:user:10001:v1
```

过期时间设计：
- 热点数据：短过期（5-30分钟）
- 冷数据：长过期（1-24小时）
- 配合：LRU + 主动刷新

【第四步：异常处理（重点）】
问题1：先写DB还是先删缓存？
- 方案A：先删缓存，再写DB
  → 问题：删缓存后DB还没写完，读请求会打爆DB
- 方案B：先写DB，再删缓存（推荐）
  → 问题：删缓存失败，DB和缓存不一致
  → 解决：延迟双删（写完DB后，sleep一下再删一次）

问题2：删缓存失败怎么办？
→ 解决方案：
1. 异步重试：把删除任务丢进MQ，异步重试
2. 订阅binlog：用Canal监听DB变化，自动更新缓存
3. 定时任务：定时扫描对比，补偿不一致

【第五步：最终方案】
```
最佳实践：Cache Aside + 延迟双删 + MQ补偿

读流程：
1. Redis命中 → 直接返回
2. Redis未命中 → 读MySQL → 写Redis → 返回

写流程：
1. 写MySQL
2. 删除Redis缓存
3. 延迟N秒（如300ms）后，再删一次Redis
4. 如果删除失败 → 丢MQ重试
```
```

### 场景3：如何设计一个分库分表方案

```
【第一步：澄清需求】
"确认一下：
- 当前数据量多少？（100万/1亿/100亿）
- 预计增长量？（每月X万）
- 核心查询模式？（按用户查/按时间查/多维度查询）
- 迁移窗口？（能停机多久）"

【第二步：技术选型】
分库分表中间件：
1. ShardingSphere-JDBC：客户端分片，应用层引入Jar
2. ShardingSphere-Proxy：代理层，MySQL协议
3. MyCat：老牌方案，社区不活跃
4. DRDS：阿里云付费方案

选型：
- 小规模/快速上线 → ShardingSphere-JDBC
- 多语言/统一入口 → ShardingSphere-Proxy
- 有阿里云预算 → DRDS

【第三步：数据设计】
分片键选择原则：
1. 尽量选择查询条件中必带的字段
2. 字段值分布均匀（避免热点）
3. 不频繁更新（更新需要同步改多个分片）

常见分片策略：
- 哈希分片：user_id % 4，数据均匀
- 范围分片：create_time，按时间分片
- 复合分片：user_id哈希 + 时间范围

表结构设计：
- 每个分片表结构相同
- 唯一ID需要自己生成（Snowflake）
- 避免跨分片JOIN（用ES/异构索引表）

【第四步：接口设计】
跨分片查询解决方案：
1. 异构索引表：按查询维度再建一套索引表
   如：按商品ID分片，再建一张按用户ID的索引表
2. 外部搜索：Elasticsearch存储全量数据，支持复杂查询
3. 内存聚合：各分片查完，应用层聚合

【第五步：异常处理】
问题1：扩容怎么办？（原来4库扩到8库）
→ 方案：一致性哈希环 + 虚拟节点
→ 或者：提前规划好容量，一次性分够

问题2：跨分片事务怎么办？
→ 分布式事务：Seata AT/TCC
→ 或者：避免跨分片事务，从设计上规避

问题3：分片后分页怎么办？
→ 限制最大页数（如最多查1000条）
→ 或：ES替代深度分页
```
