# CAP定理与BASE理论

## 这个问题为什么存在？

> 分布式系统的本质是**网络分区不可避免**。交换机故障、机房光纤被挖断、节点宕机，这些都是常态而非异常。问题是：**当网络分区发生时，你是保一致性还是保可用性？**

CAP定理给出了答案：三者不可兼得，你必须做出选择。

但CAP定理也带来了误解——很多人以为"三选二"，实际上**P（分区容忍性）基本是必须选的**，真正的选择是在C和A之间权衡。

BASE理论则是对CAP的**工程化落地**：既然强一致性很难保，那就退而求其次，保基本可用和最终一致性。

---

## 它是怎么解决问题的？

### CAP定理：不可能三角

```
         ┌──────────┐
         │          │
   C────│ 分布式系统 │────A
   (一致)│          │(可用)
         │    P     │
         └──────────┘
          (分区容忍)
```

| 选项 | 含义 | 牺牲什么 | 典型系统 |
|---|---|---|---|
| **CA** | 单点系统，无网络分区 | P（只能跑在单机） | MySQL单机 |
| **CP** | 网络分区时保一致性，拒绝服务 | A（部分节点不可用） | ZooKeeper、etcd |
| **AP** | 网络分区时保可用，允许数据不一致 | C（最终一致） | Cassandra、DynamoDB |

**关键认知**：CAP不是"三选二"，而是**P必选，C和A权衡**。

```
真实场景：
  网络正常 → CP系统 → 既一致又可用（CA都有）
  网络分区 → CP系统 → 拒绝服务（保C舍A）
            → AP系统 → 返回旧数据（保A舍C）
```

### BASE理论：CAP的工程化落地

BASE是对CAP中AP方案的延伸，通过**牺牲强一致性**获得**高可用性**：

```
BA（Basically Available）— 基本可用
         ↓
S（Soft state）— 软状态（数据可以暂时不一致）
         ↓
E（Eventually consistent）— 最终一致性
```

**BASE的核心思想**：不要求强一致性，但保证经过一段时间后数据达到一致。

```
            请求1 → 主库写入成功
                ↓
            主库 → 异步复制 → 从库（延迟 100ms）
                ↓
            请求2 → 从库读取 → 旧数据（短暂不一致）
                ↓
            100ms 后 → 从库数据一致 → 最终一致
```

## 它和相似方案的本质区别是什么？

### CAP vs ACID

| | CAP | ACID |
|---|---|---|
| 适用场景 | 分布式系统 | 数据库事务 |
| C的含义 | 所有节点数据一致 | 事务执行前后数据一致 |
| A的含义 | 每个请求都有响应 | 全部or全部不执行 |
| 设计目标 | 分布式权衡 | 数据库正确性 |

**本质区别**：CAP是**分布式系统的权衡框架**，ACID是**数据库事务的保证机制**。分布式系统用CAP做架构选择，数据库用ACID保证事务正确。

### CP系统 vs AP系统

| | CP系统 | AP系统 |
|---|---|---|
| 一致性 | 强一致 | 最终一致 |
| 可用性 | 分区时拒绝服务 | 分区时仍可用 |
| 延迟 | 需要多数派确认，延迟高 | 本地写入即可，延迟低 |
| 适用场景 | 金融、锁服务 | 社交、内容分发 |

**本质区别**：CP系统用**延迟换一致性**，AP系统用**短暂不一致换高可用**。

## 正确使用方式

### 场景1：注册中心选AP

```
Eureka（AP）— 网络分区时，客户端用本地缓存继续工作
              — 牺牲：数据暂时不一致
              — 获得：高可用，不中断业务
```

### 场景2：分布式锁选CP

```
ZooKeeper（CP）— 网络分区时，少数派节点拒绝服务
                — 牺牲：可用性（锁服务暂时不可用）
                — 获得：强一致性（锁状态不会冲突）
```

### 场景3：电商下单用BASE

```
下单 → 创建订单（主库）
    → 异步 → 扣减库存（消息队列）
    → 异步 → 发送通知（消息队列）
              — 不要求实时一致，但最终会一致
```

## 边界情况和坑

### 坑1：误解"三选二"

很多人背"CAP三选二"，但**现代分布式系统基本都选P**。

```
错误理解：我有3个选项，选其中2个
正确理解：P必选，我在C和A之间权衡
```

**面试话术**：CAP不是三选二，而是**网络分区时，C和A只能保一个**。实际系统中，网络分区是常态，所以P基本是必选的。

### 坑2：BASE不是"不保证一致性"

BASE的"最终一致性"是有**时间上限**的，不是无限期不一致。

```
错误理解：BASE就是不管一致性，随便写
正确理解：BASE保证在经过一段时间后，数据一定会达到一致
```

### 坑3：把CAP当性能指标

CAP是**理论上的权衡框架**，不是性能指标。一个CP系统也可以很高可用，只要网络不分区。

```
错误理解：CP系统可用性一定差
正确理解：CP系统在网络正常时，可用性可以和AP系统一样高
          只有在网络分区时，CP系统才会拒绝服务
```
