# 垃圾收集器

> 没有一种收集器能同时满足「最大吞吐量」和「最短停顿时间」。重点：CMS 四阶段与重新标记的原因、G1 的 Region 设计与 SATB、ZGC 的染色指针与指针自愈。

---

## 这个问题为什么存在？

新生代用复制算法、老年代用标记-整理，这只解决了「用什么算法回收」的问题。

**核心问题**：面对不同场景——后台计算（要吞吐量）、Web 服务（要低延迟）、实时系统（要超低停顿）——需要完全不同的收集器实现。没有万能方案。

---

## 它是怎么解决问题的？

### 1. 按分代划分的收集器

```
Young Gen（新生代）           Old Gen（老年代）
┌──────────────┐            ┌──────────────┐
│  Serial        │            │  Serial Old  │
│  ParNew        │ ← 配合 ─→ │  CMS         │
│  Parallel      │            │  Parallel Old│
│  G1 (Young)   │            │  G1 (Old)    │
└──────────────┘            └──────────────┘
          ↓ 不分代
    ZGC / Shenandoah（整堆并发）
```

### 2. Serial / Serial Old

单线程收集，**必须 STW（Stop-The-World）**。

- **特点**：简单、线程切换开销最小
- **适合**：客户端应用、堆很小（< 100MB）的场景

### 3. Parallel Scavenge + Parallel Old（JDK 8 默认）

**目标：吞吐量最大化**（吞吐量 = 用户代码时间 / 总时间）。

```bash
# 吞吐量 = 1 / (1 + GCTimeRatio)
-XX:GCTimeRatio=19   # GC 时间占总时间的 5%（1/(1+19)）
```

- **适合**：后台计算、报表生成（对停顿不敏感，吞吐量优先）

### 4. CMS（Concurrent Mark Sweep）⭐

**目标：低停顿**（适合 Web 应用）。四个阶段：

```
1. 初始标记（STW，很快）
   └── 标记 GC Roots 直接关联的对象
       │
2. 并发标记（和应用线程并发）
   └── 顺着引用链，标记所有存活对象
       │ （此时用户线程在跑，引用可能变化）
       ▼
3. 重新标记（STW，较快）
   └── 修正并发标记期间引用变化的对象
       │ （用 Incremental Update 算法）
       ▼
4. 并发清除（和应用线程并发）
   └── 回收未标记的对象
```

**为什么并发标记后需要「重新标记」？** 并发标记阶段用户线程在运行，引用关系可能变化（比如 A → B 的引用断了）。这些变化如果不修正，会漏标（把存活对象当垃圾回收 → 严重错误）。重新标记阶段 STW，修正这些变化。

**CMS 的两大坑**：

| 坑 | 说明 |
|------|------|
| **内存碎片** | CMS 用标记-清除（不整理），多次 GC 后碎片严重 → 大对象分配失败 → 触发 Full GC（用 Serial Old 整理，STW 极长） |
| **Concurrent Mode Failure** | 并发清除阶段用户线程还在创建对象，老年代不够放了 → 强制 Full GC |

```bash
# 预防 Concurrent Mode Failure：提前启动 CMS
-XX:CMSInitiatingOccupancyFraction=75  # 老年代占 75% 时启动（默认 92% 太晚）
```

### 5. G1（Garbage First）⭐⭐（JDK 9+ 默认）

**设计目标**：可预测的停顿时间。

#### G1 的革命性设计：Region

```
传统收集器：连续内存
┌──────┬──────┬──────┬──────┐
│ Eden │ S0   │ S1   │ Old  │  ← 必须整块扫描
└──────┴──────┴──────┴──────┘

G1：把堆分成大小相等的 Region（1~32MB）
┌──┐┌──┐┌──┐┌──┐┌──┐
│ R ││ R ││ R ││ R ││ R │  ← 每次只回收价值最高的 Region
└──┘└──┘└──┘└──┘└──┘
  R = Region
  - Eden Region
  - Survivor Region
  - Old Region
  - Humongous Region（大对象，占 ≥ 50% Region 大小）
```

**核心思想**：每次 GC 只回收「垃圾最多」的 Region（Garbage First 名字的由来），控制停顿时间。

#### G1 的四个阶段

```
1. 初始标记（STW，借助 Minor GC 完成，几乎无额外停顿）
   └── 标记 GC Roots 直接关联的对象
       │
2. 并发标记（并发）
   └── 从 GC Roots 开始，标记存活对象
       │ 记录引用变化（SATB 算法，Snapshot-At-The-Beginning）
       ▼
3. 最终标记（STW，很短）
   └── 处理 SATB 记录的引用变化
       ▼
4. 筛选回收（STW，但可以并行，停顿可控）
   └── 选择「回收价值最高」的 Region 进行回收
       用复制算法（把存活对象复制到新 Region，无碎片）
```

**SATB（Snapshot-At-The-Beginning）vs CMS 的 Incremental Update**：

| | CMS（Incremental Update） | G1（SATB） |
|--|---------------------------|---------------|
| **记录什么** | 新增的引用（黑人问号） | 被删除的旧引用 |
| **效率** | 需要重新扫描 | 只需要处理 SATB 缓冲区 |
| **效果** | 重新标记阶段可能较慢 | 重新标记阶段更快 |

#### G1 的「可预测停顿」如何实现？

```bash
# 设置期望最大停顿时间（默认 200ms）
-XX:MaxGCPauseMillis=200
```

G1 维护一个「回收价值列表」：每个 Region 的垃圾占比、回收所需时间，都记录下来。每次 GC，选择能在 `MaxGCPauseMillis` 时间内回收完的 Region。

**以停顿时间为导向**，而不是像 Parallel GC 以吞吐量为导向。

### 6. ZGC（JDK 11+，生产可用 JDK 15+）⭐⭐⭐

**目标**：超低停顿（< 10ms），**停顿时间不随堆大小增加**。

#### ZGC 的核心技术：染色指针（Colored Pointer）

```
普通对象指针（OOP）：
┌─────────────────────────────────────────┐
│            64 位对象地址                   │
└─────────────────────────────────────────┘

ZGC 染色指针（64 位，高 18 位存元数据）：
┌──┬──┬──┬──────────────────────────────┐
│18│ 1│ 1│  42 位对象地址                │
│位│位│位│                              │
│  │  │  └── Marked1/Marked0（标记位）│
│  │  └── Remapped（重映射位）       │
│  └── Finalizable（终结位）         │
└── 高 18 位未使用（ZGC 只用 42 位地址，支持 4TB 堆）
```

**为什么把 GC 状态存在指针里？** 传统 GC 把标记位存在对象头里，并发标记时需要访问对象头 → 需要内存屏障 → 开销大。ZGC 把标记位存在指针本身（染色指针）→ 访问指针时直接读标记位 → 开销极小。

#### ZGC 的三个关键阶段

```
1. 并发标记（并发，几乎无 STW）
   └── 遍历对象图，在指针上标记
       │
2. 并发转移（并发，这是 ZGC 最牛的地方）
   └── 把存活对象从旧 Region 复制到新 Region
       │ 关键：对象移动了，但旧地址的指针仍然可用！
       │ （通过「转发表 Forwarding Table」实现指针自愈）
       ▼
3. 并发重映射（并发）
   └── 修正所有指向旧地址的指针 → 指向新地址
```

**「指针自愈」**（ZGC 的核心黑科技）：

```
转移前：线程 A 持有 oldObject 的指针（指向地址 X）

转移时：ZGC 把 oldObject 复制到地址 Y
         设置 X → Y 的映射（Forwarding Table）

线程 A 访问 oldObject：
  1. 访问地址 X
  2. 发现 X 有 Forwarding 映射 → 指向 Y
  3. 修正 A 的指针（改成指向 Y）（自愈！）
  4. 下次访问直接到 Y（不需要再查 Forwarding Table）
```

**这就意味着：转移阶段不需要 STW**！对象移动了，通过 Forwarding Table 和指针自愈，用户线程可以继续访问旧地址（透明无感知）。

#### ZGC 的性能

| 堆大小 | G1 停顿 | ZGC 停顿 |
|--------|----------|----------|
| 4 GB | ~200ms | < 10ms |
| 32 GB | ~500ms | < 10ms |
| 128 GB | ~2s | < 10ms |

**ZGC 的停顿时间和堆大小无关**（染色指针 + 并发转移）。这是 ZGC 革命性的地方。

## 深入原理

### G1 vs CMS：为什么 CMS 被废弃（JDK 14 移除）？

| 维度 | CMS | G1 |
|------|------|-----|
| **内存布局** | 分代（连续） | Region（不连续） |
| **算法** | 标记-清除（有碎片） | 标记-整理+复制（无碎片） |
| **停顿可预测** | ❌ | ✅ `MaxGCPauseMillis` |
| **大堆** | 不适合（碎片问题严重） | ✅ 适合（Region 无碎片） |
| **Full GC** | Serial Old（单线程，超长停顿） | 多线程（并行回收） |

**为什么 CMS 的 Full GC 是灾难？** CMS 用标记-清除，不整理内存 → 碎片越来越多 → 大对象分配失败 → 触发 Full GC → 用 Serial Old（单线程标记-整理）→ 大堆下停顿可达数十秒。

### ZGC vs G1：选型建议

| 维度 | G1 | ZGC |
|------|-----|------|
| **停顿时间** | 可控（~200ms） | 极低（< 10ms） |
| **停顿来源** | 筛选回收需要 STW | 只标记 Root 集时需要极短 STW |
| **吞吐量** | 高（比 ZGC 高 ~15%） | 稍低（染色指针有开销） |
| **适用场景** | 堆 4~32GB，停顿 200ms 可接受 | 堆 > 32GB，要求超低停顿 |
| **JDK 版本** | JDK 9+ 默认 | JDK 11 实验，JDK 15 生产可用 |

**选型建议（JDK 17+）**：

```
堆 < 4GB       → Parallel Scavenge（吞吐量优先）
堆 4~32GB     → G1（停顿可控，综合性能最好）
堆 > 32GB     → ZGC（超低停顿，JDK 17+ 推荐）
对停顿极度敏感 → ZGC（金融交易、实时系统）
```

### Shenandoah：ZGC 的替代品

Shenandoah 是 Red Hat 开发的低延迟收集器，设计和 ZGC 类似（并发转移），但：
- **不分代**：所有对象一律平等处理
- **吞吐量比 ZGC 低**：因为没有染色指针这个黑科技
- **优点**：比 ZGC 更早可用（JDK 12+）

```bash
# JDK 12+ 启用 Shenandoah
-XX:+UseShenandoahGC
```

## 正确使用方式

### 1. 根据场景选择收集器

```bash
# JDK 8 及之前（默认 Parallel Scavenge + Parallel Old）
java -XX:+UseParallelGC MyApp

# JDK 9~14（默认 G1）
java -XX:+UseG1GC MyApp

# JDK 15+（默认 G1，但 ZGC 已生产可用）
java -XX:+UseZGC MyApp   # ZGC（JDK 15+）
java -XX:+UseShenandoahGC MyApp   # Shenandoah（JDK 12+）

# 小堆场景（< 1GB）
java -XX:+UseSerialGC -Xmx512m MyApp
```

### 2. G1 的关键参数

```bash
-XX:+UseG1GC                     # 启用 G1
-XX:MaxGCPauseMillis=200          # 期望最大停顿（默认 200ms）
-XX:G1HeapRegionSize=4m           # Region 大小（1~32MB，需为 2 的幂）
-XX:InitiatingHeapOccupancyPercent=45  # 老年代占堆 45% 时开始并发标记（默认 45%）
-XX:G1ReservePercent=10           # 保留空间百分比（防止 to-space 溢出）
```

**`-XX:MaxGCPauseMillis` 是「软目标」**：G1 会**尽量**满足这个目标，但不保证 100% 做到。如果设置得太小（比如 20ms），G1 会把堆分成更多 Region → 每次回收更少 Region → 吞吐量下降。

### 3. ZGC 的关键参数（JDK 17+）

```bash
-XX:+UseZGC                 # 启用 ZGC（JDK 15+）
-XX:+UseZGCDS               # 类卸载（默认开启）
-Xmx64g                    # 堆大小（ZGC 适合大堆）
-XX:ZCollectionInterval=5   # 触发 GC 的间隔（秒，可选）
```

**ZGC 的「无使用」场景**：ZGC 的染色指针需要 64 位地址（高 18 位），32 位 JVM 不支持。

## 边界情况和坑

### 1. `System.gc()` 触发 Full GC

```java
// ❌ 不要手动调用 System.gc()
System.gc();  
// → 触发 Full GC（所有收集器都会）
// → 长时间 STW（尤其是 CMS 的 Serial Old 回退）
```

**解决**：加上 `-XX:+DisableExplicitGC`，让 JVM 忽略 `System.gc()` 调用。

### 2. G1 的「to-space overflow」

```
G1 并发标记完成后，准备把存活对象复制到新 Region，
但发现没有足够的空闲 Region 了 → to-space overflow
 → 触发 Full GC（单线程，长时间停顿）
```

**解决**：

```bash
# 增大堆
-Xmx64g   # 原来 32g，增大到 64g

# 提前启动并发标记
-XX:InitiatingHeapOccupancyPercent=35  # 默认 45%，提前到 35%
```

### 3. 对象过大导致 Humongous Allocation

```java
// ❌ 大对象直接进入 Old Gen（G1 中 > 50% Region 大小）
byte[] hugeArray = new byte[10 * 1024 * 1024];  // 10MB，假设 Region=4MB
// → 分配在 Humongous Region
// → 无法被 Young GC 回收（直接进入 Old Gen）
// → 只有 Full GC 才能回收
```

**解决**：

```bash
# 增大 Region 大小（让大对象不再是「大对象」）
-XX:G1HeapRegionSize=8m   # 8MB Region，10MB 数组不再是「大对象」

# 或：避免分配大对象（改数据结构设计）
```

### 4. ZGC 的「多重映射」（了解即可）

> ZGC 为了支持并发转移，使用了**多重映射（Multi-Mapping）**：
> 同一个物理内存，映射到**三个不同的虚拟地址**（Marked0、Marked1、Remapped）。
> 这是 ZGC 能实现「并发转移」的底层原理。

**对应用透明**（不需要做任何事），但需要注意：ZGC 不支持压缩指针（`-XX:+UseCompressedOops`）——因为高 18 位被用作染色指针，无法压缩。
