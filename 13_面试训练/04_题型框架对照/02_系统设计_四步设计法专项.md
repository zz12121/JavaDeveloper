# 系统设计/架构题 — 四步设计法专项训练

> 框架：确认需求 → 顶层设计 → 核心模块 → 权衡取舍

---

## 一、为什么要学系统设计

```
系统设计是大厂/中高级岗位的必考题：
- 大厂（阿里/字节/美团/腾讯）：必问
- 中高级岗位（15K+）：高频
- 中级岗位（10-15K）：偶尔问

你目标30-40万年薪，系统设计必须会。
```

## 二、四步设计法详解

### 第一步：确认需求和约束

```
这是最容易忽略，但最重要的步骤。

面试官给的问题往往是模糊的：
「设计一个秒杀系统」「设计一个短链接」

你需要先问清楚：

关键问题清单：
1. QPS多少？（100/1000/10万/100万）
2. 数据量多大？（1万/100万/10亿）
3. 需要强一致性吗？（金钱相关必须强一致）
4. 需要99.99%可用吗？（核心链路必须高可用）
5. 有没有特殊限制？（成本/技术栈/时间）

技巧：
「我确认一下，这个系统预计QPS是多少？数据规模大概多大？」
→ 问清楚再动手，显得你专业。
```

### 第二步：顶层设计

```
根据QPS和数据量，确定整体架构。

架构维度：
1. 高并发：怎么扛住流量？
2. 高可用：挂了怎么办？
3. 可扩展：未来增长怎么办？

分层架构模板：
┌─────────────────────────────────────┐
│          用户请求                    │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│  接入层：CDN / Nginx / API Gateway  │
│  限流 / 鉴权 / 路由                  │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│  应用层：业务服务（微服务/单体）      │
│  业务逻辑 / 服务编排                  │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│  数据层：缓存 / 数据库 / 消息队列     │
│  Redis / MySQL / Kafka              │
└─────────────────────────────────────┘

常用技术：
高并发 → 缓存 + 队列 + 异步 + 限流
高可用 → 主从 + 副本 + 降级 + 熔断
可扩展 → 读写分离 + 分库分表 + 服务拆分
```

### 第三步：核心模块设计

```
这是最核心的部分，每个关键模块展开讲。

每个模块的描述模板：
1. 这个模块负责什么？
2. 选用什么技术方案？
3. 为什么选这个方案（对比其他方案）？
4. 数据怎么存储？
5. 接口怎么设计？

示例（秒杀系统 - 库存扣减模块）：
"核心模块是库存扣减，需要解决两个问题：
1. 防止超卖：多个用户同时买，最后一个商品
2. 高并发：万人抢100个商品

方案对比：
方案A（数据库乐观锁）：update set stock=stock-1 where id=? and stock>0
   优点：简单，数据一致
   缺点：高并发下DB压力大

方案B（Redis预减库存）：先扣Redis，异步同步DB
   优点：性能高
   缺点：Redis挂了怎么办？

最终方案C（Redis+Lua+MQ）：
1. 用Redis预减库存，Lua脚本保证原子性
2. 扣成功后发MQ消息
3. MQ消费者异步写DB
4. Redis库存为0，直接返回秒杀结束
这样既能抗住高并发，又能保证最终一致性。"
```

### 第四步：权衡取舍

```
面试官想听的不只是你"会做什么"，
更想听你"知道取舍"。

常见取舍维度：

性能 vs 一致性：
- 强一致 → 牺牲性能
- 最终一致 → 性能好

成本 vs 可用性：
- 多副本 → 成本高，但可用性高
- 单机 → 成本低，但可用性低

简单 vs 扩展：
- 快速上线 → 单体架构
- 长期扩展 → 微服务架构

取舍话术：
"这里有一个取舍：方案A优点是XX，缺点是XX，
我选方案A，因为对于这个场景XX更重要。"
```

## 三、高频系统设计题详解

### 题目1：设计一个秒杀系统

```
【第一步：确认需求】
"我确认一下：
- QPS大概多少？（秒杀商品100件，预计峰值10万人参与）
- 是否需要强一致性？（库存必须准确，不能超卖）
- 可用性要求？（不能挂了）

假设：QPS 10万，可用性99.9%，库存必须准确。"

【第二步：顶层设计】
整体分三层：
1. 接入层：限流（令牌桶）+ CDN静态化
2. 服务层：业务逻辑 + 消息队列削峰
3. 数据层：Redis缓存 + MySQL存储

核心链路：
用户请求 → CDN（静态资源） → 网关限流 → 验证码过滤 → Redis预扣库存 → 发送MQ → 异步下单 → 通知用户

【第三步：核心模块】
模块1：限流（防刷）
- 网关层用令牌桶，限制10万QPS
- 验证码过滤刷单用户
- 接口防重（Token机制）

模块2：库存扣减（防超卖）
- Redis预扣库存，Lua脚本保证原子性
- 判断：stock > 0 才扣减
- 扣成功发MQ，失败直接返回

模块3：订单处理（异步下单）
- MQ消费者异步创建订单
- 库存扣减成功 + 支付成功 = 真正减库存
- 库存不足/超时未支付 → 库存回滚

【第四步：权衡取舍】
- 为什么用MQ削峰？
  → 如果10万QPS直接打DB，DB必挂。MQ可以把峰值流量平滑到后续处理。
- 为什么不直接用Redis扣库存就完了？
  → Redis挂了怎么办？MQ作为缓冲层，Redis挂了还能从MQ恢复。
```

### 题目2：设计一个分布式ID生成器

```
【第一步：确认需求】
"确认一下：
- 每秒需要生成多少个ID？（1万/100万）
- 是否需要趋势递增？（业务需要按时间排序）
- 是否需要无中心？（多机器部署）"

【第二步：顶层设计】
方案对比：
1. UUID：趋势无序，无法作为主键
2. 数据库自增：单点，性能有限
3. Snowflake：趋势递增，无中心，高性能

选Snowflake

【第三步：核心模块】
Snowflake结构（64位）：
- 符号位（1位）：0
- 时间戳（41位）：毫秒数，可用69年
- 机器ID（10位）：2^10=1024台机器
- 序列号（12位）：每毫秒2^12=4096个

关键问题1：时钟回拨
- 原因：机器时间被回拨
- 解决：方案1：记录上次的max时间，回拨后等待
         方案2：拒绝生成，返回错误
- 我倾向方案1，加上序列号偏移

关键问题2：机器ID分配
- 方案：配置中心统一分配（Zookeeper/Etcd）
- 或者：启动时用ZK抢注，注册后心跳保活

【第四步：权衡取舍】
- 为什么不直接用UUID？
  → UUID无序，作为主键会导致B+树频繁分裂，影响写入性能。
  → Snowflake趋势递增，B+树追加写入，性能好。
```

### 题目3：设计一个延迟任务系统

```
【第一步：确认需求】
"确认一下：
- 任务规模？（每秒多少任务）
- 延迟精度？（秒级/分钟级）
- 是否需要重试？"

【第二步：顶层设计】
方案对比：
1. 数据库轮询：简单，但性能差，延迟不准确
2. JDK ScheduledThreadPool：单机，重启丢失
3. 消息队列延迟：RabbitMQ延迟队列/Kafka延迟插件
4. Redis ZSet：利用ZRANGEBYSCORE
5. 时间轮算法：Netty/Hazelcast使用

选方案3+4组合

【第三步：核心模块】
任务注册：
- 任务ID + 执行时间 + 任务类型 + 参数
- 存MySQL（持久化）+ 存Redis ZSet（待执行）

任务执行：
- 定时线程扫描Redis ZSet（ZANGEBYSCORE 0 now）
- 取出任务 → 路由到对应处理器 → 执行
- 成功 → 删除记录
- 失败 → 判断是否重试（重试次数+1）

时间轮（用于更精确的延迟）：
- 轮：秒/分/时/天 四个时间轮
- 每格：存储该时间到期的任务
- 推进：每秒推进秒轮，秒轮归零时把任务丢到下一轮

【第四步：权衡取舍】
- 为什么用Redis ZSet + MySQL双存储？
  → Redis保证实时性，MySQL保证持久化。
  → Redis挂了可以从MySQL恢复。
```

### 题目4：设计一个接口限流系统

```
【第一步：确认需求】
"确认一下：
- 在哪层限流？（网关/应用）
- 限流维度？（用户/IP/接口/总QPS）
- 限流后怎么处理？（拒绝/排队/降级）"

【第二步：顶层设计】
分层限流：
1. 接入层：Nginx/Lua限流
2. 网关层：Sentinel/Spring Cloud Gateway
3. 应用层：Guava RateLimiter/AOP注解

整体架构：
请求 → 限流配置中心（推配置） → 本地限流规则 → Redis集中统计 → 限流处理

【第三步：核心模块】
限流算法实现：
1. 计数器：
   - 维护一个计数器，1秒内请求数超过阈值就限流
   - 简单，但有临界问题（0.9秒和1.1秒各来N个请求）

2. 滑动窗口：
   - 改进计数器，用时间窗口统计
   - Redis sorted set：score=时间戳，统计窗口内数量

3. 令牌桶（推荐）：
   - 以固定速率往桶里放令牌
   - 请求来了取令牌，取到就通过
   - 允许突发流量（桶满时）
   - 实现：Guava RateLimiter

4. 漏桶：
   - 以固定速率从桶里漏出
   - 无论请求多少，输出速率固定
   - 实现：MQ削峰就是漏桶思想

限流处理：
- 快速失败：返回429 Too Many Requests
- 排队：放进MQ，异步处理
- 降级：返回默认值/缓存数据

【第四步：权衡取舍】
令牌桶 vs 漏桶：
- 令牌桶：允许突发，入口流量可以突然增大
- 漏桶：输出速率恒定，入口突发无效
- 选择：网关入口用令牌桶，允许正常用户突发
         下游保护用漏桶，保护服务不被冲垮
```
