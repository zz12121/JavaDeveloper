# 运维实战

> K8s 不是「部署完就不管」。日常运维要会看日志、会排查 Pod 异常、会滚动更新、会在不中断服务的前提下发布新版本。

---

## 这个问题为什么存在？

不会 K8s 运维时，典型灾难是：

```
场景 A：Pod 一直 CrashLopBackOff
  → 不知道怎么看日志
  → 不知道怎么进容器调试
  → 只能删了重来，还是失败

场景 B：滚动更新导致服务中断
  → 直接 kubectl set image deployment/xxx ...
  → 新 Pod 还没就绪，旧 Pod 已经被删了
  → 服务中断 30 秒

场景 C：线上问题无法排查
  → 没有日志收集
  → Pod 重启后，之前的日志丢了
  → 无法追踪「上一个版本的 Pod 为什么 OOM」

场景 D：误删资源
  → kubectl delete deployment user-service
  → 整个服务没了，恢复要 30 分钟
```

**K8s 运维要解决的核心问题**：

1. **可观测性**：Pod 为什么挂？日志在哪？指标在哪？
2. **滚动更新**：不中断服务的前提下发布新版本
3. **快速回滚**：新版本有问题时，30 秒内回滚
4. **安全生产**：防止误删、防止配置错误

---

## 它是怎么解决问题的？

### 核心工具：`kubectl`

```bash
# 查看资源（最常用）
kubectl get pods
kubectl get svc
kubectl get deploy

# 查看详情（排查问题必用）
kubectl describe pod <pod-name>
# 输出：状态、事件（Events）、容器配置

# 查看日志
kubectl logs <pod-name>
kubectl logs -f <pod-name>          # 实时跟踪
kubectl logs --previous <pod-name>  # 查看上一次容器的日志（Pod 重启过）

# 进入容器调试
kubectl exec -it <pod-name> -- /bin/bash

# 端口转发（本地访问集群内服务）
kubectl port-forward svc/mysql 3306:3306
# 本地连 localhost:3306 即可访问集群内 MySQL

# 复制文件
kubectl cp <pod-name>:/var/log/app.log ./app.log
```

### Pod 状态大全（排查问题的地图）

| 状态 | 含义 | 排查方向 |
|------|------|---------|
| `Pending` | 还没调度到 Node | `describe pod` 看 Events（资源不足？污点？） |
| `ContainerCreating` | 容器在创建中（拉镜像？） | `describe pod` 看 Events（镜像拉取失败？） |
| `Running` | 正常运行 | 看日志 `kubectl logs` |
| `CrashLopBackOff` | 容器反复崩溃 | `kubectl logs --previous` 看上一次崩溃的日志 |
| `ImagePullBackOff` | 镜像拉取失败 | 检查镜像名、镜像标签、imagePullSecrets |
| `Eror` | Pod 无法启动 | `describe pod` 看 Events |
| `Terminating` | Pod 删除中（卡住了） | `kubectl delete pod <name> --force --grace-period=0` |

### 滚动更新（Rolling Update）

```yaml
apiVersion: apps/v1
kind: Deployment
spec:
  replicas: 6
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 2        # 最多多 2 个 Pod
      maxUnavailable: 1  # 最多少 1 个 Pod
```

**滚动更新流程**（假设 `replicas: 6`，`maxSurge: 2`，`maxUnavailable: 1`）：

```
旧版本：6 个 Pod（全部就绪）

第 1 步：创建 2 个新 Pod（maxSurge=2）
  → 现在有 6 旧 + 2 新 = 8 个 Pod

第 2 步：等新 Pod 就绪，删除 1 个旧 Pod（maxUnavailable=1）
  → 现在有 5 旧 + 2 新 = 7 个 Pod

第 3 步：等待，直到所有旧 Pod 被替换
  → 最终：0 旧 + 6 新 = 6 个 Pod
```

**效果**：全程有 Pod 对外服务，服务不中断。

### 快速回滚

```bash
# 查看历史版本
kubectl rollout history deployment/user-service

# 回滚到上一个版本
kubectl rollout undo deployment/user-service

# 回滚到指定版本
kubectl rollout undo deployment/user-service --to-revision=3

# 查看回滚状态
kubectl rollout status deployment/user-service
```

## 它和相似方案的本质区别是什么？

### K8s 运维 vs 直接操作 Docker

| 维度 | K8s 运维（kubectl） | 直接操作 Docker |
|------|----------------------|-------------------|
| 操作对象 | 整个集群（Deployment、Service） | 单台机器上的容器 |
| 滚动更新 | 内置（strategy.rollingUpdate） | 需要写脚本 |
| 回滚 | 内置（`rollout undo`） | 需要手动记录历史版本 |
| 日志收集 | 需要额外组件（EFK） | `docker logs` 本地查看 |
| 适合场景 | 生产环境、多节点集群 | 本地开发、单节点 |

### K8s 原生运维 vs Helm

| 维度 | K8s 原生（kubectl + YAML） | Helm（包管理器） |
|------|--------------------------|----------------|
| 应用管理 | 单独管理每个资源 | 把一组资源打包成「Chart」 |
| 版本管理 | 需要手动记录 | Chart 有版本号 |
| 配置管理 | ConfigMap（需要手动维护） | Values.yaml（模板化配置） |
| 回滚 | `kubectl rollout undo` | `helm rollback` |
| 适合场景 | 简单应用、学习 K8s | 复杂应用、生产环境 |

**推荐**：学 K8s 先学原生 kubectl，再生产环境用 Helm 管理应用。

## 正确使用方式

### 1. 用 `readinessProbe` 保证滚动更新不中断服务

```yaml
apiVersion: apps/v1
kind: Deployment
spec:
  template:
    spec:
      containers:
        - name: user-service
          image: user-service:1.0.0
          readinessProbe:        # 就绪探针（流量只转发到就绪的 Pod）
            httpGet:
              path: /actuator/health
              port: 8080
            initialDelaySeconds: 10
            periodSeconds: 5
          livenessProbe:         # 存活探针（失败则重启 Pod）
            httpGet:
              path: /actuator/health
              port: 8080
            initialDelaySeconds: 30
            periodSeconds: 10
```

**`readinessProbe` vs `livenessProbe`**：

| 探针 | 失败后果 | 适用场景 |
|------|---------|---------|
| `readinessProbe` | 从 Service 的 Endpoints 中摘除（不转发流量） | Pod 启动中、临时过载 |
| `livenessProbe` | 杀掉容器，Pod 重启 | 死锁、死循环 |

### 2. 用 `--record` 记录变更原因

```bash
# 更新镜像，并记录原因
kubectl set image deployment/user-service user-service=user-service:1.0.1 --record

# 查看变更历史（包含原因）
kubectl rollout history deployment/user-service
# 输出：
# REVISION  CHANGE-CAUSE
# 1         <none>
# 2         kubectl set image deployment/user-service user-service=user-service:1.0.1 --record=true
```

### 3. 用 Resource Quota 防止资源超卖

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: mem-cpu-quota
  namespace: default
spec:
  hard:
    requests.cpu: "4"        # default namespace 所有 Pod 的 requests.cpu 总和不能超过 4 核
    requests.memory: 8Gi
    limits.cpu: "8"
    limits.memory: 16Gi
```

**效果**：防止某个 namespace 把所有 Node 的资源吃光。

## 边界情况和坑

### 1. `Terminating` 状态的 Pod 删不掉

```
现象：
  kubectl delete pod <pod-name>
  → Pod 状态一直是 Terminating
```

**原因**：Pod 有 `finalizers`（清理钩子），但钩子执行失败了。

**解决**（强制删除，危险！）：

```bash
kubectl delete pod <pod-name> --force --grace-period=0
```

**注意**：强制删除后，Pod 可能在 Node 上还有残留进程。

### 2. 滚动更新时，`maxSurge=0` 会导致更新变慢

```yaml
strategy:
  rollingUpdate:
    maxSurge: 0          # ❌ 不创建新 Pod，先删旧 Pod
    maxUnavailable: 1
```

**问题**：每次先删 1 个旧 Pod，再创建 1 个新 Pod → 更新时间长。

**推荐配置**：

```yaml
    maxSurge: 25%         # 最多多 25% 的 Pod（6 个副本 → 最多 8 个）
    maxUnavailable: 25%
```

### 3. `livenessProbe` 配置太严格，导致 Pod 频繁重启

```
场景：
  livenessProbe:
    initialDelaySeconds: 10   # 只等 10 秒
    timeoutSeconds: 1         # 超时 1 秒
    periodSeconds: 5
```

**问题**：应用启动需要 30 秒，但 `initialDelaySeconds=10` → 探针在应用启动完成前就开始检查 → 检查失败 → Pod 被重启 → 无限重启。

**正确做法**：

- `initialDelaySeconds` 设为应用启动时间的 1.5 倍
- `timeoutSeconds` 设为接口正常响应时间的 3 倍
- 用 `/actuator/health` 这种轻量接口，不要用 `/` 或业务接口

### 4. 删除 PVC 不会删除 PV（默认）

```bash
kubectl delete pvc my-pvc
# → PV 状态变成 Released，不会自动删除数据
```

**解决**：设置 PVC 删除时同时删除 PV：

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
spec:
  ...
  volumeMode: Filesystem
  ...
# 在 PV 上设置：
  persistentVolumeReclaimPolicy: Delete  # PVC 删除时，PV 也删除
```

## 我的理解

K8s 运维最容易被忽视的是**「探针配置」**。很多人直接复制网上的 YAML，不根据自己的应用调整：

- 应用启动慢（大型 Java 应用启动要 60 秒）→ `initialDelaySeconds` 要设 90
- 接口响应慢（报表导出要 30 秒）→ `timeoutSeconds` 要设 60
- 不要用一个接口做 `readiness` 和 `liveness` → 临时过载会导致 Pod 被重启

**我的 K8s 运维 Checklist**：

```
✅ 所有 Deployment 都配置了 resource.requests/limits
✅ 所有容器都配置了 readinessProbe + livenessProbe
✅ 用 --record 记录每次变更
✅ 用 ResourceQuota 限制 namespace 资源
✅ 重要应用用 PDB（Pod Disruption Budget）保证最少可用副本
✅ 日志收集用 EFK（Elasticsearch + Fluentd + Kibana）
```

**面试高频考点**：

1. Pod 的常见状态有哪些，怎么排查？—— `Pending`（资源不足）、`CrashLopBackOff`（看日志）、`ImagePullBackOff`（镜像问题）
2. 滚动更新的流程是什么？—— 结合 `maxSurge` 和 `maxUnavailable` 讲
3. `readinessProbe` vs `livenessProbe` 的区别？—— 高频
4. 怎么快速回滚？—— `kubectl rollout undo`
5. 你们生产环境怎么部署 K8s 应用？—— 开放题，结合 Helm / Argo CD

*K8s 运维不是「会几条 kubectl 命令」，是理解「控制循环」和「声明式 API」—— 告诉我你想要什么状态，K8s 负责实现它。*
