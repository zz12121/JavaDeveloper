# 抽象类与接口

> 抽象类和接口是 Java 的两种"抽象机制"——抽象类用于 is-a 关系和模板方法，接口用于 can-do 关系和能力契约。理解两者的设计意图差异，比记住语法重要得多。

---

## 这个问题为什么存在？

没有这两种机制时，公共行为只能靠继承一个普通类，能力声明只能靠标记接口（没有方法约束）：

```
问题 1：公共行为没有抽象约束
→ 父类写了 abstract 方法，但编译器不强制子类实现 → 运行时才发现没实现

问题 2：一个类需要多种能力
→ Java 是单继承，一个类只能 extends 一个父类
→ 需要多种能力（可比较 + 可序列化 + 可运行）怎么办？

问题 3：接口演进破坏兼容性
→ 给接口加一个新方法，所有实现类都要改（成千上万个实现类不可能全部修改）
```

抽象类和接口分别解决上述问题 1+2、问题 3。

---

## 它是怎么解决问题的？

### 抽象类：is-a 关系 + 模板方法

```java
// 抽象类：不能实例化，可以有抽象方法和具体方法
abstract class DataProcessor {
    // 模板方法：定义算法骨架，用 final 防止破坏
    final void process() {
        readData();       // 固定步骤
        transformData();  // 抽象，子类实现
        writeData();      // 固定步骤
    }

    abstract void transformData();  // 子类必须实现

    void readData() { /* 通用实现 */ }
    void writeData() { /* 通用实现 */ }
}

// 子类只需要关心变化的部分
class XmlProcessor extends DataProcessor {
    @Override
    void transformData() {
        System.out.println("处理 XML");
    }
}
```

**抽象类的核心语义**：is-a 关系。"X is-a Y" 的时候用抽象类。

### 接口：can-do 关系 + 能力契约

```java
// 接口：描述能力，不关心实现
interface Runnable { void run(); }
interface Comparable<T> { int compareTo(T o); }
interface Serializable { }  // 标记接口，没有方法

// 一个类可以实现多个接口
class Task implements Runnable, Comparable<Task> {
    public void run() { /* ... */ }
    public int compareTo(Task o) { /* ... */ }
}
```

**接口的核心语义**：can-do 关系。"X can-do Y" 的时候用接口。

### Java 8+：接口也能有实现了

Java 8 引入 `default` 方法，缩小了接口和抽象类的差距：

```java
interface Collection<E> {
    // 抽象方法：必须实现
    boolean add(E element);

    // default 方法：可选覆写
    default boolean isEmpty() {
        return size() == 0;  // 用抽象方法 size() 实现
    }

    // static 方法：接口的工厂/工具方法
    static <E> List<E> of(E... elements) {
        return new ImmutableCollections.List<>(elements);
    }
}
```

**为什么引入 default 方法？**

Java 8 给 `Collection` 加了 `stream()` 方法，但 `Collection` 有成千上万个实现类。如果直接加抽象方法，所有实现类都要改——这是 API 演进的兼容性难题。`default` 方法提供了一个有默认实现的扩展点，现有实现类不用动。

## 深入原理

### 抽象类 vs 接口：设计意图对比

| 维度 | 抽象类 | 接口 |
|------|---------|------|
| 关系语义 | is-a（是什么） | can-do（能做什么） |
| 状态 | 可以有实例字段 | Java 8+ 有 static final 字段 |
| 多继承 | ❌ 单继承 | ✅ 多实现 |
| 构造器 | ✅ 有（用于子类初始化） | ❌ 没有构造器 |
| 方法实现 | 任意（抽象/具体/private/protected） | Java 8+：default/static/private |
| 演进能力 | 加具体方法 → 子类自动继承 | `default` 方法 → 实现类不强制改 |
| 典型场景 | 模板方法模式 | 能力契约（Runnable/Serializable） |

### 接口默认方法冲突

当一个类实现了两个有相同 `default` 方法的接口，必须显式解决冲突：

```java
interface A {
    default void hello() { System.out.println("A"); }
}
interface B {
    default void hello() { System.out.println("B"); }
}

class C implements A, B {
    // 编译错误：hello() 的默认实现冲突
    @Override
    public void hello() {
        A.super.hello();  // 显式指定用 A 的实现
        // 或者完全自己实现
    }
}
```

**冲突解决规则**：
```
1. 类方法（继承的）> 接口 default 方法（类优先原则）
2. 如果两个接口的 default 方法相同 → 必须显式覆写
3. 用 Interface.super.method() 选择调用哪个父接口的默认实现
```

### 抽象类的构造函数链

```java
abstract class Parent {
    Parent() { System.out.println("parent"); }
}

class Child extends Parent {
    Child() {
        super();      // 隐式插入，可以省略
        System.out.println("child");
    }
}

new Child();
// 输出：
// parent
// child
```

抽象类不能实例化，但**构造函数仍然有意义**——给子类初始化字段用。抽象类的构造函数只能被子类用 `super()` 调用。

## 正确使用方式

### 1. 选择抽象类：is-a + 共享状态

```java
// ✓ 正确：Dog is-a Animal，共享 name/age 字段
abstract class Animal {
    protected String name;
    protected int age;

    abstract void makeSound();

    // 公共实现
    public void breathe() {
        System.out.println(name + " 在呼吸");
    }
}

class Dog extends Animal {
    @Override
    void makeSound() { System.out.println("汪汪！"); }
}
```

### 2. 选择接口：can-do + 无状态

```java
// ✓ 正确：Task can-do Runnable，不一定 is-a 某个特定类
interface Runnable { void run(); }

// ✓ 正确：Comparable 是能力，不关心实现细节
interface Comparable<T> { int compareTo(T o); }
```

### 3. 接口 + 抽象类组合（Java 集合框架的经典设计）

```java
// 接口定义能力
interface List<E> { ... }

// 抽象类提供骨架实现（Skeletal Implementation）
abstract class AbstractList<E> implements List<E> {
    // 用迭代器实现 addAll（统一实现）
    public boolean addAll(Collection<? extends E> c) { ... }
}

// 具体类继承骨架实现，只需实现核心方法
class ArrayList<E> extends AbstractList<E> implements List<E> {
    // 只需实现 size()、get()、set() 等核心方法
    // addAll() 等通用方法已从 AbstractList 继承
}
```

这个模式叫**骨架实现（Skeletal Implementation）**，详见《Effective Java》第 20 条。

## 边界情况和坑

### 1. 抽象类可以有构造器，但不能直接 new

```java
abstract class Parent {
    Parent() { System.out.println("parent init"); }
}

Parent p = new Parent();      // ❌ 编译错误：抽象类不能实例化
Parent p = new Child();       // ✅ 可以指向子类实例
```

### 2. 抽象方法不能是 private/static/final

```java
abstract class Wrong {
    private abstract void m1();  // ❌ private + abstract 矛盾
    static abstract void m2();   // ❌ static + abstract 矛盾
    final abstract void m3();    // ❌ final + abstract 矛盾
}
```

抽象方法必须能被子类覆写，所以不能是 `private`、`static`、`final`。

### 3. 接口方法的隐式修饰符

```java
interface MyInterface {
    void method1();           // 隐式：public abstract
    // private void method2();  // ❌ Java 8 之前接口方法不能是 private

    default void method3() {}  // Java 8+：可以有方法体
    private void helper() {}    // Java 9+：private 方法可以有方法体（辅助方法）
}
```

接口中的方法**默认是 `public abstract`**，字段**默认是 `public static final`**。

### 4. 抽象类 vs 接口：滥用抽象类

```java
// ❌ 不好：只是想复用代码，没有 is-a 关系
abstract class Bird {
    void fly() { System.out.println("飞"); }
}
class Plane extends Bird {}  // Plane is-a Bird？语义错误！

// ✅ 好：用接口描述能力
interface CanFly { void fly(); }
class Bird implements CanFly { ... }
class Plane implements CanFly { ... }
```
