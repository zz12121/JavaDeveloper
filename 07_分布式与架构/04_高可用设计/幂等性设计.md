# 幂等性设计

## 这个问题为什么存在？

> 网络是不可靠的。调用超时后，调用方会重试，但**被调用方可能已经处理成功**。问题是：**怎么保证重试不会导致重复操作？**

没有幂等性，你会遇到：
- 支付时网络超时，用户重试 → 扣了两次钱
- 消息队列消费超时，MQ重试 → 重复扣库存
- 前端按钮双击 → 创建两个相同订单

---

## 它是怎么解决问题的？

### 幂等性定义

```
f(f(x)) = f(x)  → 多次调用，结果一样

例子：
  - 查询操作：天然幂等（查一次和查十次结果一样）
  - 删除操作：幂等（删一次和删十次结果一样）
  - 更新操作：不一定（set age=18 是幂等的；incr age 不是）
  - 插入操作：不幂等（插入两次，两条记录）
```

### 五大幂等方案

#### 方案1：唯一约束（数据库）

```
原理：利用DB的唯一索引，重复插入会报错
适用：插入操作（创建订单、创建用户）
```

```java
// 订单表：订单号唯一索引
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    order_no VARCHAR(64) UNIQUE,  -- 唯一索引
    status VARCHAR(20)
);

// 插入订单（幂等）
public void createOrder(Order order) {
    try {
        orderMapper.insert(order);  // 重复插入会报DuplicateKeyException
    } catch (DuplicateKeyException e) {
        // 已存在，直接查询返回
        return orderMapper.selectByOrderNo(order.getOrderNo());
    }
}
```

**优点**：简单，DB保证
**缺点**：只能用于插入，且需要唯一业务号

#### 方案2：Token机制（前端+后端）

```
流程：
  1. 前端请求创建订单前，先拿一个Token（后端生成，存Redis）
  2. 前端提交订单时，带Token
  3. 后端用Lua脚本原子检查+删除Token
  4. Token不存在 → 重复提交，拒绝
```

```java
// 1. 获取Token
@GetMapping("/getToken")
public String getToken() {
    String token = UUID.randomUUID().toString();
    redis.set("idempotent:token:" + token, "1", "NX", "EX", 300);
    return token;
}

// 2. 提交订单（带Token）
@PostMapping("/createOrder")
public Order createOrder(@Header("Idempotent-Token") String token, Order order) {
    // Lua脚本：原子检查+删除
    String lua = """
        if redis.call('get', KEYS[1]) == ARGV[1] then
            return redis.call('del', KEYS[1])
        else
            return 0
        end
        """;
    Long result = jedis.eval(lua,
            Collections.singletonList("idempotent:token:" + token),
            Collections.singletonList(token));
    if (result == 0) {
        throw new RuntimeException("重复提交或Token过期");
    }

    // Token验证通过，创建订单
    return orderService.create(order);
}
```

#### 方案3：状态机（有限状态自动机）

```
订单状态：PENDING → CONFIRMED → PAID → SHIPPED
规则：只能从当前状态 → 下一个状态，不能跳
```

```java
// 状态机：保证状态流转的幂等性
public void confirmOrder(Long orderId) {
    Order order = orderMapper.selectById(orderId);
    if (!order.getStatus().equals("PENDING")) {
        // 已经是CONFIRMED了，直接返回（幂等）
        return;
    }
    order.setStatus("CONFIRMED");
    orderMapper.updateStatus(order);
}
```

#### 方案4：乐观锁（版本号）

```
原理：更新时检查版本号，一致才更新成功
适用：更新操作
```

```java
// 乐观锁：更新库存（幂等）
public boolean deductInventory(Long id, int amount, int oldVersion) {
    String sql = """
        UPDATE inventory
        SET stock = stock - ?, version = version + 1
        WHERE id = ? AND version = ?
        """;
    int rows = jdbcTemplate.update(sql, amount, id, oldVersion);
    return rows > 0;  // 返回true，表示更新成功（幂等，重试也会成功）
}
```

#### 方案5：消费确认（消息队列）

```
Kafka消费：
  1. 消费消息，处理业务
  2. 业务处理完，提交offset（标记已消费）
  3. 如果消费超时，Kafka会重新投递 → 业务要幂等
```

```java
@KafkaListener(topics = "order-topic")
public void consume(Order order) {
    // 1. 先检查是否处理过（用订单号唯一索引）
    if (orderService.isProcessed(order.getOrderNo())) {
        return;  // 已处理，直接返回（幂等）
    }

    // 2. 处理业务
    orderService.process(order);

    // 3. 标记已处理（写入去重表）
    orderService.markProcessed(order.getOrderNo());
}
```

## 它和相似方案的本质区别是什么？

| 方案 | 适用场景 | 优点 | 缺点 |
|---|---|---|---|
| 唯一约束 | 插入操作 | 简单，DB保证 | 只能用于插入 |
| Token机制 | 前端防重 | 彻底防重（前端+后端） | 需要前后端配合 |
| 状态机 | 状态流转 | 保证状态正确流转 | 需要设计状态机 |
| 乐观锁 | 更新操作 | 并发高，冲突少 | 冲突频繁会重试多次 |
| 消费确认 | MQ消费 | MQ保证不丢消息 | 业务要幂等 |

**本质区别**：不同方案解决不同场景的幂等问题，要组合使用。

## 正确使用方式

### 场景1：支付防重（Token机制 + 唯一约束）

```java
// 1. 支付前，获取Token
String token = paymentService.getToken();

// 2. 前端提交支付，带Token
paymentService.pay(orderId, token);

// 3. 后端：Token验证 + 订单号唯一约束
public void pay(Long orderId, String token) {
    // Token验证（Lua脚本，原子操作）
    if (!validateToken(token)) {
        throw new RuntimeException("重复支付");
    }

    // 更新订单状态（乐观锁）
    Order order = orderMapper.selectById(orderId);
    if (!order.getStatus().equals("PENDING")) {
        return;  // 已支付，直接返回（幂等）
    }
    order.setStatus("PAID");
    orderMapper.updateStatus(order);
}
```

### 场景2：MQ消费防重（去重表）

```java
@KafkaListener(topics = "inventory-deduct")
public void consume(DeductMessage msg) {
    // 去重表：msgId唯一约束
    try {
        deduplicateMapper.insert(new Deduplicate(msg.getMsgId()));
    } catch (DuplicateKeyException e) {
        log.info("消息已消费，跳过: {}", msg.getMsgId());
        return;  // 已消费，直接返回（幂等）
    }

    // 没消费过，处理业务
    inventoryService.deduct(msg.getProductId(), msg.getAmount());
}
```

## 边界情况和坑

### 坑1：Token泄露，被恶意重用

```
场景：Token被截获，恶意调用
后果：重复提交成功
```

**解决方案**：
1. **Token一次性**（用完即删）
2. **Token和用户信息绑定**（Redis存 `token:userId`，验证时检查）
3. **HTTPS传输**（防截获）

### 坑2：状态机设计不合理

```
错误：状态流转没限制，PENDING可以直接跳到SHIPPED
后果：业务逻辑错误
```

**解决方案**：
1. **状态机用枚举定义**（不能乱跳）
   ```java
   public enum OrderStatus {
       PENDING,
       CONFIRMED,
       PAID,
       SHIPPED;

       public boolean canTransferTo(OrderStatus next) {
           // 定义合法流转
           return (this == PENDING && next == CONFIRMED)
                   || (this == CONFIRMED && next == PAID)
                   || (this == PAID && next == SHIPPED);
       }
   }
   ```

### 坑3：MQ消费幂等，但业务处理失败

```
场景：消费消息，业务处理失败（如DB宕机）
后果：offset没提交，MQ重试 → 业务还是失败 → 死循环
```

**解决方案**：
1. **消费成功才标记去重**（去重表在业务成功后插入）
2. **重试次数上限**（超过3次，进入死信队列，人工处理）

---

## 我的理解

幂等性的核心是**容忍重试**。网络不可靠，重试是必然的，业务必须能容忍重复操作。

实际生产中，常用组合：
- **前端防重**：Token机制（按钮置灰 + 后端Token验证）
- **接口防重**：唯一约束（订单号、msgId）
- **状态流转**：状态机（保证状态正确）
- **MQ消费**：去重表（防重复消费）

**面试最高频追问**：
1. "怎么保证幂等性？" → Token机制、唯一约束、状态机、乐观锁
2. "Token机制流程？" → 获取Token → 提交带Token → 后端验证Token（Lua脚本原子操作）
3. "MQ消费怎么幂等？" → 去重表（msgId唯一约束）

---

*幂等性不是"防止重复操作"，而是"容忍重复操作"。*
