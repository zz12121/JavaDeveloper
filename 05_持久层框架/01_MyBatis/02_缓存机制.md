# 缓存机制

## 这个问题为什么存在？

> 同一个 SqlSession 内，连续两次执行相同的查询，数据库会被访问几次？

答案：**1 次**。第二次直接从内存拿结果。这个「内存」就是 **MyBatis 的一级缓存**。

如果没有缓存：每次查询都访问数据库 → 网络 IO + 数据库查询开销 → 性能差。

但缓存引入了新的问题：**脏读**（缓存了旧数据）和 **内存溢出**（缓存无上限）。

---

## 它是怎么解决问题的？

### 一级缓存（Local Cache）

```
一级缓存
  → 作用域：SqlSession 级别（默认开启，无法关闭）
  → 存储介质：PerpetualCache（底层是 HashMap）
  → 生命周期：SqlSession 关闭 → 缓存清空
  → 命中条件（必须同时满足）：
    ① SQL 语句完全相同（包括参数）
    ② 同一个 SqlSession
    ③ 相同的查询参数
    ④ 没有开启分页（RowBounds 不同也不命中）
  → 失效时机：
    → 执行了任意 INSERT/UPDATE/DELETE（不管是不是同一张表）
    → 调用 sqlSession.clearCache()
    → SqlSession 关闭
```

**源码关键**：`BaseExecutor` 中的 `localCache`（PerpetualCache 实例）

```java
// BaseExecutor.java
@Override
public <E> List<E> query(MappedStatement ms, Object parameter, ...) {
    BoundSql boundSql = ms.getBoundSql(parameter);
    // 生成缓存 key：由 SQL + 参数 + RowBounds 共同决定
    CacheKey key = createCacheKey(ms, parameter, rowBounds, boundSql);
    return query(ms, parameter, rowBounds, resultHandler, key, boundSql);
}

@SuppressWarnings("unchecked")
@Override
public <E> List<E> query(MappedStatement ms, Object parameter, ..., CacheKey key, ...) {
    // 1. 先查缓存
    List<E> list = (List<E>) localCache.getObject(key);
    if (list != null) {
        // 处理存储过程输出参数（如果有）
        handleLocallyCachedOutputParameters(ms, key, parameter);
        return list;   // ← 缓存命中，直接返回
    }
    // 2. 缓存未命中，查数据库
    list = queryFromDatabase(ms, parameter, rowBounds, resultHandler, key, boundSql);
    return list;
}
```

### 二级缓存（Second Level Cache）

```
二级缓存
  → 作用域：Mapper（namespace）级别，跨 SqlSession
  → 默认：关闭（需要在 XML 中显式开启）
  → 开启方式：<cache/> 或 @CacheNamespace
  → 存储介质：可自定义（默认 PerpetualCache，可换 Ehcache/Redis）
  → 生效条件：
    ① 开启了二级缓存（<cache/>）
    ②  SqlSession 提交或关闭后，缓存才写入（不是实时写入！）
  → 失效时机：
    ① 同一 namespace 下执行了 INSERT/UPDATE/DELETE
    ② flushCache=true（查询语句也可以设置刷新缓存）
  → 配置选项：
    <cache eviction="LRU" flushInterval="60000" size="512" readOnly=true/>
      → eviction：回收策略（LRU/FIFO/SOFT/WEAK）
      → flushInterval：刷新间隔（毫秒），不设置则不定时刷新
      → size：最多缓存对象数（不是字节数！）
      → readOnly：true=返回只读引用（快，但多线程修改有风险）/ false=序列化拷贝（慢，但安全）
```

**二级缓存执行流程**：

```
SqlSession1 查询 → 数据库 → 结果写入二级缓存（session1.close() 时才写入！）
       ↓
SqlSession2 查询 → 先查二级缓存 → 命中 → 直接返回（不访问数据库）
       ↓
SqlSession3 执行 UPDATE（同一 namespace）→ 清空该 namespace 的二级缓存
```


### 二级缓存执行全流程图

```
┌─────────────────────────────────────────────────────┐
│                   CachingExecutor                     │
├─────────────────────────────────────────────────────┤
│  query()                                           │
│    → 先查 TransactionalCacheManager                 │
│        → 再查 delegate（BaseExecutor）               │
│            → 再查数据库                              │
│    → 查询结果写入 TransactionalCache                 │
│        → commit() 时才真正写入二级缓存                │
└─────────────────────────────────────────────────────┘

SqlSession.close()
    ↓
CachingExecutor.commit()
    ↓
TransactionalCache.flushPendingEntries()   ← 真正写入二级缓存
```

**为什么 commit 时才写入？** 防止事务回滚后缓存了脏数据。

## 它和相似方案的本质区别是什么？

| 对比维度 | MyBatis 一级缓存 | MyBatis 二级缓存 | Redis 缓存 | Ehcache |
|---------|----------------|----------------|------------|---------|
| 作用域 | SqlSession（线程内） | Mapper namespace（跨线程） | 跨 JVM | 跨 JVM（可分布式） |
| 存储位置 | JVM 堆内存 | JVM 堆内存（默认） | 独立服务 | 堆内存 / 磁盘 / 远程 |
| 线程安全 | 是（每个线程独立 SqlSession） | 是（Cache 接口实现需线程安全） | 是 | 是 |
| 分布式支持 | ❌ | ❌（默认）/ ✅（Redis 作为二级缓存介质） | ✅ | ✅（Terracotta） |
| 脏数据风险 | 低（SqlSession 内） | 高（多 SqlSession 交叉操作） | 低（有过期机制） | 低 |

**本质区别**：一级缓存是「性能优化」（减少同会话内重复查询），二级缓存是「跨会话共享」（但引入了一致性复杂度）。**实际项目中二级缓存用得很少**——分布式环境下用 Redis 更合适。

## 正确使用方式

### 一级缓存：正常用，注意失效时机
```java
try (SqlSession session = sqlSessionFactory.openSession()) {
    UserMapper mapper = session.getMapper(UserMapper.class);
    User u1 = mapper.selectById(1);   // ← 查数据库
    User u2 = mapper.selectById(1);   // ← 命中一级缓存（同一个 SqlSession）
    System.out.println(u1 == u2);    // true（同一个对象引用）
}
```

### 二级缓存：谨慎开启
```xml
<!-- UserMapper.xml -->
<mapper namespace="com.example.mapper.UserMapper">
    <!-- 开启二级缓存（使用默认配置） -->
    <cache/>

    <!-- 或者精细配置 -->
    <cache eviction="LRU"
           flushInterval="60000"
           size="1024"
           readOnly="true"/>
</mapper>
```

```java
// 必须序列化和反序列化（readOnly=false 时）
public class User implements Serializable { ... }

// 测试二级缓存
try (SqlSession s1 = factory.openSession()) {
    User u1 = s1.getMapper(UserMapper.class).selectById(1);
    // s1 不关闭，二级缓存不会写入！
}
try (SqlSession s2 = factory.openSession()) {
    User u2 = s2.getMapper(UserMapper.class).selectById(1);
    // 二级缓存未命中（s1 未关闭，缓存未写入）
}
```

### Spring 整合后的坑
```java
// ❌ 一级缓存在这里「看起来」不生效
@Autowired
private UserMapper userMapper;

userMapper.selectById(1);   // SqlSession1 查询
userMapper.selectById(1);   // SqlSession2 查询（Spring 每次用新 SqlSession！）
// 原因：Spring 整合后，每次 Mapper 调用都会从 TransactionSynchronizationManager 获取 SqlSession
//       如果没有事务，每次都是新的 SqlSession → 一级缓存永远不共享
```

## 边界情况和坑

### 坑1：Spring 环境下一级缓存「失效」
```
原因：Spring 通过 SqlSessionTemplate 管理 SqlSession，
     无事务时每次 Mapper 调用都开新 SqlSession → 一级缓存无法跨调用共享
解决：开启事务（@Transactional）→ 同一事务内用同一个 SqlSession → 一级缓存生效
```

### 坑2：二级缓存和延迟加载冲突
```
<!-- ❌ 这样配置会报错 -->
<cache readOnly="false"/>   <!-- 需要序列化 -->
<!-- 如果实体类有延迟加载的关联对象（proxy），序列化时会触发 LazyInitializationException -->
解决：要么 readOnly=true，要么确保关联对象已初始化
```

### 坑3：二级缓存穿透问题
```
问题：缓存未命中时还是会查数据库（二级缓存不防穿透）
解决：这不是二级缓存的职责，需要在应用层（如 Service）加分布式缓存（Redis）
```

### 坑4：`flushCache=true` 的默认行为
```
<!-- SELECT 默认 flushCache=false -->
<!-- INSERT/UPDATE/DELETE 默认 flushCache=true -->

<!-- 如果手动设置 SELECT 的 flushCache=true -->
<select id="selectById" flushCache="true" ...>
<!-- 每次查询都清空缓存 → 缓存永远不命中，相当于关闭缓存 -->
```

### 坑5：多表查询的二级缓存不一致
```
问题：UserMapper 和 OrderMapper 都查询 user 表
  → UserMapper 的二级缓存不知道 OrderMapper 更新了 user 表
  → 导致脏读
解决：
  ① 用 <cache-ref namespace="..."/> 共享缓存（但耦合度高）
  ② 放弃二级缓存，用 Redis
```
