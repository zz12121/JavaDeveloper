# AB实验平台设计

> 产品上线新功能后，怎么证明"这个改动真的有效"？AB 实验是最核心的方法论。但设计一个靠谱的 AB 实验平台，涉及流量分桶、指标计算、统计显著性检验、实验隔离等工程挑战。

---

## 题目一：AB实验平台设计

### 业务背景

```
典型AB实验场景：

1. 推荐算法优化
   实验组 A：当前推荐算法
   实验组 B：新推荐算法
   指标：点击率 CTR、转化率 CVR、用户停留时长

2. 首页改版
   实验组 A：旧版首页布局
   实验组 B：新版首页布局
   指标：页面跳出率、下单转化率、GMV

3. 定价策略
   实验组 A：原价
   实验组 B：降价 5%
   指标：订单量、GMV、毛利率

4. push通知文案
   实验组 A：文案 A
   实验组 B：文案 B
   指标：打开率、次日留存

核心约束：
  - 流量正交：一个用户在同一时间只能在一个实验的一个组中
  - 流量互斥：同一维度的实验不能同时跑（首页改版不能同时跑两个）
  - 分层分桶：不同维度的实验可以并行（推荐算法实验 + push文案实验）
  - 统计严谨：实验结果必须经过显著性检验，不能靠拍脑袋
  - 实验可回滚：随时可以关闭实验，用户回到对照组
  - 指标可配置：不同的实验关注不同的业务指标
  - 实验透明：产品经理/数据分析师可以自助配置和查看实验
```

---

### 方案对比

| 方案 | 特点 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|---------|
| 特性开关（Feature Flag） | 简单的 on/off 开关 | 简单快速 | 只能做功能开关，不能做对比实验 | 灰度发布 |
| 自研分流（Hash 分桶） | 用户ID Hash 取模分组 | 完全可控 | 需要自己实现统计检验 | 中等规模、有数据分析团队 |
| 开源方案（Gihtub Flagsmith/Unleash） | 特性管理平台 | 功能丰富 | AB 实验功能偏弱 | Feature Flag 为主 |
| 商业方案（火山引擎 DataTester/Optimizely） | 完整 AB 实验平台 | 统计模型成熟、可视化好 | 收费、数据出域问题 | 大型企业 |
| 自建完整平台 | 分流 + 数据采集 + 统计分析 | 全部可控、数据不出域 | 开发成本高 | 数据敏感型、自有平台 |

**生产推荐**：中小规模自研分流模块 + 数据分析复用现有数仓；大规模直接用火山引擎 DataTester 或 Optimizely。

### 方案详解：自研 AB 实验平台

#### 核心概念

```
层（Layer）：实验的维度，不同层之间流量正交
  Layer 1: 推荐算法实验
  Layer 2: UI 改版实验
  Layer 3: 定价策略实验
  → 同一个用户可以在 3 个层中各分配到不同的实验组

实验（Experiment）：一次具体的对比实验
  Experiment 1: Layer 1 下的 "新推荐算法 v2"
  Experiment 2: Layer 2 下的 "首页改版 v3"

组（Variant）：实验中的不同分组
  control（对照组）: 当前方案
  treatment_1（实验组1）: 新方案 A
  treatment_2（实验组2）: 新方案 B

流量分配：
  实验流量比例：实验占用层的多少流量（如 50%）
  组流量比例：实验内部各组的流量（如 50% 对照 / 25% 实验A / 25% 实验B）
```

#### 分流架构

```
用户请求
   │
   ▼
┌─────────────────────────────────────┐
│          AB SDK / API 网关           │
│                                     │
│  1. 获取 userId                      │
│  2. 对每个 Layer：                    │
│     hash(userId + layerId) % 100    │
│     → 判断用户落在哪个实验的哪个组    │
│  3. 返回配置（variant + 实验参数）    │
└─────────────────────────────────────┘
   │
   ▼
实验配置（feature flags + 实验参数）
   │
   ├──→ 推荐服务：根据 variant 选择推荐算法
   ├──→ 前端：根据 variant 渲染不同UI
   └──→ 定价服务：根据 variant 选择价格策略
```

#### 分流服务核心代码

```java
/**
 * 分流引擎 —— 核心分流逻辑
 * 保证同一个用户在同一实验中始终分到同一组
 */
@Service
public class AssignmentService {

    @Autowired
    private ExperimentCache experimentCache;
    @Autowired
    private AssignmentLogMapper logMapper;
    @Autowired
    private StringRedisTemplate redisTemplate;

    /**
     * 获取用户在某个实验中的分组
     * @param userId       用户ID
     * @param layerId      实验层ID
     * @return 分组信息
     */
    public AssignmentResult getAssignment(Long userId, Long layerId) {
        // 第一步：查询缓存（Redis）
        String cacheKey = "ab:" + layerId + ":" + userId;
        String cached = redisTemplate.opsForValue().get(cacheKey);
        if (cached != null) {
            return AssignmentResult.fromJson(cached);
        }

        // 第二步：计算分桶
        int bucket = hashToBucket(userId, layerId, 1000); // 1000个桶

        // 第三步：查找该桶落入的实验和分组
        List<Experiment> experiments = experimentCache.getRunningExperiments(layerId);
        for (Experiment exp : experiments) {
            if (bucket >= exp.getTrafficStart() * 10
                    && bucket < exp.getTrafficEnd() * 10) {
                // 找到了实验，继续确定分组
                String variant = assignVariant(bucket, exp);
                AssignmentResult result = new AssignmentResult(
                        exp.getId(), exp.getExperimentName(), variant, exp.getConfig(variant));

                // 写缓存
                redisTemplate.opsForValue().set(cacheKey, result.toJson(), 24, TimeUnit.HOURS);
                // 写日志（异步）
                asyncLogAssignment(exp.getId(), variant, userId);
                return result;
            }
        }

        // 没有命中任何实验 → 走默认（对照组）
        return AssignmentResult.defaultResult();
    }

    /**
     * Hash 分桶 —— MurmurHash 保证均匀分布
     * 同一 userId + layerId 始终映射到同一个桶
     */
    private int hashToBucket(Long userId, Long layerId, int totalBuckets) {
        String key = userId + "_" + layerId;
        // MurmurHash3（Guava 提供）
        int hash = Hashing.murmur3_32_fixed().hashUnencodedChars(key).asInt();
        // 取正数后取模
        return Math.abs(hash % totalBuckets);
    }

    /**
     * 确定分组
     */
    private String assignVariant(int bucket, Experiment exp) {
        int localBucket = (bucket - exp.getTrafficStart() * 10);
        int totalBuckets = (exp.getTrafficEnd() - exp.getTrafficStart()) * 10;
        List<Variant> variants = exp.getVariants();

        int cumulative = 0;
        for (Variant v : variants) {
            cumulative += v.getTrafficRatio() * totalBuckets / 100;
            if (localBucket < cumulative) {
                return v.getVariantName();
            }
        }
        return variants.get(variants.size() - 1).getVariantName(); // 兜底
    }
}
```

    @Autowired
    private ExperimentCache experimentCache;

    private final ThreadLocal<Map<String, String>> variantCache = new ThreadLocal<>();

    /**
     * 初始化 —— 在请求入口调用一次，缓存当前请求的分组结果

    public void init(Long userId) {
        Map<String, String> variants = new HashMap<>();
        List<Layer> layers = experimentCache.getAllLayers();

            variants.put(result.getExperimentName(), result.getVariantName());

            return defaultVariant;

        return variants.getOrDefault(experimentName, defaultVariant);
    }

    }

    /**
     * 清理 ThreadLocal
     */
    public void clear() {
        variantCache.remove();
    }
}

### 统计分析引擎

#### 核心统计方法

```java
/**
 * AB 实验统计分析服务
 * 两种核心方法：
 * 1. 比率类指标（CTR/CVR）：用 Z检验（两样本比例检验）
 * 2. 数值类指标（客单价/停留时长）：用 T检验（两样本均值检验）
 */
@Service
public class ExperimentAnalysisService {

    /**
     * 比率类指标检验（如 CTR）
     *
     * 原假设 H0：实验组和对照组的比率没有显著差异
     * 备择假设 H1：实验组的比率显著高于对照组（单尾检验）
     *
     * 统计量 Z = (p1 - p2) / sqrt(p*(1-p) * (1/n1 + 1/n2))
     * 其中 p = (x1 + x2) / (n1 + n2)
     */
    public RatioTestResult testRatio(long controlN, long controlSuccess,
                                      long treatmentN, long treatmentSuccess,
                                      double confidenceLevel) {
        double p1 = (double) controlSuccess / controlN;
        double p2 = (double) treatmentSuccess / treatmentN;
        double p = (double) (controlSuccess + treatmentSuccess)
                / (controlN + treatmentN);

        // 标准误
        double se = Math.sqrt(p * (1 - p) * (1.0/controlN + 1.0/treatmentN));

        // Z 统计量
        double z = (p2 - p1) / se;

        // 提升幅度
        double lift = (p2 - p1) / p1;

        // P 值（查正态分布表）
        double pValue = 1 - normalCdf(z);

        // 是否显著
        boolean significant = pValue < (1 - confidenceLevel);

        // 置信区间
        double zAlpha = normalInv(1 - (1 - confidenceLevel) / 2);
        double ciLower = lift - zAlpha * se / p1;
        double ciUpper = lift + zAlpha * se / p1;

        return new RatioTestResult(p1, p2, lift, ciLower, ciUpper, z, pValue, significant);
    }

    /**
     * 数值类指标检验（如客单价）
     * 使用 Welch's T-Test（不假设方差相等）
     */
    public ValueTestResult testValue(List<Double> controlValues, List<Double> treatmentValues,
                                      double confidenceLevel) {
        // 计算均值和方差
        double mean1 = mean(controlValues);
        double mean2 = mean(treatmentValues);
        double var1 = variance(controlValues, mean1);
        double var2 = variance(treatmentValues, mean2);
        int n1 = controlValues.size();
        int n2 = treatmentValues.size();

        // T 统计量
        double se = Math.sqrt(var1/n1 + var2/n2);
        double t = (mean2 - mean1) / se;

        // 自由度（Welch-Satterthwaite 公式）
        double df = Math.pow(var1/n1 + var2/n2, 2)
                / (Math.pow(var1/n1, 2)/(n1-1) + Math.pow(var2/n2, 2)/(n2-1));

        // P 值
        double pValue = 1 - tDistCdf(t, df);

        // 提升幅度
        double lift = (mean2 - mean1) / mean1;

        return new ValueTestResult(mean1, mean2, lift, t, pValue, df,
                pValue < (1 - confidenceLevel));
    }

    /**
     * 最小样本量计算（实验开始前评估）
     * 回答：需要多少样本才能检测到 X% 的提升？
     *
     * 公式：n = (Z_α + Z_β)² × (p×(1-p) × 2) / δ²
     * 其中 δ = p × expectedLift
     */
    public long calculateMinSampleSize(double baselineRate, double expectedLift,
                                        double confidenceLevel, double power) {
        double alpha = 1 - confidenceLevel;
        double beta = 1 - power;
        double zAlpha = normalInv(1 - alpha/2);
        double zBeta = normalInv(power);
        double delta = baselineRate * expectedLift;

        double n = Math.pow(zAlpha + zBeta, 2) * baselineRate * (1-baselineRate) * 2
                / (delta * delta);
        return (long) Math.ceil(n);
    }
}
```

### 高频追问

#### Q1：实验流量互相重叠怎么办？（互斥实验）

```
场景：产品经理同时在首页做了两个实验
  实验 A：改版顶部 Banner
  实验 B：改版底部推荐区域

问题：如果同一个用户同时参与两个实验，无法判断效果归因

解决方案：
1. 互斥实验组（Mutex Group）
   将实验 A 和 B 放在同一个互斥组中
   一个用户在同一互斥组中只能参与一个实验

2. 代码实现
   // 在互斥组内先分配实验，再在实验内分配组
   int bucket = hash(userId, mutexGroupId);
   if (bucket < 50) {
       // 实验 A 区域
       String variant = hash(userId, experimentAId) → control/treatment
   } else {
       // 实验 B 区域
       String variant = hash(userId, experimentBId) → control/treatment
   }
```

#### Q2：用户中途切换分组（实验中分流不稳定）怎么办？

```
问题：
  用户今天分到对照组，明天分到实验组 → 数据污染

原因：
  1. 用户未登录，分流用 deviceId → 清了 Cookie 换了 deviceId
  2. 做了哈希算法升级，导致桶位变化
  3. 代码 Bug，分桶逻辑不幂等

解决方案：
1. 登录用户用 userId 分流（稳定）
2. 未登录用户用 deviceId 分流 + localStorage 持久化
3. 登录后绑定：用 userId 重新计算分组，覆盖之前的 deviceId 分组
4. 分组结果持久化到 Redis + DB，首次计算后后续直接读取
5. 哈希算法一旦确定不能修改（升级必须新建层）
```

#### Q3：辛普森悖论 —— 整体数据好看但分群数据差？

```
辛普森悖论示例：
  全量看：实验组 CVR 高于对照组
  按新用户/老用户拆开看：实验组 CVR 都低于对照组

原因：实验组中新用户占比更高（新用户天然转化率低拉低了整体）

解决方案：
1. 分层分析：按用户画像（新/老、VIP/普通、地区）分别看指标
2. 实验前做 AA 测试：验证分桶是否均匀
3. 分层随机化：分流时考虑用户特征，保证各组特征分布一致
4. 回归分析：用多元回归控制混淆变量
```

#### Q4：实验需要跑多久？

```
经验规则：
1. 至少 2 周（覆盖完整周期，排除周几效应）
2. 至少 1 个完整业务周期（如电商至少包含一次大促）

计算方式：
  最小样本量 ÷ 日均流量 × 1.2（缓冲）= 最少天数

例：
  基线 CTR 3%，预期提升 10%，置信度 95%，统计效力 80%
  最小样本量 = ~150,000（每组）
  日均活跃用户 = 100,000
  每组 50,000/天 → 需要 3 天
  但按经验至少跑 14 天 → 实际跑 14 天

不要过早停实验：
  「第一天实验组数据很好，立刻全量上线」→ 大概率假阳性
  必须等达到最小样本量 + 预定运行时间后才能下结论
```

### 生产 Checklist

```
□ 分流算法是否幂等（同一用户多次请求得到同一分组）
□ 不同层的实验是否流量正交（不互相干扰）
□ 同一互斥组内的实验是否互斥（不重叠）
□ 分组结果是否持久化（不因算法升级而变化）
□ 实验配置是否有缓存（避免每次请求查 DB）
□ 是否支持灰度放量（1% → 5% → 20% → 50% → 100%）
□ 实验关闭后用户是否平滑回到默认值
□ 统计检验是否正确使用（比率用 Z 检验，数值用 T 检验）
□ 是否做了 AA 测试验证分桶均匀性
□ 实验报告是否包含分群分析（新/老用户、不同地区）
□ 实验数据是否有数据质量监控（埋点漏报/重复上报）
□ 实验平台是否有权限控制（不能随便创建线上实验）
□ 是否有实验过期自动关闭机制（防止遗忘的实验一直跑）
```
