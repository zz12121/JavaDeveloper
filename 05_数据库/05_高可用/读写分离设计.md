# 读写分离设计

## 这个问题为什么存在？

> 读多写少的业务，所有请求都打主库 → 主库CPU、IO打满。
> 但主库能承载的读请求有限。

**读写分离的核心目标**：把读请求分流到从库，减轻主库压力。

---

## 它是怎么解决问题的？

### 读写分离的架构模式

```
架构1：客户端直连（最简单）
┌─────┐   读   ┌─────┐
│客户端│ ──────→ │ 从库1  │
│     │        └─────┘
│     │   写   ┌─────┐
│     │ ──────→ │ 主库  │
└─────┘        └─────┘
问题：客户端需要自己判断"读请求走从库，写请求走主库"
→ 业务代码复杂，容易出错
```

```
架构2：代理层（推荐）
┌─────┐    ┌─────────┐    ┌─────┐
│客户端│──→ │ 代理层  │──→ │ 从库1  │
│     │    │(ProxySQL)│    └─────┘
│     │    └─────────┘    ┌─────┐
│     │──→ ────────→──→ │ 从库2  │
└─────┘                  └─────┘
优点：业务代码无感知，代理层自动路由
```

```
架构3：中间件层（ShardingSphere）
┌─────┐    ┌─────────────┐    ┌─────┐
│客户端│──→ │  中间件     │──→ │ 从库1  │
│     │    │(ShardingJDBC)│    └─────┘
└─────┘    └─────────────┘
优点：集成在应用层，不需要独立代理
```

### 读写分离的路由规则

**基本规则**：

```java
// 规则1：写请求 → 主库
jdbcTemplate.update("INSERT INTO orders(...) VALUES (...)");
jdbcTemplate.update("UPDATE orders SET ...");
jdbcTemplate.update("DELETE FROM orders ...");

// 规则2：读请求 → 从库（负载均衡）
jdbcTemplate.query("SELECT * FROM orders WHERE ...");  // → 从库

// 规则3：事务内的读 → 主库（避免主从延迟导致读不到）
@Transactional
public void process() {
    // 写了数据，紧接着读 → 必须走主库
    orderDao.insert(order);
    Order o = orderDao.selectById(order.getId());  // → 主库
}
```

**ShardingSphere配置示例**：

```yaml
masterslaves:
  ms_ds:
    masterDataSourceName: master_ds
    slaveDataSourceNames: 
      - slave_ds_0
      - slave_ds_1
    loadBalanceAlgorithm: round_robin  # 负载均衡算法
```

### 主从延迟的处理策略

**问题场景**：

```
时刻T1：主库写入订单（事务提交）
时刻T2：应用立即读订单列表（读从库）
→ 从库延迟2秒 → 读不到刚才写入的订单
→ 用户以为下单失败，重复下单
```

**解决方案1：粘性读（Sticky Read）**

```java
// 写完之后，强制读主库一段时间（比如5秒）
public Order createOrder(Order order) {
    orderDao.insert(order);  // 写主库
    
    // 把连接绑定到主库（接下来5秒内的读都走主库）
    stickyReadManager.bindToMaster(5, TimeUnit.SECONDS);
    
    return orderDao.selectById(order.getId());  // 读主库
}
```

**解决方案2：写后读关键数据走主库**

```java
// 写事务提交后，关键查询强制走主库
public void afterCreateOrder(Long orderId) {
    // 刚创建的订单，强制读主库（避免延迟）
    Order order = orderDao.selectFromMaster(orderId);
}
```

**解决方案3：接受延迟（非关键数据）**

```java
// 报表类查询，可以接受几秒延迟
public List<Order> getOrderReport() {
    // 报表查询走从库，接受延迟
    return orderDao.selectFromSlave(reportQuery);
}
```

### 代理层选型

| 中间件 | 类型 | 优点 | 缺点 |
|--------|------|------|------|
| **ProxySQL** | 独立代理 | 性能高、支持连接池复用、支持读写分离规则灵活配置 | 需要独立部署、运维成本 |
| **MyCat** | 独立代理 | 支持分库分表+读写分离 | 社区不活跃、文档少 |
| **ShardingSphere** | 中间件层（Jar包） | 无独立部署、支持分库分表+读写分离、社区活跃 | 侵入应用层 |
| **MySQL Router** | 官方轻量代理 | 简单、官方支持 | 功能简单、不支持复杂路由 |

**推荐**：
- 小团队、简单场景 → **ProxySQL**（部署简单、性能足够）
- 需要分库分表+读写分离 → **ShardingSphere**
- 不差钱 → **云数据库读写分离实例**（全托管）

---

## 它和相似方案的本质区别是什么？

### 读写分离 vs 缓存（Redis）

| 维度 | 读写分离 | 缓存（Redis） |
|------|----------|--------------|
| 解决问题 | 减轻主库读压力 | 减轻数据库全部压力 |
| 数据一致性 | 最终一致（延迟） | 可能不一致（缓存过期/击穿） |
| 适用场景 | 读多写少、可以接受延迟 | 读多写少、热点数据 |
| 复杂度 | 中（需要中间件/代理） | 中（缓存一致性处理） |

**本质区别**：读写分离是"数据库层"的优化，缓存是"应用层"的优化。可以同时使用。

### 代理层 vs 中间件层

| 维度 | 代理层（ProxySQL） | 中间件层（ShardingSphere） |
|------|-------------------|-----------------------|
| 部署方式 | 独立进程 | Jar包，集成在应用中 |
| 性能 | 网络多一跳 | 无额外网络开销 |
| 运维成本 | 需要独立运维 | 无独立运维 |
| 适用场景 | 多语言、多应用共享 | Java技术栈、应用独立部署 |

**本质区别**：代理层是"侧方位"，中间件层是"嵌入式"。

## 正确使用方式

### 1. ProxySQL 快速配置

```sql
-- 1. 添加主库和从库
INSERT INTO mysql_servers (hostgroup_id, hostname, port) 
VALUES 
  (1, 'master_host', 3306),   -- hostgroup 1 = 写组
  (2, 'slave1_host', 3306),  -- hostgroup 2 = 读组
  (2, 'slave2_host', 3306);

-- 2. 配置读写分离规则
INSERT INTO mysql_query_rules (rule_id, active, match_digest, destination_hostgroup, apply)
VALUES 
  (1, 1, '^SELECT', 2, 1),  -- SELECT → 读组
  (2, 1, '.*', 1, 1);        -- 其他 → 写组

-- 3. 加载配置到运行时
LOAD MYSQL SERVERS TO RUNTIME;
LOAD MYSQL QUERY RULES TO RUNTIME;

-- 4. 保存到持久化配置
SAVE MYSQL SERVERS TO DISK;
SAVE MYSQL QUERY RULES TO DISK;
```

### 2. ShardingSphere 读写分离配置

```yaml
# application.yml
spring:
  shardingsphere:
    datasources:
      master_ds:
        type: com.zaxxer.hikari.HikariDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        jdbc-url: jdbc:mysql://master:3306/db
        username: root
        password: xxx
      slave_ds_0:
        type: com.zaxxer.hikari.HikariDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        jdbc-url: jdbc:mysql://slave1:3306/db
        username: root
        password: xxx
    masterslave:
      name: ms
      master-data-source-name: master_ds
      slave-data-source-names: slave_ds_0
      load-balance-algorithm-class-name: org.apache.shardingsphere.core.strategy.loadbalance.RoundRobinLoadBalanceAlgorithm
```

### 3. 事务内强制走主库

```java
// Spring事务：事务内的读强制走主库
@Transactional
public void process() {
    // 写操作
    orderDao.insert(order);
    
    // 紧接着的读，必须走主库（避免主从延迟）
    // ShardingSphere会自动把事务内的读路由到主库
    Order saved = orderDao.selectById(order.getId());
}
```

## 边界情况和坑

### 坑1：事务内的读走了从库

```
现象：
@Transactional
public void process() {
    orderDao.insert(order);
    Order o = orderDao.selectById(order.getId());  // 走了从库！
}
→ 主从延迟 → 读不到刚写入的数据
```

**原因**：
- 某些中间件（如早期版本的MyCat）不判断事务上下文
- 或者配置错误（没有配置"事务内读主库"）

**解决**：
```yaml
# ShardingSphere配置：事务内读强制走主库
spring:
  shardingsphere:
    masterslave:
      load-balance-algorithm-class-name: xxx
      # 关键：开启这个配置
      allow—range—query—with—in—memory—pagination: false
```

### 坑2：主从延迟导致的数据不一致

```
场景：
1. 用户支付订单（写主库）
2. 支付回调：查询订单状态（读从库）
3. 从库延迟3秒 → 读到的还是"未支付"状态
→ 业务逻辑错误
```

**解决**：
1. 关键业务（支付回调）强制走主库
2. 或者：写完后，把订单ID加入"缓存白名单"，5秒内的读走主库

### 坑3：从库延迟监控不到位

```
现象：从库延迟越来越大，但应用层不知道
→ 所有读请求都读到旧数据
```

**解决**：

```sql
-- 1. 监控从库延迟
SHOW SLAVE STATUS\G
-- 关键指标：Seconds_Behind_Master

-- 2. 从库延迟超过阈值，自动降级到主库
-- ShardingSphere支持配置：从库延迟超过5秒，读请求自动路由到主库
```

### 坑4：读写分离导致连接池不够用

```
现象：
- 原先：应用只用1个连接池（连主库）
- 读写分离后：需要2个连接池（主库 + 从库）
→ 连接数翻倍，可能超过数据库最大连接数
```

**解决**：
1. 主库连接池可以小一点（写少）
2. 从库连接池可以大一点（读多）
3. 使用支持连接池复用的代理（如ProxySQL）

## 我的理解

读写分离是"性价比"很高的优化方案：
- 成本：只需要加从库 + 配置中间件
- 收益：读性能线性提升（加从库即可）

但有几个"坑"必须注意：
1. **主从延迟**：这是读写分离的"原罪"，必须有应对方案
2. **事务内的读**：必须走主库，否则数据不一致
3. **中间件选型**：根据团队技术栈选择，不要盲目追新

## 面试话术

**Q：读写分离怎么实现？**

"读写分离一般通过三层实现：

**第一层：代理层**（如ProxySQL）。应用只连ProxySQL，由ProxySQL根据SQL类型路由：
- `SELECT` → 从库
- `INSERT/UPDATE/DELETE` → 主库

优点：对应用透明，多语言通用。缺点：多一跳网络，需要独立运维。

**第二层：中间件层**（如ShardingSphere）。以Jar包形式集成在应用中，中间件自动路由：
- 写请求 → 主库
- 读请求 → 从库（负载均衡）
- 事务内的读 → 强制主库（避免主从延迟）

优点：无额外网络开销，功能强大（支持分库分表）。缺点：只支持Java技术栈。

选哪种取决于团队技术栈和运维能力。小团队推荐ProxySQL，Java技术栈且需要分库分表的推荐ShardingSphere。"

**Q：主从延迟怎么解决？**

"主从延迟是读写分离的'原罪'，几个解决方案：

**方案1：粘性读**。写完数据后，强制这个会话/线程在接下来5秒内读主库。

**方案2：关键业务强制走主库**。比如支付回调，必须读主库，不能走从库。

**方案3：接受延迟**。非关键业务（比如报表查询），可以接受几秒延迟，直接读从库。

**方案4：并行复制**。从库开启并行复制（`slave_parallel_workers`），减少主从延迟。

推荐组合使用：关键业务走主库 + 非关键业务走从库 + 从库开启并行复制。"

**Q：事务内的读为什么要走主库？**

"因为主从延迟。如果事务内写了数据，紧接着读这个数据，但读请求被路由到从库，而从库还没同步到这个事务，就会读不到数据。

比如：
```java
@Transactional
public void process() {
    orderDao.insert(order);  // 写主库
    Order o = orderDao.selectById(order.getId());  // 如果走从库，可能读不到！
}
```

所以，成熟的中间件（如ShardingSphere）都会自动把**事务内的读**路由到主库。这个是默认行为，不需要额外配置。"

## 本文总结

| 主题 | 核心要点 |
|------|----------|
| 读写分离目的 | 减轻主库读压力，提升整体读性能 |
| 架构模式 | 客户端直连（简单但侵入）、代理层（推荐）、中间件层（功能强） |
| 路由规则 | 写→主库、读→从库、事务内读→主库 |
| 主从延迟处理 | 粘性读、关键业务走主库、并行复制 |
| 中间件选型 | ProxySQL（简单）、ShardingSphere（功能强） |
| 核心坑点 | 事务内读走从库、延迟监控不到位、连接池不够用 |

**核心公式**：
```
读多写少场景 = 主库（写） + N个从库（读） + 中间件（路由） + 主从延迟处理
```
