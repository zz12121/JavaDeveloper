## 参考答案

### 一、最基础的 Redis 分布式锁（有缺陷）

```lua
-- 获取锁
SET lock_key <unique_value> NX PX 30000

-- 释放锁（必须原子检查 value，防止误删别人的锁）
if redis.call("get",KEYS[1]) == ARGV[1] then
    return redis.call("del",KEYS[1])
else
    return 0
end
```

**三个致命缺陷**：

```
缺陷1：锁超时，任务还在执行
  → 锁过期被其他线程获取 → 两个线程同时进入临界区

缺陷2：GC STW（Stop-The-World）
  → 线程 A 获取锁，发生 GC 停顿 60 秒
  → 锁超时自动释放
  → 线程 B 获取锁
  → GC 结束，线程 A 和 B 同时执行 → 临界区被突破

缺陷3：value 不唯一（上面已用 Lua 解决）
```

---

### 二、Redisson 的解决方案（生产推荐）

```java
RLock lock = redissonClient.getLock("order_lock");

// 1. 尝试加锁，默认 watchdog 自动续租
lock.lock();
// 等同于：lock.lockAsync(30, TimeUnit.SECONDS, ...)

try {
    // 业务逻辑
    processOrder();
} finally {
    lock.unlock();
}
```

**watchdog 续租机制**：

```
1. 加锁成功后，后台启动定时任务
2. 每 internalLockLeaseTime / 3 = 10 秒，自动执行：
   PERISTENCE_LOCK <key> <value> PX 30000
3. 只要持有锁的线程还活着，锁就不会过期
4. 线程宕机 → watchdog 线程也挂了 → 30 秒后锁自动释放
```

**优点**：
- 解决「锁超时但任务未完成」问题
- Lua 脚本保证原子性
- 自动处理锁续租，业务无感知
- 支持可重入（state 计数）

### 三、Redlock 算法（多实例，解决单点故障）

**背景**：单个 Redis master 宕机，锁丢失。

**算法流程**：

```
前提：N 个独立的 Redis master（无主从关系，通常 N=5）

1. 客户端获取当前时间戳 T1（毫秒）
2. 依次向 N 个 Redis 发送：
   SET lock_key <unique_value> NX PX <expire_time>
3. 计算总耗时 = T2 - T1
4. 判断：
   成功获取锁的数量 >= N/2 + 1  AND  总耗时 < expire_time
   → 获取锁成功，有效时间 = expire_time - 总耗时
   → 失败：向所有实例发送释放锁命令（Lua 脚本）
```

**为什么要超过半数**：
```
防止脑裂时两个客户端同时获取锁：
  5 个实例，需要 >= 3 个
  客户端 A 获取了实例 1,2,3
  客户端 B 获取了实例 3,4,5  → 不够多数，失败
```

### 四、Redlock 的争议（面试加分项）

**Martin Kleppmann 的批评（2016）**：

```
问题1：依赖系统时钟
  → 如果系统时钟发生跳跃（NTP 同步），锁的有效期计算会出错
  → 例：线程 A 获取锁，时钟向前跳 10 秒 → 锁提前过期 → 线程 B 获取锁

问题2：GC STW / 网络延迟
  → 客户端 A 获取锁后发生长时间 GC
  → 锁过期，客户端 B 获取锁
  → GC 结束，A 和 B 同时进入临界区

问题3：持久化问题
  → 客户端 A 获取锁，master 刚持久化就宕机
  → 哨兵切换，新 master 没有这条锁记录
  → 客户端 B 可以再次获取同一把锁
```

**Antirez（Redis 作者）的反驳**：

```
1. 时钟漂移：认为这是运维问题，不应该用 NTP 强行修改时钟
2. GC STW：任何分布式锁方案都无法完全规避（ZooKeeper 也有类似问题）
3. 持久化：可以用 AOF 的 fsync always，但性能很差
   → 建议：锁 key 设置 TTL，即使丢失也只影响一小段时间
```

**结论（面试回答）**：

```
Redlock 适合：
  → 对一致性要求不高（偶尔重复执行可接受）
  → 性能要求高（比 ZooKeeper 快一个数量级）

不适合：
  → 金融场景（需要强一致）
  → 建议用 ZooKeeper / etcd（基于租约机制，不依赖系统时钟）
```

### 五、ZooKeeper 分布式锁（强一致方案）

**实现原理**：

```
1. 在 /locks/order_lock 下创建临时顺序节点
   → 每个客户端创建：/locks/order_lock/lock-0000000001
                            /locks/order_lock/lock-0000000002

2. 判断自己是否是最小节点？
   → 是：获取锁成功
   → 否：监听前一个节点的删除事件（watch）

3. 释放锁：
   → 客户端断开连接（宕机）→ 临时节点自动删除 → 后一个节点收到通知 → 获取锁
```

**优点**：
- **公平锁**：按顺序获取，不会出现饥饿
- **可监听**：watch 机制，锁释放时主动通知（Redis 需要轮询或 Pub/Sub）
- **强一致**：基于 ZAB 协议，不依赖系统时钟

**缺点**：
- 性能：ZK 写操作需要过半 Ack，延迟高（~10ms vs Redis ~1ms）
- 复杂：需要维护 ZK 客户端会话

---

### 六、选型决策树（面试总结）

```
问：对一致性要求高吗？

是（金融、库存） → ZooKeeper / etcd
  → 基于租约，不依赖时钟
  → 支持 watch，公平锁

否（社交、缓存、幂等可重入） → Redisson（Redis）
  → 性能高，watchdog 自动续租
  → 偶尔重复执行可接受

对性能极致要求 → Redis + 业务幂等
  → 锁只做「尽量唯一」，业务端用唯一 ID 去重
```
