# 私服搭建

> 私服（Private Repository）不是「公司的 Maven Central」，而是**内部依赖的管理中枢、构建加速缓存、安全审计入口**。没有私服，每个开发者都从公网下载依赖，慢且不可控。

---

## 这个问题为什么存在？

没有私服时，团队会遇到这些问题：

```
问题 1：构建速度慢
  每个新人或 CI 机器都要从 Maven Central 下载依赖
  → 国外服务器，下载慢
  → 每次 `mvn clean package` 都要重新下载（如果本地仓库清空了）

问题 2：内部模块无法共享
  公司写了工具库 common-utils.jar
  → 没法 `mvn install` 到每个人本地
  → 只能手动拷贝 jar 包（回到石器时代）

问题 3：依赖来源不可控
  开发者可以引入任何公网依赖
  → 可能引入有漏洞的版本
  → 可能引入许可证不兼容的库（如 AGPL）

问题 4：发布不稳定
  生产发布时需要从公网下载依赖
  → 公网服务抖动 → 发布失败
  → 或者 Maven Central 被墙了 → 完全无法构建
```

**私服解决的核心问题**：

1. **加速构建**：缓存公网依赖，内网高速下载
2. **内部发布**：内部模块发布到私服，全公司可依赖
3. **安全管控**：统一依赖来源，审计漏洞版本
4. **稳定发布**：不依赖公网，内网私服永远在线

## 它是怎么解决问题的？

### Nexus Repository Manager OSS（最主流方案）

```
Nexus 架构：
                    ┌─────────────────────────────────┐
                    │       Nexus Repository Manager    │
                    │                                 │
开发者 ──发布──→  │  Hosted Repo（内部仓库）        │
                    │    ├─ releases（正式版）        │
开发者 ──下载──→  │    └─ snapshots（快照版）        │
                    │                                 │
                    │  Proxy Repo（代理仓库）         │
                    │    ├─ maven-central（代理 Central） │
                    │    └─ aliyun-mirror（代理阿里云）  │
                    │                                 │
                    │  Group Repo（仓库组，聚合用）    │
                    │    └─ public（聚合 hosted + proxy） │
                    └─────────────────────────────────┘
```

### 三种仓库类型

| 类型 | 作用 | 典型例子 |
|------|------|---------|
| **Hosted**（宿主仓库） | 存储内部发布的依赖 | `releases`、`snapshots` |
| **Proxy**（代理仓库） | 代理公网仓库，缓存下载的依赖 | `maven-central`、`aliyun` |
| **Group**（仓库组） | 聚合多个仓库，提供一个统一地址 | `public`（聚合所有仓库） |

**访问逻辑**：
```
开发者请求依赖 spring-core
  ↓
请求先到 Group 仓库（如 public）
  ↓
Group 按顺序查找：
  1. 本地 Hosted 仓库有没有？→ 有就返回
  2. 没有 → 查 Proxy 仓库（先查缓存，没有就从公网下载并缓存）
  ↓
返回给开发者
```

### 安装与配置（Nexus OSS）

```bash
# 1. 下载 Nexus（以 3.x 为例）
wget https://download.sonatype.com/nexus/3/latest-unix.tar.gz
tar -xzf latest-unix.tar.gz
cd nexus-3.x.x

# 2. 启动（默认端口 8081）
./bin/nexus start

# 3. 访问 http://localhost:8081
# 默认账号：admin
# 默认密码：在 sonatype-work/nexus3/admin.password 中
```

**Maven 配置连接私服**（`~/.m2/settings.xml`）：

```xml
<!-- 1. 配置私服账号（Server 标签） -->
<servers>
    <server>
        <id>nexus-releases</id>
        <username>admin</username>
        <password>your-password</password>
    </server>
    <server>
        <id>nexus-snapshots</id>
        <username>admin</username>
        <password>your-password</password>
    </server>
</servers>

<!-- 2. 配置镜像（让所有请求都走私服） -->
<mirrors>
    <mirror>
        <id>nexus</id>
        <mirrorOf>*</mirrorOf>   <!-- 所有仓库请求都走这个镜像 -->
        <url>http://localhost:8081/repository/maven-public/</url>
    </mirror>
</mirrors>
```

### 发布内部依赖到私服

```xml
<!-- pom.xml 中配置 distributionManagement -->
<distributionManagement>
    <repository>
        <id>nexus-releases</id>   <!-- 对应 settings.xml 中的 server id -->
        <url>http://localhost:8081/repository/maven-releases/</url>
    </repository>
    <snapshotRepository>
        <id>nexus-snapshots</id>
        <url>http://localhost:8081/repository/maven-snapshots/</url>
    </snapshotRepository>
</distributionManagement>
```

```bash
# 发布到私服
mvn clean deploy

# Maven 会根据版本号自动选择 releases 或 snapshots：
# - 1.0.0          → releases
# - 1.0.0-SNAPSHOT → snapshots
```

## 深入原理

### Nexus vs JFrog Artifactory

| 维度 | Nexus | JFrog Artifactory |
|------|-------|---------------------|
| 开源版本 | 有（OSS，功能够用） | 无（只有 30 天试用） |
| 支持包类型 | Maven、npm、Docker、PyPI 等 | 更多（Bower、CocoaPods、Conan 等） |
| 高可用 | 需要手动配置集群 | 原生支持多活集群 |
| 权限管理 | 基础（角色 + 用户） | 精细（到仓库、到路径） |
| 价格 | 开源免费，企业版便宜 | 企业版贵 |
| 适合场景 | 大多数中小公司 | 大型公司、多语言制品管理 |

**为什么大多数公司选 Nexus？**

够用且免费。Nexus OSS 支持 Maven、npm、Docker 镜像仓库，对 Java 团队完全够用。

### 私服 vs 不加管理的 Maven Central 直连

| 维度 | 有私服 | 无私服（直连 Central） |
|------|--------|---------------------|
| 构建速度 | 内网高速（缓存命中） | 看公网速度，国外服务器慢 |
| 稳定性 | 高（内网可控） | 低（公网服务可能宕机或被墙） |
| 内部依赖 | 可以发布和共享 | 无法共享（只能手动拷贝 jar） |
| 安全审计 | 可以集成漏洞扫描（如 OWASP） | 无法审计 |
| 成本 | 需要一台服务器 + 运维 | 零成本 |

## 正确使用方式

### 1. 用阿里云镜像加速公网依赖下载

```xml
<!-- settings.xml 中配置阿里云代理（国内团队强烈推荐） -->
<mirrors>
    <mirror>
        <id>aliyun</id>
        <mirrorOf>central</mirrorOf>   <!-- 只代理 central，不影响其他仓库 -->
        <url>https://maven.aliyun.com/repository/public</url>
    </mirror>
</mirrors>
```

**如果同时有私服和阿里云**：
```
推荐架构：
  Maven 配置私服为 mirrorOf=*
  私服中配置 Proxy Repository 指向阿里云
  → 所有请求走私服 → 私服缓存阿里云的包
```

### 2. 设置依赖白名单（安全管控）

```xml
<!-- Nexus 专业版功能：Repository Health Check -->
<!-- 开源方案：用 OWASP dependency-check-maven 插件 -->
<plugin>
    <groupId>org.owasp</groupId>
    <artifactId>dependency-check-maven</artifactId>
    <version>7.4.0</version>
    <executions>
        <execution>
            <goals>
                <goal>check</goal>   <!-- 构建时检查依赖漏洞 -->
            </goals>
        </execution>
    </executions>
</plugin>
```

### 3. 定期清理 SNAPSHOT 包

```
问题：
  snapshots 仓库会越来越大（每个 SNAPSHOT 都保存）
  → 磁盘空间不足
```

**Nexus 配置自动清理**：
```
Nexus 管理界面 → Repository → Repositories → maven-snapshots
→ Edit → Storage
→ 设置 「Maximum Snapshot Count」（如 3，只保留最新 3 个 SNAPSHOT）
→ 或者设置 TTL（Time To Live，如 30 天）
```

## 边界情况和坑

### 1. `mvn deploy` 失败：401 Unauthorized

```
报错：
Failed to execute goal org.apache.maven.plugins:maven-deploy-plugin:2.8.2:deploy
(default-deploy) on project xxx: Failed to retrieve remote metadata...
Return code is: 401, ReasonPhrase: Unauthorized.
```

**原因**：`pom.xml` 中 `distributionManagement` 的 `<id>` 和 `settings.xml` 中 `<server>` 的 `<id>` 不匹配。

**解决**：确保两者 id 完全一致。

### 2. SNAPSHOT 依赖不更新（缓存问题）

```
场景：
  同事发布了 new-version-1.0.0-SNAPSHOT
  你本地 `mvn package` 还是用的旧 SNAPSHOT
```

**解决**：
```bash
# 方法 1：强制更新 SNAPSHOT
mvn clean package -U

# 方法 2：在 pom.xml 中配置 SNAPSHOT 更新策略
<repositories>
    <repository>
        <id>nexus-snapshots</id>
        <url>http://localhost:8081/repository/maven-snapshots/</url>
        <snapshots>
            <enabled>true</enabled>
            <updatePolicy>always</updatePolicy>   <!-- 每次都检查更新 -->
        </snapshots>
    </repository>
</repositories>
```

### 3. 私服磁盘空间不足

```
Nexus 默认存储路径：sonatype-work/nexus3/blobs/
 → 所有依赖、Docker 镜像都存在这里
 → 不清理会撑满磁盘
```

**解决**：
```bash
# 1. 配置定时任务清理废弃的 SNAPSHOT
# Nexus 管理界面 → System → Tasks → Create Task
# 选择 「Purge SNAPSHOTS」

# 2. 定期运行 `mvn dependency:purge-local-repository` 清理本地仓库
mvn dependency:purge-local-repository -DreResolve=false
```

### 4. 内网私服无法访问公网（代理配置）

```
场景：
  公司内网私服无法直接访问 Maven Central
  需要通过 HTTP 代理
```

**解决**：在 `settings.xml` 中配置代理：
```xml
<proxies>
    <proxy>
        <id>http-proxy</id>
        <active>true</active>
        <protocol>http</protocol>
        <host>proxy.company.com</host>
        <port>8080</port>
        <!-- 如果代理需要认证 -->
        <username>proxy-user</username>
        <password>proxy-pass</password>
    </proxy>
</proxies>
```
