# 配置中心

## 这个问题为什么存在？

> 没有配置中心，每个微服务都有自己的 `application.yml`，改一个配置要**重新打包、重启所有实例**。  
> 而且**不同环境的配置分散**，`dev`、`test`、`prod` 的配置没人统一管理，容易出错。

**配置中心的本质**：把配置从**本地文件**迁移到**远程服务器**，支持**动态刷新**、**版本管理**、**灰度发布**。

---

## 它是怎么解决问题的？

### Nacos Config 的架构

```
┌─────────────────────────────────────────────┐
│  Nacos Config Server（配置中心）            │
│    ├─ 配置存储（MySQL / 嵌入式数据库）     │
│    ├─ 配置推送（Long Polling / WebSocket） │
│    └─ 版本管理（历史配置 + 回滚）         │
└─────────────────────────────────────────────┘
        ↑ 拉取配置          ↓ 推送变更
┌──────────────────┐   ┌──────────────────┐
│ OrderService     │   │ UserService      │
│ （客户端）       │   │ （客户端）        │
└──────────────────┘   └──────────────────┘
```

**配置结构（Nacos 的三元组）**：

```
Namespace（命名空间）→ 区分环境（dev / test / prod）
    └─ Group（分组）→ 区分项目 / 租户
        └─ Data ID（配置 ID）→ 具体的配置文件
```

---

### 配置加载流程

```
Spring Boot 启动
    ↓
1. 读取 bootstrap.yml（配置中心地址、Data ID）
    ↓
2. 从 Nacos Config Server 拉取配置
    ↓
3. 把远程配置注入到 Spring 环境（Environment）
    ↓
4. 继续启动（读到的配置就像本地的一样）
```

**bootstrap.yml vs application.yml**：

| | bootstrap.yml | application.yml |
|--|----------------|-------------------|
| 加载时机 | **最早**（配置中心地址必须在这里） | 较晚 |
| 用途 | 连接配置中心（Nacos 地址、Data ID） | 业务配置 |
| 注意 | **不能省略**，否则连不上配置中心 | 可以用远程配置覆盖 |

---

### 配置动态刷新（@RefreshScope）

**问题**：改了配置，要不要重启服务？

**Nacos Config 的解法**：`@RefreshScope`（配置变更时，重新创建 Bean）。

```java
// 1. Controller 加 @RefreshScope
@RefreshScope
@RestController
public class TestController {

    @Value("${myapp.timeout:5000}")
    private int timeout;  // ← 配置变更后，value 会自动更新

    @GetMapping("/timeout")
    public int getTimeout() {
        return timeout;
    }
}
```

**刷新原理**：

```
Nacos Config Client 长轮询（Long Polling）
    ↓ 服务端配置变更
Nacos Config Client 收到变更通知
    ↓
发布 RefreshEvent
    ↓
@RefreshScope 标记的 Bean 被销毁
    ↓
下次调用时重新创建 Bean（用新配置）
```

---

### 配置优先级（高 → 低）

```
1. Nacos Config（Profile 配置，如 myapp-dev.yml）
    ↓ 覆盖
2. Nacos Config（默认配置，如 myapp.yml）
    ↓ 覆盖
3. 本地 application-{profile}.yml
    ↓ 覆盖
4. 本地 application.yml
```

> **远程配置优先级高于本地**，所以可以用远程配置**覆盖本地配置**（方便灰度、紧急修复）。

## 它和相似方案的本质区别是什么？

### Nacos Config vs Apollo vs Spring Cloud Config

| | Nacos Config | Apollo | Spring Cloud Config |
|---|---|---|---|
| 厂商 | Alibaba | Ctrip（携程） | Spring 官方 |
| 配置存储 | MySQL / 嵌入式数据库 | MySQL | Git（版本管理强） |
| 推送机制 | Long Polling | 推（HTTP 长连接） | 拉（Spring Cloud Bus） |
| 权限管理 | ✅ | ✅（强项） | ❌（需要 Git 权限） |
| 灰度发布 | ✅ | ✅（强项） | ❌ |
| 国内流行度 | 高（Alibaba 系） | 高（携程系） | 低 |

**选型建议**：

- **Alibaba 技术栈** → Nacos Config（和服务注册发现二合一）
- **需要强权限管理 + 灰度发布** → Apollo
- **已用 Git，需要版本管理** → Spring Cloud Config

## 正确使用方式

### 1. 配置 Data ID 的命名规则

```yaml
# bootstrap.yml
spring:
  application:
    name: order-service
  cloud:
    nacos:
      config:
        server-addr: localhost:8848
        file-extension: yaml
        # Data ID = ${spring.application.name}.${file-extension}
        # 即：order-service.yaml
```

**多环境配置**：

```yaml
# bootstrap.yml
spring:
  cloud:
    nacos:
      config:
        server-addr: localhost:8848
        file-extension: yaml
        namespace: dev            # 命名空间（区分环境）
        group: DEFAULT_GROUP      # 分组
```

```
Nacos 中的配置：
  Data ID: order-service.yaml         （默认配置）
  Data ID: order-service-dev.yaml    （dev 环境配置，优先级更高）
```

---

### 2. 共享配置（多个服务共用）

```yaml
# bootstrap.yml
spring:
  cloud:
    nacos:
      config:
        server-addr: localhost:8848
        shared-configs:
          - data-id: common.yaml
            group: COMMON_GROUP
            refresh: true  # 是否动态刷新
```

> **共享配置**适合：Redis 地址、MQ 地址、日志级别等**所有服务共用**的配置。

---

### 3. 灰度发布（Nacos 控制台）

```
Nacos 控制台
  → 配置列表
    → 找到 myapp.yaml
      → 点击「编辑」
        → 修改配置
          → 勾选「Beta 发布」
            → 指定 IP 列表（哪些实例收到新配置）
```

> **灰度发布**：先让 1-2 个实例用新配置，验证没问题再全量发布。

## 边界情况和坑

### 1. bootstrap.yml 没有生效

```
问题：配置了 bootstrap.yml，但连不上 Nacos
原因：Spring Boot 2.4+ 默认不加载 bootstrap.yml
```

**解决**：引入依赖或配置启用。

```xml
<!-- Maven 依赖（Spring Boot 2.4+ 需要） -->
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-bootstrap</artifactId>
</dependency>
```

或者：

```yaml
# application.yml（Spring Boot 2.4+）
spring:
  cloud:
    bootstrap:
      enabled: true  # 启用 bootstrap.yml
```

---

### 2. @Value 和 @ConfigurationProperties 的刷新区别

```java
// ❌ @Value：需要 @RefreshScope 才能刷新
@RefreshScope
@RestController
public class TestController {
    @Value("${myapp.timeout}")
    private int timeout;  // 刷新后生效
}

// ✅ @ConfigurationProperties：自动刷新（Spring Cloud 2020+）
@ConfigurationProperties(prefix = "myapp")
public class MyAppProperties {
    private int timeout;  // 配置变更后自动更新
}
```

> **Spring Cloud 2020+**：`@ConfigurationProperties` 绑定的 Bean **自动刷新**（不需要 `@RefreshScope`）。

---

### 3. 配置中心宕机，应用还能启动吗？

```
Nacos Config Client 会缓存配置到本地：
  ${user.home}/nacos/config/fixed-{server-addr}_.../{namespace}/{dataId}
```

> **配置中心宕机**：应用启动时会读**本地缓存**，不会完全不可用。  
> 但**无法获取最新配置**（直到配置中心恢复）。

---

### 4. 敏感配置（密码、Token）怎么处理？

```
❌ 明文存储在 Nacos：
  spring.datasource.password=123456  ← 不安全
```

**解决**：Nacos 配置加密（自定义 `EnvironmentPostProcessor`）或用 **KMS**（密钥管理服务）。

```yaml
# 加密后的配置（Nacos 中存储）
spring:
  datasource:
    password: ENC(af3k9s...)  # 密文

# 应用启动时自动解密（需要自定义解密逻辑）
```

## 我的理解

**配置中心的本质**：

1. **集中管理**：所有服务的配置在一个地方管理
2. **动态刷新**：改配置不需要重启服务
3. **版本管理**：配置变更有历史，可以回滚
4. **灰度发布**：新配置先让部分实例试用

**面试高频追问**：

1. 配置中心的架构（Nacos Config / Apollo / Spring Cloud Config 的区别）
2. 配置加载顺序（bootstrap.yml → Nacos Config → application.yml）
3. 配置动态刷新的原理（@RefreshScope → 销毁 Bean → 重新创建）
4. 配置优先级（远程配置 > 本地配置）
5. 配置中心宕机后，应用还能启动吗？（本地缓存）
