# 架构评审Checklist

## 这个问题为什么存在？

> 架构评审没有标准，漏掉关键点（可用性、一致性、安全性），上线后故障频发。问题是：**架构评审该关注哪些点，才能把风险提前暴露？**

没有Checklist，你会遇到：
- 上线后流量突增，系统雪崩（没考虑限流）
- 数据不一致，用户投诉（没考虑分布式事务）
- 慢SQL把数据库打挂（没审核SQL）

---

## 它是怎么解决问题的？

### 五大维度 Checklist

```

维度1：可用性（Availability）
│  - SLA目标？（99.9%？99.99%？）
│  - 限流方案？（令牌桶/漏桶）
│  - 降级方案？（非核心功能关闭）
│  - 熔断方案？（下游故障快速失败）
│  - 超时设置？（连接超时、读取超时）
│  - 监控告警？（QPS、响应时间、错误率）
│
维度2：一致性（Consistency）
│  - 分布式事务方案？（Saga/TCC/事务消息）
│  - 数据一致性延迟？（最终一致的时间窗口）
│  - 幂等性？（重试不导致重复操作）
│  - 缓存一致性？（Cache Aside Pattern）
│
维度3：扩展性（Scalability）
│  - 瓶颈在哪？（DB？缓存？外部API？）
│  - 水平扩展？（无状态设计，负载均衡）
│  - 缓存策略？（本地缓存？分布式缓存？）
│  - 数据库扩展？（读写分离？分库分表？）
│
维度4：安全性（Security）
│  - 认证方案？（JWT/OAuth2）
│  - 权限控制？（RBAC）
│  - 敏感数据？（加密存储、脱敏展示）
│  - SQL注入？（参数化查询）
│  - HTTPS？（传输加密）
│
维度5：可维护性（Maintainability）
   - 日志规范？（TraceId串联）
   - 监控指标？（JVM、GC、线程池）
   - 告警策略？（PagerDuty/钉钉）
   - 文档？（架构图、接口文档）
```

## 它和相似方案的本质区别是什么？

| 维度 | 关注点 | 忽略的后果 |
|---|---|---|
| **可用性** | 故障了系统还能用 | 雪崩，全部用户无法使用 |
| **一致性** | 数据是对的 | 超卖、重复扣款 |
| **扩展性** | 流量增长了能扛住 | 流量一来就挂 |
| **安全性** | 数据和系统不被攻击 | 数据泄露、系统被入侵 |
| **可维护性** | 出问题时能快速定位 | 排查问题要靠"猜" |

**本质区别**：五个维度各有侧重，**缺一不可**。但不同阶段侧重点不同：
- **初创期**：扩展性 > 可用性 > 一致性 > 安全性 > 可维护性
- **成长期**：可用性 = 一致性 > 扩展性 > 安全性 > 可维护性
- **成熟期**：安全性 = 可维护性 > 可用性 = 一致性 > 扩展性

## 正确使用方式

### 架构评审流程

```

1. 准备阶段
   - 画出架构图（组件、依赖、数据流向）
   - 填写Checklist（五个维度，每个都过一遍）
   - 提前发给评审委员（1天前）

2. 评审会议（1~2小时）
   - 讲架构图（10分钟）
   - 过Checklist（每个维度，委员提问）
   - 记录Action Item（谁、做什么、什么时候完成）

3. 跟进阶段
   - Action Item跟踪（Jira/Teambition）
   - 完成后才能上线（卡点）
```

### Checklist 示例（部分）

```

✅ 可用性
   - [ ] 限流：入口有Nginx限流，接口有Redis限流
   - [ ] 降级：非核心功能（推荐、评论）可以关闭
   - [ ] 熔断：调用第三方API有熔断（Resilience4j）
   - [ ] 超时：数据库连接池超时、HTTP调用超时
   - [ ] 监控：Grafana面板，P99响应时间、错误率

✅ 一致性
   - [ ] 分布式事务：下单+扣库存用Saga模式
   - [ ] 幂等性：支付接口用Token机制
   - [ ] 缓存一致性：Cache Aside Pattern，先更新DB再删缓存

✅ 扩展性
   - [ ] 无状态设计：服务不存Session（用Redis）
   - [ ] 数据库读写分离：读多写少场景
   - [ ] 缓存：热点数据放Redis（TTL 5分钟）

✅ 安全性
   - [ ] 认证：JWT（HMAC256签名）
   - [ ] 权限：RBAC（角色+权限）
   - [ ] SQL注入：用MyBatis #{}，不用${}
   - [ ] HTTPS：全站HTTPS（Let's Encrypt）

✅ 可维护性
   - [ ] 日志：TraceId串联（SkyWalking）
   - [ ] 监控：JVM面板（Prometheus + Grafana）
   - [ ] 告警：错误率>1%告警（钉钉）
   - [ ] 文档：架构图、接口文档（Swagger）
```

## 边界情况和坑

### 坑1：Checklist流于形式（check了，但没真改）

```

场景：架构评审会议，大家走过场
       Checklist打了勾，但代码没改
结果：上线后还是挂
```

**解决方案**：
1. **Action Item跟踪**（Jira/Teambition，完不成不能上线）
2. **复审**（改完后，再评审一次）
3. **线上验证**（灰度发布，验证改进点）

### 坑2：过度设计（Availability要求99.999%）

```

场景：创业公司，要求Availability 99.999%（一年宕机5分钟）
后果：成本极高（异地多活、热备），但用户量还很小
```

**解决方案**：
1. **根据业务阶段定SLA**（创业公司99.9%足够）
2. **SLA逐步提升**（用户量上来，再提升SLA）
3. **Cost-Benefit分析**（多投入100万，SLA从99.9%提升到99.99%，值不值？）

### 坑3：忽略数据迁移（重构 schema）

```

场景：重构数据库schema，但没考虑数据迁移
结果：上线后，旧数据读不懂，报错
```

**解决方案**：
1. **灰度前，先迁移历史数据**（双写、校验）
2. **灰度期间，新旧格式都写**（保证回滚后还能读旧格式）
3. **灰度完成后，删除旧格式写入**（但要保留迁移脚本）
