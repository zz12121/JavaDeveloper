# 构建优化

> Maven 构建慢，通常不是 Maven 本身的锅，而是**依赖混乱、插件配置不当、没有用增量构建**。优化构建不只是「加快速度」，更是**缩短反馈循环**，让开发体验变好。

---

## 这个问题为什么存在？

构建慢的痛点：

```
场景 A：全量构建要 20 分钟
  → 改一行代码，要等 20 分钟才能验证
  → 开发者失去耐心，跑去刷手机
  → 反馈循环太长，开发效率暴跌

场景 B：CI 构建排队 1 小时
  → 提交一个 PR，1 小时后才知道能不能合并
  → 开发者切换到其他任务，上下文切换成本高
  → 每天有效工作时间被排队吃掉 2 小时

场景 C：依赖下载慢
  → Maven Central 在国外，下载依赖要 10 分钟
  → 新人入职第一天，光下载依赖就要半天
  → CI 机器每次构建都重新下载（没有缓存）

场景 D：构建输出不可重现
  → 本地构建成功，CI 构建失败
  → 「在我机器上能跑啊」
  → 排查 3 小时，发现是依赖版本不一致
```

**构建优化的核心目标**：
1. **缩短反馈循环**（本地构建 < 2 分钟）
2. **提高 CI 吞吐量**（减少排队时间）
3. **保证构建可重现**（本地和 CI 结果一致）

## 它是怎么解决问题的？

### 优化手段 1：用镜像加速依赖下载

```xml
<!-- settings.xml：配置阿里云镜像（国内团队必备） -->
<mirrors>
    <mirror>
        <id>aliyun</id>
        <mirrorOf>central</mirrorOf>   <!-- 只代理 central，不影响私服 -->
        <url>https://maven.aliyun.com/repository/public</url>
    </mirror>
</mirrors>
```

**更彻底的加速**：搭建 Nexus 私服，所有依赖走内网缓存。

### 优化手段 2：并行构建（Maven 3.x+）

```bash
# 并行构建（根据 CPU 核心数自动分配线程）
mvn clean install -T 1C
# -T 1C：每个 CPU 核心 1 个线程
# -T 4：固定 4 个线程

# 效果：多模块项目构建时间可缩短 30% ~ 50%
```

**限制**：
- 模块之间不能有循环依赖
- 需要在 `pom.xml` 中配置 `<parallel>true</parallel>`（Maven 3.3.1+）

### 优化手段 3：增量构建（只构建变化的模块）

```bash
# Maven 本身对增量构建支持有限
# 但可以用这些技巧：

# 1. 只构建指定模块（-pl）和依赖它的模块（-amd）
mvn install -pl user-service -amd
# -pl：指定模块
# -amd：同时构建依赖 user-service 的模块（反向依赖）

# 2. 用 -o（offline）跳过远程仓库检查
mvn install -o

# 3. 用 -DskipTests 跳过测试（开发阶段，CI 中不要用）
mvn install -DskipTests
```

### 优化手段 4：用 `maven-build-cache-extension`（Maven 3.9+）

```xml
<!-- .mvn/extensions.xml -->
<extensions>
    <extension>
        <groupId>org.apache.maven.extensions</groupId>
        <artifactId>maven-build-cache-extension</artifactId>
        <version>1.0.0</version>
    </extension>
</extensions>
```

**原理**：计算输入（源码、依赖、插件版本）的哈希，如果没变，直接复用上次的构建输出。

效果：第二次构建**零开销**（如果没改代码）。

### 优化手段 5：精简依赖（定期用 `dependency:analyze`）

```bash
# 找出「用了但没声明」和「声明了但没用」的依赖
mvn dependency:analyze

# 输出示例：
# [WARNING] Used undeclared dependencies found:
# [WARNING]    org.springframework:spring-core:jar:5.2.0:compile
#     ↑ 靠传递依赖引入的，应该在 pom.xml 中显式声明
#
# [WARNING] Unused declared dependencies found:
# [WARNING]    junit:junit:jar:4.13.2:test
#     ↑ 可以删除（如果确实没用）
```

**为什么要显式声明「用了但没声明」的依赖？**

传递依赖是不稳定的——直接依赖升级时，可能去掉某个传递依赖，你的代码就编译失败了。显式声明后，版本由你控制。

### 优化手段 6：用 `maven-surefire-plugin` 并行跑测试

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-surefire-plugin</artifactId>
    <configuration>
        <!-- 并行跑测试（根据 CPU 核心数） -->
        <parallel>methods</parallel>
        <threadCount>4</threadCount>
        <!-- 或者直接用 JUnit 5 的并行测试功能 -->
    </configuration>
</plugin>
```

## 深入原理

### Maven 构建优化 vs Gradle 构建缓存

| 维度 | Maven | Gradle |
|------|--------|--------|
| 增量构建 | 弱（主要依赖插件支持） | 强（Task 输入/输出自动检测） |
| 构建缓存 | 需要额外扩展（build-cache-extension） | 内置，且支持远程缓存 |
| 并行构建 | 支持（-T 参数） | 默认并行（Task 级并行） |
| 配置复杂度 | 低（主要靠命令行参数） | 中（需要理解 Task 依赖图） |
| 效果 | 中等（节省 30% ~ 50%） | 显著（节省 50% ~ 80%） |

**为什么 Gradle 的增量构建更强？**

Gradle 的每个 Task 都声明了输入（`@Input`）和输出（`@Output`）。只要输入没变，Task 就不会重新执行。Maven 的插件需要自己实现增量逻辑，一致性差。

### 本地构建优化 vs CI 构建优化

| 优化手段 | 本地构建 | CI 构建 |
|---------|---------|---------|
| 并行构建 | ✅ 推荐 | ✅ 推荐（但需要控制并发数） |
| 跳过测试 | ✅ 开发阶段可以用 | ❌ 禁止（CI 必须跑测试） |
| 离线模式 | ✅ 可以用 | ❌ 不能用（需要下载依赖） |
| 构建缓存 | ✅ 推荐（Maven 3.9+） | ✅ 推荐（且可以设远程缓存） |
| 增量编译 | ✅（`maven-compiler-plugin` 支持） | ✅（同左） |

## 正确使用方式

### 1. 本地开发：用 `mvn compile` 代替 `mvn install`

```bash
# ❌ 错误做法：每次都 install
mvn clean install   # 慢，且会污染本地仓库

# ✅ 正确做法：只编译（开发阶段）
mvn compile         # 只编译，不打包、不安装

# 需要跑测试时
mvn test           # 只跑测试，不打包

# 需要本地其他模块依赖时，才 install
mvn install -DskipTests   # 跳过测试，快速安装到本地仓库
```

### 2. CI 配置：用 `-B`（batch 模式）避免交互

```bash
# CI 中必须用 -B，否则某些插件会卡住等输入
mvn -B clean install
```

### 3. 用 `maven-compiler-plugin` 的增量编译

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <configuration>
        <!-- 启用增量编译（默认就是 true，但显式声明更清晰） -->
        <useIncrementalCompilation>true</useIncrementalCompilation>
    </configuration>
</plugin>
```

**增量编译原理**：只重新编译「源码变了」或「依赖的 class 变了」的文件。

### 4. 多模块项目：用 `-pl` 精确控制构建范围

```bash
# 场景：只改了 user-service 模块，不需要构建全部

# 只构建 user-service 模块
mvn install -pl user-service

# 同时构建 user-service 依赖的模块（-am = also-make）
mvn install -pl user-service -am

# 同时构建依赖 user-service 的模块（-amd = also-make-dependents）
mvn install -pl user-service -amd
```

## 边界情况和坑

### 1. 并行构建时，测试可能互相干扰

```
问题：
  并行跑测试时，多个测试用同一个端口（如 8080）
  → 端口冲突，测试失败
  → 但单机跑测试时没问题（因为串行）
```

**解决**：
- 用动态端口（`ServerSocket(0)` 随机分配）
- 用 TestContainers 管理依赖（自动分配端口）
- 配置 Surefire 按类隔离（`forkCount>1`）

### 2. `-DskipTests` vs `-Dmaven.test.skip=true` 的区别

```bash
# -DskipTests：编译测试代码，但不运行
mvn install -DskipTests
# 效果：src/test/java 会被编译，但不执行测试

# -Dmaven.test.skip=true：不编译测试代码，也不运行
mvn install -Dmaven.test.skip=true
# 效果：src/test/java 不会被编译（构建更快，但危险）
```

**危险**：`-Dmaven.test.skip=true` 会跳过测试编译，如果测试代码有语法错误，你发现不了。

### 3. 构建缓存在 `clean` 后失效

```bash
mvn clean install   # clean 会删除 target/，缓存也跟着删了
```

**解决**：不需要每次都 `clean`。只有在这些场景才需要 `clean`：
- 依赖版本变了
- 构建结果异常（怀疑是旧 class 残留）
- 准备发布到生产环境（保险起见）

### 4. Windows 下文件路径过长导致构建失败

```
报错：
Failed to execute goal ...: 文件名或扩展名太长
```

**解决**：
```xml
<!-- 1. 用 <shortenCommandLine> 选项（Maven 3.3.1+） -->
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-surefire-plugin</artifactId>
    <configuration>
        <argLine>-Xmx512m</argLine>
        <useFile>false</useFile>
        <shortenCommandLine>manifest</shortenCommandLine>
    </configuration>
</plugin>

<!-- 2. 启用 Windows 长路径支持（Windows 10+） -->
<!-- 组策略 → 计算机配置 → 管理模板 → 系统 → 文件系统 → 启用长路径 -->
```
