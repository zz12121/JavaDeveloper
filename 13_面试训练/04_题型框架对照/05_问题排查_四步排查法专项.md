# 问题排查题 — 四步排查法专项训练

> 框架：确认现象 → 缩小范围 → 定位根因 → 解决+预防

---

## 一、问题排查题的本质

```
问题排查是大厂/中高级岗位的高频题：
- 问的是：你遇到线上问题怎么办？
- 看的是：你有没有系统的排查思路
- 考的是：你对技术原理的理解深度

面试官不关心你记没记住某个命令，
关心的是你"遇到未知问题能不能分析"。

系统排查思路比记住命令更重要。
```

---

## 二、四步排查法详解

### 第一步：确认现象（从外到内）

```
这一步的目的是：搞清楚到底发生了什么。

排查思路：先问清楚，再看现象。

必问清单：
1. 是所有用户都遇到这个问题，还是部分？
2. 什么时候开始的？持续多久了？
3. 有没有规律？（一直有/偶尔/只在高峰期）
4. 有没有报错信息？（错误码/堆栈/日志）
5. 最近的变更是什么？（代码发布/配置变更/机器扩缩容）

看什么：
- 监控大盘：QPS/延迟/错误率/资源使用率
- 错误日志：异常堆栈/错误码
- 用户反馈：客诉工单/用户描述

关键：先确认问题范围，不要一上来就开始查。
```

### 第二步：缩小范围（分段排查）

```
这一步的目的是：定位问题出在哪一层。

分层排查法：

网络层
  ↓ 有问题
网关层（Nginx/网关）
  ↓ 有问题
应用层（Tomcat/Spring）
  ↓ 有问题
缓存层（Redis）
  ↓ 有问题
数据库层（MySQL）
  ↓ 有问题
外部服务（第三方API）

每个层级看的指标：
- 网络层：ping/telnet/抓包(tcpdump)
- 网关层：连接数/限流/upstream状态
- 应用层：JVM/CPU/线程池/接口响应时间
- 缓存层：命中率/内存/QPS/连接数
- 数据库层：连接数/慢查询/锁等待
- 外部服务：响应时间/错误率
```

### 第三步：定位根因（找到具体原因）

```
这一步的目的是：找到问题的根本原因。

常用方法：
1. 日志分析：grep/awk筛选关键日志
2. 调用链追踪：SkyWalking/Jaeger看慢在哪一步
3. 性能分析： Arthas profiler / jstack / jmap
4. 监控对比：出问题时间段 vs 正常时间段的差异

常见根因类型：
- 代码bug：死循环/内存泄漏/并发问题/异常未捕获
- 配置问题：JVM参数/连接池大小/超时时间
- 资源瓶颈：CPU打满/内存不足/磁盘IO高/网络IO高
- 依赖问题：DB慢/Redis慢/外部服务超时
```

### 第四步：解决+预防

```
这一步的目的：解决问题，并且防止再发生。

解决：
1. 紧急止血：重启/回滚/切换/降级
2. 根因修复：改代码/改配置/扩容

预防：
1. 加监控：同类问题能及时发现
2. 加告警：阈值告警，不要等问题大了才告
3. 加流程：发布检查清单，避免同类问题
4. 加测试：性能测试/压力测试

关键：不仅要解决问题，还要让问题不再发生。
```

## 三、高频问题排查详解

### 排查1：CPU 100%

```
【第一步：确认现象】
问：
- 是Java进程CPU高，还是其他进程？
- 是一直高，还是突然高？
- 最近有没有发布？

看：
- top 命令：看哪个进程CPU高
- w java - jar pid 看Java进程的CPU使用情况

【第二步：缩小范围】
如果是Java进程CPU高，继续排查：

1. 找到最耗CPU的线程：
   top -Hp <pid>  → 找到tid

2. 把线程ID转16进制：
   printf "%x" <tid>

3. 抓取线程栈：
   jstack <pid> | grep <tid十六进制>

【第三步：定位根因】
线程栈里的常见问题：
1. 死循环：WAITING/BLOCKED 状态的线程很多
2. GC频繁：GC线程占用CPU，堆内存不够
3. 正则回溯：正则表达式写错，导致CPU打满
4. 加密解密：大量计算密集型操作
5. 序列化/反序列化：大对象处理

常见场景：
- 单核CPU 100%：死循环
- 多核CPU 100%：GC/大量计算
- GC线程CPU高：堆内存太小/对象创建太多

【第四步：解决+预防】
紧急止血：
- kill -9 <pid> 重启（最后手段）
- jmap -dump:format=b,file=heap.hprof <pid> 保留现场

根因修复：
- 死循环：找到代码，打印日志，修复
- GC问题：调整堆大小/GC参数
- 正则问题：优化正则，使用独占锁

预防：
- 加CPU监控告警
- 加全链路追踪，定位慢请求
```

### 排查2：内存泄漏/内存溢出

```
【第一步：确认现象】
问：
- 是OOM（内存溢出）还是内存持续增长？
- OOM是什么时候开始的？
- 进程跑了多久开始OOM的？

看：
- jstat -gcutil <pid> 1000 看GC情况
  - FGC（Full GC）次数是否持续增长？
  - Old区使用率是否持续增长？

【第二步：缩小范围】
jmap -heap <pid>  看堆内存使用情况：
- Eden区很快满了 → 对象创建太多
- Old区持续增长 → 有内存泄漏
- Metaspace持续增长 → 动态类加载太多

【第三步：定位根因】
方法1：dump内存快照
jmap -dump:format=b,file=heap.hprof <pid>

用MAT分析：
- 找出占用内存最大的对象
- 找出内存泄漏的可疑对象（GC Root引用链）
- 常见泄漏：ThreadLocal未清理/静态集合类/未关闭的资源

方法2：在线分析
arthas 的 heapdump 命令
dashboard 看内存使用情况

常见泄漏场景：
1. ThreadLocal未remove
2. 静态集合类持有对象引用
3. 数据库连接/网络连接未关闭
4. 监听器/回调未注销
5. 内部类持有外部类引用（非静态内部类）

【第四步：解决+预防】
紧急止血：
- 重启Java进程（先保留dump）
- 限制堆内存 -Xmx

根因修复：
- ThreadLocal：使用完手动remove()
- 静态集合：不用了就clear()
- 连接资源：try-with-resources

预防：
- 上线前用Arthas做内存监控
- 定期dump分析
- 监控Old区增长趋势
```

### 排查3：接口超时

```
【第一步：确认现象】
问：
- 是所有接口都慢，还是某个接口慢？
- 什么时候开始慢的？
- 有没有规律？（高峰期/特定用户）

看：
- APM/SkyWalking 全链路追踪：慢在哪一步
- 日志：有没有超时错误/异常堆栈

【第二步：缩小范围】
全链路排查，找到慢在哪一层：

1. 外部调用：第三方API/支付/短信
   → 看外部服务的响应时间

2. 数据库：MySQL慢查询
   → explain分析SQL

3. Redis：key不存在/大key/热key
   → slowlog看Redis命令执行时间

4. 业务逻辑：死循环/复杂计算
   → Arthas trace方法耗时

5. JVM：GC停顿
   → jstat -gcutil 看GC时间

【第三步：定位根因】
常见原因：
1. 数据库慢查询
   → explain分析，看有没有索引失效

2. 外部服务超时
   → 看对方响应时间，是否需要降级

3. Redis慢
   → slowlog get 10 看慢命令
   → bigkey 查看大key
   → 热key 查看热key

4. 连接池耗尽
   → 看连接池使用率
   → 看有没有连接泄漏

5. GC停顿
   → 看GC日志，定位是不是GC导致的

【第四步：解决+预防】
紧急止血：
- 降级：关闭非核心功能
- 限流：减少请求量
- 扩容：加机器

根因修复：
- SQL优化：加索引/改SQL
- 外部服务：加超时/降级
- Redis优化：拆分大key/加缓存
- JVM调优：改GC参数

预防：
- 慢接口监控告警
- 数据库慢查询日志
- 外部服务响应时间监控
```

### 排查4：MySQL慢查询

```
【第一步：确认现象】
问：
- 是特定接口慢，还是全局都慢？
- 有没有规律？（高峰期/特定SQL）

看：
- slow_query_log：慢查询日志
- processlist：当前执行的SQL

开启慢查询：
set global slow_query_log='ON';
set global long_query_time=1;  -- 超过1秒记录

【第二步：缩小范围】
查看慢查询：
- show processlist：正在执行的SQL
- select * from information_schema.processlist
- 找到执行时间长的SQL

分析：
- 正在执行的SQL：看执行计划
- 结束后的SQL：看慢查询日志

【第三步：定位根因】
explain分析：
explain <SQL>

关键字段：
- type：ALL（全表扫描）> index > range > ref > const
- key：使用了哪个索引
- rows：扫描了多少行
- Extra：
  - Using filesort：需要额外排序
  - Using temporary：需要临时表
  - Using index：覆盖索引

常见问题：
1. 全表扫描（type=ALL）
   → 没走索引，需要加索引

2. 索引失效
   → 违反了最左前缀/用了函数/类型转换

3. 大量回表
   → 用select * 导致，需要覆盖索引

4. 排序开销大
   → Using filesort，需要优化排序

5. 锁等待
   → 间隙锁/长事务

【第四步：解决+预防】
紧急止血：
- kill <id> 杀掉慢SQL
- 临时加索引（生产谨慎）

根因修复：
- 加索引：alter table add index
- 改SQL：避免select * / 优化where条件
- 拆SQL：大表分批处理
- 降级：非核心接口降级

预防：
- 上线前explain分析SQL
- 定期分析慢查询日志
- 加监控：慢查询超过阈值告警
```

### 排查5：Redis变慢

```
【第一步：确认现象】
问：
- Redis变慢是什么时候开始的？
- 是读慢还是写慢？
- 什么操作变慢了？

看：
- slowlog get 10：最近10个慢命令
- info commandstats：各命令耗时统计
- info clients：客户端连接数

【第二步：缩小范围】
1. 看是哪个命令慢：
   slowlog get 10

2. 看大key：
   redis-cli --bigkeys

3. 看热key：
   redis-cli --hotkeys

4. 看内存情况：
   info memory
   - used_memory_rss：实际内存
   - mem_fragmentation_ratio：内存碎片率

【第三步：定位根因】
常见原因：
1. 大key
   → 一个key几MB甚至几GB
   → 解决：拆分key，定期清理

2. 热key
   → 同一个key被大量请求
   → 解决：热点key分散/本地缓存

3. 内存碎片
   → mem_fragmentation_ratio > 1.5
   → 解决：重启/内存整理

4. 命令复杂度
   → O(n)命令：keys/smembers/sort
   → 解决：用scan替代keys，用有序集合替代list

5. 持久化阻塞
   → AOF fsync阻塞
   → 解决：改用everysec，不要always

6. CPU单核瓶颈
   → Redis单线程，绑定单核CPU
   → 解决：开启多线程IO（4.0+）

【第四步：解决+预防】
紧急止血：
- redis-cli slowlog reset 清理慢日志
- redis-cli client kill 杀掉异常客户端

根因修复：
- 大key：拆分 + 后台异步删除del
- 热key：本地缓存 + key分散
- 内存碎片：config set activedefrag yes
- O(n)命令：用低复杂度命令替代

预防：
- 定期scan大key
- 监控慢命令
- 监控内存碎片率
- 监控连接数
```

## 四、问题排查命令速查表

| 场景 | 命令 |
|------|------|
| **CPU高** | top → top -Hp → jstack |
| **内存高** | jstat -gcutil → jmap -heap → MAT |
| **GC频繁** | jstat -gc → GC日志分析 |
| **接口慢** | Arthas trace → SkyWalking |
| **MySQL慢** | explain → slowlog → processlist |
| **Redis慢** | slowlog → bigkeys → hotkeys |
| **线程问题** | jstack → Arthas thread |
| **网络问题** | tcpdump → netstat |

## 六、问题排查面试话术

```
当面试官问："线上接口超时怎么排查？"

标准回答：

"我的排查思路分四步：

第一步，确认现象。
先确认是全局超时还是部分超时，问清楚什么时候开始的，
有没有规律，是所有接口还是某个接口。

第二步，缩小范围。
通过全链路追踪（SkyWalking/Jaeger）定位慢在哪一步，
分层排查：
- 先看是外部服务慢
- 再看是数据库慢
- 还是Redis慢
- 还是业务逻辑慢

第三步，定位根因。
根据慢的层，继续定位：
- DB慢就用explain分析SQL
- Redis慢就用slowlog看命令
- 业务逻辑慢就用Arthas trace方法

第四步，解决+预防。
紧急止血：降级/限流/扩容
根因修复：加索引/改代码/优化配置
预防：加监控告警，避免同类问题"

这样回答，面试官会觉得你：
1. 有系统的排查思路
2. 理解各层技术的原理
3. 不仅会排查，还会预防
```
