# 服务注册发现是怎么实现的？

> ⚠️ **先盲答**：服务注册发现解决什么问题？基本流程是什么？

---

## 盲答引导

1. 服务 A 怎么找到服务 B 的 IP 和端口？—— 注册中心
2. 客户端发现 vs 服务端发现，区别是什么？—— 客户端自己查 vs Gateway 代查
3. Eureka（AP）和 Nacos（同时支持 CP/AP），核心区别是什么？
4. 注册中心挂了，新的服务实例还能被发现吗？

---

## 知识链提示

```
服务注册发现
  → [[服务注册发现]]
    → 基本流程
      → 服务启动 → 注册到注册中心（IP/Port/健康检查）
      → 消费者 → 从注册中心拉取服务列表
      → 消费者 → 从列表中选一个调用（配合负载均衡）
    → 客户端发现 vs 服务端发现
      → 客户端发现（Eureka）：Consumer 直接查注册中心
        → 优点：无单点，Consumer 自己决策
        → 缺点：Consumer 需要集成注册中心 SDK
      → 服务端发现（Gateway/Nginx）：Consumer 请求 Gateway，由 Gateway 转发
        → 优点：Consumer 无需改代码
        → 缺点：Gateway 是单点（需集群 + 健康检查）
    → Eureka（AP）
      → 多个 Eureka Server 互相注册，形成集群
      → 自我保护机制：超过 85% 实例心跳丢失 → 停止清理注册表（防止误杀）
        → [[高可用]] → 宁可不可用，不可误杀
      → 心跳机制：客户端每 30s 发送心跳，不续期 → 90s 后剔除
    → Nacos（CP+AP）
      → 支持两种模式：临时实例（心跳，AP）/ 持久实例（CP）
      → DNS-F / OpenAPI / 配置中心一体
```

---

## 核心追问

| 追问 | 考点 |
|------|------|
| Eureka 的自我保护机制是什么？ | 心跳丢失 > 15% → 停止剔除，防止网络抖动误杀健康实例 |
| Nacos 为什么能做到同时 CP 和 AP？ | Raft 协议保证 CP，心跳机制实现 AP |
| 注册中心挂了，还能调用服务吗？ | 能（缓存的服务列表还能用），但新实例无法发现 |
| Nacos 和 Eureka 选型？ | Nacos（功能更全）/ Eureka（简单，无缝迁移 Spring Cloud）|

---

## 参考答案要点

**客户端发现 vs 服务端发现**：
- 客户端发现：Consumer 直连注册中心 → Eureka / Consul
- 服务端发现：Consumer → Gateway → 注册中心 → Provider → 响应（Nginx / Kubernetes Ingress）

**Eureka vs Nacos**：Eureka 是纯 AP（自我保护，优先可用），Nacos 同时支持 AP 和 CP（更灵活）。
