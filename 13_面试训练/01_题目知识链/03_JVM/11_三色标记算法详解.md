## 盲答引导

1. 三色标记的「白/灰/黑」三种颜色分别代表什么含义？
2. 并发标记时，为什么会出现「漏标」（对象消失）问题？
3. CMS 用的是「增量更新」（Incremental Update），它解决的是哪一类漏标？
4. G1 用的是「原始快照」（SATB），它和 CMS 的方式有什么本质区别？
5. ZGC 的「着色指针」是怎么做到并发标记的？为什么不需要写屏障记录？

## 知识链提示

这道题应该让你联想到：

- `并发标记漏标问题` → 白→灰→黑过程中的「消失对象」问题
- `写屏障` → CMS 的增量更新、G1 的 SATB 都用写屏障记录变化
- `G1的SATB` → Snapshot-At-The-Beginning，保证标记开始时的存活对象不被漏标
- `ZGC着色指针` → 指针上的染色位，读屏障驱动标记
- `三色标记与STW` → 初始标记和重新标记仍然需要 STW

3. G1 的 SATB：灰色对象要删除对白色对象的引用，写屏障会做什么？
4. ZGC 的标记过程中，读屏障（Load Barrier）在什么时机触发？

## 参考要点（盲答后再看）


**三色标记基础**：
| 颜色 | 含义 |
|------|------|
| 白色 | 未被标记，或标记完成后仍为白色 = 垃圾 |
| 灰色 | 自身已标记，但引用的对象还未标记 |
| 黑色 | 自身及直接引用的对象都已标记完成 |

**漏标发生的两个必要条件**（必须同时满足才会漏标）：
1. 黑色对象新引用了白色对象（黑色不重新扫描）
2. 灰色对象要删除对白色对象的引用（白色失去所有存活引用）

**三大收集器的并发标记方案对比**：

| 收集器 | 方案 | 解决条件 | 写屏障开销 |
|--------|------|---------|-------------|
| CMS | 增量更新（Incremental Update） | 打破条件1：记录黑色→白色的新引用，重新标记时重新扫描 | 中 |
| G1 | SATB（原始快照） | 打破条件2：灰色删除对白色的引用时，白色被SATB记录，标记为存活 | 中~高 |
| ZGC | 着色指针 + 读屏障 | 不需要写屏障！读屏障发现指针未标记时，触发标记 | 低（染色指针） |

**SATB vs 增量更新（面试重点）**：
```
// 场景：GC 进行时，应用线程修改引用
// 初始状态：A(黑) → B(灰) → C(白)

// 应用线程做两件事：
A.field = C;   // ① 黑色引用白色
B.field = null; // ② 灰色删除对白色的引用
// 如果不处理：C 是白色，被认为是垃圾 → 漏标！

// CMS（增量更新）：记录 ①，重新标记时重新扫描 A
// G1（SATB）：记录 ② 删除前的引用（C 入 SATB 队列），保证 C 存活
```

**ZGC 着色指针（对比理解）**：
```
// 64位指针中借多位做标记位
指针格式：[高18位保留][10位着色标记][44位对象地址]

// 读屏障（类似：从堆里读指针时发现未标记 → 触发标记）
Object readPointer(Object p) {
    if (coloredMarked(p)) return p;
    mark(p);  // 触发标记，不需要写屏障！
    return p;
}
```
