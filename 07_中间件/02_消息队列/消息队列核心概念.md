# 消息队列核心概念

## 这个问题为什么存在？

没有消息队列时，系统之间的调用是同步的、紧耦合的。

想象一个电商下单场景：订单服务需要调用库存服务、优惠券服务、短信服务、物流服务……每一个都是同步调用。任何一个服务慢了，订单服务就被拖慢；任何一个服务挂了，订单就失败。订单服务和库存服务之间形成了强绑定——这不是架构，这是"连体婴儿"。

更深层的问题有三个：

**解耦**：订单服务凭什么要知道库存服务在哪台机器上、用什么协议调用？它只想知道"库存够了没有"，而不是"怎么联系库存服务"。

**异步**：用户下单后，真的不需要等短信发出去才能看到"下单成功"吧？大部分后续操作不需要实时完成，同步等待只是在浪费用户时间。

**削峰**：秒杀场景，一秒钟来了 10000 个请求，数据库只能扛 1000。没有缓冲层，数据库直接被打死。

消息队列就是来解决这三个问题的：把同步调用变成异步消息传递，让服务之间只依赖消息协议，不直接依赖彼此的实现。

## 它是怎么解决问题的？

### 核心模型

消息队列的本质是一个**分布式、持久化的先进先出队列**，但比普通队列多了一些关键能力。

```
[生产者] → [消息队列] → [消费者]
  Producer      Broker        Consumer
```

生产者把消息发给 Broker，消费者从 Broker 拉取（或 Broker 推送）消息。双方不需要同时在线，不需要知道彼此的存在。

### 核心概念详解

#### 1. 消息模型

**点对点模型（Point-to-Point）**

```
Producer → Queue → Consumer1 (一个消息只被一个消费者消费)
                → Consumer2 (竞争消费，不会重复消费)
```

每条消息只会被一个消费者消费。适合任务分发场景，比如订单支付后，只需要一个服务去扣减库存。

**发布订阅模型（Pub/Sub）**

```
Producer → Topic → Consumer Group A (每个Group都能收到全部消息)
                → Consumer Group B
```

每条消息会被所有订阅者收到。适合事件广播场景，比如订单创建后，库存、短信、物流都要感知到。

> Kafka 用 Consumer Group 统一了这两种模型：Group 内是点对点（竞争消费），Group 之间是发布订阅（每个 Group 都收到全部消息）。

#### 2. Topic 和 Partition

**Topic** 是消息的逻辑分类，比如 `order_created`、`payment_success`。

但一个 Topic 如果只存在一个 Broker 上，吞吐量就有上限。所以 Topic 被拆成多个 **Partition（分区）**，每个 Partition 是一个有序的、不可变的消息序列。

```
Topic: order_created
├── Partition-0 (Broker-1)  [msg1, msg2, msg5, ...]
├── Partition-1 (Broker-2)  [msg3, msg4, msg7, ...]
└── Partition-2 (Broker-3)  [msg6, msg8, msg9, ...]
```

**Partition 的核心价值**：并行。不同 Partition 可以由不同 Broker 处理，消费者也可以并行消费不同 Partition。

**Partition 的有序性保证**：Partition 内有序，但跨 Partition 不保证有序。如果业务要求全局有序，只能用 1 个 Partition（但会损失并行度）。

#### 3. Consumer Group 与 Offset

**Consumer Group** 是一组消费者的集合，同一个 Group 内的消费者**竞争消费** Partition，每个 Partition 在同一时刻只会被一个消费者消费。

```
Consumer Group A:
  Consumer-1 → Partition-0, Partition-1
  Consumer-2 → Partition-2

Consumer Group B:
  Consumer-3 → Partition-0, Partition-1, Partition-2  (独立消费，不受Group A影响)
```

**Offset** 是消费者在 Partition 中的消费进度指针。消费者崩溃重启后，从上次提交的 Offset 继续消费，而不是从头开始。

> Offset 的提交时机直接决定了消费语义（后面详细讲）。

#### 4. 消息可靠性层级

消息队列的可靠性不是一个二元开关，而是一个谱系：

| 层级 | 含义 | 可能丢失的场景 |
|------|------|----------------|
| 生产端可靠 | 生产者确认消息已到达 Broker | Broker 宕机、网络中断 |
| 存储可靠 | 消息持久化到磁盘 | 磁盘损坏、只存内存 |
| 消费端可靠 | 消费者真正处理完才提交 Offset | 消费失败但 Offset 已提交 |

高可靠 = 三个层级都要有对应保障，但每一个保障都有性能代价。

### 源码关键路径（以 Kafka 为例）

Kafka 生产者的消息发送路径：

```
KafkaProducer.send()
  → 拦截器链 (ProducerInterceptor)
  → 序列化 (Serializer)
  → 分区器 (Partitioner)  # 决定发到哪个Partition
  → 追加到 RecordAccumulator (批处理缓冲区)
  → Sender 线程取出批次
  → 构建 ProduceRequest
  → NetworkClient 发送 (NIO)
  → Broker 写入 PageCache
  → 副本同步 (ISR机制)
  → 返回 ProduceResponse
```

关键设计：**RecordAccumulator 是批处理的核心**。每个 Partition 对应一个 `Deque<ProducerBatch>`，消息先追加到当前批次，批次满了或 linger.ms 到了才发送。这是 Kafka 高吞吐的关键之一。

## 深入原理

### 消息队列 vs  RPC（远程过程调用）

| 维度 | RPC (gRPC/Dubbo) | 消息队列 |
|------|-------------------|----------|
| 耦合方式 | 同步调用，强依赖服务地址和协议 | 异步消息，只依赖 Topic 名 |
| 响应模式 | 必须等待响应 | 发完即走（或异步回调） |
| 失败处理 | 调用方直接感知失败 | 消息持久化，消费者自行重试 |
| 适用场景 | 实时性要求高、需要立即返回结果 | 解耦、异步、削峰 |

本质区别：**RPC 是"对话"，消息队列是"留言"**。需要实时对话就用 RPC，不需要实时就用消息队列。两者不是替代关系，是互补关系。

### Kafka vs RabbitMQ vs RocketMQ

| 维度 | Kafka | RabbitMQ | RocketMQ |
|------|-------|-----------|----------|
| 设计目标 | 高吞吐日志流 | 低延迟、灵活路由 | 高可靠、事务消息 |
| 消息模型 | Partition + Consumer Group | Exchange + Queue + Binding | Topic + Tag + Consumer Group |
| 有序性 | Partition 内有序 | Queue 内有序 | Queue 内有序 |
| 延迟消息 | 不支持（需外部方案） | TTL + Dead Letter 支持 | 原生支持（时间轮） |
| 事务消息 | 0.11 后支持 | 不支持 | 原生支持（二阶段提交） |
| 适用场景 | 日志、流处理、大数据管道 | 低延迟业务消息、灵活路由 | 金融、电商（淘宝自用） |

**为什么选 A 不选 B？**

- 选 Kafka：吞吐量优先（日志、埋点、流处理），容忍偶尔消息丢失（通过配置可以做到不丢，但性能会降）
- 选 RabbitMQ：低延迟优先（AMQP 协议、灵活路由），消息量不大但要求灵活
- 选 RocketMQ：高可靠优先（金融场景），需要事务消息、延迟消息等高级特性

## 正确使用方式

### 正确用法：PowerPoint 排版原则

**1. 生产者：幂等 + 异步回调**

```java
// 错误：发完就不管，消息丢了不知道
producer.send(record);

// 正确：异步回调，感知发送结果
producer.send(record, (metadata, exception) -> {
    if (exception != null) {
        // 发送失败，记录日志 + 重试
        log.error("发送失败", exception);
        retryOrFallback(record);
    } else {
        // 发送成功，可以记录 metadata（topic/partition/offset）
        log.debug("发送成功: {}", metadata);
    }
});
```

**为什么正确**：网络是一定会失败的，发消息不处理回调就是在"碰运气"。

**2. 消费者：先处理，再提交 Offset**

```java
// 错误：先提交 Offset，再处理业务（At most once，可能丢消息）
consumer.commitSync();
processMessage(record);

// 正确：先处理业务，再提交 Offset（At least once，可能重复）
processMessage(record);
consumer.commitSync();
```

**为什么正确**：如果先提交再处理，处理失败了消息就不会再消费（丢了）。先处理后提交，哪怕处理失败，Offset 没提交，下次还会消费（重复了，但比丢消息好）。

**3. 幂等消费（解决重复消费）**

```java
// 利用业务唯一键 + 去重表
public void processOrderMessage(OrderMessage msg) {
    String bizKey = "order:" + msg.getOrderId();
    // Redis SET NX：如果已存在说明是重复消息
    Boolean first = redisTemplate.opsForValue()
        .setIfAbsent(bizKey, "1", Duration.ofHours(24));
    if (Boolean.FALSE.equals(first)) {
        log.warn("重复消息，跳过: {}", msg.getOrderId());
        return;
    }
    // 真正处理业务...
}
```

**为什么正确**：消息队列只能保证"不丢"，不能保证"不重"。去重必须在业务层做，这是分布式系统的铁律。

### 错误用法及后果

**错误1：消费者处理时间太长，导致 Rebalance**

```java
// 错误：处理一条消息要 1 分钟，而 max.poll.interval.ms 默认 5 分钟
// 如果处理时间超过这个阈值，消费者被认为"死亡"，触发 Rebalance
@KafkaListener(topics = "order")
public void onMessage(ConsumerRecord<String, String> record) {
    Thread.sleep(60000); // 模拟慢处理
    process(record);
}
```

**后果**：频繁 Rebalance，消费进度停滞，消息堆积。

**修复**：增大 `max.poll.interval.ms`，或者把慢处理放到线程池异步执行，消费者只负责快速 ack。

**错误2：Partition 数 < 消费者数，浪费资源**

```
3 个 Partition，启动 5 个消费者 → 有 2 个消费者永远分不到 Partition，空转
```

**后果**：资源浪费，且容易造成误解（以为并行度是 5，实际只有 3）。

**修复**：Partition 数 ≥ 消费者数，且 Partition 数够了之后，增加消费者才有意义。

## 边界情况和坑

### 坑1：消息重复（At least once 的代价）

**触发场景**：消费者处理完消息，但在提交 Offset 前崩溃。重启后，消息被重新消费。

**成因**：Offset 提交和业务处理不是原子操作。分布式系统中，只要有两个写操作（业务 DB 写入 + Offset 提交），就无法保证原子性。

**解决方案**：
- 业务幂等（推荐）：唯一键 + 去重表/Redis SET NX/MongoDB 唯一索引
- 事务消息：RocketMQ 支持，Kafka 0.11+ 也支持但实现复杂
- 接受重复：如果业务本身幂等（比如 Redis INC R 天然幂等），可以忽略

### 坑2：消息丢失

**触发场景1**：生产者发消息，没开 ACK 确认，`acks=0`，Broker 宕机，消息丢了。

**触发场景2**：消费者"自动提交"模式，`enable.auto.commit=true`，拉到消息后还没处理就提交了，然后处理失败。

**触发场景3**：RabbitMQ 收到消息存在内存里，没刷盘就宕机了。

**解决方案**：
- 生产者：`acks=all`（等所有副本确认）
- 消费者：手动提交 Offset，处理完再提交
- Broker：开启持久化（Kafka 默认持久化，RabbitMQ 需要 durab le=true）

### 坑3：消息顺序性

**触发场景**：同一个订单的创建、支付、发货三条消息，被发到不同 Partition，消费者并行处理，可能"支付"比"创建"先被处理。

**成因**：分布式系统里，哈希分区是并行的基础，但并行和有序是矛盾的。

**解决方案**：
- 局部有序：相同业务键（如 orderId）发到同一个 Partition（Kafka 的 Partitioner 支持自定义）
- 全局有序：只能用 1 个 Partition（彻底丧失并行度，慎用）
- 有序 + 并行：在 Partition 内保证有序，不同 Partition 之间允许无序（大多数业务的真实需求）

### 坑4：消息堆积

**触发场景**：生产者 TPS 10000，消费者 TPS 1000，差 10 倍，消息越堆越多。

**成因**：生产速度 > 消费速度。可能原因：消费者逻辑太慢、消费者数量不够、数据库成了瓶颈。

**解决方案**：
- 短期：增加消费者实例（不超过 Partition 数）
- 中期：优化消费者处理逻辑（异步化、批处理）
- 长期：评估是否需要扩容 Partition 数（Kafka 支持，但已有消息的 Partition 扩容需要技巧）
