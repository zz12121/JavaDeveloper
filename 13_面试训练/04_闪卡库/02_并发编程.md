## 1. JMM 内存模型

**Q: JMM 和 JVM 内存布局是一回事吗？**

**A:**
- **JVM 内存布局**：堆、栈、方法区的物理划分
- **JMM**：线程与内存交互的**抽象规范**，解决 CPU 缓存 + 编译器重排导致的可见性问题

```
主内存（所有共享变量）
    ↑↓ store/write   ↓↑ read/load
线程A工作内存    线程B工作内存（各自私有）

线程间只能通过主内存间接通信
```

**Tags:** #并发 #JMM #内存模型

---

**Q: happens-before 的定义？它保证的是"物理时间先后"吗？**

**A:**
**定义**：如果 A hb B，则 A 的结果对 B 可见，且 A 的执行顺序排在 B 之前。

**❌ 不保证物理时间上的先后**——两个 CPU 可以并行执行，只要 B 最终看到 A 的正确结果。

8 条规则：
1. 程序顺序（同一线程）
2. 监视器锁（unlock hb lock）
3. volatile（写 hb 读）
4. Thread.start()
5. Thread.join()
6. interrupt()
7. 终结规则
8. 传递性

**Tags:** #并发 #JMM #happens-before

---

**Q: volatile 可见性推导实战**

**A:**
```java
int x = 0;
volatile boolean flag = false;

// 线程A               // 线程B
x = 42;             if (flag) {  // volatile 读
flag = true;           System.out.println(x);
}                      }

// 推导：A hb B 通过 volatile 规则 + 程序顺序 + 传递性
// B 一定看到 x = 42
```

**Tags:** #并发 #JMM #volatile

---

**Q: 为什么共享的 long/double 要加 volatile？**

**A:**
Java 规范允许对非 volatile 的 64 位变量拆分为两次 32 位写。

```java
long timestamp;  // 非 volatile
// 线程A: timestamp = 0x0001_FFFFFFFF
// 线程B 可能读到: 0x0001_0000_0000（半写撕裂）
```

**防御**：加 `volatile` 或用 `AtomicLong`。

**Tags:** #并发 #JMM #内存模型

---

**Q: final 字段的初始化安全性是什么？什么情况下会失效？**

**A:**
Java 5+ 保证：构造器结束后，其他线程一定能看到 final 字段的正确值。

**⚠️ 前提**：
1. 构造器内完成初始化
2. **不要泄露 this 引用**（传给其他线程或匿名内部类）

```java
// ❌ 危险：构造器内泄露
new Thread(() -> System.out.println(this.x)).start();

// ✅ 安全：构造器外
Bad bad = new Bad();
new Thread(() -> System.out.println(bad.x)).start();
```

**Tags:** #并发 #JMM #final

## 2. volatile

**Q: volatile 能保证 i++ 的原子性吗？为什么？**

**A:**
**不能**。`i++` 包含三个操作：读 → 改 → 写，volatile 只保证单次读/写的原子性，不保证复合操作的原子性。

```java
volatile int i = 0;
i++;  // 实际执行：load → increment → store（可被其他线程打断）
```

需要原子性用 `AtomicInteger.incrementAndGet()`。

**Tags:** #并发 #volatile #CAS

---

**Q: synchronized vs volatile vs Lock vs Atomic* 对比？**

**A:**
| 机制 | 可见性 | 有序性 | 原子性 | 阻塞 |
|------|--------|--------|--------|------|
| volatile | ✅ | ✅ | 仅单次读写 | ❌ |
| synchronized | ✅ | ✅ | ✅ 互斥 | ✅ |
| Lock | ✅ | ✅ | ✅ | ✅ |
| Atomic* | ✅ | ✅ | ✅ CAS | ❌ |

## 3. CAS 与原子类

**Q: CAS 的 ABA 问题是什么？如何解决？**

**A:**
```
线程A: 读取 A → 切换
线程B: A → B → A（改回去了）
线程A: CAS(A, A) → 成功！但实际上中间被改过
```

**解决**：加版本号（版本号原子引用）

```java
AtomicStampedReference<Integer> ref = new AtomicStampedReference<>(100, 1);
ref.compareAndSet(100, 200, 1, 2);  // CAS + 版本号同时检查
```

**Tags:** #并发 #CAS #ABA问题

---

**Q: AtomicInteger 的底层实现？**

**A:**
```java
// 伪代码
private volatile int value;

public final int incrementAndGet() {
    int prev, next;
    do {
        prev = value;       // 读取
        next = prev + 1;    // 修改
    } while (!cas(value, prev, next));  // CAS 重试
    return next;
}
```

关键点：
- `value` 用 `volatile` 保证可见性
- CAS 在用户态完成，不需要进入内核态（比 synchronized 快）
- 自旋等待直到 CAS 成功

## 4. 锁机制

**Q: synchronized 锁升级过程？**

**A:**
```
无锁 → 偏向锁 → 轻量级锁 → 重量级锁

偏向锁（1个线程）：第一次 CAS 将线程ID写入对象头，之后同一线程直接进入
轻量级锁（多线程交替）：栈帧创建锁记录，CAS 替换对象头中的指针
重量级锁（竞争激烈）：向操作系统申请互斥量，线程阻塞
```

**不可逆**：只能升级，不能降级（JDK 15 开始偏向锁被废除）

**Tags:** #并发 #synchronized #锁升级

---

**Q: synchronized 和 ReentrantLock 的区别？**

**A:**
| 维度 | synchronized | ReentrantLock |
|------|-------------|---------------|
| 锁粒度 | 代码块/方法 | 任意范围 |
| 公平锁 | ❌ 不支持 | ✅ 支持 |
| 锁等待可中断 | ❌ | ✅ tryLock() |
| 多条件队列 | ❌ | ✅ newCondition() |
| 可重入 | ✅ | ✅ |
| 底层 | monitor 指令 | AQS |
| 性能（低竞争） | JIT 优化后相近 | 相近 |

**Tags:** #并发 #synchronized #ReentrantLock

**Q: 为什么 copyOnWrite 的 COW 系列不需要加锁？**

**A:**
**读不加锁，写复制**。

```java
// CopyOnWriteArrayList
public E get(int index) {
    return getArray()[index];  // 读直接访问数组快照
}

private boolean add(E e) {
    synchronized (lock) {
        Object[] elements = getArray();  // 复制一份
        Object[] newElements = Arrays.copyOf(elements, len + 1);
        newElements[len] = e;
        setArray(newElements);  // 替换引用
    }
}
```

读写完全不阻塞 → **读多写少场景性能极佳**，但写操作成本高（每次复制全数组）。

**Tags:** #并发 #CopyOnWrite #COW

## 5. AQS

**Q: AQS 的核心思想是什么？**

**A:**
```
队列同步器 = 状态(state) + FIFO 队列

state:
  - ReentrantLock：state = 重入次数
  - CountDownLatch：state = 倒计数
  - Semaphore：state = 可用许可数

acquire 流程：
  1. 尝试获取（tryAcquire）→ 成功？直接返回
  2. 失败？加入 FIFO 队列
  3. 排队等前驱节点释放 → 自己被唤醒 → 再试

release 流程：
  1. tryRelease → 修改 state
  2. 唤醒队列中的后继节点
```

**Tags:** #并发 #AQS #JUC

## 6. 并发容器

**Q: ConcurrentHashMap 为什么不支持 null key/value？**

**A:**
消除二义性：`get(key)` 返回 null 时，无法区分"key 不存在"还是"value 为 null"。

```java
// 如果允许 null：
map.put("k", null);
map.get("k");  // 返回 null，两种情况无法区分

// CHM 选择：禁止 null，让语义清晰
// null = key 不存在，没有歧义
```

Hashtable 和 synchronizedMap 允许 null，但有两个歧义共存。

**Tags:** #并发 #ConcurrentHashMap #并发容器

---

**Q: ConcurrentLinkedQueue 三大坑？**

**A:**
1. **size() 是 O(n)**：遍历链表计数，高并发循环中调用会爆炸
   → 用 `isEmpty()` 代替 `size() == 0`

2. **无界 → OOM 风险**：生产 > 消费 → 无限堆积
   → 用有界阻塞队列或 Semaphore 限流

3. **不适合生产-消费**：poll() 返回 null 不阻塞，消费者的空转浪费 CPU
   → 用 BlockingQueue（take() 会阻塞等待）

**Tags:** #并发 #ConcurrentLinkedQueue #并发容器

---

**Q: 并发 Set 怎么选？**

**A:**
```
不需要排序 → ConcurrentHashMap.newKeySet()  // O(1)，默认选
需要排序   → ConcurrentSkipListSet          // O(log n)，基于跳表
读多写极少 → CopyOnWriteArraySet            // O(n)，小集合
```

**Tags:** #并发 #并发容器 #并发Set

## 7. 线程池

**Q: 线程池参数有哪些？拒绝策略有哪些？**

**A:**
```
七大参数：
  corePoolSize     — 核心线程数
  maxPoolSize     — 最大线程数
  keepAliveTime   — 空闲线程存活时间
  unit            — 时间单位
  workQueue       — 任务队列
  threadFactory   — 线程工厂
  handler         — 拒绝策略

流程：
  任务来 → 核心线程有空？直接执行
         → 核心线程满？进队列
         → 队列满？创建临时线程
         → 最大线程满？触发拒绝策略

拒绝策略：
  AbortPolicy        — 抛RejectedExecutionException（默认）
  CallerRunsPolicy   — 由调用线程执行
  DiscardPolicy      — 丢弃新任务
  DiscardOldestPolicy — 丢弃队列最旧任务
```

**Tags:** #并发 #线程池 #JUC

---

**Q: 为什么禁止用 Executors 创建线程池？**

**A:**
```java
// ❌ FixedThreadPool / SingleThreadPool
// 队列是 LinkedBlockingQueue(Integer.MAX_VALUE) → 堆积 → OOM

// ❌ CachedThreadPool / ScheduledThreadPool
// 最大线程数 Integer.MAX_VALUE → 无限创建 → OOM

// ✅ 正确做法：用 ThreadPoolExecutor，指定有界队列 + 最大线程数
new ThreadPoolExecutor(
    10, 20, 60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(100),
    new ThreadFactoryBuilder().setNameFormat("worker-%d").build(),
    new ThreadPoolExecutor.AbortPolicy()
);
```

## 8. 并发工具类

**Q: CountDownLatch vs CyclicBarrier 核心区别？**

**A:**
```
                    CountDownLatch          CyclicBarrier
可重用              ❌ 一次性               ✅ 循环
线程角色            不对等（主线程等子任务）  对等（互相等待）
典型场景            等待所有任务完成          多轮同步
底层                AQS 共享模式            ReentrantLock+Condition
回调                无                      barrierCommand
```

记忆口诀：**门闩打开就没了，栅栏到了还能再用一轮**

**Tags:** #并发 #工具类 #CountDownLatch #CyclicBarrier

---

**Q: Semaphore 的 state 可以是负数吗？**

**A:**
**可以**。state 为负表示"有人在等"。

```java
Semaphore sem = new Semaphore(3);
sem.acquire();  // state = 2
sem.acquire();  // state = 1
sem.acquire();  // state = 0
sem.acquire();  // state = -1（有人进了等待队列）
```

**注意**：
- acquire 和 release 可以是不同线程（允许，但不推荐）
- release 不会校验上限，permit 数可以超过初始值

**Tags:** #并发 #工具类 #Semaphore

---

**Q: Phaser 和 CyclicBarrier 相比增强在哪里？**

**A:**
1. **动态注册参与者**：`register()` / `arriveAndDeregister()` 动态增删
2. **多阶段执行**：每阶段自动推进，无需手动 reset
3. **onAdvance 回调**：每个阶段完成时可以执行清理/监控逻辑

```java
Phaser phaser = new Phaser(3);  // 初始3个
phaser.register();               // → 4个
phaser.arriveAndDeregister();   // → 3个，CyclicBarrier 做不到
```

**Tags:** #并发 #工具类 #Phaser

## 9. ThreadLocal

**Q: ThreadLocal 内存泄漏的原因？**

**A:**
```
ThreadLocalMap 的 Entry 继承 WeakReference<ThreadLocal<?>>
→ key（ThreadLocal）可以被 GC 回收 → key = null
→ value 无法访问但也不被 GC 回收（强引用链：Thread → ThreadLocalMap → Entry → value）
→ 内存泄漏

防御：
1. 用完调用 remove()（最推荐）
2. 避免长期持有大对象
3. 线程池场景：每次任务结束前 remove()
```

**Tags:** #并发 #ThreadLocal #内存泄漏

---

**Q: ThreadLocal 在父子线程间传递值怎么做？**

**A:**
普通 ThreadLocal **不继承**——父线程 set()，子线程 get() 拿不到。

**解决**：用 `InheritableThreadLocal`

```java
InheritableThreadLocal<String> tl = new InheritableThreadLocal<>();
tl.set("main-value");
new Thread(() -> System.out.println(tl.get())).start();  // 打印 main-value
```

**⚠️ 注意**：线程池复用线程时，值不会自动清理，需要 remove()。

**Tags:** #并发 #ThreadLocal #InheritableThreadLocal

## 10. 死锁

**Q: 死锁的四个必要条件？如何排查？**

**A:**
```
四个必要条件：
1. 互斥：资源一次只能被一个线程持有
2. 持有并等待：持有资源的同时请求其他资源
3. 不可抢占：资源不能被强制释放
4. 循环等待：线程形成环形依赖

破坏任意一个条件即可预防

排查工具：
  jstack <pid>                  — 打印线程栈，查 WAITING/BLOCKED
  jconsole / jmc                — 可视化线程状态
  Arthas thread -b              — 查找阻塞的线程
  代码审查：按固定顺序加锁 + tryLock 超时
```

**Tags:** #并发 #死锁 #排查
