# Spring 是如何解决循环依赖的？

> ⚠️ **先盲答**：什么是循环依赖？Spring 能解决哪些循环依赖？

---

## 盲答引导

1. 什么是循环依赖？—— A 依赖 B，B 依赖 A
2. Spring 能解决构造器注入的循环依赖吗？—— 不能
3. Spring 能解决 prototype 作用域的循环依赖吗？—— 不能
4. Spring 解决单例循环依赖靠的是什么？—— 三级缓存

---

## 知识链提示

```
Spring 循环依赖
  → [[Spring循环依赖]]
    → 三种注入方式 vs 循环依赖支持
      → 构造器注入：❌ 不支持（实例化时必须拿到完整对象）
      → setter 注入（prototype）：❌ 不支持（每次获取都创建新实例）
      → setter 注入（singleton）：✅ 支持（三级缓存）
    → 三级缓存（三个 Map）
      → 一级：singletonObjects（成品 Bean，直接可用）
      → 二级：earlySingletonObjects（半成品 Bean，提前暴露，未完成属性填充）
      → 三级：singletonFactories（Lambda，ObjectFactory<?>，创建半成品）
    → 三级缓存解决循环依赖的流程（A → B → A）
      → 创建 A → 属性填充时发现需要 B → 去容器拿 B
        → B 不存在 → 创建 B → 属性填充时需要 A
          → 去容器拿 A → 此时 A 还在创建中（一级里没有）
            → 从三级查 A 的 ObjectFactory → getEarlyBeanReference() → 创建 A 的半成品 → 放入二级
          → B 拿到了 A 的半成品 → 创建完成（放入一级）
        → 回到 A 的属性填充 → 拿到完整的 B → 创建完成
```

---

## 核心追问

| 追问 | 考点 |
|------|------|
| 为什么叫「三级缓存」，而不是一个 Map？ | 每级用途不同：三级存工厂，二级存半成品，一级存成品 |
| 二级缓存存在的意义是什么？ | 避免同一个 Bean 的半成品被多次创建 |
| 为什么三级缓存的 Lambda 可以「提前暴露」？ | ObjectFactory.getObject() 调用 getEarlyBeanReference()，在这里执行 AOP 代理 |
| 循环依赖 + AOP 代理会怎样？ | 三级缓存的 getEarlyBeanReference() 会创建代理对象（提前代理）|

---

## 参考答案要点

**只能解决 setter 注入 + singleton 的循环依赖**（构造器和 prototype 不行）。

**三级缓存的本质**：把「正在创建的 Bean」提前暴露给其他 Bean 使用，而不是等到完全创建完成。

**为什么需要三级**：
- 一级：成品（直接用）
- 二级：半成品（属性未填充完成，但被提前暴露）
- 三级：工厂（需要时创建半成品）
