# 反射

> 反射让 Java 程序在运行时动态获取类信息、调用方法、访问字段。它是 Spring IoC 容器创建 Bean、MyBatis 映射 SQL 到方法、JUnit 发现测试用例的底层机制——**没有反射，就没有现代 Java 框架。**

---

## 这个问题为什么存在？

没有反射时，Java 的所有类型和方法调用在编译期就完全确定：

```java
UserService service = new UserService();   // 类名写死
service.login("zhang", "123456");          // 方法名写死
```

这带来三个根本性限制：

1. **无法根据配置动态加载类**——你不可能在编译期知道数据库驱动、消息队列客户端的全限定类名
2. **无法编写通用框架**——Spring 需要调用任意 `@Bean` 方法的构造器，但它不知道你要写哪些类
3. **无法访问私有成员**——单元测试中需要测试 private 方法，框架需要注入 private 字段

**反射的本质是：把编译期确定的事情推迟到运行期。** 代价是放弃编译期类型检查和部分性能。

---

## 它是怎么解决问题的？

### 三种获取 Class 对象的方式

```java
// 1. 类字面量（编译期已知类型，最推荐）
Class<UserService> clazz1 = UserService.class;

// 2. 对象.getClass()（已有对象，获取其运行时类型）
UserService service = new UserService();
Class<?> clazz2 = service.getClass();

// 3. Class.forName()（运行时动态加载，框架场景最常用）
Class<?> clazz3 = Class.forName("com.example.UserService");
```

三者的区别不在于"哪种更正确"，而在于**你什么时候知道类名**：

| 方式 | 何时确定类名 | 是否初始化类 | 典型场景 |
|------|------------|------------|---------|
| `.class` | 编译期 | 不初始化 | 方法参数、泛型 `Class<T>` |
| `.getClass()` | 运行时（已有对象） | 已初始化 | 获取实例的实际类型 |
| `Class.forName()` | 运行时（字符串） | **会初始化** | JDBC 驱动加载、Spring Bean 创建 |

`Class.forName()` 会初始化类（执行 `<clinit>`、static 块），而 `ClassLoader.loadClass()` 只加载不初始化。这就是为什么 JDBC 驱动要用 `Class.forName("com.mysql.cj.jdbc.Driver")`——驱动类的 static 块里有 `DriverManager.registerDriver()` 调用，不执行就不注册。

### 反射调用方法

```java
Class<?> clazz = Class.forName("com.example.UserService");
Object instance = clazz.getDeclaredConstructor().newInstance();

Method method = clazz.getMethod("login", String.class, String.class);
Object result = method.invoke(instance, "zhang", "123456");
```

正常调用是**静态绑定**（编译期确定方法地址，直接跳转），反射调用需要三步：**查找方法 → 安全检查 → 动态分派**，每一步都有开销。

### Method.invoke() 的自适应优化

JDK 对反射调用做了一个巧妙的设计——**前 15 次用 native 方法，之后生成字节码访问器**：

```
第 1~15 次调用：
  Method.invoke() → NativeMethodAccessorImpl.invoke0() → JNI 调用
  （每次都经过 JNI 边界，慢但无预热开销）

第 16 次及之后：
  Method.invoke() → GeneratedMethodAccessorN.invoke() → 直接调用目标方法
  （通过字节码生成，绕过 JNI，接近直接调用的速度）
```

这个阈值（`sun.reflect.inflationThreshold`，默认 15）的设计权衡是：如果某个反射调用只执行几次，不值得花时间生成字节码；如果频繁调用，生成字节码后的收益远超一次性的生成开销。

### 访问和修改私有成员

```java
Field field = UserService.class.getDeclaredField("password");
field.setAccessible(true);              // 覆盖访问控制检查
String value = (String) field.get(service);   // 读取 private 字段
field.set(service, "newPassword");     // 修改 private 字段
```

`setAccessible(true)` 不是"破解"，而是通过 `Field.setAccessible0()` 修改了 `Field` 对象内部的 `override` 标志位。JVM 在执行字段/方法访问时检查这个标志位——如果为 true，跳过访问控制检查。

## 深入原理

### 反射 vs 直接调用：性能到底差多少？

```java
// 基准测试（JDK 17，10 万次调用）
// 直接调用：              ~1 ms
// 反射（不缓存 Method）：  ~200 ms（每次getMethod + invoke）
// 反射（缓存 + setAccessible）：~5 ms
// 反射（16 次后字节码优化）：~3 ms
```

反射慢的原因不只是"多了一层"，而是：

1. **方法查找**：`getMethod()` 需要在类的方法表中线性搜索（参数类型匹配）
2. **安全检查**：每次 `invoke()` 检查调用者是否有权限
3. **参数装箱**：`invoke(Object, Object[])` 的参数是 Object 数组，基本类型需要自动装箱
4. **JNI 开销**（native 阶段）：跨越 Java↔Native 边界

`setAccessible(true)` 消除了第 2 项；缓存 Method 消除了第 1 项；自适应优化消除了第 4 项。优化后的反射调用接近直接调用的 3~5 倍开销，对于非热路径完全可接受。

### 反射绕过泛型擦除

[[01_Java核心/04_泛型/泛型|泛型]] 在运行时被擦除，但反射提供了两种方式获取泛型信息：

```java
public class Container<T> {
    private List<String> names = new ArrayList<>();  // 字段声明中的泛型信息保留了

    public void set(List<T> items) { }               // 方法参数的泛型类型变量 T 保留了
}

// 方式 1：getGenericType() 获取字段的声明类型（含泛型参数）
Field field = Container.class.getDeclaredField("names");
Type type = field.getGenericType();  // ParameterizedType: List<String>

// 方式 2：getGenericParameterTypes() 获取方法的泛型签名
Method method = Container.class.getMethod("set", List.class);
Type paramType = method.getGenericParameterTypes()[0];  // TypeVariable: T
```

字段声明中的具体泛型（`List<String>`）被编译到字段的 `Signature` 属性中，所以运行时可以恢复。但类型变量 `T` 本身的信息（实际传入的是什么类型）在运行时确实丢失了——这是擦除的代价。

### 动态代理：反射的高级应用

反射最强大的应用是动态代理——在运行时生成一个实现指定接口的代理类：

```java
// JDK 动态代理
InvocationHandler handler = (proxy, method, args) -> {
    System.out.println("Before: " + method.getName());
    Object result = method.invoke(target, args);  // 反射调用目标方法
    System.out.println("After: " + method.getName());
    return result;
};

UserService proxy = (UserService) Proxy.newProxyInstance(
    UserService.class.getClassLoader(),
    new Class<?>[]{UserService.class},
    handler
);
```

`Proxy.newProxyInstance()` 在运行时通过字节码生成技术（`sun.misc.ProxyGenerator`）创建了一个类 `$Proxy0`，它实现了 `UserService` 接口，所有方法调用都转发给 `InvocationHandler.invoke()`。Spring AOP 的底层就是这个机制。

**但 JDK 动态代理只能代理接口**，不能代理类。如果需要代理类（没有接口的普通类），需要 CGLIB——它在运行时生成目标类的子类，通过方法重写实现拦截。

## 正确使用方式

### 缓存反射对象

```java
// ❌ 每次调用都重新查找
public Object invoke(String methodName, Object... args) throws Exception {
    Method method = target.getClass().getMethod(methodName, ...);
    return method.invoke(target, args);
}

// ✅ 启动时缓存，运行时复用
private final Map<String, Method> methodCache = new ConcurrentHashMap<>();

public void init() {
    for (Method m : target.getClass().getDeclaredMethods()) {
        m.setAccessible(true);
        methodCache.put(m.getName(), m);
    }
}

public Object invoke(String methodName, Object... args) throws Exception {
    return methodCache.get(methodName).invoke(target, args);
}
```

### 用 Constructor.newInstance() 替代 Class.newInstance()

```java
// ❌ JDK 9+ 已废弃：只能调无参构造器，且不处理构造器异常
Object obj = clazz.newInstance();

// ✅ 推荐：可调有参构造器，构造器异常正常抛出
Constructor<?> ctor = clazz.getDeclaredConstructor(String.class, int.class);
ctor.setAccessible(true);
Object obj = ctor.newInstance("zhang", 30);
```

`Class.newInstance()` 被废弃的原因：它把构造器里的 checked exception 包装成 `InvocationTargetException`，丢失了原始异常信息，且无法调用带参数的构造器。

## 边界情况和坑

### 坑 1：反射可以破坏单例模式

```java
// 即使构造器是 private 的
Constructor<Singleton> ctor = Singleton.class.getDeclaredConstructor();
ctor.setAccessible(true);               // 绕过 private
Singleton instance2 = ctor.newInstance();  // 创建了第二个实例！
```

**成因**：`setAccessible(true)` 可以访问任何构造器，private 构造器拦不住反射。

**解法**：
- [[01_Java核心/02_面向对象/枚举|枚举单例]]是唯一天然防反射的方式——`Constructor.newInstance()` 对枚举类型会直接抛 `IllegalArgumentException`
- 非枚举单例只能在构造器中检查是否已有实例：`if (INSTANCE != null) throw new RuntimeException()`，但这种方式无法阻止序列化攻击

### 坑 2：反射绕过泛型检查，运行时才爆炸

```java
List<String> list = new ArrayList<>();
list.add("hello");

Method addMethod = list.getClass().getMethod("add", Object.class);
addMethod.invoke(list, 123);  // 成功添加 Integer！

String s = list.get(1);      // ClassCastException！
```

**成因**：泛型只在编译期检查，`add(Object)` 是运行时的真实签名。反射绕过了编译期检查，但后续代码仍按 `List<String>` 使用，类型不匹配在 `get()` 时才暴露。

### 坑 3：getDeclaredMethods() 不包含父类方法

```java
// getMethods()：获取所有 public 方法（含继承的）
// getDeclaredMethods()：获取当前类声明的所有方法（含 private，不含继承的）

// 如果要获取父类的 private 方法：
Class<?> current = clazz;
while (current != null) {
    for (Method m : current.getDeclaredMethods()) {
        // 处理
    }
    current = current.getSuperclass();
}
```

### 坑 4：JDK 9+ 模块系统限制反射

JDK 9 引入的模块系统（JPMS）默认不允许反射访问非导出的包：

```
java.lang.reflect.InaccessibleObjectException:
  Unable to make field private int java.util.ArrayList.size accessible:
  module java.base does not "opens java.util" to unnamed module
```

**解法**：启动时添加 `--add-opens java.base/java.util=ALL-UNNAMED`，这在升级 JDK 版本时是常见的兼容性问题。

### 坑 5：反射调用可变参数方法的参数传递陷阱

```java
public void varargs(String... args) { }

// 反射调用时要注意：可变参数编译后是数组
Method method = clazz.getMethod("varargs", String[].class);

// ❌ 错误：传了两个独立的 String
method.invoke(obj, "a", "b");  // 实际调用 varargs("a", "b") ✅ 没问题

// ❌ 但如果参数恰好是 Object[] 会被误解包
method.invoke(obj, new Object[]{"a", "b"});  // varargs("a", "b") ✅
method.invoke(obj, new Object[]{new String[]{"a", "b"}});  // varargs(["a", "b"]) ✅

// 规则：如果传 Object[] 且编译器无法推断，需要强制包装
method.invoke(obj, (Object) new String[]{"a", "b"});  // 明确告知这是一个参数
```
