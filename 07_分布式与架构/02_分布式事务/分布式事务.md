# 分布式事务

## 这个问题为什么存在？

> 单体应用里，多个操作在同一个数据库事务里，要么全成要么全败。拆成微服务后，订单服务调库存服务、库存服务调支付服务——任何一个环节失败，已经提交的事务无法回滚。问题是：**跨服务、跨数据库，怎么保证数据一致性？**

这本质上是在**强一致性（ACID）** 和 **可用性（CAP）** 之间找平衡。

## 它是怎么解决问题的？

### 2PC：两阶段提交（理论模型）

```
    协调者（Coordinator）
         │
    ┌────┼────────┬────────┐
    │    │        │        │
    ▼    ▼        ▼        ▼
    TM1  TM2      TM3      TM4  （参与者 / 事务管理器）
```

#### 阶段一：Prepare（投票阶段）

```
协调者 → 所有参与者：Can you commit?
参与者1 → 执行事务（不提交），写 Undo/Redo 日志，返回 Yes
参与者2 → 执行事务（不提交），返回 Yes
参与者3 → 执行失败，返回 No
```

#### 阶段二：Commit / Rollback（提交阶段）

```
所有参与者都返回 Yes → 协调者发送 Commit → 所有参与者提交
有任何参与者返回 No → 协调者发送 Rollback → 所有参与者回滚
```

### 2PC 的核心问题

1. **同步阻塞**：所有参与者都在等协调者的指令，资源（数据库连接）长时间占用。
2. **协调者单点故障**：协调者宕机后，参与者一直等待，导致资源死锁。
3. **数据不一致**：阶段二部分 Commit 成功、部分失败 → 集群状态不一致。
4. **无法彻底解决**：协调者宕机 + 参与者也宕机 → 无法判断应该提交还是回滚。

### 3PC：引入超时机制

3PC 在 2PC 的基础上加了 `CanCommit` 阶段（询问是否能提交，不执行业务逻辑），并引入超时机制：

```
CanCommit → PreCommit → DoCommit
```

如果参与者在 `PreCommit` 后超时没收到 `DoCommit`，**默认提交**。减少了协调者故障后的阻塞，但**网络分区下仍可能不一致**。

### TCC：业务层面的两阶段

TCC 不依赖数据库的事务机制，而是由**业务逻辑**实现两阶段：

```
Try 阶段：预留资源（如冻结库存 10 件，而不是扣减 10 件）
Confirm 阶段：确认预留（冻结 → 实际扣减）
Cancel 阶段：取消预留（冻结 → 释放回原库存）
```

#### TCC 的两个坑

**空回滚**：Try 还没执行，Cancel 先到了（网络超时重试）。Cancel 需要判断 Try 是否执行过，没执行过也要能处理。

```
Cancel 逻辑：
  if (try 未执行) {
      记录 "空回滚" 标记
      return;  // 什么都不做，但不能报错
  }
  执行回滚...
```

**悬挂**：Cancel 比 Try 先执行（网络严重超时）。Try 执行时，发现已经被 Cancel 了，不能再执行。

```
Try 逻辑：
  if (已执行过 Cancel) {
      return;  // 悬挂，直接退出
  }
  执行预留...
```

### 最终一致性：消息队列方案

```
本地事务：
  1. 业务操作（如创建订单）
  2. 写消息表（订单ID + 消息内容）【同一个本地事务】
  
后台任务：
  扫描消息表 → 发送到 MQ → 收到 ACK 后删除消息表记录
  
消费者：
  消费消息 → 执行对应业务（如扣减库存）→ 确认消费
```

**关键**：消息表和业务操作在同一个数据库事务里，保证"业务做了，消息一定发出去"。消费者需要**幂等**（同一条消息消费多次，结果一样）。

### Saga 模式：长流程事务

```
OrderCreated → InventoryReserved → PaymentDone → ...
    ↓            ↓                  ↓
 补偿1        补偿2              补偿3
```

如果第 3 步失败，依次执行：
```
补偿2（释放库存）→ 补偿1（取消订单）
```

Saga 适合**流程长、无法冻结资源**的场景（如跨多个微服务的业务流程）。

## 它和相似方案的本质区别是什么？

| 方案 | 一致性 | 性能 | 复杂度 | 适用场景 |
|---|---|---|---|---|
| XA（2PC） | 强一致 | 低（同步阻塞） | 低（数据库内置） | 单体应用多数据源 |
| TCC | 最终一致 | 中 | 高（需写三个接口） | 金融、扣款等强一致需求 |
| Saga | 最终一致 | 高 | 中 | 长流程业务（旅游、电商下单） |
| 本地消息表 + MQ | 最终一致 | 高 | 中 | 对一致性要求不高的业务 |
| AT（自动补偿，Seata） | 最终一致 | 中 | 低 | 希望无侵入的分布式事务 |

### Seata 的 AT 模式

Seata 通过**拦截 JDBC 操作**，自动生成反向补偿 SQL：

```
业务 SQL：UPDATE inventory SET count = count - 10 WHERE id = 1
Seata 自动记录前镜像：count = 100
Seata 自动记录后镜像：count = 90
回滚时：UPDATE inventory SET count = 100 WHERE id = 1
```

对业务代码零侵入，但依赖 Seata Server（TC——事务协调器）。

## 正确使用方式

### TCC 接口设计

```java
public interface InventoryTccService {
    @TwoPhaseBusinessAction(name = "deduct", commitMethod = "confirm", rollbackMethod = "cancel")
    boolean tryDeduct(BusinessActionContext context, Long orderId, Integer count);
    
    boolean confirm(BusinessActionContext context);  // 确认：冻结 → 扣减
    boolean cancel(BusinessActionContext context);    // 取消：冻结 → 释放
}
```

### 幂等消费（消息队列方案必须）

```java
@KafkaListener(topics = "inventory-deduct")
public void consume(ConsumerRecord<String, String> record) {
    String msgId = record.key();  // 用消息 ID 做幂等
    if (redisTemplate.opsForValue().setIfAbsent("msg:" + msgId, "1")) {
        process(record.value());  // 第一次消费，执行业务
    }
    // 重复消息，直接 ACK
}
```

## 边界情况和坑

### 2PC 协调者宕机

```
协调者在阶段一之后宕机：
  → 所有参与者都停留在 "等待协调者指令" 状态
  → 资源（数据库连接）一直不释放
  → 需要人工介入，或者新的协调者从日志恢复状态
```

Seata 的 TC 通过**事务日志持久化**解决这个问题：新 TC 启动后，从日志恢复所有未完成事务的状态。

### MQ 消息重复发送

```
生产者发送消息 → Broker 写入成功 → 网络超时 → 生产者重试 → 同一条消息发送两次
```

消费者必须幂等。常见方案：
1. 数据库唯一约束（推荐）
2. Redis `SET NX`（设置过期时间）
3. 状态机判断（已处理过的状态不再处理）

### TCC 的 Confirm / Cancel 必须幂等

```
网络超时 → 协调者重试 Confirm → 同一条 Try 被 Confirm 两次
```

Confirm 和 Cancel 都要支持幂等（通过事务 ID 判断是否已执行）。
