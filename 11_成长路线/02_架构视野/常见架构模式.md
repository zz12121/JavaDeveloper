# 常见架构模式

> 架构模式是前人经验的总结。理解每个模式的适用场景和代价，比背定义重要 100 倍。

---

## 一、架构模式总览

```
架构演进路线：
单体架构 → 垂直拆分 → SOA → 微服务 → Service Mesh → Serverless

选型核心矛盾：
- 单体：简单，但扩展困难
- 微服务：扩展灵活，但运维复杂
- Serverless：极致弹性，但冷启动+状态管理难
```

---

## 二、单体架构（Monolithic）

### 2.1 结构

```
                    ┌──────────────┐
                    │ 单体应用      │
                    │              │
                    │ Controller   │
                    │ Service      │
                    │ DAO          │
                    │              │
                    └──────┬───────┘
                           │
                       ┌───▼────────┐
                       │   MySQL     │
                       └────────────┘
```

### 2.2 优缺点

| 优点 | 缺点 |
|------|------|
| 开发简单，上手快 | 代码耦合，难以拆分 |
| 部署简单（一个包） | 扩展性差（要扩全量扩） |
| 事务管理简单（本地事务） | 技术栈绑定，难以升级 |
| 调试方便 | 单点故障影响全量 |

### 2.3 适用场景

```
✅ 项目初期 / MVP 阶段
✅ 小团队（< 10 人）
✅ 业务逻辑不复杂
✅ 对运维要求不高

❌ 高并发场景
❌ 大团队协作
❌ 需要独立扩缩容的模块
```

## 三、微服务架构（Microservices）

### 3.1 结构

```
┌────┐  ┌────┐  ┌────┐  ┌────┐
│用户│  │订单│  │商品│  │支付│   → 每个服务独立部署
└─┬──┘  └─┬──┘  └─┬──┘  └─┬──┘
  │         │        │        │
──┼─────────┼────────┼────────┼──
  │    服务注册/发现（Nacos/Eureka）    │
  │    配置中心（Nacos/Apollo）       │
  │    网关（Spring Cloud Gateway）    │
```

### 3.2 核心组件

| 组件 | 作用 | 主流方案 |
|------|------|---------|
| 服务注册/发现 | 服务自动上下线感知 | Nacos / Eureka / Consul |
| 配置中心 | 集中管理配置，动态刷新 | Nacos / Apollo / Spring Cloud Config |
| API 网关 | 路由、鉴权、限流 | Spring Cloud Gateway / Kong |
| 服务通信 | RPC 调用 | Dubbo / Feign / gRPC |
| 熔断限流 | 防止雪崩 | Sentinel / Hystrix |
| 分布式追踪 | 调用链可视化 | SkyWalking / Zipkin / Jaeger |

### 3.3 微服务拆分原则

```
拆分依据（优先级从高到低）：
1. 业务边界（DDD 限界上下文）
2. 数据独立性（能否独立数据库？）
3. 团队组织（康威定律：架构反映组织结构）
4. 部署独立性（能否独立发布？）

拆分禁忌：
❌ 拆得过细（3 人维护 50 个服务）
❌ 没有考虑数据一致性（分布式事务地狱）
❌ 没有配套的 DevOps（手动部署每个服务）
```

## 四、SOA 架构（面向服务）

```
SOA vs 微服务：

SOA：
- 重 ES（企业服务总线）
- 服务共享数据库
- 适合大型企业系统整合

微服务：
- 去 ES，直接调用
- 每个服务独立数据库
- 适合互联网业务
```

## 五、事件驱动架构（Event-Driven）

### 5.1 结构

```
┌────┐  Event  ┌────────┐  消费  ┌────┐
│服务A│ ──────→ │  消息队列  │ ─────→ │服务B│
└────┘         │(Kafka/RabbitMQ)│        └────┘
┌────┐  Event  └────────┘  消费  ┌────┐
│服务C│ ──────────────────────→ │服务D│
└────┘                                  └────┘
```

### 5.2 核心概念

| 概念 | 说明 |
|------|------|
| 事件（Event） | 已经发生的事情（订单已创建） |
| 命令（Command） | 要求做的事情（创建订单） |
| 事件溯源（Event Sourcing） | 用事件日志重建状态 |
| CQRS | 读写分离（Command/Query 分离） |

### 5.3 优缺点

| 优点 | 缺点 |
|------|------|
| 解耦（服务间不直接调用） | 一致性难以保证（最终一致性） |
| 异步（提高吞吐量） | 调试困难（调用链不直观） |
| 弹性（消费者挂了不影响生产者） | 消息队列成为单点 |

## 六、Service Mesh

### 6.1 核心思想

```
传统微服务：     Service Mesh：
┌────────┐      ┌────────┬────────┐
│ 业务逻辑 │      │ 业务逻辑 │ Sidecar│
│ 服务发现 │  →   │          │(Envoy) │
│ 负载均衡 │      └────────┴────────┘
│ 熔断限流 │              ↑
└────────┘              │ 统一管理
                       ↓
                   Control Plane
```

### 6.2 主流方案

| 方案 | 语言 | 特点 |
|------|------|------|
| Istio | Go | 最主流，功能全面 |
| Linkerd | Rust | 轻量，性能好 |
| Consul Connect | Go | 与 Consul 深度整合 |

## 七、Serverless 架构

### 7.1 核心特征

```
传统服务：              Serverless：
1. 申请服务器          1. 只写函数
2. 部署应用            2. 事件触发
3. 保持运行            3. 按需运行（用完即毁）
4. 手动扩缩容          4. 自动弹性

适用场景：
✅ 突发流量（秒杀回调）
✅ 低频任务（定时备份）
✅ 事件驱动（文件上传后处理）

不适用：
❌ 长耗时任务（超过超时限制）
❌ 有状态服务（每个请求独立）
❌ 极低延迟要求（冷启动 100ms+）
```

### 7.2 主流平台

| 平台 | 厂商 |
|------|------|
| AWS Lambda | Amazon |
| Azure Functions | Microsoft |
| 云函数 | 腾讯云/阿里云 |
| Knative | CNCF（Kubernetes 上的 Serverless） |

## 八、CQRS 架构模式

```
CQRS = Command Query Responsibility Segregation
命令查询职责分离

传统：
┌─────────────┐
│   UserService │
│  createUser() │  → 写
│  getUser()     │  → 读
└─────────────┘

CQRS：
┌─────────────┐   事件   ┌─────────────┐
│  Write Service│ ──────→ │  Read Service │
│  (主库)       │         │  (从库/ES)   │
└─────────────┘         └─────────────┘
```

**适用场景**：读写比例悬殊（如 20:1），读模型可以接受短暂延迟。

## 九、架构选型决策树

```
需要快速上线？
├── 是 → 单体架构
└── 否 → 继续

团队规模 > 10 人？
├── 是 → 微服务 / SOA
└── 否 → 单体（模块化）

需要独立扩缩容？
├── 是 → 微服务
└── 否 → 单体（加缓存）

异步场景多？
├── 是 → 事件驱动（消息队列）
└── 否 → 同步调用（RPC/REST）

流量波动巨大？
├── 是 → Serverless（函数计算）
└── 否 → 微服务 + K8s HPA
```
