# 调度机制

> 调度器（Scheduler）不是「随机分配 Pod 到 Node」，是**基于资源请求、亲和性、污点容忍的复杂决策过程**。不理解调度机制，Pod 一直 `Pending` 时你只能重启大法。

---

## 这个问题为什么存在？

Pod 调度不好会导致什么？

```
场景 A：资源不均匀
  Node A：CPU 10%，内存 20%
  Node B：CPU 99%，内存 99%
  → 因为调度时没考虑资源，Pod 都挤到 Node B 了

场景 B：高可用没保证
  3 个 Pod 副本都被调度到同一台 Node
  → Node 挂了，服务全挂

场景 C：特殊硬件浪费
  GPU 节点只有 2 台
  普通 Pod 被调度到 GPU 节点
  → GPU 被占用了，真正需要 GPU 的 Pod 没资源了

场景 D：本地存储 Pod 被迁移了
  Pod 用了 Node 本地存储
  → 被重新调度到其他 Node
  → 数据丢了
```

**调度器要解决的核心问题**：

1. **资源匹配**：Node 有足够的 CPU/内存吗？
2. **高可用**：副本打散到不同 Node / Zone
3. **亲和性**：Pod 想去哪种 Node？不想和哪种 Pod 在一起？
4. **污点容忍**：某些 Node 是「特殊」的，普通 Pod 不能调度上去

---

## 它是怎么解决问题的？

### 调度流程（简化版）

```
1. 监听：Scheduler 监听 API Server，发现有 Pod 还没调度（nodeName 为空）

2. 预选（Filtering）：
   遍历所有 Node，过滤掉不满足条件的：
   - Node 资源够吗？
   - Node 有污点（Taint）且 Pod 没容忍（Toleration）吗？
   - Node 的标签匹配 Pod 的 nodeSelector 吗？

3. 优选（Scoring）：
   对通过预选的 Node 打分：
   - 资源使用越少，分越高（Least Requested Priority）
   - Pod 和同 Deployment 的其他 Pod 是否在同一 Zone？（打散）

4. 绑定（Binding）：
   选出最高分的 Node，把 Pod 的 nodeName 字段更新为这个 Node
```

### 资源请求（Resources Requests/Limits）

```yaml
# Pod 的资源配置
apiVersion: v1
kind: Pod
spec:
  containers:
    - name: user-service
      image: user-service:1.0.0
      resources:
        requests:        # 调度依据（Scheduler 只看这个）
          cpu: "500m"     # 0.5 核
          memory: "512Mi"
        limits:           # 硬限制（超过就被 OOM Kill）
          cpu: "1"
          memory: "1Gi"
```

**`requests` vs `limits` 的区别**：

| 字段 | 作用 | 超过会怎样 |
|------|------|---------|
| `requests` | Scheduler 调度依据（Node 必须有这么多「可分配」资源） | 不影响运行时 |
| `limits` | 运行时硬限制 | CPU：限速（throttling）；内存：OOM Kill |

```
调度时：
  Scheduler 只看法 request.cpu=500m
  → Node 必须有 500m 可分配 CPU，才能调度上去

运行时：
  容器可以用到 1 核（limit.cpu=1）
  → 但如果 Node 其他容器也在用 CPU，会限速
  → 如果内存用到 1Gi（limit.memory=1Gi），容器被 Kill
```

### 节点亲和性（Node Affinity）

```yaml
apiVersion: v1
kind: Pod
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:  # 硬要求（不满足就不调度）
        nodeSelectorTerms:
          - matchExpressions:
              - key: disktype
                operator: In
                values:
                  - ssd       # 必须调度到 disktype=ssd 的 Node
      preferredDuringSchedulingIgnoredDuringExecution:  # 软要求（尽量，不满足也调度）
        - weight: 100
          preference:
            matchExpressions:
              - key: zone
                operator: In
                values:
                  - zone-a     # 尽量调度到 zone-a
```

### Pod 亲和性/反亲和性（Pod Affinity/Anti-Affinity）

```yaml
affinity:
  podAntiAffinity:    # Pod 反亲和性（打散 Pod）
    requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchLabels:
            app: user-service
        topologyKey: kubernetes.io/hostname   # 按 Node 打散
    # 效果：user-service 的 Pod 不会被调度到已经有 user-service Pod 的 Node
```

**典型场景**：打散副本，保证高可用。

### 污点和容忍（Taints and Tolerations）

```
Taint（污点）：打在 Node 上，意思是「我有污点，普通 Pod 别来」
Toleration（容忍）：写在 Pod 上，意思是「我能容忍某些污点」
```

```yaml
# Node 打污点（通常用 kubectl）
kubectl taint nodes node1 gpu=true:NoSchedule
# 效果：没有容忍 gpu=true 的 Pod 不会被调度到 node1

# Pod 写容忍
apiVersion: v1
kind: Pod
spec:
  tolerations:
    - key: "gpu"
      operator: "Equal"
      value: "true"
      effect: "NoSchedule"   # 不接受新 Pod，但不驱逐已有 Pod
```

| Effect | 含义 |
|--------|------|
| `NoSchedule` | 不调度新 Pod 到该 Node（已有 Pod 不受影响） |
| `PreferNoSchedule` | 尽量不调度（软限制） |
| `NoExecute` | 不调度新 Pod，且驱逐已有 Pod |

## 它和相似方案的本质区别是什么？

### Kubernetes 调度 vs Docker Swarm 调度

| 维度 | Kubernetes | Docker Swarm |
|------|-------------|---------------|
| 调度策略 | 丰富（亲和性、反亲和性、污点容忍） | 简单（只有 constraints） |
| 资源感知 | ✓（requests/limits） | ✓（但功能弱） |
| 自定义调度器 | ✓（可以写自己的调度器） | ✗ |
| 复杂度 | 高 | 低 |

### Kubernetes 默认调度器 vs 自定义调度器

**为什么需要自定义调度器？**

- 机器学习任务：需要 GPU 拓扑感知（NVLink 连接的两张 GPU 性能更好）
- 延迟敏感应用：需要协同调度（多个 Pod 同时调度到相邻 Node）

```yaml
# 指定用自定义调度器
apiVersion: v1
kind: Pod
spec:
  schedulerName: my-custom-scheduler
```

## 正确使用方式

### 1. 永远写 `resources.requests`

```yaml
# ❌ 不写 requests（Scheduler 默认所有 Node 都能调度）
resources:
  limits:
    cpu: "1"
    memory: "1Gi"

# ✅ 写 requests（让 Scheduler 做正确决策）
resources:
  requests:
    cpu: "500m"
    memory: "512Mi"
  limits:
    cpu: "1"
    memory: "1Gi"
```

**不写 requests 的后果**：Node 资源超卖，所有 Pod 都挤到少数 Node，导致资源不均匀。

### 2. 用 `podAntiAffinity` 打散副本

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: user-service
spec:
  replicas: 3
  template:
    spec:
      affinity:
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
            - labelSelector:
                matchLabels:
                  app: user-service
              topologyKey: kubernetes.io/hostname
      # 效果：3 个副本会尽量调度到不同 Node
```

### 3. GPU 节点用 Taint 保护

```bash
# 给 GPU 节点打污点
kubectl taint nodes gpu-node1 gpu=true:NoSchedule

# GPU Pod 写 toleration（见上文）
# 效果：普通 Pod 不会跑到 GPU 节点上浪费资源
```

## 边界情况和坑

### 1. `Pending` 但不知道为什么

```bash
kubectl describe pod <pod-name>
# 在 Events 部分会看到：
# Warning  FailedScheduling  0/3 nodes are available: 3 Insufficient cpu.
# ↑ CPU 不足
# 或
# Warning  FailedScheduling  0/3 nodes are available: 3 node(s) had taint {gpu: true}, that the pod didn't tolerate.
# ↑ 污点没容忍
```

### 2. `limits.cpu` 不是「最多能用到多少」

```
误解：limits.cpu=1 → 容器最多用 1 核
实际情况：limits.cpu=1 → 容器可以用到 1 核，但超过后**不会被 Kill**，只会被限速（throttling）

内存不一样：limits.memory 超过后**立即 OOM Kill**
```

### 3. `requests` 可以超卖（Overcommit）

```
Node 有 4 核 CPU
Scheduler 可以调度总 requests.cpu=10 核的 Pod 到这个 Node
→ 因为 requests 只是「调度依据」，不是「运行时硬限制」

风险：所有 Pod 同时用 CPU 时，Node 会 CPU 不够用
```

### 4. 亲和性语法太复杂容易写错

```yaml
# ❌ 常见错误：topologyKey 写错
podAntiAffinity:
  requiredDuringSchedulingIgnoredDuringExecution:
    - topologyKey: zone   # ✗ 不存在的 key，Pod 会一直 Pending
    # 应该是 kubernetes.io/hostname 或 topology.kubernetes.io/zone
```
