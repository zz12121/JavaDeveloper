# 枚举

> 枚举的底层是一个继承了 `Enum<T>` 的 final 类，每个枚举值是一个 `public static final` 实例，在类加载阶段初始化——这使得它天然线程安全，也是实现单例模式的最佳方式。

---

## 这个问题为什么存在？

在没有枚举之前，Java 用"整数常量"表示一组固定值：

```java
// 旧写法：整数常量模式
public static final int SEASON_SPRING = 0;
public static final int SEASON_SUMMER = 1;
public static final int SEASON_AUTUMN = 2;
public static final int SEASON_WINTER = 3;
```

**问题**：
1. **无类型安全**：`setSeason(99)` 编译通过，但值非法
2. **无命名空间**：`WEEK_MONDAY` 和 `MONTH_MONDAY` 都是 `1`，容易混淆
3. **无自描述**：`System.out.println(0)` 输出 `0`，不知道是什么含义
4. **脆弱性**：在中间插入新常量，后续所有 ordinal 值都变

```java
// 枚举解决了这些问题
enum Season { SPRING, SUMMER, AUTUMN, WINTER }
setSeason(Season.SPRING);  // 类型安全，编译器检查
```

---

## 它是怎么解决问题的？

### 枚举编译后的真相

```java
// 源码
enum Color { RED, GREEN, BLUE }

// 编译后等价于（反编译 javap -c Color.class）
final class Color extends Enum<Color> {
    public static final Color RED = new Color("RED", 0);
    public static final Color GREEN = new Color("GREEN", 1);
    public static final Color BLUE = new Color("BLUE", 2);

    private static final Color[] $VALUES;

    static {
        $VALUES = new Color[]{RED, GREEN, BLUE};
    }

    public static Color[] values() { return $VALUES.clone(); }
    public static Color valueOf(String name) { return Enum.valueOf(Color.class, name); }
}
```

**关键特性**：
- 枚举类是 `final` 的，不能被继承
- 每个枚举值是 `public static final` 的，在 `<clinit>`（类初始化）中创建
- 构造器是 `private` 的，外部不能 new
- 继承了 `Enum<T>`，获得了 `name()`、`ordinal()`、`toString()` 等方法

### 枚举可以有字段、方法、构造器

```java
enum HttpStatus {
    OK(200, "成功"),
    NOT_FOUND(404, "未找到"),
    SERVER_ERROR(500, "服务器错误");

    private final int code;
    private final String message;

    // 构造器必须是 private（不写也默认 private）
    HttpStatus(int code, String message) {
        this.code = code;
        this.message = message;
    }

    public int getCode() { return code; }
    public String getMessage() { return message; }
}
```

### 枚举实现接口

```java
enum Operation implements BiFunction<Integer, Integer, Integer> {
    ADD { public Integer apply(Integer a, Integer b) { return a + b; } },
    SUB { public Integer apply(Integer a, Integer b) { return a - b; } };
    // 每个枚举值可以有不同的实现（类似策略模式）
}
```

枚举不能继承其他类（已经继承了 `Enum<T>`），但可以实现接口。

## 深入原理

### 枚举单例——为什么是"最佳方式"？

```java
public enum DatabaseConnection {
    INSTANCE;

    private Connection conn;

    DatabaseConnection() {
        this.conn = DriverManager.getConnection("jdbc:mysql://...");
    }

    public Connection getConnection() { return conn; }
}
```

**三种单例实现对比**：

| 维度 | 双重检查锁 | 静态内部类 | 枚举 |
|------|-----------|-----------|------|
| 线程安全 | 需要 `volatile` + `synchronized` | JVM 类加载保证 | JVM 类加载保证 |
| 序列化安全 | 需要 `readResolve()` | 需要 `readResolve()` | **天然安全** |
| 反射攻击 | ❌ 可以被破坏 | ❌ 可以被破坏 | ✅ **天然防破坏** |
| 代码量 | ~15行 | ~8行 | **~3行** |
| 延迟加载 | ✅ | ✅ | ❌ 类加载即初始化 |

**为什么反射不能破坏枚举？**

```java
// 反射源码（Constructor.java）
public T newInstance(Object ... initargs) {
    if (clazz.isEnum()) {
        throw new IllegalArgumentException("Cannot reflectively create enum objects");
    }
    // ...
}
```

JDK 在 `Constructor.newInstance()` 中直接拦截了对枚举类的反射实例化。

**为什么序列化不能破坏枚举？**

```
Java 序列化对枚举的特殊处理：

普通类的序列化：写入对象的所有字段 → 反序列化时通过 Unsafe.allocateInstance() 创建新对象
枚举的序列化：只写入枚举的 name（字符串） → 反序列化时通过 Enum.valueOf() 返回已有实例

所以枚举反序列化永远不会创建新对象，天然保证了单例！
```

### ordinal() 的坑

```java
enum Priority { LOW, MEDIUM, HIGH }

// ordinal() 返回声明顺序：LOW=0, MEDIUM=1, HIGH=2
// 危险：如果在中间插入新值
enum Priority { LOW, NORMAL, MEDIUM, HIGH }
// 现在 MEDIUM=2, HIGH=3 → 所有依赖 ordinal 的代码都错了！
```

**正确做法**：用自定义字段

```java
enum Priority {
    LOW(1), MEDIUM(2), HIGH(3);
    private final int level;
    Priority(int level) { this.level = level; }
    public int getLevel() { return level; }
}
```

《Effective Java》Item 35：**永远不要用 `ordinal()` 关联枚举值，用实例字段代替。**

### 枚举与抽象方法（每个值独立实现）

```java
enum PayType {
    CASH {
        @Override
        public void pay(BigDecimal amount) {
            System.out.println("现金支付：" + amount);
        }
    },
    ALIPAY {
        @Override
        public void pay(BigDecimal amount) {
            System.out.println("支付宝支付：" + amount);
        }
    };

    public abstract void pay(BigDecimal amount);
}
```

这是**策略模式**的内建实现——比定义接口 + 多个实现类更简洁，且所有策略类封装在同一个类型里。

## 正确使用方式

### 1. 用枚举替代整数常量

```java
// ❌ 不好：整数常量
public static final int STATUS_PENDING = 0;
public static final int STATUS_ACTIVE = 1;
setStatus(99);  // 编译通过，但值非法

// ✅ 好：枚举
enum Status { PENDING, ACTIVE }
setStatus(Status.PENDING);  // 类型安全
```

### 2. 给枚举加自定义字段，不用 ordinal()

```java
// ❌ 不好：依赖 ordinal()
enum ErrorCode { NOT_FOUND, SERVER_ERROR }
int code = error.ordinal();  // 脆弱，插入新值就全错

// ✅ 好：自定义字段
enum ErrorCode {
    NOT_FOUND(404), SERVER_ERROR(500);
    private final int code;
    ErrorCode(int code) { this.code = code; }
    public int getCode() { return code; }
}
```

### 3. 单例用枚举，不用双重检查锁

```java
// ✅ 最佳实践：枚举单例
public enum ConfigManager {
    INSTANCE;
    private final Properties props = new Properties();

    public String get(String key) { return props.getProperty(key); }
}
// 使用：ConfigManager.INSTANCE.get("db.url");
```

## 边界情况和坑

### 1. 枚举不能被继承，也不能继承其他类

```java
// ❌ 编译错误
class SubRed extends Color { }  // Color 是 final

enum Bad {              // ❌ 编译错误
    ;  // 空枚举体
    extends Object {}       // 枚举已经继承了 Enum，不能再继承
}
```

### 2. values() 每次调用都 clone 数组

```java
// values() 每次返回新数组
for (Color c : Color.values()) { }  // 每次都 clone

// 如果频繁调用，考虑缓存
private static final Color[] COLORS = Color.values();  // 缓存起来
```

### 3. switch 对枚举的特殊支持

```java
enum Color { RED, GREEN, BLUE }

switch (color) {
    case RED:    break;  // 不需要写 Color.RED
    case GREEN:  break;
    case BLUE:   break;
}
```

编译后，switch 枚举实际上使用的是 `ordinal()` 值（整数比较，比字符串比较快），但 Java 语法允许省略枚举类名。

### 4. 枚举的序列化只写 name

```java
// 序列化枚举时，只写入 name 字符串
ObjectOutputStream.writeObject(Color.RED);  // 只写 "RED"

// 反序列化时：Enum.valueOf(Color.class, "RED") → Color.RED
// 所以如果枚举 name 改了，反序列化旧数据会失败
```

**解决**：如果枚举需要序列化兼容，name 不要改。或者自己实现 `readResolve()` 做兼容映射。
