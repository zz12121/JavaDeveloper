# 垃圾回收算法

> GC 算法的设计就是在「吞吐量」「延迟」「内存碎片」三者之间取舍。重点：可达性分析 vs 引用计数、标记-清除/复制/标记-整理三大算法、分代收集的底层逻辑。

---

## 这个问题为什么存在？

C/C++ 里手动 `new/delete` 管理内存，开发者必须记住每一块内存的生命周期——忘记 `delete` 就内存泄漏，提前 `delete` 就悬空指针。

**核心问题**：如何让开发者专注业务逻辑，不用手动管理内存，同时保证不再使用的对象能被自动回收？

---

## 它是怎么解决问题的？

### 1. 判断对象是否存活：可达性分析

JVM 不用「引用计数」（无法解决循环引用），而是用**可达性分析（Reachability Analysis）**：

```
从 GC Roots 开始，向下搜索引用链：
  └── 能被 GC Roots 到达的对象：存活 ✅
      （无法到达的对象：可回收 ❌）
```

**GC Roots 包括**：
- 虚拟机栈（栈帧局部变量表）中引用的对象
- 方法区中类静态属性引用的对象
- 方法区中常量引用的对象
- 本地方法栈中 JNI（Native 方法）引用的对象

**为什么不用引用计数？** 循环引用无法解决：

```java
class Node { Node next; }

Node a = new Node();
Node b = new Node();
a.next = b;
b.next = a;   // 循环引用
a = null;
b = null;
// 引用计数都不为 0，但已经无法被程序访问 → 泄漏
```

### 2. 三种基础算法

#### 标记-清除（Mark-Sweep）

```
阶段1：标记（Mark）
  └── 从 GC Roots 遍历，标记所有存活对象
阶段2：清除（Sweep）
  └── 遍历整个堆，回收未被标记的对象
```

| | 标记-清除 |
|--|------------|
| **优点** | 实现简单 |
| **缺点** | 效率低（两次遍历全堆）；产生大量内存碎片 |

#### 复制（Copying）

```
将内存分为两块（From 和 To）：
1. 只使用 From 区
2. GC 时，将 From 区中存活的对象复制到 To 区
3. 清空 From 区
4. 交换 From 和 To 的角色
```

| | 复制算法 |
|--|----------|
| **优点** | 无内存碎片；效率高（只处理存活对象） |
| **缺点** | 内存利用率只有 50%（需要预留 To 区） |

**为什么新生代用复制算法？** 弱分代假说——新生代 90%+ 的对象都活不过第一次 GC，需要复制的对象极少，复制成本很低。

#### 标记-整理（Mark-Compact）

```
阶段1：标记（Mark）— 同标记-清除
阶段2：整理（Compact）
  └── 将存活对象向一端移动，然后清理边界以外的内存
```

| | 标记-整理 |
|--|----------|
| **优点** | 无内存碎片；内存利用率 100% |
| **缺点** | 需要移动对象，效率较低 |

**为什么老年代用标记-整理？** 老年代存活率高，复制成本高；标记-整理虽然慢，但避免了碎片问题。

### 3. 分代收集理论

JVM 把堆分为新生代和老年代，对不同的代用不同的算法——这是所有现代 GC 设计的基石。

**弱分代假说（Weak Generational Hypothesis）**：
1. 绝大多数对象都是「朝生夕死」的
2. 熬过越多次 GC 的对象，越难消亡

```
新生代（Young Gen）← 复制算法（存活率低，复制成本低）
      │
      │ 对象年龄达到阈值（默认 15）
      ▼
老年代（Old Gen）  ← 标记-整理 或 标记-清除（存活率高，不适合复制）
```

### 4. GC 的触发条件

| GC 类型 | 触发条件 |
|---------|---------|
| **Minor GC**（新生代 GC） | Eden 区空间不足 |
| **Major GC**（老年代 GC） | 老年代空间不足 |
| **Full GC**（全堆 GC） | 老年代不足 / 方法区不足 / `System.gc()`（只是建议） |

## 深入原理

### 为什么复制算法要把内存对半分？

HotSpot 的新生代并不是 50:50 分——而是 **Eden:Survivor = 8:1:1**。

```
新生代（总共 10 份）
├── Edan： 8 份（新对象在这里分配）
├── Survivor From：1 份
└── Survivor To：  1 份（始终有一块是空的）
```

**为什么可以这样？** 因为 90%+ 的新对象都活不过第一次 Minor GC，Survivor 区不需要太大。

### `finalize()` 为什么被废弃？

```java
// Object 类中的方法
@Deprecated(since = "9")
protected void finalize() throws Throwable {}
```

**三个致命问题**：
1. **执行时机不确定**——JVM 不保证何时调用，甚至不保证会调用
2. **性能差**——需要 GC 对含 `finalize()` 的对象做特殊处理，拖慢 GC
3. **可以复活对象**——在 `finalize()` 里把自己赋值给静态变量，对象就「死而复生」了

**替代方案**：用 `Cleaner`（JDK 9+）或 `PhantomReference`。

### 引用类型与 GC 的关系

GC 对四种引用的处理完全不同——这是理解 GC 行为的关键：

| 引用类型 | GC 行为 | 典型场景 |
|---------|---------|---------|
| **强引用** `A a = new A()` | 只要强引用还在，永不回收 | 普通对象引用 |
| **软引用** `SoftReference` | 内存不足时回收 | 缓存（内存够用时留着，不够时释放） |
| **弱引用** `WeakReference` | 下次 GC 时一定回收 | `WeakHashMap`（防止内存泄漏） |
| **虚引用** `PhantomReference` | 随时可能回收，必须配合 `ReferenceQueue` | 堆外内存释放（`DirectByteBuffer`） |

## 正确使用方式

### 1. 避免内存泄漏——及时释放引用

```java
// ❌ 错误：静态集合持有对象引用
public class MemoryLeak {
    private static List<Object> list = new ArrayList<>();
    public void add(Object obj) { list.add(obj); }
    // obj 永远无法被 GC 回收
}

// ✅ 正确：用 WeakReference 做缓存
public class SafeCache {
    private static Map<String, WeakReference<Object>> cache = new HashMap<>();
    public void put(String key, Object val) {
        cache.put(key, new WeakReference<>(val));
    }
    public Object get(String key) {
        WeakReference<Object> ref = cache.get(key);
        return (ref == null) ? null : ref.get();
    }
}
```

### 2. 不要用 `System.gc()` 干预 GC

```java
// ❌ 误区：认为 System.gc() 一定会触发 GC
System.gc();

// 实际情况：只是「建议」JVM 执行 GC，JVM 完全可以忽略
// 而且如果开启了 -XX:+DisableExplicitGC，System.gc() 直接被忽略
```

**为什么不要调用？** 现代 JVM 的 GC 策略经过了大量调优，手动调用反而打乱 JVM 的节奏。

### 3. 大对象直接用「老年代」分配

```bash
# 超过这个大小的对象，直接在老年代分配
# 避免短命大对象在新生代复制来复制去
-XX:PretenureSizeThreshold=1M
```

## 边界情况和坑

### 1. 内存碎片导致「明明还有空间，但分配失败」

标记-清除算法不整理内存，会产生碎片：

```
回收前：[Used][Used][Free][Used][Free][Free][Used]
回收后：[Used][Used][Free][Free][Free][Free][Free]
                    ↑
                               连续空间只有 4 个 slot
```

如果此时要分配一个需要 5 个连续 slot 的对象——**分配失败，触发 Full GC**。

**解决**：老年代如果用 CMS（标记-清除），需要定期做 Full GC 来整理；或者改用 G1/ZGC（它们做整理）。

### 2. `WeakHashMap` 的坑——value 持有 key 的引用

```java
WeakHashMap<BigObject, List<Ref>> map = new WeakHashMap<>();
BigObject key = new BigObject();
map.put(key, someListThatReferencesKey);  // ❌ value 引用了 key
key = null;
// WeakHashMap 无法回收这个 entry——因为 value 还持有 key 的引用
```

**成因**：`WeakHashMap` 只保证 key 是弱引用，但 value 是强引用。如果 value 间接引用了 key，就形成了引用环。

### 3. 误用 `finalize()` 导致对象延迟回收

含 `finalize()` 方法的对象，GC 时不会立即回收——而是把对象放到一个**引用队列**里，由 `Finalizer` 线程异步执行 `finalize()`，执行完才会真正回收。

**后果**：对象要经历**两次 GC** 才能被回收，如果 `Finalizer` 线程处理不及时，会导致大量对象堆积，触发 Full GC。

### 4. 并发场景下的 GC 误差

```java
// 业务代码
public void process() {
    List<byte[]> list = new ArrayList<>();
    for (int i = 0; i < 1000; i++) {
        list.add(new byte[1024]);  // 不断分配，但 list 还持有引用
    }
    list.clear();  // 现在对象才可以回收
}
```

**坑**：GC 只判断「此时」是否有引用——如果代码刚执行完 `list.clear()`，但 GC 还没触发，内存使用量仍然很高。这是"为什么我释放了对象，内存没降下来"的根本原因。
