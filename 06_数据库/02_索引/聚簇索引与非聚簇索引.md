# 聚簇索引与非聚簇索引

## 这个问题为什么存在？

> InnoDB的主键索引查一次IO就能拿到数据，二级索引却要查两次——为什么？
> 
> MyISAM的索引和数据是分离的，而InnoDB的聚簇索引把数据存在索引里——这两种设计各有什么优劣？

聚簇索引（Clustered Index）和非聚簇索引（Secondary Index，也叫非聚簇/二级索引）的区别，本质上是在回答一个问题：

**索引里面，要不要存数据？**

```
方案A：索引只存"键→地址"（非聚簇）
  - 先查索引，拿到数据地址
  - 再去数据文件读数据
  - 两次IO

方案B：索引把数据存在叶子节点（聚簇）
  - 查索引，叶子节点就有完整数据
  - 一次IO

方案B更快，但有个问题：
  - 数据只能按一种顺序存储（按主键排序）
  - 如果还有其他索引，它们的叶子节点存什么？
```

---

## 它是怎么解决问题的？

### InnoDB的聚簇索引（主键索引）

**定义**：聚簇索引不是"一种索引类型"，而是**数据存储方式**——数据行和相邻的键值紧凑地存储在一起。

```
InnoDB的主键索引（聚簇索引）：
                     [20, 40]
                    /    |    \
               [10,15] [25,30] [45,50]
                /   \   /   \   /   \
              ┌─────┐  ┌─────┐  ┌─────┐
              │id=10│  │id=20│  │id=25│
              │Tom  │  │Jerry│  │Mike │
              │age=20│  │age=25│  │age=30│
              └─────┘  └─────┘  └─────┘
              
叶子节点 = 索引键 + 完整数据行
```

**为什么叫"聚簇"？**

```
"聚簇" = 数据行按主键顺序存储，相邻的键值，数据也存储在相邻的页中。

好处：
- 按主键范围查询极快 (WHERE id BETWEEN 10 AND 20)
  → 数据在物理上也是连续的 → 顺序IO

- 主键等值查询极快 (WHERE id = 10)
  → 索引叶子节点就有完整数据 → 一次IO
```

**聚簇索引的选择顺序**（InnoDB的规则）：

```
InnoDB选择聚簇索引的顺序：
1. 有PRIMARY KEY → 用PRIMARY KEY做聚簇索引
2. 没有PRIMARY KEY，但有UNIQUE NOT NULL → 用第一个UNIQUE NOT NULL做聚簇索引
3. 都没有 → InnoDB自动生成一个6字节的隐藏row_id (自增) 做聚簇索引
```

```sql
-- 示例1：有主键
CREATE TABLE users (
  id INT PRIMARY KEY,  -- 用这个做聚簇索引
  name VARCHAR(50)
);

-- 示例2：没有主键，有UNIQUE NOT NULL
CREATE TABLE users (
  uuid CHAR(36) NOT NULL UNIQUE,  -- 用这个做聚簇索引
  name VARCHAR(50)
);

-- 示例3：都没有
CREATE TABLE users (
  name VARCHAR(50)
);
-- InnoDB自动生成row_id做聚簇索引 (用户看不到)
```

### InnoDB的二级索引（非聚簇索引）

**定义**：所有非主键索引，都是二级索引。它们的叶子节点不存完整数据，只存**主键值**。

```
InnoDB的二级索引（例如：idx_name ON users(name)）：
                     ['Jerry', 20]
                          |
               ['Mike', 25]  ['Tom', 10]
                     |              |
              ┌───────┐       ┌───────┐
              │name=  │       │name=  │
              │'Mike' │       │'Tom'  │
              │id=25  │       │id=10  │
              └───────┘       └───────┘
              
叶子节点 = 索引键 + 主键值
```

**为什么二级索引的叶子节点存主键值，而不是数据地址？**

```
原因1：数据移动时，不需要更新所有二级索引
  - 如果存数据地址（像MyISAM那样）
  - 当数据行因为页分裂而移动时 → 所有二级索引都要更新地址！
  - 如果存主键值 → 主键不会因为页分裂而改变 → 二级索引不需要更新

原因2：聚簇索引的数据是按主键排序的
  - 通过主键值，去聚簇索引查数据（回表）→ 效率还可以
```

**回表（Look-up / Book-mark Lookup）**：

```
查询：SELECT * FROM users WHERE name = 'Tom';

执行过程：
1. 通过idx_name找到 name='Tom' 的叶子节点
   → 得到主键值：id=10

2. 根据主键id=10，去聚簇索引查找
   → 得到完整数据行

这就是"回表"：二级索引查一次，聚簇索引再查一次。
```

### MyISAM的索引（都是非聚簇索引）

**MyISAM不支聚簇索引**——它的索引和数据完全分离。

```
MyISAM的索引（例如：idx_name ON users(name)）：
                     ['Jerry']
                         |
               ['Mike']           ['Tom']
                    |                  |
              ┌───────┐          ┌───────┐
              │name=  │          │name=  │
              │'Mike' │          │'Tom'  │
              │行号=5  │          │行号=2  │
              └───────┘          └───────┘
                                     |
                                     ▼
                          去数据文件读第2行

数据文件（.MYD）：
    行1  |  id=9,  name='Jerry'
    行2  |  id=10, name='Tom'
    行3  |  id=11, name='Mike'
```

**MyISAM的索引叶子节点存"行号"（row number）** → 需要去数据文件读取完整数据。

## 它和相似方案的本质区别是什么？

### InnoDB聚簇索引 vs MyISAM非聚簇索引

```
                     InnoDB (聚簇)             MyISAM (非聚簇)
─────────────────────────────────────────────────────────────
主键查询            1次IO (索引+数据在一起)    2次IO (先索引，再数据文件)
二级索引            需要回表 (2次IO)          需要去数据文件 (2次IO)
范围查询            极快 (数据物理连续)         慢 (数据可能不连续)
插入性能            主键有序插入快              快 (追加写)
页分裂              可能发生 (UUID主键)          不发生 (索引和数据分离)
适用场景            OLTP (高并发读写)         OLAP (只读)
```

**为什么InnoDB的主键查询比MyISAM快？**

```
InnoDB: SELECT * FROM users WHERE id = 10;
  → 聚簇索引的叶子节点就有完整数据 → 1次IO

MyISAM: SELECT * FROM users WHERE id = 10;
  → 主键索引的叶子节点拿到行号 (假设行号=5)
  → 去数据文件读第5行 → 2次IO
```

**但是**，InnoDB的二级索引查询，可能比MyISAM慢：

```
InnoDB: SELECT * FROM users WHERE name = 'Tom';
  → 1. 二级索引找到主键值 (id=10)
  → 2. 回表，去聚簇索引查 → 2次IO

MyISAM: SELECT * FROM users WHERE name = 'Tom';
  → 1. 二级索引找到行号 (行号=5)
  → 2. 去数据文件读第5行 → 2次IO

看起来一样？但是：
- InnoDB的回表是"随机IO"（主键值不一定连续）
- MyISAM的行号是顺序的 → 如果按插入顺序查询，可能是顺序IO
```

**但是但是**，如果InnoDB用**覆盖索引**，就不需要回表 → 比MyISAM快！

```sql
-- InnoDB覆盖索引（不需要回表）
CREATE INDEX idx_name_age ON users(name, age);
SELECT name, age FROM users WHERE name = 'Tom';
→ 索引中就有name和age → 1次IO → 比MyISAM快！
```

### 聚簇索引 vs 堆表（Heap-organized Table）

```
                    聚簇索引 (InnoDB)          堆表 (Oracle/SQL Server默认)
─────────────────────────────────────────────────────────────
数据存储            按主键顺序存储             按插入顺序存储 (堆)
主键索引            叶子节点存数据             索引叶子节点存行地址
二级索引            叶子节点存主键             叶子节点存行地址
范围查询            快 (数据物理连续)          慢 (数据可能分散)
```

**Oracle/SQL Server的堆表**：

```
堆表：数据不按任何键排序，就按插入顺序堆在那里。

索引（不管是主键还是二级索引）的叶子节点，都存一个"行地址"（指向数据在堆里的位置）。

问题：
- 范围查询：id BETWEEN 10 AND 20
  → 这11行数据在堆表里可能完全不连续 → 随机IO → 慢！

- 数据移动：如果因为UPDATE导致行变大，原位置放不下 → 行移动到新位置
  → 所有索引都要更新行地址！（和MyISAM的问题一样）
```

**为什么InnoDB选择聚簇索引，而不是堆表？**

- OLTP场景，大多按主键查询 → 聚簇索引的主键查询极快
- 范围查询在OLTP中也常见（例如：查询某个用户的所有订单）→ 聚簇索引的范围查询极快
- 缺点是：二级索引需要回表 → 但是可以用覆盖索引优化

## 正确使用方式

### 1. 主键设计：用自增INT/BIGINT

```
✗ 错误：用UUID做主键
CREATE TABLE users (
  id CHAR(36) PRIMARY KEY DEFAULT (UUID()),
  name VARCHAR(50)
);

问题：
- UUID是无序的 → 插入时，B+树需要频繁页分裂
- 主键索引占用空间大 (CHAR(36) vs BIGINT(8字节))
- 二级索引也变大 (因为二级索引的叶子节点存主键值)

✓ 正确：用自增BIGINT
CREATE TABLE users (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  uuid CHAR(36) UNIQUE,  -- UUID只做业务标识，不做主键
  name VARCHAR(50)
);

好处：
- 插入时，新的主键值永远是最大的 → 追加到B+树末尾 → 几乎不发生页分裂
- 主键索引紧凑 → 每个节点能存更多键值 → 树更矮
- 二级索引也更小 → 每个节点能存更多主键值
```

### 2. 利用覆盖索引，避免回表

```sql
-- ✗ 慢：需要回表
CREATE INDEX idx_name ON users(name);
SELECT * FROM users WHERE name = 'Tom';
-- 1. idx_name找到主键值
-- 2. 回表，去聚簇索引查完整数据
-- → 2次IO

-- ✓ 快：覆盖索引
CREATE INDEX idx_name_age ON users(name, age);
SELECT name, age FROM users WHERE name = 'Tom';
-- idx_name_age中就有name和age → 不需要回表 → 1次IO

-- 检查是否用了覆盖索引
EXPLAIN SELECT name, age FROM users WHERE name = 'Tom';
-- Extra: Using index (覆盖索引！)
```

### 3. 避免SELECT *，减少回表开销

```sql
-- ✗ 慢：SELECT * 几乎不可能用覆盖索引
SELECT * FROM users WHERE name = 'Tom';
-- 即使有idx_name，也需要回表（因为*需要所有列）

-- ✓ 快：只查需要的列，尽量用覆盖索引
SELECT id, name FROM users WHERE name = 'Tom';
-- 如果idx_name的叶子节点包含id和name → 覆盖索引！
-- （实际上，InnoDB的二级索引叶子节点本来就包含主键值）
-- → 不需要回表 → 快！
```

### 4. 顺序主键 vs 随机主键（性能对比）

```sql
-- 顺序主键（自增INT）
CREATE TABLE orders_seq (
  id INT PRIMARY KEY AUTO_INCREMENT,
  user_id INT,
  amount DECIMAL(10,2)
);
-- 插入性能：高（追加写，不发生页分裂）

-- 随机主键（UUID）
CREATE TABLE orders_random (
  id CHAR(36) PRIMARY KEY DEFAULT (UUID()),
  user_id INT,
  amount DECIMAL(10,2)
);
-- 插入性能：低（随机插入，频繁页分裂）

-- 测试：插入1万行
-- 顺序主键：约1秒
-- 随机主键：约10秒（页分裂开销）
```

## 边界情况和坑

### 坑1：主键太大，导致二级索引膨胀

```
场景：用很长的字符串做主键
CREATE TABLE users (
  email VARCHAR(255) PRIMARY KEY,  -- 主键是email
  name VARCHAR(50)
);
CREATE INDEX idx_name ON users(name);

问题：
- 二级索引idx_name的叶子节点，要存主键的值（email）
- email最长255字符 → 每个二级索引叶子节点都很大 → 索引变大
- 内存能缓存的索引节点变少 → 缓存命中率下降 → 性能变差

解决：
CREATE TABLE users (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,  -- 主键用BIGINT
  email VARCHAR(255) UNIQUE,
  name VARCHAR(50)
);
-- 二级索引的叶子节点只存8字节的主键值 → 紧凑！
```

### 坑2：页分裂导致碎片

```
场景：用UUID做主键，插入大量数据

问题：
- UUID无序 → 插入位置随机 → 频繁页分裂
- 页分裂后，页的利用率只有50% → 碎片！
- 查询时需要读更多的页 → 性能下降

排查：
SHOW TABLE STATUS FROM your_db LIKE 'users';
-- Data_length: 数据文件大小
-- Index_length: 索引文件大小
-- Data_free: 碎片空间

解决：
1. 用自增主键（避免页分裂）
2. 定期执行OPTIMIZE TABLE（会锁表，低峰期做）
   OPTIMIZE TABLE users;
3. 或者用ALTER TABLE重建表
   ALTER TABLE users ENGINE=InnoDB;
```

### 坑3：回表导致的随机IO

```
场景：二级索引返回大量行 → 回表变成随机IO

示例：
CREATE INDEX idx_age ON users(age);
SELECT * FROM users WHERE age > 20;
-- 假设age>20的有50万行

执行过程：
1. 通过idx_age找到50万行的主键值
2. 根据这50万个主键值，去聚簇索引回表
3. 问题是：这50万个主键值可能是无序的 → 回表是随机IO！
   → 性能极差（可能比全表扫描还慢）

优化方案：
1. 如果只需要部分列 → 用覆盖索引
   CREATE INDEX idx_age_name ON users(age, name);
   SELECT name FROM users WHERE age > 20;  -- 覆盖索引，不回表

2. 如果需要所有列，但LIMIT行数少 → 先查索引，再回表
   SELECT * FROM users WHERE age > 20 LIMIT 10;
   -- 优化器可能选择先查idx_age，找到10行，再回表10次（可以接受）

3. 如果需要所有列，且返回行数多 → 考虑全表扫描
   -- 全表扫描是顺序IO，可能比大量随机回表更快
```

### 坑4：主键值改变，导致行移动

```
场景：UPDATE主键（永远不要这样做！）
UPDATE users SET id = 100 WHERE id = 10;

问题：
- 聚簇索引是按主键排序的
- 主键值改变 → 行需要从原来的页，移动到新的页
- 所有二级索引都要更新（因为二级索引的叶子节点存主键值）
- 开销极大！

规则：
- 主键永远不要UPDATE！
- 如果必须改主键 → 删除旧行，插入新行（但这不是好方案）
```
