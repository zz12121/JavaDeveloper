# 线程基础

> 线程是 Java 并发编程的起点，也是理解所有并发问题的根基。很多"高级"并发问题（可见性、原子性、指令重排）的根源都在线程的创建、调度和生命周期管理上。把线程基础想透，后面的路会平坦很多。

---

## 这个问题为什么存在？

### 为什么需要多线程？

单线程程序一次只能做一件事。但在真实世界中，程序经常需要同时处理多个任务：

- **IO 等待**：等待数据库查询返回时，CPU 是空闲的——这段时间可以被利用
- **多核利用**：现代 CPU 有多个核心，单线程只能用一个核心
- **响应性**：GUI 程序中，UI 线程被阻塞 = 界面卡死
- **业务需求**：同时服务多个用户请求、并行处理批量数据

### 为什么线程这么难？

线程不难在"创建"，难在"协同"。多个线程共享同一块内存，当它们同时读写同一个变量时，**结果不可预测**：

```
int count = 0;

// 线程A 和 线程B 同时执行 count++
// count++ 不是原子操作，它分为三步：
//   1. 读取 count 的值
//   2. 加 1
//   3. 写回 count

线程A: 读(0) → 加1 → 写(1)
线程B:          读(0) → 加1 → 写(1)  // 丢失了线程A的更新

// 两次 ++ 后，count 应该是 2，实际是 1
// 这就是经典的「竞态条件」（Race Condition）
```

**并发编程的三大核心问题**：
1. **原子性**：一个操作是否不可被中断？
2. **可见性**：一个线程的修改，另一个线程能否立即看到？
3. **有序性**：指令的执行顺序是否和代码顺序一致？

这三个问题贯穿整个并发编程——synchronized、volatile、Lock、原子类、CAS……所有的并发工具本质上都在解决这三个问题。

## 它是怎么解决问题的？

### 线程的创建方式

Java 中创建线程有 4 种方式，核心都是围绕 `Thread` 类和 `Runnable` 接口：

```java
// 1. 继承 Thread（不推荐——Java 单继承，浪费了继承位）
class MyThread extends Thread {
    @Override
    public void run() { /* 任务逻辑 */ }
}
new MyThread().start();

// 2. 实现 Runnable（推荐——解耦任务和线程）
Runnable task = () -> { /* 任务逻辑 */ };
new Thread(task).start();

// 3. 实现 Callable + Future（需要返回值时）
Callable<Integer> task = () -> { return 42; };
FutureTask<Integer> future = new FutureTask<>(task);
new Thread(future).start();
Integer result = future.get();  // 阻塞等待结果

// 4. 线程池（生产环境必须用这种方式）
ExecutorService pool = Executors.newFixedThreadPool(4);
pool.submit(() -> { /* 任务逻辑 */ });
pool.shutdown();
```

```
为什么生产环境必须用线程池？
1. 线程创建和销毁有成本（OS 级别资源分配）
2. 无限创建线程 → OOM（每线程约 1MB 栈空间）
3. 线程池复用线程、控制并发度、管理任务队列
4. Executors 的线程池实现：ThreadPoolExecutor（后面线程池章节详细讲）
```

### 线程的生命周期

```
         start()
 NEW ──────────→ RUNNABLE ←────────┐
                   │      │        │
              调度获得CPU  等待锁    │
              时间片用完  /IO/      │  wait()结束
              yield()   sleep()    │  notify/notifyAll
                   │      │        │  /锁释放
                   │      ↓        │
                   │   BLOCKED ────┘
                   │      │
                   │   WAITING
                   │   (Object.wait()
                   │    Thread.join()
                   │    LockSupport.park())
                   │      │
                   │   TIMED_WAITING
                   │   (Thread.sleep(n)
                   │    Object.wait(n)
                   │    LockSupport.parkNanos(n))
                   │      │
                   ↓      ↓
                TERMINATED
                (run()结束 / 异常退出)
```

**六种状态详解**：

| 状态 | 说明 | 触发条件 |
|------|------|---------|
| NEW | 已创建，未启动 | `new Thread()` |
| RUNNABLE | 可运行（含就绪和运行中） | `start()` |
| BLOCKED | 等待获取监视器锁 | 等待进入 synchronized 块 |
| WAITING | 无限期等待 | `wait()`、`join()`、`park()` |
| TIMED_WAITING | 有时限等待 | `sleep(n)`、`wait(n)`、`parkNanos(n)` |
| TERMINATED | 执行完毕 | `run()` 正常结束或抛异常 |

```
常见误区：
- RUNNABLE 不等于"正在运行"——线程可能在等待 CPU 调度（就绪态）
- BLOCKED 只针对 synchronized——ReentrantLock 的 await() 状态是 WAITING
- Thread.sleep() 不会释放锁，Object.wait() 会释放锁
```

### 线程的核心方法

#### start() vs run()

```java
Thread t = new Thread(() -> System.out.println("hello"));

t.run();    // 在当前线程中执行，不会创建新线程
t.start();  // 创建新线程，在新线程中执行 run()
```

**这是最基础的面试题**：`run()` 是普通方法调用，在当前线程执行；`start()` 会创建新的线程并调用 `run()`。

#### sleep() vs wait()

| 维度 | sleep() | wait() |
|------|---------|--------|
| 所属类 | Thread | Object |
| 释放锁 | 不释放 | 释放 |
| 使用位置 | 任何地方 | 必须在 synchronized 块内 |
| 唤醒方式 | 超时自动醒 | notify/notifyAll 或超时 |
| 用途 | 暂停当前线程 | 线程间通信/协作 |

```java
// sleep：不释放锁
synchronized (lock) {
    Thread.sleep(1000);  // 其他线程仍然无法获取 lock
}

// wait：释放锁
synchronized (lock) {
    lock.wait();  // 释放 lock，其他线程可以获取
    // 被唤醒后重新竞争锁
}
```

#### join()

```java
Thread t = new Thread(() -> { /* 耗时任务 */ });
t.start();
t.join();  // 当前线程阻塞，等待 t 执行完毕

// join(timeout)：最多等待指定时间
t.join(3000);  // 最多等 3 秒
```

**join() 底层就是 wait()**——调用方线程在 t 对象上 wait()，t 执行完毕后 JVM 会自动调用 t.notifyAll()。

#### yield()

```java
Thread.yield();  // 提示调度器"我可以让出 CPU"
```

**只是一个"提示"，不保证生效。** 调度器完全可以忽略 yield。实际开发中几乎不用。

#### interrupt()

```java
Thread t = new Thread(() -> {
    while (!Thread.currentThread().isInterrupted()) {
        // 检查中断标志
    }
});
t.start();
t.interrupt();  // 设置中断标志为 true
```

```
中断机制的核心设计：
1. interrupt() 只是设置中断标志，不会强制停止线程
2. 线程需要自行检查中断标志并决定如何响应
3. 如果线程在 sleep/wait/join 中被中断，会抛出 InterruptedException
   并清除中断标志

错误做法：
- Thread.stop() — 已废弃，不安全（可能释放锁时对象状态不一致）
- 直接操作线程的标志位 — 与 sleep/wait/join 配合有问题

正确做法：
- 使用 interrupt() + InterruptedException + isInterrupted()
```

**中断的三种响应方式**：

```java
// 方式1：检查中断标志
while (!Thread.currentThread().isInterrupted()) {
    // 做工作
}

// 方式2：捕获 InterruptedException，退出循环
try {
    while (true) {
        Thread.sleep(100);
        // 做工作
    }
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();  // 恢复中断标志
    // 清理资源，退出
}

// 方式3：双重检查（同时处理中断标志和阻塞中断）
while (!Thread.currentThread().isInterrupted()) {
    try {
        Thread.sleep(100);
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
        break;
    }
}
```

### 守护线程 vs 用户线程

```java
Thread daemon = new Thread(() -> {
    while (true) { /* 后台任务 */ }
});
daemon.setDaemon(true);  // 设为守护线程
daemon.start();
```

```
守护线程（Daemon Thread）：
- 当所有用户线程结束时，JVM 会自动退出，不管守护线程是否执行完毕
- 典型应用：GC 线程、日志清理线程

用户线程（User Thread）：
- 默认创建的就是用户线程
- JVM 等待所有用户线程结束后才退出

规则：如果 JVM 中只剩下守护线程，JVM 立即退出。
```

### 线程优先级

```java
Thread t = new Thread(task);
t.setPriority(Thread.MAX_PRIORITY);   // 10
t.setPriority(Thread.MIN_PRIORITY);   // 1
t.setPriority(Thread.NORM_PRIORITY);  // 5（默认）
```

```
重要事实：
1. Java 线程优先级有 10 个级别（1~10），但只映射到 OS 的优先级
2. 不同 OS 的优先级调度策略不同——Linux 基本忽略 Java 线程优先级
3. 优先级高的线程只是「更可能」被调度，不保证先执行
4. 不要依赖线程优先级来保证执行顺序
```

### 线程组（ThreadGroup）

```
ThreadGroup 是 Java 早期的线程管理机制，现在基本不推荐使用。
问题：
1. 线程组的 stop()/suspend() 有安全问题（和 Thread.stop 同理）
2. 现代并发编程用 ExecutorService 管理线程，不需要 ThreadGroup
3. 官方文档明确表示 ThreadGroup 已过时

了解即可，面试中不需要深入。
```

## 和相似方案的区别

### 线程 vs 进程

| 维度 | 进程 | 线程 |
|------|------|------|
| 资源 | 独立的内存空间 | 共享进程的内存 |
| 创建成本 | 高（OS 分配地址空间） | 低（共享进程资源） |
| 通信 | IPC（管道、共享内存、Socket） | 直接读写共享变量 |
| 切换成本 | 高（需要切换地址空间） | 低（只需切换寄存器和栈） |
| 安全性 | 高（内存隔离） | 低（需要同步机制） |
| 崩溃影响 | 一个进程崩溃不影响其他 | 一个线程崩溃可能导致整个进程崩溃 |

### Java Thread vs Virtual Thread

| 维度 | 平台线程 | 虚拟线程（Java 21+） |
|------|---------|---------------------|
| 映射 | 1:1 映射到 OS 线程 | M:N 映射到少量载体线程 |
| 创建成本 | 高 | 极低 |
| 栈大小 | 固定 ~1MB | 动态增长，几百字节起步 |
| 数量上限 | 几千 | 百万级 |
| 阻塞行为 | OS 线程阻塞 | 自动卸载，释放载体线程 |
| 使用方式 | 通常通过线程池 | 通常不用池，用完即弃 |

### Runnable vs Callable

| 维度 | Runnable | Callable<V> |
|------|----------|-------------|
| 返回值 | 无 | 有（V） |
| 异常 | 不能抛 checked exception | 可以抛 checked exception |
| 使用方式 | `new Thread(r).start()` | 配合 FutureTask / 线程池 |
| Java 版本 | Java 1.0 | Java 5 |

## 正确使用方式

### 1. 不要用 Executors 的快捷方法创建线程池

```java
// ✗ 不推荐
Executors.newFixedThreadPool(10);       // 队列无界 → OOM
Executors.newCachedThreadPool();        // 最大 Integer.MAX_VALUE → OOM
Executors.newSingleThreadExecutor();    // 队列无界 → OOM

// ✓ 推荐：手动创建 ThreadPoolExecutor，明确所有参数
new ThreadPoolExecutor(
    2,                      // corePoolSize
    4,                      // maximumPoolSize
    60L, TimeUnit.SECONDS,  // keepAliveTime
    new LinkedBlockingQueue<>(1000),  // 有界队列
    new ThreadFactoryBuilder().setNameFormat("biz-pool-%d").build(),
    new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略
);
```

### 2. 正确的中断处理模式

```java
// 正确模板
public void run() {
    try {
        while (!Thread.currentThread().isInterrupted()) {
            doWork();
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();  // 恢复中断标志
    } finally {
        cleanup();  // 释放资源
    }
}
```

### 3. 线程命名

```java
// ✗ 线程名是 Thread-1、Thread-2……出问题时无法定位
new Thread(task).start();

// ✓ 给线程起有意义的名字
new Thread(task, "order-processing-" + orderId).start();

// 用 ThreadFactory 批量命名
ThreadFactory factory = r -> new Thread(r, "biz-worker-" + counter.getAndIncrement());
```

### 4. 异常处理

```java
// 线程中抛出的未捕获异常，主线程不会感知到
// 需要设置 UncaughtExceptionHandler
Thread t = new Thread(() -> {
    throw new RuntimeException("boom");
});
t.setUncaughtExceptionHandler((thread, ex) -> {
    logger.error("线程 {} 异常", thread.getName(), ex);
});
t.start();

// 线程池中设置全局 Handler
ExecutorService pool = Executors.newFixedThreadPool(4);
((ThreadPoolExecutor) pool).setUncaughtExceptionHandler(
    (t, e) -> logger.error("线程池异常", e)
);
```

## 边界情况和坑

### 1. start() 只能调用一次

```java
Thread t = new Thread(task);
t.start();  // ✓
t.start();  // IllegalThreadStateException！
```

Thread 对象一旦进入 TERMINATED 状态，就不能再 start()。**不要复用 Thread 对象**——这也是线程池存在的理由之一。

### 2. ThreadLocal 导致的内存泄漏

```java
// 线程池中 ThreadLocal 的经典坑
ExecutorService pool = Executors.newFixedThreadPool(4);

pool.submit(() -> {
    threadLocal.set(largeObject);
    try {
        doWork();
    } finally {
        // ✗ 忘记清理！线程被回收到池中，threadLocal 仍然持有 largeObject
        // ✓ 必须清理
        threadLocal.remove();
    }
});
```

线程池中的线程是复用的，ThreadLocal 的值不会自动清除。**每次使用后必须 remove()**，否则内存泄漏。

### 3. sleep(0) 的作用

```java
Thread.sleep(0);  // 让出当前时间片，重新参与 CPU 竞争
```

和 `yield()` 类似，只是一个"建议"。实际开发中几乎不会用。

### 4. 线程的生命周期状态不准确

```java
Thread.State state = t.getState();  // 获取快照
// 注意：getState() 返回的是调用瞬间的状态
// 线程可能在调用后立即切换状态
// 不要基于 getState() 做关键决策
```

### 5. join() 的意外行为

```java
Thread t = new Thread(() -> { /* 任务 */ });
t.start();
t.join(5000);  // 等待 5 秒

// 如果 t 在 5 秒内结束，join 立即返回
// 如果 5 秒后 t 还没结束，join 也会返回——不会抛异常
// 需要检查 t.isAlive() 判断是否超时
if (t.isAlive()) {
    // 超时了，线程还在运行
}
```

### 6. 守护线程中创建的线程也是守护线程

```java
Thread daemon = new Thread(() -> {
    Thread child = new Thread(() -> { /* 子线程 */ });
    // child 也是守护线程（继承创建者的守护状态）
    child.start();
    // 如果主线程和 daemon 都结束了，child 也会被强制终止
});
daemon.setDaemon(true);
```

**守护线程中不要做 IO 操作或资源清理**——可能中途被终止导致资源不一致。

### 7. 未捕获异常导致线程静默死亡

```java
// 线程中抛出的未捕获异常
new Thread(() -> {
    throw new RuntimeException("boom");  // 线程直接终止，没有任何输出
}).start();
// 主线程完全感知不到异常！
```

不设置 UncaughtExceptionHandler 的话，异常会被打印到 System.err，但调用方完全不知情。**线程池中的任务异常如果不处理，会被静默吞掉**。

## 我的理解

线程基础是并发编程的地基。很多开发者的并发问题其实源于对线程基础的不理解：

- **竞态条件** → 不理解线程共享内存和指令的非原子性
- **死锁** → 不理解线程的锁获取顺序和 BLOCKED 状态
- **内存泄漏** → 不理解线程池中线程的复用和 ThreadLocal 的生命周期
- **任务丢失** → 不理解线程的异常处理和 UncaughtExceptionHandler

**面试中线程基础的高频考点**：

1. **线程状态转换图**（几乎必问，必须能画出来）
2. **start() vs run()**（经典基础题）
3. **sleep() vs wait()**（高频，考察锁的释放）
4. **中断机制**（interrupt/InterruptedException/isInterrupted 三者的关系）
5. **守护线程 vs 用户线程**（JVM 退出条件）
6. **Runnable vs Callable**（返回值和异常的区别）
7. **线程创建方式**（4 种，以及为什么推荐线程池）
