# 并发工具类

> CountDownLatch、CyclicBarrier、Semaphore、Phaser——JUC 提供的四大并发工具类，是 AQS 之上最经典的应用层封装。它们各自解决一个具体的并发协作问题，理解它们的本质区别，远比记住 API 重要。

---

## 这个问题为什么存在？

### 并发编程的核心难题：线程间如何"协作"

我们聊了很多「互斥」——synchronized、Lock、CAS，解决的是"同一时刻只能一个线程操作"。

但实际场景中，线程之间更多需要的是**协作**：

```
问题 1：主线程等 3 个子任务全部完成，才能汇总结果
  → 怎么等？Thread.join()？那子任务又有子任务呢？嵌套 join？

问题 2：5 个线程各自准备数据，全部就绪后同时出发
  → 怎么同步"同时出发"这个动作？

问题 3：数据库连接池最多 20 个连接，超过就排队
  → 怎么控制并发访问的数量？

问题 4：10 个阶段的多线程任务，每个阶段需要所有线程都完成才能进入下一阶段
  → 怎么动态管理"阶段"？
```

```
手动实现这些协作逻辑的痛点：

1. 用 wait/notify？容易漏notify、容易虚假唤醒、代码复杂
2. 用 volatile 标志位？轮询等待浪费 CPU
3. 用 Thread.join()？不支持复杂的同步模式
4. 自己写？ bug 满天飞，边界情况处理不完

JUC 并发工具类 = 把常见的线程协作模式封装成高质量、经过充分测试的组件
```

### 工具类全景图

```
JUC 并发工具类
├── CountDownLatch    — 一次性倒计时门闩（不可重置）
├── CyclicBarrier     — 循环栅栏（可重用，支持回调）
├── Semaphore         — 信号量（控制并发数）
└── Phaser            — 分阶段执行器（动态注册/注销）
```

**选型速查：**

| 场景 | 工具 | 关键词 |
|------|------|--------|
| 等待 N 个任务完成 | CountDownLatch | 一次性、不可重置 |
| 多线程到齐后一起行动 | CyclicBarrier | 可重用、barrierAction |
| 限制同时访问的数量 | Semaphore | 许可证、限流 |
| 多阶段并行任务 | Phaser | 动态注册、arrive/await |

### CountDownLatch：倒计时门闩

#### 核心思想

```
想象一个起跑线的倒计时：3、2、1、GO！

- count 初始化为一个正整数（比如 3）
- 每个线程完成任务后调用 countDown()，count 减 1
- 等待的线程调用 await()，阻塞直到 count 变为 0
- count 到 0 后不能重置（一次性）
```

#### 基本用法

```java
// 主线程等待 3 个子任务全部完成
CountDownLatch latch = new CountDownLatch(3);

for (int i = 0; i < 3; i++) {
    new Thread(() -> {
        try {
            // 执行任务...
            Thread.sleep(1000);
        } finally {
            latch.countDown();  // 任务完成，count - 1
        }
    }).start();
}

latch.await();  // 主线程阻塞，直到 count == 0
System.out.println("所有任务完成！");
```

#### 内部实现（基于 AQS）

```
CountDownLatch 内部用 AQS 的共享模式：

1. new CountDownLatch(3)
   → state = 3

2. countDown()
   → state = state - 1（CAS）
   → 当 state == 0 时，唤醒所有等待线程

3. await()
   → 如果 state > 0，当前线程入 AQS 等待队列并阻塞
   → 如果 state == 0，直接返回

关键点：用 AQS 的 state 做计数器，countDown 就是 releaseShared
```

#### 典型应用场景

```java
// 场景 1：并发测试——等待所有线程就绪后同时开始
CountDownLatch readyLatch = new CountDownLatch(N);
CountDownLatch startLatch = new CountDownLatch(1);
CountDownLatch doneLatch = new CountDownLatch(N);

for (int i = 0; i < N; i++) {
    new Thread(() -> {
        readyLatch.countDown();   // 报告：我准备好了
        startLatch.await();        // 等待起跑信号
        doTest();
        doneLatch.countDown();     // 报告：我做完了
    }).start();
}

readyLatch.await();   // 等所有人就绪
startLatch.countDown(); // 发令枪！
doneLatch.await();     // 等所有人完成
```

```java
// 场景 2：微服务启动检查——等待所有依赖服务可用
CountDownLatch healthCheck = new CountDownLatch(3);

checkService("mysql", healthCheck);
checkService("redis", healthCheck);
checkService("kafka", healthCheck);

healthCheck.await(10, TimeUnit.SECONDS);  // 最多等 10 秒
if (healthCheck.getCount() > 0) {
    throw new StartupException("部分依赖服务未就绪");
}
```

#### 常见陷阱

```
陷阱 1：countDown 不在 finally 中
  → 任务抛异常后 countDown 没执行，await 永远阻塞
  → 解决：必须在 finally 中 countDown

陷阱 2：忘记调用 countDown
  → 编译不报错，运行时死锁，极难排查
  → 解决：代码审查时重点关注

陷阱 3：count 初始化为 0
  → await() 直接返回， latch 完全不生效
  → 这是逻辑 bug，不是 API 问题

陷阱 4：重复调用 await()
  → count == 0 后，多次 await() 都直接返回，这是正确行为
  → 但如果 count > 0 就被重复 await，会多线程阻塞（这没问题）
```

### CyclicBarrier：循环栅栏

#### 核心思想

```
想象一队人去旅行：
- 每个人走自己的路（并行执行）
- 到达集合点后等其他人（await）
- 人齐了一起出发（barrierAction 回调）
- 去下一个景点，重复这个过程（可循环）

和 CountDownLatch 的关键区别：
- CountDownLatch：计数递减到 0 后结束，不可重用
- CyclicBarrier：计数达到 parties 后重置，可循环使用
```

#### 基本用法

```java
// 3 个线程到齐后同时执行，可重复 2 轮
CyclicBarrier barrier = new CyclicBarrier(3, () -> {
    System.out.println("=== 所有线程到齐，开始本轮任务 ===");
});

for (int i = 0; i < 3; i++) {
    final int threadId = i;
    new Thread(() -> {
        for (int round = 1; round <= 2; round++) {
            System.out.println("线程" + threadId + " 第" + round + "轮准备中...");
            try {
                barrier.await();  // 等待其他线程到齐
                System.out.println("线程" + threadId + " 第" + round + "轮执行中");
            } catch (Exception e) {
                e.printStackTrace();
            }
        }
    }).start();
}
```

```
输出：
线程0 第1轮准备中...
线程1 第1轮准备中...
线程2 第1轮准备中...
=== 所有线程到齐，开始本轮任务 ===
线程2 第1轮执行中
线程0 第1轮执行中
线程1 第1轮执行中
...（第 2 轮同理）
```

#### 内部实现

```
CyclicBarrier 不基于 AQS，而是用 ReentrantLock + Condition：

内部结构：
- parties：总参与线程数
- count：当前还在等待的线程数
- lock：ReentrantLock（保护 count 的修改）
- condition：Condition（让等待的线程阻塞）
- generation：Generation 对象（标记当前轮次，用于支持 reset）
- barrierCommand：可选的回调（所有线程到齐后执行）

await() 流程：
1. 获取 lock
2. count--
3. 如果 count == 0：
   a. 执行 barrierCommand（如果有）
   b. generation.breaked = false
   c. 重置 count = parties
   d. 创建新 generation
   e. condition.signalAll()（唤醒所有等待线程）
4. 如果 count > 0：
   a. condition.await()（阻塞当前线程）

reset() 流程：
1. 中断当前等待的所有线程（BrokenBarrierException）
2. 重置 count = parties
3. 创建新 generation
```

#### BrokenBarrierException

```
当某个线程在 await 时被中断或超时：
→ barrier 进入 broken 状态
→ 其他所有 await 的线程抛出 BrokenBarrierException
→ 这是一个"传染性"的设计：一个人出问题，所有人都要知道

为什么这样设计？
因为 CyclicBarrier 的语义是"大家一起行动"
如果一个人掉队了，继续让其他人一起行动可能没意义
```

#### CountDownLatch vs CyclicBarrier 对比

```
                    CountDownLatch          CyclicBarrier
计数方向            递减到 0                递减到 0 后重置
可重用              ❌ 一次性                ✅ 循环使用
等待方向            1 个线程等 N 个          N 个线程互相等
回调                无                      有（barrierCommand）
线程角色            有"主线程"和"工作线程"    所有线程对等
重置                不支持                  reset()
底层实现            AQS 共享模式             ReentrantLock + Condition
典型场景            等待所有子任务完成        多线程分阶段协作
```

**记忆口诀**：Latch 是门闩（打开就没了），Barrier 是栅栏（到了就继续，下一轮还能用）。

### Semaphore：信号量

#### 核心思想

```
想象一个停车场：
- 有 5 个车位（permits = 5）
- 来一辆车，拿一个 permit（acquire）
- 走一辆车，还一个 permit（release）
- 车位满了，后来的车排队等

信号量本质上就是一个"许可证池"——控制同时访问共享资源的线程数量
```

#### 基本用法

```java
// 数据库连接池模拟：最多 3 个并发连接
Semaphore semaphore = new Semaphore(3);

for (int i = 0; i < 10; i++) {
    new Thread(() -> {
        try {
            semaphore.acquire();     // 获取许可（没有就排队）
            System.out.println(Thread.currentThread().getName() + " 获取连接");
            Thread.sleep(2000);      // 模拟使用连接
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        } finally {
            semaphore.release();     // 释放许可
        }
    }).start();
}
```

#### 公平 vs 非公平

```java
Semaphore fair = new Semaphore(3, true);   // 公平：按等待时间排队
Semaphore unfair = new Semaphore(3, false); // 非公平：允许插队（默认）
```

```
公平模式：
  acquire() 时如果许可不够，进入 FIFO 队列排队
  → 线程获取许可的顺序 = 它们请求的顺序
  → 不会饥饿，但吞吐量较低

非公平模式：
  acquire() 时直接尝试 CAS 抢许可
  → 刚 release 的许可可能被新来的线程抢走
  → 吞吐量高，但可能导致饥饿

和 ReentrantLock 一样的公平策略，底层同样是 AQS
```

#### 内部实现（基于 AQS）

```
Semaphore 内部用 AQS 的共享模式：

1. new Semaphore(3)
   → state = 3（许可数量存在 AQS 的 state 中）

2. acquire()
   → 尝试 CAS：state = state - 1
   → 如果 state < 0，进入 AQS 等待队列阻塞
   → 注意：state 可以是负数，表示等待中的线程数

3. release()
   → state = state + 1（CAS）
   → 如果 state <= 0（之前有人在等），唤醒等待线程

注意：state 的值可以超过 permits（release 多次）
但这不代表许可数量增加，只是 release 不会校验上限
```

#### 实际应用场景

```java
// 场景 1：接口限流——同一时刻最多 100 个请求在处理
Semaphore rateLimiter = new Semaphore(100);

public Response handleRequest(Request req) {
    if (!rateLimiter.tryAcquire(500, TimeUnit.MILLISECONDS)) {
        return Response.reject("系统繁忙，请稍后重试");
    }
    try {
        return doHandle(req);
    } finally {
        rateLimiter.release();
    }
}
```

```java
// 场景 2：资源池管理——限制对稀缺资源的并发访问
Semaphore printerSemaphore = new Semaphore(2); // 只有 2 台打印机

public void print(Document doc) {
    printerSemaphore.acquire();
    try {
        usePrinter(doc);
    } finally {
        printerSemaphore.release();
    }
}
```

#### 常见陷阱

```
陷阱 1：acquire 后不 release
  → 许可逐渐耗尽， semaphore 变成"死锁"
  → 解决：务必在 finally 中 release

陷阱 2：release 的线程和 acquire 的线程不一致
  → 这是允许的！ Semaphore 不要求"谁拿到谁释放"
  → 但逻辑上容易混乱，建议同一资源持有者负责释放

陷阱 3：用 tryAcquire 代替 acquire 但忘记处理失败
  → tryAcquire 返回 false 时，线程应该走降级逻辑
  → 不能直接执行需要许可才能执行的操作

陷阱 4：初始化 permits = 0
  → 所有 acquire 都会阻塞
  → 可以用于"先创建信号量，稍后由其他线程 release 来启动"
  → 但这不是常规用法，容易让人困惑
```

### Phaser：分阶段执行器

#### 核心思想

```
Phaser 是 CountDownLatch + CyclicBarrier 的增强版：
- 支持动态注册/注销参与者（不需要提前知道线程数）
- 支持多阶段执行（每阶段自动推进）
- 支持每个阶段的到达监控（onAdvance 回调）

适用场景：多轮次的并行计算，每轮参与线程数可能变化
```

#### 基本用法

```java
// 3 个学生参加 2 门考试，每门考试必须所有人交卷才能进入下一门
Phaser phaser = new Phaser(3);  // 3 个参与者

for (int i = 0; i < 3; i++) {
    final int studentId = i;
    new Thread(() -> {
        // 第 1 门考试
        System.out.println("学生" + studentId + " 开始语文考试");
        doExam("语文");
        phaser.arriveAndAwaitAdvance();  // 交卷，等其他人

        // 第 2 门考试
        System.out.println("学生" + studentId + " 开始数学考试");
        doExam("数学");
        phaser.arriveAndAwaitAdvance();  // 交卷，等其他人

        phaser.arriveAndDeregister();    // 考完注销
    }).start();
}

phaser.register();  // 主线程也注册，等待所有阶段完成
phaser.arriveAndAwaitAdvance();  // 等待所有阶段结束
```

#### 核心方法

```
arrive()                    — 到达当前阶段，不等待其他人，直接进入下一阶段
arriveAndAwaitAdvance()     — 到达当前阶段，等待其他人（最常用）
arriveAndDeregister()       — 到达当前阶段，注销自己（不再参与后续阶段）
register()                  — 动态注册新参与者
bulkRegister(int)           — 批量注册
awaitAdvance(int phase)     — 等待指定阶段的完成
forceTermination()          — 强制终止（所有 await 抛出异常）
```

#### 进阶用法：onAdvance 回调

```java
Phaser phaser = new Phaser(3) {
    @Override
    protected boolean onAdvance(int phase, int registeredParties) {
        System.out.println("=== 阶段 " + phase + " 完成，参与人数: " + registeredParties + " ===");
        return registeredParties == 0;  // 返回 true 表示 Phaser 终止
        // 返回 false 表示继续下一阶段
    }
};
```

```
onAdvance 的返回值决定了 Phaser 的生命周期：
- 返回 true：Phaser 终止，后续 arrive 立即返回
- 返回 false：Phaser 继续，进入下一阶段

终止条件通常是：registeredParties == 0（所有参与者都注销了）
```

#### Phaser vs CyclicBarrier 对比

```
                    CyclicBarrier           Phaser
参与线程数          固定（构造时指定）        动态（可随时注册/注销）
可重用              ✅                      ✅（且更灵活）
回调                barrierCommand（每轮）    onAdvance（每阶段）
提前退出            不支持                   arriveAndDeregister()
复杂度              简单                    更复杂，API 更多
适用场景            固定线程数的多轮任务      参与者数动态变化的分阶段任务
```

## 和相似方案的区别

### 四大工具类横向对比

```
维度          CountDownLatch    CyclicBarrier     Semaphore        Phaser
─────────────────────────────────────────────────────────────────────────
核心语义      倒计数到0         集合点同步         许可证限流        分阶段同步
计数方向      递减              递减（到0重置）    可增可减          递减（到0推进）
可重用        ❌               ✅                ✅（只要还permit）  ✅
线程角色      不对等（主/从）    对等              不对等（竞争）      对等
回调          无                barrierCommand    无                onAdvance
动态参与者    ❌               ❌                —                 ✅
底层          AQS 共享模式      Lock+Condition    AQS 共享模式      独立实现
```

### 什么时候用哪个？

```
选择决策树：

Q1: 需要限制并发数量吗？
   → 是 → Semaphore

Q2: 需要等 N 个任务完成（一次性）吗？
   → 是 → CountDownLatch

Q3: 需要多轮同步（循环使用）吗？
   → 是，参与线程数固定 → CyclicBarrier
   → 是，参与线程数变化 → Phaser

Q4: 需要"所有人到齐后一起行动"吗？
   → 是 → CyclicBarrier（简单）或 Phaser（复杂）
```

## 正确使用方式

### 最佳实践

```
1. acquire/release、countDown、await 务必配对
   → 用 try-finally 保证 release/countDown 一定执行

2. 指定超时时间
   → await(timeout, unit)
   → tryAcquire(timeout, unit)
   → 永远不要无限期等待，生产环境必须有超时

3. 响应中断
   → await/interrupt 的异常要正确处理
   → Thread.currentThread().interrupt() 恢复中断状态

4. 选择合适的公平策略
   → 大多数场景用非公平（性能好）
   → 公平性有业务要求时用公平
```

### 实战：多服务并行启动检查

```java
public class StartupHealthChecker {

    public static void main(String[] args) throws Exception {
        String[] services = {"MySQL", "Redis", "Kafka", "ElasticSearch"};
        CountDownLatch latch = new CountDownLatch(services.length);
        Semaphore semaphore = new Semaphore(2); // 最多 2 个并发检查

        for (String service : services) {
            new Thread(() -> {
                try {
                    semaphore.acquire();
                    boolean healthy = checkHealth(service);
                    System.out.println(service + ": " + (healthy ? "✓" : "✗"));
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                } finally {
                    semaphore.release();
                    latch.countDown();
                }
            }).start();
        }

        if (!latch.await(15, TimeUnit.SECONDS)) {
            System.err.println("超时！部分服务未响应");
        }
    }
}
```

## 边界情况和坑

### 死锁场景

```java
// 反例：CyclicBarrier 的参与者数不匹配
CyclicBarrier barrier = new CyclicBarrier(3);

// 线程 A 和线程 B 各调 await()
// 第 3 个线程永远不会来 → 所有线程永久阻塞
// 这种 bug 在测试环境可能不容易复现（线程调度顺序不同）
```

### 资源泄漏

```java
// 反例：Semaphore release 后许可数超过初始值
Semaphore sem = new Semaphore(3);
sem.release();  // 没有对应的 acquire
sem.release();  // 现在有 5 个许可了
// 如果代码中某人 release 两次但只 acquire 一次
// 许可数量会膨胀，限流失效
```

### BrokenBarrierException 的连锁反应

```
CyclicBarrier 中一个线程异常退出：
→ barrier 被 broken
→ 其他所有等待的线程收到 BrokenBarrierException
→ 如果不处理，整个协作流程崩溃

正确处理方式：
try {
    barrier.await();
} catch (BrokenBarrierException e) {
    // 1. 记录日志
    // 2. 通知监控系统
    // 3. 执行降级逻辑或重试
}
```

### CountDownLatch 的 count 被减到负数

```java
CountDownLatch latch = new CountDownLatch(3);
latch.countDown();
latch.countDown();
latch.countDown();
latch.countDown();  // count = -1，不会报错！
// getCount() 返回 -1
// 但 await() 在 count <= 0 时都直接返回
// 所以多减不会影响正确性，但说明逻辑有 bug
```

---
