## 1. CAP 理论

**Q: CAP 理论是什么？为什么说"三选二"是伪命题？**

**A:**
CAP 理论（Brewer's Theorem）指出分布式系统无法同时满足：
```
C（Consistency）：一致性 — 所有节点在同一时刻看到相同的数据
A（Availability）：可用性 — 每个请求都能在有限时间内得到响应
P（Partition Tolerance）：分区容错性 — 系统在网络分区时仍能运行
```

**为什么"三选二"是伪命题**：
- 网络分区（网络故障）不可避免 —— 分布式系统必须容忍 P
- 实际上是在 **C 和 A 之间做权衡**：
  - CP 系统：优先保证一致性（如 Zookeeper、HBase）
  - AP 系统：优先保证可用性（如 Cassandra、Eureka）

**不是"三选二"**：
```
      满足 P
    ↙       ↘
  CA         CP
 /             \
All            网络分区时
 Systems        选 C 放弃 A
（单点系统）     或选 A 放弃 C
```

**Tags:** #分布式 #CAP #一致性 #可用性

---

**Q: 什么场景选 CP？什么场景选 AP？如何判断？**

**A:**
**CP 场景（强一致优先）**：
- 金融交易（银行转账、支付）
- 库存扣减（超卖问题）
- 分布式锁
- 配置管理
- 选型：Zookeeper、etcd、HBase

**AP 场景（高可用优先）**：
- 社交feed、评论系统
- 日志收集、监控数据
- DNS 服务
- 选型：Cassandra、DynamoDB、Eureka

**判断标准**：
```
问自己：分区时，业务能容忍什么？

能容忍返回旧数据？ → AP
不能容忍数据不一致？ → CP

例子：
  - 用户下单 → 必须强一致（CP）
  - 点赞数统计 → 可容忍短暂不一致（AP）
```

## 2. BASE 理论

**Q: BASE 理论是什么？和 ACID 是什么关系？**

**A:**
BASE 是对 CAP 中 AP 的一致性妥协方案：
```
BASE = Basically Available + Soft state + Eventually consistent

Basically Available（基本可用）：
  → 分布式系统在故障时，允许暂时降级，但核心功能可用

Soft state（软状态）：
  → 数据状态可以在一段时间内不一致
  → 数据同步有延迟

Eventually consistent（最终一致性）：
  → 数据在一段时间后最终会达到一致状态
  → 不需要实时强一致
```

**与 ACID 的对比**：
| 维度 | ACID（传统数据库） | BASE（分布式系统） |
|------|------------------|-------------------|
| 一致性 | 强一致性 | 最终一致性 |
| 可用性 | 低 | 高 |
| 响应时间 | 等待提交完成 | 立即响应 |
| 典型场景 | 银行、订单 | 社交、缓存、日志 |

**实际应用**：大多数互联网系统采用 BASE + 补偿机制处理分布式事务

**Tags:** #分布式 #BASE #最终一致性

---

**Q: 强一致性、弱一致性、最终一致性的区别？**

**A:**
```
一致性强度（从强到弱）：

强一致性（Linearizability）：
  → 写入后立即可读，所有节点立即同步
  → 分布式锁、事务提交后立即可见

顺序一致性（Sequential Consistency）：
  → 所有进程看到相同的操作顺序
  → 不保证实时，但保证程序顺序

因果一致性（Causal Consistency）：
  → 有因果关系的操作必须按因果顺序
  → 无因果关系的操作可以乱序

最终一致性（Eventual Consistency）：
  → 数据在无更新时，最终会达到一致
  → 存在不一致窗口期（延迟）

单调读一致性（Monotonic Read）：
  → 读到的数据不会比之前更旧

单调写一致性（Monotonic Write）：
  → 写操作按顺序执行
```

**Tags:** #分布式 #一致性模型 #强一致性 #最终一致性

## 3. 分布式一致性协议

**Q: 2PC（两阶段提交）的原理和缺陷？**

**A:**
2PC（Two-Phase Commit）是一种原子提交协议：
```
阶段一：Prepare（投票阶段）
  协调者向所有参与者发送 Prepare 请求
  参与者执行事务（但不提交），然后投票 YES/NO
  如果全部 YES → 进入阶段二

阶段二：Commit（提交阶段）
  协调者发送 Commit/Rollback 给所有参与者
  参与者提交/回滚并释放锁
```

**缺陷**：
1. **同步阻塞**：Prepare 阶段参与者持有锁，其他事务无法写入
2. **单点故障**：协调者宕机，参与者无限等待
3. **数据不一致**：部分 Commit 后协调者宕机，未提交的参与者无法回滚
4. **无法应对参与者宕机**：超时机制不可靠

**Tags:** #分布式 #2PC #两阶段提交

---

**Q: 3PC（三阶段提交）相比 2PC 改进了什么？**

**A:**
3PC 在 2PC 之间插入一个 **CanCommit** 阶段，减少阻塞：
```
阶段一：CanCommit（询问阶段）
  协调者询问参与者：能否执行？
  参与者返回 YES/NO（不执行事务）

阶段二：PreCommit（预提交阶段）
  协调者发送 PreCommit，所有参与者执行但不提交
  如果有 NO → 发 Abort 回滚

阶段三：DoCommit（真正提交）
  协调者发送 Commit，参与者提交
  协调者超时默认 Commit（避免无限等待）
```

**改进**：
- 引入超时机制，避免同步阻塞
- 协调者超时后自动提交（2PC 是回滚）

**但仍有缺陷**：
- 网络分区时，可能出现部分提交（脑裂问题）
- 需要更复杂的网络协议保证

**Tags:** #分布式 #3PC #三阶段提交

**Q: TCC（Try-Confirm-Cancel）的原理？**

**A:**
TCC 是一种业务层面的补偿事务模式：
```
Try：预留资源（冻结/锁定）
  → 检查业务可行性
  → 预留业务资源（如冻结库存、预扣余额）
  → 失败 → Cancel

Confirm：确认执行
  → 真正使用预留的资源
  → 幂等操作

Cancel：取消释放
  → 释放 Try 阶段预留的资源
  → 幂等操作
```

**与 2PC 的区别**：
- 2PC 是数据库层面的协议，TCC 是业务层面的
- TCC 不需要数据库支持分布式事务
- TCC 每个操作都需要提供 Try/Confirm/Cancel 三个接口

**示例（转账场景）**：
```
Try：冻结账户A 100元，冻结账户B 100元
Confirm：账户A扣100，账户B加100
Cancel：解冻账户A，解冻账户B
```

**TCC 的问题**：
- 业务侵入性大（每个业务都要实现 Try/Confirm/Cancel）
- 需要考虑空回滚、幂等、悬挂问题

**Tags:** #分布式 #TCC #分布式事务 #补偿事务

**Q: Paxos 算法的核心原理？**

**A:**
Paxos 是分布式共识算法，用于在分布式系统中就某个值达成一致：
```
角色：
  Proposer（提议者）：提出提案
  Acceptor（接受者）：投票决定接受哪个提案
  Learner（学习者）：学习最终被选定的值

两个阶段：

阶段一：Prepare（准备阶段）
  Proposer 提出提案编号 n，发送 Prepare(n) 给多数派 Acceptor
  Acceptor 收到 Prepare(n)：
    - 如果 n > 任何已响应的提案编号 → 承诺不再接受编号 < n 的提案
    - 返回已接受的编号最大的提案（如果有）

阶段二：Accept（接受阶段）
  Proposer 收到多数派响应后：
    - 如果有已接受的提案 → 用编号最大的提案值 v
    - 如果没有 → 用自己的提案值
    发送 Accept(n, v) 给多数派 Acceptor

  Acceptor 收到 Accept(n, v)：
    - 如果未承诺拒绝 n → 接受 (n, v)

多数派接受 → 提案被选定
```

**关键保证**：
- **一致性**：只有一个值会被选定
- **有效性**：只有 Proposer 提出的值才会被选定
- **终止性**（活性）：只要多数派存活，最终会选定值

**Tags:** #分布式 #Paxos #共识算法

**Q: Raft 算法和 Paxos 的区别？为什么 Raft 更易实现？**

**A:**
Raft 是 Paxos 的简化版，将问题分解为三个子问题：
```
Raft 三大子问题：
  1. Leader 选举（Leader Election）
  2. 日志复制（Log Replication）
  3. 安全性（Safety）
```

**Leader 选举**：
```
任期（Term）：逻辑时钟，每个任期最多一个 Leader

选举流程：
  1. 节点启动 → 候选人（Candidate）
  2. 向其他节点请求投票（RequestVote）
  3. 获得多数票 → 成为 Leader
  4. Leader 定期发送心跳保持权威

投票规则：
  - 候选人 term 必须 >= 自己的 term
  - 候选人数据必须比自己的数据更新
  - 每个节点只能投一票
```

**日志复制**：
```
Leader 接收客户端请求：
  1. 将命令追加到本地日志
  2. 并行发送 AppendEntries RPC 给所有 Follower
  3. 多数派确认 → Apply 到状态机
  4. 返回成功给客户端

日志条目的组成：
  <term, index, command>
  term = 产生这条日志的任期
  index = 日志在日志序列中的位置
```

**为什么比 Paxos 易实现**：
- Paxos 论文艰深，实现者需要自己补充细节
- Raft 先设计可理解性，按顺序描述三大子问题
- 日志复制模型比 Paxos 的"值槽位"更直观
- 大量工业实现（etcd、Kubernetes 使用 etcd/raft）

**Tags:** #分布式 #Raft #共识算法 #Leader选举

## 4. 分布式锁

**Q: 分布式锁的实现方案有哪些？各自的优缺点？**

**A:**
| 方案 | 优点 | 缺点 |
|------|------|------|
| Redis SETNX | 性能高，支持集群 | 非公平锁，可能有 GC STW 问题 |
| Redisson | 封装完善，有 watchdog | 依赖 Redis |
| ZooKeeper | 公平锁，有顺序，可等待 | 性能低于 Redis |
| etcd | 支持 lease，支持 TTL | 实现复杂度高 |
| Chubby（Google） | 工业级可靠性 | 闭源，部署复杂 |

**Redis vs ZooKeeper 对比**：
```
Redis 锁：
  - 获取锁：SET key value NX PX timeout
  - 释放锁：Lua 脚本原子检查 value + 删除
  - 优点：性能高
  - 缺点：无法感知锁释放事件，需要轮询/PubSub

ZooKeeper 锁：
  - 获取锁：在 /locks 下创建临时有序节点
  - 判断是否最小节点？是则获取锁
  - 释放锁：删除临时节点
  - 优点：公平锁，可等待锁释放事件
  - 缺点：性能较低（ZK 是同步协议）
```

**Tags:** #分布式 #分布式锁 #Redis #ZooKeeper

## 5. 分布式 ID

**Q: 分布式 ID 有哪些生成方案？**

**A:**
| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| UUID | 128 位随机数 | 无需中心化，本地生成快 | 无序，占用空间大 |
| 数据库自增 | 主键 AUTO_INCREMENT | 有序，递增 | 单点瓶颈，扩展难 |
| Redis INCR | INCR 命令原子递增 | 性能高，可设置步长 | 依赖 Redis |
| Snowflake | 时间戳 + 机器ID + 序列号 | 有序，不依赖第三方 | 时钟回拨问题 |
| 百度 UidGenerator | Snowflake 变种，支持并发 | 支持每秒百万级 | 实现复杂 |
| 美团 Leaf | Snowflake + 数据库号段 | 高可用，支持分段获取 | 需要额外组件 |

**Snowflake 算法**：
```
64 bit = 1 bit(固定0) + 41 bit(时间戳) + 10 bit(机器ID) + 12 bit(序列号)

时间戳：可使用 69 年
机器ID：最多 1024 个节点
序列号：每毫秒最多 4096 个 ID
```

**时钟回拨问题**：
- 方案一：发现时钟回拨 → 等待追上
- 方案二：使用 etcd/Redis 的授时服务
- 方案三：Leaf 使用数据库分配号段

**Tags:** #分布式 #分布式ID #Snowflake #UUID

## 6. 微服务架构

**Q: 微服务 vs 单体架构的区别？什么时候该拆微服务？**

**A:**
**单体架构**：
```
所有模块在一个进程内
优点：部署简单、调试方便、团队协作简单
缺点：代码耦合严重、扩展困难、技术栈单一
```

**微服务架构**：
```
每个服务独立进程，专注单一功能
优点：独立部署、技术异构、弹性扩展
缺点：运维复杂、分布式事务、网络延迟
```

**拆分的信号**：
```
1. 团队规模 > 10 人（2 pizza team）
2. 模块独立发布频率差异大
3. 技术栈需求不同
4. 性能瓶颈不同
```

**拆分的反模式**：
- 服务拆分过细（mini-service 地狱）
- 服务间循环依赖
- 分布式事务滥用

**Tags:** #微服务 #架构 #单体 #服务拆分

**Q: 服务注册与发现是怎么工作的？**

**A:**
**核心组件**：
```
服务注册中心：
  - Eureka（N Netflix，开源停止维护）
  - Consul（HashiCorp，支持多数据中心）
  - Nacos（阿里，支持配置管理 + 注册发现）
  - Zookeeper（强一致性，适合金融场景）
```

**两种模式**：
```
客户端发现（Client-Side Discovery）：
  客户端查询注册中心获取所有服务实例列表
  自己负载均衡选择目标
  优点：少一跳网络调用
  缺点：客户端复杂，需要每种语言实现

服务端发现（Server-Side Discovery）：
  客户端请求 API Gateway / Load Balancer
  网关查询注册中心并转发
  优点：客户端简单
  缺点：多一跳，有单点风险
```

**注册中心的数据同步**：
```
心跳机制：
  服务启动 → 注册到注册中心
  服务定期发送心跳 → 证明存活
  心跳超时 → 注册中心删除该服务

注册中心 CAP 选择：
  - Eureka（AP）：优先可用性
  - Consul/Zookeeper（CP）：优先一致性
```

**Tags:** #微服务 #服务发现 #注册中心 #Eureka #Nacos

**Q: API Gateway 是什么？有什么作用？**

**A:**
**API Gateway 是微服务的统一入口**

**核心功能**：
```
1. 路由转发 — 根据 path、header、参数路由到对应服务
2. 负载均衡 — Round Robin、权重、最少连接数
3. 认证授权 — JWT 验证、OAuth2、API Key
4. 限流熔断 — 令牌桶 / 漏桶限流，Sentinel/Hystrix
5. 请求/响应转换 — 协议转换、数据格式转换
6. 日志监控 — 统一日志收集，请求链路追踪
```

**常见实现**：
- Spring Cloud Gateway（Spring 官方）
- Kong（基于 Nginx + Lua）
- Zuul（Netflix，已停止维护）
- APISIX（国产高性能）

**Tags:** #微服务 #API网关 #Gateway #SpringCloud

**Q: 熔断和限流是怎么工作的？区别是什么？**

**A:**
**熔断器模式（Circuit Breaker）**：
```
防止级联故障，类似于电路保险丝

状态机：闭合（Closed）→ 打开（Open）→ 半开（Half-Open）

闭合状态：请求正常通过，失败计数超阈值 → 打开熔断器
打开状态：请求直接返回降级响应（fail-fast）
半开状态：放行少量请求尝试，成功 → 关闭，失败 → 继续打开

常用实现：Resilience4j、Sentinel、Hystrix
```

**限流（Rate Limiting）**：
```
控制单位时间内请求数量，保护系统不被压垮

常见算法：
1. 固定窗口 — 固定时间窗口内计数，简单但有边界突刺
2. 滑动窗口 — 移动时间窗口，更精确
3. 令牌桶 — 允许突发流量，Guava RateLimiter
4. 漏桶 — 严格限制速率，不允许突发
5. 滑动日志 — 精确但内存占用大
```

**区别**：
| 维度 | 熔断 | 限流 |
|------|------|------|
| 目的 | 防止级联故障 | 保护系统资源 |
| 触发条件 | 下游服务异常 | 请求量过大 |
| 效果 | 快速失败 | 延迟处理 |

**Tags:** #微服务 #熔断器 #限流 #Sentinel #Hystrix

## 7. 分布式事务

**Q: 分布式事务有哪些解决方案？各自适用场景？**

**A:**
**分布式事务的挑战**：CAP 理论告诉我们，一致性和可用性无法兼得。
```
XA 方案（强一致）：
  - 2PC / 3PC
  - 需要数据库支持 XA 协议（MySQL XA）
  - 性能差，适合短事务

TCC 方案（最终一致）：
  - Try-Confirm-Cancel
  - 业务侵入性大
  - 适合对数据一致性要求高的场景（金融、库存）

可靠消息最终一致性：
  - RocketMQ 事务消息
  - 本地消息表 + 定时任务
  - 适合异步场景

Saga 模式（最终一致）：
  - 编排式 Saga / 协同式 Saga
  - 适用于长事务、跨服务调用链
  - 需要补偿机制

最大努力通知：
  - 不保证一定成功，但尽力通知
  - 适合不重要的异步回调
```

**选型建议**：
```
高频小额支付 → TCC（强一致）
订单 → RocketMQ 事务消息（可靠消息）
日志同步 → 本地消息表（异步，最终一致）
外部回调 → 最大努力通知
```

**Tags:** #分布式 #分布式事务 #TCC #RocketMQ

## 8. 负载均衡

**Q: 常见的负载均衡算法有哪些？**

**A:**
| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| Round Robin | 轮询 | 简单，均匀 | 不考虑服务器性能差异 |
| Weighted RR | 按权重轮询 | 适配不同性能服务器 | 需要手动配置权重 |
| Random | 随机选择 | 实现简单 | 可能不均匀 |
| Least Connections | 选择连接数最少的 | 动态感知负载 | 需要维护连接状态 |
| IP Hash | hash(ip) % servers | 同 IP 路由同一后端 | 可能不均匀 |
| Consistent Hash | 一致性哈希 | 最小化服务变更影响 | 实现复杂 |

**一致性哈希的原理**：
```
1. 将服务器映射到哈希环上（0 ~ 2^32）
```
