## 场景 A：线程阻塞导致 TPS 下降

### 现象

```
监控告警：订单服务接口响应时间从 200ms 飙升到 3s
TPS：从 500 降到 50
CPU 使用率：20%（不高 → 不是 CPU 密集）
```

**关键推断**：CPU 不高 + 响应慢 → 大概率是**锁等待/IO阻塞**（不是死循环）

### 定位过程

**Step 1：快速看线程状态分布**

```bash
jstack <pid> | grep -c "RUNNABLE"     # 运行中
jstack <pid> | grep -c "BLOCKED"       # 阻塞等待锁
jstack <pid> | grep -c "WAITING"       # 等待条件
```

**Step 2：找到阻塞的「罪魁祸首」**

```bash
# Arthas 在线诊断
thread -n 5              # CPU 最高的 5 个线程
thread -b                # 找出当前阻塞其他线程的线程
thread --state BLOCKED   # 所有阻塞线程
```

**Step 3：分析 jstack 输出**

```
"http-nio-8080-exec-3" #23 prio=5 os_prio=0 tid=0x... nid=0x... BLOCKED
   java.lang.Thread.State: BLOCKED (on object monitor)
    at com.xxx.OrderService.createOrder(OrderService.java:45)
    - waiting to lock <0x000000076ab5c2f0> (a com.xxx.OrderLock)
```

→ 找 `locked <...>` 对应的线程，看它在干什么。

### 根因

```java
// 问题代码
public synchronized Order createOrder(OrderRequest req) {
    validate(req);         // 不需要锁
    deductStock(req);      // 需要锁，~5ms
    Order order = createOrder(req); // 需要锁，~5ms
    sendMessage(order);    // ❌ IO 操作，100-500ms，在锁内！
    log.info("order created"); // 不需要锁
}
```

锁粒度太大，IO 操作放在了锁内，每个请求多等 100-500ms。

### 解决方案

```java
// 方案1：缩小锁范围
public Order createOrder(OrderRequest req) {
    validate(req);
    Order order;
    synchronized(lock) {
        deductStock(req);
        order = createOrder(req);
    }
    sendMessage(order);  // 锁外
    return order;
}

// 方案2：ReentrantLock + tryLock（推荐）
private final ReentrantLock lock = new ReentrantLock();

public Order createOrder(OrderRequest req) {
    validate(req);
    if (!lock.tryLock(3, TimeUnit.SECONDS)) {
        throw new BizException("系统繁忙");
    }
    try {
        deductStock(req);
        return createOrder(req);
    } finally {
        lock.unlock();
    }
}
```

效果：锁持有时间从 ~500ms → ~10ms，TPS 从 50 → 800+。

## 场景 B：线程池配置不当导致任务堆积

### 现象

```
线程池活跃线程数一直 = 核心线程数
队列长度持续增长
最终 OOM：unable to create new native thread
```

### 定位

```java
// 问题代码
ExecutorService pool = Executors.newFixedThreadPool(10);
// 默认使用 LinkedBlockingQueue(Integer.MAX_VALUE)
// 任务只增不减，最终 OOM
```

### 根因

`newFixedThreadPool` 的队列是无界队列（容量 `Integer.MAX_VALUE`），任务会无限堆积。

### 解决方案

```java
// 正确姿势：自定义 ThreadPoolExecutor，有界队列 + 拒绝策略
ThreadPoolExecutor pool = new ThreadPoolExecutor(
    10,                       // corePoolSize
    20,                       // maximumPoolSize
    60, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(1000),  // 有界队列
    new ThreadFactoryBuilder().setNameFormat("order-pool-%d").build(),
    new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略：调用线程执行
);
```

## 场景 C：CAS + 自旋导致的 CPU 飙高

### 现象

```
CPU 使用率飙升到 95%+
接口响应正常，但机器扛不住
```

### 定位

```bash
# Arthas
thread -n 5    # 发现多个线程一直在 RUNNABLE
profiler start    # CPU 火焰图
profiler stop     # 发现大量时间在 Unsafe.compareAndSwap*
```

### 根因

```java
// 高并发下 LongAdder 的 base CAS 竞争激烈
// 或者自定义的 CAS 自旋没有退避策略
while (!atomicRef.compareAndSet(expect, update)) {
    // ❌ 没有任何退避，空转消耗 CPU
}
```

### 解决方案

```java
// 加上退避策略
int retries = 0;
while (!atomicRef.compareAndSet(expect, update)) {
    retries++;
    if (retries > 100) {
        // 超过重试次数，降级为锁
        synchronized(lock) {
            if (atomicRef.get() == expect) {
                atomicRef.set(update);
            }
        }
        break;
    }
    Thread.yield();  // 让出 CPU 时间片
}
```

## 追问链

### 追问 1：怎么判断是锁竞争而不是 GC 导致的？

> "首先看 GC 日志（`-XX:+PrintGCDetails`），如果 Full GC 频率正常就排除 GC。然后看 jstack 输出中 `BLOCKED` 线程数量，如果大量线程都在 `waiting to lock <同一个对象>`，基本确认是锁竞争。还可以用 Arthas 的 `thread -b` 直接找出阻塞源头。"

→ 关联：03_JVM/06_垃圾回收算法

### 追问 2：notify() 之后线程会直接获取锁吗？

> "不会。notify() 是把线程从 `_WaitSet` 移到 `_EntryList`，线程状态是 `BLOCKED`，需要**重新竞争锁**。这也是为什么 `wait()/notify()` 必须在 synchronized 块里调用。"

→ 关联：02_并发编程/04_synchronized

### 追问 3：线程池的队列用无界还是有界？

> "生产环境**必须用有界队列**。无界队列在任务生产速度 > 消费速度时，队列无限增长，最终 OOM。`CallerRunsPolicy` 是最安全的拒绝策略，让调用线程自己执行任务，起到自然的限流效果。"

→ 关联：02_并发编程/10_线程池

### 追问 4：怎么监控线程池的运行状态？

> "通过 `ThreadPoolExecutor` 的 API：`getActiveCount()`、`getQueue().size()`、`getCompletedTaskCount()`、`getLargestPoolSize()`。生产环境建议配合 Micrometer 暴露 metrics 到 Prometheus，设告警规则：队列长度 > 阈值、活跃线程数持续等于最大线程数。"

→ 关联：08_工程化/监控告警

## 排查 Checklist

```
□ CPU 高吗？  → 高：看线程堆栈，找热点方法
                → 低（<30%）：看线程状态，大概率锁等待/IO阻塞
□ GC 频繁吗？ → 看_gc 日志_，jstat -gcutil <pid> 1000
□ 线程池健康吗？ → 队列长度、拒绝次数、活跃线程数
□ 有死锁吗？ → jstack -l <pid> 或 Arthas thread -b
□ 连接池满了吗？ → 数据库连接池监控、HTTP 连接池监控
```
