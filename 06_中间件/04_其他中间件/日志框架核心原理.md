# Java 日志框架核心原理

> 日志是线上问题排查的第一手资料。Java 日志生态以 SLF4J 门面 + Logback/Log4j2 实现为核心，理解其架构有助于正确配置和排查日志相关问题。面试重点在于门面模式、日志级别、MDC 链路追踪、异步日志。

---

## 一、日志框架演进

```
System.out.println()
    ↓ 问题：无法分级、无法关闭、无法输出到文件
java.util.logging（JDK 1.4）
    ↓ 问题：性能差、功能弱
Log4j（Apache，2001）
    ↓ 问题：单点依赖，切换日志框架需改代码
SLF4J（门面模式，2005）+ Logback（参考实现）
    ↓ Logback 是 Log4j 作者写的升级版
Log4j2（Apache，2014）
    ↓ 性能更优（Disruptor 异步）、功能更强
现状：SLF4J（门面）+ Logback 或 Log4j2（实现）
```

### 1.1 为什么需要门面模式？

```
问题：项目依赖了 A 库用 Log4j，B 库用 JUL，C 库用 Commons Logging
    → 日志框架冲突，输出混乱，无法统一配置

解决：SLF4J 作为门面（接口层），底层实现可替换
    → 所有代码面向 SLF4J API 编程
    → 实际输出由 Logback 或 Log4j2 完成
    → 切换实现只需换依赖，不改一行业务代码
```

## 二、SLF4J 门面

### 2.1 核心接口

```java
// 获取 Logger（推荐用类名）
private static final Logger log = LoggerFactory.getLogger(MyClass.class);

// 或 Lombok 简写
@Slf4j
public class MyClass {
    // 自动生成 private static final Logger log = LoggerFactory.getLogger(MyClass.class);
}
```

### 2.2 日志级别

```
ERROR（错误） > WARN（警告） > INFO（信息） > DEBUG（调试） > TRACE（追踪）

级别规则：
- 只输出 >= 当前设置级别的日志
- 设置为 INFO → 只输出 ERROR、WARN、INFO
- 设置为 DEBUG → 输出 ERROR、WARN、INFO、DEBUG
```

| 级别 | 用途 | 示例 |
|------|------|------|
| **ERROR** | 系统错误，需要立即处理 | 数据库连接失败、NPE |
| **WARN** | 潜在问题，不影响主流程 | 参数为空使用默认值、接口超时重试 |
| **INFO** | 重要业务信息 | 订单创建成功、用户登录 |
| **DEBUG** | 调试信息，开发环境使用 | SQL 参数、方法入参出参 |
| **TRACE** | 最详细的信息，一般不用 | 循环中的每一步状态 |

### 2.3 参数化日志

```java
// 推荐：参数化（{} 占位符），字符串拼接只在日志确实输出时执行
log.debug("User {} login from IP {}", username, ip);

// 不推荐：字符串拼接，无论是否输出都会执行
log.debug("User " + username + " login from IP " + ip);

// 原因：如果级别是 INFO，debug 日志不输出
//      但字符串拼接的开销已经产生了（创建 StringBuilder + toString）
//      参数化写法避免了这个开销
```

### 2.4 Lombok @Slf4j 常见问题

```java
// Lombok 的 log 变量名是固定的 "log"
// 如果需要多个 Logger：
@CustomLog   // private static final Logger log = LoggerFactory.getLogger(MyClass.class);
private static final Logger bizLog = LoggerFactory.getLogger("BIZ_LOG");
```

## 三、Logback 架构与配置

### 3.1 核心组件

```
Logger（记录器）
    ↓ 关联
Level（日志级别）
    ↓ 关联
Appender（输出目的地）
    ↓ 关联
Filter（过滤器）
    ↓ 关联
Layout（格式化）+ Encoder（编码）
```

| 组件 | 职责 | 配置 |
|------|------|------|
| **Logger** | 日志记录器，按包名/类名层级组织 | `<logger name="com.example" level="DEBUG">` |
| **Appender** | 日志输出目的地 | Console / File / RollingFile / Async |
| **Layout** | 日志格式化 | PatternLayout |
| **Encoder** | 编码输出 | PatternLayoutEncoder |
| **Filter** | 过滤日志 | LevelFilter / ThresholdFilter |

### 3.2 Logger 继承关系

```
root（根 Logger，默认级别 DEBUG）
├── com（继承 root）
│   └── com.example（继承 com）
│       ├── com.example.service（继承 com.example）
│       └── com.example.controller（继承 com.example）
└── org（继承 root）
    └── org.springframework（可单独设置级别）

<!-- 配置示例 -->
<configuration>
    <!-- root 默认 INFO -->
    <root level="INFO">
        <appender-ref ref="CONSOLE" />
        <appender-ref ref="FILE" />
    </root>

    <!-- 特定包设置 DEBUG 级别 -->
    <logger name="com.example.service" level="DEBUG" additivity="false">
        <appender-ref ref="FILE" />
    </logger>

    <!-- 降低框架日志 -->
    <logger name="org.springframework" level="WARN" />
    <logger name="org.apache.kafka" level="WARN" />
</configuration>
```

> **additivity="false"**：日志不向父 Logger 传播，避免重复输出。

### 3.3 日志格式

```xml
<!-- 常用 Pattern -->
<pattern>
    %d{yyyy-MM-dd HH:mm:ss.SSS}  ← 时间
    [%thread]                      ← 线程名
    %-5level                       ← 级别（左对齐 5 字符）
    %logger{36}                    ← Logger 名（最长 36 字符）
    - %msg%n                       ← 消息 + 换行
</pattern>

<!-- 推荐格式（带颜色） -->
<pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} %highlight(%-5level) [%boldMagenta(%thread)] %cyan(%logger{36}) - %msg%n</pattern>

<!-- 生产环境格式（方便 ELK 解析） -->
<pattern>{"time":"%d{ISO8601}","level":"%-5level","thread":"%thread","logger":"%logger","msg":"%msg"}%n</pattern>
```

### 3.4 滚动文件策略

```xml
<appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
    <file>logs/app.log</file>
    <rollingPolicy class="ch.qos.logback.core.rolling.SizeAndTimeBasedRollingPolicy">
        <!-- 按天 + 大小滚动 -->
        <fileNamePattern>logs/app.%d{yyyy-MM-dd}.%i.log.gz</fileNamePattern>
        <!-- 单文件最大 100MB -->
        <maxFileSize>100MB</maxFileSize>
        <!-- 保留 30 天 -->
        <maxHistory>30</maxHistory>
        <!-- 总大小上限 10GB -->
        <totalSizeCap>10GB</totalSize>
    </rollingPolicy>
    <encoder>
        <pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n</pattern>
    </encoder>
</appender>
```

## 四、MDC 链路追踪

### 4.1 原理

```
MDC（Mapped Diagnostic Context）= 线程绑定的 Map

请求进入时 → MDC.put("traceId", "abc123")
日志输出时 → Pattern 中 %X{traceId} 自动取出
请求结束时 → MDC.clear()

效果：同一请求的所有日志都带相同的 traceId，可以 grep 快速串联
```

### 4.2 实现方式

```java
// 方案1：Filter（最简单）
@Component
public class TraceFilter implements Filter {
    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
                         FilterChain chain) throws IOException, ServletException {
        String traceId = UUID.randomUUID().toString().replace("-", "");
        MDC.put("traceId", traceId);
        try {
            chain.doFilter(request, response);
        } finally {
            MDC.clear();
        }
    }
}

// 方案2：SpringBoot + Micrometer Tracing（推荐，支持 Reactor）
// 依赖：spring-boot-starter-actuator + micrometer-tracing-bridge-brave
// 自动在日志中注入 traceId 和 spanId

// logback-spring.xml 中引用
<pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] [%X{traceId}] %-5level %logger{36} - %msg%n</pattern>
```

### 4.3 异步线程 MDC 传递

```java
// 问题：@Async / 线程池中 MDC 丢失
// 解决：使用 TaskDecorator 传递 MDC

@Bean
public TaskExecutor taskExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(10);
    executor.setTaskDecorator(new MdcTaskDecorator());  // 关键
    executor.initialize();
    return executor;
}

public class MdcTaskDecorator implements TaskDecorator {
    @Override
    public Runnable decorate(Runnable runnable) {
        Map<String, String> contextMap = MDC.getCopyOfContextMap();
        return () -> {
            try {
                if (contextMap != null) {
                    MDC.setContextMap(contextMap);
                }
                runnable.run();
            } finally {
                MDC.clear();
            }
        };
    }
}
```

## 五、异步日志

### 5.1 Logback AsyncAppender

```xml
<!-- Logback 异步方式：AsyncAppender 包装同步 Appender -->
<appender name="ASYNC_FILE" class="ch.qos.logback.classic.AsyncAppender">
    <!-- 队列大小，默认 256 -->
    <queueSize>1024</queueSize>
    <!-- 队列满时不丢弃任何日志 -->
    <discardingThreshold>0</discardingThreshold>
    <!-- 不丢失 INFO 及以上级别的日志 -->
    <neverBlock>true</neverBlock>
    <!-- 实际输出 -->
    <appender-ref ref="FILE" />
</appender>
```

### 5.2 Log4j2 AsyncLogger（Disruptor）

```
Log4j2 异步日志 = LMAX Disruptor（无锁环形队列）

性能对比：
- 同步日志：~1万条/秒
- Logback AsyncAppender：~10万条/秒
- Log4j2 AsyncLogger：~1800万条/秒（号称最快）

原因：Disruptor 用环形数组 + CAS 替代锁，消除线程竞争
```

```xml
<!-- Log4j2 异步配置（Log4j2.xml） -->
<Configuration>
    <Appenders>
        <RollingFile name="FILE" fileName="logs/app.log"
            filePattern="logs/app-%d{yyyy-MM-dd}-%i.log.gz">
            <PatternLayout pattern="%d %p %c{1.} [%t] %m%n"/>
            <Policies>
                <TimeBasedTriggeringPolicy/>
                <SizeBasedTriggeringPolicy size="100MB"/>
            </Policies>
        </RollingFile>
    </Appenders>
    <Loggers>
        <!-- 全局异步：所有 Logger 都异步 -->
        <AsyncRoot level="info">
            <AppenderRef ref="FILE"/>
        </AsyncRoot>

        <!-- 混合模式：部分 Logger 异步 -->
        <AsyncLogger name="com.example" level="debug" includeLocation="false">
            <AppenderRef ref="FILE"/>
        </AsyncLogger>
    </Loggers>
</Configuration>
```

### 5.3 异步日志注意事项

| 注意点 | 说明 |
|--------|------|
| **日志丢失** | 队列满时会丢弃日志（配置 `discardingThreshold=0` 或 `neverBlock=true`） |
| **异常丢失** | 异步线程的异常可能与原始请求上下文脱节 |
| **JVM 崩溃** | 队列中未输出的日志会丢失（接受范围） |
| **includeLocation** | 获取调用行号有性能开销，生产环境建议关闭 |

### 6.1 方案

public class SensitiveDataConverter extends ClassicConverter {
    @Override
    public String convert(ILoggingEvent event) {
        String msg = event.getFormattedMessage();
        // 手机号脱敏
        msg = msg.replaceAll("(1\\d{2})\\d{4}(\\d{4})", "$1****$2");
        // 身份证脱敏
        msg = msg.replaceAll("(\\d{6})\\d{8}(\\d{4})", "$1********$2");
        // 邮箱脱敏
        msg = msg.replaceAll("(\\w{3})\\w+@(\\w+)", "$1***@$2");
        return msg;
    }
}

## 七、Logback vs Log4j2 对比

| 维度 | Logback | Log4j2 |
|------|---------|--------|
| **性能** | 好（AsyncAppender） | 更好（Disruptor AsyncLogger） |
| **SpringBoot 默认** | 是 | 需要排除 Logback 再引入 |
| **配置格式** | logback.xml / logback-spring.xml | log4j2.xml / log4j2-spring.xml |
| **自动重载** | 支持（`scan="true"`） | 支持 |
| **Fluentd/JSON** | 需要额外依赖 | 原生支持 |
| **Lambda 延迟求值** | 不支持 | 支持（`log.debug(() -> expensiveMethod())`) |
| **GC 压力** | 格式化产生临时 String | 可配置使用 StringBuilder 复用 |

## 九、常见问题

### 9.1 日志冲突（SLF4J 绑定冲突）

```
现象：启动时报错
    SLF4J: Class path contains multiple SLF4J bindings.
    SLF4J: Found binding in [jar:...logback-classic-1.2.11.jar]
    SLF4J: Found binding in [jar:...log4j-slf4j-impl-2.17.2.jar]

原因：classpath 中有多个 SLF4J 实现的 jar 包

解决：只保留一个实现
    mvn dependency:tree | grep log
    排除多余的日志依赖
```

### 9.2 日志打不出来的排查步骤

```
1. 确认日志级别：Logger 的级别是否 >= 配置的级别？
2. 确认 additivity：是否被父 Logger 的配置覆盖？
3. 确认 Appender：有没有配置 appender-ref？
4. 确认文件权限：日志目录是否有写入权限？
5. 确认依赖冲突：是否只有一个 SLF4J 实现？
```

## 十、面试高频问题

### Q1：为什么要用 SLF4J？直接用 Logback 不行吗？

> SLF4J 是门面模式，解耦了日志 API 和实现。好处：① 所有代码面向 SLF4J 编程，切换底层实现（Logback → Log4j2）只需换依赖不改代码；② 解决依赖冲突问题，第三方库用不同日志框架时，通过 SLF4J Bridge 统一适配到同一个实现；③ SpringBoot 默认使用 SLF4J + Logback。

### Q2：log.info("user: " + user) 和 log.info("user: {}", user) 有什么区别？

> 后者是参数化日志，性能更好。`"user: " + user` 无论日志是否输出都会执行字符串拼接（创建 StringBuilder + toString），而 `"user: {}", user` 只在日志确实输出时才格式化。当日志级别高于当前配置（如 INFO 级别配置下调用 `log.debug`），参数化方式避免了无谓的字符串拼接开销。

### Q3：什么是 MDC？怎么用？

> MDC（Mapped Diagnostic Context）是 SLF4J 提供的线程级上下文 Map。请求进入时设置 traceId（`MDC.put("traceId", uuid)`），日志 Pattern 中用 `%X{traceId}` 引用，同一请求的所有日志都带相同的 traceId。可以快速 grep 串联一个请求的完整日志链路。异步线程中 MDC 需要手动传递（TaskDecorator）。

### Q4：Logback 和 Log4j2 怎么选？

> SpringBoot 默认 Logback，满足 90% 场景。选 Log4j2 的理由：① 极高并发场景需要 Disruptor 异步日志的性能优势；② 需要 JSON 格式日志（原生支持）；③ 需要 Lambda 延迟求值。切换只需排除 `spring-boot-starter-logging`，引入 `spring-boot-starter-log4j2`。

### Q5：生产环境日志配置有什么要注意的？

> ① 日志级别设 INFO，业务包可 DEBUG 但必须有开关；② 文件滚动策略（按天+按大小，保留 30 天，总大小限制）；③ 使用异步日志减少对业务线程的阻塞；④ 日志格式结构化（JSON），方便 ELK 采集解析；⑤ 敏感信息脱敏（手机号/身份证/密码）；⑥ 开启 MDC traceId 实现全链路追踪；⑦ 监控 ERROR 日志告警（接入钉钉/邮件）；⑧ 避免循环日志（Object.toString 导致无限递归）。

## 知识关联

- **AOP**：日志切面常用 AOP 实现方法入参/出参/耗时的统一记录
- **Filter/Interceptor**：MDC traceId 通常在 Filter 中设置
- **ThreadLocal**：MDC 底层基于 ThreadLocal，存在跨线程传递问题
- **Disruptor**：Log4j2 AsyncLogger 使用 LMAX Disruptor 无锁队列实现高性能
- **ELK**：Elasticsearch + Logstash + Kibana，日志采集分析的标准方案
- **Prometheus + Grafana**：日志量监控、错误率告警
