# 降级与熔断

## 这个问题为什么存在？

> 服务 A 调用服务 B，B 挂了或者响应慢，A 的请求全部堆积，线程池 / 连接池耗尽，导致 A 也挂了——**雪崩**。降级和熔断就是解决这个问题：**在依赖出问题时，主动切断或降低服务质量，保护自身不崩**。

两者的区别：
- **熔断**：检测到下游故障时，主动断开调用（不再发请求），给下游恢复时间
- **降级**：系统压力过大时，关闭非核心功能，释放资源给核心功能

## 它是怎么解决问题的？

### 熔断器状态机

```
        ┌──────────┐
        │          │
        ▼          │
    ┌─────┐    ┌─────┐    ┌─────┐
    │Closed│───▶│Open │───▶│Half-Open│
    └─────┘    └─────┘    └─────┘
      ▲            │            │
      │            │            │
      └────────────┴────────────┘
      恢复成功     超时后 尝试一半
```

| 状态 | 行为 |
|---|---|
| **Closed（关闭）** | 正常调用，统计失败率 |
| **Open（打开）** | 所有请求快速失败（FallBack），不调用下游 |
| **Half-Open（半开）** | 放一部分请求试下游是否恢复，成功则关闭熔断，失败则回到 Open |

### 熔断触发条件（以 Sentinel 为例）

```
10 秒内调用 ≥ 10 次（最小请求数）
并且错误率（异常比例）≥ 50%
→ 触发熔断，进入 Open 状态
```

Open 状态持续 N 秒（如 10s）后，自动进入 Half-Open，放行少量请求试探。

### 降级策略

降级不依赖错误率，而是**主动**关闭功能：

```
系统负载 > 阈值 → 降级非核心接口（返回兜底数据）
CPU > 80%     → 降级推荐、统计等可延后功能
线程池已满     → 降级新请求，直接 Fallback
```

典型 Fallback 设计：
```java
// 查询商品详情，库存服务挂了
public StockVO getStockFallback(Long skuId) {
    return StockVO.builder()
        .skuId(skuId)
        .stock(-1)           // -1 表示"数据暂时不可用"
        .fromCache(true)
        .build();
}
```

## 它和相似方案的本质区别是什么？

| | 熔断（Circuit Breaker） | 降级（Degrade） | 限流（Rate Limit） |
|---|---|---|---|
| 触发条件 | 下游错误率超阈值 | 系统资源不足（主动） | 请求量超阈值 |
| 保护对象 | 调用方自己 | 调用方自己 | 被调用方（或调用方） |
| 行为 | 快速失败，不发起调用 | 返回兜底数据 | 拒绝或排队 |
| 恢复 | 自动（Half-Open 试探） | 手动或自动（资源恢复后） | 自动（下一时间窗口） |

**Hystrix vs Sentinel**：

| | Hystrix | Sentinel |
|---|---|---|
| 熔断策略 | 错误率 + 慢调用 | 错误率 + 慢调用 + 异常数 |
| 降级维度 | 线程池隔离 | QPS / 线程数 / 系统负载 |
| 控制台 | Hystrix Dashboard（简单） | Sentinel Dashboard（功能丰富） |
| 状态 | 已停止维护 | 活跃维护（阿里开源） |

## 正确使用方式

### Sentinel 注解方式

```java
@Service
public class OrderService {
    // value：资源名；fallback：降级方法
    @SentinelResource(value = "createOrder",
                      fallback = "createOrderFallback")
    public OrderVO createOrder(Long userId, Long skuId) {
        // 正常业务逻辑
        return doCreate(userId, skuId);
    }

    // 降级方法，参数和返回值必须一致，可多一个 Throwable
    public OrderVO createOrderFallback(Long userId, Long skuId, Throwable t) {
        return OrderVO.builder().code(500).msg("服务繁忙，请稍后重试").build();
    }
}
```

### Sentinel 规则配置（熔断）

```java
List<DegradeRule> rules = new ArrayList<>();
DegradeRule rule = new DegradeRule()
    .setResource("createOrder")
    .setGrade(RuleConstant.DEGRADE_GRADE_RT)  // 按响应时间熔断
    .setCount(500)          // RT > 500ms 触发
    .setTimeWindow(10);     // 熔断 10 秒
rules.add(rule);
DegradeRuleManager.loadRules(rules);
```

## 边界情况和坑

### Fallback 里再调用下游 → 死循环

```java
@SentinelResource(value = "getStock", fallback = "getStockFallback")
public StockVO getStock(Long skuId) {
    return stockClient.get(skuId);  // 可能抛异常
}

public StockVO getStockFallback(Long skuId) {
    return stockClient.get(skuId);  // ❌ Fallback 里又调下游，还是会失败
}
```

**正确做法**：Fallback 只返回本地兜底数据，不发起任何远程调用。

### 熔断后「误杀」正常请求

```
下游短暂抖动（网络闪断 2 秒）→ 错误率超 50% → 熔断 10 秒
→ 下游其实已经恢复了，但熔断器还在 Open 状态
```

解决：适当调大「最小请求数」，避免少量请求就触发熔断。

### Sentinel 规则默认内存存储

Sentinel 的规则（熔断、限流）默认存在内存里，**重启后丢失**。生产环境需要配置 `DataSource`（如 Nacos、Apollo、ZooKeeper）持久化规则。

```java
// 从 Nacos 读取规则
ReadableDataSource<String, List<FlowRule>> flowRuleDataSource =
    new NacosDataSource<>(properties, groupId, dataId,
        source -> JSON.parseArray(source, FlowRule.class));
FlowRuleManager.register2Property(flowRuleDataSource.getProperty());
```

## 我的理解

熔断是「下游病了，我不调了」；降级是「我资源不够了，先保核心」。

面试常问 **Sentinel vs Hystrix**：Hystrix 用线程池隔离，开销大；Sentinel 用 **信号量（并发计数）**，轻量，并且支持多种降级维度（RT、异常比例、异常数）。
