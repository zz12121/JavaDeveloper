## 知识链提示

```
慢查询分析
  → [[06_数据库/04_SQL优化/慢查询分析与优化]]
    → 排查步骤
      → 1. 开启慢查询日志：slow_query_log = ON, long_query_time = 1（秒）
      → 2. 用 mysqldumpslow 分析慢查询日志
        → 按查询时间排序 / 按出现次数排序
      → 3. 对每条慢 SQL 跑 EXPLAIN
        → [[06_数据库/04_SQL优化/EXPLAIN执行计划]] → 看 type / key / rows / Extra
      → 4. 分析索引设计是否合理
        → [[06_数据库/02_索引/索引设计原则]] → 联合索引、覆盖索引
      → 5. 分析 SQL 写法是否有问题
        → SELECT * → 不覆盖索引，回表多
        → 深分页：LIMIT 1000000, 20 → 先查 ID 再 JOIN
        → 子查询 → 改写成 JOIN
    → 常见优化手段
      → 加索引（联合索引注意最左前缀）
      → 覆盖索引（Extra: Using index）
      → 改写 SQL（避免子查询、深分页）
      → 分库分表（数据量太大时）
        → [[06_数据库/04_SQL优化/分库分表方案]]
```

## 核心追问

| 追问 | 考点 |
|------|------|
| 深分页（LIMIT 1000000, 20）怎么优化？ | 先查 ID 再 JOIN / 游标分页 |
| `SELECT COUNT(*)` 很慢怎么办？ | 用估算值 / 缓存计数 |
| 联合索引 (a,b,c)，WHERE a=1 ORDER BY c 会 filesort 吗？ | 会，因为 b 没出现在 WHERE 里 |
| 覆盖索引能解决所有慢查询吗？ | 不能，覆盖索引只能优化查询，不能优化写入 |

---

## 参考答案要点

**排查流程**：
1. 开启慢查询日志 → 找到慢 SQL
2. `EXPLAIN` 分析 → 看 type / key / rows / Extra
3. 检查索引设计 → 是否走了索引？是否覆盖索引？
4. 检查 SQL 写法 → SELECT * / 深分页 / 子查询

**三大优化方向**：
- 索引层面：加索引、联合索引、覆盖索引
- SQL 层面：改写子查询、优化深分页、避免 SELECT *
- 架构层面：读写分离、分库分表
