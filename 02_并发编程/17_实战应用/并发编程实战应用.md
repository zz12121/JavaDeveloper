# 并发编程实战应用

> 前面的章节讲了线程、锁、JMM、CAS、工具类……这些知识点如果不去结合真实场景，就是一堆散落的零件。本章把零件组装成机器——用 8 个真实业务场景，演示并发编程在实际项目中的正确用法和常见陷阱。

## 这个问题为什么存在？

### 知道 ≠ 会用

```
学完并发编程后的状态：

知道 synchronized 怎么用

知道 volatile 能保证可见性

知道线程池有 7 个参数
→ 但核心线程数设多少？队列选哪种？拒绝策略怎么选？

知道 CompletableFuture 能编排异步任务

"会用"和"知道"之间差的就是：实战
```

## 实战场景一：接口聚合查询

### 场景描述

```
商品详情页需要聚合多个下游服务的数据：
- 商品基础信息（商品服务）
- 库存信息（库存服务）
- 价格信息（价格服务）
- 优惠活动（营销服务）
- 用户评价（评价服务）

这 5 个接口互相独立，串行调用 5 次网络 RTT ≈ 500ms
并行调用 → 总耗时 = 最慢的那个 ≈ 100ms

目标：并行调用，总超时 3 秒，部分失败不阻塞整体
```

### 方案：CompletableFuture 并行编排

```java
public ProductDetail getProductDetail(Long productId, Long userId) {

    // 1. 并行发起 5 个查询
    CompletableFuture<ProductInfo> productFuture = CompletableFuture.supplyAsync(
        () -> productService.getById(productId), ioPool);

    CompletableFuture<StockInfo> stockFuture = CompletableFuture.supplyAsync(
        () -> stockService.getByProductId(productId), ioPool);

    CompletableFuture<PriceInfo> priceFuture = CompletableFuture.supplyAsync(
        () -> priceService.getByProductId(productId, userId), ioPool);

    CompletableFuture<PromoInfo> promoFuture = CompletableFuture.supplyAsync(
        () -> promoService.getActivePromo(productId), ioPool);

    CompletableFuture<List<ReviewInfo>> reviewFuture = CompletableFuture.supplyAsync(
        () -> reviewService.getByProductId(productId, 0, 10), ioPool);

    // 2. 统一等所有完成，超时 3 秒
    CompletableFuture.allOf(productFuture, stockFuture, priceFuture,
                             promoFuture, reviewFuture)
        .orTimeout(3, TimeUnit.SECONDS)   // JDK 9+
        .join();

    // 3. 汇总结果（部分失败用 fallback）
    ProductDetail detail = new ProductDetail();
    detail.setProduct(getOrFallback(productFuture, ProductInfo.EMPTY));
    detail.setStock(getOrFallback(stockFuture, StockInfo.EMPTY));
    detail.setPrice(getOrFallback(priceFuture, PriceInfo.DEFAULT));
    detail.setPromo(getOrFallback(promoFuture, null));
    detail.setReviews(getOrFallback(reviewFuture, Collections.emptyList()));

    return detail;
}

// 工具方法：获取结果或返回默认值
private <T> T getOrFallback(CompletableFuture<T> future, T fallback) {
    try {
        return future.getNow(fallback);  // 已完成返回结果，否则返回 fallback
    } catch (CompletionException e) {
        log.warn("子任务异常: {}", e.getCause().getMessage());
        return fallback;
    }
}
```

### 关键决策

```
1. 为什么用 CompletableFuture 而不是虚拟线程？
   → 如果项目已经是 JDK 21+，虚拟线程更简单
   → 如果还在 JDK 8~17，CompletableFuture 是标准方案

2. 为什么用 allOf + join 而不是每个都 get？
   → allOf 可以统一设超时
   → 每个单独 get 需要写 5 遍 try-catch

3. 为什么用自定义 ioPool 而不是 ForkJoinPool.commonPool()？
   → commonPool 默认只有 CPU-1 个线程，I/O 密集任务会互相阻塞
   → ioPool 应该设大一些（50~200）

4. 为什么用 orTimeout 而不是单个 future.completeOnTimeout？
   → 整体超时语义：3 秒内必须全部返回
   → 单个超时语义：每个接口独立超时
   → 业务需要的是整体超时
```

## 实战场景二：缓存一致性

### 场景描述

```
需求：数据库更新后，缓存同步更新

方案对比：
1. 先更新 DB 再删缓存 → 偶发不一致（读线程读到旧缓存时 DB 已更新）
2. 先删缓存再更新 DB → 偶发不一致（删缓存后、更新 DB 前，读线程回填旧值）
3. 延迟双删 → 接近最终一致，但有短暂的脏读窗口

没有银弹，关键在于理解每种方案的 tradeoff
```

### 方案：延迟双删 + 分布式锁

```java
public void updateProduct(Product product) {
    String cacheKey = "product:" + product.getId();
    String lockKey = "lock:product:" + product.getId();

    // 1. 先删缓存
    redisTemplate.delete(cacheKey);

    // 2. 更新数据库
    productMapper.updateById(product);

    // 3. 延迟再删一次（防止步骤 1~2 之间有读线程回填旧值）
    CompletableFuture.runAsync(() -> {
        try {
            Thread.sleep(500);  // 延迟 500ms
            redisTemplate.delete(cacheKey);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }, ioPool);
}

// 读缓存逻辑：加锁防止缓存击穿
public Product getProduct(Long id) {
    String cacheKey = "product:" + id;
    String lockKey = "lock:product:" + id;

    // 1. 读缓存
    Product product = redisTemplate.opsForValue().get(cacheKey);
    if (product != null) {
        return product;
    }

    // 2. 缓存未命中，获取分布式锁
    String lockValue = UUID.randomUUID().toString();
    try {
        Boolean locked = redisTemplate.opsForValue()
            .setIfAbsent(lockKey, lockValue, 10, TimeUnit.SECONDS);
        if (Boolean.TRUE.equals(locked)) {
            // 3. 双重检查：拿到锁后再查一次缓存
            product = redisTemplate.opsForValue().get(cacheKey);
            if (product != null) {
                return product;
            }

            // 4. 查数据库并回填缓存
            product = productMapper.selectById(id);
            if (product != null) {
                redisTemplate.opsForValue().set(cacheKey, product, 30, TimeUnit.MINUTES);
            }
            return product;
        } else {
            // 没拿到锁，短暂等待后重试
            Thread.sleep(50);
            return getProduct(id);  // 递归重试（生产环境用循环+重试次数限制）
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
        return productMapper.selectById(id);  // 降级到直接查 DB
    } finally {
        // 释放锁（用 Lua 脚本保证原子性）
        unlockWithLua(lockKey, lockValue);
    }
}
```

### 并发要点

```
1. 为什么需要分布式锁？
   → 防止缓存击穿：热点 key 失效瞬间，大量请求同时打到 DB
   → 锁粒度：按 key 级别，不要用全局锁

2. 为什么需要双重检查？
   → 拿到锁之前可能有其他线程已经回填了缓存
   → 避免重复查 DB

3. 为什么释放锁要用 Lua 脚本？
   → 保证"比较 lockValue + 删除"的原子性
   → 防止锁过期后被其他线程持有，自己误删别人的锁
```

## 实战场景三：异步任务链

### 场景描述

```
用户下单流程：
1. 校验库存 → 2. 创建订单 → 3. 扣减库存 → 4. 发送通知 → 5. 记录日志

其中步骤 4 和 5 不需要同步返回，可以异步执行
步骤 1~3 必须同步，且需要事务保证一致性
```

### 方案：同步核心链路 + 异步非核心步骤

```java
@Transactional
public OrderResult createOrder(OrderRequest request) {
    // 同步核心链路
    // 1. 校验库存（乐观锁）
    boolean stockOk = stockService.deduct(request.getProductId(), request.getQuantity());
    if (!stockOk) {
        throw new BusinessException("库存不足");
    }

    // 2. 创建订单
    Order order = Order.create(request);
    orderMapper.insert(order);

    // 3. 异步执行非核心步骤
    CompletableFuture.runAsync(() -> {
        // 4. 发送通知（MQ 发送，本身已是异步）
        notificationService.sendOrderCreated(order);
    }, ioPool).exceptionally(ex -> {
        log.error("发送通知失败，orderId={}", order.getId(), ex);
        // 不影响主流程，记录日志即可
        return null;
    });

    CompletableFuture.runAsync(() -> {
        // 5. 记录操作日志
        orderLogService.record(order, "CREATE");
    }, ioPool).exceptionally(ex -> {
        log.error("记录日志失败，orderId={}", order.getId(), ex);
        return null;
    });

    return OrderResult.success(order);
}
```

### 关键决策

```
1. 为什么不用 @Async 注解？
   → @Async 依赖 Spring 的线程池配置
   → 线程池参数不可控，容易踩坑
   → 显式用 CompletableFuture 更清晰

2. 为什么通知和日志要用异步？
   → 核心链路 RT = 校验(5ms) + 创建(10ms) + 扣减(5ms) = 20ms
   → 加上通知(100ms) + 日志(50ms) = 170ms
   → 异步化后核心链路保持 20ms

3. 异步任务失败了怎么办？
   → exceptionally 捕获异常，记录日志
   → 通知失败 → 可以用 MQ 的重试机制补偿
   → 日志失败 → 可以用本地消息表补偿
   → 核心原则：非核心步骤失败不能影响核心流程
```

## 实战场景四：批量数据处理

### 场景描述

```
需求：从数据库分页查询 100 万条数据，逐条调用远程服务处理

朴素做法：查 1 条 → 处理 1 条 → 100 万次串行 → 太慢

优化思路：
- 分批查询（每批 1000 条）
- 每批内并行处理（控制并发数）
- 批次间串行（保证顺序）
```

### 方案：分批 + 并行 + 限流

```java
public void processAllData(int totalRecords) {
    int batchSize = 1000;
    int totalBatches = (totalRecords + batchSize - 1) / batchSize;
    Semaphore semaphore = new Semaphore(50);  // 每批最多 50 并发

    for (int batch = 0; batch < totalBatches; batch++) {
        int offset = batch * batchSize;

        // 1. 分批查询
        List<Data> dataList = dataMapper.selectByOffset(offset, batchSize);
        if (dataList.isEmpty()) break;

        // 2. 每批内并行处理
        List<CompletableFuture<Void>> futures = new ArrayList<>();
        for (Data data : dataList) {
            futures.add(CompletableFuture.runAsync(() -> {
                try {
                    semaphore.acquire();
                    remoteService.process(data);
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    throw new RuntimeException("被中断", e);
                } finally {
                    semaphore.release();
                }
            }, ioPool));
        }

        // 3. 等待当前批次全部完成，再处理下一批
        CompletableFuture.allOf(futures.toArray(new CompletableFuture[0])).join();
        log.info("批次 {}/{} 处理完成", batch + 1, totalBatches);
    }
}
```

## 实战场景五：定时任务并发控制

### 场景描述

```
需求：每 5 分钟执行一次数据同步任务
问题：如果上次执行还没结束，新的定时任务又触发了怎么办？

风险：
- 两个实例同时操作同一批数据 → 重复处理
- 资源竞争导致数据库连接池耗尽
- 内存中积累大量未完成的任务 → OOM
```

### 方案：分布式锁 + 单线程执行

```java
@Scheduled(fixedRate = 5 * 60 * 1000)
public void syncData() {
    String lockKey = "lock:data-sync";
    String lockValue = UUID.randomUUID().toString();

    // 1. 尝试获取分布式锁（只锁 10 分钟）
    Boolean locked = redisTemplate.opsForValue()
        .setIfAbsent(lockKey, lockValue, 10, TimeUnit.MINUTES);

    if (!Boolean.TRUE.equals(locked)) {
        log.info("上一次同步尚未完成，跳过本次执行");
        return;
    }

    try {
        doSync();
    } finally {
        unlockWithLua(lockKey, lockValue);
    }
}

// 如果不需要跨实例互斥（单机部署），可以用更简单的方式：
private final AtomicBoolean running = new AtomicBoolean(false);

@Scheduled(fixedRate = 5 * 60 * 1000)
public void syncDataLocal() {
    if (!running.compareAndSet(false, true)) {
        log.info("上一次同步尚未完成，跳过");
        return;
    }
    try {
        doSync();
    } finally {
        running.set(false);
    }
}
```

## 实战场景六：并发安全的数据结构选型

### 场景描述

```
项目里经常需要用到线程安全的数据结构，但很多开发者不管场景直接用
Collections.synchronizedXXX() 或 ConcurrentHashMap，这不一定是最优选择。
```

### 选型指南

```
┌─────────────────────────────────────────────────────────────┐
│  场景                         │  推荐数据结构              │
├─────────────────────────────────────────────────────────────┤
│  高并发读写 Map                 │  ConcurrentHashMap       │
│  读多写少 Map                  │  CopyOnWriteArrayList     │
│  高并发计数器                   │  LongAdder               │
│  高并发队列（生产-消费）         │  LinkedBlockingQueue     │
│  延迟任务                      │  DelayQueue              │
│  有序并发集合                   │  ConcurrentSkipListMap   │
│  只需要计数（不需要存值）        │  AtomicInteger/LongAdder │
│  需要原子更新 Map 中的值        │  CHM.computeIfAbsent()   │
└─────────────────────────────────────────────────────────────┘
```

### 常见误用

```java
// ❌ 误用 1：用 synchronized 代替 ConcurrentHashMap
Map<String, Object> cache = Collections.synchronizedMap(new HashMap<>());
// 问题：每个操作都锁整个 Map，高并发下性能差
// 正确：ConcurrentHashMap 分段锁，高并发下性能好

// ❌ 误用 2：computeIfAbsent 里的逻辑有副作用
ConcurrentHashMap<String, List<String>> map = new ConcurrentHashMap<>();
map.computeIfAbsent("key", k -> {
    // 如果这里的逻辑涉及 I/O 或长时间操作
    // 会阻塞其他线程对同一个 key 的访问
    return fetchDataFromRemote(k);  // 危险！网络调用可能慢
});

// ✅ 正确：先获取数据，再放入 Map
List<String> data = fetchDataFromRemote("key");
map.putIfAbsent("key", data);

// ❌ 误用 3：CopyOnWriteArrayList 用于频繁修改场景
List<Event> events = new CopyOnWriteArrayList<>();
for (int i = 0; i < 10000; i++) {
    events.add(new Event());  // 每次添加都复制整个数组！
}
// CopyOnWriteArrayList 只适合读多写极少（配置列表、监听器列表）
// 频繁修改用 Collections.synchronizedList 或 ConcurrentLinkedQueue
```

## 实战场景七：线程池参数调优

### 场景描述

```
线程池的 7 个参数怎么设？这是面试和实际开发中最常被问到的问题。
```

### 参数决策框架

```
核心线程数（corePoolSize）：

  CPU 密集型：N（CPU 核心数）
    → 为什么？每个线程都需要 CPU，设多了反而因上下文切换变慢
    → Runtime.getRuntime().availableProcessors()

  I/O 密集型：2N 或 2N + 1
    → 为什么？线程在等待 I/O 时 CPU 空闲，可以多放一些线程
    → 但也不要太多，太多会导致内存压力大

  实际公式：N = CPU 核数 × (1 + W/C)
    → W = 等待时间（I/O 等待、sleep、锁等待）
    → C = 计算时间
    → W/C = 1（一半等一半算）→ N = 2 × CPU
    → W/C = 10（ mostly waiting）→ N = 11 × CPU

最大线程数（maximumPoolSize）：
  → 通常 = corePoolSize（不使用弹性扩容）
  → 弹性场景：maxPoolSize > corePoolSize + 合适的队列

工作队列（workQueue）：
  → 有界队列（ArrayBlockingQueue）：防止任务堆积导致 OOM
  → 不建议用无界队列（LinkedBlockingQueue 无参构造）
  → 队列长度：根据任务处理速度和积压容忍度设

拒绝策略（rejectedExecutionHandler）：
  → AbortPolicy（默认）：抛异常，调用方处理
  → CallerRunsPolicy：让提交任务的线程自己执行（适合降级）
  → DiscardPolicy：静默丢弃（适合日志等可丢失任务）
  → DiscardOldestPolicy：丢弃最老的（适合最新数据更重要的场景）

线程空闲时间（keepAliveTime）：
  → 非核心线程的存活时间，超过就被回收
  → 通常 60~120 秒
  → 如果 allowCoreThreadTimeOut = true，核心线程也会被回收
```

### 线程池监控

```java
// 线程池监控指标（定期采集，接入监控系统）
public Map<String, Object> monitorThreadPool(ThreadPoolExecutor pool) {
    Map<String, Object> metrics = new LinkedHashMap<>();
    metrics.put("activeCount", pool.getActiveCount());           // 正在执行任务的线程数
    metrics.put("currentPoolSize", pool.getPoolSize());         // 当前线程池大小
    metrics.put("corePoolSize", pool.getCorePoolSize());        // 核心线程数
    metrics.put("maxPoolSize", pool.getMaximumPoolSize());      // 最大线程数
    metrics.put("queueSize", pool.getQueue().size());           // 队列中等待的任务数
    metrics.put("completedTaskCount", pool.getCompletedTaskCount());  // 已完成任务数
    metrics.put("totalTaskCount", pool.getTaskCount());         // 总提交任务数
    metrics.put("rejectedCount", rejectedCounter.get());        // 被拒绝的任务数
    return metrics;
}

// 告警规则：
// activeCount == maxPoolSize 持续 30 秒 → 线程池满载告警
// queueSize > maxQueueSize * 0.8 → 队列积压告警
// rejectedCount > 0 → 拒绝策略触发告警
```

## 实战场景八：优雅停机

### 场景描述

```
服务发布/重启时，如果直接 kill 进程：
- 正在处理的请求会被中断 → 返回 500 给用户
- 线程池中的任务丢失 → 数据不一致
- 数据库连接未释放 → 连接池泄漏

优雅停机：让正在处理的请求完成，线程池中的任务执行完毕，再关闭进程
```

### 方案：Spring Boot 优雅停机

```yaml
# application.yml
server:
  shutdown: graceful    # 开启优雅停机

spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s  # 最多等 30 秒
```

```java
// 自定义线程池的优雅关闭
@Configuration
public class ThreadPoolConfig {

    @Bean(destroyMethod = "shutdown")
    public ExecutorService bizThreadPool() {
        return new ThreadPoolExecutor(
            10, 50, 60, TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(1000),
            new ThreadFactoryBuilder().setNameFormat("biz-pool-%d").build(),
            new ThreadPoolExecutor.CallerRunsPolicy()
        );
    }
}

// 如果需要在停机时执行清理逻辑：
@PreDestroy
public void cleanup() {
    // 1. 停止接收新请求（Spring Boot graceful shutdown 已处理）
    // 2. 等待正在处理的请求完成
    // 3. 释放资源
    log.info("服务开始优雅停机...");
}
```

### 优雅停机的并发要点

```
1. 为什么需要优雅停机？
   → 保护用户体验（正在处理的请求不要被中断）
   → 保护数据一致性（任务执行到一半不要丢弃）
   → 保护资源（数据库连接、文件句柄要正确释放）

2. shutdown() vs shutdownNow()
   → shutdown()：不再接受新任务，等待已提交的任务完成
   → shutdownNow()：尝试中断所有正在执行的任务，返回未开始的任务列表
   → 优先用 shutdown()，更安全

3. 等待任务完成的超时设置
   → 不能无限等待（进程可能永远关不掉）
   → 超时后用 shutdownNow() 强制关闭
   → 记录哪些任务未完成，方便后续补偿

4. 和消息队列的配合
   → 优雅停机前，暂停消费者（停止拉取新消息）
   → 等待正在处理的消息完成
   → 向 MQ 发送确认（ACK）
   → 最后关闭消费者
```

## 和相似方案的区别

## 正确使用方式

### 并发编码十条军规

```
1. 优先使用并发容器，而不是 synchronized 包装
2. 锁的粒度尽量小，持锁时间尽量短
3. 线程池参数要明确指定，不要用 Executors 快速方法
4. 所有阻塞操作都要设超时
5. acquire/release、countDown 必须在 finally 中
6. volatile 不能替代锁，它只保证可见性，不保证原子性
7. 异常不能吞掉，异步任务的异常要显式处理
8. 共享可变状态越少越好，优先用不可变对象
9. 优雅停机是必修课，不是可选项
10. 并发代码必须写单元测试 + 压测，靠"看"看不出 bug
```

## 边界情况和坑

### 生产环境常见并发 Bug

```
Bug 1：SimpleDateFormat 线程不安全
  → 多线程共享一个 SimpleDateFormat → 日期格式化结果错乱
  → 解决：用 DateTimeFormatter（线程安全）或 ThreadLocal

Bug 2：HashMap 并发 put 导致死循环（JDK 7）
  → 多线程同时扩容 → 链表成环 → get() 死循环
  → JDK 8 用红黑树解决了，但并发场景仍要用 ConcurrentHashMap

Bug 3：HashSet 的 add() 并发不安全
  → 多线程同时 add → 元素丢失（数组越界覆盖）
  → 解决：ConcurrentHashMap.newKeySet() 或 Collections.synchronizedSet()

Bug 4：ArrayList 的 add() 并发不安全
  → 多线程同时 add → 数组越界（size 没加够但实际元素已写入）
  → 解决：CopyOnWriteArrayList（读多写少）或 Collections.synchronizedList()

Bug 5：Boolean 的 ++ 操作不是原子的
  → count++ 实际是 read + modify + write 三步
  → 解决：AtomicInteger.incrementAndGet()

Bug 6：double-checked locking 忘记 volatile
  → 可能返回未完全初始化的对象
  → 解决：单例的 instance 声明加 volatile
```

### 线上问题排查思路

```
1. CPU 100%
   → jstack 看线程转储，哪个线程一直在 RUNNING
   → 常见原因：死循环、GC 频繁、锁竞争

2. 响应时间突然变长
   → jstack 看大量线程是否在 WAITING/BLOCKED
   → 常见原因：锁竞争、线程池满、数据库慢查询、外部服务超时

3. OOM
   → -XX:+HeapDumpOnOutOfMemoryError 生成堆转储
   → 用 MAT 或 VisualVM 分析
   → 常见原因：ThreadLocal 泄漏、线程池任务堆积、内存缓存没有上限

4. 线程数异常增长
   → jstack 统计线程数和状态分布
   → 常见原因：线程泄漏（创建了没回收）、连接池泄漏、定时任务堆积
```

## 我的理解（面试考点）

### 高频面试题

```
Q1: 生产环境怎么处理接口聚合查询？
→ CompletableFuture 并行调用 + allOf 统一超时
→ exceptionally/handle 处理单接口异常
→ getNow + fallback 处理超时接口
→ 强调：不要用串行、不要用 commonPool

Q2: 线程池参数怎么设？
→ CPU 密集型：N（CPU 核数）
→ I/O 密集型：2N 或 N × (1 + W/C)
→ 队列：有界队列，长度根据任务处理速度设
→ 拒绝策略：CallerRunsPolicy（降级）或自定义
→ 强调：不用 Executors 快速方法（有 OOM 风险）

Q3: 如何保证缓存和数据库一致性？
→ 先更新 DB 再删缓存（推荐）或 延迟双删
→ 分布式锁防止缓存击穿
→ MQ 异步更新缓存保证最终一致性
→ 强调：没有强一致方案，只有最终一致

Q4: 线上并发问题怎么排查？
→ CPU 100% → jstack 看线程状态
→ 响应变慢 → 看线程池指标（activeCount、queueSize）
→ OOM → HeapDump + MAT 分析
→ 强调：要有监控体系（线程池指标、GC 日志、APM 链路追踪）

Q5: 聊聊你理解的并发编程最佳实践？
→ 能说出上面的"十条军规"中的 5 条以上
→ 结合实际项目举例（接口聚合、缓存更新、异步通知）
→ 承认并发 bug 难复现，强调"防御性编码"
```

### 一句话总结

```
并发编程的实战核心就三条：
1. 找到正确的工具（CHM 还是 synchronized？Future 还是虚拟线程？）
2. 控制好边界（超时、队列大小、线程数上限）
3. 做好防御（优雅停机、异常处理、监控告警）

理论决定上限，实战决定下限。
能说出原理是 P6，能把坑踩过并总结出来是 P7。
```
