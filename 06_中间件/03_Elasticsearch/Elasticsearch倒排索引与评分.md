# Elasticsearch 倒排索引与评分

## 这个问题为什么存在？

数据库的 B+ 树索引是给"精确查询"设计的——比如 `WHERE id = 1`、`WHERE age > 20`。它解决的问题是"快速定位到某些行"。

但当你要搜索"文章内容包含'分布式事务'"时，问题变了：你不知道哪些文档包含这个词，你需要**在所有文档的所有词里，找到包含'分布式事务'的那些**。

B+ 树做不了这个——它只能按"字段值"找，不能按"字段内容里是否包含某个词"找。

这就是**倒排索引（Inverted Index）** 要解决的问题：

> 正排索引：文档 → 它包含哪些词
> 倒排索引：词 → 哪些文档包含它

同时，搜索不只是"找到包含词的文档"，还要**按相关性排序**（最相关的排最前面）。这需要一套**评分算法**（BM25 / TF-IDF）。

## 它是怎么解决问题的？

### 倒排索引结构

倒排索引由两部分组成：**词项字典（Term Dictionary）** + **倒排列表（Postings List）**。

```
正排（文档 → 词）:
  Doc-1: "Elasticsearch 是一个分布式搜索引擎"
  Doc-2: "分布式系统需要一致性协议"

倒排（词 → 文档）:
  "Elasticsearch" → [Doc-1]
  "分布式"         → [Doc-1, Doc-2]
  "搜索引擎"        → [Doc-1]
  "系统"           → [Doc-2]
  "一致性"          → [Doc-2]
```

**搜索"分布式"**：直接查倒排索引，找到 `[Doc-1, Doc-2]`，O(1) 时间。

#### 词项字典（Term Dictionary）

所有词项（Term）按**字典序排序**存储，配合 **FST（Finite State Transducer，有限状态传感器）** 压缩存储，支持快速查找。

```
FST 压缩示例：
  存储 "abc", "abd", "acd" 三个词
  不压缩: 每个词独立存储，有重复前缀
  FST:    共享前缀 "ab"，只存一次，大幅减少内存占用
```

**为什么用 FST？** 倒排索引的词项字典要放在内存里（保证查询速度），FST 能做到**比原始文本小 2-5 倍**的压缩比，同时支持 O(len) 的查找速度。

#### 倒排列表（Postings List）

每个词项对应一个倒排列表，记录包含该词项的文档 ID 列表，以及**词频、位置**等信息。

```
Term: "分布式"
Postings:
  [DocID=1, TF=3, Positions=[5, 18, 42],   ← Doc-1 里"分布式"出现3次
   DocID=5, TF=1, Positions=[12],            ← Doc-5 里出现1次
   ...]
```

**为什么存位置？** 支持**短语搜索**（Phrase Query）。搜索 `"分布式 事务"` 时，需要两个词在文档里的位置是相邻的，这需要位置信息才能判断。

**压缩**：倒排列表通常很大（一个常见词可能对应百万级 DocID），Lucene 使用 **Roaring Bitmap** 或 **Frame Of Reference (FOR)** 编码压缩，可以做到把 1 百万个 DocID 压缩到几 MB。

### BM25 评分算法（默认相似度算法）

Elasticsearch 5.x 之后，默认用 **BM25**（Best Matching 25）替代了旧的 TF-IDF。

#### TF-IDF 的问题

**TF（Term Frequency）**：词在文档里出现的次数越多，越相关。
**IDF（Inverse Document Frequency）**：词越罕见（出现在越少文档里），越能代表文档的主题。

**TF-IDF 的问题**：TF 没有上限。如果一个词在文档里出现 100 次，TF=100，评分会被这个文档"刷爆"，哪怕这个词只是堆砌的。

#### BM25 的改进

BM25 对 TF 做了**饱和处理**（Sigmoid 曲线）：

```
TF 贡献 = (TF * (k1 + 1)) / (TF + k1)

k1 默认 1.2：
  TF=1   → 贡献 ≈ 0.85
  TF=5   → 贡献 ≈ 0.98
  TF=100 → 贡献 ≈ 0.99  （已经饱和，再增加 TF 也没用）
```

**核心思想**：一个词出现 1 次和出现 5 次，相关性差别很大；但出现 5 次和出现 100 次，差别不大。BM25 通过饱和函数表达了这个直觉。

**IDF 部分**（和 TF-IDF 类似）：

```
IDF = log((N - n + 0.5) / (n + 0.5) + 1)

N = 总文档数
n = 包含该词的文档数
```

**n 越小（词越罕见），IDF 越大**，评分越高。

**字段长度归一化**：

```
Norm = 1 / sqrt(numFieldTerms)

文档越短（词越少），归一化值越大，评分越高。
直观理解：同样出现"分布式"，在 10 个词的标题里比在 1000 个词的正文里更有意义。
```

### 源码关键路径：Lucene 搜索流程

```java
// Lucene 搜索核心流程（简化）
IndexSearcher searcher = new IndexSearcher(indexReader);

// 1. 解析查询（Query Parser）
Query query = parser.parse("分布式事务");

// 2. 创建 Weight（计算 IDF 等静态分数）
Weight weight = searcher.createWeight(query, ScoreMode.TOP_SCORES, 1.0f);

// 3. 创建 Scorer（遍历倒排列表，计算 TF + 字段长度）
Scorer scorer = weight.scorer(reader.getContext());

// 4. 遍历匹配文档，计算 BM25 评分
while (scorer.iterator().nextDoc() != DocIdSetIterator.NO_MORE_DOCS) {
    float score = scorer.score();  // BM25 评分
    // ...
}
```

**关键优化**：Lucene 使用 **Skip List** 加速多 Term 查询（比如 `分布式 AND 事务`，需要对两个倒排列表做"与"操作，Skip List 可以跳过不可能的 DocID）。

## 它和相似方案的本质区别是什么？

### 倒排索引 vs B+ 树索引

| 维度 | B+ 树（MySQL） | 倒排索引（ES/Lucene） |
|------|----------------|----------------------|
| 适合查询 | 精确匹配、范围查询 | 全文搜索、模糊匹配 |
| 索引结构 | 树形，按字段值排序 | 词项字典 + 倒排列表 |
| 模糊搜索 | `LIKE '%xx%'` 全表扫描 | 直接查倒排索引，O(1) |
| 相关性排序 | 不支持 | 支持（BM25 / TF-IDF） |

**本质区别**：B+ 树解决的问题是"快速找到某个值对应的行"，倒排索引解决的问题是"快速找到包含某个词的所有文档 + 按相关性排序"。两者适用场景完全不同。

### BM25 vs TF-IDF

| 维度 | TF-IDF | BM25 |
|------|--------|-------|
| TF 处理 | 线性增长（无上限） | 饱和函数（有上限） |
| 字段长度归一化 | 简单除以词数 | 更复杂的归一化（b 参数可调） |
| 默认算法（ES版本） | 2.x 及之前 | 5.x 及之后 |

**为什么选 BM25 不选 TF-IDF？** TF-IDF 的 TF 线性增长容易被"词堆砌"攻击（SEO 垃圾文档），BM25 的饱和函数更合理。

## 正确使用方式

### 正确用法

**1. 使用 `match` 查询（会用倒排索引 + BM25 评分）**

```json
GET /books/_search
{
  "query": {
    "match": {
      "content": "分布式事务"
    }
  }
}
// ES 会自动：
// 1. 对 "分布式事务" 做分词 → ["分布式", "事务"]
// 2. 查倒排索引，找到包含这些词的文档
// 3. 用 BM25 算法计算相关性评分
// 4. 按评分降序返回
```

**为什么正确**：`match` 查询是 ES 的**全文搜索标准查询**，会自动分词、使用倒排索引、计算 BM25 评分。

**错误对比**：`term` 查询是**精确匹配**（不做分词，直接找倒排索引里有没有这个词），用于 keyword 类型字段；对 text 类型字段用 `term` 查询，大概率查不到结果（因为 text 字段存的是分词后的词项，不是原始文本）。

**2. 控制评分：Boost 特定字段**

```json
GET /books/_search
{
  "query": {
    "multi_match": {
      "query": "Elasticsearch",
      "fields": ["title^3", "content"]  // title 的匹配权重 ×3
    }
  }
}
```

**为什么正确**：用户在标题里搜到"Elasticsearch"比在正文中搜到更有意义，应该给标题更高的权重。

**3. 使用 `match_phrase` 做短语搜索**

```json
GET /books/_search
{
  "query": {
    "match_phrase": {
      "content": "分布式事务"
    }
  }
}
// 要求："分布式" 和 "事务" 在文档中位置相邻（slop 参数可控制允许间隔几个词）
```

**为什么正确**：`match` 查询只要求"包含这些词"，不要求顺序和相邻；`match_phrase` 要求词的顺序和相邻性，适合"精确短语"搜索。

### 错误用法及后果

**错误1：对 text 字段用 `term` 查询**

```json
GET /books/_search
{
  "query": {
    "term": {
      "content": "分布式事务"  // content 是 text 类型，存的是分词后的词项
    }                           // "分布式事务" 作为一个整体，倒排索引里大概率没有
  }
}
```

**后果**：查不到结果（或结果极少）。

**修复**：text 字段用 `match` 查询；如果要精确匹配，把字段改为 `keyword` 类型，或用 `.keyword` 子字段。

**错误2：不使用分页，一次返回太多结果**

```json
GET /books/_search
{
  "query": { "match": { "content": "搜索" } },
  "size": 10000  // 一次返回 10000 条
}
```

**后果**：ES 要从各 Shard 取 10000 条数据，协调节点要汇总排序，内存压力极大。

**修复**：用 `from + size` 做浅分页（< 10000），深分页用 `search_after`。

## 边界情况和坑

### 坑1：分词器选择不当，搜索结果不准

**场景**：默认标准分词器（standard analyzer）对中文按字分词。

```
"我爱中国" → ["我", "爱", "中", "国"]

搜索 "中国" → ["中", "国"] → 能匹配，但精度差（"中" 和 "国" 分开匹配，可能匹配到不相关的文档）
```

**修复**：用 IK 分词器（`ik_max_word` 最细粒度，`ik_smart` 最粗粒度）。

```json
PUT /books
{
  "mappings": {
    "properties": {
      "content": {
        "type": "text",
        "analyzer": "ik_max_word",
        "search_analyzer": "ik_smart"  // 搜索时用粗粒度（提高召回率）
      }
    }
  }
}
```

### 坑2：评分被"短文档"带偏

**现象**：搜索结果里，很多很短的文档（比如只有标题）排在很长的内容文档前面。

**原因**：BM25 的字段长度归一化，短文档有优势。

**修复**：调整 `b` 参数（归一化强度），或自定义 Similarity：

```json
PUT /books
{
  "settings": {
    "similarity": {
      "my_bm25": {
        "type": "BM25",
        "b": 0.5  // 默认 0.75，调小 = 降低短文档优势
      }
    }
  },
  "mappings": {
    "properties": {
      "content": {
        "type": "text",
        "similarity": "my_bm25"
      }
    }
  }
}
```

### 坑3：倒排列表太大，内存不够

**场景**：一个非常常见的词（比如"的"），倒排列表可能有几百万个 DocID，占内存几十 MB。

**后果**：查询时加载倒排列表到内存，可能导致 OOM 或 GC 停顿。

**修复**：
- 去掉停用词（stop words）："的"、"是"、"在" 等无意义词不索引
- 用 IK 分词器时配置停用词词典

## 面试话术

**Q：倒排索引是什么？**
"倒排索引是 Elasticsearch/Lucene 的核心数据结构。正排是'文档包含哪些词'，倒排是'词在哪些文档里'。搜索时直接查倒排索引，O(1) 找到包含某个词的所有文档。倒排索引由词项字典（用 FST 压缩存在内存里）和倒排列表（用 FOR 或 Roaring Bitmap 压缩存在磁盘上）组成。"

**Q：ES 的默认评分算法是什么？**
"BM25（Best Matching 25），替代了旧的 TF-IDF。核心改进是对 TF 做了饱和处理——一个词出现 1 次和 5 次差别很大，但 5 次和 100 次差别不大，BM25 用饱和函数表达了这个直觉。同时还考虑了 IDF（词越罕见越重要）和字段长度归一化（短文档的词权重更高）。"

**Q：match 和 term 查询的区别？**
"match 是全文搜索查询，会做分词、查倒排索引、计算 BM25 评分，用于 text 类型字段。term 是精确匹配查询，不做分词，直接找倒排索引里有没有精确的词项，用于 keyword 类型字段。对 text 字段用 term 查询基本查不到结果。"

**Q：怎么优化 ES 的搜索性能？**
"几个方向：1) 合理设置 Shard 数（单个 Shard 10GB~50GB）；2) 不要用通配符在前面的 wildcard 查询（无法用倒排索引）；3) 深度分页用 search_after 替代 from+size；4) 热数据用 SSD，JVM 堆内存不超过 50% 留给文件系统缓存。"

## 本文总结

倒排索引是 ES 的核心：词项字典（FST 压缩，放内存） + 倒排列表（FOR/Roaring Bitmap 压缩，放磁盘）。

BM25 是默认评分算法：TF（饱和函数，防止词堆砌） + IDF（稀有词权重高） + 字段长度归一化（短文档优势）。

**正确使用**：text 字段用 `match` 查询（全文搜索），keyword 字段用 `term` 查询（精确匹配）。中文必须用 IK 分词器。

**主要坑**：分词器选择不当（中文必须用 IK）、对 text 用 term 查询（查不到）、深度分页（用 search_after）。

倒排列表压缩是 ES 能处理海量数据的关键——不压缩的话，倒排索引会比原始文本还大。
