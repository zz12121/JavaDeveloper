# 垃圾回收

## 这个问题为什么存在？

> 从「没有这个东西会怎样」出发，解释问题的根源。

在没有垃圾回收（Garbage Collection, GC）的语言中（如C/C++），开发者需要手动管理内存：

```cpp
// C++ 中手动管理内存
void process() {
    Object* obj = new Object();  // 分配内存
    // 使用obj...
    delete obj;  // 必须手动释放，否则内存泄漏
}
```

**痛点**：
1. **内存泄漏**：忘记释放内存，导致可用内存越来越少
2. **悬空指针**：释放后继续使用指针，导致程序崩溃
3. **双重释放**：同一块内存释放两次，导致堆损坏
4. **内存碎片**：频繁分配和释放不同大小的对象，导致内存碎片化

**核心问题**：如何让开发者**专注于业务逻辑**，而不用担心内存管理？如何自动识别并回收**不再使用的对象**？

垃圾回收的本质是**自动内存管理**：通过可达性分析，找到不再被引用的对象，并回收其占用的内存。

## 它是怎么解决问题的？

> 不止讲「是什么」，要讲清楚「为什么这样设计能解决问题」。

### 核心机制

#### 1. 判断对象是否存活的算法

**引用计数算法（Reference Counting）**：
```
每个对象有一个引用计数器：
- 被引用时 +1
- 引用失效时 -1
- 计数器为 0 时，对象可被回收
```

**优点**：实现简单，判定效率高。
**致命缺陷**：**无法解决循环引用问题**。
```java
// 循环引用问题
class Node {
    Node next;
}

Node a = new Node();
Node b = new Node();
a.next = b;
b.next = a;  // a 和 b 互相引用

a = null;
b = null;
// 虽然 a 和 b 已经无法被访问，但引用计数不为 0，无法回收
```

**可达性分析算法（Reachability Analysis）**：
```
从 GC Roots 开始，向下搜索引用链：
- 能被 GC Roots 到达的对象：存活
- 无法被 GC Roots 到达的对象：可回收
```

**GC Roots 包括**：
1. 虚拟机栈（栈帧中的局部变量表）中引用的对象
2. 方法区中类静态属性引用的对象
3. 方法区中常量引用的对象
4. 本地方法栈中 JNI（Native 方法）引用的对象

**为什么 Java 选择可达性分析而非引用计数？**
- 引用计数无法解决循环引用
- 可达性分析可以准确识别所有存活对象

#### 2. 垃圾收集算法

**标记-清除算法（Mark-Sweep）**：
```
阶段1：标记（Mark）
  - 从 GC Roots 开始，标记所有可达对象

阶段2：清除（Sweep）
  - 遍历整个堆，回收未被标记的对象
```

**优点**：实现简单。
**缺点**：
1. **效率不高**：需要两次遍历整个堆
2. **内存碎片**：清除后产生大量不连续的内存碎片

**复制算法（Copying）**：
```
将内存分为两块（From 和 To）：
1. 只使用 From 区
2. GC 时，将 From 区中存活的对象复制到 To 区
3. 清空 From 区
4. 交换 From 和 To 的角色
```

**优点**：
1. **无内存碎片**：复制后内存连续
2. **效率高**：只需要遍历存活对象

**缺点**：
1. **内存浪费**：需要预留 To 区，实际使用内存减半
2. **存活对象多时效率低**：需要复制大量对象

**为什么新生代使用复制算法？**
- **弱分代假说**：绝大多数对象都是朝生夕死的
- 新生代中存活对象通常很少（<10%），复制成本低

**标记-整理算法（Mark-Compact）**：
```
阶段1：标记（Mark）
  - 同标记-清除算法

阶段2：整理（Compact）
  - 将存活对象向一端移动
  - 清理边界以外的内存
```

**优点**：无内存碎片。
**缺点**：需要移动对象，效率较低。

**为什么老年代使用标记-整理或标记-清除？**
- 老年代中存活对象多，复制成本高
- 标记-整理可以避免内存碎片

#### 3. 分代收集理论

**弱分代假说（Weak Generational Hypothesis）**：
1. 绝大多数对象都是朝生夕死的
2. 熬过越多次垃圾收集的对象越难以消亡

**基于分代假说的设计**：
- **新生代（Young Generation）**：对象存活率低，适合复制算法
- **老年代（Old Generation）**：对象存活率高，适合标记-整理或标记-清除

**分代收集的好处**：
- 针对不同年龄的对象采用不同的收集策略
- 提高 GC 效率，减少 STW（Stop-The-World）时间

#### 4. 垃圾收集器分类

**按线程数分**：
- **串行收集器（Serial）**：单线程收集
- **并行收集器（Parallel）**：多线程收集

**按工作模式分**：
- **并发收集器（Concurrent）**：与应用线程并发执行
- **独占收集器（STW）**：暂停所有应用线程

**按碎片处理方式分**：
- **压缩收集器**：整理内存，无碎片
- **非压缩收集器**：不整理内存，有碎片风险

### 源码关键路径

#### Object.finalize() 方法

```java
// java.lang.Object
public class Object {
    protected void finalize() throws Throwable {
        // 对象被回收前，JVM 会调用这个方法
        // 可以在这里做资源释放
    }
}
```

**为什么不推荐使用 finalize()？**
1. **执行时机不确定**：无法保证何时执行
2. **性能差**：JVM 需要特殊处理
3. **可能导致对象复活**：在 finalize() 中重新建立引用

**替代方案**：使用 `Cleaner` 或 `PhantomReference`。

#### 垃圾回收的触发条件

**新生代 GC（Minor GC）触发条件**：
- 新生代（Eden 区）空间不足时

**老年代 GC（Major GC / Full GC）触发条件**：
1. 老年代空间不足
2. 方法区（元空间）空间不足
3. 通过 Minor GC 后进入老年代的对象大小 > 老年代剩余空间
4. 调用 `System.gc()`（只是建议，不一定执行）

## 它和相似方案的本质区别是什么？

> 对比不是列表，要解释「为什么选 A 不选 B」。

### 引用计数 vs 可达性分析

| 维度 | 引用计数 | 可达性分析 |
|------|----------|------------|
| **判断方式** | 计数器是否为 0 | 是否可达 GC Roots |
| **循环引用** | 无法处理 | 可以处理 |
| **效率** | 高（即时回收） | 低（需要遍历） |
| **实现复杂度** | 低 | 高 |

**为什么选可达性分析？**
- 引用计数无法解决循环引用问题
- Java 中对象之间的循环引用很常见

### 标记-清除 vs 复制 vs 标记-整理

| 维度 | 标记-清除 | 复制 | 标记-整理 |
|------|----------|------|----------|
| **效率** | 中等 | 高（存活对象少时） | 低 |
| **内存碎片** | 有 | 无 | 无 |
| **内存利用率** | 100% | 50% | 100% |
| **适用场景** | 老年代 | 新生代 | 老年代 |

**为什么新生代用复制，老年代用标记-整理？**
- 新生代存活对象少，复制成本低
- 老年代存活对象多，复制成本高

## 正确使用方式

> 不是 API 手册，要解释「为什么这样用是对的」。

### 1. 避免内存泄漏

```java
// ❌ 错误：静态集合持有对象引用
public class MemoryLeak {
    private static List<Object> list = new ArrayList<>();

    public void add(Object obj) {
        list.add(obj);  // 对象永远不会被回收
    }
}

// ✅ 正确：及时清理
public class Safe {
    private List<Object> list = new ArrayList<>();

    public void add(Object obj) {
        list.add(obj);
    }

    public void clear() {
        list.clear();  // 释放引用
    }
}
```

**为什么这样用是对的？**
- 静态集合的生命周期与 JVM 一样长
- 及时清理引用，让垃圾收集器可以回收对象

### 2. 使用软引用和弱引用

```java
// 软引用：内存不足时会被回收
SoftReference<byte[]> softRef = new SoftReference<>(new byte[1024 * 1024]);
byte[] data = softRef.get();  // 可能返回 null（已被回收）

// 弱引用：GC 时一定会被回收
WeakReference<Object> weakRef = new WeakReference<>(new Object());
Object obj = weakRef.get();  // 可能返回 null（已被回收）
```

**为什么用软引用缓存？**
- 内存敏感的场景（如图片缓存）
- 内存不足时自动释放，避免 OOM

### 3. 避免频繁创建大对象

```java
// ❌ 错误：在循环中创建大对象
public void process() {
    for (int i = 0; i < 100000; i++) {
        byte[] data = new byte[1024 * 1024];  // 每次循环创建 1MB 对象
        // ...
    }
}

// ✅ 正确：复用对象
public void process() {
    byte[] data = new byte[1024 * 1024];
    for (int i = 0; i < 100000; i++) {
        // 复用 data
    }
}
```

**为什么这样用？**
- 频繁创建大对象会触发频繁 GC
- 复用对象可以减少 GC 压力

### 4. 选择合适的垃圾收集器

```bash
# 1. 串行收集器（单线程，适合客户端模式）
-XX:+UseSerialGC

# 2. 并行收集器（多线程，适合吞吐量优先）
-XX:+UseParallelGC

# 3. CMS 收集器（并发，适合低延迟）
-XX:+UseConcMarkSweepGC

# 4. G1 收集器（JDK 9+ 默认，适合大堆、低延迟）
-XX:+UseG1GC

# 5. ZGC（JDK 11+，超低延迟）
-XX:+UseZGC
```

**为什么这样选择？**
- **吞吐量优先**：Parallel GC
- **低延迟优先**：CMS、G1、ZGC
- **小堆**：Serial GC
- **大堆（> 4GB）**：G1、ZGC

## 边界情况和坑

> 不是列举，要解释「坑的成因」。

### 坑 1：过多的临时对象导致频繁 GC

```java
// 问题代码
public String process(List<String> list) {
    String result = "";
    for (String s : list) {
        result += s;  // 每次循环创建新的 String 对象
    }
    return result;
}

// 导致：大量临时对象，频繁触发 GC
```

**成因**：
- `String` 是不可变类，每次拼接都会创建新对象
- 大量临时对象会增加 GC 压力

**解决方案**：使用 `StringBuilder`。

### 坑 2：误解 `System.gc()`

```java
// 错误理解
System.gc();  // 以为一定会触发 GC

// 实际情况
// System.gc() 只是"建议" JVM 执行 GC，不一定会执行
// 而且即使执行，也不保证能回收所有垃圾
```

**成因**：
- `System.gc()` 只是建议，不是强制命令
- JVM 有自己的 GC 策略

**解决方案**：不要依赖 `System.gc()`，让 JVM 自己管理。

### 坑 3：finalize() 导致对象复活

```java
public class Resurrect {
    private static Resurrect saved;

    @Override
    protected void finalize() {
        saved = this;  // 在 finalize() 中复活对象
    }
}
```

**成因**：
- `finalize()` 中可以让对象重新被引用
- 导致对象"死而复生"，违反 GC 的初衷

**解决方案**：不要依赖 `finalize()`，使用 `Cleaner` 或 `PhantomReference`。

### 坑 4：误用强引用导致内存泄漏

```java
// 问题代码
public class Cache {
    private Map<String, Object> cache = new HashMap<>();

    public void put(String key, Object value) {
        cache.put(key, value);  // 强引用，对象永远不会被回收
    }
}

// 导致：缓存无限增长，最终 OOM
```

**成因**：
- 强引用不会被 GC 回收
- 缓存中的对象永远无法被回收

**解决方案**：使用 `WeakHashMap` 或设置缓存过期时间。

### 坑 5：G1 GC 的 Mixed GC 触发条件复杂

```bash
# G1 GC 的 Mixed GC 触发条件
# - 老年代占用超过 InitiatingHeapOccupancyPercent（默认 45%）
# - 需要回收的老年代 Region 数量足够多

# 如果设置不当，可能导致：
# 1. Mixed GC 迟迟不触发 → Full GC
# 2. Mixed GC 触发太频繁 → 影响吞吐量
```

**成因**：
- G1 GC 的调优参数很多，设置不当会导致性能问题

**解决方案**：
- 监控 GC 日志，调整 `-XX:InitiatingHeapOccupancyPercent`
- 使用 `-XX:MaxGCPauseMillis` 设置目标暂停时间

## 我的理解（可选）

> 用自己的话重新表述一遍，检验是否真正理解。

垃圾回收的本质是**自动内存管理**，它通过**可达性分析**识别不再被使用的对象，并回收其占用的内存。

**核心思想**：
- **分代收集**：根据对象的存活周期，将堆分为新生代和老年代
- **不同算法**：新生代用复制算法，老年代用标记-整理或标记-清除

**为什么需要 GC？**
- 避免手动管理内存的痛点（内存泄漏、悬空指针等）
- 让开发者专注于业务逻辑

**最容易被忽略的点**：
- `System.gc()` 只是建议，不是强制
- 静态集合、缓存等容易导致内存泄漏
- 选择合适的垃圾收集器对性能影响巨大

## 面试话术总结

1. **如何判断对象是否可以被回收？**
   "Java 使用可达性分析算法。从 GC Roots 开始向下搜索，能被 GC Roots 到达的对象是存活的，无法被到达的对象是可回收的。GC Roots 包括：虚拟机栈中引用的对象、方法区中静态属性引用的对象、方法区中常量引用的对象、本地方法栈中 JNI 引用的对象。"

2. **垃圾收集算法有哪些？**
   "主要有三种：1）标记-清除算法：标记存活对象，清除未被标记的对象，缺点是会产生内存碎片；2）复制算法：将内存分为两块，将存活对象复制到另一块，优点是效率高、无碎片，缺点是内存利用率低；3）标记-整理算法：标记存活对象，将存活对象向一端移动，优点是无碎片，缺点是效率较低。"

3. **为什么要分代收集？**
   "根据弱分代假说，绝大多数对象都是朝生夕死的，熬过越多次 GC 的对象越难以消亡。分代收集将堆分为新生代和老年代，针对不同年龄的对象采用不同的收集策略：新生代使用复制算法（存活对象少），老年代使用标记-整理或标记-清除算法（存活对象多）。"

4. **G1 收集器有什么特点？**
   "G1 是 JDK 9+ 的默认收集器，特点：1）面向大堆（> 4GB）；2）可预测的停顿时间；3）将堆分为多个 Region，避免全堆扫描；4）支持并发收集，减少 STW 时间。"

5. **如何避免内存泄漏？**
   "1）及时清理静态集合、缓存中的引用；2）使用软引用或弱引用；3）避免在循环中创建大量临时对象；4）使用 `ThreadLocal` 后及时调用 `remove()`；5）选择合适的垃圾收集器，并监控 GC 日志。"
