# 负载均衡策略

## 这个问题为什么存在？

> 微服务有多个实例（如订单服务有3个实例），服务消费者需要选一个实例调用。问题是：**怎么选，才能最大化吞吐、最小化延迟？**

没有负载均衡，你会遇到：
- 所有请求都打到第一个实例 → 热点，其他实例闲置
- 某个实例故障了，还往它那里发请求 → 调用失败
- 实例性能不同（4核 vs 8核），但分配流量一样 → 资源浪费

---

## 它是怎么解决问题的？

### 七种核心算法

```
算法1：轮询（Round Robin）— 按顺序轮流分配
算法2：随机（Random）— 随机选一个
算法3：加权轮询（Weighted Round Robin）— 性能好的多分配
算法4：加权随机（Weighted Random）— 性能好的随机概率高
算法5：一致性Hash（IP Hash）— 同一IP总是打到同一实例（会话保持）
算法6：最少连接数（Least Connections）— 选当前连接数最少的
算法7：自适应（Zone Avoidance）— 过滤性能差的Zone，再轮询
```

#### 1. 轮询（Round Robin）

```java
// 轮询：按顺序分配
public class RoundRobinLoadBalancer {
    private final List<String> instances;
    private AtomicInteger index = new AtomicInteger(0);

    public String choose() {
        int idx = index.getAndIncrement() % instances.size();
        return instances.get(idx);
    }
}
```

**优点**：简单，每个实例均匀分配
**缺点**：实例性能不同，但分配一样；某实例故障，还会继续分配

#### 2. 随机（Random）

```java
// 随机：随便选一个
public String chooseRandom() {
    int idx = ThreadLocalRandom.current().nextInt(instances.size());
    return instances.get(idx);
}
```

**优点**：简单，和轮询差不多
**缺点**：和轮询一样，没有权重，没有故障检测

#### 3. 加权轮询（Weighted Round Robin）

```
实例A（权重10）  → 10个请求
实例B（权重5）   → 5个请求
实例C（权重5）   → 5个请求
```

```java
// 加权轮询：按权重分配
public class WeightedRoundRobin {
    private final Map<String, Integer> weights;
    private final AtomicInteger currentWeight = new AtomicInteger(0);

    public String choose() {
        // 简化版：按权重比例，循环分配
        int totalWeight = weights.values().stream().mapToInt(i -> i).sum();
        int idx = currentWeight.getAndIncrement() % totalWeight;

        // 找到对应权重的实例（略）
        return findInstanceByWeight(idx);
    }
}
```

#### 4. 一致性Hash（IP Hash）

```
客户端IP → Hash → 固定打到某个实例
好处：同一个用户的请求，总是到同一个实例（会话保持）
```

```java
// 一致性Hash：相同IP总是选同一个实例
public String chooseByIp(String clientIp) {
    int hash = Math.abs(clientIp.hashCode());
    int idx = hash % instances.size();
    return instances.get(idx);
}
```

**问题**：实例扩容时，大量IP会重新Hash，缓存失效。

**解决方案**：一致性Hash环（康瑟树），但这里不展开。

## 它和相似方案的本质区别是什么？

| 算法 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| 轮询 | 简单，均匀 | 不感知性能、故障 | 实例性能一致，无故障 |
| 随机 | 简单，均匀 | 不感知性能、故障 | 实例性能一致，无故障 |
| 加权轮询 | 考虑性能差异 | 权重静态，不感知实时负载 | 实例性能差异大 |
| 加权随机 | 考虑性能差异 | 权重静态 | 实例性能差异大 |
| 一致性Hash | 会话保持 | 扩容时缓存失效 | 需要会话保持（如购物车） |
| 最少连接数 | 感知实时负载 | 需要维护连接数 | 长连接场景（如WebSocket） |
| 自适应 | 过滤差Zone，感知健康 | 复杂 | 生产推荐（Ribbon默认） |

**本质区别**：
- **静态算法**（轮询、随机、加权）：不感知实例实时状态，简单但可能分配不当
- **动态算法**（最少连接数、自适应）：感知实例实时负载，复杂但更合理

## 正确使用方式

### Ribbon（Spring Cloud）：配置负载均衡策略

```yaml
# 对"order-service"这个服务，用随机策略
order-service:
  ribbon:
    NFLoadBalancerRuleClassName: com.netflix.loadbalancer.RandomRule
    # 其他策略：
    # RoundRobinRule（轮询，默认）
    # WeightedResponseTimeRule（响应时间加权，推荐）
    # ZoneAvoidanceRule（自适应，过滤差Zone）
```

### Nginx：负载均衡配置

```nginx
# 轮询（默认）
upstream backend {
    server backend1.example.com;
    server backend2.example.com;
}

# 加权轮询
upstream backend {
    server backend1.example.com weight=5;  # 5/6的流量
    server backend2.example.com weight=1;  # 1/6的流量
}

# 一致性Hash（按IP）
upstream backend {
    ip_hash;  # 同一IP总是到同一backend
    server backend1.example.com;
    server backend2.example.com;
}

# 最少连接数
upstream backend {
    least_conn;
    server backend1.example.com;
    server backend2.example.com;
}
```

## 边界情况和坑

### 坑1：热点问题（Hot Spot）

```
场景：缓存失效，大量请求同时打到某个实例
后果：这个实例被打挂，连锁反应（所有实例逐个被打挂）
```

**解决方案**：
1. **客户端缓存**（如Ribbon的本地缓存，不每次都请求注册中心）
2. **限流**（每个实例限流，超出的请求拒绝）
3. **熔断**（实例响应慢，熔断，不再发请求过去）

### 坑2：实例故障，但负载均衡还往那里发请求

```
场景：实例A挂了，但注册中心还没剔除（心跳超时还没到）
后果：部分请求失败
```

**解决方案**：
1. **重试机制**（Ribbon默认重试下一个实例）
2. **熔断机制**（Hystrix/Resilience4j，实例故障率超阈值就熔断）
3. **缩短心跳超时**（但会增加网络负担）

### 坑3：一致性Hash扩容，缓存失效

```
场景：3个实例 → 扩容到4个
后果：Hash重新计算，几乎全部缓存失效（缓存雪崩）
```

**解决方案**：
1. **一致性Hash环**（一致性Hash的高级版本，扩容时只失效1/n的缓存）
2. **不扩容，而是替换**（新实例替换老实例，IP不变）

## 我的理解

负载均衡的核心是**把流量均匀分配到多个实例**，但"均匀"不是目的，目的是**最大化吞吐、最小化延迟**。

不同算法有不同适用场景：
- **实例性能一致** → 轮询或随机
- **实例性能差异大** → 加权轮询或加权随机
- **需要会话保持** → 一致性Hash
- **长连接** → 最少连接数
- **生产环境** → 自适应（ZoneAvoidance，Ribbon默认）

实际生产中，**Ribbon的ZoneAvoidanceRule（自适应）** 是默认策略，会过滤性能差的Zone，再轮询。

**面试最高频追问**：
1. "负载均衡算法有哪些？" → 轮询、随机、加权、一致性Hash、最少连接数
2. "一致性Hash是什么？" → 同一IP总是打到同一实例，适合会话保持
3. "Ribbon默认策略是什么？" → ZoneAvoidanceRule（自适应，过滤差Zone）

---

*负载均衡不是"均匀分配"那么简单，要感知实例性能、健康状态、会话保持需求。*
