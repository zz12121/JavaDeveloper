# 并发容器

> `java.util.concurrent` 包提供了多种线程安全的容器实现，每种都有自己适用的场景和 tradeoff。本文档覆盖 `ConcurrentHashMap`、`CopyOnWriteArrayList`、`ConcurrentSkipListMap` 之外的并发容器，并提供完整的选型指南。

---

## 这个问题为什么存在？

```
Collections.synchronizedXXX() 的根本问题：
1. 锁粒度 = 整个集合 → 读操作也要等锁
2. 复合操作不安全 → if (!contains(x)) add(x) 不是原子的
3. 迭代必须手动加锁 → 忘了就线程不安全

JUC 并发容器的核心设计理念：
→ 细粒度并发控制（CAS / 分段锁 / 写时复制 / 无锁算法）
→ 支持原子复合操作（putIfAbsent / computeIfAbsent）
→ 弱一致性迭代器（不需要加锁，不抛 CME）
```

## JUC 并发容器全景图

```
Map：
├── ConcurrentHashMap            — 分段锁/CAS，高并发 Map（⭐ 最常用）
├── ConcurrentSkipListMap        — 跳表实现，并发有序 Map（⭐ 详见独立文档）
└── ConcurrentHashMap.newKeySet() — 并发 Set（基于 CHM）

List：
├── CopyOnWriteArrayList         — 写时复制，读多写极少（⭐ 详见独立文档）
└── Collections.synchronizedList — 简单加锁（不推荐）

Queue/Deque（非阻塞）：
├── ConcurrentLinkedQueue         — 无锁链表队列
├── ConcurrentLinkedDeque         — 无锁链表双端队列

Queue/Deque（阻塞）：
├── ArrayBlockingQueue            — 有界数组阻塞队列
├── LinkedBlockingQueue           — 链表阻塞队列
├── SynchronousQueue               — 零容量直接传递队列
├── PriorityBlockingQueue          — 优先级阻塞队列
├── DelayQueue                    — 延迟队列
└── LinkedTransferQueue           — 高性能传输队列

Set：
├── ConcurrentHashMap.newKeySet() — 无序并发 Set
├── ConcurrentSkipListSet         — 有序并发 Set（基于跳表）
└── CopyOnWriteArraySet           — 读多写少 Set（基于 COWArrayList）
```

## ConcurrentLinkedQueue（无锁队列）

### 核心思想

```
ConcurrentLinkedQueue = 基于 CAS 的无锁链表队列

特点：
- 完全没有锁（synchronized / ReentrantLock）
- 所有操作通过 CAS 实现
- 队列无界（没有容量限制）
- 不支持阻塞操作（不会等待，满了/空了就返回）
```

### 内部结构

```
  head → Node(A) → Node(B) → Node(C) ← tail

入队（offer）：
  1. CAS(tail.next, null, newNode)   // 把新节点接到尾部
  2. CAS(tail, oldTail, newNode)      // 移动 tail 指针
  → 如果 CAS 失败，说明其他线程已经修改，自旋重试

出队（poll）：
  1. if (head == tail) return null    // 队列为空
  2. CAS(head, oldHead, head.next)    // 移动 head 指针
  → 如果 CAS 失败，自旋重试

所有操作 O(n) 时间复杂度，但实际 n 很小（只涉及头尾节点）
```

### 使用示例

```java
ConcurrentLinkedQueue<String> queue = new ConcurrentLinkedQueue<>();

// 入队（不阻塞，永远成功）
queue.offer("task1");
queue.offer("task2");

// 出队（不阻塞，空队列返回 null）
String task = queue.poll();  // "task1"
String empty = queue.poll(); // "task2"

// 查看队首（不移除）
String peek = queue.peek();  // null（队列已空）

// 注意：没有 take() 方法！不会阻塞等待
```

### 和阻塞队列的对比

```
              ConcurrentLinkedQueue    ArrayBlockingQueue
──────────────────────────────────────────────────────────
实现方式       CAS 无锁                ReentrantLock + Condition
队列大小       无界                    有界（构造时指定）
队列为空       poll() 返回 null         take() 阻塞等待
队列已满       offer() 永远成功         put() 阻塞等待
内存溢出风险   ✅ 有（无界堆积）        ❌ 无（有界保护）
适用场景       非阻塞高并发             生产-消费模式
```

### 边界情况

```
坑 1：size() 是 O(n) 操作！
  → size() 需要遍历整个链表计数
  → 绝对不要在高并发循环中调用 size()
  → 用 isEmpty() 代替 size() == 0

坑 2：无界队列可能 OOM
  → 生产者速度 > 消费者速度 → 队列无限增长
  → 如果需要流量控制，用阻塞队列（有界）
  → 或者配合 Semaphore 限流

坑 3：不适合生产-消费模式
  → 生产者没有"等待"语义
  → 消费者空转（不断 poll 返回 null）浪费 CPU
  → 生产-消费用阻塞队列
```

## ConcurrentLinkedDeque（无锁双端队列）

```
ConcurrentLinkedDeque = ConcurrentLinkedQueue 的双向版本

额外能力：
- addFirst(e) / addLast(e)    — 两端添加
- pollFirst() / pollLast()    — 两端移除
- peekFirst() / peekLast()    — 两端查看

适用场景：
- 工作窃取算法（Work Stealing）
  → 每个 worker 有自己的双端队列
  → 从自己的队列尾部取任务
  → 空闲时从其他 worker 的队列头部"偷"任务
  → ForkJoinPool 的任务队列就是这个实现

使用场景较少，大多数时候用 ConcurrentLinkedQueue 就够了
```

## 并发 Set

### ConcurrentHashMap.newKeySet()

```java
// Java 8+ 提供的并发 Set
Set<String> set = ConcurrentHashMap.newKeySet();

// 原理：Set 的底层就是 Map（value 固定为 Boolean.TRUE）
// 等价于：
new ConcurrentHashMap<String, Boolean>().keySet(Boolean.TRUE)

// 支持 Set 操作 + CHM 的并发能力
set.add("hello");           // O(1)
set.contains("hello");      // O(1)
set.remove("hello");        // O(1)
set.addAll(list);           // 批量添加

// Java 8+ 还支持带权重的 Set
Set<String> set = ConcurrentHashMap.newKeySet(16);  // 初始容量

// 为什么没有 ConcurrentHashSet 类？
// 因为 Set = Map 的 key 集合，不需要单独实现
// CHM.keySet() 在 JDK 8 之前返回弱一致性视图
// JDK 8+ 的 newKeySet() 返回真正的并发 Set
```

### ConcurrentSkipListSet

```java
// 有序并发 Set（基于跳表）
ConcurrentSkipListSet<String> set = new ConcurrentSkipListSet<>();

set.add("banana");
set.add("apple");
set.add("cherry");

// 自动按字典序排序
set.forEach(System.out::println);
// apple, banana, cherry

// 支持 Set 操作 + 跳表的有序能力
set.headSet("banana");     // [apple]
set.subSet("apple", "cherry");  // [apple, banana]

// 适用：需要排序的 Set
// 不适合：大量元素（跳表内存开销大）
```

### CopyOnWriteArraySet

```java
// 基于 CopyOnWriteArrayList 的 Set
CopyOnWriteArraySet<String> set = new CopyOnWriteArraySet<>();

// add 调用 addIfAbsent，去重
set.add("hello");
set.add("hello");  // 重复添加，自动去重

// 适用：小集合 + 读多写极少（事件监听器注册表）
// 不适合：大集合（addIfAbsent 是 O(n)）
```

### 并发 Set 选型

```
需要 Set？
├── 不需要排序 → ConcurrentHashMap.newKeySet()（默认选择）
├── 需要排序   → ConcurrentSkipListSet（基于跳表）
└── 读多写极少  → CopyOnWriteArraySet（小集合）
```

## 选型指南

### 按数据结构选型

```
需要 Map？
├── 不需要排序 → ConcurrentHashMap（⭐ 默认选择）
├── 需要排序/范围查询 → ConcurrentSkipListMap（⭐ 详见独立文档）
└── 读多写极少 → 也用 CHM（CHM 本身就是读多写的优化）

需要 List？
├── 读多写极少 → CopyOnWriteArrayList（⭐ 详见独立文档）
├── 读写都频繁 → Collections.synchronizedList（或用 Queue 替代）
└── 频繁插入/删除 → 考虑用 ConcurrentLinkedDeque

需要 Set？
├── 无序 → ConcurrentHashMap.newKeySet()（⭐ 默认选择）
├── 有序 → ConcurrentSkipListSet
└── 读多写少 → CopyOnWriteArraySet（小集合）

需要 Queue？
├── 不需要阻塞等待 → ConcurrentLinkedQueue（⭐ 非阻塞高并发）
├── 生产-消费模式 → ArrayBlockingQueue / LinkedBlockingQueue（详见"阻塞队列"章节）
├── 需要优先级 → PriorityBlockingQueue
├── 需要延迟 → DelayQueue
└── 需要双端 → ConcurrentLinkedDeque
```

### 常见误用提醒

```
❌ 高并发 Map 用 Hashtable / synchronizedMap
   → 用 ConcurrentHashMap

❌ 高并发 List 用 synchronizedList + 遍历不加锁
   → 用 CopyOnWriteArrayList 或 ConcurrentLinkedDeque

❌ 用 ArrayList 在多线程中 add
   → 数组越界、元素丢失、CME

❌ 用 HashSet 在多线程中 add
   → 元素丢失（size 没加够但数据已写入）
   → 用 ConcurrentHashMap.newKeySet()

❌ ConcurrentLinkedQueue 的 size() 放在循环里
   → O(n) 每次！用 isEmpty()
```

---

## 边界情况速查

```
坑 1：ConcurrentHashMap 不允许 null key/value
   → get(key) 返回 null = key 不存在（二义性问题）
   → CHM 直接禁止 null，消除歧义

坑 2：ConcurrentLinkedQueue 的 size() 是 O(n)
   → 别在循环里调用

坑 3：CopyOnWriteArrayList 迭代器是快照
   → 看不到最新修改，不抛 CME

坑 4：ConcurrentSkipListMap 同 key 会覆盖
   → 排行榜场景用 Map<Integer, Set<String>>

坑 5：所有并发容器的迭代器都是弱一致性的
   → 不保证看到最新修改，这是设计 tradeoff
   → 如果需要强一致，额外加同步
```

## 我的理解（面试考点）

```
Q1: 说说常用的并发容器？
→ Map: ConcurrentHashMap（默认）、ConcurrentSkipListMap（需要排序）
→ List: CopyOnWriteArrayList（读多写少）
→ Set: ConcurrentHashMap.newKeySet()（默认）
→ Queue: ConcurrentLinkedQueue（非阻塞）、阻塞队列（生产消费）

Q2: 为什么没有 ConcurrentHashSet？
→ Set = Map 的 key 集合，CHM.newKeySet() 就是并发 Set

Q3: ConcurrentLinkedQueue 和阻塞队列的区别？
→ 无锁 vs 加锁、无界 vs 有界、不阻塞 vs 阻塞
→ 高并发非阻塞用 LinkedQueue，生产消费用阻塞队列

Q4: 如何选择并发容器？
→ Map 首选 CHM，需要排序用 SkipListMap
→ List 读多写少用 COWArrayList
→ Set 用 CHM.newKeySet()
→ Queue 按是否需要阻塞选择
```
