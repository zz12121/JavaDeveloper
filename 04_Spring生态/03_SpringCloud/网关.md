# 网关

## 这个问题为什么存在？

> 没有网关，每个服务都要**自己处理**鉴权、限流、日志、跨域... 重复劳动，且容易不一致。  
> 客户端也要知道**所有后端服务的地址**，耦合严重。

**网关的本质**：**统一入口**，把跨切面的逻辑（鉴权、限流、日志）集中到一层处理。

```
客户端 → 网关（统一入口）
          ├─ 鉴权（JWT 校验）
          ├─ 限流（防止刷爆）
          ├─ 日志（请求追踪）
          ├─ 跨域（CORS）
          └─ 路由（转发到具体服务）
```

---

## 它是怎么解决问题的？

### Spring Cloud Gateway 的架构

**核心概念**：

```
Route（路由）：转发规则（路径 + 目标服务）
Predicate（断言）：匹配条件（路径、Method、Header...）
Filter（过滤器）：处理逻辑（鉴权、改请求、改响应）
```

**工作流程**：

```
客户端请求
    ↓
Gateway Handler Mapping（匹配 Route）
    ↓
Gateway Web Handler（执行 Filter 链）
    ↓
Predicates 匹配 → 找到 Route
    ↓
执行「前置 Filter」（鉴权、日志、改请求头）
    ↓
转发请求到目标服务（Http Client）
    ↓
收到响应
    ↓
执行「后置 Filter」（改响应头、日志）
    ↓
返回响应给客户端
```

### 路由配置（YAML）

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://user-service  # lb:// 表示用负载均衡
          predicates:
            - Path=/api/users/**
          filters:
            - StripPrefix=1  # 去掉第一段路径（/api）
            # /api/users/1 → 转发到 /users/1

        - id: order-service
          uri: lb://order-service
          predicates:
            - Path=/api/orders/**
            - Method=GET,POST
          filters:
            - StripPrefix=1
```

**predicates 常用匹配条件**：

| Predicate | 说明 | 示例 |
|-----------|------|------|
| `Path` | 路径匹配 | `Path=/api/users/**` |
| `Method` | HTTP Method | `Method=GET,POST` |
| `Header` | 请求头匹配 | `Header=X-Request-Id, \d+` |
| `Query` | 查询参数匹配 | `Query=token` |
| `After` | 在时间之后 | `After=2024-01-01T00:00:00+08:00` |
| `Weight` | 权重路由（灰度） | `Weight=group1, 80` |

### Filter（过滤器）

**内置 Filter**：

```yaml
filters:
  # 1. 添加请求头
  - AddRequestHeader=X-Request-Id, {id}

  # 2. 添加请求参数
  - AddRequestParameter=source, gateway

  # 3. 修改路径
  - RewritePath=/api/(?<segment>.*), /$\{segment}

  # 4. 限流
  - RequestRateLimiter:
      redis-rate-limiter.replenishRate: 10   # 每秒 10 个请求
      redis-rate-limiter.burstCapacity: 20    # 突发容量 20

  # 5. 熔断
  - name: CircuitBreaker
    args:
      name: myCircuitBreaker
      fallbackUri: /fallback
```

**自定义全局 Filter**（鉴权场景）：

```java
@Component
public class AuthFilter implements GlobalFilter, Ordered {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange,
                             GatewayFilterChain chain) {
        String token = exchange.getRequest()
                         .getHeaders()
                         .getFirst("Authorization");

        if (token == null || !validateToken(token)) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();  // 中断请求
        }

        // 校验通过，继续下一个 Filter
        return chain.filter(exchange);
    }

    @Override
    public int getOrder() {
        return -100;  // 越小越先执行（鉴权要第一个执行）
    }
}
```

### 熔断（CircuitBreaker）

**问题**：后端服务挂了，网关还一直转发 → 大量请求堆积，拖垮网关。

**Spring Cloud CircuitBreaker**（基于 Resilience4J）：

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://user-service
          predicates:
            - Path=/api/users/**
          filters:
            - name: CircuitBreaker
              args:
                name: userCircuitBreaker
                fallbackUri: /fallback/users
```

```java
// 降级方法
@GetMapping("/fallback/users")
public ResponseEntity<String> fallback() {
    return ResponseEntity.status(503)
            .body("服务暂时不可用，请稍后重试");
}
```

**熔断器的三种状态**：

```
CLOSED（关闭）
  ↓ 错误率超过阈值（如 50%）
OPEN（打开）→ 直接走降级，不转发请求
  ↓ 等待一段时间（如 10s）
HALF_OPEN（半开）→ 放一个请求试试
  ↓ 成功
CLOSED（恢复）
  ↓ 失败
OPEN（再次打开）
```

### 限流（RequestRateLimiter）

**基于 Redis 的令牌桶限流**：

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://user-service
          predicates:
            - Path=/api/users/**
          filters:
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 10   # 每秒补充 10 个令牌
                redis-rate-limiter.burstCapacity: 20    # 桶容量 20
                key-resolver: "#{@userKeyResolver}"     # 限流维度
```

**限流维度（KeyResolver）**：

```java
@Bean
public KeyResolver userKeyResolver() {
    // 按用户 ID 限流（从请求参数取 user_id）
    return exchange -> Mono.just(
        exchange.getRequest()
                .getQueryParams()
                .getFirst("user_id")
    );
}

@Bean
public KeyResolver ipKeyResolver() {
    // 按 IP 限流
    return exchange -> Mono.just(
        exchange.getRequest()
                .getRemoteAddress()
                .getAddress()
                .getHostAddress()
    );
}
```

## 它和相似方案的本质区别是什么？

### Spring Cloud Gateway vs Zuul

| | Spring Cloud Gateway | Zuul 1.x | Zuul 2.x |
|---|---|---|---|
| 编程模型 | 响应式（WebFlux） | 阻塞（Servlet） | 响应式（Netty） |
| 性能 | 高 | 低（阻塞模型） | 中高 |
| 维护状态 | 活跃 | Netflix 已停止维护 | 不活跃 |
| 适用场景 | 新项目（推荐） | 旧项目迁移 | 不推荐（生态弱） |

> **Zuul 1.x 的致命问题**：阻塞模型，一个请求占用一个线程，高并发下线程池打满。

---

### Spring Cloud Gateway vs Nginx

| | Spring Cloud Gateway | Nginx |
|---|---|---|
| 语言 | Java | C |
| 性能 | 中（基于 Netty） | 高（C + 事件驱动） |
| 动态路由 | ✅（配置中心 + 动态刷新） | ❌（需要 reload） |
| 鉴权逻辑 | ✅（Java 代码，灵活） | ❌（Lua 脚本） |
| 适用场景 | 内网微服务网关 | 边界网关（对外） |

**最佳实践**：**Nginx 做边界网关**（对外），**Spring Cloud Gateway 做内网网关**（微服务间）。

## 正确使用方式

### 1. 跨域配置（CORS）

```yaml
spring:
  cloud:
    gateway:
      globalcors:
        cors-configurations:
          '[/**]':
            allowed-origins:
              - "https://www.example.com"
            allowed-methods:
              - GET
              - POST
            allowed-headers:
              - "*"
            allow-credentials: true
            max-age: 3600
```

---

### 2. 请求日志 Filter

```java
@Component
@Slf4j
public class LoggingFilter implements GlobalFilter {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange,
                             GatewayFilterChain chain) {
        long start = System.currentTimeMillis();

        return chain.filter(exchange).then(
            Mono.fromRunnable(() -> {
                long cost = System.currentTimeMillis() - start;
                log.info("请求 {} 耗时 {}ms",
                         exchange.getRequest().getPath(), cost);
            })
        );
    }
}
```

---

### 3. 灰度发布（权重路由）

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: user-service-v1
          uri: lb://user-service-v1
          predicates:
            - Weight=user-group, 90  # 90% 流量到 v1
        - id: user-service-v2
          uri: lb://user-service-v2
          predicates:
            - Weight=user-group, 10  # 10% 流量到 v2（灰度）
```

## 边界情况和坑

### 1. 响应体只能读取一次

```java
// ❌ 错误：响应体读取后，下游 Filter 读不到
ServerHttpResponse response = exchange.getResponse();
DataBufferFactory bufferFactory = response.bufferFactory();
// 如果在这里读了 response.getBody()，后面的 Filter 会报错
```

**解决**：用 `cachedBody` 或者 `modifyResponseBody`。

```java
// ✅ 正确：用 modifyResponseBody 修改响应体
routes:
  - id: modify-response
    uri: lb://user-service
    predicates:
      - Path=/api/users/**
    filters:
      - name: ModifyResponseBody
        args:
          contentType: application/json
          bodyClass: String
          rewriteFunction: com.example.ModifyResponseFunction
```

---

### 2. 过滤器执行顺序

```
GlobalFilter（全局过滤器）按 getOrder() 排序（越小越先执行）
    ↓
GatewayFilter（路由过滤器）按配置顺序执行
```

> **Order 值越小，越先执行（前置处理）**；  
> **但响应时，Order 越小反而越后执行（后置处理）**。  
> 这和 Spring MVC 的拦截器是一样的逻辑。

---

### 3. WebSocket 转发

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: websocket
          uri: ws://localhost:8081
          predicates:
            - Path=/ws/**
```

> Spring Cloud Gateway **支持 WebSocket 转发**，但配置要用 `ws://` 或 `wss://`。

---

### 4. 超时配置

```yaml
spring:
  cloud:
    gateway:
      httpclient:
        connect-timeout: 1000   # 连接超时（毫秒）
        response-timeout: 5000   # 响应超时（毫秒）
```

> 默认**没有超时**，如果后端服务挂了，请求会一直挂起。  
> **生产环境必须配置超时**。

## 我的理解

**Spring Cloud Gateway 的本质**：

1. **统一入口**：所有请求先到网关，再路由到具体服务
2. **跨切面逻辑集中处理**：鉴权、限流、日志、跨域
3. **基于 WebFlux**：响应式、非阻塞，性能优于 Zuul 1.x
4. **三大核心**：Route（路由）、Predicate（匹配）、Filter（处理）

**面试高频追问**：

1. Spring Cloud Gateway 的工作流程（匹配 Route → 执行 Filter → 转发 → 后置 Filter）
2. 熔断器的三种状态（CLOSED / OPEN / HALF_OPEN）
3. 限流算法（令牌桶 vs 漏桶 vs 固定窗口）
4. Spring Cloud Gateway vs Zuul vs Nginx 的区别
5. 过滤器执行顺序（Order 值越小越先执行，但后置处理反序）
