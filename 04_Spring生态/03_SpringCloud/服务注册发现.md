# 服务注册发现

## 这个问题为什么存在？

> 没有服务注册发现，服务 A 调用服务 B 要**硬编码 IP 地址**：  
> `restTemplate.getForObject("http://192.168.1.10:8080/api/order", ...)`  
> 问题是：**服务 B 扩容了、IP 变了、下线了，服务 A 都要改配置并重启**。

**服务注册发现的本质**：把「服务地址」从代码里抽出来，放到一个**共享注册表**里，服务启动时自动注册，调用时自动查询。

---

## 它是怎么解决问题的？

### 服务注册发现的流程

```
┌─────────────────────────────────────────────┐
│ 1. 服务启动 → 注册到注册中心                 │
│    OrderService (192.168.1.10:8080)       │
│    OrderService (192.168.1.11:8080)       │
├─────────────────────────────────────────────┤
│ 2. 调用方查询 → 从注册中心拉取服务列表        │
│    UserService 查 OrderService → 拿到 2 个地址 │
├─────────────────────────────────────────────┤
│ 3. 负载均衡 → 选一个地址调用                 │
│    UserService → 192.168.1.10:8080       │
├─────────────────────────────────────────────┤
│ 4. 心跳续约 → 服务定期发心跳（每 5s）       │
│    注册中心：30s 没收到心跳 → 标记不可用      │
│    注册中心：90s 没收到心跳 → 剔除实例        │
└─────────────────────────────────────────────┘
```

---

### Nacos 的核心架构

**Nacos** = **服务注册发现** + **配置中心**（二合一）。

```
┌─────────────┐
│  Nacos Server（注册中心）                     │
│    ├─ Service Registry（服务注册表）          │
│    ├─ Health Check（健康检查）                │
│    └─ Push/Pull（推送/拉取服务列表）         │
└─────────────┘
        ↑ 注册（Register）       ↓ 发现（Discover）
┌──────────────┐          ┌──────────────┐
│ OrderService │          │ UserService  │
│ (服务 A)     │          │ (服务 B)     │
└──────────────┘          └──────────────┘
```

**服务注册**：

```java
// Spring Cloud Alibaba Nacos
// 1. 引入依赖
// spring-cloud-starter-alibaba-nacos-discovery

// 2. 配置文件
spring:
  cloud:
    nacos:
      discovery:
        server-addr: localhost:8848
        service: order-service  # 服务名

// 3. 启动类加 @EnableDiscoveryClient
@EnableDiscoveryClient
@SpringBootApplication
public class OrderApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderApplication.class, args);
    }
}
```

> **服务注册是自动完成的**：Spring 启动时，`NacosAutoServiceRegistration` 监听到 `WebServerInitializedEvent`，自动把本服务的 IP + 端口注册到 Nacos。

### 服务发现的源码路径

**问题**：服务调用方怎么拿到服务列表？

```java
// 1. NacosDiscoveryClient.getInstances()
public List<ServiceInstance> getInstances(String serviceId) {
    // 从 Nacos 客户端拉取服务实例列表
    List<Instance> instances = namingService.getAllInstances(serviceId);
    // 转换成 Spring 的 ServiceInstance 格式
    return hostToServiceInstanceList(instances, serviceId);
}

// 2. 调用方使用（配合 @LoadBalanced）
@Bean
@LoadBalanced  // ★ 关键：让 RestTemplate 有负载均衡能力
public RestTemplate restTemplate() {
    return new RestTemplate();
}

// 3. 调用（用服务名，不是 IP）
String url = "http://order-service/api/order/" + orderId;
Order order = restTemplate.getForObject(url, Order.class);
```

> **`@LoadBalanced` 做了什么**？  
> 它给 `RestTemplate` 加了一个 `@LoadBalanced` 注解标记的 `LoadBalancerInterceptor`，  
> 这个拦截器会把 `http://order-service/...` 中的 `order-service` 替换成**实际 IP 地址**（从 Nacos 拉取 + 负载均衡）。

---

### 负载均衡（Ribbon / Spring Cloud LoadBalancer）

**问题**：服务有 3 个实例，调用方选哪一个？

**Spring Cloud 2020+ 默认用 `Spring Cloud LoadBalancer`**（淘汰了 Ribbon）。

```java
// 配置负载均衡策略
spring:
  cloud:
    loadbalancer:
      configurations: round-robin  # 轮询（默认）
      # configurations: random     # 随机
```

**自定义负载均衡策略**：

```java
@Bean
public ReactorLoadBalancer<ServiceInstance> customLoadBalancer(
    Environment environment,
    LoadBalancerClientFactory factory) {
    String name = environment.getProperty(LoadBalancerClientFactory.PROPERTY_NAME);
    return new RoundRobinLoadBalancer(
        factory.getLazyProvider(name, ServiceInstanceListSupplier.class), name);
}
```

## 它和相似方案的本质区别是什么？

### Nacos vs Eureka

| | Nacos | Eureka |
|---|---|---|
| 厂商 | Alibaba | Netflix（已停止维护） |
| 配置中心 | ✅ 内置 | ❌ 需要 Config Server |
| 一致性协议 | CP + AP（可切换） | AP |
| 健康检查 | TCP / HTTP / MySQL / 自定义 | Client 心跳 |
| 适用场景 | 国内主流（阿里系） | 旧项目（Netflix 系） |

**CP vs AP 的区别**：

```
CP（Consistency + Partition tolerance）：
  → 注册中心集群分区时，不返回不一致的数据
  → 牺牲可用性，保证一致性
  → 适合：服务注册发现（不能拿到过期地址）

AP（Availability + Partition tolerance）：
  → 注册中心集群分区时，返回旧数据（可用）
  → 牺牲一致性，保证可用性
  → 适合：服务注册发现（允许短暂不一致，客户端有重试）
```

> Nacos 支持**切换**：AP 模式（基于 Distro）和 CP 模式（基于 Raft）。

---

### Nacos vs Consul

| | Nacos | Consul |
|---|---|---|
| 语言 | Java | Go |
| 配置中心 | ✅ | ✅ |
| 健康检查 | 多种 | 多种 |
| 多数据中心 | ✅ | ✅（强项） |
| 学习曲线 | 中 | 高（需要了解 Serf、Raft） |

## 正确使用方式

### 1. 服务分组和集群

```yaml
spring:
  cloud:
    nacos:
      discovery:
        server-addr: localhost:8848
        service: order-service
        group: PROD_GROUP        # 分组（不同组服务不可见）
        cluster-name: HZ_CLUSTER  # 集群名（优先调用同集群服务）
```

**集群优先策略**（减少跨机房调用）：

```java
// 配置：优先调用同集群服务
spring:
  cloud:
    nacos:
      discovery:
        cluster-name: HZ
    loadbalancer:
      nacos:
        enabled: true  # 启用 Nacos 负载均衡规则（同集群优先）
```

---

### 2. 健康检查和自我保护

```yaml
# Nacos Server 配置
nacos:
  discovery:
    # 客户端心跳间隔（默认 5s）
    heart-beat-interval: 5
    # 心跳超时（默认 15s，超过标记为不健康）
    heart-beat-timeout: 15
    # 剔除实例时间（默认 30s）
    ip-delete-timeout: 30
```

> **重要**：生产环境不要设置太短的超时，否则网络抖动会导致服务被误剔除。

---

### 3. 临时实例 vs 持久化实例

```yaml
spring:
  cloud:
    nacos:
      discovery:
        # 临时实例（默认，AP 模式）
        ephemeral: true
        # 持久化实例（CP 模式，不会因为心跳超时被剔除）
        # ephemeral: false
```

| | 临时实例 | 持久化实例 |
|---|----------|-------------|
| 注册方式 | 客户端注册 | 服务端注册 |
| 心跳 | 客户端发心跳 | 服务端检查心跳 |
| 适用场景 | 普通服务 | 关键服务（不能因心跳超时被剔除） |

## 边界情况和坑

### 1. 服务下线但注册中心不知道

```java
// ❌ 进程被 kill -9，没有机会发送注销请求
// Nacos 只能靠心跳超时 → 30s 后才标记不可用
// 这段时间内请求可能路由到已下线的服务 → 报错
```

**解决**：使用 **优雅停机**。

```java
// Spring Boot 优雅停机
server:
  shutdown: graceful  # 等待请求处理完再关闭

spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s
```

> 优雅停机时，Spring 会发送 **服务注销请求** 到 Nacos，让注册中心立即移除实例。

---

### 2. 本地开发时服务发现干扰

```yaml
# ❌ 本地开发时，不想注册到 Nacos（避免别人调到自己本地）
spring:
  cloud:
    nacos:
      discovery:
        register-enabled: false  # 禁用注册
```

---

### 3. 服务名写错导致调用失败

```java
// ❌ 服务名写错（大小写敏感）
restTemplate.getForObject("http://Order-Service/...", ...);
// 实际注册的服务名是 order-service（小写）

// ✅ 正确：保持服务名一致
spring:
  application:
    name: order-service  # 服务名（小写，用 "-" 分隔）
```

---

### 4. Nacos 集群节点全部宕机

```
Nacos 集群全部宕机 → 客户端本地缓存服务列表 → 仍然可以调用（但无法发现新服务）
```

> Nacos 客户端会**缓存服务列表**到本地文件（`${user.home}/nacos/naming/`）。  
> 注册中心全部宕机时，**已发现的服务仍然可用**，但新服务无法注册。

## 我的理解

**服务注册发现的本质**：

1. **注册**：服务启动时把自己的地址写到注册表
2. **发现**：调用方从注册表拉取服务列表
3. **心跳**：服务定期发心跳证明自己还活着
4. **负载均衡**：调用方从多个实例中选一个

**面试高频追问**：

1. 服务注册发现的流程（注册 → 发现 → 心跳 → 负载均衡）
2. Nacos 的 CP 和 AP 模式区别
3. Nacos vs Eureka 的区别（配置中心、一致性协议、健康检查）
4. 优雅停机为什么重要（避免请求路由到已下线服务）
5. `@LoadBalanced` 的原理（LoadBalancerInterceptor 拦截请求，替换服务名为 IP）
