## 知识链提示

这道题应该让你联想到：

- `最小暴露原则` → 能 private 就 private
- `防御性拷贝` → 返回可变对象时要克隆
- `this$0机制` → 成员内部类的内存泄漏风险
- `JavaBean规范` → 框架友好 vs 业务合理是两回事
- `不可变对象` → String 为什么字段是 final 的

## 参考要点（盲答后再看）


**封装的本质**：

```
封装 ≠ getter/setter
封装 = 把"可以改变的内部实现"藏起来，只暴露"稳定的接口"

检验标准：内部实现改变时，调用方代码要不要改？
如果改 = 封装做得差
如果不用改 = 封装做得好
```

**访问修饰符范围**：

| 修饰符 | 同类 | 同包 | 不同包子类 | 任意位置 |
|--------|------|------|-----------|---------|
| private | ✅ | ❌ | ❌ | ❌ |
| (默认) | ✅ | ✅ | ❌ | ❌ |
| protected | ✅ | ✅ | ✅ | ❌ |
| public | ✅ | ✅ | ✅ | ✅ |

**protected 的坑**：
```java
class Parent {
    protected void method() { }
}

class Child extends Parent {
    void test() {
        method();                    // ✅ 可以：通过子类实例调用
        new Parent().method();       // ❌ 编译错误！不能通过父类实例调用
    }
}
```

**防御性拷贝**：
```java
class Profile {
    private final int[] scores;

    // 入参防御
    Profile(int[] scores) {
        this.scores = scores.clone();  // 防止外部数组被修改
    }

    // 返回防御
    public int[] getScores() {
        return scores.clone();  // 不暴露内部引用
    }
}
```

**JavaBean vs 真正封装**：
```java
// JavaBean 规范（框架友好）
class User {
    private String password;
    public String getPassword() { return password; }  // ❌ 密码不该有 getter
    public void setPassword(String p) { this.password = p; }
}

// 业务封装（真正合理）
class User {
    private String passwordHash;
    public boolean checkPassword(String input) {
        return BCrypt.checkpw(input, passwordHash);  // ✅ 不暴露密码
    }
}
```
