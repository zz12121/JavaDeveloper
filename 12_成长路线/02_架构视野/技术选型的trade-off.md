# 技术选型的trade-off

> 没有最好的技术，只有最合适的选型。选型的本质是在多个维度之间做权衡。

---

## 一、trade-off 的本质

```
技术选型 = 在多个维度之间做权衡

核心维度：
性能 ←→ 开发效率
一致性 ←→ 可用性
复杂度 ←→ 功能丰富度
学习成本 ←→ 长期收益
自建 ←→ 用开源 ←→ 买 SaaS
```

**关键认知**：每个选项都有代价，架构师的价值不是"选最好的"，而是"选最合适的"。

---

## 二、常见 trade-off 场景

### 2.1 一致性 vs 可用性（CAP 定理）

```
CP 系统（一致性 + 分区容错）：
- 例子：ZooKeeper、Etcd、HBase
- 场景：金融交易、分布式锁

AP 系统（可用性 + 分区容错）：
- 例子：Cassandra、DynamoDB、Eureka
- 场景：购物车、社交网络

CA 系统（一致性 + 可用性）：
- 例子：单机 MySQL（无分区）
- 场景：小规模系统
```

**面试问题**：_Redis 是 CP 还是 AP？_
> Redis Cluster 在网络分区时，少数派节点会停止接受请求（CP 倾向）；但 Redis 本身不保证强一致性（异步复制），所以是 **AP 倾向**。具体问题具体分析。

### 2.2 性能 vs 开发效率

```
高性能方案：
- 手写 SQL、用 Redis 缓存、用 gRPC 通信
- 优点：延迟低、吞吐高
- 缺点：开发慢、维护成本高

高效率方案：
- 用 ORM、用 REST API、用消息队列
- 优点：开发快、代码易读
- 缺点：性能有损耗

选型原则：
- 核心链路（QPS > 1万）：性能优先
- 后台管理（QPS < 100）：效率优先
```

### 2.3 简单 vs 灵活

```
简单方案（YAGNI 原则）：
- 单体架构、单库、无分布式事务
- 优点：开发快、运维简单
- 缺点：扩展困难

灵活方案（面向未来）：
- 微服务、分库分表、SAGA 事务
- 优点：扩展灵活
- 缺点：复杂度指数上升

选型原则：
- 团队 < 10 人：简单优先
- 团队 > 50 人：灵活优先（多团队并行开发）
- 不确定时：先简单，后续用绞杀者模式迁移
```

### 2.4 自建 vs 开源 vs 商业产品

| 维度 | 自建 | 开源 | 商业产品 |
|------|------|------|---------|
| 成本 | 高（人力） | 低（但需要运维） | 中（授权费） |
| 可控性 | 完全可控 | 可控（可改源码） | 不可控 |
| 成熟度 | 低（踩坑多） | 高（社区验证） | 高（厂商保障） |
| 运维成本 | 高 | 中 | 低（托管） |
| 适用场景 | 核心壁垒技术 | 通用基础设施 | 非核心业务 |

**选型建议**：
```
核心业务壁垒 → 自建（如阿里 TMDC 调度系统）
通用基础设施 → 开源（如 Redis、Kafka、MySQL）
非核心支撑系统 → 商业产品（如 OA、HR 系统）
```

## 三、技术选型框架

### 3.1 选型评分表

```
对每个候选方案打分（1-5 分）：

| 维度 | 权重 | 方案 A | 方案 B |
|------|--------|--------|--------|
| 功能匹配度 | 30% | 5 | 4 |
| 性能 | 20% | 4 | 5 |
| 团队熟悉度 | 20% | 5 | 2 |
| 社区活跃度 | 10% | 5 | 3 |
| 运维成本 | 10% | 3 | 4 |
| 扩展性 | 10% | 4 | 5 |
| **加权总分** | **100%** | **4.4** | **3.7** |

→ 方案 A 胜出
```

### 3.2 选型决策树

```
这个技术是解决核心问题吗？
├── 是 → 自建 或 深度定制开源
└── 否 → 继续

团队有能力运维吗？
├── 是 → 开源（自主运维）
└── 否 → 商业产品（托管服务）

有成熟开源方案吗？
├── 是 → 用开源（不要重复造轮子）
└── 否 → 自驾（做好长期投入准备）
```

## 四、典型技术选型案例

### 4.1 服务框架：Dubbo vs Spring Cloud

| 维度 | Dubbo | Spring Cloud |
|------|-------|--------------|
| 性能 | 高（Netty 通信） | 中（Servlet 通信） |
| 生态 | 较窄（阿里系） | 完整（Spring 全家桶） |
| 学习曲线 | 陡（ZooKeeper + Netty） | 缓（Spring 生态） |
| 适用场景 | 内部 RPC 性能要求高 | 快速交付、生态整合 |

**选型建议**：
- 已有 Spring 生态 → Spring Cloud
- 性能敏感 + 团队有经验 → Dubbo
- 新项目 → Spring Cloud Alibaba（整合两者优势）

### 4.2 数据库：MySQL vs PostgreSQL

| 维度 | MySQL | PostgreSQL |
|------|-------|-------------|
| 读性能 | 高 | 中 |
| 写性能 | 中 | 高（MVCC 更优） |
| JSON 支持 | 弱 | 强（JSONB 类型） |
| 复杂查询 | 弱 | 强（窗口函数、CTE） |
| 社区 | 极大 | 大 |
| 适用场景 | 互联网业务（读多写少） | 数据分析、复杂查询 |

### 4.3 消息队列：Kafka vs RabbitMQ vs RocketMQ

| 维度 | Kafka | RabbitMQ | RocketMQ |
|------|-------|------------|-----------|
| 吞吐量 | 极高（10万+ QPS） | 中（万级 QPS） | 高（10万级 QPS） |
| 延迟 | ms 级 | μs 级 | ms 级 |
| 顺序消息 | 分区内有序 | 支持 | 支持 |
| 事务消息 | 不支持 | 不支持 | 支持 |
| 适用场景 | 日志、流处理 | 业务消息、低延迟 | 金融交易（阿里系） |

## 五、选型后的验证

```
选型不是一锤子买卖，需要持续验证：

1. POC（概念验证）：用真实场景验证
2. 压测：性能是否达标？
3. 灰度：先上 5% 流量
4. 复盘：3 个月后回顾选型是否正确
5. 回退计划：选型失败如何迁移？
```
