# 高级特性

## 这个问题为什么存在？

> Spring Boot 除了「能跑起来」，还需要**监控应用状态**、**自定义扩展**、**事件驱动**等高级能力。  
> 这些特性是区分「会用 Spring Boot」和「精通 Spring Boot」的关键。

**核心问题**：Spring Boot 提供了哪些开箱即用的高级特性？

---

## 它是怎么解决问题的？

### Actuator（应用监控）

**问题**：应用上线后，怎么知道它的健康状态、内存使用、数据库连得上吗？

**Spring Boot 的解法**：`spring-boot-actuator`，提供一系列**开箱即用的监控端点**。

```xml
<!-- Maven 依赖 -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
```

**常用端点**：

| 端点 | 说明 | 是否需要认证 |
|------|------|--------------|
| `/actuator/health` | 应用健康状态（磁盘、数据库、Redis...） | 默认公开 |
| `/actuator/info` | 应用信息（版本、Git 提交） | 默认公开 |
| `/actuator/metrics` | 指标（JVM 内存、GC、线程池） | 默认不公开 |
| `/actuator/env` | 环境变量 + 配置属性 | 默认不公开（敏感） |
| `/actuator/beans` | 容器中的所有 Bean | 默认不公开 |
| `/actuator/mappings` | 所有 MVC 端点映射 | 默认不公开 |

**配置暴露端点**：

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,info,metrics,env  # 暴露的端点
  endpoint:
    health:
      show-details: always  # 显示详细健康信息
```

### 自定义 Starter（自动配置封装）

**问题**：公司内部有多个项目，都要用同一套 Redis 配置、同一套日志配置... 每个项目都 copy 配置 → 维护噩梦。

**Spring Boot 的解法**：**自定义 Starter**，把公共配置封装成一个依赖。

**Starter 的结构**：

```
mycompany-spring-boot-starter
├── pom.xml
└── src/main/java
    └── com/mycompany/autoconfigure/
        ├── MyServiceAutoConfiguration.java  （自动配置类）
        └── MyServiceProperties.java          （属性配置类）
    resources/
    └── META-INF/
        └── spring/
            └── org.springframework.boot.autoconfigure.AutoConfiguration.imports
            （Spring Boot 2.7+：注册自动配置类）
```

**示例：自定义 Redis Starter**

```java
// 1. 属性配置类
@ConfigurationProperties(prefix = "mycompany.redis")
public class MyRedisProperties {
    private String host = "localhost";
    private int port = 6379;
    // getter / setter
}

// 2. 自动配置类
@Configuration
@ConditionalOnClass(RedisTemplate.class)
@EnableConfigurationProperties(MyRedisProperties.class)
public class MyRedisAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean
    public RedisTemplate<String, Object> redisTemplate(
        RedisConnectionFactory factory) {
        RedisTemplate<String, Object> template = new RedisTemplate<>();
        template.setConnectionFactory(factory);
        return template;
    }
}

// 3. 注册自动配置（META-INF/spring/...imports）
com.mycompany.autoconfigure.MyRedisAutoConfiguration
```

**使用方**：只需要 `pom.xml` 引入依赖，然后在 `application.yml` 写配置即可。

### 事件机制（ApplicationEvent）

**问题**：一个操作完成后，需要触发多个后续动作（如用户注册 → 发邮件 + 发优惠券 + 初始化用户数据），但不能硬编码依赖。

**Spring Boot 的解法**：事件发布-订阅机制（观察者模式）。

```java
// 1. 定义事件
public class UserRegisteredEvent extends ApplicationEvent {
    private final User user;
    private final String source;

    public UserRegisteredEvent(Object source, User user) {
        super(source);
        this.user = user;
    }
    // getter
}

// 2. 发布事件
@Service
public class UserService {
    @Autowired
    private ApplicationEventPublisher eventPublisher;

    public User register(UserDTO dto) {
        User user = userRepository.save(toEntity(dto));
        // 发布事件
        eventPublisher.publishEvent(new UserRegisteredEvent(this, user));
        return user;
    }
}

// 3. 监听事件（可以多个监听器）
@EventListener
@Async  // 异步执行（需要 @EnableAsync）
public void sendWelcomeEmail(UserRegisteredEvent event) {
    emailService.sendWelcome(event.getUser());
}

@EventListener
public void initUserProfile(UserRegisteredEvent event) {
    profileService.initProfile(event.getUser());
}
```

**事件机制的好处**：

```
不用事件：
  register() {
      userService.save();
      emailService.send();     // 硬编码依赖
      couponService.send();
      profileService.init();
  }

用事件：
  register() {
      userService.save();
      eventPublisher.publishEvent(event);  // 只管发布
  }
  // 新增监听者：加一个 @EventListener，不用改 register()
```

### Spring Boot 的测试支持

**问题**：写单元测试要手动启动 Spring 容器，很慢；mock 依赖要手动创建 mock 对象。

**Spring Boot 的解法**：`@SpringBootTest` + `@MockBean`。

```java
@SpringBootTest
class UserServiceTest {

    @Autowired
    private UserService userService;

    @MockBean  // 自动创建 mock 对象，替换容器中的真实 Bean
    private UserRepository userRepository;

    @Test
    void testFindById() {
        // 设置 mock 行为
        when(userRepository.findById(1L))
            .thenReturn(Optional.of(new User(1L, "zhang")));

        // 测试
        User user = userService.findById(1L);
        assertEquals("zhang", user.getName());
    }
}
```

**切片测试**（只加载部分配置，更快）：

```java
@WebMvcTest(UserController.class)  // 只加载 Web 层
class UserControllerTest {
    @Autowired
    private MockMvc mockMvc;

    @MockBean
    private UserService userService;

    @Test
    void testGetUser() throws Exception {
        when(userService.findById(1L)).thenReturn(new User(1L, "zhang"));

        mockMvc.perform(get("/users/1"))
               .andExpect(status().isOk())
               .andExpect(jsonPath("$.name").value("zhang"));
    }
}
```

## 它和相似方案的本质区别是什么？

### Actuator vs 传统监控（JMX）

| | Actuator | JMX |
|--|-----------|-----|
| 访问方式 | HTTP 端点 | JMX 客户端（如 JConsole） |
| 集成难度 | 开箱即用 | 需要手动暴露 MBean |
| 适合场景 | 微服务监控（HTTP 可访问） | 传统单体应用 |

---

### @EventListener vs 直接调用

| | @EventListener | 直接调用 |
|--|----------------|----------|
| 耦合度 | 低（发布方不知道谁处理） | 高（硬编码依赖） |
| 可扩展性 | 高（加监听器即可） | 低（要改代码） |
| 执行方式 | 可以异步（`@Async`） | 同步 |
| 适用场景 | 解耦的后续动作 | 强一致的逻辑 |

## 正确使用方式

### 1. 自定义 Health Indicator

```java
@Component
public class RedisHealthIndicator implements HealthIndicator {

    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    @Override
    public Health health() {
        try {
            redisTemplate.opsForValue().get("health-check");
            return Health.up().withDetail("redis", "available").build();
        } catch (Exception ex) {
            return Health.down().withDetail("redis", ex.getMessage()).build();
        }
    }
}

// 访问 /actuator/health → 显示 Redis 的健康状态
```

---

### 2. 自定义配置属性（类型安全的配置）

```java
@ConfigurationProperties(prefix = "myapp")
@Validated
public class MyAppProperties {

    @NotNull
    private String appKey;

    @Min(1)
    @Max(100)
    private int retryCount = 3;

    // getter / setter
}

@Configuration
@EnableConfigurationProperties(MyAppProperties.class)
public class MyAppConfig {
    @Autowired
    private MyAppProperties props;
}
```

---

### 3. 自定义事件

```java
// 继承 ApplicationEvent（Spring 4.2+ 可以不继承）
public class OrderCreatedEvent {
    private final Order order;

    public OrderCreatedEvent(Order order) {
        this.order = order;
    }
    // getter
}

// 发布（Spring 4.2+ 可以发布任意对象，不一定非要继承 ApplicationEvent）
@Autowired
private ApplicationEventPublisher publisher;

public void createOrder(OrderDTO dto) {
    Order order = orderRepository.save(toEntity(dto));
    publisher.publishEvent(new OrderCreatedEvent(order));
}
```

## 边界情况和坑

### 1. Actuator 端点暴露的安全风险

```yaml
# ❌ 错误：暴露了敏感端点
management:
  endpoints:
    web:
      exposure:
        include: "*"  # 暴露所有端点，包括 /env、/beans

# ✅ 正确：只暴露必要的端点
management:
  endpoints:
    web:
      exposure:
        include: health,info,metrics
```

> **生产环境建议**：  
> - 不要暴露 `/env`、`/beans`、`/mappings` 等敏感端点  
> - 暴露的端点要加权限控制（Spring Security）

---

### 2. 事件监听器中的异常处理

```java
@EventListener
public void handleEvent(MyEvent event) {
    throw new RuntimeException("处理失败");
    // 默认：异常会传播到发布者 → publisher.publishEvent() 抛异常
}
```

**解决**：用 `@Async` 异步处理，或者 `try-catch`。

```java
@EventListener
@Async  // 异步执行，异常不会传播到发布者
public void handleEvent(MyEvent event) {
    // ...
}
```

---

### 3. 自定义 Starter 的自动配置不生效

**原因**：没有在 `META-INF/spring/...imports` 文件中注册自动配置类。

```
# Spring Boot 2.7+：META-INF/spring/
# org.springframework.boot.autoconfigure.AutoConfiguration.imports
com.mycompany.autoconfigure.MyServiceAutoConfiguration
```

> Spring Boot 2.7 之前用 `spring.factories`，2.7 之后推荐用 `imports` 文件。
