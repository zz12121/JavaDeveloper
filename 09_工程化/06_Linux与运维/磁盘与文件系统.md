# 磁盘与文件系统

> 磁盘满了导致服务宕机，是线上最「低级」却最「高频」的事故。熟练掌握磁盘排查命令，能在事故发生的第一时间定位问题。

---

## 这个问题为什么存在？

Java 应用最常见的磁盘问题：

```
事故场景 A：磁盘突然 100%
原因：日志文件没有轮转，或者有一个大文件被删除了但进程还在写入（lsof | grep deleted）

事故场景 B：IO 突然变慢
原因：磁盘满了（ext4 在利用率 > 95% 时性能急剧下降），或者磁盘硬件故障

事故场景 C：文件删除了但空间没释放
原因：文件仍被进程占用（lsof | grep deleted）
```

**为什么不能只靠「看日志」？**

磁盘问题是「系统层」问题，应用日志不会告诉你「磁盘满了」或者「IO wait 飙高」。这些信息需要从系统命令获取。

---

## 它是怎么解决问题的？

### 磁盘空间查看：`df` 和 `du`

#### `df`：查看文件系统总览（全局视角）

```bash
# 人类可读格式
df -h

# 输出：
Filesystem      Size  Used Avail Use% Mounted on
/dev/sda1        50G   20G   28G  42% /
tmpfs           7.8G     0  7.8G   0% /dev/shm
/dev/sdb1       200G  180G   20G  90% /data
```

**高频场景**：`/data` 分区使用率 90%，需要找出哪个目录占用了最多空间。

#### `du`：查看目录/文件占用（钻取视角）

```bash
# 查看当前目录下各子目录的大小
du -sh */

# 查看 /data 下最大的 10 个目录
du -h /data | sort -rh | head -10

# 只显示第一层目录的大小
du -h --max-depth=1 /data
```

**`df` vs `du` 的区别**：

| 命令 | 视角 | 速度 | 用途 |
|------|------|------|------|
| `df` | 文件系统级别（inode/块设备） | 快（读超级块） | 看「还有多少空间」 |
| `du` | 文件级别（遍历目录树） | 慢（遍历文件） | 看「哪个目录占空间」 |

### inode 用完：`df -i`

```
问题场景：df -h 显示还有 50% 空间，但创建文件报错 "No space left on device"
原因：inode 用完了（通常是小文件太多，比如日志轮转产生大量小文件）
```

```bash
# 查看 inode 使用情况
df -i

# 输出：
Filesystem      Inodes  IUsed   IFree IUse% Mounted on
/dev/sda1      3276800 310000 2966800   10% /
```

**解决**：找出哪个目录有很多小文件：

```bash
# 统计每个目录下的文件数（粗略估计 inode 占用）
for dir in /*; do echo "$dir: $(find $dir -xdev 2>/dev/null | wc -l)"; done | sort -t: -k2 -rn | head -10
```

### 磁盘 IO 性能：`iostat` 和 `iotop`

#### `iostat`：磁盘 IO 统计

```bash
# 每秒刷新一次，共显示 5 次
iostat -x 1 5

# 关键输出：
Device   r/s    w/s    rkB/s    wkB/s  %util
sda      12.3   45.6   1024     2048    15.2
```

| 指标 | 含义 | 异常阈值 |
|------|------|---------|
| `r/s`, `w/s` | 每秒读写次数（IOPS） | 取决于磁盘类型（SSD 通常 > 10000） |
| `rkB/s`, `wkB/s` | 每秒读写吞吐量 | — |
| `await` | IO 平均等待时间（包括队列时间 + 服务时间） | > 100ms 说明磁盘跟不上 |
| `%util` | 磁盘繁忙百分比 | > 80% 说明磁盘是瓶颈 |

#### `iotop`：按进程查看 IO 占用（类似 `top`，但是 IO 视角）

```bash
# 实时查看 IO 占用最高的进程
iotop

# 只显示有 IO 活动的进程
iotop -o
```

### 文件系统结构基础

```
Linux 文件系统层级：
应用读写文件
    ↓
VFS（虚拟文件系统，统一接口）
    ↓
具体文件系统（ext4 / XFS / ZFS）
    ↓
块设备（/dev/sda1）
```

**为什么需要 VFS？**

让应用不用关心底层是 ext4 还是 XFS，「一切皆文件」的抽象就是 VFS 提供的。

## 深入原理

### `du` 遍历 vs `df` 读超级块

`df` 只读取文件系统的超级块（superblock），O(1) 时间。`du` 需要遍历目录树中的所有文件，O(n) 时间。

**实际差异**：在有很多小文件的目录下，`du` 可能需要几分钟，而 `df` 瞬间完成。

### ext4 vs XFS vs ZFS

| 维度 | ext4 | XFS | ZFS |
|------|------|-----|-----|
| 最大文件系统 | 1EB | 8EB | 256ZB |
| 单文件最大 | 16TB | 8EB | 16EB |
| 性能 | 中等 | 大文件性能好 | 很高，但内存占用高 |
| 数据完整性 | 中等（日志） | 中等 | 很高（校验和 + 自愈） |
| 管理复杂度 | 低 | 低 | 高 |

**Java 应用推荐**：
- 普通应用：`ext4`（足够用，运维熟悉）
- 大文件存储（日志/数据文件）：`XFS`（性能更好）
- 需要数据完整性的场景（数据库）：`ZFS`（但内存占用高，需要权衡）

## 正确使用方式

### 1. 线上磁盘满的标准排查流程

```bash
# Step 1: 确认哪个分区满了
df -h

# Step 2: 找到占用空间最大的目录
du -h --max-depth=1 /data | sort -rh | head -10

# Step 3: 逐层钻取，找到大文件
du -h --max-depth=1 /data/logs | sort -rh | head -10

# Step 4: 确认大文件是否可以删除
ls -lh /data/logs/bigfile.log

# Step 5: 如果文件正在被写入，不要用 rm，用清空
# rm 会导致进程继续写入已删除的文件（空间不释放）
> /data/logs/bigfile.log   # 清空文件内容，空间立即释放
```

### 2. 查找并清理「已删除但未释放」的文件

```bash
# 找出被删除但仍被进程占用的文件
lsof | grep deleted

# 输出：
java  12345  myapp  42w   REG  8,1  10G   /data/logs/app.log (deleted)
# 含义：PID 12345 的 java 进程，打开了 /data/logs/app.log（已删除），文件描述符是 42

# 解决方案：
# 方案 A：重启进程（最彻底）
systemctl restart myapp

# 方案 B：通过 /proc 清空文件（不需要重启）
echo "" > /proc/12345/fd/42   # 通过文件描述符清空文件内容
```

### 3. 用 `find` 定期清理旧日志

```bash
# 删除 7 天前的日志文件
find /data/logs -name "*.log" -mtime +7 -delete

# 更安全：先预览，再执行
find /data/logs -name "*.log" -mtime +7 -ls   # 先预览
find /data/logs -name "*.log" -mtime +7 -delete  # 确认无误后删除
```

### 4. 监控磁盘 inode 使用率

```bash
# 加入定时任务，inode 使用率 > 80% 时告警
df -i | awk '$5 > 80 {print "WARNING: inode usage on", $1, "is", $5}'
```

## 边界情况和坑

### 1. `du` 和 `df` 显示不一致

```
现象：df -h 显示磁盘满了，但 du -sh / 只有 50% 的占用
原因：
  1. 文件被删除了但进程还在占用（lsof | grep deleted）
  2. 有文件在另一个 mount namespace 里（Docker 容器场景）
  3. 文件系统有保留空间（ext4 默认保留 5% 给 root 用户）
```

**解决**：

```bash
# 检查保留空间
tune2fs -l /dev/sda1 | grep "Reserved block count"

# 减少保留空间（谨慎！）
tune2fs -m 1 /dev/sda1   # 保留 1%（默认 5%）
```

### 2. `du` 统计很慢

```bash
# 问题：/data 下有百万级文件，du 需要遍历很久
du -h /data

# 解决：先用 ncdu（可视化的 du，支持导航）
ncdu /data
# 或者用 find + awk 粗略估计
find /data -type f -exec stat -c "%s" {} \; | awk '{sum += $1} END {print sum/1024/1024/1024 " GB"}'
```

### 3. 文件系统只读（Read-Only）

```
现象：写入文件时报错 "Read-only file system"
原因：内核检测到磁盘错误，自动将文件系统设为只读（保护数据）
```

**解决**：

```bash
# 查看内核日志，确认原因
dmesg | grep -i error

# 尝试重新挂载为读写
mount -o remount,rw /

# 如果失败，需要 fsck（需要在单用户模式下进行，生产环境谨慎！）
```

### 4. LVM 扩容后 `df` 看不到

```bash
# 问题：用 lvextend 扩容了逻辑卷，但 df -h 显示没变
lvextend -L +10G /dev/vg00/lvol1
df -h   # 大小没变！

# 原因：lvextend 只扩容了逻辑卷，文件系统还没扩容
# 解决：扩容文件系统
resize2fs /dev/vg00/lvol1   # ext4
xfs_growfs /data              # XFS（挂载点路径）
```
