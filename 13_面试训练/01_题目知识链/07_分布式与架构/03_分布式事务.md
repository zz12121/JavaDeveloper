# 分布式事务有哪些解决方案？

> ⚠️ **先盲答**：什么是分布式事务？为什么单机数据库事务在分布式场景下不够用了？

---

## 盲答引导

1. 分布式事务的「原子性」为什么难保证？—— 涉及多个节点/数据库
2. 2PC（两阶段提交）的两个阶段是什么？—— Prepare → Commit
3. 2PC 有什么问题？—— 协调者单点 / 同步阻塞 / 数据不一致风险
4. TCC / Saga / 本地消息表 / AT 模式 分别是什么？各自适合什么场景？

---

## 知识链提示

```
分布式事务
  → [[分布式事务]]
    → 2PC（两阶段提交）
      → Prepare：所有参与者投票（准备好，锁资源）
      → Commit/Rollback：协调者决定提交或回滚
      → 问题
        → 协调者单点（挂了，参与者锁资源一直不释放）
        → 同步阻塞（所有参与者在提交阶段必须等待）
        → 数据不一致（部分参与者收到 commit，部分没收到）
      → XA 协议：数据库层面的 2PC（MySQL XA / Oracle XA）
    → TCC（Try-Confirm-Cancel）
      → Try：预留资源（冻结/预扣）
      → Confirm：确认使用（资源真正扣减）
      → Cancel：取消（回滚预留）
      → Seata TCC 模式
    → Saga 模式
      → 编排式：按顺序执行本地事务，失败则逆序补偿
      → 适用于长事务（跨几十个服务的场景）
      → 无隔离性（业务补偿）
    → 本地消息表（事务性消息）
      → RocketMQ / Kafka 的事务消息
      → 本地事务 + 消息表，先落库再发消息
    → AT 模式（Seata 默认）
      → 自动生成反向 SQL 补偿（对业务零侵入）
      → 无 AT 补偿能力，降级为 MT
```

---

## 核心追问

| 追问 | 考点 |
|------|------|
| 为什么说 2PC 的「协调者」是单点？ | 协调者挂了，所有参与者的 Prepare 状态锁资源 |
| TCC 的「空回滚」和「悬挂」是什么？ | 空回滚：Try 没执行，Cancel 先执行；悬挂：Confirm 先执行，Try 后执行 |
| 本地消息表和 RocketMQ 事务消息有什么区别？ | 本地消息表需要自己实现，RocketMQ 内置事务消息 |
| AT 模式和 TCC 模式哪个性能更好？ | AT 模式（自动补偿，SQL 改写），TCC（资源预留，性能更好） |

---

## 参考答案要点

**四大方案对比**：

| 方案 | 性能 | 侵入性 | 隔离性 | 适用场景 |
|------|------|--------|--------|---------|
| 2PC/XA | 低（同步阻塞） | 低（数据库支持） | 强 | 强一致性要求 |
| TCC | 高 | 高（业务改造） | 中 | 高并发、跨服务 |
| Saga | 高 | 中（补偿逻辑） | 低 | 长事务 |
| 本地消息表 | 中 | 中 | 低 | 最终一致性 |
