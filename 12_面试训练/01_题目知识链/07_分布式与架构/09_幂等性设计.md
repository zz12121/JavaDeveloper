# 如何设计幂等性？

> ⚠️ **先盲答**：什么是幂等性？哪些操作天然幂等？哪些不幂等？

---

## 盲答引导

1. 幂等性和防重复提交是一回事吗？有什么区别？
2. 哪些 HTTP 方法是天然幂等的？哪些不是？—— GET/DELETE/PUT 是，POST 不是
3. 接口幂等的实现方案有哪些？—— 数据库唯一索引 / 状态机 / Token 机制 / 去重表
4. 下单接口怎么设计幂等？—— 订单号作为业务唯一键

---

## 知识链提示

```
幂等性
  → [[幂等性]]
    → 定义：同一操作执行多次，结果和执行一次相同
    → 天然幂等：DELETE（再删返回 404）、PUT（覆盖）、GET（读）、HEAD
    → 非天然幂等：POST（创建）、PATCH（修改）
    → 典型非幂等场景
      → 支付扣款：重复扣款
      → 订单创建：重复下单
      → 消息消费：重复消费
      → [[消息队列]] 消费幂等
    → 幂等实现方案
      → 数据库唯一索引：订单号唯一 → 重复插入报错（DupKey）
      → 状态机：订单状态流转（待支付→已支付，不可逆）
      → Token 机制：先获取 Token，提交时验证 Token（只生效一次）
        → Redis SETNX 存 Token
      → 去重表：把每次请求的 ID + 参数哈希存表
      → 乐观锁：UPDATE SET amount = amount - #{delta}, version = version + 1 WHERE version = #{old}
```

---

## 核心追问

| 追问 | 考点 |
|------|------|
| 消息队列重复消费怎么保证幂等？ | 消费前查数据库 / Redis 去重 / 每条消息带全局唯一 ID |
| 支付回调怎么设计幂等？ | 支付流水号（out_trade_no）做唯一索引，重复回调返回成功 |
| 乐观锁的 CAS 在高并发下有性能问题吗？ | 有，自旋失败会多，但比 synchronized 轻量 |
| Token 机制的 Token 存 Redis，Redis 挂了怎么办？ | Token 验证失败 → 降级为不幂等（接受小概率重复提交）|

---

## 参考答案要点

**五种幂等方案**：

| 方案 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| 唯一索引 | 订单/支付流水 | 简单可靠 | 侵入数据库 |
| 状态机 | 有明确状态流转的实体 | 语义清晰 | 需要设计好状态流转 |
| Token | 防重复提交 | 通用 | 多一次请求 |
| 去重表 | 高并发幂等 | 可靠 | 多一次 DB 写入 |
| 乐观锁 | 更新操作 | 无额外存储 | 并发高时可能重试 |

**实战组合**：唯一索引（数据库层面兜底）+ 业务层 Token（防前端重复提交）。
