**IPC 方式（7 种）**：

| 方式 | 速度 | 能否跨机器 | 典型场景 |
|------|------|-----------|---------|
| 管道（Pipe） | 中 | ❌ | 父子进程通信（ls \| grep） |
| 命名管道（FIFO） | 中 | ❌ | 无父子关系的进程通信 |
| 共享内存 | **最快** | ❌ | 高频大数据交换（如数据库） |
| 消息队列 | 中 | ❌ | 异步解耦（进程间） |
| Socket | 慢 | ✅ | 跨机器通信（最通用） |
| Signal | 快 | ❌ | 进程间简单通知（如 kill -9） |
| Semaphore / Mutex | 快 | ❌ | 进程间同步（类似线程锁） |

**共享内存为什么最快？**：

```
其他 IPC（管道、消息队列、Socket）：
  发送方 → 写内核缓冲区 → 接收方 → 读内核缓冲区
  （两次拷贝：用户态 → 内核态 → 用户态）

共享内存：
  发送方和接收方直接读写同一块物理内存
  （零拷贝！只需要一次内存写入）
  风险：需要自己处理同步（用 Semaphore / Mutex）
```

**CPU 调度算法**：

```
① FCFS（先来先服务）：
  - 谁先到谁先执行
  - 缺点：短任务可能等很久（被长任务堵住）

② SJF（最短作业优先）：
  - 选运行时间最短的任务先执行
  - 优点：平均等待时间最短
  - 缺点：长任务可能永远得不到执行（饥饿）

③ 优先级调度：
  - 给每个任务分配优先级，高优先级先执行
  - 缺点：低优先级任务饥饿
  - 解决：动态调整优先级（老化算法）

④ 时间片轮转（RR）：
  - 每个任务分配固定时间片（如 100ms）
  - 时间片用完 → 放到队列末尾
  - 优点：公平，适合交互式系统
  - 缺点：时间片设太大 → 响应慢；设太小 → 切换开销大

⑤ 多级反馈队列（MLFQ）★★★：
  - 多个队列，优先级从高到低
  - 新任务进入最高优先级队列
  - 执行一个时间片后没结束 → 降级到下一队列
  - 低优先级队列时间片更大（照顾长任务）
  - 定期把所有任务提升到最高队列（防止饥饿）
  - 被认为是最好的通用调度算法
```

**Java 线程的上下文切换**：

```java
// Object.wait() / LockSupport.park()
// → 当前线程进入 WAITING / TIMED_WAITING
// → 操作系统把 CPU 让给其他线程
// → 涉及：用户态 → 内核态 → 上下文切换 → 内核态 → 用户态

// Thread.yield()
// → 建议操作系统重新调度（不一定真的让出 CPU）
// → 不放弃锁，只是让出 CPU

// Thread.sleep(0)
// → 和 yield() 类似，但保证当前线程休息至少 0ms
// → 触发重新调度
```
