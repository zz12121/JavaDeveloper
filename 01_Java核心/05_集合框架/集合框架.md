# 集合框架

> 集合框架不是「API 速查表」，是 Java 设计者解决「如何组织和高效访问数据」这个问题的系统性答案。理解设计动机，比背下 API 重要一百倍。

---

## 这个问题为什么存在？

没有集合框架之前，你需要什么？

- 需要「可变长序列」？自己写一个动态数组，手动扩容。
- 需要「键值对映射」？自己写一个哈希表，处理冲突。
- 需要「不重复集合」？自己写一个数组，每次插入前遍历检查。

**问题不是「能不能写」，而是「每个人都在重复造同一个轮子」**——而且大多数人造出来的轮子有 bug。

Java 集合框架（JCF）解决的是两个根本问题：

1. **数据结构的抽象**：统一接口，让代码不依赖具体实现。换 LinkedList 改 ArrayList，调用方不用动。
2. **通用算法的复用**：`Collections.sort()`、`Collections.shuffle()` 对任何 `List` 都有效，不需要为每种数据结构重写。

## 核心脉络

### 接口继承体系

```
Iterable              — 可遍历（iterator()）
  └── Collection      — 单值集合根接口
        ├── List      — 有序、可重复、可索引
        ├── Set       — 无序（除 LinkedHashSet）、去重
        └── Queue    — 队列操作（FIFO / 优先级）
  Map                  — 键值对（注意：Map 不继承 Collection）
```

**为什么 Map 不继承 Collection？** 因为 Map 的核心操作（put/get）跟 Collection（add/remove）语义完全不同，硬塞在一起会让两个接口都变得别扭。这是设计上的诚实——不愿意为了「看起来统一」而牺牲语义清晰。
