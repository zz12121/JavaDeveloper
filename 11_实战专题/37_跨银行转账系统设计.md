# 跨银行转账系统设计

> 跨行转账核心：分布式事务（TCC/SAGA）+ 幂等设计 + 对账系统。核心矛盾是**跨行操作无法用本地事务保证一致性，需要分布式事务补偿**。

---

## 场景 A：TCC方案实现

### 现象

```
跨行转账涉及两个银行系统
本行扣款成功，跨行转账失败 → 用户钱没了，投诉
跨行转账成功，本行扣款失败 → 银行资损
两个操作必须用分布式事务保证一致性
```

### 解决方案

```java
// 1. TCC 接口定义
public interface TransferTccService {
    // Try：尝试冻结资金
    boolean tryTransfer(String bizId, Long fromUserId, Long toUserId, BigDecimal amount);
    
    // Confirm：确认转账（真正扣款+入账）
    boolean confirmTransfer(String bizId);
    
    // Cancel：取消转账（解冻资金）
    boolean cancelTransfer(String bizId);
}

// 2. Try 阶段实现
@Service
public class TransferTccServiceImpl implements TransferTccService {
    
    @Autowired
    private AccountMapper accountMapper;
    
    @Autowired
    private TransferRecordMapper recordMapper;
    
    @Override
    @Transactional
    public boolean tryTransfer(String bizId, Long fromUserId, Long toUserId, BigDecimal amount) {
        // 1. 幂等检查：事务状态表
        TransferRecord record = recordMapper.selectByBizId(bizId);
        if (record != null) {
            return record.getStatus() == TransferStatus.TRY_SUCCESS;
        }
        
        // 2. 校验账户余额
        Account fromAccount = accountMapper.selectForUpdate(fromUserId);
        if (fromAccount.getBalance().compareTo(amount) < 0) {
            throw new BizException("余额不足");
        }
        
        // 3. 冻结资金（不真正扣款，只冻结）
        accountMapper.freezeAmount(fromUserId, amount);
        
        // 4. 记录事务状态（TRY成功）
        TransferRecord newRecord = new TransferRecord();
        newRecord.setBizId(bizId);
        newRecord.setFromUserId(fromUserId);
        newRecord.setToUserId(toUserId);
        newRecord.setAmount(amount);
        newRecord.setStatus(TransferStatus.TRY_SUCCESS);
        newRecord.setCreateTime(new Date());
        recordMapper.insert(newRecord);
        
        return true;
    }
    
    @Override
    @Transactional
    public boolean confirmTransfer(String bizId) {
        TransferRecord record = recordMapper.selectByBizId(bizId);
        if (record == null || record.getStatus() != TransferStatus.TRY_SUCCESS) {
            throw new BizException("事务状态异常，无法确认");
        }
        
        // 1. 确认扣款（从冻结转到实际扣款）
        accountMapper.confirmDeduct(record.getFromUserId(), record.getAmount());
        
        // 2. 调用跨行通道（实际跨行转账）
        boolean success = crossBankChannel.transfer(
                record.getFromUserId(), record.getToUserId(), record.getAmount());
        
        if (!success) {
            throw new BizException("跨行转账失败，触发Cancel");
        }
        
        // 3. 跨行入账（对方账户加钱）
        accountMapper.addBalance(record.getToUserId(), record.getAmount());
        
        // 4. 更新事务状态为 CONFIRMED
        recordMapper.updateStatus(bizId, TransferStatus.CONFIRMED);
        
        return true;
    }
    
    @Override
    @Transactional
    public boolean cancelTransfer(String bizId) {
        TransferRecord record = recordMapper.selectByBizId(bizId);
        if (record == null) {
            return true;  // 没记录，直接返回成功
        }
        
        if (record.getStatus() == TransferStatus.CONFIRMED) {
            return true;  // 已确认，无需取消
        }
        
        // 解冻资金
        accountMapper.unfreezeAmount(record.getFromUserId(), record.getAmount());
        
        // 更新状态为 CANCELED
        recordMapper.updateStatus(bizId, TransferStatus.CANCELED);
        
        return true;
    }
}
```

**TCC 关键点**：
- Try 阶段只冻结资源，不真正执行业务
- Confirm 和 Cancel 必须幂等（可能重复调用）
- 事务状态表记录中间状态，用于恢复

## 场景 B：SAGA方案实现

### 现象

```
TCC 方案侵入性强，需要改造所有参与方接口
跨行通道（如银联）不支持 TCC，只提供正向接口
需要一种补救式的分布式事务方案
SAGA：先执行正向操作，失败了用补偿操作回滚
```

### 解决方案

```java
// 1. SAGA 编排器实现
@Service
public class TransferSagaCoordinator {
    
    @Autowired
    private AccountMapper accountMapper;
    
    @Autowired
    private CrossBankChannel crossBankChannel;
    
    @Autowired
    private SagaLogMapper sagaLogMapper;
    
    // SAGA 执行：一系列正向操作 + 对应的补偿操作
    public boolean transfer(Long fromUserId, Long toUserId, BigDecimal amount) {
        String sagaId = UUID.randomUUID().toString();
        
        try {
            // 步骤1：本行扣款
            sagaLogMapper.log(sagaId, "deduct", fromUserId, amount, SagaStatus.STARTED);
            accountMapper.deduct(fromUserId, amount);
            sagaLogMapper.log(sagaId, "deduct", fromUserId, amount, SagaStatus.SUCCESS);
            
            // 步骤2：跨行转账
            sagaLogMapper.log(sagaId, "crossTransfer", toUserId, amount, SagaStatus.STARTED);
            boolean success = crossBankChannel.transfer(fromUserId, toUserId, amount);
            if (!success) {
                throw new BizException("跨行转账失败");
            }
            sagaLogMapper.log(sagaId, "crossTransfer", toUserId, amount, SagaStatus.SUCCESS);
            
            // 步骤3：对方账户入账（如果跨行通道没做的话）
            sagaLogMapper.log(sagaId, "credit", toUserId, amount, SagaStatus.STARTED);
            accountMapper.addBalance(toUserId, amount);
            sagaLogMapper.log(sagaId, "credit", toUserId, amount, SagaStatus.SUCCESS);
            
            return true;
            
        } catch (Exception e) {
            // 失败：执行补偿操作（反向顺序）
            compensate(sagaId, fromUserId, toUserId, amount);
            return false;
        }
    }
    
    // 补偿操作：反向执行
    private void compensate(String sagaId, Long fromUserId, Long toUserId, BigDecimal amount) {
        // 查询已成功的步骤，执行对应补偿
        List<SagaLog> logs = sagaLogMapper.selectBySagaId(sagaId);
        
        for (int i = logs.size() - 1; i >= 0; i--) {
            SagaLog log = logs.get(i);
            if (log.getStatus() != SagaStatus.SUCCESS) {
                continue;  // 没成功的步骤不需要补偿
            }
            
            switch (log.getAction()) {
                case "deduct":
                    // 补偿：退款
                    accountMapper.addBalance(log.getUserId(), log.getAmount());
                    sagaLogMapper.log(sagaId, "refund", log.getUserId(), log.getAmount(), SagaStatus.COMPENSATED);
                    break;
                    
                case "crossTransfer":
                    // 补偿：跨行冲正（调用冲正接口）
                    crossBankChannel.reverse(log.getUserId(), log.getAmount());
                    sagaLogMapper.log(sagaId, "reverse", log.getUserId(), log.getAmount(), SagaStatus.COMPENSATED);
                    break;
                    
                case "credit":
                    // 补偿：扣回入账
                    accountMapper.deduct(log.getUserId(), log.getAmount());
                    sagaLogMapper.log(sagaId, "debit", log.getUserId(), log.getAmount(), SagaStatus.COMPENSATED);
                    break;
            }
        }
    }
}
```

**SAGA vs TCC**：
- SAGA：侵入性低，补偿操作由业务提供，适合跨行等不可控场景
- TCC：强一致，但需要参与方支持 Try/Confirm/Cancel 接口
- 选择：跨行转账用 SAGA（银联不提供 TCC 接口）

## 场景 C：幂等设计（防重复转账）

### 现象

```
网络超时，用户重复点击转账按钮
同一个转账请求被发送多次
钱被扣了多次 → 用户投诉，需要人工退款
必须保证转账接口幂等：同一请求只执行一次
```

### 解决方案

```java
// 1. 幂等实现：基于唯一业务流水号（bizId）
@RestController
public class TransferController {
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    @Autowired
    private TransferService transferService;
    
    @PostMapping("/transfer")
    public Result transfer(@RequestBody TransferRequest request) {
        // 1. 生成或获取幂等键（由客户端生成，或服务端生成后返回）
        String idempotentKey = "transfer:idempotent:" + request.getBizId();
        
        // 2. 用 Redis SET NX EX 实现幂等锁（5分钟有效期）
        Boolean locked = redisTemplate.opsForValue()
                .setIfAbsent(idempotentKey, "PROCESSING", 5, TimeUnit.MINUTES);
        
        if (locked == null || !locked) {
            // 可能正在处理中，或已处理完成
            String status = redisTemplate.opsForValue().get(idempotentKey);
            if ("SUCCESS".equals(status)) {
                return Result.success("转账已成功，请勿重复提交");
            } else if ("PROCESSING".equals(status)) {
                return Result.error("转账处理中，请稍后查询");
            } else {
                return Result.error("重复请求");
            }
        }
        
        try {
            // 3. 执行转账
            boolean success = transferService.transfer(
                    request.getFromUserId(),
                    request.getToUserId(),
                    request.getAmount());
            
            if (success) {
                redisTemplate.opsForValue().set(idempotentKey, "SUCCESS", 5, TimeUnit.MINUTES);
                return Result.success("转账成功");
            } else {
                redisTemplate.delete(idempotentKey);  // 失败允许重试
                return Result.error("转账失败");
            }
            
        } catch (Exception e) {
            redisTemplate.delete(idempotentKey);  // 异常允许重试
            throw e;
        }
    }
}
```

```sql
-- 2. DB 层幂等：唯一索引保证
-- 转账流水表，biz_id 唯一
CREATE TABLE transfer_record (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    biz_id VARCHAR(64) NOT NULL COMMENT '业务流水号（幂等键）',
    from_user_id BIGINT NOT NULL,
    to_user_id BIGINT NOT NULL,
    amount DECIMAL(15, 2) NOT NULL,
    status VARCHAR(20) NOT NULL,
    create_time DATETIME NOT NULL,
    UNIQUE KEY uk_biz_id (biz_id)
);

-- 插入时如果 biz_id 重复，报唯一索引冲突错误
INSERT INTO transfer_record (biz_id, from_user_id, to_user_id, amount, status, create_time)
VALUES (?, ?, ?, ?, 'PROCESSING', NOW());
```

**幂等设计要点**：
- **客户端生成 bizId**：用户点击转账时生成，携带在请求中
- **服务端双重校验**：Redis 做分布式锁，DB 唯一索引兜底
- **结果缓存**：转账结果缓存到 Redis，重复请求直接返回结果

## 场景 D：对账系统设计

### 现象

```
本行扣款成功，跨行通道转账失败 → 数据不一致
跨行通道转账成功，本行没收到回调 → 数据不一致
日终发现账不平，需要人工对账
必须建立自动化对账系统，及时发现差异
```

### 解决方案

```java
// 1. 对账任务：对比本行流水和跨行通道流水
@Component
public class ReconciliationScheduler {
    
    @Autowired
    private TransferRecordMapper localMapper;
    
    @Autowired
    private CrossBankChannel crossBankChannel;
    
    @Autowired
    private ReconciliationMapper reconMapper;
    
    @Scheduled(cron = "0 30 2 * * ?")  // 每天凌晨2:30执行（跨行通道日终后）
    public void reconcile() {
        LocalDate yesterday = LocalDate.now().minusDays(1);
        
        // 1. 获取本行昨天的转账流水
        List<TransferRecord> localRecords = localMapper.selectByDate(yesterday);
        
        // 2. 获取跨行通道昨天的转账流水
        List<CrossBankRecord> remoteRecords = crossBankChannel.queryRecords(yesterday);
        
        // 3. 对账：以本行流水为基准
        for (TransferRecord local : localRecords) {
            CrossBankRecord remote = findMatchingRecord(remoteRecords, local.getBizId());
            
            if (remote == null) {
                // 差异：本行有，跨行通道没有 → 可能跨行转账失败
                handleDifference(local, null, "LOCAL_ONLY");
                
            } else if (!isAmountMatch(local, remote)) {
                // 差异：金额不匹配
                handleDifference(local, remote, "AMOUNT_MISMATCH");
                
            } else if (!isStatusMatch(local, remote)) {
                // 差异：状态不匹配
                handleDifference(local, remote, "STATUS_MISMATCH");
            }
        }
        
        // 4. 检查跨行通道有但本行没有的记录（挂账）
        for (CrossBankRecord remote : remoteRecords) {
            TransferRecord local = localMapper.selectByBizId(remote.getBizId());
            if (local == null) {
                handleDifference(null, remote, "REMOTE_ONLY");
            }
        }
    }
    
    private void handleDifference(TransferRecord local, CrossBankRecord remote, String diffType) {
        // 记录差异
        ReconciliationRecord diff = new ReconciliationRecord();
        diff.setLocalBizId(local != null ? local.getBizId() : null);
        diff.setRemoteBizId(remote != null ? remote.getBizId() : null);
        diff.setDiffType(diffType);
        diff.setStatus("PENDING");  // 待处理
        diff.setCreateTime(new Date());
        reconMapper.insert(diff);
        
        // 告警（金额大于阈值）
        if (isAmountLarge(local, remote)) {
            alarmService.send("发现大额对账差异: bizId=" + 
                    (local != null ? local.getBizId() : remote.getBizId()));
        }
    }
}
```

```sql

    local_amount DECIMAL(15, 2),
```

**对账优化**：
- **实时对账**：跨行回调时立即核对，发现差异及时处理
- **日终对账**：每天凌晨跑全量核对（兜底）
- **差错处理**：小额差异自动修复，大额差异人工介入

## 核心参数估算

```
转账量估算：
  假设：日均转账 100万笔，峰值 10万笔/小时
  跨行占比 30% → 日均跨行 30万笔
  
分布式事务耗时：
  TCC：Try 50ms + Confirm 100ms = 150ms/笔
  SAGA：扣款 50ms + 跨行 500ms + 入账 50ms = 600ms/笔
  
对账耗时：
  30万笔流水，逐笔对比 → 约 30分钟（单线程）
  优化：多线程并行对比 → 压缩到 5分钟内
```

## 追问链

### 追问 1：TCC 和 SAGA 怎么选？

> "看参与方是否可控。**可控参与方**（本行系统）：用 TCC，强一致，性能好。**不可控参与方**（银联、第三方支付）：用 SAGA，因为对方不支持 TCC 接口。实际生产中常混合使用：本行内部用 TCC，跨行用 SAGA。"

### 追问 2：幂等键（bizId）怎么生成？

> "两种方案：1. **客户端生成**：用户点击转账时，前端生成 UUID 作为 bizId，携带在请求中。优点：服务端无状态；缺点：客户端需保证生成唯一。2. **服务端生成**：转账接口返回 bizId，后续查询/通知用这个 bizId。推荐方案一（客户端生成），更符合幂等语义。"

### 追问 3：对账发现差异怎么处理？

> "分场景处理：1. **本行有、跨行无**：查询跨行通道状态，如果确实失败，本行退款（SAGA 补偿）。2. **跨行有、本行无**：挂账处理，人工核实后补录或退回。3. **金额/状态不匹配**：人工介入，核对原始凭证。4. **技术优化**：小额差异自动修复（如 0.01 元误差），大额告警人工处理。"

### 追问 4：跨行转账回调丢失怎么办？

> "用**对账机制兜底**：1. 回调是异步通知，可能丢失（网络故障、对方系统故障）。2. 本行先记流水为'处理中'状态。3. 每天日终对账，发现'处理中'超时的流水，主动查询跨行通道状态。4. 定时任务每小时轮询未完成的转账，主动查询状态并更新。"

## 排查 Checklist

```
□ TCC 的三个方法都幂等吗？ → Confirm/Cancel 可重复调用
□ SAGA 补偿操作能回滚吗？ → 每个正向操作都有对应补偿
□ 幂等键（bizId）唯一吗？ → Redis SET NX + DB 唯一索引
□ 转账流水记录了状态吗？ → 每次状态变更都记录日志
□ 对账任务执行了吗？ → 检查昨天的对账记录
□ 对账差异及时处理了吗？ → PENDING 状态不应超24小时
□ 跨行回调丢了能发现吗？ → 定时轮询未完成的转账
□ 并发转账会超扣吗？ → 账户表用 SELECT FOR UPDATE 或乐观锁
□ 分布式事务超时了怎么办？ → 定时任务恢复超时事务
```
