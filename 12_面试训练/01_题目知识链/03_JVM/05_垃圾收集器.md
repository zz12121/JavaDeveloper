## 知识链提示

```
垃圾收集器
  → [[垃圾收集器]]
    → 新生代收集器
      → Serial（单线程，STW）→ 客户端/单核
      → ParNew（多线程版 Serial）→ CMS 的默认新生代收集器
      → Parallel Scavenge（吞吐量优先）→ 追求高吞吐量（= 运行时间/(运行+GC时间)）
    → 老年代收集器
      → Serial Old / Parallel Old（老年代版）
      → CMS（并发标记清除，4步）
        → 初始标记（STW）→ 并发标记 → 重新标记（STW）→ 并发清除
        → 优点：并发收集，用户线程停顿短
        → 缺点：浮动垃圾 / 内存碎片 / 对 CPU 敏感
      → G1（Garbage First）
        → 把堆分成多个 Region（大小相同，1MB~32MB）
        → 追踪每个 Region 的回收价值（回收收益最大的 Region）
        → 整体：标记-整理，局部：复制 → 无内存碎片
        → [[G1调优]]
      → ZGC（不分代，JDK 11+，大内存低延迟场景）
        → 着色指针（Colored Pointers）+ 读屏障
        → 全程并发，STW 时间 < 1ms
        → [[ZGC原理]]
```

## 核心追问

| 追问 | 考点 |
|------|------|
| CMS 的「浮动垃圾」是什么？怎么处理？ | 并发清理时产生的垃圾，只能下次GC清理 |
| G1 为什么能做到可预测停顿？ | 可以设置 -XX:MaxGCPauseMillis 控制目标停顿时间 |
| ZGC 为什么可以不分代？ | 着色指针分区，并发标记 + 重新映射 |
| 生产环境怎么选垃圾收集器？ | 低延迟选 G1/ZGC，吞吐量选 Parallel |

---

## 参考答案要点

**常用组合**：
- JDK 8 默认：`Parallel Scavenge + Parallel Old`
- 低延迟场景：`ParNew + CMS`（已逐步被 G1 取代）
- JDK 11+：`G1`（默认）
- 超大内存 + 超低延迟：`ZGC`（100MB~4TB）

**CMS vs G1**：
- CMS：分老年代的「标记-清除」，有内存碎片
- G1：把堆分成多个 Region，整体「标记-整理」，局部「复制」，无碎片
