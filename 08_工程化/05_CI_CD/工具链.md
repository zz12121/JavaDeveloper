# 工具链

> CI/CD 工具链的选择，决定了「从代码到上线」的效率和可靠性。没有最好的工具，只有最适合团队规模和习惯的工具。

---

## 这个问题为什么存在？

**没有 CI/CD 工具时**：

```
开发提交代码
    ↓
手动打包：mvn clean package
手动上传：scp target/app.jar server:/opt/
手动重启：ssh server systemctl restart app
手动通知：企业微信发消息「我发布完了」
```

**问题**：
1. **重复劳动**：每次发布都要做一遍，浪费时间
2. **人为失误**：漏打包某个模块、传错服务器、忘记重启
3. **不可追溯**：谁发布了什么版本，无法审计
4. **无法回滚**：发布出现问题，手动回退到上一个版本（容易出错）

**CI/CD 工具的价值**：把「发布流程」代码化、自动化、可追溯。

---

## 它是怎么解决问题的？

### Jenkins：老牌通用 CI/CD 引擎

```
Jenkins 架构：
┌─────────────────────────────────────┐
│          Jenkins Master             │
│  任务调度 / UI 管理 / 插件管理      │
└──────────────┬──────────────────────┘
               │ 分发任务
    ┌──────────┼──────────┐
    ▼          ▼          ▼
┌────────┐ ┌────────┐ ┌────────┐
│ Agent1 │ │ Agent2 │ │ Agent3 │  ← 执行器（可以是任意机器）
└────────┘ └────────┘ └────────┘
```

**核心概念**：

| 概念 | 说明 |
|------|------|
| Job（任务） | 一个 CI/CD 流程（如「构建 + 测试 + 部署」） |
| Pipeline | 用代码（Jenkinsfile）定义流程（推荐） |
| Agent | 执行器（可以是 Docker 容器、远程机器） |
| Plugin | 扩展功能（Jenkins 有上千个插件） |
| Workspace | 每次构建的工作目录（默认保留） |

**Jenkinsfile 示例（声明式）**：

```groovy
pipeline {
    agent any

    stages {
        stage('Build') {
            steps {
                sh 'mvn clean package -DskipTests'
                archiveArtifacts artifacts: 'target/*.jar', fingerprint: true
            }
        }
        stage('Test') {
            steps {
                sh 'mvn test'
            }
        }
        stage('Deploy') {
            when {
                branch 'main'   // 只有 main 分支才部署
            }
            steps {
                sh 'scp target/app.jar user@server:/opt/'
                sh 'ssh user@server systemctl restart app'
            }
        }
    }

    post {
        success {
            slackSend channel: '#dev', message: '构建成功 ✅'
        }
        failure {
            mail to: 'team@company.com', subject: '构建失败 ❌'
        }
    }
}
```

### GitLab CI：代码仓库内置的 CI/CD

**核心概念**：`.gitlab-ci.yml` 文件定义在仓库根目录，GitLab 自动执行。

```yaml
# .gitlab-ci.yml
stages:
  - build
  - test
  - deploy

build_job:
  stage: build
  image: openjdk:17
  script:
    - mvn clean package -DskipTests
  artifacts:
    paths:
      - target/*.jar
    expire_in: 1 day

test_job:
  stage: test
  image: openjdk:17
  script:
    - mvn test
  coverage: '/Total.*?([0-9]{1,3})%/'   # 从输出中提取覆盖率

deploy_job:
  stage: deploy
  only:
    - main
  script:
    - scp target/app.jar $SERVER:/opt/
    - ssh $SERVER systemctl restart app
```

**GitLab CI 的优势**：
- 配置即代码（和代码一起版本控制）
- 天然和代码仓库集成（不需要额外配置 Webhook）
- 支持多阶段、并行执行、手动卡点

### GitHub Actions：GitHub 原生 CI/CD

**核心概念**：`.github/workflows/*.yml` 定义工作流。

```yaml
# .github/workflows/ci.yml
name: Java CI

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
    - uses: actions/checkout@v3       # 拉取代码
    - uses: actions/setup-java@v3     # 安装 JDK
      with:
        java-version: '17'
        distribution: 'temurin'
    - name: Build with Maven
      run: mvn clean package -DskipTests
    - name: Test
      run: mvn test
    - name: Upload artifact
      uses: actions/upload-artifact@v3
      with:
        name: app-jar
        path: target/*.jar
```

**GitHub Actions 的优势**：
- 市场（Marketplace）有海量现成 Action（`uses: ...`）
- 矩阵构建（Matrix Build）：一次定义，多版本并行测试
- 直接集成 GitHub（PR 状态、Checks）

### Argo CD：Kubernetes 原生 CD 工具

**核心概念**：GitOps —— 以 Git 仓库为「单一事实来源」，K8s 集群状态自动同步 Git 仓库中定义的期望状态。

```
Argo CD 工作流：
┌─────────────┐
│  Git 仓库    │  ← 期望状态（K8s YAML）
└──────┬──────┘
       │  watch（监听变更）
       ▼
┌─────────────┐
│  Argo CD     │  ← 对比期望状态 vs 实际状态
└──────┬──────┘
       │  自动同步（或手动同步）
       ▼
┌─────────────┐
│  K8s 集群   │  ← 实际状态
└─────────────┘
```

**Argo CD 的优势**：
- 可视化：Web UI 显示应用状态和健康状况
- 自动回滚：Git revert 即可回滚（不需要重新部署）
- 多集群管理：一个 Argo CD 管理多个 K8s 集群

## 它和相似方案的本质区别是什么？

### Jenkins vs GitLab CI vs GitHub Actions

| 维度 | Jenkins | GitLab CI | GitHub Actions |
|------|---------|-----------|-----------------|
| 部署方式 | 独立部署（需要自己维护） | GitLab 内置（不需要额外部署） | GitHub 内置（不需要额外部署） |
| 配置方式 | Jenkinsfile（Groovy） | `.gitlab-ci.yml`（YAML） | `.github/workflows/*.yml`（YAML） |
| 插件生态 | 极丰富（但质量参差） | 中等（内置功能够用） | 极丰富（Marketplace） |
| 学习曲线 | 陡（概念多，插件配置复杂） | 中等 | 低（YAML 直观，有示例） |
| 适用场景 | 复杂流程、多仓库协作、老项目 | GitLab 用户、私有化部署 | GitHub 用户、开源项目 |
| 维护成本 | 高（需要专人维护 Master 和 Agent） | 低（SaaS 版零维护） | 零（完全 SaaS） |

**选择建议**：
- 已用 GitLab → GitLab CI（集成最好）
- 已用 GitHub → GitHub Actions（集成最好）
- 复杂流程、多仓库、有专门运维团队 → Jenkins（灵活性最高）
- 小团队、想零维护 → GitHub Actions / GitLab CI SaaS

### Argo CD vs Spinnaker

| 维度 | Argo CD | Spinnaker |
|------|---------|------------|
| 复杂度 | 低（专注 K8s） | 高（支持多平台：K8s / ECS / GCE） |
| 学习曲线 | 低 | 陡 |
| GitOps 支持 | 原生支持 | 需要额外配置 |
| 适用场景 | K8s 环境 | 多云环境（AWS + GCP + K8s） |

**选择建议**：纯 K8s 环境 → Argo CD；多云、复杂发布策略 → Spinnaker。

## 正确使用方式

### 1. 敏感信息不要写在配置文件里

```yaml
# 错误：密码明文
script:
  - scp app.jar user:mysecretpassword@server:/opt/

# 正确：用环境变量 / Secret
script:
  - scp app.jar $USER@$SERVER:/opt/
  env:
    USER: admin
    SERVER: 192.168.1.100
    # 密码存在 CI/CD 工具的 Secret 管理里，不在代码中
```

**各工具的 Secret 管理**：
- Jenkins：Credentials → Secret Text / SSH Key
- GitLab CI：Settings → CI/CD → Variables
- GitHub Actions：Settings → Secrets and variables → Actions

### 2. 用「制品（Artifact）」在阶段间传递文件

```yaml
# GitLab CI 示例
build_job:
  stage: build
  script:
    - mvn package
  artifacts:
    paths:
      - target/app.jar    # 传递给后续 stage
    expire_in: 1 day     # 1 天后自动清理

deploy_job:
  stage: deploy
  script:
    - scp target/app.jar $SERVER:/opt/   # 直接使用 build_job 产生的 artifact
```

**为什么不用「共享存储」？**
Artifact 是「自包含」的，不依赖外部环境，更可靠。

### 3. 给部署加速：缓存依赖

```yaml
# GitHub Actions 示例：缓存 Maven 本地仓库
- name: Cache Maven dependencies
  uses: actions/cache@v3
  with:
    path: ~/.m2/repository
    key: ${{ runner.os }}-m2-${{ hashFiles('**/pom.xml') }}
    restore-keys: ${{ runner.os }}-m2-
```

**效果**：第二次构建不需要重新下载依赖，速度提升 5-10 倍。

### 4. 关键环境用「手动卡点」

```yaml
# GitLab CI 示例：生产环境部署需要手动点击确认
deploy_prod:
  stage: deploy
  script:
    - ansible-playbook deploy.yml
  when: manual   # 需要手动点击「播放」按钮才执行
  only:
    - main
```

## 边界情况和坑

### 1. Jenkins Pipeline 的「构建编号」坑

```
问题：多次构建使用同一个 Workspace，导致旧文件干扰新构建
解决：在 Jenkinsfile 开头加 `cleanWs()`（清理 Workspace）
```

```groovy
pipeline {
    agent any
    stages {
        stage('Clean') {
            steps {
                cleanWs()   // 清理 Workspace
            }
        }
        // ... 后续 stage
    }
}
```

### 2. GitHub Actions 的「超时」问题

```
问题：默认超时时间是 6 小时，长时间运行的任务会被强制终止
解决：在 workflow 中设置 timeout-minutes
```

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    timeout-minutes: 30   # 30 分钟超时
    steps:
      - run: mvn package
```

### 3. GitLab CI Runner 的「并发」问题

```
问题：多个 Job 同时在一个 Runner 上执行，端口冲突
解决：用 Docker Executor（每个 Job 独立容器，环境隔离）
```

```toml
# config.toml（GitLab Runner 配置）
[[runners]]
  executor = "docker"
  [runners.docker]
    image = "openjdk:17"
    privileged = false
```

### 4. Argo CD 的「自动同步」风险

```
问题：开启了 auto-sync，有人误提交错误的 YAML，直接同步到生产
解决：开启 sync-window（只允许特定时间同步）+ 手动审批关键环境
```

```yaml
# AppProject 配置：只允许工作日 9-18 点自动同步
spec:
  syncWindows:
    - kind: allow
      schedule: '0 9 * * 1-5'
      duration: 9h
```
