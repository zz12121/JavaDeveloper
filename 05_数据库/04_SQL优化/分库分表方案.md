# 分库分表方案

## 这个问题为什么存在？

> 单表数据量超过千万级，查询开始变慢，写入也变慢。
> 即使加了索引，数据量大到一定程度，B+树层级变深，磁盘IO次数变多。

**什么时候需要分库分表？**

| 数据量 | 表现 | 是否需要分 |
|--------|------|------------|
| < 500万行 | 性能正常 | 不需要 |
| 500万-1000万行 | 开始变慢，但可接受 | 可以考虑 |
| > 1000万行 | 明显变慢，索引效果减弱 | 需要分 |
| > 5000万行 | 性能很差，必须分 | 必须分 |

**核心原因**：
1. B+树层级变深（3层 → 4层 → 磁盘IO次数+1）
2. 二级索引变大，占用内存多
3. 备份、DDL操作变慢

---

## 它是怎么解决问题的？

### 垂直拆分 vs 水平拆分

```
垂直拆分（按列拆分）
┌────────────────────────────────┐
│  原表：users                                 │
│  id, name, email, password, age, address    │
└────────────────────────────────┘
                    ↓ 垂直拆分
┌────────────────┐  ┌──────────────────────┐
│  users_base     │  │  users_profile         │
│  id, name,     │  │  id, email,           │
│  password      │  │  age, address         │
└────────────────┘  └──────────────────────┘
优点：减少IO（常用字段放一个表）
缺点：需要JOIN，管理复杂

水平拆分（按行拆分）
┌────────────────────────────────┐
│  原表：orders                                 │
│  id, user_id, status, created_at             │
└────────────────────────────────┘
                    ↓ 水平拆分
┌────────────────┐  ┌────────────────┐
│  orders_0       │  │  orders_1       │
│  id % 2 = 0    │  │  id % 2 = 1    │
└────────────────┘  └────────────────┘
优点：每个表数据量小，性能提升
缺点：跨分片查询、分布式事务问题
```

### 分片策略

**策略1：哈希分片（hash）**

```sql
-- 按user_id哈希分16个库，每个库16个表
shard_key = hash(user_id) % 16;  -- 分到哪个库
table_key = hash(user_id) % 16;  -- 分到哪个表

-- 优点：数据分布均匀
-- 缺点：扩缩容需要数据迁移（所有数据重新分片）
```

**策略2：范围分片（range）**

```sql
-- 按user_id范围分片
user_id 1-1000000     → db_0.orders
user_id 1000001-2000000 → db_1.orders

-- 优点：扩缩容方便（加一个新范围即可）
-- 缺点：数据分布不均匀（热点数据问题）
```

**策略3：一致性哈希（推荐）**

```
一致性哈希：
1. 把[0, 2^32-1]看成一个环
2. 把节点（数据库实例）映射到环上
3. 把数据key哈希映射到环上
4. 数据存储在顺时针方向的第一个节点上

扩缩容时：只影响相邻的节点，不需要全部重新分片
```

### 分库分表带来的问题

**问题1：分布式事务**

```sql
-- 下单：需要扣减库存（库存库）+ 创建订单（订单库）
BEGIN;
UPDATE inventory SET stock = stock - 1 WHERE id = 1;  -- 库存库
INSERT INTO orders(...) VALUES (...);                  -- 订单库
COMMIT;
-- 问题：跨库事务，不能用本地事务
```

**解决**：分布式事务方案（XA、TCC、Saga、本地消息表）
→ 详见`07_分布式/分布式事务.md`

**问题2：跨分片查询**

```sql
-- 查询用户在全部订单（数据分布在16个分片）
SELECT * FROM orders WHERE user_id = 123;
-- 如果user_id是分片键：只查一个分片（很快）
-- 如果按order_id分片：需要查所有16个分片（很慢）
```

**解决方案**：
1. 避免跨分片查询（精心设计分片键）
2. 使用基因法（分片键嵌入其他ID中）
3. 使用异构索引（空间换时间）

**问题3：扩容问题**

```
哈希分片扩容：
原：4个分片（% 4）
新：8个分片（% 8）
→ 需要把原有数据重新分片（大量数据迁移）

一致性哈希扩容：
加入新节点 → 只影响顺时针方向的第一节点
→ 数据迁移量小
```

## 它和相似方案的本质区别是什么？

### 垂直拆分 vs 水平拆分

| 维度 | 垂直拆分 | 水平拆分 |
|------|---------|---------|
| 拆分维度 | 按列（字段）拆分 | 按行（记录）拆分 |
| 适用场景 | 字段太多，常用字段和不常用字段分离 | 单表数据量太大 |
| 带来的问题 | 需要JOIN查询 | 分布式事务、跨分片查询 |
| 推荐程度 | 优先考虑（问题少） | 不得已而为之 |

**本质区别**：垂直拆分是"列纬度"，水平拆分是"行纬度"。

### 哈希分片 vs 范围分片 vs 一致性哈希

| 维度 | 哈希分片 | 范围分片 | 一致性哈希 |
|------|---------|---------|------------|
| 数据分布 | 均匀 | 可能不均匀（热点） | 均匀 |
| 扩缩容 | 需要全部重新分片 | 方便（加一个范围） | 只影响相邻节点 |
| 查询需求 | 适合点查 | 适合范围查询 | 适合点查 |
| 复杂度 | 低 | 低 | 中 |

**本质区别**：权衡"数据分布均匀性"和"扩缩容成本"。

## 正确使用方式

### 1. 评估是否需要分库分表

```
1. 单表数据量是否超过1000万行？
   → 否 → 不需要分，先优化索引和SQL

2. 索引和SQL是否已经优化到极限？
   → 否 → 先优化，可能不需要分表

3. 数据库服务器IO、CPU是否已经到瓶颈？
   → 否 → 可能是索引/SQL问题，不是量级问题

4. 预估未来2年数据增长是否会超过5000万行？
   → 否 → 暂时不需要分，但要持续监控

如果以上问题都指向"需要分" → 开始设计分库分表方案
```

### 2. 选择分片键（Shard Key）

**原则**：
1. **分片键应该是最核心的查询条件**（80%的查询都用这个字段）
2. **分片键应该离散度高**（数据分布均匀）
3. **避免数据热点**（比如按创建时间分片，最近的表会成为热点）

```sql
-- 订单表
-- 错误：按created_at分片
-- → 最近的表成为热点，老表闲置

-- 正确：按user_id分片
-- → 数据分布均匀，80%的查询都是"查某个用户的订单"
```

### 3. 使用ShardingSphere（推荐）

**ShardingSphere**是Apache顶级项目，支持数据分片、读写分离、分布式事务。

**配置示例**（sharding-rule.yaml）：

```yaml
shardingRule:
  tables:
    orders:
      actualDataNodes: ds${0..1}.orders_${0..1}
      databaseStrategy:
        inline:
          shardingColumn: user_id
          algorithmExpression: ds${user_id % 2}
      tableStrategy:
        inline:
          shardingColumn: user_id
          algorithmExpression: orders_${user_id % 2}
```

**Java代码中使用**（完全透明，就像操作单表）：

```java
// 配置数据源（ShardingSphere代理）
DataSource ds = new ShardingDataSource(config);

// 后续JdbcTemplate、MyBatis等正常使用
// ShardingSphere自动路由到正确的分片
jdbcTemplate.update("INSERT INTO orders(user_id, status) VALUES (?, ?)", 
                   123, "PAID");
// → 自动路由到 ds1.orders_1
```

### 4. 数据迁移方案

**停机迁移**（简单，但有停机时间）：

```
1. 停止写入
2. 用工具（如pt-online-schema-change）迁移数据
3. 切换分片规则
4. 恢复写入
```

**平滑迁移**（无停机，但复杂）：

```
1. 双写：新数据同时写入老表和新分片表
2. 迁移工具：迁移老数据到分片表
3. 验证：对比老表和新分片表的数据一致性
4. 切换读流量：先读老表，逐步切到分片表
5. 停止老表写入
```

## 边界情况和坑

### 坑1：分片键选择错误，导致大量跨分片查询

```
场景：
- 订单表按user_id分片
- 运营需要查询"今天的所有订单"（跨分片）

结果：
- 需要查询所有分片，性能极差
- 分片的好处完全没用上
```

**解决**：
1. 如果这类查询多，考虑异构索引（再建一张按时间分片的表）
2. 或者把这类查询放到OLAP系统（如ClickHouse）

### 坑2：扩容时数据迁移量大

```
场景：
- 原先按月分12个分片
- 数据量增长，需要扩到24个分片
- 所有数据都需要重新分片

结果：
- 数据迁移量大，耗时长
- 迁移期间性能受影响
```

**解决**：
1. 提前规划，预留足够分片数（比如一开始就分64片，但只放在2个物理库）
2. 使用一致性哈希（扩缩容时只迁移部分数据）

### 坑3：分布式事务的复杂性

```
场景：
- 下单：扣库存（库存库）+ 创建订单（订单库）
- 扣库存成功，创建订单失败 → 数据不一致
```

**解决**：分布式事务方案
- XA协议（强一致，但性能差）
- TCC（Try-Confirm-Cancel，最终一致）
- Saga（长流程事务，补偿机制）
- 本地消息表（基于消息队列，最终一致）

需要根据业务一致性要求选择方案。

### 坑4：分片数设置不合理

```
分片数太少（比如只有4个）：
→ 后续扩容麻烦（需要重新分片）

分片数太多（比如1024个）：
→ 运维复杂（管理1024张表）
→ 跨分片查询耗时长（需要查1024个分片）
```

**经验规则**：
- 前期：分16-32个分片（放在2-4个物理库）
- 预留：可以后续加物理库，但分片数不变（只是把分片迁移到新库）
