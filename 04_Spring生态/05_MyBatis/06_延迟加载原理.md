# 延迟加载原理

## 这个问题为什么存在？

> 查询 `User` 时，你不需要它的 `List<Order>` 属性，但 MyBatis 默认把关联的订单也查出来了——浪费性能。

**延迟加载**（Lazy Loading）：用到关联属性时才触发查询，不用就不查。

JPA/Hibernate 天然支持延迟加载（实体类被 CGLIB 代理）。
MyBatis 呢？**XML 中可以配置，但需要额外的代理支持**。

---

## 它是怎么解决问题的？

### 延迟加载的三种实现方式

```
方式一：CGLIB 代理（MyBatis 默认方式）
  → 对关联属性（如 user.orders）生成一个 CGLIB 代理对象
  → 当调用代理对象的任何方法时（如 get(0)）
  → 触发回调：执行嵌套查询 SQL，加载真实数据
  → 第一次调用后，数据被缓存，后续调用直接返回

方式二：JDK 动态代理（接口类型用这个）
  → 关联属性是接口类型时，用 JDK 动态代理

方式三：Serrializable 代理（对象实现了 Serializable）
  → 使用 SerializedProxy（MyBatis 内置）
```

### 源码关键路径

```java
// Configuration.java — 延迟加载相关配置
public class Configuration {
    protected boolean lazyLoadingEnabled = false;   // ← 总开关
    protected boolean aggressiveLazyLoading = false; // ← 激进模式（3.4.0 默认 false）
    // aggressiveLazyLoading=true：调用对象的任何方法都会触发加载（不仅是关联属性的方法）
    // aggressiveLazyLoading=false：只有调用关联属性的方法才触发加载
}

// ProxyFactory — 创建延迟加载代理的工厂
// 默认实现：CglibProxyFactory（需要引入 cglib 依赖）
// 可选实现：JavassistProxyFactory

// CglibProxyFactory.java（简化）
public Object createProxy(Object target, ResultLoaderMap lazyLoader, ...) {
    // 为 target 对象创建 CGLIB 子类代理
    Enhancer enhancer = new Enhancer();
    enhancer.setSuperclass(target.getClass());
    enhancer.setCallback(new ResultLoaderInterceptor(lazyLoader, ...));
    return enhancer.create();  // ← CGLIB 代理对象
}

// ResultLoaderInterceptor.java — CGLIB 方法拦截器
public Object intercept(Object obj, Method method, Object[] args, ...) {
    // 判断：这个方法调用是否触发了延迟加载
    String property = getPropertyByMethod(method);  // 通过方法名反查属性名
    if (lazyLoader.hasLoader(property)) {
        // 执行嵌套 SQL，加载数据
        lazyLoader.load(property);  // ← 真正触发查询
    }
    // 执行原方法（现在数据已经加载了）
    return method.invoke(target, args);
}
```

### 配置方式

```xml
<!-- mybatis-config.xml -->
<settings>
    <!-- 开启延迟加载 -->
    <setting name="lazyLoadingEnabled" value="true"/>
    <!-- 是否激进（调用任何方法都触发加载）-->
    <setting name="aggressiveLazyLoading" value="false"/>
    <!-- 延迟加载触发的方法名白名单（默认：equals/hashCode/toString 不触发） -->
    <setting name="lazyLoadTriggerMethods" value="equals,clone,hashCode,toString"/>
</settings>
```

### XML 中配置延迟加载

```xml
<resultMap id="userWithOrders" type="User">
    <id column="id" property="id"/>
    <result column="name" property="name"/>
    <!-- select：触发延迟加载的 SQL ID -->
    <!-- column：传给嵌套查询的参数（user.id → selectOrdersByUserId 的参数） -->
    <!-- fetchType：覆盖全局设置 -->
    <collection property="orders"
               select="selectOrdersByUserId"
               column="id"
               fetchType="lazy"/>   <!-- lazy / eager -->
</resultMap>

<select id="selectUserById" resultMap="userWithOrders">
    SELECT id, name FROM user WHERE id = #{id}
</select>

<select id="selectOrdersByUserId" resultType="Order">
    SELECT * FROM orders WHERE user_id = #{userId}
</select>
```

## 它和相似方案的本质区别是什么？

| 对比维度 | MyBatis 延迟加载 | JPA/Hibernate 延迟加载 | 手动分两次查询 |
|---------|------------------|----------------------|--------------|
| 实现方式 | CGLIB/JDK 代理 | CGLIB 代理（字节码增强） | 无代理（手写代码） |
| 粒度控制 | 每个关联属性单独配置 | 注解级（@OneToMany(fetch=...)） | 完全手动 |
| N+1 问题 | 有（需手动 JOIN 避免） | 有（需 EntityGraph 或 JOIN FETCH） | 无 |
| 复杂度 | 中（需配 fetchType） | 中（需理解代理机制） | 低（但代码多） |

**本质区别**：MyBatis 的延迟加载是 **可选的优化手段**（默认关闭），JPA 是 **默认的加载策略**（需要手动改成 EAGER）。

## 正确使用方式

### 全局开启 + 局部覆盖

```xml
<!-- 全局开启延迟加载 -->
<setting name="lazyLoadingEnabled" value="true"/>

<!-- 某个关联属性强制立即加载 -->
<collection property="orders"
           select="selectOrdersByUserId"
           column="id"
           fetchType="eager"/>  <!-- 覆盖全局的 lazy -->
```

### Spring 整合后的注意事项

```java
@Transactional  // ← 必须在事务内访问延迟加载属性！
public User getUserWithLazyOrders(int id) {
    User user = userMapper.selectUserById(id);
    // 此时 orders 是 CGLIB 代理，数据未加载
    System.out.println(user.getOrders().size());  // ← 触发加载（需要事务未提交）
    return user;
}
// ❌ 如果在事务外访问延迟属性 → LazyInitializationException
```

## 边界情况和坑

### 坑1：事务外访问延迟属性

```
异常：LazyInitializationException（类似 Hibernate 的异常）
原因：延迟加载触发时需要 SqlSession，事务提交后 SqlSession 已关闭
解决：在 @Transactional 方法内访问延迟属性，或用 EAGER 加载
```

### 坑2：toString() 触发所有延迟加载

```xml
<!-- 默认情况下，调用 toString() 会触发所有延迟属性的加载 -->
<!-- 因为 lazyLoadTriggerMethods 默认包含 toString -->

<!-- 解决：排除 toString -->
<setting name="lazyLoadTriggerMethods" value="equals,clone,hashCode"/>
```

### 坑3：序列化时的坑

```
如果延迟加载的属性还没被触发，对象就被序列化（如返回 JSON）：
  → 序列化会调用 getter 方法
  → 触发延迟加载
  → 如果此时 SqlSession 已关闭 → 异常

解决：
  ① 在事务内先把对象「用一遍」（触发所有延迟加载）再返回
  ② 用 DTO 做返回对象（不暴露实体类的延迟属性）
  ③ 配置 Jackson 忽略延迟属性：@JsonIgnoreProperties({"handler", "hibernateLazyInitializer"})
```

### 坑4：CGLIB 依赖缺失

```
MyBatis 3.4+ 默认使用 Javassist 做代理（不再依赖 CGLIB）
但如果用老版本，需要手动加 CGLIB 依赖

解决：确认 mybatis.version，3.4+ 无需额外依赖
```

### 坑5：final 类无法延迟加载

```
CGLIB 通过继承实现代理 → final 类无法被继承 → 无法创建代理
MyBatis 会抛异常：Cannot subclass final class xxx

解决：去掉类的 final 修饰符，或改用 EAGER 加载
```

## 延迟加载触发流程图

```
User user = userMapper.selectUserById(1);
  ↓
user.orders 是 CGLIB 代理对象（数据未加载）
  ↓
user.getOrders()   ← 第一次调用
  ↓
ResultLoaderInterceptor.intercept()
  ↓
lazyLoader.load("orders")
  ↓
执行嵌套 SQL：select * from orders where user_id = 1
  ↓
结果设置到 user.orders（真实数据）
  ↓
后续 user.getOrders() 直接返回（不再触发 SQL）
```

## 我的理解

MyBatis 的延迟加载是 **CGLIB 代理 + 触发方法回调** 的经典应用。

- 配置 `fetchType="lazy"` 后，关联属性被替换成 CGLIB 代理对象
- 第一次调用该属性的任何方法（如 `getOrders().size()`）时，触发嵌套 SQL 执行
- 数据加载后被缓存，后续调用不再触发 SQL

**和 JPA 的区别**：JPA 的延迟加载更「智能」（字节码增强，更细粒度），但调试更困难；MyBatis 的延迟加载更「显式」（XML 中配置），可控性更强。

**实际项目建议**：
- 一对多、多对多关联：**默认用延迟加载** ✅
- 一对一关联：如果确定会用，**用 EAGER 或 JOIN 一次查出** ✅
- 返回 JSON 时：**用 DTO，不要直接序列化实体类** ⚠️
