## 场景 A：TraceId生成与传递

### 现象描述

跨服务调用没有关联TraceId，无法串联调用链，排查问题时只能逐服务查日志。TraceId在异步线程、MQ消费场景中丢失，下游服务收不到TraceId，链路断裂。

### 根因分析

未使用统一TraceId生成规则（如UUID、雪花算法），各服务生成的ID格式不一致；TraceId未在跨进程调用中传递（如HTTP Header、Dubbo Attachment）；MDC（Mapped Diagnostic Context）未在线程池、异步任务中传递，导致上下文丢失。

### 解决方案

```java
// 1. TraceId生成工具类
public class TraceIdGenerator {
    private static final String TRACE_ID = "traceId";
    
    // 生成全局唯一TraceId（雪花算法简化版）
    public static String generate() {
        return System.currentTimeMillis() + "-" + ThreadLocalRandom.current().nextInt(10000);
    }
    
    // 设置TraceId到MDC
    public static void setTraceId(String traceId) {
        MDC.put(TRACE_ID, traceId);
    }
    
    // 获取当前TraceId
    public static String getTraceId() {
        return MDC.get(TRACE_ID);
    }
}

// 2. HTTP调用传递TraceId（RestTemplate拦截器）
import org.springframework.http.HttpRequest;
import org.springframework.http.client.ClientHttpRequestInterceptor;

public class TraceIdInterceptor implements ClientHttpRequestInterceptor {
    @Override
    public ClientHttpResponse intercept(
            HttpRequest request, byte[] body, ClientHttpRequestExecution execution) throws IOException {
        String traceId = TraceIdGenerator.getTraceId();
        if (traceId != null) {
            request.getHeaders().add("X-Trace-Id", traceId); // 传递到下游
        }
        return execution.execute(request, body);
    }
}

// 3. 线程池传递TraceId（自定义线程池）
import java.util.concurrent.ThreadPoolExecutor;

public class TraceThreadPoolExecutor extends ThreadPoolExecutor {
    @Override
    public void execute(Runnable command) {
        String traceId = TraceIdGenerator.getTraceId();
        super.execute(() -> {
            TraceIdGenerator.setTraceId(traceId); // 传递TraceId到新线程
            try {
                command.run();
            } finally {
                MDC.clear(); // 清理MDC
            }
        });
    }
}
```

## 场景 B：SkyWalking/Agent方案

### 现象描述

手动埋点侵入业务代码，每个接口都要加埋点注解，维护成本高；SkyWalking Agent未适配自定义组件（如内部RPC框架），链路中缺失这部分调用。接入后发现监控数据不全，无法定位性能瓶颈。

### 根因分析

使用AOP手动埋点导致业务代码耦合，新增接口容易遗漏；SkyWalking Agent默认只支持标准组件（Spring MVC、Dubbo、Redis），自定义组件未实现插件，无法被埋点。未配置Agent的排除路径，静态资源、健康检查也被埋点，浪费资源。

### 解决方案

```java
// 1. SkyWalking Agent配置（agent.config）
// agent.service_name=order-service
// collector.backend_service=127.0.0.1:11800
// exclude_paths=/health,/static/** 排除无需埋点的路径

// 2. 自定义SkyWalking插件（监控内部RPC框架）
import net.bytebuddy.implementation.bind.annotation.*;
import org.apache.skywalking.apm.agent.core.plugin.interceptor.enhance.EnhancedInstance;

public class InternalRpcInterceptor {
    // 拦截内部RPC调用入口
    @RuntimeType
    public static Object intercept(@Origin String method, @AllArguments Object[] args) {
        // 创建Span，记录RPC调用
        AbstractSpan span = ContextManager.createLocalSpan("internal-rpc/" + method);
        try {
            return method.invoke(null, args);
        } finally {
            ContextManager.stopSpan();
        }
    }
}

// 3. 插件定义文件（skywalking-plugin.def）
// internal-rpc=com.example.skywalking.InternalRpcInstrumentation
```

## 场景 C：采样率设计

### 现象描述

生产环境全采样导致监控数据量过大，ES存储每天增长500GB，成本飙升；低采样（如1%）导致重要链路（如支付、下单）丢失，排查问题无数据。高峰期监控组件占用CPU超过20%，影响业务。

### 根因分析

未根据场景设置差异化采样率：核心链路（支付、下单）需要全采样，非核心链路（查询、心跳）低采样；固定采样率无法应对流量波动，高峰期限流不足；未考虑业务重要性，重要场景采样率不足。

### 解决方案

```java
// 1. 动态采样率策略（基于业务场景）
public class SamplingStrategy {
    // 核心链路：全采样
    private static final Set<String> CORE_PATHS = Set.of("/pay", "/order/create");
    // 非核心链路：10%采样
    private static final double NON_CORE_RATE = 0.1;
    
    public static boolean shouldSample(String path) {
        if (CORE_PATHS.contains(path)) {
            return true; // 核心链路全采样
        }
        // 非核心链路按概率采样
        return ThreadLocalRandom.current().nextDouble() < NON_CORE_RATE;
    }
}

// 2. SkyWalking采样率配置（agent.config）
// agent.sample_n_per_3_secs=1000 每3秒采样1000个请求（约10%）

// 3. Java代码：接入采样策略到SkyWalking
import org.apache.skywalking.apm.agent.core.sampling.SamplingService;

public class CustomSamplingService extends SamplingService {
    @Override
    public boolean trySampling(String operationName) {
        String path = parsePathFromOperation(operationName);
        return SamplingStrategy.shouldSample(path);
    }
}
```

## 场景 D：低开销保障

### 现象描述

接入监控后接口RT增加10%+，CPU使用率上升；监控组件未限制资源，导致OOM；埋点日志输出过多，磁盘IO飙升。

### 根因分析

埋点逻辑过重，采集了过多无用指标（如每个方法的参数详情）；未做监控组件自身的资源限制（如线程数、队列大小）；未关闭不必要的埋点（如健康检查、静态资源）。

### 解决方案

```java
// 1. 监控开销控制（SkyWalking配置）
// agent.keep_tracing 关闭不必要的跟踪
// thread_pool.queue_size=1000 限制队列大小，避免OOM
// profile.duration=10 性能分析最大时长10分钟

// 2. Java代码：精简埋点指标
import org.apache.skywalking.apm.toolkit.trace.ActiveSpan;

public class LeanSpanExample {
    public void processOrder(Order order) {
        ActiveSpan.tag("order_id", order.getId().toString()); // 只埋关键标签
        // 避免埋大对象（如order详情），浪费存储
        // 错误：ActiveSpan.tag("order_detail", order.toString());
    }
}

// 3. 监控组件健康检查
public class MonitorHealthChecker {
    public static boolean isHealthy() {
        // 检查SkyWalking Agent是否正常
        try {
            ContextManager.getRuntimeContext();
            return true;
        } catch (Exception e) {
            return false;
        }
    }
}
```

## 排查 Checklist

```text
□ TraceId是否统一？ → 全局统一生成规则，格式一致
□ 跨进程是否传递？ → HTTP Header、MQ属性、RPC Attachment都传递
□ 异步场景是否丢失？ → 线程池、@Async都传递MDC上下文
□ Agent是否覆盖核心组件？ → Spring MVC、RPC、Redis、MQ都埋点
□ 采样率是否合理？ → 核心链路100%，非核心10%，动态可调
□ 开销是否可控？ → RT增加<5%，CPU占用<5%
□ 存储是否足够？ → ES容量规划，采样后存储<100GB/天
□ 链路是否完整？ → 网关→服务→DB全链路可追踪
□ 监控组件是否健康？ → 定期检查Agent状态，避免无数据
□ 排除路径是否配置？ → 健康检查、静态资源不埋点
```

## 追问链

### 追问 1：全链路监控选SkyWalking还是Zipkin？

> "看需求：SkyWalking功能更全（性能分析、告警、拓扑图），适合生产环境大规模使用；Zipkin更轻量，适合小团队或测试环境。如果已经用Spring Cloud Sleuth，选Zipkin；如果需要国产化和更多功能，选SkyWalking。"

### 追问 2：异步场景下如何传递TraceId？

> "三种方案：1. MDC + 自定义线程池：线程池执行前传递MDC上下文；2. TransmittableThreadLocal（TTL）：阿里开源，支持线程池上下文传递；3. SkyWalking原生支持：Agent自动增强线程池，无需改代码。推荐用TTL或SkyWalking Agent。"

### 追问 3：采样率如何动态调整？

> "通过配置中心动态调整：1. 配置中心存储各链路的采样率；2. 监控组件监听配置变更；3. 无重启生效。示例：支付链路高峰期限流，采样率从100%降到50%。SkyWalking支持通过Backend API动态调整采样率。"

### 追问 4：监控数据如何存储和查询？

> "存储：SkyWalking默认用ES（支持大规模存储），也可用MySQL（小规模）。查询：SkyWalking UI查看拓扑图、链路详情、性能指标；ES直接查询：用Kibana查询trace_id索引；告警：SkyWalking配置规则，链路超时、错误率超标触发告警。"
