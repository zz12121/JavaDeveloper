# 基础语法

> Java 基础语法不只是"入门第一课"——很多看似高级的问题（并发 Bug、GC 停顿、序列化异常）根子都在基本类型、值传递、自动装箱这些概念里。

---

## 这个问题为什么存在？

任何语言都需要回答几个根本问题：

- **数据怎么表示？** → 基本类型 vs 引用类型
- **流程怎么控制？** → 条件、循环、分支
- **一段代码影响的范围是什么？** → 变量作用域和生命周期

Java 的基础语法有几个**与其他语言显著不同的设计决策**，这些决策直接影响后续所有框架和工具的行为：

1. **强类型 + 静态编译**：类型错误在编译期暴露，不像 Python/JS 到运行时才炸
2. **只有值传递**：没有 C++ 的引用传递，也没有 C# 的 ref/out 参数——这个限制深刻影响了 Java 的 API 设计
3. **一切皆对象（除了基本类型）**：这个"除了"导致了自动装箱/拆箱，引出了一堆坑

> 异常处理是基础语法的独立维度，详见 [[01_Java核心/03_异常处理/异常处理|异常处理]]。

---

## 它是怎么解决问题的？

### 基本数据类型（8 种）

| 分类 | 类型 | 大小 | 范围 | 说明 |
|------|------|------|------|------|
| 整数 | `byte` | 1 字节 | -128 ~ 127 | 网络协议、文件格式 |
| 整数 | `short` | 2 字节 | -32768 ~ 32767 | 几乎不用 |
| 整数 | `int` | 4 字节 | ±21 亿 | **默认整数类型** |
| 整数 | `long` | 8 字节 | ±9.2 × 10¹⁸ | 时间戳、ID |
| 浮点 | `float` | 4 字节 | ~7 位有效数字 | 几乎不用 |
| 浮点 | `double` | 8 字节 | ~15 位有效数字 | **默认浮点类型** |
| 字符 | `char` | 2 字节 | 0 ~ 65535 | UTF-16 编码单元 |
| 布尔 | `boolean` | ~ | true/false | JVM 规范不规定大小 |

```
为什么没有 unsigned？
Java 的设计哲学：简单优先。unsigned 带来溢出行为、类型转换等额外规则，
对大多数业务场景没有实际收益。需要无符号时用 long 或 BigInteger。

为什么 char 是 2 字节？
Java 诞生时 Unicode 还在 3.0 之前，16 位被认为足以表示所有字符。
后来 Unicode 扩展到 21 位（emoji、生僻字），导致一个 char 可能无法表示
一个完整的 Unicode 码点——这是 [[01_Java核心/01_基础语法/字符串|字符串处理]] 很多坑的根源。
```

### 整数溢出：一个常被忽略的致命问题

```java
// 静默溢出，不报错！
int max = Integer.MAX_VALUE;  // 2147483647
int result = max + 1;         // -2147483648（不是 2147483648）
```

Java 整数运算溢出时**不会抛异常，也不会报错**，直接"绕回"。这在以下场景特别危险：

```
典型踩坑场景：
1. 计算数组/集合大小：(end - start + 1) 当 start = 0 时没问题，
   但当 end = Integer.MAX_VALUE 时溢出
2. hashCode 计算：31 * h + val 连乘容易溢出
3. 中间运算：小时 × 60 × 60 × 1000 转毫秒，大数值直接溢出

防御手段：
- 用 Math.addExact() / Math.multiplyExact()，溢出时抛 ArithmeticException
- 用 long 承接中间结果
- 用 BigInteger 处理超大数
```

### 浮点数精度问题

```java
System.out.println(0.1 + 0.2);  // 0.30000000000000004
System.out.println(1.0 - 0.9);  // 0.09999999999999998
```

**这是 IEEE 754 浮点标准的问题，不是 Java 的 Bug。** 计算机用二进制表示小数，而 0.1 在二进制中是无限循环的——就像 1/3 在十进制中是 0.333... 一样。

### 值传递：Java 唯一的参数传递方式

这是面试高频题，也是理解 Java 行为的基础。

```
核心结论：Java 只有值传递。
- 基本类型：传的是值的拷贝
- 引用类型：传的是引用地址的拷贝（不是对象本身的拷贝）

这意味着什么？
方法内部能修改引用指向的对象内容，但不能让外部引用指向新对象。

public void swap(Object a, Object b) {
    Object tmp = a;
    a = b;    // 只改了局部变量，外部不受影响
    b = tmp;
}
// 调用后，外部变量不会交换——这就是值传递的证据
```

### 自动装箱与拆箱（Autoboxing）

Java 5 引入的语法糖，让基本类型和包装类型之间可以隐式转换：

```java
Integer a = 127;    // 自动装箱：Integer.valueOf(127)
int b = a;          // 自动拆箱：a.intValue()
```

**缓存陷阱**——Integer.valueOf() 对 -128 ~ 127 的值做了缓存：

```java
Integer x = 127;
Integer y = 127;
System.out.println(x == y);   // true（同一缓存对象）

Integer m = 128;
Integer n = 128;
System.out.println(m == n);   // false（不同对象！）

// 正确的比较方式
System.out.println(m.equals(n));  // true
```

```
为什么是 -128 ~ 127？
这是 IntegerCache 的默认范围，可通过 -XX:AutoBoxCacheMax 调整。
设计理由：大多数程序中，小整数的出现频率远高于大整数。
缓存这些值可以减少对象创建，降低 GC 压力。
```

### 流程控制

#### for 循环的四种形态

```java
// 1. 传统 for
for (int i = 0; i < list.size(); i++) { }

// 2. 增强 for（for-each）
for (String s : list) { }

// 3. 迭代器（需要删除元素时必须用这个）
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if (condition) it.remove();  // 唯一安全的遍历时删除方式
}

// 4. Java 16+ 的 forEach + lambda
list.forEach(s -> { });
```

```
遍历时删除元素的经典错误：
for (String s : list) {
    if (condition) list.remove(s);  // ConcurrentModificationException!
}
// for-each 底层用迭代器，直接修改集合会让迭代器的 modCount 不一致。
// 正确做法：用 Iterator.remove() 或 removeIf()。
```

#### switch 表达式（Java 14+）

```java
// 旧写法（语句，需要 break）
switch (day) {
    case MONDAY:
    case FRIDAY:
        System.out.println("工作日");
        break;
    default:
        System.out.println("其他");
}

// 新写法（表达式，用 -> 和 yield）
String result = switch (day) {
    case MONDAY, FRIDAY -> "工作日";
    case SATURDAY, SUNDAY -> "周末";
    default -> "其他";
};
```

旧版 switch 忘记 break 就会 fall-through 执行后续 case。Java 14+ 的 switch 表达式用 `->` 语法彻底消除了这个问题。

### 数组

```
数组的几个关键特性：
1. 长度固定——创建后不能改变（需要变长用 [[01_Java核心/05_集合框架/集合框架|ArrayList]]）
2. 协变（covariant）——String[] 是 Object[] 的子类型
   Object[] arr = new String[3];
   arr[0] = 123;  // 编译通过，运行时 ArrayStoreException
3. 数组也是对象——有 .length 属性（不是方法），继承自 Object

数组 vs ArrayList 的选择：
- 长度固定、性能敏感、基本类型数组 → 用数组（int[] vs ArrayList<Integer>）
- 需要动态增删、需要 [[01_Java核心/04_泛型/泛型|泛型]] → 用 ArrayList
```

### 变量作用域

```java
public void demo() {
    int x = 1;          // 局部变量，方法结束销毁

    {
        int y = 2;      // 块级作用域
    }
    // y 在这里不可见

    for (int i = 0; i < 10; i++) {
        // i 只在 for 循环内有效
    }
    // i 在这里不可见

    try {
        int z = 3;
    } catch (Exception e) {
        // e 只在 catch 块内有效
    }
    // z 在这里不可见——try 块内声明的变量，外面看不到
}
```

## 深入原理

### 编译期常量与类加载

```java
class Constants {
    // 编译期常量：不会触发 Constants 类的初始化
    public static final int A = 100;
    System.out.println(Constants.A);  // 不打印 Constants 的 static 块

    // 运行时常量：会触发 Constants 类的初始化
    public static final long NOW = System.currentTimeMillis();
    System.out.println(Constants.NOW); // 会触发 Constants 的 <clinit>
}
```

编译期常量（基本类型 + String 字面量 + 不可变表达式）在编译时被内联到调用处，不会触发类加载。`[[01_Java核心/01_基础语法/final关键字|final]]` 的编译期优化就是基于这个机制。

### 字符串拼接的编译器优化

```java
// 编译前
String s = "hello" + "world" + "java";

// 编译后（javac 优化）
String s = "helloworldjava";  // 编译器合并常量
```

```
注意：Java 9 之后，String 拼接 "a" + b + "c" 编译器会自动优化为
invokedynamic 调用 StringConcatFactory，性能接近 StringBuilder。
但在循环内多次拼接，手动 StringBuilder 仍然是更安全的选择。
```

### 数组协变 vs 泛型

数组是协变的（`String[]` 是 `Object[]` 的子类型），而泛型是不变的（`List<String>` 不是 `List<Object>` 的子类型）。这是 Java 早期的妥协设计——泛型出现前需要数组协变来处理多态，但这导致了运行时的 ArrayStoreException。泛型用类型擦除解决了这个问题（虽然擦除本身也有妥协，详见 [[01_Java核心/04_泛型/泛型|泛型]]）。

## 正确使用方式

### 1. 基本类型优先

```java
int count = 0;              // ✓ 堆上零开销
Integer count = 0;          // ✗ 自动装箱，多一个对象
```

当你在 [[01_Java核心/05_集合框架/集合框架|集合]] 中需要对象时才用包装类型。局部变量和字段声明优先用基本类型。

### 2. 常量用 static final + 大写命名

```java
// 类常量
public static final int MAX_SIZE = 1024;
public static final String DEFAULT_NAME = "unknown";

// 如果值在编译期就能确定，static final 会让编译器做内联优化。
// 如果值在运行时才能确定（如 UUID），static final 仍然有意义——语义上表示不可变。
// 详见 [[01_Java核心/01_基础语法/final关键字|final关键字]] 关于编译期常量的完整说明。
```

### 3. 字符串拼接用 StringBuilder

```java
// 循环内字符串拼接
String result = "";
for (String s : list) {
    result += s;  // ✗ 每次循环创建新 StringBuilder + 新 String
}

// 正确做法
StringBuilder sb = new StringBuilder();
for (String s : list) {
    sb.append(s);  // ✓ 同一个 StringBuilder 上操作
}
String result = sb.toString();
```

### 4. 浮点数正确使用

```
1. 金额计算 → 用 BigDecimal，且构造时必须传字符串：
   new BigDecimal("0.1")   ✓
   new BigDecimal(0.1)     ✗（0.1 本身已经有精度损失）
2. 比较浮点数 → 不要用 ==，用误差范围：
   Math.abs(a - b) < 1e-9
3. 真需要精确分数 → 用 Apache Commons Math 的 Fraction
```

## 边界情况和坑

### 1. Integer 缓存导致的 == 误判

```java
Integer x = 127, y = 127;
x == y;   // true（缓存对象）

Integer m = 128, n = 128;
m == n;   // false！不同对象
```

**包装类型比较永远用 equals()**。`==` 比较的是引用地址，在缓存范围之外一定返回 false。

### 2. 浮点数的 NaN 和 Infinity

```java
double nan = 0.0 / 0.0;    // NaN
double inf = 1.0 / 0.0;    // Infinity

nan == nan;                  // false！NaN 不等于任何值，包括自己
Double.isNaN(nan);           // 正确判断方式

// NaN 传播：任何涉及 NaN 的运算结果都是 NaN
nan + 1;    // NaN
nan * 0;    // NaN
```

NaN 是浮点数的一个特殊值，表示"不是一个数"（Not a Number）。IEEE 754 规定 NaN 不等于自身，所以不能用 `==` 判断，必须用 `Double.isNaN()`。

### 3. 三元表达式的自动装箱陷阱

```java
Object x = true ? 1 : 2.0;  // 结果是 Double 1.0，不是 Integer 1！
```

三元表达式 `? :` 的两个分支类型必须统一。`int` 和 `double` 统一到 `double`，触发自动装箱为 `Double`。

### 4. 数组协变导致的运行时异常

```java
Object[] arr = new String[3];  // 编译通过（String[] 是 Object[] 的子类型）
arr[0] = 42;                   // 编译通过，运行时 ArrayStoreException
```

编译器无法在编译期阻止你往 String 数组里放 Integer，但运行时会检查。**这是 Java 早期泛型缺失时的妥协设计**——泛型出来后，[[01_Java核心/04_泛型/泛型|集合]] 用类型擦除解决了这个问题（虽然擦除本身也有妥协）。

### 5. 遍历时删除导致 ConcurrentModificationException

```java
// 错误：for-each 底层用迭代器，直接修改集合会导致 modCount 不一致
for (String s : list) {
    if (condition) list.remove(s);  // ConcurrentModificationException!
}

// 正确：用 Iterator.remove() 或 removeIf()
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if (condition) it.remove();  // 安全
}

// 或 Java 8+ 的 removeIf
list.removeIf(s -> condition);
```

### 6. try 块内声明的变量作用域陷阱

```java
// 编译错误：z 在 try 外不可见
try {
    int z = 3;
} catch (Exception e) {
    // 想在这里用 z → 编译错误
}
// 想在 finally 用 z → 也编译错误

// 正确：在 try 之前声明
int z;
try {
    z = 3;
} catch (Exception e) {
    // z 不可用（可能未赋值）
} finally {
    // z 也不可用
}
```
