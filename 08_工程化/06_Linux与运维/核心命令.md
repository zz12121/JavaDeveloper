# 核心命令

> Linux 的哲学是「组合小工具完成复杂任务」。管道（`|`）和重定向是这个哲学的核心——不理解它们，就等于没学 Linux。

---

## 这个问题为什么存在？

Windows 的用户习惯用「一个软件解决一个问题」：下载用迅雷，解压用 WinRAR，搜索用 Everything。

Linux 的设计哲学完全不同：**每个命令只做一件事，但做好这件事**。复杂任务通过「管道」把多个命令串联起来完成。

```
Windows 思维：打开一个 GUI 软件，点几下鼠标完成任务
Linux 思维：  ls -lh | grep "\.log$" | awk '{print $9}' | xargs rm
              ↑        ↑                ↑                  ↑
            列出文件  过滤日志文件       提取文件名          批量删除
```

**为什么这种哲学更高效？**

1. **组合性**：`grep` 不关心输入来自哪里（文件、管道、网络），它只负责「过滤文本」
2. **脚本化**：所有操作都可以写成脚本，自动化重复任务
3. **远程友好**：SSH 到服务器，没有 GUI，只能用命令

---

## 它是怎么解决问题的？

### 管道（`|`）：命令间通信的桥梁

```
命令A的输出 → 管道 → 命令B的输入
         echo "hello" | wc -c
                       ↑
    echo 输出 "hello\n" → wc -c 统计字节数 → 输出 6
```

**本质**：管道是一个「内存缓冲区」，把前一个命令的 `stdout` 重定向到后一个命令的 `stdin`。

```
进程A（echo）         管道（内核缓冲区）        进程B（wc）
stdout → 写入管道    →  ← 从管道读取 →    stdin
```

### 重定向：控制输入输出的去向

```bash
# 输出重定向
echo "hello" > file.txt      # 覆盖写入
echo "world" >> file.txt     # 追加写入

# 错误重定向
ls not_exist 2> error.log    # 只重定向 stderr
ls not_exist 2>&1           # 把 stderr 合并到 stdout

# 输入重定向
sort < input.txt             # 从文件读取输入
```

### 核心命令详解

#### `find`：文件查找（按属性，不是按内容）

```bash
# 按名称查找
find /var/log -name "*.log"

# 按时间查找（7天前修改的文件）
find /var/log -name "*.log" -mtime +7

# 按大小查找（大于 100MB 的文件）
find /var/log -size +100M

# 组合条件（查找 + 执行动作）
find /var/log -name "*.log" -mtime +7 -delete   # 直接删除
find /var/log -name "*.log" -exec cp {} /backup/ \;  # 复制到备份目录
```

**为什么不用 `ls *.log`？**
`ls` 只能查找当前目录，`find` 可以递归查找子目录，并且支持复杂的过滤条件。

#### `grep`：文本过滤（按内容）

```bash
# 基本用法
ps aux | grep java

# 反向过滤（排除 grep 自身）
ps aux | grep java | grep -v grep

# 显示行号
grep -n "ERROR" app.log

# 递归查找目录下的所有文件
grep -r "OutOfMemoryError" /var/log/
```

#### `awk`：文本处理语言（不止是命令）

```bash
# 基本用法：按列提取
ls -lh | awk '{print $9, $5}'    # 输出：文件名 文件大小

# 条件过滤
ls -lh | awk '$5 > 1000000 {print $9}'   # 输出大小超过 1MB 的文件名

# 内置变量
awk '{sum += $5} END {print "Total:", sum}'   # 求和
```

**`awk` 的本质**：它是一个完整的文本处理语言，支持变量、数组、函数、控制流。但 90% 的场景只需要 `{print $N}`。

#### `sed`：流编辑器（批量替换）

```bash
# 替换文件中的字符串（不改变原文件，输出到 stdout）
sed 's/old/new/g' file.txt

# 直接修改文件（-i 参数）
sed -i 's/old/new/g' file.txt

# 删除空行
sed -i '/^$/d' file.txt

# 删除包含某模式的行
sed -i '/DEBUG/d' app.log    # 删除所有包含 DEBUG 的行
```

#### `xargs`：把 stdin 转成命令行参数

```bash
# 问题：find 输出的文件名包含空格时会出问题
find . -name "*.txt" | rm   # 错误！rm 会把每个空格分隔的单词当作一个参数

# 解决：用 xargs
find . -name "*.txt" | xargs rm

# 更安全：处理包含空格的文件名
find . -name "*.txt" -print0 | xargs -0 rm
```

**为什么需要 `xargs`？**
很多命令（如 `rm`、`cp`）不接受 stdin 输入，只接受命令行参数。`xargs` 的作用就是把 stdin 转换成命令行参数。

## 它和相似方案的本质区别是什么？

### Linux 命令 vs Windows PowerShell

| 维度 | Linux Shell（bash） | PowerShell |
|------|---------------------|------------|
| 数据传递单位 | 文本（字符串） | 对象（.NET Object） |
| 管道语义 | 文本流 | 结构化对象流 |
| 学习曲线 | 陡（需要理解文本处理工具链） | 陡（需要理解 .NET 对象模型） |
| 远程管理 | SSH + 命令 | WinRM + PowerShell Remoting |
| 脚本语法 | 独特（bash 语法） | 类似 C# |

**为什么 Linux 管道传递的是文本？**

历史原因：1970年代的 Unix 设计，所有工具都用文本作为输入/output 格式。好处是「通用性」——任何命令的输出都可以作为任何命令的输入。坏处是「解析成本」——每个命令都要解析文本。

PowerShell 改进了这一点：管道传递的是「对象」，不需要解析。但代价是「依赖性」——发送方和接收方必须就对象的 schema 达成一致。

### 直接执行 vs 通过管道执行

```bash
# 直接执行：一次处理一个文件
for file in *.log; do
  grep "ERROR" "$file"
done

# 管道 + xargs：批量处理
ls *.log | xargs grep "ERROR"
```

**性能差异**：`xargs` 版本会把多个文件合并成一个命令行，减少进程创建次数，速度更快。

## 正确使用方式

### 1. 管道长度控制：不要为了炫技写 10 级管道

```bash
# 不好的写法：难以调试，哪一级出错很难定位
cat access.log | grep "404" | awk '{print $7}' | sort | uniq -c | sort -rn | head -10 | awk '{print $2}'

# 好的写法：拆分成多行，或者用变量暂存中间结果
grep "404" access.log | awk '{print $7}' > /tmp/urls.txt
sort /tmp/urls.txt | uniq -c | sort -rn | head -10
```

### 2. 总是用 `grep -v grep` 排除自身

```bash
# 问题：grep 进程也会出现在 ps 输出中
ps aux | grep java
# 输出包含：myuser  12345  ...  grep --color=auto java

# 解决
ps aux | grep java | grep -v grep
# 或者（更优雅）
ps aux | grep "[j]ava"   # 正则表达式，匹配 "java" 但不匹配 "grep java"
```

### 3. 用 `find -print0` + `xargs -0` 处理怪异文件名

```bash
# 问题：文件名包含空格、换行符
touch "my file.txt"
find . -name "*.txt" | xargs rm   # 错误！会把 "my" 和 "file.txt" 当作两个文件

# 解决
find . -name "*.txt" -print0 | xargs -0 rm
```

**`-print0` 的作用**：用 null 字符（`\0`）分隔文件名，而不是换行符。`xargs -0` 对应地用 null 字符分割输入。

### 4. 用 `tee` 同时输出到屏幕和文件

```bash
# 场景：长时间运行的命令，想同时看输出和保存到日志
tar -czf backup.tar.gz /data | tee backup.log

# 等价但更麻烦的写法
tar -czf backup.tar.gz /data | tee >(cat > backup.log)
```

## 边界情况和坑

### 1. 管道中的 `cd` 不生效

```bash
# 错误理解：认为管道中的每个命令都在同一个 shell 中执行
pwd         # 输出：/home/user
cd /tmp | pwd   # 期望输出 /tmp，实际还是 /home/user
```

**原因**：管道中的每个命令都在子 shell 中执行，`cd` 只影响子 shell，不影响父 shell。

### 2. `xargs` 默认的命令行长度限制

```bash
# 问题：文件太多时，xargs 会分成多次执行
find /var/log -name "*.log" | xargs rm   # 如果文件太多，会执行多次 rm

# 解决：用 -P 参数并行执行（谨慎！）
find /var/log -name "*.log" | xargs -P 4 rm   # 4 个进程并行删除
```

### 3. `grep` 的递归搜索会卡死在二进制文件上

```bash
# 问题：grep -r 会尝试搜索二进制文件，输出乱码
grep -r "keyword" /path/

# 解决：只搜索文本文件
grep -rI "keyword" /path/   # -I 忽略二进制文件
```

### 4. `rm` 通过 `xargs` 执行时没有确认提示

```bash
# 危险！没有确认提示，直接删除
find /var/log -name "*.log" | xargs rm -f

# 防御性写法：先预览，再执行
find /var/log -name "*.log" | xargs echo rm   # 先打印要执行的命令
find /var/log -name "*.log" | xargs rm -f     # 确认无误后再执行
```

## 我的理解

Linux 命令设计的精髓是**组合（Composition）**：

- 每个命令是一个「纯函数」：输入确定，输出确定，没有副作用（除了明确写入文件/网络）
- 管道是「函数调用」：前一个命令的 stdout 是后一个命令的 stdin
- 这种设计的威力在于**涌现性**：简单工具组合起来，能完成复杂任务

```
单个命令  →  功能有限
管道组合  →  功能涌现
脚本封装  →  可复用工具
```

**和 Java Stream API 的关系**：

Linux 管道 `cat file | grep "error" | wc -l` 和 Java Stream `Files.lines(path).filter(s -> s.contains("error")).count()` 在概念上完全一样——都是「流式处理」。

理解 Linux 命令，对理解现代编程中的「流式处理」「函数式编程」有直接帮助。

**面试高频考点**：
1. 如何用一条命令找出某目录下最大的 10 个文件 —— `du -ah | sort -rh | head -10`
2. 如何批量替换多个文件中的字符串 —— `find . -name "*.txt" | xargs sed -i 's/old/new/g'`
3. 管道和重定向的区别 —— 管道连接两个进程，重定向连接进程和文件

*掌握这些命令，不只是为了「会用 Linux」，而是理解「组合式设计」的威力。这种思维方式，在写代码、设计架构时同样适用。*
