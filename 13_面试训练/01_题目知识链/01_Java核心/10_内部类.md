## 盲答引导

1. 成员内部类和局部内部类，为什么能访问外部类的 `private` 字段？

## 知识链提示

这道题应该让你联想到：

- `[[内部类引用机制]]` → 编译器为每个内部类生成 `Outer$Inner.class`，自动持有外部类引用
- `[[effective final]]` → JDK 8 为什么放宽了 final 限制？
- `[[局部类与匿名类]]` → 匿名内部类的局限性（只能继承一个类或实现一个接口）
- `[[Lambda vs 匿名内部类]]` → Lambda 的字节码和匿名内部类完全不同
- `[[内存泄漏]]` → 内部类持有外部类引用，如果内部类是 Handler/Runnable，可能导致 Activity 泄漏

## 核心追问

1. 下面代码输出什么？
   ```java
   class Outer {
       private int x = 10;
       class Inner {
           int x = 20;
           void print() { System.out.println(this.x); }
       }
   }
   new Outer().new Inner().print();
   ```
2. 内部类编译后生成的 `.class` 文件，名字是什么？内部类里如何引用外部类字段？
3. `new Thread(() -> System.out.println("hi")).start()` 和 `new Thread(new Runnable() {...}).start()` 字节码有什么不同？
4. 为什么 Java 不允许在局部内部类/匿名内部类里修改局部变量的值？（effective final 的本质原因）
5. Android 开发中，为什么 Handler 匿名类会导致 Activity 内存泄漏？生命周期差异在哪里？

## 参考要点（盲答后再看）

<details>
<summary>点击展开</summary>

**四类内部类**：

| 类型 | 语法 | 编译后文件 | 能访问外部 |
|------|------|-----------|-----------|
| 静态内部类 | `static class A` | `Outer$A.class` | 仅静态成员 |
| 成员内部类 | `class A` | `Outer$A.class` | 全部成员（包括 private） |
| 局部内部类 | 方法里 `class A` | `Outer$1A.class` | 外部全部 + 局部（需final） |
| 匿名内部类 | `new X() {}` | `Outer$1.class` | 同局部内部类 |

**内存区别**：
```java
// 成员内部类：持有外部类引用
Outer outer = new Outer();
Outer.Inner inner = outer.new Inner();
// inner 对象里有一个隐式字段：Outer this$0
// inner 生命周期可能比 outer 长 → 内存泄漏风险！

// 静态内部类：不持有外部类引用
// 适合作为工具类/组合类
```

**字节码层面（编译器自动生成）**：
```java
// 源码
class Outer {
    private int x = 10;
    class Inner {
        void print() { System.out.println(x); }
    }
}

// 编译后 Inner.class 大致等价于：
class Inner {
    final Outer this$0;  // 编译器生成，持有外部类引用
    Inner(Outer this$0) { this.this$0 = this$0; }
    void print() { System.out.println(this$0.x); }
}
```

**Lambda vs 匿名内部类**：

```java
// 匿名内部类：会生成单独的 .class 文件
Runnable r1 = new Runnable() {
    @Override public void run() {}
};  // 编译后：Outer$1.class

// Lambda：使用 invokedynamic，不生成额外 .class
Runnable r2 = () -> {};  // 编译后：只有 LambdaMetafactory 生成调用点
// Lambda 不会持有外部类引用（只捕获需要的变量）
```

**effective final 本质原因**：
```java
void method() {
    int x = 10;  // 局部变量在栈上
    Runnable r = () -> System.out.println(x);
    // 编译器把这个 x「拷贝」进 Runnable 里
    // 如果后面 x 被修改了，Runnable 里的值就对不上了
    // 所以 JDK 8 要求「值不变」，JDK 8 之后放宽为 effective final
}
```
