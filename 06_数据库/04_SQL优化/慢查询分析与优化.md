# 慢查询分析与优化

## 这个问题为什么存在？

> 一个慢查询，可能拖垮整个数据库。

真实场景：

```sql
-- 一个慢查询撑爆连接池
SELECT * FROM orders WHERE user_id = 12345;  
-- 没建索引，全表扫描（100万行）
-- 执行时间：8秒
-- 并发10个这样的查询 → 连接池被打满 → 整个系统不可用
```

**问题的本质**：
1. 慢查询不会"报错"，只表现为"慢"
2. 慢查询往往在数据量增大后"突然"出现
3. 慢查询的影响是指数级的（连接池、锁、IO都受影响）

---

## 它是怎么解决问题的？

### 慢查询日志（Slow Query Log）

**开启慢查询日志**：

```sql
-- 查看当前配置
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query_time';

-- 开启慢查询日志
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;  -- 超过1秒的记录到慢查询日志
SET GLOBAL log_queries_not_using_indexes = 'ON';  -- 记录未使用索引的查询

-- 永久配置（my.cnf）
-- [mysqld]
-- slow_query_log = 1
-- slow_query_log_file = /var/lib/mysql/slow.log
-- long_query_time = 1
-- log_queries_not_using_indexes = 1
```

**慢查询日志格式**：

```
# Time: 2024-01-01T10:00:00.123456Z
# User@Host: root[root] @ localhost [127.0.0.1]
# Query_time: 5.123456  Lock_time: 0.000789  Rows_sent: 100  Rows_examined: 1000000
SELECT * FROM orders WHERE user_id = 12345;
```

关键字段解读：

| 字段 | 含义 | 优化方向 |
|------|------|----------|
| Query_time | 查询执行时间（秒） | 主要优化目标 |
| Lock_time | 等待锁的时间（秒） | 锁竞争问题 |
| Rows_sent | 返回给客户端的行数 | 是否返回了过多数据 |
| Rows_examined | 扫描的行数 | 扫描行数 >> 返回行数 → 索引问题 |

**Rows_examined / Rows_sent 比值**：

```
比值 >> 1：索引没用好，扫描了很多行才找到目标行
比值 ≈ 1：索引使用良好
```

### 分析工具：mysqldumpslow

通过 shell 调用，不在 SQL 中运行：

```bash
# 按查询时间排序，显示前10条
mysqldumpslow -s t -t 10 /var/lib/mysql/slow.log

# 按扫描行数排序
mysqldumpslow -s r -t 10 /var/lib/mysql/slow.log

# 按查询次数排序
mysqldumpslow -s c -t 10 /var/lib/mysql/slow.log

# 抽象化（把具体值替换成N，方便分组）
mysqldumpslow -a -s t -t 10 /var/lib/mysql/slow.log
```

输出示例：

```
Count: 150  Time=5.12s (768s)  Lock=0.00s (0s)  Rows=100.0 (15000), root[root]@localhost
  SELECT * FROM orders WHERE user_id = N
```

**含义**：
- Count: 150 → 这个查询模式出现了150次
- Time=5.12s (768s) → 平均5.12秒，总计768秒
- Rows=100.0 (15000) → 平均返回100行，总计15000行

### 分析工具：pt-query-digest（推荐）

需要通过 shell 调用，不在 SQL 中运行：

```bash
# 分析慢查询日志，输出报告
pt-query-digest /var/lib/mysql/slow.log > slow_report.txt

# 只分析SELECT语句
pt-query-digest --filter '$event->{fingerprint} =~ m/^select/i' /var/lib/mysql/slow.log

# 分析最近的慢查询（最近1小时）
pt-query-digest --since 3600 /var/lib/mysql/slow.log

# 保存到数据库，方便长期追踪
pt-query-digest --review h=localhost,D=slow_query,t=query_review /var/lib/mysql/slow.log
```

### 使用 EXPLAIN 分析慢查询

```sql
-- 假设慢查询是：
SELECT * FROM orders WHERE user_id = 12345 AND status = 1;

-- 用EXPLAIN分析
EXPLAIN SELECT * FROM orders WHERE user_id = 12345 AND status = 1;

-- 关键字段：
-- type：访问类型（ALL/index/range/ref/eq_ref/const）
-- key：实际使用的索引
-- rows：预估扫描的行数
-- Extra：附加信息（Using filesort/Using temporary/Using index等）
```

**EXPLAIN 字段详解**：

| 字段 | 含义 | 优化目标 |
|------|------|----------|
| id | 查询的序列号 | 了解复杂查询的执行顺序 |
| select_type | 查询类型（SIMPLE/PRIMARY/SUBQUERY等） | 判断子查询或联合查询 |
| table | 访问的表 | - |
| partitions | 匹配的分区 | - |
| type | 访问类型 | 目标是ref/eq_ref/const，避免出现ALL |
| possible_keys | 可能使用的索引 | - |
| key | 实际使用的索引 | 如果是NULL，说明没用索引 |
| key_len | 使用的索引长度 | 判断联合索引用了几列 |
| ref | 索引的哪些列被使用 | - |
| rows | 预估扫描行数 | 越小越好 |
| filtered | 百分比 | 越高越好 |
| Extra | 附加信息 | 避免出现Using filesort/Using temporary |

**访问类型（type）从好到坏**：

```
system > const > eq_ref > ref > range > index > ALL

const：主键或唯一索引的等值查询（最多返回1行）
ref：非唯一索引的等值查询（可能返回多行）
range：范围查询（BETWEEN、IN、>、<等）
index：全索引扫描（比ALL好一点，因为只扫描索引树）
ALL：全表扫描（最差，需要优化）
```

## 它和相似方案的本质区别是什么？

### 慢查询日志 vs 性能_schema

| 维度 | 慢查询日志 | performance_schema |
|------|------------|-------------------|
| 开启方式 | 需要手动开启 | 默认开启（MySQL 5.7+） |
| 影响性能 | 轻微（写磁盘） | 轻微（内存中） |
| 分析粒度 | 慢查询级别（>long_query_time） | 更细粒度（可以按语句、线程、等待事件分析） |
| 使用难度 | 简单（有现成工具分析） | 复杂（需要写SQL查询） |
| 推荐场景 | 常规慢查询分析 | 深入性能诊断 |

**本质区别**：慢查询日志关注"哪些查询慢"，performance_schema关注"为什么慢"。

### EXPLAIN vs EXPLAIN ANALYZE

| 维度 | EXPLAIN | EXPLAIN ANALYZE（MySQL 8.0.18+） |
|------|---------|-------------------------------|
| 预估 vs 实际 | 预估执行计划 | 实际执行计划 + 实际耗时 |
| 是否需要执行 | 不需要 | 需要执行 |
| 准确性 | 预估，可能不准 | 实际数据，更准确 |
| 对生产环境影响 | 无 | 有（实际执行查询） |

**本质区别**：EXPLAIN是"预演"，EXPLAIN ANALYZE是"实战回放"。

## 正确使用方式

### 1. 通过 shell 调用分析工具（不在 SQL 中运行）

```bash
# 1. 找出最慢的10条SQL
mysqldumpslow -s t -t 10 /var/lib/mysql/slow.log

# 2. 对每条慢SQL，用EXPLAIN分析
# （这一步在MySQL客户端中执行，不是shell）

# 3. 针对性优化：
# - 没索引 → 加索引
# - 索引没用上 → 改写SQL或调整索引
# - 扫描行数太多 → 优化WHERE条件或索引
# - Using filesort → 优化ORDER BY或加索引
# - Using temporary → 优化GROUP BY或 DISTINCT
```

### 2. 优化步骤（标准流程）

```
1. 开启慢查询日志
   → SET GLOBAL slow_query_log = 'ON';

2. 等待业务高峰，收集慢查询
   → 最低等待1天，最好等待1周（覆盖所有查询模式）

3. 用mysqldumpslow或pt-query-digest分析
   → 找出最慢的、出现频率最高的SQL

4. 对每条慢SQL，用EXPLAIN分析
   → 找出问题（没用索引、扫描行数多、filesort等）

5. 针对性优化
   → 加索引、改写SQL、调整索引、分库分表等

6. 上线后继续监控
   → 确保优化有效，没有引入新问题
```

### 3. 常见优化手段

**优化1：加索引**

```sql
-- 慢查询：全表扫描
SELECT * FROM orders WHERE user_id = 12345;

-- 优化：加索引
CREATE INDEX idx_user_id ON orders(user_id);

-- 验证：EXPLAIN显示type=ref，扫描行数大幅减少
```

**优化2：改写SQL**

```sql
-- 慢查询：对索引字段用函数，导致索引失效
SELECT * FROM orders WHERE DATE(created_at) = '2024-01-01';

-- 优化：改写SQL，让索引生效
SELECT * FROM orders 
WHERE created_at >= '2024-01-01' 
  AND created_at < '2024-01-02';
-- 可以用上created_at索引
```

**优化3：避免SELECT ***

```sql
-- 慢查询：返回所有字段，包括大字段（text/blob）
SELECT * FROM orders WHERE user_id = 12345;

-- 优化：只查需要的字段，减少IO和网络传输
SELECT id, user_id, status, created_at 
FROM orders WHERE user_id = 12345;
```

**优化4：优化ORDER BY**

```sql
-- 慢查询：Using filesort
SELECT * FROM orders WHERE user_id = 12345 
ORDER BY created_at DESC;

-- 优化：加联合索引，避免filesort
CREATE INDEX idx_user_id_created ON orders(user_id, created_at);

-- 验证：EXPLAIN的Extra列不再显示Using filesort
```

## 边界情况和坑

### 坑1：long_query_time设置不合理

```
问题1：设置太长（如10秒）
→ 只能捕获极慢的查询，很多"亚健康"查询（2-3秒）被漏掉
→ 这些查询在高并发下也会拖垮数据库

问题2：设置太短（如0.1秒）
→ 捕获太多查询，慢查询日志膨胀
→ 分析工具运行缓慢
```

**合理设置**：

```sql
-- 一般业务：1秒
SET GLOBAL long_query_time = 1;

-- 高并发、低延迟业务：0.1秒
SET GLOBAL long_query_time = 0.1;
```

### 坑2：mysqldumpslow在Windows下不能用

```bash
# Windows下没有mysqldumpslow（Perl脚本）
# 替代方案1：用pt-query-digest（需要Perl环境）
pt-query-digest /path/to/slow.log

# 替代方案2：直接查看慢查询日志（用文本编辑器打开）
# 但这样无法统计和排序

# 替代方案3：用performance_schema
SELECT * FROM performance_schema.events_statements_history_long 
WHERE timer_wait > 1000000000  -- 超过1秒（单位是皮秒）
ORDER BY timer_wait DESC;
```

### 坑3：EXPLAIN的预估可能不准确

```sql
-- EXPLAIN显示的rows是预估的，可能和实际相差很大

-- 例如：
-- EXPLAIN显示rows=1000
-- 实际扫描了5000行

-- 原因：
-- 1. 统计信息过期（需要ANALYZE TABLE）
-- 2. 数据分布不均匀（某个user_id有10万行，其他只有10行）
-- 3. 优化器的成本估算模型不够精确
```

**解决**：

```sql
-- 1. 更新统计信息
ANALYZE TABLE orders;

-- 2. 用EXPLAIN ANALYZE（MySQL 8.0.18+）看实际执行情况
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 12345;

-- 3. 如果优化器选错索引，用FORCE INDEX强制用某个索引
SELECT * FROM orders FORCE INDEX(idx_user_id) 
WHERE user_id = 12345;
```

### 坑4：优化单个查询，但系统整体性能没提升

```
现象：
- 优化了最慢的10条SQL
- 但数据库整体负载还是很高

原因：
- 大量"中等速度"的查询（0.5-1秒）累积起来，负载也很高
- 或者，索引加太多，写入性能下降

解决：
1. 不仅要优化最慢的查询，还要优化"出现频率最高"的查询
2. 权衡索引数量和写入性能（单表索引不超过5-7个）
3. 考虑缓存（如Redis），减少数据库查询次数
```
