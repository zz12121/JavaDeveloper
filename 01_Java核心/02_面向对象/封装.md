# 封装

> 封装的本质不是"给字段加 private + getter/setter"，而是**把"可以改变的内部实现"藏起来，只暴露"稳定的接口"**。理解这个意图，才算真正理解封装。

---

## 这个问题为什么存在？

没有封装时，程序的数据和操作是分离的——任何人可以随意修改任何数据：

```java
// 没有封装：银行账户余额可以被任何人随意修改
class Account {
    public double balance;  // ← 没有任何约束
}

account.balance = -999999;  // 任何代码都能执行，没有任何约束
```

封装要解决的问题：

```
面向过程的根本问题：
- 数据和操作数据的函数分离：谁都可以改 struct 里的数据
- 模块边界模糊：函数和全局变量交织，依赖难以追踪
- 复用靠复制代码，代码一处改，处处都要改

面向对象的解法：
把数据和对数据的操作绑在一起（类），
只暴露受控的访问方式（访问修饰符），
把"可以改变的内部实现"藏起来。
```

---

## 它是怎么解决问题的？

### 访问控制修饰符：封装的工具

| 修饰符 | 可见范围 | 设计意图 |
|---------|----------|------------|
| `private` | 仅本类 | 最严格，完全隐藏 |
| （包私有） | 同包 | 默认，包级封装 |
| `protected` | 同包 + 子类 | 给子类扩展空间 |
| `public` | 所有人 | 最宽松，对外稳定接口 |

**设计原则**：**最小暴露原则（Principle of Least Privilege）**——能 private 就 private，必要时才放宽。

### getter/setter 不等于封装

一个常见误区：给所有字段加 getter/setter，以为就是封装了。

```java
// 这不是封装，是伪封装
class Person {
    private String name;
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
    // 跟 public 字段几乎没有区别！
}
```

真正的封装是：**字段的修改必须通过有业务语义的方法**。如果一个 setter 里什么都不做，只是直接赋值，和 public 字段几乎等价。

```java
// 好的封装：Name 有业务约束
class Person {
    private String name;

    public Person(String name) {
        setName(name);  // 构造器也走校验逻辑
    }

    public void setName(String name) {
        if (name == null || name.isBlank()) {
            throw new IllegalArgumentException("姓名不能为空");
        }
        this.name = name;
    }

    public String getName() { return name; }  // 只读，无副作用
}
```

### 封装的四大价值

```
1. 修改内部实现时，只要接口不变，调用方不需要改
   → 例：把字段存储从 String 改为 String[]（多名字），调用方无感知

2. 可以在方法里做参数校验，防止非法状态
   → 例：Account.withdraw() 里检查余额

3. 单元测试的边界更清晰
   → 只需测暴露的接口，内部实现变化不影响测试

4. 多线程安全更容易保障
   → 同步逻辑集中在方法内部，不需要外部协调
```

### JavaBean 规范的误区

JavaBean 规范要求字段 private + getter/setter，这让很多框架（Spring、MyBatis）能直接操作对象。但**JavaBean 规范不等于封装最佳实践**：

```java
// JavaBean 式的封装（框架友好，但业务上可能不合理）
class User {
    private String password;
    public String getPassword() { return password; }  // ← 密码不应该有 getter！
    public void setPassword(String p) { this.password = p; }
}

// 更好的封装：不暴露密码原文
class User {
    private String passwordHash;
    public boolean checkPassword(String input) {
        return BCrypt.checkpw(input, passwordHash);
    }
}
```

## 深入原理

### 编译器如何处理 private

`private` 是编译期检查，`protected` / `public` 是运行时检查的。

```java
// 编译期：编译器检查 private 的访问
class Outer {
    private int secret;

    class Inner {
        void peek() { System.out.println(secret); }  // ✓ 编译通过（内部类特权）
    }
}

// 运行时：protected / public 的覆写遵循运行时多态规则
// private 方法不参与多态，编译期就绑定了目标方法（invokespecial）
```

编译器在允许内部类访问外部类 private 成员时，会在外部类生成一个**合成方法**（`access$XXX`），让内部类能绕过 private 限制。这是 Java 语言层面的"friend"机制。详见 [[01_Java核心/02_面向对象/内部类|内部类]]。

### 封装与不变性的关系

封装和 [[01_Java核心/01_基础语法/final关键字|final]] 是不可分割的：

```java
// 封装 + final = 真正的不可变对象
public final class ImmutablePoint {
    private final int x;        // final：创建后不能改
    private final int y;

    public ImmutablePoint(int x, int y) {
        this.x = x;
        this.y = y;
    }

    public int getX() { return x; }  // 只有 getter，没有 setter
    public int getY() { return y; }
}
```

如果类不是 final 的，子类可以绕过不变性约束。所以 String 必须同时是 final 的和字段 private final。

## 正确使用方式

### 1. 最小暴露原则

```
设计类时的检查清单：
□ 所有字段 private？
□ 对外暴露的方法是否都是接口契约（不会随便改）？
□ 有没有不必要的 public 方法？
□ 包内共享的方法，是不是可以用 protected 或包私有？
□ 构造器是否需要限制（单例用 private 构造器）？
```

### 2. 业务语义优于简单 getter/setter

```java
// 不好：暴露了实现细节，且破坏了业务约束
class Account {
    private double balance;
    public double getBalance() { return balance; }
    public void setBalance(double b) { this.balance = b; }  // ← 业务上不应该存在
}

// 好：只暴露有业务语义的操作
class Account {
    private double balance;

    public boolean withdraw(double amount) {
        if (amount <= 0 || amount > balance) return false;
        balance -= amount;
        return true;
    }

    public void deposit(double amount) {
        if (amount <= 0) throw new IllegalArgumentException();
        balance += amount;
    }

    public double getBalance() { return balance; }  // 只读
}
```

### 3. 防御性拷贝

当方法返回可变对象的引用时，需要返回拷贝，防止外部修改内部状态：

```java
class Profile {
    private final int[] scores;

    public Profile(int[] scores) {
        this.scores = scores.clone();  // 构造器也要防御：传入的数组可能被外部修改
    }

    public int[] getScores() {
        return scores.clone();  // 不返回原数组引用
        // 或者返回不可变视图：Collections.unmodifiableList(...)
    }
}
```

详见 [[01_Java核心/02_面向对象/克隆与拷贝|克隆与拷贝]] 中深拷贝的相关内容。

## 边界情况和坑

### 1. 包私有（默认修饰符）的陷阱

```java
// PackageA/MyClass.java
class MyClass {           // 没有修饰符 = 包私有
    void doSomething() { }
}

// PackageB/Test.java
class Test {
    void test() {
        new MyClass().doSomething();  // 编译错误！不同包不可见
    }
}
```

**最常见的坑**：以为"没写修饰符就是 private"，实际上它是包级可见。如果需要跨包访问，必须显式写 `public`。

### 2. protected 的访问范围

```java
// PackageA/Parent.java
package packageA;
public class Parent {
    protected void method() { }
}

// PackageB/Child.java
package packageB;
public class Child extends Parent {
    void test() {
        method();                         // ✓ 子类可以访问继承来的 protected 方法
        new Parent().method();            // ✗ 编译错误！不能通过父类实例访问
        new Child().method();             // ✓ 可以通过子类实例访问
    }
}
```

`protected` 的访问规则：**同包 ✓，不同包的子类可以访问继承来的成员，但不能通过父类引用访问**。这是 Java 语言规范里最容易误解的规则之一。

### 3. setter 引发的不可变性破坏

```java
class Point {
    private int x;
    private int y;

    public void setX(int x) { this.x = x; }  // ← 这直接破坏了"不可变"的设计意图
}

// 如果本来想设计不可变对象，就不应该有 setter
// 需要"修改"时，返回新对象：
class ImmutablePoint {
    private final int x;
    private final int y;

    public ImmutablePoint withX(int newX) {
        return new ImmutablePoint(newX, this.y);  // 返回新对象，不修改原对象
    }
}
```

### 4. 内部类破坏封装

成员内部类持有外部类的引用（`this$0`），可以访问外部类的所有 private 成员——这实际上**在同一个类的边界内**，没有破坏封装。但如果内部类被发布到外部（比如作为返回值），外部代码就可以通过内部类访问本应是 private 的字段。

```java
class Outer {
    private int secret = 42;

    public Inner getInner() {
        return new Inner();  // ← 发布内部类，secret 实际上可间接访问
    }

    class Inner {
        public int peek() { return secret; }  // 可以访问 private
    }
}
```

使用 [[01_Java核心/01_基础语法/final关键字|static 内部类]] 可以避免这个问题——它不持有外部类引用。
