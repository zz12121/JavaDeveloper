## 场景 A：TCC实现

### 现象

```
订单创建成功后扣减库存，库存扣减失败但订单已创建，数据不一致
TCC Try阶段成功，Confirm阶段超时，用户看到订单创建失败但库存已扣
TCC Cancel阶段执行失败，导致资源悬挂，后续无法重试
```

### 根因分析

TCC（Try-Confirm-Cancel）模式要求三个阶段的幂等性和一致性。Try阶段预留资源，Confirm阶段确认执行，Cancel阶段释放资源。问题根因在于：1) Try与Confirm之间没有事务状态机保障，Confirm失败无重试；2) Cancel阶段未处理幂等，重复执行导致数据错误；3) 网络超时导致状态不一致，事务协调器无法判断参与者实际状态。

### 解决方案

```java
/**
 * TCC分布式事务实现：支持幂等、重试、状态机
 */
@Component
public class TccTransactionManager {
    // 事务状态机：记录事务状态（TRYING/CONFIRMED/CANCELLED）
    @Autowired
    private TransactionStateMachine stateMachine;

    /**
     * 执行TCC事务
     * @param context 事务上下文
     */
    public void executeTcc(TccContext context) {
        String xid = context.getXid();
        List<TccParticipant> participants = context.getParticipants();

        try {
            // 1. Try阶段：所有参与者执行Try
            for (TccParticipant participant : participants) {
                stateMachine.updateState(xid, TransactionState.TRYING);
                boolean success = participant.tryExecute();
                if (!success) {
                    throw new TccException("Try阶段失败：" + participant.getName());
                }
            }

            // 2. Confirm阶段：所有参与者执行Confirm
            for (TccParticipant participant : participants) {
                confirmWithRetry(participant, xid);
            }
            stateMachine.updateState(xid, TransactionState.CONFIRMED);

        } catch (Exception e) {
            // 3. Cancel阶段：回滚所有参与者
            for (TccParticipant participant : participants) {
                cancelWithRetry(participant, xid);
            }
            stateMachine.updateState(xid, TransactionState.CANCELLED);
            throw new TccException("TCC事务回滚完成", e);
        }
    }

    /**
     * Confirm阶段带重试（幂等）
     */
    private void confirmWithRetry(TccParticipant participant, String xid) {
        int retryCount = 0;
        while (retryCount < 3) {
            try {
                // 幂等性检查：若已Confirm，直接返回成功
                if (stateMachine.isConfirmed(xid, participant.getName())) {
                    return;
                }
                participant.confirm();
                stateMachine.markConfirmed(xid, participant.getName());
                return;
            } catch (Exception e) {
                retryCount++;
                if (retryCount >= 3) {
                    throw new TccException("Confirm阶段重试失败", e);
                }
                try {
                    Thread.sleep(1000 * retryCount); // 指数退避
                } catch (InterruptedException ie) {
                    Thread.currentThread().interrupt();
                }
            }
        }
    }

    /**
     * Cancel阶段带重试（幂等）
     */
    private void cancelWithRetry(TccParticipant participant, String xid) {
        int retryCount = 0;
        while (retryCount < 3) {
            try {
                // 幂等性检查：若已Cancel，直接返回成功
                if (stateMachine.isCancelled(xid, participant.getName())) {
                    return;
                }
                participant.cancel();
                stateMachine.markCancelled(xid, participant.getName());
                return;
            } catch (Exception e) {
                retryCount++;
                if (retryCount >= 3) {
                    // 记录死信，人工介入
                    log.error("Cancel阶段重试失败，xid={}, participant={}", xid, participant.getName(), e);
                    return;
                }
                try {
                    Thread.sleep(1000 * retryCount);
                } catch (InterruptedException ie) {
                    Thread.currentThread().interrupt();
                }
            }
        }
    }
}
```

## 场景 B：SAGA实现

### 现象

```
长事务流程（订单→库存→物流→积分），物流失败导致前面全部回滚，性能差
SAGA正向操作成功，补偿操作执行失败，事务状态不一致
补偿操作顺序错误，先补偿库存再补偿订单，导致库存释放但订单仍存在
```

### 根因分析

SAGA模式将长事务拆分为多个本地事务，每个本地事务有对应的补偿操作。问题根因在于：1) 补偿操作未按正向操作的逆序执行，导致数据状态错误；2) 补偿操作未实现幂等，重复执行导致数据混乱；3) 补偿操作失败未记录死信，无法人工介入恢复。

### 解决方案

```java
/**
 * SAGA分布式事务实现：支持正向/补偿操作、逆序回滚、状态机
 */
@Component
public class SagaTransactionManager {
    @Autowired
    private SagaStateRepository stateRepository;

    /**
     * 执行SAGA事务
     * @param sagaDefinition SAGA定义（包含步骤列表）
     */
    public void executeSaga(SagaDefinition sagaDefinition) {
        String sagaId = sagaDefinition.getSagaId();
        List<SagaStep> steps = sagaDefinition.getSteps();
        int currentStep = 0;

        try {
            // 1. 正向执行：按顺序执行每个步骤
            for (; currentStep < steps.size(); currentStep++) {
                SagaStep step = steps.get(currentStep);
                stateRepository.saveStepState(sagaId, step.getName(), SagaState.EXECUTING);

                boolean success = step.execute();
                if (!success) {
                    throw new SagaException("正向操作失败：" + step.getName());
                }
                stateRepository.saveStepState(sagaId, step.getName(), SagaState.COMPLETED);
            }

        } catch (Exception e) {
            // 2. 补偿回滚：按逆序执行补偿操作
            log.error("SAGA事务失败，开始补偿回滚，sagaId={}", sagaId, e);
            compensateSteps(sagaId, steps, currentStep);
            throw new SagaException("SAGA事务回滚完成", e);
        }
    }

    /**
     * 补偿回滚：按正向操作的逆序执行补偿
     * @param completedStepIndex 失败步骤的索引
     */
    private void compensateSteps(String sagaId, List<SagaStep> steps, int completedStepIndex) {
        // 逆序遍历：从失败步骤的前一步开始，向前补偿
        for (int i = completedStepIndex - 1; i >= 0; i--) {
            SagaStep step = steps.get(i);
            // 仅对已完成的步骤执行补偿
            if (stateRepository.getStepState(sagaId, step.getName()) == SagaState.COMPLETED) {
                compensateWithRetry(sagaId, step);
            }
        }
    }

    /**
     * 补偿操作带重试（幂等）
     */
    private void compensateWithRetry(String sagaId, SagaStep step) {
        int retryCount = 0;
        while (retryCount < 3) {
            try {
                // 幂等性检查
                if (stateRepository.isCompensated(sagaId, step.getName())) {
                    return;
                }
                step.compensate(); // 执行补偿操作
                stateRepository.markCompensated(sagaId, step.getName());
                return;
            } catch (Exception e) {
                retryCount++;
                if (retryCount >= 3) {
                    // 记录死信，人工介入
                    stateRepository.saveDeadLetter(sagaId, step.getName(), e.getMessage());
                    log.error("补偿操作失败，已记录死信，sagaId={}, step={}", sagaId, step.getName(), e);
                    return;
                }
                try {
                    Thread.sleep(1000 * retryCount);
                } catch (InterruptedException ie) {
                    Thread.currentThread().interrupt();
                }
            }
        }
    }
}
```

## 场景 C：事务状态机

### 现象

```
事务协调器宕机，恢复后无法确定事务状态，导致数据不一致
事务状态丢失，参与者无法查询自己所属事务的状态
事务超时后状态未自动清理，数据库积压大量历史事务记录
```

### 根因分析

事务状态机是分布式事务的核心，记录事务的全局状态和各参与者的本地状态。问题根因在于：1) 状态机未持久化，协调器宕机后状态丢失；2) 状态机未提供查询接口，参与者无法主动查询状态；3) 状态机未设置超时清理机制，历史数据堆积。

### 解决方案

```java
/**
 * 分布式事务状态机：持久化、查询、超时清理
 */
@Component
public class TransactionStateMachine {
    @Autowired
    private TransactionStateRepository repository;

    // 事务超时时间：30分钟
    private static final long TRANSACTION_TIMEOUT = 30 * 60 * 1000;

    /**
     * 创建事务记录
     */
    public void createTransaction(String xid, List<String> participants) {
        TransactionRecord record = new TransactionRecord();
        record.setXid(xid);
        record.setState(TransactionState.INIT);
        record.setCreateTime(System.currentTimeMillis());
        record.setParticipants(participants);
        repository.save(record);
    }

    /**
     * 更新事务状态
     */
    public void updateState(String xid, TransactionState newState) {
        TransactionRecord record = repository.findByXid(xid);
        if (record == null) {
            throw new TransactionException("事务不存在：" + xid);
        }
        // 状态机转换校验：INIT -> TRYING -> CONFIRMED/CANCELLED
        if (!isValidStateTransition(record.getState(), newState)) {
            throw new TransactionException("非法状态转换：" + record.getState() + " -> " + newState);
        }
        record.setState(newState);
        record.setUpdateTime(System.currentTimeMillis());
        repository.save(record);
    }

    /**
     * 查询事务状态（供参与者查询）
     */
    public TransactionState queryState(String xid) {
        TransactionRecord record = repository.findByXid(xid);
        if (record == null) {
            return null;
        }
        // 检查是否超时
        if (isTimeout(record)) {
            record.setState(TransactionState.TIMEOUT);
            repository.save(record);
        }
        return record.getState();
    }

    /**
     * 超时事务清理任务（定时执行）
     */
    @Scheduled(cron = "0 0 2 * * ?") // 每天凌晨2点执行
    public void cleanupTimeoutTransactions() {
        long timeoutTime = System.currentTimeMillis() - TRANSACTION_TIMEOUT;
        List<TransactionRecord> timeoutRecords = repository.findByCreateTimeBefore(timeoutTime);
        for (TransactionRecord record : timeoutRecords) {
            // 仅清理已终态的事务（CONFIRMED/CANCELLED/TIMEOUT）
            if (record.getState().isFinal()) {
                repository.delete(record);
                log.info("清理超时事务：xid={}", record.getXid());
            } else {
                // 非终态事务标记为超时，触发回滚
                record.setState(TransactionState.TIMEOUT);
                repository.save(record);
                log.warn("标记超时事务：xid={}", record.getXid());
            }
        }
    }

    private boolean isTimeout(TransactionRecord record) {
        return System.currentTimeMillis() - record.getCreateTime() > TRANSACTION_TIMEOUT;
    }

    private boolean isValidStateTransition(TransactionState from, TransactionState to) {
        // 状态机转换规则
        switch (from) {
            case INIT:
                return to == TransactionState.TRYING;
            case TRYING:
                return to == TransactionState.CONFIRMED || to == TransactionState.CANCELLED;
            default:
                return false;
        }
    }
}
```

## 场景 D：空回滚与悬挂

### 现象

```
TCC Try阶段超时，事务协调器触发Cancel，但Try实际未执行，导致空回滚
Cancel阶段执行后，Try阶段请求才到达，导致资源悬挂
幂等控制不当，重复执行Confirm导致数据重复提交
```

### 根因分析

空回滚根因：Try阶段网络超时，协调器认为Try失败触发Cancel，但实际Try已执行或完全未执行，Cancel执行时无资源可释放或释放未预留的资源。悬挂根因：Try请求网络延迟，Cancel先执行，Try后到达，导致Try预留的资源无法被Confirm或Cancel。幂等控制不当：Confirm/Cancel未做幂等校验，重复执行导致数据错误。

### 解决方案

```java
/**
 * TCC防悬挂与空回滚组件：记录Try执行状态，拦截无效操作
 */
@Component
public class TccIdempotentGuard {
    @Autowired
    private TccActionRecordRepository recordRepository;

    /**
     * Try阶段前置检查：记录Try开始执行
     */
    public boolean beforeTry(String xid, String branchId) {
        TccActionRecord record = recordRepository.findByXidAndBranchId(xid, branchId);
        if (record == null) {
            record = new TccActionRecord();
            record.setXid(xid);
            record.setBranchId(branchId);
            record.setState(ActionState.TRYING);
            record.setTryTime(System.currentTimeMillis());
            recordRepository.save(record);
            return true;
        }
        // 已存在记录，说明Try已执行，幂等返回
        return false;
    }

    /**
     * Confirm阶段前置检查：防悬挂、防空回滚
     */
    public boolean beforeConfirm(String xid, String branchId) {
        TccActionRecord record = recordRepository.findByXidAndBranchId(xid, branchId);
        if (record == null) {
            // 空回滚：Try未执行，直接返回成功（幂等）
            log.warn("空回滚检测：xid={}, branchId={}", xid, branchId);
            return false;
        }
        if (record.getState() == ActionState.CONFIRMED) {
            // 已Confirm，幂等返回
            return false;
        }
        if (record.getState() == ActionState.CANCELLED) {
            // 已Cancel，无法Confirm，抛出异常
            throw new TccException("资源已Cancel，无法Confirm");
        }
        // 更新状态为Confirming
        record.setState(ActionState.CONFIRMING);
        recordRepository.save(record);
        return true;
    }

    /**
     * Cancel阶段前置检查：防悬挂、防空回滚
     */
    public boolean beforeCancel(String xid, String branchId) {
        TccActionRecord record = recordRepository.findByXidAndBranchId(xid, branchId);
        if (record == null) {
            // Try未执行，记录空回滚标记，防止后续Try执行（悬挂）
            record = new TccActionRecord();
            record.setXid(xid);
            record.setBranchId(branchId);
            record.setState(ActionState.CANCELLED);
            record.setCancelTime(System.currentTimeMillis());
            recordRepository.save(record);
            log.warn("空回滚标记：xid={}, branchId={}", xid, branchId);
            return false;
        }
        if (record.getState() == ActionState.CANCELLED) {
            // 已Cancel，幂等返回
            return false;
        }
        if (record.getState() == ActionState.CONFIRMED) {
            // 已Confirm，无法Cancel，抛出异常
            throw new TccException("资源已Confirm，无法Cancel");
        }
        // 更新状态为Cancelling
        record.setState(ActionState.CANCELLING);
        recordRepository.save(record);
        return true;
    }

    /**
     * Try阶段后置处理：标记Try成功，检查是否已Cancel（悬挂）
     */
    public void afterTry(String xid, String branchId) {
        TccActionRecord record = recordRepository.findByXidAndBranchId(xid, branchId);
        if (record != null && record.getState() == ActionState.CANCELLED) {
            // 悬挂：Try执行前已Cancel，立即执行Cancel补偿
            log.warn("悬挂检测：xid={}, branchId={}，立即补偿Cancel", xid, branchId);
            // 调用Cancel逻辑...
        } else if (record != null) {
            record.setState(ActionState.TRY_SUCCESS);
            recordRepository.save(record);
        }
    }
}
```

## 核心参数估算

```
事务性能参数：
  TCC模式：Try阶段RT 20ms，Confirm阶段RT 10ms，整体RT 50ms
  SAGA模式：每步骤RT 30ms，10步骤长事务整体RT 300ms
  事务协调器QPS：单机支持1万TPS，集群10万TPS

资源占用参数：
  事务状态存储：每条记录约200字节，10万并发事务约20MB内存
  事务超时时间：30分钟，超时后自动回滚或清理
  重试间隔：指数退避（1s, 2s, 4s），最大重试3次

一致性参数：
  最终一致性延迟：< 5秒（含网络延迟+重试）
  幂等性窗口：24小时（防止重复提交）
  死信队列：补偿失败记录，人工介入处理
```

## 排查 Checklist

```
□ TCC Try阶段是否幂等？ → 同一xid+branchId仅执行一次Try
□ Confirm/Cancel是否幂等？ → 重复执行结果一致，无数据错误
□ 空回滚是否处理？ → Try未执行时Cancel直接返回成功
□ 悬挂是否处理？ → Try执行前已Cancel，立即补偿Cancel
□ 事务状态是否持久化？ → 协调器宕机恢复后状态不丢失
□ 补偿操作是否逆序？ → SAGA补偿按正向操作逆序执行
□ 补偿失败是否有死信？ → 记录死信，人工介入处理
□ 事务超时是否设置？ → 30分钟超时，自动回滚或清理
□ 重试间隔是否合理？ → 指数退避，避免雪崩
□ 参与者能否查询事务状态？ → 提供查询接口，避免状态不一致
```

## 追问链

### 追问 1：TCC和SAGA模式如何选择？

> TCC适合**短事务**、**强一致性**场景（如订单+库存），要求所有参与者在秒级完成。SAGA适合**长事务**、**最终一致性**场景（如旅游预订：机票+酒店+租车），允许分钟级完成。TCC需要业务改造（预留资源接口），SAGA只需补偿接口，侵入性更低。

### 追问 2：如何处理TCC的悬挂问题？

> 悬挂根因是Try请求延迟，Cancel先执行。解决方案：1) **记录Cancel状态**：Cancel执行时若Try未执行，记录空回滚标记。2) **Try前置检查**：Try执行前检查是否已Cancel，若已Cancel则立即执行Cancel补偿，避免资源预留。3) **请求超时控制**：Try请求设置超时时间，超时后不再执行，避免无效预留。

### 追问 3：事务协调器宕机如何恢复？

> 1) **状态持久化**：事务状态存储到DB或Redis，协调器宕机后新实例加载未完成事务。2) **超时回滚**：协调器恢复后，扫描超时事务（如创建时间>30分钟且未终态），触发回滚。3) **参与者补偿**：参与者定期查询事务状态，若发现事务已回滚但本地未回滚，主动执行补偿。

### 追问 4：SAGA补偿操作失败怎么办？

> 1) **重试机制**：补偿操作失败自动重试3次，指数退避。2) **死信队列**：重试失败后记录到死信队列，人工介入处理。3) **对账修复**：定时对账，发现数据不一致则自动修复或告警。4) **设计可补偿业务**：业务设计时确保补偿操作可重试、无副作用（如库存释放可重复执行）。
