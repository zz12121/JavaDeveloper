# 高并发秒杀设计

> 秒杀的核心矛盾是：**库存有限，请求无数**。所有设计都围绕「用最小的代价过滤掉最多无效请求，同时保证不超卖」展开。

## 场景 A：库存超卖（最严重 bug）

### 现象

```
卖出 100 件商品
实际库存只有 50
超卖 50 件
用户付款后无法履约 → 退款 → 差评 → 资损
```

### 根因

```
库存 1，多个线程同时读到有库存，都去下单
```

### 解决方案

```java
// 1. Redis 原子扣减（第一道防线）
public Result seckill(Long userId, Long productId) {
    String stockKey = "seckill:stock:" + productId;
    String userKey = "seckill:user:" + productId;
    
    // 每个用户只能秒杀一次（用 SETNX 防重复）
    if (!redis.setnx(userKey, "1")) {
        return Result.error("您已参与过秒杀");
    }
    redis.expire(userKey, 60 * 60);  // 1小时过期
    
    // 原子扣减库存（核心）
    Long stock = redis.decr(stockKey);
    if (stock < 0) {
        redis.incr(stockKey);  // 回补库存
        return Result.error("商品已售罄");
    }
    
    // 库存扣减成功，发送消息异步下单
    mq.send("seckill:order", new SeckillMsg(userId, productId));
    return Result.success("秒杀成功");
}
```

```sql
-- 2. DB 层兜底（第二道防线）
UPDATE seckill_stock 
SET stock = stock - 1, sold = sold + 1
WHERE product_id = ? AND stock > 0;

-- 检查影响行数
if (affectedRows == 0) {
    throw new SeckillException("库存不足");
}
```

**防超卖关键**：
- `stock > 0` 条件必须加在 UPDATE 语句里，不是先 SELECT 再 UPDATE
- 先扣 Redis，失败直接返回；DB 作为最终兜底

## 场景 B：恶意请求 / 爬虫刷接口

### 现象

```
接口 QPS 10万+
有效请求只有 1000（库存数量）
其余 99000 都是爬虫/脚本/刷子
```

### 解决方案

```
限流策略（多层级）：
1. 网关层：IP 限流（单 IP 每秒 N 次）
2. 业务层：用户 ID 限流（同一用户每秒 N 次）
3. 接口层：验证码（图形验证码/短信验证码）
4. 识别层：风险识别（风控引擎，识别机器行为）
```

```java
// 验证码限流
@RestController
public class SeckillController {
    
    @Autowired
    private RedisRateLimiter rateLimiter;
    
    public Result seckillPage(Long productId, String token) {
        // 1. 图形验证码（防机器）
        if (!verifyCaptcha(token)) {
            return Result.error("请先验证");
        }
        
        // 2. 风控检查
        if (riskService.isRisky(userId)) {
            return Result.error("操作异常");
        }
        
        // 3. 验证码通过后才暴露秒杀接口地址
        return Result.success(generateSeckillToken(productId, userId));
    }
    
    public Result doSeckill(String seckillToken, Long productId) {
        // 4. 验证秒杀 token（一次性，防止 URL 被收藏/分享）
        if (!verifySeckillToken(seckillToken, userId, productId)) {
            return Result.error("秒杀令牌无效");
        }
        
        // 5. 执行秒杀
        return seckillService.execute(userId, productId);
    }
}
```

## 场景 C：热点数据导致 Redis 卡顿

### 现象

```
Redis CPU 使用率 100%
Redis 响应时间从 1ms 飙升到 500ms
大量请求超时
```

### 根因

热点商品，所有请求都打同一个 Redis key（库存 key）。

### 解决方案

```
1. 库存分桶（最有效）
   库存 1000 → 分 10 个桶，每桶 100
   key: seckill:stock:{productId}:{bucketId}
   用户随机分配桶

2. Redis Cluster（水平扩展）
   多节点分担压力

3. 本地缓存热点数据
   库存扣减前用本地锁过滤大部分请求
```

```java
// 分桶实现
public Result seckill(Long userId, Long productId) {
    // 1. 用本地缓存过滤无效请求（Guava Cache / Caffeine）
    LoadingCache<Long, Boolean> localBlacklist = 
        Caffeine.newBuilder()
                .maximumSize(10_000)
                .expireAfterWrite(1, TimeUnit.HOURS)
                .build(id -> {
                    // 查 DB 看是否还有资格
                    return hasSeckillQuota(id);
                });
    
    if (!localBlacklist.get(userId)) {
        return Result.error("无秒杀资格");
    }
    
    // 2. 随机分桶
    int bucketId = userId.hashCode() % 10;
    String stockKey = "seckill:stock:" + productId + ":" + bucketId;
    
    Long stock = redis.decr(stockKey);
    if (stock < 0) {
        return Result.error("商品已售罄");
    }
    // ...
}
```

## 场景 D：下单后用户不付款（库存占用）

### 现象

```
秒杀成功，库存已扣
用户不付款
库存被占用 15 分钟
真正想买的用户买不了
```

### 解决方案

```
方案1：下单减库存（已采用）→ 需配合付款超时机制
方案2：预占库存 → 付款成功后真正扣减
方案3：付款倒计时 → 超时自动释放库存
```

```java
// 定时任务：超时未付款，释放库存
@Scheduled(cron = "0/10 * * * * ?")
public void releaseTimeoutOrders() {
    // 查找超时订单（10分钟未付款）
    List<Order> timeoutOrders = orderMapper
        .selectTimeoutUnpaid(10, TimeUnit.MINUTES.toSeconds(10));
    
    for (Order order : timeoutOrders) {
        // 1. 关闭订单
        orderMapper.updateStatus(order.getId(), OrderStatus.CLOSED);
        
        // 2. 释放 Redis 库存（加回去）
        String stockKey = "seckill:stock:" + order.getProductId();
        redis.incr(stockKey);
        
        // 3. 释放用户购买资格
        redis.del("seckill:user:" + order.getProductId() + ":" + order.getUserId());
        
        log.info("超时释放库存: orderId={}, productId={}", 
                 order.getId(), order.getProductId());
    }
}
```

## 秒杀核心参数

```
关键参数估算：
  假设：10000 库存，100000 用户参与，持续 1 小时
  平均 QPS = 100000 / 3600 ≈ 28 req/s
  峰值 QPS = 平均 × 10 ≈ 280 req/s
  
  实际峰值可能更高（秒杀开始前集中刷新页面）
  预留 3-5 倍 buffer：峰值按 1000-1500 req/s 设计

参数配置：
  Redis 连接池：core=50, max=200
  Tomcat 线程池：core=100, max=500, queue=1000
  MQ 消费者：concurrent=50
```

## 追问链

### 追问 1：分布式锁在秒杀中怎么用？

> "一般不用分布式锁做库存扣减（太慢）。分布式锁的用途：1. 支付回调时的幂等校验。2. 定时任务释放库存时的互斥。库存扣减用 Redis 的 DECR 原子操作就够了，比分布式锁快 10 倍以上。"

### 追问 2：MQ 消息丢了怎么办？

> "1. 生产端：开启事务模式（但影响吞吐量）或手动 confirm 模式。2. 消费端：先扣库存再 ACK，失败自动重试。3. 死信队列：消息处理 N 次失败后进入死信队列，人工处理。4. 最终兜底：定时扫描超时未处理的秒杀记录，主动补偿。"

### 追问 3：怎么保证不超卖又不少卖？

> "Redis DECR 保证原子扣减（不会超卖）。DB 的 `UPDATE ... WHERE stock > 0` 是双重保险。Redis 和 DB 定期对账（Redis 库存 vs 实际已付款订单数），发现差异告警，人工介入。"

### 追问 4：秒杀开始瞬间的流量怎么扛？

> "1. 秒杀按钮开始前灰度开放（逐步放量）。2. CDN 缓存秒杀详情页。3. 网关层直接拦截非秒杀时段的请求。4. 验证码过滤掉一半机器流量。5. Redis 原子操作快速过滤无效请求。核心思路：**在用户付钱之前，每一层都尽力过滤无效请求**。"

## 排查 Checklist

```
□ Redis 库存扣减是原子的吗？ → DECR / Lua 脚本
□ DB 层有兜底防超卖吗？ → UPDATE WHERE stock > 0
□ 用户重复秒杀能拦住吗？ → SETNX userKey
□ MQ 消费失败会重试吗？ → 死信队列兜底
□ 超时未付款会释放库存吗？ → 定时任务扫描
□ 热点 key 有分桶吗？ → 多 key 分担压力
□ 限流规则合理吗？ → IP + 用户 ID 双维度
□ 系统瓶颈在哪？ → Redis / DB / MQ 分别压测
□ 有对账机制吗？ → Redis vs DB vs 订单表
```
