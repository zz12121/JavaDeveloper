# TCP/IP 协议族

> **核心认知**：TCP/IP 不是两个协议，而是一个协议族（Protocol Suite）。它定义了数据如何在网络中从一台计算机传输到另一台计算机——从应用层发出请求，到物理层变成电信号，中间每一层各司其职。

---

## 1. 为什么需要分层？

如果网络通信不分层，每个程序都需要自己处理：数据如何变成电信号、经过路由器怎么转发、丢包了怎么重传、对方端口怎么找……这根本不可行。

**分层的核心思想**：每层只关心自己的事，上层调用下层的服务，下层为上层提供服务。

```
你写代码发送一个 HTTP 请求，背后发生了什么：

应用层：构造 HTTP 请求报文（GET /index.html HTTP/1.1）
  ↓
传输层：把报文切块，加上端口号（TCP:80），保证可靠传输
  ↓
网络层：加上 IP 地址（目标服务器的 IP），选择路由路径
  ↓
数据链路层：加上 MAC 地址，通过网卡发送出去
  ↓
物理层：变成电信号/光信号/电磁波，在网线/光纤/空气中传输
```

## 2. 两种分层模型

### 2.1 OSI 七层模型（理论模型）

由 ISO（国际标准化组织）于 1984 年提出，是**理论参考模型**，实际网络中并不严格按照它实现。

```
┌─────────────────────────────────┐
│   7  应用层 (Application)        │  HTTP / FTP / DNS / SMTP
│   6  表示层 (Presentation)       │  数据格式转换、加密、压缩
│   5  会话层 (Session)            │  建立、管理、终止会话
├─────────────────────────────────┤
│   4  传输层 (Transport)          │  TCP / UDP
├─────────────────────────────────┤
│   3  网络层 (Network)            │  IP / ICMP / ARP
├─────────────────────────────────┤
│   2  数据链路层 (Data Link)      │  以太网 / Wi-Fi / PPP
│   1  物理层 (Physical)           │  网线 / 光纤 / 集线器
└─────────────────────────────────┘
```

### 2.2 TCP/IP 四层模型（实际使用的模型）

TCP/IP 模型把 OSI 的上三层合并为"应用层"，下两层合并为"网络接口层"。

```
┌─────────────────────────────────┐
│   应用层 (Application)           │  HTTP / DNS / FTP / SMTP
├─────────────────────────────────┤
│   传输层 (Transport)             │  TCP / UDP
├─────────────────────────────────┤
│   网际层 (Internet)              │  IP / ICMP / ARP
├─────────────────────────────────┤
│   网络接口层 (Network Interface)  │  以太网 / Wi-Fi / 网卡驱动
└─────────────────────────────────┘
```

### 2.3 两种模型的对应关系

| OSI 七层 | TCP/IP 四层 | 对应协议 | 数据单元 (PDU) |
|----------|------------|---------|---------------|
| 7. 应用层 | \multirow{3}{*}{应用层} | HTTP、DNS、FTP、SMTP、SSH | \multirow{3}{*}{报文 (Message)} |
| 6. 表示层 | | TLS/SSL、JPEG、ASCII | |
| 5. 会话层 | | NetBIOS、RPC | |
| 4. 传输层 | 传输层 | TCP、UDP | 段 (Segment/Datagram) |
| 3. 网络层 | 网际层 | IP、ICMP、ARP、RARP | 包 (Packet) |
| 2. 数据链路层 | \multirow{2}{*}{网络接口层} | 以太网、Wi-Fi、PPP | 帧 (Frame) |
| 1. 物理层 | | 网线、光纤、集线器 | 比特 (Bit) |

## 3. 每一层的职责详解

### 3.1 应用层

**职责**：为用户的应用程序提供网络服务。

**常见协议**：

| 协议 | 端口 | 用途 | 特点 |
|------|------|------|------|
| HTTP | 80 | 网页浏览 | 无状态、明文 |
| HTTPS | 443 | 安全网页 | HTTP + TLS 加密 |
| DNS | 53 | 域名解析 | UDP 为主，域名→IP |
| FTP | 20/21 | 文件传输 | 两个连接（控制+数据） |
| SMTP | 25 | 发送邮件 | 推模式 |
| POP3 | 110 | 接收邮件 | 拉模式，邮件下载到本地 |
| IMAP | 143 | 接收邮件 | 拉模式，邮件保留在服务器 |
| SSH | 22 | 远程登录 | 加密传输 |
| Telnet | 23 | 远程登录 | 明文，不安全 |
| DHCP | 67/68 | 自动分配 IP | UDP |

**面试角度**：
- DNS 用的是 TCP 还是 UDP？→ 默认 UDP（域名解析数据量小，UDP 快），区域传输用 TCP
- HTTP 和 HTTPS 的区别？→ HTTPS = HTTP + TLS，端口不同，需要证书，加密传输

### 3.2 传输层

**职责**：提供端到端的通信服务（进程到进程），通过端口号区分同一台机器上的不同应用。

**核心协议**：

| 维度 | TCP | UDP |
|------|-----|-----|
| 连接 | 面向连接（三次握手） | 无连接 |
| 可靠性 | 可靠（确认、重传、排序） | 不可靠（尽最大努力交付） |
| 传输方式 | 字节流 | 报文（数据报） |
| 速度 | 较慢 | 快 |
| 头部大小 | 20~60 字节 | 8 字节 |
| 广播/多播 | 不支持 | 支持 |
| 应用场景 | HTTP、FTP、SMTP | DNS、视频流、游戏 |

**端口号**：
```
0~1023：知名端口（需要 root 权限）
  80:HTTP, 443:HTTPS, 22:SSH, 3306:MySQL, 6379:Redis

1024~49151：注册端口
  8080:Tomcat, 3389:RDP

49152~65535：动态/私有端口（客户端临时使用）
```

**面试角度**：
- TCP 为什么是三次握手不是两次？→ 确认双方收发能力、防止历史连接、协商 ISN
- TCP 和 UDP 的区别和使用场景？→ 见上表，关键是 trade-off

### 3.3 网际层

**职责**：负责数据包从源主机到目的主机的路由和转发。

**核心协议**：

| 协议 | 功能 |
|------|------|
| IP | 寻址和路由，将数据包从源地址送到目的地址 |
| ICMP | 网络诊断（ping、traceroute），报告错误 |
| ARP | IP 地址 → MAC 地址（通过广播获取） |
| RARP | MAC 地址 → IP 地址（已淘汰） |
| IGMP | 多播组管理 |

**IP 地址**：
```
IPv4：32 位（4 字节），如 192.168.1.1
  - A 类：1.0.0.0 ~ 126.255.255.255（大型网络）
  - B 类：128.0.0.0 ~ 191.255.255.255（中型网络）
  - C 类：192.0.0.0 ~ 223.255.255.255（小型网络）
  - D 类：224.0.0.0 ~ 239.255.255.255（多播）
  - 私有地址：10.x.x.x / 172.16~31.x.x / 192.168.x.x

IPv6：128 位（16 字节），如 2001:0db8:85a3:0000:0000:8a2e:0370:7334
  - 解决 IPv4 地址耗尽问题
  - 简化头部、内置 IPSec、更好的多播支持
```

### 3.4 网络接口层

**职责**：负责将数据帧发送到物理网络，处理 MAC 地址、帧的封装和解封装。

**核心概念**：

| 概念 | 说明 |
|------|------|
| MAC 地址 | 48 位，全球唯一，固化在网卡中（如 00:1A:2B:3C:4D:5E） |
| 以太网帧 | 前导码(8B) + 目的MAC(6B) + 源MAC(6B) + 类型(2B) + 数据(46~1500B) + FCS(4B) |
| MTU | 最大传输单元，以太网通常 1500 字节 |
| ARP | 通过广播获取目标 IP 对应的 MAC 地址 |

## 4. 数据封装与解封装

数据从上到下发送时，每一层都会加上自己的头部（有时还有尾部）：

```
发送方（封装）：

  应用层数据（HTTP 请求）
  ↓ 加 TCP 头（源端口、目的端口、序列号、确认号……）
  TCP 段（Segment）
  ↓ 加 IP 头（源 IP、目的 IP、TTL、协议类型……）
  IP 包（Packet）
  ↓ 加以太网帧头（目的 MAC、源 MAC）+ 帧尾（FCS 校验）
  以太网帧（Frame）
  ↓ 变成比特流，通过物理介质发送

接收方（解封装）：反过来，逐层去掉头部，最终拿到应用层数据。
```

**MTU 与分片**：
- 以太网 MTU = 1500 字节
- 若 IP 包 > MTU，需要在网络层分片（Fragmentation）
- TCP 通过 MSS（Max Segment Size）协商来避免分片：MSS = MTU - IP头(20) - TCP头(20) = 1460

## 5. 常见面试问题

### Q1: 从浏览器输入 URL 到页面展示，经历了什么？

**答**：
1. **DNS 解析**：域名 → IP 地址（递归/迭代查询）
2. **TCP 连接**：三次握手建立连接
3. **TLS 握手**（HTTPS）：协商加密套件、交换密钥
4. **发送 HTTP 请求**：GET /index.html HTTP/1.1
5. **服务器处理请求**：路由、业务逻辑、查询数据库
6. **返回 HTTP 响应**：200 OK + HTML 内容
7. **浏览器解析渲染**：HTML → DOM Tree、CSS → CSSOM → Render Tree → Layout → Paint
8. **TCP 断开**：四次挥手（或保持 Keep-Alive 长连接）

### Q2: HTTP 和 RPC 的区别？

| 维度 | HTTP | RPC |
|------|------|-----|
| 传输协议 | HTTP/1.1、HTTP/2 | TCP（如 Dubbo）或 HTTP/2（如 gRPC） |
| 序列化 | JSON、XML | Protobuf、Hessian（更高效） |
| 调用方式 | RESTful 风格（资源导向） | 方法调用风格（面向动作） |
| 耦合度 | 松耦合 | 较紧密（需要接口定义） |
| 生态 | 跨语言友好、通用 | 服务间调用效率高 |

### Q3: 在浏览器地址栏输入一个 URL 后按回车，到显示完整页面的过程中，涉及哪些网络协议？

**答**：DNS（UDP）→ ARP → TCP（三次握手）→ TLS（HTTPS 时）→ HTTP → 可能涉及 SPDY/HTTP2 → TCP（四次挥手断开）。

### Q4: 为什么 OSI 七层模型没有被实际使用？

**答**：
1. OSI 模型提出时，TCP/IP 已经被广泛使用（学术界 vs 工业界）
2. OSI 分层过于细碎（表示层、会话层功能在实际中可以合并到应用层）
3. 实现复杂，性能开销大
4. TCP/IP 模型更实用，"够用就好"
