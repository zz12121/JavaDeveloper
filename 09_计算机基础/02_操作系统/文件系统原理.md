# 文件系统原理

> **核心认知**：文件系统是操作系统管理磁盘数据的抽象层。理解 inode、文件描述符、零拷贝等机制，是理解 Java NIO（FileChannel/MappedByteBuffer）、MySQL 存储引擎、Linux 系统编程的基础。

---

## 1. 文件系统概述

### 1.1 什么是文件系统

**文件系统（File System）是操作系统在磁盘上组织和管理数据的方式。**

```
磁盘的层次结构：

磁盘 → 分区（Partition） → 文件系统（ext4/xfs/ntfs） → 文件/目录

一个分区只能有一种文件系统
一个磁盘可以有多个分区
```

### 1.2 常见文件系统

| 文件系统 | 平台 | 特点 |
|---------|------|------|
| ext4 | Linux | 最常用的 Linux 文件系统，日志式 |
| XFS | Linux | 大文件和高并发 IO 性能好 |
| Btrfs | Linux | 支持 CoW、快照、数据校验 |
| NTFS | Windows | Windows 默认文件系统 |
| APFS | macOS | Apple 文件系统，支持快照和加密 |
| tmpfs | Linux | 基于内存的文件系统（/dev/shm） |
| procfs | Linux | 虚拟文件系统（/proc，内核信息） |
| sysfs | Linux | 虚拟文件系统（/sys，设备信息） |

## 2. inode

### 2.1 什么是 inode

**inode（Index Node）是文件系统中存储文件元数据的数据结构。**

```
文件 = 文件名 + inode + 数据块

文件名 → 目录项（directory entry）→ inode 号 → inode → 数据块
```

**inode 中存储的元数据**：

| 属性 | 说明 |
|------|------|
| 文件大小 | 字节数 |
| 权限 | rwxr-xr-x（属主/属组/其他） |
| 所有者 | UID/GID |
| 时间戳 | atime（访问）、mtime（修改）、ctime（inode 变化） |
| 链接数 | 硬链接计数 |
| 数据块指针 | 指向实际数据块的位置 |

### 2.2 inode 的数据块指针

```
inode 中有 15 个数据块指针：

  12 个直接指针 → 直接指向数据块（每个指针指向 4KB 数据块）
   1 个一级间接指针 → 指向一个"指针块"（包含 1024 个指针）→ 数据块
   1 个二级间接指针 → 指向"指针块"→"指针块"→ 数据块
   1 个三级间接指针 → 三级嵌套

文件大小上限（4KB 块大小）：
  直接指针：12 × 4KB = 48KB
  一级间接：1024 × 4KB = 4MB
  二级间接：1024² × 4KB = 4GB
  三级间接：1024³ × 4KB = 4TB
  总计：约 4TB+
```

### 2.3 inode 的限制

```bash
# 查看分区 inode 数量和使用情况
df -i

# inode 耗尽的场景：
# 大量小文件（每个文件消耗一个 inode，即使文件只有 1 字节）
# 此时磁盘空间可能还有剩余，但无法创建新文件

# 查看文件的 inode 号
ls -i file.txt
stat file.txt
```

### 2.4 硬链接 vs 软链接

| 维度 | 硬链接（Hard Link） | 软链接（Symbolic Link） |
|------|-------------------|----------------------|
| 原理 | 指向同一个 inode | 指向文件名的路径 |
| 创建 | `ln source target` | `ln -s source target` |
| 删除源文件 | 不影响（链接数 -1） | 软链接失效（红字） |
| 跨分区 | ❌ 不可以（inode 是分区内的） | ✅ 可以（存的是路径） |
| 目录 | ❌ 不可以 | ✅ 可以 |
| inode 数 | 不增加 | 新建一个 inode |

## 3. 文件描述符

### 3.1 什么是文件描述符

**文件描述符（File Descriptor, fd）是进程打开文件的抽象标识，是一个非负整数。**

```
每个进程维护一个"文件描述符表"：

  fd 0 → stdin  （标准输入）
  fd 1 → stdout （标准输出）
  fd 2 → stderr （标准错误）
  fd 3 → /path/to/file.log
  fd 4 → socket（网络连接）
  ...

  Linux 默认限制：每个进程最多 1024 个文件描述符
  可通过 ulimit -n 65535 调整
```

### 3.2 文件描述符与 Java 的关系

```java
// Java 的 FileDescriptor 就是 Linux 的 fd
// FileInputStream/FileOutputStream/FileChannel 底层都持有 fd

// Netty 底层通过 FileDescriptor 操作
FileChannel.open(Paths.get("file.txt"), StandardOpenOption.READ);

// TCP 连接也是 fd
Socket socket = serverSocket.accept();  // 返回新的 fd
```

### 3.3 文件描述符限制

```bash
# 查看当前进程的 fd 限制
ulimit -n

# 查看系统级别的 fd 限制
cat /proc/sys/fs/file-max

# 查看某个进程打开了多少 fd
ls /proc/<pid>/fd | wc -l

# 查看某个进程打开了哪些文件
ls -la /proc/<pid>/fd

# 线上问题：Too many open files
# 原因：文件描述符耗尽
# 解决：增大 ulimit -n，检查代码是否正确关闭文件/Socket
```

## 4. 文件 IO 过程

### 4.1 标准 IO 流程

```
应用程序调用 read(fd, buffer, size)

1. 用户态 → 内核态（系统调用）
2. 内核检查页缓存（Page Cache）
   ├── 命中 → 直接从页缓存拷贝到用户空间
   └── 未命中 → 从磁盘读取到页缓存 → 拷贝到用户空间
3. 内核态 → 用户态（返回）

标准写入流程：
  write(fd, buffer, size)
  → 数据先写入页缓存（异步刷盘）
  → 后台线程定期刷盘（fsync 强制刷盘）
```

### 4.2 Page Cache（页缓存）

**Linux 将空闲的物理内存作为磁盘缓存（Page Cache）**，减少磁盘 IO。

```
读取流程：
  1. 先查 Page Cache
  2. 命中 → 直接返回（内存读取，纳秒级）
  3. 未命中 → 从磁盘读取到 Page Cache → 返回

写入流程：
  1. 先写入 Page Cache（立即返回）
  2. 内核定期将脏页刷回磁盘
  3. 可调用 fsync() 强制刷盘

为什么 Java 的 FileChannel 写入后需要 force(true)？
  → 因为数据可能还在 Page Cache 中，没写入磁盘
  → force(true) 会调用 fsync() 确保刷盘
```

### 4.3 Buffer IO vs Direct IO

| 维度 | Buffer IO（标准 IO） | Direct IO |
|------|---------------------|-----------|
| Page Cache | 经过 | 绕过 |
| 性能 | 重复读取快（缓存命中） | 适合一次性读取大文件 |
| 数据一致性 | 需要手动 fsync | 写完即落盘 |
| 使用场景 | 普通文件读写 | 数据库（自己管理缓存） |

```
MySQL InnoDB 使用 Direct IO：
  → InnoDB 有自己的 Buffer Pool 缓存机制
  → 不需要操作系统的 Page Cache
  → 使用 O_DIRECT 标志绕过页缓存
```

## 5. 零拷贝

### 5.1 传统 IO 的拷贝过程

```
读取文件并发送到网络（如静态文件服务）：

  磁盘 → 内核页缓存（DMA 拷贝）→ 用户缓冲区（CPU 拷贝）
  → Socket 缓冲区（CPU 拷贝）→ 网卡（DMA 拷贝）

  共 4 次拷贝（2 次 DMA + 2 次 CPU）+ 4 次上下文切换
```

### 5.2 mmap（内存映射）

```
mmap 将文件映射到用户空间的虚拟内存：

  磁盘 → 内核页缓存（DMA）→ 用户空间虚拟内存（共享同一物理页）

  读取：直接访问用户空间虚拟内存（零 CPU 拷贝）
  发送：write() 将页缓存数据发送到 Socket 缓冲区（1 次 CPU 拷贝）

  共 3 次拷贝（2 DMA + 1 CPU）+ 4 次上下文切换

Java 中：
  MappedByteBuffer buffer = channel.map(MapMode.READ_ONLY, 0, size);
  // buffer 直接指向文件在内存中的映射，read() 不需要拷贝
```

### 5.3 sendfile

```
sendfile 完全在内核态完成数据传输：

  磁盘 → 内核页缓存（DMA）→ Socket 缓冲区（CPU 拷贝）→ 网卡（DMA）

  共 3 次拷贝（2 DMA + 1 CPU）+ 2 次上下文切换
  （不需要用户态参与）

Java 中：
  FileChannel.transferTo(0, size, destChannel);
  // 底层调用 sendfile 系统调用
```

### 5.4 sendfile + DMA Scatter/Gather

```
进一步优化：CPU 不参与拷贝，只传递描述信息：

  磁盘 → 内核页缓存（DMA）
  → DMA 控制器直接从页缓存读取描述信息 → 网卡

  共 2 次拷贝（都是 DMA）+ 2 次上下文切换
  CPU 零参与！

  这是终极的"零拷贝"——CPU 不参与任何数据搬运。
```

## 6. 文件系统与 Java 的关系

| 操作系统概念 | Java 对应 |
|-------------|---------|
| 文件描述符（fd） | FileDescriptor / FileChannel |
| Page Cache | FileChannel.read() 默认经过页缓存 |
| mmap | MappedByteBuffer |
| sendfile | FileChannel.transferTo() |
| Direct IO | O_DIRECT（未直接暴露，通过 JNI 或 Netty 使用） |
| inode | Files.getAttribute()（获取文件元数据） |
| 硬链接/软链接 | Files.createLink() / Files.createSymbolicLink() |
| 文件锁 | FileChannel.lock() / FileLock |

```java
// Java NIO 中的零拷贝示例
try (FileChannel source = FileChannel.open(Paths.get("big.txt"));
     WritableByteChannel dest = Channels.newChannel(socket.getOutputStream())) {
    // transferTo 底层调用 sendfile
    long transferred = source.transferTo(0, source.size(), dest);
}

// MappedByteBuffer 示例
try (FileChannel channel = FileChannel.open(Paths.get("data.bin"),
     StandardOpenOption.READ, StandardOpenOption.WRITE)) {
    MappedByteBuffer buffer = channel.map(MapMode.READ_WRITE, 0, 1024);
    buffer.putInt(0, 42);  // 直接写入文件（经过页缓存）
}
```

## 7. 面试高频问题

### Q1: inode 是什么？inode 耗尽怎么解决？

**答**：inode 是文件系统中存储文件元数据（大小、权限、时间戳、数据块指针）的数据结构，每个文件对应一个 inode。大量小文件会导致 inode 耗尽（磁盘空间还有但无法创建新文件）。解决：删除无用的小文件、使用支持更多 inode 的文件系统格式、或使用 noatime 减少 inode 写入。

### Q2: 什么是零拷贝？Java 中如何使用？

**答**：零拷贝减少数据在内核空间和用户空间之间的拷贝次数。Java 中 `FileChannel.transferTo()` 底层调用 sendfile，数据不经过用户空间；`MappedByteBuffer` 底层调用 mmap，将文件映射到用户虚拟内存。Netty 的 FileRegion 也封装了零拷贝。适用于大文件传输（静态资源服务、日志传输）。

### Q3: 为什么数据库要用 Direct IO 而不是 Page Cache？

**答**：数据库（如 MySQL InnoDB）有自己的 Buffer Pool 缓存机制。如果再经过操作系统的 Page Cache，同一份数据会被缓存两次（Buffer Pool + Page Cache），浪费内存。使用 Direct IO 绕过 Page Cache，让数据库完全自主管理缓存策略。Redis 也类似，使用 Direct IO 写入 AOF 文件。

### Q4: 什么是 Page Cache？对 Java 应用有什么影响？

**答**：Page Cache 是操作系统利用空闲物理内存缓存磁盘数据的机制。Java 的 FileInputStream/FileChannel 默认使用 Buffer IO，数据经过 Page Cache。重复读取同一文件时直接从 Page Cache 返回（内存速度），非常快。但 JVM 关闭后 Page Cache 失效，下次启动需要重新从磁盘加载。MySQL 使用 O_DIRECT 绕过 Page Cache。
