# 锁机制

> synchronized 是 JVM 内置的锁，ReentrantLock 是 JDK 提供的显式锁。两者解决同一问题（互斥），但实现路径完全不同——理解它们的差异，才能在正确场景选对工具。

---

## 这个问题为什么存在？

多线程同时读写共享变量会产生竞态条件，结果依赖线程执行时序，不可预测。需要一个机制保证「同一时刻只有一个线程能执行某段代码」。

synchronized 是 Java 最原始的互斥方案，语法简洁但功能有限。JDK 5 引入 JUC（java.util.concurrent）后，出现了更灵活的 Lock 体系。两者并存至今，各有适用场景。

---

## 核心脉络

### synchronized：JVM 内置锁

synchronized 的锁信息存在 **Java 对象头（Mark Word）** 里，根据竞争程度自动升级：

```
无锁（001）
  ↓ 第一个线程进入（无竞争）
偏向锁（101）   ← 只记录线程 ID，无 CAS 开销
  ↓ 第二个线程尝试获取
轻量级锁（00）  ← CAS + 自旋，用户态解决
  ↓ 竞争激烈（自旋超时 / 线程数超 CPU 核数）
重量级锁（10）  ← 操作系统 mutex，线程阻塞（内核态）
```

synchronized 对程序员透明——不需要手动释放，锁升级/降级由 JVM 全权处理。详解见 [[02_并发编程/03_锁机制/synchronized详解|synchronized 详解]]。

### Lock：JDK 显式锁

`java.util.concurrent.locks.Lock` 接口提供了比 synchronized 更灵活的控制：

```
Lock 接口核心方法：
  lock()               — 阻塞获取，不可中断
  lockInterruptibly()  — 阻塞获取，响应中断
  tryLock()            — 非阻塞尝试获取
  tryLock(timeout)     — 超时获取
  unlock()             — 手动释放（必须在 finally 中）
  newCondition()       — 创建条件变量（可多个）
```

核心实现是 `ReentrantLock`（基于 AQS），详解见 [[02_并发编程/03_锁机制/Lock详解|Lock 详解]]。

### 两者的本质区别

| 维度 | synchronized | Lock（ReentrantLock） |
|------|---------------|------------------------|
| 用法 | 关键字，自动释放 | 接口，手动 `unlock()` |
| 中断响应 | ✗ 拿不到就死等 | ✓ `lockInterruptibly()` |
| 超时获取 | ✗ | ✓ `tryLock(timeout)` |
| 公平锁 | ✗ 只支持非公平 | ✓ 构造参数可选 |
| 条件变量 | 1 个（wait/notify） | 多个（Condition） |
| 实现层 | JVM（C++） | JDK（Java + AQS） |
| 锁升级 | ✓ 偏向→轻量→重量 | ✗ 无升级概念 |
