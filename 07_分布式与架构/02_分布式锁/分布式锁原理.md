# 分布式锁原理

## 这个问题为什么存在？

> 单机锁（synchronized、ReentrantLock）只管一个JVM里的线程。但分布式环境下，**多个服务实例同时修改同一资源**，单机锁完全失效。问题是：**怎么在分布式环境下实现互斥？**

没有分布式锁，你会遇到：
- 两个订单服务同时扣同一商品的库存 → 超卖
- 定时任务在多台机器上同时触发 → 重复执行
- 分布式ID生成器，两台机器拿到相同的workerId → ID冲突

---

## 它是怎么解决问题的？

### 分布式锁的四大核心要求

```
1. 互斥性（Mutual Exclusion）
   → 同一时刻，只有一个客户端能持有锁

2. 不死锁（Deadlock Free）
   → 持有锁的客户端挂了，锁不能永远不释放（需要超时机制）

3. 可重入（Reentrant）
   → 同一线程可以多次获取同一把锁（避免自己死锁自己）

4. 高性能（High Performance）
   → 加锁、释放锁的延迟要低，支持高并发
```

### 分布式锁的三种实现方案

```
方案一：Redis  — 性能最高，但极端场景有bug
方案二：ZooKeeper — 可靠性最高，但性能差
方案三：数据库   — 实现最简单，但性能最差
```

---

## 它和相似方案的本质区别是什么？

| | Redis分布式锁 | ZooKeeper分布式锁 | 数据库分布式锁 |
|---|---|---|---|
| 性能 | 极高（内存操作） | 中（磁盘+网络） | 低（磁盘IO） |
| 可靠性 | 有bug（主从切换时可能丢失锁） | 高（ZAB协议保证） | 中（依赖DB可用性） |
| 实现复杂度 | 中（需要Lua脚本） | 高（需要临时顺序节点） | 低（唯一约束） |
| 超时释放 | ✅ 靠TTL | ✅ 靠会话超时 | ✅ 靠超时记录 |
| watchdog | ✅ Redisson支持 | ❌ 不需要（会话机制） | ❌ 需要自己实现 |
| 适用场景 | 高并发，锁冲突不频繁 | 一致性要求高 | 并发量低，简单场景 |

**本质区别**：
- **Redis锁**：AP系统，性能优先，牺牲强一致性（主从切换时可能丢失锁）
- **ZK锁**：CP系统，一致性优先，性能差但可靠
- **DB锁**：最简单的实现，但性能瓶颈明显

## 正确使用方式

### 通用原则：永远设超时

```java
// ❌ 错误：加锁不设超时，客户端挂了锁永远不释放
redis.set("lock:order:123", "me");

// ✅ 正确：加锁必须设超时（TTL）
redis.set("lock:order:123", "me", "NX", "EX", 30);  // 30秒超时
```

### 通用原则：释放锁要验证身份

```java
// ❌ 错误：直接删除，可能删除别人的锁
redis.del("lock:order:123");

// ✅ 正确：用Lua脚本，先验证value再删除
String lua = """
    if redis.call('get', KEYS[1]) == ARGV[1] then
        return redis.call('del', KEYS[1])
    else
        return 0
    end
    """;
redis.eval(lua, Collections.singletonList("lock:order:123"), "me");
```

### 通用原则：超时时间要大于业务执行时间

```
错误估算：业务执行10秒，锁超时设5秒 → 业务还没执行完，锁就被别人抢走了
正确估算：锁超时 > 业务执行时间 × 2（留有余地）
```

## 边界情况和坑

### 坑1：锁超时，但业务没执行完

```
场景：
  T0: 客户端A加锁成功，超时30秒
  T20: 客户端A业务还在执行（网络慢、GC停顿）
  T30: 锁超时，Redis自动删除
  T31: 客户端B加锁成功
  T32: 客户端A业务执行完，删除锁 → 把客户端B的锁删了！
```

**解决方案**：
1. **watchdog机制**（Redisson）：后台线程每隔一段时间（如10秒）续期一次
2. **业务幂等**：就算锁失效，业务也要能容忍重复执行

### 坑2：Redis主从切换导致锁丢失

```
场景：
  客户端A在master节点加锁成功
  master还没同步给slave，就挂了
  slave晋升为新master，客户端B在新master上加锁成功
  结果：客户端A和B同时持有锁！
```

**解决方案**：
1. **Redlock算法**（不推荐，争议大，性能差）
2. **ZooKeeper锁**（CP系统，强一致，不存在这个问题）
3. **接受风险**（大部分业务可以容忍，锁的是"偶尔重复"）

### 坑3：GC停顿导致锁超时

```
场景：
  客户端A加锁成功，超时30秒
  发生了Full GC，停顿了40秒
  GC结束后，锁已经超时被删除了
  客户端A恢复后，还以为自己持有锁
```

**解决方案**：
1. **减少Full GC频率**（堆内存调优、G1/ZGC）
2. **watchdog续期**（GC期间也续期，但需要GC停顿时间短于续期间隔）
3. **业务幂等**（最终兜底）
