# 配置管理

## 这个问题为什么存在？

> 一个应用要在「开发环境」「测试环境」「生产环境」跑，数据库地址、Redis 地址、日志级别... 每个环境都不一样。  
> 如果没有配置管理，每次换环境都要改代码、重新打包 → 不可接受。

**配置管理的本质**：把**会变的东西**从代码里抽出来，放到外部，按环境切换。

---

## 它是怎么解决问题的？

### 配置文件加载顺序（优先级从低到高）

Spring Boot 会按以下顺序加载配置，**后面的覆盖前面的**：

```
1. 默认配置（application.yml，打包在 Jar 里）
      ↓ 被覆盖
2. 打包在 Jar 内的 profile 配置（application-{profile}.yml）
      ↓ 被覆盖
3. 打包在 Jar 外的默认配置（classpath:/config/application.yml）
      ↓ 被覆盖
4. 打包在 Jar 外的 profile 配置（classpath:/config/application-{profile}.yml）
      ↓ 被覆盖
5. 操作系统环境变量
      ↓ 被覆盖
6. Java 系统属性（-D 参数）
      ↓ 被覆盖
7. 命令行参数（--key=value）
```

**结论**：**命令行参数优先级最高**，可以覆盖所有配置文件。

---

### 配置文件的格式

**YAML（推荐）**：层级清晰，支持复杂结构。

```yaml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/test
    username: root
    password: 123456
    hikari:
      maximum-pool-size: 10

myapp:
  features:
    - feature-a
    - feature-b
```

**Properties（传统）**：简单，但层级深时不直观。

```properties
spring.datasource.url=jdbc:mysql://localhost:3306/test
spring.datasource.hikari.maximum-pool-size=10
myapp.features[0]=feature-a
myapp.features[1]=feature-b
```

---

### 配置属性的绑定（@ConfigurationProperties）

**问题**：配置文件中是 `kebab-case`（`max-pool-size`），Java 类是 `camelCase`（`maxPoolSize`），怎么映射？

```java
// 配置文件
spring.datasource.hikari.maximum-pool-size=10

// Java 类
@ConfigurationProperties(prefix = "spring.datasource.hikari")
public class HikariConfig {
    private int maximumPoolSize;  // ← 自动映射（kebab-case → camelCase）

    // getter / setter
}
```

**宽松绑定规则**：

| 配置文件中的写法 | Java 字段 | 是否匹配 |
|------------------|-----------|----------|
| `max-pool-size` | `maxPoolSize` | ✅ |
| `max_pool_size` | `maxPoolSize` | ✅ |
| `maxPoolSize` | `maxPoolSize` | ✅ |
| `MAX_POOL_SIZE` | `maxPoolSize` | ✅ |

### 源码路径：PropertySources 加载

```java
// ConfigFileApplicationListener（Spring Boot 2.4 之前）
// 新版本改为 ConfigDataEnvironmentPostProcessor

protected void addPropertySources(ConfigurableEnvironment environment,
                                 ResourceLoader resourceLoader) {
    // 1. 加载 application.properties / application.yml
    //    （从 classpath:/, classpath:/config/, file:./, file:./config/）
    load(environment, resourceLoader);

    // 2. 加载 profile 对应的配置文件
    //    application-{profile}.properties / application-{profile}.yml
    loadForProfile(environment, resourceLoader, profile);
}
```

> 配置文件的加载是在 **`prepareEnvironment()` 阶段**完成的，早于 Bean 的创建。  
> 所以 `@Value` 和 `@ConfigurationProperties` 可以正常读取配置。

## 它和相似方案的本质区别是什么？

### @Value vs @ConfigurationProperties

| | @Value | @ConfigurationProperties |
|--|--------|--------------------------|
| 绑定方式 | 单个属性 | 批量绑定（前缀 + 字段名） |
| 宽松绑定 | ❌ 不支持 | ✅ 支持 |
| 元数据支持 | ❌ | ✅（IDE 自动补全） |
| 校验（@Validated） | ❌ | ✅ |
| 适用场景 | 单个简单属性 | 一组相关属性（如数据源配置） |

```java
// @Value：适合单个属性
@Value("${myapp.timeout:5000}")  // 有默认值
private int timeout;

// @ConfigurationProperties：适合一组属性
@ConfigurationProperties(prefix = "myapp.datasource")
public class DataSourceConfig {
    private String url;
    private String username;
    private String password;
}
```

---

### Profile vs 外部化配置

| | Profile | 外部化配置 |
|--|---------|-------------|
| 目的 | 区分环境（dev/test/prod） | 不重新打包就能改配置 |
| 实现 | `application-{profile}.yml` | 命令行参数 / 外部配置文件 |
| 使用场景 | 环境差异大的配置（数据库地址） | 运维调参（JVM 参数、端口号） |

**最佳实践**：

```yaml
# application.yml（通用配置）
spring:
  profiles:
    active: @activatedProperties@  # Maven 替换

# application-dev.yml（开发环境）
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/dev

# application-prod.yml（生产环境）
spring:
  datasource:
    url: jdbc:mysql://prod-db:3306/prod
```

运行时指定 Profile：

```bash
java -jar app.jar --spring.profiles.active=prod
```

## 正确使用方式

### 1. 类型安全的配置属性

```java
@ConfigurationProperties(prefix = "myapp")
@Validated
public class MyAppProperties {

    @NotNull  // 校验：这个属性必须配置
    private String appKey;

    @Min(1)
    @Max(100)
    private int retryCount = 3;  // 默认值

    // getter / setter
}
```

### 2. 多环境配置

```yaml
# application.yml
spring:
  profiles:
    active: dev  # 默认激活 dev

---
spring:
  profiles: dev
myapp:
  datasource:
    url: jdbc:mysql://localhost:3306/dev

---
spring:
  profiles: prod
myapp:
  datasource:
    url: jdbc:mysql://prod-db:3306/prod
```

> 用 `---` 分隔符可以在**一个文件里写多个 Profile**。

### 3. 命令行覆盖配置

```bash
# 覆盖配置文件中的属性
java -jar app.jar --server.port=8081 --spring.datasource.url=jdbc:mysql://newhost:3306/db
```

## 边界情况和坑

### 1. YAML 的陷阱：同名 Key 覆盖

```yaml
# ❌ YAML 中同名 Key 会被后面的值覆盖
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/db1
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/db2  # 前面的被覆盖！
```

> YAML 规范允许同名 Key，但**后面的覆盖前面的**。  
> 解决：不要重复写同名顶层 Key。

---

### 2. 配置文件的加载位置

```
file:./config/           ← 最高优先级（Jar 所在目录的 config/）
file:./
classpath:/config/
classpath:/              ← 最低优先级
```

> 如果把配置文件放在 **Jar 同级的 `config/` 目录**，可以不用重新打包就能改配置。  
> 这是生产环境的最佳实践。

---

### 3. @Value 的 ${} 和 #{} 区别

```java
// ${}：取配置属性
@Value("${myapp.name}")        // 从配置文件读取
private String name;

// #{}：SpEL 表达式
@Value("#{systemProperties['user.home']}")  // 取系统属性
private String userHome;

@Value("#{${myapp.max}-10}")  // SpEL 运算
private int calculated;
```

---

### 4. 配置属性不生效

```java
// ❌ 忘记加 @EnableConfigurationProperties
@Configuration
public class MyConfig {
    @Bean
    public MyService myService(
        @ConfigurationProperties(prefix = "myapp") DataSource ds) {
        return new MyService(ds);
    }
}
```

**解决**：

```java
// ✅ 方式1：@Component + @ConfigurationProperties（推荐）
@Component
@ConfigurationProperties(prefix = "myapp")
public class MyProperties { ... }

// ✅ 方式2：@EnableConfigurationProperties
@Configuration
@EnableConfigurationProperties(MyProperties.class)
public class MyConfig { ... }
```
