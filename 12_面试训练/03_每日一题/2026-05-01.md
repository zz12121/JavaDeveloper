## 参考答案

### 一、缓存穿透（Cache Penetration）

**定义**：查询一个**数据库中根本不存在**的数据，缓存永远不会命中，请求每次都透传到 DB。

```
请求 key="-1"（不存在）
  ↓ 查 Redis → miss（不存在）
  ↓ 查 DB   → 也不存在
  ↓ 不写缓存（因为没有数据）
  ↓ 下次同样的 key → 再次穿透 DB
```

**攻击场景**：恶意用户用大量不存在的 id 疯狂请求，绕过缓存直打 DB。

**解决方案**：

| 方案 | 原理 | 适用场景 |
|------|------|----------|
| **缓存空值** | 查到 null 时，缓存一个空值（TTL 短，如 60s） | key 种类有限、攻击量不大 |
| **布隆过滤器** | 请求前先过布隆过滤器，不存在的 key 直接拦截 | 数据量大、恶意攻击场景 |
| 接口层限流/鉴权 | 从入口拦截非法请求 | 有业务语义的 id 格式校验 |

**布隆过滤器的坑**：

```
优点：内存极小（bitmap），查询 O(1)
缺点：有误判率（存在的 key 可能被误判为不存在 → 不会发生；不存在的 key 可能误判为存在 → 仍会穿透，但概率可调）
     不支持删除（解决：Counting Bloom Filter）
     需要预热（初始化时把所有合法 key 写入）
```

### 二、缓存击穿（Cache Breakdown / Hotspot Invalid）

**定义**：某个**热点 key 在缓存中过期**的瞬间，大量并发请求同时打到 DB，造成 DB 瞬间压力暴增。

```
热点 key 过期
  ↓ 1000 个并发请求同时到达
  ↓ 全部 miss
  ↓ 全部去查 DB（1000 个并发查询！）
  ↓ DB 撑不住 → 崩溃
```

**解决方案**：

#### 方案 1：互斥锁（Mutex Lock）

```java
// 缓存击穿解决：分布式锁
String value = redis.get(key);
if (value == null) {
    if (redis.setnx("lock:" + key, "1", 3, TimeUnit.SECONDS)) {
        try {
            value = db.query(key);
            redis.set(key, value, 30, TimeUnit.MINUTES);
        } finally {
            redis.del("lock:" + key);
        }
    } else {
        Thread.sleep(50);
        return getCacheValue(key);  // 重试
    }
}
return value;
```

**优点**：DB 只有一个请求，保证一致性
**缺点**：性能下降（其他线程等待），有死锁风险（锁要设超时）

#### 方案 2：逻辑过期（不设物理过期时间）

```java
// value 里包含逻辑过期时间
{"data": {...}, "expire": "2026-05-01 10:00:00"}

// 读取时：
if (逻辑时间已过期) {
    异步线程去更新缓存（互斥锁保证只有一个线程更新）
    return 旧数据（接受短暂数据不一致）
}
```

**优点**：高可用，不会阻塞请求
**缺点**：短时间内返回旧数据（接受最终一致性才适用）

### 三、缓存雪崩（Cache Avalanche）

**定义**：**大量 key 在同一时刻集中失效**（或 Redis 宕机），导致大量请求同时打到 DB，造成 DB 雪崩。

```
22:00 整点活动结束 → 10万个 key 全部 TTL=1h → 23:00 同时过期
  ↓ 大量请求 miss
  ↓ 全部打到 DB
  ↓ DB CPU 飙升 → 请求堆积 → 超时 → 服务雪崩
```

**解决方案**：

| 方案 | 原理 | 说明 |
|------|------|------|
| **随机 TTL 打散** | `TTL = 基准时间 + random(0, N)` | 最简单有效，错峰过期 |
| **多级缓存** | 本地缓存（Caffeine）+ Redis 双层 | 即使 Redis 层失效，本地缓存兜底 |
| **Redis 高可用** | 主从 + Sentinel / Cluster | 避免 Redis 宕机导致的完全雪崩 |
| **限流 + 降级** | 熔断器（Sentinel/Hystrix）兜底 | DB 压力过大时主动降级，保核心链路 |
| **缓存预热** | 启动时提前加载热点数据 | 避免冷启动雪崩 |

---

### 四、三者对比总结

| 问题 | 本质 | 关键词 | 核心解法 |
|------|------|--------|----------|
| **穿透** | 查不存在的数据 | null、恶意请求、数据库也没有 | 布隆过滤器 / 缓存空值 |
| **击穿** | 热点 key 过期 | 热点、单个 key、瞬间高并发 | 互斥锁 / 逻辑过期 |
| **雪崩** | 大量 key 同时过期 | 大量、集中过期、Redis 宕机 | 随机 TTL / 多级缓存 / 限流 |

**记忆口诀**：
```
穿透 → 查的根本没有 → 过滤/空值挡在前
击穿 → 热点 key 过期 → 锁/不过期守住门
雪崩 → 集体过期/宕机 → 打散/多层/限流保平安
```
