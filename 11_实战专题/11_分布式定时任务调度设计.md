# 分布式定时任务调度平台设计

> 公司内部跑批任务散落各服务，没有统一调度，出问题靠猜。需要一个平台统一管理：定时触发、分片执行、故障转移、重试告警、可视化监控。

---

## 业务背景

```
现状痛点：
  各服务各自写 @Scheduled / cron
  → 不知道有哪些任务在跑
  → 任务失败没人知道
  → 多实例部署重复执行
  → 无法动态调整执行时间
  → 任务之间有依赖关系（先对账，再清算，再出报表），手动控制顺序

需求：
  - 统一管理：所有任务在平台上注册、配置、启停
  - 分片执行：大任务分片并行，缩短总耗时（如对账 100 万条数据分 10 片并行处理）
  - 故障转移：执行节点挂了，自动转移到其他节点
  - 失败重试：失败自动重试（指数退避），超过阈值告警
  - 监控告警：任务执行耗时、成功/失败率、可视化日志
  - 动态调参：不用重启服务就能改 cron 表达式
```

## 方案对比

| 方案 | 分片 | 故障转移 | 重试 | 监控 | 动态配置 | 复杂度 | 推荐场景 |
|------|:---:|:---:|:---:|:---:|:---:|:---:|---------|
| Spring @Scheduled | 不支持 | 不支持 | 不支持 | 无 | 需重启 | 低 | 单机、简单定时 |
| Quartz 集群 | 支持（DB 分片） | 支持（DB 抢锁） | 支持 | 基础 | 需重启 | 中 | 中小规模，DB 可靠 |
| XXL-JOB | 支持 | 支持 | 支持 | Web 控制台 | 支持 | 中 | **推荐，国内最主流** |
| ElasticJob | 支持（弹性分片） | 支持 | 支持 | 支持 | 支持 | 中高 | 大数据量、弹性扩缩容 |
| Airflow | DAG 支持 | 支持 | 支持 | 完善 | 支持 | 高 | 数据管道、复杂依赖 |

**生产推荐**：XXL-JOB（中小团队首选，生态成熟，上手快）或 ElasticJob（大数据量场景，弹性分片更强）。

## 核心设计：XXL-JOB 架构

### 整体架构

```
                     ┌─────────────────────┐
                     │    XXL-JOB Admin     │
                     │   （调度中心）        │
                     │                      │
                     │  ┌────────────────┐  │
                     │  │  DB（MySQL）    │  │
                     │  │  任务配置表     │  │
                     │  │  执行日志表     │  │
                     │  │  注册节点表     │  │
                     │  └────────────────┘  │
                     │                      │
                     │  调度线程池          │
                     │  时间轮触发          │
                     │  路由策略            │
                     └──────────┬───────────┘
                                │
                      触发任务执行（HTTP）
                                │
              ┌─────────────────┼─────────────────┐
              │                 │                 │
              ▼                 ▼                 ▼
       ┌───────────┐    ┌───────────┐    ┌───────────┐
       │  Executor  │    │  Executor  │    │  Executor  │
       │  (业务A)   │    │  (业务B)   │    │  (业务C)   │
       │            │    │            │    │            │
       │  对账任务  │    │  报表任务  │    │  清算任务  │
       │  库存同步  │    │  数据导出  │    │  风控扫描  │
       └───────────┘    └───────────┘    └───────────┘
       （自动注册到 Admin）
```

### 核心流程

```
1. 任务注册
   业务服务启动时，Executor 自动注册到 Admin
   Admin 记录执行器地址列表

2. 触发调度
   Admin 调度线程池扫描任务表
   匹配到触发时间的任务 → 按路由策略选一个 Executor
   通过 HTTP 调用 Executor 的 run 方法

3. 执行任务
   Executor 收到请求 → 在业务线程池中执行 JobHandler
   执行过程中定期向 Admin 回调进度

4. 结果处理
   执行成功 → 记录成功日志
   执行失败 → 根据重试策略自动重试
   超过重试次数 → 告警（邮件/钉钉/企微）
```

### 分片广播设计

```
场景：对账任务需要处理 1000 万条交易记录
     → 单机处理需要 2 小时
     → 分 10 片并行处理，20 分钟搞定

分片流程：
┌─────────────┐
│  XXL-JOB    │  触发时携带分片参数
│  Admin      │  total=10, index=0~9
└──────┬──────┘
       │
   ┌───┼───────────────────────┐
   │   │                       │
   ▼   ▼                       ▼
┌────────┐ ┌────────┐    ┌────────┐
│节点 A  │ │节点 B  │    │节点 C  │
│分片 0  │ │分片 1  │    │分片 2  │
│分片 3  │ │分片 4  │    │分片 5  │
│分片 6  │ │分片 7  │    │分片 8  │
│分片 9  │ │        │    │        │
└────────┘ └────────┘    └────────┘

Admin 向所有在线的 Executor 广播触发请求
每个 Executor 拿到不同的分片序号
各节点只处理自己负责的分片
```

```java
@XxlJob("reconcileJobHandler")
public void reconcileJobHandler() throws Exception {
    // 获取分片参数
    ShardingUtil.ShardingVO sharding = ShardingUtil.getShardingVo();
    int total = sharding.getTotal();    // 总分片数
    int index = sharding.getIndex();    // 当前分片序号

    // 查询时带上分片条件
    List<TradeRecord> records = tradeMapper.selectForReconcile(
        new Date(),        // 对账日期
        index,             // 分片序号
        total              // 总分片数
    );

    // SQL 中的分片逻辑：
    // SELECT * FROM trade_record
    // WHERE reconcile_date = #{date}
    //   AND MOD(id, #{total}) = #{index}
    // LIMIT 10000;

    log.info("分片 [{}/{}] 开始处理，共 {} 条记录", index, total, records.size());

    for (TradeRecord record : records) {
        reconcileService.process(record);
    }

    log.info("分片 [{}/{}] 处理完成", index, total);
}
```

### 故障转移与路由策略

```
故障转移场景：
  Admin 触发任务 → 选择 Executor-A 执行
  Executor-A 挂了（HTTP 调用超时 / 连接拒绝）
  → Admin 自动切换到 Executor-B 重新触发
  → 用户无感知

路由策略：
┌──────────────────────────────────────────────┐
│ 路由策略              │ 说明                   │
├───────────────────────┼───────────────────────┤
│ 第一个（FIRST）        │ 固定选第一个在线节点    │
│ 最后一个（LAST）       │ 固定选最后一个在线节点  │
│ 轮询（ROUND）         │ 按顺序轮流分配          │
│ 随机（RANDOM）        │ 随机选一个节点          │
│ 一致性哈希（CONSISTENT）│ 相同参数路由到同一节点  │
│ LFU（最不经常使用）    │ 优先选执行次数最少的    │
│ LRU（最近最久未使用）  │ 优先选最久没执行的      │
│ 故障转移（FAILOVER）  │ 按 LFU 路由，失败自动切换│
│ 忙碌转移（BUSYFAILOVER）│ 忽略忙碌节点，失败切换  │
│ 分片广播（SHARDING）  │ 广播到所有在线节点      │
└──────────────────────────────────────────────┘
```

### 失败重试与告警

```java
@XxlJob("reportJobHandler")
public void reportJobHandler() throws Exception {
    // XXL-JOB 内置重试机制（在 Admin 配置）：
    // - 失败重试次数：3 次
    // - 失败告警邮件：xxx@company.com

    // 业务层面的重试：
    int maxRetries = 3;
    for (int i = 0; i < maxRetries; i++) {
        try {
            reportService.generateDailyReport();
            return;  // 成功直接返回
        } catch (RetryableException e) {
            log.warn("报表生成失败，第 {} 次重试: {}", i + 1, e.getMessage());
            if (i == maxRetries - 1) {
                // 最终失败，抛出异常触发 XXL-JOB 告警
                throw e;
            }
            Thread.sleep(calculateBackoff(i));  // 指数退避
        }
    }
}

private long calculateBackoff(int retryCount) {
    // 1s, 2s, 4s
    return (long) Math.pow(2, retryCount) * 1000;
}
```

```
告警配置（Admin 控制台）：
  - 任务失败告警 → 邮件 / 钉钉群 / 企微群
  - 超时告警 → 任务执行超过 N 分钟
  - 执行结果统计 → 每日汇总报告
```

## 任务间依赖：DAG 编排

```
场景：每日结算流程
  1. 订单对账（依赖：无）
  2. 支付对账（依赖：无）
  3. 佣金计算（依赖：订单对账 + 支付对账）
  4. 供应商结算（依赖：佣金计算）
  5. 日报表生成（依赖：供应商结算）

DAG 图：
  订单对账 ──┐
             ├──→ 佣金计算 ──→ 供应商结算 ──→ 日报表生成
  支付对账 ──┘

实现方式：
  方案1：XXL-JOB 子任务（XXL-JOB 支持在 JobHandler 里触发另一个任务）
  方案2：Airflow / DolphinScheduler（专业 DAG 调度平台）
  方案3：自研编排（每个任务完成后发 MQ，下游任务监听执行）
```

```java
// 方案1：XXL-JOB 子任务依赖
@XxlJob("commissionCalcHandler")
public void commissionCalcHandler() {
    // 1. 执行佣金计算
    commissionService.dailyCalc();

    // 2. 触发下游任务：供应商结算
    XxlJobHelper.triggerChildJob("settlementJobHandler");
}
```

```java
// 方案3：基于 MQ 的任务编排
@Component
public class TaskOrchestrator {

    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    // 订单对账完成后发送事件
    public void onOrderReconcileComplete(String date) {
        // 记录完成状态到 Redis
        String key = "dag:daily_settlement:" + date;
        redisTemplate.opsForSet().add(key, "order_reconcile");

        // 检查依赖是否都完成
        Set<String> completed = redisTemplate.opsForSet().members(key);
        if (completed != null && completed.contains("order_reconcile")
                              && completed.contains("pay_reconcile")) {
            // 所有前置依赖完成，触发佣金计算
            rocketMQTemplate.syncSend("task-trigger-topic",
                "commission_calc:" + date);
            // 清理状态
            redisTemplate.delete(key);
        }
    }
}
```

## 生产踩坑记录

### 坑 1：分片数据倾斜

```
场景：MOD(id, 10) 分 10 片
     → id 是自增的，部分分片分到更多数据（id 分布不均匀）
     → 某个分片跑了 1 小时，其他分片 10 分钟就完了

解决：
1. 用业务字段分片：MOD(merchant_id, 10)，商户分布更均匀
2. 动态分片：先 COUNT 预估数据量，自动计算最优分片数
3. 工作窃取（ElasticJob 支持）：快的节点自动拿慢的节点的任务
```

### 坑 2：任务执行中途挂了，部分数据处理了

```
场景：分片任务跑到一半，OOM 或节点重启
     → 已处理的数据无法回滚
     → 重跑又从头开始，已处理的重复处理

解决：
1. 幂等处理：每条记录处理前检查是否已处理
2. 进度记录：记录最后处理的 ID/时间戳，重跑时从断点继续
3. 事务粒度：小批量提交（每 1000 条 commit 一次），减少重放范围
```

```java
// 断点续跑
@XxlJob("syncJobHandler")
public void syncJobHandler() {
    String progressKey = "job:sync:progress:" + ShardingUtil.getShardingVo().getIndex();

    // 1. 获取上次处理到的位置
    Long lastId = redisTemplate.opsForValue().get(progressKey, Long.class);
    if (lastId == null) lastId = 0L;

    // 2. 从断点继续查询
    List<DataRecord> records = dataMapper.selectByIdGreaterThan(lastId, BATCH_SIZE);
    for (DataRecord record : records) {
        syncService.process(record);
        lastId = record.getId();
    }

    // 3. 更新进度
    redisTemplate.opsForValue().set(progressKey, lastId);
}
```

### 坑 3：大量任务同时触发撑爆线程池

```
场景：凌晨 2 点，100 个任务同时触发
     → Executor 线程池被打满
     → 任务排队等待，互相影响

解决：
1. 错峰配置：不同任务分散到不同时间点（02:00, 02:10, 02:20...）
2. 线程池隔离：不同优先级的任务用不同线程池
3. 阻塞策略：XXL-JOB 支持「丢弃后续调度」「覆盖之前调度」「串行阻止」
4. 资源限制：给每个任务配置最大并发数
```

### 坑 4：Admin 单点故障

```
场景：XXL-JOB Admin 挂了 → 所有任务都不触发了

解决：
1. Admin 部署两个实例，前端 Nginx 负载均衡
2. DB 做主从（Admin 无状态，多实例共享同一个 DB）
3. Admin 挂了不影响正在执行的任务（Executor 已经接到了触发请求）
4. Admin 恢复后自动继续调度
```

## 追问链

### 追问 1：XXL-JOB 和 Quartz 集群有什么区别？

> "核心区别在架构：Quartz 是去中心化的，靠 DB 行锁抢占执行权，性能瓶颈在 DB；XXL-JOB 是中心化的，Admin 统一调度，通过 HTTP 触发 Executor，DB 压力小。另外 XXL-JOB 自带 Web 管理界面，支持分片广播、失败重试、路由策略、动态修改 cron，这些 Quartz 要自己实现。简单说：Quartz 适合简单场景，XXL-JOB 适合生产环境。"

### 追问 2：任务执行时间不确定（比如依赖外部接口），怎么处理超时？

> "XXL-JOB 支持配置超时时间，超时后自动中断（但注意 Java 线程中断是协作式的，需要代码检查中断标志）。更实用的做法：1. 设置合理的超时阈值（比正常执行时间多 50% buffer）。2. 超时后发告警，但不要自动杀进程（可能造成脏数据）。3. 关键外部调用设置独立的超时和重试（如 OkHttp client timeout=5s, retry=2）。4. 外部接口返回 slow 的告警提前介入。"

### 追问 3：如何保证任务不重复执行？

> "三个层面保证：1. XXL-JOB 层面：路由策略选故障转移或轮询，同一时刻只有一个节点被选中。2. 业务层面：关键操作用幂等设计（唯一索引/状态机/Redis SETNX）。3. 数据层面：任务处理前检查处理标记，避免重入。最可靠的方案是业务幂等——调度框架可能会出 bug，但业务幂等永远兜底。"

### 追问 4：任务执行记录太多，日志表怎么处理？

> "日志表增长很快（每天几百万条）。处理方式：1. 按月分表（xxl_job_log_202601, xxl_job_log_202602...）。2. 只保留近 30 天的热数据，历史数据归档到 HDFS/ES。3. 成功日志保留 7 天，失败日志保留 90 天（排查需要）。4. XXL-JOB 自带日志清理功能，可以配置自动清理策略。"

### 追问 5：如果不想引入 XXL-JOB 这种重量级框架，轻量级怎么做？

> "轻量级方案：1. Spring @Scheduled + Redis 分布式锁（SETNX），最简单。2. ShedLock（专门为 Spring 定时任务加分布式锁的库，一行注解搞定）。3. 数据库轮询方案：用一张任务表记录下次执行时间和状态，定时任务每分钟扫一次，CAS 更新状态。适合任务数量不多（<50）、对可靠性要求不极端的场景。"

□ 路由策略选对了吗？ → 分片任务用 SHARDING，单机任务用 FAILOVER
□ 多实例会重复执行吗？ → 路由策略 + 业务幂等
□ 任务超时会告警吗？ → 配置超时阈值 + 告警渠道
□ 失败重试次数合理吗？ → 根据业务重要性配置 0~5 次
□ 日志表膨胀了吗？ → 配置自动清理，保留近 30 天
□ 分片数据均匀吗？ → 检查分片键的分布
□ Admin 高可用了吗？ → 至少 2 个实例 + Nginx LB
□ 任务间依赖怎么管？ → DAG 或子任务触发
