# 质量门禁

> 质量门禁（Quality Gate）是 CI/CD 流水线中的"守门人"，确保不合格的代码不能进入下一阶段。没有质量门禁的流水线只是"自动化搬运"——把低质量代码更快地搬到了生产环境。

---

## 这个问题为什么存在？

没有质量门禁的团队是什么样的？

```
场景 A：代码"能跑就行"
  开发者提交代码 → CI 只做编译和打包 → 直接部署到生产
  → 线上出了空指针异常，排查发现是未处理的边界 case
  → 单元测试覆盖率只有 30%，这个 case 根本没被测试覆盖到

场景 B：代码评审靠"自觉"
  没有自动化检查，全靠人工 Code Review
  → Reviewer 疲劳，漏掉了 SQL 注入风险
  → 生产环境被注入，数据泄露

场景 C：安全漏洞"事后发现"
  项目依赖了 log4j2 2.14.0（CVE-2021-44228），没人知道
  → 安全团队扫描才发现，但已经运行了半年
```

**根本原因**：软件交付是多人协作的持续过程，人工检查无法做到"每次提交都覆盖所有质量维度"。质量门禁的本质是**把质量标准从"人的自觉"变成"系统的强制"**。

---

## 它是怎么解决问题的？

### 三层门禁模型

质量门禁按「距离生产环境的远近」分为三层，逐层加严：

```
┌─────────────────────────────────────────────┐
│           第三层：发布门禁（Release Gate）       │
│   集成测试 + 性能测试 + 安全审计 + 灰度验证      │
├─────────────────────────────────────────────┤
│           第二层：集成门禁（Integration Gate）    │
│   代码审查 + 集成测试 + 代码覆盖率 + 安全扫描     │
├─────────────────────────────────────────────┤
│           第一层：提交门禁（Commit Gate）         │
│   编译检查 + 单元测试 + 代码格式 + 静态分析       │
└─────────────────────────────────────────────┘
```

**核心设计**：越靠近生产环境，检查越全面、越严格。第一层必须快（<5分钟），否则开发者体验极差。

### 第一层：提交门禁（Commit Gate）

每次提交自动触发，确保代码"基本可用"：

| 指标 | 阈值示例 | 工具 |
|------|---------|------|
| 编译成功 | 必须通过 | Maven/Gradle |
| 单元测试通过率 | 100% | JUnit/TestNG |
| 代码格式检查 | 0 Error | Checkstyle |
| 静态代码分析 | 0 Critical Bug | SonarQube/SpotBugs |
| 提交信息规范 | 符合 Conventional Commits | commitlint |

提交门禁失败 → **立即阻断**，开发者必须修复后重新提交。

### 第二层：集成门禁（Integration Gate）

合并到主分支时触发，确保代码"质量达标"：

| 指标 | 阈值示例 | 工具 |
|------|---------|------|
| 代码审查（Code Review） | 至少 1 人 Approve | GitLab/GitHub PR |
| 单元测试覆盖率 | ≥ 80%（新增代码 90%） | JaCoCo |
| 集成测试通过率 | 100% | Testcontainers/SpringBootTest |
| 安全漏洞扫描 | 0 High/Critical | OWASP Dependency-Check |
| API 兼容性检查 | 无 Breaking Change | Swagger/Diff |

集成门禁失败 → **合并被阻止**，必须修复后重新提交 MR/PR。

### 第三层：发布门禁（Release Gate）

部署到生产环境前触发，确保"上线安全"：

| 指标 | 阈值示例 | 工具 |
|------|---------|------|
| 性能基线 | 响应时间 P99 < 500ms | JMeter/Gatling |
| 压力测试 | 吞吐量不低于基线 90% | wrk/k6 |
| 安全审计 | 通过 | SAST/DAST |
| 灰度验证 | 错误率 < 0.1% | APM（SkyWalking/Prometheus） |
| 回归测试 | 通过 | Selenium/Appium |

发布门禁失败 → **阻止发布**，必须修复后重新走发布流程。

### 代码覆盖率：JaCoCo 配置

```xml
<!-- Maven JaCoCo 插件配置 -->
<plugin>
    <groupId>org.jacoco</groupId>
    <artifactId>jacoco-maven-plugin</artifactId>
    <version>0.8.11</version>
    <executions>
        <execution>
            <goals>
                <goal>prepare-agent</goal>
            </goals>
        </execution>
        <execution>
            <id>report</id>
            <phase>test</phase>
            <goals>
                <goal>report</goal>
            </goals>
        </execution>
        <execution>
            <id>check</id>
            <phase>verify</phase>
            <goals>
                <goal>check</goal>
            </goals>
            <configuration>
                <rules>
                    <rule>
                        <!-- 增量覆盖率：只检查本次变更的代码 -->
                        <element>PACKAGE</element>
                        <limits>
                            <limit>
                                <counter>LINE</counter>
                                <value>COVEREDRATIO</value>
                                <!-- 新增代码行覆盖率不低于 90% -->
                                <minimum>0.90</minimum>
                            </limit>
                            <limit>
                                <counter>BRANCH</counter>
                                <value>COVEREDRATIO</value>
                                <minimum>0.80</minimum>
                            </limit>
                        </limits>
                    </rule>
                </rules>
            </configuration>
        </execution>
    </executions>
</plugin>
```

覆盖率类型从弱到强：行覆盖率 → 分支覆盖率 → 条件覆盖率。看分支覆盖率比看行覆盖率更有意义——一个 if/else 只走了一个分支，行覆盖率 100% 但分支覆盖率只有 50%。

### 静态代码分析：SonarQube

```yaml
# sonar-project.properties
sonar.qualitygate.wait=true  # 流水线等待质量门禁结果

# 质量门禁规则示例：
# - 新增代码覆盖率 ≥ 80%
# - 重复率 ≤ 3%
# - 可靠性评级 ≥ A（0 Bug）
# - 安全性评级 ≥ A（0 Vulnerability）
# - 代码异味密度 ≤ 5%
```

代码异味等级：

| 级别 | 说明 | 示例 |
|------|------|------|
| Blocker | 必须修复 | 空指针风险、SQL 注入、硬编码密码 |
| Critical | 强烈建议修复 | 未关闭的资源、空 catch 块 |
| Major | 建议修复 | 过长方法（>80行）、圈复杂度过高 |
| Minor | 可选修复 | 魔法数字、命名不规范 |

### 安全扫描：三层防线

```
第一层 SAST（静态分析）—— 编译期扫源码
  工具：SonarQube Security / SpotBugs + FindSecBugs / Semgrep
  检查：SQL 注入、XSS、硬编码密钥、不安全反序列化

第二层 SCA（成分分析）—— 扫依赖的已知漏洞
  工具：OWASP Dependency-Check / Snyk / Trivy
  场景：log4j2 漏洞（CVE-2021-44228）爆发时，快速定位受影响项目

第三层 DAST（动态分析）—— 运行时模拟攻击
  工具：OWASP ZAP / Burp Suite
  通常在预发布环境执行
```

### CI/CD 平台集成示例

**GitLab CI/CD**：

```yaml
stages:
  - build
  - test
  - quality-gate
  - deploy

build:
  stage: build
  script:
    - mvn clean compile

unit-test:
  stage: test
  script:
    - mvn test
  coverage: '/Total.*?([0-9]{1,3})%/'

sonarqube-check:
  stage: quality-gate
  script:
    - mvn verify sonar:sonar -Dsonar.qualitygate.wait=true
  allow_failure: false  # 质量门禁失败则阻断

deploy:
  stage: deploy
  script:
    - kubectl apply -f k8s/
  when: manual  # 门禁通过后手动触发
```

**Jenkins Pipeline**：

```groovy
pipeline {
    agent any
    stages {
        stage('Build') {
            steps { sh 'mvn clean compile' }
        }
        stage('Test') {
            steps { sh 'mvn test' }
            post {
                always { junit '**/target/surefire-reports/*.xml' }
            }
        }
        stage('Quality Gate') {
            steps { sh 'mvn verify sonar:sonar' }
        }
        stage('Deploy') {
            when { branch 'main' }
            steps { sh 'kubectl apply -f k8s/' }
        }
    }
}
```

## 深入原理

### Checkstyle vs SpotBugs vs SonarQube 的本质区别

这三个工具经常被混淆，但它们的检查维度完全不同：

- **Checkstyle**：只查"代码风格"——命名规范、缩进、import 顺序、最大行数。类似语文老师改错别字，不关心逻辑对不对
- **SpotBugs**：查"代码逻辑缺陷"——空指针、资源泄漏、死锁。通过分析字节码找 Bug，类似数学老师检查解题过程
- **SonarQube**：全维度覆盖——代码风格 + 逻辑缺陷 + 安全漏洞 + 重复率 + 复杂度。它是门禁的"总管"，可以把 Checkstyle 和 SpotBugs 的结果汇总到一起

实际项目中三者配合使用：Checkstyle 管风格，SpotBugs 管逻辑，SonarQube 做总门禁。

### 覆盖率的"数字游戏"

行覆盖率 80% 不代表质量高。看这个例子：

```java
public String process(Order order) {
    if (order == null) {                    // 分支A
        throw new IllegalArgumentException();
    }
    if (order.getAmount() > 1000) {         // 分支B
        return discount(order);              // 分支C
    }
    return order.toString();                 // 分支D
}
```

只测 `process(null)` 和 `process(new Order(500))` → 行覆盖率 100%，但分支覆盖率只有 50%（分支B的true路径、分支C、分支D都没走到）。

**为什么增量覆盖率比全量覆盖率更重要？** 历史代码覆盖率差是既成事实，强行补全性价比极低。增量覆盖率（只看本次提交变更的代码）才能真正反映"新写的代码质量如何"。

### SAST vs SCA vs DAST 的检测时机

```
代码编写 → 编译 → 测试 → 部署 → 运行
  │         │       │       │       │
  └─ SAST ──┘       │       │       │  扫源码，找逻辑漏洞
            └─ SCA ──┘       │       │  扫依赖，找已知CVE
                    └─ DAST ──┘       │  扫运行中的应用，模拟攻击
```

SAST 最早介入，成本最低；DAST 最晚但最接近真实攻击场景。三者互补，不是替代关系。

## 正确使用方式

### 渐进式引入策略

不要一开始就上全套门禁，否则团队会抵触：

```
阶段 1（基础）：编译 + 单元测试通过          ← 几乎零成本，立即收益
阶段 2（增强）：+ 代码覆盖率 ≥ 80% + Checkstyle  ← 加代码质量维度
阶段 3（完善）：+ SonarQube 扫描 + 安全检查      ← 加安全和综合质量维度
阶段 4（高级）：+ 性能基线 + 灰度验证            ← 加发布级保障
```

### 覆盖率阈值设置建议

```
整体覆盖率：
- 核心业务模块 ≥ 80%
- 工具类/通用组件 ≥ 90%
- Controller 层 ≥ 60%（重点测 Service 层）

增量覆盖率（推荐）：
- 新增代码行覆盖率 ≥ 90%
- 新增代码分支覆盖率 ≥ 80%

不要盲目追求 100%：
- getter/setter 不需要测试
- 简单的 CRUD 可以适当放宽
- 异常分支的覆盖性价比低时可以豁免
```

### 质量门禁的"好味道"

```
✓ 快速反馈（< 10 分钟）
✓ 规则透明（团队共同制定，写在 README 或 wiki 里）
✓ 可追踪（每次构建的质量报告可回溯）
✓ 可调整（定期回顾和优化阈值）
✓ 自动化（无需人工干预）
```

## 边界情况和坑

### 覆盖率"造假"

```java
// 为了覆盖率而写的"废测试"
@Test
void testProcess() {
    service.process(new Order());  // 只测了正常路径，异常路径一个没覆盖
}
```

**成因**：只看行覆盖率不看分支覆盖率。行覆盖率 100% 但只走了一个分支，隐藏了大量未覆盖的边界 case。

**解决**：门禁必须检查分支覆盖率（BRANCH），不要只看行覆盖率（LINE）。JaCoCo 配置中同时设置 LINE 和 BRANCH 的最低阈值。

### 扫描工具误报导致门禁频繁失败

**成因**：SonarQube 把 try-catch 中的 `catch(Exception e)` 标为 Critical（认为不应该捕获通用异常），但实际上这是合理的兜底逻辑。

**解决**：配置排除规则（`// NOSONAR` 注释），但要定期审查——排除规则太多意味着门禁在退化。

### 门禁太慢阻塞交付

**成因**：质量门禁串行执行，编译 5 分钟 + 测试 10 分钟 + SonarQube 扫描 8 分钟 + 安全扫描 5 分钟 = 28 分钟。开发者等不起，开始绕过门禁。

**解决**：
- 代码风格检查和单元测试并行执行
- SonarQube 用增量分析（只扫本次变更的代码）
- SCA 扫依赖用缓存，依赖没变就跳过

### 门禁失败后的"绕过文化"

**成因**：门禁太严格 + 没有豁免机制，紧急修复时只能手动跳过。一旦有人成功绕过，其他人会效仿，门禁形同虚设。

**解决**：
- 分级门禁：Blocker（必须通过）和 Major（可由 Tech Lead 审批豁免）
- 豁免记录留痕：谁豁免的、为什么、影响范围
- 定期回顾豁免记录，调整不合理的门禁规则
