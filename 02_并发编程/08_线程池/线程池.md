# 线程池

> 线程是昂贵资源——创建、启动、销毁都有开销。线程池的核心思想是**线程复用 + 任务排队 + 可控并发数**，用有限的线程处理大量的任务。

---

## 这个问题为什么存在？

每次来一个任务就 `new Thread(task).start()`，代价极高：

- 操作系统分配 PCB（进程控制块）
- 分配 1MB 栈内存（默认 `-Xss`）
- 用户态 → 内核态切换

高并发场景下，线程数暴增 → 内存耗尽 → 系统假死。

Java 提供了两套线程池实现：
- `ThreadPoolExecutor`：基于**共享队列**模型，核心/临时线程都从同一个队列取任务
- `ForkJoinPool`：基于**工作窃取（work-stealing）** 模型，每个线程有自己的双端队列，适合分治型任务

---

## 核心脉络

### ThreadPoolExecutor 的 7 个参数

```java
new ThreadPoolExecutor(
    int corePoolSize,      // ① 核心线程数（常住）
    int maximumPoolSize,   // ② 最大线程数（临时线程上限）
    long keepAliveTime,    // ③ 临时线程空闲存活时间
    TimeUnit unit,         // ④ 时间单位
    BlockingQueue<Runnable> workQueue,  // ⑤ 任务队列
    ThreadFactory threadFactory,          // ⑥ 线程工厂
    RejectedExecutionHandler handler      // ⑦ 拒绝策略
);
```

**任务提交流程**（面试必问）：

```
任务 submit(task)
  │
  ├─ 1. 当前线程数 < corePoolSize？
  │     是 → 创建核心线程，执行 task
  │     否 → 继续
  │
  ├─ 2. 队列 workQueue.offer(task) 成功？
  │     是 → 任务入队，等待执行
  │     否 → 继续
  │
  ├─ 3. 当前线程数 < maximumPoolSize？
  │     是 → 创建临时线程，执行 task
  │     否 → 继续
  │
  └─ 4. 执行拒绝策略（handler.rejectedExecution）
```

> **关键点**：队列满了之后，才会创建「超过 corePoolSize」的线程。

### ForkJoinPool 的分治模型

`ForkJoinPool` 是为**分治任务**设计的，核心思想是工作窃取：

```
普通线程池（共享队列）：
  Thread1 ──→  ┌──────────┐  ← Thread2 也来这里取
  Thread2 ──→  │ 共享队列  │
                └──────────┘

ForkJoinPool（工作窃取）：
  Thread1  ─→  [deque1]  ← 空闲线程从这里偷（从队尾偷）
  Thread2  ─→  [deque2]
  Thread3  ─→  [deque3]
```

**适合场景**：大任务可以递归拆分成小任务（归并排序、并行计算、文件遍历）。
