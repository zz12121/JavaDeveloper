## 1. MySQL 架构

**Q: MySQL 的一条 SQL 是怎么执行的？**

**A:**
```
连接层：连接管理、认证、安全
  ↓
SQL 层：
  (1) 解析器（Parser）      — 词法/语法解析 → AST
  (2) 预处理器（Preprocessor）— 语义检查（表/字段存在？）
  (3) 优化器（Optimizer）     — 生成执行计划（代价模型）
  (4) 执行器（Executor）     — 调用存储引擎执行
  ↓
存储引擎层（InnoDB / MyISAM）：
  (5) 执行计划 → 读写 Buffer Pool / 磁盘

InnoDB 的关键：
  - Buffer Pool：缓存数据页和索引页
  - Redo Log：事务持久性保证
  - Undo Log：事务回滚 + MVCC
```

**Q: InnoDB vs MyISAM 的核心区别？**

**A:**
| 维度 | InnoDB | MyISAM |
|------|--------|--------|
| 事务 | ✅ 支持 ACID | ❌ 不支持 |
| 外键 | ✅ 支持 | ❌ 不支持 |
| 锁粒度 | 行锁（高并发） | 表锁 |
| 并发 | 乐观并发控制 + MVCC | 表锁，高并发差 |
| 全文索引 | ✅（MySQL 5.6+）| ✅ 原生支持 |
| 崩溃恢复 | 自动（Redo Log） | 慢（需要完整修复） |
| 存储结构 | 聚簇索引（数据即索引） | 非聚簇（索引文件 + 数据文件分离） |
| 适用场景 | OLTP（主流选择） | OLAP/只读/小表 |

## 2. 索引

**Q: 为什么 MySQL 用 B+ 树而不是 B 树/Hash/红黑树？**

**A:**
```
B+ 树 vs 其他结构：

红黑树：        — 树高不可控（平衡因子 2），深度 O(log n)，百万数据 ~20 层
Hash：           — 等值查询 O(1)，但范围查询要全表扫描，无法排序

B 树：           — 每个节点存数据和索引，非叶子节点也存数据
B+ 树：          — 只有叶子节点存数据，非叶子节点只存索引（键）
  → 相同节点大小下，B+ 树的非叶子节点能存更多索引
  → 树高更矮，磁盘 IO 更少
  → 所有叶子节点形成有序链表 → 范围查询友好

InnoDB 选 B+ 树的原因：
  1. 磁盘友好：每个节点一个数据页（16KB），非叶子节点只存键
  2. 范围查询友好：叶子链表 → 不需要回表
  3. 查询稳定：每次查询都要到叶子，复杂度固定
```

**Tags:** #数据库 #索引 #B+树

**Q: 聚簇索引 vs 非聚簇索引？回表查询？**

**A:**
```
聚簇索引（Clustered Index）：
  - 主键索引 = 数据本身
  - 叶子节点存储完整的行数据
  - 每个表只有一个聚簇索引

非聚簇索引（Secondary Index）：
  - 非主键索引
  - 叶子节点存储：索引列的值 + 主键值
  - 需要回表：拿主键值 → 再查聚簇索引 → 获取完整行数据

回表查询：
  SELECT * FROM users WHERE name = 'Alice';
  (1) 先在 name 索引中找 'Alice' → 得到主键 ID=100
  (2) 再用 ID=100 查聚簇索引 → 取完整行数据

减少回表：用覆盖索引
  SELECT name, age FROM users WHERE name = 'Alice';
  → 如果 (name, age) 是联合索引，name 和 age 都在索引里
  → 不需要回表 → 性能更快（索引覆盖）
```

**Tags:** #数据库 #聚簇索引 #非聚簇索引 #回表

**Q: 索引失效的常见场景？**

**A:**
```
失效场景（最常见）：

1. 最左前缀原则
   - 联合索引 (a, b, c)，查询只用 b 或 c → 失效
   - 查询：a = 1 AND c = 1 → 只用到 a 的索引

2. 隐式类型转换
   - 字段类型是 VARCHAR，传入 INT
   - WHERE phone = 13800138000（数字）→ 索引失效
   - 字符串列加了数字，MySQL 会把列转成数字再比较

3. 使用函数/表达式
   - WHERE YEAR(created_at) = 2024 → 失效
   - WHERE created_at >= '2024-01-01' → 有效

4. LIKE 前缀通配符
   - WHERE name LIKE '%Alice%' → 失效（无法用索引顺序）
   - WHERE name LIKE 'Alice%' → 有效（可以用索引）

5. OR 连接
   - WHERE name = 'A' OR age = 20 → 前面用了索引，后面 age 的索引失效
   - 解决：拆成两个查询 + UNION

6. 不等于判断
   - WHERE status != 'active' → 索引失效
   - 解决：反过来查 active 或用 IN
```

## 3. 事务

**Q: ACID 各自靠什么保证？**

**A:**
```
原子性（Atomicity）：
  → Undo Log（回滚日志）
  → 事务回滚时，逆向执行已做的修改

一致性（Consistency）：
  → 应用层保证（数据库本身无法完全保证）
  → 约束（外键、唯一索引）+ 代码逻辑

隔离性（Isolation）：
  → 锁机制 + MVCC
  → 控制并发事务之间的干扰程度

持久性（Durability）：
  → Redo Log（重做日志）
  → 事务提交时，先写 Redo Log 到磁盘，再返回成功
  → 崩溃时用 Redo Log 恢复
```

**Tags:** #数据库 #事务 #ACID #UndoLog #RedoLog

**Q: MVCC 的原理？ReadView 是什么？**

**A:**
```
MVCC（Multi-Version Concurrency Control）：
  → 同一行数据可以同时存在多个版本
  → 不同事务看到不同的数据版本

InnoDB 实现：
  每行有两个隐藏列：
    DB_TRX_ID — 最近修改的事务ID
    DB_ROLL_PTR — 指向 Undo Log 旧版本的指针（版本链）
    DB_ROW_ID — 主键（如果没有显式主键）

版本链：
  UPDATE 时，旧数据写入 Undo Log，通过 roll_ptr 串成链表

ReadView（快照读）：
  当事务执行 SELECT 时，生成 ReadView：
    m_ids：活跃事务 ID 列表
    min_trx_id：最小活跃事务 ID
    max_trx_id：创建 ReadView 时最大事务 ID + 1
    creator_trx_id：当前事务 ID

可见性判断：
  1. trx_id == creator_trx_id → 可见（自己改的）
  2. trx_id < min_trx_id → 可见（已提交）
  3. trx_id >= max_trx_id → 不可见（快照生成后开启的事务）
  4. trx_id 在 m_ids 中 → 不可见（活跃事务未提交）
  5. 否则 → 通过 roll_ptr 读 Undo Log，重复判断
```

**Tags:** #数据库 #MVCC #ReadView #InnoDB

**Q: Read Committed vs Repeatable Read 的 MVCC 区别？**

**A:**
```
Read Committed（RC）：
  - 每次读取都生成新的 ReadView
  - 只能读到已提交事务的修改
  - 问题：同一事务内两次 SELECT 可能看到不同的数据（不可重复读）

Repeatable Read（RR）：
  - 事务第一次读取时生成 ReadView，后续复用同一个
  - 整个事务期间看到的快照一致
  - InnoDB 默认隔离级别

关键区别：ReadView 生成时机
  RC：每次 SELECT 重新生成
  RR：事务开始时生成，后续 SELECT 复用

RR 解决了脏读和不可重复读，但可能有幻读问题
InnoDB 的 Next-Key Lock（临键锁）在 RR 级别下锁住范围，防止幻读
```

## 4. 锁

**Q: InnoDB 的行锁是怎么实现的？**

**A:**
```
InnoDB 行锁索引记录锁：

表锁：LOCK TABLE（DDL/大表操作时用）
行锁：InnoDB 引擎层面，基于索引实现

行锁类型：
  Record Lock（记录锁）：锁住某一条索引记录
    SELECT * FROM users WHERE id = 1 FOR UPDATE;

  Gap Lock（间隙锁）：锁住索引之间的间隙（防止幻读）
    SELECT * FROM users WHERE id BETWEEN 10 AND 20 FOR UPDATE;
    → 锁住 (10, 20) 这个区间，新插入 id=15 会被阻塞

  Next-Key Lock（临键锁）：Record Lock + Gap Lock
    InnoDB RR 级别的默认锁
    锁住某条记录 + 它前面的间隙

行锁的触发条件：
  - 查询条件有索引 → 索引记录锁
  - 查询条件无索引 → 锁住整个表（所有记录 + 间隙）

⚠️ 锁住索引：如果 UPDATE WHERE name = 'Alice'
  name 有索引 → 锁 name 索引记录
  如果 name 没索引 → 锁聚簇索引（主键） → 全表锁定！
```

**Tags:** #数据库 #InnoDB #行锁 #GapLock #Next-KeyLock

**Q: 死锁是怎么产生的？如何排查？**

**A:**
```
死锁产生的条件（同时满足）：
  1. 互斥：资源一次只能被一个事务持有
  2. 持有并等待：持有资源 A，请求资源 B
  3. 不可抢占：资源不能被强制释放
  4. 循环等待：A→B，B→A

典型死锁场景：
  事务 A：UPDATE t SET x=1 WHERE id=1（锁 id=1）
          UPDATE t SET x=2 WHERE id=2（锁 id=2）
  事务 B：UPDATE t SET x=2 WHERE id=2（锁 id=2）
          UPDATE t SET x=1 WHERE id=1（等待 A 释放 id=1）→ 死锁！

解决：InnoDB 自动检测死锁（等待图检测循环依赖）
  → 回滚代价最小的事务（undo log 少）

排查命令：
  SHOW ENGINE INNODB STATUS;  → 查看最近死锁的事务详情
  information_schema.INNODB_LOCKS    → 查看锁信息
  information_schema.INNODB_LOCK_WAITS  → 查看锁等待

预防死锁：
  1. 按固定顺序访问表/记录（多个事务用同样顺序加锁）
  2. 减少锁的持有时间
  3. 降低隔离级别（但影响数据一致性）
  4. 添加合理索引（减少锁的范围）
```

## 5. SQL 优化

**Q: EXPLAIN 各字段含义？**

**A:**
```
EXPLAIN SELECT * FROM users WHERE name = 'Alice';

id            — 查询中 SELECT 的顺序（子查询按从内到外编号）
select_type   — 查询类型
  SIMPLE          — 简单 SELECT
  PRIMARY         — 外层查询
  SUBQUERY        — 子查询
  DERIVED         — FROM 中的子查询（派生表）
  UNION           — UNION 后的查询

type           — 访问方式（从好到差）
  system > const > eq_ref > ref > range > index > ALL
  const          — 主键/唯一索引等值查询（最快）
  ref            — 普通索引等值查询
  range          — 索引范围查询（> < BETWEEN IN）
  index          — 全索引扫描
  ALL            — 全表扫描（最差，要优化！）

key            — 实际使用的索引
rows           — 预估扫描的行数
Extra          — 额外信息
  Using index     — 覆盖索引，无需回表
  Using where     — 服务层过滤（不是引擎层）
  Using filesort — 文件排序（需要优化）
  Using temporary — 临时表
  Using index condition — 索引条件下推（ICP）
```

**Q: 深分页问题怎么优化？**

**A:**
```
深分页问题：LIMIT 100000, 10 → 扫描 100010 行，只返回 10 行

原因：
  SELECT * FROM orders ORDER BY id LIMIT 100000, 10;
  → 先用 id 索引排序，再回表取 100000 条完整数据

优化方案：

1. 延迟关联（改用索引）
  SELECT * FROM orders o
  INNER JOIN (SELECT id FROM orders ORDER BY id LIMIT 100000, 10) t
  ON o.id = t.id;
  → 子查询只查 id（索引覆盖），减少回表

2. 游标分页（基于上一页最大ID）
  SELECT * FROM orders WHERE id > 100000 ORDER BY id LIMIT 10;
  → O(1)，性能稳定，不随页数增加而变慢

3. 记录总数（不上翻页按钮）
  SELECT COUNT(*) FROM orders → 返回总数
  → 前端不上翻页按钮，用"上一页/下一页"

4. Elasticsearch / 瀑布流（大数据量）
  → 分页交给搜索引擎，不走 MySQL
```

## 6. 主从复制

**Q: MySQL 主从复制原理？**

**A:**
```
主从复制（异步）：

Master（主库）：
  1. 事务执行 → 写入 binlog（Binary Log）
  2. 提交事务 → binlog 记录完成

Slave（从库）：
  1. I/O Thread：连接主库，拉取 binlog → 写入 relay log（中继日志）
  2. SQL Thread：读取 relay log → 在从库执行 SQL

binlog 格式：
  STATEMENT — 记录 SQL 语句（可能有函数不确定性）
  ROW       — 记录行变化（最安全，MySQL 5.7+ 默认）
  MIXED     — 自动选择（默认 STATEMENT，特殊场景用 ROW）

GTID（Global Transaction Identifier）：
  每个事务有一个全局唯一 ID
  格式：server_uuid:transaction_id
  → 解决了传统复制位置（file:position）难管理的问题

半同步复制：
  主库等待至少一个从库确认收到 binlog 后才提交
  → 减少主从数据不一致的风险
  → 性能略低于异步复制
```

**Q: 读写分离的延迟问题怎么解决？**

**A:**
```
读写分离延迟问题：
  主库写入 → 从库还没同步完 → 读请求到从库 → 数据不一致

常见场景：
  - 批量导入后立即查询
  - 用户刚注册就查询自己的订单
  - 大促期间从库延迟

解决方案：

1. 强制路由主库
   - 写操作 + 写后立即读 → 强制走主库
   - 适合：对一致性要求高的场景

2. 延迟感知
   - 应用层记录写操作时间戳
   - 读取时检查：从库延迟 > 阈值？换主库

3. 减少主从延迟
   - 从库配置足够硬件（SSD、大内存）
   - 从库关闭 binlog（log_slave_updates=OFF）
   - 使用并行复制（slave_parallel_workers）
   - 读写分离 + 消息队列解耦（最终一致性）

4. 业务层面容错
   - 允许短暂不一致（如统计数据）
   - 前端加提示（如"数据可能有几秒延迟"）
```
