# 进程与网络

> 线上问题排查，80% 的时间花在「找到问题进程/连接」上。熟练掌握进程和网络排查命令，是区分「开发工程师」和「运维工程师」的分水岭。

---

## 这个问题为什么存在？

Java 开发最常见的问题场景：

```
用户投诉：系统变慢了
你：...（不知道从哪入手）

正确排查路径：
1. 用 top 找到 CPU/内存占用最高的进程
2. 用 jstack 拿到那个 Java 进程的线程栈
3. 用 netstat/ss 看这个进程的网络连接
4. 用 strace 跟踪系统调用，找到卡在哪
```

**为什么不能直接看日志？**

日志是「应用层」的信息，告诉你「发生了什么」。但性能问题往往是「系统层」的——IO 阻塞、网络延迟、内存不够触发 GC。这些信息不会出现在应用日志里。

## 它是怎么解决问题的？

### 进程查看三剑客：`top` / `htop` / `ps`

#### `top`：实时进程监控

```
top 输出解读：
top - 14:30:00 up 45 days,  3:22,  2 users,  load average: 0.08, 0.12, 0.15
Tasks: 156 total,   1 running, 155 sleeping,   0 stopped,   0 zombie
%Cpu(s):  2.3 us,  1.2 sy,  0.0 ni, 96.2 id,  0.3 wa,  0.0 hi,  0.0 si
MiB Mem :  15892.5 total,   234.8 free,  8234.1 used,  7423.6 buff/cache
MiB Swap:   2048.0 total,   2048.0 free,      0.0 used.  7365.4 avail Mem

    PID USER      PR  NI    VIRT    RES    SHR S  %CPU  %MEM     TIME+ COMMAND
 123456 myapp     20   0   12.3g   2.1g   3456 S  45.6   8.9  23:45.67 java
```

**关键指标解读**：

| 指标 | 含义 | 异常表现 |
|------|------|---------|
| `load average` | 1/5/15 分钟平均负载 | 持续 > CPU核数 = 过载 |
| `%CPU` | 进程占用 CPU 百分比 | 持续 100% = 死循环或密集计算 |
| `RES` | 物理内存占用（不含 swap） | 接近总内存 = 内存泄漏 |
| `TIME+` | 进程累计占用 CPU 时间 | 短时间内暴增 = 异常计算 |
| `S` | 进程状态 | `D` = 不可中断睡眠（通常是 IO 卡死） |

**交互命令**（在 `top` 界面中按）：
```
P  — 按 CPU 排序（找 CPU 热点）
M  — 按内存排序（找内存大户）
1  — 显示所有 CPU 核心的使用率
c  — 显示完整命令路径
k  — kill 掉某个进程
```

#### `ps`：快照式进程查看

```bash
# 查看 Java 进程
ps aux | grep java

# 以树状结构显示进程关系
ps auxf | grep java

# 查看某个进程的线程（找出是哪个线程 CPU 高）
ps -eLf | grep java
```

**`ps aux` 各列含义**：
```
USER   PID  %CPU %MEM VSZ    RSS   TTY   STAT START   TIME COMMAND
myapp  123  45.6 8.9  12.3g  2.1g  ?     Sl   14:00  23:45 java ...
```

### 网络连接排查：`netstat` / `ss`

**`netstat` 已过时，推荐用 `ss`**（更快，功能更强）。

```bash
# 查看所有 TCP 连接
ss -tan

# 查看监听端口
ss -tuln

# 查看某个进程的网络连接（找出这个 Java 进程和谁在通信）
ss -tanp | grep java
```

**输出解读**：
```
State      Recv-Q Send-Q Local Address:Port  Peer Address:Port  Process
ESTAB      0      0     192.168.1.100:8080  192.168.1.200:54321  users:(("java",pid=123,fd=42))
```

| 字段 | 含义 |
|------|------|
| `State` | TCP 连接状态（ESTAB = 已建立，LISTEN = 监听中，TIME_WAIT = 等待关闭） |
| `Recv-Q` | 接收队列（内核已收到但未交给应用的字节数） |
| `Send-Q` | 发送队列（应用已写入内核但未发出的字节数） |
| `Local Address:Port` | 本地 IP 和端口 |
| `Peer Address:Port` | 对端 IP 和端口 |

**高频问题：`TIME_WAIT` 堆积**

```
现象：ss -tan | grep TIME_WAIT | wc -l  输出 20000+
原因：短连接频繁建立/断开，主动断开方会进入 TIME_WAIT 状态（2MSL，通常 60秒）
解决：
  1. 改用长连接（连接池）
  2. 调整内核参数：net.ipv4.tcp_tw_reuse = 1
```

### 打开文件查看：`lsof`

```bash
# 查看某个进程打开了哪些文件（找出是哪个文件被占满）
lsof -p 12345

# 查看某个端口被哪个进程占用（端口冲突时必用）
lsof -i :8080

# 查看被删除但未释放的文件（磁盘空间被占满但找不到大文件时）
lsof | grep deleted
```

**经典场景**：删除了一个大文件，但 `df -h` 显示磁盘空间没有释放。

**原因**：文件被进程打开（文件描述符未关闭），`rm` 只是删除了目录项，内核等到进程关闭文件描述符才真正释放磁盘空间。

**解决**：重启占用文件的进程，或者 `> /path/to/bigfile` 清空文件内容。

### 系统调用跟踪：`strace`

```bash
# 跟踪某个进程的所有系统调用（找出进程卡在哪）
strace -p 12345

# 跟踪某个命令的执行（找出启动慢的原因）
strace -f java -jar app.jar 2>&1 | grep -v ENOENT | head -100
```

**`strace` 输出解读**：
```
read(3, "some data...", 4096) = 128   # 从文件描述符 3 读取了 128 字节
write(1, "hello", 5) = 5               # 向 stdout 写入 5 字节
connect(4, {sa_family=AF_INET...}, 16) = -1 EINPROGRESS  # 非阻塞连接中
```

**高频用途**：
- 进程卡住不动 → `strace -p PID` 看卡在哪个系统调用
- 启动慢 → `strace java -jar app.jar 2>&1 | ts` 找出最耗时的系统调用
- 文件找不到 → `strace java -jar app.jar 2>&1 | grep ENOENT`

## 它和相似方案的本质区别是什么？

### `top` vs `htop`

| 维度 | `top` | `htop` |
|------|-------|---------|
| 交互性 | 差（需要记键盘快捷键） | 好（支持鼠标、颜色、横向滚动） |
| 批量操作 | 不支持 | 支持（F5 树状视图，F9 kill 多个进程） |
| 默认安装 | 几乎所有 Linux 发行版 | 需要额外安装 |
| 性能 | 更轻量 | 稍重 |

**建议**：服务器上用 `top`（一定存在），本地开发用 `htop`（体验更好）。

### `netstat` vs `ss`

`netstat` 是通过读取 `/proc/net/tcp` 等文件实现的，效率低下（O(n) 遍历）。`ss` 是通过 `netlink` 套接字直接从内核获取信息，效率更高（O(1)）。

**实际差异**：当服务器有 10万+ 连接时，`netstat` 可能需要几秒，而 `ss` 几乎瞬间完成。

### `lsof` vs `/proc/<pid>/fd/`

```bash
# 用 lsof
lsof -p 12345

# 直接读 /proc
ls -l /proc/12345/fd/
```

两者功能几乎完全相同。`lsof` 是封装好的工具，输出更友好；直接读 `/proc` 更底层，在 `lsof` 不存在时可以作为备选方案。

## 正确使用方式

### 1. 用 `top` + `jstack` 定位 Java CPU 热点

```bash
# Step 1: 用 top 找到 CPU 最高的 Java 进程
top
# 假设 PID = 12345，CPU 占用 200%

# Step 2: 找到这个进程中 CPU 最高的线程
top -H -p 12345
# 假设线程 TID = 67890（十进制）

# Step 3: 把线程 ID 转成 16 进制（jstack 输出的是 16 进制）
printf "%x\n" 67890
# 输出：10932

# Step 4: 用 jstack 拿到线程栈，找 nid=0x10932 的线程
jstack 12345 | grep -A 20 "nid=0x10932"
```

**这是线上 CPU 飙高时标准排查流程，必须熟练掌握。**

### 2. 用 `ss` + `awk` 统计连接数

```bash
# 统计各个状态的 TCP 连接数
ss -tan | awk '{print $1}' | sort | uniq -c | sort -rn

# 统计连接最多的前 10 个 IP
ss -tan | awk '{print $5}' | cut -d: -f1 | sort | uniq -c | sort -rn | head -10
```

### 3. 用 `watch` 实时监控

```bash
# 每隔 1 秒刷新一次 CPU 占用最高的进程
watch -n 1 "ps aux --sort=-%cpu | head -10"

# 实时监控某个端口的连接数
watch -n 1 "ss -tan | grep :8080 | wc -l"
```

## 边界情况和坑

### 1. `ps aux` 输出被截断

```bash
# 问题：COMMAND 列只显示前 80 个字符
ps aux | grep java   # 输出被截断，看不到完整的 Java 启动参数

# 解决
ps auxww | grep java   # -w 不限制宽度
# 或者
cat /proc/12345/cmdline | tr '\0' ' '   # 直接读 /proc
```

### 2. `top` 的 `%CPU` 超过 100%

```
误解：%CPU 超过 100% 是异常
真相：%CPU 是「占用 CPU 核数的百分比」，4 核机器上，单进程最多 400%
```

### 3. `lsof` 输出太多，卡死

```bash
# 问题：lsof 扫描所有进程的所有打开文件，输出几十万行
lsof

# 解决：加过滤条件
lsof -p 12345         # 只看某个进程
lsof -i :8080         # 只看某个端口
lsof +D /tmp          # 只看某个目录下的文件
```

### 4. `strace` 对多线程程序输出混乱

```bash
# 问题：strace -p 跟踪主进程，但输出混入了所有线程的系统调用
strace -p 12345

# 解决：跟踪特定线程
strace -p 67890   # 直接跟踪线程 TID
```

## 我的理解

进程和网络排查的本质是**「自顶向下」的钻取**：

```
系统慢 → top 找进程 → top -H 找线程 → jstack 找线程栈 → 定位到具体代码行
```

这套方法论，比「瞎猜」或「看日志碰运气」效率高 10 倍。

**和 Java 工具链的配合**：

| Linux 命令 | Java 工具 | 配合用途 |
|-----------|----------|---------|
| `top` | `jstack` | 找到 CPU 高的线程 → 定位到 Java 线程栈 |
| `jstat -gc` | `top` | 确认高 CPU 是否由 GC 引起 |
| `ss` | `netstat` | 确认连接池是否泄漏 |
| `lsof` | `jmap` | 确认是否有文件句柄泄漏 |

**面试高频考点**：
1. 线上 Java 进程 CPU 100%，如何排查 —— 必须能口述完整流程（top → top -H → printf → jstack）
2. `TIME_WAIT` 状态过多怎么办 —— 改用连接池 + 调整内核参数
3. 磁盘空间满了但找不到大文件 —— `lsof | grep deleted`，重启占用已删除文件的进程

*这些命令不是「背」的，是「用」的。建议在自己机器上故意制造一些问题（死循环、端口占用、大文件删除），然后用这些命令去排查，形成肌肉记忆。*
